## Answers to Protocol Questions

### 1. Are commands integration tested?

**Yes.** Commands are integration-tested via `NewExecutorTest()` in `executor_test.go:46-73`, which creates a real `task.Executor` and runs actual Taskfiles. This is closer to integration testing than unit testing since it exercises the full task execution pipeline.

Example from `executor_test.go:452-460`:
```go
func TestDeps(t *testing.T) {
    t.Parallel()
    NewExecutorTest(t,
        WithExecutorOptions(
            task.WithDir("testdata/deps"),
            task.WithSilent(true),
        ),
        WithPostProcessFn(PPSortedLines),
    )
}
```

True CLI integration tests are limited to `signals_test.go` (lines 25-188), which spawns separate processes and sends signals to verify signal handling.

### 2. Are golden tests used?

**Yes, extensively.** The project uses the `goldie/v2` library for golden file testing. Golden files are stored in `testdata/{testname}/testdata/*.golden` directories.

Key evidence:
- `task_test.go:25`: `"github.com/sebdah/goldie/v2"`
- `executor_test.go:166-169`: Goldie setup with custom `NormalizedEqual` function
- `executor_test.go:99-101`: Fixture templating with `WithFixtureTemplating()`

Example golden file path: `testdata/deps/testdata/TestDeps.golden`

The `writeFixture()` method (`task_test.go:69-102`) writes fixture files and the `gen:fixtures` task updates them when behavior changes.

### 3. How are mocks structured?

Mocks are **generated by mockery** and stored centrally in `internal/fingerprint/checker_mock.go` (320 lines).

The mocks implement interfaces `StatusCheckable` and `SourcesCheckable`:

Factory functions create mocks with automatic cleanup:
```go

func NewMockStatusCheckable(t interface {
    mock.TestingT
    Cleanup(func())
}) *MockStatusCheckable {
    mock := &MockStatusCheckable{}
    mock.Mock.Test(t)
    t.Cleanup(func() { mock.AssertExpectations(t) })
    return mock
}
```

### 4. Is behavior tested or implementation details?

**Behavior is tested**, not implementation details. Tests verify:
- **Output content** — via golden files comparing stdout/stderr

- **Error conditions** — via `WithSetupError()` and `WithRunError()` options

Tests do not assert on internal state or implementation-specific fields. The `ExecutorTest` wrapper hides implementation details behind task execution.

## Architectural Decisions

1. **Functional options pattern** for test configuration (`task_test.go:126-294`) — Allows flexible test composition without setter methods.

2. **Centralized golden fixture directory** (`testdata/{name}/testdata/`) — Each test directory contains its own golden files alongside Taskfiles.

3. **Cross-platform normalization** via `normalizeOutput()` (`task_test.go:316-323`) — Converts CRLF→LF and backslash→forward slash for consistent comparison on Windows.

4. **Thread-safe output capture** via `SyncBuffer` (`task_test.go:383-392`) — Prevents race conditions when running parallel tests.

5. **Post-processing pipeline** via `PostProcessFn` (`task_test.go:300`) — Allows tests to normalize output (e.g., `PPSortedLines`) before comparison.

## Tradeoffs

1. **Real executor vs mocks** — Most tests use real `task.Executor` instead of mocks. This provides high confidence but can be slower and less isolated. Only `internal/fingerprint/task_test.go` uses mocks.

2. **Golden file maintenance** — Extensive golden tests require running `task gen:fixtures` when behavior changes legitimately. The benefit is confident refactoring; the cost is maintaining fixture files.

3. **Filesystem operations in tests** — Many tests operate on real files in `testdata/` directories. This can cause flakiness if tests don't clean up properly, though `fileContentTest.Run()` does remove files before running (`task_test.go:411-413`).

4. **No interface mocking for Executor** — The `Executor` interface is not mocked in tests; all executor tests run the real implementation. This limits unit test isolation.

## Failure Modes / Edge Cases

1. **Missing cleanup** — If a test crashes before cleanup, stale files may remain in `testdata/` directories, potentially causing subsequent test failures.

2. **Golden file drift** — When legitimate behavior changes, developers must remember to run `task gen:fixtures` to update golden files, or tests will fail.

3. **Cross-platform path handling** — Despite `normalizeOutput()`, some path comparisons may still fail on Windows due to backslash handling in template data (`task_test.go:97-98`).

4. **Parallel test interference** — While `SyncBuffer` protects output capture, tests sharing the same `testdata/` directory could interfere if they run concurrently without proper locking.

5. **Signal test fragility** — `signals_test.go` relies on timing and process groups, making it potentially flaky in CI environments.

## Future Considerations

1. **Increase mock usage** — Consider adding mocks for the `Executor` interface to enable faster, more focused unit tests for edge cases.

2. **Extract interface for Executor** — If not already done, creating an `Executor` interface would enable dependency injection and easier mocking.

3. **Testify suite integration** — Using `testify/suite` could provide more structured test organization with shared setup/teardown.

4. **Benchmark tests** — No benchmarking tests observed; adding benchmarks for hot paths (task resolution, dependency sorting) could guide optimizations.

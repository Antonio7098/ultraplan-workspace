## Executive Summary

Five agent-/workflow-runtimes converge on a deceptively similar surface ("we checkpoint state") but reveal three structurally distinct execution-state models. **temporal** (9/10) treats history as the journal and mutable state as a materialized projection, branching via fork-tokens — the only example here that achieves mature, scale-proven durability. **agent-framework** (8/10) and **langgraph** (8/10) take the Pregel superstep view and capture *committed* state into pluggable storage; langgraph goes further with first-class time travel and source-tagging while agent-framework mirrors the model twice (Python and .NET) with graph/topology validation. **letta** (5/10) swaps snapshots for a per-step relational event log (Run/Step/Message) over Postgres/SQLite — durable and clear but lacks branching and replay. **openhands** (4/10) deliberately splits responsibility, keeping execution state inside the sandbox and persisting only an event mirror plus metadata; durability depends on a container staying alive.

Across the five sources, the recurring design choices are: capture at a well-defined execution boundary (superstep, step, transaction) rather than ad-hoc; keep a single-parent linear chain for cheap replay; exclude in-flight / pending state from the snapshot; attach structural validation (graph-signature hash, topology match, schema version) so a stale snapshot cannot be silently accepted; commit atomic writes via rename-or-CAS; and delegate prune/retention to a higher layer — where most sources either omit it entirely or warn loudly about operator footguns.

## Core Thesis

There is no single "checkpoint architecture" — there are three, each with very different failure economics:

1. **Event-sourced execution** (temporal): events are the truth; mutable state is a cache derived by replay. Resume = re-derive. Branches = fork the event stream. Best for multi-year durable workflows and complex fan-out.
2. **Pregel-style committed-state snapshots** (agent-framework, langgraph): at each superstep boundary the runner commits pending state and serializes the now-deterministic slice — a channel-state projection plus in-flight writes — into pluggable storage. Resume = load and continue from the next superstep. Time travel = load an arbitrary ancestor with a known parent link.
3. **Step-grained transactional log** (letta): every agent step writes a small, well-typed set of rows (`runs`, `steps`, `messages`) in one transaction; "checkpoint" is implicit in the head of `agent.message_ids`. Resume = re-read the head. There is no snapshotted state object.
4. **Read-model mirror + live sandbox** (openhands): the app server persists events as a *trajectory/export*, while execution state lives in the Docker sandbox. Resume = unpause the container. Restore-from-snapshot is not an app-server concern.

The chosen model is determined less by taste than by **three constraints**: (a) whether the system can replay the same events to produce the same state — if yes, event-source; if no, snapshot; (b) whether tool calls and external effects are deterministic and re-runnable — if yes, replay is cheap; if no, snapshots are safer; (c) whether the user controls the storage backend — if yes, pluggable serializers and atomic writes dominate; if no, durable framework-internal storage wins.

## Approach Models

### A. Event-sourced execution (temporal)

**Shape:** The history table (`history_node`) is the journal; the `executions` row is a materialized index. Every domain call closes a transaction via `MutableStateImpl.CloseTransactionAsMutation` (`service/history/workflow/mutable_state_impl.go:7484-7528`) or `CloseTransactionAsSnapshot` (`service/history/workflow/mutable_state_impl.go:7530-7573`). Snapshot vs. mutation is chosen per persistence mode — creates / continues-as-new / resets use snapshot, in-place updates use mutation. Branches are tokens with an `Ancestors` slice (`common/persistence/history_manager.go:37-127`, `branch_manager.go:23-227`).

**Why it scales:** Operations on a workflow reduce to "append event + write the mutation delta"; the materialization is amortized. Branching is just a new token. CAS via `db_record_version` (`common/persistence/cassandra/mutable_state_store.go:107-123`) plus shard `RangeID` fencing (`common/persistence/sql/execution.go:40-58`) gives genuine optimistic concurrency.

**Why it costs:** A cold rebuild is linear in event count (`MutableStateRebuilderImpl.applyEvents` at `service/history/workflow/mutable_state_rebuilder.go:103-775`). The `MutableStateImpl` monolith is ~10 200 lines. Without history trimming this would break at scale; with it, retention has to be branch-aware (`common/persistence/history_manager.go:212-308`).

### B. Pregel-style committed-state snapshot (agent-framework, langgraph)

**Shape:** At the end of every superstep, the runner commits pending state, then serializes a deterministic slice — channels (langgraph) or shared state + per-executor state + pending requests (agent-framework) — into a storage backend. A single `previous_checkpoint_id` / `parent_checkpoint_id` link forms a linear chain per thread.

**Why it scales:** Captures are small (channel values are typically narrow) and one-per-superstep which is tight enough for replay but loose enough to be I/O-cheap. Reads are a single key load. Branching is second-class but cheap to add because the storage layer already has the pointers.

**Why it costs:** Pluggable storage means the runner must be defensive (agent-framework swallows checkpoint failures to avoid aborting the run, `_runner.py:236-238`; langgraph drains `put_writes` futures before the next checkpoint, `_loop.py:1515-1524`). Schema evolution is hard once captures are out in user-controlled files / Postgres JSONB.

### C. Step-grained relational log (letta)

**Shape:** Every agent step performs four discrete, transactional writes: a `Step` row (`letta/orm/step.py:20`), a `StepMetrics` row (`letta/orm/step_metrics.py:20`), a `messages` insert via `MessageManager.create_many_messages_async` (`letta/services/message_manager.py:477`), and an update of `agent.message_ids` (`letta/services/agent_manager.py:999`) or `conversation_messages.in_context` (`letta/services/conversation_manager.py:752`). There is no snapshotted state object — current state is the head of `agent.message_ids`. A `Run` is the top-level envelope.

**Why it scales:** Every row has a clear purpose and is independently queryable. Composite indexes on `(agent_id, conversation_id, sequence_id)` and `(run_id, sequence_id)` (`letta/orm/message.py:27-37`) keep cursor reads fast. Operational data (token usage, errors) lives next to the step that produced it.

**Why it costs:** No `parent_step_id` / `parent_run_id` anywhere; branching is only via `ConversationManager.fork_conversation` (`letta/services/conversation_manager.py:103-172`) at the conversation level. The in-source TODO at `letta/orm/agent.py:69-70` admits the JSON `message_ids` column is "dangerously flexible". Retention has no scheduled job.

### D. Read-model mirror + live sandbox (openhands)

**Shape:** The V1 app server stores each event as a one-file-per-id JSON blob (`openhands/app_server/event/event_service_base.py:190-197`) plus a SQL `AppConversationInfo` row with rolled-up `MetricsSnapshot` (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:120-125`). Pause / resume is implemented by pausing and unpausing the Docker sandbox container (`openhands/app_server/sandbox/docker_sandbox_service.py:517-548`) and the persistent workspace volume (`openhands/app_server/sandbox/docker_sandbox_service.py:566`).

**Why it scales (for the studied boundary):** Storage and query are pluggable; the event blob is interchangeable (filesystem / AWS / GCS). Resume is instant. History is replayed timestamp-sorted for UI/export only.

**Why it costs:** The actual execution state is in `openhands-sdk==1.29.0` (out of scope for this source) and in the live sandbox. If the container is lost, the conversation is archived read-only (`app_conversation_models.py:173-181`); no in-source path rebuilds an agent from the event log. Ordering relies on ISO-8601 string sort (`event_service_base.py:121-131`).

## Pattern Catalog

### Pattern 1 — Capture only the *committed* slice

**Problem:** Capturing in-flight state means a replay races with pending writes and yields a non-deterministic reproduction.

**Where seen:** agent-framework `_runner.py:218-225` runs `state.commit()` before capture; the `.NET` `StateManager.ExportStateAsync` asserts no queued updates (`StateManager.cs:245-248`). langgraph splits `Checkpoint.channel_values` from `pending_writes` (`base/__init__.py:31` `PendingWrite` tuple), so a partial superstep can be replayed. temporal's `CloseTransactionAsSnapshot` returns `softassert.UnexpectedInternalErr` if buffered events exist (`mutable_state_impl.go:7539-7546`).

**Why it works:** The replayable state is a fixed point given the events up to that point. Everything that could still change is staged separately.

**When to copy:** Any system whose state is mutated incrementally during a step.

**When it is overkill:** Read-only agents where every interaction is atomic and there are no in-flight tool calls.

**Evidence:** `agent-framework/python/packages/core/agent_framework/_workflows/_checkpoint.py:56-58`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StateManager.cs:245-248`; `langgraph/libs/langgraph/langgraph/pregel/_loop.py:432-447`; `temporal/service/history/workflow/mutable_state_impl.go:7530-7573`.

### Pattern 2 — Linear parent-chained lineage

**Problem:** Users want time travel, replay, and history views; without a chain these are ad-hoc over an unordered blob store.

**Where seen:** langgraph `parent_checkpoint_id` column (DDL at `libs/checkpoint-postgres/.../base.py:47-56`; `libs/checkpoint-sqlite/.../__init__.py:147`); agent-framework Python `previous_checkpoint_id` (`_checkpoint.py:75`) and .NET `Parent: CheckpointInfo?` (`Checkpoint.cs:39`); temporal `BranchInfo.Ancestors` slice (`branch_manager.go:188-226`).

**Why it works:** A single pointer per capture is enough for ordered iteration, replay, and arbitrary-ancestor loading. The cost is one extra column.

**When to copy:** Any user-facing capture that will be replayed, restored, or inspected.

**When it is overkill:** Internal scratch checkpoints that are always overwritten; append-only event logs (letta) where lineage is implicit in `sequence_id`.

**Evidence:** `langgraph/libs/checkpoint/langgraph/checkpoint/base/__init__.py:139-146`; `agent-framework/python/packages/core/agent_framework/_workflows/_checkpoint.py:30-117`; `temporal/common/persistence/history_manager.go:37-127`.

### Pattern 3 — Source-tagged provenance for each checkpoint

**Problem:** Operators need to know *why* a checkpoint exists (input? loop tick? user-triggered fork? post-step pause?) to reason about history.

**Where seen:** langgraph `CheckpointMetadata.source ∈ {input, loop, update, fork}` (`base/__init__.py:38-86`); temporal `WorkflowExecutionInfo` carries `TransitionHistory []VersionedTransition` plus `PreviousTransitionHistory` (`data_interfaces.go:378-398`; `mutable_state_impl.go:7859-7862`). agent-framework returns two checkpoint-source modes (graph vs functional) but does not tag inside the schema.

**Why it works:** The tag becomes a query / filter handle — `get_state_history` can show only updates, only forks, etc.

**When to copy:** When the same storage holds captures from multiple sources (UI edits, automated loops, forks).

**When it is overkill:** A system with one path that always writes the same thing.

**Evidence:** `langgraph/libs/langgraph/langgraph/pregel/_loop.py:706`, `:959`, `:1016`; `temporal/common/persistence/transitionhistory/transition_history.go:81-125`.

### Pattern 4 — Atomic persistence via rename or CAS

**Problem:** A crash mid-write must not leave a corrupted checkpoint or a half-applied mutation.

**Where seen:** agent-framework Python `FileCheckpointStorage._write_atomic` uses `tmp + os.replace` (`_checkpoint.py:317-322`); agent-framework .NET `FileSystemJsonCheckpointStore` rolls back index on failure (`FileSystemJsonCheckpointStore.cs:164-178`) and uses `FileShare.None` for process exclusivity (`FileSystemJsonCheckpointStore.cs:60-65`); temporal uses optimistic CAS on `db_record_version` (`mutable_state_store.go:107-123`) plus shard `RangeID` fencing (`sql/execution.go:40-58`); letta commits per-step in a SQL transaction (`MessageManager.create_many_messages_async` at `letta/services/message_manager.py:477`); openhands makes event files idempotent by id (`event_service_base.py:190-197`).

**Why it works:** Either the old value or the new value is visible — never a half-state.

**When to copy:** Anywhere a single logical capture is composed of multiple writes.

**When it is overkill:** Always-eventual-consistent read-models where a torn write is naturally hidden.

**Evidence:** `agent-framework/python/packages/core/agent_framework/_workflows/_checkpoint.py:317-322`; `agent-framework/python/packages/azure-cosmos/agent_framework_azure_cosmos/_checkpoint_storage.py:422-433`; `temporal/common/persistence/cassandra/mutable_state_store.go:22-31`.

### Pattern 5 — Pluggable storage behind a typed interface

**Problem:** Users will run on in-memory (tests), file (local dev), Postgres (production), or cloud blobs (scale). Each backend has different atomicity, durability, and indexability properties.

**Where seen:** agent-framework Python `CheckpointStorage` Protocol (`_checkpoint.py:119-189`); agent-framework .NET `ICheckpointStore<TStoreObject>` + `ICheckpointManager` (`Checkpointing/ICheckpointStore.cs:12-46`); langgraph `BaseCheckpointSaver` (`base/__init__.py:176`); openhands `EventServiceBase` with filesystem/AWS/GCS implementations; temporal's per-archetype `ExecutionManager` pluggable across Cassandra/MySQL/Postgres/SQLite.

**Why it works:** The runner never depends on a concrete backend. New backends plug in.

**When to copy:** Any framework that has to ship in dev and in prod. Each backend comes with conformance tests (langgraph) or filesystem locking (agent-framework).

**When it is overkill:** A single-tenant system with a fixed deployment.

**Evidence:** `langgraph/libs/checkpoint/langgraph/checkpoint/base/__init__.py:176-580`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/ICheckpointStore.cs:12-46`; `openhands/app_server/event/` (`aws_event_service.py`, `google_cloud_event_service.py`, `filesystem_event_service.py`).

### Pattern 6 — Durability / cadence knobs

**Problem:** Always writing on every step is wasteful; never writing is unsafe. Users want a per-deployment knob.

**Where seen:** langgraph `Durability = Literal["sync","async","exit"]` (`types.py:87-93`) drives `_put_checkpoint`, `put_writes`, and exit-mode drain (`_loop.py:1515-1524`, `:1769-1771`); temporal runs *synchronous* per-transaction persistence (every domain call) — durability is not user-tunable; agent-framework turns on/off via `RunnerContext.has_checkpointing()` (`_runner_context.py:97-275`); functional @workflow in agent-framework writes per *live* step (cache hits don't save, `_functional.py:419-433`).

**Why it works:** Cost vs. fidelity becomes an explicit deployment-time choice rather than an architectural accident.

**When to copy:** When the same framework is used in dev (cheap, frequent) and in prod (durability before next step).

**When it is overkill:** A system that always knows what level it wants.

**Evidence:** `langgraph/libs/langgraph/langgraph/types.py:87-93`; `agent-framework/python/packages/core/agent_framework/_workflows/_functional.py:961-1009`.

### Pattern 7 — Structural validation on restore

**Problem:** A checkpoint from a structurally different workflow silently produces wrong results.

**Where seen:** agent-framework Python `_compute_signature_hash` mixes name, step names, `co_code`, `co_names` (`_functional.py:1144-1164`); the runner raises `WorkflowCheckpointException` on mismatch (`_runner.py:275-279`). agent-framework .NET `WorkflowInfo.IsMatch` validates start executor, executor bindings, edge list, request ports, output-tag sets (`Checkpointing/WorkflowInfo.cs:46-103`). temporal's checksum is best-effort — a mismatch is metricked but the load proceeds (`mutable_state_impl.go:546-560`).

**Why it works:** Refusing a stale checkpoint is louder than silently mis-running.

**When to copy:** Any system where workflow definitions evolve over time.

**When it is overkill:** A server where the workflow binary is pinned and immutable.

**Evidence:** `agent-framework/python/packages/core/agent_framework/_workflows/_runner.py:275-279`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/WorkflowInfo.cs:46-103`.

### Pattern 8 — Per-executor (or per-component) hooks

**Problem:** Generic capture cannot know what state lives in a domain-specific executor (an LLM agent's full conversation, a tool's pending request, a Magentic plan).

**Where seen:** agent-framework Python `executor.on_checkpoint_save()` / `executor.on_checkpoint_restore(state)` (`_executor.py:493-517`); agent-framework .NET `OnCheckpointingAsync` / `OnCheckpointRestoredAsync` (`Executor.cs:338-347`) called on every executor at superstep end (`InProcessRunnerContext.cs:394-400`); langgraph captures only channels (a similarly channel-shaped abstraction); temporal exposes CHASM per-node `Metadata` + `Data` blobs (`mutable_state_impl.go:7510-7511`) and HSM sub-machines under `executionInfo.SubStateMachinesByType` (`mutable_state_impl.go:651-662`).

**Why it works:** Each component owns its own capture protocol; the runner walks the executor graph and calls the hook.

**When to copy:** When the executor graph includes components whose in-memory state is more than the framework knows about.

**When it is overkill:** A pure dataflow where the channel IS the state.

**Evidence:** `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/Executor.cs:338-347`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/WorkflowHostExecutor.cs:256-294`; `temporal/service/history/workflow/mutable_state_impl.go:651-662`.

### Pattern 9 — Two-tier restore for streaming vs. runtime

**Problem:** REST-time resume should defer re-emitting pending events until an attached subscriber; cold load should re-emit immediately.

**Where seen:** agent-framework .NET `RestoreCheckpointAsync` (public, republishes) vs `RestoreCheckpointCoreAsync` (used by `ResumeStreamAsync`, defers) (`InProcessRunner.cs:380-388`; `ICheckpointingHandle.cs:24-36`).

**Why it works:** Two callers get two correct behaviors; neither has to reimplement the lifecycle.

**When to copy:** When both interactive and batch surfaces share a checkpoint format.

**When it is overkill:** Single-surface deployments.

**Evidence:** `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunner.cs:380-388`.

### Pattern 10 — Atomic per-event / per-step log keyed by id

**Problem:** Re-delivery (network retry, idempotent producer) must not duplicate captures.

**Where seen:** openhands names event files by event UUID (`event_service_base.py:190-197`); letta uses `otid` as idempotency key (`letta/data_sources/redis_client.py:252-...`); langgraph uses UUIDv6 monotonically-time-ordered ids (`base/id.py:79-109`); agent-framework uses UUIDs and stores the previous-id link to detect anomalies.

**Why it works:** Idempotent writers can produce the same capture twice without observable harm.

**When to copy:** When the producer is a remote sandbox / worker.

**When it is overkill:** Synchronous in-process writers.

**Evidence:** `openhands/app_server/event/event_service_base.py:190-197`; `langgraph/libs/checkpoint/langgraph/checkpoint/base/id.py:79-109`.

### Pattern 11 — Branch via fork token / new ancestor slice

**Problem:** Reset, replay-with-alt-input, and time-travel all need a way to address an alternative timeline.

**Where seen:** temporal `ForkHistoryBranch` produces a new branch token whose `Ancestors` slice references upstream ranges (`common/persistence/history_manager.go:37-127`); langgraph `source: "fork"` in `bulk_update_state` at `main.py:1817-1822` and the time-travel path at `_loop.py:959`; letta `ConversationManager.fork_conversation` shares messages by junction (`letta/services/conversation_manager.py:103-172`); openhands `parent_conversation_id` / `sub_conversation_ids` conversation nesting (`app_conversation_models.py:131-132`).

**Why it works:** Branching becomes a metadata operation — the events themselves can be shared.

**When to copy:** Any system that wants reset / replay / experimentation.

**When it is overkill:** Single-path workflows that never rewind.

**Evidence:** `temporal/common/persistence/history_manager.go:37-127`; `langgraph/libs/langgraph/langgraph/pregel/main.py:1817-1822`; `temporal/service/history/ndc/workflow_resetter.go:103-261`.

### Pattern 12 — Channel-state vs. call-state capture

**Problem:** Capturing call stacks / live objects is brittle (closures, native handles, transient caches). Capturing only the channel state — the materialized slice after commit — is replay-deterministic.

**Where seen:** langgraph (channels only) at `libs/checkpoint/.../__init__.py:92-123`; agent-framework similarly captures *committed* shared state and per-executor state, not call stacks (`_runner.py:212-238`).

**Why it works:** Channel state is portable; node functions are reproducible by re-applying history.

**When to copy:** When users want to author new node implementations without touching the checkpoint format.

**When it is overkill:** A one-node, one-function agent where the channel IS the function's return.

**Evidence:** `langgraph/libs/checkpoint/langgraph/checkpoint/base/__init__.py:92-123`.

### Pattern 13 — Pending writes staged in a separate table

**Problem:** Partial-superstep failures must be replayable without re-running tasks that already committed.

**Where seen:** langgraph `pending_writes` table keyed by `(thread_id, ns, checkpoint_id, task_id, idx)`; `WRITES_IDX_MAP` reserves negative indices for `ERROR / SCHEDULED / INTERRUPT / RESUME` (`base/__init__.py:795`); agent-framework captures `pending_request_info_events` inside the same payload; temporal separates `buffered_events` from `history_node` (`schema/postgresql/v12/temporal/schema.sql:83-93`; `mutable_state_impl.go:7687`).

**Why it works:** Re-apply only the writes whose task succeeded; errors / interrupts route differently.

**When to copy:** Any async / parallel task graph.

**When it is overkill:** Strictly sequential synchronous workflows.

**Evidence:** `langgraph/libs/langgraph/langgraph/pregel/_loop.py:432-447`; `temporal/schema/postgresql/v12/temporal/schema.sql:83-93`.

### Pattern 14 — Lazy stub for first-run exit-mode writes

**Problem:** In `durability="exit"` mode, no parent exists yet on the first run; without an anchor, delta writes have nothing to chain to.

**Where seen:** langgraph `_loop.py:1230-1258` writes an empty stub at `step=-2` so accumulated delta writes can be anchored.

**Why it works:** The anchor is cheap (empty) and lets the same delta-write logic run on first-run and resume.

**When to copy:** Whenever "first run vs. resume" semantics can differ in dangerous ways.

**When it is overkill:** Single-mode systems.

**Evidence:** `langgraph/libs/langgraph/langgraph/pregel/_loop.py:1230-1258`.

### Pattern 15 — Sub-workflow checkpoint rehydration

**Problem:** A nested workflow's pending requests and execution contexts are part of the parent's state.

**Where seen:** agent-framework Python `WorkflowExecutor.on_checkpoint_restore` re-registers pending request_info events on the sub-workflow's runner context (`_workflow_executor.py:483-535`); agent-framework .NET `WorkflowHostExecutor.OnCheckpointingAsync` snapshots its `CheckpointManager` and pending response ports under reserved keys (`Specialized/WorkflowHostExecutor.cs:256-294`).

**Why it works:** Without this, resume loses cross-graph in-flight requests.

**When to copy:** Any framework with nested / sub-graph execution.

**When it is overkill:** Flat-only agents.

**Evidence:** `agent-framework/python/packages/core/agent_framework/_workflows/_workflow_executor.py:473-535`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/WorkflowHostExecutor.cs:256-294`.

### Pattern 16 — Distributed-safe checkpoint by phase-by-phase

**Problem:** Rebalancing a shard or recovering from a leader crash without losing in-flight work.

**Where seen:** temporal's queue-reader checkpoint (`queue_base.go:292-354`) plus shard lock (`sql/execution.go:40-58`); versioned-transition staleness checks (`transitionhistory/transition_history.go:81-125`); HSM state machines have `DBRecordVersion` for CAS.

**Why it works:** The runtime can rebuild state by replay if a worker dies mid-write.

**When to copy:** Multi-replica orchestrators.

**When it is overkill:** Single-instance.

**Evidence:** `temporal/service/history/queues/queue_base.go:83-419`; `temporal/common/persistence/cassandra/mutable_state_store.go:107-123`.

## Tradeoffs

### Event-sourced (temporal) vs. snapshot (agent-framework, langgraph)

**Benefit:** Pure event-source has trivially correct branching, deterministic replay from any fork point, and the journal doubles as audit log. temporal's `history_node` rows + `branchToken` make time travel and reset the same operation.

**Cost:** Cold rebuild is linear in event count. The `MutableStateImpl` monolith is the size it is because mutable state is a large projection. Cheaper storage per capture requires more compute per restore.

**Best fit:** Long-running workflows with frequent resets, multi-region replication, and a need for principled audit history.

**Failure mode:** Mutable state can grow past size limits; `enforceMutableStateSizeCheck` (`context.go:1120-1157`) force-terminates workflows that exceed `MutableStateSizeLimitError`.

**Alternative in another source:** langgraph's per-superstep channel-state capture is similar in spirit but bounded by the per-step channel projection rather than by the whole workflow.

### Pregel snapshot (agent-framework, langgraph) vs. step log (letta)

**Benefit:** Capturing at superstep boundaries gives free resume and free time travel (with a parent link). The capture is small.

**Cost:** The runner has to *commit* pending state before capture (langgraph), which means a partial-step failure may lose partial work. Per-step PostgreSQL writes (letta) avoid the "what's the committed slice?" question because the unit of durability *is* the step.

**Best fit:** Workflow-shaped agents with branching desired; multi-step LLM agents with single-message user turns.

**Failure mode:** A mismatched graph (Python signature hash; .NET `WorkflowInfo.IsMatch`) refuses to load — operator must debug from a stack trace that doesn't say *what* changed.

**Alternative in another source:** letta's per-step log avoids the snapshot entirely but loses the parent-chain time-travel machinery.

### Read-model mirror + live sandbox (openhands) vs. in-source durability

**Benefit:** Pause/unpause is instant; the storage backend only handles the read model; logs can be regenerated on demand.

**Cost:** If the sandbox dies, the conversation becomes archived read-only. There is no in-source restore-from-events. Ordering relies on string-sorted ISO timestamps.

**Best fit:** Sandboxed-agent runtimes where the sandbox *is* the durable unit, and where the app server is a control-plane / dispatcher only.

**Failure mode:** Sandbox lifecycle drift; late webhook events processed twice because of file-key collisions on rare UUID clashing; no automated recovery.

**Alternative in another source:** A snapshot-based approach (temporal / agent-framework / langgraph) where the storage layer carries the truth; the sandbox becomes a runtime rather than a durability boundary.

### Pluggable storage (everywhere) vs. fixed (letta, openhands)

**Benefit:** Pluggable backends (agent-framework, langgraph, temporal, openhands) let users choose their durability tier.

**Cost:** Each backend has different atomicity / consistency properties — the framework must be defensive. agent-framework swallows checkpoint failures (`_runner.py:236-238`); langgraph drains futures before the next save; openhands uses per-event idempotency. None of these guarantees transfer cleanly across backends.

**Best fit:** A framework shipping in dev + prod.

**Failure mode:** A naive `delete()` on file storage scales fine; the same call on a hot Postgres table is destructive; Cosmos `load` raises on id collisions.

## Decision Guide

### Choose a model

- **If your workflow is event-deterministic, fully reproducible from history, and you need multi-year durability / cross-region replication / principled reset → event-source (temporal-style).** You get free audit history, branches-as-fork-tokens, and durable replay. Pay for cold-restart cost.
- **If you have channel-style intermediate state (LLM messages, tool returns) but want fast capture and resume → Pregel-style superstep snapshot (langgraph-style).** You get cheap resume, optional branching via `bulk_update_state`, and durability-mode knobs.
- **If you have explicit per-step relational rows and a small, well-typed data model → step-grained log (letta-style).** You get operational queries (token usage, error type) for free but you give up parent-chain replay.
- **If execution lives in a sandbox the framework does not own → read-model + sandbox (openhands-style).** Be honest about it: persist only what you can rebuild.

### Choose where checkpointing belongs

- **Centralized in the core** (agent-framework, langgraph, temporal): every storage backend implements the same contract; conformance tests catch divergence.
- **Per-host shim** (agent-framework foundry-hosting, letta `ConversationManager`): when retention / pruning are tenant-specific, push the policy into a higher layer and keep core durable-but-raw.
- **Sandbox-owned** (openhands): only when the sandbox is the durability boundary. Document it loudly.

### Choose a deletion / pruning model

- **History trim + branch-reference-aware delete** (temporal): required if you ever fork.
- **Per-turn prune-to-latest** (agent-framework foundry-hosting): simple but destroys intra-turn chain.
- **Retention-as-background-job** (letta's gap): needed when the head-of-list pattern otherwise grows unbounded.
- **Explicit `prune` API with loud footgun warnings** (langgraph): correct contract, optional implementation.

## Practical Tips

### Patterns to copy

1. **Capture at a defined execution boundary, commit pending state first** (Pattern 1). Add a per-resource dirty check (Pattern 4 + temporal's `StartTransaction` reject at `mutable_state_impl.go:7451-7463`) so callers cannot start a transaction on uncommitted state.
2. **Single parent link per checkpoint, period.** Even if you never expose branching, the parent column is cheap and unlocks replay and history views.
3. **Atomic writes via rename or CAS.** `os.replace` for files, optimistic CAS for SQL.
4. **Validation on restore.** A struct hash on the function signatures (agent-framework Python) or a topology match (agent-framework .NET) costs little and prevents silent corruption.
5. **Per-executor hooks** if your executor graph has components whose internal state the framework cannot introspect (LLM agents, sub-workflows, plans).
6. **Durability-mode knobs** (`sync` / `async` / `exit` like langgraph). Makes dev cheap and prod durable without forking the architecture.
7. **Source-tagged provenance.** Future you debugging a fork vs. an automated loop tick will thank you.
8. **Branch tokens with an `Ancestors` slice** if you ever plan to reset / time-travel; it is a metadata-only operation.
9. **Loud footgun docs** for unsafe operations. langgraph's `prune` warning (`base/__init__.py:387-414`) is a model — three safe-pruning recipes spelled out.

### Patterns to avoid or delay

1. **Naive `keep_latest` pruning on a thread that uses delta / sparse-state capture.** langgraph's docstring says you will silently corrupt state. Run the ancestor walk or force a snapshot first.
2. **Pickle payloads without an explicit "not a security boundary" disclaimer.** agent-framework warns; honor it. If data is untrusted, type your payloads (Pydantic / dataclass / protobuf) and lose the round-trip-fidelity instead.
3. **JSON-string-sorted event ordering.** openhands does this (`event_service_base.py:121-131`) and it is fragile to clock skew.
4. **Using ISO string comparison for a sequence number.** Use a monotonic counter (letta's `sequence_id` bigint; langgraph's `channel_versions`; temporal's `next_event_id`).
5. **Cross-process file storage with `FileShare.None`.** agent-framework .NET intentionally blocks this (`FileSystemJsonCheckpointStore.cs:60-65`); if you want cooperative hosting, swap to `fcntl.flock`.
6. **JSON-column in-context pointers where you also have a relational alternative** (letta's `agent.message_ids`). The inline TODO at `letta/orm/agent.py:69-70` admits it.
7. **Snapshot-on-every-domain-call with no `durability` knob.** Either commit to it (temporal) or expose the knob (langgraph). Halfway modes are the worst.

### Decision rules

- **Replay must be deterministic for a given event sequence → event-source.**
- **Replay must tolerate non-deterministic tools → snapshot + capture tool state in hooks.**
- **Users will reach into the storage backend (files / Postgres / S3) → pluggable storage with conformance tests.**
- **Run a worker / orchestrator fleet → CAS + fencing (temporal-style).**
- **Long-running workflows with planned resets → branch tokens.**
- **Per-step audit (which model, which tokens, which error) → step-row log (letta-style).**

### Caution signs

- The runner swallows storage errors silently (agent-framework `_runner.py:236-238` is intentional but documented; unwarned equivalents are a warning sign).
- The schema's `version` field exists but no consumer reads it (agent-framework `version="1.0"` at `_checkpoint.py:88`; langgraph `LATEST_VERSION` declared in two places).
- There are two parallel version constants (langgraph: `checkpoint.base.LATEST_VERSION = 2` vs `pregel/_checkpoint.py:21`'s `= 4`).
- The "prune" / "compact" method exists in the base class but is unimplemented in every shipped backend (langgraph conformance gap).
- The in-context pointer is a JSON column with an inline TODO admitting it should be relational (letta `agent.message_ids`).
- Restore relies on ISO-8601 string comparison (openhands `event_service_base.py:121-131`).
- A read-only / archived state exists for "lost container" with no recovery path (openhands `SandboxStatus.MISSING`).
- Branching is implemented as `fork` only at the conversation / graph level (letta `fork_conversation`; openhands sub-conversations; agent-framework storage layer supports `withParent` filter but no caller writes branching links).

## Notable Absences

- **agent-framework has no branching / fork API.** The .NET storage layer supports parent-child queries (`ICheckpointManager.RetrieveIndexAsync(sessionId, withParent?)` at `ICheckpointManager.cs:41`), but no caller writes a branching checkpoint. The data structure is dead infrastructure for branching today (`agent-framework.md:107`).
- **agent-framework has no built-in retention in core.** The only auto-pruning is in foundry hosting (`_responses.py:702-712`); storage grows unbounded for everything else.
- **agent-framework .NET `ICheckpointManager` is internal.** Only `CheckpointManager` facade exposed with two factories (`CheckpointManager.cs:13-60`); custom backends must re-derive the abstraction.
- **langgraph declares `prune`, `copy_thread`, `delete_for_runs` on the base but no bundled saver implements them.** Conformance specs exist (`test_prune.py`, `test_copy_thread.py`, `test_delete_for_runs.py`) but Postgres/SQLite/InMemory all raise `NotImplementedError` (`langgraph.md:107, 134`).
- **letta has no `parent_step_id` or `parent_run_id` column.** All parent–child linkage is implicit in the FK chain `agent → run → step → messages`.
- **letta has no replay-from-step / fork-from-step API.** "Branch from step N" requires a manual export-and-rebuild.
- **letta has no scheduled retention for `runs` / `steps` / `messages`.** Only Redis SSE TTL (`stream_ttl=10800s`, `max_stream_length=10000`).
- **letta has a JSON-column `agent.message_ids`** flagged in-source as "dangerously flexible with the JSON type" (`letta/orm/agent.py:69-70`).
- **openhands has no restore-from-events path.** Resume is via sandbox pause/unpause only. `SandboxStatus.MISSING` is read-only archival with no recovery.
- **openhands has no event-log pruning or retention.**
- **temporal rebuild cost is linear in event count**; without history trimming this would not scale. The `MutableStateImpl` is ~10 200 lines (`mutable_state_impl.go:1-10247`).

## Per-Source Notes

### agent-framework — 8/10

Two full-fidelity parallel implementations (Python and .NET), keyed off a superstep boundary in their respective Pregel loops. Captures the committed slice (messages in flight, executor state, shared state, pending external requests, iteration counter, graph signature hash) via pluggable `CheckpointStorage` (Python) / `ICheckpointStore<TStoreObject>` (.NET). Each checkpoint records a single parent link. Python wire format is JSON + base64-pickle with a `RestrictedUnpickler` allowlist; .NET uses `System.Text.Json` over `PortableValue` with deferred deserialization. Both support full execution restore (re-emit pending events) but neither supports branching or time travel. Foundry hosting has the only auto-pruning (`_delete_not_latest_checkpoints`); durable-task activities disable checkpointing entirely (state lives in the durable task framework).

### langgraph — 8/10

The versioned, parent-linked channel-state capture taken after every Pregel superstep plus on entry/exit/branch events. `BaseCheckpointSaver` defines six must-implement methods plus four optional extensions. `CheckpointMetadata.source ∈ {input, loop, update, fork}` tags every saved tuple; `durability ∈ {"sync","async","exit"}` drives when `_put_checkpoint` actually fires. Channel values are split between an inline JSON column and a `(channel, version)`-keyed blob table (Postgres) or inline-only SQLite row; writes for the current superstep are stored separately as `pending_writes` so partial supersteps can be replayed without re-running succeeded tasks. A new beta `DeltaChannel` reduces blob size by storing a sentinel per intermediate step and reconstructing state via a parent-walk (`get_delta_channel_history`). Time travel, branching (`update_state` / `bulk_update_state`), subgraph checkpoint scoping via `checkpoint_ns`, and parent-config plumbing are first-class.

### letta — 5/10

Layered, append-only event log over relational tables (`runs`, `steps`, `messages`, `messages.conversation_messages`, plus optional Redis SSE streams). Each agent step performs four discrete, transactional writes: a `Step` row, a `StepMetrics` row, a `messages` insert, and an update of `agent.message_ids` (or `conversation_messages.in_context`). Granularity is one `Step` row per LLM call, capped by `max_steps` per `Run`. No native time-travel, branching, or `fork_from_step` API; the closest analog is `ConversationManager.fork_conversation` (a one-way soft branch) and Redis SSE stream reconnection (transport-level replay of an active run). Execution always resumes from the head of `agent.message_ids`. Explicit "rollback to previous state" on exception (`letta_agent_v3.py:1513`); cancellation writes synthetic approval-denial messages to keep the tail coherent. Production-grade Postgres/SQLite via SQLAlchemy with full migration history; no built-in retention for `runs` / `steps` / `messages`.

### openhands — 4/10

The V1 app server splits responsibility: execution runs inside a separate sandbox (`openhands-sdk==1.29.0`), and the app server owns only the durable record. The persisted model is an append-only per-event JSON log (one file per event id, under `{persistence_dir}/{user_id}/v1_conversations/{conversation_id}/`) plus a SQL `AppConversationInfo` metadata record (title, parent/child links, tags, rolled-up `MetricsSnapshot`) and rolled-up stats on each `ConversationStateUpdateEvent(key='stats')`. Resume = sandbox pause/unpause (`docker_sandbox_service.py:517-548`); durable workspace volume (`openhands-workspace-{sandbox_id}`) survives container restarts. Parent/child conversation nesting exists but is conversation-level, not point-in-time. No restore-from-events path; no pruning or retention for the event log.

### temporal — 9/10

Event-sourced execution with materialized mutable state. "Checkpoint" is the immutable append-only `history_node` table plus a single mutable-state row (`executions`) overwritten on every commit. Three checkpoint levels: per-transaction persistence (synchronous on every domain call, snapshot vs mutation by mode), per-shard queue-reader checkpoint (timer-driven `checkpoint()` at `queue_base.go:292-354`), and history-branch checkpoint (transactional on event append). Resume is event replay (`StateRebuilderImpl.Rebuild`, `ApplyEvents`), not snapshot deserialization. Branching: `ForkHistoryBranch` creates a new branch token whose `Ancestors` slice references upstream ranges; `VersionedTransition` enables state-based replication alongside event-based. Pruning: three coordinated mechanisms (`TrimHistoryBranch`, `rangeCompleteTasks`, branch-reference-aware `DeleteHistoryBranch`). Proven at scale via Uber Cadence lineage.

## Open Questions

- **Why are `prune`, `copy_thread`, `delete_for_runs` declared on langgraph's base but unimplemented across Postgres/SQLite/InMemory?** They appear in the conformance test catalog (`test_prune.py`, `test_copy_thread.py`, `test_delete_for_runs.py`) yet no shipped saver overrides them. No design doc explaining the deferral (`langgraph.md:166`).
- **What is the relationship between `checkpoint.base.LATEST_VERSION = 2` and `langgraph._checkpoint.LATEST_VERSION = 4`?** Two parallel version constants; the latter bumps `Checkpoint.v` on write. Undocumented split (`langgraph.md:167`).
- **Where does `v=1/2/3 → v=4` migration take place in langgraph?** The only on-read migration is for `pending_sends`. No documentation of the lineage boundary (`langgraph.md:168`).
- **Does any active write path still emit `pending_sends`?** The base Checkpoint TypedDict does not list it but `_load_checkpoint_tuple` reads it (`langgraph.md:170`).
- **Is the `JSON` column used in langgraph Postgres?** It is declared but `checkpoints.put` sets it to `None` (`langgraph.md:171`).
- **Why does agent-framework store pickle when it disclaims it as "not a security boundary"?** The `RestrictedUnpickler` narrows but does not eliminate the surface (`agent-framework.md:147`).
- **Why does agent-framework have no built-in retention?** Operators must wire it themselves; the foundry-hosting shim is the only example (`agent-framework.md:185`).
- **Why does letta have no `parent_step_id` / `parent_run_id`?** The schema has every other relation but no self-referential FKs; no decision recorded (`letta.md:235`).
- **What is the operational story for very long-running letta agents?** No retention policy, no soft-delete + archive, no scheduled cleanup (`letta.md:236`).
- **What is the relationship between `ConversationManager.fork_conversation` and `_step_checkpoint_finish`?** The fork creates a new Conversation + junction rows but never calls `_checkpoint_messages` (`letta.md:237`).
- **Is the `WebhookService` retry-safe in letta?** No retry / backoff / dedup observed; fire-and-forget (`letta.md:239`).
- **Where is the real point-in-time execution state in openhands?** Lives in `openhands-sdk==1.29.0` (out of scope). Search for `checkpoint` / `snapshot` returns only `MetricsSnapshot` and frontend thumbnails (`openhands.md:111`).
- **What happens to `BufferEvents` after a Temporal `CloseTransactionAsSnapshot`?** The path returns `softassert.UnexpectedInternalErr` (`mutable_state_impl.go:7539-7546`); does that path exist in every Cassandra / SQL back-end? (`temporal.md:213`)
- **How does CHASM interact with time-skipping?** A `CONSIDER`/TODO comment explicitly says CHASM support is incomplete (`mutable_state_impl.go:7674-7676`).
- **Backward compat with old `next_event_id` CAS:** `templateUpdateWorkflowExecutionQueryDeprecated` still in codebase; deprecation plan not visible (`temporal.md:209`).
- **What is the public API to query arbitrary historical state in temporal?** Admin `DescribeMutableState` returns only current. Arbitrary past states not exposed (`temporal.md:212`).

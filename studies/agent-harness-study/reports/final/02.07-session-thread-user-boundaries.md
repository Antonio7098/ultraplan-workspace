## Executive Summary

The six sources converge on a small but well-defined pattern: every mature system in the study names a **primary durable identity** (AgentSession, Organization, Namespace, Project, User/Org tuple) and treats finer-grained scopes (threads, runs, sessions) as sub-keys beneath it. The primary identity is almost always prefixed into the storage primary key so that two requests with the same inner ID cannot collide across tenants — agent-framework's .NET Cosmos provider, langfuse's `(id, projectId)` keys, openhands' `(user_id, org_id)` index on `conversation_metadata_saas`, and Temporal's `namespace_id` in every `executions` PK all rely on the same trick.

The **divergence** is in where the boundary is enforced and how the application pays for it.

- **Storage-enforced** systems (Temporal, agent-framework's Cosmos `.NET`, openhands' SAAS stores) make the schema refuse cross-tenant rows.
- **Application-enforced** systems (langfuse, letta, openhands OSS, agent-framework Python core) trust the framework's authorization filter or repository helper, and a missing filter in a new query silently leaks.

Five of the six land at 7/10 — the consistent deduction is "no DB-level RLS, no built-in `user_id`/`tenant_id` field, or both" — and one source (agent-framework) earns 8/10 because the .NET hosted path adds a hard runtime guard (HTTP 403 on resume-identity mismatch) that the other systems only approximate.

The dimension's hardest question — *"can two users with similar runs accidentally share state?"* — receives a confident "no" only in three places: Temporal namespaces (storage-enforced), openhands SAAS (`AuthError` on missing `user_id`), and agent-framework .NET hosted (403 on identity mismatch). Everywhere else, the answer depends on which provider is configured.

## Core Thesis

**Session and user boundaries are easy to declare and hard to enforce.** Every source in this study treats the *unit of identity* (session, conversation, thread, workflow) as a first-class typed object, but treats the *tenant axis* (user, organization, namespace) as configuration that the host wires in. Sources separate by *whether they make the tenant axis mandatory at construction time* (Temporal's namespace validator rejects requests without one; openhands SAAS raises `AuthError` on missing `user_id`) versus *whether they make it optional with a fail-closed default* (agent-framework's in-memory providers accept any `session_id`; letta's default-actor fallback collapses anonymous callers into the global default user).

The interesting patterns are not about declaring the boundary but about (a) **prefixing the tenant into the storage key** so collisions are structurally impossible, (b) **building an interceptor chain that runs before any handler** so authorization cannot be forgotten in a new endpoint, (c) **separating "cross-thread memory" into an explicit, opt-in primitive** that defaults to per-thread only, and (d) **using an admin/internal sentinel** that gates the one path that legitimately crosses user boundaries (webhook callbacks, server-to-server RPC). Forks are universally safe — every system implements them by minting a new identity, never by aliasing the source.

## Approach Models

The six sources cluster into five distinct approach models. The clustering is by **where the boundary is enforced and where the developer must remember to enforce it**, not by language or framework choice.

### 1. SDK with per-invocation isolation context (agent-framework)

The `AgentSession` (Python) and `AgentSession` (.NET) are the durable state-bearers. Each call constructs a one-shot `SessionContext` that holds only `session_id` and `service_session_id` — every provider writes into `session.state[provider.source_id]`, so two providers of the same type on the same session never collide. Cross-thread memory is explicit via a `scope` argument on every memory provider; tenant isolation is delegated to whichever provider is wired in (Cosmos `.NET` uses hierarchical partition, Mem0/Redis use validated filter tags, Foundry Memory requires `scope` up front).

### 2. Library with minimal primitives and an Auth handler (langgraph)

The framework defines just three things: `thread_id` + `checkpoint_ns` for short-term state, a tuple `namespace` for long-term store items, and an `Auth` handler whose `FilterType` return value the server applies as a SQL/JSONB predicate. There is no `user_id` column anywhere; cross-user isolation is entirely the host's responsibility. The trade-off is that the *interface* is tiny and the *conformance suite* locks it down, but the *production enforcement* depends on the developer writing an `@auth.on.threads.read` handler.

### 3. Two-level application tenancy with composite keys (langfuse)

Every entity carries `projectId` (and most also carry `orgId`). Composite primary keys `@@id([id, projectId])` make per-project ID namespaces independent. Authorization is application-enforced via tRPC middleware and a centralized ClickHouse `project_id` filter factory; storage-level enforcement is structural (composite PKs) but not access-controlling. This is the standard "SaaS platform" pattern: deep, consistent, but convention-only.

### 4. ORM with a centralized access predicate (letta)

`OrganizationMixin` is on every entity. `apply_access_predicate(query, actor, access_type)` is the single chokepoint that translates an actor into a `WHERE` clause. Managers take `actor: PydanticUser` as an explicit argument so the call graph is reviewable. The risk is the inverse: when `actor=None`, the system emits only a `logger.warning("SECURITY: ... bypasses organization filtering.")` and returns the row.

### 5. Dual-mode SaaS/OSS with hard SAAS path (openhands)

The base app_server ships permissive OSS defaults (`DefaultUserAuth` always returns `None`); the `enterprise/` extension overrides them with a SAAS path that raises `AuthError` on missing user_id, applies a `(user_id, org_id)` composite index, and gates every store behind an effective-org-id precedence chain. The OSS path is documented as single-host single-user; the seam between them is silent and the comment on the base `_secure_select` claims a filter that does not exist.

### 6. Namespace-as-tenant with per-API role metadata (temporal)

The namespace is the only tenant axis. It is enforced three times: in the SQL schema (every PK starts with `namespace_id`), in the gRPC interceptor chain (`NamespaceValidateIntercept` runs before any handler), and in the authorizer (`ScopeNamespace` vs `ScopeCluster` per API). Per-user identity is layered on top via JWT `Namespaces map[string]Role`; per-workflow audit (`created_by_subject`) does not exist.

## Pattern Catalog

Patterns that appear in multiple sources, or that one source demonstrates unusually clearly. Each pattern explains the problem it solves, which sources demonstrate it, why it works, when to copy, when it is overkill, and the evidence that supports it.

### 1. Prefix the tenant into the storage primary key

**Problem.** Two requests with the same inner ID (e.g., `"default"`, a UUID hash, a user-supplied slug) from different tenants must never collide.

**Sources.** agent-framework (.NET Cosmos hierarchical `TenantId+UserId+ConversationId`), langfuse (`TraceSession @@id([id, projectId])`, `InAppAgentConversation @@id([id, projectId])`), openhands (`(user_id, org_id)` composite index on `conversation_metadata_saas`), Temporal (every PK prefixed by `(shard_id, namespace_id, …)`).

**Why it works.** The DBMS itself refuses to insert a row whose key collides with an existing row in another tenant. Cross-tenant collision becomes structurally impossible; the developer cannot forget a filter because there is no "forgetting" — the key just doesn't match.

**When to copy.** Always for systems that expect more than one tenant to share a database, and where the tenant axis is in fact a partition (not just a row attribute).

**When overkill.** Single-tenant OSS deployments where the cost of maintaining composite keys in every query outweighs the benefit. In agent-framework Python core, in-memory history is intentionally keyed by `session_id` alone; this is acceptable because the core is meant to be hosted by an integrator that will add tenant scoping externally.

**Evidence.** `agent-framework/dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs:197-215`; `langfuse/packages/shared/prisma/schema.prisma:234,392`; `openhands/enterprise/migrations/versions/118_add_conversation_metadata_saas_indexes.py:15-56`; `temporal/schema/postgresql/v12/temporal/schema.sql:30-93`.

### 2. Authorization interceptor chain at the API boundary

**Problem.** Authorization checks added ad-hoc in handlers can be forgotten; a new endpoint bypasses every existing check.

**Sources.** Temporal (most explicit: mask-errors → service-error → namespace-validate → log → metrics → auth → handover → redirection → telemetry → health → state-validate → rate-limit → sdk-version → caller-info, all before the handler), agent-framework `.NET` (HTTP pipeline with `ClaimsIdentitySessionIsolationKeyProvider`), langfuse (tRPC `enforceUserIsAuthedAndProjectMember` and `enforceIsAuthedAndOrgMember`), letta (FastAPI `HeaderParams` dependency resolved before every router).

**Why it works.** Centralizing auth means a new endpoint that forgets the check is impossible to ship. The pattern is "validate at the door, trust inside."

**When to copy.** Always. Even when the application is single-tenant, run the same chain with a noop authorizer so adding auth later is a config flip rather than a code change.

**When overkill.** Pure in-process library code where there is no API boundary (langgraph does not need this because the SDK is consumed by the developer's host).

**Evidence.** `temporal/service/frontend/fx.go:274-303`; `temporal/common/rpc/interceptor/namespace_validator.go:233-322`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Hosting.AspNetCore/ServiceCollectionExtensions.cs:42-70`; `langfuse/web/src/server/api/trpc.ts:284-373`; `letta/server/rest_api/dependencies.py:26-83`.

### 3. Cross-thread memory as an explicit, opt-in primitive that defaults to per-thread

**Problem.** Memory that crosses threads "accidentally" leaks between users; memory that is per-thread by default forces developers to opt in when they actually need sharing.

**Sources.** agent-framework (Python `FileMemoryProvider(scope=...)` defaults to `session_id`; Foundry memory provider raises if `scope is None`; Mem0/Redis use filter tags that must be explicitly supplied), langgraph (`Store` is documented as "cross-thread memory" and `Runtime.store` is shared by design while checkpoints are strictly per-thread), letta (`isolated_block_labels` for per-conversation overrides; `fork_conversation` shares messages explicitly; archival memory is per-agent).

**Why it works.** Naming the boundary (e.g., `scope`, `namespace`, `isolated_block_labels`) forces the developer to decide. There is no "default that happens to leak." Tests like `test_scope_isolates_memories_across_sessions` and `test_explicit_scope_shares_memories_across_sessions` lock the contract.

**When to copy.** When your system has any concept of "long-term memory" or "shared knowledge" beyond the immediate conversation. Default to per-thread, expose the cross-thread primitive, and require an explicit argument.

**When overkill.** Stateless tool APIs, single-turn agents, or read-only RAG pipelines that do not store per-user state.

**Evidence.** `agent-framework/python/packages/core/agent_framework/_harness/_file_memory.py:176-215`; `agent-framework/python/packages/foundry/agent_framework_foundry/_memory_provider.py:128-138`; `langgraph/libs/checkpoint/langgraph/store/base/__init__.py:700-717`; `letta/services/conversation_manager.py:894-957`.

### 4. Actor as a typed object threaded through managers, never as a thread-local or global

**Problem.** Auth context stored on `threading.local()` or read from a global makes the call graph unreviewable: a manager can leak state across requests and no one can tell from the signature.

**Sources.** letta (every manager takes `actor: PydanticUser` as an explicit argument; `User.user_id` and `User.organization_id` are read off the actor), openhands (`UserContext` is an abstract base injected per-request via `request.state`; three concrete subclasses: `AuthUserContext`, `SpecifyUserContext`, `ADMIN`).

**Why it works.** Static review of the call graph can find missing actors — every manager signature declares it. The pattern also makes testing easy: pass a fake actor.

**When to copy.** When the actor drives authorization decisions (which is most multi-user systems). The cost is verbosity in every manager signature, but the benefit is verifiability.

**When overkill.** Single-user scripts, batch jobs, or stateless functions.

**Evidence.** `letta/server/rest_api/routers/v1/agents.py:184`; `letta/services/conversation_manager.py:67-101`; `openhands/app_server/user/user_context.py:13-49`; `openhands/app_server/user/auth_user_context.py:152-166`.

### 5. Secure-default-deny on missing tenant/user identity

**Problem.** A request that arrives without an authenticated user should fail closed, not be silently rewritten to a "default user" that shares data with every other anonymous caller.

**Sources.** openhands (`_apply_user_and_org_filter` raises `AuthError('User authentication required')` when `get_user_id()` returns `None`; `validate_session_key_ownership` 403s on mismatch), agent-framework `.NET` (`AgentFrameworkResponseHandler.CreateAsync` compares persisted `HostedSessionContext(UserId, ChatId)` against the live request and throws 403 `hosted_session_identity_mismatch`).

**Why it works.** The cost of one false 403 is much lower than the cost of one cross-tenant leak. A loud failure during development catches misconfigurations early.

**When to copy.** Always for hosted, multi-tenant systems. Document the OSS-vs-SAAS boundary explicitly (as openhands does) rather than silently.

**When overkill.** Single-user tools, local-only development servers, or any system where the threat model does not include "two callers may share a process."

**Evidence.** `openhands/enterprise/server/utils/saas_app_conversation_info_injector.py:76-78`; `openhands/enterprise/server/utils/saas_pending_message_injector.py:84-86`; `agent-framework/dotnet/src/Microsoft.Agents.AI.Foundry.Hosting/AgentFrameworkResponseHandler.cs:79-110`.

### 6. Admin/internal sentinel for the one path that legitimately crosses boundaries

**Problem.** Background tasks (webhook callbacks, server-to-server RPC, scheduled jobs) need to operate across users. Without an explicit escape, the only way to do this is to weaken the per-user filter, which leaks.

**Sources.** openhands (`SpecifyUserContext.ADMIN = SpecifyUserContext(user_id=None)`; the SAAS conversation service skips `_apply_user_and_org_filter` when `user_context == ADMIN`; ordered by `as_admin` which raises if a non-admin context is already on the request), Temporal (`internalClaimMapper` hard-codes `Principal{Type:"temporal", Name:"internal"}` with `System: RoleAdmin`; inbound `temporal-principal-*` headers are stripped unconditionally before the authorizer runs), langfuse (admin override path via `ADMIN_API_KEY` that emits `sendAdminAccessWebhook` audit signal).

**Why it works.** A sentinel value (instead of an unset user_id) makes the bypass grep-able. Listing every site that installs the sentinel is a security review exercise. Stripping inbound headers prevents the caller from impersonating the sentinel.

**When to copy.** Whenever background work needs to cross user boundaries. The single best practice is `headers.StripPrincipal(ctx)` (Temporal) so callers cannot inject the sentinel themselves.

**When overkill.** Systems with no background work and no server-to-server RPC.

**Evidence.** `openhands/app_server/user/specifiy_user_context.py:52-65`; `openhands/enterprise/server/utils/saas_app_conversation_info_injector.py:70-73`; `temporal/common/authorization/claim_mapper.go:61-78`; `temporal/common/authorization/interceptor.go:158`; `langfuse/web/src/features/public-api/server/createAuthedProjectAPIRoute.ts:164-251`.

### 7. Prefix-keyed IDs to prevent type confusion

**Problem.** A bug that passes a `user_id` where a `run_id` was expected is invisible at the type level if both are strings.

**Sources.** letta (`PrimitiveType` enum drives every entity ID prefix; `user-…`, `org-…`, `agent-…`, `conv-…`, `run-…`, `block-…`, `passage-…`, `job-…`; `PRIMITIVE_ID_PATTERNS` enforces `<prefix>-<uuid4>`).

**Why it works.** A regex mismatch raises immediately. Audit fields like `_created_by_id` can assert that the value starts with `user-`, blocking developer errors that populate them with non-user IDs.

**When to copy.** When you have multiple ID types in the same database and they are passed through shared plumbing. Especially useful for `created_by` / `last_updated_by` audit columns.

**When overkill.** When the ID type is always wrapped in a typed object (e.g., `User` is a class, not a string).

**Evidence.** `letta/schemas/enums.py:4-42`; `letta/validators.py:12-19`; `letta/orm/base.py:30-85`; `letta/schemas/letta_base.py:79-89`.

### 8. Conformance / isolation test suite that locks the contract across backends

**Problem.** When a system has multiple storage backends (in-memory, SQLite, Postgres, Cosmos), an isolation contract that's only tested in one backend can regress silently in another.

**Sources.** langgraph (most explicit: `libs/checkpoint-conformance/langgraph/checkpoint/conformance/spec/test_delete_thread.py`, `test_copy_thread.py`, `test_get_tuple.py`; `capabilities.py` declares `BASE_CAPABILITIES` vs `EXTENDED` per backend), langfuse (`in-app-agent-persistence.servertest.ts:1248-1332` tests cross-user isolation and cross-project id reuse), openhands (`test_user_isolation`, `test_batch_get_with_user_isolation`, `test_search_and_count_with_user_isolation` in `test_sql_app_conversation_start_task_service.py`), Temporal (`TestCrossNamespaceCommands_Authorized`, `_Unauthorized`, `_DisabledFeature` in `interceptor_test.go:341-476`), agent-framework (`tests/core/test_harness_file_memory.py:226-254` tests `test_scope_isolates_memories_across_sessions` and `test_explicit_scope_shares_memories_across_sessions`).

**Why it works.** Tests are the executable specification. If a future contributor adds a new SQLite backend, they must run the conformance suite; the test failure is the bug.

**When to copy.** Whenever you ship more than one storage backend for the same logical interface.

**When overkill.** Single-backend systems where the storage layer is not user-pluggable.

**Evidence.** see source citations above.

### 9. Fork = new ID + new entity, never alias the source

**Problem.** "Forking" a session by pointing the new ID at the source's rows sounds efficient but corrupts audit trails and confuses deletion.

**Sources.** agent-framework (`AgentSession.to_dict` / `from_dict` round-trip; framework actively avoids replaying the source's `service_session_id` so a forked session won't re-enter the original hosted conversation), letta (`fork_conversation` creates a new `Conversation` row but shares `Message` rows via a `ConversationMessage` junction; uniqueness on `(conversation_id, message_id)` allows many-to-many without conflict), openhands (`parent_conversation_id` linkage; `_inherit_configuration_from_parent` propagates `sandbox_id`/repo/LLM but not events), Temporal (`ContinueAsNew`, retry, reset, child workflow all generate a fresh `RunID` under the same `(NamespaceID, WorkflowID)`).

**Why it works.** A new ID + new entity is honest: deletions and audits don't cross-contaminate. Sharing messages or configuration via an explicit junction table (letta) or inheritance hook (openhands) is the right way to express "this new thing reuses part of the old one."

**When to copy.** Whenever you offer a "fork" or "branch" feature. Avoid the temptation to alias.

**When overkill.** Stateless copies where there is nothing to share.

**Evidence.** `agent-framework/python/packages/core/agent_framework/_sessions.py:69-147`; `letta/services/conversation_manager.py:103-172`; `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1014-1042`; `temporal/service/history/interfaces/engine.go:64,132`.

### 10. Defensive validation as identity-agnostic correctness

**Problem.** Names with dots, empty strings, reserved prefixes (`langgraph`, `_internal`, etc.) silently corrupt storage or trigger injection.

**Sources.** langgraph (namespace validator: rejects empty tuple, non-string labels, periods, empty strings, and the `langgraph` root), agent-framework (file-system helpers encode risky session IDs and reject literal path separators and Windows reserved names; `FileMemoryProvider` rejects filenames matching internal markers like `_description.md` sidecars and `memories.jsonl`), langfuse (NextAuth session tokens are session strings, but observability `TraceSession.id` has its own validated format).

**Why it works.** Validation does not claim to enforce identity, but it prevents the most common ways an identity bug (e.g., `thread_id = "../etc/passwd"`) escalates to a security incident. It is cheap and orthogonal.

**When to copy.** Always, but especially for any string that becomes part of a file path, SQL identifier, or partition key.

**When overkill.** Pure string outputs that never touch storage.

**Evidence.** `langgraph/libs/checkpoint/langgraph/store/base/__init__.py:1255-1275`; `agent-framework/python/packages/core/agent_framework/_sessions.py:1087-1134`; `agent-framework/python/packages/core/agent_framework/_harness/_file_memory.py:84-90`.

## Key Differences

The differences are not quality differences; they are responses to different constraints. Each is framed as "X uses Y because..., while Z uses W because...".

### Library vs. application

**langgraph** uses minimal primitives (`thread_id`, namespace tuple) and an `Auth` handler because it is a *library* consumed by many hosts; the framework refuses to make policy choices for the host. **Temporal** uses namespace + per-API metadata because it is a *service* with a single auth boundary and can afford to own the policy. The trade-off: langgraph's conformance suite is more important than its enforcement, while Temporal's interceptor chain is more important than its schema.

### Hosted vs. self-hosted

**agent-framework** splits its boundary model across the SDK (per-invocation context, provider-scoped state) and the .NET hosting layer (claims-based isolation key, 403 on identity mismatch). The Python core is provider-agnostic; the .NET hosting layer is opinionated. **openhands** does the same split differently: the app_server ships permissive OSS defaults and the SAAS extension overrides them with a hard path. The common pattern is "library is permissive, hosted path is strict" — but the seam is visible in agent-framework (a class hierarchy) and silent in openhands (the base class docstring lies about a filter that is only present in the subclass).

### Single dominant tenant axis vs. composite

**letta** commits to one tenant axis (`organization_id`) and uses a single chokepoint (`apply_access_predicate`). **Temporal** commits to one tenant axis (`namespace`) and uses three enforcement layers (schema, interceptor, authorizer). **langfuse** uses two axes (org + project) and enforces them in two layers (composite keys + middleware). The choice is driven by the *minimum granularity the host must isolate to*: a single tenant per deployment gets one axis; an organization-with-projects model gets two.

### Session-centric vs. workflow-centric

**agent-framework**, **langgraph**, and **letta** center on a `Session`/`thread_id`/`Conversation` as the primary state-bearer — the application developer picks the ID. **Temporal** and **openhands** center on a *workload* (workflow, conversation) where the ID is generated server-side (or via stable external identity like Keycloak `sub`). The trade-off: workload-centric systems have fewer "caller picked the wrong ID" bugs because the workload has its own validation; session-centric systems are easier to fork because the ID space is opaque to the platform.

### With cross-thread memory vs. without

**agent-framework**, **langgraph**, and **letta** all have explicit cross-thread memory primitives (`scope`, `Runtime.store`, `isolated_block_labels` + `fork_conversation`). **openhands** has none — the answer to "is cross-thread memory explicit?" is "there is no cross-conversation memory service." **Temporal** has none — cross-workflow coordination must use external stores or Nexus endpoints. The difference is whether the system positions itself as an agent runtime (where long-term memory is a core feature) or as a workflow orchestrator / multi-tenant SaaS shell (where long-term memory is the application's job).

### Storage-level vs. application-level enforcement

**Temporal** and **agent-framework `.NET` Cosmos** make the schema enforce: composite PKs that cannot be inserted cross-tenant. **langfuse**, **letta**, **openhands OSS**, and **agent-framework Python core** rely on application-level filters: composite keys exist (langfuse) or the column exists (letta) but the *database itself* does not refuse a cross-tenant row — only the application does. **openhands SAAS** is the hybrid: SAAS path has both composite keys (storage-level) and validated filter helpers (application-level).

## Tradeoffs

### Application-enforced isolation is portable but fragile

**Benefit.** Same query works across Postgres, ClickHouse, SQLite, Cosmos, in-memory; no schema-per-tenant operational complexity.

**Cost.** A new query that forgets the `projectId`/`createdByUserId`/`actor` filter silently leaks; the database will not catch it.

**Best fit.** Multi-backend systems where the storage layer is pluggable (langfuse, langgraph), or systems where the tenant axis is configuration rather than schema (letta's `organization_id`).

**Failure mode.** New repository function added without `apply_access_predicate` (letta) or without the `projectId` filter (langfuse).

**Alternative.** langfuse explicitly mentions considering Postgres RLS in its "Future Considerations"; openhands SAAS does both application and storage. **Stronger approach:** add a lint/test guard that fails any Prisma/ClickHouse query lacking a `project_id` predicate (langfuse), or move isolation to PostgreSQL RLS keyed on a session-level `letta.organization_id` (letta).

### Storage-enforced isolation is durable but rigid

**Benefit.** The DBMS itself refuses cross-tenant rows; the developer cannot forget a filter.

**Cost.** Every query path must thread the tenant key; composite PKs make ad-hoc admin queries awkward; renaming or merging tenants is operationally expensive.

**Best fit.** Systems where the tenant axis is a true partition and never changes (Temporal namespaces; multi-tenant SaaS with a stable per-customer database).

**Failure mode.** Tenant axis becomes a row attribute instead of a partition key (e.g., openhands OSS).

**Alternative.** Hybrid: storage-level composite key + application-level filter helpers + a sentinel escape for background work. This is openhands SAAS and agent-framework Cosmos `.NET`.

### Per-invocation context that doesn't leak

**Benefit.** Provider-contributed state is scoped to a single `run()` call; persisted state goes via `provider.after_run → session.state`. The framework prevents invocation-scope state from leaking into session scope.

**Cost.** Every model invocation pays the cost of constructing and tearing down a `SessionContext`.

**Best fit.** SDKs that wrap multiple LLM calls per user request (agent-framework); less relevant for one-shot APIs.

**Failure mode.** Developers stash long-lived state in the context instead of in `session.state`.

**Alternative.** A single mutable session dict without per-invocation boundary (letta's `actor` threaded through managers is similar but method-scoped, not invocation-scoped).

### `thread_id` as a string vs. typed object

**Benefit.** A `str` is cheap, easy to log, and trivially serializable.

**Cost.** Two callers passing `"default"` collide silently; no compiler help.

**Best fit.** Langgraph where the thread ID is a developer-supplied conversation handle and the framework provides a conformance suite to lock behavior.

**Failure mode.** `thread_id = "default"` in two users' code paths; letta's prefix-keyed IDs (`conv-<uuid4>`) avoid this by enforcing a regex at the type level.

**Alternative.** letta's prefix-keyed primitive IDs enforce type-safety at the storage layer.

### Default-actor fallback for OSS convenience

**Benefit.** A developer can spin up the server with no auth header and it just works.

**Cost.** Two anonymous callers share the global default user and the global default org; a leaked `user_id` header that points at a non-existent user silently rewrites to default.

**Best fit.** OSS, single-tenant, single-user local development.

**Failure mode.** A self-hosted deployment exposes the OSS-mode server without reverse-proxy auth.

**Alternative.** `settings.no_default_actor=True` (letta), which raises `NoResultFound` if the header is missing — opt-in but the safe default for any deployment that may be exposed.

### Cross-thread memory primitive

**Benefit.** Opting into cross-thread memory is explicit; tests like `test_scope_shares_memories_across_sessions` lock the contract.

**Cost.** Developers must pick a scope/namespace strategy; mis-scoping leaks across users.

**Best fit.** Long-running agent systems where memory matters across turns (letta, agent-framework, langgraph).

**Failure mode.** Picking `scope="shared"` accidentally because it was the example in the docs.

**Alternative.** No cross-thread memory primitive at all (openhands, Temporal) — the application's responsibility to wire external storage if needed.

## Decision Guide

### Choosing an approach

| If your system is… | Use | Why |
|--------------------|-----|-----|
| A library consumed by many hosts | langgraph's Auth handler + namespace tuples | The library cannot enforce policy; the conformance suite locks the contract |
| A hosted SaaS with stable tenants | openhands SAAS pattern + Temporal composite-PK pattern | Storage-level enforcement + secure-default-deny + admin sentinel |
| A self-hosted OSS product | letta's `no_default_actor=False` with a documented seam | Convenience for single-user; flip the flag for production |
| An agent runtime with memory | agent-framework `scope` parameter + letta `isolated_block_labels` | Cross-thread memory is explicit and tests cover it |
| A workflow orchestrator | Temporal's namespace + per-API metadata | Namespace is the universal tenant axis; per-API metadata scales |
| An application with two-level tenancy (org + project) | langfuse composite `(id, projectId)` keys + middleware | Composite PKs + tRPC middleware + centralized filter factory |

### When to copy what

| Pattern | Copy when… | Delay until… |
|---------|------------|--------------|
| Prefix-keyed IDs | You have multiple ID types sharing plumbing | Single typed wrapper class suffices |
| Interceptor chain at the API boundary | You have an API surface | Pure in-process library code |
| Per-invocation context | You wrap multiple LLM calls per request | One-shot tool APIs |
| Actor as a typed object | Authorization drives manager behavior | Stateless functions |
| Secure-default-deny | Hosted multi-tenant | Single-user local-only |
| Admin sentinel | Background work crosses boundaries | No background work |
| Conformance suite | Multiple storage backends | Single backend |
| Cross-thread memory primitive | Long-running agents | Stateless tool APIs |
| Storage-level composite PKs | Multi-tenant database | Single-tenant or per-tenant database |
| `to_dict`/`from_dict` fork | Sessions need to be checkpointable | Pure ephemeral state |

### Caution signs

These signal that the studied design area is becoming brittle, over-coupled, under-specified, hard to test, hard to operate, or hard to evolve.

1. **`user_id`/`tenant_id` is not a first-class field on the primary state object.** This is what langgraph, agent-framework Python core, and openhands OSS share. It means cross-user isolation is the integrator's problem; the framework cannot refuse a mis-routed call. Fix: add `isolation`/`isolation_key` to the state object's constructor (as agent-framework Foundry does).

2. **The base SQL `WHERE` clause "filters for X and user if available" — and the user filter is never actually applied.** openhands' `SQLAppConversationInfoService._secure_select` carries a docstring that lies; the SAAS subclass overrides with the real filter. Fix: either ship the filter or update the comment; a `pre-commit` check that fails any TODO marker in a security-sensitive file helps.

3. **A noop authorizer or noop claim mapper is the default.** Temporal's `noopAuthorizer.Authorize` returns `DecisionAllow` and `noopClaimMapper.GetClaims` returns `System: RoleAdmin`. A misconfigured server is wide-open. Fix: a startup assertion that the authorizer is "real" (not a noop).

4. **Composite primary keys exist but no row-level security policy.** langfuse's `TraceSession @@id([id, projectId])` is structural, not access-controlling. Fix: Postgres RLS keyed on a session-level `letta.organization_id`-style variable.

5. **A "default user" fallback in OSS mode.** letta's `get_actor_or_default_async` collapses all anonymous callers to `user-00000000-0000-4000-8000-000000000000`. Two anonymous users share identity. Fix: make `no_default_actor=True` the default for any deployment that may be exposed.

6. **Three different "session" concepts in the same codebase.** langfuse has `Session` (NextAuth auth), `TraceSession` (observability), `InAppAgentConversation` (agent thread). Developers wiring state to the wrong scope is a real risk. Fix: rename one (e.g., `AuthSession`, `TraceSession`, `AgentConversation`) or document the disambiguation centrally.

7. **`copy_thread` is declared but unimplemented in production savers.** langgraph declares it on `BaseCheckpointSaver`, the conformance suite tests it, but neither Postgres nor SQLite overrides it. Production users must rely on the LangGraph Server endpoint. Fix: implement in the OSS savers or remove from the base interface.

8. **The session ID is just a `str` with no `validate_thread_id` helper.** Langgraph allows `"default"`, `""`, `None` as thread IDs. Two callers passing the same string collide. Fix: a regex validator (as letta does with `PRIMITIVE_ID_PATTERNS`).

9. **An `ADMIN` sentinel with no audit log of where it's installed.** openhands installs `SpecifyUserContext.ADMIN` in `webhook_router.py:251-252`, every SAAS-side V1 callback processor (`github_v1_callback_processor.py:252`, `jira_v1_callback_processor.py:113`, etc.), and `session_auth.py:60-66`. A grep enumerates them but a single ADR or runtime audit log does not. Fix: central document + runtime emit of an audit event when the sentinel is installed.

10. **A `assert` is used for security invariants.** openhands' `save_app_conversation_info` uses `assert existing is None or existing.user_id == user_id_uuid`. Assertions are stripped under `python -O`. Fix: explicit `HTTPException` or `AuthError`.

11. **The base class docstring describes a check that the subclass implements.** openhands' `SQLAppConversationInfoService` class comment: "Security and permission checks are handled by wrapper services." This is honest — but only if the wrapper is always wired. Fix: a startup check that the SAAS subclass is loaded when `AppMode.SAAS`.

12. **Storage path encodes user identity in plain text.** openhands puts events at `{persistence_dir}/{user_id}/v1_conversations/{conversation_id.hex}/` and falls back to the conversation's `created_by_user_id` if `user_id` is missing — meaning the fallback writes another user's events into a directory keyed by a different user. Fix: refuse the fallback when `user_id` is `None` in SAAS.

## Practical Tips

### Patterns to copy

1. **Prefix the tenant axis into every storage primary key** when designing multi-tenant schemas. Composite `(shard_id, namespace_id, workflow_id, run_id)` (Temporal) and `(TenantId, UserId, ConversationId)` (agent-framework Cosmos) are the gold standard.
2. **Build an authorization interceptor chain at the API boundary** that runs *before* any handler. Even if your application is single-tenant today, run the chain with a noop authorizer so adding auth later is a config flip.
3. **Define `actor` as a typed object threaded through managers** (letta, openhands). The signature review is your safety net.
4. **Make cross-thread memory an explicit, opt-in primitive** with a name like `scope` or `isolated_block_labels`. Default to per-thread; require the argument to share.
5. **Use a sentinel (`SpecifyUserContext.ADMIN` or `Principal{Type:"temporal", Name:"internal"}`) for background work** that legitimately crosses user boundaries. Strip inbound headers that could impersonate the sentinel.
6. **Add an isolation conformance suite** if you have multiple storage backends. langgraph's `libs/checkpoint-conformance/` is the model.
7. **Prefix-keyed IDs** (`user-…`, `conv-…`, `run-…`) prevent the `user_id`-vs-`run_id` typo bug.
8. **Secure-default-deny on missing tenant identity.** Raise `AuthError` when `get_user_id()` returns `None` (openhands SAAS, agent-framework .NET hosted).
9. **Fork = new ID + new entity, never alias.** agent-framework's `to_dict`/`from_dict`, letta's `fork_conversation`, openhands' `parent_conversation_id`, Temporal's `ContinueAsNew` all mint a fresh identity.
10. **Type-tagged session serialization** (agent-framework's `_STATE_TYPE_REGISTRY`) lets custom state round-trip across forks without losing type information.

### Patterns to avoid or delay until needed

1. **Default-actor fallback for hosted deployments.** letta's `no_default_actor=False` is fine for OSS but lethal in production. Flip the default.
2. **Noop authorizer as the production default.** Temporal's noop authorizer is convenient for self-hosted single-user but should be replaced before any multi-user exposure.
3. **Sharing a `Runtime.store` instance across processes** without a real Postgres/SQLite backend. langgraph's in-memory store is process-local and a second process (or a reload) starts empty.
4. **`enable_studio_auth = true` in production.** langgraph's `disable_studio_auth` flag is easy to leave on by accident.
5. **Per-user audit columns without enforcement.** letta records `_created_by_id` but does not enforce it as an authorization boundary. If you want enforcement, treat it like any other tenant axis.
6. **`assert` for security invariants.** Stripped under `python -O`.
7. **`EnforceCrossNamespaceCommands=true`** without a thorough review of `validateCrossNamespaceCall` (Temporal). The default is off for a reason.
8. **OSS mode for multi-user exposure.** openhands' OSS `DefaultUserAuth` always returns `None`. Deploy behind a reverse proxy that injects user_id, or switch to SAAS mode.

### Decision rules

- **One tenant axis?** Use a single column (e.g., `organization_id` or `namespace_id`) and make it composite-PK-anchored. Source: letta, Temporal.
- **Two tenant axes (org + project)?** Use composite `(id, projectId)` keys plus an `orgId` column. Source: langfuse.
- **Library, not service?** Defer to an `Auth` handler returning server-applied `FilterType`. Source: langgraph.
- **SDK with hosted and self-hosted modes?** Put the hard guard in the hosting layer, leave the core permissive. Source: agent-framework.
- **Workflows and runs, not conversations?** Use a tri-tuple `(NamespaceID, WorkflowID, RunID)` and don't invent a "thread" concept. Source: Temporal.
- **Agent runtime with memory?** Add `scope` or `isolated_block_labels` as an explicit primitive. Source: agent-framework, letta.
- **Multi-user OSS?** Document the seam (as openhands does) and require an opt-in for hardened mode. Source: openhands.

## Anti-Patterns / Caution Signs

1. **No first-class `user_id`/`tenant_id` on the primary state object.** agent-framework Python core, langgraph, openhands OSS all share this. Cross-user isolation is the integrator's problem; the framework cannot refuse a mis-routed call.
2. **A docstring that lies about a filter that doesn't exist.** openhands' `SQLAppConversationInfoService._secure_select` says "filters for V1 and user if available" but the user filter is only in the SAAS subclass.
3. **Noop authorizer / noop claim mapper as the default.** Temporal's noop grant `System: RoleAdmin`. A misconfigured server is wide-open.
4. **Composite primary keys without row-level security.** langfuse's `TraceSession @@id([id, projectId])` is structural, not access-controlling.
5. **Default-user fallback collapsing anonymous callers.** letta's `get_actor_or_default_async` rewrites no-header requests to `user-00000000-…`.
6. **Three "session"-like concepts in the same codebase.** langfuse has `Session` (NextAuth), `TraceSession` (observability), `InAppAgentConversation` (agent). Wiring state to the wrong scope is a real risk.
7. **`copy_thread` declared but unimplemented in production savers.** langgraph's Postgres/SQLite savers inherit `NotImplementedError` from the base.
8. **No `validate_thread_id` helper for a string-typed ID.** Langgraph allows `"default"`, `""`, `None` as thread IDs.
9. **An `ADMIN` sentinel with no central audit log.** openhands installs `SpecifyUserContext.ADMIN` in ~6 places; a grep enumerates them but a structured audit does not exist.
10. **`assert` for security invariants.** Stripped under `python -O`. openhands uses `assert` for the conversation-ownership invariant.
11. **The base class docstring describes a check that only the subclass implements.** openhands' "Security and permission checks are handled by wrapper services" — only true if the SAAS wrapper is wired.
12. **Storage path encodes user identity with a silent fallback to a different identity.** openhands' `get_conversation_path` falls back to the conversation's `created_by_user_id` when `user_id` is `None`, which can write one user's events under another user's directory.
13. **`acopy_thread`/`aprune`/`adelete_for_runs` capability gap.** langgraph declares these capabilities in the conformance suite but neither Postgres nor SQLite overrides them.
14. **A bulk operation that silently skips cross-tenant IDs with only a log line.** letta's `bulk_update_block_values_async` logs `"skipping during bulk update"` and returns a partial result; the client doesn't see the gap unless it reads logs.
15. **`session_api_key` as a long-lived credential.** openhands treats session keys as sandbox-lifetime proof only — the comment block at `session_auth.py:1-16` is explicit, and `validate_session_key` enforces it.

## Notable Absences

Things that are conspicuously absent or underdeveloped across the sources:

1. **No source has DB-level RLS.** langfuse, letta, openhands OSS all rely on application filters; Temporal and openhands SAAS rely on composite-PK structural isolation, which is not RLS. agent-framework's Cosmos `.NET` hierarchical partition is the closest analogue. This is the dimension's biggest open risk.

2. **No source has a single, comprehensive isolation overview doc.** langgraph's "Future Considerations" suggests adding one; openhands mentions an `ADMIN` central document but it doesn't exist yet. A new contributor trying to understand the full model has to read every interceptor and every store.

3. **No source implements `acopy_thread` in production savers** (langgraph). The conformance suite tests it; the OSS savers raise `NotImplementedError`.

4. **No source has a focused "two users, similar runs, no leakage" end-to-end test** for the conversation info service (openhands has it for the start-task service only).

5. **No source unifies cross-thread memory into a single primitive.** agent-framework uses `scope`, langgraph uses `namespace` tuples, letta uses `isolated_block_labels` + `fork_conversation`, openhands has none, Temporal has none. The vocabulary is inconsistent.

6. **No source has a documented graceful-degradation story for the noop authorizer** (Temporal). The default is "wide open" with no startup warning.

7. **No source has a documented read-after-write consistency story** (langgraph). Postgres uses `autocommit=True` by default; transactional read-your-writes is undocumented.

8. **No source has a per-user audit chain.** Temporal logs `WorkflowID` and `RunID` but not the JWT `Subject`. Letta records `_created_by_id` but doesn't enforce it. Compliance use-cases (who created workflow X?) require sidecars.

9. **No source has a "session" concept that is shared between auth and observability** without naming collisions. Langfuse tries and ends up with three "session"-like concepts. Either disambiguate or unify.

10. **No source implements a `validate_thread_id` helper** that warns when a developer uses a literal `"default"` or empty string. Langgraph and openhands would both benefit.

11. **No source has an end-to-end test that exercises the `ADMIN`/internal sentinel path with an audit assertion.** openhands installs it in ~6 places; no test verifies the audit trail.

12. **No source has a documented rollback story for the `V0 → V1` split** (openhands). The `conversation_metadata_saas` insert can fail after the base save commits, leaving a conversation without SAAS metadata.

## Per-Source Notes

**agent-framework (8/10)** — The strongest score in the study. Two cooperating implementations of the same boundary model (Python + .NET) with consistent per-invocation isolation context. The .NET hosting layer adds a hard runtime guard (`hosted_session_identity_mismatch` 403) that the Python core lacks. Memory providers validate `scope`/`user_id`/`agent_id`/`application_id` up front and raise if missing. Standout: hierarchical Cosmos partition `TenantId+UserId+ConversationId` is the only storage-level tenant boundary in the study besides Temporal. Concern: in-memory providers have no built-in guard.

**langfuse (7/10)** — Clear two-level tenancy with composite keys, RBAC, and middleware guards. The centralized `project_id` filter factory for ClickHouse is a strong pattern. Three "session"-like concepts (`Session`, `TraceSession`, `InAppAgentConversation`) create naming friction. Concern: no Postgres RLS, so isolation is convention. `visibilityScope = PROJECT` is declared but not yet wired into read paths.

**langgraph (7/10)** — Library with minimal primitives and an `Auth` handler returning server-applied `FilterType`. Conformance suite locks the isolation contract across backends. `Store` is documented as "cross-thread memory" and `Runtime.store` is shared by design. Concern: `copy_thread` declared but unimplemented in OSS savers; `user_id`/`tenant_id` not first-class; `thread_id` is just a string.

**letta (7/10)** — Single dominant `organization_id` axis with centralized `apply_access_predicate`. Actor (`User`) threaded through every manager. Prefix-keyed IDs prevent type confusion. `isolated_block_labels` and `fork_conversation` make cross-thread memory explicit. Concern: default-actor fallback collapses anonymous callers; `organization_id` is `String` not FK in `Job`; bulk operations skip cross-org IDs silently with only a log line.

**openhands (7/10)** — Dual-mode (OSS permissive + SAAS hard). SAAS path has secure-default-deny, `(user_id, org_id)` composite index from migration 118, effective-org-id precedence chain, and `ADMIN` sentinel for callback bypass. `session_api_key` is sandbox-scoped and only valid for RUNNING sandboxes. Concern: OSS `_secure_select` docstring lies; fork inherits config but not history; sub-conversation lifecycle half-managed.

**temporal (7/10)** — Namespace as the universal tenant axis. `namespace_id` in every PK, gRPC interceptor chain validates before any handler, per-API role metadata. No user table; identities live in the user's IdP. Cross-namespace commands are a feature flag. Concern: noop authorizer silently bypasses auth; no per-user audit chain.

## Open Questions

These are questions the source reports surfaced but did not fully answer, or that emerge from cross-source comparison.

1. **Is there a principled way to make the `ADMIN`/internal sentinel greppable?** openhands installs `SpecifyUserContext.ADMIN` in ~6 places; Temporal hard-codes `internalClaimMapper`. Neither has a single audit document.

2. **What happens to a forked conversation when the source is deleted?** letta's fork shares `Message` rows via a junction; agent-framework's `to_dict`/`from_dict` is a copy. Both are "safe" but the cleanup story differs.

3. **Can the OSS SAAS seam be made explicit?** openhands' base `SQLAppConversationInfoService` ships a docstring that lies about a user filter. A startup check (or a config-time assertion) would close the gap.

4. **Should cross-thread memory be unified into a single primitive?** agent-framework's `scope`, langgraph's `namespace` tuple, letta's `isolated_block_labels` are three different vocabularies for the same idea. A library that unifies them would be valuable.

5. **Is `EnableTokenNamespaceEnforcement` (Temporal) safe to default on?** The natural progression is to default it on, but the test matrix in `namespace_validator_test.go:578-619` shows matched, mismatched, and disabled cases — default-on may have unintended consequences for callers that legitimately need token-from-token flexibility.

6. **What is the correct posture for `acopy_thread` in OSS savers?** langgraph declares it on the base, tests it in the conformance suite, but Postgres/SQLite raise `NotImplementedError`. Should the base declare it (forcing every saver to either implement or reject) or remove it from the base (treating it as LangGraph Server-only)?

7. **How do you test "two users with similar runs cannot see each other's state" at scale?** openhands' `test_user_isolation` family covers the start-task service. An end-to-end conversation-info-service test, plus a cross-tenant bulk-operation test (letta), would harden the dimension's hardest question.

8. **Is there a documented pattern for `disable_studio_auth` (langgraph) escaping production?** The flag is in the schema but no warning is emitted if it's `true` in production.

9. **What is the right cleanup story for the `conversation_metadata` vs `conversation_metadata_saas` duplication?** openhands carries both tables; the base `created_by_user_id` is no longer authoritative. A migration to deprecate one would simplify the model.

10. **How should cross-workflow memory be modeled when the system has no shared store?** Temporal has none; cross-workflow must use Nexus endpoints or external stores. Is there a Temporal-native pattern?

11. **Should `Project` (letta) become a first-class entity?** Today `project_id` is a free-form string with no FK, no quota, no lifecycle. Promoting it to a `projects` table would enable per-project isolation, billing, and cleanup.

12. **What is the proper handling of "user moves between orgs"?** letta's `Job.organization_id` is nullable; a user's old jobs remain in the old org. No source has a clean migration story.

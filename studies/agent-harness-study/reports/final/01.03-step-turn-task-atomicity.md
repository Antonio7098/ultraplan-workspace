## Executive Summary

Across the eight sources, the **atomic unit of execution is never the same shape for persistence, tracing, retry, and UI** — but the shape of the divergence, and whether the divergence is explicit or accidental, is what separates the better-engineered systems from the weaker ones. Ratings cluster in two bands: a 7/10 band (agent-framework, crewai, letta, openhands) where the model is clear and tested but layered without unification, and an 8/10 band (langgraph, openai-agents-sdk, pydantic-ai, temporal) where the model is mature, observable, and has explicit transactional or schema-versioned safety nets for mid-step crashes.

The dominant pattern: **every source names at least one "atomic unit" in code, and most name two or three**. The agent-framework has four nested units (chat-loop iteration → tool invocation → agent-loop iteration → executor superstep), each with its own counter and event vocabulary; langgraph has three (write / task / superstep); letta and pydantic-ai have three to four (Step / Run / Message / StepMetrics, or graph node / run_step / model request / tool call); temporal has at least six (DB transaction / Executable / Task / Workflow Task turn / HSM MachineTransition / CHASM Transition). Only openai-agents-sdk has deliberately collapsed to a single unit — the **turn** — and only openhands has deliberately refused to define one inside the app_server (it treats the SDK's opaque `Event` as the unit).

The most consequential design choice is **whether tool calls are their own atomic units**. pydantic-ai says yes at the concurrency/retry layer (`_call_tools`, per-tool `max_retries`) and at the durable-execution layer (Temporal `call_tool_activity`); langgraph says no, riding tool calls inside the node's `writes` deque and inheriting the task's retry budget; openai-agents-sdk aggregates them into the turn (`SingleStepResult.new_step_items`); letta aggregates them into the Step row. This single decision drives whether the framework can answer "which tool crashed?" without parsing messages.

The most mature safety pattern is **schema-versioned resume** (openai-agents-sdk's `CURRENT_SCHEMA_VERSION = "1.11"` with `frozenset(SUPPORTED_SCHEMA_VERSIONS)`) plus **rewind-on-retry** (openai-agents-sdk's `rewind_session_items`, langgraph's `match_cached_writes`, pydantic-ai's Temporal `RetryPolicy`). These three together let a crashed mid-step state be recovered deterministically. Letta has the inverse model: it relies on `Step` rows with `PENDING / SUCCESS / FAILED / CANCELLED` and accepts that crashed Steps sit in `PENDING` forever until a sweep (which doesn't exist).

## Core Thesis

**Atomic unit design is a product-shape decision, not an aesthetic choice.** Three product questions determine the answer:

1. **Is the atomic unit a graph node, a state-machine transition, or a linear step?** langgraph and agent-framework say *node / superstep* (BSP graph); temporal says *state-machine transition with rollback* (HSM / CHASM); pydantic-ai says *typed graph node with capability hooks*; openai-agents-sdk says *single turn with a 4-way tagged outcome*; letta and crewai say *named Step / Task with an explicit lifecycle FSM*; openhands says *opaque Event* and refuses to define one.
2. **Are tool calls atomic?** This is the *single biggest* product question. pydantic-ai: yes (per-call retry, per-call span, per-call durable activity). langgraph: no (cached at the node level). openai-agents-sdk: no (aggregated into the turn; recovered via `_current_turn_persisted_item_count` and fingerprint rewind). Most others: aggregated, best-effort.
3. **Must the atomic unit survive a process crash?** If yes, the framework needs a transactional outbox (temporal CHASM, langgraph put/put_writes), or a versioned snapshot (openai-agents-sdk RunState), or a sweepable pending row (letta Step). If no, an in-memory counter suffices.

The convergence is real but shallower than it looks: every framework names an atomic unit and has a stop predicate for it. The divergence is structural: a graph product needs different atomicity than a chat product; a chat product needs different atomicity than a multi-agent orchestration product; an orchestration product needs different atomicity than a sandbox-driven webhook product.

## Approach Models

The eight sources cluster into **five distinct approach archetypes** for atomicity. The archetype is usually a consequence of the product's unit of atomic progress (see Core Thesis) rather than a free choice.

### 1. Named Step with Explicit Lifecycle FSM (letta, crewai P&E, agent-framework Executor)

The persisted atomic unit is a named row (`Step`, `TodoItem`, `Executor`) with an explicit finite-state machine on `status`. Each transition has a dedicated update method that fires webhooks and emits events.

- **letta** (`letta/schemas/step.py:16-65`): `StepStatus ∈ {PENDING, SUCCESS, FAILED, CANCELLED}`. The Step is created with `PENDING` at `_step_checkpoint_start` (`letta/agents/letta_agent_v2.py:946-962`) and updated to terminal state via `update_step_success_async` / `update_step_error_async` / `update_step_cancelled_async` (`letta/services/step_manager.py:368-555`). Messages persist only at step success; this is the explicit atomicity choice.
- **crewai** (`utilities/planning_types.py:27-42`): `TodoItem.status ∈ {pending, running, completed, failed}` with `mark_running` / `mark_completed` / `mark_failed` methods. The new P&E executor (`experimental/agent_executor.py:126-161`) carries `AgentExecutorState` with `_finalize_lock` + `_finalize_called` and `_execution_lock` to prevent double-finalize — explicit safety guards absent from the legacy `CrewAgentExecutor`.
- **agent-framework** (`_workflows/_executor.py:30`): `Executor.execute` emits `executor_invoked` / `executor_completed` / `executor_failed` / `executor_bypassed` events. The `_bypassed` event is explicitly the "what did not run" signal on cache hit — a deliberate replay-awareness pattern.

**Distinguishing features**: explicit lifecycle FSM, separate update methods, webhooks fired outside DB session, "named step is the answer to what completed."

**Failure mode**: when the FSM is the only durable signal, a process crash between PENDING and the terminal update leaves an orphaned PENDING row. None of the three sources has a sweep that re-classifies stale PENDING rows.

### 2. BSP / Pregel Superstep (langgraph, agent-framework)

The persisted atomic unit is the **superstep**: one batch of parallel task executions plus the channel update that ends the batch. Tasks within the tick are concurrent; channel updates *between* tasks are atomic.

- **langgraph** (`libs/langgraph/langgraph/pregel/main.py:2974-2978`): explicit comment "Similarly to Bulk Synchronous Parallel / Pregel model, computation proceeds in steps, while there are channel updates." `tick()` → runner → `after_tick()` → `_put_checkpoint({"source": "loop"})` (`libs/langgraph/langgraph/pregel/_loop.py:706`). Two-tier storage: `put` for the `Checkpoint` snapshot, `put_writes` for per-task writes indexed by `(task_id, idx)` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:31,92-123`).
- **agent-framework** (`_workflows/_runner.py:99-156`): `Runner.run_until_convergence` is documented as "A class to run a workflow in Pregel supersteps." `State.set()` writes to `_pending`; `commit()` flushes pending to committed at the superstep boundary (`_workflows/_state.py:6-127`). Checkpoints serialize committed state only.

**Distinguishing features**: BSP barrier with explicit `apply_writes` boundary, two-tier storage (checkpoint + pending_writes), deterministic task-id hashing for replay, `Durability = "sync" \| "async" \| "exit"` as a first-class axis in langgraph.

**Failure mode**: superstep is *large* for fine-grained recovery. A tick can produce many writes; if the process dies after some `put_writes` but before `_put_checkpoint`, the `_put_exit_delta_writes` synthetic stub at `libs/langgraph/langgraph/pregel/_loop.py:1201-1292` recovers correctness but at the cost of an extra checkpoint row.

### 3. State-Machine Transition with Transactional Outbox (temporal)

The persisted atomic unit is the **DB transaction** wrapping Mutable State + History Events + queued Tasks. Multiple atomic-unit abstractions coexist (DB transaction / Executable / Task / Workflow Task turn / HSM MachineTransition / CHASM Transition), but the persistence boundary is unambiguous.

- **temporal** (`docs/architecture/history-service.md:294-322`): "Consistency of Mutable State and History Tasks is achieved via use of database transactions. ... Transfer Task queue processor ... ensures Transfer Tasks ... result in the required Workflow or Activity Task being created in Matching Service (Transactional Outbox Pattern)."
- **CHASM** (`docs/architecture/chasm.md:216-238`): "A Transition is the atomic unit of state change in CHASM. ... All Field writes and Task schedules from a transition commit together as a single database write, or roll back entirely."

**Distinguishing features**: transactional outbox (state + tasks commit together), `VersionedTransition` logical clock for cross-DC total ordering, Speculative WT for zero-DB-write turns, DLQ for terminal failures, panic-recover at Executable boundary.

**Failure mode**: HSM/CHASM overlap during the migration; `IsRetryableError` always returns false (deliberate — "never retry task while holding the goroutine"). Multiple abstractions increase cognitive load.

### 4. Single-Unit Discriminated Step Loop (openai-agents-sdk)

The persisted atomic unit is the **turn**: one model call + the synchronous/concurrent tool calls that consume its tool calls, ending in one of four `next_step` outcomes. Persistence, tracing, and UI agree on the turn; retry is finer (model call).

- **openai-agents-sdk** (`src/agents/run_internal/run_steps.py:178-218`): `SingleStepResult` with `next_step` tagged union (`NextStepHandoff | NextStepFinalOutput | NextStepRunAgain | NextStepInterruption`). `CURRENT_SCHEMA_VERSION = "1.11"` (`src/agents/run_state.py:131`) and `SUPPORTED_SCHEMA_VERSIONS = frozenset(...)` form a fail-fast resume contract. `_current_turn_persisted_item_count` + `_last_processed_response` (`src/agents/run_state.py:257-264`) let `save_result_to_session` recover from a partial turn save.

**Distinguishing features**: tagged-union step outcome in the type system, schema-versioned resume, rewind-on-retry for both session and server-managed conversation, soft cancel via `after_turn`, span closure in `finally`.

**Failure mode**: tool calls are aggregated not atomic; retry granularity is model call not turn (no per-tool idempotency token); realtime / voice / Codex siblings lack `RunState` parity.

### 5. Typed Graph Node + Capability Hook Chain (pydantic-ai)

The persisted atomic unit is the **graph node** (`BaseNode` subclass) wrapped in `BaseStatePersistence.record_run`. The agent loop is itself a graph (`UserPromptNode` → `ModelRequestNode` ↔ `CallToolsNode` → `SetFinalResult`), and a 12-hook capability middleware chain wraps every transition.

- **pydantic-ai** (`pydantic_graph/pydantic_graph/persistence/__init__.py:32-41`): `SnapshotStatus = {'created', 'pending', 'running', 'success', 'error'}`. `record_run` context manager sets status to `running` / `success` / `error` with `start_ts` / `duration` (`pydantic_graph/pydantic_graph/persistence/in_mem.py:55-74`). `iter_from_persistence` resumes from the next pending snapshot (`pydantic_graph/pydantic_graph/graph.py:260-309`).
- **Tool atomicity** (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1862-1965`): `_call_tools` is itself an atomic scope with three concurrency modes (`parallel` / `sequential` / `parallel_ordered_events`). Each tool has its own `max_retries` and `ctx.retries[name]` (`tool_manager.py:177-181`).
- **Durable wrappers** (`pydantic_ai/durable_exec/temporal/_function_toolset.py:43-105`): `TemporalFunctionToolset.call_tool_activity` wraps each tool call as a separate Temporal activity with per-tool `ActivityConfig`.

**Distinguishing features**: capability hook chains (`before_node_run` / `wrap_node_run` / `after_node_run` / `on_node_run_error`), `ValidatedToolCall` split (validate phase / execute phase), three durable-engine wrappers sharing one base, `for_run_step` propagates per-tool retries across steps.

**Failure mode**: agent loop does *not* snapshot graph nodes by default — recovery is `message_history` reconstruction, not node-snapshot replay. `run_step` is not in the snapshot schema. Three atomic units (`run_step` / `usage.requests` / `usage.tool_calls`) are incremented in different functions and can drift under refactor.

### Side-cluster A: Event-Sourced Append-Only (openhands)

The persisted atomic unit is the **opaque Event** from the SDK, persisted as one JSON file/blob/object per event. There is no batching or transactional boundary across multiple events; partial mid-event saves are possible only if a write is interrupted.

- **openhands** (`openhands/app_server/event/filesystem_event_service.py:33-36`): `_store_event` calls `path.write_text(content)` for one event. The path is `v1_conversations/{conversation_id_hex}/{event_id_hex}.json`. Batch save uses `asyncio.gather(*[event_service.save_event(...)])` (`openhands/app_server/event_callback/webhook_router.py:464-466`) with no transaction. The webhook handler returns `Success` even on partial save failure (`openhands/app_server/event_callback/webhook_router.py:519-522`).
- **AppConversationStartTask** (`openhands/app_server/app_conversation/app_conversation_models.py:262-271`): 7-state state machine `WORKING → WAITING_FOR_SANDBOX → PREPARING_REPOSITORY → RUNNING_SETUP_SCRIPT → SETTING_UP_SKILLS → STARTING_CONVERSATION → READY (or ERROR)`, persisted per-yield as the coarse checkpoint trail.

**Distinguishing features**: append-only persistence (no in-place mutation), idempotent save by UUID-keyed filename, opaque `EventKind` literal auto-derived from SDK subclasses.

**Failure mode**: no transactional boundary across events; "step"/"turn" vocabulary absent; `turn_count` always `None`; callbacks fire after HTTP 200; orphan start task on mid-yield crash has no automatic resume.

## Pattern Catalog

The following patterns recur in multiple sources or are demonstrated unusually clearly by one. Each pattern is named with the question it solves and the evidence supporting it.

### P1. State-Machine Lifecycle on the Atomic Unit

**Problem solved**: how to say exactly what completed and what did not, without parsing logs.

**Demonstrated by**: letta (`StepStatus` 4-state, `letta/schemas/enums.py:268-274`), crewai (`TodoItem.status`, `utilities/planning_types.py:27-42`), agent-framework (`WorkflowRunState` + `Executor` events, `_workflows/_events.py:58-68`), pydantic-ai (`SnapshotStatus` 5-state, `pydantic_graph/pydantic_graph/persistence/__init__.py:32-41`), temporal (`TaskState` 5-state FSM, `common/tasks/state.go:6-17`; CHASM `Transition` with rollback, `service/history/hsm/tree.go:579-629`), openhands (`AppConversationStartTaskStatus` 7-state, `app_conversation_models.py:262-271`).

**Why it works**: a closed enum with documented transitions gives every reader a stable vocabulary. The transition itself becomes the unit of replay.

**When to copy**: any system where the atomic unit is named and persisted. Skipping this means relying on log parsing.

**When overkill**: ephemeral, in-memory execution that doesn't need resume (e.g., short-lived CLI runs).

### P2. Discriminated Union for Step Outcome

**Problem solved**: how to model "step ends in one of N different states" without exceptions or sentinel returns.

**Demonstrated by**: openai-agents-sdk (`SingleStepResult.next_step` 4-way tagged union, `src/agents/run_internal/run_steps.py:154-218`), agent-framework (`MiddlewareTermination` with optional `result` attribute, `_middleware.py:72-80`; `NextStepHandoff` / `NextStepFinalOutput` equivalent via `RequestInfoEvent`).

**Why it works**: the compiler enforces exhaustiveness; the outer loop branches on the discriminant; streaming and non-streaming can share the same dispatch.

**When to copy**: any state machine where the next-step outcome is one of a fixed set of structurally different shapes (final output vs handoff vs run-again vs interrupt).

**When overkill**: a single terminal state with a side-channel (e.g., exceptions).

### P3. Two-Tier Storage (Checkpoint + Pending Writes)

**Problem solved**: how to keep the checkpoint row small while preserving in-flight writes for replay.

**Demonstrated by**: langgraph (`put` for `Checkpoint` snapshot, `put_writes` for per-task writes indexed by `(task_id, idx)` with negative `WRITES_IDX_MAP` indices for control writes, `libs/checkpoint/langgraph/checkpoint/base/__init__.py:795`; `libs/checkpoint/langgraph/checkpoint/memory/__init__.py:497-509`), temporal (transactional outbox: history events + queued tasks in one DB write, `docs/architecture/history-service.md:294-322`).

**Why it works**: the snapshot stays small enough to write quickly; in-flight writes are replayed on resume via `_reapply_writes_to_succeeded_nodes` (`libs/langgraph/langgraph/pregel/_loop.py:724-737`).

**When to copy**: any system where the atomic unit produces multiple writes that need to commit independently but must be replayable together.

**When overkill**: a single record per atomic unit (letta's Step row).

### P4. Rewind-on-Retry for State + Server-Managed Side Effects

**Problem solved**: how to retry a model call without leaving orphan items in the session or server-managed conversation.

**Demonstrated by**: openai-agents-sdk (`rewind_session_items` pops a serialized tail suffix matched by item fingerprint, `src/agents/run_internal/session_persistence.py:416-468`; `server_conversation_tracker.rewind_input` rewinds server-managed deltas, `src/agents/run_internal/run_loop.py:1876-1880`; `wait_for_session_cleanup` verifies the rewound items are actually gone, `src/agents/run_internal/session_persistence.py:518-557`), langgraph (`match_cached_writes` for replay-cache, `libs/langgraph/langgraph/pregel/_loop.py:1526-1539, 1780-1793`).

**Why it works**: the retry layer can pop partial state without knowing the session's internals; server-managed conversations stay in lockstep with the runner's view.

**When to copy**: any system that delegates state to an external service (OpenAI Responses `conversation_id`, server-side cache, etc.).

**When overkill**: a system with no server-managed state and no session.

### P5. Span-as-Context via ContextVar

**Problem solved**: how to wrap every model call, tool call, and node in an OTel span without threading a span object through every function signature.

**Demonstrated by**: openai-agents-sdk (`Span.start(mark_as_current=True)` registers in a contextvar, `src/agents/tracing/spans.py:100-108`; tool calls and model calls become children of whichever span is current, `src/agents/run_internal/tool_execution.py:1033`), pydantic-ai (`_CURRENT_RUN_CONTEXT` ContextVar + `set_current_run_context`, `pydantic_ai_slim/pydantic_ai/_run_context.py:275-305`; `@trace_method` decorator for per-method spans), letta (`tracer.start_span("agent_step", start_time=step_start_ns)`, `letta/agents/letta_agent_v2.py:944-945`; `trace_id` column on `Step` row).

**Why it works**: hooks, capability middleware, and tool bodies can read the current span / run context without it being passed through every signature.

**When to copy**: any framework with hooks / middleware / capability system. Skip the parameter threading entirely.

**When overkill**: a system with no hooks or capability layer.

### P6. Frozen Dataclass Boundaries

**Problem solved**: how to make the contract between layers explicit and impossible to mutate by accident.

**Demonstrated by**: crewai (`StepExecutionContext` and `StepResult` are `@dataclass(frozen=True)`, `utilities/step_execution_context.py:13,44`; comment: "carries only final results from dependencies — never LLM message histories"), pydantic-ai (`RunContext` and `GraphAgentState` are Pydantic models with strict field declarations; `ValidatedToolCall` separates `args_valid` from execution), openai-agents-sdk (`RunState` is a Pydantic model with strict schema versioning).

**Why it works**: the type system prevents in-place mutation of inputs that downstream code relies on; tests can compare instances by value.

**When to copy**: any interface between two layers where the consumer must not modify the producer's state.

**When overkill**: high-throughput in-memory mutation where copying would dominate runtime cost.

### P7. Per-Step Tool Lifecycle via run_step Counter

**Problem solved**: how to scope per-tool retry counters and dynamic-toolset re-evaluation to one model round-trip without conflating them with graph traversal.

**Demonstrated by**: pydantic-ai (`GraphAgentState.run_step` incremented in `ModelRequestNode._prepare_request`, `_agent_graph.py:856`; `ToolManager.for_run_step` propagates per-tool retries only when `run_step` advances, `tool_manager.py:120-145`), openai-agents-sdk (turn-counter via `_current_turn += 1` immediately before `run_single_turn_streamed`, `src/agents/run_internal/run_loop.py:875-876`).

**Why it works**: dynamic toolsets are re-evaluated once per step (not once per graph node, which might fire multiple LLM calls); per-tool retries survive across steps without leaking.

**When to copy**: any framework where toolsets can be mutated mid-run.

**When overkill**: static toolsets.

### P8. Tool Calls as Separate Atomic Units

**Problem solved**: how to give tool calls their own retry budget, span, and durable activity.

**Demonstrated by**: pydantic-ai (`_call_tools` treats each `ToolCallPart` as its own atomic unit with three concurrency modes, `_agent_graph.py:1862-1965`; per-tool `max_retries` + `ctx.retries[name]`; `TemporalFunctionToolset.call_tool_activity` wraps each tool call as a separate Temporal activity, `durable_exec/temporal/_function_toolset.py:43-105`).

**Why it works**: a flaky tool can be retried in place; billing and observability are per-tool; cross-tool partial completion is recoverable.

**When to copy**: any framework where tool failures are common, tools have side effects, or the framework ships a durable-execution wrapper.

**When overkill**: read-only tools with no failure mode (filesystem read, in-memory lookup).

### P9. Helper-Method Checkpoint Pattern

**Problem solved**: how to make the agent loop's progress visible at named, individually-traceable points.

**Demonstrated by**: letta (`_step_checkpoint_start`, `_step_checkpoint_llm_request_start`, `_step_checkpoint_llm_request_finish`, `_step_checkpoint_finish`, `letta/agents/letta_agent_v2.py:940-1034`; `_request_checkpoint_start`, `_request_checkpoint_ttft`, `_request_checkpoint_finish`, `letta/agents/letta_agent_v2.py:916-938`), pydantic-ai (capability hook chains `before_node_run` → `wrap_node_run(handler)` → `on_node_run_error` → `after_node_run`, `pydantic_ai_slim/pydantic_ai/capabilities/hooks.py:115-128, 899-937`).

**Why it works**: each named checkpoint is a named trace span and a named place to attach metrics.

**When to copy**: any agent loop with a non-trivial inner structure (LLM call + tool execution + compaction).

### P10. Origin-Tagged Events for Replay-Awareness

**Problem solved**: how to distinguish framework-orchestrated events from user-executor events in the same stream.

**Demonstrated by**: agent-framework (`WorkflowEvent.origin = FRAMEWORK | EXECUTOR` carried by `_event_origin_context: ContextVar`, `_workflows/_events.py:38-55`; `executor_bypassed` event on cache hit is the explicit "what did not run" signal, `_workflows/_events.py:340-342`).

**Why it works**: a UI consumer can filter framework noise from user events; replay can distinguish "skipped via cache" from "never scheduled."

**When to copy**: any framework with executor-or-node abstractions that can be cached or bypassed.

### P11. Transactional Outbox for State + Tasks

**Problem solved**: how to commit application state and queued follow-up work in one atomic operation.

**Demonstrated by**: temporal (Mutable State + History Events + queued Tasks in one DB transaction; Transfer Tasks later move work to Matching Service, `docs/architecture/history-service.md:294-322`), langgraph (two-tier storage with `put_after_previous` ordering).

**Why it works**: eliminates the "applied state without task" or "task without applied state" inconsistency class entirely.

**When to copy**: any system where state changes generate follow-up work that must be durably queued.

### P12. Schema-Versioned Resume Contract

**Problem solved**: how to evolve the persistence schema without breaking older SDKs in production.

**Demonstrated by**: openai-agents-sdk (`CURRENT_SCHEMA_VERSION = "1.11"` with 12 released versions, `src/agents/run_state.py:131-149`; `SUPPORTED_SCHEMA_VERSIONS = frozenset(...)`; older SDKs reject newer versions, `src/agents/run_state.py:128-130`), crewai (`_migrate` patches missing discriminator fields based on `crewai_version`, `state/runtime.py:89-167`).

**Why it works**: forward-compat is fail-fast; downgrade is explicit; rollback strategy is recoverable.

**When to copy**: any persisted-state object that survives across SDK versions.

**When overkill**: in-memory state only.

## Key Differences

### Convergence across sources

1. **The atomic unit is never the same shape for persistence, tracing, retry, and UI** — every source has at least two layers, and most have three or four.
2. **The persisted unit is the one that's most carefully defined** — all sources except openhands have an explicit lifecycle FSM on the persisted unit.
3. **The retried unit is often finer than the persisted unit** — letta retries LLM calls inside a Step; openai-agents-sdk retries the model call inside a turn; pydantic-ai retries per-tool inside a step.
4. **Tool calls are aggregated at the message/persistence layer in 5 of 8 sources** (agent-framework, crewai, langgraph, letta, openai-agents-sdk), and first-class atomic in 3 (pydantic-ai via `_call_tools` and Temporal wrappers, openhands via opaque Event but no per-tool row, langgraph via `CachePolicy` at the node level).

### Divergence based on product shape

| Divergence axis | Reason |
|---|---|
| **Single unit vs. layered units** | openai-agents-sdk collapses to *turn* because the SDK is narrow (no multi-agent, no graph); agent-framework / crewai / langgraph / pydantic-ai / temporal layer multiple units because they expose multi-agent or graph surfaces |
| **Atomic tool calls vs. aggregated** | pydantic-ai's per-tool retry makes sense because it ships a durable-execution wrapper where per-tool idempotency matters; openai-agents-sdk aggregates because server-managed `conversation_id` dominates the retry story |
| **Transactional outbox vs. best-effort** | temporal and langgraph have transactional outbox; letta, crewai, openhands, agent-framework, openai-agents-sdk use best-effort (in-memory state + versioned snapshot, or append-only event log) |
| **Framework-managed resume vs. external-engine resume** | openai-agents-sdk, letta, langgraph, agent-framework, pydantic-ai's graph all have framework-managed resume (RunState, Step, Checkpoint, WorkflowCheckpoint, NodeSnapshot); pydantic-ai's durable exec defers to Temporal/DBOS/Prefect for resume; temporal's resume is the engine itself |
| **Schema-versioned vs. in-place evolution** | openai-agents-sdk schema-versioned (`CURRENT_SCHEMA_VERSION`); crewai uses `_migrate` patching; others rely on in-place evolution with optional manual migration |
| **Public vs. private atomic unit** | openai-agents-sdk exposes the turn as the public contract; agent-framework has four units but only `AgentResponse` is public-facing; pydantic-ai exposes graph nodes but keeps `run_step` private |

## Tradeoffs

### T1. Multi-layer atomicity without unification vs. single-unit contract

**Benefit**: each layer has its own retry/durability/observability contract (langgraph's put/put_writes, agent-framework's superstep + executor + chat loop).

**Cost**: the user must reason about which layer a question applies to; there is no single "what completed" answer (agent-framework's four counters; langgraph's three; temporal's six).

**Best-fit context**: graph products, multi-agent orchestration, systems with multiple execution surfaces.

**Failure mode**: documentation drift; counters that don't correlate; an unifier that almost-but-not-quite-works (`SingleStepResult.session_step_items` vs `new_step_items` in openai-agents-sdk).

**Alternative seen in**: openai-agents-sdk's single-turn contract; pydantic-ai's "graph node is the answer" with run_step as a sub-counter.

### T2. Per-tool atomic units vs. aggregated tool calls

**Benefit**: per-tool retry, per-tool span, per-tool durable activity, per-tool idempotency (pydantic-ai's `call_tool_activity`).

**Cost**: requires the framework to expose per-tool retry budget configuration; aggregated systems get this "for free" via the model-call retry.

**Best-fit context**: durable execution; non-idempotent tools; observability requirements.

**Failure mode**: cached at the node level (langgraph) means a tool change inside the node is not observable; aggregated at the turn level (openai-agents-sdk) means a tool that succeeded but the model call failed will re-execute on retry.

**Alternative seen in**: openai-agents-sdk's `_current_turn_persisted_item_count` + `_last_processed_response` (counter + fingerprint rewind); langgraph's `match_cached_writes` (cache the node output).

### T3. Transactional outbox vs. best-effort snapshot

**Benefit**: outbox eliminates "state without task" / "task without state" inconsistency (temporal CHASM, langgraph two-tier).

**Cost**: requires transactional storage; harder to retrofit; harder to debug ("which transaction did this state come from?").

**Best-fit context**: mission-critical workflow; cross-service atomicity requirements.

**Failure mode**: best-effort snapshot can have a window where state is applied but task is not queued; best-effort event log (openhands) can have partial batch saves with no recovery.

**Alternative seen in**: agent-framework's `WorkflowCheckpoint` (best-effort, with `graph_signature_hash` validation on restore); openai-agents-sdk's `RunState` (best-effort, with schema-versioned resume).

### T4. Append-only event log vs. stateful row

**Benefit**: append-only is replay-friendly and audit-friendly; one event = one file is cheap to test.

**Cost**: no transactional boundary across multiple events; partial state on mid-batch failure; cannot answer "what is the current state" without replaying.

**Best-fit context**: observability-first products; webhooks; analytics.

**Failure mode**: openhands' `asyncio.gather` batch save can leave a partial event set with no rollback; corrupt files silently dropped.

**Alternative seen in**: letta's `Step` row (single record per atomic unit, updated in place); langgraph's `Checkpoint` + `pending_writes` (two-tier, but not append-only).

### T5. Framework-managed resume vs. external-engine resume

**Benefit**: framework-managed keeps the contract surface narrow and the resume logic versioned (openai-agents-sdk RunState).

**Cost**: the framework's resume logic is coupled to its persistence schema; cannot delegate to a battle-tested engine.

**Best-fit context**: narrow SDK; library users; versioned public contract.

**Failure mode**: framework-managed resume can drift from the actual execution semantics (openai-agents-sdk realtime / voice / Codex siblings lack RunState parity).

**Alternative seen in**: pydantic-ai's Temporal wrappers (`call_tool_activity`); temporal itself.

## Decision Guide

### When you need true durable execution

Choose **temporal-style** (transactional outbox + state-machine transitions with rollback) or **pydantic-ai durable exec** (Temporal / DBOS / Prefect wrappers around model requests and tool calls). Both commit state and tasks atomically and have explicit recovery semantics. The cost is operational complexity (Temporal requires a cluster; DBOS / Prefect require their respective orchestrators).

### When you need a narrow public contract

Choose **openai-agents-sdk** (single turn unit + schema-versioned resume) or **pydantic-ai's graph mode** (typed graph nodes with capability hooks). Both expose one named atomic unit to the user and have explicit failure semantics. The cost is that finer-grained observability (per-tool) requires going through OTel spans rather than the public contract.

### When you need explicit step status

Choose **letta** (`Step` row with 4-state FSM) or **agent-framework Executor** (with `executor_invoked` / `completed` / `failed` / `bypassed` events). Both make the persisted unit a named row with an explicit lifecycle. The cost is the orphaned-PENDING-row failure mode — neither has a sweep that re-classifies stale PENDING rows.

### When you need graph-style atomicity

Choose **langgraph** (BSP superstep + two-tier storage + `Durability` axis) or **pydantic-ai graph mode** (typed nodes + capability hooks). Both have explicit channel-update barriers and replay-friendly checkpointing. The cost is the cognitive load of multiple atomic units (langgraph's three: write / task / superstep; pydantic-ai's four: node / run_step / model request / tool call).

### When you need fine-grained tool retry

Choose **pydantic-ai** (`_call_tools` with three concurrency modes + per-tool `max_retries`) or **pydantic-ai's Temporal wrappers** (`call_tool_activity` with per-tool `ActivityConfig`). The cost is the configuration surface (per-tool `max_retries` + `tool_activity_config`).

### When you need versioned durability

Choose **openai-agents-sdk** (`CURRENT_SCHEMA_VERSION` + `frozenset(SUPPORTED_SCHEMA_VERSIONS)`). The schema-versioning is the most explicit in the survey; forward-compat is fail-fast. The cost is the inability to downgrade an SDK after a schema bump.

## Practical Tips

### Patterns to copy

1. **State-machine lifecycle on the atomic unit.** The closed-enum approach (letta's `StepStatus`, pydantic-ai's `SnapshotStatus`, temporal's `TaskState`) is the most-tested pattern in the survey. Every named unit should have a 3-5 state FSM with documented transitions.
2. **Schema-versioned resume contract.** openai-agents-sdk's `CURRENT_SCHEMA_VERSION` + `SUPPORTED_SCHEMA_VERSIONS` is the gold standard. Without it, persistence evolution is a permanent liability.
3. **Two-tier storage (Checkpoint + Pending Writes).** langgraph's `put` + `put_writes` keeps the snapshot small while preserving in-flight writes for replay. The negative-index `WRITES_IDX_MAP` is a clever reuse of the saver's primary key.
4. **Span-as-context via ContextVar.** openai-agents-sdk and pydantic-ai demonstrate this; it eliminates the parameter threading that bloats every function signature in legacy systems.
5. **Rewind-on-retry for state + server-managed side effects.** openai-agents-sdk's `rewind_session_items` + `server_conversation_tracker.rewind_input` is the only pattern that keeps the runner and the server in lockstep.
6. **Helper-method checkpoint pattern.** letta's `_step_checkpoint_start` / `_llm_request_start` / `_llm_request_finish` / `_finish` makes every inner step a named trace span.
7. **Origin-tagged events.** agent-framework's `WorkflowEvent.origin = FRAMEWORK | EXECUTOR` + `executor_bypassed` event is the only pattern that explicitly answers "which side of the boundary produced this?" and "what did not run on this tick?"
8. **Transactional outbox for state + tasks.** temporal's CHASM is the gold standard; langgraph's two-tier storage is the lightweight alternative.

### Patterns to avoid or delay until needed

1. **Multiple atomic-unit vocabularies without a unifying type.** agent-framework's four units and temporal's six units are real cognitive load; only adopt if the layering is genuinely justified (multi-agent, multi-runtime, multi-engine).
2. **Best-effort max_function_calls.** agent-framework's "best-effort" cap is the most surprising behavior in the survey; if you need hard limits, use `tool_choice="none"` or `max_iterations` instead, and document it.
3. **Append-only event log with no transactional boundary.** openhands' `asyncio.gather` batch save can leave partial state with no recovery; only adopt if observability is the primary product.
4. **Shared mutable message list with no transactional boundary.** crewai's legacy `CrewAgentExecutor.messages: list[LLMMessage]` is the failure mode the new P&E executor explicitly avoids with `StepExecutionContext` (frozen dataclass).
5. **Tool cache write not transactional with execution.** crewai's `crew_agent_executor.py:1002-1005` cache write is the documented failure mode — a tool succeeds, the process crashes, the cache misses, the tool re-executes on retry.

### Decision rules

| If you need... | Choose | Why |
|---|---|---|
| A single named atomic unit | openai-agents-sdk (turn) | Narrow contract, schema-versioned resume |
| A state-machine lifecycle on the unit | letta (Step), pydantic-ai (Snapshot), temporal (Task) | Explicit FSM + tests for transitions |
| A BSP / superstep atomic unit | langgraph, agent-framework | Channel barrier + checkpoint per superstep |
| Graph-style atomicity with type inference | pydantic-ai (`BaseNode`) | Edges inferred from `BaseNode.run()` return types |
| True transactional outbox | temporal CHASM, langgraph two-tier | State + tasks commit atomically |
| Per-tool retry budget | pydantic-ai | `max_retries` per tool + `ctx.retries[name]` |
| Per-tool durable activity | pydantic-ai Temporal wrappers | `call_tool_activity` per tool |
| Versioned durable schema | openai-agents-sdk | `CURRENT_SCHEMA_VERSION` + `frozenset(SUPPORTED_SCHEMA_VERSIONS)` |
| Append-only audit trail | openhands, crewai events | Replay-friendly, idempotent saves |

## Anti-Patterns / Caution Signs

1. **Multiple atomic-unit vocabularies without a unifying type.** agent-framework has four counters (`attempt_idx`, `iteration`, `iteration_count`, `iteration` again) that don't correlate. A refactor that moves one but not the others desyncs them.
2. **Persisted unit ≠ retried unit without a documented rewind story.** openai-agents-sdk handles this with `_current_turn_persisted_item_count` + fingerprint rewind; most others don't.
3. **Tool cache write not transactional with tool execution.** crewai's `crew_agent_executor.py:1002-1005` is the documented failure mode.
4. **Mid-event callbacks running outside any transaction boundary.** openhands' `asyncio.create_task(_run_callbacks_in_bg_and_close, ...)` lets the HTTP 200 succeed before callbacks complete; silent failure mode.
5. **Best-effort `max_function_calls`.** agent-framework's `_tools.py:2636-2644` explicitly states the cap is checked *after* the batch — "the current batch always runs to completion even if it overshoots."
6. **Orphaned PENDING rows on process crash.** letta, openhands (start task), and agent-framework (executor) all leave named atomic units in an in-flight state with no automatic sweep.
7. **Schema evolution without versioning.** crewai's `_migrate` patches missing discriminator fields, but the patch is keyed on `crewai_version`; openhands' `_load_event` returns `None` on schema drift (silent event loss).
8. **Shared mutable message list as the persisted unit.** crewai's legacy `CrewAgentExecutor.messages` is the failure mode the P&E path explicitly avoided.
9. **In-memory counters that aren't persisted.** agent-framework's `_ensure_not_running` raises on concurrent `agent.run` (one instance at a time), but the counter is in-memory — a process restart loses the lock.
10. **No test for mid-step crash recovery.** Searched `tests/integration_test_cancellation.py` (letta — only Redis-based cancellation), `tests/test_run_impl_resume_paths.py` (openai-agents-sdk — only resume paths, not crash paths). The "rollback to previous state" claim in `letta/agents/letta_agent_v3.py:1513` is by convention, not by enforced contract.
11. **Webhooks returning 200 on partial failure.** openhands' `webhook_router.py:519-522` logs and returns `Success` on partial save — the sandbox never retries, so the partial state is permanent.
12. **Schema-versioned resume that breaks downgrades.** openai-agents-sdk's `SUPPORTED_SCHEMA_VERSIONS = frozenset(...)` is fail-fast: downgrading an SDK after a schema bump loses the ability to resume the new snapshot.

## Notable Absences

1. **No source unifies persistence / tracing / retry / UI at one atomic-unit boundary.** Every source makes the user consult two or three different counters or row types to answer "what completed and what did not."
2. **No source has a sweep that re-classifies orphaned PENDING rows.** letta, openhands (start task), agent-framework (executor) all leave atomic units in `PENDING` / `WAITING_FOR_SANDBOX` / etc. with no automatic recovery. Operators must detect and triage manually.
3. **No source ships a first-class `Tool Call` atomic unit at the public contract surface.** pydantic-ai's per-tool retry and Temporal wrappers exist, but they are internal — the public contract is the turn / step / node. Only langgraph's `CachePolicy` exposes per-tool state at the public level (via the cache key).
4. **No source has a public `Turn` abstraction as a standalone type.** pydantic-ai uses `run_id` (one run = one turn) and `conversation_id` (multi-run conversation), but no `Turn` class. openai-agents-sdk has the concept but not the type.
5. **No source makes mid-event-callback atomicity explicit.** openhands runs callbacks in background after HTTP 200; agent-framework runs middleware in pipeline order; pydantic-ai runs capability hooks synchronously. None treats "callback" as a transactional unit.
6. **No source has a public atomic-step API in the SDK sense.** temporal exposes RPCs (`StartWorkflow`, `SignalWorkflow`, `RespondWorkflowTaskCompleted`); the atomic unit visible to the SDK is the Workflow Task turn, not the transition. langgraph exposes `stream_mode` (7 modes); the atomic unit visible depends on the mode.
7. **No source answers "what completed and what didn't" across multi-transaction scenarios.** temporal's DB transaction is atomic, but across transactions (e.g., a Workflow Task that schedules 5 activities then fails), the system relies on history events + tasks to reconstruct progress — explicit "transaction receipt" is not exposed beyond `StateTransitionCount`.

## Per-Source Notes

### agent-framework

- **Atomic unit vocabulary**: four layered units (chat-loop iteration, tool invocation, agent-loop iteration, executor superstep).
- **Where implemented**: `_tools.py` (FunctionInvocationLayer), `_harness/_loop.py` (AgentLoopMiddleware), `_workflows/_runner.py` + `_executor.py` (superstep + executor), `_workflows/_checkpoint.py` (persistence).
- **Main mechanism**: explicit per-layer atomicity with dedicated counters and bounded loops.
- **Supporting**: Middleware pipelines (Agent + Function), exception-to-structured-error-result conversion, state staging (committed / pending), checkpoint at superstep boundary.
- **Standout patterns**: layered atomicity as design principle, `RequestInfoEvent` persisted as first-class atomic state on checkpoint, origin-tagged events, `MiddlewareTermination` as structured exit.
- **Tradeoffs / failure modes**: `max_function_calls` is best-effort, four layers can't answer "what completed" at a single boundary, `fresh_context` truncates `conversation_id`, no unified replay semantic, agent-loop crash semantics rely on `PerServiceCallHistoryPersistingMiddleware`.
- **Gaps**: no unifying abstraction across four layers, agent-loop crash semantics implicit, no transactional semantics on agent-loop crash.

### crewai

- **Atomic unit vocabulary**: three coexisting units (iteration, TodoItem / step, Task) plus event-sourced architecture.
- **Where implemented**: `crew_agent_executor.py` (legacy ReAct), `experimental/agent_executor.py` (P&E), `step_executor.py` (per-step worker), `state/runtime.py` (RuntimeState checkpoint).
- **Main mechanism**: event-sourced architecture; tasks / events as primary unit with checkpoint triggers.
- **Supporting**: frozen dataclass isolation (StepExecutionContext), guardrail retry loop, event DAG with adjacency, branch-based checkpoint lineage.
- **Standout patterns**: frozen-dataclass boundaries (StepExecutionContext / StepResult), status state machine on entities, branch-based checkpoint lineage, Plan-and-Act isolation per `arxiv 2503.09572`.
- **Tradeoffs / failure modes**: two parallel executors with different contracts, tool cache write not transactional with execution, messages shared mutable list, default checkpointing too coarse.
- **Gaps**: no per-iteration checkpoint without `["*"]`, legacy `CrewAgentExecutor` still primary on some paths.

### langgraph

- **Atomic unit vocabulary**: three layered units (write, task, superstep) plus explicit `Durability = "sync" | "async" | "exit"` axis.
- **Where implemented**: `_loop.py` (PregelLoop), `_algo.py` (task scheduling), `_runner.py` (task execution), `_retry.py` (retry wrapper), `checkpoint/` (persistence).
- **Main mechanism**: superstep barrier with two-tier storage (`put` / `put_writes`), ordered put-after-previous drain.
- **Supporting**: deterministic task-id hashing, reserved write keys for control flow (INTERRUPT / ERROR / RESUME), `_reapply_writes_to_succeeded_nodes`, delta-channel bookkeeping.
- **Standout patterns**: deterministic task-id hashing for replay / time-travel, two-tier storage (Checkpoint + PendingWrite), `_coerce_pending_error` deserialization, ordered put-after-previous durability.
- **Tradeoffs / failure modes**: superstep large for fine-grained recovery, `put_writes` fire-and-forget, retry per task not per write, stream modes are mode-dependent.
- **Gaps**: no canonical atomic event for observability, no transactional step boundaries, v1 / v2 stream contracts coexist.

### letta

- **Atomic unit vocabulary**: three layered units (Step, Run, Message) plus `StepProgression` 6-stage in-memory FSM.
- **Where implemented**: `step.py` / `orm/step.py` (Step), `run.py` (Run), `services/step_manager.py` (status updates), `letta_agent_v3.py` (agent loop).
- **Main mechanism**: Step row with 4-state lifecycle, separate `StepMetrics` row, OTel `trace_id` correlation.
- **Supporting**: StepProgression FSM, helper-method checkpoints, auto-resolved model patching, webhook firing outside DB session.
- **Standout patterns**: StepProgression 6-stage FSM, two DB IDs (trace_id + request_id) cross-correlate Step, webhook fired outside DB session, auto-mode patches Step row.
- **Tradeoffs / failure modes**: Step larger than retry boundary, no automatic resume of crashed Step, tool calls aggregated (no per-tool breakdown), messages persist only on Step success.
- **Gaps**: no canonical atomic event spanning Step / StepMetrics / OTel, no test for mid-step crash recovery.

### openai-agents-sdk

- **Atomic unit vocabulary**: single unit (turn) with `SingleStepResult.next_step` as a 4-way tagged union; sub-units (model request, tool call, span) inherit the turn.
- **Where implemented**: `run_internal/run_steps.py` (SingleStepResult), `run_internal/run_loop.py` (turn execution), `run_state.py` (RunState), `run_internal/model_retry.py` (retry with rewind).
- **Main mechanism**: turn = model call + tool side-effects; schema-gated durable resume.
- **Supporting**: tagged-union step outcome, span-as-context (ContextVar), persist-counter idempotence, capability-tagged retry policies, soft cancel.
- **Standout patterns**: tagged-union step outcome (4-way), schema-version-gated resume (fail-fast), rewind-on-retry for server-managed conversations, span-as-context, soft cancel preferred.
- **Tradeoffs / failure modes**: tool calls aggregated not atomic, retry granularity is model call not turn, schema versioning fail-fast, realtime / voice / Codex siblings lack RunState parity.
- **Gaps**: no test for crash between tool execution and session save, no tool-call idempotency token.

### openhands

- **Atomic unit vocabulary**: five units in the app_server (Event, AppConversationStartTask, AppConversationInfo, PendingMessage, EventCallback); no step / turn in this layer.
- **Where implemented**: `event/event_service_base.py` (Event), `app_conversation/live_status_app_conversation_service.py` (start task), `event_callback/sql_event_callback_service.py` (callbacks).
- **Main mechanism**: one event = one JSON file / blob / object; append-only persistence.
- **Supporting**: async-generator-as-task-progress, opaque EventKind typing, per-event UUID in filename, pre-start pending message buffering.
- **Standout patterns**: async-generator-as-task-progress, opaque EventKind typing via Literal subclass enumeration, hex-stripped UUIDs in storage, lock-bypass on missing Redis.
- **Tradeoffs / failure modes**: granular events with no transaction, webhook returns Success even on partial save, `turn_count` always `None`, callbacks run after HTTP 200, no automatic resume of orphaned start task.
- **Gaps**: "step" / "turn" vocabulary absent, no retry queue for failed event saves.

### pydantic-ai

- **Atomic unit vocabulary**: four units (graph node, run_step, model request, tool call) plus durable-engine wrappers (Temporal / DBOS / Prefect).
- **Where implemented**: `pydantic_graph/` (graph nodes + persistence), `pydantic_ai/_agent_graph.py` (agent loop), `tool_manager.py` (tool execution), `durable_exec/` (engine wrappers).
- **Main mechanism**: graph node = durable unit + `BaseStatePersistence.record_run` + capability hook chains.
- **Supporting**: per-step tool resolution via run_step, two retry budgets (output + per-tool), tool validation / execution split, capability hooks.
- **Standout patterns**: capability hook chains (`before_node_run` / `wrap_node_run` / `after_node_run` / `on_node_run_error`), context-var current run context, `SkipModelRequest` short-circuit, three durability models sharing one base.
- **Tradeoffs / failure modes**: three atomic units coupled by hand, run_step not in snapshot schema, agent loop does not snapshot graph nodes by default.
- **Gaps**: agent-loop nodes not snapshotted by default, run_step not on snapshot schema, no public Turn abstraction.

### temporal

- **Atomic unit vocabulary**: six nested units (DB transaction, Executable, Task, Workflow Task turn, HSM MachineTransition, CHASM Transition).
- **Where implemented**: `service/history/workflow/mutable_state_impl.go` (DB transactions), `service/history/queues/executable.go` (Executable lifecycle), `service/history/tasks/` (Task types), `service/history/hsm/tree.go` (HSM), `chasm/` (CHASM).
- **Main mechanism**: transactional outbox pattern; tasks written atomically with state.
- **Supporting**: 5-state Task FSM, Operation log + compact, VersionedTransition logical clock, Speculative WT for latency, panic-recover at Executable boundary.
- **Standout patterns**: transactional outbox for state + task durability, VersionedTransition total ordering, Speculative WT for zero-DB-write turns, DLQ as terminal failure destination.
- **Tradeoffs / failure modes**: multiple atomic abstractions increase cognitive load, `IsRetryableError` always false, `MutableState` rejects dirty with `Unavailable`.
- **Gaps**: HSM / CHASM overlap during migration, no global ordering across shards except per-namespace version.

## Open Questions

1. **Can the system say exactly what completed and what did not?** Across all sources, the answer is "partially." Every framework can say what its *own* atomic unit did, but the cross-layer question (e.g., "did the LLM call succeed but the tool call fail?") requires correlating two or three different state pieces.
2. **Is there a unified "atomic step" abstraction in any framework?** No. Even temporal's CHASM Transition is a step in that direction, but it coexists with the older HSM, MutableState, and Executable abstractions during the migration. None of the frameworks has a single type that says "this is one unit of work, persisted, retried, traced, and shown to the user."
3. **How does the system recover from a mid-step crash if the process dies before the checkpoint writes?** agent-framework, openai-agents-sdk, pydantic-ai's graph mode all rely on the *previous* checkpoint being intact; the current step is lost. langgraph has the `_reapply_writes_to_succeeded_nodes` partial-recovery mechanism, but only for write-level recovery, not step-level. temporal's transactional outbox is the only true answer: the transaction either commits or doesn't.
4. **What is the canonical atomic event for observability?** Every source has *some* named event (Step, RunState, Checkpoint, NodeSnapshot, Task, WorkflowTaskInfo, Event), but no single event type spans all four observability axes. No source has a unified "atomic event" record joining step / metrics / OTel trace / UI state.
5. **Does the framework guarantee `function_call` / `function_result` pairing on persistence?** With `require_per_service_call_history_persistence=True` (agent-framework), history is persisted after the model call but a mid-batch tool crash can leave a `function_call` without a `function_result`. Recoverable on next run via history replay, but the design is implicit. No source has an explicit "function call pair" invariant at the persistence layer.
6. **Is there a test for mid-step crash recovery in any source?** Searched `tests/integration_test_cancellation.py` (letta — only Redis-based cancellation), `tests/test_run_impl_resume_paths.py` (openai-agents-sdk — only resume paths), `tests/test_function_invocation_logic.py` (agent-framework — only `max_function_calls`). The "rollback to previous state" claims are by convention, not by enforced contract.
7. **Are there any frameworks that recover from a `PENDING` Step row after a process crash?** None of letta, openhands (start task), agent-framework (executor) has a sweep that re-classifies stale in-flight rows. Operators must detect and triage manually.

## Executive Summary

Across the five sources, mutation discipline varies along a spectrum that correlates with the system's threat model rather than its domain. Three convergent themes emerge: (1) every system converges on a small, well-defined set of mutation entry points — channel `update()` in langgraph, `State.set`/`QueueStateUpdateAsync` in agent-framework, `SqlalchemyBase.create_async`/`update_async`/`delete_async` in letta, service interfaces in openhands, and `Transition.Apply`/`MutableStateImpl.UpdateWorkflowStateStatus` in temporal; (2) observability is universally prioritized — temporal records operation logs and OTel spans, agent-framework emits superstep events, langgraph exposes value streams, letta wraps nearly every manager in `@trace_method` plus `log_event`, and openhands persists webhook events; (3) per-execution serialization is implemented at three different layers (in-process semaphore, distributed Redis lock, row-level OCC). The major divergence is the *formality* of transition guards: temporal and langgraph reject illegal transitions via typed predicates and dedicated error types; letta enforces the discipline for *some* subsystems (files, jobs) and logs-and-commits for others (runs, steps); agent-framework and openhands rely on documentation plus runtime checks. No source achieves the ideal "impossible states are unrepresentable"; instead each balances enforcement against authoring ergonomics and the cost of type-system machinery.

## Core Thesis

**Mutation discipline is a product of the durability promise, not the language.** Systems whose failure is expensive (temporal — multi-year workflows with replication) invest in declared transitions, row-level optimistic concurrency, and atomic version counters. Systems whose workflow steps are short and recoverable (langgraph, agent-framework) can rely on per-step batching with sequenced `update()` calls. Systems whose primary hazard is user-driven HTTP traffic (openhands, letta) end up with mixed models — strict where one hot path dominates (Letta's Block optimistic lock), permissive everywhere else. The lesson is that you cannot study mutation discipline in isolation: it is downstream of durability requirements, concurrency expectations, and the cost of one lost update.

## Approach Models

### `Staging + Per-Step Commit` — agent-framework

Writes are buffered in a per-executor "pending" buffer (`State._pending` in Python at `python/packages/core/agent_framework/_workflows/_state.py:25-104`; `_queuedUpdates` in .NET at `dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StateManager.cs:177-233`) and committed exactly once per superstep by the runner. Concurrent writes to the same key inside a superstep follow last-write-wins, documented in code rather than prevented by exception. Adoption is universal across the Python and .NET runtimes but absent from the agent-session layer (a plain mutable dict).

### `Channel as Sole Mutator + Version Vector + Per-Step Batching` — langgraph

Every state change is a call to `BaseChannel.update(values) -> bool` (`libs/langgraph/langgraph/channels/base.py:90`), invoked exclusively from `apply_writes` (`libs/langgraph/langgraph/pregel/_algo.py:232-345`). Even when a `BackgroundExecutor` runs node bodies concurrently, their writes are batched by channel and applied in deterministic sorted order. A monotonic version vector (`channel_versions`) plus per-node `versions_seen` provides optimistic "MVCC" semantics for triggering. Illegal transitions (`LastValue` getting multi-writes, multiple `Overwrite` per step) raise `InvalidUpdateError` with `ErrorCode.INVALID_CONCURRENT_GRAPH_UPDATE` (`libs/langgraph/errors.py:36-99`).

### `Centralized ORM Funnel + Optimistic Lock + Distributed Lock` — letta

A single chokepoint (`SqlalchemyBase.create_async`/`update_async`/`delete_async` at `letta/orm/sqlalchemy_base.py:560-791`) enforces actor attribution, `updated_at`, soft-delete, deadlock retry, and `StaleDataError → ConcurrentUpdateError` mapping. Three superimposed safeguards layer on top: SQLAlchemy `version_id_col` on the `Block` ORM (`letta/orm/block.py:56-61`); Redis distributed lock per conversation with 5-minute TTL (`letta/data_sources/redis_client.py:194-229`); and a hard-guarded state machine for files (`letta/services/file_manager.py:200-274`). Job and File state machines validate transition tuples inside SQL `WHERE` clauses; Run and Step status transitions only log.

### `Service Interface + Direct Mutation + Last-Writer-Wins` — openhands

State is divided between durable metadata (`AppConversationInfo` at `sources/openhands/openhands/app_server/app_conversation/app_conversation_models.py:112-181`) and runtime-derived status (`AppConversation` rebuilt at `sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:630-663`). Service methods mutate model fields directly via `setattr` (`live_status_app_conversation_service.py:2138-2145`) and call `merge` + `commit` (`sql_app_conversation_info_service.py:347-388`). Target locks are surgical: per-conversation Redis lock for export (`live_status_app_conversation_service.py:2354-2401`) and per-user in-process lock for profile writes (`settings_router.py:313-320`). No general OCC.

### `Declared Transitions + Validator + OCC + Per-Execution Semaphore` — temporal

Two parallel state-machine frameworks (HSM at `service/history/hsm/sm.go:32-66` and CHASM at `chasm/statemachine.go:24-79`) implement `Transition[S comparable, SM StateMachine[S], E any]` with explicit `Sources` lists. `MachineTransition` (`hsm/tree.go:579-629`) atomically bumps `TransitionCount`, snapshots `prevLastUpdatedVersionedTransition`, defers rollback on error, and forces cache invalidation. Below that, a validator (`workflow/mutable_state_state_status.go:16-125` plus `common/persistence/workflow_state_status_validator.go:30-85`) rejects illegal `WorkflowExecutionState × WorkflowExecutionStatus` combos before persistence; row-level `DBRecordVersion` (`common/persistence/persistence_interface.go:414-500`) plumbed into `Condition` catches same-shard races; shard `RangeID` (`service/history/shard/context_impl.go:212-217`) catches host-failover races; and a `PrioritySemaphore(1)` (`service/history/workflow/context.go:43`) serialises within one execution.

## Pattern Catalog

### Pattern: Single mutation entry point (universal)

**Problem.** When state lives in many places, impossible combinations creep in.
**Sources.** All five. Langgraph funnels everything through `apply_writes`; agent-framework through `State.set` and `QueueStateUpdateAsync`; letta through `create_async`/`update_async`/`delete_async`; openhands through `save_app_conversation_info` and the SQL service layer; temporal through `MachineTransition`/`MutableContext`.
**Why it works.** Restricting writers narrows the audit surface and lets tests pin behavior at one boundary.
**When to copy.** Always. Cost is low; benefit grows with codebase size.
**When it is overkill.** Trivial apps where the only writer is the request handler.
**Evidence.** `libs/langgraph/langgraph/pregel/_algo.py:232-345`; `python/packages/core/agent_framework/_workflows/_state.py:30-104`; `letta/orm/sqlalchemy_base.py:560-791`; `chasm/statemachine.go:24-79`.

### Pattern: Staging + atomic commit at a synchronization point

**Problem.** Mid-transaction reads must never see partial writes.
**Sources.** agent-framework stages writes in `_pending`/`_queuedUpdates` and commits at the superstep boundary (`python/packages/core/agent_framework/_workflows/_runner.py:140-146`; `dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunner.cs:341-369`). langgraph batches per-superstep writes through `apply_writes` (`libs/langgraph/langgraph/pregel/_algo.py:232-345`). Temporal's CHASM `CloseTransaction` (`chasm/tree.go:1630-1681`) syncs sub-components, handles lifecycle, and serialises in a single pass. Temporal HSM `MachineTransition` (`hsm/tree.go:600-607`) defers rollback on error.
**Why it works.** It converts "is the system consistent?" from a property of all the writers to a property of one moment.
**When to copy.** Any time a logical step happens across multiple fields/keys/channels.
**When it is overkill.** Single-key writes (the lock IS the staging).
**Evidence.** `chasm/tree.go:579-629`, `python/packages/core/agent_framework/_workflows/_state.py:90-100`, `libs/langgraph/langgraph/pregel/_algo.py:316-345`.

### Pattern: Monotonic version counter (per channel / per row)

**Problem.** Detect stale reads and determine trigger eligibility.
**Sources.** langgraph: per-channel `channel_versions` plus per-node `versions_seen` (`libs/langgraph/langgraph/pregel/_algo.py:262-269, 1260-1277`). Temporal: `VersionedTransition{TransitionCount, NamespaceFailoverVersion}` flowing through `NextTransitionCount()` (`workflow/mutable_state_impl.go:1185-1198`); letta: SQLAlchemy `version_id_col` on `Block` (`letta/orm/block.py:56-61`).
**Why it works.** A monotone counter is cheaper than an epoch clock and suffices for both "did this channel move since node X last ran?" and "is this row stale?".
**When to copy.** Always — counters are cheap, irreplaceable, and integrate with replication.
**When it is overkill.** In-process data with a single writer.
**Evidence.** `libs/checkpoint/langgraph/checkpoint/base/__init__.py:692-711`; `service/history/hsm/tree.go:619-626`; `letta/orm/block.py:56-61`.

### Pattern: Explicit transition guard (Sources → Destination)

**Problem.** Prevent illegal state moves from being representable.
**Sources.** Temporal (HSM and CHASM) is the strongest instance (`hsm/sm.go:32-66`, `chasm/statemachine.go:24-79`). Langgraph encodes allowed transitions via channel types (`LastValue` rejects multi-writes at `libs/langgraph/langgraph/channels/last_value.py:56-67`; `BinaryOperatorAggregate` rejects duplicate `Overwrite` at `libs/langgraph/langgraph/channels/binop.py:117-127`; `NamedBarrierValue` rejects unknown values at `libs/langgraph/langgraph/channels/named_barrier_value.py:56-67`). Letta enforces for `FileProcessingStatus` and `JobStatus` (`letta/services/file_manager.py:200-274`, `letta/services/job_manager.py:84-94`) but not for `RunStatus` (logged only at `letta/services/run_manager.py:342-356`).
**Why it works.** Source-state lists make the *legal moves* the unit of code rather than scattered `if` blocks. Tests that fail on adding an illegal move are extremely stable.
**When to copy.** Whenever an enum has more than two values and the *next* state depends on the *current* state.
**When it is overkill.** Pure read-only status displays.
**Risk.** Guard tables need maintenance as transitions are added.
**Evidence.** `chasm/lib/activity/statemachine.go:38-87`; `libs/langgraph/langgraph/channels/binop.py:109-130`; `letta/services/file_manager.py:200-274`.

### Pattern: Optimistic concurrency at the row level

**Problem.** Two writers to the same row must not silently overwrite each other.
**Sources.** Temporal: `DBRecordVersion` plus SQL `FOR UPDATE` row locks (`common/persistence/sql/sqlplugin/mysql/execution.go:28-32`). Letta: SQLAlchemy `version_id_col` on `Block`, mapping `StaleDataError → ConcurrentUpdateError` (`letta/orm/sqlalchemy_base.py:779-780`) to HTTP 409 (`letta/server/rest_api/app.py:584`).
**Why it works.** Roll-your-own version columns plus a refused-on-mismatch path surface conflicts at the storage boundary.
**When to copy.** Any table where the simultaneous-edit window is non-trivial.
**When it is overkill.** Append-only tables or single-writer-per-row.
**Risk.** Retry storms if many clients contend.
**Evidence.** `common/persistence/persistence_interface.go:414, 448, 488`; `letta/orm/block.py:56-61`.

### Pattern: Per-execution / per-conversation distributed lock

**Problem.** Multiple workers must not interleave writes that belong to one logical session.
**Sources.** Letta's `acquire_conversation_lock` (`letta/data_sources/redis_client.py:194-229`). OpenHands' export lock (`live_status_app_conversation_service.py:2354-2401`).
**Why it works.** Linearizing one conversation's writes trades throughput for predictable ordering. TTL bounds the damage from a crashed worker.
**When to copy.** Whenever a per-session mutation must be serial (e.g. message append, profile save).
**When it is overkill.** Stateless requests.
**Risk.** Lock loss on Redis outage (Letta's `NoopAsyncRedisClient` at `letta/data_sources/redis_client.py:593-625` silently degrades — anti-pattern).
**Evidence.** `letta/data_sources/redis_client.py:194-348`; `sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2354-2401`.

### Pattern: Per-execution in-process semaphore

**Problem.** Within one process, two requests against the same workflow should serialize.
**Sources.** Temporal's `workflow.ContextImpl.lock: locks.PrioritySemaphore(1)` (`service/history/workflow/context.go:43, 73, 84-93`).
**Why it works.** Single-execution write mutex is tighter than redis and keeps the cache hot.
**When to copy.** Hot single-key hot paths where the write cost dominates.
**Risk.** Does not extend cross-process; pair with row-level OCC for safety.
**Evidence.** `service/history/workflow/context.go:43-100`; `service/history/workflow/mutable_state_impl.go:7451-7463` (refuses new tx if previous is dirty).

### Pattern: Operation log / append-only transition record

**Problem.** Need a deterministic replay-by-record for replication and debugging.
**Sources.** Temporal HSM `OperationLog` with `TransitionOperation` and `DeleteOperation` plus deterministic `compact()` (`service/history/hsm/tree.go:87-148, 709-805`).
**Why it works.** A co-ordered log per node lets replication ship one stream instead of replaying events; `compact()` removes operations under deleted ancestors.
**When to copy.** When you need deterministic cluster failover or replay-during-debug.
**Evidence.** `service/history/hsm/tree.go:619-626, 709-805`.

### Pattern: Event-based mutation observability

**Problem.** Operators need to know what changed.
**Sources.** All five, at differing scopes. Letta: `@trace_method` plus `log_event` (`letta/otel/tracing.py:228, 484`) plus terminal-only run callback (`letta/services/run_manager.py:450-510`). Agent-framework: superstep start/complete events with `StateUpdated` flag (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcStepTracer.cs:17-26`). Temporal: OTel `chasm.transition` events (`chasm/statemachine.go:55-67`) and `state_transition_count` histogram (`common/metrics/metric_defs.go:703`).
**When to copy.** Any user-facing system. The cheap form is method-level tracing; the powerful form is per-transition events.
**Evidence.** `chasm/statemachine.go:55-67`; `letta/services/run_manager.py:500-503`; `python/packages/core/agent_framework/_workflows/_events.py:104-130`.

### Pattern: Reserved namespace for framework-owned keys

**Problem.** Users will write into framework-owned slots.
**Sources.** Agent-framework reserves `_`-prefixed keys with mixed enforcement (`python/packages/core/agent_framework/_workflows/_state.py:20-23` vs `_functional.py:294-301`). Langgraph reserves `NO_WRITES`, `PUSH`, `RESUME`, `INTERRUPT`, `RETURN`, `ERROR` (`libs/checkpoint/langgraph/checkpoint/serde/types.py:12-16`). Letta separates bookkeeping into `is_deleted`, audit fields, and side tables (`letta/orm/base.py:13-85`).
**Why it works.** Side-stepping the reserved namespace is impossible if the framework reads first and overwrites unconditionally.
**When to copy.** Whenever the framework owns a small portion of shared state.
**Risk.** Quiet overwrites of user data when enforcement is uneven (agent-framework footgun).
**Evidence.** `python/packages/core/agent_framework/_workflows/_functional.py:294-301`; `libs/checkpoint/langgraph/checkpoint/serde/types.py:12-16`.

### Pattern: Send the receiver an explicit error code

**Problem.** Generic `RuntimeError` for "two writers, one slot" hides the cause.
**Sources.** Langgraph: `InvalidUpdateError` with `ErrorCode.INVALID_CONCURRENT_GRAPH_UPDATE` (`libs/langgraph/errors.py:34-99`). Temporal: `ErrInvalidTransition` (`hsm/sm.go:9-10`), `serviceerror.NewFailedPrecondition` (`chasm/statemachine.go:14`), `ErrStaleReference` (`hsm/tasks.go:64-81`). Letta: `ConcurrentUpdateError` -> 409 (`letta/errors.py:72-78` + `letta/server/rest_api/app.py:584`).
**Why it works.** Programmatic error types are debuggable; clients can branch on `errors.Is(err, ErrInvalidTransition)` rather than string-match.
**When to copy.** Always for any class of "this should be impossible" — tests pinning the error type are the cheapest insurance.
**Evidence.** `libs/langgraph/langgraph/errors.py:34-99`; `service/history/hsm/sm.go:9-10`.

### Pattern: Soft-assert on invariant violation (fail loud in dev, log in prod)

**Problem.** Strict assertions panic production; silent warnings are ignored.
**Sources.** Temporal's `softassert.UnexpectedInternalErr` (`common/softassert/serviceerror.go:21-56`) logged on internal inconsistency in CHASM at `chasm/tree.go:450, 572, 580, 608, 634, 690, 753, 806, 830, 899, 911, 933, 983, 1043, 1057, 1123, 1132`.
**Why it works.** Keeps production stable while logging indicates drift.
**Risk.** If the drift goes unaddressed, the inconsistency is real.
**Evidence.** `common/softassert/softassert.go:28-45`.

## Key Differences

### Formality of the FSM

**Where they differ.** Temporal and langgraph reject illegal transitions with typed errors (CHASM/HSM `ErrInvalidTransition`; langgraph `InvalidUpdateError`); letta enforces for files/jobs and *logs but commits* for runs/steps; agent-framework and openhands have no FSM at all.

**Why.** Temporal's threat model is a multi-year workflow with strict SLA — illegal moves must never be persisted. LangGraph's threat model is reproducible graph runs — wrong writes make replays non-deterministic. Letta has heterogeneous subsystems with different stakes. Agent-framework and openhands optimize for authoring speed over invariant enforcement.

**What to learn.** Match FSM formality to the cost of one bad transition. Temporal sets the ceiling; openhands sets the floor; the middle is an explicit choice.

### Lock placement: in-process vs database vs distributed

**Where they differ.** Temporal uses all three layers (semaphore in `workflow.ContextImpl`, OCC via `DBRecordVersion`, lease via `RangeID`). Langgraph relies on Pregel-loop serialization plus monotonic versioning — no DB locks. Letta combines ORM funnel + Redis lock + `version_id_col` on Block. Agent-framework uses asyncio single-thread (Python) or `ConcurrentDictionary` per key (.NET). OpenHands uses service-level `setattr`/`merge` with surgical Redis locks for export and process-local locks for settings.

**Why.** Lock placement maps to the consistency boundary: cross-process requires OCC or a distributed lock; cross-workflow requires a per-execution semaphore; cross-statement requires transactional semantics inside the DB call.

**What to learn.** Identify *each* concurrency boundary and pick the lock that fits its scope. A single approach usually under-covers.

### Observability channels

**Where they differ.** Temporal emits both per-transition OTel events and metrics (`state_transition_count` histogram); langgraph exposes streams (`CONFIG_KEY_SEND: deque.extend` + `StreamMux`); letta uses `@trace_method` + webhook callbacks; agent-framework uses superstep events; openhands uses webhook event persistence plus WebSocket broadcasts.

**Why.** Cost. OTel spans in tight loops (temporal, letta) are gated by debug mode; per-step event emission (agent-framework, langgraph) is cheap; webhook fan-out (letta, openhands) is expensive but reaches external integrations.

### Handling of the same-key-write race

**Where they differ.** Temporal HSM and langgraph channels raise exceptions; langgraph `LastValue` errors on multi-write with a specific error code; agent-framework uses last-write-wins (Python) or per-key lock (.NET); openhands uses pure last-writer-wins.

**Why.** Some systems choose "make the user fix the graph" (temporal, langgraph); others choose "let the most recent user win" (agent-framework Python, openhands). The .NET variant of agent-framework deliberately diverged from Python to enforce.

**What to learn.** Pick last-write-wins explicitly; the surprise is when the choice is *implicit* (openhands).

### What the FSM guards against

**Where they differ.** Temporal guards (a) transition source-destination pairs, (b) closed/paused ancestors, (c) typed reducer issues, (d) workflow `state × status` invariants. Langgraph guards channel-level invariants. Letta guards (a) state × status for files and jobs only, (b) terminal-state predicates via `is_terminal`. Agent-framework guards only the cross-superstep visibility boundary. Openhands guards message-send against sandbox state at the API boundary.

**Why.** Each system has different "this would cause permanent corruption" boundaries. OpenHands picks the API boundary because that's where the consequence (a 503) is visible to the user.

## Tradeoffs

| Choice | Benefit | Cost | Best-fit context | Failure mode | Alternative |
|--------|---------|------|------------------|--------------|-------------|
| Two-tier staging (agent-framework) | Atomic superstep transitions; failure leaves no half-applied state | Hides writes from writer's own subsequent code unless ordering is explicit | Synchronous single-machine agent runtimes | Iteration-boundary ordering bugs | Inline mutations (langgraph's apply_writes) |
| Channel as sole mutator (langgraph) | Local semantic authority per channel | User-supplied reducers must obey purity contract | Replayable graph runtimes | Silent replay divergence | Pure staging (agent-framework) |
| Centralized ORM funnel (letta) | Single chokepoint for actor tracking, retries, OCC mapping | Bypasses through raw SQL still possible; refactor risk for bulk ops | Mixed-tenant SaaS with audit needs | Audit bypass via `BatchCreateAsync` | Per-DAO methods with explicit checks |
| Direct `setattr` + `merge` (openhands) | Simple authoring; broad reuse | Lost updates on concurrent writes; no automatic OCC | App servers where most edits are user-driven and unique | Stale metadata after concurrent PATCH | OCC version column |
| Declared transitions (temporal) | Illegal moves are unrepresentable | Maintenance cost on transition tables; framework duplication (HSM vs CHASM) | Long-running critical workflows | Forgotten transition arms a bug pathway | Runtime exception model (langgraph) |
| Per-execution semaphore (temporal) | Tight lock-down of cache and writes | Single-point contention; doesn't extend cross-process | Hot path with one execution per workflow | Hot key starvation under high load | Channel-level lock (agent-framework .NET) |
| Per-conversation Redis lock (letta) | Linear-per-session writes | TTL bottleneck; cross-region lock latency | Session-rich applications | 5-min tail latency after worker crash | Row-level OCC + retry |
| Operation log (temporal) | Deterministic replication and replay | Log compaction logic is non-trivial | Multi-cluster replication | Compact edge cases | Per-event replication (langgraph `put_writes`) |
| `InvalidUpdateError` style errors (langgraph) | Programmatic handling | User code must catch and decide | Strongly-typed graph runtimes | Caller silently drops the error | Soft-log-and-commit (letta runs) |
| Mixed enforcement (agent-framework) | Authoring flexibility | Footguns where the framework silently overwrites | Multi-language SDK with permissive defaults | User key shadowed by reserved `_`-prefix | Strict enforcement (langgraph channels) |

## Decision Guide

Use this decision tree when designing mutation discipline for a new system:

1. **Is the workflow long-running or replicated?**
   - Yes → declare transitions (Temporal HSM/CHASM style), add OCC at the row, lease at the shard. Set the bar at 9/10.
   - No → continue.

2. **Will multiple workers process the same logical session?**
   - Yes → add a distributed lock for the unit (conversation, session, run). Pair with row-level OCC if cross-table updates exist (Letta's `Block` pattern).
   - No → in-process serialization suffices (Temporal's `PrioritySemaphore` or `ConcurrentDictionary`).

3. **Is replay determinism required?**
   - Yes → funnel all writes through one entry point (langgraph `apply_writes`) and treat reducer purity as a contract.
   - No → staging is sufficient; commit at synchronization points.

4. **Are illegal transitions costly to recover from?**
   - Yes → typed transition guards with declared sources lists (Temporal/CHASM pattern). The cost of authoring a transition table is cheaper than the cost of debugging a multi-cluster state corruption.
   - No → runtime exceptions at the channel boundary (langgraph `InvalidUpdateError`).

5. **Is there user-driven mutation?**
   - Yes → adopt an ORM-style chokepoint with actor tracking (Letta `SqlalchemyBase`) and audit fields. Add optimistic lock to the hot rows only; the rest can rely on deadlock retry.

6. **Will the app serve a UI?**
   - Yes → event/stream observability of mutations (letta webhooks; agent-framework superstep events; temporal OTel) is non-negotiable. Sync UI via delta events, not via polling.

7. **Are there reserved framework-controlled slots (e.g. checkpoint keys, internal IDs)?**
   - Yes → reserve a namespace (langgraph `NO_WRITES`, `INTERRUPT`, etc.; agent-framework `_`-prefixed keys) and **enforce consistently** in *all* writes, not just user-facing ones.

## Practical Tips

### Patterns to copy

1. **Funnel through one mutation entry point.** A single SQL base class or one `apply_writes` function gives you a checklist for actor tracking, retries, OCC mapping, and audit fields. (`letta/orm/sqlalchemy_base.py:560-791`; `libs/langgraph/langgraph/pregel/_algo.py:232-345`.)

2. **Stage then commit.** When more than one field changes per logical step, write to a "pending" buffer and commit at a synchronization point. The cost of writing the buffer is small; the gain is that "did this step fully apply?" becomes a yes/no. (`python/packages/core/agent_framework/_workflows/_state.py:25-104`.)

3. **Use a typed transition with explicit source-states.** A `Transition[S comparable, SM StateMachine[S]]` interface with `Sources` and `Destination` is the cheapest way to make illegal moves rejected by the code, not by review. (`hsm/sm.go:32-66`.)

4. **Add row-level optimistic concurrency to shared edits.** SQLAlchemy `version_id_col`, EF Core `RowVersion`, or a manual integer column + `WHERE version = ?` is enough to surface conflicts as retryable exceptions. (`letta/orm/block.py:56-61`; `common/persistence/persistence_interface.go:414`.)

5. **Use a typed error for illegal transitions.** A `ConcurrentUpdateError`, `InvalidUpdateError`, or `ErrInvalidTransition` lets callers programmatically branch and lets you write one regression test per illegal move. (`libs/langgraph/langgraph/errors.py:34-99`.)

6. **Make observation the same shape as state.** Temporal emits an OTel span event per transition; Letta has `log_event` inside the same call sites that mutate; agent-framework emits `SuperStepCompletedEvent` with `StateUpdated`. The mutation and the trace live together.

7. **Reserve framework-owned keys.** Once you have internal slots (`_executor_state`, `INTERRUPT`, `_meta`), reserve the namespace. (`libs/checkpoint/langgraph/checkpoint/serde/types.py:12-16`.)

8. **Distinguish "impossible" from "expensive".** Defensive `validateAccess` walks the ancestor tree (Temporal CHASM at `chasm/tree.go:486-563`) is cheap and prevents cross-closed-state writes — pay this cost; skip strict reducer purity checks.

### Patterns to avoid or delay until needed

1. **Two parallel state-machine frameworks.** Temporal's HSM + CHASM duality is acknowledged friction; don't replicate it unless you have a real reason to keep the old abstraction.

2. **In-memory only flags for state coupling.** `_terminated` and `deleteAfterClose` in CHASM (`chasm/tree.go:113-128`) are deliberately transaction-scoped, but their semantics are invisible to a casual reader. Prefer persisting the flag or making the scope obvious.

3. **Last-write-wins on a list that the framework also rewrites.** Letta's `agent.message_ids` (`letta/services/agent_manager.py:1650-1667`) plus `update_message_ids_async` is a tested desync risk (`tests/managers/test_cancellation.py:121-142`); if you let user code rewrite a list, version it.

4. **Soft-log-and-commit on status.** `letta/services/run_manager.py:342-356` logs illegal status updates but commits them anyway. This is OK for observability but it is *not* a transition guard. Don't conflate the two.

5. **Direct `setattr` + `merge` without OCC.** OpenHands' `save_app_conversation_info` (`sql_app_conversation_info_service.py:347-388`) is fine for an MVP. The moment two endpoints can write to the same row, add a version column.

6. **Silent degradation on lock failure.** `NoopAsyncRedisClient` (`letta/data_sources/redis_client.py:593-625`) papers over Redis outages and is tested-skip-mode in CI. Either fail-closed at startup or accept the loss of concurrency control with eyes open.

7. **Per-step batching with no fallback.** Langgraph assumes a single-writer-per-superstep. If you extend concurrency to multiple cores, also extend the channel-level guards to handle parallel writes (you'll either keep exceptions or move to a CRDT).

### Decision rules

- **Source of truth lives where the row is committed.** Don't keep two parallel state spaces unless the live one is a cache explicitly named as such (OpenHands' `LiveStatusAppConversationService` does this correctly). When you do, document the reconciliation direction.

- **If a transition is missing from the table, it doesn't exist.** Treat the transition table as the source of legality — don't add post-hoc validation in service methods.

- **If a transition is critical (post-completion, post-cancel), check the *ancestor*.** CHASM's `validateAccess` walks the parent chain (`chasm/tree.go:486-563`); this avoids the "the child is fine but the parent is closed" mistake.

- **Errors should be programmable, not strings.** `InvalidUpdateError`, `ErrInvalidTransition`, `ConcurrentUpdateError` all carry enough information for a client to retry or branch.

- **Audit fields come for free — if your ORM has them, set them automatically.** Every write time-stamped with `updated_at` and `last_updated_by_id` is gold for postmortems (`letta/orm/base.py:13-85`).

### Caution signs

- **You have an enum with more than two values and your service methods do `obj.status = new_status`.** Add a transition table.

- **Two endpoints mutate the same row.** Add `version` or `updated_at` to your save path.

- **A "this should be impossible" log line appears in your bug report.** It isn't impossible — push the check into a guard.

- **Your tests do round-trip `merge` + read** but no transition-rejection tests.

- **Two replicas race for the same resource without a lock, OCC, or atomic counter.** Lost updates are coming.

- **Your "reserved" keys have partial enforcement** (agent-framework: `State.set` accepts `_`-prefixed; `FunctionalWorkflow.RunContext.set_state` does not). Pick one and enforce everywhere.

- **Your mutation funnel has unit tests but your real callers bypass it.** Audit raw SQL or direct service-method calls into the ORM; they bypass retries and audit fields.

- **Your observability is method-level but transitions happen between methods.** Add per-transition events.

- **Your application-level predicate for "did this transition happen?" is "diff these two snapshots".** Compute it forward in a transition event instead.

## Notable Absences

- **No source achieves truly impossible states.** All five represent illegal combinations in their data model — temporal at the in-memory `terminated` flag, langgraph at the `UntrackedValue`/`EphemeralValue` semantic, letta at the desync between `Run` and `agent.message_ids`, openhands at the `MISSING + RUNNING` case, agent-framework at the underscore-key collision. The state machines push illegal moves to errors but cannot fully *delete* them from representation.

- **No source has first-class mutation events comparable to langgraph's `StreamMux`.** Letta has method-level tracing; temporal has OTel spans. None expose a per-update, ordered event stream that a consumer can replay exactly as the engine saw mutations.

- **Only temporal uses per-execution in-process locking.** The rest rely on either batching (langgraph), asyncio single-thread (agent-framework Python), `ConcurrentDictionary` (agent-framework .NET, openhands wherever applicable), or external Redis (letta, openhands export).

- **Per-key mutation audit log is absent from agent-framework.** `SuperStepCompletedEvent` carries `StateUpdated: true|false` but not "who changed what". Diagnosing "this key has the wrong value" requires checking the checkpoint, not an event stream.

- **Soft-asserts on bad invariants** are unique to temporal (`common/softassert/serviceerror.go:21-56`). Other sources either raise or silently log — neither is informative.

- **OpenHands lacks any ORM-level OCC.** It's the only source where a `with_for_update` or version predicate is absent at every save site, not just the hot ones.

- **Type-driven enforcement is missing from Python and openhands.** Both rely on dynamic checks where Go generics (Temporal) or compiled C# (agent-framework .NET) can encode transitions into the type system.

## Per-Source Notes

### agent-framework

Two parallel runtimes (Python and .NET) converge on the *same* shape: two-tier staging with per-superstep commit. Notable asymmetries are deliberate: Python uses documented last-write-wins inside a superstep (`python/packages/core/agent_framework/_workflows/_state.py:36-42`); .NET throws `InvalidOperationException` if more than one update for the same key is queued (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StateScope.cs:78-82`). Run-status enums (`RunStatus`, `WorkflowRunState`) are plain enums without FSM guards. The underscore-prefix reservation is enforced at the *user-facing* layer only, creating a documented footgun. Strength: explicit `commit()`/`discard()` API with tests covering both happy and sad paths. Weakness: no OCC, Python `State` not thread-safe.

### langgraph

Channels are the single mutator interface (`BaseChannel.update`). The architecture's signature feature is the monotonic version vector with per-node `versions_seen`, which makes triggers a pure function of state. Illegal transitions raise `InvalidUpdateError` with structured `ErrorCode.INVALID_CONCURRENT_GRAPH_UPDATE`. Schema-versioned checkpoints plus deterministic compaction of `OperationLog`-style delta channels (`DeltaChannel`) bound replay depth. Concurrent threads can append to per-thread `deque[...].extend` safely; the commit pipeline is the single mutator. Weakness: reducer purity contract is not enforced; `InMemorySaver` is non-thread-safe across processes.

### letta

Heterogeneous system. Strong FSM discipline for files and jobs (state transitions validated inside SQL `WHERE` clauses); weak FSM discipline for runs and steps (illegal updates logged but committed). Optimistic locking only on the `Block` ORM (`version_id_col`). Per-conversation distributed lock via Redis with 5-min TTL serializes one-hot conversations. The implicit approval state (reconstructed from `agent.message_ids[-1]`) is a documented risk. Soft-delete via `is_deleted` on most tables; mixed soft-vs-hard delete in conversation cleanup. Weakness: `Run.status` and `Step.status` updates are not guarded; `message_ids` desync after cancellation is explicitly tested.

### openhands

Service-interface mutations with direct object mutation + last-writer-wins persistence. Most data uses `merge` + `commit` without OCC; one `conversation_version` field is a record discriminator, not an update counter. Targeted Redis locks for export; targeted process-local locks for settings. Enums exist (`SandboxStatus`, `AppConversationStartTaskStatus`, `V1ExecutionStatus`) but no transition validator. Webhook event persistence provides strong observability for *agent* events; metadata PATCH updates are not first-class events. Weakness: impossible states are representable (`MISSING` + `RUNNING` requires UI precedence logic to disambiguate).

### temporal

The reference model. Two parallel state-machine frameworks (HSM, CHASM) with `Transition[S comparable, SM, E]` carrying `Sources` lists. Persistence-layer validator (`workflow_state_status_validator.go`) plus in-memory state machine plus OCC (`DBRecordVersion`) plus lease (`RangeID`) plus per-execution semaphore (`PrioritySemaphore(1)`) form a defense-in-depth stack. HSM `MachineTransition` defers rollback on error (`hsm/tree.go:600-607`). CHASM `CloseTransaction` is one atomic pass (`chasm/tree.go:1630-1681`). OTel span events gate on `telemetry.DebugMode()` for production cost. Replication path bypasses the validator with a documented trust-the-source convention. Weakness: framework duplication; in-memory only flags create soft contracts.

## Open Questions

1. **Is there a way to encode "transition source-states" into Python type system?** Both agent-framework Python and openhands would benefit from declared transitions; the gap is whether any third-party library could provide a typed-state-machine pattern at acceptable cost.

2. **What is the empirical frequency of `ConcurrentUpdateError` in Letta's `Block` writes?** The framework reports "low contention" but the deadlock-retry path is commented-out in tests. Without production data, the cost of "always-on OCC" is speculative.

3. **Does langgraph's reducer purity actually cause silent replay divergence in production?** The contract is documented; whether the check is *enforced* is open.

4. **What is the migration cost of consolidating temporal's HSM and CHASM?** Both share the `Transition[S, SM, E]` signature; the difference is context APIs. Estimate?

5. **Can agent-framework Python's `State` be made thread-safe without restructuring the commit protocol?** Adding locks would change the contract — readers might block during a slow commit — so the question is whether the trade-off is acceptable for, say, multi-threaded executor dispatch.

6. **Does OpenHands' choice of best-effort post-switch model persistence (`sources/openhands/openhands/app_server/app_conversation/app_conversation_router.py:593-617`) cause user-visible inconsistency?** The test asserts the inconsistency is accepted; is it actually seen by users?

7. **What is the failure mode when a workflow's `version` (Letta `Block`) drifts across an upstream/downstream boundary that doesn't share the lock?** Cross-service writes can corrupt the version field in a way that the local lock won't catch.

## Executive Summary

Across seven agent frameworks, "reflection" decomposes into four distinct approach models:

1. **First-class loop + judge primitive** (agent-framework 8/10) — a dedicated `LoopEvaluator` middleware and `LoopAgent` that wrap an inner agent, invoke a `should_continue` predicate, and feed structured `JudgeVerdict` feedback back into the next iteration. Magentic One adds a stall counter and a JSON progress-ledger with its own retry budget.
2. **Exception-driven reflection contract** (pydantic-ai 9/10) — a single `ModelRetry` exception unifies every retry site (tools, validators, hooks, unknown tools, empty responses). Two-tier budget (per-tool + global output) raises `UnexpectedModelBehavior` with distinct messages on exhaustion.
3. **Multi-loop bounded retries** (crewai 7/10) — three independent retry loops (guardrail reask, structured-output converter, plan-and-act replan) each with its own counter, evidence format, and exit signal. Per-index retry bookkeeping. No unified LLM-as-judge on the final answer.
4. **Error-channel feedback + retry policy** (langgraph 8/10, openai-agents-sdk 6/10) — failure surfaces as a `ToolMessage(status="error")` or a `function_call_output` that the model sees on the next turn. Bounded by `RetryPolicy.max_attempts` + `recursion_limit` / `max_turns`. No first-class reflection prompt or critic agent.

Two outliers sit at the edges: **letta (6/10)** implements reflection almost entirely as *context-pressure compaction* (an LLM-driven summarizer), with self-correction only on context overflow and tool-rule violations — no critic primitive. **openhands (2/10)** has no agent-side reflection at all in this repo: the agent loop is delegated to external `openhands-sdk` / `openhands-agent-server` packages, and the V1 orchestrator only ships bounded HTTP-timeout retries and a deferred title-polling callback.

The core dimension question — *can the system recover from bad model output without hallucinating success?* — receives a **strong "yes"** when three ingredients appear together: (a) an explicit per-loop retry budget validated at construction, (b) structured evidence supplied to the next model call (typed errors, judge gap analysis, validation error JSON), and (c) exhaustion that raises a typed exception or surfaces a deterministic failsafe rather than returning the in-progress failure chain. pydantic-ai and agent-framework satisfy all three. crewai satisfies (a) and (b) for the per-loop case but lacks a final-answer judge. langgraph satisfies (a) and the tool-error half of (b) but binds semantic re-ask to graph topology choices. openai-agents-sdk and letta satisfy (a) and partial (b) but rely on the model to recover. openhands satisfies none at the orchestrator level.

## Core Thesis

**Reflection in agent frameworks is best understood as a stack of bounded retry loops, each with its own counter, evidence format, and exit signal.** The frameworks converge on three architectural moves — explicit budgets, structured evidence, and typed exhaustion — but diverge sharply on *whether a quality-critic loop exists*. Only agent-framework ships a first-class judge (`AIJudgeLoopEvaluator` + `JudgeVerdict`); pydantic-ai deliberately leaves critic logic to user-supplied capability hooks. The remaining frameworks treat the model's own next turn as the critic and accept that "the same model that failed is the one that decides whether the re-prompt worked."

Three patterns recur across the strongest implementations:

1. **Two retry budgets, not one.** A per-tool/per-iteration counter for transient errors plus a per-run/global counter for output-level recovery. (pydantic-ai's per-tool vs `output_retries_used`; agent-framework's per-loop vs Magentic progress-ledger retry count; langgraph's per-node `RetryPolicy.max_attempts` vs run-wide `recursion_limit`.)
2. **Short-circuit before the predicate runs.** `max_iterations` is checked before the predicate evaluates, so an over-eager judge cannot also be a runaway cost. (agent-framework `_loop.py:657-664`, `LoopAgent.cs:196-200`.)
3. **Evidence goes through the model's own message channel.** Failures become `ToolMessage` / `function_call_output` / `RetryPromptPart` / heartbeat user messages rather than a shadow prompt template. The model sees its own prior response + the error, in the schema it already understands.

The most common weakness is **the missing reflection verdict**: when the loop eventually stops, only the last model message is returned; the evidence that *caused* termination (judge verdict, retry counter, validation reason) is dropped on the floor. pydantic-ai comes closest to closing this gap with `capture_run_messages()` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2064-2099`); agent-framework flags it as an open gap.

## Approach Models

The seven sources cluster into five approach models. Within each cluster, the mechanisms are recognizably the same; across clusters they are different abstractions with overlapping intent.

**Cluster A — Explicit loop + judge (agent-framework).** `LoopAgent` / `AgentLoopMiddleware` wraps an inner agent, runs it, calls a `LoopEvaluator` (`should_continue` predicate) that returns either `Stop`, `Continue(feedback)`, or `ContinueWithMessages(...)`. `AIJudgeLoopEvaluator` is a built-in sub-case that uses a chat-client judge returning a structured `JudgeVerdict`. Magentic adds a richer self-correction system on top: a `progress_ledger` JSON object with `is_request_satisfied` / `is_progress_being_made` / `is_in_loop` / `next_speaker` / `instruction_or_question`, each with a *reason* field; stall counter; reset signal; per-progress-ledger retry budget with linear backoff. Evidence (`agent-framework.md:16-18`): `LoopAgent.cs:65`, `_loop.py:212`, `AIJudgeLoopEvaluator.cs:44`, `MagenticManager.cs:66-106`, `_magentic.py:709-727`.

**Cluster B — Exception-driven reflection contract (pydantic-ai).** A single exception (`ModelRetry`) unifies every retry site. The exception is raised by tool bodies, output functions, output validators, capability hooks, and edge-case handlers (unknown tools, empty responses, tool timeouts). The graph catches it, builds a `RetryPromptPart` carrying structured `list[ErrorDetails]` (`loc` + `msg` + `type` + `input`), appends it as a fresh `ModelRequest`, and gives the model another chance. Two retry budgets: per-tool (`RunContext.retries[name]`) and global (`GraphAgentState.output_retries_used`). Both raise `UnexpectedModelBehavior` with distinct messages on exhaustion. Evidence (`pydantic-ai.md:18-25, 38-50`): `_agent_graph.py:127`, `_agent_graph.py:162-179`, `tool_manager.py:177-181`, `messages.py:1407-1468`.

**Cluster C — Multi-loop bounded retries (crewai).** Three independent loops each with its own counter and exit signal: (1) guardrail reask loop wraps `agent.execute_task` in `for attempt in range(max_attempts)` with `guardrail_max_retries=3`; (2) structured-output converter recurses with `current_attempt + 1` up to `max_attempts=3`; (3) plan-and-act replan maintains `replan_count` capped by `PlanningConfig.max_replans=3`, driven by `PlannerObserver`'s structured `StepObservation` (one of four routing outcomes: `goal_achieved` / `replan_now` / `refine_and_continue` / `continue_plan`). Auxiliary recovery paths handle max-iteration termination (`handle_max_iterations_exceeded` → "force final answer"), context-length overflow, parser errors, and execution exceptions. Evidence (`crewai.md:18-23`): `task.py:1246-1463`, `utilities/converter.py:84-178`, `experimental/agent_executor.py:813-895, 2591-2651`.

**Cluster D — Error-channel feedback + retry policy (langgraph, openai-agents-sdk).** Failures are converted into a message that the model sees on the next turn (`ToolMessage(status="error")` for langgraph, `function_call_output` with failure text for openai-agents-sdk), bounded by an explicit retry counter (`RetryPolicy.max_attempts` for langgraph, `ModelRetrySettings.max_retries` + `max_turns` for openai-agents-sdk). langgraph adds a *second* self-correction surface: a per-node `error_handler` that is scheduled automatically when retries are exhausted, with a frozen `NodeError` context that survives checkpoint resume. openai-agents-sdk adds an opt-in `error_handlers["max_turns"]` callback that receives the `RunErrorData` snapshot. Evidence (`langgraph.md:19-22, 44-50`): `types.py:416-435`, `state.py:857-870`, `pregel/_retry.py:600-612, 841-854`; (`openai-agents-sdk.md:21-23, 71-73`): `retry.py:114-228`, `model_retry.py:511-607`, `error_handlers.py:80-166`.

**Cluster E — Compaction/context-pressure driven (letta).** Self-correction targets *context pressure* rather than *output quality*: a token-budget-triggered summarizer rewrites the in-context message history when it crosses `SUMMARIZATION_TRIGGER_MULTIPLIER * context_window` (default 0.9). On `ContextWindowExceededError`, a bounded retry loop compacts and re-issues the request up to `max_summarizer_retries=3`. Tool-rule violations produce a structured `[ToolConstraintError]` heartbeat that is *injected into the next turn*. There is no critic / reflection agent; the only LLM-in-the-loop for self-evaluation is the summarizer itself. Evidence (`letta.md:18-23`): `letta_agent_v3.py:1093-1299`, `compact.py:220-413`, `helpers.py:501-505`.

**Outlier F — None at the orchestrator level (openhands).** The V1 orchestrator (`openhands/`) is a thin proxy over the externally-shipped `openhands-sdk` / `openhands-agent-server` packages; the agent loop lives there, not here. Within `openhands/`, the only self-correction-shaped loop is `SetTitleCallbackProcessor._poll_for_title` (4 polls × 3s) for a deferred side effect — not agent reflection. Validation is overwhelmingly silent (best-effort delivery, drop-on-parse, `_skip_invalid_profiles`); escalation is missing entirely. Evidence (`openhands.md:16-23, 33-38`): `set_title_callback_processor.py:32-79, 132-138`, `app_conversation_router.py:467-470`, `filesystem_event_service.py:23-31`.

## Pattern Catalog

Patterns that appear in multiple sources or demonstrate unusually clearly how the dimension's problem is solved.

### P1. Two-tier retry budget (per-tool + global)

**Problem.** A single global counter conflates "this tool's arguments were wrong" with "the final output is wrong." The first is often recoverable with a few attempts; the second is often a structural problem the model cannot fix.

**Where.** pydantic-ai is the cleanest example: `RunContext.retries: dict[str, int]` per-tool (`_run_context.py:60-77`) plus `GraphAgentState.output_retries_used: int` per-run (`_agent_graph.py:127`). agent-framework mirrors the pattern with per-loop `max_iterations` plus Magentic progress-ledger retry count. langgraph uses per-node `RetryPolicy.max_attempts` plus run-wide `recursion_limit`. openai-agents-sdk has per-request `max_retries` plus per-run `max_turns=10`.

**Why it works.** The two budgets fail independently. A bad tool call can spend its tool budget, the global counter stays at zero, and a different tool can succeed. Conversely, an output that *validates* but is *wrong* consumes the output budget, which is the right counter to watch.

**When to copy.** Any agent that exposes both tools and structured output. The cost of a unified counter is that a noisy tool blocks the entire run; the cost of two is one extra integer per run + a test that proves isolation (see `pydantic-ai.md` `test_concurrent_runs_output_retry_isolation` at `tests/test_agent.py:10766-10808`).

**When overkill.** Pure tool-only agents (no structured output) only need one counter. Pure single-turn completion agents do not need a reask loop at all.

### P2. Structured evidence goes through the model's own message channel

**Problem.** Failure feedback is most useful when the model can pattern-match it against its own prior response. A shadow "reflection prompt template" requires the model to mentally translate two contexts; a `function_call_output` does not.

**Where.** pydantic-ai's `RetryPromptPart` carries `list[ErrorDetails]` (`loc`/`msg`/`type`/`input`) and is appended to message history alongside the model's prior response (`messages.py:1407-1468`, `_agent_graph.py:752-757`). langgraph's `ToolMessage(status="error", ...)` is injected into the messages channel (`tool_node.py:339-441, 1005-1012`). openai-agents-sdk uses `function_call_output` items (`turn_resolution.py:259-281`). letta's `[ToolConstraintError]` heartbeat is converted to a user message (`helpers.py:501-505`, `letta_agent_v3.py:1987-2036`). crewai's guardrail reask prompt embeds the prior raw task output + validation error verbatim (`task.py:1310-1313`).

**Why it works.** The model already knows how to interpret tool results. Forcing it through the same channel avoids a new prompt schema and keeps the original response visible — which is the most important evidence ("here is what I produced, here is what was wrong with it").

**When to copy.** Always. The only reason to use a separate reflection prompt is when the feedback is so domain-specific that the model's natural response channel cannot carry it (e.g., a long-form critique that does not fit a tool-result schema). Even then, consider whether a custom message type would work better than a shadow template.

**Evidence / caveat.** pydantic-ai's `RetryPromptPart.model_response()` strips `input` for top-level `NativeOutput` errors to avoid duplicating large JSON the model already produced (`messages.py:1454-1462`) — a sensible compaction that preserves the error path while not bloating context.

### P3. Short-circuit the budget before the predicate runs

**Problem.** A `should_continue` predicate that always returns `True` will burn tokens indefinitely if the only safeguard is the predicate itself.

**Where.** agent-framework: `max_iterations` short-circuits **before** the predicate runs (`_loop.py:657-664`, `LoopAgent.cs:196-200`). pydantic-ai: `consume_output_retry()` is the first thing the retry path does (`_agent_graph.py:162-179`); if the budget is exhausted, no retry prompt is built. langgraph: `_evaluate_retry` enforces `attempt > max_retries → RetryDecision(retry=False)` before retrying (`model_retry.py:382`). crewai: `for attempt in range(max_attempts)` is a Python `for` loop — the predicate (the loop body) cannot run more than `max_attempts` times by construction.

**Why it works.** Cap-first ordering guarantees that "predicate is buggy" cannot turn into "tokens burn forever." The alternative — running the predicate once and then checking the budget — leaves a window where a malicious or runaway predicate can chain one more iteration than the cap allows.

**When to copy.** Every time the loop has an external callable predicate. The cost is one extra integer comparison per iteration; the benefit is provable termination.

### P4. Typing exhaustion with a distinct exception or stop reason

**Problem.** "Loop ended because budget was exhausted" is operationally different from "loop ended because predicate said stop" or "loop ended because tool raised." Callers and tests need to distinguish these.

**Where.** pydantic-ai: `UnexpectedModelBehavior` carries `__cause__` from the original validation error (`exceptions.py:203-229`); test asserts the exact message format `tests/test_agent.py:10760` and `tests/test_tools.py:3636`. agent-framework: loop exhaustion logs `"Maximum iterations reached"` (`_tools.py:2664-2685`); Magentic exhaustion routes through `ResetAndReplanAsync` once and then terminates with a manager-authored message (`_magentic.py:1222-1258`). langgraph: handler failure fails the run rather than silently retrying (`tests/test_retry.py:2099`). letta: 13-member `StopReasonType` enum (`letta_stop_reason.py:9-49`); every retry/decide branch sets a typed reason; `RunStatus` is derived. openai-agents-sdk: `MaxTurnsExceeded` exception with `RunErrorDetails` snapshot (`exceptions.py:30-43, 56-63`).

**Why it works.** Operators can distinguish "stuck in a tool loop" from "ran out of turns legitimately." Tests can assert the exact reason. Dashboards can graph "retry exhaustion rate" without conflating it with "successful completion."

**When to copy.** Always. An untyped `return None` after `max_attempts` iterations is a footgun — it silently breaks "did the model recover or did we give up?" questions.

### P5. Fresh context snapshot for iteration N+1 (advanced)

**Problem.** After several iterations, accumulated context may itself be the cause of failure — old tool calls pollute the model's reasoning.

**Where.** agent-framework only: `fresh_context=True` snapshots the session once at the start of the run and restores a clone before each re-invocation (`_loop.py:415-417, 423-437, 506-510`, `LoopAgent.cs:151-156, 500-508`). The downside: only works for sessions that support serialization; hosted-conversations services that store messages by reference cannot cleanly clone (`LoopAgentOptions.cs:42-49`).

**Why it works.** Iteration N+1 sees the pristine baseline, not the accumulated transcript. This makes the reflection loop a true "given the same input, produce a better answer" cycle rather than a "given the conversation, change your mind" cycle.

**When to copy.** When correctness-of-iteration is more important than continuity-of-conversation. Most chat agents want continuity; reask-on-validation agents (e.g., graders, judges) want isolation.

**When risky.** When the model needs to remember earlier decisions. Snapshot-and-reset is destructive to multi-turn reasoning.

### P6. Retry-after-failure preserves the original response in history

**Problem.** "Please try again" without showing what was tried is a poor signal.

**Where.** pydantic-ai: every retry site appends the response (or partial result) to history *before* the `RetryPromptPart` (`_agent_graph.py:752-757, 828-835, 980-982`). langgraph: `ToolMessage` includes the original `ToolCallItem` (`turn_resolution.py:1949-1951`). agent-framework: the judge's `gap_analysis` is substituted into the next turn's user message alongside the original request (`AIJudgeLoopEvaluator.cs:173`). crewai: the guardrail reask prompt embeds the prior raw task output (`task.py:1310-1313`).

**Why it works.** The model sees exactly what it produced and exactly why it failed. It can revise the specific element rather than restart from scratch.

**When to copy.** Always. The cost is one extra message in history per retry; the benefit is far better recovery quality.

### P7. Force-final-answer on iteration cap (anti-pattern to think about)

**Problem.** What do you do when the retry budget is exhausted? Raise an exception and stop, or take one more shot?

**Where.** crewai is the canonical example: `handle_max_iterations_exceeded` appends a `force_final_answer` prompt that says "ignore all previous instructions, stop using any tools, and just return your absolute BEST Final answer" (`agent_utils.py:293-329`, `translations/en.json:46-47`). agent-framework's function-call loop forces `tool_choice="none"` on `max_iterations` exhaustion and emits a deterministic failsafe (`_tools.py:2664-2696`). pydantic-ai and openai-agents-sdk raise typed exceptions instead.

**Why it might work.** Best-effort answers are sometimes useful even when validation fails. The user can decide whether to retry the whole run.

**Why it is dangerous.** The directive to "ignore prior instructions" can produce a contradicting answer — the model may fabricate a clean-looking response that contradicts its in-progress plan. For agents that have made side-effecting tool calls, this is a hallucinated-success footgun.

**When to copy carefully.** Read-only agents where the worst case is a wrong answer. Never copy for side-effecting agents — raise an exception instead.

### P8. Per-index retry bookkeeping (advanced)

**Problem.** A list of N guardrails — should the budget be per guardrail or per attempt across all guardrails?

**Where.** crewai: `Task._guardrail_retry_counts: dict[int, int]` tracks each guardrail independently (`task.py:288-291`); tests confirm `tests/test_task_guardrails.py:440, 518, 554, 773-775`.

**Why it works.** One guardrail failing does not consume the budget for the others.

**When to copy.** When validation is composable (multiple orthogonal checks). Skip for monolithic validation.

### P9. Replay-safety for state-modifying retries

**Problem.** A transient 5xx retry on a tool call that mutates external state may re-run the side effect.

**Where.** openai-agents-sdk: `_with_replay_safe_approval` decision modifier requires `previous_response_id` or `conversation_id` requests to get explicit `replay_safety == "safe"` blessing (`retry.py:201-228, 412-419`).

**Why it works.** Defaulting to "not safe" means the runner cannot accidentally replay a side-effectful request after a network blip.

**When to copy.** Any framework that retries calls with side effects. The default must be conservative.

### P10. Idempotent approvals across compaction

**Problem.** After summarization evicts the approval request, the next approval payload would normally be processed as a new request, possibly re-running the tool.

**Where.** letta: `validate_persisted_tool_call_ids` scans recent tool-return messages for the matching `tool_call_id` and short-circuits to a keep-alive turn (`helpers.py:230-294`); integration test `tests/integration_test_human_in_the_loop.py:1456-1568`.

**Why it works.** Compaction is destructive to in-flight state; the agent needs a way to recognize "this approval is for a request that has already been handled" without losing the user's input.

**When to copy.** When the agent supports compaction *and* human-in-the-loop. Both features are individually useful; their interaction is where bugs hide.

## Key Differences

The seven sources diverge along five axes. Some are valid responses to different constraints; others are simply different design opinions.

### D1. Explicit reflection primitive vs. error-channel-only

agent-framework and pydantic-ai ship explicit reflection primitives (`LoopEvaluator`, `ModelRetry`). langgraph and openai-agents-sdk use the error channel only — the next model turn is the reflection. crewai has explicit *retry* loops but no critic primitive; the planner re-asks itself, but no separate judge scores the final answer. letta and openhands have no reflection primitive in this dimension's sense.

**Why it matters.** An explicit reflection primitive lets the framework expose retry counters, attach the verdict to the response, and prevent runaway loops structurally. An error-channel approach is simpler but trusts the same model that just failed to decide when to stop retrying.

**Where this is a real tradeoff.** Library frameworks that target both simple and advanced users (openai-agents-sdk, langgraph) prefer error-channel so the surface stays small. End-user platforms (pydantic-ai) prefer explicit primitives because the developer wants to know how many retries happened.

### D2. Per-tool vs. global budget granularity

pydantic-ai separates per-tool and global output budgets; a noisy tool does not consume the global counter. agent-framework separates loop iterations from progress-ledger parse retries; the two are independent. langgraph separates `RetryPolicy.max_attempts` from `recursion_limit`. openai-agents-sdk separates transport retries from turn cap. crewai separates three different retry counters (guardrail / converter / replan) but each counter is global to its loop.

**Why it matters.** Two-tier budgets prevent one bad layer from blocking recovery at another layer. Single-tier budgets are easier to reason about but mix independent failure modes.

### D3. LLM-as-judge for final answer vs. only on validation

agent-framework's `AIJudgeLoopEvaluator` is a true LLM-as-judge: it asks a structured `JudgeVerdict` after each iteration. crewai's `LLMGuardrail` is an LLM-judge for *validation*, not for *the final answer*; the open-source `HallucinationGuardrail` is a no-op placeholder. pydantic-ai deliberately does not include a critic model — users build it with capability hooks. langgraph, openai-agents-sdk, letta, and openhands have no critic primitive in the framework.

**Why it matters.** A judge that scores "did this answer the question?" catches successful-but-wrong outputs that pass surface validation. Without it, the framework can only say "this output passed validation" — not "this output is good."

### D4. Feedback goes through model's message channel vs. shadow reflection prompt

All sources use the model's own message channel for failure feedback (see P2). None use a shadow reflection prompt template. The closest thing to a separate prompt is agent-framework's Magentic progress ledger, which asks the model to fill out a structured JSON object (`is_request_satisfied`, `is_progress_being_made`, etc.) — but this is a structured-output call, not a separate prompt template.

### D5. Reask runs through middleware/agent decorator vs. graph topology

agent-framework (Python) implements reask as a middleware in the agent pipeline (`AgentLoopMiddleware`); langgraph implements it as graph topology (an `error_handler` node and `add_conditional_edges`). pydantic-ai implements it as an exception caught by the central graph node. crewai implements it as three separate `for attempt in range(...)` loops at three different layers.

**Why it matters.** Middleware is composable but order-sensitive. Graph topology is explicit but verbose. Exception-driven is uniform but requires every retry site to raise the exception. Multi-loop is pragmatic but spreads the surface.

## Tradeoffs

For each major design choice, what the framework gives up vs. what it gains.

### Explicit reflection primitive

- **Benefit.** Retry counters are first-class; the framework can attach the verdict to the response; the user can see *why* a loop stopped.
- **Cost.** Larger surface area; tests must cover both the success and exhaustion paths; abstractions may evolve (agent-framework's `LoopAgent` is marked `[Experimental]`).
- **Best fit.** Library frameworks targeting advanced users; platforms where the user wants observability into how many retries happened.
- **Failure mode.** A buggy evaluator can mask failures (agent-framework's "first ShouldReinvoke wins" rule means a buggy evaluator silently disables other evaluators, `LoopAgent.cs:34-37`).
- **Alternative.** Error-channel-only (simpler, smaller surface) — openai-agents-sdk.

### Two-tier retry budget

- **Benefit.** One noisy layer does not consume the other's budget.
- **Cost.** One extra integer per run; tests must prove isolation; user-facing API has more knobs.
- **Best fit.** Agents with both tools and structured output.
- **Failure mode.** Concurrent runs sharing the same retry state — pydantic-ai specifically guards against this with `OutputToolset.for_run_step` shallow-copy (`tool_manager.py:120-145`); without that guard, two runs could see `[0, 1, 2]` interleaved.

### Force-final-answer on iteration cap

- **Benefit.** Best-effort answers are sometimes useful.
- **Cost.** Directive prompt ("ignore all previous instructions") can produce hallucinated-success answers.
- **Best fit.** Read-only agents.
- **Failure mode.** Side-effecting tools whose partial state is now represented as a clean-looking final answer. crewai's `translations/en.json:47` is the canonical example.
- **Alternative.** Raise a typed exception (pydantic-ai, openai-agents-sdk) — caller decides what to do.

### LLM-as-judge on the final answer

- **Benefit.** Catches successful-but-wrong outputs.
- **Cost.** Doubles the LLM cost on every iteration; the judge is itself an LLM that can fail.
- **Best fit.** High-stakes agents where correctness matters more than cost.
- **Failure mode.** Judge returns "answered=True" prematurely to escape the loop.
- **Alternative.** Trust the next model turn (simpler) — most frameworks.

### Fresh-context reset on iteration N+1

- **Benefit.** Iteration N+1 sees the pristine baseline, not the accumulated transcript.
- **Cost.** Loses continuity; requires session-serialization support.
- **Best fit.** Reask-on-validation agents (graders, judges).
- **Failure mode.** Multi-turn reasoning the user wanted preserved is silently dropped.
- **Alternative.** Session-preserving (default in most frameworks).

### Per-index retry bookkeeping

- **Benefit.** One failing validation does not consume the budget for orthogonal validations.
- **Cost.** More bookkeeping; tests must cover per-index state.
- **Best fit.** Composable validation pipelines.
- **Alternative.** Single counter (simpler) — most frameworks.

### Best-effort delivery (openhands)

- **Benefit.** One bad profile/event/hook/message does not break the user's whole session.
- **Cost.** Real failures hide. A broken title endpoint looks identical to "title not generated yet."
- **Best fit.** Layered orchestrators where the user already has another signal for failure.
- **Failure mode.** Silent data loss — `_process_pending_messages` deletes queued messages after a single failed POST (`live_status_app_conversation_service.py:1994-2006`).

## Decision Guide

How to pick between the approach models for a new framework or a new agent.

### Pick explicit reflection primitive when:

- You are building a library where users want observability into retry counts.
- You need a critic that scores the *final* answer, not just validation.
- You can afford the surface area (tests, docs, version compatibility for evolving abstractions).
- Examples: agent-framework, pydantic-ai (as capability hooks).

### Pick exception-driven reflection when:

- You want every retry site to be uniform and the budget to be centralized.
- You expect users to add many domain-specific retry triggers (validation rules, capability hooks).
- You are willing to require users to wrap `ModelRetry` in the appropriate exception type at boundary sites.
- Example: pydantic-ai.

### Pick multi-loop bounded retries when:

- Validation is a first-class user concern (guardrails, structured output, plan steps).
- Each loop has a distinct evidence format and exit signal.
- You are willing to maintain three independent retry counters.
- Example: crewai.

### Pick error-channel feedback when:

- You want a small framework surface and trust the model to recover from errors.
- Your user base is primarily building simple agents.
- Examples: langgraph, openai-agents-sdk.

### Pick compaction/context-pressure when:

- Your primary failure mode is context-window overflow, not output quality.
- Your users have long-running conversations.
- You do not have budget for a critic model on every step.
- Example: letta.

### Pick externalized agent loop when:

- You want to separate orchestration concerns from LLM mechanics.
- Your agent runtime is a separate product with its own release cadence.
- Be aware that this makes it hard to add reflection/correction primitives from the orchestrator side.
- Example: openhands (open-source repo only; the actual loop is in `openhands-sdk`).

### Decision rules

1. **If you have structured output**, you need at minimum a per-output retry budget (pydantic-ai's `output_retries_used` or equivalent). Skipping this is the most common failure mode.
2. **If you have tools**, you need at minimum a per-tool retry budget. Either per-tool (pydantic-ai) or per-node (langgraph) works.
3. **If you have both**, you need two budgets. Do not unify them.
4. **If your model can produce silent-success-but-wrong answers** (e.g., free-form text generation with no schema), you need either an LLM-judge (agent-framework's `AIJudgeLoopEvaluator`) or accept that the framework cannot detect this class of failure.
5. **If your agents make side-effecting tool calls**, never use force-final-answer. Raise a typed exception on budget exhaustion and let the caller decide.
6. **If your framework supports sessions**, decide upfront whether reflection iterations should snapshot or accumulate. agent-framework offers `fresh_context=True` for the snapshot case; most frameworks accumulate.
7. **If your agent loop is long-running**, add a streaming service that synthesizes an error event when the inner loop dies without a terminal marker (letta's pattern). Without it, dropped streams look identical to "task is still running."

## Practical Tips

Concrete advice for applying these patterns.

### Patterns to copy

- **Two-tier retry budget.** Per-tool + global. Costs one extra integer; gains independent failure surfaces.
- **Typed exhaustion exception.** `UnexpectedModelBehavior` / `MaxTurnsExceeded` / `StopReasonType.max_steps` etc. Callers can branch on the reason; tests can assert the exact message.
- **Structured evidence in next prompt.** Validation error JSON, judge gap analysis, prior output excerpt. The model recovers better when it sees what it produced.
- **Short-circuit before predicate runs.** Cap-first ordering. Costs one integer comparison; provable termination.
- **Event hooks for observability.** crewai's `LLMGuardrailCompletedEvent` and `PlanReplanTriggeredEvent`; letta's stop-reason persistence. Operators need to see retries, not just final outcomes.
- **Concurrent run isolation.** pydantic-ai's `OutputToolset.for_run_step` shallow-copy. Two concurrent runs on the same agent must not share retry counters.
- **Streaming terminal-event guarantee.** letta's pattern: always emit a terminal marker, even on inner-loop failure.

### Patterns to avoid or delay until needed

- **Unbounded loop with `max_iterations=None`.** agent-framework supports it; do not opt in unless you have an external kill switch.
- **Force-final-answer on iteration cap.** Crewai's directive prompt is dangerous for side-effecting agents. Use only for read-only agents with explicit user consent.
- **Single global counter for tool + output retries.** They fail independently; unifying them creates cross-coupling.
- **Best-effort delivery that silently drops on failure.** openhands' `_process_pending_messages` pattern. Prefer requeue-on-failure with backoff.
- **`validation_error` is a free-form string.** pydantic-ai's structured `ErrorDetails` (loc/msg/type/input) gives the model more signal than "validation failed."
- **Recursive retry path.** crewai's `_execute_core` recursion in LiteAgent risks message-list growth. Prefer an iterative loop with explicit budget.

### Caution signs

The dimension area is becoming brittle, over-coupled, under-specified, hard to test, hard to operate, or hard to evolve when:

- **The retry budget is not validated at construction.** (agent-framework validates via `ArgumentOutOfRangeException` at `LoopAgent.cs:123`; pydantic-ai normalizes at `agent/__init__.py:368-518`.) If your framework accepts `max_iterations=-1` without complaint, expect surprise.
- **The retry counter is not exposed to the caller.** pydantic-ai's `ctx.retry` / `ctx.max_retries` / `ctx.last_attempt` are exemplary; agent-framework flags this as an open gap.
- **Exhaustion returns the in-progress failure chain.** agent-framework's `tool_choice="none"` failsafe is the correct pattern; returning the original `function_call` chain is hallucinated success.
- **Per-iteration logging is absent or buried.** agent-framework's `ILogger` + OpenTelemetry events are the model.
- **Reflection verdict is not attached to the response.** agent-framework explicitly flags this; pydantic-ai's `capture_run_messages()` is the partial workaround.
- **Concurrent runs share retry state.** pydantic-ai explicitly tests isolation at `tests/test_agent.py:10766-10808`; most frameworks do not.
- **Streaming drops mid-run without terminal marker.** Without it, callers cannot distinguish "still running" from "lost."
- **Force-final-answer on side-effecting agents.** A clean-looking answer that contradicts in-progress tool state is the worst kind of hallucinated success.

## Anti-Patterns / Caution Signs

Things to watch out for that indicate the dimension is being poorly served.

- **Hidden retry loops.** A retry that happens inside a library function the user cannot see or configure. (openhands' `_poll_for_title` is visible; openhands' failure paths inside `httpx` POSTs are not.)
- **Unbounded retries on user input.** crewai's `reasoning_handler.py:296` accepts `max_attempts=None` and loops until the planner emits `READY` — a footgun if the planner is stuck.
- **Silent message-list growth.** crewai's LiteAgent path appends the error string on every retry without compaction (`lite_agent.py:718-723`).
- **Recursive retry path.** crewai's `_execute_core` recursion (`lite_agent.py:726`); risks stack overflow and message bloat.
- **Single retry counter for tool + output.** Cross-couples two failure modes that should fail independently.
- **Shadow reflection prompt template** that hides the model's own message channel. (None of the seven sources do this; it is a thing to avoid if encountered elsewhere.)
- **`failure_error_function` bypassed when not using the decorator.** openai-agents-sdk's `FunctionTool` only honors `failure_error_function` when bound via the `@function_tool` decorator (`tool.py:540-568, 599-650`). Direct invocation silently bypasses the hook.
- **Cache invalidation interaction with retries.** langgraph's cache may return a cached payload without re-validating against a fresh retry policy (`_runner.py:1526-1539, 1780-1793`); `test_arun_with_retry_timeout_discards_stale_executor_writes` is the regression test.
- **Parser errors not counted against guardrail budget.** crewai's `handle_output_parser_exception` consumes the iteration counter but not `guardrail_retry_count` (`agent_utils.py:660-695`); a successful task with hundreds of parser retries could exhaust `max_iter` while looking fine to a guardrail-based monitor.
- **No per-attempt observability.** agent-framework has this; openai-agents-sdk plumbs `failed_retry_attempts_out` but it is unclear where it surfaces (`model_retry.py:618`, `run_loop.py:1458`); langgraph's `_AttemptEvent` is private and used only by `langgraph-server` (no OSS consumer found).

## Notable Absences

What is missing from the field.

- **LLM-as-judge on the final answer.** Only agent-framework ships a first-class judge (`AIJudgeLoopEvaluator`). crewai's `HallucinationGuardrail` is a no-op in OSS (`tasks/hallucination_guardrail.py:73-77`). pydantic-ai leaves it to user capability hooks. langgraph retired its reflection/reflexion examples (`examples/reflection/reflection.ipynb:9-13` redirects to consolidated docs). openai-agents-sdk has no critic primitive. letta and openhands do not have one.
- **Per-re-ask budget on top of `recursion_limit`.** langgraph's only outer bound is the coarse `recursion_limit` (`_internal/_config.py:32`); there is no per-re-ask budget for "tool failed N times in this run." Application code must compose `RemainingSteps` to bound that cycle.
- **Reflection verdict attached to the final response.** When agent-framework's loop terminates, only the last model message is returned; the `JudgeVerdict` that caused termination is dropped. pydantic-ai's `capture_run_messages()` is a partial workaround but not the same thing.
- **Documented reflection playbook.** CrewAI's docs at `docs/edge/en/concepts/...` were not inspected for this study, but the source tree has no documented playbook — operators must read code to find the budgets.
- **Multi-model reflection on persistent failure.** pydantic-ai's `FallbackModel` is transport-only (it kicks in on transport errors, not on validation retries). A retry that always fails on one model will keep failing — no built-in escalation to a different model for reask.
- **Test that a real LLM recovers from a re-ask.** openai-agents-sdk's test `tests/test_agent_runner.py:4458-4489` asserts the failure message reaches the model but uses a `FakeModel` script. None of the seven frameworks has a real-LLM recovery test in the cited evidence.

## Per-Source Notes

A condensed view of each source's contribution. The detailed evidence is in the per-source analysis files; this is the takeaway.

### agent-framework (8/10)

Dedicated `LoopEvaluator` middleware (Python) / `LoopAgent` (C#) with optional `AIJudgeLoopEvaluator`. Magentic One adds a stall counter and a JSON progress-ledger with its own retry budget (`MagenticManager.cs:70-105`). Function-invocation layer has a separate budget: consecutive tool errors are counted and force `tool_choice="none"` after the cap. Strongest points: structured `JudgeVerdict` substituted into next turn; `fresh_context` snapshot reset; multi-language parity. Weakest: `LoopAgent` is `[Experimental]`; verdict not attached to returned `AgentResponse`; no logged counter for unbounded loops.

### crewai (7/10)

Three independent bounded retry loops (guardrail, converter, plan-and-act replan) plus auxiliary recovery paths (max-iteration force-final-answer, context overflow, parser errors, execution exceptions). Per-index retry bookkeeping (`_guardrail_retry_counts`). Event-bus hooks (`LLMGuardrailStartedEvent`, `PlanReplanTriggeredEvent`). `force_final_answer` directive prompt for max-iteration exhaustion. Strongest: typed exceptions on every exhaustion path; rich event telemetry. Weakest: no LLM-as-judge on final answer; LiteAgent recursive path risks message-list growth; force-final-answer can contradict prior plan.

### langgraph (8/10)

Declarative `RetryPolicy` NamedTuple combined with per-node `error_handler`. Tool-error feedback via `ToolMessage(status="error")`. `recursion_limit` is the only outer bound; `GraphRecursionError` on exhaustion. `NodeTimeoutError` deliberately not `OSError` so the default `RetryPolicy` retries it. Cancellation triage via `asyncio.Task.cancelling()` distinguishes user-initiated from framework-initiated. `NodeError` survives checkpoint resume. Strongest: handler composition (`Command(update={...}, goto=...)`); retry+handler composition tested. Weakest: no per-re-ask budget on top of `recursion_limit`; no critic primitive; reflection examples retired.

### letta (6/10)

Compaction-as-reflection on context overflow: `SUMMARIZATION_TRIGGER_MULTIPLIER=0.9` triggers summarizer; `max_summarizer_retries=3` retries with compaction. Tool-rule violations produce `[ToolConstraintError]` heartbeat injected into next turn. `StopReasonType` enum (13 members); `RunStatus` derived. Streaming service synthesizes `event: error` so dropped streams are not mistaken for success. Idempotent approvals across compaction. Strongest: stop-reason taxonomy; streaming terminal guarantee. Weakest: three coexisting loops with overlapping-but-distinct semantics (legacy vs V2/V3); no critic primitive; legacy `_get_ai_reply` uses blocking `time.sleep`.

### openai-agents-sdk (6/10)

Error-channel feedback only via `function_call_output` items. `RetryPolicy` with `RetryDecision(retry, _hard_veto, _replay_safe)` modifiers. `DEFAULT_MAX_TURNS=10`. `MaxTurnsExceeded` exception with `RunErrorDetails` snapshot; opt-in `error_handlers["max_turns"]` callback. `reset_tool_choice=True` as anti-loop guard. Per-tool/per-input/output guardrails with `AllowBehavior` / `RejectContentBehavior` / `RaiseExceptionBehavior` discriminator. Strongest: replay-safety annotations; transport retry and validation retry kept separate. Weakest: no reflection prompt template; default `tool_not_found_behavior="raise_error"` short-circuits recovery; per-tool call has no per-attempt counter.

### openhands (2/10)

V1 orchestrator only; agent loop is external in `openhands-sdk` / `openhands-agent-server` PyPI packages. Within `openhands/`, only `SetTitleCallbackProcessor._poll_for_title` (4 polls × 3s) is a self-correction-shaped loop, and it is a deferred side effect, not agent reflection. Validation is overwhelmingly silent (best-effort delivery, drop-on-parse, `_skip_invalid_profiles`). `EventCallbackStatus.ERROR` enum value exists but is never assigned. `_classify_error_type` is post-mortem observability, not a corrective trigger. Strongest: bounded polling for sandbox startup; HTTP status codes as a tiny failure state machine. Weakest: no reflection prompt; no critic; no per-step retry decorator; best-effort delivery hides failures.

### pydantic-ai (9/10)

Exception-driven reflection: single `ModelRetry` exception unifies every retry site (tools, validators, hooks, unknown tools, empty responses, tool timeouts). Two-tier budget: per-tool (`RunContext.retries[name]`) plus global (`GraphAgentState.output_retries_used`). Structured evidence: `RetryPromptPart` carries `list[ErrorDetails]` (loc/msg/type/input). `UnexpectedModelBehavior` carries `__cause__` on exhaustion. `last_attempt` signal lets tools/validators switch to fallback on the final try. Capability hooks as composition points for user overrides. HTTP retries are a separate concern (`TenacityTransport`). Strongest: structured evidence; capability-hook composition; concurrent-run isolation; `capture_run_messages` for post-mortem. Weakest: no formal critic model; default budget=1 may be insufficient; empty-response blind retry can burn budget.

## Executive Summary

Across eight frameworks spanning agent runtimes, workflow orchestration, and a control-plane proxy, **the runtime owns control flow in every case** — the question is only *how* that ownership is encoded. Sources cluster into three approach families:

1. **Typed-next-step runtimes** that build an explicit transition value before each outer-loop iteration — most prominently OpenAI Agents SDK's `NextStepHandoff | NextStepFinalOutput | NextStepRunAgain | NextStepInterruption` discriminated union (`src/agents/run_internal/run_steps.py:154-218`), agent-framework's `FunctionRequestResult.action ∈ {"return", "continue", "stop"}` (`python/packages/core/agent_framework/_tools.py:2209`), Pydantic AI's static four-node graph (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2130-2139`), and letta's `_decide_continuation` returning a typed `(continue, reason, stop_reason)` tuple (`letta/agents/letta_agent_v3.py:1967-2036`).
2. **Graph runtimes with cooperative runtime overrides** — LangGraph's `PregelLoop.tick()` with `RunControl.request_drain()` + `recursion_limit` + `interrupt_before/after` (`libs/langgraph/langgraph/pregel/_loop.py:592-674`, `runtime.py:79-104`), and Temporal's typed `Transition[S, SM, E]` values with `Sources`/`Destination` enforced via `Apply` (`chasm/statemachine.go:14-79`, `service/history/hsm/sm.go:18-66`).
3. **DSL/state-machine runtimes with literal-typed routers** — CrewAI's `AgentExecutor(Flow[AgentExecutorState])` with `@router` methods returning `Literal[...]` labels (`lib/crewai/src/crewai/experimental/agent_executor.py:608-2123`), and agent-framework's parallel workflow superstep loop driven by typed `WorkflowEvent`s.

The **outlier** is OpenHands, which is a control-plane proxy: it pushes configuration into an external agent-server rather than running a loop (`openhands/app_server/app_conversation/app_conversation_service_base.py:614-652`), so its control authority is bounded by what the SDK exposes over HTTP. **Temporal is also an outlier by purpose**: it is a workflow-orchestration server, not an agent harness; its "user code" is the SDK worker, and its state-machine authority is the most rigorous in the survey (atomic state-and-task commit, versioned transitions, `ErrInvalidTransition` / `ErrStaleReference` rejection).

Convergence: every framework (except OpenHands' boundary) gives the runtime explicit authority to *stop*, *rewrite*, or *pause* the model's next action — through tool-choice reset (agent-framework, OpenAI Agents SDK), forced final non-tool calls, `tool_choice="none"` overrides, recursive guards, approval gating, parallel-tool truncation, and typed error classes. Every framework is testable without an LLM via mock chat clients, `FakeModel` substitutes, or in-memory engines (`chasmtest.Engine`, `MockChatClient`, `TestModel`). Transitions are explicit in seven of eight sources; only CrewAI's legacy `CrewAgentExecutor` and `LiteAgent` show the *scattered* alternative within an otherwise-explicit codebase.

The single most important takeaway: **runtime-owned control flow is the dominant pattern, and the variation is in the type vocabulary used to encode transitions** — string literals, `Literal` types, dataclass unions, typed `Transition` values, and `Enum` classes all coexist, with a clear trend toward typed payloads that can carry data (handoff target, approval items, final output) rather than just labels.

## Core Thesis

**The runtime owns control flow in every surveyed framework, and the strongest implementations encode transitions as typed values with payload-carrying variants, runtime-only transition functions, and explicit override primitives that the model cannot bypass.** Sources diverge on the type vocabulary used to encode transitions, on whether the loop is graph-based or imperative, and on whether transitions are string-keyed or struct-keyed — but they agree that:

- The model may pick *what* (which tool, which next speaker, which text) but never *whether to continue, when to stop, or how the loop is dispatched*.
- Runtime override authority is layered: budget caps, approval gates, tool-choice resets, error handlers, and durability primitives all act on the model from outside.
- Control flow must be testable without an LLM — and the strongest frameworks expose a public test surface (LangGraph's `prepare_next_tasks`, agent-framework's `MockChatClient`, OpenAI Agents SDK's hand-built `ProcessedResponse`, Temporal's `chasmtest.Engine`, Pydantic AI's `TestModel`) that does not require a model at all.

The "control-flow ownership" question is therefore not *whether the runtime owns it* but *how well the type vocabulary documents and enforces that ownership* — and the 9/10 sources show that a small, typed, payload-carrying transition vocabulary with an explicit `apply`/transition function is the strongest shape.

## Approach Models

Eight sources produce five recognizable control-flow models. Naming them surfaces the convergence that the per-source ratings hide.

### Model A — Typed-next-step imperative loop

Sources: **openai-agents-sdk**, **agent-framework** (inner chat-client loop), **letta**.

A single `while` loop drives iterations. At each iteration the runtime builds a typed next-step value, then a small dispatch (one `isinstance` chain or a single transition function) reads the value and acts. The model emits structured output items; the runtime classifies them and decides.

- OpenAI Agents SDK: `SingleStepResult.next_step` is a `NextStepHandoff | NextStepFinalOutput | NextStepRunAgain | NextStepInterruption` dataclass union (`src/agents/run_internal/run_steps.py:154-218`). The outer `while True` reads the discriminant and dispatches in typed branches (`src/agents/run.py:1367-1497`). Streaming twin duplicates the dispatch intentionally (`src/agents/run_internal/run_loop.py:1065-1164`, documented in `AGENTS.md:55-67`).
- agent-framework (Python `FunctionInvocationLayer`): the inner step returns `FunctionRequestResult.action: Literal["return", "continue", "stop"]` (`python/packages/core/agent_framework/_tools.py:2209`), with action rewrites on approval flow (`_tools.py:2374-2377`) and forced non-tool final call after budget exhaustion (`_tools.py:2664-2696`). The .NET mirror is `LoopEvaluation.Stop()/Continue()` (`LoopEvaluation.cs:46-79`) plus `RunStatus` enum.
- letta: `_decide_continuation` returns `(continue, reason, stop_reason)` (`letta/agents/letta_agent_v3.py:1967-2036`); `StopReasonType` (`letta/schemas/letta_stop_reason.py:9-22`) is the typed vocabulary with `.run_status` mapping to `RunStatus`.

**Why it works:** a small, payload-carrying type vocabulary makes the outer loop a 30-line dispatch instead of a 700-line state machine. Adding a new transition means adding a new variant to the union — a *type error* for every consumer that does not handle it, not a runtime miss.

**Where it stutters:** dispatch duplication across streaming / non-streaming paths (OpenAI Agents SDK); string literals scattered through the file instead of a single enum (agent-framework); near-duplicate drivers across v1/v2/v3 (letta).

### Model B — Graph runtime with BSP supersteps

Sources: **langgraph**, **pydantic-ai** (v1 + v2 beta), **agent-framework** (workflow superstep + Pregel-style).

Control flow is encoded as a graph; the runtime walks the graph in BSP supersteps. Static edges compile to direct `ChannelWrite`s; conditional edges compile to `BranchSpec`s; dynamic routing (`Send` / `Command(goto=)`) is deferred into a topic channel (`__pregel_tasks`) consumed at the next tick.

- LangGraph: `PregelLoop.tick()` → `prepare_next_tasks()` (`libs/langgraph/langgraph/pregel/_loop.py:592-674`, `_algo.py:392-513`). Channel-version deltas drive PULL triggers (`_algo.py:1260-1277`); TASKS topic drives PUSH triggers (`_algo.py:441-466`). Runtime override: `RunControl.request_drain()` (`_loop.py:650-652`), `recursion_limit` → `GraphRecursionError` (`_loop.py:600-602`), `interrupt_before/after` (`_algo.py:155-185`).
- agent-framework workflows: `Runner.run_until_convergence` is `while self._iteration < self._max_iterations:` draining messages per superstep (`python/packages/core/agent_framework/_workflows/_runner.py:99-156`); .NET twin is `StreamingRunEventStream.RunLoopAsync` (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:87-140`).
- Pydantic AI: four-node static graph `UserPromptNode → ModelRequestNode → CallToolsNode → SetFinalResult | End[FinalResult]` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2110-2139`); v2 graph builder adds typed `Step`/`Join`/`Decision`/`Fork` primitives (`pydantic_graph/pydantic_graph/graph_builder.py:382-396`).

**Why it works:** the graph schema is a type-checkable declaration; static edges are inspectable and visualisable; BSP supersteps make checkpointing and replay trivial. Every transition has a clear superstep boundary.

**Where it stutters:** graph models can concentrate state-machine logic in a single dispatch function (`CallToolsNode._run_stream` in Pydantic AI, `prepare_next_tasks` in LangGraph) that becomes the heaviest function in the codebase. Routing logic can be duplicated in two layers (LangGraph's `map_command` vs `_control_branch`).

### Model C — State-machine with typed `Transition` values

Source: **temporal**.

Every state machine exposes `Transition[S, SM, E]` — a value type with `Sources []S`, `Destination S`, and `apply` function. The only legal mutation path is `Tree.MachineTransition` (HSM) or `Transition.Apply` (ChASM); `Sources` membership is checked before `apply` runs; state-and-task commit is atomic (transactional outbox).

- ChASM `Transition.Apply` (`chasm/statemachine.go:52-78`) verifies `Sources`, runs user `apply`, then sets `Destination`. Tasks emitted via `ctx.AddTask` are committed atomically (`docs/architecture/chasm.md:233-238`).
- HSM `Tree.MachineTransition` (`service/history/hsm/tree.go:579-629`) increments `TransitionCount` and supports rollback on error.
- V1 `workflow_task_state_machine.go` encodes the workflow-task lifecycle through `ApplyWorkflowTaskScheduledEvent`, `ApplyWorkflowTaskStartedEvent`, etc. (`service/history/workflow/workflow_task_state_machine.go:62,120,169,247,256`).

**Why it works:** the type system enforces "what can follow what" at construction time; the engine is the only authority that can reject a transition (`ErrInvalidTransition`, `ErrStaleReference`, `ErrWorkflowZombie`); atomicity means a malicious or buggy transition cannot drop tasks silently.

**Where it stutters:** two engines coexist (V1 + ChASM); V1 path's `mutable_state_impl` and executor pair (`transfer_queue_active_task_executor.go` / `timer_queue_active_task_executor.go`) split transition responsibility across files; HSM callback already deprecated (`components/callbacks/statemachine.go:36-39`).

### Model D — Flow DSL with `Literal`-typed routers

Source: **crewai**.

A `Flow[State]` runtime dispatches on string labels. Each `@router` method returns a `Literal[...]` that names the next method; the runtime drains routers sequentially then runs listeners in parallel via `asyncio.gather` (`lib/crewai/src/crewai/flow/runtime/__init__.py:2984-3110`).

- `AgentExecutor(Flow[AgentExecutorState])` is the modern agent executor (`lib/crewai/src/crewai/experimental/agent_executor.py:164-167`). Routers return `Literal["step_observed_low", "step_observed_medium", "step_observed_high"]` (`:608-673`), `Literal["force_final_answer", "continue_reasoning", "continue_reasoning_native"]` (`:2112-2123`), `Literal["has_todos", "all_todos_complete"]` (`:976-1016`).
- Static validation: `_get_router_return_events` reads the `Literal[...]` from method return annotations (`lib/crewai/src/crewai/flow/dsl/_router.py:86-88`).

**Why it works:** a Flow user can express 30+ typed transitions without writing a state machine; the DSL validates router emit sets at definition time. The plan-and-execute overlay (`generate_plan` → `StepExecutor` per todo → `PlannerObserver.observe` → `_synthesize_final_answer_from_todos`) is a structural commitment to a well-known pattern (`arxiv 2503.09572`).

**Where it stutters:** transitions are routed through string labels, so reading a transition means following the string across files. The legacy `CrewAgentExecutor` (`lib/crewai/src/crewai/agents/crew_agent_executor.py:340-468`) and `LiteAgent` (`lib/crewai/src/crewai/lite_agent.py:860-972`) are hand-rolled ReAct loops with scattered transitions — a real but unfixed inconsistency.

### Model E — Control-plane proxy

Source: **openhands**.

The repo is the *control plane*, not the agent runtime. Control flow is bounded by the HTTP boundary to the external `openhands-agent-server` (pinned `1.29.0` at `pyproject.toml:60-61`). The app-server pushes three runtime override surfaces into the agent-server: `ConversationSettings` (max_iterations / confirmation_mode / security_analyzer), `ConfirmationPolicyBase` (`AlwaysConfirm` / `ConfirmRisky` / `NeverConfirm`), and `LLMSecurityAnalyzer` (`app_conversation_service_base.py:614-652`). The lifecycle state machine is `AppConversationStartTaskStatus` (`app_conversation_models.py:262-271`).

**Why it works:** the boundary is clean — every action crosses it as an authenticated HTTP request — so the app-server's own control surface is small and testable (`tests/unit/app_server/test_live_status_app_conversation_service.py:66-867+`). The agent-server handles loop, stuck detection, tool dispatch.

**Where it stutters:** the actual loop is opaque from this repo; `max_iterations` enforcement happens inside the SDK and is not observable from here. Webhook-driven termination means the app-server relies on the agent-server to push status updates; a stopped webhook stops the app-server's view of the world.

## Pattern Catalog

These patterns appear in multiple sources or in unusual clarity in one. Each entry explains what problem the pattern solves, which sources demonstrate it, when to copy it, and when it is overkill.

### Pattern 1 — Typed transition vocabulary with payload-carrying variants

**Problem:** a free-form "next step" string forces every consumer to pattern-match on labels and creates the risk that a new transition silently breaks old dispatchers.

**Sources:** openai-agents-sdk (dataclass union `NextStep*`), agent-framework (`Literal["return", "continue", "stop"]`), letta (`StopReasonType` × `RunStatus`), temporal (`Transition[S, SM, E]` value type).

**Why it works:** the type system enforces exhaustiveness. In openai-agents-sdk, the outer loop ends with `else: raise AgentsException` for unknown variants (`src/agents/run.py:1494-1497`). In Temporal, `Apply` returns `ErrInvalidTransition` if the source state is not in the declared `Sources` set (`chasm/statemachine.go:70-78`).

**Copy when:** you have a small (<10) closed set of transition kinds that must be exhaustive.

**Avoid when:** you need N-way branching with dynamic labels (CrewAI's Flow DSL handles that case better).

### Pattern 2 — Runtime-only transition function

**Problem:** scattering `if next == "continue": ...` branches across the codebase makes transitions hard to audit.

**Sources:** letta's `_decide_continuation` (`letta/agents/letta_agent_v3.py:1967-2036`), OpenAI Agents SDK's `check_for_final_output_from_tools` (`src/agents/run_internal/turn_resolution.py:594-626`), LangGraph's `prepare_next_tasks` (`libs/langgraph/langgraph/pregel/_algo.py:392-513`), agent-framework's `_process_function_requests` (`python/packages/core/agent_framework/_tools.py:2209`).

**Why it works:** a single named function that maps `(state, model_output) → transition` is auditable in isolation and is the natural place to enforce runtime override authority.

**Copy when:** you have a non-trivial decision (more than two outcomes) per step.

**Avoid when:** the decision is genuinely trivial (e.g., binary continue/stop) — the overhead of a named function is not justified.

### Pattern 3 — Runtime authority to rewrite model choice

**Problem:** a model that loops on `tool_choice="required"` or emits an unknown tool name can hijack the loop.

**Sources:** agent-framework rewrites `tool_choice="none"` after budget exhaustion (`python/packages/core/agent_framework/_tools.py:2592,2635,2673,2844`), resets `"required"` after one iteration (`_tools.py:2647-2652`); OpenAI Agents SDK's `maybe_reset_tool_choice` (`src/agents/run_internal/run_loop.py:1346,1830`); letta's parallel-tool truncation (`letta/agents/letta_agent_v3.py:1337-1342`), Anthropic parallel override (`letta/agents/letta_agent_v3.py:1119-1149`), and prefilled-arg merging (`letta/agents/helpers.py:465-493`); Pydantic AI's `wrap_model_request` / `after_model_request` middleware (`pydantic_ai_slim/pydantic_ai/capabilities/abstract.py:486-536`).

**Why it works:** the runtime can short-circuit the model's choice without re-asking the model. letta's `ToolRulesSolver` is the most extreme example — it intersects child/parent/conditional/max-count rules and synthesizes a `ToolConstraintError` result rather than running the call (`letta/agents/letta_agent_v3.py:1780, 1826`).

**Copy when:** you have a tool registry that can answer "is this call legal?" faster than the model can.

**Avoid when:** the rewrite logic becomes its own DSL — at that point it deserves a first-class `Policy` class (letta's `ToolRulesSolver` is the boundary case).

### Pattern 4 — Human-in-the-loop as a first-class loop state

**Problem:** approvals are usually implemented as exceptions or external hooks, which makes them hard to reason about and easy to bypass.

**Sources:** agent-framework's `MiddlewareTermination` exception (`python/packages/core/agent_framework/_middleware.py:72-79`) and `LoopAgent` halting on pending approval (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:189-193`); OpenAI Agents SDK's `NextStepInterruption` (`src/agents/run_internal/run_steps.py:169-174`) and tri-state `RunContextWrapper._approvals` (`src/agents/run_context.py:29-211`); letta's `requires_approval` pause with HTTP 409 (`letta/agents/letta_agent_v3.py:1709`, `letta/server/rest_api/routers/v1/agents.py:1801-1806`); Pydantic AI's `ApprovalRequired` exception (`pydantic_ai_slim/pydantic_ai/exceptions.py:116-178`); LangGraph's `interrupt_before`/`interrupt_after` and `langgraph.types.interrupt()` (`libs/langgraph/langgraph/pregel/_loop.py:659-664`, `types.py:811-934`); OpenHands' `ConfirmRisky` / `AlwaysConfirm` policies (`openhands/app_server/app_conversation/app_conversation_service_base.py:641-652`).

**Why it works:** pause is encoded as a transition variant (NextStepInterruption, MiddlewareTermination, ApprovalRequired), not as an exception the loop must catch and translate. Resume is encoded as state — OpenAI Agents SDK's `RunState._current_step: NextStepInterruption | None` (`src/agents/run_state.py:254-255`) with versioned schema `CURRENT_SCHEMA_VERSION = "1.11"` (`src/agents/run_state.py:131-149`).

**Copy when:** you need durable pause/resume — every source above pairs the pause primitive with a serialisable state.

**Avoid when:** your pause is purely cosmetic (a `time.sleep` while waiting for an event) — first-class primitives pay off when the resume path needs to survive a process restart.

### Pattern 5 — Capability / middleware chain around the loop

**Problem:** cross-cutting concerns (retries, telemetry, moderation, caching) want to intercept every step but should not be hard-coded into the loop.

**Sources:** Pydantic AI's `Capability` chain with 17+ typed hook points (`pydantic_ai_slim/pydantic_ai/capabilities/abstract.py:405-947`); agent-framework's three middleware layers (`FunctionMiddlewarePipeline`, `AgentMiddlewarePipeline`, `ChatMiddlewarePipeline` at `_middleware.py:876-1100`).

**Why it works:** each capability declares which stage(s) it wraps (`wrap_run`, `wrap_node_run`, `wrap_model_request`, etc.); the `CombinedCapability` chain builds the middleware at composition time (`pydantic_ai_slim/pydantic_ai/capabilities/combined.py:247-401`). Adding a hook is a new class, not a loop edit.

**Copy when:** you have a closed set of stages and want to let users add cross-cutting behaviour.

**Avoid when:** you have <3 cross-cutting concerns — the middleware overhead is not worth it.

### Pattern 6 — LLM-free test surface for control flow

**Problem:** if control flow can only be exercised through a real model, you cannot test the runtime's override authority under failure.

**Sources:** agent-framework's `MockChatClient` / `MockBaseChatClient` (`python/packages/core/tests/core/conftest.py:219-260`) and dedicated `tests/core/test_function_invocation_logic.py:1000-1399`; OpenAI Agents SDK's `FakeModel` (`tests/fake_model.py`) and `tests/test_run_step_execution.py:1-3552`; Pydantic AI's `TestModel` (`pydantic_ai_slim/pydantic_ai/models/test.py:62`) and `FunctionModel` (`pydantic_ai_slim/pydantic_ai/models/function.py:46`); LangGraph's `prepare_next_tasks(for_execution=False)` overload (`libs/langgraph/langgraph/pregel/_algo.py:348-389`, `tests/test_algo.py:6-43`); Temporal's `chasmtest.Engine` (`chasm/chasmtest/test_engine.go:82-323`); CrewAI's `_build_executor` with `model_construct` (`lib/crewai/tests/agents/test_agent_executor.py:23-54`); letta's `ToolRulesSolver` exhaustive unit suite (`tests/test_tool_rule_solver.py:34-1133`); OpenHands' mocked agent-server boundary (`tests/unit/app_server/test_live_status_app_conversation_service.py:66-867+`).

**Why it works:** the test surface is the proof that the runtime owns control flow. If you cannot drive the loop without a model, the loop is model-owned by construction. Every 8/10+ source exposes this surface explicitly.

**Copy when:** always. If your framework cannot be tested without a model, you cannot test the runtime's override authority under model failure.

### Pattern 7 — Force a final non-tool call after budget exhaustion

**Problem:** a model that exhausts `max_iterations` mid-batch may leave `function_call` items without matching `function_call_result` items, producing an orphan.

**Sources:** agent-framework's forced `tool_choice="none"` final call after budget exhaustion (`python/packages/core/agent_framework/_tools.py:2664-2696`); OpenAI Agents SDK's `MaxTurnsExceeded` with optional `error_handlers["max_turns"]` override (`src/agents/run.py:1057-1144`, `src/agents/run_error_handlers.py:53`).

**Why it works:** the runtime cleans up the model's bookkeeping instead of relying on the next model call to recover.

**Copy when:** you have a budget that may fire mid-batch (i.e., mid parallel tool execution).

**Avoid when:** every batch is a single tool call — the cap will land between iterations naturally.

### Pattern 8 — Per-step / per-iteration durability primitives

**Problem:** a run that crashes mid-iteration loses context that is expensive to rebuild.

**Sources:** Temporal's atomic state-and-task commit (`docs/architecture/chasm.md:233-238`); OpenAI Agents SDK's versioned `RunState` snapshot (`src/agents/run_state.py:131-149, 184-321`); agent-framework's `PerServiceCallChatHistoryPersistingChatClient` (`dotnet/src/Microsoft.Agents.AI/ChatClient/PerServiceCallChatHistoryPersistingChatClient.cs:200`); LangGraph's `ChannelWrite` + checkpointing via `GetNextVersion` (`libs/langgraph/langgraph/pregel/_algo.py:1260-1277`); Pydantic AI's `BaseStatePersistence` (`pydantic_graph/pydantic_graph/persistence/__init__.py:113-136`) with `durable_exec/{prefect,temporal,dbos}/` adapters (`pydantic_ai_slim/pydantic_ai/durable_exec/temporal/_function_toolset.py:43-69`); letta's per-step checkpoint pattern (`letta/schemas/step.py:68-74`, `letta/agents/letta_agent_v2.py:941-1003`).

**Why it works:** durability is at the step boundary, not the run boundary. A crashed tool execution still leaves a recoverable transcript.

**Copy when:** tool calls can take minutes and the user might disconnect.

**Avoid when:** tool calls are sub-second and idempotent — the persistence overhead is not justified.

### Pattern 9 — Runtime-only re-derivation of next state from worker output

**Problem:** a worker that returns "what I did" should not be able to invent "what should happen next."

**Sources:** Temporal's V1 path (`mutable_state_impl` re-derives the next state from `RespondWorkflowTaskCompletedRequest`); OpenAI Agents SDK's `process_model_response` (`src/agents/run_internal/turn_resolution.py:1551-1999`) classifies raw model output into `ProcessedResponse`; letta's `_handle_ai_response` (`letta/agents/letta_agent_v3.py:1345`) parses model output before runtime decides continuation; Pydantic AI's `process_tool_calls` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1537-1860`) partitions tool calls by kind before runtime routing.

**Why it works:** the runtime reads a structured description of what happened, *then* decides what to do. The model cannot lie about its outcome — the runtime re-derives.

**Copy when:** the worker / model can produce structured output that the runtime can validate.

**Avoid when:** the output is free-form text and parsing is unreliable.

### Pattern 10 — String-keyed transition vocabulary

**Problem:** sometimes you want N-way branching with dynamic labels and do not want to enumerate every variant in advance.

**Sources:** CrewAI's Flow DSL with `Literal["..."]` return types (`lib/crewai/src/crewai/experimental/agent_executor.py:608-2123`); agent-framework's `_process_function_requests` returning `"return" / "continue" / "stop"` (`python/packages/core/agent_framework/_tools.py:2209`).

**Why it works:** the DSL validates router emit sets at definition time (CrewAI's `_get_router_return_events` at `lib/crewai/src/crewai/flow/dsl/_router.py:86-88`); the small vocabulary in agent-framework's case is auditable.

**When to copy:** CrewAI's pattern when you need N-way branching with parallel listeners (`or_(...)` and `and_(...)` combinators at `_conditions.py:22-27`).

**When it is risky:** scattering string literals across multiple files (agent-framework's `"return" / "continue" / "stop"` literals at `_tools.py:2209,2265,2273,2337,2627,2632,2645,2802,2806`) creates the maintenance hazard that a typed enum would solve.

## Key Differences

### Where sources converge

- **Runtime owns the loop.** Every source except OpenHands has a clear runtime owner — letta's `for i in range(max_steps)`, OpenAI Agents SDK's `while True`, agent-framework's three layered loops, LangGraph's `PregelLoop.tick()`, Pydantic AI's `Graph.run`, CrewAI's `Flow._execute_listeners`, Temporal's `MachineTransition`/`Apply`, and OpenHands' control-plane proxy.
- **Runtime has multi-layer override authority.** Tool-choice resets, parallel-tool truncation, prefilled-arg merging, approval gating, recursion limits, drain signals, and typed error classes all let the runtime reject or rewrite the model's choice. The LLM cannot bypass these.
- **Transitions are typed.** The vocabulary varies (`Literal[...]`, dataclass union, enum, `Transition[S, SM, E]` value type, free-text strings in OpenHands), but the trend is toward typed payloads that can carry data.
- **Control flow is testable without an LLM.** Every source exposes a non-LLM test surface — `MockChatClient`, `FakeModel`, `TestModel`, `chasmtest.Engine`, `_build_executor`, `RunControl`-driven tests, in-memory scheduler tests.
- **Pause/resume is a first-class state.** OpenAI Agents SDK's `RunState._current_step`, agent-framework's `MiddlewareTermination` + `ExternalRequest/Response`, letta's `requires_approval` + idempotent retry, LangGraph's `interrupt()`, Pydantic AI's `ApprovalRequired`, OpenHands' `AlwaysConfirm` all encode pause as a state, not an exception.

### Where sources diverge

- **Loop shape.** Imperative `while` (OpenAI Agents SDK, agent-framework inner loop, letta, OpenHands) vs BSP graph (LangGraph, agent-framework workflows, Pydantic AI, Temporal ChASM) vs Flow DSL (CrewAI). The choice tracks product shape: imperative loops suit single-agent runtimes; BSP graphs suit multi-agent orchestration; Flow DSLs suit user-extensible graphs.
- **Type vocabulary for transitions.** String literals (`"return" / "continue" / "stop"`), `Literal[...]` (`Literal["step_observed_low", "step_observed_medium", "step_observed_high"]`), dataclass unions (`NextStep*`), enums (`StopReasonType`), and typed value types (`Transition[S, SM, E]`). The strongest implementations (Temporal ChASM, OpenAI Agents SDK) carry payload data; the weakest (agent-framework's scattered string literals) do not.
- **Streaming / non-streaming parity.** OpenAI Agents SDK duplicates the dispatch between streaming and non-streaming `while True` loops and documents the duplication (`AGENTS.md:55-67`); agent-framework's `FunctionInvocationLayer` runs the same `for attempt_idx` loop for both streaming and non-streaming (`python/packages/core/agent_framework/_tools.py:2567,2717`); Pydantic AI's `_run_stream` is async-generator-based and shares the dispatch logic.
- **Approval granularity.** OpenAI Agents SDK exposes a tri-state `RunContextWrapper._approvals` with permanent and per-call_id granularity (`src/agents/run_context.py:29-211`); agent-framework exposes `function_approval_request` as a typed function-call content type (`_tools.py:2228-2246`); letta exposes `RequiresApprovalToolRule` (`letta/schemas/tool_rule.py:13-373`) auto-appended at agent creation; CrewAI exposes before/after hooks with `False` to deny; OpenHands exposes `ConfirmRisky` vs `AlwaysConfirm` at the policy level.
- **Loop-budget surface.** Temporal exposes `BusinessIDConflictPolicy` + `WithSpeculative()` options (`chasm/engine.go:68-82,146-150`); agent-framework exposes `max_iterations` + `max_function_calls` + `max_consecutive_errors_per_request` + `max_rounds` + `max_reset_count` + `max_stall_count` + `MaxIterationsPerRequest` + `LoopAgentOptions.MaxIterations`; letta exposes `max_steps` with `StopReasonType.max_steps`; OpenAI Agents SDK exposes `DEFAULT_MAX_TURNS = 10` with `error_handlers["max_turns"]` override; CrewAI exposes `max_iter` per agent, `max_step_iterations` per step, `_max_iterations` per Flow.
- **Lifecycle state machine.** Temporal encodes the workflow task lifecycle as typed events (`ApplyWorkflowTaskScheduledEvent`, etc.); agent-framework encodes workflow lifecycle as typed `WorkflowEvent`s with `SuperStepStartedEvent` / `SuperStepCompletedEvent`; letta encodes step lifecycle as `StepProgression = {START, STREAM_RECEIVED, RESPONSE_RECEIVED, STEP_LOGGED, LOGGED_TRACE, FINISHED}` plus `StepStatus = {PENDING, SUCCESS, FAILED, CANCELLED}`; OpenHands encodes conversation start as `AppConversationStartTaskStatus`.
- **Where to look for test surface.** agent-framework and letta ship dedicated test files (`tests/core/test_function_invocation_logic.py`, `tests/test_tool_rule_solver.py`); OpenAI Agents SDK's `tests/test_run_step_execution.py` is 3552 lines; Pydantic AI's `tests/graph/` and `tests/graph/beta/` exercise v1/v2 graph iteration; LangGraph's `tests/test_algo.py:1-67` is thinner but `tests/test_pregel.py` covers routing indirectly; CrewAI uses `model_construct` to bypass Pydantic validators (`lib/crewai/tests/agents/test_agent_executor.py:23-54`); Temporal's `chasmtest.Engine` is a complete in-memory `chasm.Engine` impl.

## Tradeoffs

### Choice 1 — Imperative loop vs graph runtime

| Side | Imperative loop (OpenAI Agents SDK, agent-framework inner, letta, OpenHands) | Graph runtime (LangGraph, Pydantic AI, agent-framework workflows, Temporal ChASM) |
|---|---|---|
| Benefit | Single-threaded reasoning; small loop body; explicit dispatch | Static analysability; checkpoint-friendly; visualisable; checkpointing at superstep boundary |
| Cost | State machine logic concentrates in one function | One dispatch function (`prepare_next_tasks`, `CallToolsNode._run_stream`, `MachineTransition`) becomes the heaviest in the codebase |
| Best-fit | Single-agent runtimes; tight feedback between model and runtime | Multi-agent orchestration; long-running workflows; replay & debug needs |
| Failure mode | Dispatch duplication across streaming/non-streaming paths | Routing logic duplicated across layers (LangGraph's `map_command` vs `_control_branch`); v1/v2 coexistence (Pydantic AI beta graph, Temporal V1 + ChASM) |
| Alternative | Flow DSL (CrewAI) | Control-plane proxy (OpenHands) |

### Choice 2 — Typed payload union vs string-keyed state machine

| Side | Typed payload union (OpenAI Agents SDK's `NextStep*` dataclasses; Temporal's `Transition[S, SM, E]`; letta's `StopReasonType` + tuple) | String-keyed (CrewAI Flow's `Literal[...]`; agent-framework's `"return"/"continue"/"stop"`) |
|---|---|---|
| Benefit | Type system enforces exhaustiveness; payload can carry data (handoff target, approval items, final output) | Lightweight; readable; DSL-friendly; N-way branching without enumerating every variant |
| Cost | Adding a variant requires updating every consumer's `isinstance` ladder; the union is not extensible at runtime | String scattering; risk of typos; harder to refactor; less compiler help |
| Best-fit | Closed-set transitions with payload (handoff, final output, interruption) | User-extensible graphs where listeners can be added without editing the union |
| Failure mode | Dispatch code duplication across paths (OpenAI Agents SDK's streaming vs non-streaming) | Inconsistent label vocabulary (agent-framework's `"return"` vs `"stop"` literals scattered across `_tools.py`) |
| Alternative | Enum (`StopReasonType`, `RunStatus`, `StepProgression` in letta) | |

### Choice 3 — Pause as state vs pause as exception

| Side | State (OpenAI Agents SDK's `RunState._current_step: NextStepInterruption \| None`; letta's `requires_approval`; LangGraph's `interrupt_before/after`; Pydantic AI's typed approval flow) | Exception (agent-framework's `MiddlewareTermination`; Pydantic AI's `ApprovalRequired`) |
|---|---|---|
| Benefit | Serializable, versioned, replay-friendly; resume is a state query not a control-flow recompute | Simple; the loop can `try`/`except` |
| Cost | Requires the framework to know the variant set; harder to add new pause kinds | Exception-as-control-flow hides the pause from the outer loop; resume is implicit |
| Best-fit | Durable execution; long-running workflows; UI-driven approval | Transient pauses; tool-level errors |
| Failure mode | Multiple pause kinds overlap and need coordination | Exceptions cross middleware boundaries silently |

### Choice 4 — Capability middleware vs explicit transition function

| Side | Middleware (Pydantic AI's 17+ Capability hook points; agent-framework's three middleware pipelines) | Transition function (letta's `_decide_continuation`; OpenAI Agents SDK's `check_for_final_output_from_tools`) |
|---|---|---|
| Benefit | Cross-cutting concerns (telemetry, retries, moderation, caching) plug in without loop edits | Single auditable function; no indirection; the decision is in one place |
| Cost | Middleware ordering is configuration-driven and can silently reorder (`CapabilityOrdering` in Pydantic AI's `combined.py:55-60`); debugging requires knowing the chain | Cross-cutting concerns have to be hand-wired into the function |
| Best-fit | Public framework where users add their own concerns | Internal / private framework with a small, fixed set of decisions |
| Failure mode | Ordering conflicts raise `Conflicting positions` only at capability construction time | The function becomes a god-method with dozens of branches |

### Choice 5 — Per-step durability vs per-iteration only

| Side | Per-step (Temporal's atomic state-and-task; OpenAI Agents SDK's `RunState` snapshot; agent-framework's per-service-call history persistence; letta's per-step checkpoint pattern) | Per-iteration only (CrewAI's `_finalize_lock` + per-step output) |
|---|---|---|
| Benefit | A crashed tool execution still leaves a recoverable transcript | Simpler; less overhead |
| Cost | Schema versioning; persistence overhead; serialization discipline | Cannot recover from mid-tool crash |
| Best-fit | Tool calls can take minutes; user can disconnect; checkpointing required | Single-process runtimes with sub-second tool calls |

## Decision Guide

When choosing a control-flow model for a new agent framework, the following decision rules emerge from the source reports.

### Decision 1 — Loop shape: imperative vs graph vs Flow DSL

- **Pick imperative** (`while True` reading a typed next-step union) if your framework is a *single-agent* runtime with one chat client per iteration. OpenAI Agents SDK is the canonical example: 730 lines of loop body, every transition is an `isinstance` branch, the `NextStep*` union is the contract.
- **Pick BSP graph** (Pregel-style supersteps) if you have *multi-agent orchestration* and need durable execution. LangGraph and Temporal ChASM show how BSP makes checkpointing and replay tractable.
- **Pick Flow DSL** (`Flow[State]` with `@router` methods returning `Literal[...]`) if your *users* need to define their own graphs. CrewAI reuses its Flow runtime for both user graphs and the agent executor.
- **Pick control-plane proxy** if your agent runtime is a separate process you do not own. OpenHands shows the cost (opaque inner loop) and the benefit (clean HTTP boundary, testable surface).

### Decision 2 — Transition vocabulary

- **Pick typed payload union** (`@dataclass` variants or `Transition[S, SM, E]`) if the transition carries data (handoff target, final output, approval items). OpenAI Agents SDK and Temporal ChASM demonstrate the type-safety benefit.
- **Pick `Literal[...]` strings** if you have N-way branching and want the dispatch code to be a simple `if-elif` on the return value. CrewAI's Flow DSL is the example.
- **Pick enum** (`StopReasonType`, `RunStatus`, `StepProgression`) if you need exhaustiveness for terminal states. letta's `.run_status` mapping shows how to compose enums without losing readability.
- **Avoid string literals scattered across multiple files.** agent-framework's `"return" / "continue" / "stop"` literals at `_tools.py:2209,2265,2273,2337,2627,2632,2645,2802,2806` would be cleaner as a single enum.

### Decision 3 — Pause mechanism

- **Pick first-class state** (`RunState._current_step` with versioned schema, `NextStepInterruption` variant) if your framework needs durable pause/resume. OpenAI Agents SDK's `CURRENT_SCHEMA_VERSION = "1.11"` (`src/agents/run_state.py:131-149`) is the strongest example.
- **Pick first-class exception** (`MiddlewareTermination`, `ApprovalRequired`) if your pause is transient and lives within a single loop iteration. agent-framework's `MiddlewareTermination` propagates past the middleware pipeline by design (`_middleware.py:72-79`).
- **Pick policy object** (letta's `RequiresApprovalToolRule`) if your pause is configurable per tool. The policy can be auto-appended at agent creation (`letta/services/agent_manager.py:487-488`).

### Decision 4 — Override authority surface

- **Pick typed override primitives** (`middleware_pipeline.execute`, `check_for_final_output_from_tools`, `_decide_continuation`) if your users need to extend the loop. Pydantic AI's Capability chain shows how to make the override surface discoverable.
- **Pick in-loop checks** (`_check_run_cancellation`, `_check_max_iterations`, `_check_within_limits_or_complete`) if your override surface is small and fixed. agent-framework and letta use this pattern.

### Decision 5 — Test surface

- **Always expose an LLM-free test surface.** Every 8/10+ source does this. If your loop cannot be driven without a model, you cannot test runtime override authority under model failure. The minimum is a `MockChatClient` / `FakeModel` / in-memory engine that exercises the same dispatch code as production.

## Practical Tips

### Patterns to copy

1. **Single transition function.** letta's `_decide_continuation` is a model to follow — a single named function that maps `(state, model_output) → (continue, reason, stop_reason)`. Make it the only place the loop's continuation is decided.
2. **Typed payload union for transitions.** OpenAI Agents SDK's `NextStepHandoff | NextStepFinalOutput | NextStepRunAgain | NextStepInterruption` shows that a 4-variant union is enough for a multi-agent runtime. Dataclasses (not enums) let variants carry data.
3. **Tool-choice reset.** agent-framework's `tool_choice="none"` rewrite after budget exhaustion and OpenAI Agents SDK's `maybe_reset_tool_choice` are both small, well-tested runtime-authority primitives that defeat model-level loops.
4. **Force a final non-tool call.** agent-framework's pattern of forcing a `tool_choice="none"` call after budget exhaustion (`_tools.py:2664-2696`) prevents orphaned `function_call` items.
5. **Tri-state approval map.** OpenAI Agents SDK's `_ApprovalRecord` with True / False / None per tool + call_id (`src/agents/run_context.py:29-40`) is the cleanest approval primitive in the survey.
6. **Capability chain for cross-cutting concerns.** Pydantic AI's 17+ typed hook points (`pydantic_ai_slim/pydantic_ai/capabilities/abstract.py:405-947`) let users add telemetry, retries, moderation, caching without editing the loop.
7. **Per-step persistence.** OpenAI Agents SDK's versioned `RunState` schema (`src/agents/run_state.py:131-149, 184-321`), Temporal's atomic state-and-task commit, letta's `StepProgression` lifecycle are all durable-execution primitives that pay off when the user can disconnect.
8. **Mock everything.** `MockChatClient`, `FakeModel`, `TestModel`, `chasmtest.Engine`, `_build_executor` with `model_construct` — pick one and use it aggressively.

### Patterns to avoid or delay

1. **Scattered string literals.** agent-framework's `"return" / "continue" / "stop"` literals across 8 sites are the canonical anti-pattern. Promote to an enum once the vocabulary is stable.
2. **Three parallel executors.** CrewAI's `CrewAgentExecutor` / `AgentExecutor` / `LiteAgent` show how a deprecation window creates maintenance drag — the legacy path keeps accumulating subtle semantic drift.
3. **Three parallel drivers.** letta's `LettaAgent` / `LettaAgentV2` / `LettaAgentV3` face the same problem — heartbeat semantics differ between v2 and v3, parallel-tool overrides differ between v1 and v3.
4. **V1 + V2 coexistence.** LangGraph's `_internal` module is still the source of truth for transition primitives; Pydantic AI's v2 graph builder ships in beta; Temporal's V1 path encodes transitions across multiple large files. Aim for a single authoritative runtime.
5. **Per-step exception as control flow.** Use exceptions for actual errors; use state for pause/resume. Mixing the two makes the loop harder to reason about.
6. **Duplicated dispatch across streaming/non-streaming.** OpenAI Agents SDK's `AgentRunner.run` and `start_streaming` both have ~30-line `isinstance` ladders. The fix is a single `apply_turn_result(turn_result, ctx) -> Action` helper.
7. **Free-text hook events.** OpenHands' `HookEventResponse.event_type: str` lets typos through to the agent-server. Mirror the SDK's vocabulary as a typed enum.

### Decision rules

- **If the loop has <4 transition kinds and no payload:** `Literal[...]` is enough. agent-framework's `"return" / "continue" / "stop"` works.
- **If the loop has 4–10 transition kinds and carries data:** `@dataclass` union. OpenAI Agents SDK's `NextStep*` works.
- **If the loop has N-way branching and parallel listeners:** Flow DSL with `Literal[...]` routers. CrewAI's Flow works.
- **If the framework needs durable execution:** typed `Transition` value type with `Sources`/`Destination` and atomic commit. Temporal's ChASM works.
- **If the framework needs user-extensible graphs:** BSP graph with typed `Send` / `Command(goto=...)`. LangGraph works.
- **If the framework is a control-plane proxy:** explicit lifecycle enum + typed REST endpoints + string-keyed policy mapping. OpenHands works (with the noted caveats).

### Caution signs

- **Dispatch logic concentrated in one function.** agent-framework's `CallToolsNode._run_stream` is `# noqa: C901`. If your loop has the same `# noqa`, refactor.
- **String literals for transitions.** If you find yourself pattern-matching on `"return"`/`"continue"`/`"stop"` at multiple sites, promote to an enum.
- **Streaming / non-streaming parity tests.** If your streaming path does not have parity tests with the non-streaming path, they will diverge. agent-framework's `test_streaming_stops_at_max_iterations` and `test_max_iterations_limit` (one streaming, one not) are the pattern.
- **Test surface requires a model.** If your unit tests require a real LLM or VCR cassettes, your control flow is not testable in isolation.
- **Multi-version coexistence.** letta's three drivers, CrewAI's three executors, Temporal's V1 + ChASM are all red flags for ongoing maintenance cost.
- **Pause as exception.** If your pause mechanism is `try`/`except`, the resume path is implicit and hard to test.
- **Approval granularity is policy-level only.** OpenHands' `AlwaysConfirm` vs `ConfirmRisky` is binary. If users want per-tool approval, you need a finer primitive (OpenAI Agents SDK's tri-state or letta's `RequiresApprovalToolRule`).
- **Loop budget surface has >5 named caps.** agent-framework has `max_iterations`, `max_function_calls`, `max_consecutive_errors_per_request`, `max_rounds`, `max_reset_count`, `max_stall_count`, `MaxIterationsPerRequest`, `LoopAgentOptions.MaxIterations`. Document the relationship between them or unify behind a single `LoopBudget`.

## Anti-Patterns / Caution Signs

These are the concrete failure shapes that the source reports name as brittleness, coupling, or operational risk.

- **"Scattered `if/elif` over action strings"** (agent-framework `_tools.py:2209,2265,2273,2337,2627,2632,2645,2802,2806`). Hard to refactor; consumer code cannot pattern-match exhaustively.
- **"Three parallel executors / drivers"** (CrewAI `CrewAgentExecutor` / `AgentExecutor` / `LiteAgent`; letta `LettaAgent` / `LettaAgentV2` / `LettaAgentV3`). Maintenance drag; subtle semantic drift.
- **"Duplicated dispatch across streaming/non-streaming"** (OpenAI Agents SDK `AgentRunner.run:1367-1497` and `start_streaming:1065-1164`). Two ~30-line `isinstance` ladders; parity must be maintained manually.
- **"Routing logic in two layers"** (LangGraph `map_command` vs `_control_branch`). The same routing vocabulary in two places.
- **"God-method dispatch"** (Pydantic AI `CallToolsNode._run_stream` at `# noqa: C901`). Refactor once the function passes 100 lines.
- **"Pause as exception"** (`MiddlewareTermination`, `ApprovalRequired`). Resume is implicit; serialisation is fragile.
- **"Force-final-answer only as code path, not as state"** (CrewAI `_check_max_iterations` returns `"force_final_answer"` but the flow still routes through listeners — `_finalize_lock` papers over the race).
- **"Per-step persistence with no schema versioning"** (some CrewAI paths). Schema drift breaks resume.
- **"Capability ordering as configuration"** (Pydantic AI `CapabilityOrdering`; conflicts raise only at construction time). Silent reorder is possible.
- **"Streamed `is_complete` set in many places"** (OpenAI Agents SDK's many `streamed_result.is_complete = True` sites). Easy to miss a path.
- **"`tool_choice` reset between turns only"** (OpenAI Agents SDK `maybe_reset_tool_choice`). A model that packs many tool calls into one response with `tool_choice="required"` slips through.
- **"Free-text hook events"** (OpenHands `HookEventResponse.event_type: str`). Typos pass through to the agent-server.
- **"Webhook-driven termination with no liveness sweep"** (OpenHands). If webhooks stop firing, the app-server falls behind.
- **"`max_function_calls` checked after a parallel batch"** (agent-framework `_tools.py:2636-2644`). Can overshoot the cap by one batch's worth.
- **"V1 + V2 coexistence"** (LangGraph `_internal` module still source of truth; Pydantic AI beta graph; Temporal V1 + ChASM). Maintenance cost.
- **"Loop-budget surface >5 named caps"** (agent-framework). Document or unify.

## Notable Absences

These are gaps where the surveyed frameworks do *not* cover a control-flow concern well, or where the question is unanswered.

- **Mid-turn interruption in streaming.** OpenAI Agents SDK explicitly only has `after_turn` cancel (`src/agents/result.py:702-925`). No framework exposes a public API to inject an approval decision between LLM token chunks.
- **Intra-response `tool_choice` reset.** OpenAI Agents SDK's `maybe_reset_tool_choice` runs once per turn, not mid-response (`src/agents/run_internal/run_loop.py:1346,1830`). A model that emits many tool calls in one response with `tool_choice="required"` slips through.
- **Per-tool loop budget.** agent-framework has `max_function_calls` but it is checked after a batch and can overshoot (`python/packages/core/agent_framework/_tools.py:2636-2644`).
- **Direct user control over `max_function_calls` overshoot magnitude.** Documented but not bounded.
- **Shared observable contract across languages.** agent-framework's Python harness and .NET `LoopContext` have similar surfaces (`_loop.py:131-138`, `LoopContext.cs:17-90`) but no shared contract — porting a custom evaluator requires re-implementation.
- **First-party agent-loop inspection in control-plane proxies.** OpenHands cannot read iteration count or last-action time; only the projected `execution_status` enum is observable.
- **SDK-side confirmation policy change at runtime.** OpenHands exposes the conversation start, not mid-run change (`app_conversation_service_base.py:688-693` strongly suggests yes but no endpoint in the router does it).
- **Hook runtime invocation in OpenHands' boundary.** `hook_loader.py:43-148` loads hooks but no evidence in the repo that they are attached to the running agent.
- **Stale-state retry budget.** Temporal retries once (`service/history/statemachine_environment.go:200-220`); a budget-based retry with circuit-breaking could be more robust.
- **Default-on trace event for state transitions.** Temporal's `chasm.transition` span event is gated on `telemetry.DebugMode()` (`chasm/statemachine.go:56`).
- **Typed hook event vocabulary in OpenHands.** `HookEventResponse.event_type: str` is not a typed enum (`app_conversation_models.py:340`).

## Per-Source Notes

Each entry captures the *one thing* a reader should take away from the source report for control-flow design.

- **agent-framework** — Three layered loops (`FunctionInvocationLayer` → workflow superstep → `LoopAgent`) with three-action vocabulary (`return` / `continue` / `stop`). The strongest 9/10: layered, parallel Python/.NET, deeply tested. The single design lesson: *separate the chat-client loop, the workflow superstep, and the re-invocation loop into three bounded state machines with distinct transition authorities*.
- **crewai** — Flow DSL with `Literal`-typed routers; `_finalize_lock` papers over a Flow runtime race. The design lesson: *a Flow runtime reuses the same `@router`/`@listen` DSL for user graphs and the agent executor — leverage your existing state machine rather than building a parallel one*.
- **langgraph** — BSP/Pregel superstep loop with `RunControl.request_drain()` + `recursion_limit` + `interrupt_before/after`. The design lesson: *encode the runtime's override authority at superstep boundaries, with three checkpoints: drain, recursion, interrupt*.
- **letta** — `for i in range(max_steps)` + `_decide_continuation` + `ToolRulesSolver`. The design lesson: *put the policy logic in a separate object (`ToolRulesSolver`) and the decision in a single function (`_decide_continuation`); let the runtime rewrite model output (prefilled args, rule violations) rather than only reject it*.
- **openai-agents-sdk** — `NextStepHandoff | NextStepFinalOutput | NextStepRunAgain | NextStepInterruption` dataclass union + `RunState` snapshot. The design lesson: *payload-carrying transition variants + versioned pause/resume state is the strongest pattern for durable multi-agent runtimes*.
- **openhands** — Control-plane proxy with explicit `AppConversationStartTaskStatus` + three runtime override surfaces (`ConfirmationPolicy`, `SecurityAnalyzer`, `max_iterations`). The design lesson: *if your agent runtime is a separate process, push typed configuration and rely on HTTP boundary contracts; the app-server's own control surface should be small and testable*.
- **pydantic-ai** — Static four-node graph + Capability middleware chain + 8 typed control-flow exceptions. The design lesson: *closed-set graph schema + capability middleware is a clean way to make cross-cutting concerns pluggable without editing the loop*.
- **temporal** — V1 + HSM + ChASM with typed `Transition[S, SM, E]` values; atomic state-and-task commit; `Sources`/`Destination` enforcement; `chasmtest.Engine`. The design lesson: *for durable execution, encode every transition as a typed value with explicit source/destination and commit state + tasks atomically — the runtime is the only authority that can reject a stale move*.

## Open Questions

The per-source reports surface questions that the source material does not answer; these would need additional research.

1. **Shared observable contract across languages** — agent-framework's Python harness and .NET `LoopContext` are similar but not identical (`_loop.py:131-138`, `LoopContext.cs:17-90`). No cross-language contract.

## Executive Summary

The eight sources cluster into **five distinct execution archetypes** and disagree primarily on (a) whether the agent loop is graph-shaped, queue-shaped, or step-shaped, (b) whether the loop is the *only* model or one of several layered models, and (c) how durable / resumable the loop is.

The dominant observation: **every source names its model explicitly in code and has at least one iteration cap or termination predicate**. Beyond that, the models diverge widely. LangGraph and Agent Framework are the cleanest single-runtime stories (BSP / Pregel superstep, named and tested). OpenAI Agents SDK is the cleanest step-loop story (discriminated `next_step` union + `RunState` durability). Pydantic AI pushes the typed-graph state machine to a sibling library and wraps it in a 12-hook capability chain. CrewAI deliberately ships four layered models. Letta pairs a bounded ReAct loop with a six-layer lifecycle (HTTP → streaming → persistence → scheduler → sandbox → multi-agent). Temporal is a five-model composition built for a different problem (durable distributed workflows, not single-agent chat). OpenHands is a webhook-driven app-server that deliberately delegates the actual agent loop to a pinned sandbox image.

Ratings cluster in a narrow band: 7/10 (crewai, letta, openhands) → 8/10 (agent-framework, langgraph, pydantic-ai) → 9/10 (openai-agents-sdk, temporal). No source scored below 7 — the floor is high because *every* source names its model and exposes events for it. The difference between 7 and 9 is mostly about layering cleanness, the breadth of the public surface, and how cleanly contributors can extend the system.

## Core Thesis

**The execution model that an agent framework chooses is determined by three product-shape questions, not by aesthetic preference:**

1. **What is the unit of atomic progress?** A state-machine event (temporal), a BSP superstep (langgraph, agent-framework), a typed node transition (pydantic-ai), a discriminated `next_step` (openai-agents-sdk), a `for i in range(max_steps)` iteration (letta), a router emission (crewai), or an externally-emitted agent event (openhands).
2. **Is the loop the system, or is the loop one of several models?** Single-model systems (langgraph, openai-agents-sdk, pydantic-ai) are easier to teach. Layered systems (temporal, letta, openhands) handle more product surfaces but ask more of the contributor. CrewAI is a deliberately layered system that hides its layer count.
3. **Does the loop need to be durable across restarts?** If yes, the model gains a versioned state object (`RunState`, `MutableState`, `RunState`-like artifact) and a persistence boundary. If no, the loop can be in-memory-only and rely on `max_steps` / `recursion_limit` for safety.

The convergence is real but shallow: every source *names* its model and has a stop predicate. The divergence is structural: a step loop is wrong for a graph product; a graph is wrong for a queue product; a queue is wrong for a single-agent chat product. Most "what should I learn from this?" decisions reduce to choosing the model that matches your unit of atomic progress and the depth of the layering you need.

## Approach Models

The eight sources implement five distinct execution archetypes. The choice is rarely arbitrary — it follows from the product's unit of atomic progress and its durability / layering needs.

### 1. Pregel / BSP superstep graph (langgraph, agent-framework)

Both langgraph and agent-framework name the **Bulk Synchronous Parallel / Pregel** model explicitly. The control loop is: in each superstep, (1) select nodes whose subscribed channels advanced, (2) run them in parallel, (3) atomically apply their writes, (4) checkpoint. The loop terminates when no channel version advances or a `recursion_limit` / `max_iterations` is hit.

- **langgraph** (`libs/langgraph/langgraph/pregel/main.py:449`): class `Pregel`, with the comment "Similarly to Bulk Synchronous Parallel / Pregel model, computation proceeds in steps, while there are channel updates" at `libs/langgraph/langgraph/pregel/main.py:2974-2978`. Channels (`BaseChannel.update`/`consume`/`finish` at `libs/langgraph/langgraph/channels/base.py:19-121`) are the only inter-actor mechanism. `StateGraph`, `@entrypoint`, and `create_react_agent` all compile to a `Pregel` instance.
- **agent-framework** (`sources/agent-framework/python/packages/core/agent_framework/_workflows/_runner.py:31`): class `Runner` is documented as "A class to run a workflow in Pregel supersteps." The .NET mirror is `InProcessRunner` driving `ISuperStepRunner.RunSuperStepAsync` (`sources/agent-framework/dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunner.cs:212-241`). `DEFAULT_MAX_ITERATIONS = 100` (`_const.py:4`) is the safety cap; the .NET side does not have a numeric cap and relies on `HasUnprocessedMessages` going false.

**Distinguishing features**: BSP barrier, channel-based data flow, version-vector PULL optimization, "one checkpoint per superstep."

### 2. Turn-based discriminated step loop (openai-agents-sdk)

A `while True` loop calls a single-turn function that returns a `SingleStepResult` whose `next_step` is a discriminated union of `NextStepFinalOutput | NextStepHandoff | NextStepRunAgain | NextStepInterruption` (`src/agents/run_internal/run_steps.py:154-218`). The outer loop branches on the discriminant; streaming and non-streaming share the discriminant so the state machine does not fork.

**Distinguishing features**: discriminated-union state machine in the type system, versioned `RunState` (`CURRENT_SCHEMA_VERSION = "1.11"`), first-turn guardrail cancellation that pre-empts in-flight model work (`src/agents/run.py:1219-1245`), `after_turn` cancel mode for graceful drain.

### 3. Typed graph state machine (pydantic-ai)

A graph of `BaseNode` subclasses (`UserPromptNode` → `ModelRequestNode` ↔ `CallToolsNode` → `End(FinalResult)`) executed by `_GraphIterator.iter_graph` (`pydantic_graph/pydantic_graph/graph_builder.py:646-825`). Edges are inferred from `BaseNode.run()` return-type annotations (`pydantic_graph/pydantic_graph/basenode.py:104-136`). A 12-hook capability middleware chain (`pydantic_ai_slim/pydantic_ai/capabilities/abstract.py:100-958`) wraps every transition.

**Distinguishing features**: type-inferred edges, task-group + memory-object-stream runtime with `ErrorMarker` recovery, capability tier system (`'outermost' | 'innermost'`), `UsageLimits` (request/tool_calls/input_tokens/output_tokens/total_tokens) (`pydantic_ai_slim/pydantic_ai/usage.py:263-422`), parallel-tool sibling cancellation on `BaseException`.

### 4. Plan-and-Execute / Flow DAG (crewai)

A decorator-defined DAG (`@start` / `@listen` / `@router`) implemented as a recursive listener dispatch (`_execute_listeners` at `lib/crewai/src/crewai/flow/runtime/__init__.py:3239`). Router methods return `Literal[...]` labels that the runtime uses to fire `@listen` handlers (`lib/crewai/src/crewai/flow/dsl/_router.py:86-88`). The Plan-and-Execute `AgentExecutor` reuses this Flow runtime as its inner state machine.

**Distinguishing features**: literal-typed router emissions, locked state proxies (`LockedDictProxy`/`LockedListProxy`) for async/sync mixing, two-loop pattern (outer plan loop + inner `StepExecutor` loop with per-step message isolation).

### 5. Bounded ReAct step loop (letta)

A `for i in range(max_steps)` loop where each iteration makes one LLM call (via `LettaLLMAdapter`) and, if the model returns a tool call, executes it via `ToolExecutionManager`. Continuation is decided by `request_heartbeat` (returned from the tool) plus `ToolRulesSolver` (`letta/helpers/tool_rule_solver.py:24-298`). Wrapped in a six-layer lifecycle: HTTP request → streaming/SSE transport → Run/Step state machine → APScheduler + Postgres advisory-lock poller → tool-sandbox dispatch → multi-agent sleeptime.

**Distinguishing features**: heartbeat flag, `StopReasonType` taxonomy (`end_turn`, `max_steps`, `tool_rule`, `requires_approval`, `context_window_overflow_in_system_prompt`, …), leader-elected scheduler, Redis conversation lock.

### 6. Layered orchestration: workflow + webhook router (openhands)

The app-server (`openhands/`) is a wrapper around an external `agent-server` image. The 8-state `AppConversationStartTaskStatus` workflow drives startup; an event-callback router consumes agent-server webhooks (`POST /webhooks/events/{conversation_id}`); a polling fallback (`poll_agent_servers`) runs when the deployment cannot be reached via webhook.

**Distinguishing features**: 8-state async-generator workflow, `EventCallbackProcessor` ABC with `ClassVar[EventKind]` discriminator, webhook/poll duality gated by `web_url`, per-sandbox session API key, hidden agent loop (pinned to `ghcr.io/openhands/agent-server:1.29.0-python`).

### 7. Transactional state machine + persistent task queue (temporal)

A deliberate composition of five models: (1) transactional workflow state machine (`MutableStateImpl` + `WorkflowTaskStateMachine`), (2) per-category queue processor (`Queue.processEventLoop`), (3) shard ownership loop, (4) long-poll matcher engine, (5) CHASM HSM engine. The transactional outbox writes history events + tasks in one persistence call (`docs/architecture/history-service.md:319-321`).

**Distinguishing features**: transactional outbox, per-shard `RangeID` fencing, alert/monitor/mitigator triad, two-tier scheduler stack (`InterleavedWeightedRoundRobin` → `ExecutionAwareScheduler` → `FIFO`), CAS-guarded `Start/Stop` for double-start safety.

## Pattern Catalog

Each pattern below is implemented in two or more sources, or one source demonstrates it unusually clearly.

### P1. Named termination predicate (universal)

Every source exposes a loop terminator in code: `not has_messages()` (agent-framework `letta/agents/letta_agent_v2.py:1250`), `recursion_limit` / `out_of_steps` (langgraph `libs/langgraph/langgraph/pregel/main.py:3017-3026`), `max_turns` with `MaxTurnsExceeded` (openai-agents-sdk `src/agents/run.py:1058-1144`), `max_steps` (letta `letta/agents/letta_agent_v2.py:233`), `request_limit` (pydantic-ai `pydantic_ai_slim/pydantic_ai/usage.py:273`), `max_method_calls` (crewai `flow/runtime/__init__.py:3178-3185`).

**When to copy**: always. The named predicate is the single most useful piece of execution-model documentation — it tells a contributor "the loop ends when X."

**When it is overkill / risky**: a one-shot tool call with no iteration can skip this. For research-style agents that need many turns, do not rely on a single global cap; use per-tool budgets too (letta's gap).

### P2. Streaming twin that shares the state machine (langgraph, openai-agents-sdk, pydantic-ai, agent-framework, letta)

Five sources expose both a non-streaming and a streaming entry point. In four of them, the streaming variant does **not** fork the state machine — the same discriminant / channel / next-step is used, and streaming adds a queue or sentinel.

- openai-agents-sdk: `run_single_turn` + `run_single_turn_streamed` both return `SingleStepResult` with the same `next_step`; streaming adds `asyncio.Queue` + `QueueCompleteSentinel`.
- pydantic-ai: `agent.run` and `agent.run_stream` share the graph; `run_stream` short-circuits on first final output.
- langgraph: `stream` and `ainvoke` share the BSP driver; only the wire format changes (`v1` / `v2` / `v3`).
- agent-framework: `InProcessRunner` is consumed by `StreamingRunEventStream` and `LockstepRunEventStream`; the `ExecutionMode` enum selects delivery shape, not the execution model itself.
- letta: `step` and `stream` are parallel methods on `BaseAgentV2` with the same loop body.

**When to copy**: when you have any non-trivial model interaction. The benefit is that contributors only learn the state machine once; the cost is that adding a `next_step` variant requires editing both paths.

**When it is overkill**: trivial single-turn `complete()` calls.

**Evidence**: see `src/agents/run.py:768-1497` vs. `src/agents/run_internal/run_loop.py:670-1239`; `agent/abstract.py:438-446` vs. `agent/abstract.py:653-841`; `InProcessRunner.cs:212-241` + `StreamingRunEventStream.cs:56-141`.

### P3. Versioned durable state object (openai-agents-sdk, temporal, langgraph, pydantic-ai via durable_exec, agent-framework)

Four sources expose a serializable state object with explicit schema versioning:

- openai-agents-sdk: `RunState` with `CURRENT_SCHEMA_VERSION = "1.11"` (`src/agents/run_state.py:131`).
- temporal: `MutableStateImpl` + history events; CHASM is migrating to a versioned `Node` tree.
- langgraph: checkpoint per superstep; `_migrate_checkpoint` callback for old formats (`libs/langgraph/langgraph/pregel/_loop.py:1641-1642`).
- agent-framework: `_hash_graph_signature` fingerprints the graph (`_workflow.py:1078-1080`); `CheckWorkflowMatch` rejects mismatched graph + checkpoint (`InProcessRunner.cs:435-436`).

**When to copy**: when pause/resume or crash-recovery is in the product spec. A versioned state object is the cleanest way to make resumptions safe across SDK upgrades.

**When it is overkill**: a stateless per-request API where each call is independent.

### P4. One or more iteration / request caps (universal, but with variance)

All eight sources have at least one safety cap:

| Source | Cap | Default | Type |
|--------|-----|---------|------|
| agent-framework | `DEFAULT_MAX_ITERATIONS` | 100 | Numeric |
| crewai | `max_method_calls`, `max_iter * 10`, `max_replans` | varies | Numeric |
| langgraph | `recursion_limit` | 25 (default) | Numeric |
| letta | `max_steps` | `DEFAULT_MAX_STEPS` | Numeric |
| openai-agents-sdk | `max_turns` | optional | Numeric |
| openhands | `startup_timeout` | 120 s | Time |
| pydantic-ai | `UsageLimits.request_limit` | 50 | Numeric + per-tool-call limit |
| temporal | `slice + checkpoint` cycle, `forceNewSliceDuration` | 5 min | Time + structural |

**When to copy**: always. Different caps protect different risks: numeric caps prevent runaway iteration; time caps prevent slow-leak resource exhaustion; per-tool-call caps localize damage from a single tool.

**When it is overkill**: never. The variation is in *which* caps you need, not in whether you need any.

### P5. Cooperative hand-off pattern between model and stream (pydantic-ai, openai-agents-sdk, langgraph, agent-framework)

The streaming path needs to yield tokens to the consumer *while* the model is still running. Three concrete patterns are visible:

- **Event-based** (pydantic-ai `ModelRequestNode.stream`): `stream_ready` / `stream_done` `asyncio.Event`s; consumer awaits `stream_ready`, yields, then `stream_done.set()`.
- **Queue + sentinel** (openai-agents-sdk): `RunResultStreaming._event_queue: asyncio.Queue[StreamEvent | QueueCompleteSentinel]`.
- **Generator with FIRST_COMPLETED** (langgraph `PregelRunner.tick`): `concurrent.futures.wait(..., FIRST_COMPLETED)` between yields.

**When to copy**: any time the streaming path intersects with a concurrency primitive. The hand-off is the load-bearing piece — without it, the consumer blocks the model or the model blocks the consumer.

**When it is risky**: when the hand-off is implicit (e.g., bare `await future`); the contributor will not see the boundary.

### P6. Discriminated state / event union in the type system (openai-agents-sdk, pydantic-ai, langgraph, agent-framework, crewai)

Five sources encode the state machine as a typed union so that adding a variant fails at compile time / static-check time:

- openai-agents-sdk: `SingleStepResult.next_step: NextStepHandoff | NextStepFinalOutput | NextStepRunAgain | NextStepInterruption`.
- pydantic-ai: `BaseNode | End[NodeRunEndT]`, with `assert_never` for exhaustiveness.
- langgraph: `tick` returns `status: Literal["done", "out_of_steps", "draining", ...]`.
- agent-framework: `WorkflowRunState` enum + status events as separate channels.
- crewai: `@router` methods return `Literal[...]`; emit list captured at Flow-definition time.

**When to copy**: when the loop has more than 2-3 terminal states. The discriminator prevents the silent "what if a new state is added and I forget to update a dispatch site?" bug.

**When it is overkill**: a binary `while condition: ...` loop.

### P7. Pluggable concurrency executor (langgraph, openai-agents-sdk, agent-framework, temporal, pydantic-ai)

Five sources separate "what to do" from "how to run it concurrently":

- langgraph: `PregelRunner.tick` dispatches to `BackgroundExecutor` (sync, thread pool) or `AsyncBackgroundExecutor` (async, semaphore + asyncio tasks).
- openai-agents-sdk: `RunResultStreaming._run_loop_task = asyncio.create_task(start_streaming(...))`.
- agent-framework: `asyncio.gather` (Python) / `Task.WhenAll` (.NET) per edge, plus `ExecutionMode.OffThread` / `Lockstep` / `Subworkflow`.
- temporal: `Scheduler` interface with `InterleavedWeightedRoundRobin` → `ExecutionAwareScheduler` → `FIFO` stack.
- pydantic-ai: `_call_tools` picks one of three modes: `'sequential'`, `'parallel_ordered_events'`, or default race.

**When to copy**: as soon as a node can fan out across multiple tools or actors. The pluggable executor is the only way to keep the *control* model unchanged while the *concurrency* model varies.

**When it is overkill**: single-tool, single-step agents.

### P8. Event bus / event-callback subscription (crewai, openhands, openai-agents-sdk-realtime, temporal, letta, agent-framework)

Multiple sources implement an event bus or callback dispatch for observability and side effects:

- crewai: `CrewAIEventsBus` with dedicated daemon event loop + `ThreadPoolExecutor` (`lib/crewai/src/crewai/events/event_bus.py:94-189`).
- openhands: `EventCallbackProcessor` ABC + `SQLEventCallbackService.execute_callbacks` dispatching on `(status, event_kind, callback_id)`.
- openai-agents-sdk realtime: `OpenAIRealtimeWebSocketModel._emit_event` iterates `list(self._listeners)`.
- temporal: `NotifierImpl.dequeueHistoryEventNotifications` loop dispatches to subscribers.
- letta: `StepProgression (START, STEP_LOGGED, STREAM_RECEIVED, RESPONSE_RECEIVED, LOGGED_TRACE, FINISHED)`.
- agent-framework: per-superstep `SuperStepStartedEvent` / `SuperStepCompletedEvent`.

**When to copy**: when you need observability, plug-in points, or side effects that do not belong in the core loop. The event bus is also the cleanest way to implement webhook delivery (openhands) and analytics (temporal, letta).

### P9. Per-step / per-superstep persistence boundary (langgraph, agent-framework, temporal, openai-agents-sdk, pydantic-ai-durable_exec)

Five sources have a persistence boundary at a named, predictable point in the loop:

- langgraph: `_put_checkpoint({"source": "loop"})` in `after_tick` (`libs/langgraph/langgraph/pregel/_loop.py:706`).
- agent-framework: checkpoint in `InProcessRunner.RunSuperstepAsync` after every `DeliverMessagesAsync` (`InProcessRunner.cs:332`).
- temporal: `MutableState.CloseTransaction*` writes events + tasks atomically (`docs/architecture/history-service.md:319-321`).
- openai-agents-sdk: `RunState` snapshot on `NextStepInterruption` (`src/agents/run_state.py:184-321`).
- pydantic-ai: `durable_exec/` integration via the capability system (referenced in `pydantic_ai_slim/pydantic_ai/durable_exec/AGENTS.md`).

**When to copy**: when the loop needs to be replayable. The boundary is what makes a step-loop a "durable" step-loop.

**When it is overkill**: stateless / single-shot runs.

### P10. Process / state machine that survives exceptions (openai-agents-sdk, pydantic-ai, crewai)

Three sources have first-class recovery from in-flight failures:

- openai-agents-sdk: `NextStepInterruption` triggers `RunState` save; resume via `state.approve(approval_item)` / `state.reject(...)` (`src/agents/run_state.py:332-339`).
- pydantic-ai: `ErrorMarker` recovery — exceptions in `_run_tracked_task` are emitted as `ErrorMarker(task_result.error)`; the caller can `override_next(...)` or re-raise (`pydantic_graph/pydantic_graph/graph_builder.py:660-667`).
- crewai: `recover_from_parser_error` increments `iterations` and routes back to `"initialized"`; `recover_from_context_length` summarizes and retries (`lib/crewai/src/crewai/experimental/agent_executor.py:2678-2715`).

**When to copy**: when the loop will be exposed to user input that can fail in predictable ways. Recovery-as-control-flow is more robust than try/except around the whole loop.

### P11. Two-tier concurrency: outer step loop + inner parallel tool dispatch (openai-agents-sdk, pydantic-ai, langgraph, crewai, agent-framework, letta)

Six sources separate "advance the step" from "run this step's tools":

- openai-agents-sdk: outer `while True` advances `current_turn`; inner `execute_tools_and_side_effects` runs tools, handoffs, and guardrails in parallel via `asyncio.gather`.
- pydantic-ai: outer graph state machine advances; inner `_call_tools` picks one of three parallel modes.
- langgraph: outer BSP superstep; inner `BackgroundExecutor` runs selected tasks in parallel.
- crewai: outer plan loop; inner `StepExecutor` runs a multi-turn LLM + tool call with its own bounded `for _ in range(max_step_iterations)`.
- agent-framework: outer superstep; inner `asyncio.gather` per edge.
- letta: outer `for i in range(max_steps)`; inner per-step `_handle_ai_response` (one LLM, at most one tool).

**When to copy**: when one step can run multiple tools or actors. The two-tier separation keeps the *control* model clean and the *concurrency* model pluggable.

**When it is overkill**: when there is never more than one tool per step.

### P12. Capability / middleware / hook chain (pydantic-ai, openai-agents-sdk, agent-framework, langgraph)

Four sources wrap the loop in a middleware chain:

- pydantic-ai: 12-hook `AbstractCapability` chain with `CombinedCapability` composition and `'outermost' | 'innermost'` tier system (`pydantic_ai_slim/pydantic_ai/capabilities/abstract.py:76-84`).
- openai-agents-sdk: `RunHooks` lifecycle (`src/agents/lifecycle.py`) called from `AgentRunner.run` body.
- agent-framework: `AgentLoopMiddleware` ("re-runs an agent in a loop by calling `call_next()` repeatedly") in the Python harness; .NET side uses decorator pattern (`FunctionInvocationDelegatingAgent.cs:15-87`).
- langgraph: callback manager (`libs/langgraph/langgraph/callbacks.py`) emits `RawResponsesStreamEvent` / `RunItemStreamEvent` / `AgentUpdatedStreamEvent`.

**When to copy**: when the loop has cross-cutting concerns (telemetry, tracing, retries, rate-limiting, approval). A middleware chain is more composable than scattering flags on the loop entry point.

**When it is overkill / risky**: when there is only one cross-cutting concern — a single hook is enough; a full chain is overkill. The risk is that ordering bugs in the chain become contributor hazards (pydantic-ai's `_ordering.py` solver is non-trivial).

## Key Differences

These are the differences that matter when choosing a model. They are *not* quality differences in most cases — they are responses to different product shapes.

### K1. Single runtime vs. layered composition

- **Single runtime**: langgraph (BSP), openai-agents-sdk (turn loop), pydantic-ai (graph), agent-framework (superstep).
- **Layered composition**: temporal (5 models), letta (6 layers), openhands (workflow + webhook + poll), crewai (4 models).

A single runtime is easier to teach and easier to extend — contributors learn one state machine. A layered composition handles more product surfaces (durability + long-poll + sandboxing + multi-tenancy) but asks more of the contributor. The choice is product-driven, not aesthetic.

### K2. Public graph surface vs. hidden graph

- **Public**: langgraph (`StateGraph.compile()`), pydantic-ai's `pydantic_graph` library, agent-framework's `WorkflowBuilder`, crewai's `Flow` (in the public sense). Users can add nodes and edges.
- **Hidden**: pydantic-ai's `Agent` (the user sees `AgentRun`, not `Graph`; the graph is built inside `build_agent_graph` at `_agent_graph.py:2110-2139`), agent-framework's `AIAgent` (the graph is the workflow; the agent is a single-turn chat call).

A public graph surface is a power-user feature: it lets the user build anything. A hidden graph is a safety / simplicity feature: the framework guarantees the agent is the agent. Most "what should I learn from this?" choices reduce to "do my users need to build their own graph?"

### K3. Step-loop vs. graph for the agent

- **Step loop**: openai-agents-sdk, letta, agent-framework's `AIAgent`, agent-framework's `Agent` in Python.
- **Graph**: langgraph, agent-framework's `Workflow`, pydantic-ai, crewai's `Flow`, temporal, openhands (workflow for startup).

A step loop is right for a single agent. A graph is right for multiple actors or for tool results that should feed back into the model. CrewAI is the unique case where the *plan* and the *step* are both expressed as a graph (the inner Flow runtime drives a step loop).

### K4. Durability model: in-memory, versioned, or sharded

- **In-memory**: letta, pydantic-ai (default; durable_exec is opt-in), crewai (Flow persistence is optional), agent-framework (checkpoints are opt-in via `checkpoint_id`).
- **Versioned single-process**: openai-agents-sdk (`RunState` with `CURRENT_SCHEMA_VERSION="1.11"`), langgraph (one checkpointer, migrate on load).
- **Sharded distributed**: temporal (`MutableStateImpl` + `RangeID` fencing + DLQ), openhands (per-sandbox session API key + Redis stream bridge).

The choice follows from "is this a single-process product or a distributed product?" Temporal's model is not a model for chat; it is a model for distributed workflows that *need* to survive a regional outage. Openhands' sharded model exists because sandboxes are inherently per-conversation.

### K5. State machine in the type system vs. runtime

- **Type system**: openai-agents-sdk (`@dataclass` union with static dispatch), pydantic-ai (`BaseNode | End` + `assert_never`), langgraph (typed `tick` status), agent-framework (`WorkflowRunState` enum), crewai (`@router` with `Literal[...]`).
- **Runtime-only**: letta (the `request_heartbeat` flag is a bool), openhands (8-state enum is a value, not a type).

A type-system state machine is harder to misuse. A runtime state machine is more flexible but is a contributor hazard. The cost of the type-system approach is that adding a variant requires editing every dispatch site (openai-agents-sdk's known maintenance cost).

### K6. Concurrency model: cooperative vs. preemptive

- **Cooperative**: langgraph (cancellation through `RunControl.drain_requested` mid-superstep), pydantic-ai (`wrap_run` short-circuit), openai-agents-sdk (`after_turn` cancel mode), agent-framework (`ExecutionMode.Lockstep` accumulates events per superstep).
- **Preemptive**: openai-agents-sdk (`cancel(mode="immediate")` cancels tasks), pydantic-ai (`CancelScope` in `_GraphIterator.iter_graph`), agent-framework (Task.WhenAll cancellation propagates).
- **Both**: most sources have both. The combination is "preempt in-flight tasks, but check at the next superstep / next step."

A preemptive cancel is needed for long-running tool calls. A cooperative cancel is needed for clean state (you cannot cancel mid-database-commit). The most mature sources use both.

## Tradeoffs

Every major design choice has a cost. This section captures the benefit / cost / best-fit / failure-mode for each.

### T1. BSP superstep barrier (langgraph, agent-framework)

- **Benefit**: deterministic ordering, one-checkpoint-per-superstep, trivial parallel-within-step fan-out.
- **Cost**: an extra superstep between cause and effect. ReAct agents pay one extra superstep per tool call.
- **Best fit**: long-running multi-actor graphs where determinism + checkpointing are required.
- **Failure mode**: cross-edge ordering anomalies; "I just wrote a value, why is it not visible?"
- **Alternative**: openai-agents-sdk's turn loop, where the LLM and tool call are in the same turn.

### T2. Discriminated `next_step` union (openai-agents-sdk)

- **Benefit**: static exhaustiveness, single dispatch site per variant, clean first-class interruption.
- **Cost**: adding a new variant requires editing every dispatch site (~7 sites in `run.py` + `run_loop.py` + `turn_resolution.py`).
- **Best fit**: when the state machine has 3-5 named states and the variants are stable.
- **Failure mode**: a new variant added in one file and missed in another fails loudly at the typed unknown-variant error (`run.py:1495-1497`).
- **Alternative**: langgraph's BSP loop (no explicit states, just "no candidates").

### T3. Capability middleware chain (pydantic-ai)

- **Benefit**: cross-cutting behavior in one place, composable, ordering is solved once.
- **Cost**: 12 hook points is high cognitive load; ordering requires a constraint solver (`_ordering.py`).
- **Best fit**: when many cross-cutting concerns (tracing, retries, rate-limit, instrumentation) need to be added without editing the core loop.
- **Failure mode**: a contributor who skips `wrap_node_run` (e.g., bare `async for node in agent_run` at `run.py:189-201`) loses all middleware behavior.
- **Alternative**: langgraph's callback manager (smaller surface, less powerful).

### T4. Layered composition (temporal, letta, openhands, crewai)

- **Benefit**: each layer can be replaced, scaled, or tested independently.
- **Cost**: the contributor must learn multiple models; "what advances execution?" requires naming which layer.
- **Best fit**: when the product has multiple independent surfaces (durability + sandboxing + multi-tenancy).
- **Failure mode**: layer boundaries that leak (letta's `letta/agent.py` 89 KB legacy class, openhands' dual webhook/poll execution-status sources).
- **Alternative**: a single runtime with clear extensions (openai-agents-sdk).

### T5. Bounded `for i in range(max_steps)` (letta)

- **Benefit**: trivially safe; billing-predictable; one-line enforcement.
- **Cost**: no graceful termination; a heartbeating tool burns the full budget; no per-tool-call budget.
- **Best fit**: server-side, per-request, paid services where each turn is a known cost.
- **Failure mode**: a misconfigured agent that always heartbeats is killed at `max_steps`; a tool that needs more than `max_steps` calls cannot get them.
- **Alternative**: a richer `UsageLimits` system (pydantic-ai) with request / tool_calls / token budgets.

### T6. Hidden graph (pydantic-ai's `Agent`, agent-framework's `AIAgent`)

- **Benefit**: simpler mental model for the user; framework guarantees correctness.
- **Cost**: power users cannot extend the graph; everything goes through the capability system.
- **Best fit**: products where 95% of users just want a "model + tools + maybe an output schema" agent.
- **Failure mode**: a user with a genuinely novel control-flow need (e.g., "I want to call two models in parallel and merge") hits a wall and has to fork the framework.
- **Alternative**: langgraph's public `StateGraph` for power users.

### T7. Transactional outbox (temporal)

- **Benefit**: history events and tasks persist atomically; the system never has "I persisted the event but not the timer that should fire."
- **Cost**: every state change writes the full event + task set in one DB transaction; throughput is bounded by DB write rate.
- **Best fit**: distributed workflows where missed tasks are catastrophic.
- **Failure mode**: a long transaction holding row locks; mitigated by `forceNewSliceDuration` and shard fencing.
- **Alternative**: langgraph's per-superstep checkpoint (no transactional outbox; checkpointer is a pluggable interface).

### T8. Webhook + polling fallback (openhands)

- **Benefit**: works in any deployment topology (public web URL, NAT, localhost).
- **Cost**: dual execution-status sources can drift; polling is at `polling_interval` granularity (default 15s).
- **Best fit**: deployments that may not have a public webhook URL.
- **Failure mode**: localhost dev runs polling; production may run webhook; a contributor confused by "why is my event not appearing?" reads the wrong code path.
- **Alternative**: pure webhook (requires public URL) or pure polling (no real-time guarantee).

## Decision Guide

A new contributor choosing an execution model should answer four questions in order.

### Q1. Is the product a single agent, a graph of actors, or a distributed workflow?

- **Single agent** → step loop (openai-agents-sdk, letta, pydantic-ai's `Agent`).
- **Graph of actors** → BSP / typed graph (langgraph, agent-framework's `Workflow`, pydantic-ai's `pydantic_graph`).
- **Distributed workflow** → queue / state machine (temporal, openhands' startup workflow).

### Q2. Does the loop need to be durable across restarts?

- **No** → in-memory loop with `max_steps` / `recursion_limit` (letta, pydantic-ai default).
- **Yes, single-process** → versioned `RunState` (openai-agents-sdk) or per-superstep checkpoint (langgraph, agent-framework).
- **Yes, distributed** → transactional outbox + shard fencing (temporal) or webhook + per-sandbox state (openhands).

### Q3. Does the user need to build their own graph?

- **No** → hidden graph (pydantic-ai's `Agent`, agent-framework's `AIAgent`).
- **Yes** → public graph DSL (langgraph's `StateGraph`, pydantic-ai's `pydantic_graph`, agent-framework's `WorkflowBuilder`).

### Q4. Does the loop need cross-cutting behavior (tracing, retries, rate-limit, approval)?

- **No** → single hook point or no hooks (letta's `RunHooks`).
- **Yes, simple** → callback manager (langgraph).
- **Yes, complex** → capability middleware (pydantic-ai) or `RunHooks` + lifecycle (openai-agents-sdk).

If you answered **single agent, durable single-process, no public graph, simple cross-cutting** — start with a step loop + a `RunState` snapshot (openai-agents-sdk's model). If you answered **graph of actors, durable single-process, public graph, complex cross-cutting** — start with a BSP superstep loop (langgraph) or a typed graph (pydantic-ai). If you answered **distributed workflow, durable distributed, public topology, complex cross-cutting** — start with a transactional state machine + queue (temporal's model). Anything in between is layered composition, and the cost is that the contributor learns more.

## Practical Tips

- **Name the model in code.** The single highest-value documentation is a module-level comment that says "this is a Pregel superstep loop" or "this is a turn step loop with discriminated next-step." Every source in this study that scores ≥ 8 has this.
- **Expose the loop terminator as a named predicate.** `has_messages()`, `recursion_limit`, `max_steps`, `request_limit`. Make it discoverable.
- **Cap iteration count with at least one numeric guard.** Universal; the variation is in defaults. 50 (pydantic-ai), 100 (agent-framework), 25 (langgraph default `recursion_limit`).
- **Share the state machine between streaming and non-streaming.** All five streaming twins in this study reuse the same state. The only addition is a queue or event.
- **Encode the state machine in the type system.** Discriminated unions + `assert_never` (pydantic-ai) or `match` with `UnknownVariantError` (openai-agents-sdk `run.py:1495-1497`).
- **Separate "what advances" from "how it runs concurrently."** A `Scheduler` / `Runner` / `Executor` separation lets you swap concurrency without changing the control model (langgraph's `PregelRunner` + `BackgroundExecutor`).
- **Make the iteration cap per-layer, not global.** A global `max_method_calls` (crewai) counts every listener invocation; a per-tool-call budget (pydantic-ai) localizes damage.
- **Add a versioned durable state object before you need it.** The cost of adding `RunState` + `CURRENT_SCHEMA_VERSION="1.11"` later is breaking all existing snapshots. Build the version field on day one.
- **Use an event bus for observability, not a return value.** crewai's `CrewAIEventsBus`, openhands' `EventCallbackProcessor`, openai-agents-sdk's stream events all work this way. The return value is for the result; the event bus is for everything else.
- **Test the BSP barrier contract, not just end-to-end.** langgraph's `test_delta_channel_supersteps_bound.py`, `test_interleave_arrival_order.py` — these tests fail in ways end-to-end tests cannot.

## Anti-Patterns / Caution Signs

- **Global listener-call counter as a runaway-loop defense** (crewai `max_method_calls = max_iter * 10`). It conflates legitimate fan-out with runaway loops. A per-method budget or a fingerprint-based cycle detector is more precise.
- **Hidden agent loop in a pinned Docker image** (openhands). The trade is sandbox isolation; the cost is that contributors cannot read or test the loop. Acceptable when the loop is provably stable and versioned.
- **`while True` outer loop with `max_chaining_steps` as the only termination** (letta legacy `letta/agent.py:775` plus `max_chaining_steps` at line 800). Cancellation relies on the legacy interface's `is_pause_request` flag pattern, which is harder to reason about than a typed state machine.
- **Bare `async for node in agent_run` that skips capability hooks** (pydantic-ai `run.py:189-201`). Emits a `UserWarning` but does not refuse to run. A contributor who copies a tutorial will lose all middleware behavior.
- **`validate_graph_structure=False` on a hand-built graph** (pydantic-ai `_agent_graph.py:2139`). A bug in a node's return-type annotation is caught at runtime, not build time. The trade is "the agent's hand-built topology is trusted." Fine for a stable four-node graph; risky as the graph grows.
- **Dual execution-status sources without explicit reconciliation** (openhands `webhook` + `poll`). A sandbox in `RUNNING` with a disconnected webhook shows `execution_status=None` until the next poll. A reconciliation heartbeat is needed.
- **Two parallel language implementations without a shared schema** (agent-framework Python + .NET). A new feature must be implemented twice, in `_edge_runner.py` and `EdgeRunner.cs` and friends. The trade is performance; the cost is contributor surface.
- **Streaming with three format versions** (langgraph `v1` / `v2` / `v3`). API versioning is not the same as execution-model versioning, but the surface area is real. Document the choice and the wire format.
- **Per-superstep event-storm with unbounded channel** (agent-framework `StreamingRunEventStream.cs:38-44`). Pathological fan-out can buffer many events before the consumer drains. The `Lockstep` mode is the explicit countermeasure; do not skip it.
- **`FunctionInvokingChatClient` reused as the in-framework tool loop** (agent-framework .NET). The framework intentionally leaves tool-iteration to the external decorator, so users must compose it themselves. The pattern is fine for library authors; risky for application authors who expect it "just to work."
- **Multiple parallel agent implementations still in active use** (letta V2, V3, legacy `Agent`, `LettaAgent`, batch, voice). Each has its own stop-reason vocabulary; extending the system means porting to all of them.

## Notable Absences

- **No shared cross-language execution engine.** agent-framework ships Python and .NET implementations that mirror each other by convention, not by a shared core. Pydantic-ai uses the same `pydantic_graph` for Python only; openai-agents-sdk is Python only.
- **No cycle detector / fingerprint-based loop detector** in any source. All runaways are bounded by a numeric cap and detected post-hoc. Pydantic-ai's `AgentToolUseTracker` is informational only; agent-framework's `WorkflowConvergenceException` is a last-resort guard.
- **No cycle / repeat-tool detection** in openai-agents-sdk's `AgentRunner.run`. The `AgentToolUseTracker` records tool usage but is not used to gate the loop.
- **No public description of the BSP "barrier" contract for custom channels** in langgraph. The contract is enforced via `consume()` / `finish()` (`libs/langgraph/langgraph/channels/base.py:101-121`) but not documented in `docs/`. A contributor writing a non-trivial channel will not find a guide.
- **No chaos-engineering / runbook tests for partial-failure recovery** in any of the eight sources. The closest is langgraph's `_checkpointer_put_after_previous` ordering test and pydantic-ai's `test_parallel_tool_exception_cancels_sibling_tasks`.
- **No documented SLA on Flow method call latency** in crewai. Reentrancy / cancellation only happens for OR racing.
- **No unified lifecycle hooks** for the openai-agents-sdk realtime session that mirror the turn loop's `RunHooks`. The realtime session exposes `RealtimeSessionEvent`s but does not provide a `RunHooks`-equivalent.
- **No shared schema version** for openai-agents-sdk Codex `Thread` resume. `RunState` has a version; Codex resume relies on the subprocess's own `thread_id` only.
- **No centralized "execution model" doc** in openhands. The architectural intent is distributed across `app_server/README.md`, `sandbox/README.md`, `event/README.md`, `event_callback/README.md`, and inline docstrings.
- **No public diagram of `Crew → Agent → AgentExecutor → Flow`** in crewai. The composition is implemented in code; a contributor has to read four files.

## Per-Source Notes

A short recap of each source's model and the lesson it teaches, in the order they appear in the rating table.

- **agent-framework (8/10)** — The cleanest BSP superstep story in two parallel language implementations. Lessons: name the model in code; expose `superstep_started`/`superstep_completed` events; cap iteration count; checkpoint at every superstep; separate delivery shape (`ExecutionMode`) from execution model.
- **crewai (7/10)** — Deliberately layered: Flow DAG + Plan-and-Execute + ReAct + Crew scheduler. Lessons: decorator-defined DAG with literal-typed routers is a powerful way to expose a state machine; locked state proxies for async/sync mixing; per-step isolation (`StepExecutor` owns its message list); centralized router emit registry would catch renames.
- **langgraph (8/10)** — The reference BSP implementation: `Pregel` class, channels as the only inter-actor mechanism, three-phase Plan/Execute/Update. Lessons: channels enforce the BSP contract via `consume()`/`finish()`; trigger optimization via channel-version diff; one runtime, three compiled front-ends (StateGraph, @entrypoint, create_react_agent).
- **letta (7/10)** — Bounded ReAct step loop wrapped in a six-layer lifecycle. Lessons: heartbeat flag is simpler than a typed state machine for chat products; `StopReasonType` taxonomy is essential for billing and observability; leader-elected scheduler + Redis conversation lock + sandbox dispatch form a coherent operational surface; multiple parallel agent implementations is a contributor hazard.
- **openai-agents-sdk (9/10)** — The cleanest turn step loop with discriminated `next_step` union. Lessons: encode the state machine in the type system; share the state machine between streaming and non-streaming; versioned `RunState` is the cleanest pause/resume surface; first-turn guardrail cancellation that pre-empts in-flight model work; schema-version-gated identity matching for legacy snapshots.
- **openhands (7/10)** — Layered orchestration around a hidden agent-server loop. Lessons: 8-state workflow as async generator; event-callback router with `EventCallbackProcessor` ABC; webhook/poll duality gated by `web_url`; per-sandbox session API key; pending-message queue to decouple message arrival from sandbox readiness. The hidden loop is a real cost.
- **pydantic-ai (8/10)** — Typed graph state machine wrapped in a 12-hook capability middleware. Lessons: edges inferred from return-type annotations is a clean way to declare a state machine in the type system; `ErrorMarker` recovery instead of exception groups; parallel-tool sibling cancellation on `BaseException` is a non-obvious correctness fix; v1 `BaseNode` running inside v2 `GraphBuilder` via `NodeStep` shim is a clean migration pattern; `UsageLimits` with multiple budget dimensions is the right model for paid services.
- **temporal (9/10)** — Five-model composition for distributed workflows. Lessons: transactional outbox writes events + tasks in one DB call; per-shard `RangeID` fencing; alert/monitor/mitigator triad; two-tier scheduler stack; CAS-guarded `Start/Stop`; `CHASM` engine for non-workflow "Application State Machines"; the matching service's long-poll + look-ahead goroutine pattern is reusable.

## Open Questions

These are gaps that future studies or framework evolutions should resolve.

1. **Is there a shared "barrier" contract for custom channels in langgraph, or is it enforced only by convention?** `consume()` / `finish()` are documented in code, but not in `docs/`. A contributor writing a non-trivial channel has no guide.
2. **What is the failure mode if `wrap_run` swallows a `BaseException` in pydantic-ai?** The `BaseExceptionGroup` unwrap stack ensures exception groups are not double-wrapped, but there is no stress test for this path.
3. **How does the app-server behave when the agent-server inside an openhands sandbox crashes mid-turn?** The webhook may or may not deliver a terminal event; the polling fallback may reconcile minutes later. No explicit crash handler.
4. **What enforces `StopReasonType.context_window_overflow_in_system_prompt` in letta?** The enum value exists; the code that sets it was not located in the search.
5. **What is the exact Redis conversation-lock TTL in letta?** `StreamingService.create_agent_stream` acquires the lock but the TTL is not documented in the code.
6. **What evidence exists that the Plan-and-Execute Flow in crewai is durable under failure?** `experimental/agent_executor.py` has `RecursionError` protection, replan caps, `_execution_lock`, `_finalize_lock`, and `state.execution_log`, but no documented chaos-engineering test.
7. **Is there a single canonical "execution model" doc in temporal?** The architecture documents describe slices but the cross-cutting narrative is implicit.
8. **Is the queue model the intended replacement for, or augmentation of, the workflow task state machine for non-Workflow ASMs in temporal?** Documented in `docs/architecture/chasm.md:25-51` but the runtime ownership between "executor in queue" and "executor in HSM registry" is not explicitly contrasted.
9. **How does `LiteAgent` in crewai differ from `Agent` in terms of model selection?** No comment at `lite_agent.py:1-100` explains why it is excluded from Plan-and-Execute.
10. **What is the actual model inside openhands' pinned `agent-server` image?** The repo cannot answer this; the SDK is pinned to `openhands-agent-server==1.29.0` and `openhands-sdk==1.29.0` but those packages live outside this source tree.

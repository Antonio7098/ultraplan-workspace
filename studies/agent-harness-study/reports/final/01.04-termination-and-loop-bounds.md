## Executive Summary

Across eight frameworks, termination is universally **runtime-driven** (counters, budgets, predicates) rather than model-driven; no source treats the LLM's own "I'm done" signal as a sufficient terminator without a parallel hard cap. All sources converge on three primitives: a hard step/iteration counter, a typed status/exception taxonomy that distinguishes success from exhaustion, and a config surface (often per-layer) for the user to tune the caps. The most significant divergence is **what the counter counts**: openai-agents-sdk counts "model invocations" (turns), langgraph counts "supersteps" (Pregel tick cycles), pydantic-ai counts "LLM requests" (`UsageLimits.request_limit`), crewai counts "iterations" (with a wall-clock timeout alongside), letta counts "steps" (with a rich `StopReasonType` taxonomy), temporal counts "attempts" on a per-task basis (with `RetryState` semantics), and agent-framework has three independent counters (function-call, workflow-superstep, agent-re-invocation) that compose multiplicatively.

The deepest cross-cutting gap is the **absence of stuck-loop detection**: with the partial exception of crewai's single-step repeated-tool-call guard, none of the sources compare consecutive responses, hash channel state, or otherwise detect "the model is emitting identical outputs forever" before the hard cap fires. Wall-clock enforcement is also uneven — present in langgraph (per-superstep + per-node), openai-agents-sdk (tool timeout), pydantic-ai (tool timeout), and crewai (agent-level), but absent from agent-framework's three loops and letta's main agent loop. Termination observability is consistently strong: every framework persists or surfaces a structured reason (exception type, enum, telemetry attribute), and most use a `Literal[...]` or `Enum` rather than free-form strings.

The frameworks split cleanly into three archetypes: **library-style step loops** (agent-framework, openai-agents-sdk, pydantic-ai) that give the user a single scalar bound and trust them to configure it; **layered / orchestrator frameworks** (langgraph, crewai, letta) that stack multiple orthogonal budgets (counter + wall-clock + drain + predicate); and **application / server frameworks** (temporal, openhands) where termination is event-sourced or wired through a black-box SDK and the application's job is to forward, persist, and surface.

## Core Thesis

**Termination is multi-budget, runtime-driven, and observable — but never "smart."** Every studied framework implements a hard counter as the load-bearing safety net, paired with a typed taxonomy of reasons, but none implements a generic stuck-loop detector. The most mature designs (langgraph 8/10, openai-agents-sdk 8/10) treat termination as a **layered budget model**: count + wall-clock + idle + drain, with explicit precedence rules. The least mature (openhands 5/10) is a configuration forwarder over an opaque SDK loop and has no first-class observability for "why" the loop stopped. Frameworks that lean on **forced-final-answer** backstops (agent-framework, crewai) trade off loud exhaustion for guaranteed user-visible output. Frameworks that lean on **exception-only** exhaustion (pydantic-ai, openai-agents-sdk) trade off guaranteed output for clean call-site branching. There is no source that combines all three: hard cap + forced fallback + typed exception + stuck-loop detector + persisted reason + drain API.

## Approach Models

The eight sources cluster into four distinct approach models for this dimension. The choice of model is driven primarily by whether the framework is a **library** (caller composes the loop), an **orchestrator** (framework owns the loop), or an **application server** (framework owns everything including the runtime).

### 1. Single-counter-with-backstop (openai-agents-sdk, agent-framework function-loop)

A single integer counter (`max_turns` or `max_iterations`) is incremented once per model invocation, with a final "best-effort" call if the cap is hit so the user almost always gets an answer.

- **openai-agents-sdk**: `max_turns=DEFAULT_MAX_TURNS=10` checked at `run.py:1057-1058` and `run_internal/run_loop.py:875-881`; exhaustion raises `MaxTurnsExceeded` (sync) or stores for deferred raising (streaming). The `error_handlers` TypedDict (`run_error_handlers.py:50-54`) lets users convert exhaustion into a controlled `final_output`.
- **agent-framework**: function-calling loop at `_tools.py:2567-2696` is bounded by `max_iterations=40`; on exhaustion it issues a final call with `tool_choice="none"` so the model is forced to produce text. Workflow and harness loops have separate caps (`100` and `10`).

**Why choose this model**: simple to reason about; one knob; the user always gets a final answer. **Why avoid**: coarse — a tool-heavy turn and a text-heavy turn consume the same budget.

### 2. Layered-budget (langgraph, crewai, letta, pydantic-ai, agent-framework workflow)

Three or four orthogonal caps stacked together, each catching a different failure mode.

- **langgraph**: `recursion_limit` (superstep count, default 10007) + `step_timeout` (per-superstep wall-clock) + `TimeoutPolicy` (per-node `run_timeout` + `idle_timeout`) + `RunControl.request_drain` (cooperative cancel). Each lives at a different layer of the call stack.
- **crewai**: `max_iter` + `max_execution_time` (whole-task wall-clock) + `max_rpm` (rate limit) + repeated-tool-call detection + `Flow.max_method_calls` (recursion-style cap on method dispatch).
- **letta**: `max_steps` + `_check_run_cancellation` (per-step poll) + `_check_credits` (per-step async task) + tool-rule enforcement (`is_terminal_tool`, `is_continue_tool`, `get_uncalled_required_tools`).
- **pydantic-ai**: `UsageLimits(request_limit | tool_calls_limit | *_tokens_limit)` + `AgentRetries(tools, output)` + per-tool `max_retries` + `tool_timeout`. All four budgets are independent.
- **agent-framework workflow**: `Runner(max_iterations=100)` + per-executor conditions + build-time `_validate_self_loops`.

**Why choose this model**: each layer catches a failure mode the others miss (count catches patience, wall-clock catches hangs, idle catches frozen progress, drain catches external shutdown). **Why avoid**: more knobs to reason about; user must understand which layer fires when.

### 3. Event-sourced (temporal)

Termination is an explicit append to history; six terminal events (`Completed`, `Failed`, `TimedOut`, `Terminated`, `Canceled`, `ContinuedAsNew`) each close the workflow via `Add*WorkflowEvent` and `UpdateWorkflowStateStatus(...COMPLETED)` (`mutable_state_impl.go:4834-5554`). Retry policy is a per-namespace dynamic config (`common/dynamicconfig/constants.go:2513-2524`) with `MaximumAttempts=0` meaning "no cap" by default.

**Why choose this model**: crash-safe and replayable; the termination state is reconstructible from history. **Why avoid**: high operational overhead for short-lived loops; background event loops have only the shutdown channel as a terminator.

### 4. Forwarder / SDK-opaque (openhands)

The app server does not own the loop; it forwards `max_iterations` and `max_budget_per_task` to the SDK (`openhands.sdk`) via `ConversationSettings.create_request()` and reacts to a webhook carrying `ConversationExecutionStatus` (`FINISHED`/`ERROR`/`STUCK`). The `is_terminal()` method is the only termination interface visible in this source.

**Why choose this model**: separation of concerns — the SDK owns the loop, the app server owns persistence and analytics. **Why avoid**: observability gaps (last_error captured only for analytics, not surfaced on `AppConversation`), error classification by string match (`webhook_router.py:70-90`), no mid-conversation budget updates.

## Pattern Catalog

The patterns below appear in multiple sources or are demonstrated unusually clearly by one source. Each solves a specific problem in the dimension.

### Pattern 1: Hard-cap counter as the universal safety net

**Problem**: every loop needs a terminator that the user cannot accidentally disable.

**Sources**: all eight. Each defines a constant or default (`DEFAULT_MAX_TURNS=10`, `max_iterations=40`, `recursion_limit=10007`, `DEFAULT_MAX_STEPS=50`, `MaximumAttempts=0`, `request_limit=50`, `max_iter=25`, etc.).

**Why it works**: the counter is simple to reason about, easy to expose in config, and immune to model misbehavior — the model cannot "talk its way out" of an integer check. It is the floor under every other termination mechanism.

**When to copy**: always. The cost is essentially zero (a comparison per iteration) and the safety is non-trivial.

**When it is overkill**: when the loop is genuinely meant to be unbounded (e.g., a long-lived service). Even then, a "log every N" counter is useful for observability.

**Evidence**: `run.py:1057-1058` (openai-agents-sdk); `pregel/_loop.py:600-602` (langgraph); `letta/agents/letta_agent_v3.py:328` (letta); `crewai/utilities/agent_utils.py:280-290` (crewai); `usage.py:264-365` (pydantic-ai); `mutable_state_impl.go:4834-5554` (temporal).

### Pattern 2: Typed exception/status hierarchy for termination reasons

**Problem**: callers need to branch on "why did this stop" without parsing log strings.

**Sources**: all eight.

- openai-agents-sdk: `AgentsException` → `MaxTurnsExceeded`, `ModelRefusalError`, `ToolTimeoutError`, four `*GuardrailTripwireTriggered` variants (`exceptions.py:46-173`).
- langgraph: `GraphRecursionError`, `GraphDrained`, `GraphInterrupt`, `NodeTimeoutError`, `NodeCancelledError` (`errors.py:54-241`).
- letta: `StopReasonType` with 13 values mapped to `RunStatus` via `run_status` property (`letta/schemas/letta_stop_reason.py:9-49`).
- temporal: `enumspb.RetryState` enum (`RETRY_STATE_MAXIMUM_ATTEMPTS_REACHED`, `RETRY_STATE_TIMEOUT`, etc.) persisted on `Failed`/`TimedOut` events.
- agent-framework: `WorkflowConvergenceException`, `MiddlewareTermination`, `ToolException`, `ChatResponse.finish_reason` typed as `Literal["stop", "length", "tool_calls", "content_filter"]`.
- pydantic-ai: `UsageLimitExceeded`, `UnexpectedModelBehavior`, `ContentFilterError`, `IncompleteToolCall`, `ModelAPIError` — each with a distinct message and (where applicable) `body`.
- crewai: distinguishes `AgentFinish` (success / forced-final-answer) from `TimeoutError`, `RecursionError`, `Exception` (`a2a/wrapper.py:761-819`).

**Why it works**: callers can `try/except MaxTurnsExceeded` or `match status` instead of parsing messages. The error code can also feed telemetry (`OtelAttr.FINISH_REASONS` in agent-framework, `error_code` in langgraph's `create_error_message`).

**When to copy**: always. Free-form status strings are an anti-pattern.

**Evidence**: openai-agents-sdk `exceptions.py:46-173`; langgraph `errors.py:54-241`; letta `letta_stop_reason.py:9-49`; temporal `retry.go:45-110`.

### Pattern 3: Forced-final-answer backstop on iteration exhaustion

**Problem**: when the cap is hit, the user wants *some* answer, not an empty response.

**Sources**: agent-framework and crewai.

- **agent-framework**: after `max_iterations` is exhausted, the function-calling loop issues a final model call with `tool_choice="none"` so the model is forced to produce text (`_tools.py:2664-2696`, `_tools.py:2835-2858`). The caller gets a normal `ChatResponse`, not an exception.
- **crewai**: `handle_max_iterations_exceeded` injects the `force_final_answer` prompt (`translations/en.json:47`) and makes one more LLM call that is expected to return `AgentFinish` (`utilities/agent_utils.py:293-350`). Exhaustion collapses to `AgentFinish` (graceful), in contrast to wall-clock timeout which raises `TimeoutError`.

**Why it works**: the user gets a usable answer even when the agent couldn't converge. The LLM is told "this is your last try, give your best answer" instead of just stopping.

**When to copy**: when the loop is conversational (the user is waiting for a reply). When the loop is transactional (a CI job, a database migration), prefer an exception so the caller knows it didn't complete.

**When it is risky**: the extra LLM call adds latency and cost. If the model is the one looping, the forced call may also fail or loop in the same way — agent-framework's tool-choice reset is a mitigation but not a guarantee.

**Evidence**: agent-framework `_tools.py:2664-2696`; crewai `agent_utils.py:293-350`.

### Pattern 4: Cooperative drain / cancel as a third termination class

**Problem**: hard caps and wall-clock timeouts don't catch the "operator wants to stop now" case (SIGTERM, HTTP cancellation, admin shutdown).

**Sources**: langgraph (most explicit), openai-agents-sdk, letta, openhands.

- **langgraph**: `RunControl.request_drain(reason)` writes a single attribute polled at the top of every `tick()` (`pregel/_loop.py:650-652`); on exit, `GraphDrained(reason)` is raised (`main.py:3027-3030`). Documented as the SIGTERM handling mechanism.
- **openai-agents-sdk**: `RunResultStreaming.cancel(mode="immediate" | "after_turn")` (`result.py:648-694`); honored at three sites in the run loop (`run_internal/run_loop.py:842-845`, `:1109-1112`, `:1161-1164`).
- **letta**: per-step `_check_run_cancellation(run_id)` polled at the start of every step (`letta_agent_v3.py:1031-1035`); idempotency check at the run layer (`run_manager.py:649-667`).
- **openhands**: `SendMessage` rejects non-`RUNNING` sandboxes with 409/410/503 (`app_conversation_router.py:508-529`); no explicit "stop" endpoint.

**Why it works**: the drain check is cheap (single attribute read), so it can run every tick without performance cost. It separates "the system is stopping" from "the agent exhausted its budget" — two operationally distinct events that should be observable differently.

**When to copy**: when the agent runs inside a long-lived process that may receive SIGTERM. When the agent is a one-shot CLI invocation, drain adds complexity without value.

**Evidence**: langgraph `runtime.py:79-104`, `_loop.py:650-652`; openai-agents-sdk `result.py:648-694`; letta `run_manager.py:649-667`.

### Pattern 5: Termination reason persisted at multiple storage layers

**Problem**: when the loop ends, the user (and the UI, and the analytics, and the next process) need to know *why*.

**Sources**: letta (three layers), langgraph (checkpoint + exception), openai-agents-sdk (RunState schema), temporal (history events).

- **letta**: `Step.stop_reason` (per step) → `Run.stop_reason` (terminal transition) → `agent.last_stop_reason` (filterable). `RunStatus` is derived from `StopReasonType.run_status` (`letta/schemas/letta_stop_reason.py:24-49`).
- **langgraph**: `step` is persisted in checkpoint metadata (`_loop.py:1108`); final checkpoint is flushed on exit (`_loop.py:1301-1311`). But the *reason* (out_of_steps vs draining vs done) is **not** persisted on the checkpoint — only the exception differentiates them.
- **openai-agents-sdk**: `RunState._max_turns` and `RunState._current_turn` are schema-versioned and round-trippable; the runner rehydrates them on HITL resume (`run_state.py:131-149`, `:722`).
- **temporal**: every termination is an explicit history event with full attributes; `WorkflowExecutionInfo` summary is exposed via `DescribeWorkflowExecution`.

**Why it works**: a `last_stop_reason` field on the agent lets the UI filter, retry, or display meaningfully. A checkpoint without a reason is a debugging dead-end.

**When to copy**: when the agent state outlives the process (server, long-lived service). For a one-shot CLI, exceptions are enough.

**Evidence**: letta `step_manager.py:339-414`, `run_manager.py:398-410`; openai-agents-sdk `run_state.py:131-149`, `:722`; langgraph `_loop.py:1108`, `:1301-1311`.

### Pattern 6: Cap checked before predicate (precedence rule)

**Problem**: when both a hard cap and a user predicate can stop the loop, which fires first?

**Sources**: agent-framework (`AgentLoopMiddleware._evaluate_stop` checks cap before `should_continue` at `_harness/_loop.py:653-664`), langgraph (`step > stop` checked at top of `tick()` before any node work, `_loop.py:600-602`).

**Why it works**: the cap is the safety net; the predicate is the policy. If the predicate fires first, the cap may never be tested. The explicit precedence (cap → predicate) means a runaway predicate cannot defeat the safety net.

**When to copy**: when both mechanisms exist. Document the precedence in the docstring.

**Evidence**: agent-framework `_harness/_loop.py:653-664`; langgraph `_loop.py:600-602`.

### Pattern 7: `tool_choice="required"` reset (anti-stuck guard for one specific pattern)

**Problem**: a model under `tool_choice="required"` will keep calling tools forever.

**Sources**: agent-framework (`_tools.py:2647-2652` resets after one iteration); openai-agents-sdk (`reset_tool_choice=True` default, `agent.py:367-369`, `tool_execution.py:541-549`).

**Why it works**: after one tool call, reset to default selection so the model can decide to stop naturally.

**When to copy**: when your framework exposes `tool_choice` configuration. This is a targeted, narrow defense — it does not catch arbitrary stuck patterns.

**Evidence**: agent-framework `_tools.py:2647-2652`, `:2819-2823`; openai-agents-sdk `agent.py:367-369`.

### Pattern 8: Repeated-tool-call detection (narrow stuck-loop guard)

**Problem**: a model that calls the same tool with the same args indefinitely.

**Sources**: crewai only (`tools/tool_usage.py:728-738` — `_check_tool_repeated_usage` compares `last_used_tool` for exact name + args equality). Telemetry counter `tool_repeated_usage` is incremented.

**Why it works**: the agent is forced to try a different tool via the "I tried reusing the same input, I must stop using this action input" prompt (`translations/en.json:49`).

**When to copy**: when the tool set is large enough that the model has alternatives.

**When it is risky**: false positives on legitimately identical read-only queries (e.g., `get_user(name="alice")` twice in a row). Single-step comparison (only `last_used_tool`) misses rotation-based loops where the model cycles among three tools.

**Evidence**: crewai `tools/tool_usage.py:728-738`.

### Pattern 9: Schema-versioned serialization for resume across processes

**Problem**: when an agent pauses for HITL and resumes in a different process, the budget and counter must survive.

**Sources**: openai-agents-sdk (most explicit), langgraph (via checkpoints), letta (via run_id).

- **openai-agents-sdk**: `RunState` carries `_max_turns`, `_current_turn`, `_tool_use_tracker_snapshot`, schema version `1.11`, with `null` support for `max_turns=None` (`run_state.py:131-149`, `:722`). Resume picks up `current_turn = run_state._current_turn` regardless of caller kwarg (`run.py:508`, `:1713`).

**Why it works**: the budget is part of the durable state, not a runtime parameter. HITL can last days; the loop bound cannot reset.

**When to copy**: when the agent supports HITL or long-lived pauses.

**Evidence**: openai-agents-sdk `run_state.py:131-149`, `:722`; `run.py:508`, `:1713`.

### Pattern 10: Continue-as-new as a soft loop terminator

**Problem**: a long-running workflow accumulates history until it hits hard limits; the engine should suggest a way to wrap up gracefully.

**Sources**: temporal (most explicit); langgraph (implicit via `versions_seen[INTERRUPT]` liveness check).

- **temporal**: `SuggestContinueAsNew` returns true at `HistorySizeSuggestContinueAsNew = 4*1024*1024` or `HistoryCountSuggestContinueAsNew = 4*1024` events (`common/dynamicconfig/constants.go:390-395`, `:432-437`); the user's worker is expected to call `workflow.NewContinueAsNewError(...)`. The hard backstop is `forceTerminateWorkflow` at `HistorySizeLimitError = 50*1024*1024` (`context.go:1042-1199`).

**Why it works**: the engine nudges the user to wrap up while there is still time to do it cleanly; the hard terminate is reserved for cases the user ignored.

**When to copy**: when the framework is event-sourced and has a concept of "history" that can grow unbounded.

**Evidence**: temporal `common/dynamicconfig/constants.go:390-437`; `workflow_task_state_machine.go:1454-1478`; `context.go:1042-1199`.

## Key Differences

### What converges despite different implementation choices

1. **A hard counter is universal.** Every source implements some form of step/iteration/turn/attempt counter. The variance is in *what* it counts, not in *whether* it exists.
2. **A typed reason for termination is universal.** Every source uses an enum, exception class, or `Literal[...]` type to distinguish stop reasons — none relies on free-form strings.
3. **The cap is configurable, never hard-coded into the loop body.** Defaults are constants (`DEFAULT_MAX_TURNS=10`, `recursion_limit=10007`, `max_iterations=40`) but the loop reads from configuration.
4. **`finish_reason` is carried but rarely interpreted as a stop signal.** Agent-framework (`_types.py:1648-1672`), pydantic-ai (`messages.py:114-120`), and langgraph all surface `finish_reason` to the user but only interpret `length` and `content_filter` as control signals; `stop` and `tool_call` are not used as terminators.
5. **Final-state observability is uniformly strong.** Every framework populates a structured result object or terminal event with enough context to diagnose the stop.

### Why sources diverge based on product shape, maturity, user needs

1. **Library vs application**: openai-agents-sdk, pydantic-ai, and agent-framework are libraries that must defer termination policy to the caller; they expose `max_turns` / `request_limit` / `max_iterations` but do not enforce wall-clock on the agent loop itself. Langgraph is also a library but more opinionated — its default `recursion_limit=10007` is "essentially unbounded for sane runs" (a deliberate ergonomic choice) but the per-superstep `step_timeout` is a hard safety net. Letta and openhands are application servers that own the runtime and therefore must enforce wall-clock, sandbox status, and per-namespace budgets.

2. **One-shot vs resume-capable**: openai-agents-sdk supports HITL resume and therefore schema-versioned `RunState`; langgraph supports checkpoint-based resume and therefore flushes the final checkpoint on every termination path; letta supports cancellation and therefore idempotency checks at the run layer. Crewai, pydantic-ai, and agent-framework are one-shot and therefore don't need this machinery.

3. **Graph-of-nodes vs step-loop**: langgraph and pydantic-ai express the loop as a graph of nodes with a driver (`tick()` / `next()`). The graph structure makes the loop boundary implicit — termination is when the driver returns nothing or when the graph reaches an `End` node. agent-framework and letta use a traditional `while` / `for i in range(max_steps)` loop. The graph approach makes interrupt (langgraph's `interrupt_before/after`, pydantic-ai's `CallDeferred`/`ApprovalRequired`) natural but makes "I want a hard step counter" awkward — pydantic-ai solves this with `UsageLimits.request_limit`; langgraph with `recursion_limit`.

4. **Event-sourced vs stateful**: temporal's termination model is fundamentally different — workflows are not step-bounded; the loop terminator is the next history event. This makes Continue-as-new natural (it is just another history event) and makes stuck-detection irrelevant (the workflow is doing what history says). It also makes "the user changed their mind" termination a different category (terminate vs cancel vs timeout).

5. **SDK-forwarder vs full-stack**: openhands is a unique case — the loop is owned by `openhands.sdk` (imported but not present in the source tree). The app server's contribution is configuration forwarding and webhook-driven analytics. This creates a clean separation but also an observability gap — the app server cannot tell you *why* the loop stopped beyond the enum string.

6. **Per-tool vs global budget**: pydantic-ai has the most layered budget model (`UsageLimits` for tokens/requests, `AgentRetries` for retries, per-tool `max_retries`); langgraph is per-node (`TimeoutPolicy`); openai-agents-sdk is per-agent (`max_turns`). The choice reflects whether the framework expects the same budget to apply to all components or each component to have its own envelope.

7. **Multi-version drift vs single code path**: letta has three concurrent agent implementations (`LettaAgent`, `LettaAgentV2`, `LettaAgentV3`) each with its own termination copy-paste logic. The drift is a documented liability. Openai-agents-sdk and pydantic-ai keep a single implementation with async/sync mirrored logic.

## Tradeoffs

### Library-style single counter (openai-agents-sdk, agent-framework)

- **Benefit**: simple to reason about, one knob, always get an answer (in the forced-final-answer case).
- **Cost**: coarse — a tool-heavy turn consumes the same budget as a text-heavy turn; user must size for the worst case.
- **Best-fit context**: conversational agents where the user is waiting and a rough answer is acceptable.
- **Failure mode**: a model that loops on tool calls exhausts the budget before producing output; no semantic detection before the cap.
- **Alternative**: layered budgets (langgraph, crewai, pydantic-ai) catch the failure mode at the cost of more knobs.

### Layered budgets (langgraph, crewai, letta, pydantic-ai)

- **Benefit**: each layer catches a failure mode the others miss (count catches patience, wall-clock catches hangs, idle catches frozen progress, drain catches shutdown).
- **Cost**: more knobs to reason about; user must understand which layer fires when; configuration drift across layers.
- **Best-fit context**: long-lived or production-grade agents where downtime is expensive.
- **Failure mode**: the budgets may interact in surprising ways (langgraph's `step_timeout` is per-superstep, so a run with 10006 fast steps is bounded by `recursion_limit`; a run with one slow step is bounded by `step_timeout`).
- **Alternative**: single counter + exception (openai-agents-sdk) — simpler but coarser.

### Event-sourced (temporal)

- **Benefit**: crash-safe; termination is reconstructible from history; Continue-as-new is a natural soft terminator.
- **Cost**: every termination must persist an event; high operational overhead for short loops; background event loops have only the shutdown channel as a terminator.
- **Best-fit context**: long-running workflows (hours to years) with crash-safety requirements.
- **Failure mode**: a misconfigured Continue-as-new spin can run forever within the run-timeout.
- **Alternative**: step-loop frameworks are simpler but lose replay semantics.

### Forwarder (openhands)

- **Benefit**: clean separation between loop implementation and persistence/analytics.
- **Cost**: observability gaps (last_error captured only for analytics, not surfaced); error classification by string match; no mid-conversation budget updates.
- **Best-fit context**: when the loop is owned by a trusted SDK and the application's role is plumbing.
- **Failure mode**: if the SDK is updated to emit a new terminal state, the app server's `is_terminal()` check may miss it until updated.
- **Alternative**: own the loop in the application server (letta, langgraph as application).

### Forced-final-answer vs exception-only on exhaustion

- **agent-framework / crewai**: exhaustion collapses to a "best effort" answer via one more LLM call. The user gets *something*.
- **pydantic-ai / openai-agents-sdk**: exhaustion raises a typed exception (`UsageLimitExceeded` / `MaxTurnsExceeded`). The user gets *an error*.
- **Best-fit context**: forced-final-answer fits conversational agents where partial output is better than no output; exception-only fits transactional agents where partial output is worse than no output.

### Multiplicative vs additive composition of caps

- **agent-framework**: three independent counters (`max_iterations` on three loops) compose multiplicatively — a complex agent bounded by `5 × 40 × 100` worst-case.
- **langgraph / temporal**: budgets are orthogonal (steps vs wall-clock vs drain) — they don't compose, they each catch different failures.
- **pydantic-ai**: layered but non-composing (`UsageLimits.request_limit` vs per-tool `max_retries` — both can fire, neither depends on the other).

## Decision Guide

Use this guide when choosing how to bound loops in your own agent.

### 1. Library or application?

- **Library** (caller composes): choose a **single counter** (openai-agents-sdk model). The caller will size it. Do not enforce wall-clock — they may want unbounded.
- **Application server** (you own the runtime): choose **layered budgets** (langgraph model). You must defend against runaway loops, hung workers, and operator shutdown.

### 2. One-shot or HITL-resume?

- **One-shot**: a single exception type or status field is enough.
- **HITL-resume**: persist the cap and current counter in durable state. Use a schema version. Test round-trip (`openai-agents-sdk` `test_max_turns_none_round_trips`).

### 3. Conversational or transactional?

- **Conversational** (user waiting for a reply): implement **forced-final-answer** on exhaustion. The user is better served by a partial answer than an exception.
- **Transactional** (CI job, data migration): raise an exception. The caller needs to know it didn't complete.

### 4. Per-component or single envelope?

- **Single envelope** (one budget for the whole agent): use a single counter (openai-agents-sdk).
- **Per-component** (each tool, node, or step has its own limit): use layered budgets (pydantic-ai `UsageLimits` + `AgentRetries`, langgraph `TimeoutPolicy`).

### 5. Need stuck-loop detection?

If yes: **none of the eight sources fully solve this**. The closest patterns are:
- crewai's `_check_tool_repeated_usage` for exact-args repetition (single-step only).
- agent-framework's `tool_choice="required"` reset (one specific pattern).
- pydantic-ai's `UsageLimits.request_limit` as a hard backstop (no semantic detection, just a cap).

You will need to implement your own hash-of-last-response or cosine-similarity check before the cap fires.

### 6. Need operator-controlled termination?

Implement cooperative drain (langgraph `RunControl.request_drain`, openai-agents-sdk `cancel(mode)`). Poll at the top of each loop iteration. Distinguish "drained" from "exhausted" in the exception type and the persisted status.

### 7. Need audit trail?

Persist the stop reason at multiple layers (letta model): per-step, per-run, and on the agent itself. Use an enum so the UI can filter. Surface the reason in the API response, not just in analytics.

## Practical Tips

### Patterns to copy

1. **A typed `Literal[...]` or `Enum` for stop reasons, persisted at every relevant layer.** Free-form strings are an anti-pattern. Langraph's `Literal[...]` status field catches typo bugs at type-check time; letta's `StopReasonType.run_status` property centralizes success/failure mapping.
2. **A hard cap as the floor, with a forced-final-answer or handler hook as the ceiling.** agent-framework and crewai demonstrate that exhaustion can be graceful without being silent.
3. **A cooperative drain API for operator shutdown.** langgraph's `RunControl.request_drain` is the cleanest pattern — single-attribute write, polled at superstep boundaries, raises a distinct exception (`GraphDrained`).
4. **Pre-flight budget check + post-response budget check** (pydantic-ai `UsageLimits.check_before_request` + `check_tokens`). Pre-flight catches "this will exceed the limit" before the model call; post-response catches "the model just exceeded the limit." Both are needed because token counts are not known until the response arrives.
5. **Per-tool retry counter on the run context** (pydantic-ai `RunContext.retries[name]`). This lets a single tool fail up to N times without affecting the global budget.
6. **`count_tokens_before_request` where the provider supports it** (pydantic-ai). Avoids wasted model calls on requests that are guaranteed to exceed the limit.
7. **Schema-versioned serialization for resume** (openai-agents-sdk `RunState`). Allow `null` for "disable the cap" and round-trip it through the schema.
8. **Test coverage for boundary conditions**: `max_turns=0`, `max_turns=1`, `max_turns=None` (openai-agents-sdk `tests/test_max_turns.py`). Resume with exhausted budget. Cancel during streaming.
9. **`tool_choice="required"` reset as a defensive default** (agent-framework, openai-agents-sdk). Prevents one specific runaway pattern.
10. **Document the precedence between cap and predicate** (agent-framework `_harness/_loop.py:653-664`). When both can fire, the cap must fire first.

### Patterns to avoid or delay until needed

1. **Building your own stuck-loop detector from scratch.** It is hard to get right (false positives, semantic similarity). Use a cap as the safety net; layer a detector only when you see the pattern in production.
2. **Hard-coding termination behavior into the loop body.** Always read from configuration. Defaults are fine; constants in the loop are not.
3. **Returning success-shaped results for exhausted runs without an exhaustion marker.** agent-framework's function loop returns a normal `ChatResponse` on exhaustion — the caller cannot tell success from failure without reading logs. Stamp the response or raise an exception.
4. **String-match error classification** (openhands `webhook_router.py:70-90`). It will mis-classify on minor wording changes. Demand a structured payload from the SDK.
5. **Maintaining parallel agent implementations** (letta V1/V2/V3). The drift is a maintenance liability and the stop-reason semantics are subtly different across versions.
6. **`max_iterations=None` without an explicit "disable the cap" semantic** (openai-agents-sdk's `max_turns=None` is intentional and tested; letta's `max_steps=0` falls into the empty range and stamps `end_turn`, which is surprising).
7. **Per-iteration wall-clock budgets that subdivide the agent-level budget.** crewai's `max_execution_time` is a thread/async barrier around the whole task, not per-iteration. Subdividing means a single hung tool can consume the whole budget.

### Decision rules

- **If the loop is conversational and one-shot**: use openai-agents-sdk's model — single counter, typed exception, optional handler hook.
- **If the loop is a long-lived agent or has resume**: use langgraph's model — layered budgets, checkpoint persistence, cooperative drain.
- **If the loop is event-sourced or has a "history" concept**: use temporal's model — explicit terminal events, Continue-as-new as a soft terminator, force-terminate as a backstop.
- **If the loop is owned by a separate SDK**: use openhands' model — forward configuration, surface the SDK's terminal state, persist it at every layer, but do not try to re-implement the loop.
- **If the loop must support per-tool retry budgets**: use pydantic-ai's model — `UsageLimits` + `AgentRetries` + per-tool `max_retries` + `tool_timeout`.
- **If the loop must catch repeated-tool calls**: copy crewai's `_check_tool_repeated_usage` — but extend it to a sliding window.

### Caution signs that termination design is becoming brittle

1. **The same word (`max_iterations`, `max_steps`, `max_turns`) means different things in different parts of the framework.** agent-framework has this problem — three "max_iterations" with multiplicative composition. Rename or document the layering explicitly.
2. **The user cannot distinguish "the loop said stop" from "the model said stop" without reading logs.** Add a `termination_reason` field to the result object, or raise a typed exception.
3. **Exhaustion is communicated only by an exception message.** Use an enum; allow `match`/`switch`/`except` on the type.
4. **The default counter is so high that production loops never trip it.** langgraph's 10007 default is the documented "no, really, no limit" choice; it should be lowered via env var in production.
5. **Background event loops only exit on shutdown.** Temporal's queue readers and shard ownership loops fit this pattern. A bug in the loop body can hang the process.
6. **Multiple parallel implementations of the loop exist.** Letta V1/V2/V3 demonstrates that the drift between them is a maintenance hazard.
7. **No test exercises the boundary condition of `cap=0` or `cap=1`.** openai-agents-sdk tests `max_turns=None` and `max_turns=0` explicitly; most frameworks do not.
8. **Wall-clock timeout is per-iteration instead of per-run.** A single hung LLM call can consume the whole budget.
9. **The forced-final-answer fallback is undocumented.** agent-framework's final non-tool call is in the source but not in the docstring.
10. **Termination reason is not persisted on the durable state.** langgraph's checkpoint lacks the `out_of_steps` vs `done` distinction; the user must look at the exception.

## Anti-Patterns / Caution Signs

1. **Three different `max_iterations` with multiplicative composition** (agent-framework): `_tools.py:90` (40), `_workflows/_const.py:4` (100), `_harness/_loop.py:117` (10). A user setting all three gets a worst-case of `40 × 10 × 100 = 40,000` model invocations.
2. **String-match error classification** (openhands `webhook_router.py:70-90`): a model emitting "budget" inside a longer message gets mis-classified as `budget_exceeded`.
3. **`max_iterations=0` semantics are surprising** (crewai): `has_reached_max_iterations(0, 0)` returns `True` immediately, so the very first loop iteration forces the final-answer path.
4. **Function-loop exhaustion returns a normal `ChatResponse`** (agent-framework `_tools.py:2664-2696`): the caller cannot distinguish "the model gave up" from "the model answered" without reading logs.
5. **Background event loops only exit on shutdown** (temporal queue readers, shard ownership loops): a bug in the loop body can hang the process; there is no in-loop iteration cap.
6. **`defaultMaximumAttempts = 0` means retry forever** (temporal `common/backoff/retrypolicy.go:22`): any caller that builds a retry policy without explicitly setting a max will retry indefinitely (only an expiration interval caps it).
7. **`recursion_limit=10007` is effectively "no limit for sane runs"** (langgraph `_internal/_config.py:32`): production must lower this via `LANGGRAPH_DEFAULT_RECURSION_LIMIT` env var; the default is too high to catch runaway loops.
8. **`request_limit=50` default is generous** (pydantic-ai `usage.py:273`): a runaway loop can burn a non-trivial amount of money before tripping. The recommended pattern (`docs/agent.md:682`) is "set this yourself for production."
9. **No global "max steps" knob** (pydantic-ai): the framework is opinionated about per-tool / per-output retries / per-request limits but does not provide a single "max model turns" knob. Users typically layer `request_limit=…` on top of `retries=…` to get an effective ceiling.
10. **Termination reason not persisted on checkpoint** (langgraph `_loop.py:1108`): `step` is in metadata, but the *reason* (out_of_steps vs draining vs done) is not. After exhaustion, `get_state` returns the same shape as a normal checkpoint.
11. **`max_iterations` and `max_budget_per_task` apply only at conversation start** (openhands): no evidence of mid-conversation updates taking effect; the SDK would have to honor a live update path.
12. **Concurrent agent implementations drift** (letta V1/V2/V3): each carries its own stop-reason copy-paste with subtle differences (V3 catches `LLMEmptyResponseError`, V2 doesn't).
13. **`max_iterations` is hard-coded into the request schema, not the agent** (letta `schemas/letta_request.py:43-46`): a caller that sets `max_steps=1000` can keep the worker occupied far longer than the agent's intended workload; no per-agent ceiling enforced server-side.
14. **Single-step repeated-tool detection misses rotation-based loops** (crewai `tools/tool_usage.py:728-738`): an agent that rotates among three tools indefinitely is not detected.
15. **Forced-final-answer LLM call may itself fail** (crewai `agent_utils.py:334-340`): `handle_max_iterations_exceeded` can raise `ValueError("Invalid response from LLM call - None or empty.")` — best-effort, not guaranteed.

## Notable Absences

The following are gaps that no source in the study addresses — useful "what to invent" notes for someone designing a new framework.

1. **Generic stuck-loop detection (hash / cosine similarity) at the LLM-response level.** Every source relies on the hard cap. The closest is crewai's exact-args repeated-tool detection, which is narrow and single-step.
2. **A global "max model turns" knob that combines request count + retry count.** pydantic-ai is closest (`UsageLimits.request_limit` + `AgentRetries`), but the two budgets are independent and the user must reason about both.
3. **Wall-clock timeout on the agent loop itself** (as opposed to per-LLM-call). agent-framework has none; letta has none; openai-agents-sdk has `ToolTimeoutError` (per-tool only).
4. **Exhaustion marker on the success-shaped result.** agent-framework returns a normal `ChatResponse` on exhaustion; the user must read logs. A `loop_exhausted: bool` field on the response would close this.
5. **Termination reason persisted on the durable state.** langgraph persists `step` but not the reason; letta is the only source that persists the reason at multiple layers (`Step.stop_reason` → `Run.stop_reason` → `agent.last_stop_reason`).
6. **Cooperative drain mid-superstep.** langgraph polls only at tick boundaries; a slow node cannot be interrupted by drain.
7. **Per-run wall-clock budget** (as opposed to per-superstep). langgraph has `step_timeout` per-superstep and `TimeoutPolicy.run_timeout` per-node, but no per-run budget.
8. **Cross-resume budget aggregation.** openai-agents-sdk's `RunState` carries `_max_turns` but the budget resets on each `invoke`/`stream` call; there is no aggregate cap across resumes.
9. **Build-time cycle detection.** agent-framework warns on self-loops at build time but cannot detect data-dependent cycles. A full SCC analysis would catch more.
10. **Pre-flight tool-call budget enforcement** (drop parallel calls mid-batch when budget is hit). agent-framework documents that `max_function_calls` is checked post-batch.
11. **Stop reason surfaced on the API response.** openhands captures `last_error` only for analytics, not on `AppConversation`. The user-visible state is the enum string, not the message.
12. **Mid-conversation budget updates.** openhands settings apply only at conversation start; no live update path.
13. **Per-tool turn counters distinct from global turn counter.** openai-agents-sdk uses a single `max_turns`; a user that wants "max 3 calls to `web_search`" cannot express this.
14. **Per-namespace / per-tenant budget isolation.** temporal has per-namespace retry policies; the agent frameworks do not.
15. **A "soft stuck" signal for application code to opt into.** langgraph exposes `IsLastStep` / `RemainingSteps` but only `create_react_agent` consults them; generalizing the pattern would let applications decide their own exit semantics.

## Open Questions

1. **Is there a defensible reason that no framework implements stuck-loop detection?** Is the cost (false positives, semantic-similarity design) higher than the benefit (catching the `infinite_retry_tool` pattern)? Or is it that the hard cap is "good enough" for the current generation of agents?
2. **Should `recursion_limit` be a per-run cap or a per-resume cap?** Langgraph's `RunState` does not aggregate across resumes; a long thread can hit `recursion_limit` per call indefinitely. Should the framework track cumulative steps?
3. **Should the forced-final-answer pattern (agent-framework, crewai) be combined with the typed-exception pattern (openai-agents-sdk, pydantic-ai)?** Both have merit; the first guarantees output, the second guarantees clean call-site branching. Can a single design do both — issue one more LLM call *and* stamp a `termination_reason` on the result?
4. **Should wall-clock timeout be per-iteration or per-run?** Crewai's `max_execution_time` is per-run (whole task); langgraph's `step_timeout` is per-superstep; agent-framework's loops have no wall-clock. The choice has operational consequences (a hung tool can consume the whole budget in crewai's model; a fast-but-many-steps run can exceed wall-clock without tripping langgraph's `step_timeout`).
5. **What is the right default for `recursion_limit` / `max_turns` / `max_iterations`?** Langgraph's 10007 is "effectively unbounded"; openai-agents-sdk's 10 is conversational; pydantic-ai's 50 is generous. Is there a principled way to choose a default that is "high enough to not surprise users, low enough to catch runaway loops"?
6. **Should termination reasons be part of the schema or part of the exception?** Openai-agents-sdk persists `max_turns` in `RunState`; letta persists `stop_reason` at three layers; langgraph persists `step` but not the reason. Is the cost of schema versioning worth the benefit of cross-resume budget enforcement?
7. **Is string-match error classification acceptable when the SDK contract is not yet stable?** Openhands' classifier is fragile; the alternative is to demand a structured payload from the SDK. At what point does the SDK evolve enough that the app server can rely on a structured payload?
8. **What is the relationship between `max_iterations` and `max_budget_per_task` in OpenHands?** Are they multiplicative (a run hits whichever is lower) or additive (a run must stay under both)? The source does not clarify.

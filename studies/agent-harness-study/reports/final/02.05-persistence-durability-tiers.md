## Executive Summary

All six sources organize persistence as a stack of tiers spanning in-memory caches through relational databases and external object storage, but they diverge sharply on three axes: whether the tiers are *operator-visible*, whether defaults are *durable*, and whether they *fail loudly* when misconfigured. LangGraph (`8/10`) and Temporal (`8/10`) ship the most explicit durability contracts — LangGraph via a typed `Durability` literal that decouples write timing from storage backend, and Temporal via named `DefaultStore` / `VisibilityStore` / `Archival` slots with startup schema checks. Langfuse and Letta sit at `7/10` with multi-store models that are durable-by-default but rely on implicit operator knowledge (env flags, factory wiring) rather than a single durability map. Agent-framework (`6/10`) and OpenHands (`6/10`) expose rich pluggable interfaces but ship non-durable defaults and include silent no-op stores that can mask misconfiguration.

The dominant failure mode across the sample is "silent no-op persistence": in-memory stores installed without warning (Agent-framework `HarnessAgent`, OpenHands `InMemoryFileStore`, Letta `NoopAsyncRedisClient`), per-process caches that look like distributed caches (Langfuse `ClickhouseReadSkipCache`, Temporal `XDC` event blob cache), or async write paths that drop work on SIGKILL (LangGraph `durability="async"`, Temporal Elasticsearch bulk processor, Langfuse `ClickhouseWriter`). Recovery is well-engineered in every framework (writes are batched, retried, and flushed on shutdown), but the recovery logic is *about* the durable tier — it does nothing for state that lives only in memory. Operators can answer "can I tell which state is production-durable?" only if they read every config schema, factory, and storage interface; none of the six frameworks offers a single runtime query that enumerates tier membership.

## Core Thesis

Durability tiering in agent harnesses is universally a *layered* problem (in-memory → file → relational → distributed), but the *observable* structure of the layers varies from explicit (LangGraph `Durability` literal, Temporal named stores) to implicit (Agent-framework and Langfuse relying on store-class names). The strongest designs pair a typed tier identifier with a forced choice — let an operator pick "in-memory only" but make that choice declared and visible at startup, rather than letting a fallback silently take effect. None of the frameworks achieves "mature, observable, proven under failure" (`9-10`) because none exposes a single durability matrix that operators can read or query.

## Approach Models

The six sources cluster into three approach models for organizing persistence tiers.

### Explicit Tiering (`8/10` class)

**LangGraph** decouples *when* writes happen from *where* they land. `Durability = "sync" | "async" | "exit"` (`libs/langgraph/langgraph/types.py:87`) controls timing against a separately chosen `BaseCheckpointSaver` (InMemory / Sqlite / Postgres / custom). Operators see both knobs (`graph.invoke(durability="sync", checkpointer=...)`) and can reason about them independently. The `BaseStore` contract adds TTL plumbing (`libs/checkpoint/langgraph/store/base/__init__.py:545-565`) and ships a per-backend sweeper (`libs/checkpoint-postgres/langgraph/store/postgres/base.py:807-879`).

**Temporal** names its tiers in config: `Config.Persistence.DefaultStore`, `VisibilityStore`, `SecondaryVisibilityStore`, `Archival` (`common/config/config.go:256-268`, `common/config/config.go:493-558`). Each tier has explicit SQL/Cassandra/Elasticsearch backend factories (`common/persistence/client/fx.go:181-214`, `common/persistence/visibility/factory.go:237-301`) plus a schema-version compatibility check at startup (`temporal/fx.go:912-925`). Shard recovery uses persisted `RangeId` to acquire ownership (`service/history/shard/context_impl.go:1164-1197`).

### Implicit Multi-Store (`7/10` class)

**Langfuse** wires five storage systems — Postgres for control plane (`packages/shared/prisma/schema.prisma:9-14`), ClickHouse for telemetry (`packages/shared/clickhouse/migrations/clustered/0001_traces.up.sql:23`), S3 for raw event payloads (`packages/shared/src/env.ts:232`), Redis for queues and caches, plus an in-process LRUCache (`packages/shared/src/server/cache/localCache.ts:18-164`). There is no `Durability` literal; operators read the `LANGFUSE_*` env schema (`packages/shared/src/env.ts:1-516`) to know what survives what.

**Letta** has a single relational DB as its durability spine (`letta/server/db.py:21`, `:58`) plus an optional Redis coordinate layer (`letta/data_sources/redis_client.py:663`) and pluggable provider-trace backends (`letta/services/provider_trace_backends/factory.py:30`). Postgres vs SQLite is inferred from env presence (`letta/settings.py:492`), not declared.

### Pluggable Tier-by-Tier (`6/10` class)

**Agent-framework** offers the largest stack — `AgentSession` (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentSession.cs:39`), `ChatHistoryProvider` (InMemory / Cosmos / Valkey / Vector / Mem0), `ICheckpointStore` (InMemory / File / Cosmos), and Durable Task entities (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:13`). The interfaces are clean, but the `HarnessAgent` default falls into the lowest durability tier (`dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgent.cs:193-199`), and durability is opt-in.

**OpenHands** has three orthogonal abstractions (`FileStore`, `EventServiceBase`, SQLAlchemy `AsyncSession`) with env-driven backend selection (`openhands/app_server/file_store/__init__.py:8-21`, `openhands/app_server/services/db_session_injector.py:167-241`). The default config is local-only and silently non-durable across redeploys.

## Pattern Catalog

### Pluggable store interface with backend factory

Every source except Letta defines an abstract store contract and a factory that selects a concrete backend at module load or DI time.

- LangGraph: `BaseCheckpointSaver` and `BaseStore` with `InMemorySaver`, `SqliteSaver`, `PostgresSaver` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:176-580`).
- Temporal: `DataStoreFactory` returning core store interfaces (`common/persistence/persistence_interface.go:30-51`); SQL/Cassandra/Elasticsearch factories in `common/persistence/sql/factory.go:43-151`, `common/persistence/cassandra/factory.go:29-126`.
- Agent-framework: `ChatHistoryProvider`, `ICheckpointStore`, `AgentFileStore` (`dotnet/src/Microsoft.Agents.AI.Abstractions/ChatHistoryProvider.cs:51-477`, `dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/ICheckpointStore.cs:12-46`).
- OpenHands: `FileStore` and `EventServiceBase` with local/S3/GCS/in-memory (`openhands/app_server/file_store/files.py:7-30`, `openhands/app_server/event/event_service_base.py:31-52`).
- Langfuse: `StorageService` with five concrete implementations (`packages/shared/src/server/services/StorageService.ts:222-255`).
- Letta: `StorageBackend` for object storage (`letta/services/memory_repo/storage/base.py:7-82`); relational layer is a single SQLAlchemy engine.

**Why it works**: A single contract + many backends keeps the call-site stable while letting operators swap durability. **When to copy**: when the application must run both locally (in-memory file) and in production (managed DB/object store). **When overkill**: a small library whose only deployment target is a known production backend. **Evidence**: every source in the sample, see also `patterns/SmartObjectLifecycle.md` for the wider object-vs-row split.

### Silent no-op fallback for optional subsystems

Half the sources ship a silent-degrading implementation when a peer system is unavailable.

- Agent-framework: `Purview` cache no-ops when no `IDistributedCache` is registered (`dotnet/src/Microsoft.Agents.AI.Purview/CacheProvider.cs:25`).
- Langfuse: `evalJobConfigCache` returns `false` if Redis is absent (`packages/shared/src/server/evalJobConfigCache.ts:27-44`); `recentProjectMarks` is an in-process `Map` that is never persisted (`packages/shared/src/server/redis/ingestionFailureTracking.ts:14`).
- Letta: `NoopAsyncRedisClient` lets locks return `None` and streams return empty (`letta/data_sources/redis_client.py:562-660`).
- OpenHands: `get_file_store` returns `InMemoryFileStore` for any unknown `file_store_type` with no warning (`openhands/app_server/file_store/__init__.py:20-21`).

**Why it works**: zero-friction local development. **When it works**: dev/test environments where the operator knows the limitation. **When risky**: production deployments where no human reads the warning. **Caution sign**: a code path whose only failure mode is `return None` / `return false` / `return []` with a single `log.warn` call.

### Schema-versioned migrations at startup

Long-lived frameworks reconcile schema on boot.

- Temporal: `VerifyVersion` enforces schema compatibility at startup (`temporal/fx.go:912-925`, `common/persistence/schema/version.go:9-30`; SQLite version check is a no-op TODO: `common/persistence/sql/sqlplugin/sqlite/db.go:119-124`).
- LangGraph: Postgres `MIGRATIONS` list applied by `setup()` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:43-108`, `…/store/postgres/base.py:1088-1160`).
- Letta: `alembic upgrade head` gates container startup (`letta/server/startup.sh:54-61`); 167 migrations.
- OpenHands: `command.upgrade(cfg, 'head')` in `OssAppLifespanService.__aenter__` (`openhands/app_server/app_lifespan/oss_app_lifespan_service.py:14-35`).
- Agent-framework: explicit `schemaVersion` field with hard-fail on major mismatch (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateJsonConverter.cs:38-41`).

**Why it works**: forces a contract between code version and data version; prevents the "code thinks the schema is v3, data is v1" footgun. **When to copy**: any store whose schema outlives a release. **When overkill**: a transient cache.

### Atomic local writes with temp + fsync + rename

The few frameworks that write to local files serialize through a single helper that survives concurrent writers.

- OpenHands: `LocalFileStore.write` does pid/thread temp file + `f.flush()` + `os.fsync()` + `os.replace()` (`openhands/app_server/file_store/local.py:31-42`).
- Agent-framework: `FileSystemJsonCheckpointStore` line-appends to the index and flushes before the write returns (`dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/FileSystemJsonCheckpointStore.cs:148-160`).
- Temporal: filestore history archiver uses JSON history files under a URI directory (`common/archiver/filestore/history_archiver.go:171-183`).

**Why it works**: avoiding torn writes is the cheapest correctness win available; standard pattern from `man rename(2)`. **When to copy**: any non-transactional file write. **When overkill**: transactional stores that already have atomic single-row updates.

### TTL sweepers as opt-in background threads

Two sources ship TTL-aware stores where the sweeper is a separately-started thread rather than auto-spawned.

- LangGraph: `PostgresStore.start_ttl_sweeper()` / `SqliteStore.start_ttl_sweeper()` are opt-in (`libs/checkpoint-postgres/langgraph/store/postgres/base.py:807-879`, `libs/checkpoint-sqlite/langgraph/store/sqlite/base.py:977-1046`).
- Langfuse: ingestion-failure tracking uses a Redis sorted set keyed by expiry score (`packages/shared/src/server/redis/ingestionFailureTracking.ts:82-87`); no in-process TTL sweeper needed because Redis handles eviction.
- Agent-framework: Cosmos TTL via `MessageTtlSeconds` (`dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs:88`); Durable Task TTL via self-signal (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:197-215`).

**Why it works**: TTL is the cleanest "what survives how long" answer, but each store needs its own sweeper because the storage engine does not always do it. **Caution sign**: a sweeper that the operator must remember to start.

### Write-ahead log via staged S3 upload

The deepest durability pattern in the sample belongs to Langfuse: the API uploads the full event payload to S3 first, and only enqueues the S3 key + scope in Redis (`packages/shared/src/server/ingestion/processEventBatch.ts:201-518`). This means Redis loss does not lose events, and the BullMQ payload stays small. The trade-off is that an S3 outage stalls ingestion (`packages/shared/src/env.ts:293-298`); the mitigation is `markProjectS3Slowdown` (`packages/shared/src/server/redis/s3SlowdownTracking.ts:39-56`).

### Explicit Durability knob decoupled from storage tier

LangGraph's `Durability = "sync" | "async" | "exit"` (`libs/langgraph/langgraph/types.py:87`) is a unique pattern. The default is "async", which submits checkpoints to a `BackgroundExecutor` (`libs/langgraph/langgraph/pregel/_executor.py:40-223`); "sync" blocks until the write completes (`libs/langgraph/langgraph/pregel/main.py:3002-3003`); "exit" defers writes to context manager `__exit__` (`libs/langgraph/langgraph/pregel/_loop.py:1301-1311`). The storage tier is selected independently via the `BaseCheckpointSaver` passed at compile time. **This is the cleanest abstraction in the sample**: operators tune write latency without changing durability backend, or vice versa.

### RangeID-based shard ownership as durability anchor

Temporal's `RangeId` per shard is incremented in persisted storage on each ownership transfer (`service/history/shard/context_impl.go:1164-1197`). Combined with persisted queue state restored into in-memory scheduling state (`service/history/shard/context_impl.go:1811-1846`), this is what makes multi-instance safe restart possible. No other source has an equivalent primitive.

## Key Differences

### Default durability at first-run

| Source | First-run default |
|--------|-------------------|
| agent-framework | In-memory chat history + cwd-relative files (`dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgent.cs:193-318`) |
| langfuse | Postgres + ClickHouse + S3 + Redis (production stack if configured) |
| langgraph | Async durability + `InMemorySaver` / `InMemoryStore` (lowest durability) |
| letta | Postgres if pg env present, else SQLite (semi-durable via file) |
| openhands | SQLite file + local filesystem events under `~/.openhands` (single-host only) |
| temporal | Cassandra / MySQL / PostgreSQL only (SQLite is dev-only) |

This split is the single biggest predictor of "operator can tell which state is durable at first glance." Langfuse and Temporal refuse to start in a non-durable mode for production; the others accept the dev default and trust the operator to swap backends.

### Operator visibility into tier membership

- **Visible**: LangGraph's `Durability` value is printed in logs and validated at every entry point (`libs/langgraph/langgraph/pregel/main.py:2817-2820`); Temporal's `Config.Persistence` is a struct that can be printed at startup (`common/config/config.go:256-268`).
- **Reachable by reading config**: Langfuse `LANGFUSE_*` env vars (`packages/shared/src/env.ts:1-516`); Agent-framework `HarnessAgentOptions` (`dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgentOptions.cs:130-257`).
- **Hidden in factory wiring**: OpenHands' `get_file_store` selector (`openhands/app_server/file_store/__init__.py:8-21`) is reachable only by code inspection; Letta's `database_engine` (`letta/settings.py:492`) is inferred rather than declared.

### Single durability knob vs per-subsystem knobs

Only Langfuse lacks any top-level durability knob — every concern is tuned via `LANGFUSE_*` env vars individually. The richest knobbed designs are LangGraph (per-call `durability`) and Temporal (per-store type). Agent-framework and OpenHands have per-store config but no global "make everything durable" override. Letta's migration gate is the closest analogue to a global enforce at startup.

### Recovery semantics on graceful shutdown

- Langfuse: `ClickhouseWriter.shutdown()` calls `flushAll(true)` (`worker/src/services/ClickhouseWriter/index.ts:99-110`).
- Temporal: shard context state already lives in persistence; in-memory queue state is rebuilt from persisted high-watermarks (`service/history/shard/context_impl.go:1811-1846`).
- LangGraph: `durability="async"` chains `_put_checkpoint_fut` futures; `durability="exit"` writes only on `__exit__` (`libs/langgraph/langgraph/pregel/_loop.py:1182-1189`, `:1301-1311`).
- Agent-framework: Durable Task entities use self-signal TTL cleanup (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:197-215`).
- OpenHands: no application-level shutdown flush — Alembic migration on boot is the only recovery hook (`openhands/app_server/app_lifespan/oss_app_lifespan_service.py:14-35`).
- Letta: rollback on cancel/exception per-request, no global flush (`letta/server/db.py:88-116`).

### Handling of schema and version change

Temporal is the most defensive, requiring production startup verification against the persisted schema version (`common/persistence/schema/version.go:9-30`). LangGraph relies on its `MIGRATIONS` version table (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:85-108`). Agent-framework takes the strongest line by *rejecting* state with a major version mismatch (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateJsonConverter.cs:38-41`). OpenHands and Letta rely on Alembic additive migrations. Langfuse has no equivalent — schema drift across Postgres and ClickHouse is governed by independent migration directories with no shared manifest (gap noted at `packages/shared/prisma/migrations/*` and `packages/shared/clickhouse/migrations/*`).

## Tradeoffs

### Explicit Durability knob vs implicit per-store config

- **Benefit (explicit knob, LangGraph)**: operator tunes write timing without touching storage; reasoning is per-call.
- **Cost**: a new dimension to learn; misuse warnings are easy to silence.
- **Best-fit**: long-running graph executions where checkpoint frequency matters.
- **Failure mode (LangGraph)**: `InMemorySaver` is the loop's default, and "no warning" looks like "no problem."
- **Alternative**: per-store configuration (Temporal, Agent-framework, OpenHands) — less abstract but tied to the storage class.

### Implicit vs explicit backend selection at startup

- **Benefit (implicit, OpenHands / Letta)**: zero-config local dev; minimal env surface.
- **Cost**: silent footguns — running in prod with the dev default.
- **Best-fit**: dev/test rigs and CLI tools.
- **Failure mode**: an operator sees "the server started" and assumes the right store is wired.
- **Alternative (explicit, Langfuse / Temporal / Agent-framework)**: each backend is named in config or DI; the operator is forced to choose.

### Multi-store richness vs single-DB simplicity

- **Benefit (multi-store, Langfuse / LangGraph / Temporal)**: each tier uses the best engine for its workload (analytical, transactional, blob).
- **Cost**: more components to back up, observe, and version-migrate; cross-system consistency is the operator's problem.
- **Best-fit (single-DB, Letta)**: small surface area, one ORM, one migration tool; pays the cost of putting file text in `file_contents.text`.
- **Failure mode**: a silent misconfiguration in any tier (which Redis, which S3 bucket, which DB) compounds across the system.

### Async durability vs sync durability

- **Benefit (async, LangGraph `durability="async"`, Temporal bulk processor, Langfuse `ClickhouseWriter`)**: throughput at the cost of last-write loss on SIGKILL.
- **Cost**: the operator must reason about "in-flight but not committed" writes.
- **Best-fit**: high-throughput ingestion pipelines where re-drive is acceptable.
- **Failure mode**: SIGKILL of the worker leaves in-memory writes unrecoverable; can be mitigated by S3 staging (Langfuse) or by follow-up retries driven from durable storage.

### In-memory cache for latency vs Redis cache for consistency

- **Benefit (in-memory, Langfuse `LocalCache`, Temporal `XDC` cache)**: cheaper reads; one fewer network hop.
- **Cost**: per-pod divergence on restart; multi-instance freshness window equal to local TTL.
- **Best-fit**: caches with TTL-only semantics and a backing Redis for cross-instance invalidation.
- **Failure mode**: an operator assumes a cache hit rate that only holds in single-pod deployments.

## Decision Guide

When choosing how to organize persistence tiers in an agent harness:

1. **Pick the storage class first.** Decide whether the dominant state is *relational* (use SQL/ORM + migrations), *checkpoint-shaped* (use LangGraph-style `BaseStore`/`BaseCheckpointSaver` separation), or *event/observation-shaped* (use Langfuse/Temporal-style split with analytics DB).
2. **Decide whether the tier is visible.** LangGraph's explicit `Durability` enum is appropriate when checkpoint timing matters; implicit per-store config is appropriate when storage choices are obvious from the deployment platform.
3. **Make every tier observable at startup.** Each source that survives degradation (silent no-op fallbacks) accumulates risk. A simple rule: log the resolved tier class for every state category at boot.
4. **Distinguish durable from ephemeral in code.** Agent-framework's separation of `AgentSession` (caller-serialized) from `ChatHistoryProvider` (durable) is a useful boundary. State that crosses a process boundary should live behind a contract; state that does not should be marked `ephemeral`.
5. **Use migrations for every durable store that has a schema.** Even simple additive changes benefit from being declared. Alembic-style migrations or in-tree `MIGRATIONS` lists both work.
6. **Treat object-storage writes transactionally with metadata.** The Langfuse S3-staging pattern is the strongest ingestion durability primitive in the sample — write the payload to durable storage first, then queue a small reference.
7. **Pair async writers with a flush on shutdown.** Langfuse `flushAll(true)`, LangGraph `BackgroundExecutor.__exit__`, and OpenHands' Alembic-on-boot are all examples; pick the one that matches your storage class.
8. **Use RangeID / ownership records for multi-instance durability.** Temporal's pattern is the cleanest way to make state acquisition atomic across processes.
9. **Avoid dropping TTL semantics in the silent no-op fallback.** A cache that silently degrades must either log its degradation or refuse to start.
10. **Test recovery explicitly.** LangGraph's conformance suite (`libs/checkpoint-conformance/langgraph/checkpoint/conformance/initializer.py:59-99`) and Temporal's CI matrix (`.github/workflows/run-tests.yml:58-62`) are the strongest examples; the smaller frameworks rely on end-to-end tests that rarely exercise crash recovery.

## Practical Tips

### Patterns to copy

- **Force a tier choice at compile / startup time.** When the operator must instantiate a `BaseCheckpointSaver` or a `DefaultStore`, the chance of accidentally running on `InMemorySaver` is reduced.
- **Use atomic local writes** (`fsync` + rename) for any file-backed store. The cost is microseconds; the benefit is no torn writes on crash.
- **Stage writes through an external durable store before queueing.** Langfuse's S3-staging pattern survives Redis loss without operator intervention.
- **Use schema-versioned migrations with explicit version checks.** Temporal's `VerifyVersion` is the closest a framework gets to "prove you can read your own data."
- **Use RangeID for multi-instance ownership.** This is what makes Temporal's restart safe across pods.
- **Pair the durable tier with a TTL mechanism.** Either store-native (Cosmos `MessageTtlSeconds`) or a sweeper (PostgresStore `start_ttl_sweeper`).
- **Make `durability=...` configurable per call.** The decoupling of write timing from storage tier is the cleanest abstraction in the sample.

### Patterns to avoid or delay

- **Silent in-memory fallbacks.** They compound risk: a noisy error in one tier becomes a silent loss in two.
- **Defaulting the highest-level agent to in-memory stores.** Agent-framework's `HarnessAgent` and OpenHands' local default both show that "developer experience first" is a portability trap.
- **Burying storage selection in factory wiring.** Make the selection visible: a config field, a CLI flag, an env var — anything that forces the operator to type the choice.
- **Putting TTL sweeper startup behind an opt-in the operator can forget.** Either spawn it automatically or document the requirement prominently.
- **Asymmetric DLQ retention.** Langfuse uses `removeOnFail: 100_000` for some queues and `100` for others (`packages/shared/src/server/redis/ingestionQueue.ts:66`, `…/batchActionQueue.ts:26`); this is the kind of design that breaks an outage postmortem.
- **Async-only checkpoint flows with no flush hook.** LangGraph's `durability="async"` plus `BackgroundExecutor` is fine because `__exit__` chains the future; smaller frameworks without that hook can lose in-flight writes.

### Decision rules

- **For a long-running agent harness with rich state**: pick the LangGraph model (explicit Durability + pluggable saver). It gives the cleanest operational story.
- **For an ingestion-heavy observability harness**: pick Langfuse's model (Postgres/ClickHouse/S3/Redis split with S3 staging). It scales by separating concerns.
- **For a workflow engine with strict per-step durability**: pick Temporal's model (named stores + RangeID ownership + schema version check).
- **For a small agent framework with one relational backend**: pick Letta's model (single DB + Alembic + pluggable trace tier). Avoid Redis-as-no-op unless you really need zero-friction dev.

### Caution signs

- **In-memory stores selected without a WARNING log.** OpenHands' `get_file_store` and Letta's `NoopAsyncRedisClient` are examples.
- **Async write paths with no flush hook and no redo log.** A bulk processor that holds pending requests in-memory without an external WAL is exactly this.
- **DLQ retention caps that differ between queues.** Operators won't know which queue retried longest.
- **Encrypted-secret stores whose decryption key is stored in the same file system tier.** OpenHands' `.keys` file under `~/.openhands` (`openhands/app_server/utils/encryption_key.py:85`) is durable for single-host but couples to a single file.
- **`AgentSession`-like objects that the harness promises to persist but does not.** Agent-framework's `HarnessAgent` defaults to in-memory and never auto-saves (`dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgent.cs:193-199`). This is the most common silent-drop pattern.
- **Migrations that succeed but don't verify compatibility.** SQLite in Temporal (`common/persistence/sql/sqlplugin/sqlite/db.go:119-124`) is a TODO; the rest of the system has to make do without it.
- **`durability` modes whose names do not match their semantics.** LangGraph's `async` and `exit` are intuitive, but a future contributor must read `_loop.py:1182-1189` and `:1301-1311` to see exactly which writes are persisted when.

## Anti-Patterns / Caution Signs

The following patterns appear in the sample and should be treated as warnings:

1. **Silent fallback to in-memory store when config missing.** Examples: OpenHands `InMemoryFileStore` (`openhands/app_server/file_store/__init__.py:20`), Letta `NoopAsyncRedisClient` (`letta/data_sources/redis_client.py:663-681`), Langfuse `evalJobConfigCache` (`packages/shared/src/server/evalJobConfigCache.ts:27-44`). The risk is identical: an operator sees a working server while durability is silently disabled.

2. **Async-write future that is not chained or flushed.** LangGraph's `durability="async"` chains futures via `_delta_write_futs` (`libs/langgraph/langgraph/pregel/_loop.py:1507-1524`) — this is correct. Temporal Elasticsearch bulk processor does not chain to a durable WAL (`common/persistence/visibility/store/elasticsearch/processor.go:37-48`) — this is fragile. Langfuse `ClickhouseWriter` keeps batches in process memory but flushes on shutdown (`worker/src/services/ClickhouseWriter/index.ts:99-110`) — this is mid-grade.

3. **Production-targeted adapter that ships an in-memory checkpointer as default.** Agent-framework `HarnessAgent` defaults to `InMemoryChatHistoryProvider` (`dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgent.cs:193-199`); LangGraph's `InMemorySaver` is the loop's default when no checkpointer is passed (`libs/langgraph/langgraph/pregel/_loop.py:1501-1505`). Both ship "Only use for debugging or testing" docstrings — neither forbids it at runtime.

4. **Per-process cache that bypasses Redis invalidation.** Langfuse `LocalCache` does TTL-only invalidation with cross-container consistency relying on Redis eviction (`packages/shared/src/server/ingestion/modelMatch.ts:29-30`). This is documented and intended, but any new operator assumes cross-pod cache freshness.

5. **In-memory rate-limit maps with no persistence.** Langfuse `recentProjectMarks` (`packages/shared/src/server/redis/ingestionFailureTracking.ts:14`) is a plain in-process `Map`; a worker restart resets the rate-limit window.

6. **Encryption keys that are persisted to the same dir as their ciphertext.** OpenHands' `.keys` (`openhands/app_server/utils/encryption_key.py:29-86`) is durable only if `persistence_dir` is durable.

7. **Schema version check that no-ops.** Temporal SQLite `VerifyVersion` returns `nil` with a TODO (`common/persistence/sql/sqlplugin/sqlite/db.go:119-124`); a schema mismatch won't be caught.

8. **DLQ retention caps that differ by queue and aren't documented.** Langfuse uses two different `removeOnFail` values (`packages/shared/src/server/redis/ingestionQueue.ts:66`, `…/batchActionQueue.ts:26`); an operator reading one queue's docs won't understand the other's retention.

9. **TTL sweeper thread that is opt-in rather than auto-spawned.** LangGraph's `PostgresStore.start_ttl_sweeper()` (`libs/checkpoint-postgres/langgraph/store/postgres/base.py:807-879`) is the textbook case: forgetting to call it leaves expired items in the table forever.

10. **TTL option that the producer advertises but the backend ignores.** Agent-framework's Valkey has no TTL even though the abstract `BaseStore` supports TTL configuration (`dotnet/src/Microsoft.Agents.AI.Valkey/ValkeyChatHistoryProvider.cs:25-29`).

## Notable Absences

- **No framework exposes a "durability tier matrix" endpoint** at runtime. An operator who wants to know which stores are wired must read source code or config; none of the six frameworks prints a startup summary tagged by tier.
- **No framework ships conformance tests as a mandatory gate.** LangGraph ships a conformance suite (`libs/checkpoint-conformance/langgraph/checkpoint/conformance/initializer.py:59-99`), but it is opt-in. Temporal's CI matrix covers SQL/Cassandra/Elasticsearch/OpenSearch/SQLite (`.github/workflows/run-tests.yml:58-62`), but it tests adapters, not the durability contract.
- **No framework has a built-in "force durable" global override.** Each tier is configured independently; there is no equivalent to a read-only mode for storage.
- **No framework offers an atexit-style flush for SIGKILL.** Only SIGTERM-trapped shutdown flushes are present (Langfuse `ClickhouseWriter.shutdown()`, LangGraph `BackgroundExecutor.__exit__`); SIGKILL remains unbounded.
- **No framework has a uniform way to enumerate process-local state at boot.** Per-process caches like `recentProjectMarks` and the `WorkerManager.workers` map are runtime-only; an operator cannot list them.
- **No agent-framework session auto-save equivalent.** The closest analogue is `SerializeSessionAsync` on `AIAgent` (`dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgent.cs:187`), which is caller-driven.
- **No multi-engine migration manifest.** Langfuse's Postgres and ClickHouse schemas evolve independently (`packages/shared/prisma/migrations/*` and `packages/shared/clickhouse/migrations/*`); Temporal at least ties SQL/Cassandra to a single concept of `schema_version`.

## Per-Source Notes

### agent-framework (`6/10`)

**Approach**: Pluggable multi-tier with non-durable defaults.

**Where implemented**: `dotnet/src/Microsoft.Agents.AI.Abstractions/`, `dotnet/src/Microsoft.Agents.AI/`, `dotnet/src/Microsoft.Agents.AI.Workflows/`, `dotnet/src/Microsoft.Agents.AI.DurableTask/`, plus extension packages in `dotnet/src/Microsoft.Agents.AI.{CosmosNoSql,Valkey,Mem0,VectorData,DurableTask}/`.

**Mechanism**: Multiple `XProvider` interfaces — `ChatHistoryProvider`, `ICheckpointStore`, `AgentFileStore` — each with several concrete implementations, all wired through DI. State model is two-tier: lightweight in-process `AgentSessionStateBag` versus durable external storage.

**Strengths**: widest stack in the sample (memory → file → Cosmos → Valkey → Durable Task → vector stores → Mem0); per-store filter pipelines; durable agent entity TTL via self-signaling.

**Concerns**: `HarnessAgent` defaults to in-memory and cwd-relative stores; `AgentSession` is never auto-persisted; per-service-call persistence is opt-in (`RequirePerServiceCallChatHistoryPersistence`, `dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgentOptions.cs:152`); Valkey has no TTL; durable compaction is delegated to the underlying store.

### langfuse (`7/10`)

**Approach**: Implicit multi-store with at-least-once recovery.

**Where implemented**: `packages/shared/src/server/services/StorageService.ts:222-1649`; `packages/shared/prisma/schema.prisma:10-14`; `packages/shared/clickhouse/migrations/clustered/*`; `packages/shared/src/server/redis/*`; `worker/src/services/ClickhouseWriter/index.ts:42-110`.

**Mechanism**: Postgres is control-plane, ClickHouse is observability source-of-truth, S3 is the write-ahead log for ingestion events, Redis backs queues and caches, and in-process LRUCache is the L1 latency layer. Recovery is at-least-once via S3 staging + BullMQ retries.

**Strengths**: S3 staging pattern is the strongest WAG in the sample; DLQ sweep every 10 minutes; `safeMultiDel` for Redis cluster compatibility; prismaGlobal tag to globalThis in dev.

**Concerns**: No top-level durability knob; many silent no-op paths (`evalJobConfigCache`, `recentProjectMarks`, `ClickhouseReadSkipCache`); DLQ retention caps differ between queues; Postgres and ClickHouse migrations are independent.

### langgraph (`8/10`)

**Approach**: Explicit Durability literal decoupled from storage tier.

**Where implemented**: `libs/langgraph/langgraph/types.py:87-93`; `libs/langgraph/langgraph/pregel/_loop.py:1117,1182-1189,1301-1311`; `libs/checkpoint/langgraph/checkpoint/base/__init__.py:176-580`; `libs/checkpoint-{sqlite,postgres,conformance}/...`; `libs/cli/langgraph_cli/schemas.py:9-233`.

**Mechanism**: `Durability = "sync" | "async" | "exit"` controls write timing; `BaseCheckpointSaver` and `BaseStore` are orthogonal storage contracts. SQLite + WAL for single-instance; Postgres + JSONB/BYTEA for multi-instance; conformance suite validates custom backends.

**Strengths**: cleanest abstraction in the sample; explicit warnings on misuse; DeltaChannel history walker works on every tier; versioned migrations + `prune` API + encrypted serializer wrapper.

**Concerns**: `InMemorySaver` is the loop default (no runtime enforcement against it); `EphemeralValue` and `UntrackedValue` channels silently drop on persistence; `durability="exit"` mid-run state is lost on SIGKILL; ShallowPostgresSaver is deprecated but still exported.

### letta (`7/10`)

**Approach**: Single relational DB spine + pluggable trace tier + optional Redis.

**Where implemented**: `letta/server/db.py:21,58`; `letta/orm/base.py:15-17`; `letta/data_sources/redis_client.py:663-681`; `letta/services/provider_trace_backends/factory.py:30-41`; `letta/services/memory_repo/storage/base.py:7-82`; `letta/server/startup.sh:54-61`.

**Mechanism**: SQLAlchemy async ORM on Postgres (always, in the runtime engine) with Alembic migrations applied at container startup; Redis with silent no-op fallback; provider-trace backends support dual-write for migrations; object storage for git-backed memory repos.

**Strengths**: one ORM, one migration tool for the dominant state; pluggable trace tier with explicit factory; object-store abstraction; soft-delete via `is_deleted` mixin.

**Concerns**: Postgres-vs-SQLite is implicit from env; runtime engine always builds Postgres regardless of inferred `database_engine`; Redis no-ops when absent; no startup assertion of a durable configuration.

### openhands (`6/10`)

**Approach**: Two orthogonal persistence families (FileStore blob + SQLAlchemy structured) plus an event-service abstraction, all env-driven.

**Where implemented**: `openhands/app_server/file_store/{files,local,memory,s3,google_cloud}.py`; `openhands/app_server/event/{event_service_base,filesystem_event_service,aws_event_service,google_cloud_event_service}.py`; `openhands/app_server/services/db_session_injector.py:27-320`; `openhands/app_server/app_lifespan/oss_app_lifespan_service.py:14-35`; `openhands/app_server/utils/sql_utils.py:9-92`.

**Strengths**: clear discriminated-union injectors; request-scoped `AsyncSession`; secrets encrypted at rest with deterministic key for cross-pod consistency; SQLite in-memory test rig for every SQL service.

**Approach**: Named store categories (`DefaultStore`, `VisibilityStore`, `SecondaryVisibilityStore`, archival) backed by SQL or Cassandra with explicit schema versioning.

**Mechanism**: Persistence config exposes named stores with backend-specific settings (Cassandra / MySQL / PostgreSQL); schema-version check at startup; shard recovery uses persisted `RangeId` for ownership; cluster metadata is read from persistence and merged over static config; archival is configurable per URI scheme (file/S3/GCS/custom).

**Strengths**: most production-grade durability tiering in the sample; compatibility checks allow actual schema version to be newer than expected (rollback support); explicit retry/rate-limit/metrics wrappers around stores; CI matrix covers SQL/Cassandra/Elasticsearch/OpenSearch/SQLite.

**Concerns**: SQLite schema compatibility check is a TODO; Elasticsearch bulk processor uses in-memory ACK map (not a durable queue); no-op DLQ writer can silently drop tasks if injected; custom datastore durability is outside core enforcement; no single operator-facing tier matrix.

## Open Questions

- **Durability reporting endpoint**: would any framework commit to a startup log line that says "active tiers: DB=postgres@host, cache=redis-cluster@host, archive=s3@bucket, in-memory-caches=[X,Y,Z]"? This would close the operator-clarity gap that all six sources share.

- **SIGKILL flush**: the gap between graceful shutdown flush (well-engineered everywhere) and SIGKILL (uncovered) is a structural weakness in every framework that uses in-memory write staging. Is there a way to use `signalfd` or `prctl(PR_SET_PDEATHSIG)` to flush on parent-process death?

- **Cross-tier transactional consistency**: Langfuse moves events through S3 → BullMQ → ClickHouseWriter; ClickhouseWriter and Postgres mutations happen after the queue is acked. Is there a write-ahead-log primitive that covers this without S3?

- **Multi-engine migration manifest**: Postgres migrations (Langfuse, Letta, OpenHands, Temporal) and ClickHouse migrations (Langfuse) evolve independently. Would a shared migration ordinal across engines be useful, or does it make operator reasoning harder?

- **Conformance tests as a gate**: LangGraph ships conformance tests but does not require them for production. Would a "LangGraph conformance certified" badge (or Temporal equivalent) help operators reason about custom storage tiers?

- **`ValkeyTTL` parity with Cosmos**: Agent-framework's Valkey has no TTL (`dotnet/src/Microsoft.Agents.AI.Valkey/ValkeyChatHistoryProvider.cs:25-29`) while Cosmos has `MessageTtlSeconds` (`dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs:88`). Is the inconsistency intentional, or is it waiting on a config option?

- **Encryption key durability**: OpenHands' `.keys` file (`openhands/app_server/utils/encryption_key.py:85`) and Temporal's `JWT_SECRET` (`temporal/fx.go:604-606`) are per-cluster secrets. What is the recommended backup/recovery story for these?

- **`AgentSession` auto-save hook**: Agent-framework deliberately does not auto-save (`dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgent.cs:178-188`). Would an opt-in `ISessionStore` interface close the silent-drop pattern that beginners hit?

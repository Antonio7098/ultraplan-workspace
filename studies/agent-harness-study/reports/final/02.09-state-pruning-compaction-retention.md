## Executive Summary

State growth is bounded in five distinct ways across the six sources, and the choice is largely dictated by *product shape* (agent harness vs workflow engine vs telemetry platform) and *replay requirements* (forensic audit vs human rewind vs idempotent retry). Only **langfuse** (`9/10`) and **temporal** (`8/10`) ship a clear, opt-in retention pipeline that runs to scheduled cadence with replay-safe deletion; **agent-framework** (`8/10`) and **letta** (`7/10`) invest in compaction primitives but rely on per-provider or per-user delete endpoints for actual row removal; **langgraph** (`6/10`) exposes the right abstractions but ships no default implementation of its `prune` contract and no TTL on checkpoints; **openhands** (`3/10`) is essentially absent at the storage layer — events are append-only JSON files with no retention hook at all.

The strongest convergent pattern is **deferred deletion via a queue with staged, idempotent execution** (temporal's bitmasked stages, langfuse's `pending_deletions` → BullMQ → CH DELETE flow, letta's `is_deleted` → hard-delete split). The strongest divergent pattern is **annotation versus deletion**: agent-framework flags excluded messages but never removes them (replay fidelity at the cost of memory pressure), while temporal / langfuse / letta physically delete and rely on external archival or recall indexes for replay.

A pragmatic year-of-runtime guarantee requires *all four* of: (1) a TTL or retention field somewhere authoritative, (2) a scheduled sweeper, (3) a stable, replay-safe delete path, and (4) a user-facing "delete my data" endpoint. Only langfuse and temporal score on all four; letta is missing the scheduled sweeper for recall memory; agent-framework and langgraph are missing uniform retention contracts; openhands is missing everything.

## Core Thesis

Pruning, compaction, and retention are three different solutions to the same problem — **state is growing faster than the operator's willingness to pay for storage or attention**. The mature designs (langfuse, temporal, agent-framework) split the solution into orthogonal layers: a *retention configuration surface* (per-project, per-namespace, per-channel), a *trigger* (time-based cron, size-based threshold, event-based timer), and a *delete executor* (physical `DELETE`, lightweight delete + mask compaction, in-place annotation). The immature designs (openhands) collapse all three layers into "nothing happens automatically," which works at small scale and fails quietly at year-of-runtime.

The most important *design choice* is annotation vs deletion. Annotation (agent-framework) preserves replay and audit but never frees memory until a downstream consumer opts in. Deletion (temporal, langfuse) frees memory cleanly but creates a window where the data is logically gone yet physically present (lightweight delete) or where replay simply stops working past the boundary. Letta's split — keep the row, flip the junction's `in_context` flag — is the cleanest compromise: data is queryable through recall but excluded from active context, so a year-long agent's `messages` table grows linearly but the LLM only ever sees a bounded slice.

## Approach Models

The six sources cluster into **five approach models** for bounding state growth. The cluster is driven primarily by *what kind of state* each system holds (LLM messages vs workflow events vs telemetry rows vs checkpointed graphs) and *what "replay" means to its users*.

### 1. Annotation over deletion (agent-framework)

agent-framework's `_compaction.py:582-597` and `_compaction.py:1249-1305` define compaction as in-place mutation: messages are flagged via `EXCLUDED_KEY` and `EXCLUDE_REASON_KEY`, summaries carry forward-pointers (`_summary_of_message_ids`, `_summary_of_group_ids`), and originals carry back-pointers (`_summarized_by_summary_id`). The full original message list is retained; downstream `included_messages` filters project the active subset at read time when `skip_excluded=True` is set on the history provider. This is the most "replay-fidelity-first" model in the sample: a consumer can always reconstruct what was sent to the model on turn N, even after compaction. The cost is that storage only shrinks when a downstream consumer opts in to filtering — agent-framework's default `InMemoryHistoryProvider` and `FileHistoryProvider` do not (`_sessions.py:814-893, 894-1134`), so the model retains everything in RAM and on disk until the process exits.

The `CompactionStrategy` Protocol (`_compaction.py:51-63`) plus six concrete strategies (`TruncationStrategy`, `SlidingWindowStrategy`, `SelectiveToolCallCompactionStrategy`, `ToolResultCompactionStrategy`, `SummarizationStrategy`, `TokenBudgetComposedStrategy`) and the two-phase `ContextWindowCompactionStrategy` (`_compaction.py:1307-1419`) make the trigger side rich. Retention is the weak half: only `RedisHistoryProvider` enforces a hard `max_messages` cap via `LTRIM` (`_history_provider.py:151-165`); Cosmos relies on container-side TTL; in-memory and file have nothing.

### 2. Project-scoped time-based retention (langfuse)

Langfuse is a telemetry platform, not a conversational harness, so the dimension reduces to "rows growing in ClickHouse + Postgres + S3." The single knob is `Project.retentionDays: Int?` (`packages/shared/prisma/schema.prisma:126`), validated to `0` or `>=3` (`web/src/features/auth/lib/projectRetentionSchema.ts:3-10`) and gated by the `data-retention` entitlement (`web/src/features/entitlements/constants/entitlements.ts:16`). Two complementary mechanisms:

- **Per-project BullMQ path**: daily cron at `15 3 * * *` (`packages/shared/src/server/redis/dataRetentionQueue.ts:43`) fans out one `DataRetentionProcessingJob` per project; the handler re-reads retention before deleting (`worker/src/ee/dataRetention/handleDataRetentionProcessingJob.ts:28-39`) and walks a five-store cascade (ClickHouse traces/observations/scores/events, S3 media, S3 ingestion events, Postgres media + junctions) (`worker/src/ee/dataRetention/handleDataRetentionProcessingJob.ts:84-103`).
- **Batch retention cleaner** (opt-in via `LANGFUSE_BATCH_DATA_RETENTION_CLEANER_ENABLED=true`): `BatchDataRetentionCleaner` issues one bulk DELETE per table per run with hash-keyed project params (`worker/src/features/batch-data-retention-cleaner/index.ts:62-96, 152-399`); a `PeriodicExclusiveRunner` (`worker/src/utils/PeriodicExclusiveRunner.ts`) holds a Redis lock per table name so multiple worker instances do not double-delete.

The disk-reclaim half is the `DeletedMaskCleaner` (`worker/src/features/deleted-mask-cleaner/index.ts:42-263`): ClickHouse lightweight deletes leave tombstones in `patch-YYYYMM` parts until the next mutation or `ALTER TABLE ... APPLY DELETED MASK` runs, and this worker inspects `system.parts` for old patch partitions and submits one `APPLY DELETED MASK` per tick. Without it, physical disk follows the deletion only on background merges — a slow leak.

### 3. Three orthogonal axes, no defaults (langgraph)

LangGraph refuses to pick a single answer. Instead, it exposes:

- **`Durability = "sync" | "async" | "exit"`** (`libs/langgraph/langgraph/types.py:87-93`). `"exit"` skips intermediate super-step checkpoints, writing only the final one (`libs/langgraph/langgraph/pregel/_loop.py:1116-1118`). This is the framework's answer to "we don't want to checkpoint every node."
- **`DeltaChannel`** (`libs/langgraph/langgraph/channels/delta.py:25-204`) is a sentinel-only compaction primitive: between snapshots the channel writes only a `MISSING` sentinel, and snapshot cadence is bounded by per-channel `snapshot_frequency` (default `1000`) plus a system-wide `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT` (default `5000`, env-overridable via `LANGGRAPH_DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT`; `libs/langgraph/langgraph/_internal/_config.py:33-35`). Replay walks the parent chain via `BaseCheckpointSaver.get_delta_channel_history` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-690`), with per-backend fast paths.
- **Saver-level retention**: `delete_thread` (mandatory), `delete_for_runs` (optional), `prune(strategy="keep_latest"|"delete")` (optional) (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:320-415`). The store layer ships TTL with refresh-on-read and a background sweeper (`sweep_interval_minutes` default `5`); the checkpoint layer has no TTL — `checkpoints`, `checkpoint_blobs`, `checkpoint_writes` lack an `expires_at` column (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:43-91`).

The `prune` contract is documented but unimplemented — no built-in saver overrides `aprune`/`prune`; the conformance suite `test_prune.py:34-194` would fail on Postgres or SQLite today. `prune(strategy="keep_latest")` is documented to *break* `DeltaChannel` replay because the kept checkpoint is "rarely a snapshot point itself" (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:387-413`); the framework refuses to ship a default implementation that would silently corrupt channel state. `ShallowPostgresSaver` was a "drop history" escape hatch but is deprecated as of `2.0.20` and slated for `3.0.0` removal (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/shallow.py:192-197, 551-556`); the migration path is `durability="exit"`.

### 4. Persisted-summary compaction (letta)

Letta commits hardest to the idea that a compaction event is *itself* a first-class object. The compaction result is a `Message(role=summary)` written into the same `messages` table as ordinary messages, with `compaction_stats` (trigger, mode, context_tokens_before/after, messages_count_before/after) embedded in the JSON body (`letta/services/summarizer/compact.py:434-466`, `letta/schemas/letta_message.py:411`, `letta/system.py:207-237`). The `EventMessage(event_type="compaction")` is emitted before compaction begins so streaming clients can show a "compacting…" state (`letta/agents/letta_agent_v3.py:818-846`).

Four strategies with explicit fallback chains (`letta/services/summarizer/compact.py:189-348`):

- `all` → `sliding_window` (fallback if `all` fails to reduce)
- `sliding_window` (incremental eviction %, raises `ValueError` if no assistant message)
- `self_compact_all` (agent's own LLM does the summarization)
- `self_compact_sliding_window` (sliding window with agent's own LLM, falls back to `all`)

Trigger threshold is `SUMMARIZATION_TRIGGER_MULTIPLIER = 0.9` of context window (`letta/constants.py:83`), with a model-aware override that pushes GPT-5 to proactive 90% compaction regardless of model size to avoid `max_output_tokens` overruns (`letta/services/summarizer/thresholds.py:27-41`). Compaction fires either on a `ContextWindowExceededError` mid-step retry (`letta/agents/letta_agent_v3.py:1218-1294`) or a post-step check (`letta/agents/letta_agent_v3.py:1439-1509`). The provider-overload fallback routes Anthropic → Bedrock Opus and ZAI → Baseten GLM-5 (`letta/services/summarizer/summarizer.py:714-816`).

The retention half is the weakness: `messages`, `passages`, `runs`, and `BlockHistory` rows are only ever removed by explicit user-initiated delete endpoints. The `Base` ORM mixin has no `expires_at` (`letta/orm/base.py:14-18`). The only background sweeper in the codebase is OAuth session expiry (`letta/services/mcp/oauth_utils.py:260-272`, `letta/services/mcp_manager.py:1071-1087`). A year-long agent accumulates rows in the `messages` table; recall search still works (they're not deleted) but the SQL `count` and pagination cost grow linearly.

### 5. Immutable history + retention-driven deletion (temporal)

Temporal's model is the closest to "industrial workflow engine." History events are append-only and immutable (`common/persistence/execution_manager.go:606-628`); compaction in the LLM sense is not a concept. Retention is per-namespace (`Namespace.Retention()`, `common/namespace/namespace.go:289-296`), with global/local minima enforced at write time (`common/namespace/const.go:6-13`, `service/frontend/namespace_handler.go:1111-1128`). The delete driver is a five-stage bitmask (`Visibility` / `Replication` / `Current` / `MutableState` / `History`, `service/history/tasks/delete_workflow_execution_stage.go:10-26`) that lets the timer-queue processor retry partial work safely (`service/history/shard/context_impl.go:903-1095`). The timer is scheduled at close time as `closeTime + ns.Retention() + jitter` (`service/history/workflow/task_generator.go:357-367`) with `RetentionTimerJitterDuration=30m` (`common/dynamicconfig/constants.go:2074-2078`) to spread fan-out.

Three scavengers close the loop when the timer path loses a task:

- **`HistoryScavenger`** (`service/worker/scanner/history/scavenger.go:118-378`) reconciles orphan branches; with `HistoryScannerVerifyRetention=true` it also calls `adminClient.DeleteWorkflowExecution` when retention+buffer has passed.
- **`ExecutionsScavenger`** (`service/worker/scanner/executions/task.go:86-253`, Cassandra-only) validates mutable-state invariants and force-deletes retention-past workflows.
- **`TaskQueueScavenger`** (`service/worker/scanner/taskqueue/scavenger.go:60-65`) drops empty or idle task queues.

The `BuildHistoryGarbageCleanupInfo` annotation (`common/persistence/data_interfaces.go:1420-1427`) lets the history scavenger verify no live mutable state references the branch before deletion — a small but important replay-safety property. The `ContinueAsNew` pattern (`common/dynamicconfig/constants.go:380-417`) is the operational answer to "history is getting long": when `limit.historySize.error=50MB` is reached, the workflow is expected to roll over to a fresh run rather than rely on retention.

### 6. Absent (openhands)

OpenHands V1's app server stores every event as `{event_id_hex}.json` under `{persistence_dir}/{user_id}/v1_conversations/{conversation_id_hex}/` (`openhands/app_server/event/event_service_base.py:83-89`, `openhands/app_server/event/filesystem_event_service.py:33-42`). The `EventService` abstract has no `delete_event` method (`openhands/app_server/event/event_service.py:18-70`); the event router exposes only read endpoints (`openhands/app_server/event/event_router.py:29-110`). The `LLMSummarizingCondenser` from `openhands.sdk` (`openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`) condenses the LLM context window but does not touch persisted events.

`delete_app_conversation` (`openhands/app_server/app_conversation/app_conversation_router.py:892-978`) deletes the SQL metadata rows and best-effort calls the agent-server DELETE, but the event JSON files are never explicitly removed by app-server code paths — orphaned until out-of-band cleanup. `pause_old_sandboxes` (`openhands/app_server/sandbox/sandbox_service.py:203-244`) limits per-user compute, not storage. The only event-count cap is `export_max_events=10000` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:2304-2313`), which gates the export operation rather than deleting anything.

## Pattern Catalog

### A. Annotation with bidirectional trace metadata (agent-framework)

**What problem it solves**: compaction should not break replay — the original message stream must remain reconstructible even after eviction. **Where it appears**: agent-framework only. **Why it works**: every strategy that excludes messages mutates them in place with `_excluded=True` and `_exclude_reason=<reason>` (`_compaction.py:582-597`); summaries carry `_summary_of_message_ids` and `_summary_of_group_ids` (`_compaction.py:33-34, 894-913`); originals carry `_summarized_by_summary_id` (`_compaction.py:35, 288-293`). A consumer can walk back from a summary to the originals, or from an excluded message to the summary that replaced it. **When to copy**: when the consumer wants "forensic" replay — i.e., a developer can be told "what exactly did the model see on turn N." **When overkill or risky**: in any system where memory pressure is the binding constraint, because excluded messages still occupy RAM until a downstream `skip_excluded=True` consumer filters them out. **Evidence**: `_compaction.py:33-35, 582-597, 894-913, 1067-1086`, `tests/core/test_compaction.py:393-466` (summarization tests assert the back-link).

### B. Compaction summary as a persisted first-class Message (letta)

**What problem it solves**: compaction must be observable to the user, queryable for replay, and have a stable schema for downstream analytics. **Where it appears**: letta only. **Why it works**: the compaction result is a `Message(role=summary)` written into the same `messages` table and `conversation_messages` junction as ordinary messages (`letta/services/summarizer/compact.py:434-466`). `compaction_stats` is packed into the JSON body so any consumer that can read messages can read compaction history (`letta/system.py:207-237`, `letta/schemas/letta_message.py:411`). The pre-compaction `EventMessage(event_type="compaction")` (`letta/agents/letta_agent_v3.py:818-846`) lets streaming clients render a "compacting…" state. **When to copy**: when end users need to see *that* and *when* compaction happened, and when developers need a single query to debug "why did the model behave this way?" **When overkill**: when the compaction result is opaque to the user (some chat UIs hide system messages anyway) and only LLM input/output matters. **Evidence**: `letta/services/summarizer/compact.py:434-466`, `letta/schemas/letta_message.py:411`, `letta/tests/integration_test_conversations_sdk.py:1418-1456`.

### C. Bitmasked staged deletion (temporal)

**What problem it solves**: deletion must be replay-safe and retriable across stages that touch different stores, with no risk of partial work corrupting state. **Where it appears**: temporal (visibility, replication, current, mutable state, history); letta uses a softer version with `is_deleted` plus eventual `hard_delete_async`. **Why it works**: the `DeleteWorkflowExecutionStage` bitmask (`service/history/tasks/delete_workflow_execution_stage.go:10-26`) records which stages have completed; each retry resumes from the next unprocessed stage (`service/history/shard/context_impl.go:914-1094`). Cross-cluster retention is intentionally skipped at the replication stage so each cluster self-deletes on its own clock (`service/history/deletemanager/delete_manager.go:138-160`). **When to copy**: any system where deletion touches multiple backends and partial failures must be safely retriable. **When overkill**: a single-store system with atomic row deletion. **Evidence**: `service/history/shard/context_impl.go:903-1095`, `service/history/timer_queue_task_executor_base.go:81-147`, `service/history/deletemanager/delete_manager_test.go:175-234`.

### D. Scheduled cron + opt-in batch cleaner (langfuse)

**What problem it solves**: per-project retention must be correct-by-default for small fleets and fast-at-scale for large fleets, without forcing every operator to pick. **Where it appears**: langfuse (per-project BullMQ cron at 03:15 UTC + opt-in `BatchDataRetentionCleaner`); langgraph's TTL sweeper for the store layer follows the same shape (`sweep_interval_minutes` default 5). **Why it works**: the daily cron fans out one job per project with re-fetched retention (`worker/src/ee/dataRetention/handleDataRetentionProcessingJob.ts:28-39`); the batch cleaner chunks by project hash with collision detection (`worker/src/features/batch-data-retention-cleaner/index.ts:62-79`), runs under a Redis-locked `PeriodicExclusiveRunner`, and emits one bulk DELETE per table per run (`worker/src/features/batch-data-retention-cleaner/index.ts:152-399`). **When to copy**: any multi-tenant system where per-tenant retention is the primary contract but operators of large fleets need a faster path. **When overkill**: a single-tenant system where a per-entity TTL is sufficient. **Evidence**: `packages/shared/src/server/redis/dataRetentionQueue.ts:43`, `worker/src/features/batch-data-retention-cleaner/index.ts:113-399`, `worker/src/utils/PeriodicExclusiveRunner.ts`.

### E. Two-phase token-budget pipeline (agent-framework)

**What problem it solves**: a single token threshold forces the framework to choose between dropping tool calls (preserving recency) and dropping user/assistant exchanges (preserving tool history). **Where it appears**: agent-framework's `ContextWindowCompactionStrategy` (`_compaction.py:1307-1419`). **Why it works**: two `TokenBudgetComposedStrategy` instances are composed: tool-eviction fires at 50% of the input budget (collapsing tool-call groups via `ToolResultCompactionStrategy`), then truncation fires at 80% (dropping oldest non-system groups). Each phase preserves a different invariant. **When to copy**: when the cost of breaking tool-call/result pairings is high (OpenAI / Azure APIs reject them) and the recency ordering of human exchanges matters. **When overkill**: simple chat UIs where a single truncation threshold suffices. **Evidence**: `_compaction.py:1307-1419`, `tests/core/test_compaction.py:1057-1207`.

### F. Sentinel + parent-chain walk for DeltaChannel (langgraph)

**What problem it solves**: per-super-step checkpoint snapshots are expensive (every node writes the full channel value), but per-super-step reducer replay is also expensive (recompute from origin on every checkpoint read). **Where it appears**: langgraph's `DeltaChannel` (`libs/langgraph/langgraph/channels/delta.py:25-204`) with a per-backend `get_delta_channel_history` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-690`, `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:201-477`). **Why it works**: between snapshots the channel writes only a `MISSING` sentinel; the snapshot cadence is bounded by per-channel `snapshot_frequency` (default 1000) and a system-wide `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT` (default 5000). On read, the saver walks the parent chain from the latest checkpoint back to the nearest snapshot ancestor. **When to copy**: any system where channel state is large, super-step cadence is high, and replay from a non-snapshot ancestor is rare but must be possible. **When overkill or risky**: when the system cannot afford the two-stage SQL cost on cold reads, or when the application has no need for time-travel past the latest snapshot. **Evidence**: `libs/langgraph/langgraph/channels/delta.py:74-204`, `libs/langgraph/tests/test_delta_channel_supersteps_bound.py`, `libs/langgraph/tests/test_delta_channel_exit_mode.py`.

### G. TTL refresh-on-read in a single roundtrip (langgraph store, letta token cache)

**What problem it solves**: items should expire after a TTL of "last touched" rather than "first written," without paying for a separate UPDATE per read. **Where it appears**: langgraph's Postgres store (`libs/checkpoint-postgres/langgraph/store/postgres/base.py:266-285`) and SQLite store (`libs/checkpoint-sqlite/langgraph/store/sqlite/base.py:262-266`); letta's Redis token-count cache with 1h TTL (`letta/services/context_window_calculator/token_counter.py:49-253`). **Why it works**: a CTE (`passed_in`) merges the lookup and the conditional `UPDATE expires_at = NOW() + (ttl_minutes || ' minutes')::interval` so refresh and read are atomic. **When to copy**: when items should live as long as they remain interesting; when the cost of a per-read write is too high; when "touched" semantically means "successfully fetched." **When risky**: a perpetually-read item never expires — an attacker can keep data alive by polling. **Evidence**: `libs/checkpoint-postgres/langgraph/store/postgres/base.py:266-285`, `libs/checkpoint-sqlite/langgraph/store/sqlite/base.py:262-266`, `letta/services/context_window_calculator/token_counter.py:49-262`.

### H. Pre-flight count + min/max timestamp gating before DELETE (langfuse)

**What problem it solves**: a noisy `DELETE FROM huge_table WHERE id IN (...)` is expensive even when no rows match, and ClickHouse does not partition-prune on `id IN (long-list)`. **Where it appears**: langfuse's `deleteTraces` (`packages/shared/src/server/repositories/traces.ts:966-1032`), `deleteObservationsByTraceIds` (`packages/shared/src/server/repositories/observations.ts:1257-1318`). **Why it works**: the function first computes the affected time window (`min(timestamp), max(timestamp)`) and adds it to the WHERE clause so the DELETE becomes a partition cut. The `cnt = 0` short-circuit (`traces.ts:1004-1010`) skips the DELETE statement entirely when no rows match. **When to copy**: any analytical store (ClickHouse, BigQuery, Druid) where partition pruning matters and the natural query is "delete by id list." **When overkill**: a transactional store where primary-key lookups are cheap. **Evidence**: `packages/shared/src/server/repositories/traces.ts:966-1105`, `packages/shared/src/server/repositories/observations.ts:1257-1434`.

### I. S3-first deletion with idempotent retries (langfuse)

**What problem it solves**: deletion order matters when S3 holds the byte content and Postgres holds the metadata row. **Where it appears**: langfuse's `deleteMediaFiles` (`packages/shared/src/server/media-deletion.ts:73-116`); letta's Turbopuffer dual-write with `strict_mode` opt-in (`letta/services/message_manager.py:1084-1087`, `letta/services/passage_manager.py:883-891`). **Why it works**: the comment at `media-deletion.ts:84-87` explicitly states that S3 deletes are idempotent and PG re-deletes are guarded by `in (...mediaIds)`, so partial failure self-heals on retry. The downside (orphaned S3 if PG-first) is avoided by deleting S3 first. **When to copy**: any system where metadata and blob storage must agree. **When risky**: when S3 is the source of truth (then the order should be reversed, with compensating transactions). **Evidence**: `packages/shared/src/server/media-deletion.ts:73-116`, `worker/src/features/media-retention-cleaner/index.ts:35-209`.

### J. Compaction stats as observability payload (letta, agent-framework partial)

**What problem it solves**: without telemetry, an operator cannot tell whether compaction is firing too often, too rarely, or successfully. **Where it appears**: letta's `compaction_stats` in the summary payload (`letta/system.py:207-237`, `letta/schemas/letta_message.py:411`); agent-framework's `_exclude_reason` taxonomy (`truncation` / `sliding_window` / `tool_call_compaction` / `tool_result_compaction` / `summarized` / `token_budget_fallback` / `token_budget_fallback_strict` — `_compaction.py:717-1157`); langfuse's per-cleaner metric prefix `langfuse.batch_data_retention_cleaner.*` (`worker/src/features/batch-data-retention-cleaner/index.ts:220-229`). **Why it works**: every exclusion is tagged with a stable reason string; the taxonomy is uniform; the aggregation surface is implicit (one mapping from reason string to OTel counter). **When to copy**: any system with multiple compaction strategies — without a reason taxonomy, post-hoc debugging requires correlating log lines. **When overkill**: a single-strategy system. **Evidence**: `letta/system.py:207-237`, `_compaction.py:717, 763, 816, 897, 1083, 1146, 1157`, `worker/src/features/batch-data-retention-cleaner/index.ts:220-229`.

### K. Soft-delete as universal mutability primitive (letta)

**What problem it solves**: hard-delete cascades are dangerous; soft-delete with eventual hard-delete lets the system recover from partial failures and gives users an "undo" window. **Where it appears**: letta's `Base.is_deleted` (`letta/orm/base.py:18`) and `hard_delete_async` (`letta/orm/sqlalchemy_base.py:673-706`); langfuse's `pending_deletions` table (`packages/shared/prisma/schema.prisma:1788-1803`). **Why it works**: soft-delete flips a flag; hard-delete is reserved for explicit user-initiated teardown. Cascade rules fire only when the user calls the hard-delete endpoint. **When to copy**: any system where accidental deletion is expensive and the operator benefits from a recovery window. **When risky**: when no operator ever sweeps soft-deleted rows — they accumulate forever. Letta is exactly in this state for `messages` and `passages`. **Evidence**: `letta/orm/base.py:14-18`, `letta/services/conversation_manager.py:556-609`.

### L. Atomic group boundaries (agent-framework)

**What problem it solves**: OpenAI / Azure APIs reject messages where a tool_call appears without its tool_result, or where a reasoning prefix is detached from its assistant message. **Where it appears**: agent-framework's `group_messages` (`_compaction.py:120-243`). **Why it works**: every strategy that excludes messages respects the group boundary — `tool_call + tool_result + reasoning_prefix` are kept or evicted as a unit. ADR-0019 calls this the "critical correctness constraint" (`docs/decisions/0019-python-context-compaction-strategy.md:54`). **When to copy**: any framework that compacts LLM context with tool-use messages. **When overkill**: pure-text conversations. **Evidence**: `_compaction.py:120-243`, `tests/core/test_compaction.py:107-135`.

### M. SDK-as-runtime scanner (temporal)

**What problem it solves**: long-running sweepers should be observable, retriable, and respect the same persistence guarantees as user workflows. **Where it appears**: temporal's `worker/scanner` subsystem runs SDK workers for scavengers (`service/worker/scanner/scanner.go:117-290`). **Why it works**: scavengers are SDK workflows with activities, heartbeats, and ContinueAsNew — TimeSkipping applies naturally and progress is observable through Temporal's own machinery. **When to copy**: any system where background sweepers need the same durability and observability as user-facing work. **When overkill**: a small system where a cron + script is sufficient. **Evidence**: `service/worker/scanner/scanner.go:117-290`, `service/worker/scanner/history/scavenger.go:118-378`, `service/worker/deletenamespace/workflow.go`.

### N. Per-namespace retention with validation floors (temporal, langfuse)

**What problem it solves**: a single retention value must be enforceable, validated against floors, and discoverable in the API. **Where it appears**: temporal's `Namespace.Retention()` with `MinRetentionGlobal = 1 * 24 * time.Hour` (`common/namespace/const.go:6-13`) and langfuse's `projectRetentionSchema` accepting `0` or `>=3` (`web/src/features/auth/lib/projectRetentionSchema.ts:3-10`). **Why it works**: validation rejects bad values at write time so the runtime never has to handle "impossible" retentions; minimums protect against operator foot-guns. **When to copy**: any multi-tenant or multi-environment system. **Evidence**: `common/namespace/const.go:6-13`, `service/frontend/namespace_handler.go:1111-1128`, `web/src/features/auth/lib/projectRetentionSchema.ts:3-10`.

### O. Compaction that never deletes source rows (letta)

**What problem it solves**: replay must remain possible even after the in-context window has been compacted; recall search must continue to find "what did the user say 6 months ago?" **Where it appears**: letta's junction-table flip (`letta/services/conversation_manager.py:752-797`); partially in agent-framework's annotation model (which keeps the row but only conditionally exposes it). **Why it works**: compaction only updates the `in_context` flag on `conversation_messages`; the row in `messages` is untouched. `search_messages_async` (`letta/services/message_manager.py:1142-1170`) continues to find it. **When to copy**: when the application needs both "compact for active context" and "searchable history for recall." **When risky**: when row growth is the binding constraint (letta hits this). **Evidence**: `letta/services/conversation_manager.py:752-797`, `letta/services/message_manager.py:1045-1138`.

## Key Differences

The sources diverge on five axes. None of the divergences is a quality difference in isolation — each is a valid response to a different constraint.

### Divergence 1: Annotation vs physical deletion

- **Annotation**: agent-framework. Messages are flagged, never removed; replay is preserved by construction; memory pressure grows linearly with conversation length unless the downstream consumer opts in.
- **Physical deletion**: langfuse, temporal, letta (for `passages`/`messages` hard-delete), openhands-via-conversation-DELETE. Data is removed from the primary store; replay beyond the deletion point is impossible without external archival.
- **Hybrid (junction flip, row kept)**: letta (for in-context compaction). Data is queryable but excluded from the active context.

**Why they differ**: agent-framework is a *library* with multiple downstream storage choices — it cannot physically delete from a store it does not own (Redis, Cosmos, file). It deliberately pushes the choice onto the host application via `skip_excluded`. Langfuse and Temporal own the storage and own the retention pipeline. Letta owns the messages table but values recall over storage cap.

### Divergence 2: Per-tenant vs per-system retention

- **Per-namespace**: temporal (`Namespace.Retention()`).
- **Per-project**: langfuse (`Project.retentionDays`).
- **Per-agent**: letta (`CompactionSettings` on the agent).
- **Per-channel**: langgraph (`DeltaChannel.snapshot_frequency`, plus a system-wide superstep bound).
- **Per-session / per-thread**: agent-framework (per-provider; only `RedisHistoryProvider` enforces it).
- **Per-sandbox / per-user**: openhands (only sandbox quota; no per-conversation retention).

**Why they differ**: the natural unit of policy mirrors the natural unit of isolation. A workflow engine isolates by namespace; a multi-tenant telemetry platform isolates by project; an agent harness isolates by agent; a checkpointing library isolates by channel. None of these is wrong; the question is whether the operator can discover the right knob.

### Divergence 3: Scheduled cron vs event-driven timer vs user-initiated

- **Scheduled cron**: langfuse (`15 3 * * *`), temporal (`DeleteHistoryEventTask` at close-time + retention + jitter).
- **Event-driven**: agent-framework's `CompactionProvider.before_run` / `after_run` lifecycle; letta's `ContextWindowExceededError` retry and `post_step_context_check`.
- **User-initiated**: openhands (no automatic retention); letta's manual `/agents/{id}/summarize` and `/conversations/{id}/compact` endpoints; agent-framework's per-provider `clear()` methods.

**Why they differ**: cron is right when the operator owns the storage and wants a predictable deletion cost. Event-driven is right when the agent owns the compaction and must respond to model failures. User-initiated is right when retention is rare and the user should be in the loop.

### Divergence 4: Replay as a first-class property

- **Compaction breaks replay**: temporal (after retention elapses, history is gone); langfuse (after retention elapses, telemetry is gone).
- **Compaction preserves replay**: agent-framework (annotation), letta (junction flip + persisted summary), langgraph's `DeltaChannel` (parent-chain walk).
- **No compaction, so replay is trivially preserved**: openhands.

**Why they differ**: temporal's value proposition is deterministic replay for workflows that must complete *before* retention elapses — long retention is an operational concession, not a feature. Langfuse's replay value is debugging individual traces, not replaying months of history. Agent-framework and letta treat the message stream as a long-lived audit object; replay must work indefinitely.

### Divergence 5: User-driven deletion surface

- **Project/namespace level**: langfuse (`PUT /projects/{id}` retention, `DELETE /projects/{id}` cascade), temporal (`Operator.DeleteNamespace` → `DeleteNamespaceWorkflow` → `DeleteExecutionsWorkflow`).
- **Per-conversation / per-thread**: agent-framework (`RedisHistoryProvider.clear`, `CosmosHistoryProvider.clear`, `delete_thread`), langgraph (`BaseCheckpointSaver.delete_thread`), letta (`/conversations/{id}`, `/agents/{id}`, `/messages/{id}`, etc.).
- **Per-message**: letta (`/messages/{id}`), agent-framework (no general-purpose filter; only `MemoryContextProvider.history_message_filter`).
- **Account level (GDPR)**: langfuse has no `user_id` anonymization; letta has `delete_actor_by_id_async` but it is not wired to a public REST endpoint; openhands has no `DELETE /users/me` at all in V1; agent-framework has no `delete_user` helper on the base `HistoryProvider`.

**Why they differ**: account-level deletion is a compliance primitive that most agent harnesses do not implement well. Langfuse comes closest through its `PendingDeletion` model + admin API, but it has no anonymization path — deletion is binary removal.

## Decision Guide

Use the following rules when choosing between the approach models.

1. **If the system is a workflow engine that must replay deterministically**, pick **temporal's per-namespace retention with bitmasked staged deletion**. The replay-vs-size tradeoff is resolved by `ContinueAsNew`, not by compaction.

2. **If the system is a multi-tenant telemetry or observability platform**, pick **langfuse's project-scoped retention with the two-tier BullMQ + opt-in batch pattern**. The batch path is opt-in for a reason: small installations do not need it, large ones must enable it.

3. **If the system is an agent harness that owns its storage**, pick **letta's persisted-summary compaction plus soft-delete for recall**. The summary-as-Message pattern gives first-class observability; the junction flip preserves recall; the soft-delete primitive is universal.

4. **If the system is an agent harness that does *not* own its storage** (i.e., it is a library with multiple backends), pick **agent-framework's annotation model with explicit `skip_excluded` opt-in on history providers**. Accept that retention is the consumer's problem; provide a Protocol so consumers can implement their own.

5. **If the system is a checkpointing library for graph execution**, pick **langgraph's three-axis model**: `Durability` for write cadence, `DeltaChannel` for snapshot cadence, `delete_thread`/`prune` for retention. Do not ship a default `prune` implementation until it is replay-safe under `DeltaChannel` — the framework explicitly refuses to for good reason.

6. **If the system has no retention story at all** (like openhands V1), at minimum implement (a) a `delete_event_directory(conversation_id)` call in the conversation-delete path, (b) a `DELETE /users/me` endpoint that cascades, and (c) a per-conversation event-count cap on exports. Without these, the system cannot be operated for a year.

7. **For summarization specifically**, prefer **letta's strategy + fallback chain** (`compact.py:189-348`) over a single strategy. The fallback chain is what makes summarization production-grade — `sliding_window` may fail to reduce if no assistant message exists; `all` may exceed the goal if the system prompt is too large. The chain turns a hard failure into a soft degradation.

8. **For retention validation**, prefer **temporal's minimum-retention floor + langfuse's null/zero as "indefinite"** convention. Reject bad values at write time; never let the runtime encounter "impossible" retentions.

9. **For compaction observability**, prefer **letta's `compaction_stats` embedded in the summary payload** over a separate metrics endpoint. The payload is queryable by anything that can read messages; a metrics endpoint is invisible to most consumers.

10. **For deletion ordering across stores**, prefer **langfuse's "S3 first, then Postgres"** (`media-deletion.ts:84-87`). S3 deletes are idempotent; PG re-deletes are guarded; orphans self-heal.

## Practical Tips

### Patterns to copy

- **Compaction annotation with bidirectional trace metadata** (`_compaction.py:33-35`). When a message is excluded, store both the forward-link (which summary covers it) and the back-link (which message IDs were summarized). This costs one string per message and pays for itself the first time a developer asks "why did the model ignore turn 47?"
- **Bitmasked staged delete** (`service/history/tasks/delete_workflow_execution_stage.go:10-26`). When deletion touches more than one store, model the work as a sequence of named stages with a persisted bitmask. A retry can resume from the next unprocessed stage.
- **Pre-flight min/max timestamp before DELETE** (`packages/shared/src/server/repositories/traces.ts:966-1032`). A 30-line guard that turns a 10-minute DELETE into a 10-second one. Mandatory for any analytical store.
- **Atomic group boundaries** (`_compaction.py:120-243`). When compacting LLM context, keep tool-call + tool-result + reasoning prefix as a unit. Most LLMs reject split messages.
- **Scheduled cron + opt-in batch** (`packages/shared/src/server/redis/dataRetentionQueue.ts:43`, `worker/src/features/batch-data-retention-cleaner/index.ts:113-399`). The cron is correct-by-default; the batch path is opt-in for scale.
- **Compaction summary as first-class Message** (`letta/services/summarizer/compact.py:434-466`). Makes compaction observable, queryable, and replayable.
- **`CompactionProvider` lifecycle hook** (`_compaction.py:1249-1305`). Compaction should be a `ContextProvider` with `before_run` and `after_run`, not an ad-hoc sweep.
- **Strategy + fallback chain** (`letta/services/summarizer/compact.py:189-348`). Every strategy has a documented fallback; transient failures degrade rather than block.
- **`BuildHistoryGarbageCleanupInfo` annotation** (`common/persistence/data_interfaces.go:1420-1427`). When a deletion can leave orphans, annotate the data with the back-reference needed to verify it's safe.
- **Skip-list guard for deletion processors** (`packages/shared/src/server/deletionGuard.ts:5-47`). Even with per-project retention, a `LANGFUSE_DELETE_SKIP_PROJECT_IDS` env var gives operators a circuit breaker.

### Patterns to avoid or delay until needed

- **Per-channel `snapshot_frequency`**. Powerful but adds a per-channel config knob that operators must understand. Default to a single system-wide bound (langgraph's `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT`) and only expose per-channel tuning if profiling shows it matters.
- **Self-compaction modes** (`self_compact_all`, `self_compact_sliding_window` in letta). Letting the agent's own LLM summarize itself is elegant but coupling-prone. Default to a separate, cheaper summarizer model.
- **Voice/sleeptime message-count buffer** (`letta/agents/voice_agent.py:88-113`). Decouples voice from the main agent's token-aware compaction. Either unify or document the rationale.
- **Opt-in `BatchDataRetentionCleaner` as the only fast path**. Most operators do not read changelogs; if the daily cron is too slow for a fleet, the batch path should be auto-promoted at a project-count threshold, not silently behind a flag.
- **Custom tokenizer without a default** (`_compaction.py:66-70`). agent-framework's `CharacterEstimatorTokenizer` undercounts for CJK and code. Ship a `tiktoken` implementation in core by default; let users override, not the other way around.
- **Open-ended `_partial_evict_message_buffer` mode** (`letta/services/summarizer/summarizer.py:185-186`). Marked TODO; do not let legacy code paths accumulate without a sunset date.
- **Lightweight delete without a mask cleaner**. langfuse's primary tables get lightweight deletes but the `DeletedMaskCleaner` is opt-in (`LANGFUSE_CLICKHOUSE_DELETED_MASK_CLEANER_ENABLED`). Without it, disk reclaims only on background merges — a slow leak.

### Decision rules

- **Pick annotation when**: replay is mandatory, the storage is not yours, and memory pressure is tolerable up to a single process lifetime.
- **Pick staged deletion when**: the storage is yours, deletion touches multiple stores, and partial failure must be retriable.
- **Pick soft-delete with eventual hard-delete when**: regulatory hold or undo windows are required.
- **Pick TTL refresh-on-read when**: items are cache-like and "last touched" is the right semantic.
- **Pick TTL from write time + scheduled sweeper when**: items are records and "created at" is the right semantic.
- **Pick pre-compaction `EventMessage` when**: streaming clients need to render a "compacting…" UI state.
- **Pick compaction stats embedded in summary when**: downstream consumers want to query compaction history with the same tools they use for messages.
- **Pick separate metrics endpoint when**: the summary payload is opaque to operators (e.g., the framework never returns the summary to the user).

### Caution signs

- **Compaction is opt-in and the default wires no CompactionProvider** (agent-framework: `_harness/_agent.py:112-113`). A default harness agent will not compact unless token params are passed.
- **`InMemoryHistoryProvider` and `FileHistoryProvider` have no retention** (agent-framework: `_sessions.py:814-893, 894-1134`). Two of four providers grow forever.
- **Workflow checkpoints have no TTL** (agent-framework: `_workflows/_checkpoint.py:119-189`). Long-running workflows accumulate checkpoint JSON files.
- **`prune` is documented but unimplemented in built-in savers** (langgraph: `libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`). Operators who read the docs will assume a default exists; it does not.
- **`pending_deletions` grows on CH delete failures** (langfuse: `worker/src/queues/traceDelete.ts:105-119`). A flaky CH backend produces a slow PG leak.
- **Compaction can succeed without reducing token count** (letta: `letta/services/summarizer/compact.py:410`). Manual endpoint catches this; in-loop path does not.
- **`delete_app_conversation` does not delete event files** (openhands: `app_conversation_router.py:892-978`). Orphans accumulate on disk.
- **No `DELETE /users/me` or GDPR-style endpoint** (openhands: `user/user_router.py:21-58`; agent-framework: no `delete_user_data` on the base `HistoryProvider`). Compliance gap.
- **Visibility TTL is not server-managed** (temporal: `schema/elasticsearch/visibility/index_template_v7.json:9-13`). The server emits no warnings if the index is left unmanaged.
- **ExecutionsScavenger is Cassandra-only and disabled by default** (temporal: `common/dynamicconfig/constants.go:3257-3262`). SQL persistence loses the safety net.
- **Sweeper exceptions are swallowed with `logger.exception`** (langgraph: `libs/checkpoint-postgres/langgraph/store/postgres/aio.py:347-351`). A persistent DB error in the sweep loop will keep the thread alive but do nothing; no escalation, no alarm.
- **`CharacterEstimatorTokenizer` is the default** (agent-framework: `_compaction.py:66-70`). Mis-counts CJK and code; production use should pass a `tiktoken`-backed tokenizer.
- **`memory_warning_threshold = 0.75` is default-OFF** (letta: `letta/settings.py:103`). The warning system is dormant out of the box.

## Notable Absences

- **GDPR / account-level data deletion**: langfuse has no `user_id` anonymization path; letta has `delete_actor_by_id_async` but it is not wired to a documented public REST endpoint; openhands has no `DELETE /users/me` at all in V1; agent-framework has no `delete_user_data` on the base `HistoryProvider`. Only langfuse's `pending_deletions` + admin API comes close.
- **Compaction telemetry**: agent-framework has no metrics module that surfaces compaction frequency, summary length, or excluded-message counts. The `_exclude_reason` taxonomy is uniform but is one mapping away from OTel counters.
- **Default `prune` implementation** (langgraph): the conformance suite `test_prune.py:34-194` is aspirational; no built-in saver implements `aprune`.
- **Default token-accurate tokenizer** (agent-framework): the default is `CharacterEstimatorTokenizer` (4 chars/token), which mis-counts for CJK and code. A `tiktoken` implementation exists in samples but is not the default.
- **Summary caching** (agent-framework): the default prompt says "Incorporate any previously provided summary" (`_compaction.py:938`) but the implementation rebuilds the prompt from scratch.
- **Background sweeper for `messages` and `passages`** (letta): only OAuth sessions have a sweeper (`letta/services/mcp/oauth_utils.py:260-272`).
- **Automatic sweep of stale soft-deleted rows** (letta): soft-delete accumulates without scheduled hard-delete.
- **TTL on checkpoint rows** (langgraph): `checkpoints`, `checkpoint_blobs`, `checkpoint_writes` have no `expires_at` column.
- **Server-managed SQL visibility TTL** (temporal): the server has no DELETE-by-time API for SQL visibility; only row-level delete exists.
- **Event-level TTL or `delete_event` API** (openhands): the `EventService` abstract has no delete method.
- **`summarize` / `compact` / `prune_conversation` step in the codebase** (langfuse): no agent-harness-style summarization; the matches for "summarize" are unrelated to retention.
- **Per-message redaction hook on standard history providers** (agent-framework): only `MemoryContextProvider.history_message_filter` offers pre-persist rewriting.
- **Bulk / namespace-scoped deletion** (langgraph): `delete_thread` is per-thread; no `delete_threads_where(metadata @> ...)` for compliance workflows.

## Per-Source Notes

### agent-framework

The strongest message-history compaction model in the sample, with first-class `CompactionStrategy` Protocol (`_compaction.py:51-63`), six named strategies, a budget-aware composer (`TokenBudgetComposedStrategy`), a two-phase budget pipeline (`ContextWindowCompactionStrategy`), and a `CompactionProvider` lifecycle hook (`_compaction.py:1182-1305`). Compaction preserves replay through bidirectional trace metadata (`_compaction.py:33-35`). Retention is the weak half: only `RedisHistoryProvider` enforces `max_messages` via `LTRIM`; `InMemoryHistoryProvider` and `FileHistoryProvider` have no `max_messages`, no `clear()`, no TTL; workflow checkpoints grow unbounded; `CharacterEstimatorTokenizer` is the only built-in tokenizer.

### langfuse

The most complete retention pipeline in the sample. Project-scoped `retentionDays` (`schema.prisma:126`) drives both a per-project BullMQ cron at `15 3 * * *` (`dataRetentionQueue.ts:43`) and an opt-in `BatchDataRetentionCleaner` for scale. Five-store cascade (ClickHouse traces/observations/scores/events, S3 media, S3 ingestion events, Postgres media + junctions). `DeletedMaskCleaner` physically reclaims disk via `APPLY DELETED MASK`. Pre-flight min/max timestamp gating. DeletionGuard with skip-list. The default behavior is unbounded retention (`retentionDays=null`), and `pending_deletions` grows on CH delete failures.

### langgraph

Three orthogonal axes: `Durability` literal (`types.py:87-93`), `DeltaChannel` sentinel+parent-chain (`channels/delta.py:25-204`), and saver-level `delete_thread`/`prune`. The store layer ships TTL with refresh-on-read and a sweeper; the checkpoint layer does not. `prune` is documented but unimplemented in built-in savers; `prune(strategy="keep_latest")` actively breaks `DeltaChannel` replay per the framework's own docstring warning. `InMemorySaver`/`InMemoryStore` have no retention. `ShallowPostgresSaver` is deprecated as of `2.0.20`.

### letta

First-class compaction subsystem with four strategies (`all`, `sliding_window`, `self_compact_all`, `self_compact_sliding_window`) and explicit fallback chains. Compaction result is a persisted `Message(role=summary)` with `compaction_stats` embedded. Per-agent `CompactionSettings` with model-aware trigger threshold (GPT-5 family at proactive 90%). Provider-overload fallback decouples summarizer availability from agent availability. The retention half is weak: `messages` and `passages` grow unbounded; only OAuth sessions have a sweeper; no `expires_at` anywhere on the ORM; legacy `PARTIAL_EVICT_MESSAGE_BUFFER` mode is marked TODO.

### openhands

Storage-layer pruning and compaction are absent. Events are append-only JSON files; the `EventService` abstract has no `delete_event` method; no event-level TTL, no scheduled GC. The in-process `LLMSummarizingCondenser` from `openhands.sdk` condenses the LLM context but never touches persisted events. `delete_app_conversation` does not delete event files on disk. No `DELETE /users/me`. `pause_old_sandboxes` frees compute, not storage. The only event-count cap is `export_max_events=10000` which gates exports rather than deleting.

### temporal

Per-namespace `Retention()` with validation floors; five-stage bitmasked idempotent delete (`Visibility` / `Replication` / `Current` / `MutableState` / `History`); three scavengers (history, executions-Cassandra-only, task-queue). Cross-cluster retention is intentionally uncoordinated. `ContinueAsNew` is the operational answer to "history is getting long." `BuildHistoryGarbageCleanupInfo` lets scavengers verify orphan branches. SQL visibility has no server-side TTL; ES relies on operator-managed ILM with only `search.idle.after=365d` as the server knob; Cassandra TTLs capped at 10 years; `ExecutionsScavenger` disabled by default.

## Open Questions

1. **agent-framework**: Is there a unified retention contract for `HistoryProvider`? Today only `RedisHistoryProvider` enforces `max_messages`; `InMemoryHistoryProvider` and `FileHistoryProvider` have none. Should the base class expose `max_messages` / `max_age_seconds` and a default `clear()`?

2. **agent-framework**: Does the `Compact-then-persist pipeline` (ADR-0019 §"Boundary vs In-Run Compaction") ever persist a *reduced* transcript? Today `_compaction.py:1275-1305` only annotates in place during `after_run` and relies on `skip_excluded` on the next `before_run`.

3. **agent-framework**: Is there a service-side compaction contract? ADR-0019 says "the service is responsible for its own context window management and compaction" but no machine-readable contract was found.

4. **agent-framework**: Does `MemoryContextProvider.history_message_filter` need to be elevated to the standard `HistoryProvider` base class? Today only the memory provider offers pre-persist rewriting.

5. **langfuse**: What is the operator-facing contract for the `seconds_past_cutoff` gauge and the `delete_failures` counter? Currently the metrics exist but no UI surfaces them; operators must rely on Datadog/SigNoz.

6. **langfuse**: Does the `event_log` ClickHouse table participate in any retention sweep? It is not in the `BATCH_DATA_RETENTION_TABLES` list (`worker/src/features/batch-data-retention-cleaner/index.ts:18-24`) and not referenced in `handleDataRetentionProcessingJob.ts`.

7. **langfuse**: Does the `pending_deletions` table ever get hard-pruned, e.g., after N days regardless of CH delete status? No evidence found.

8. **langgraph**: Is the conformance suite's `prune` test suite ever exercised by a built-in saver? Grep returns zero overrides of `aprune`/`prune` in `libs/checkpoint*`.

9. **langgraph**: What is the operator-facing contract for "remove all checkpoints older than N days"? No env var, no config key, no built-in sweeper for checkpoints.

10. **langgraph**: Is there an in-server SQL visibility index/partition rotation policy in temporal? None found; only per-namespace table layout.

11. **letta**: Where, if anywhere, is `delete_actor_by_id_async` exposed to the public REST surface? `letta/services/user_manager.py:79-84` exists but no router wiring was located.

12. **letta**: Is there a per-organization retention SLA or operator job that bulk-cleans old `messages` / `runs` rows? Only OAuth sessions surfaced.

13. **letta**: How is `compaction_stats` validated across schema evolution? The packing code in `letta/system.py:207-237` embeds `compaction_stats` as JSON but there is no migration policy for that schema when compaction parameters change.

14. **letta**: Does `update_in_context_messages` correctly handle the case where compaction produces a summary but the `in_context_message_ids` argument is missing summary message IDs? The function sets `in_context = message_id in in_context_set` — if summary is omitted, it would be flipped to `False` even though it should remain in context (`letta/services/conversation_manager.py:752-797`).

15. **openhands**: No evidence was found for any time-based event pruning, background/scheduled GC, or user-account deletion endpoint. The agent-server (external `openhands-agent-server==1.29.0` package) may prune its own storage; this is out of scope.

16. **openhands**: Is `OH_MAX_API_SECRETS_COUNT` and friends (`openhands/app_server/constants.py:16-27`) a retention primitive in disguise? They are character/count caps on user secrets, not retention policies for conversation history.

17. **temporal**: Is there a configurable upper bound on per-namespace retention at the cluster level? Validation only rejects values below `MinRetentionGlobal`; very large retentions are accepted as-is, which can outrun any storage backend.

18. **temporal**: Is there an in-server SQL visibility index/partition rotation policy? None found; only per-namespace table layout in `common/persistence/sql/...`.

19. **temporal**: Does the relationship between `DeleteExecutionTask` category (transfer) and the visibility/replication ordering have schema-level enforcement? Understanding requires tracing three executors; no schema-level guard.

20. **Cross-source**: None of the six sources ships a GDPR-style "delete all data for user X" endpoint with both row deletion and vector-store / S3-blob reconciliation. Letta comes closest with `delete_messages_by_ids_async` + Turbopuffer dual-write, but the dual-write is best-effort with `strict_mode` opt-in.

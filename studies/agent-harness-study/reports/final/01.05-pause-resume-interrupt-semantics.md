## Executive Summary

Across the five studied agent frameworks, "pause/resume" means at least four distinct things depending on whether the system is a graph runtime, a server-backed chat agent, a sandboxed execution host, or a distributed workflow engine. What converges is the strong instinct that pause must be **persisted, durable, and restartable**: every source except openhands writes a row or event that survives a process crash and from which a resume can begin. What diverges is *where* the pause lives ã at a superstep boundary, at a step boundary, at the OS/runtime container boundary, or as a first-class state-machine transition in a workflow engine.

Mature runtimes (agent-framework, langgraph, temporal) treat pause as a **state-machine event** anchored in a persistent log or checkpoint and bound to a typed correlation key (request_id, interrupt_id, PauseInfo). Less mature runtimes (letta, openhands) treat pause as a **DB row flip** (`Run.status = completed/stop_reason = requires_approval`, or `SandboxStatus = PAUSED`), with the in-process loop polled cooperatively. None of the five provides full multi-party resume arbitration with explicit ownership; concurrency safety comes from per-instance event-loop serialization, checkpointer-level single-writer, or no coordination at all.

The "approve a pending action tomorrow and continue from the exact same state" question is answerable "yes, with caveats" in agent-framework, langgraph, and temporal (all rated 9/10), "yes for short windows with idempotency" in letta (rated 7/10), and "only at the container boundary; agent-internal state is opaque" in openhands (rated 6/10).

## Core Thesis

> **Pause/resume is durable when the pause is a typed, persisted state transition with a deterministic correlation key ã not when it is a flag flipped in memory or a side-effect of an external API call.**

Three of the five sources (agent-framework, langgraph, temporal) implement pause as a first-class event/state with deterministic keys and replay-safe persistence. The remaining two (letta, openhands) implement pause as a DB-row transition that is durable but lacks a deterministic correlation primitive, leaving "approve the right request from the right caller tomorrow" to application-level idempotency. The biggest design lever a team has in this dimension is **where pause lives**: inside the in-process loop (cheap, but loses crash safety), at the orchestrator layer with persisted events (durable, replayable, but requires an event log), or at the OS/runtime container (durable for the container, opaque for the agent inside).

## Approach Models

The five sources cluster into four distinct approach models for pause/resume. Within each cluster, individual choices diverge around durability granularity and concurrency semantics.

### Cluster A ã Pregel/BSP superstep with persisted checkpoint + explicit interrupt API

**Members:** agent-framework, langgraph

Both expose the same underlying architecture: a Pregel-style superstep loop with three separate pause surfaces, each persisted.

| Mechanism | agent-framework | langgraph |
|-----------|-----------------|-----------|
| Dynamic interrupt inside node/step | `WorkflowContext.request_info` emits a `request_info` event and settles into `IDLE_WITH_PENDING_REQUESTS`; functional API raises `WorkflowInterrupted` (`BaseException`) | `interrupt(value)` raises `GraphInterrupt`; `Command(resume=...)` injects a `(NULL_TASK_ID, RESUME, value)` pending write |
| Static interrupt at compile time | (None ã interrupt is always dynamic and HITL-shaped) | `interrupt_before` / `interrupt_after` raise `GraphInterrupt` based on `should_interrupt` channel-version check |
| Cooperative drain | (None ã pause is exclusively HITL-shaped) | `RunControl.request_drain()` flips a flag; loop returns at next superstep boundary with `GraphDrained` |
| Checkpoint schema | `WorkflowCheckpoint` dataclass with `graph_signature_hash`, `messages`, `state`, `pending_request_info_events`; refuses restore on topology change | `Checkpoint` plus `WRITES_IDX_MAP` with reserved negative keys (`INTERRUPT=-3`, `RESUME=-4`) |
| Deterministic id | `auto::<index>` fallback for functional API; `request_id` is the universal correlation key | xxh3-derived `Interrupt.id` from namespace; per-task `__resume__` pending writes |
| Cross-process durable layer | `agent_framework_durabletask` package translates the same model onto Azure Durable Functions via `TaskOrchestrationContext.WaitForExternalEvent` | Postgres / SQLite `BaseCheckpointSaver` implementations |

**Why this model works:** the superstep boundary is the *only* place state is committed, so replay determinism is enforced by the architecture rather than by ad-hoc calls. The interrupt is always a typed event with a deterministic id, which makes "approve the right request tomorrow" a lookup against a persisted field.

**Why it diverges internally:** agent-framework's functional API layers per-step caching on top of superstep persistence (so completed steps are not re-executed on resume ã `_functional.py:170`), while langgraph re-executes the entire node body and relies on the application to make nodes idempotent. This is a deliberate tradeoff: langgraph keeps the runtime simple at the cost of putting idempotency on the caller; agent-framework invests in runtime caching at the cost of per-step cache management.

### Cluster B ã Run-as-state-machine with persisted step rows + idempotency tokens

**Member:** letta

Letta's model is fundamentally different: there is no superstep loop, no checkpoint schema, no reserved write keys. Instead, every `_step` writes a `Step` row, every LLM call writes a `Message` row, and the run row carries `status` and `stop_reason`. Pause is *the* terminal state when `stop_reason = requires_approval`; the next inbound `ApprovalCreate` is matched against the persisted `ApprovalRequestMessage` via `validate_approval_tool_call_ids`.

| Mechanism | Implementation |
|-----------|----------------|
| Dynamic interrupt (HITL approval) | `StopReasonType.requires_approval`; run row goes to `RunStatus.completed` with pending `ApprovalRequestMessage` |
| Client-side tool pause | Functionally identical to approval; tool runs client-side and returns `ToolReturn` |
| Cooperative cancellation | `cancel_run` flips `RunStatus.cancelled`; agent loop polls `_check_run_cancellation(run_id)` once per step |
| Resume input | Re-send `ApprovalCreate` or `ToolReturn` with same `run_id`; idempotency check survives summarization (`helpers.py:230-310`) |
| Background streaming reattach | Redis SSE stream `sse:run:{run_id}` with 3 h TTL; `GET /v1/runs/{run_id}/stream?starting_after=N` |
| Idempotency | OTID hash via `derive_request_token(otids)`; `try_recover_duplicate_request` recovers same-OTID retries |

**Why this model works:** the database row is the source of truth, so pause and resume do not need a custom serialization layer. Idempotency via OTID is a clean API contract for client retries.

**Why it is fragile:** pause granularity is fixed at the step boundary ã an in-flight LLM call or tool execution always runs to completion. Two lock systems (Redis conversation lock + per-step DB poll) coexist without a unifying primitive. Cancel and deny are conflated: cancelling a paused-on-approval run materializes a denial `ToolReturn`, so the trail is identical to a denied approval.

### Cluster C ã Runtime-container pause with session key rotation

**Member:** openhands

OpenHands is the only source where pause/resume is an **OS/runtime operation**, not an agent-runtime concept. The agent inside the sandbox is treated as opaque.

| Mechanism | Implementation |
|-----------|----------------|
| Pause primitive | Docker `container.pause()`, remote `POST /pause`, or `psutil.Process.suspend()` ã synchronous, no handshake with agent |
| Security control | `session_api_key_hash = None` on pause; new key minted and returned on resume |
| Persistence | `v1_remote_sandbox` row + `conversation_metadata` row; runtime status re-queried on every `get_sandbox` |
| Resume determinism | Deterministic at sandbox boundary; agent-internal resume point depends on opaque SDK (ACP `acp_isolate_data_dir` + `base_state.json`) |
| Approval gate | None at app-server layer; `confirmation_mode` setting is forwarded to agent server as `ConfirmationPolicy` |
| Multi-party safety | None ã no ownership check beyond `created_by_user_id`; rate-limit exempts `/resume` |

**Why this model works:** the sandbox boundary is the most reliable pause point in the stack ã it works regardless of what the agent is doing. Session key rotation is a strong operational safeguard against leaked secrets.

**Why it is limited:** the app server cannot reason about the agent's internal pause state. "Approve a pending tool call tomorrow" requires the agent server to expose that affordance ã which it currently does not, as a first-class API. The dropped `is_paused` column (`012.py:19-29`) is concrete evidence that earlier attempts to cache pause state locally failed because external terminations left the cache stale.

### Cluster D ã Event-sourced workflow state machine with stamp-based staleness

**Member:** temporal

Temporal is the only source where pause is a **first-class event type** in the workflow history itself, not a flag on a checkpoint row.

| Mechanism | Implementation |
|-----------|----------------|
| Workflow pause | `EVENT_TYPE_WORKFLOW_EXECUTION_PAUSED` appended to history; `mutable_state_impl.go:3196-3247` sets status `PAUSED` and bumps activity stamps |
| Workflow unpause | `EVENT_TYPE_WORKFLOW_EXECUTION_UNPAUSED` appended; workflow task rescheduled; buffered signals/timers delivered |
| Activity pause | `ActivityInfo.Paused=true` + `PauseInfo`; transfer queue drops task if `ai.Paused` (`transfer_queue_active_task_executor.go:259-262`) |
| Activity unpause | `UnpauseActivity` clears `Paused`, regenerates retry task with current stamp |
| In-flight interrupt | Heartbeat RPC returns `ActivityPaused`; worker is responsible for stopping |
| Idempotency | `RequestId` on pause (`pauseworkflow/api.go:61-67`); precondition-only on unpause |
| Cross-cluster | Pause bumps activity stamps so existing task replication machinery propagates the change |
| World-changed handling | Signals/timers/cancels buffered while paused; delivered atomically on unpause |

**Why this model works:** the pause event *is* the persistence. There is no checkpoint row to lose, no secondary state to cache, and no race between "did the pause commit" and "did the event commit" ã they are the same thing. `mutable_state_rebuilder.go:671-678` re-applies paused events on history replay, so crash recovery is automatic.

**Why it is limited:** pause does not synchronously interrupt in-flight activities; the heartbeat mechanism is cooperative. Pause is per-run, not per-workflow-id, so continue-as-new and reset naturally create fresh RUNNING runs. No way to pause a child workflow from a parent workflow command.

## Pattern Catalog

A pattern belongs here if it appears in multiple sources or is unusually clear in one.

### Pattern: Persisted pause state is the source of truth, not in-memory state

- **Problem:** in-memory pause state is lost on crash; the next process has no way to know what was paused, on whose behalf, and with what correlation key.
- **Sources:** agent-framework (`WorkflowCheckpoint` dataclass, `_checkpoint.py:30`), langgraph (`Checkpoint` + `WRITES_IDX_MAP`), letta (`Run` + `Step` rows), temporal (`EVENT_TYPE_WORKFLOW_EXECUTION_PAUSED`).
- **Why it works:** durability follows automatically from the storage decision. If the pause state is in the same atomic transaction as the agent state, "resume" is just "load row N+1".
- **When to copy:** always, for any system that may outlive a single process invocation.
- **When it is overkill:** single-process dev tools where the entire runtime fits in memory and restarts are rare.
- **Counter-example:** openhands deliberately does NOT cache pause state in the DB (`012.py:19-29` drops the `is_paused` column); the runtime is the source of truth. This is the right call when the runtime is external and may be terminated without app-server awareness.

### Pattern: Pause is a typed event with a deterministic correlation key

- **Problem:** without a stable key, "approve the right request" requires the caller to know internal state.
- **Sources:**
  - agent-framework: `request_id` is the universal key; auto-derived `auto::<index>` fallback (`_functional.py:229-234`).
  - langgraph: xxh3-derived `Interrupt.id` from namespace; per-task `__resume__` pending write (`_algo.py:1300-1314`).
  - letta: `ApprovalRequestMessage.id` matched against incoming `ApprovalResponseMessage.approval_request_id` (`helpers.py:121-148`).
  - temporal: `RequestId` for pause idempotency; `PauseInfo.Identity` / `Reason` / `RequestId` for audit.
- **Why it works:** the key is the contract between the pause-side and the resume-side. Determinism (xxh3 hash, auto-index, request_id) lets the caller target a specific pending interrupt without ambiguity.
- **When to copy:** any system where pause may be answered by a different process, machine, or operator than the one that initiated it.
- **When it is overkill:** single-threaded synchronous agents where the in-process state is implicitly known.

### Pattern: Cooperative drain / interrupt at a known boundary, not mid-execution

- **Problem:** synchronous interrupt of an LLM HTTP call or tool execution is unsafe and platform-dependent.
- **Sources:**
  - langgraph: `RunControl.request_drain()` flips a flag checked at the top of `tick()`; loop returns at next superstep boundary.
  - letta: `_check_run_cancellation(run_id)` polled once per step; SSE wrapper polls every 0.5 s.
  - openhands: pause at container boundary; agent-server is suspended mid-execution by `container.pause()` (unsafe but accepted).
  - temporal: heartbeat-driven; worker checks `ActivityPaused` on next heartbeat.
- **Why it works:** the boundary is well-defined and replayable. The cost is latency ã between "cancel called" and "agent stopped" can be one step / one heartbeat / one SSE poll.
- **When to copy:** any agent runtime where the underlying provider (LLM, tool, RPC) does not expose a clean cancel primitive.
- **When it is risky:** when the step itself can take minutes (a slow tool, a long context window). The user perceives "cancel" as immediate; the system delivers it at the next boundary.

### Pattern: Reserve dedicated schema slots for control signals

- **Problem:** `INTERRUPT` / `RESUME` / `ERROR` writes must not collide with user state writes in the checkpoint or message log.
- **Sources:**
  - langgraph: `WRITES_IDX_MAP` reserves `INTERRUPT=-3, RESUME=-4, ERROR=...` in checkpoint pending-writes (`checkpoint/base/__init__.py:795`).
  - agent-framework: framework reserves state keys prefixed with `_` (`_functional.py:283-298`); user code that writes to `_foo` is rejected.
  - temporal: `EVENT_TYPE_WORKFLOW_EXECUTION_PAUSED/UNPAUSED` are top-level enum entries with their own factory (`historybuilder/event_factory.go:1052-1082`).
- **Why it works:** the namespace collision is resolved at design time, not at runtime.
- **When to copy:** any persistence layer that mixes user state with control state.
- **When it is overkill:** ad-hoc storage that does not share a single schema.

### Pattern: Multi-layer pause (in-process + persisted + durable external)

- **Problem:** different pause scenarios have different durability requirements. A 30-second HITL gate does not need crash safety; a 3-day approval does.
- **Sources:**
  - agent-framework: in-process HITL (`request_info`) ã file/in-memory checkpoints ã `agent_framework_durabletask` (Azure Durable Functions). Three explicit layers.
  - langgraph: in-memory `InMemorySaver` ã `SqliteSaver` ã `PostgresSaver`. Single layer with a swappable backend.
  - letta: in-memory asyncio.Event ã Redis conversation lock ã DB-row `Run.status` (3 implicit layers).
- **Why it works:** callers can pick the durability they need without paying for what they don't.
- **When to copy:** any system that serves both interactive and long-running workloads.
- **When it is overkill:** single-purpose systems where pause durability is fixed at design time.

### Pattern: Graph/workflow topology validation on restore

- **Problem:** restoring a checkpoint against a changed graph silently corrupts replay.
- **Sources:**
  - agent-framework: SHA-256 of canonical graph JSON stored in `graph_signature_hash`; restore rejects with `WorkflowCheckpointException("Workflow graph has changed")` (`_runner.py:274-279`).
  - agent-framework functional: `_compute_signature_hash` mixes `__code__.co_code`/`co_names` (`_functional.py:1144+`).
- **Why it works:** the topology check is cheap (hash compare) and gives a clear failure mode.
- **When to copy:** any checkpoint-based system where the topology can evolve.
- **Evidence:** tested by `test_resume_fails_when_graph_mismatch` (`test_checkpoint_validation.py:40-62`).

### Pattern: World-changed detection via application-level re-validation, not runtime check

- **Problem:** the framework cannot know whether the external world is still consistent with the paused state.
- **Sources:**
  - agent-framework: contracts this explicitly ã "the framework remembers what it knew; you remember what the world knows"; recommends an `on_world_changed` hook (gap, `_functional.py`).
  - langgraph: node body re-executes on resume; whatever world-checking the application does, it happens there.
  - letta: `validate_approval_tool_call_ids` checks the request matches the response (`helpers.py:121-148`); summarizer protects trailing approval from eviction.
  - temporal: signals/timers buffered while paused, delivered on unpause ã the workflow sees them in order and can react.
- **Why it works:** shifts the responsibility to the layer that knows the semantics.
- **When to copy:** always ã there is no universal way to detect "the world changed".
- **When it is risky:** when the application does not implement re-validation and silently proceeds with stale assumptions.

## Key Differences

### What converges

- **Pause is persisted, not in-memory.** All five sources write some row/event/file when the agent pauses, and resume begins by reading it.
- **Pause is keyed by a deterministic correlation id.** `request_id`, `interrupt_id`, `ApprovalRequestMessage.id`, `RequestId`, `PauseInfo.RequestId` ã every source has one.
- **Resume is restart-safe.** All five can resume after process restart (openhands' agent-internal resume is opaque, but the sandbox boundary is restart-safe).
- **Cooperative cancellation is preferred over forced cancellation.** All five rely on the next boundary / next heartbeat, not synchronous interrupt.
- **Multi-party resume is weakly modeled.** No source provides a true "claim this pending request" primitive; concurrency safety comes from event-loop serialization (agent-framework), single-writer checkpointer (langgraph), conversation lock (letta), no coordination (openhands), or state-machine precondition + lease (temporal).

### Why sources diverge

| Dimension | Cause of divergence |
|-----------|---------------------|
| Pause granularity | langgraph/agent-framework pause at superstep boundaries (deterministic but coarse); letta pauses at step boundaries (coarser ã between LLM call and tool execution); openhands pauses at container boundary (coarsest ã opaque to agent); temporal pauses at workflow-task boundary (fine within history events, coarse for in-flight activities). |
| Persistence substrate | Pregel runtimes store checkpoint blobs; letta stores typed `Step`/`Message` rows; openhands stores identity only; temporal stores events as the primary log. The substrate choice follows the system's overall architecture, not a pause-specific decision. |
| Approval vs. cancel | letta conflates them (cancel of a paused-on-approval run = deny); agent-framework separates them (only HITL pause; cancel is the durable-task layer's job); temporal explicitly separates them (cancel during pause is recorded but not delivered; the two compose). |
| Multi-party safety | Single-writer checkpointer (langgraph) and lease-based precondition (temporal) are the strongest; event-loop serialization (agent-framework) is medium; conversation-lock + DB-row polling (letta) is weak; no coordination (openhands) is weakest. |
| World-changed handling | langgraph expects the application to handle staleness in the node body; agent-framework has no built-in mechanism but documents the contract; letta validates tool-call ids but does not detect external staleness; openhands relies on session-key invalidation to force re-authentication; temporal buffers external signals/timers and delivers them on unpause. |
| Cross-process durability | temporal's pause state is in workflow history and replicated across clusters (`mutable_state_impl.go:3229-3244`); agent-framework's durable layer is Azure-specific (Durable Functions); langgraph uses Postgres/SQLite; letta uses Redis for stream + Postgres for state; openhands delegates entirely to the runtime. |
| Pause as a "first-class event" | Only temporal records pause as an event type with its own factory and rebuilder path. The Pregel runtimes use `INTERRUPT` writes as a reserved key inside a generic checkpoint. letta uses a typed stop reason. openhands uses a sandbox status enum. The deeper the event integration, the more replay-safe the pause. |

## Tradeoffs

### Pause at the superstep boundary vs. finer granularity

- **Benefit:** deterministic replay; single atomic commit point; easy to reason about.
- **Cost:** latency between "user wants to pause" and "agent is paused" can be up to one step.
- **Best fit:** graph runtimes where steps are short and steps themselves are the unit of progress.
- **Failure mode:** long-running steps (slow tools, long LLM streaming) make pause feel unresponsive.
- **Alternative:** temporal's stamp-based approach pauses at the *task-dispatch* boundary, not the *task-execution* boundary, which is finer but requires heartbeats for in-flight interrupt.

### Cooperative cancellation vs. forced cancellation

- **Benefit:** safe ã never tears down an in-flight resource; predictable replay from the next committed state.
- **Cost:** latency ã between cancel and agent-stop can be one LLM call, one tool execution, or one heartbeat cycle.
- **Best fit:** LLM-driven agents where the provider does not expose a clean cancel primitive.
- **Failure mode:** a stuck LLM provider (no heartbeat, no error) blocks the cancel indefinitely. Tested by `tests/integration_test_cancellation.py:179-188` in letta.
- **Alternative:** openhands accepts forced cancel at the container boundary; agent-server is suspended mid-execution. This is the strongest possible pause but loses the agent-internal state.

### Multi-tier pause model (in-process + persisted + durable external)

- **Benefit:** callers choose the durability they need; interactive workloads pay less, long-running workloads pay more.
- **Cost:** three layers = three sets of failure modes, three sets of tests, three sets of operational concerns.
- **Best fit:** platforms that serve both interactive chat and long-running automation (agent-framework explicitly targets this).
- **Failure mode:** subtle bugs at layer boundaries ã e.g., the openhands `is_paused` column drop shows what happens when the persisted layer and the runtime layer disagree.
- **Alternative:** single-layer with a pluggable backend (langgraph's saver pattern).

### Typed correlation key vs. implicit ordering

- **Benefit:** the key is the contract; resume is a lookup, not a search.
- **Cost:** callers must surface the key in their UI/API; a missing or stale key produces a hard error rather than a heuristic guess.
- **Best fit:** multi-party systems where multiple operators may answer pending requests (langgraph's `Command(resume={interrupt_id: value})`, agent-framework's `run(responses={request_id: value})`).
- **Failure mode:** when multiple `interrupt()` calls exist in the same node and the caller does not specify which one to resume, langgraph raises `RuntimeError("When there are multiple pending interrupts, you must specify the interrupt id when resuming...")` (`_loop.py:904-908`).

### Event-sourced pause (temporal) vs. checkpointed pause (langgraph/agent-framework)

- **Benefit of event-sourcing:** the pause event is the log; replay is automatic; cross-cluster replication is built-in.
- **Cost:** every pause/unpause is a history event that the workflow must replay forever.
- **Benefit of checkpointing:** compact storage; no replay cost on hot path.
- **Cost:** checkpoint schema must evolve carefully; topology validation needed; resume depends on storage backend availability.
- **Failure mode:** temporal has a documented race between in-flight workflow task and pause (`tests/pause_workflow_execution_test.go:853-983`); agent-framework's `previous_checkpoint_id` chain can grow without bound.

## Decision Guide

### Choose **Pregel + checkpoint + interrupt API** (agent-framework / langgraph) when:

- You are building a **graph or workflow runtime** with explicit node/step boundaries.
- You need a **declarative interrupt** option (`interrupt_before`/`interrupt_after`) for debuggability.
- You want **pluggable durability** ã in-memory for dev, SQLite/Postgres for production.
- You accept that **node bodies must be idempotent** under re-execution (langgraph) or invest in per-step caching (agent-framework functional API).

### Choose **Run-as-state-machine with persisted steps** (letta) when:

- You are building a **server-backed chat agent** where each turn is a discrete step.
- You need **client-side tool execution** as a first-class pattern (letta treats it identically to approval).
- You want **cursor-based stream reattachment** for long-running background runs.
- You accept **step-boundary pause granularity** and **cooperative cancellation**.

### Choose **runtime-container pause** (openhands) when:

- You are building a **sandboxed agent host** where the agent's internal state is opaque by design.
- You need **strong operational guarantees** about credentials (session key rotation).
- You have **external approval/control flows** outside the agent (e.g., GitOps, CI/CD) that already answer "approve" before the agent is invoked.
- You accept that **agent-internal pause is the agent's problem**, not yours.

### Choose **event-sourced workflow state machine** (temporal) when:

- You are building a **long-running workflow engine** where individual activities may run for hours or days.
- You need **first-class replication** across clusters.
- You want **buffered signals/timers/cancels** during pause (the workflow sees them in order on unpause).
- You accept **heartbeat-driven in-flight interrupt** (the worker must cooperate).

## Practical Tips

### Patterns to copy

1. **Pause as a typed event with a deterministic correlation key.** The single most important pattern across all five sources. Adopt it even in a single-process system ã the day you need a second process, you'll thank yourself.
2. **Persist pause state in the same atomic transaction as agent state.** Avoid the "paused? yes/no" flag pattern ã use the absence of a "next" event as the implicit pause state.
3. **Validate graph/topology on restore.** Cheap (hash compare), prevents silent corruption, gives a clear error.
4. **Reserve schema slots for control signals** (negative indices, underscore prefixes, dedicated event types) so user state cannot collide.
5. **Use cooperative cancellation at a known boundary.** Document the latency honestly ã "pause takes effect within one step / one heartbeat / one superstep".
6. **Buffer external events (signals, timers, cancels) during pause and deliver on unpause.** Temporal's strongest pattern; letta's summarizer-aware pause is a small-scale version.
7. **Rate-limit-exempt the resume path.** Openhands does this deliberately; LRU-style concurrency sweeps should not starve the resume call.

### Patterns to avoid or delay

1. **In-memory-only pause state.** Lost on every crash; forces the caller to re-detect pause on every resume.
2. **Conflated "cancel" and "deny".** Letta does this; the trail becomes ambiguous and debugging "why did this tool call not run?" is hard.
3. **Two parallel lock systems without a unifying primitive.** Letta has this; the conversation lock and the per-step DB poll can disagree.
4. **Caching pause state in a column the runtime doesn't control.** Openhands tried this, dropped the column (`012.py:19-29`), and now always asks the runtime. Save yourself the migration.
5. **Built-in idempotency without an explicit contract.** langgraph's node re-execution model is simple but means the application owns idempotency. If you don't have a clear story for it, document the gap.
6. **Synchronous interrupt of LLM HTTP requests.** No source does this; don't be the first.

### Decision rules

- **If your agent may live across a process restart:** pause must be persisted, and the pause state must be in the same transaction as the agent state.
- **If multiple operators may answer pending requests:** you need a deterministic correlation key (request_id, interrupt_id) and a typed resume API.
- **If your agent runs inside a sandbox:** pause at the sandbox boundary; treat the agent-internal state as opaque; rotate session keys on resume.
- **If your agent uses long-running LLM calls or tools:** use cooperative cancellation; document the latency; do not promise immediate stop.
- **If your workflow may run across multiple clusters:** pause must be a first-class event with replication support ã temporal's model is the cleanest.
- **If your pause may outlast a process restart by days:** the persistence layer must support buffered external events (signals, timers) and replay-safe resume.

## Anti-Patterns / Caution Signs

- **Pause state lives only in memory.** Check for `is_paused` flags on objects that are not persisted.
- **Cancel and deny share a code path.** If cancelling a paused-on-approval run produces a "denied" trail, you have this problem.
- **No graph/topology validation on restore.** If you can load a checkpoint from a different workflow version silently, this is missing.
- **Resume API does not specify which pending request to answer.** If the caller can only do "resume", not "approve request X", multi-party coordination is undefined.
- **Session keys are reused across pause/resume cycles.** A leaked key remains valid; you should rotate on resume (openhands pattern).
- **Pause status is cached locally and the runtime is the source of truth.** The dropped `is_paused` column in openhands is the cautionary tale.
- **Two status enums that must be reconciled at every call site.** Letta has `RunStatus`, `StepStatus`, `StopReasonType`, `JobStatus`; the more of these you have, the more places they can drift.
- **Cooperative cancellation without a documented latency bound.** "Pause takes effect at the next step" is fine; "pause takes effect within one SSE poll (0.5 s)" is better; "pause takes effect at the next LLM call completion" is risky.
- **Auto-resume on subsequent requests.** Openhands deliberately does not do this; auto-resume can hide genuine operator intent.
- **No `Idempotency-Key` header on the resume RPC.** Openhands does not have one; a duplicate `POST /resume` reissues a new session key each time.

## Notable Absences

These are gaps that appeared in multiple sources or are conspicuous by their absence.

- **No multi-party resume arbitration primitive.** None of the five sources provides a "claim this pending request" or "first-responder wins" abstraction. Concurrency safety comes from event-loop serialization, single-writer checkpointer, conversation lock, or no coordination.
- **No built-in `on_world_changed` hook.** Every source punts world-staleness detection to the application layer. agent-framework documents the contract explicitly but does not implement it; langgraph expects it in the node body; letta validates tool-call ids but does not detect external staleness.
- **No first-class "fork paused run into two branches" API.** Temporal supports `Reset` which terminates the current run and starts a new one, but this is not "fork"; langgraph's `update_state` with `source="fork"` creates a new checkpoint lineage, but the original is not paused.
- **No documented SLO for "time from cancel call to agent loop exit".** Letta has a 0.5 s SSE-poll bound but the agent-loop poll can be much longer; temporal has a heartbeat-driven bound but no SLA.
- **No automatic node-idempotency in langgraph.** Documented explicitly as application responsibility.
- **No general-purpose `pause` primitive in agent-framework that is not HITL-shaped.** Pausing requires an expected response; pure "sleep until timestamp" is delegated to the durable-task host.
- **No `RequestPauseChildWorkflowExecution` command in temporal.** Pause is API-driven only, not invocable from inside a parent workflow.
- **No automatic cleanup of abandoned paused entities.** Openhands' paused sandboxes can sit forever; letta's paused runs accumulate `Step` rows; agent-framework's checkpoint chain grows without bound.
- **No app-server-level human-in-the-loop approval API in openhands.** `confirmation_mode` is forwarded as a hint; the app server has no list/approve/reject endpoints.
- **No `RETRY_STATE_PAUSED` in temporal.** Comment `TODO: uncomment once RETRY_STATE_PAUSED is supported` (`mutable_state_impl.go:6817-6818`) shows this is acknowledged future work.
- **No universal "halt without expecting a response" primitive in agent-framework's Python surface.** The .NET mirror has `DurableHaltRequestedEvent`; the Python side does not yet expose an equivalent.

### agent-framework (9/10)
- Three explicit layers: in-process HITL (`request_info`) ã file/in-memory checkpoint ã Azure Durable Task. Each is separately tested (`test_request_info_and_response.py`, `test_checkpoint_validation.py`, `test_09_dt_workflow_hitl.py`).

### langgraph (9/10)

- `WRITES_IDX_MAP` reserves `INTERRUPT=-3, RESUME=-4` so control writes cannot collide with user writes.
- xxh3-derived `Interrupt.id` from namespace allows multiple interrupts per node; legacy payloads round-trip via `Interrupt.from_ns`.
- `RunControl.request_drain()` is cooperative; `GraphDrained` exception propagates; checkpoint persisted.
- Lifecycle callbacks `GraphInterruptEvent` / `GraphResumeEvent` are first-class (`callbacks.py:42-77`).
- Subgraph interrupts bubble up as a combined `GraphInterrupt` (single tuple to root); subgraph attribution requires decoding `Interrupt.id`.
- Node bodies re-execute from the top on resume; application owns idempotency.
- `_put_checkpoint` exit-mode logic uses `metadata is self.checkpoint_metadata` identity check (`_loop.py:1075`) ã flagged for replacement.
- Cross-process resume requires Postgres/SQLite `BaseCheckpointSaver`; no built-in lock for "another worker is currently running this thread".

### letta (7/10)
- Three pause primitives: HITL approval (`StopReasonType.requires_approval`), client-side tool, cooperative cancel.
- `Run` + `Step[]` + `Message[]` is the persistence model; the database row is the source of truth.
- Pause granularity is fixed at the step boundary ã in-flight LLM calls and tools run to completion.
- OTID-based idempotency: same OTID recovers the in-flight run; different OTID is treated as new work.
- Background streaming survives process restart via Redis SSE stream `sse:run:{run_id}` with 3 h TTL; `GET /v1/runs/{run_id}/stream?starting_after=N` allows reattachment.
- Two lock systems: Redis conversation lock + per-step DB poll of `Run.status`. Not unified.
- `cancel_run` is idempotent for non-approval runs (`if run.stop_reason not in {requires_approval}: return`); cancellation of a paused-on-approval run materializes a denial `ToolReturn`, conflating cancel and deny.
- Multi-agent pause/resume via `SleeptimeMultiAgentV3/V4` is delegated but opaque ã no per-sub-agent `run_id` exposed in the public API.
- Lettuce integration is a stub (`lettuce_client_base.py:9-101`); local cancel marks the DB, remote cancel must come from Lettuce.

### openhands (6/10)
- Pause/resume is a sandbox-level operation: Docker `container.pause()`, remote `POST /pause`, or `psutil.Process.suspend()`.
- Session API key is **invalidated on pause** and a **new one returned on resume** (`remote_sandbox_service.py:501-540`); strong operational safeguard.
- `is_paused` column was added in migration 011 and dropped in migration 012 because external terminations left the cache stale ã runtime `/list` is now the source of truth.
- `pause_old_sandboxes` is a sweep that runs on every `start_sandbox` and `resume_sandbox` ã concurrency control is heuristic, not transactional.
- No human-in-the-loop approval API at the app-server layer; `confirmation_mode` is forwarded to the agent server as `ConfirmationPolicy` (`AlwaysConfirm` / `ConfirmRisky` / `NeverConfirm`).
- `app_conversation_router.py:170-172` collapses STARTING, ERROR, MISSING into 404 for paused sandboxes, masking the resume-able from the non-resume-able cases.
- Rate-limit explicitly exempts `POST /sandboxes/{id}/resume` (`middleware.py:18`); bursty reopen polling cannot starve the resume path.
- The agent server inside the sandbox is treated as opaque; whether a paused agent can truly resume at the same step depends on the SDK's own session persistence (ACP `acp_isolate_data_dir` + `base_state.json`).
- `update_openapi.py` only knows V0 paths; V1 resume endpoints are not surfaced in the public OpenAPI document.

### temporal (9/10)
- Pause/unpause are first-class history event types (`EVENT_TYPE_WORKFLOW_EXECUTION_PAUSED/UNPAUSED`) with their own factory and rebuilder paths.
- Atomic pause via `GetAndUpdateWorkflowWithNew` ã `UpdateWorkflowExecutionAsActive` (same transaction as all workflow state mutations).
- Stamp-based staleness detection: pause bumps `ai.Stamp`, transfer queue drops task if `ai.Paused` (`transfer_queue_active_task_executor.go:259-262`).
- Heartbeat-driven in-flight interrupt: `RecordActivityTaskHeartbeat` returns `ActivityPaused`; worker is responsible for stopping.
- Idempotency on pause via `RequestId` (`pauseworkflow/api.go:61-66`); precondition-only on unpause (`unpauseworkflow/api.go:58-62`).
- Signals/timers/cancels buffered while paused; delivered atomically on unpause. Order preserved.
- Per-run pause: reset while paused starts a fresh RUNNING run; pause does not carry over to continue-as-new.
- Pause bumps activity stamps so the existing task replication machinery propagates the paused state to passive clusters.
- Pause does not synchronously interrupt in-flight activities; relies on worker heartbeats.
- `RETRY_STATE_PAUSED` not yet supported; retry-while-paused reported as `RETRY_STATE_IN_PROGRESS` (`mutable_state_impl.go:6803-6819`).
- `PauseActivityExecution`/`UnpauseActivityExecution` plural forms explicitly stubbed as `Unimplemented` in frontend (`workflow_handler.go:7306-7317`) ã future replacements for the singular API.

## Open Questions

These are questions raised in one or more source reports that the source did not answer and that future study could address.

- **What is the operational guarantee when the world changes during a multi-day HITL pause?** agent-framework rehydrates internal state and pending requests deterministically but does not call any executor "freshen" hook. The framework contract is "I remember what I knew; you remember what the world knows" but there is no `on_world_changed` primitive.
- **Is there a documented way to fork a checkpoint into two independent runs?** agent-framework's `previous_checkpoint_id` chain structure could in principle support it, but `CheckpointStorage` has no `fork` or `branch` method. langgraph's `update_state` with `source="fork"` is the closest analog and it operates on state, not on paused runs.
- **How does resume-after-process-restart interact with `current_run_id` in letta's legacy `LettaAgent`?** The legacy agent uses a per-instance `self.current_run_id`; if the process restarts, that field is lost. The agent_id-based re-derivation is implicit and not documented.
- **Can two clients collaborate on resuming a single run in letta?** The shared Redis stream allows read fan-out, but write fan-out (one client approves tool A while another approves tool B) is undefined. The current `approvals` payload is treated as a single batch.
- **What does the agent server in openhands do internally on resume?** The app server treats the agent server as opaque. The only evidence is a comment at `live_status_app_conversation_service.py:1875-1877` acknowledging the SDK's role.
- **What is the propagation latency of temporal's pause from server to in-flight activity?** Documented mechanism is heartbeat round-trip; no explicit SLA in source.
- **Is there a way to pause a child workflow from a parent workflow command in temporal?** No `RequestPauseChildWorkflowExecution` command found; pause is API-driven only.
- **How does temporal's pause interact with StartDelay timers or cron schedules?** No explicit handling found; timer fires but workflow task is not scheduled.
- **Can multiple clients safely resume the same run in langgraph?** Not safely ã no CAS-style resume on `Command(resume=...)`; concurrent resumes race first-write-wins.
- **What is the maximum size of a checkpoint payload in agent-framework?** No documented limit; `FileCheckpointStorage` writes a single JSON file per checkpoint; large state (LLM context, embeddings) will linearly increase save/load latency with no batching or sharding.

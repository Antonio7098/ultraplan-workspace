## Executive Summary

Across the six sources, state taxonomy is the dimension where library-shaped frameworks and application-shaped products most visibly diverge, but they converge on a small set of mechanics. Every source recognises the same eight-to-twelve categories — conversation, execution, tool/task, memory, artifact, approval, eval, runtime-config, identity/coordination, cache, secret, and schema — and each has named owner modules with explicit durability assignments. What differs is the *shape* of the ownership boundary:

- **Library frameworks** (langgraph, agent-framework) expose state through abstract protocols (`BaseChannel`, `HistoryProvider`, `BaseStore`) where the *persistence seam* is part of the public API. Crash-survival is enforced by conformance-style contracts, schema versioning, and per-key versioning.
- **Application servers** (langfuse, letta, openhands) place state into concrete storage tiers (Postgres/ClickHouse/S3/Redis, or Postgres + Redis, or SQL + filesystem) and document ownership in schema files and dependency-direction rules. Crash-survival is enforced by foreign keys, soft-delete, and durable coordination locks.
- **Durable execution engines** (temporal) model state as an event-sourced projection of a History branch, fenced by `RangeID` and `DBRecordVersion`. Mutation is a function of the History log, not of the in-memory view.

The dimension's central question — *which state must survive a crash?* — is answered clearly by every source, but the *guarantee* it offers varies: langgraph (9/10) offers conformance-tested checkpointing; temporal (7/10) offers range-fenced conditional writes; agent-framework (8/10), langfuse (8/10), and letta (8/10) offer typed durable schemas with operational safeguards; openhands (7/10) splits the question across a control-plane/data-plane boundary. The dominant convergence is **append-only event logs as the source of truth, with derived projections and tiered caches**; the dominant divergence is **how and where the event log is stored**.

## Core Thesis

A mature state taxonomy requires three things that no single source fully nails at 10/10: (1) an explicit, named owner per state class; (2) a durable-versus-ephemeral decision that is mechanically enforced, not merely documented; and (3) a recovery protocol that explains exactly which state survives a crash and how the rest is reconstructed. The strongest sources achieve this by treating the persistence boundary as a public contract (langgraph's `BaseCheckpointSaver` conformance suite), by fencing every durable write with a conditional update token (temporal's `RangeID`/`DBRecordVersion`), or by anchoring all conversation history to an append-only event log that is replayed into derived projections (langfuse's `InAppAgentEvent`, temporal's `History`, openhands's per-event JSON files, agent-framework's `WorkflowCheckpoint`). The weakest source (openhands) is weak because half the state lives in a sandbox that this source cannot see — a structural, not implementation, deficit. The dimension's score ranges 7-9 not because any source lacks a taxonomy, but because each pays a different tax for being a library, an application, or a durable engine.

## Approach Models

The six sources cluster into three approach models. Sources within a cluster solve the dimension's problem with similar mechanics; differences between clusters are valid responses to different product shapes.

### Cluster A — Library frameworks (state as a first-class public API surface)

- **agent-framework** and **langgraph** both expose persistence as a pluggable protocol — `HistoryProvider` / `BaseCheckpointSaver` / `BaseStore` / `BaseCache` — and place the *shape* of state in the framework's public contract.
- They differ on granularity. LangGraph uses *channels* as the per-key state machine with explicit reducers (`LastValue`, `BinaryOperatorAggregate`, `Topic`, `DeltaChannel`); agent-framework uses a *namespaced dict* on `AgentSession` keyed by `source_id`, where each provider picks its own type and registers a `(de)serializer`.
- **Why this works**: a developer adding a new state category cannot accidentally pick the wrong durability tier — the protocol *is* the tier. Crash-survival is provable via conformance tests (langgraph's `checkpoint-conformance` suite, `agent-framework`'s round-trip tests).
- **Why it is overkill for some**: the per-channel or per-`source_id` machinery pays a tax in serialization cost and mental model overhead that an application ORM does not pay. A library that exposes the abstraction layer commits to maintaining it forever.

### Cluster B — Application servers (state as concrete rows in tiered stores)

- **langfuse**, **letta**, and **openhands** each define a fixed taxonomy of durable models/tables and route them to specific storage backends. The taxonomy is documented in schema files (Prisma, SQLAlchemy, SQLAlchemy + files), not in code abstractions.
- They differ on the number of tiers and the placement of agent-specific state:
  - langfuse uses **four** tiers (Postgres for control-plane, ClickHouse for telemetry, S3 for raw events, Redis for queues/caches), with the in-app agent using event-sourcing on top.
  - letta uses **two primary** tiers (Postgres for everything durable, Redis for ephemeral coordination) with optional ClickHouse / socket telemetry backends and a git-backed memory repo on an external `StorageBackend`.
  - openhands uses **two** tiers (SQL metadata + filesystem event log) within the app_server, plus an external sandbox that owns runtime execution state.
- **Why this works**: the developer reads a schema file and sees the entire durable state. Multi-store pointer-indirection (langfuse, letta's git repo) lets one logical entity span stores without losing durability.
- **Why it is brittle**: schema changes touch every model and every migration. Cross-store consistency is eventual, not transactional. None of these sources has a single canonical state-map document — it must be reconstructed from schema + migrations + queue contracts.

### Cluster C — Durable execution engine (state as event-sourced projection)

- **temporal** stands alone. State is a projection of `History` events; the in-memory `MutableStateImpl` is reconstructed on demand and fenced by `RangeID` (shard) and `DBRecordVersion` (per-execution).
- **Why this works**: the History branch is the canonical artifact; replay is the recovery protocol. Conditional updates with monotonic fences prevent concurrent writers from corrupting each other across ownership transfers. `CHASM` generalises the model for new state archetypes without changing the executions-row layout.
- **Why it is unique**: the entire architecture is *about* state taxonomy — every other source treats state taxonomy as a supporting concern. Temporal treats it as the product.

## Pattern Catalog

The following patterns appear in multiple sources or are unusually clearly demonstrated in one.

### Pattern 1 — Append-only event log as the source of truth for conversation state

**Problem.** Conversation history must survive process crashes, schema changes, partial writes, and replay. A mutable message table drifts; an in-memory list is lost; a single JSON file is slow to parse.

**Sources.**
- langfuse: `InAppAgentEvent` is an append-only, `sequenceNumber`-ordered Postgres log; conversation messages are *derived* by replaying events through `getMessagesFromEvents` (`persistence.ts:464`).
- temporal: `history_node` + `history_tree` Cassandra tables are the canonical artifact; `MutableStateImpl` is a projection over History (`docs/architecture/history-service.md:113-132`).
- openhands: file-backed `Event` JSON log per conversation (`event_service_base.py:66-84`); trajectory export replays it.
- langgraph: `Checkpoint` with `pending_writes` is an append-style log of unconsumed writes; `BaseCheckpointSaver.put_writes` (`checkpoint/base/__init__.py:176-722`) persists them.
- agent-framework: `WorkflowCheckpoint` captures committed-only state per superstep (`_checkpoint.py:30-117`), and `DurableAgentState.conversation_history` is an append of request/response entries (`_durable_agent_state.py:365-474`).

**Why it works.** Append-only logs eliminate the update-vs-insert ambiguity and make replay the natural recovery protocol. They also enable compaction, time-travel debugging, and event-sourced projections without dual-writes.

**When to copy.** Any time conversation history must survive crashes and the read pattern tolerates replay cost. Particularly valuable when tool-call lifecycles are interleaved with messages (langfuse, openhands).

**When it is overkill.** For toy agent demos or short-running tools where the entire conversation fits in one process and a JSONL file suffices.

**Evidence.** Per-source reports cited above; core abstractions at `langfuse:packages/shared/prisma/schema.prisma:239`, `temporal:api/persistence/v1/executions.pb.go:135`, `openhands:openhands/app_server/event/event_service_base.py:66`.

### Pattern 2 — Source-of-truth with derived projections (info vs live)

**Problem.** Some queries need cheap reads; others need full-fidelity state. Storing both invites drift; storing only one makes some queries expensive.

**Sources.**
- openhands: `AppConversationInfo` (durable) vs `AppConversation` (info + live sandbox + live execution status) (`app_conversation_models.py:112`, `:173`).
- openhands: `SandboxRecord` (durable id/owner) vs `SandboxInfo` (live runtime) (`sandbox_models.py:33`, `:48`).
- langfuse: `InAppAgentConversation` is the durable record; messages are derived from `InAppAgentEvent` (`persistence.ts:464`).
- agent-framework: `AgentSession.state` is durable; `SessionContext` is per-invocation and derived (`_sessions.py:154-348`).
- letta: `Agent` row is durable; `AgentState` Pydantic is rebuilt via `to_pydantic` (`letta/orm/agent.py:214-315`).

**Why it works.** The boundary between "must persist" and "can be re-fetched live" is explicit. Live state (sandbox status, agent execution status) is fetched over HTTP and never persisted.

**When to copy.** Any time a logical entity spans a process boundary (e.g., sandbox + control plane) or a read pattern wants a cheap cached value.

**Evidence.** Per-source reports cited above.

### Pattern 3 — Tiered persistence by access pattern

**Problem.** One database cannot serve high-volume appends, low-latency coordination, and durable control-plane reads with the same operational profile.

**Sources.**
- langfuse: Postgres (control), ClickHouse (telemetry, `ReplicatedReplacingMergeTree(event_ts, is_deleted)` at `0001_traces.up.sql:23`), S3 (raw events at `processEventBatch.ts:275`), Redis (queues + caches at `queues.ts:342`).
- letta: Postgres (durable), Redis (locks + idempotency + SSE streams at `redis_client.py:194-289`), ClickHouse + Unix socket + Postgres (multi-backend dual-write for provider traces at `telemetry_manager.py:50-80`).
- agent-framework: file (JSONL), Azure Durable Entity (cloud), Redis (history), and an in-memory default (`_sessions.py:814-1134`, `redis/_history_provider.py:20-194`).
- temporal: Cassandra/MySQL/Postgres/SQLite (durable), in-memory `common/cache.Cache` (workflow/events/namespace), `dynamicconfig.Setting[T]` (runtime config).
- openhands: SQL (metadata), filesystem (events + settings), Docker (live sandbox state).

**Why it works.** Each tier picks the right durability/latency/throughput trade-off. Cross-tier references are tracked via pointers (langfuse's `event_log` table maps `(project_id, entity_type, entity_id)` to `bucket_name/bucket_path` at `0007_add_event_log.up.sql:1`).

**When to copy.** Whenever a single database becomes the operational bottleneck for one specific access pattern (e.g., high-volume telemetry, low-latency locks).

**When it is risky.** Multi-store consistency becomes eventual; cross-store FKs are impossible; operators must learn N systems. langfuse explicitly admits cross-store references without FK (`schema.prisma:1101`).

**Evidence.** Per-source reports; langfuse `packages/shared/clickhouse/migrations/clustered/0007_add_event_log.up.sql:1`, letta `letta/services/telemetry_manager.py:50-80`.

### Pattern 4 — Atomic write via temp file + fsync + rename

**Problem.** A crash mid-write can leave a file in a partially-written state, corrupting durable state.

**Sources.**
- openhands: `LocalFileStore.write` writes temp file, `f.flush()` + `os.fsync`, then `os.replace` (`local.py:26-43`).
- agent-framework: `TodoFileStore._save_state_sync` (`_todo.py:432-444`), `FileCheckpointStorage.save` (`_checkpoint.py:317-322`), `_memory._atomic_write_text` (`_memory.py:119-132`) all use the temp + `os.replace` pattern, documented as "a crash, OOM, or disk-full mid-write leaves the previous file intact".
- letta: relies on Postgres transactions rather than filesystem atomicity.

**Why it works.** POSIX rename is atomic within a filesystem; a crash before rename leaves the original intact, after rename leaves the new file intact.

**When to copy.** Any time state lives in flat files and durability matters. Document the pattern in a comment so future contributors do not "simplify" it away.

**Evidence.** `openhands:local.py:31-43`, `agent-framework:_todo.py:432-444`, `agent-framework:_checkpoint.py:317-322`.

### Pattern 5 — Pending vs committed buffering at the seam

**Problem.** Concurrent writers within the same logical step need a place to stage writes that aren't yet visible; the seam between in-flight and durable state must be explicit.

**Sources.**
- agent-framework: `State` class with `_committed` and `_pending` buffers; `commit()` flushes at superstep boundaries, `discard()` rolls back (`_workflows/_state.py:6-127`). Checkpoints capture only committed state (`_checkpoint.py:56-58`).
- langgraph: `pending_writes` are stored alongside `Checkpoint`; on resume, `channels_from_checkpoint` reconstructs via `saver.get_delta_channel_history` (`pregel/_checkpoint.py:136-226`).
- temporal: `WorkflowMutation` (incremental) vs `WorkflowSnapshot` (full) envelope; `RangeID`/`DBRecordVersion` conditional writes gate the durability (`data_interfaces.go:344-398`).
- agent-framework: durable activity bridge returns diff format `{shared_state_updates, shared_state_deletes, sent_messages, events, pending_request_info_events}` against pre-snapshot (`_workflows/activity.py:37-186`).

**Why it works.** A crash between "wrote pending" and "committed" is recoverable by replaying pending writes; a crash after "committed" but before "checkpointed" is still recoverable because the next superstep will re-derive from the prior checkpoint. The diff-format response lets the durable layer reconstruct state without trusting the in-memory layer.

**When to copy.** Whenever multiple writers may modify the same logical state within one tick/step.

**Evidence.** `agent-framework:_workflows/_state.py:6-127`, `langgraph:pregel/_loop.py:1507-1524`, `temporal:common/persistence/data_interfaces.go:344-398`.

### Pattern 6 — Schema versioning at the durable boundary

**Problem.** Schema migrations happen. Old data must still deserialize; new data must not be silently downgraded.

**Sources.**
- agent-framework: `DurableAgentState.SCHEMA_VERSION = "1.1.0"` (`_durable_agent_state.py:395`); `from_dict` warns and resets on mismatch (`_durable_agent_state.py:426-430`). JSON schema with `schemaVersion` discriminator (`schemas/durable-agent-entity-state.json:208-217`).
- langgraph: `LATEST_VERSION = 4` (`pregel/_checkpoint.py:21`); `_migrate_checkpoint` (`pregel/main.py:1135-1142`) is the migration hook.
- letta: 167 Alembic revisions under `alembic/versions/` cover every model and index.
- openhands: `conversation_version='V1'` scopes the `conversation_metadata` table (`sql_app_conversation_info_service.py:65`).
- temporal: protobuf schema files in `api/persistence/v1/*.pb.go`; schema evolution is centralized via generated code.

**Why it works.** A versioned deserializer can detect "old data" vs "future data" and either migrate forward or refuse to read. Without versioning, silent corruption becomes inevitable.

**When to copy.** Whenever state is persisted as a structured blob (JSON, protobuf, pickle). Particularly important when state outlives a process restart — by definition, the deserializer may be a newer version than the serializer.

**Evidence.** `agent-framework:_durable_agent_state.py:395`, `langgraph:pregel/_checkpoint.py:21`, `openhands:sql_app_conversation_info_service.py:65`.

### Pattern 7 — Per-session locks stored in a `WeakKeyDictionary`

**Problem.** Per-session state needs per-session locks, but locks for dead sessions leak memory in long-running services.

**Sources.**
- agent-framework: `_todo.py:482-492`, `_memory.py:1019-1038`, `_background_agents.py:294-299` keep per-session `asyncio.Lock` instances in `weakref.WeakKeyDictionary` keyed by `AgentSession`. Comment at `_todo.py:483-484` is explicit: "so per-session locks are evicted automatically when the session is GC'd".

**Why it works.** `WeakKeyDictionary` evicts entries when the key (the `AgentSession`) is garbage-collected. No timer-based cleanup; no manual eviction.

**When to copy.** Any time you cache per-session locks or per-session resources in a long-running server.

**Evidence.** `agent-framework:_todo.py:482-492`.

### Pattern 8 — Idempotency via client-supplied token

**Problem.** Network retries can cause duplicate agent runs; the server needs to detect and either replay or reject.

**Sources.**
- letta: `set_otid_run_mapping` hashes all message OTIDs and stores `OTID_RUN_PREFIX:{hash} -> run_id` in Redis (`redis_client.py:250-289`); duplicate POSTs return the original run stream.
- openhands: pending messages queued under `task-{uuid}` are rekeyed to the real conversation id once known (`pending_message_service.py:168-185`).
- langfuse: events grouped by `entityType-body.id`; update events sorted last to survive out-of-order processing (`processEventBatch.ts:213`, `:434`).
- temporal: `dbRecordVersion` + `NextEventID` conditional updates make retries safe by construction (`data_interfaces.go:348`, `:373`).

**Why it works.** A hash of input content → stored run id acts as a content-addressable idempotency layer. Clients can retry without worrying about double-billing or duplicate work.

**When to copy.** Any time an HTTP endpoint may be retried by the client (which is to say: always).

**Evidence.** `letta:letta/data_sources/redis_client.py:250-289`, `openhands:pending_message_service.py:168-185`, `langfuse:processEventBatch.ts:213`.

### Pattern 9 — Source-of-truth duplication with reconciliation

**Problem.** A denormalized cache (e.g., a JSON column listing current messages) speeds reads but drifts from the relational source of truth.

**Sources.**
- letta: `Agent.message_ids` (JSON cache) vs `ConversationMessage.in_context` + `position` (relational truth); reconciler is `ConversationManager.update_in_context_messages` (`conversation_manager.py:750-797`).
- langfuse: `event_log` ClickHouse table is the pointer index into S3; ClickHouse traces are derived from S3 by the worker (`processEventBatch.ts:281-289`).
- openhands: denormalized metrics columns on `conversation_metadata` are reduced from `ConversationStateUpdateEvent` stats events; hardcoded `total_tokens=0` (`sql_app_conversation_info_service.py:367`) is a live example of drift.

**Why it works.** The denormalized form supports cheap O(1) reads; the relational form is the source of truth and can be used to repair drift.

**When it is risky.** If the reconciler is called only at specific code paths (letta), drift can persist for arbitrary intervals. Openhands's metrics drift is unchecked by design (best-effort `process_stats_event` swallows exceptions).

**Evidence.** `letta:letta/services/conversation_manager.py:750-797`, `openhands:sql_app_conversation_info_service.py:367`.

### Pattern 10 — Pin + release + finalizer for in-memory mutable state

**Problem.** An in-memory mutable projection must not outlive its ownership boundary; otherwise concurrent writers can corrupt each other.

**Sources.**
- temporal: `cacheImpl` pins items, attaches a `finalizer.Finalizer` that calls `wfContext.Clear()` on evict, and supports `GetOrCreateWorkflowExecution` / `GetOrCreateCurrentExecution` (`service/history/workflow/cache/cache.go:60-148`).
- langgraph: `InMemoryCache` with TTL eviction; `PregelLoop.put_writes` waits on `_delta_write_futs` before the next `_checkpointer_put_after_previous` (`pregel/_loop.py:1507-1524`).
- langfuse: `LocalCache` LRU with `ttlAutopurge:false`, `allowStale:false`, and a disposer that records evictions (`localCache.ts:18`, `:31`).

**Why it works.** Pin prevents eviction mid-request; finalizer guarantees cleanup; TTL bounds memory growth.

**When to copy.** Any time you cache mutable per-entity state in a long-running process.

**Evidence.** `temporal:service/history/workflow/cache/cache.go:60-148`, `langgraph:pregel/_loop.py:199-205`.

### Pattern 11 — Pluggable serialization with allowlists

**Problem.** `pickle` and similar formats are convenient but unsafe — an attacker who can write to a checkpoint file can achieve code execution.

**Sources.**
- agent-framework: hybrid JSON + base64-pickle for checkpoints; `_RestrictedUnpickler` with allowlist of built-in safe types + `agent_framework.*` + `openai.types.*` (`_checkpoint_encoding.py:18-145`). Docs explicitly disclaim pickle safety: "defense in depth, not a security boundary".
- langgraph: `STRICT_MSGPACK_ENABLED` gate plus `allowed_msgpack_modules` enforces a per-graph module allowlist, mitigating deserialization RCE (`serde/jsonplus.py:88-95`).
- langfuse: events grouped + deduped by `entityType-body.id`; messages sorted by event timestamp on ingest (`processEventBatch.ts:213`).

**Why it works.** Allowlisting the deserialization target prevents RCE via crafted payloads. The cost is verbosity in the serializer.

**When it is risky.** The allowlist must be maintained as new types are added; missing a type means the checkpoint cannot be restored. The agent-framework docs are explicit that this is not a security boundary — the storage backend must be access-controlled.

**Evidence.** `agent-framework:_checkpoint_encoding.py:18-145`, `langgraph:serde/jsonplus.py:88-95`.

### Pattern 12 — Strict ownership boundaries via dependency direction

**Problem.** A monorepo with shared schemas and runtime code can rot if any package can reach into any other.

**Sources.**
- langfuse: queue name/payload schemas centralized in `packages/shared/src/server/queues.ts`; schema ownership in `@langfuse/shared`; one-directional dependency graph (`.agents/AGENTS.md`).
- temporal: each module has a clear owner — `common/persistence` for durable, `service/history/workflow` for in-memory projections, `common/dynamicconfig` for runtime config. The architecture document pins this (`docs/architecture/history-service.md`).
- letta: ORM/Pydantic split is the canonical boundary; ORM is storage, Pydantic is wire (`letta/orm/sqlalchemy_base.py:979-991`).

**Why it works.** A one-directional dependency graph is mechanically enforceable (e.g., via import lint rules). When violated, the rule fires.

**When to copy.** Any multi-package monorepo where state ownership is distributed.

**Evidence.** `langfuse:.agents/AGENTS.md`, `temporal:docs/architecture/history-service.md`.

## Key Differences

The differences across sources are best understood as responses to product shape, not as quality differences.

### Library vs application vs engine

- **Library frameworks** (langgraph, agent-framework) treat state as a *protocol* — the persistence seam is part of the public API. They expose `BaseChannel`/`HistoryProvider`/`BaseStore` and ship multiple concrete implementations.
- **Application servers** (langfuse, letta, openhands) treat state as a *schema* — durable models live in Prisma/SQLAlchemy/SQL files, and concrete stores are wired in `config.py`.
- **Durable execution engines** (temporal) treat state as a *projection* — the History branch is canonical, the in-memory view is derived.

### Granularity of the state machine

- LangGraph works at the *channel* level: each schema field is a `BaseChannel` subclass with explicit reducers.
- agent-framework works at the *provider* level: each `ContextProvider`/`HistoryProvider`/`Store` writes into a `source_id` namespace under `AgentSession.state`.
- letta works at the *model* level: each `__pydantic_model__` is the unit of persistence.
- openhands works at the *service* level: each `EventService`/`SettingsStore`/`SandboxService` is the unit of storage abstraction.
- temporal works at the *execution* level: per-workflow mutable state is the unit of conditional write.

### Crash recovery strategy

- langgraph: conformance tests guarantee that any `BaseCheckpointSaver` implementation handles the same set of invariants (`PUT`, `PUT_WRITES`, `GET_TUPLE`, `LIST`, `DELETE_THREAD`, etc. at `capabilities.py:15-50`).
- agent-framework: `WorkflowCheckpoint.version="1.0"` + `register_state_type()` registry + `_STATE_TYPE_REGISTRY` (`_sessions.py:42-147`) ensure cold-start processes can round-trip unknown types.
- langfuse: S3 raw events are the durable source-of-truth; ClickHouse is rebuilt from S3 by the worker; if S3 write fails, ingestion aborts (`processEventBatch.ts:284`, `:324`).
- letta: Postgres deadlock retries with exponential backoff (`_DEADLOCK_MAX_RETRIES = 3`, `letta/orm/sqlalchemy_base.py:40-41`); OTID idempotency for HTTP retries.
- openhands: control-plane durable (SQL + files); data-plane state is fetched live from the sandbox agent-server over HTTP.
- temporal: `RangeID`/`DBRecordVersion` conditional writes; `appliedEvents` map is in-memory only and explicitly TODO-persisted.

### Cache discipline

- langgraph: explicit `BaseCache` with TTL; node-result cache, opt-in.
- temporal: explicit `common/cache.Cache` interface; workflow/events/namespace caches; pin + finalizer pattern.
- langfuse: explicit `LocalCache` (in-memory LRU); Redis caches for API key + eval config.
- agent-framework: implicit dict-based caches (`AgentSession.state[source_id]`).
- letta: in-memory trackers on `LettaAgentV3` are not snapshotted and drift on crash.
- openhands: in-process `AppServerConfig` rebuilt from env per process; no application-level cache.

### Eval state

Eval state is the dimension's most consistent *absence*. langfuse stores `JobConfiguration` and `JobExecution` for offline evals (`schema.prisma:1046`, `:1083`); letta stores `Step.feedback` as a free-text string (`step.py:78-86`); openhands's closest analog is `event_callback_result` (`sql_event_callback_service.py:76`); langgraph and agent-framework leave eval entirely to the caller; temporal has no eval concept. No source treats eval as a first-class state category with its own owner module.

### Approval state

Approval state is more developed. langgraph has dedicated `Interrupt` records inside `pending_writes` with `id = xxh3_128_hexdigest` of checkpoint namespace (`types.py:572`). agent-framework has `ToolApprovalState` with rules, queued requests, and collected responses (`_tool_approval.py:86-217`). letta encodes approvals on `Message` (`role="approval"`, `approval_request_id`, `approve`, `denial_reason`). openhands stores the *configuration* (`conversation_settings.confirmation_mode`, `security_analyzer`) but defers per-action records to the SDK. temporal treats approvals as Update state machines and HSM-based Callback state machines.

## Tradeoffs

| Choice | Benefit | Cost | Best-fit | Failure mode | Alternative |
|--------|---------|------|----------|--------------|-------------|
| Append-only event log as source of truth | Replay is the recovery protocol; compaction-friendly; time-travel-friendly | Read path requires replay or materialised view; storage grows unbounded without GC | Long-running agents, multi-turn conversations, regulated audits | Forgetting to compact leads to runaway storage; using wrong cursor leads to replaying partial state | Mutable message table (letta) — cheaper reads, no replay |
| Tiered persistence (N storage backends) | Each tier picks the right durability/latency/cost | Cross-store consistency is eventual; ops surface multiplies; no FKs across stores | High-volume telemetry + low-latency coordination + durable control | Operators debug at 3am without docs; cascade deletes miss dangling references | Single Postgres (letta) — simpler ops, less flexibility |
| Schema versioning at the durable boundary | Old data still deserialises; new schema flagged on mismatch | More code per state class; migrations need explicit support | Anything that outlives a code deploy | Migrations are forgotten; `from_dict` resets silently (agent-framework) | No versioning — simpler code, eventual corruption |
| Atomic file writes via temp + rename | Crash-safe file durability | Doesn't work across filesystems; relies on POSIX rename | Local filesystem durability; event logs; settings | Container restart with overlay FS may lose temp | Postgres durability (letta) — strong but heavier |
| Namespaced dict (`source_id`) for session state | Easy to extend; no rigid base class hierarchy | No schema enforcement at write time; type errors surface at read time | Library frameworks with many optional providers | A buggy provider corrupts another's namespace | ORM with per-class tables (letta) — strong typing, more code |
| Hybrid JSON+pickle checkpoint encoding | Best of both worlds: JSON-readable + pickle-expressive | Pickle is RCE vector; allowlist must be maintained | Internal/trusted storage | Attacker writes a checkpoint → code execution | Pure JSON / msgpack with strict allowlist (langgraph) |
| Pin + finalizer for in-memory mutable state | Guarantees cleanup on eviction | Finalizer cost; pin leaks if not released | Long-running services with hot caches | Forgotten pin leaks memory; forgotten release triggers premature eviction | TTL-only — simpler but no guarantee |
| Conditional updates anchored to durable fence | Safe across ownership transfers | Extra round-trip; LWT contention in Cassandra | Distributed durable execution | Fence drift → `ConditionFailedError` | Optimistic locking on Block only (letta) — limited coverage |
| Wall-clock staleness for run liveness | Cheap heuristic; no heartbeat infrastructure | Slow-but-alive run reclaimed; truly dead run blocks until timeout | Foreground-only agent (langfuse v1) | User waits 150s for "stuck" recovery | Heartbeat-based liveness (future langfuse) |
| Idempotency via OTID/content hash | Safe retries; content-addressable | OTID must be deterministic across retries; collisions possible | HTTP endpoints with retry semantics | Hash collision → wrong run returned | Server-side dedup with no client token — easier, weaker |
| Pluggable stores behind ABCs + DI | Easy to swap file/SQL/cloud; testable | Boilerplate; more files | Pluggable persistence (openhands, langfuse, langgraph) | ABC and concrete impl drift over time | Single concrete impl — simpler but inflexible |

## Decision Guide

Use these decision rules when designing the state taxonomy for a new system.

### Rule 1: Pick a cluster based on product shape.

- If you are building a **library / framework**, expose persistence as a protocol. LangGraph's `BaseCheckpointSaver` and agent-framework's `HistoryProvider` are the templates. The protocol *is* the contract.
- If you are building an **application server**, define durable schemas in an ORM/Prisma layer and wire concrete stores in config. langfuse and letta are the templates. Document the tier boundaries explicitly.
- If you are building a **durable execution engine**, anchor everything to a History log with conditional updates. Temporal is the template. Don't try to fake it without proper fencing.

### Rule 2: Pick the granularity of the state machine.

- **Field-level channels** (langgraph): use when reducers are central to the product — multi-agent, parallel branches, conditional updates. Cost: cognitive overhead for users.
- **Provider-level namespace** (agent-framework): use when you have many optional providers and want to add new ones without changing the base class. Cost: no static type safety.
- **Model-level ORM** (letta): use when state is naturally object-shaped (Agent, Conversation, Run) and a single Postgres is sufficient. Cost: many tables, many migrations.
- **Service-level abstraction** (openhands): use when control-plane/data-plane is split across process boundaries. Cost: dual representation invites drift.

### Rule 3: Decide durability tier before writing the first migration.

For every state class, ask: *can this survive a crash?* If yes, where? If no, can it be re-derived? Document the answer in a single canonical state map. Three sources explicitly noted this is missing — langfuse, openhands, temporal.

### Rule 4: Use append-only logs for conversation history.

Across langfuse, temporal, openhands, agent-framework, and langgraph, the durable conversation history is an append-only log. langfuse's `InAppAgentEvent`, temporal's `History` events, openhands's per-event JSON files, agent-framework's `WorkflowCheckpoint` and `DurableAgentState.conversation_history`, langgraph's `pending_writes`. Don't store mutable message rows; store events and derive messages.

### Rule 5: Fence every durable write with a conditional token.

Temporal's `RangeID` + `DBRecordVersion`, langgraph's per-channel monotonic versions, letta's `Block.version` — all ensure that concurrent writers cannot corrupt each other. Even a single-process server with eventual multi-process scaling benefits from this discipline.

### Rule 6: Keep runtime config out of the durable boundary.

agent-framework's `Settings` (`_settings.py:1-293`) is built from env + .env + overrides at startup. temporal's `dynamicconfig.Setting[T]` is process-local with optional file-backed refresh. Don't store runtime config in the same store as durable state unless you have a versioning scheme.

### Rule 7: For eval state, default to a single free-text column.

Letta's `Step.feedback` (`step.py:78-86`) is the simplest viable model: a string column accepting "positive" / "negative" with last-writer-wins semantics. Don't over-engineer until you have multiple evaluators or rubric structures to model.

### Rule 8: For approval state, model it as an interrupt.

langgraph's `Interrupt` records inside `pending_writes` (`types.py:533-588`) are the cleanest model: a stable `id` derived from checkpoint namespace, a value, and a write-time marker. agent-framework's `ToolApprovalState` is similar but with extra rules + queued requests. Both are durable; both can be replayed.

### Rule 9: For caches, prefer TTL + size bound + finalizer.

temporal's `cacheImpl` with `finalizer.Finalizer` (`service/history/workflow/cache/cache.go:60-148`) is the template. langgraph's `BaseCache` with TTL and `CachePolicy` is a library version of the same idea. Avoid unbounded dicts.

### Rule 10: For secrets, use frozen models with redaction.

openhands's `Secrets` with `frozen=True` + `MappingProxyType` + `SecretStr` + redacted serialization (`secrets_models.py:36-107`) is a sound baseline. letta's PBKDF2 + Fernet-style encryption (`value_enc` columns) is the durable-store equivalent. never log secrets; never store them unencrypted.

## Practical Tips

### Patterns to copy

- **Append-only event log for conversation state** (langfuse, temporal, openhands, langgraph, agent-framework) — five sources agree.
- **Atomic write via temp + rename** for file-backed durability (agent-framework, openhands) — pattern is uniform and well-documented.
- **Schema versioning at the durable boundary** (agent-framework `SCHEMA_VERSION = "1.1.0"`, langgraph `LATEST_VERSION = 4`, letta Alembic) — without it, deserialisation is a guessing game.
- **Schema-version-aware deserialization** (agent-framework `DurableAgentState.from_dict` reads `schemaVersion` first) — explicit "migrate forward, never read backwards".
- **Pending-vs-committed buffering** at superstep or step boundaries (agent-framework `State`, langgraph `pending_writes`, temporal `WorkflowMutation`) — separates in-flight from durable.
- **Pluggable serialization with allowlists** for non-JSON-native types (agent-framework `_RestrictedUnpickler`, langgraph `STRICT_MSGPACK_ENABLED`) — defense in depth, but document it as such.
- **Idempotency via OTID / content hash** (letta `OTID_RUN_PREFIX`, openhands task-id rekeying, langfuse `entityType-body.id` grouping) — survives client retries.
- **Pin + finalizer for in-memory mutable state** (temporal `cacheImpl`) — guarantees cleanup on eviction.
- **Per-session locks in `WeakKeyDictionary`** (agent-framework `_todo.py`, `_memory.py`, `_background_agents.py`) — automatic GC, no timer.
- **Strict ownership boundaries via dependency direction** (langfuse, temporal) — mechanically enforceable.
- **Strict msgpack allowlist** for serialization (langgraph `STRICT_MSGPACK_ENABLED` + `allowed_msgpack_modules`) — RCE mitigation.

### Patterns to avoid or delay until needed

- **Multi-tier persistence** (langfuse's 4 tiers): powerful but operationally expensive. Delay until a single database demonstrably bottlenecks one access pattern.
- **Hybrid JSON+pickle checkpoint encoding**: trade clarity for expressiveness. Only adopt if the storage backend is access-controlled and the allowlist is maintained.
- **Wall-clock staleness** for run liveness (langfuse): a 150-second timeout is a v1 heuristic. Move to heartbeat-based liveness when background/resumable runs become a requirement.
- **Hardcoded metrics columns** (openhands `total_tokens=0`): prefer a materialised view over denormalised columns; drift is otherwise invisible.
- **In-memory trackers on a long-running agent class** (letta `LettaAgentV3.context_token_estimate`): unsnapshotted state is lost on crash and silently drifts.
- **OTID idempotency with no schema for the request token**: if the token doesn't include a version, a breaking schema change can collide.
- **Frozen-empty shim fields** (openhands `Settings.secrets_store`): documents in-progress migrations but invites misuse. Plan to remove.
- **`actor=None` warnings** (letta `sqlalchemy_base.py:256-258`): a security warning without a hard fail is a footgun.

### Caution signs

- **Schema changes without versioning**: silent corruption when old data meets new code.
- **Cross-store references without FKs**: dangling references after cascade deletes.
- **Hardcoded metrics or counters that drift**: silent observability failures.
- **In-memory state on long-lived agent classes**: data loss on crash, replay drift.
- **`dict[str, Any]` with no schema enforcement**: type errors at read time, not at write time.
- **Pickle/unsafe deserialisation with allowlist**: easy to forget new types in the allowlist, or to accidentally let attackers write checkpoints.
- **Stale-read checks only on a subset of tables** (letta optimistic locking on Block only): other tables silently race.
- **`EphemeralData` labelled "eventually correct"** (temporal `user_data_manager.go:94-98`): know the staleness window.
- **Wall-clock staleness for run liveness**: a slow-but-alive run can be reclaimed; a dead run blocks.
- **Documented state taxonomy that requires reconstruction from schema + migrations + queue contracts**: langfuse, openhands, temporal all flagged this. Build the canonical map.
- **`agent_framework` `pickle` checkpoints in less-trusted storage**: the docs admit the unpickler is not a security boundary.
- **In-flight `asyncio.Task` handles stored in process memory** (agent-framework `BackgroundAgentsProvider`): orphans are marked `LOST` on restart.
- **Per-service-call history persistence with no recovery protocol**: silently drifts if `conversation_id` round-trip breaks.

## Anti-Patterns / Caution Signs

- **Mutable message tables as the source of truth for conversation history** — replay is harder; partial writes corrupt; ordering becomes ambiguous.
- **Single-row mutations as the only durable boundary** — without conditional updates, concurrent writers race.
- **Hardcoded "stale run" timeouts** without heartbeat — false positives reclaim live work; false negatives block dead work.
- **Metrics columns maintained by `try/except: pass`** (openhands `process_stats_event`) — failures are silent, drift accumulates.
- **Storing `total_tokens=0` on every save** (openhands `sql_app_conversation_info_service.py:367`) — visible drift, broken metrics.
- **Freezing a field but keeping the serializer empty** (openhands `Settings.secrets_store`) — migration seam that survives in production longer than intended.
- **Logging a security warning instead of raising** (letta `actor=None`) — invitations to bypass.
- **Denormalising cache columns without a reconciler** — drift is unrecoverable.
- **Treating "ephemeral" as a license to leak** (letta `LettaAgentV3` trackers) — drift accumulates silently.
- **Schema migration without `LATEST_VERSION`** — silent corruption across deploys.
- **Cached mutable state without pin + finalizer** — eviction mid-request → stale data.

## Notable Absences

The dimension's most consistent absence is **eval state**. No source has a first-class `EvaluationResult` model with a dedicated owner module; letta's `Step.feedback` is a free-text string, langfuse's `JobConfiguration`/`JobExecution` are for offline batch evals, openhands's `event_callback_result` is closest but undocumented as an eval. A second absence is **canonical state map documentation**: langfuse, openhands, and temporal all flagged that the full taxonomy must be reconstructed from code. A third absence is **encrypted secrets rotation** (letta's `value_enc` columns lack a `key_version` discriminator; losing `LETTA_ENCRYPTION_KEY` permanently destroys data). A fourth absence is **cross-process session recovery for `AgentSession.state`** (agent-framework has no `SessionStore` protocol — only message history is file-backed, not arbitrary provider state). A fifth absence is **operational hooks for in-memory tracker snapshotting** (letta `LettaAgentV3`'s `turns`, `logprobs`, `response_messages` cannot be snapshotted without redesign).

## Open Questions

- **Conformance as a release gate**: langgraph's `checkpoint-conformance` suite is run for `InMemorySaver` only (`checkpoint-conformance/tests/test_validate_memory.py:17-21`); Postgres/SQLite conformance is presumably external. Promoting it to CI would harden the contract.
- **Transactional outbox for cache + checkpoint** (langgraph): today, `BaseCache` is best-effort; an option to write `cache_key → writes` atomically with `pending_writes` would close the duplication window.
- **Eval state as first-class**: no source has a dedicated `EvaluationResult` model. Is the dimension still emergent, or is the eval dimension expected to live elsewhere?
- **Cross-process session recovery** (agent-framework): no `SessionStore` protocol for `AgentSession.state`. A `SessionStore` protocol mirroring `HistoryProvider`'s shape would let provider state cross processes.
- **Pickle-free checkpoints** (agent-framework): `_checkpoint_encoding.py` relies on pickle for non-JSON-native values. A Pydantic-native path would let checkpoint storage be safely exposed to less-trusted environments.
- **Heartbeat-based liveness** (langfuse): wall-clock staleness for runs is a v1 heuristic; the in-app agent cannot resume interrupted turns.
- **Canonical state map**: langfuse, openhands, temporal all noted the absence of a single document enumerating every state class, owner, and durability. Is this a documentation debt or a deliberate choice?
- **Approval state in execution engines**: temporal treats approvals as Update state machines and Callback HSMs. Is that the right model, or should there be a first-class `Approval` class?
- **Schema versioning alignment**: agent-framework has `SCHEMA_VERSION` for `DurableAgentState`, langgraph has `LATEST_VERSION` for `Checkpoint`, letta has Alembic, openhands has `conversation_version='V1'`. None of these are aligned across sources — is cross-source schema compatibility even a goal?
- **Encrypted secrets key rotation**: letta's `value_enc` columns lack a `key_version` discriminator; losing `LETTA_ENCRYPTION_KEY` permanently destroys data. Is rotation a near-term need?

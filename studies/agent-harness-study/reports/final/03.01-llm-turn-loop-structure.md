## Executive Summary

Five of the seven sources ship a first-class LLM turn loop; one (`openhands`) deliberately moved the loop into an external SDK and is a thin orchestrator; one (`langgraph`) replaces the turn loop with a Bulk Synchronous Parallel graph engine where "turn" is emergent across supersteps. The remaining five split along two distinguishable axes: **loop location** (chat-client mixin vs. agent layer vs. dedicated agent base class) and **loop shape** (imperative `while`-loop with explicit phase boundaries vs. declarative node-graph).

Across all sources the loop owns the same five responsibilities the dimension asks about — prompt assembly, model call, output interpretation, tool execution, and iteration decision — but they distribute them across the codebase differently. **Where to place the loop is the single highest-leverage architectural decision** in this dimension: sources that put it at the chat-client mixin get tool calling for free across every provider (agent-framework `FunctionInvocationLayer`), while sources that put it at a graph node layer get fork/join/Send semantics for free (langgraph `Pregel`, pydantic-ai `pydantic_graph`).

Persistence of turn state is the second largest differentiator. Sources converge on a tiered model — in-memory mutable state for the duration of one loop, checkpointed or message-history appends per turn, optional durable external store — but they diverge on what counts as a "turn boundary." Sources that make "turn" a first-class persistable primitive (letta's `Step` ORM row, openai-agents-sdk's schema-versioned `RunState`) recover from crashes and support HITL pause/resume; sources that only persist message history (pydantic-ai slim, crewai) have to delegate durability to external adapters or Flow checkpointing.

The dimension's heuristic — "can you draw the loop without reading five files?" — is satisfied by langgraph, pydantic-ai, and openai-agents-sdk. It is just met by agent-framework (loop spans `_tools.py` and `_sessions.py`) and letta (`_step()` is ~700 lines). crewai is borderline because of three near-duplicate loops; openhands is not applicable because the loop is in an external package.

## Core Thesis

**The LLM turn loop is a durability-and-observation seam, not just a control-flow loop.** Sources that model "turn" as a typed record with id, start/end timestamps, usage, and finish reason (`Step`, `RunState`, `PregelExecutableTask`, `SingleStepResult`, `GraphAgentState.run_step`, `SingleStepResult`, `pregel_call_counter`) unlock replay, HITL pause/resume, observability, and tool-tracking across crashes; sources that model "turn" as just an integer counter (`while not isinstance(formatted_answer, AgentFinish)`) trade that for simplicity but lose the seam. The imperative-vs-declarative split is secondary to the typed-turn-record seam.

The other convergence point is that **every mature source has a way to bound the loop** (max iterations, max turns, max steps, recursion limit, content-length retry) and **every mature source separates the loop body from the agent class** — either through composition (agent-framework's mixin chain, agent-framework's `BaseChatClient` → `FunctionInvocationLayer`), through a graph runtime (langgraph's Pregel, pydantic-ai's `pydantic_graph`), or through dependency injection (crewai's `BaseAgentExecutor`, openai-agents-sdk's `Runner.run`). Agent-class-specific logic lives in node functions or step implementations, not in the loop itself.

## Approach Models

The seven sources collapse into four main approach models. Where one source fits cleanly, the row notes it; rows are not exclusive — a source can demonstrate multiple models at different layers (agent-framework has both a layered chat-client loop and an agent-level `AgentLoopMiddleware`).

### A. Loop-at-Chat-Client (mixin layer)

**agent-framework** is the canonical example. The model-call loop lives on `FunctionInvocationLayer`, a mixin applied to every `BaseChatClient` subclass (`_clients.py:975-999`), and the agent's `run()` is a thin façade (`_agents.py:899-1021`). Tool calling is "free" for any chat client; the agent never re-implements it. Termination policy is a tri-state action enum (`"return" | "stop" | "continue"`) with a forced clean-text tail (`tool_choice="none"` final pass) after the iteration cap, preventing orphaned `function_call` items.

This model answers "where does the loop live?" by inverting the usual assumption: any chat client (OpenAI, Azure, Anthropic, Bedrock, Ollama, etc.) gets the loop automatically. Tradeoff: the agent-level `AgentLoopMiddleware` (`_harness/_loop.py:117`) is a separate outer loop with its own `DEFAULT_MAX_ITERATIONS = 10`, and the two share the same name — a constant-collision hazard.

### B. Imperative While-Loop at the Agent Layer

**crewai**, **letta**, and **openai-agents-sdk** place the loop on the agent (or its executor). All three use an explicit `while ... :` over a phase sequence; all three expose the loop body as either a function (crewai's `_invoke_loop_*`, openai-agents-sdk's `AgentRunner.run`), a method (letta's `_step()`), or a coroutine (letta's `async def _step`).

Differences within this model:

- **crewai** has three near-duplicate loops (the deprecated `CrewAgentExecutor._invoke_loop_*`, the Flow-based `AgentExecutor`, and the nested `StepExecutor`) reusing shared helpers (`get_llm_response`, `process_llm_response`, `handle_max_iterations_exceeded`, `handle_output_parser_exception`, `handle_context_length`).
- **letta** has three generations of the same loop class (`LettaAgentV1/V2/V3`) selected by `AgentLoop.load()` factory and shares the `BaseAgentV2` interface.
- **openai-agents-sdk** has one loop shape (`while True` over `NextStep*` routing) duplicated as sync + streaming variants, kept in lock-step by AGENTS.md policy.

### C. Graph-Based Declarative Loop

**langgraph** and **pydantic-ai** both replace the imperative `while` with a node graph.

- **langgraph**: the loop is the Pregel Bulk Synchronous Parallel engine (`libs/langgraph/langgraph/pregel/main.py:2974-3003` — `while loop.tick(): for _ in runner.tick(...): ... loop.after_tick()`). The agent is a `StateGraph` with conditional edges; a user-perceived "turn" is two supersteps (model → tools) glued by `should_continue` routing (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:830-859`). Per-superstep `put_writes` + `_put_checkpoint` is the durability primitive.
- **pydantic-ai**: a 4-node agent graph `UserPromptNode → ModelRequestNode ⇄ CallToolsNode → SetFinalResult` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2130-2138`), driven by the generic `_GraphIterator` runtime (`pydantic_graph/pydantic_graph/graph_builder.py:646-825`). A turn is `ModelRequestNode → CallToolsNode`; finalization is `End(FinalResult)` or `SetFinalResult` (streaming-only).

Both make the loop declarative and free for visualization (`Graph.render()` in pydantic-ai, the LangGraph Studio graph view). Neither has a hand-written `while` to maintain for the agent loop itself.

### D. External/Loose-Coupled Loop

**openhands** is the outlier. The actual turn loop lives in `openhands-sdk` (an installed package), and the OpenHands V1 FastAPI server is a thin orchestrator that proxies user messages to a sandbox-hosted `agent_server` (gated by `SandboxStatus == RUNNING`) and persists `Event` JSON files one-per-file. The source has no `while ... :`, no `agent.step()`, no `conversation.run()` in the tree — every dimension question is answered "unknown from this source, see SDK." This is a valid pattern (clean separation of orchestrator from agent runtime), but it means the orchestrator cannot intervene mid-turn beyond mid-conversation model switching (`switch_profile`, `switch_acp_model`).

## Pattern Catalog

Patterns below were chosen because they appeared in multiple sources or because one source demonstrates them unusually clearly.

### 1. Loop placement as a layering decision

**Problem.** "Where does the turn loop live?" determines whether tool-calling is provider-special or general-purpose.
**Where it appears.** agent-framework (chat-client mixin); langgraph (graph engine); pydantic-ai (graph engine); letta (agent base class); openai-agents-sdk (Runner closure).
**Why it works.** Each placement maps a natural failure boundary. Mixin layer = "loop runs wherever the model is called." Graph engine = "loop runs wherever a state machine has work." Runner closure = "loop runs wherever the user invokes `run()`."
**When to copy it.** Pick the placement closest to the unit you want to be reusable. If you want tool-calling for free across providers, place the loop at the chat-client layer (agent-framework). If you want fork/join/Send for free, place it at a graph engine (langgraph, pydantic-ai). If you want one loop per agent invocation, keep it at the agent closure (openai-agents-sdk).
**When it is overkill.** If you ship one provider and one agent type, the layering discussion is unnecessary; an imperative loop on the agent class suffices.
**Evidence.** `python/packages/core/agent_framework/_tools.py:2551-2696`; `libs/langgraph/langgraph/pregel/main.py:2974-3003`; `src/agents/run.py:768-1504`; `letta/agents/base_agent_v2.py:18-105`.

### 2. Typed turn record as a durability seam

**Problem.** Crashes between model calls lose work; HITL pause/resume needs a "where are we?" snapshot; observability needs per-turn usage/cost.
**Where it appears.** letta `Step` ORM row (`letta/schemas/step.py:16-66`); openai-agents-sdk `RunState` schema-versioned at `1.11` (`src/agents/run_state.py:185-322`); langgraph `PregelExecutableTask` with `(name, input, writes, triggers, id, path)` (`libs/langgraph/langgraph/types.py:627-641`); pydantic-ai `GraphAgentState` with `run_step`, `message_history`, `usage` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:122-180`); openai-agents-sdk `SingleStepResult` (`src/agents/run_internal/run_steps.py:178-218`).
**Why it works.** A typed record names the boundary between "what the loop has done" and "what the model will see next," and it gives every downstream concern (persistence, retry, replay, observability) a stable handle.
**When to copy it.** Whenever you need HITL pause/resume, cross-process replay, or cost-per-turn accounting. crewai and pydantic-ai slim both defer this — they hold turn state only as in-memory `messages: list[LLMMessage]` and depend on external persistence (Flow checkpoint or `durable_exec/{temporal,dbos,prefect}`) for durability.
**When it is overkill.** If you have no pause/resume need and no crash-recovery need, an integer counter (`agent.iterations`) is simpler.
**Evidence.** `letta/agents/letta_agent_v2.py:941-1034`; `src/agents/run_state.py:131-149`; `pydantic_ai_slim/pydantic_ai/_agent_graph.py:122-180`.

### 3. Streaming + non-streaming dual implementation

**Problem.** The same turn loop needs to serve blocking `run()` and streaming `stream()` / `astream()` callers.
**Where it appears.** agent-framework (streaming mirror at `_tools.py:2700-2866`); openai-agents-sdk (sync `AgentRunner.run` at `src/agents/run.py:768-1504` vs streaming `start_streaming` at `src/agents/run_internal/run_loop.py:671-1199`); crewai (`CrewAgentExecutor` sync + `_ainvoke_loop_*` mirror); pydantic-ai (`ModelRequestNode._make_request` vs `ModelRequestNode.stream`).
**Why it works.** Streaming lets the caller see partial model output as it arrives; non-streaming is simpler to reason about for tests and for callers that don't want partial updates.
**When to copy it.** OpenAI Agents SDK explicitly acknowledges the maintenance burden (AGENTS.md policy: "Keep streaming and non-streaming paths behaviorally aligned"). The burden is real; the trade-off is unavoidable for caller-facing flexibility.
**When it is overkill.** langgraph sidesteps it by routing partial output through `_event_queue.put_nowait` inside the generic runner — its single loop shape covers both styles.
**Evidence.** `src/agents/run.py:768-1504` vs `src/agents/run_internal/run_loop.py:671-1199`; `python/packages/core/agent_framework/_tools.py:2551-2696` vs `:2700-2866`.

### 4. Multi-tool execution with explicit mode selection

**Problem.** Parallel tool execution must coexist with short-circuit `result_as_answer` tools and ordered-event emission.
**Where it appears.** crewai (`ThreadPoolExecutor(max_workers=min(8, n))` with `contextvars.copy_context().run` at `crew_agent_executor.py:742-767`, override for `result_as_answer` at `agent_executor.py:1714-1786`); pydantic-ai (`_call_tools` with `parallel_execution_mode ∈ {sequential, parallel, parallel_ordered_events}` at `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1862-1965`); langgraph (v2 routes per-call via `Send("tools", ToolCallWithContext(...))` at `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:846-859`); letta (`asyncio.gather(*[_run_one(spec)])` at `letta/agents/letta_agent_v3.py:1857`, with client-side truncation when `parallel_tool_calls=false` at `:1337-1342`); openai-agents-sdk (parallel local function tools capped by `tool_execution.max_function_tool_concurrency`).
**Why it works.** Multiple tool calls in one model response is the default for modern providers; serializing them silently is a regression; parallelizing them blindly is a determinism break.
**When to copy it.** Always distinguish at least "parallel" and "sequential" modes. The three-mode split in pydantic-ai (`sequential`, `parallel`, `parallel_ordered_events`) is the cleanest.
**When it is overkill.** If your agent uses one tool that always short-circuits or one tool that always takes minutes, don't bother with parallel.
**Evidence.** `crewai/src/crewai/utilities/agent_utils.py:1374-1663`; `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1862-1965`.

### 5. Loop termination forcing a clean-text tail after cap

**Problem.** When the iteration cap is hit, the conversation can have orphaned `function_call` items without matching `function_result` items, causing provider-side errors on the next turn.
**Where it appears.** agent-framework sets `tool_choice="none"` and makes a final plain model call after `max_iterations` (`_tools.py:2664-2696`); crewai `handle_max_iterations_exceeded` makes one extra LLM call to force a final answer (`agent_utils.py:293-350`); openai-agents-sdk's `error_handlers["max_turns"]` can convert `MaxTurnsExceeded` into a synthetic final output (`run.py:1066-1144`); pydantic-ai incrementally resubmits with an empty `ModelRequest` on empty responses (`_agent_graph.py:1102-1173`).
**Why it works.** A cap that leaves the model in the middle of a tool call is worse than a hard cap. Forcing one final text-only turn produces a clean final state.
**When to copy it.** Always. Choose between hard termination (raise), synthetic final result (openai-agents-sdk), or one-more-LLM-call (agent-framework, crewai) based on whether your users prefer exception-style or graceful-style failure.
**Evidence.** `python/packages/core/agent_framework/_tools.py:2664-2696`; `lib/crewai/src/crewai/utilities/agent_utils.py:293-350`.

### 6. Per-turn usage + per-turn span as first-class observability

**Problem.** Aggregated-only usage obscures which turns were expensive; one model call is not the same observability unit as one loop iteration.
**Where it appears.** agent-framework (`aggregated_usage = add_usage_details(...)` per pass at `_tools.py:2605, 2685`; OTel chat-completion span per model call at `observability.py:1596`, agent-invoke span per `agent.run` at `:1874`); openai-agents-sdk (`turn_usage_start = snapshot_usage(...)` at `src/agents/run.py:1165`, usage-delta on span close at `:1273-1278`; `turn_span` factory at `src/agents/tracing/create.py:139-152`); letta (`StepMetrics` per step with `step_start_ns`, `llm_request_start_ns`, `llm_request_ns`, `tool_execution_ns` at `letta/schemas/step_metrics.py:13-26`).
**Why it works.** Cost analysis, retry triage, and replay debugging all need per-turn numbers, not just rollup.
**When to copy it.** Whenever you charge per turn, charge per model call, or do any post-mortem on a failed run.
**When it is overkill.** If your loop is a one-shot `while not isinstance(result, AgentFinish)` and you only ever run it interactively, a single total may suffice.
**Evidence.** `python/packages/core/agent_framework/_tools.py:2605`, `src/agents/run.py:1165-1278`.

### 7. Service-managed conversation tracking as a delta calculator

**Problem.** Re-sending the full conversation on every turn wastes tokens; re-sending nothing loses tool-call IDs and reasoning item references.
**Where it appears.** openai-agents-sdk (`OpenAIServerConversationTracker` at `src/agents/run_internal/oai_conversation.py` — only the delta is shipped when `conversation_id` or `previous_response_id` is set; `Session` auto-disabled to avoid double-recording); agent-framework (`service_session_id` propagation via `_update_continuation_state` at `_tools.py:1883-1901`, eager stream capture at `_agents.py:1085-1099`); langgraph (channel writes per superstep via `put_writes`).
**Why it works.** Server-managed history is the cheapest way to scale to long traces without re-tokenizing; client delta tracking is the cleanest way to avoid double-recording.
**When to copy it.** Whenever you use a provider that supports `conversation_id` / `previous_response_id` / `store=True` (OpenAI Responses, Anthropic Bedrock conversations, etc.).
**Evidence.** `src/agents/run_internal/oai_conversation.py`; `python/packages/core/agent_framework/_tools.py:1883-1901`.

### 8. Two-tier termination: outer hard cap + per-state advisory

**Problem.** A single hard cap is bad UX (the user gets cut off mid-thought); a per-state soft cap allows graceful wrap-up.
**Where it appears.** langgraph (`recursion_limit` per `SyncPregelLoop` at `_loop.py:1677` is hard; `RemainingStepsManager.get(scratchpad) = stop - step` at `libs/langgraph/langgraph/managed/is_last_step.py:18-22` is per-state advisory that injects "Sorry, need more steps…" message); agent-framework (`max_iterations` at `_tools.py:90` is hard, `max_function_calls` at `_tools.py:1394` is per-iteration soft cap; `errors_in_a_row >= 3` triggers `tool_choice="none"` at `:2248-2270`); letta (`max_steps=DEFAULT_MAX_STEPS=50` at `letta/constants.py:75` is hard; `_decide_continuation` reads approval/tool-rule state for advisory continuation); openai-agents-sdk (`max_turns=10` is hard; `on_*` hooks and per-tool `max_usage_count` are advisory).
**Why it works.** A hard cap protects against runaway loops; an advisory cap lets the agent wrap up gracefully.
**When to copy it.** Always have a hard outer cap (`max_iterations`, `max_steps`, `max_turns`, `recursion_limit`); consider adding a per-state advisory cap if you want graceful over-cap behavior.
**Evidence.** `libs/langgraph/langgraph/pregel/_loop.py:1677`, `libs/langgraph/langgraph/managed/is_last_step.py:18-22`; `python/packages/core/agent_framework/_tools.py:90, 1394`.

### 9. Loops that survive `asyncio` cancellation cleanly

**Problem.** Cancellation in the middle of a model call or a parallel tool batch can leave state inconsistent.
**Where it appears.** langgraph (`AsyncPregelLoop.__aexit__` schedules `asyncio.create_task(self.stack.__aexit__(...))` and sets `e.args = (*e.args, exit_task)` so consumers can await teardown at `libs/langgraph/langgraph/pregel/_loop.py:1953-1962`); openai-agents-sdk (`_cancel_mode == "immediate"` raises at any await, `"after_turn"` lets the current turn complete then breaks at `src/agents/result.py:496`); pydantic-ai (`ModelRequestNode.stream()` and `_call_tools` cancel siblings via `cancel_and_drain` on `CancelledError` at `pydantic_ai_slim/pydantic_ai/_agent_graph.py:729-756, 1947-1956`); letta (best-effort at step boundaries via `_check_run_cancellation` at `letta/agents/letta_agent_v2.py:750-757`).
**Why it works.** Cooperating between cancellation and the loop is a correctness fix that affects what observability sees after the fact.
**When to copy it.** If you expose a `cancel()` / `interrupt()` surface, decide which kind ("immediate" vs "after_turn") and document it. openai-agents-sdk's choice to expose both is a noteworthy pattern.
**Evidence.** `src/agents/result.py:496`; `libs/langgraph/langgraph/pregel/_loop.py:1953-1962`; `letta/agents/letta_agent_v2.py:750-757`.

## Key Differences

These are differences between sources that look like quality differences but are really responses to different constraints.

| Concern | Source A choice | Source B choice | Why they differ |
|---------|-----------------|-----------------|-----------------|
| Loop placement | agent-framework puts loop at chat-client mixin | openai-agents-sdk puts loop at Runner closure | agent-framework optimizes for "any provider gets tool-calling for free"; openai-agents-sdk optimizes for "one agent type, one Runner" |
| Loop shape | langgraph + pydantic-ai use graph nodes | crewai + letta use imperative `while` | Graph sources want declarative visualization, fork/join, and durable per-node checkpoints; imperative sources want a simpler on-ramp |

| Multi-tool parallelism | pydantic-ai 3 modes; crewai 2 | letta + agent-framework client-side enforce (truncate to first call) on providers that don't support parallel | Sources that ship with multiple provider SDKs have to negotiate provider parity; sources that ship with one provider family (OpenAI Responses) get free parallelism |
| Streaming/non-streaming | openai-agents-sdk + crewai have two implementations | langgraph routes through one runner | langgraph has a generic engine, so streaming is a runner feature not a duplicate path |
| Termination policy | agent-framework: forced clean-text tail | pydantic-ai: incremental empty resubmit | Different provider-failure models; the agent-framework choice protects OpenAI/Azure APIs that reject orphan tool calls, while pydantic-ai's resubmit handles Anthropic-style empty responses |
| Final-output condition type | openai-agents-sdk: `NextStep*` discriminated union | langgraph: graph edge to END | A typed union works for any loop shape; a graph edge is implicit in the topology |
| Coordination of multiple loops | agent-framework has TWO `DEFAULT_MAX_ITERATIONS` (tool-loop 40, harness-loop 10) | letta has THREE loop generations (`V1/V2/V3`) | Both expose the cost of "loop spans multiple layers but they share a name" — the industry has not converged on loop-naming conventions across layers |
| Conversation management | openai-agents-sdk auto-disables `Session` when server-managed id is set | agent-framework requires explicit `require_per_service_call_history_persistence=True` | Both prevent double-recording; the SDKs differ on whether the safe default is "let provider manage" (OpenAI Agents) or "let client manage" (agent-framework) |

## Tradeoffs

### Placement: chat-client mixin vs. agent layer vs. graph engine

- **Chat-client mixin (agent-framework)** — Benefit: tool-calling works across every provider without re-implementation. Cost: loop body is buried in the chat-client layer, so tracing "one agent turn" requires reconciling two layers (the agent loop and the inner function-invocation loop). Best-fit: multi-provider orchestration. Failure mode: developer confusion about which `DEFAULT_MAX_ITERATIONS` applies to which loop. Alternative: imperative loop at the agent layer (openai-agents-sdk).

### Shape: imperative `while` vs. declarative graph

- **Imperative `while` (crewai, letta, openai-agents-sdk, agent-framework)** — Benefit: one local read of the function tells you the entire loop body. Cost: "one more turn is just one more iteration" hides dependency on external state (graph topology, RunState schema, conversation tracking). Best-fit: single-purpose agent; one provider; one path. Failure mode: drift between sync/streaming variants (openai-agents-sdk AGENTS.md policy). Alternative: declarative graph (langgraph, pydantic-ai).
- **Declarative graph (langgraph, pydantic-ai)** — Benefit: visualization for free; conditional routing composes naturally; fork/join is a graph primitive. Cost: the "turn" is not a first-class concept — it is two supersteps (model → tools) glued by an edge; some advanced patterns need careful edge-graph debugging. Best-fit: multi-agent, conditional, or tool-fan-out-heavy flows. Failure mode: graph structure validation is off (pydantic-ai `_agent_graph.py:2139`) which is safe today but trades safety for speed. Alternative: imperative loop with hooks (crewai).

### Persistence: in-memory `messages` vs. typed turn record vs. external durable adapter

- **In-memory only (crewai messages, pydantic-ai slim)** — Benefit: smallest surface; no schema versioning; tests are easy. Cost: crashes lose work; HITL requires external coordinator; cost-per-turn is invisible. Best-fit: library code that callers integrate into their own durable runtimes. Failure mode: caller assumes the loop will hold state across restarts. Alternative: typed turn record (letta Step, openai-agents-sdk RunState).
- **Typed turn record (letta `Step`, openai-agents-sdk `RunState`)** — Benefit: cross-process replay, schema-versioned resumption, per-turn usage/failure inspection. Cost: schema migration stories; every change to the record shape needs a version bump. Best-fit: hosted multi-tenant services, HITL workflows, anything with audit/compliance. Failure mode: schema becomes coupled to every layer; migrations are heavy. Alternative: external durable adapter (Temporal, DBOS, Prefect) on top of in-memory loop.
- **External durable adapter (pydantic-ai `durable_exec/`, langgraph checkpoints via CheckpointSaver)** — Benefit: framework-agnostic; pluggable backend. Cost: caller must pick an adapter and deploy it; SLA on durability lives outside the framework. Best-fit: enterprise deployments with existing workflow orchestrator. Failure mode: lock-in to the adapter's introspection model. Alternative: typed turn record.

### Termination: hard cap vs. forced tail vs. incremental retry

- **Hard cap raise (crewai RuntimeError, openai-agents-sdk MaxTurnsExceeded)** — Benefit: caller catches and decides. Cost: bad UX for chat-style callers who wanted a graceful answer. Failure mode: callers forget to handle and get an exception they did not expect.
- **Forced clean-text tail (agent-framework)** — Benefit: conversation ends in a usable state. Cost: makes the loop non-deterministic (the final tool_choice=none turn may produce anything). Failure mode: silent over-budget behavior.
- **Incremental retry (pydantic-ai empty resubmit)** — Benefit: handles provider-side empty responses. Cost: may loop forever if the model keeps producing empty output. Failure mode: needs an additional error-in-a-row cap (which pydantic-ai has via `output_retries_used` and `ToolManager._check_max_retries`).

### Streaming duality: dual implementations vs. unified core

- **Dual implementations (openai-agents-sdk, agent-framework, crewai)** — Benefit: each variant optimizes for its call style. Cost: behavioral drift must be policed (AGENTS.md, double tests). Failure mode: one path gets a fix the other doesn't.
- **Unified engine that exposes streaming (langgraph Runner)** — Benefit: one loop body, both styles. Cost: streaming is a runner feature, not a public loop decision; some advanced patterns become harder to debug. Failure mode: streaming surfaces leaks into the sync path.

## Decision Guide

When designing a turn loop for a new agent harness, the decisions in order of leverage are:

### 1. Where does the loop live?

- **Chat-client mixin** — Choose if you must support multiple provider SDKs and want tool-calling to be uniform. Trade off against the difficulty of reasoning about the loop across two layers (the agent's `run()` and the client's `FunctionInvocationLayer`).
- **Agent layer / agent base class** — Choose if you ship one provider or one agent type and want a simple, drawable loop. agent-framework (chat-client), crewai (mixed), letta (`BaseAgentV2`), openai-agents-sdk (`Runner`) all fit here.
- **Graph runtime** — Choose if you want declarative visualization, conditional routing, fork/join, and per-superstep checkpoints. langgraph's Pregel and pydantic-ai's `pydantic_graph` both qualify; both have `_GraphIterator`-style drivers.

### 2. Is "turn" a first-class record?

- **Yes** — Required if you need HITL pause/resume, cross-process replay, per-turn cost accounting, or schema-versioned migration. Examples: letta `Step`, openai-agents-sdk `RunState`. Adds schema versioning cost.
- **No** — Sufficient if your loop runs in one process, crashes are non-critical, and you have no HITL requirement. Examples: crewai `messages: list[LLMMessage]`, pydantic-ai slim `GraphAgentState.message_history`. Defer durability to caller.

### 3. How do you bound the loop?

- **`max_iterations` (or `max_steps` / `max_turns`)** — Hard outer cap, mandatory. Default values in the wild: agent-framework tool-loop 40; agent-framework harness-loop 10; crewai 25 (× 10 = 250 method calls); letta 50; openai-agents-sdk 10; langgraph `recursion_limit` 25.
- **Per-state advisory cap** — Lets the agent wrap up gracefully. langgraph's `RemainingStepsManager`, agent-framework's `errors_in_a_row`, pydantic-ai's `output_retries_used` are all variants of this.
- **Forced clean-text tail after cap** — Required if your providers reject orphan tool calls (OpenAI). Recommended if you ship with multiple providers.

### 4. How do you handle streaming?

- **Dual implementation** — Choose if you want full caller-side control over stream shape. Need disciplined mirroring (tests for both paths, or AGENTS.md policy).
- **Unified engine** — Choose if you want one loop body. langgraph's `PregelRunner.tick` yields between completions so streaming falls out of the same loop.

### 5. How do you support multiple tool calls per turn?

- **Always parallel by default, with a mode flag** — pydantic-ai (`sequential | parallel | parallel_ordered_events`). Recommended.
- **Parallel with `result_as_answer` override** — crewai (`ThreadPoolExecutor` at 742-767, override at 1097-1106 / 1773-1784).
- **One Send per tool call (graph)** — langgraph v2 (per-call `Send("tools", ToolCallWithContext(...))` at 846-859).
- **Sequential with client-side truncation when parallel is disabled** — letta (V3 truncates to first call when `parallel_tool_calls=false` at 1337-1342).

### 6. How do you expose hooks?

- **Global hooks (before/after LLM, before/after tool)** — crewai `llm_hooks.py` / `tool_hooks.py`.
- **Per-turn spans** — openai-agents-sdk `turn_span`, agent-framework OTel per-model-call spans.
- **Capability middleware around model call** — pydantic-ai `wrap_model_request` (`_agent_graph.py:671-676, 816-820`).
- **Graph-level middleware** — langgraph `before_node_run` / `wrap_node_run` / `after_node_run` via `AgentRun.next(node)`.

### 7. How do you cancel a running loop?

- **Immediate (raise at any await)** — openai-agents-sdk `_cancel_mode == "immediate"` at `src/agents/result.py:496`.
- **After turn (let current turn complete then break)** — openai-agents-sdk `_cancel_mode == "after_turn"`.
- **Best-effort at loop step boundaries** — letta (`_check_run_cancellation` at `letta/agents/letta_agent_v2.py:750-757`).
- **Stack-tracked via exit task** — langgraph (`AsyncPregelLoop.__aexit__` schedules the teardown at `libs/langgraph/langgraph/pregel/_loop.py:1953-1962`).

## Practical Tips

### Patterns to copy

- **Pick a `Step`-like ORM row** as the turn durability seam (letta). It pays off immediately for HITL pause/resume and per-turn usage reporting, and it forces a clean separation between "turn key" and "turn content."
- **Use a tri-state action enum** in the loop body (`"return" | "stop" | "continue"` — agent-framework). It expresses four-state outcomes (finalize, force-text, retry, continue) in a single return.
- **Force a clean-text tail** when the iteration cap is hit (agent-framework, crewai). Prevents orphaned `function_call` items from poisoning the conversation.
- **Use a single inner-step coroutine** that returns an async generator (letta `_step() -> AsyncGenerator[LettaMessage | dict, None]`). One body, three consumers (`step`, `stream`, `build_request` dry-run).
- **Make streaming fall out of a generator that yields between completions** (langgraph `PregelRunner.tick`). One implementation, both styles.
- **Use a `Send`-style fan-out** for parallel tool calls when your loop is already graph-shaped (langgraph v2). The discriminated payload shape (`__type="tool_call_with_context"`) is a small cost; running each tool as its own superstep task is a clean concurrency primitive.
- **Stamp a `step_id` (UUID) on every artifact** that a step produces (messages, ORM rows, OTel spans, ProviderTrace, webhook payloads — letta). The reuse-this-id-on-approval-resume pattern (letta `letta_agent_v3.py:1019-1027`) is the cheapest way to make HITL recovery clean.
- **Use `StepProgression` (or an equivalent) as a failure-path sentinel** (letta `letta/schemas/step.py:68-74`). Lets the loop's `finally` block decide between "log success", "log stop reason", and "log error" deterministically.
- **Keep the streaming and non-streaming loops behaviorally aligned** with explicit AGENTS.md policy and parallel test suites (openai-agents-sdk `tests/test_max_turns.py`, `tests/test_agent_runner.py`, `tests/test_agent_runner_streamed.py`).

### Patterns to avoid or delay until needed

- **Don't ship three near-duplicate loops** (crewai). One imperative loop, one streaming variant, one graph — anything more than that becomes a maintenance tax that compounds.
- **Don't put the loop at the chat-client layer unless you really need it cross-provider** (agent-framework). The layering cost is real — debugging "what is one turn?" in agent-framework requires reading `_tools.py`, `_sessions.py`, and `_agents.py`.
- **Don't rely on provider-managed conversation tracking if your provider doesn't support it.** openai-agents-sdk's `OpenAIServerConversationTracker` is a great pattern, but it only works for OpenAI Responses / Anthropic Bedrock / similar. Agent-framework's `service_session_id` is the multi-provider equivalent.
- **Don't use an in-memory `messages` list as your durability boundary** (crewai) unless you are explicitly designing a library (not a service). External Flow checkpointing is fine for a library; it is brittle for a service.
- **Don't have two unrelated constants share the same name** (agent-framework `DEFAULT_MAX_ITERATIONS` exists at `_tools.py:90` and `_harness/_loop.py:117`). Rename one (`TOOL_LOOP_DEFAULT_MAX_ITERATIONS` vs `HARNESS_LOOP_DEFAULT_ITERATIONS`).
- **Don't gate per-call persistence behind an opt-in flag unless you have to** (agent-framework `require_per_service_call_history_persistence=True`). The default flow loses work on mid-loop crashes.

### Decision rules (cheat sheet)

- **One provider, one agent type, no HITL, no replay** → imperative `while` loop at the agent layer (openai-agents-sdk `Runner.run` shape).
- **Multiple providers, want uniform tool-calling** → loop at the chat-client mixin (agent-framework `FunctionInvocationLayer`).
- **Need conditional routing, fork/join, durable per-node checkpoints** → graph runtime (langgraph `Pregel`, pydantic-ai `pydantic_graph`).
- **Need HITL pause/resume, per-turn cost, cross-process replay** → typed turn record (letta `Step`, openai-agents-sdk `RunState`).
- **Run as an orchestrator that delegates to a sandbox or a sub-agent runtime** → external-loop pattern (openhands) — own only the boundary.
- **Need per-call and per-iteration caps separately** → two-tier termination policy (agent-framework `max_iterations` + `max_function_calls`).
- **Need multiple tool calls in one turn** → pick a mode that matches your determinism needs: parallel (fastest, least deterministic), sequential (slowest, most deterministic), parallel_ordered_events (parallel but ordered — pydantic-ai).

## Caution Signs

These are signals that the LLM turn loop is becoming brittle, over-coupled, under-specified, hard to test, hard to operate, or hard to evolve.

### Loop body too long to draw in five files

- `_step()` reaches ~700 lines (letta `letta/agents/letta_agent_v3.py:895-1592`).
- `run.py` reaches ~737 lines (openai-agents-sdk `src/agents/run.py:768-1504`).
- Three near-duplicate ReAct loops (crewai).

**Indicator.** You cannot say what one turn does without scrolling or grep'ing. **Remedy.** Extract sub-phases as named methods (letta's `_execute_tools_in_step`, `_emit_step_finish_telemetry`, `_maybe_compact`, `_apply_continuation_decision` candidates).

### Loop logic spread across files

- agent-framework's loop is at `_tools.py` (turning) + `_sessions.py` (persistence) + `_agents.py` (façade).
- pydantic-ai's loop is at `_agent_graph.py` (nodes) + `pydantic_graph/graph_builder.py` (runtime) + `agent/abstract.py:438-446` (driver) + `run.py` (consumer).

**Indicator.** Reading "what happens between `for i in range(...)` and `await response`" requires three files. **Remedy.** Either consolidate into one module (langgraph `pregel/main.py` + `_loop.py` are colocated) or extract a `TurnOrchestrator` class.

### Loop and agent are 1:1

- crewai's loop bakes in agent-type assumptions; `CrewAgentExecutor` is deprecated but still ships.
- letta has three generations of one loop class.

**Indicator.** Forking the loop is the only way to add an agent type. **Remedy.** Lift the loop to an interface (letta `BaseAgentV2`) or to a graph runtime (langgraph, pydantic-ai).

### Two `max_iterations` named constants with the same value

- agent-framework has `DEFAULT_MAX_ITERATIONS = 40` (`_tools.py:90`) and `DEFAULT_MAX_ITERATIONS = 10` (`_harness/_loop.py:117`); langgraph has `recursion_limit: int = 25` default + `RemainingStepsManager` per-state; openai-agents-sdk has `DEFAULT_MAX_TURNS = 10` + per-tool `max_usage_count`.

**Indicator.** Grep for `max_iter` returns multiple unrelated hits. **Remedy.** Prefix or rename per layer; do not let "max iterations" be ambiguous in the codebase.

### Turn persistence not stable across processes

- crewai's `messages: list[LLMMessage]` is wiped at every `invoke()` unless `_resuming=True`.
- pydantic-ai's slim package has no built-in turn snapshot protocol; durable persistence is delegated to `durable_exec/{temporal,dbos,prefect}/`.

**Indicator.** A crash between turns loses work; HITL pause/resume requires external cooperation. **Remedy.** Add a typed turn record (letta `Step`, openai-agents-sdk `RunState`) or expose a durably resumption protocol (pydantic-ai `durable_exec/`).

### Streaming and non-streaming paths diverge

- openai-agents-sdk documents this in AGENTS.md as a known cost.

**Indicator.** New feature works in sync but not streaming (or vice versa). **Remedy.** Either unify into a single engine that emits via queue (langgraph pattern) or add a parity test that's enforced in CI.

### Loop has no `turn_id`, only an integer counter

- crewai (`state.iterations`), letta V2 (`heartbeat_reason`), pydantic-ai (`state.run_step`), agent-framework (`attempt_idx`).

**Indicator.** Replay/resume requires matching `(agent_id, iter_n)` pairs; you cannot trace a turn across processes by id alone. **Remedy.** Switch to a UUID per turn (letta `step-<uuid4>`, openai-agents-sdk would benefit from `_current_turn_uuid` even though it has the integer `_current_turn`).

### Loop terminates silently with no finish reason

- Several sources set a default `stop_reason="end_turn"` or default the loop to "completed" without capturing why.

**Indicator.** Post-mortem on a stuck run shows "loop exited, no reason recorded." **Remedy.** Always set an explicit `stop_reason` (letta's `StopReasonType` enum at `letta/schemas/letta_stop_reason.py:9-22` is exemplary: 12 typed reasons with a `run_status` mapper at `:24-49`).

### Per-tool policy silently downgrades parallelism

- crewai falls back to sequential if any tool has `result_as_answer=True` or `max_usage_count` set (no error, no warning).
- letta truncates to the first tool call when `parallel_tool_calls=False` with just a warning log.

**Indicator.** A user expecting parallelism gets sequential execution and doesn't know it. **Remedy.** Add an explicit config flag for "force parallel" (pydantic-ai `parallel_execution_mode`) or document the downgrade.

### Crashes inside `_step` leave inconsistent state

- letta explicitly rolls back message persistence on exception ("message persistence does not happen in the case of an exception" at `letta/agents/letta_agent_v3.py:1513`) but leaves the Step row in an error state.

**Indicator.** Post-crash, messages exist without a Step, or Steps exist without messages. **Remedy.** Use the `StepProgression` sentinel (letta `StepProgression`) to make `finally` decide deterministically.

## Notable Absences

The following are commonly expected in this dimension but absent or under-documented in the sources:

- **First-class `Turn` class in agent-framework, crewai (legacy), pydantic-ai slim**. These sources have a "turn" (an iteration of their loop) but no type with `id, start, end, finish_reason, usage`; reconstruct turn boundaries from message ordering.
- **Per-turn UUID in openai-agents-sdk**. `RunState` has the integer `_current_turn` but no `_current_turn_uuid`. Cross-SDK or cross-version resume matching depends on integer collisions.
- **Stable mid-loop cancellation in agent-framework, crewai legacy**. The chat-client loop has no documented `asyncio.CancelledError` handling surface; the harness loop sits on top of it. Letta's cancellation is best-effort at step boundaries (long LLM calls run to completion).
- **Built-in turn snapshot primitive in pydantic-ai slim**. Durable persistence is explicitly delegated to `durable_exec/{temporal,dbos,prefect}/_agent.py` — without one of those adapters, a slim-only user gets no checkpointing.
- **Loop phase extraction in letta V3**. `_step()` mixes (a) message refresh, (b) request building, (c) LLM call + retry + fallback, (d) response handling, (e) parallel/serial tool execution, (f) continuation decision, (g) persistence, (h) post-step compaction, (i) span/trace emission. The "can you draw the loop" test fails at ~700 lines.
- **Aggregated-only usage in agent-framework, pydantic-ai**. Usage is captured per pass but applied to `response.usage_details` at the end. Per-turn usage requires wrapping `ChatMiddleware` or inspecting `StepMetrics` (letta) / `turn_span` (openai-agents-sdk).
- **Turn grouping in openhands event log**. Events are stored one-per-file sorted by ISO timestamp; no in-source projection of "one turn = one model call + N tool calls."

## Per-Source Notes

**agent-framework** — Strong layered design with the loop placed at `FunctionInvocationLayer` mixin (`_clients.py:975-999`). Reuses across every chat client subclass. Forced `tool_choice="none"` tail after `max_iterations` prevents orphaned tool calls. Loop is split across `_tools.py` (turning) and `_sessions.py` (persistence), which just meets the "five files" bar. Per-call durability is opt-in (`require_per_service_call_history_persistence=True`); default flow only persists on `after_run`, so a crash mid-loop loses work. The two `DEFAULT_MAX_ITERATIONS` constants (tool-loop 40, harness-loop 10) are a naming hazard.

**crewai** — Three near-duplicate loops (`CrewAgentExecutor` legacy, `AgentExecutor` Flow-based, `StepExecutor`) over shared helpers (`get_llm_response`, `process_llm_response`, `handle_max_iterations_exceeded`, `handle_output_parser_exception`, `handle_context_length`). Excellent hook/event observability on every path. Auto-downgrade from native tool calling to ReAct text tool calling on provider rejection. Loop persistence delegated to outer Flow checkpointing rather than the loop itself. `CrewAgentExecutor` is deprecated but still ships. Native-tool parallelism is silently downgraded to sequential if any tool has `result_as_answer` or `max_usage_count`.

**langgraph** — The loop is the Pregel BSP superstep engine, not a hand-written LLM turn loop. Per-superstep `put_writes` + `_put_checkpoint` is the durability primitive. "Turn" is not first-class — it is two supersteps (model → tools) glued by `should_continue` conditional routing. Recursion-limit hard cap + per-state `RemainingSteps` advisory cap. Three durability tiers (`sync`, `async`, `exit`). Per-task `PregelExecutableTask` carries `(name, input, writes, triggers, id, path)`. Subgraph limits are not unified across nested loops.

**letta** — Three coexisting generations (`LettaAgentV1/V2/V3`) selected by `AgentLoop.load()` factory. `Step` ORM row is the durable turn key (step_id reused across StepMetrics, OTel span, ProviderTrace, webhook). `StepProgression` state machine for failure cleanup. Idempotent resume on approval (reuses prior step_id). `_step()` is a ~700-line god coroutine. V2 forbids no-tool-call responses (raises LLMError); V3 allows them. Compaction sub-loop shared with main loop uses same `step_id`. Best-effort cancellation at step boundaries.

**openai-agents-sdk** — Flat `while True` with `NextStep*` discriminated union (FinalOutput / Handoff / RunAgain / Interruption). Sync + streaming variants in lock-step. Schema-versioned `RunState` at `1.11` for HITL pause/resume; fail-fast on unsupported versions. `turn_span` trace per iteration. `SingleStepResult` dataclass unifies the turn body return. `OpenAIServerConversationTracker` ships only deltas when server-managed id is set. Cancel-mode choice (`immediate` vs `after_turn`). Two large loop bodies (737 + 528 lines). No UUID per turn.

**openhands** — Loop is in `openhands-sdk` (external). This source is a FastAPI orchestrator that proxies `send-message` to a sandbox-hosted `agent_server`. Documented as "intentionally a thin proxy." One-event-per-file storage sorted by ISO timestamp. `SandboxStatus == RUNNING` gates send-message. Pending-message queue replays messages after `READY`. Custom agent-server image version is checked (`_verify_agent_server_version`). Cannot intervene mid-turn beyond `switch_profile` / `switch_acp_model`.

**pydantic-ai** — 4-node agent graph (`UserPromptNode → ModelRequestNode ⇄ CallToolsNode → SetFinalResult`) driven by generic `pydantic_graph` runtime. `GraphAgentState` is the snapshot primitive. Retry is a new node (`_build_retry_node`), not a flag. Capability middleware around model call (`wrap_model_request`). Cooperative `stream_ready`/`stream_done` events for cancellation safety. Three named parallel execution modes (`sequential`, `parallel`, `parallel_ordered_events`). Streaming + non-streaming paths in `ModelRequestNode` are parallel implementations. No built-in turn snapshot inside slim (delegated to `durable_exec/{temporal,dbos,prefect}/`).

## Open Questions

These are gaps the per-source reports flagged and that warrant further study outside this dimension.

- **Cross-subgraph `recursion_limit` budget in langgraph.** Each subgraph owns its own `SyncPregelLoop`/`AsyncPregelLoop` with its own step counter; nested subgraph limits are not unified. (`langgraph.md` flagged this.)
- **Mid-request cancellation latency for long LLM calls in letta.** Best/worst-case latency is bounded by (a) step granularity and (b) inflight LLM call duration; no metric or SLO target found in code. (`letta.md` flagged this.)
- **`run_step` semantics under `SkipModelRequest` in pydantic-ai.** `state.run_step += 1` is unconditional; `SkipModelRequest` short-circuits before the model call. Unclear whether `run_step` counts "turns attempted" or "turns completed." (`pydantic-ai.md` flagged this.)
- **Conversation-locked compatibility retry budget in openai-agents-sdk.** 3 backoff retries on `conversation_locked` (`model_retry.py:43, 558-570, 673-688`) before user-configured retry policy. Could be a study on its own.
- **Aggregated usage visibility per turn in agent-framework.** Captured per pass but applied at end of loop; per-turn breakdown requires wrapping `ChatMiddleware`.
- **Final-output condition type and rate in pydantic-ai.** Two terminal paths (`End(FinalResult)` for non-streaming, `SetFinalResult` for streaming) — could they be unified? (`pydantic-ai.md` flagged unification as future consideration.)
- **Hook-blocked LLM call in crewai.** A `before_llm_call` hook that returns `False` short-circuits with `ValueError("LLM call blocked by before_llm_call hook")` (`agent_utils.py:421-424`). How does this interact with the auto-downgrade flow?

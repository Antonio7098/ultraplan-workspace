## Executive Summary

Both LangGraph and Temporal treat replay not as a debugging aid but as **the primary mechanism by which mutable state is reconstructed from a persisted source of truth**, and both rate **9/10** on this dimension. The convergence is striking: in each system the answer to "can execution be replayed?" is "yes, and it is the only way state is ever reconstructed after restart, time-travel, reset, or cross-cluster sync." In both, the answer to "are LLM calls replayed from logs or re-called?" is also the same: **re-called, by design** — neither system records model tokens as part of its source of truth, and both push model-output determinism up to the user (LangGraph via an opt-in `CachePolicy` keyed on `args_key`; Temporal by routing model calls through the SDK worker, which re-invokes the user workflow function against the recorded event stream).

The interesting divergence is *what the source of truth is shaped like*:

- **LangGraph (9/10)** stores a **channel-versioned checkpoint graph**: each step persists a `Checkpoint` (`v`, `id`, `ts`, `channel_values`, `channel_versions`, `versions_seen`, `pending_sends`, `updated_channels`) plus per-task `checkpoint_writes` and per-channel-per-version `checkpoint_blobs`. Replay is driven by `is_replaying = CONFIG_KEY_CHECKPOINT_ID in config[CONF]` (`libs/langgraph/langgraph/pregel/_loop.py:308`), `ReplayState` (`libs/langgraph/langgraph/_internal/_replay.py:14-90`) handles subgraph branching, and `DeltaChannel.replay_writes` (`libs/langgraph/langgraph/channels/delta.py:139-157`) folds ancestor writes through a batching-invariant reducer. Determinism comes from `xxh3_128` content-hashed task IDs (`libs/langgraph/langgraph/pregel/_algo.py:1404-1409`), monotonic `uuid6(clock_seq=step)` checkpoint IDs (`libs/checkpoint/langgraph/checkpoint/base/id.py:79-109`), and strictly sorted iteration (`_algo.py:256`). Model-output replay is opt-in via `CachePolicy` (`libs/langgraph/langgraph/pregel/_algo.py:663-687`).
- **Temporal (9/10)** stores an **append-only event log keyed by opaque history branch tokens**: every replay path (workflow reset, replication gap-fill, conflict resolution, admin recovery, import) funnels through `MutableStateRebuilderImpl.ApplyEvents` (`service/history/workflow/mutable_state_rebuilder.go:70-101`), which dispatches to a per-event-type `case` block (`mutable_state_rebuilder.go:152-693`). Replay is *five* independent code paths converging on one primitive. Determinism is unconditional for server-side mutable state because the events are pre-recorded; verification uses `rebuildVersionHistory.Equal(replayVersionHistory)` after conflict-resolution rebuild (`service/history/ndc/conflict_resolver.go:166-168`). The SDK worker replays user code against the same history via `worker.NewWorkflowReplayer()` and fails fast on non-determinism in tests (`temporaltest/server.go:50`).

The platform-specific patterns worth copying from each are largely orthogonal: from LangGraph, **the deterministic-task-id hashing scheme and the per-channel-version blob store** (`libs/langgraph/langgraph/pregel/_algo.py:1395-1409`, `libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-649`); from Temporal, **the single-rebuilder-many-callers architectural discipline and the explicit `SetReplayEventBatchID` hook that keeps replay distinct from live event generation** (`service/history/workflow/mutable_state_impl.go:1369-1395`). What both converge on — *event/channel-sourced source of truth, replay as the only way to reconstruct state, model calls always re-called, replay tests for every back-end* — is the load-bearing content of this dimension.

## Core Thesis

> Replay in a workflow runtime is not a feature; it is the architecture. A platform earns a high score on this dimension by making the persisted event/checkpoint stream the **only** source of truth for mutable state, by funneling every reconstruction path (time-travel, reset, replication, conflict resolution, crash recovery) through **one replay primitive**, and by drawing the determinism boundary **above** the user code: workflow replay is deterministic by construction, model-output replay is opt-in or the user's responsibility. LangGraph and Temporal both reach 9/10 because they share that architecture, and the small differences — channel-version graphs with parent-chain walks (LangGraph) vs. branch-tokened append-only event streams (Temporal) — are valid responses to different product shapes (single-process Python agent loops vs. multi-cluster Go history services), not quality differences.

## Approach Models

The two sources land in adjacent but distinct approach models for replay. The boundary tracks whether the system organizes replay around **state versions** (channel versions in a checkpoint) or **event versions** (history branch tokens keyed by event id + version). Both make replay the *only* state-construction path, but the operational primitives differ.

### A. Channel-versioned checkpoint graph (LangGraph)

LangGraph's source of truth is a **directed acyclic graph of checkpoints**, each carrying per-channel version counters, per-task writes, and per-channel blob snapshots. Replay is triggered when a caller supplies `config["configurable"]["checkpoint_id"]`; the `PregelLoop.__init__` sets `self.is_replaying` and steers hydration (`libs/langgraph/langgraph/pregel/_loop.py:308`, `:1606-1686`). On time travel, a **fork checkpoint** is created with `metadata.source = "fork"` (`libs/langgraph/langgraph/pregel/_loop.py:866-959`), so history is append-only and never mutated. The `_reapply_writes_to_succeeded_nodes` mechanism restores writes to tasks that previously finished (`libs/langgraph/langgraph/pregel/_loop.py:724-737`), while pending tasks with empty writes re-execute. `ReplayState` propagates the replay point across subgraph boundaries (`libs/langgraph/langgraph/_internal/_replay.py:14-90`); subsequent invocations of the same subgraph fall back to the latest checkpoint so newly-produced checkpoints during replay are visible.

Determinism is engineered at three layers:

1. **Task ID determinism** — `_xxhash_str` (or `_uuid5_str` for v1 checkpoints) over `(checkpoint_id, namespace, step, name, kind, …parts)` (`libs/langgraph/langgraph/pregel/_algo.py:1395-1409`). Two runs of the same workflow produce identical task IDs.
2. **Checkpoint ID determinism** — `uuid6(clock_seq=step)`, monotonic per saver instance (`libs/checkpoint/langgraph/checkpoint/base/id.py:79-109`). The lexicographic ordering is equivalent to chronological ordering, which is what makes `ORDER BY checkpoint_id DESC` in Postgres correct.
3. **Iteration determinism** — `tasks = sorted(tasks, key=lambda t: task_path_str(t.path[:3]))` (`libs/langgraph/langgraph/pregel/_algo.py:256`), `updated_channels = sorted(updated_channels)` (`libs/langgraph/langgraph/pregel/_checkpoint.py:120`), and `sorted(triggered_nodes)` in `prepare_next_tasks` (`libs/langgraph/langgraph/pregel/_algo.py:482`).

The standout mechanism is **`DeltaChannel.replay_writes`** (`libs/langgraph/langgraph/channels/delta.py:139-157`): writes are stored separately from the channel value (a `_DeltaSnapshot` sentinel sits in `channel_values`), and reconstruction walks the parent chain via `BaseCheckpointSaver.get_delta_channel_history` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-649`), which returns `{writes: oldest→newest, seed: nearest ancestor's value}`. The reducer must satisfy `reducer(reducer(s, xs), ys) == reducer(s, xs + ys)` — the formal batching-invariance contract the user signs when using DeltaChannel (`libs/langgraph/langgraph/channels/delta.py:40-49`). For Postgres, the two-stage SQL in `_build_delta_stage1_sql` + `_build_delta_stage2_sql` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:444-550`) does page-scan + per-channel `UNION ALL`.

**LLM replay is intentionally absent.** Pregel never records model responses; `CachePolicy` (`libs/langgraph/langgraph/pregel/_algo.py:663-687`) is an opt-in per-node writes cache keyed on `identifier(proc) + args_key`. When the policy matches, the cached `task.writes` are re-applied without invoking the runnable. The line is drawn at the runnable boundary: writes inside a runnable are cached as a batch and never broken apart per token.

### B. Event-sourced append-only history with branch tokens (Temporal)

Temporal's source of truth is an **append-only event log keyed by history branch tokens**. The architecture doc states it explicitly: "The system functions via event sourcing: an append-only history of events is stored for each workflow execution, and all required workflow state can be recreated at any time by replaying this history" (`docs/architecture/README.md:32`). Replay is **five** independent code paths that all funnel through `MutableStateRebuilderImpl.ApplyEvents` (`service/history/workflow/mutable_state_rebuilder.go:70-101`):

1. **Workflow reset** — `workflowResetterImpl.replayResetWorkflow` forks a branch and rebuilds up to the reset point (`service/history/ndc/workflow_resetter.go:434-508`).
2. **Replication gap-filling** — `HistoryReplicatorImpl.BackfillHistoryEvents` (`service/history/ndc/history_replicator.go:210-231`).
3. **Conflict resolution** — `ConflictResolverImpl.rebuild` streams events from a non-current branch (`service/history/ndc/conflict_resolver.go:116-183`), then asserts `rebuildVersionHistory.Equal(replayVersionHistory)` (`:166-168`).
4. **Admin recovery** — `RebuildMutableState` RPC → `workflowRebuilderImpl.rebuild` → `RebuildWithCurrentMutableState` (`service/history/history_engine.go:993-1006`, `service/history/workflow_rebuilder.go:64-104, 201-232`).
5. **Cross-cluster import** — `mutableStateMapper.GetOrRebuildMutableState` (`service/history/ndc/history_importer.go:249-253`).

Each replay paginates the branch via `StateRebuilderImpl.Rebuild` (`service/history/ndc/state_rebuilder.go:103-150`) using `ReadHistoryBranchByBatch` and applies events in order. **There is no second, divergent replay implementation.** The events cache (`service/history/events/cache.go:23-144`) keys events by `(NamespaceID, WorkflowID, RunID, EventID, Version)` with a dynamic-config TTL; cache is best-effort, persistence is authoritative.

Determinism is unconditional for server-side mutable state because the events are pre-recorded. The `MutableStateImpl.SetReplayEventBatchID` hook (`service/history/workflow/mutable_state_impl.go:1369-1395`) is the single explicit knob that distinguishes replay from live event generation: it records the batch first-event ID so that `GenerateEventLoadToken` references the original batch, preventing replay from generating load tokens pointing to batches that do not yet exist in cache. The flag is reset on transaction close (`mutable_state_impl.go:8357`).

Reset is a **curated reapply**, not an automatic replay: `reapplyEvents` cherry-picks which event types to re-apply (`service/history/ndc/workflow_resetter.go:842-1039`) — signals, updates, cancel requests, options updates, terminations (non-resetter), and any HSM-registered event type via `def.CherryPick(root, event, ...)` (`:1015-1031`). Identity-terminated events are skipped via `IsTerminatedByResetter` (`:1164-1169`) so the resetter does not re-apply its own terminations. Updates are dedup'd across branches by `(runIdForDeduplication, eventId, version)` (`workflow_resetter.go:854-860, 1033-1036`) and by `targetBranchUpdateRegistry.Find(attr.Request.Meta.UpdateId)` (`:890-891`).

**CHASM is an explicit exception.** `chasm/lib/workflow/workflow.go:177-178` states "Unlike HSM callbacks, CHASM replicates entire trees rather than replaying events, so deterministic cross-cluster IDs based on event version are not needed." This is a documented architectural choice: CHASM buys replication simplicity at the cost of event-level replay granularity.

### Why the two approach models converge on "replay = state reconstruction"

Both LangGraph and Temporal treat replay as the *only* way mutable state is materialized, regardless of whether the trigger is a crash, a fork, a reset, a replication sync, or a conflict resolution. The mechanism differs (channel versions vs. event versions; composite primary keys vs. branch tokens; `ReplayState` vs. `reapplyEvents`), but the *contract* is identical: write a deterministic record, re-apply it, get the same state back. This convergence is the load-bearing content of the dimension — anything less than "replay is the only construction path" is replay-as-debugging, not replay-as-architecture.

## Pattern Catalog

Patterns below appear in at least one source with unusually clear evidence and have direct analogues in the other source.

### P1 — One replay primitive, many callers

**Problem.** When the codebase has multiple replay code paths (time-travel, reset, replication, conflict, recovery, import), they tend to diverge. Each new caller adds its own event-handling switch statement, and over time no two callers agree on what an event means.

**Where.** Temporal's `MutableStateRebuilderImpl.ApplyEvents` (`service/history/workflow/mutable_state_rebuilder.go:70-101`) is called by every one of the five replay paths. The per-event-type switch (`mutable_state_rebuilder.go:152-693`) is the single source of truth for event semantics. LangGraph's analogue is `PregelLoop.__init__` setting `is_replaying = CONFIG_KEY_CHECKPOINT_ID in config[CONF]` (`libs/langgraph/langgraph/pregel/_loop.py:308`), reused for both `Pregel.__call__` time-travel and resume-after-interrupt, distinguished by `is_time_traveling` (`libs/langgraph/langgraph/pregel/_loop.py:1019-1057`).

**Why it works.** The replay primitive is a *narrow interface* (give me events, give me state, give me back state). Adding a new replay path means wiring to the existing primitive, not duplicating it. For Temporal this means `RebuildMutableState` for corruption recovery costs ~zero new code; for LangGraph this means the resume path "is just time travel from the current head."

**When to copy.** Always. The cost is essentially zero (one interface), and the benefit compounds over time as new operational needs arise (admin recovery, migration, replication).

**When it is overkill.** Almost never. The only exception is when the underlying state representation is itself heterogeneous enough that no single rebuilder can sensibly handle all event types — Temporal's CHASM decision is a deliberate carve-out (`chasm/lib/workflow/workflow.go:177-178`), not a counterexample.

### P2 — Content-hashed identifiers for deterministic replay

**Problem.** Two runs of the same workflow (e.g., original + replay) must produce identical task IDs / event IDs so that downstream dedup, ordering, and write-through work correctly. Random UUIDs break this.

**Where.** LangGraph uses `xxh3_128` over `(checkpoint_id, namespace, step, name, kind, …parts)` for task IDs (`libs/langgraph/langgraph/pregel/_algo.py:1404-1409`), with `_uuid5_str` (SHA-1) for v1 checkpoints (`:1395-1401`). Temporal records the *batch* in `MutableStateImpl.SetReplayEventBatchID` (`service/history/workflow/mutable_state_impl.go:1369-1395`), so the event-id used to compute downstream tokens is the same as the one used in the original generation.

**Why it works.** Identifiers are functions of the workflow's logical state, not random or wall-clock values. Given the same checkpoint lineage, the hash is identical; given the same event batch, the event ID is identical.

**When to copy.** Whenever identifiers cross replay boundaries (task IDs, event IDs, write keys, dedup keys). Don't copy wall-clock or random IDs into these slots — LangGraph's `random.random()` suffix in `get_next_version` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:543-552`) is harmless because ordering is by the integer prefix, but a similar suffix in a *key* slot would break replay.

**When it is overkill.** When identifiers never cross replay boundaries (e.g., log messages, transient cache keys).

### P3 — Append-only lineage with branch/fork checkpoints

**Problem.** Replay must be able to *branch* (time-travel, reset, replication) without destroying prior history. Mutating the original chain corrupts audit trails and breaks later diffs.

**Where.** LangGraph creates fork checkpoints with `metadata.source = "fork"` (`libs/langgraph/langgraph/pregel/_loop.py:866-959`) and asserts the original chain is preserved in `test_replay_creates_branch_preserving_old_checkpoints` (`libs/langgraph/tests/test_time_travel.py:2985-3109`). Temporal's `ForkHistoryBranch` constructs a new branch token inheriting ancestors up to the fork node (`common/persistence/history_manager.go:36-127`); `DeleteHistoryBranch` ref-counts branches so pruning does not destroy active ancestors (`common/persistence/history_manager.go:130-210`).

**Why it works.** Branching is the only way to preserve the past while exploring alternatives. Each branch has its own replayable event stream but shares ancestors, so reconstruction from any branch is cheap and complete.

**When to copy.** When your system has *any* replay path that creates a new execution from an existing one (time-travel, reset, retry-with-different-code, fork, replication). Append-only lineage is the assumption under which branch ref-counting, conflict resolution, and audit trails all work.

**When it is overkill.** When lineage is genuinely never branched (single linear log) — but in that case, you have a simpler system than either LangGraph or Temporal and don't need this dimension to score high.

### P4 — Idempotency marker on reapplied events / replayed writes

**Problem.** Replay that crosses a boundary (reset, replication, conflict resolution) can re-apply events that were already applied. Without dedup, replays double-write.

**Where.** Temporal uses `IsResourceDuplicated(runIdForDeduplication, …)` keyed on `(runId, eventId, version)` (`service/history/ndc/workflow_resetter.go:854-860, 1033-1036`) and `targetBranchUpdateRegistry.Find(attr.Request.Meta.UpdateId)` for cross-branch dedup (`:890-891`). LangGraph relies on composite-primary-key `ON CONFLICT DO NOTHING` for regular channels and `ON CONFLICT DO UPDATE` for the four control channels (`ERROR`, `SCHEDULED`, `INTERRUPT`, `RESUME` — `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-159`, `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:445-482`).

**Why it works.** Either the storage layer enforces idempotency (LangGraph) or the application layer tags each reapplied event with a dedup marker (Temporal). Both prevent the same event from being applied twice on the same branch.

**When to copy.** Whenever replay crosses a logical boundary where the same event could be applied to two different mutable states (cross-cluster replication, cross-thread migration, reset-with-update-reapply). Storage-layer dedup is preferable when feasible; application-layer dedup is necessary when the storage layer cannot express the right composite key.

**When it is risky.** If the dedup marker is in memory only (Temporal's `ms.IsResourceDuplicated` is in-memory — `service/history/ndc/workflow_resetter.go:858`), a passive cluster that rebuilds independently must rebuild its own dedup set by replaying the same events. This is acceptable for a single-cluster primary but becomes a coordination problem at scale.

### P5 — Two-tiered determinism: workflow replay deterministic, model replay opt-in

**Problem.** A workflow runtime that calls an LLM cannot deterministically replay the model output unless the model itself is deterministic. But forcing the runtime to record every model token breaks the platform's separation of concerns and adds latency/cost to every call.

**Where.** LangGraph draws the line at the runnable boundary: `CachePolicy` (`libs/langgraph/langgraph/pregel/_algo.py:663-687`) is an opt-in per-node writes cache keyed on `args_key`. When the policy matches, the cached `task.writes` are re-applied without invoking the runnable (`libs/langgraph/langgraph/pregel/_loop.py:1526-1539`). Otherwise the runner re-invokes the runnable on every replay (`libs/langgraph/langgraph/pregel/main.py:2980-2998`). Temporal routes LLM calls through the SDK worker; the server never sees them. SDK replay (`worker.NewWorkflowReplayer().ReplayWorkflowHistory` — `service/worker/scheduler/replay_test.go:21-42`) re-invokes the user workflow function against recorded events; `temporaltest/server.go:50` enables fail-fast non-determinism in tests.

**Why it works.** The platform guarantees that workflow *structure* is deterministic (channel versions, event ordering, task IDs), and the user is responsible for guaranteeing that workflow *contents* are deterministic (model temperature, prompt caching, idempotent side effects). The contract is honest: the framework will replay deterministically up to the model boundary, after which it's the user's problem.

**When to copy.** Whenever your runtime sits below user code that calls non-deterministic external systems (LLMs, third-party APIs, time-dependent services). Don't try to make the runtime itself deterministic at the model boundary — that's a tar pit. Do expose a hook for users who *want* opt-in determinism (LangGraph's `CachePolicy`; Temporal's worker-side replayer for test-time verification).

**When it is risky.** When users assume the framework will give them replay-verify semantics. Both sources are explicit that this is *not* the contract — `temporaltest/server.go:50` enables fail-fast only in tests, and LangGraph's `CachePolicy` is opt-in. Documentation must be loud.

### P6 — Conformance harness for replay correctness

**Problem.** When the source-of-truth store has multiple back-ends (InMemory, Postgres, SQLite, Shallow), each back-end must satisfy the same replay contract. Without a conformance test suite, divergence creeps in.

**Where.** LangGraph ships `libs/checkpoint-conformance/langgraph/checkpoint/conformance/` as a separate library with capability detection (`capabilities.py:66-96`), per-saver PUT/PUT_WRITES/GET_TUPLE/LIST/DELETE_THREAD and optional DELETE_FOR_RUNS/COPY_THREAD/PRUNE/DELTA_CHANNEL_HISTORY tests (`validate.py:45-128`), plus round-trip tests (`test_put.py:39-51`) and list-ordering tests (`test_list.py:97-…`). Temporal ships per-event-type replay tests (50+ `TestApplyEvents_EventTypeXxx` cases at `service/history/workflow/mutable_state_rebuilder_test.go:164-2158+`), reset replay tests (`service/history/ndc/workflow_resetter_test.go:283-343`, `resetter_test.go:111-202+`), admin rebuilder tests (`service/history/workflow_rebuilder_test.go:175-216`), and integration tests (`tests/reset_workflow_test.go:45-1014+`).

**Why it works.** The contract is executable. Adding a new back-end (or a new event type in Temporal) means running the suite and seeing it pass. Subtle ordering bugs surface as conformance failures, not production incidents.

**When to copy.** Whenever the source-of-truth store is pluggable. Don't ship a back-end without running the conformance suite against it. The harness is cheap to write (one interface, one test per method) and pays for itself the first time it catches a SQL syntax drift.

**When it is risky.** The harness can give false confidence if it only tests storage round-trip, not workflow-level equivalence. LangGraph's `test_time_travel.py` and `test_time_travel_async.py` (100+ tests) live in the framework tests, not the conformance suite, and they run only against the bundled checkpointer fixtures (`libs/langgraph/langgraph/tests/test_time_travel.py:1-…`). The conformance suite does *not* test workflow-level replay equivalence — see "Notable Absences."

### Why these divergences are not quality differences

- **Channel versions vs. event versions.** LangGraph stores the *result* of every superstep (a checkpoint with channel state); Temporal stores the *input* to every superstep (the event). Both are valid: LangGraph optimizes for fast replay (one round-trip to load a checkpoint), Temporal optimizes for replay-as-verification (events are the contract the user code is checked against).
- **Subgraphs vs. child workflows.** LangGraph has a Subgraph concept where the same node appears in a loop with different inputs; this needs `ReplayState` to share the replay point across iterations. Temporal expresses hierarchy via child-workflow events on a single branch, so the rebuilder handles it uniformly.
- **One replay path vs. five.** LangGraph's two paths are time-travel and resume; Temporal's five paths include replication and conflict resolution, which exist because Temporal is a multi-cluster system. The number of paths tracks operational complexity, not engineering quality.
- **Storage-layer dedup vs. application-layer dedup.** LangGraph's `ON CONFLICT` enforcement is elegant but requires SQL; Temporal's `IsResourceDuplicated` works with any KV store. Both prevent double-application.

## Decision Guide

Use the questions below to choose between the two approach models for a new system, or to evaluate whether an existing system has earned a 9/10.

**1. Is replay the *only* way mutable state is reconstructed?**

- **Yes, and it is exercised by operational paths beyond user-driven time-travel (reset, replication, recovery).** → You are closer to Temporal's model. Invest in a single rebuilder primitive and discipline new callers to funnel through it.
- **Yes, but the only trigger is user-driven time-travel / resume.** → You are closer to LangGraph's model. A `PregelLoop`-style replay flag plus checkpoint hydration is sufficient.
- **No — replay is a debugging aid, and crash recovery reconstructs state via a different path.** → You have a latent bug. Two reconstruction paths will diverge. Pick one and make it the only path.

**2. Does your runtime call non-deterministic external systems (LLMs, third-party APIs)?**

- **Yes.** → Draw the determinism boundary at the runnable boundary (LangGraph) or the SDK boundary (Temporal). Do not try to make the runtime itself deterministic at the model boundary. Expose an opt-in cache (LangGraph's `CachePolicy`) or an SDK-side replayer (Temporal's `worker.NewWorkflowReplayer()`).
- **No.** → You can deterministically replay everything, but you still need an explicit mechanism to surface "the replay produced a different result than the recorded execution." Temporal's `rebuildVersionHistory.Equal(replayVersionHistory)` is one such mechanism.

**3. Is your state storage pluggable across multiple back-ends (InMemory, Postgres, SQLite, custom)?**

- **Yes.** → Ship a conformance harness (LangGraph's `libs/checkpoint-conformance`) and run it against every back-end. Tests should cover round-trip, ordering, and pruning. Capability detection (which methods does this back-end support?) is worth the investment.
- **No — single storage backend.** → You don't need a conformance suite, but you still need per-event-type replay tests if your state is event-sourced. Temporal's 50+ `TestApplyEvents_EventTypeXxx` cases are the model.

**4. Do you have multi-cluster replication, conflict resolution, or admin recovery?**

- **Yes.** → You need branch tokens (Temporal's `ForkHistoryBranch`) or fork checkpoints (LangGraph's `source="fork"`), append-only lineage, and explicit replay-time hooks (`SetReplayEventBatchID`). Reset/recovery should be `reapplyEvents`-style curated, not full-event replay.
- **No — single-cluster.** → Append-only lineage is still valuable for time-travel, but you can get away with a simpler fork-checkpoint model.

**5. Does your system have a "reducer" concept (channels, sub-states, accumulated state)?**

- **Yes.** → You need a batching-invariance contract (LangGraph's `reducer(reducer(s, xs), ys) == reducer(s, xs + ys)`) and a deterministic replay walk (`DeltaChannel.replay_writes`, `get_delta_channel_history`). Document the contract loudly; a static check on user-defined reducers catches non-batching-invariant reducers at compile time.
- **No — state is event-list-shaped.** → You don't need a reducer contract. The event log is the state.

## Practical Tips

### Patterns to copy

1. **One rebuilder primitive, many callers.** Adding a new replay path should be wiring, not reimplementation. Both sources demonstrate this discipline; the cost of replicating it is one interface.
2. **Content-hashed identifiers.** Task IDs, event IDs, write keys — all should be functions of logical state, not wall-clock or random values. LangGraph's `xxh3_128` over `(checkpoint_id, namespace, step, name, kind, …parts)` is the model.
3. **Append-only lineage with branching.** Use fork checkpoints or branch tokens; never mutate prior history. Both sources enforce this.
4. **Idempotency markers on reapplied events.** Either storage-layer composite PK with `ON CONFLICT` (LangGraph) or application-layer `(runId, eventId, version)` dedup (Temporal). Without one, replays double-write.
5. **Two-tiered determinism: workflow deterministic, model opt-in.** Don't try to make the runtime deterministic at the model boundary. Expose a hook (LangGraph's `CachePolicy`, Temporal's SDK replayer) for users who want model-output replay.
6. **Conformance harness with capability detection.** Pluggable storage back-ends need a contract test suite. LangGraph's `libs/checkpoint-conformance` is the model; Temporal's per-event-type tests are the equivalent for event-sourced state.
7. **Explicit replay-time hooks to distinguish replay from live.** Temporal's `SetReplayEventBatchID` keeps load tokens stable during replay; LangGraph's `is_replaying` flag steers `_reapply_writes_to_succeeded_nodes` to skip control writes. Both are cheap and prevent replay-time bugs.

### Patterns to avoid or delay until needed

1. **Naive `prune` without ancestor awareness.** LangGraph's `BaseCheckpointSaver.prune` docstring explicitly warns that naive `keep_latest` silently corrupts DeltaChannel state (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`). If you ship a prune operation, document the contract or refuse the call when an ancestor is needed.
2. **Random suffixes in *key* slots.** LangGraph's `random.random()` suffix in `get_next_version` is harmless because ordering is by the integer prefix. Don't use the same pattern for *keys*.
3. **Cross-cluster update dedup via in-memory only.** Temporal's `ms.IsResourceDuplicated` is in-memory only (`service/history/ndc/workflow_resetter.go:858`); a passive cluster that rebuilds independently must rebuild its own dedup set. If you need cross-cluster dedup, persist the marker.
4. **`ShallowPostgresSaver`-style "no replay" back-ends.** LangGraph deprecated this for `durability="exit"` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/shallow.py:192-196`). Operational needs change; a "no replay" back-end is a future trap.
5. **Replay-time assertions comparing to recorded model tokens.** Neither source does this. It is theoretically desirable but practically impossible without per-call recording, which breaks the platform's separation of concerns.

### Decision rules

- **One replay primitive or five?** Start with one. Add more only when a new operational need arises (replication, conflict, recovery). Resist the temptation to write a second rebuilder "because it's simpler."
- **Storage-layer dedup or application-layer dedup?** Storage-layer when the natural composite key fits the schema. Application-layer when it doesn't, or when the storage is opaque.
- **Content-hashed IDs or monotonic IDs?** Content-hashed for *task* IDs (so two runs of the same workflow produce the same IDs); monotonic for *checkpoint* / *batch* IDs (so the natural sort order is chronological). LangGraph uses both.
- **Fork checkpoint or branch token?** Fork checkpoint for state-versioned graphs (the fork inherits parent versions); branch token for event-sourced logs (the branch points to a span of events).
- **Replay-verify or replay-only?** Replay-verify on conflict resolution (where divergence is the bug you're trying to catch); replay-only on time-travel and resume (where the user trusts the recorded state).

### Caution signs

1. **Two or more replay code paths that don't funnel through one primitive.** The codebase will diverge. Find the primitive or create one.
2. **Storage back-ends without conformance tests.** Subtle ordering bugs (e.g., Postgres `ORDER BY` vs. InMemory `sorted()`) will surface in production. Ship a conformance harness.
3. **Random or wall-clock IDs in *key* slots.** Replay will produce different keys; dedup and idempotency will break.
4. **Replays that re-apply all events including resetter-written terminations.** Temporal's `IsTerminatedByResetter` exists because this is a real failure mode (`service/history/ndc/workflow_resetter.go:1164-1169`).
5. **Pruning without ancestor awareness.** DeltaChannel-style state walks break silently; no error is raised. Document the contract or refuse the call.
6. **LLM calls documented as "deterministic by default."** Neither LangGraph nor Temporal does this. If your docs do, you have a documentation bug.
7. **`is_replaying` flag that doesn't get reset after the first tick.** LangGraph resets `is_replaying = False` after the first tick (`libs/langgraph/langgraph/pregel/_loop.py:704`) so subsequent ticks run normally. Forgetting this causes replay to leak into normal execution.
8. **Replay correctness verified only by happy-path tests.** The interesting bugs are in conflict resolution, multi-cluster replication, and admin recovery. Test those paths explicitly.

## Anti-Patterns / Caution Signs

The following are patterns observed in the sources (or implied by their absence) that indicate the dimension is becoming brittle:

1. **Conformance suite tests storage round-trip, not workflow-level equivalence.** LangGraph's `libs/checkpoint-conformance` tests `put/get/list/copy/delete/prune/delta-channel-history` but does *not* test that re-running a workflow produces the same state. The high-level replay tests (`libs/langgraph/tests/test_time_travel.py`) live in the framework tests and run only against bundled checkpointer fixtures. A new back-end could pass conformance and still produce wrong replay results. **Caution sign:** if your conformance suite passes but you can't answer "what does this back-end do on time-travel?," your harness is incomplete.

2. **`ShallowPostgresSaver`-style "no replay" back-ends shipped as a feature.** LangGraph deprecated this for `durability="exit"` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/shallow.py:192-196`), but the surface is still there. Users who pick it for performance reasons will lose replay without realizing. **Caution sign:** if your back-end catalog includes "fast / no-replay" variants, you have a UX trap.

3. **Naive `prune` without awareness of DeltaChannel-style state walks.** LangGraph's `BaseCheckpointSaver.prune` docstring explicitly warns about this (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`). The failure mode is silent — `get_delta_channel_history` simply returns no `seed`, and the channel reconstructs as empty. **Caution sign:** if your `prune` API is a one-liner that doesn't check for delta-needing channels, it will corrupt state.

4. **`id=None` messages racing with replay.** LangGraph's `DeltaChannel.put_writes` calls `ensure_message_ids` before the background thread serialises them (`libs/langgraph/langgraph/pregel/_loop.py:451-458`) specifically because reducers that assign IDs inside `apply_writes` race with serialisation and `get_state()` replays produce a different UUID per call (`libs/langgraph/langgraph/pregel/_messages.py:429-432`). **Caution sign:** if your reducer assigns IDs and your serialisation is async, replays will produce different IDs.

5. **Time-travel fork vs. resume ambiguity.** When a client resumes with an explicit `checkpoint_id` pointing at the current head (e.g. LangGraph Studio sending `checkpoint: {checkpoint_id}` alongside `Command(resume=...)`), `is_replaying` is True but `is_time_traveling` is False. The loop must NOT pass `ReplayState` to subgraphs in that case (`libs/langgraph/langgraph/pregel/_loop.py:1019-1057`). **Caution sign:** if your "replay" and "resume" share code paths without a distinguishing flag, subgraphs will walk back to stale checkpoints.

6. **Cache key collisions.** LangGraph's `CachePolicy` key is `xxh3_128_hexdigest(args_key.encode())`. If two distinct inputs produce the same `args_key` via user-defined `key_func`, cached writes leak between calls (`libs/langgraph/langgraph/pregel/_algo.py:677-684`). **Caution sign:** if your cache key is user-defined and you don't validate it, collisions are silent.

7. **Server-side replay determinism vs. replay-with-different-time boundary implicit.** Temporal's reset overrides activity `ScheduledTime`/`FirstScheduledTime` to `now` (`service/history/ndc/workflow_resetter.go:577-578`) so post-reset timing is not bit-identical with pre-reset. The boundary is implicit. **Caution sign:** if your docs say "replay is bit-identical" but you override time-derived fields, your contract is misleading.

8. **CHASM-style opt-outs without end-to-end testing.** Temporal's CHASM does not replay events (`chasm/lib/workflow/workflow.go:177-178`); the determinism contract is split when both HSM and CHASM are involved. The report explicitly notes "No clear evidence found of an end-to-end determinism test that exercises both layers simultaneously." **Caution sign:** if your system has a documented opt-out from replay, you owe the world a test that exercises the boundary.

## Notable Absences

These are capabilities the dimension's prompt asks about but for which neither source provides strong evidence.

1. **Replay-verify comparing new run outputs to recorded writes.** Neither LangGraph nor Temporal does this at the framework level. The closest LangGraph has is `test_replay_interrupt_stable_across_replays` (`libs/langgraph/tests/test_time_travel.py:395-442`), which uses value equality across replays but only because the underlying model is a deterministic function. The closest Temporal has is the conflict-resolution equality check (`service/history/ndc/conflict_resolver.go:166-168`), which compares version histories, not model outputs. **Why it matters:** users who want "did the model produce the same answer?" must implement it themselves (e.g., via LangGraph's `CachePolicy` or Temporal's SDK replayer in tests).

2. **Published SLO on replay cost vs. original run.** Neither source publishes benchmarks. Empirically, LangGraph replay = hydration cost + un-resumed-task execution cost; Temporal replay = events cache lookup + persistence paging + event application. No benchmarks are linked in either report.

3. **Cross-thread replay.** LangGraph's `ReplayState` is single-instance per parent execution (`libs/langgraph/langgraph/_internal/_replay.py:21-23`). A user wanting to replay against a different thread's checkpoint must do it via `copy_thread`, which has its own determinism caveats around delta channels (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:361-372`). Temporal's equivalent (cross-cluster replay) is documented (`service/history/ndc/history_replicator.go:210-231`) but the boundary is operational, not user-facing.

4. **Input validation for `config["configurable"]["checkpoint_id"]`.** LangGraph's `PregelLoop` falls back to a synthetic empty checkpoint when the ID doesn't exist (`libs/langgraph/langgraph/pregel/_loop.py:1614`). A bogus checkpoint_id silently replays from empty — no error, no warning.

5. **What happens if `durability` changes mid-run.** LangGraph captures the durability flag at loop entry (`libs/langgraph/langgraph/pregel/_loop.py:313, 1662`) but a user toggling durability across replays would get inconsistent stub/anchor behavior. No clear evidence.

6. **Replay correctness under events-cache miss (Temporal).** The events cache is best-effort (`service/history/events/cache.go:118-144`); without it, replay falls back to `ReadHistoryBranch` per event. No clear evidence of replay correctness under cache misses.

7. **Automated production replay-verify runs (Temporal).** `temporaltest/server.go:50` enables fail-fast non-determinism in tests, but in production, the SDK's replay-verify is opt-in via `Replayer.ReplayWorkflowHistory`. No clear evidence of automated production runs.

8. **Static check on user-defined reducer batching-invariance (LangGraph).** `DeltaChannel` requires `reducer(reducer(s, xs), ys) == reducer(s, xs + ys)` (`libs/langgraph/langgraph/channels/delta.py:40-49`). No static check exists; non-batching-invariant reducers produce inconsistent replay at runtime.

9. **CHASM/HSM end-to-end determinism test (Temporal).** CHASM replicates trees rather than events; for workflows that include both HSM-managed sub-machines and CHASM nodes, the determinism contract is split. No clear evidence of a test that exercises both layers simultaneously.

10. **Reverse history replay impact (Temporal).** `ReadHistoryBranchReverse` is used by `GetWorkflowExecutionHistoryReverse` for UI pagination. Not a replay path, but adjacent. No clear evidence of any determinism impact.

## Open Questions

The following questions emerged from comparing the two sources and could not be answered from the per-source reports alone.

1. **What is the right model for cross-runtime replay?** LangGraph's `ReplayState` is single-instance per parent execution; Temporal's cross-cluster replay is operational, not user-facing. A user wanting to "replay this run but with a different model / different tools" has no first-class API in either. Would a generalized "replay against alternative inputs" primitive be valuable, and where would it live?

2. **How should replay correctness be verified end-to-end?** LangGraph's conformance suite tests storage round-trip; Temporal's tests cover per-event-type semantics. Neither tests "this workflow, with this model, produces the same answer when replayed." Is there a class of test that could close this gap without recording model outputs?

3. **What is the cost-vs-benefit of recording model outputs?** Both sources explicitly choose *not* to record model tokens. If the cost (storage, latency) were acceptable, would a built-in model-output cache (LangGraph's `CachePolicy` generalized, Temporal's worker-side cache) materially improve the user experience? The reports flag this as a "feature opportunity, not a missing primitive."

4. **How should `durability` / `replay-mode` evolve across replays?** LangGraph captures `durability` at loop entry; Temporal's reset mutates `now()`-derived fields. If a user replays the same execution with different durability / replay-mode settings, what should the framework guarantee? The reports flag this as a gap.

5. **What is the operational SLO for replay latency?** Neither source publishes benchmarks. For a 1000-step workflow, is replay 10x faster than the original run (events cache warm), or 0.1x (every event must be re-applied)? Users planning disaster-recovery runbooks would benefit from published numbers.

6. **How should pruning interact with DeltaChannel-style state walks?** LangGraph's `BaseCheckpointSaver.prune` docstring warns about this but provides no safe default. Temporal doesn't have an equivalent because its event log is the source of truth (pruning is `DeleteHistoryBranch` with ref-counting). What is the right contract for a system that wants both cheap pruning *and* replay correctness?

7. **What is the right boundary for "deterministic replay" claims?** Temporal claims server-side replay is "events are recorded → mutable state reconstructs identically" but reset mutates `now()`-derived fields. LangGraph claims workflow replay is deterministic but models are re-called. Both sources are honest about the boundary; the open question is whether a more precise vocabulary (e.g., "state-equivalent replay" vs. "byte-identical replay" vs. "model-output replay") would help users reason about correctness.

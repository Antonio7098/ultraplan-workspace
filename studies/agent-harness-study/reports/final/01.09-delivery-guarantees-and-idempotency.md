## Executive Summary

None of the four sources claims system-wide "exactly-once" semantics, and that absence is itself the first design lesson: each platform deliberately **avoids** the term and instead builds a layered model in which state-machine bookkeeping is *effectively-once* while user-execution side effects remain *at-least-once* (or *best-effort*). The interesting divergence is in *where* the dedup boundary lives and *how* it is exposed:

- **temporal (8/10)** pushes the boundary into the persistence layer through Request-ID-keyed conditional writes (`service/history/api/startworkflow/api.go:319-368`, `common/persistence/cassandra/queue_v2_store.go:32`) and surfaces `RequestID` at every public RPC. User activity bodies are explicitly outside the contract: the README states activity code "must either be idempotent or non-retryable (i.e. at least once or at most once)" (`docs/architecture/README.md:35`).
- **langgraph (7/10)** pushes the boundary into the storage schema via a composite `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` primary key with `ON CONFLICT DO NOTHING` for regular channels and `ON CONFLICT DO UPDATE` for the four control channels (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-159`, `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:445-482`). The user-visible knob is `durability = "sync" | "async" | "exit"` (`libs/langgraph/langgraph/types.py:87-93`).
- **agent-framework (6/10)** delegates most delivery semantics to Durable Task / Durable Entities. Only the per-session entity path inherits a real guarantee ("Durable Entities serializes access to each entity instance" — `docs/features/durable-agents/README.md:28`). Streaming is explicitly at-least-once and tells clients to dedupe (`dotnet/samples/04-hosting/DurableAgents/ConsoleApps/07_ReliableStreaming/README.md:160`). The in-process agent loop has no such model.
- **openhands (5/10)** is a heterogeneous application where every subsystem chose its own model: provider webhooks use Redis `SET NX EX` keyed on provider-stable IDs (`enterprise/server/routes/integration/gitlab.py:107`); event storage uses deterministic file paths (`openhands/app_server/event/event_service_base.py:190-197`); pending messages are at-most-once with delete-after-attempt (`openhands/app_server/pending_messages/pending_message_models.py:12-18`). There is no central doc.

Two patterns appear across all four sources: (1) an idempotency primitive tied to a stable identifier, supplied or inferred, for the request, and (2) a separation between a guaranteed-against-duplicates state layer and an at-least-once effects layer. The way each source composes them is the subject of the rest of this report.

## Core Thesis

> Delivery guarantees in agent frameworks are not a single value but a *composition*. The state machine and storage layer can be made effectively-once through a stable idempotency key plus a composite primary key or conditional write, but the user-execution side (activity body, node function, tool call) stays at-least-once by design. The platforms that score highest on this dimension treat the idempotency key as a first-class input to every public API (temporal's `RequestID`, agent-framework's `correlationId`), move the dedup boundary into the persistence layer rather than the application layer (langgraph's composite PK, temporal's `IF NOT EXISTS`, openhands' deterministic event-id paths), and tell users loudly that retries on side effects are the user's responsibility. The platforms that score lower treat dedup as in-memory state (openhands' callback results, agent-framework's `signal_entity` without an idempotency-key short-circuit), tolerate silent loss (openhands' pending messages), or hide the contract behind jargon (agent-framework's "Reliable Streaming" label).

## Approach Models

The four sources cluster into three approach models. The cluster boundaries track whether the source is a *workflow runtime* (temporal, langgraph), an *SDK that delegates to a runtime* (agent-framework), or an *application composed of subsystems* (openhands).

### A. Persistence-layer conditional writes with caller-supplied idempotency keys (temporal)

Temporal treats every public RPC as a conditional operation whose key is either caller-supplied (RequestID on signal, update, start, pause, worker-deployment) or generated as a unique ID on a stable column (`create_request_id` on `current_executions`, `requestID + idx` on Nexus callbacks, `VectorClock`/`TransitionCount` on replication tasks). The persistence interface returns success-on-conflict or a typed error carrying the conflicting IDs (`common/persistence/data_interfaces.go:113-125`, `common/persistence/cassandra/queue_v2_store.go:30-72`). Because the conditional write is the API contract, retries from a crashed client converge to the same persisted state regardless of which path tries.

A second sublayer — *stamp-and-clock* — rejects tasks that arrive at the history service from a stale matching decision: `RecordActivityTaskStarted` returns `NewObsoleteMatchingTask` if the stamp differs from what history expects (`service/history/api/recordactivitytaskstarted/api.go:178-184`). Without this, retry storms could cause two pollers to act on the same `scheduledEventID`.

A third sublayer — *state-machine idempotency* — covers cases that don't have a natural composite key. Workflow Updates keep an in-memory `updates map[string]*Update` rebuilt from history events on registry loss (`service/history/workflow/update/registry.go:226-235`); Signal dedup maintains a `pendingSignalRequestedIDs` set inside mutable state (`service/history/workflow/mutable_state_impl.go:2499-2518`); Nexus operation completion matches on `RequestID` and returns `ErrStaleReference` for terminal operations (`components/nexusoperations/executors.go:739`).

Agent-execution side effects are explicitly *out of scope*: `docs/architecture/README.md:35` documents this as a user contract.

### B. Storage-enforced composite-key dedup (langgraph)

LangGraph puts the dedup boundary in the **storage schema**: every checkpoint write carries the tuple `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` as a composite primary key, and the SQL back-ends issue `ON CONFLICT (...) DO NOTHING` for regular channel writes and `ON CONFLICT (...) DO UPDATE` for the four control channels (`ERROR`, `SCHEDULED`, `INTERRUPT`, `RESUME` — `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-159`, `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:445-482`). The Postgres saver even picks the query per call based on `all(w[0] in WRITES_IDX_MAP for w in writes)` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:347-379`).

The same boundary does double duty as in-flight retry safety: `_retry.py:600-682` calls `task.writes.clear()` before each attempt so a partially-completed retry cannot leak state across attempts; combined with the storage-layer `ON CONFLICT`, the node does not have to be aware of retries.

Durability is exposed as the only knob the user turns: `durability = "sync"` blocks the next checkpoint on the previous plus all delta writes; `"async"` (default) submits writes but does not await them; `"exit"` skips per-step persistence entirely (`libs/langgraph/langgraph/types.py:87-93`, `libs/langgraph/langgraph/pregel/_loop.py:459-498`). Cache short-circuit is wired in *before* the first retry attempt via `match_cached_writes` (`libs/langgraph/langgraph/pregel/_retry.py:714-718`).

External side effects performed inside node bodies are not in the contract. LangGraph offers no idempotency-key infrastructure for user code, no transactional bundling of node execution and checkpoint write, and no saga coordinator.

### C. Deterministic-path idempotency for object storage plus Redis-NX dedup at provider edges (openhands, with agent-framework as a partial sibling)

Where langgraph and temporal rely on relational conditional writes, openhands leans on **idempotency-by-key-path** for object stores and **atomic Redis SET** for webhook ingestion.

- *Event-store writes* go to `<conversation_id>/<event_id.hex>.json` (`openhands/app_server/event/event_service_base.py:190-197`) — three implementations (FS, S3, GCS) all write to a deterministic key, so concurrent writes with the same `event_id` are last-writer-wins. No conditional-write primitive (`IfNoneMatch`, `if_generation_match`) is used.
- *Webhook ingestion* uses Redis `SET NX EX` keyed on the most stable identifier the provider supplies (GitLab `object_attributes.id`, Bitbucket `x_request_uuid`, Slack `client_msg_id`, Jira signature — `enterprise/server/routes/integration/gitlab.py:96-113`, `enterprise/server/routes/integration/bitbucket.py:88-105`, `enterprise/server/routes/integration/slack.py:315-320`, `enterprise/server/routes/integration/jira.py:308-316`). HMAC verification runs *before* the dedup call so a forged payload cannot poison the dedup key.
- *Webhook enrollment* uses pre-existence checks (`check_webhook_exists_on_repository` — `openhands/app_server/integrations/bitbucket_data_center/service/webhooks.py:37-52`) so an idempotent create-or-update is achieved by reads-then-writes.
- *Conversation export* uses a Redis distributed lock with periodic TTL refresh so concurrent exports fail fast (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:2354-2419`). When Redis is unavailable, the lock falls open by default and a misconfigured env can produce duplicate exports.
- *Pending messages* are stored in SQL and bulk-deleted after attempt: this is **explicit at-most-once** by docstring ("deleted after processing, regardless of success or failure" — `openhands/app_server/pending_messages/pending_message_models.py:12-18`).

Agent-framework adopts a *partial* version of this: its `correlationId` is paired with a deterministic per-session entity key (`dotnet/src/Microsoft.Agents.AI.DurableTask/RunRequest.cs:37`, `DefaultDurableAgentClient.cs:23-29`), and the Reliable Streaming sample echoes the Redis-Streams-via-cursor pattern of openhands but adds a per-chunk monotonic `sequence` field for client-side dedup (`dotnet/samples/04-hosting/DurableAgents/AzureFunctions/08_ReliableStreaming/RedisStreamResponseHandler.cs:88-93`). Both at-least-once channels are honest about the contract.

## Pattern Catalog

The patterns below appear in at least two of the four sources or are unusually clearly demonstrated by one.

### P1 — Composite primary key as the dedup primitive

**Problem.** Two retries of the same request — possibly from different processes — must converge to a single persisted row.

**Where.** langgraph (Postgres + SQLite composite `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` — `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:66-76`, `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:152-162`); openhands (deterministic `<conversation_id>/<event_id.hex>.json` path — `openhands/app_server/event/event_service_base.py:190-197`); temporal (`(namespace, workflowID, run_id) + last_write_version` on `current_executions` — `common/persistence/cassandra/mutable_state_store.go:399-444`).

**Why it works.** The schema, not the application, enforces dedup. Every back-end exposes identical semantics; an orchestrator (or another replica) can re-issue the same write and stay safe.

**When to copy.** Whenever the entity being persisted has a natural composite key available *and* the storage is relational or key-blob. Do not copy when keys would have to be invented or when writes need to span rows (use a transactional outbox instead).

**When it is overkill.** For ephemeral data whose loss is acceptable, the extra index cost and write-conflict classification are not worth it.

### P2 — Caller-supplied (or auto-generated) Request-ID dedup

**Problem.** A request reaching the platform may be retried by the network layer; the platform must recognize retries as the same logical operation.

**Where.** temporal (`RequestID` on signal, update, start, pause, worker-deployment — `service/frontend/workflow_handler.go:4210-4214`, `service/history/api/signalworkflow/api.go:45-89`); agent-framework (`correlationId` per `RunRequest` — `dotnet/src/Microsoft.Agents.AI.DurableTask/RunRequest.cs:37`; auto-generated when the caller omits, which is a partial regression); langgraph (`task_id` plus persistence-layer dedup — `libs/langgraph/langgraph/pregel/_runner.py:603`).

**Why it works.** Concurrency control maps to a single string in the request envelope. Replay, retry, and timeout all converge to the same outcome.

**When to copy.** When designing a public API that may be retried by the network or the caller.

**When it is risky.** Auto-generation (agent-framework style) is fine for response correlation but does not help *input* dedup unless the caller captures the value across retries. A consumer that retries the HTTP call from scratch gets a fresh GUID every time, and the framework has no way to know. Counter-evidence: openhands routes that take a provider-supplied UUID explicitly rather than generating one inside the route avoid this trap.

### P3 — Conditional-write primitive in the persistence layer

**Problem.** Even with a composite key, two writers may race; the storage has to refuse the second writer or convert it into a no-op success.

**Where.** temporal (Cassandra `INSERT ... IF NOT EXISTS` on queue messages — `common/persistence/cassandra/queue_v2_store.go:32`; SQL `ON CONFLICT` in `common/persistence/sql/execution.go:81-208`); langgraph (Postgres `ON CONFLICT ... DO NOTHING` for regular channels, `DO UPDATE` for special channels — `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-159`); openhands (Redis `SET NX EX` for webhook dedup — five providers, `enterprise/server/routes/integration/*.py`).

**Why it works.** Atomicity is provided by the data store, which is faster and more correct than a lock on the application side.

**When to copy.** Whenever the storage layer supports it (every relational DB, Cassandra, Redis `SET NX`). For object stores that lack conditional writes (S3, GCS), consider a deterministic path plus a separate dedup store, or accept last-writer-wins with versioning.

### P4 — At-most-once at the boundary, at-least-once inside, effectively-once at the persistence layer

**Problem.** The user wants a single mental model that survives crashes.

**Where.** All four sources adopt this layered model. temporal: RequestID dedup at the boundary (`current_executions` create), at-least-once for activity execution (`docs/architecture/README.md:35`), effectively-once persistence via conditional writes. agent-framework: HTTP/MCP trigger is at-least-once, durable-entity per session is serialized, replay of orchestrator side effects is effectively-once. langgraph: state-write is effectively-once through `put_writes`/`ON CONFLICT`, node bodies are at-least-once. openhands: webhooks effectively-once within TTL, pending messages at-most-once, event storage effectively-once by path.

**Why it works.** Splitting the model along the natural ownership boundary (boundary node, executor, storage) lets each layer pick the cheapest primitive that gives the right guarantee.

**When to copy.** Always. The alternative — a single primitive (e.g., a global idempotency store) — tends to be slow and over-fits.

### P5 — Per-attempt write clearing for node retries

**Problem.** A retry on a partial-failure node should not leak partial writes into the next attempt.

**Where.** langgraph (`task.writes.clear()` at the start of each retry iteration — `libs/langgraph/langgraph/pregel/_retry.py:615`, `:738`).

**Why it works.** The buffer that feeds the storage layer is clean per attempt; the storage layer's `ON CONFLICT` then makes the writes idempotent.

**When to copy.** Whenever you have a node-execution model with retries and a persistence layer that uses idempotent-key writes.

**Evidence limit.** This pattern is only demonstrated clearly in langgraph. agent-framework replays via Durable Task and trusts the platform's history-replay semantic; temporal's `RetryPolicy` covers scheduling, not in-memory writes.

### P6 — Visibility invariant: drain in-flight writes before checkpoint commit

**Problem.** A checkpoint written before its delta writes would mislead a reader who walks the checkpoint chain.

**Where.** langgraph (`_delta_write_futs` and `_error_handler_write_futs` drained before `await checkpointer.aput(...)` — `libs/langgraph/langgraph/pregel/_loop.py:1507-1524`, `:1759-1778`).

**Why it works.** It is the only place the framework asserts a wait-for-condition-before-publish primitive. The code comment labels it the "visibility invariant".

**When to copy.** Whenever you have a write buffer plus a commit point. Without this, anti-dependencies (a checkpoint `B` that reads a delta of `A`) can become inconsistent.

**Evidence limit.** Currently visible only in langgraph; the other sources either do not need it (because they do not have a write buffer separate from the commit point) or rely on the underlying runtime (agent-framework on Durable Task, temporal on event-sourced history).

### P7 — Fail-closed dedup at the edge with provider-stable IDs

**Problem.** Provider webhooks retry with varying headers; the dedup key must be stable across redeliveries.

**Where.** openhands (GitLab `object_attributes.id`, Bitbucket `x_request_uuid`, Slack `client_msg_id`, Jira `x-hub-signature`, fallback SHA-256 of body — `enterprise/server/routes/integration/gitlab.py:96-113`, `enterprise/server/routes/integration/bitbucket.py:88-105`, `enterprise/server/routes/integration/slack.py:315-320`, `enterprise/server/routes/integration/jira.py:308-316`).

**Why it works.** Edge dedup is bounded (TTL 60-300 s); inside the TTL the same logical event is suppressed; outside the TTL, the provider has long since given up retrying.

**When to copy.** When ingesting events from an external system with its own retry semantics.

**When to delay.** When the provider has no stable per-event ID (e.g., generic POSTs). Then hash the body and accept that any two non-identical redeliveries pass through.

**Caveat.** All five openhands providers use this pattern **except GitHub**, where the route delegates straight to `receive_message` (`enterprise/server/routes/integration/github.py:91-93`) and ignores `X-GitHub-Delivery`. This is the most visible inconsistency.

### P8 — User-side dedup enabled, not enforced, on side effects

**Problem.** Activity / node side effects may run twice after a crash; the framework needs to give the user the right hook.

**Where.** temporal (`docs/architecture/README.md:35` documents the user contract); langgraph (no surface; cache via `CachePolicy` is the closest primitive — `libs/langgraph/langgraph/types.py:518-527`); agent-framework (no surface in-process; depends on user-defined tool idempotency).

**Why it works.** The platform exposes a policy (`CachePolicy` in langgraph, retry policy + activity options in temporal) that the user wires into a `(node, input) -> idempotency_key` mapping.

**When to copy.** Whenever side effects leave the framework's storage boundary.

## Key Differences

Some differences are quality differences; others are valid responses to product constraints.

**Where the idempotency boundary lives.** langgraph and temporal both move dedup into the persistence schema; agent-framework moves it into the runtime (Durable Entities serialize per-session operations); openhands moves it into per-subsystem helpers at the application layer. The library-vs-application split predicts the answer: libraries that control the storage layer use schema constraints; applications that delegate storage to S3/Redis use path keys and `SET NX`.

**How the idempotency key is supplied.** temporal makes `RequestID` a first-class input that *defaults* to a generated UUID when absent. agent-framework auto-generates `correlationId` on `RunRequest` and does not document that the caller must capture it on retry; this turns the framework-generated correlationId from an idempotency primitive into a response-correlation primitive. langgraph generates `task_id` per task but persists it in the storage row so it acts as an idempotency key automatically. openhands reads provider-supplied IDs off the wire; only when those are missing does it fall back to a hash.

**How durability is exposed.** temporal hides it (every write is conditional; durability is essentially "the strongest we can manage"). langgraph makes it the user-tunable knob (`sync` vs. `async` vs. `exit`). agent-framework delegates to Azure Functions / Durable Task durability modes. openhands has no knob — per-subsystem code chooses hardcoded modes (60-second edge dedup, delete-after-send pending messages, bulk upload then watermark export).

**Where crash recovery happens.** temporal: re-replays from history events (event sourcing). langgraph: re-reads the last persisted checkpoint and re-prepares tasks. agent-framework: trusts Durable Task's orchestrator replay. openhands: no universal recovery — each subsystem either replays (callbacks re-fire on every webhook), loses (pending message deleted), or relies on a deterministic write (event-id path).

**Whether the platform claims exactly-once.** None of the four does, publicly. But the *internal* invariants vary: temporal claims exactly-once by RequestID within each RPC and is honest that activity bodies are at-least-once; langgraph claims first-write-wins by composite key and exposes that as a conformance test; agent-framework claims effective-once within an entity and honest at-least-once for reliable streaming; openhands claims edge dedup-with-TTL-window but does not centralize the model.

**Cross-cluster consistency.** Only temporal surfaces a cross-cluster replication story with `VectorClock` + `TransitionCount` dedup (`service/history/ndc/replication_task.go:118-236`); the other three sources are single-region.

## Tradeoffs

This section captures the design tradeoffs behind each approach.

**Persistence-layer dedup vs. application-layer lock.** Persistence-layer dedup (temporal, langgraph) requires the storage to support conditional writes; it scales naturally with replication and survives process crashes. An application-layer lock (most openhands subsystems without Redis NX, in-process signals in agent-framework) is faster on the happy path but does not survive process restarts and is harder to reason about under concurrency. Best-fit: persistence layer for state-machine operations; application-layer when the storage layer cannot help (e.g., S3 without `IfNoneMatch`).

**Composite-key writes vs. monotonic sequence numbers.** langgraph uses composite keys; temporal uses monotonic task IDs under a range-id (`service/history/shard/task_key_generator.go:151-184`). The composite-key approach is robust under out-of-order arrivals; the monotonic approach enables efficient backlog GC (ack-level barrier at `service/matching/ack_manager.go:97-128`). Best-fit: composite for low-volume state writes; monotonic when the queue has high throughput and ordering matters.

**Durability modes exposed vs. hidden.** langgraph exposes a knob (`sync/async/exit`) that the user can tune to a workload. temporal hides the knob and always gives the strongest guarantee it can. Exposed knobs give the user control but require the user to understand durability; hidden knobs simplify the API but limit throughput. Best-fit: hidden when the only correct value is "strongest possible"; exposed when weaker modes are useful (e.g., high-throughput ephemeral graphs).

**Caller-supplied vs. framework-generated idempotency keys.** temporal uses both: caller-supplied when present, framework-generated with a `RequestID` request envelope otherwise. agent-framework generates `correlationId` automatically but the caller cannot use it for input dedup without capturing it across retries. Best-fit: support both; document explicitly which one (caller-supplied) is the deduplication primitive, not the response-correlation primitive.

**Edge dedup TTL vs. provider redelivery window.** openhands uses 60-300 seconds; this is correct when providers retry within that window. If a provider retries later (e.g., on a 5xx from the load balancer that does not surface to the client), the dedup window is exceeded and a duplicate event runs. Best-fit: TTL must be ≥ observed redelivery window of every provider; longer TTLs waste Redis memory.

**Background task offload vs. inline processing.** openhands's Bitbucket DC and Slack routes offload to FastAPI `BackgroundTasks` (`enterprise/server/routes/integration/bitbucket_dc.py:702`) so the response returns 200 quickly and avoids provider timeouts. The cost is that failures inside the background task are unawaited and only logged. Inline processing (Bitbucket Cloud — `enterprise/server/routes/integration/bitbucket.py:115`) gives better error visibility at the cost of webhook timeout risk. Best-fit: background when the provider has a tight webhook timeout; inline when the work is short enough to complete within it.

**Outbox vs. dual-write.** agent-framework's `DurableAgentContext.ScheduleNewOrchestration` is committed atomically with the agent state update (`dotnet/src/Microsoft.Agents.AI.DurableTask/DurableAgentContext.cs:80-87`) — this is the transactional-outbox pattern in miniature. langgraph has no equivalent transactional bundling of node execution and checkpoint write; a worker crash between node return and `commit()` causes a re-run (`libs/langgraph/langgraph/pregel/_runner.py:574-613`). Best-fit: outbox when state and side-effect ordering matter; accept the re-run when they do not.

**In-memory dedup table vs. storage-backed dedup.** temporal's Update Registry (`service/history/workflow/update/registry.go:226-235`) and signal-requested set (`service/history/workflow/mutable_state_impl.go:2499-2518`) live in memory and are rebuilt from history events on loss. This is fast and keeps the hot path simple; the cost is that a window exists (admitted-but-unsent Updates, in particular) where state can be lost and the client must retry with the same ID (`docs/architecture/workflow-update.md:70-83`). Best-fit: in-memory when rebuild-from-history is cheap and a retry contract exists; storage-backed when the rebuild path is expensive or unavailable.

## Decision Guide

Use the heuristics below when implementing delivery guarantees in a new agent system.

**Choose temporal's pattern when:**
- The system exposes a public API that may be retried by callers or networks.
- The state machine is durable event-sourced (append-only history).
- Users will pay the cost of being told their activity body must be idempotent.
- The downside of a duplicate side effect is severe (financial, regulatory).
- Cross-cluster consistency is required.

**Choose langgraph's pattern when:**
- The state layer is one of a small set of relational stores (Postgres, SQLite).
- You want callers to choose between throughput and durability (`sync/async/exit`).
- Idempotent keying is naturally per-task (`task_id` is cheap to expose).
- You can rely on the conformance suite to pin invariants across storage back-ends.

**Choose openhands' pattern when:**
- You are gluing together external systems (webhooks, third-party APIs) and do not control their retry semantics.
- The dedup window can be bounded by the external system's redelivery interval.
- You have Redis available and the dedup TTL is short.
- You accept that edge dedup is the *outermost* defense and downstream subsystems may still re-run.

**Choose agent-framework's pattern when:**
- The system already has a Durable Task / Durable Entities substrate.
- "Per-session serialization" is sufficient (you do not need cross-session exactly-once).
- You can offload replay-side-effect reasoning to `is_replaying` checks.

**Avoid the "framework auto-generates the ID" pattern when:**
- The ID is meant to be an idempotency key but is exposed to the caller *after* the request. The caller must capture it for retries, which most callers will not.

**Avoid the "in-memory dedup table only" pattern when:**
- The process can crash. Either back the table with storage (langgraph, temporal approach) or rebuild it from history (temporal approach for Updates).

## Practical Tips

When implementing delivery guarantees in an agent framework, prefer the following patterns.

- Treat the idempotency key as a **first-class input** to every public RPC, not a response-only field. Auto-generate when absent but make caller-supply the documented dedup mechanism.
- Put the dedup boundary in the **persistence layer** when you control the storage; use composite primary keys or conditional writes (`INSERT ... IF NOT EXISTS`, `ON CONFLICT ... DO NOTHING`).
- Clear `task.writes` at the **start of each retry attempt**, then let storage-level dedup handle the case of multiple attempts landing. This is much simpler than per-node idempotency code.
- Expose **durability as a knob only when** weaker modes are useful. A single "strongest possible" mode is simpler to explain and easier to operate.
- At webhook edges, **HMAC-verify before dedup** — never the other way around. A forged payload should never poison a dedup key.
- Use a **composite-key store for control channels** with `ON CONFLICT ... DO UPDATE` (so the latest control state wins) and a separate insert-only path for data writes (so attempts are not lost).
- **Document explicitly** which subsystems are at-least-once vs. effectively-once. Without this, contributors assume every subsystem is durable.
- **Drain delta-write futures** at the checkpoint-commit point to enforce a visibility invariant: a published checkpoint must reflect all in-flight writes that produced it.
- Use **deterministic event-id paths** for object stores that lack conditional writes. This gives last-writer-wins idempotency for free; do not store events at random paths and rely on read-side dedup.
- **Back in-memory dedup tables with a rebuild-from-history path** so a registry clear is recoverable rather than lossy.
- **Reshape the at-least-once nature of agent execution** in derivative documentation. The danger is a "Beginner's Guide" that promises exactly-once.

## Anti-Patterns / Caution Signs

The four sources demonstrate recurring hazards.

- **Auto-generated idempotency key tied to the response.** agent-framework's `correlationId` is generated when the caller omits; HTTP-trigger retries get a fresh GUID and the run is duplicated. Symptom: history grows unbounded across retries.
- **In-memory dedup table without rebuild path.** temporal's Update Registry mitigates this by rebuilding from history events (`service/history/workflow/update/registry.go:168-224`) — but admitted-but-unsent Updates are still lost on registry clear (`docs/architecture/workflow-update.md:70-83`). Symptom: platform occasionally returns `WorkflowUpdateAbortedErr` and asks the client to retry.
- **Silent loss under crash.** openhands pending messages delete themselves whether or not the inner POST succeeded (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1997-2002`). Symptom: user submits a message; intermittent network failure; message is gone.
- **Background task without error surfacing.** openhands `_run_callbacks_in_bg_and_close` (`openhands/app_server/event_callback/webhook_router.py:513-573`) returns 200 and lets exceptions inside `execute_callbacks` propagate to unawaited background tasks. Symptom: provider marks delivery successful, downstream callbacks silently fail.
- **No conditional writes on object store.** openhands event store writes are unconditional FS / S3 / GCS operations (`openhands/app_server/event/aws_event_service.py:54-62`, `openhands/app_server/event/google_cloud_event_service.py:57-62`). Symptom: a partial write can leave a corrupted JSON file that downstream readers silently drop.
- **Mixed add_agent registration semantics.** agent-framework's `AgentFunctionApp.add_agent` silently skips a duplicate (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:572-576`); `DurableAIAgentWorker.add_agent` raises (`python/packages/durabletask/agent_framework_durabletask/_worker.py:113-115`). Symptom: a contributor writes code expecting one behavior and gets the other.
- **Per-attempt write-buffer leakage.** langgraph explicitly clears `task.writes` before each retry attempt (`libs/langgraph/langgraph/pregel/_retry.py:600-682`); a framework that copied langgraph's retry loop without the clear would allow partial writes from a failed attempt to feed the next attempt.
- **TTL shorter than provider redelivery window.** openhands uses 60-second dedup TTLs; if a provider retries after 90 seconds, the dedup key has expired. Symptom: occasional duplicate side effects that look unsystematic.
- **Persisting only the success path.** openhands `event_callback_result` records results but does not consult them on retry (`openhands/app_server/event_callback/sql_event_callback_service.py:208-250`). Symptom: every webhook redelivery re-runs every active callback.
- **Default-fall-open on dedup outage.** openhands's `_get_cached_value` returns `None` on Redis outage (`enterprise/server/services/automation_event_service.py:486-510`) and `try_acquire_redis_lock` falls open silently (`openhands/app_server/utils/redis_lock.py:30-33`). Symptom: dedup silently disappears during incidents.
- **Inconsistent dedup keys across siblings.** openhands uses `SET NX EX` for GitLab/Bitbucket/Slack and `SETEX` for Jira (`enterprise/server/routes/integration/jira.py:316`), with different semantics. Symptom: contributors copy the wrong pattern.

## Notable Absences

What none of the four sources does — but probably should.

- **A general-purpose idempotency-key store for user code.** langgraph relies on `CachePolicy` keyed on `(args, kwargs)`; agent-framework and openhands have no analogous primitive; temporal relies on the user to handle it. A `runtime.idempotency_key(node=..., input=...)` that participated in storage dedup would make side-effecting nodes effectively-once.
- **A central "delivery guarantees" doc.** Every source documents the contract piecemeal — temporal per-subsystem (`workflow-update.md`, `nexus.md`, `README.md`), langgraph in code comments + types, agent-framework in sample READMEs, openhands in docstrings. A top-level `docs/delivery.md` would let callers stop guessing.
- **Conditional writes on object stores.** openhands uses S3 `put_object` without `IfNoneMatch`; an `IfNoneMatch: *` PUT would close the partial-write window.
- **Worker-to-worker hand-off across processes for langgraph.** The runner (`libs/langgraph/langgraph/pregel/_runner.py`) assumes a single in-process event loop; distributed execution would need a `SELECT ... FOR UPDATE SKIP LOCKED`-style claim.
- **Bounded retry budgets visible to operators.** openhands's webhook dedup has a TTL (visible), but its background callbacks have *no* retry cap. temporal's outbound queue retries are bounded only by workflow retention, which is opaque to operators (`docs/architecture/nexus.md:348-353`).
- **Notification when dedup silently fails.** openhands's `_get_cached_value` returns `None` on Redis outage (`enterprise/server/services/automation_event_service.py:486-510`); the loss-of-dedup condition is not surfaced.
- **Per-attempt write-buffer clearing generalized across the industry.** Only langgraph demonstrates this; the rest rely on either platform-level replay (agent-framework, temporal) or no safety at all.
- **A cross-source standard for at-least-once replay.** Each platform invents its own term (correlationId, RequestID, task_id, client_msg_id) for what is conceptually the same primitive.

## Per-Source Notes

### agent-framework

Idempotency is **delegated** to Durable Task / Durable Entities / Azure Functions. The framework's contribution: a `correlationId` per `RunRequest` (`dotnet/src/Microsoft.Agents.AI.DurableTask/RunRequest.cs:37`), per-session `TaskEntity<DurableAgentState>` which serializes concurrent calls (`docs/features/durable-agents/README.md:28`), a poll-for-response loop that matches by correlation ID (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentRunHandle.cs:65-67`), and a deterministic-replay-aware orchestrator that gates `set_custom_status` calls on `is_replaying` (`python/.../orchestrator.py:755-757`). ### langgraph

Idempotency is **storage-enforced** via composite primary key `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` and Postgres / SQLite `ON CONFLICT` (DO NOTHING for regular, DO UPDATE for control channels — `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-159`, `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:445-482`). ### openhands

Each subsystem solved its own redelivery story. **Provider webhooks** (GitLab, Bitbucket Cloud, Bitbucket DC, Jira, Slack) use Redis `SET NX EX` with provider-stable IDs and a SHA-256 fallback — all five implementations verified in `enterprise/server/routes/integration/*.py`. **Event storage** (FS / S3 / GCS) uses deterministic `<conversation_id>/<event_id.hex>.json` paths (`openhands/app_server/event/event_service_base.py:190-197`). **Conversation export** uses a Redis distributed lock with TTL refresh (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:2354-2419`). **Pending messages** are at-most-once with bulk delete after attempt (`openhands/app_server/pending_messages/pending_message_models.py:12-18`). **Event callbacks** record results in `event_callback_result` but do not consult them on retry, so a redelivered event re-runs every active callback processor. **Webhook enrollment** is idempotent-by-construction (existence checks before write). **GitHub** has no dedup layer; **`Automation` forward** is fire-and-forget; the doc model is per-subsystem and reconstructed from code comments.

### temporal

The most disciplined layered model. **Per-RPC RequestID dedup** at the history layer collapses duplicate signal / update / start / pause / worker-deployment / nexus-completion calls (`service/history/api/startworkflow/api.go:319-368`, `service/history/api/updateworkflow/api.go:161-164`, `service/history/api/signalworkflow/api.go:45-89`, `tests/pause_workflow_execution_test.go:1281-1323`, `components/nexusoperations/completion.go:186-203`). **Conditional writes** are the persistence-layer primitive: Cassandra `INSERT ... ## Open Questions

- What happens to openhands events when the agent_server rejects a partial file write? Currently `Event.model_validate_json` silently returns `None` (`openhands/app_server/event/filesystem_event_service.py:25-31`), so the event disappears from history. Is silent drop the intended behavior, or should the storage surface a corruption error?
- Does agent-framework's in-process agent loop ever guarantee anything? The corpus showed correlation IDs only on the Durable Task path; the in-process `AIAgent.RunCoreAsync` has no documented dedup or transactional boundary.
- What does langgraph's `durability="exit"` look like under crash? The mode "defers all write persistence until graph completion"; if a worker crashes before completion, the entire persisted history is lost, not just the in-flight writes.
- How often do openhands provider webhooks see redeliveries beyond the 60-second TTL window? The current TTL was chosen empirically; an observability gap on "we deduped" vs. "we let it through" makes this hard to retune.
- Should temporal's Update `stateAdmitted` window persist sooner? The doc acknowledges the gap (`docs/architecture/workflow-update.md:75-83`) but the implementation is described as a TODO.
- Is langgraph's `task_id` guaranteed unique across runs? The composite-key dedup is local to the checkpoint; if a tool reuses `task_id` across runs, the rows are independent, not coalesced.
- Does agent-framework's cancellation gap (`throw new NotSupportedException("Cancellation is not supported for durable agents.")` — `dotnet/src/Microsoft.Agents.AI.DurableTask/DurableAIAgent.cs:97`) affect any planner that includes durable agents in a cancellation-aware workflow?

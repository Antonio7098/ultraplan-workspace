## Executive Summary

Across the five sources, "what triggers execution" splits cleanly into three architectural tribes. **Temporal** (9/10) and **agent-framework** (9/10) treat scheduling as a first-class durability problem — every timed trigger is a persisted task with replay-safe semantics, idempotency keys, and observable status. **crewai** (6/10) and **openhands** (6/10) are libraries/applications that deliberately delegate scheduling to an external host: the user gets `kickoff()` and an HTTP router, but no scheduler, no cron surface, and durability is either opt-in (crewai checkpointing) or survives only because state is in DB rows (openhands `AppConversationStartTask`). **LangGraph** (5/10) sits in between: the open-source engine has mature channel-trigger and retry semantics, but cron, delayed-run, and webhook delivery are owned by the closed-source `langgraph_api.server` and only surface here as typed HTTP contracts.

What converges: every source recognizes that background work and interactive work must share an execution engine (one `kickoff`, one `Pregel`, one `WorkflowBackoffTimerTask`, one `AgentEntity.run`), that idempotency deserves an explicit key, and that trigger observation is non-negotiable for operators. What diverges is *who* owns the scheduler infrastructure, *where* triggers are persisted, and *how much* explicit idempotency the caller must supply. Library-style projects defer to the operator; platform-style projects impose a contract.

## Core Thesis

A trigger model in an agent system is a contract about *who owns the loop*. The strongest scheduling/tigger designs (temporal, agent-framework) keep **triggers and trigger-derived state in the same persistence layer as execution state**, so durability, idempotency, and observation cannot drift apart. Mid-tier designs (crewai, openhands) keep execution state in the application and persist triggers as ordinary rows; this works for HTTP/queue workloads but loses the implicit replay-safety that platform-style designs provide. The bottom tier (langgraph OSS side) reveals what the engine can and cannot do without an owned scheduler — and the answer is that channel triggers and per-task retries are mature, but cron/queue/webhook semantics must be added by a separate server.

## Approach Models

The five sources group into four approach models for this dimension. The vocabulary below is what each source actually exposes — not abstractions imposed from outside.

### 1. Durable-entity trigger host (agent-framework)

agent-framework has no trigger surface on `AIAgent` (`agent-framework:46`); every trigger is a Durable Task primitive — HTTP, MCP tool, orchestration, activity, entity — routed through a per-session entity keyed `dafx-{name}` (`python/packages/durabletask/agent_framework_durabletask/_models.py:219-243`). The same `AgentEntity.run` executes regardless of which trigger arrived. Background vs. foreground differs only in caller contract: foreground (`wait_for_response=True`) blocks and polls by `correlation_id`, fire-and-forget (`wait_for_response=False`) returns HTTP 202 and lets the durable state record the eventual outcome. Orchestrations can pause indefinitely via `ctx.wait_for_external_event(...)` (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:863-900`).

The .NET hosting layer registers triggers through `IFunctionMetadataTransformer` (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/DurableAgentFunctionMetadataTransformer.cs:37-68`), since isolated workers cannot use C# attribute decorators at runtime. TTL is implemented as a self-signaled deletion so the host scheduler owns it (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211-215`).

### 2. Persisted-task scheduler (temporal)

Temporal has a strictly layered design. Triggers fall into five surfaces: external RPC (`service/frontend/workflow_handler.go:538`), per-task `WorkflowBackoffTimerTask` rows (`service/history/tasks/workflow_delay_timer.go:14`), user-facing Schedules (implemented twice: V1 legacy workflow at `service/worker/scheduler/workflow.go:230`; V2 CHASM component at `chasm/lib/scheduler/generator_tasks.go:50`), internal timer queue (`service/history/timer_queue_factory.go:33-79`), and background scanners launched at boot (`service/worker/scanner/scanner.go:117`). All five converge on persisted tasks indexed by `(CategoryID, VisibilityTimestamp, TaskID)` (`service/history/queues/queue_scheduled.go:54-87`).

Idempotency is layered too: `RequestId` on start, `ConflictToken` on update (`service/worker/scheduler/workflow.go:89`), catch-up window to drop missed actions safely (`service/worker/scheduler/workflow.go:706-716`), and explicit duplicate-execute drops in the CHASM invoker (`chasm/lib/scheduler/invoker_tasks.go:129`).

### 3. Engine + external scheduler (langgraph)

The OSS engine (`Pregel`) has only one trigger: a Python call to `invoke`/`stream`/`update_state` (`libs/langgraph/langgraph/pregel/main.py:449`). Internally, channel-triggered node re-firing is solved cleanly by `PregelNode.triggers` (`libs/langgraph/langgraph/pregel/_read.py:107-109`) and `PUSH_TRIGGER` for fan-out (`libs/langgraph/langgraph/pregel/_algo.py:516`). Retry is per-task with backoff/jitter (`libs/langgraph/langgraph/pregel/_retry.py:573, 685`), and timeouts split into step-level and per-node `TimeoutPolicy` (`libs/langgraph/langgraph/types.py:444`).

External triggers — cron, `after_seconds`, batch, webhook — live in `langgraph_api.server` (`libs/sdk-py/langgraph_sdk/_async/client.py:117`). The SDK exposes typed `CronClient` and `RunsClient` surfaces (`libs/sdk-py/langgraph_sdk/_async/cron.py:30`, `libs/sdk-py/langgraph_sdk/_async/runs.py:441`) that forward scheduling parameters into the server. `Durability = Literal["sync", "async", "exit"]` (`libs/sdk-py/langgraph_sdk/schema.py:104-108`) is forwarded but enforced server-side.

### 4. Library with opt-in durability (crewai)

crewai is deliberately scheduler-less. Every entry point — `Crew.kickoff` (`lib/crewai/src/crewai/crew.py:980`), `LiteAgent.kickoff` (`lib/crewai/src/crewai/lite_agent.py:477`), `Flow.kickoff` (`lib/crewai/src/crewai/flow/runtime/__init__.py:2110`), `Agent.kickoff_with_a2a` (`lib/crewai/src/crewai/a2a/wrapper.py:175`) — runs the same execution body. Trigger vocabulary splits into: direct kickoff calls, A2A server endpoints, polling loops (`lib/crewai/src/crewai/a2a/updates/polling/handler.py:45-200`), conversational turn (`lib/crewai/src/crewai/experimental/conversational_mixin.py:259`), OS-signal-to-event conversion (`lib/crewai/src/crewai/telemetry/telemetry.py:176-234`), and reactive checkpoint listener (`lib/crewai/src/crewai/state/checkpoint_listener.py:43`).

Durability is opt-in via `CheckpointConfig` with `JsonProvider | SqliteProvider` discriminated union (`lib/crewai/src/crewai/state/checkpoint_config.py:159-233`); without it, a process crash loses in-flight runs.

### 5. Application with durable middleware state (openhands)

openhands exposes a clean trigger taxonomy through `ConversationTrigger` (`openhands/app_server/app_conversation/app_conversation_models.py:37-48`) covering GUI, resolvers, suggested tasks, Jira/Slack/Linear/Bitbucket, MICROAGENT_MANAGEMENT, REMOTE_API_KEY, and AUTOMATION — but it has **no scheduler, no queue worker, no cron**. HTTP requests (router) and webhooks (`openhands/app_server/event_callback/webhook_router.py:333`) are the only external triggers. Long-running starts persist as `AppConversationStartTask` rows (`openhands/app_server/app_conversation/app_conversation_models.py:281-309`) updated after each state transition (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`). Background work uses `asyncio.create_task` for continuations (`openhands/app_server/app_conversation/app_conversation_router.py:405`) and `refresh_lock_periodically` for export jobs (`openhands/app_server/utils/redis_lock.py:36-53`). Restart does **not** resume in-flight start tasks; only new requests do.

## Pattern Catalog

The patterns below appear in two or more sources or are unusually clearly demonstrated in one. Each entry names the problem, the participating sources, why it works, when to copy, when it is overkill, and supporting evidence.

### Pattern 1: Persisted-trigger table keyed by time

**Problem**: A scheduled or delayed trigger must survive process restart.

**Sources**: temporal, agent-framework.

**Mechanism**: temporal keeps every scheduled task as a row in the `tasks` table with `(CategoryID, VisibilityTimestamp, TaskID)` (`service/history/queues/queue_scheduled.go:54-87`); agent-framework keeps TTL as a self-signaled durable entity call (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211-215`) and orchestrations as `schedule_new_orchestration` instances persisted by the host (`python/packages/durabletask/agent_framework_durabletask/_workflows/client.py:81`).

**Why it works**: A persisted `(category, time, id)` tuple is process- and shard-agnostic. The scheduler can index by time, scan by category, and reconstruct trigger provenance across any infrastructure churn. Shard ownership can change without losing triggers because the table is the source of truth.

**When to copy**: When the system has any timed work that must outlive the process that created it.

**When it is overkill/risky**: For ephemeral conversational systems with sub-second lifetimes. The schema migration burden, category router, and pagination logic are heavy if you just want `setTimeout`.

**Evidence**: temporal `service/history/tasks/category.go:18-87`, `service/history/queues/queue_scheduled.go:43-301`; agent-framework `python/packages/durabletask/agent_framework_durabletask/_models.py:219-243`, `dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211-215`.

### Pattern 2: Correlation-ID pickup for in-flight responses

**Problem**: A caller that initiated a trigger asynchronously needs to know which response belongs to which call, especially on retry.

**Sources**: agent-framework, openhands.

**Mechanism**: agent-framework generates `correlation_id` per HTTP/MCP request (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:721`) and the entity stores it on every `DurableAgentStateEntry` (`python/packages/durabletask/agent_framework_durabletask/_models.py:198-200`). The HTTP poll loop matches responses by exact id, so a stale completion cannot be misattributed (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:1103`). openhands has an analogous mechanism with `AppConversationStartTask.id` persisted per submission and a per-row status machine (`openhands/app_server/app_conversation/app_conversation_models.py:281-309`).

**Why it works**: Separation of "request" identity from "execution" identity. The trigger layer can replay/duplicate without confusing which response goes to which caller; the executor layer can run more than one in-flight request per session because each row carries its own id.

**When to copy**: Whenever the system accepts external triggers from a fan-in that may produce duplicate submissions (HTTP retries, webhook replays, queue redelivery).

**When it is overkill**: For a single-process interactive tool that never has in-flight requests.

**Evidence**: agent-framework `python/packages/durabletask/agent_framework_durabletask/_entities.py:151`, `python/packages/durabletask/agent_framework_durabletask/_models.py:97-200`; openhands `openhands/app_server/app_conversation/app_conversation_models.py:262-309`.

### Pattern 3: Idle-trigger cancel-cleanup via TTL self-signal

**Problem**: A durable session can sit idle for days; how is it cleaned up without an external sweep?

**Sources**: agent-framework.

**Mechanism**: The first entity interaction schedules a future `SignalEntity` to itself with `SignalTime = scheduledTime` (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211-215`). Subsequent interactions only bump `ExpirationTimeUtc`. The deletion check decides whether to delete or reschedule (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:120-195`).

**Why it works**: Uses the same durable signal infrastructure as user work, so cleanup inherits retry, idempotency, and observability for free.

**When to copy**: When retention windows are essential and you have a durable signal infrastructure to lean on.

**When it is overkill**: If you do not have a durable signal infrastructure already, rolling one just to do TTL is expensive.

**Evidence**: `dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:120-215`, `docs/features/durable-agents/durable-agents-ttl.md:30-33`.

### Pattern 4: Channel-triggered node re-firing

**Problem**: A node in a dataflow graph must re-run when its inputs change.

**Sources**: langgraph (central to its engine); analogous in crewai via event-bus.

**Mechanism**: langgraph attaches a `triggers: list[str]` of channel names to each `PregelNode` (`libs/langgraph/langgraph/pregel/_read.py:107-109`); the runner's plan step selects nodes whose `triggers` overlap with channels written in the previous step (`libs/langgraph/langgraph/pregel/_algo.py:266, 477`). PUSH triggers are reserved (`libs/langgraph/langgraph/pregel/_algo.py:516`) so control flow is segregated from dataflow. crewai's event bus (`lib/crewai/src/crewai/events/event_bus.py:94`) plays a similar role inside a `kickoff`: a `CrewKickoffCompletedEvent` fans out to handlers by type rather than node-id, and `Depends(...)` orders them (`lib/crewai/src/crewai/events/depends.py:43-105`).

**Why it works**: A single primitive (channel-write / event-emit) drives both dataflow and control flow. The runner does not need a separate "should I run?" check.

**When to copy**: Whenever execution is dataflow-shaped (DAG, state machine, channel algebra).

**When it is overkill**: For simple sequential code without data dependencies.

**Evidence**: langgraph `libs/langgraph/langgraph/pregel/_read.py:97-109`, `libs/langgraph/langgraph/pregel/_algo.py:266-516`; crewai `lib/crewai/src/crewai/events/event_bus.py:570-730`, `lib/crewai/src/crewai/events/depends.py:43-105`.

### Pattern 5: Replay-safe orchestrator with skip-on-replay side effects

**Problem**: When a durable execution resumes from a checkpoint, side effects (live-status publish, external event raise) must not re-fire.

**Sources**: agent-framework, temporal.

**Mechanism**: agent-framework gates any "real" side effect behind `ctx.is_replaying` (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:757-758`); temporal's V1 scheduler uses `SideEffect` + `MutableSideEffect` for non-deterministic spec parsing (`service/worker/scheduler/workflow.go:524, 596, 1300`).

**Why it works**: A replayed step looks identical to a fresh step at the API level. The skip-on-replay gate is the only place where "do I actually want this side effect now?" is answered.

**When to copy**: Every durable orchestrator that does I/O.

**When it is overkill**: For non-durable runtimes; the cost is more gates to maintain.

**Evidence**: agent-framework `python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:733-870`; temporal `service/worker/scheduler/workflow.go:514-642, 1300`.

### Pattern 6: Catch-up window to bound replay

**Problem**: When a host catches up after a long downtime, replaying every missed cron tick would create a runaway fan-out.

**Sources**: temporal.

**Mechanism**: `getCatchupWindow` enforces min/max bounds (`service/worker/scheduler/workflow.go:1305-1314`); actions outside the window are explicitly dropped and counted (`metrics.ScheduleMissedCatchupWindow`, `:706-716`).

**Why it works**: The user-visible contract becomes "at-most-catchup-window-late" rather than "every-tick-ever". Dropped ticks are observable (metric) and bounded (max).

**When to copy**: Whenever a scheduled trigger can be missed (any durable cron-style scheduler).

**When it is overkill**: For lightweight timers that are inherently idempotent and cheap to replay.

**Evidence**: `service/worker/scheduler/workflow.go:706-716, 1305-1314`.

### Pattern 7: `Idempotency-Key` / `RequestId`-style trigger token

**Problem**: A retried HTTP request should not create two workflows/runs.

**Sources**: temporal (strongest), agent-framework (partial via `correlation_id`), openhands (only on tags), crewai (partial via `pending_feedback` insert-or-replace).

**Mechanism**: temporal's `StartWorkflowExecution` requires `RequestId` and respects `WorkflowIdReusePolicy` (`service/history/api/startworkflow/api.go:86`); agent-framework dedups via `correlation_id` post-hoc (`python/packages/durabletask/agent_framework_durabletask/_entities.py:151`); crewai's `SQLiteFlowPersistence.save_pending_feedback` uses `INSERT OR REPLACE` on `flow_uuid` (`flow/persistence/sqlite.py:229-244`).

**Why it works**: A request token is a cheap single field that, checked before any side effect, removes the duplicate-trigger burden.

**When to copy**: Any system that accepts triggers from unreliable transports.

**When it is overkill**: For single-process in-process invocations with no transport-layer retries.

**Evidence**: temporal `service/history/api/startworkflow/api.go:86`, `service/history/history_engine.go:410`; agent-framework `python/packages/durabletask/agent_framework_durabletask/_models.py:97-200`; crewai `lib/crewai/src/crewai/flow/persistence/sqlite.py:96-244`.

### Pattern 8: Async-generator start-task state machine

**Problem**: A long-running start operation needs visible state per transition.

**Sources**: openhands (defining use); agent-framework has analogous persistence after every superstep.

**Mechanism**: `LiveStatusAppConversationService._start_app_conversation` is an async generator that yields after each meaningful state change and persists via `save_app_conversation_start_task` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`). The router consumes one yield synchronously and the rest in a background task (`openhands/app_server/app_conversation/app_conversation_router.py:380-405`). Each transition persists to DB, so polling clients see the state machine (`openhands/app_server/app_conversation/app_conversation_models.py:262-271`).

**Why it works**: Yielding = checkpoint. Persisting after each yield turns an async coroutine into a queryable state machine.

**When to copy**: When a long-running operation needs operator visibility and restart-survival but the host offers no native state-machine primitive.

**When it is overkill**: For fast (< 1s) operations or for systems with a native durable state-machine.

**Evidence**: openhands `openhands/app_server/app_conversation/live_status_app_conversation_service.py:309-538`, `openhands/app_server/app_conversation/app_conversation_router.py:363-410`; agent-framework `python/packages/durabletask/agent_framework_durabletask/_entities.py:172-173, _workflows/orchestrator.py:733-870`.

### Pattern 9: `ContinueAsNew`-based forever-loop workflows

**Problem**: A long-running service workflow (scanner, scheduler, batcher) accumulates unbounded history.

**Sources**: temporal (defining use); analogous in agent-framework via Durable Task's `ContinueAsNew`.

**Mechanism**: temporal's V1 scheduler uses `IterationsBeforeContinueAsNew` plus `forceCAN` signal (`service/worker/scheduler/workflow.go:282-388`); on CAN, `State.LastProcessedTime` is restored from history. Scanner workflows use `infiniteDuration = 20 * 365 * 24h` (`service/worker/scanner/workflow.go:19`).

**Why it works**: History size grows with work done; CAN resets history while preserving state, so the trigger loop runs forever inside a bounded history.

**When to copy**: Any always-on system workflow.

**When it is overkill**: For workflows that complete in seconds.

**Evidence**: temporal `service/worker/scheduler/workflow.go:282-388`, `service/worker/scanner/workflow.go:19-62`; agent-framework uses ContinueAsNew at the Durable Task layer (host implementation).

### Pattern 10: `PUSH_TRIGGER` / special tokens to segregate control flow from data

**Problem**: User-defined dataflow risks colliding with internal control flow.

**Sources**: langgraph; analogous surface in temporal via `ConflictToken`.

**Mechanism**: langgraph reserves `PUSH` as a `TASKS` channel (`libs/langgraph/langgraph/pregel/main.py:803-808`) and uses `PUSH_TRIGGER = (PUSH,)` (`libs/langgraph/langgraph/pregel/_algo.py:516`); the validator raises on user-named collisions (`libs/langgraph/langgraph/pregel/main.py:803-806`). temporal's `ConflictToken` (`service/worker/scheduler/workflow.go:89`) plays a similar role in schedule updates.

**Why it works**: Naming a token `__pregel_*` or using a discriminator field makes accidental collision impossible and segregation explicit.

**When to copy**: Any user-extensible trigger system.

**When it is overkill**: For closed systems with no user extensibility.

**Evidence**: langgraph `libs/langgraph/langgraph/pregel/main.py:803-808`, `libs/langgraph/langgraph/pregel/_algo.py:516`; temporal `service/worker/scheduler/workflow.go:89, 1283`.

### Pattern 11: Trigger captured as a first-class enum

**Problem**: Analytics, conditional logic, and integration authorization need to know *how* a trigger arrived.

**Sources**: openhands (canonical).

**Mechanism**: `ConversationTrigger` enum with 11 values (`openhands/app_server/app_conversation/app_conversation_models.py:37-48`), persisted as a string column on `app_conversation_info` (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:84`), and guarded against override on webhooks (`openhands/app_server/event_callback/webhook_router.py:212-213`).

**Why it works**: A typed enum prevents string typos; the DB column survives restart; "don't override existing" makes webhook replays safe.

**When to copy**: When multiple integrations can create the same logical entity.

**When it is overkill**: For single-integration systems.

**Evidence**: `openhands/app_server/app_conversation/app_conversation_models.py:37-48, 220`, `openhands/app_server/app_conversation/sql_app_conversation_info_service.py:84, 560`, `openhands/app_server/event_callback/webhook_router.py:191-232`.

### Pattern 12: Pending-message queue bounded by per-conversation cap

**Problem**: Messages can arrive before the conversation is ready; how do you buffer them safely?

**Sources**: openhands (defining); agent-framework has implicit buffer via durable entity state.

**Mechanism**: `POST /api/v1/conversations/{id}/pending-messages` writes to `pending_messages` table; hard cap of 10 per conversation returns HTTP 429 (`openhands/app_server/pending_messages/pending_message_router.py:85-91`); once READY, `_process_pending_messages` re-points `task-id` → real conversation id, replays sequentially, then deletes (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1925-2006`).

**Why it works**: The cap bounds blast radius; the delete-after-replay prevents unbounded growth; the conversion handles the sandboxes-still-starting race.

**When to copy**: When work-in-progress has a finite prep window before it can accept inputs.

**When it is overkill**: For low-latency systems where prep is sub-second.

**Evidence**: openhands `openhands/app_server/pending_messages/pending_message_service.py:27-72`, `openhands/app_server/pending_messages/pending_message_router.py:85-91`, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:525-533, 1925-2006`.

## Key Differences

### Where the trigger vocabulary lives

- **temporal/agent-framework** own the trigger vocabulary in the platform (entities, tasks, schedulers).
- **crewai** scatters its trigger vocabulary across seven modules (`crew.py`, `flow/runtime/__init__.py`, `a2a/wrapper.py`, `a2a/utils/task.py`, `state/checkpoint_listener.py`, `telemetry/telemetry.py`, `conversational_mixin.py`).
- **openhands** consolidates triggers into routers + webhooks + the `ConversationTrigger` enum.
- **langgraph** consolidates triggers in the SDK contract and pushes implementation behind `langgraph_api.server`.

### How idempotency is named

- **temporal**: `RequestId` + `WorkflowIdReusePolicy` + `ConflictToken` + catch-up window (multiple explicit mechanisms at different layers).
- **agent-framework**: `correlation_id` post-hoc only; duplicate HTTP triggers both append a state entry.
- **crewai**: `flow_uuid` insert-or-replace for pending feedback; checkpoint rows are append-only.
- **openhands**: tag-merge is shallow idempotent; pending-message delete-after-replay; no event-id dedup on webhooks.
- **langgraph**: `multitask_strategy` (`reject`/`interrupt`/`rollback`/`enqueue`) on same thread; cache-lookup short-circuit in retry; no `Idempotency-Key` header.

### How restart-survival is achieved

- **temporal** and **agent-framework**: triggers are persisted in the same store as execution state. Restart is structurally transparent — `scheduledQueue.notify(time.Time{})` re-arms (`service/history/queues/queue_scheduled.go:138`); entity state is persisted before return (`python/packages/durabletask/agent_framework_durabletask/_entities.py:172-173`).
- **crewai**: restart-survival requires explicit `CheckpointConfig` or `@persist`. Without it, in-flight work is lost.
- **openhands**: state-machine rows survive restart, but `asyncio.create_task` continuations do not. A start task in mid-`WAITING_FOR_SANDBOX` stays stuck forever; only a fresh request unsticks it.
- **langgraph**: engine-side restart-survival is via `Checkpointer` (`libs/langgraph/langgraph/pregel/main.py:732`); scheduler-side restart-survival is server-only and unobservable in OSS.

### How foreground and background diverge

- **temporal**: share the same primitive task layer (timer/transfer/outbound); diverge in driver (interactive poll vs. durable scheduler).
- **agent-framework**: share `AgentEntity.run`; diverge in caller contract (foreground polls by correlation_id, fire-and-forget returns 202).
- **langgraph**: share `PregelLoop` + `PregelRunner.tick`; diverge in caller contract (`runs.wait` blocks, `runs.create` returns Run row).
- **crewai**: share `kickoff`; diverge only via `asyncio.to_thread` and the unique `CrewAIEventsLoop` daemon thread for async event handlers.
- **openhands**: share the `AppConversationStartTask` row; diverge in HTTP status code (foreground returns 200 immediately, background emits ERROR on failure) and persistence surface.

### Why the sources diverge

These are not quality differences; they are responses to constraints:

- **Library vs. platform**: crewai is a library; it cannot own a scheduler without becoming a platform. It deliberately delegates.
- **Application vs. framework**: openhands is a user-facing app, not an orchestration platform. Its triggers are HTTP requests; a scheduler would not help end users.
- **Open-source vs. proprietary**: langgraph ships the engine publicly; the scheduler is licensed and lives in `langgraph_api.server`. The OSS side tells you what the engine guarantees; the contract tells you what the server promises.
- **Hosted-functions runtime vs. SDK**: agent-framework's Durable Task integration is host-agnostic (Azure Functions + standalone DTS) and uses metadata transformers because isolated workers cannot use attribute decorators.
- **Mature platform vs. library evolution**: Temporal has two scheduler implementations coexisting because it has a 10-year migration window; the others have not yet encountered this trade-off.

## Tradeoffs

The biggest design choices each source made, with both sides.

### Trigger storage: persisted task vs. ephemeral coroutine

- **Benefit** (temporal, agent-framework): durability and idempotency are structural; restart cannot lose triggers.
- **Cost**: a persisted task framework requires migrations, category routing, and schema discipline.
- **Best-fit**: any system that promises at-least-once trigger execution across restarts.
- **Failure mode**: schema drift; category ID renumbering breaks live data (`service/history/tasks/category.go:18-28` carries a warning comment).
- **Alternative**: ephemeral `asyncio.create_task` continuations (openhands, crewai scheduling) — cheap, but restart leaves them in limbo.

### Dual scheduler rollout (temporal)

- **Benefit**: the V1→V2 CHASM migration can be staged via dynamic config (`service/worker/scheduler/fx.go:92-95`); users opt in incrementally.
- **Cost**: doubled maintenance surface; sentinel workflows needed to atomically reserve V1 vs. CHASM keyspace (`service/frontend/workflow_handler.go:3664-3692`).
- **Best-fit**: mature platforms with a large installed base.
- **Failure mode**: dual-source bugs during the transition window.
- **Alternative**: a one-shot replacement (Crew AI extracts CLI in a single change, `lib/crewai/src/crewai/cli/__init__.py:19-27`) is cheaper but only viable when no migration contract exists.

### Single-engine runtime (crewai)

- **Benefit**: only one execution body to reason about; test surface is smaller; sync/async duality is trivial.
- **Cost**: no scheduler; no per-mode SLA; background memory writes must drain in `kickoff` finally.
- **Best-fit**: library authors.
- **Failure mode**: a queue caller cannot tell whether a job forked or ran from scratch because `restore_from_state_id` silently falls back (`lib/crewai/src/crewai/flow/runtime/__init__.py:2374-2379`).
- **Alternative**: separate engines per trigger family (agent-framework, temporal) — more code, more clarity per layer.

### HTTP-driven with no scheduler (openhands)

- **Benefit**: simple trigger model; no periodic loop to maintain; immediately obvious.
- **Cost**: no in-process cron, no automatic retry; restart leaves stuck tasks.
- **Best-fit**: low-traffic apps with operator-driven recovery.
- **Failure mode**: a stuck `WORKING` start task is silently stuck forever until a client retries.
- **Alternative**: a minimal in-process loop in `OssAppLifespanService.__aenter__` is the suggested future fix (openhands `Future Considerations`).

### TTL via self-signal (agent-framework)

- **Benefit**: cleanup inherits the durable signal infrastructure for free.
- **Cost**: minimum delay capped at 5 minutes (`docs/features/durable-agents/durable-agents-ttl.md:33`); per-agent overrides unsupported.
- **Best-fit**: any system with a durable signal infrastructure already.
- **Failure mode**: if every worker is down when the signal fires, deletion is delayed until one reattaches.
- **Alternative**: periodic reaper — easier to scale, but introduces its own scheduler.

### Continue-as-new scheduler workflow (temporal V1)

- **Benefit**: scheduler state is fully observable in workflow history; queries (`Describe`, `ListMatchingTimes`) are cheap.
- **Cost**: each schedule is a running workflow with full history; `IterationsBeforeContinueAsNew` (`service/worker/scheduler/workflow.go:285`) must be tuned.
- **Best-fit**: systems where scheduler state visibility matters.
- **Failure mode**: history bloat if CAN settings are too lax.
- **Alternative**: CHASM component-based scheduler (temporal V2) avoids history; trades state-history observability for typed-component observability.

### Correlation-id post-hoc matching (agent-framework)

- **Benefit**: a single id pattern covers HTTP, MCP, and orchestrator; tests are simple.
- **Cost**: duplicate triggers both append a state entry; the caller must dedup by id (`python/packages/durabletask/agent_framework_durabletask/_entities.py:151`).
- **Best-fit**: systems where the caller is expected to enforce idempotency.
- **Failure mode**: a misclassified duplicate becomes two `conversation_history` entries.
- **Alternative**: pre-execution idempotency token (temporal `RequestId`).

### `Idempotency-Key` header / `RequestId`-based start

- **Benefit**: dedup is enforced before any side effect.
- **Cost**: the SDK must synthesize and forward the token; replay-safe handlers must remember it.
- **Best-fit**: any system exposed over HTTP retries.
- **Failure mode**: lost tokens become duplicate triggers.
- **Alternative**: post-hoc correlation matching — cheaper to implement, harder to reason about.

## Decision Guide

Use this section when choosing an approach.

### Choose a durable-entity trigger host (agent-framework model) when:

- Triggers must survive process restart and shard ownership changes.
- Multiple integration sources (HTTP, MCP, queue, timer, event) funnel into the same agent surface.
- Long-running sessions (hours to days) need durable state per session.
- You already have, or are willing to deploy, Durable Task or equivalent infrastructure.

### Choose a persisted-task scheduler (temporal model) when:

- The system is a *platform* that other people schedule against.
- Trigger idempotency must be guaranteed server-side across `(RequestId, ConflictToken, catch-up window)`.
- You can afford to maintain dual scheduler implementations during migration.
- Schedules have rich policies (`overlap`, `missed-catchup`, `buffer-overrun`) that need explicit semantics.

### Choose an engine + external scheduler (langgraph model) when:

- The OSS engine is the artifact you ship; the scheduler is proprietary.
- In-process channel-triggered and per-task retry semantics are the main user-visible concern.
- The trigger contract is more important than the implementation (callers are typed-RPC clients).

### Choose a library with opt-in durability (crewai model) when:

- You are a library author and the caller is expected to bring their own scheduler.
- The user's restart tolerance is mixed (some workflows tolerate loss, some don't).
- You want a clean sync/async duality over one execution body.

### Choose an HTTP-driven app with durable middleware state (openhands model) when:

- You are building a user-facing app where most triggers are HTTP and operator-driven recovery is acceptable.
- Trigger classification matters for analytics (guarded "don't override existing" semantics).
- Background work is short and rare enough that fire-and-forget is OK.
- A scheduler is overkill for your traffic pattern.

### Rules of thumb

1. **If you promise any `kickoff_at(t)` and `t` can outlive the process, persist it.** crewai and openhands demonstrate the failure mode; temporal and agent-framework demonstrate the fix.
2. **If you accept triggers over HTTP, give the caller an idempotency key.** temporal demonstrates the correct shape; langgraph's absence of one in OSS is its main weakness.
3. **If your triggers are diverse, funnel them through one entity per session.** agent-framework's per-session durable entity is the cleanest version of this.
4. **If you promise `at-least-once` delivery, also promise `exactly-once` effects.** Otherwise you will need post-hoc dedup (which is what agent-framework's `correlation_id` does today).
5. **If you promise "restart-safe", every intermediate state must be persisted.** agent-framework persists after every run (`_entities.py:172-173`); openhands persists after every yield (`:300-307`); temporal persists after every task.
6. **Treat schedulers as infrastructure that deserves tests, not afterthoughts.** agent-framework has integration tests against the Durable Task Scheduler; langgraph's open-source side has no in-source cron firing test.

## Practical Tips

Concrete patterns to copy, delay, and avoid.

### Patterns to copy

- **Trigger ID + persisted state per trigger** (temporal `RequestId`, agent-framework `correlation_id`). One field, one indexed column, one dedup query.
- **Self-signal TTL** if you have a durable signal infrastructure (agent-framework).
- **Channel triggers per node** in dataflow runtimes (langgraph).
- **Catch-up window with explicit drop + metric** (temporal `ScheduleMissedCatchupWindow`).
- **Async-generator start-task state machine with per-yield persistence** (openhands) when you cannot adopt a full durable platform.
- **Per-conversation pending-message queue with hard cap** (openhands) for any startup race.
- **Skip-on-replay side-effect gate** (agent-framework `ctx.is_replaying`) in any durable orchestrator.
- **`PUSH_TRIGGER` reserved-sentinel** (langgraph) to keep control flow separate.
- **`ContinueAsNew` + dynamic config** (temporal) for always-on system workflows.

### Patterns to delay until needed

- **CHASM-style scheduler component** until the simpler V1-style workflow scheduler starts to hit history-bloat limits.
- **Per-namespace workers** until the cluster has >100 namespaces.
- **Sentinel workflow for keyspace reservation** only when dual implementations coexist.
- **`Idempotency-Key` header** until there is a documented retry expectation.
- **`@persist` everywhere** — only on flows that touch durable state.

### Patterns to avoid

- **Restart-dangling `asyncio.create_task` continuations without a supervisor.** openhands and crewai both have this exact failure mode.
- **Silent fallback on `restore_from_state_id` miss.** It hides bugs in queue-driven callers.
- **Global hook registries on a long-lived process** (crewai's `_before_llm_call_hooks` etc.). They break multi-tenancy.
- **Status in custom payload on Azure Functions above 16 KB** (agent-framework cap). The framework already detects this and omits `events`; replicate this pattern.
- **`wait_for_response=False`** without a built-in notification sink. agent-framework's fire-and-forget uses a callback side channel; do not pretend it is fire-and-forget.
- **Cross-loop hang band-aids** for entity handlers (agent-framework `_async_bridge`). The band-aid works but couples the framework to a single loop.
- **Webhook flood with no event-id dedup** (openhands). Every fan-out doubles.
- **Single-shot `asyncio.create_task` polling fallback in `__init__`** (openhands `remote_sandbox_service.py:54`). Replace with a supervisor that can be cancelled and re-acquired.

## Anti-Patterns / Caution Signs

If you see these in a codebase, the trigger layer is becoming brittle:

- **Restarts leave in-flight tasks stuck forever.** If a task row is in a non-terminal state but no background coroutine is processing it, restart-survival is broken. openhands exhibits this; agent-framework avoids it by persisting before return.
- **Trigger vocabulary scattered across many modules with no shared abstraction.** crewai's seven-module trigger surface is fragile; if you find yourself adding a trigger in a new module, you have already crossed the line.
- **"Don't override existing trigger" guards without a documented override path.** openhands's `detect_automation_trigger` returns `current_trigger` if not None — good — but has no UI/API to set it explicitly other than the original create call.
- **Background tasks created without observable lifecycle.** If you grep for `asyncio.create_task` and find no supervisor wrapping it, the task is fire-and-forget.
- **Idempotency keys missing on HTTP triggers.** langgraph's OSS side lacks `Idempotency-Key`; this is its biggest weakness. If your API has the same gap, fix it before scaling.
- **Catch-up replay unbounded.** temporal's catch-up window is bounded (`service/worker/scheduler/workflow.go:1305-1314`); without one, a long downtime will fan out every missed tick.
- **TTLs scheduled via `asyncio.create_task(sleep(N))` rather than persistent timers.** If `N >` process lifetime, the task is lost on restart.
- **Cron with `enabled=False` that does not delete the row.** langgraph's `enabled: bool` keeps the row, so a future `enabled=True` resumes from the original schedule — surprising. If you must keep the row, surface the resume semantics.
- **Webhook delivery with no signing/HMAC.** langgraph's `webhook_secret` is not in the public schema; never accept webhooks without HMAC.
- **Retry budget not observable.** langgraph's `RetryPolicy.max_attempts` is opaque at runtime; you cannot introspect "how many attempts did this run consume?" If you cannot expose the metric, operators will guess.

## Notable Absences

Things that no source in this study implements, even though the dimension's evidence list suggests they might:

- **Built-in framework-level cron/timer trigger.** agent-framework relies on the host's Durable Task primitives; langgraph implements cron in `langgraph_api.server` only; crewai and openhands have no scheduler at all. Only temporal exposes a cron surface, and it has two implementations.
- **In-process queue broker.** No source ships with Celery/Dramatiq/RQ/SQS integration in-tree. crewai and openhands explicitly defer to the host.
- **Built-in outbox/transactional pattern.** agent-framework's `signal_entity + entity run` is at-least-once, not transactional. crewai notes the gap (no two-phase commit between kickoff and checkpoint). temporal's `WorkflowIdReusePolicy.REJECT_DUPLICATE` is the closest pattern.
- **Restart-resume for in-flight async coroutines.** openhands's `asyncio.create_task` continuations are lost on restart; the start-task DB row survives but no coroutine resumes it. agent-framework avoids this by pinning everything in durable entities.
- **Universal idempotency key.** Only temporal's `RequestId` is universal. agent-framework's `correlation_id` is post-hoc. langgraph's `multitask_strategy` only protects one thread at a time.
- **Webhook signing.** langgraph's OSS schema has no `webhook_secret`. temporal does not expose a generic user-facing webhook surface.
- **Process-level scheduler that survives restart without external infrastructure.** crewai, openhands, and langgraph all require external infrastructure (Celery, k8s cron, langgraph server) for that capability.
- **Trigger observability hooks in OSS.** temporal has full metrics; agent-framework relies on the host's dashboard; langgraph has `RunStatus` and stream-mode events; crewai has event-bus dispatch but no first-class OTel trigger span; openhands logs background errors but does not surface them as the trigger's status.
- **Cancel handler.** agent-framework explicitly rejects `cancellationToken` on `DurableAIAgent.RunCoreAsync` (`dotnet/src/Microsoft.Agents.AI.DurableTask/DurableAIAgent.cs:95-99`) — a notable absence for a durable host.

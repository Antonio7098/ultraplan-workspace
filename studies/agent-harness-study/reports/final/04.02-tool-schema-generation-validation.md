## Executive Summary

Eight sources converge on a small set of mature, well-tested patterns for tool schemas. The dominant pattern is **schema generated from Python type annotations, validated by Pydantic at the call boundary, transformed per-provider before the wire** — used in some form by agent-framework (Python), crewai, openai-agents-sdk, and pydantic-ai, with the highest possible maturity in pydantic-ai. Two sources deliberately deviate: **langgraph delegates everything to `langchain_core`** (no schema generation of its own), and **opa uses hand-coded `*ast.Builtin` struct literals** with a release-time JSON dump. Three sources add a **provider transformation/healing layer** before the wire (crewai sanitizers for OpenAI/Anthropic/Bedrock, letta's `enable_strict_mode` + per-provider cleaners, pydantic-ai's `JsonSchemaTransformer` family). Letta's layered model — three generation sources, one canonical persisted schema, five layers of pre-flight mutation, shallow output validation — is the most distinctive. Agent-framework stands out as a **dual-stack reference**: Pydantic on Python and `Microsoft.Extensions.AI` reflection on .NET, both with first-class `JsonSchema` objects and round-trip tests.

The headline lessons are: (1) **descriptions are the highest-leverage, lowest-cost investment** — pydantic-ai makes them mandatory via `require_parameter_descriptions`, opa enforces them with a CI test, openhands writes them as model-instructional prose; (2) **strict-mode is a per-provider policy, not a global one** — every multi-provider source exposes per-provider toggles (crewai, letta, pydantic-ai) or per-tool toggles (openai-agents-sdk), and **none** treats strict mode as universal; (3) **schema-validation error-to-retry plumbing is the single biggest differentiator** — pydantic-ai converts `ValidationError` to a `RetryPromptPart` carrying the tool name + call ID, crewai appends a `build_schema_hint` to the error, openai-agents-sdk uses `ModelBehaviorError` + `default_tool_error_function`, agent-framework and openhands delegate recovery to the next LLM turn without structured framing; (4) **portability is preserved by treating the schema as a transparent `dict`/`JsonElement`** and never mutating it — every source that multi-targets providers keeps the canonical schema immutable and does the per-provider work in a separate wrapper.

## Core Thesis

A library/tool author who wants an LLM to **correctly** call a function needs four guarantees working together: (a) a JSON Schema the model can read, (b) descriptions that disambiguate intent, (c) a validator that runs *before* the tool body executes, and (d) a structured error path that nudges the model to retry with corrected input. The eight sources cover the design space along exactly these four axes:

- **Generation:** Pydantic v2 introspection (the default), function-name+AIFunction reflection (agent-framework .NET / MEAI), AST walking with docstring parsing (letta), handwritten `*ast.Builtin` literals (opa), or no generation at all (langgraph).
- **Description quality:** enforced-with-error (pydantic-ai `require_parameter_descriptions`, opa `TestAllBuiltinsHaveDescribedArguments`), enforced-but-soft (letta logs but does not raise), opportunistic (crewai, agent-framework, openhands, openai-agents-sdk, langgraph).
- **Validator placement:** Pydantic at the call boundary (pydantic-ai, openai-agents-sdk, crewai, openhands via FastMCP), a function-invocation-layer validation (agent-framework Python, langgraph via langchain_core), or annotation-based coercion with no schema validation (letta).
- **Retry/correction plumbing:** structured `RetryPromptPart`/`ModelBehaviorError`/build_schema_hint (pydantic-ai, crewai, openai-agents-sdk), warn-only (letta), or natural LLM self-correction with no scaffolding (agent-framework, openhands, opa).

The dimensions correlate: the source with the most sophisticated retry plumbing (pydantic-ai) also has the strictest registration-time validation (`require_parameter_descriptions`). Sources that delegate validation to an external library (langgraph, agent-framework .NET) often lack rich correction plumbing by design.

## Approach Models

The eight sources fall into four distinct approach families, each with its own center-of-gravity for the four guarantees listed above.

### 1. Generated-Pydantic, validated-at-boundary (the default)
**Sources:** pydantic-ai, openai-agents-sdk, agent-framework (Python), crewai, openhands (via FastMCP).

The dominant pattern. Each tool function's Python signature is fed through `inspect.signature` + `typing.get_type_hints` into a `pydantic.create_model` call, whose `model_json_schema()` becomes the wire payload. Validation happens at every tool invocation through Pydantic's `SchemaValidator` (or equivalent), so the framework reuses the same schema for "show to model" and "validate model output." Per-provider customization is layered either before (openai-agents-sdk's `ensure_strict_json_schema`) or after (pydantic-ai's `JsonSchemaTransformer`) the canonical schema is generated. Source-specific evidence:
- `pydantic_ai_slim/pydantic_ai/_function_schema.py:125-200` builds the TypedDict core schema + `pydantic_ai_slim/pydantic_ai/tool_manager.py:215-250` validates every call.
- `src/agents/function_schema.py:263-407` builds the dynamic `BaseModel`; `src/agents/tool.py:503-510` re-applies `ensure_strict_json_schema`.
- `python/packages/core/agent_framework/_tools.py:484-514` builds the Pydantic model; `_tools.py:1097-1144` validates.
- `lib/crewai/src/crewai/tools/base_tool.py:200-247` builds the schema; `_tools.py:272-293` validates with `build_schema_hint`.
- `openhands/app_server/mcp/mcp_router.py:147-163` introspects `Annotated[T, Field(...)]` via FastMCP; `mcp_router.py:43` configures `mask_error_details=True`.

### 2. Reflection-driven, framework-delegated (the polyglot mode)
**Sources:** agent-framework (.NET side only), opa.

Here the framework does *not* own the schema generator. Agent-framework delegates .NET schema generation to `Microsoft.Extensions.AI.AIFunctionFactory.Create(...)`, which reflects on `[Description]`-annotated delegate parameters (`dotnet/src/Microsoft.Agents.AI/Skills/Programmatic/AgentInlineSkillScript.cs:45-47`). OPA goes further — schemas are hand-coded Go struct literals, and the "transformation" step is just `Builtin.MarshalJSON` emitting a Rego-native type tree (`v1/types/types.go:787-801`) plus a release-time dump to `builtin_metadata.json` via `internal/cmd/genbuiltinmetadata/main.go:26-81`. Both sources rely on external libraries (MEAI, Go runtime) for validation. Source-specific evidence:
- `v1/ast/builtins.go:780-796` shows the hand-written `Count` builtin; `v1/ast/builtins_test.go:33-62` is the CI test enforcing descriptions.
- `dotnet/src/Microsoft.Agents.AI.Tools.Shell/LocalShellExecutor.cs:340-361` shows the `AIFunctionFactory.Create` pattern reused everywhere on .NET.

### 3. Schema-as-data with transformation pipeline (the layered mode)
**Sources:** letta, crewai (partially), openai-agents-sdk (partially), pydantic-ai (transformers).

Letta is the canonical example: one persisted `tool.json_schema`, then five layers mutate it before the LLM call — `enable_strict_mode` adds `strict: true` and makes optional fields nullable (`letta/helpers/tool_execution_helper.py:89-162`), `runtime_override_tool_json_schema` injects `request_heartbeat` (`tool_parser_helper.py:78-113`), per-provider converters (`anthropic_client.py:1412`, `google_vertex_client.py:348`, `openai_client.py:384-415`) further clean for wire. Crewai and pydantic-ai follow the same pattern with one mutator and a sanitizer per provider (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:572-635`, `pydantic_ai_slim/pydantic_ai/profiles/openai.py:263-419`). Source-specific evidence:
- `letta/functions/schema_validator.py:12-202` is the `SchemaHealth` classifier that gates strict-mode healing.
- `crewai`'s `sanitize_tool_params_for_{openai,anthropic,bedrock}_strict` chain is the producer-side equivalent.
- `pydantic_ai`'s `JsonSchemaTransformer.walk()` is the consumer-side (request-prep) equivalent.

### 4. Schema passthrough only (the delegating mode)
**Source:** langgraph.

LangGraph explicitly does not generate schemas. It wraps functions in `langchain_core.tools.create_tool(...)`, lets the chat-model provider adapter translate `BaseTool.args_schema` for the LLM, and only contributes *error plumbing* — `tool_node.py:959-966` catches `pydantic.ValidationError`, translates to `ToolInvocationError`, lets the configurable `handle_tool_errors` turn it into a `ToolMessage(status="error")`. Source-specific evidence:
- `libs/prebuilt/langgraph/prebuilt/tool_node.py:779-786` registers tools verbatim; validation runs inside `tool.invoke(...)` (langchain_core).
- `tool_node.py:339-380` defines `ToolInvocationError`; `tool_node.py:383-391` is the default handler.

## Pattern Catalog

### Pattern 1: Pydantic introspection → JSON Schema → wire
**Problem:** Schemas must be generated, validated, and forwarded without drift between three consumers (model, validator, client-facing wire).
**Solution:** Function signature → `pydantic.create_model(...)` → `model_json_schema()` → cache → forward.
**Demonstrated by:** pydantic-ai (`_function_schema.py:103-281`), openai-agents-sdk (`function_schema.py:263-407`), agent-framework Python (`_tools.py:484-514`), crewai (`base_tool.py:200-247`, `base_tool.py:707-747`), openhands via FastMCP (`mcp_router.py:147-163`).
**Why it works:** One source of truth eliminates schema-vs-implementation drift; Pydantic's JSON Schema is a portable, widely-supported dialect; subsequent per-provider mutation (strictness, keywords) layers on top without touching the source.
**When to copy:** Any Python tool-calling library whose tools are authored as Python functions and whose provider supports JSON Schema. This is the 80%-solution.
**When it is overkill or risky:** When tools must be authored in a non-Python language (C#/Go/JS); when the provider enforces a JSON Schema subset that Pydantic does not emit without rewrites; when authoring tools as JSON Schema dicts directly (MCP servers, OpenAPI imports). Pydantic-ai's own docstring at `_function_schema.py:1-4` flags brittleness in the dependency on `_generate_schema`.

### Pattern 2: Per-provider transformation, not per-provider schema
**Problem:** A canonical schema must satisfy providers with conflicting strict-mode constraints (OpenAI strict, Anthropic defaults, Gemini keyword denylists).
**Solution:** Keep the canonical JSON Schema immutable; apply a per-provider transformer at request-prep that emits a wire-compatible version.
**Demonstrated by:** pydantic-ai's `JsonSchemaTransformer.walk()` (`models/__init__.py:1288-1309`) with provider-specific implementations for OpenAI (`profiles/openai.py:263-419`), Anthropic (`providers/anthropic.py:121-187`), Bedrock (`providers/bedrock.py:61-122`), Gemini (`profiles/google.py:105-146`); crewai's `sanitize_tool_params_for_*_strict` pipeline (`utilities/pydantic_schema_utils.py:572-635`); letta's per-provider converters (`anthropic_client.py:1412`, `google_vertex_client.py:348`, `openai_client.py:384-415`); agent-framework's per-provider wire serialization (`ChatCompletions/Models/Tool.cs:65-70`, `FoundryPromptAgentConverter.cs:126-136`).
**Why it works:** Transformer preserves a single source of truth while emitting provider-tailored schemas. Lossy fields (e.g. `minLength`, regex lookaround) are saved into `description` text so the information reaches the model in human-readable form (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:374-383`).
**When to copy:** Any library targeting 2+ LLM providers with disjoint strict-mode grammars. Especially valuable when the source-of-truth schema is owned by the user (Pydantic model) but wire compatibility is the library's concern.
**When overkill:** Single-provider products (just use the provider's recommended schema dialect). When providers converge on a shared subset (uncommon today).

### Pattern 3: Mandatory descriptions at registration
**Problem:** Without descriptions, the model guesses argument intent from names alone; hallucinated values multiply; tool calls become unreliable.
**Solution:** Assert during tool registration that every parameter has a description; fail with a `UserError` listing missing params.
**Demonstrated by:** pydantic-ai's `require_parameter_descriptions=True` flag (`_function_schema.py:175-176,209-216`); opa's `TestAllBuiltinsHaveDescribedArguments` CI test (`v1/ast/builtins_test.go:33-62`); crewai's per-tool `description=` is conventional but not enforced (every built-in `agent_tools/*.py` declares them); openhands writes them as explicit model-instructional prose (`mcp_router.py:159-162`).
**Why it works:** A single failure at registration is infinitely cheaper than production-time hallucination. Letta attempts this in `validate_google_style_docstring` but is **soft-fail only** (`schema_generator.py:411-420` log rather than raise), which the letta report explicitly calls out as a gap.
**When to copy:** Always, when the registry of tools is under your control. For library authors, default the flag to `True`.
**When it is overkill:** Hand-curated small tool sets where the developer treats descriptions as documentation anyway; or when the layer is a server rather than a library.

### Pattern 4: Validation error → structured retry prompt
**Problem:** When the model emits invalid arguments, the tool throws and the conversation either crashes or relies on the model to notice the error.
**Solution:** Catch `ValidationError` at the boundary, reformat as a structured prompt part that names the tool and lists available params, return to the model inside the conversation with the original `tool_call_id`.
**Demonstrated by:** pydantic-ai (`tool_manager.py:183-191, 341-350` — `RetryPromptPart(tool_name=..., tool_call_id=...)` with bound `max_retries`); openai-agents-sdk (`tool.py:1609-1618` — `ModelBehaviorError` through `default_tool_error_function`); crewai (`base_tool.py:272-293` + `structured_tool.py:90-108` — `build_schema_hint` appended to error); langgraph (`tool_node.py:339-380, 383-391` — `ToolInvocationError` → `ToolMessage(status="error")`).
**Why it works:** The model sees a reminder of what the schema actually accepts (not just "I tried X and it failed"), making self-correction cheap. A retry budget (`max_retries` / `_max_parsing_attempts`) prevents infinite loops.
**When to copy:** Any tool layer that expects an LLM to call back into itself with corrected arguments. Pair the retry prompt with a budget cap.
**When overkill:** One-shot non-agentic tool calls (the cost of building the re-prompt outweighs the benefit). When the tool is non-deterministic and re-prompting is unlikely to produce a useful answer.

### Pattern 5: Strict mode as a per-provider flag, default-on where supported
**Problem:** Strict mode forces `additionalProperties: false`, all-properties-required, and a constrained JSON Schema subset — which the OpenAI Responses API and Anthropic accept but most open-source providers do not.
**Solution:** Per-provider capability map; default to strict for providers that support it; opt-out at registration time for the rest.
**Demonstrated by:** agent-framework (`FoundryPromptAgentConverter.cs:131-135` defaults Foundry to strict, Chat Completions exposes but does not default — `Tool.cs:78-80`); letta (`LLMConfig.strict` per-agent flag at `llm_config.py:116-119`, Anthropic strict allowlist at `anthropic_constants.py:13-27`); pydantic-ai (`openai_supports_strict_tool_definition`/`bedrock_supports_strict_tool_definition` profile flags); openai-agents-sdk (`strict_json_schema=True` default on `FunctionTool` — `tool.py:1859, 1907`); crewai (strict-by-default for native path at `agent_utils.py:1600-1627`, `anthropic/completion.py:490-535`).
**Why it works:** Cross-provider portability is preserved; per-tool opt-out avoids forcing callers to choose between strict mode and compatibility.
**When to copy:** Any multi-provider harness. The default should be "strict where supported"; the override should be a per-tool flag, not global.

### Pattern 6: Defense-in-depth validation outside the schema
**Problem:** Schema validation catches type errors but not business-rule violations (`count > 0` even though `count: int` is satisfied).
**Solution:** Run a second-layer `args_validator`/model-retry hook *after* Pydantic validation and *before* tool body execution.
**Demonstrated by:** pydantic-ai (`args_validator` parameter at `tools.py:84-94`; `ModelRetry` raises convert to `RetryPromptPart` — `tool_manager.py:183-191`); openai-agents-sdk (`failure_error_function` chain at `tool.py:1519-1618`); crewai (`ToolUsage._validate_tool_input` cascading parsers at `tool_usage.py:884-930`); openhands (`validate_secret_name` etc. at `constants.py:97-160` as example domain validators).
**Why it works:** JSON Schema validates type; semantic rules belong in code. Carrying the retry plumbing (not just the validation) means a model-steerable error: not just "invalid" but "must be positive integer, got -1."
**When to copy:** Any tool surface where semantic constraints matter (financial operations, security checks, business workflows).

### Pattern 7: Schema round-trip preservation through wire formats
**Problem:** Schemas published to MCP, OpenAI Chat Completions, and OpenAI Responses must be identical after serialization.
**Solution:** Treat the schema as a transparent `dict`/`JsonElement`; the same object reference flows through every wire layer; round-trip tests pin the contract.
**Demonstrated by:** agent-framework's `dotnet/tests/Microsoft.Agents.AI.AGUI.UnitTests/AIToolExtensionsTests.cs:193-215` (`RoundTrip_AIFunctionToAGUIToolBackToDeclaration_PreservesMetadata`); pydantic-ai's snapshot tests (`tests/profiles/test_openai.py:97-173`, `tests/profiles/test_anthropic.py:49-199`); openai-agents-sdk's `FunctionTool.__post_init__` deep-copies before strictifying (`tool.py:507-510`).
**Why it works:** Tests catch schema mutation drift that hand-written codebases miss; immutability discipline prevents inadvertent shared-state bugs.
**When to copy:** Any library that ships tools through multiple serialization layers.

### Pattern 8: Healthcare for upstream-provided schemas
**Problem:** MCP servers, OpenAPI imports, and SDK-based schemas may contain keywords (e.g. `oneOf`, `default`, `additionalProperties: true`, `format: regex`) that strict-mode or specific providers reject.
**Solution:** Classify the schema's health (`STRICT_COMPLIANT` / `NON_STRICT_ONLY` / `INVALID`) before forwarding; heal what can be healed (e.g. optional fields → nullable); strip what can't; warn or refuse the rest.
**Demonstrated by:** letta's `SchemaHealth` enum + `validate_complete_json_schema` (`schema_validator.py:12-202`); letta's `enable_strict_mode` healer (`tool_execution_helper.py:89-162`); letta's MCP normalizer (`schema_generator.py:586-691`); crewai's recursive sanitizers (`utilities/pydantic_schema_utils.py:586-635`); openai-agents-sdk's `ensure_strict_json_schema` (`strict_schema.py:18-149`); pydantic-ai's per-provider transformers (especially OpenAI strict mode drops).
**Why it works:** Tells the developer (and the user-facing error) what went wrong, instead of letting the provider reject mid-request with a vague error.
**When to copy:** Whenever you accept third-party schemas (MCP, OpenAPI imports, plugin descriptions).

### Pattern 9: Default-stripping for strict providers
**Problem:** OpenAI strict mode does not allow `default` to coexist with `required`-in-`anyOf`; some fields legitimately have defaults but cannot be declared under strict.
**Solution:** Strip `default: None` in strict transforms; surface the field as required.
**Demonstrated by:** openai-agents-sdk's `ensure_strict_json_schema` (`src/agents/strict_schema.py:121-122`); pydantic-ai's OpenAI transformer (`profiles/openai.py:344-351`).
**Why it works:** Keeps a schema that's strict-compatible at the cost of one bit of metadata the model can interpret anyway.
**When to copy:** When targeting strict-mode providers (OpenAI Responses, Anthropic strict). Document the lossiness in `is_strict_compatible` so callers can opt out per tool.

### Pattern 10: Injected-argument filtering from error messages
**Problem:** Tools that take runtime-injected parameters (`InjectedState`, `InjectedStore`, `ToolRuntime`) must not let the model forge their values via `tool_call["args"]`, and any validation errors on injected args must not leak those arg names back to the model.
**Solution:** Per-tool precomputation of injected-arg keys at registration; defensive strip before invocation; `loc`-tuple-based filter on `ValidationError.errors()` to keep only model-controlled args in the error report.
**Demonstrated by:** langgraph's `_InjectedArgs` dataclass (`tool_node.py:1967-2030`), `_filter_validation_errors` (`tool_node.py:510-563`), `stripped_args` filter (`tool_node.py:1424-1429`).
**Why it works:** Single invariant — "injected args are system-controlled and never visible to the LLM" — expressed at three points (registration, dispatch, error report). The defensive strip is doubly-enforced by a free-floating comment about the threat model.
**When to copy:** Any tool-calling layer that injects context, store, or runtime into tool bodies.

### Pattern 11: Approval wrapper as a schema-level interposition hook
**Problem:** Dangerous tools (shell execution, file access) need user approval before invocation, but approval is orthogonal to schema correctness.
**Solution:** Wrap the `AIFunction` in a delegation layer that returns `ToolApprovalRequestContent` instead of executing; the wrapped function's schema is unchanged.
**Demonstrated by:** agent-framework's `ApprovalRequiredAIFunction(fn)` (`LocalShellExecutor.cs:328-361`, `DockerShellExecutor.cs:266-293`, `FileAccessProvider.cs:305-310`); the `ToolApprovalAgent` wraps the agent and the workflow `InvokeFunctionToolExecutor` halts via `ExternalInputRequest` (`InvokeFunctionToolExecutor.cs:80-113`).
**Why it works:** Schema stays the same; the framework can hang approval policies off the wrapper without touching tool code.
**When to copy:** Whenever a tool set contains side-effecting operations; coupling approval to schema (rather than configuration) keeps the entry point discoverable.

## Key Differences

### Source maturity varies more than approach
Pydantic-ai scores 9/10 and openhands scores 5/10 — but openhands' "thin server-side wrapper around MCP tools" is a different *kind* of system, not an inferior version. Pydantic-ai is a programmable agent toolkit where schema validation is a first-class concern; openhands is a server-as-thin-wrapper whose primary agent tools live in an external `openhands-tools` package and cannot be assessed. The 5/10 reflects what we can verify *inside* the openhands repo (prescriptive descriptions, FastMCP typed introspection, validation testing gaps), not the quality of the agent experience.

### Generated vs handwritten schemas is a real axis
**Generated (5 sources):** agent-framework (Python), crewai, openai-agents-sdk, pydantic-ai, openhands (via FastMCP/Pydantic). Plus letta for source-text upload, MCP and AST cases.
**Handwritten (1 source):** opa.
**Delegated (1 source):** langgraph (langchain_core generates).
**Reflection-without-typed-source (1 source):** agent-framework .NET (MEAI reflects on `[Description]`-annotated delegates).

The generated-pattern winners (pydantic-ai, openai-agents-sdk, crewai) all score 8–9. Handwritten (opa) scores 7 because the discipline (CI test, metadata dump) compensates. Delegated (langgraph) drops to 6 because nobody owns the schema behavior in that layer.

### Validation strictness gradient
**Strict at the boundary:** pydantic-ai (`extra_behavior='forbid'` default in TypedDict — `_function_schema.py:346-352`), openai-agents-sdk (strict-mode default — `tool.py:408`), crewai (`additionalProperties=False` — `pydantic_schema_utils.py:181-217`), agent-framework Python (`_validate_arguments_against_schema` — `_tools.py:1097-1144`).
**Strict in chunks:** openhands (strict for secrets and LLM profiles, lenient for stored profiles), opa (strict compile-time + lenient runtime by default).
**Lenient / coerce-only:** letta (annotation-based coercion at runtime — `ast_parsers.py:117-168`); deeply so — an `int` parameter accepts `"five"` if `int("five")` doesn't fail, missing descriptions degrade silently.
**Delegated:** langgraph (schemes delegated to langchain_core); agent-framework .NET (delegated to MEAI).

The lenient approach (letta) does not scale to provider-strict mode; the strict approach (pydantic-ai, openai-agents-sdk) requires a Pydantic-or-equivalent dependency that may not be appropriate in languages like Go (opa's choice).

### Provider portability mechanisms
- **Sanitizer/transformer at the boundary (4 sources):** pydantic-ai, crewai, openai-agents-sdk (default for OpenAI, opt-out for the rest), agent-framework (.NET per-provider converters).
- **Per-provider cleaners at request time (1 source):** letta — distinct from "transformer" because cleaners drop more keywords and order matters.
- **Schemas-as-data + capability allowlist (1 source):** opa — `Capabilities.Builtins` / `Capabilities.AllowNet`.
- **Treat-schema-as-JsonElement-passthrough (multi-provider in agent-framework):** schema is never rewritten, only wrapped.
- **No portability layer (1 source):** langgraph — delegates to chat-model provider adapter.

### Retry / correction plumbing quality
- **Rich (named, structured, budget-bounded):** pydantic-ai — `RetryPromptPart` with tool name + call ID, `max_retries` cap, `args_valid` event flag.
- **Rich (hint-augmented error):** crewai — `build_schema_hint(expected properties + required)`, `_max_parsing_attempts` cap.
- **Rich (typed exception + formatter):** openai-agents-sdk — `ModelBehaviorError` + `default_tool_error_function` per tool.
- **Translated to message (defaults safe):** langgraph — `ValidationError` → `ToolMessage(status="error")`.
- **Implicit (no scaffolding):** agent-framework, openhands, opa — let the LLM read the failure text and retry.

The pattern across all eight: **every source that scores ≥7/10 has explicit structured retry plumbing.** The pattern holds even for handwritten-schema opa, where the lack of a retry layer is the explicit reason for the deduction. Letta is the exception that proves the rule — high schema-quality practices but warned rather than retried, scores 8/10.

### Description enforcement gradient
- **Hard enforcement (CI test or registration error):** pydantic-ai (`require_parameter_descriptions`), opa (`TestAllBuiltinsHaveDescribedArguments`).
- **Soft enforcement (logged warning, no raise):** letta (`validate_google_style_docstring` raises but caught and logged).
- **Convention only (no enforcement, but built-ins comply):** crewai, agent-framework, openai-agents-sdk, openhands, langgraph, opa for the description-text-itself (but opa enforces the *presence*).

## Tradeoffs

### Schema generation: ergonomics vs. durability vs. expressiveness
- **Pydantic introspection** (pydantic-ai, openai-agents-sdk, agent-framework Python, crewai, openhands via FastMCP) buys ergonomics (one Python function gets a full schema for free). Cost: locks the project into Pydantic v2 (or v1, in langgraph's case); custom JSON-Schema keywords (`$comment`, `examples`, `patternProperties`, `oneOf` discriminators) require escaping the framework (Hyperlight's `ExecuteCodeFunction.cs:57` is the documented escape hatch in agent-framework).
- **Handwritten literals** (opa) buy expressiveness (any constraint can be expressed) and compile-time validation (the struct is impossible to construct incorrectly). Cost: ergonomic drag — adding a tool is a multi-step Go write-test-release-permission cycle.
- **Reflection without typed source** (agent-framework .NET) is a middle ground — ergonomic in C#, but downstream schema behavior depends entirely on the external MEAI library.

### Strict mode in Pydantic-ai vs. openai-agents-sdk
Both default to strict and both rely on a destructive transform. They differ in **where** the transform happens:
- **OpenAI Agents SDK** (`ensure_strict_json_schema` at `src/agents/strict_schema.py:18-149`) mutates the schema at tool-construction time, every fresh usage runs the transform. The framework breaks if the input schema was not strict-safe.
- **Pydantic AI** keeps the canonical schema unchanged and applies a per-provider transformer at request time (`JsonSchemaTransformer.walk()`). Lossy fields are preserved in `description` text (`profiles/openai.py:374-383`). The framework degrades gracefully to non-strict mode for providers that don't support it.

**Best-fit context:** OpenAI Agents SDK is appropriate when targeting OpenAI's Responses/Chat Completions/Realtime ecosystems almost exclusively. Pydantic AI is appropriate when targeting 3+ providers with disjoint strict mode grammars.

### Provider-parity vs strict-ness in agent-framework
`FoundryPromptAgentConverter.cs:131-135` defaults Foundry to strict; `Tool.cs:78-80` (OpenAI Hosting) exposes the `strict` boolean but does not set it; Anthropic/Copilot/A2A stacks forward the schema verbatim to the chat client. **Same tool, same definition, different strict behavior across providers.** This is a portability hazard for schemas that depend on strict semantics (e.g. enum-like `Literal` types). Mitigation would be to unify on a per-provider default (e.g. always-on for providers that support it).

### Schema-as-data vs schema-as-code in Letta vs OPA
- **Letta** persists `tool.json_schema` on the Tool ORM row (`letta/orm/tool.py:50`) and applies five layers of mutation per request. Trade-off: any provider-specific bug surfaces on every LLM call, but the canonical schema is durable and migratable. The system includes a `SchemaHealth` classifier and explicit `request_heartbeat` / `pre_exec_msg` injection and stripping.
- **OPA** persists `*Builtin` literals in Go source. Trade-off: schema changes require code edits, but compile-time validation guarantees no malformed builtin. The release-time dump to `builtin_metadata.json` is a one-way mirror of the in-memory registry.

**Best-fit:** schema-as-data (letta) for end-user tools where the schema is owned by an admin and must survive provider upgrades. Schema-as-code (opa) for system-internal tools where build-time correctness matters more than runtime mutability.

### Lenient vs strict model-output validation (Letta)
Letta's `_safe_load_tool_call_str` (`agents/helpers.py:378-393`) uses `json.loads` with a fallback-to-`{}` on parse errors, then coerces via Python annotations (`ast_parsers.py:117-168`). The framework trusts that strict-mode providers reject invalid calls at parse time, and Python tools get *runtime* coercion based on annotations. **Cost:** `"count": "five"` for an `int` parameter fails inside the tool body rather than at the boundary, with `ValueError` propagating to `ToolExecutionResult(status="error")`. **Benefit:** no validator-on-every-call overhead, simpler parser contract.

### Approval wrapper interposition (agent-framework .NET)
`ApprovalRequiredAIFunction` (`dotnet/src/Microsoft.Agents.AI.Tools.Shell/LocalShellExecutor.cs:361`) wraps the schema-bearing AIFunction and intercepts *before* tool invocation. The wrapped function's schema is unchanged; the user/agent sees a `ToolApprovalRequestContent` and can resume via `ExternalInputRequest`. Trade-off: approval is wired at tool-build time, not tool-call time, so policy changes require rebuilding the function.

### Delegation vs ownership (langgraph)
LangGraph owns no schema generation; it depends on `langchain-core>=1.3.1` (`libs/prebuilt/pyproject.toml:28`). Trade-off: any change in Pydantic ↔ JSON-Schema output or in `BaseTool`'s injected-arg handling propagates silently into LangGraph because there is no local schema-version check. Mitigation: an executable test suite pins behavior; the deprecated `ValidationNode` retains v1-Pydantic code paths (`tool_validator.py:29-30, 196-198`) for migration support.

### `pydantic_core.SchemaValidator` reuse (pydantic-ai vs openai-agents-sdk)
- **Pydantic-ai** builds a `pydantic_core.SchemaValidator` per tool (`_function_schema.py:125-200`) and wires it as `args_validator` for every tool path (function, MCP, external — `toolsets/function.py:632-642`, `mcp.py:1128`). Same validator, three call sites.
- **OpenAI Agents SDK** dynamically builds a Pydantic `BaseModel` (`function_schema.py:407`) and instantiates it per call (`tool.py:1978-1990`). The validator is implicit in the constructor.

**Trade-off:** pydantic-ai's approach is faster (precomputed validator) and uniform (MCP and function tools share one path). OpenAI Agents SDK's approach is simpler to test (Pydantic model is the schema is the validator).

## Decision Guide

**Choose Pydantic-AI-style (generated + transformer + retry-prompt) when:**
- Python-only toolset.
- Targeting 2+ LLM providers with disjoint strict-mode grammars.
- The team values snapshot-tested transformers over per-provider ad-hoc sanitizers.
- Per-tool `args_validator` and `max_retries` is desired.

**Choose OpenAI Agents SDK-style (generated + destructive strict transform + formatter) when:**
- Python + OpenAI Responses / Chat Completions / Realtime.
- Strict mode is the default and providers outside OpenAI are an afterthought.
- `failure_error_function` per-tool customization matters.
- `DONT_LOG_TOOL_DATA` redaction is required for tracebacks.

**Choose CrewAI-style (generated + per-provider sanitizer + `_max_parsing_attempts`) when:**
- Multi-provider strict mode is needed without per-provider transformer boilerplate.
- The two-schema contract (`args_schema` + `result_schema`) is valuable.
- The team accepts text-vs-native schema divergence as a known risk to manage.

**Choose Letta-style (three-source generator + healing pipeline + per-provider cleaner) when:**
- Tools arrive from many sources (Python function, AST of source code, MCP, TypeScript).
- MCP server schemas must be normalized and health-classified.
- Provider wire format is heterogeneous enough to justify a single canonical schema mutated through five layers.
- Annotation-based coercion is acceptable (no Pydantic dependency on tools).

**Choose Agent-framework-style (dual stack with delegated generation + raw JsonSchema passthrough) when:**
- Supporting both Python *and* .NET in one library.
- Targeting 4+ LLM providers (OpenAI Chat Completions, OpenAI Responses, Foundry, Anthropic, GitHub Copilot, A2A).
- The framework should expose schemas as transparent `JsonElement` for per-provider wrappers to consume verbatim.
- Approval wrapping (`ApprovalRequiredAIFunction`) is the primary interposition hook.

**Choose OPA-style (handwritten + CI-enforced + release-time dump) when:**
- Tool calls are policy-language function calls rather than LLM agent calls.
- Compile-time correctness matters more than per-tool ergonomics.
- The team is willing to hand-write `*NamedType` chains and run `go generate` on every addition.

**Choose LangGraph-style (full delegation to underlying chat-model layer) when:**
- The framework is an agent *orchestration* layer that doesn't want to own schema semantics.
- `langchain_core` is already a dependency.
- Error translation (`ValidationError` → `ToolMessage`) is sufficient for the use case.
- Injected-argument filtering (`_filter_validation_errors`) is needed and the user wants the framework to enforce it.

**Choose OpenHands-style (FastMCP decorator + prescriptive descriptions + masking) when:**
- The server is a thin wrapper around an external agent-tool library (`openhands-tools`).
- The team is fine exposing schemas via the MCP wire protocol and not owning provider-specific translation.
- Masked error responses (`mask_error_details=True`) are appropriate for the user-facing surface.

## Practical Tips

### Patterns to copy
1. **Always** write descriptions as model-instructional prose, not parameter-shape descriptions. Openhands shows the style (`"If labels are provided, they must be selected from the repository's existing labels. Do not invent new ones. If the repository's labels are not known, fetch them first."` — `mcp_router.py:159-162`). CrewAI's `Field(description=...)` on every agent tool follows the same pattern.
2. **`require_parameter_descriptions=True` by default**, gated behind a flag for power users. Pydantic-ai demonstrates the registration-time `UserError` (`tests/test_tools.py:1050-1077`).
3. **Per-provider transformer rather than per-provider schema.** Pydantic-ai's `JsonSchemaTransformer.walk()` (`_json_schema.py:201-208`) is the cleanest implementation; crewai sanitizers (agent-side) and agent-framework wire converters (provider-side) are the alternatives.
4. **`RetryPromptPart(tool_name=..., tool_call_id=...)`** rather than free-formatted error string. Pydantic-ai shows the canonical form (`tool_manager.py:183-191`); openai-agents-sdk's `ModelBehaviorError` + `default_tool_error_function` is the formatters-backed equivalent.
5. **`extra_behavior='forbid'` on the validator** (pydantic-ai `_function_schema.py:346-352`) so model-side and validator-side agree on whether unknowns are errors.
6. **Schema health classification** (`STRICT_COMPLIANT` / `NON_STRICT_ONLY` / `INVALID`) before forwarding — letta's `schema_validator.py:12-202`. This catches upstream schema drift before it becomes a production failure.
7. **Round-trip serialization tests** for AG-UI/OpenAI Hosting/MCP wire formats — agent-framework's `RoundTrip_AIFunctionToAGUIToolBackToDeclaration_PreservesMetadata` test (`dotnet/tests/Microsoft.Agents.AI.AGUI.UnitTests/AIToolExtensionsTests.cs:193-215`) is the template.
8. **`DONT_LOG_TOOL_DATA` redaction** for tool-call payloads in error tracebacks — openai-agents-sdk (`tool.py:1583-1593`) shows the pattern.
9. **Injected-arg filtering from validation errors** — langgraph's `_filter_validation_errors` (`tool_node.py:510-563`) is the only well-tested implementation across the eight sources.

### Patterns to avoid or delay until needed
1. **Don't** handwrite `*ast.Builtin`-style schema literals unless you have a strict CI test (like opa's `TestAllBuiltinsHaveDescribedArguments`). Otherwise adding a tool is a PR + code review + release cycle.
2. **Don't** deep-copy the schema in every code path. OpenAI Agents SDK does it once in `__post_init__` (`tool.py:507-510`) — that's the right discipline.
3. **Don't** mutate the canonical schema in place. Every transformer should emit a copy (`ensure_strict_json_schema`, `JsonSchemaTransformer.walk()`, letta's `add_pre_execution_message` returning a new dict).
4. **Don't** add custom JSON-Schema keyword extensions (`oneOf` discriminators, `patternProperties`, `$ref` reuse) unless you also build the strict-mode rewrite for them.
5. **Don't** use `additionalProperties: True` for LLM-facing tool schemas. Even non-strict-mode providers benefit from explicit close-objects.

### Decision rules
- **Python + multi-provider:** pydantic-ai or crewai, depending on whether you want transformers (pydantic-ai) or sanitizers (crewai).
- **Python + OpenAI-only:** openai-agents-sdk with `strict_json_schema=True` default.
- **Python + MCP-imported tools:** letta's generator + normalizer + per-provider cleaner pattern.
- **.NET + multi-provider:** agent-framework's `Microsoft.Extensions.AI` reflection + per-provider wire converters.
- **Polyglot policy evaluator:** opa's manual struct-literal + CI test + release-time JSON dump.
- **Thin MCP wrapper:** openhands' FastMCP decorator + `Annotated[..., Field(...)]`.

### Per-run validation depth markers
- **Validation is too shallow** if `pydantic.ValidationError` from a function-bodied validator reaches the model as raw text. Look for: letta's pattern (must be Pydantic-typed *or* annotation-coerceable, not schema-validated).
- **Description quality is degrading** if `_validate_kwargs` or its equivalent shows up in error messages as `{"loc": "value", "msg": "Input should be a valid integer"}` — the description metadata was dropped before it reached the model. Look for: pydantic-ai's `_function_schema.py:175-176` enforcement at the boundary.
- **Strict mode is being silently ignored** if `strict=true` shows up in the wire but the provider still hallucinates `additionalProperties`. Likely cause: an MCP server schema was applied without healing. Look for: letta's `validate_complete_json_schema` classification chain.
- **The schema is drifting from the implementation** if the user-visible descriptions no longer match a parameter renaming. Look for: type-driven schemas (Pydantic, FastMCP) — no need to look otherwise.

## Anti-Patterns / Caution Signs

1. **Schema-as-text vs. schema-as-native drift.** CrewAI's `_generate_description` (`base_tool.py:482-490`) inlines the JSON Schema into the tool description string but keeps `strip_null_types=False` for the native path (`agent_utils.py:184-189`). A model reading the text prompt and a model reading the native provider payload see different schemas. **Anti-pattern.** Mitigation: produce both views from a single canonical schema, or document the divergence.

2. **Generation source as a hidden dependency.** Agent-framework's .NET side delegates to `Microsoft.Extensions.AI.AIFunctionFactory` (external NuGet). Schema behavior changes upstream silently. **Caution sign:** no `schema_version` field on emitted schemas; no compile-time anchor on the schema generator.

3. **Lenient-by-default with no opt-in compliance.** OPA's `builtinErrors` (`topdown/eval.go:49, 2166`) accumulates errors and returns only the first unless `StrictBuiltinErrors(true)` is set, and the wasm target ignores strict mode altogether (`v1/rego/rego_wasmtarget_test.go:233`). **Anti-pattern** for any model-correction loop.

4. **Silent description degradation.** Letta's `validate_google_style_docstring` (`schema_generator.py:15-59`) raises `ValueError` but `generate_schema` only logs (`schema_generator.py:411-420`). Tools can ship with empty descriptions. **Caution sign:** validate_google_style_docstring is in the codebase but the generator entry point catches and logs.

5. **Positional-arg bypass of validation.** CrewAI's `_validate_kwargs` is gated on `if not args` (`base_tool.py:319`). Tools that overload `run(x, y)` skip validation entirely. **Anti-pattern** when validation is a security boundary (auth, billing, etc.).

6. **Schema immutability not enforced.** Pydantic-ai's `function_schema()` is the only source; agent-framework's Python side caches `_input_schema_cached` (`_tools.py:783-804`); langgraph reads the schema once and never re-validates it. **Caution sign:** no `__post_init__` re-validation; no round-trip test pinning the schema invariant.

7. **`default: None` stripping without notice.** OpenAI Agents SDK strips `default: None` in `ensure_strict_json_schema` (`strict_schema.py:121-122`); Pydantic-ai does the same in the OpenAI transformer (`profiles/openai.py:344-351`). **Anti-pattern** for callers that rely on `None` defaults.

8. **No rejection of injected-arg forgery.** Only langgraph's `ToolNode` strips `InjectedToolArg`-marked parameters before invocation (`tool_node.py:1424-1429`). Other frameworks trust the LLM-visible schema to omit injected args.

9. **Schema-version mismatch between persisted and runtime.** Letta warns: "if `source_code` changes without `json_schema` update, the schema is stale and the model will be told about arguments that no longer exist." (`letta/functions/schema_generator.py` future consideration.) CrewAI's `BeforeValidator(_deserialize_schema)` round-trips through persisted JSON but does not version-track.

10. **Cap on tools-with-strict for one provider.** Letta's `ANTHROPIC_MAX_STRICT_TOOLS = 15` (`anthropic_constants.py:13-27`) silently downgrades tools beyond the limit to non-strict mode. **Caution sign:** the cap is not surfaced in the schema-sending code path; users only learn about it when their strict-mode behavior breaks.

11. **Tool description not uniform between text and native paths.** CrewAI's `_generate_description` mutates `tool.description` in place (`base_tool.py:482-490`) but the native path reads `tool.description` and may strip or replace. **Anti-pattern** for any framework that uses description as a runtime semantic signal.

12. **Per-field metadata-stripping without preservation.** OpenAI Agents SDK's `ensure_strict_json_schema` strips `default: None` and several other keys (`strict_schema.py:18-149`). Pydantic-ai instead preserves them in `description` text (`profiles/openai.py:374-383`). **Caution sign:** a tool call that worked in pre-strict mode fails silently under strict because metadata was dropped without notice.

13. **No schema-shape unit tests.** OpenHands has extensive tests for secrets (`tests/unit/app_server/test_constants.py:1-163`) but none for tool JSON Schema shape. **Caution sign:** the test suite covers plumbing, FastAPI OpenAPI, and integration — but no test asserts a specific tool's `parameters` schema.

14. **Tests cover only the generator, not the validator end-to-end.** CrewAI has separate generator tests (`tests/utilities/test_converter.py:331-644`) and validator tests (`tests/tools/test_base_tool.py:250-352`), but only one test (`test_schema_validation_catches_missing_args_on_native_path` at `tests/agents/test_native_tool_calling.py:1340-1365`) asserts the native call path actually validates. **Anti-pattern.**

15. **Strict-mode per-provider without a unified default.** Agent-framework defaults Foundry strict but not OpenAI Hosting (`FoundryPromptAgentConverter.cs:131-135` vs `Tool.cs:78-80`). **Caution sign:** the same tool definition can have different strict-mode behavior across providers in the same framework.

## Notable Absences

- **No source** has a centralized schema-versioning system. Every source is one Pydantic or FastMCP or GRPC minor version away from silent schema output changes.
- **No source** has a "schema-debug" endpoint exposed for operators to inspect what the model sees. Pydantic-ai's Logfire telemetry is closest (`tests/test_logfire.py:557`) but it is debugging-by-telemetry, not a first-class UI.
- **No source** has a per-tool schema validator override that lets operators tighten constraints without changing source. The closest is openai-agents-sdk's pre-existing `FunctionTool(params_json_schema=...)` (`tests/test_function_tool.py:335-388`), but that is a one-shot override.
- **No source** exposes the round-trip schema through a wire test. Pydantic-ai has snapshot tests of transformed schemas; agent-framework has AG-UI round-trip tests; but no source has a "model saw the same schema X times in a row" assertion.
- **No source** documents how to migrate from strict mode to non-strict mode without redeploying. Pydantic-ai and openai-agents-sdk make the toggle per-tool/per-output-schema, but rollback procedures are not documented.
- **No source** has cross-provider schema-diffing (e.g. "the schema we send to OpenAI is X, to Anthropic is Y, they should match"). The per-provider transformer pattern is exactly what needs this kind of test but does not have it.
- **No source** has explicit `additionalProperties: false` enforcement at the framework level outside of strict-mode transforms. Pydantic-ai relies on the OpenAI transformer; crewai on the strict sanitizer; agent-framework on Foundry's default. The standard JSON Schema dialect without strict-mode has no enforcement at the framework boundary.
- **No source** has a "schema negotiation" protocol where the model tells the framework which JSON Schema features it supports (e.g. `oneOf`, `additionalProperties: false`). Per-provider capability maps are the closest (pydantic-ai's `is_strict_compatible`).

## Open Questions

1. **Schema versioning contract across providers.** None of the eight sources carries a `schema_version` field on emitted schemas; Pydantic v2 and FastMCP revisions can shift field ordering, defaults, or union handling silently. What is the migration story when a strict-mode provider's schema dialect changes?
2. **Cross-provider schema-diff tests.** Pydantic-ai has snapshot tests per provider; agent-framework has AG-UI round-trip tests. No source has a "schema-A sent to OpenAI vs. Anthropic should match for the same tool" test. Why is this not standard?
3. **Wasm / cross-runtime parity for strict mode.** OPA's wasm target ignores strict mode (`v1/rego/rego_wasmtarget_test.go:233`). What about language runtimes that strip reflection-based decorators (AOT for .NET, frozen modules for Python)?
4. **Description-text-vs-native-schema divergence resolution.** CrewAI documents the drift (text schema vs. native strict schema can disagree because of `strip_null_types`). What is the right resolution: align the text to the strict native, or remove the text rendering entirely when native is enabled?
5. **Injected-arg forgery protection outside langgraph.** Only langgraph strips caller-supplied injected keys. Other frameworks trust the LLM-visible schema to omit them. Is there a real attack surface here that warrants a framework-level defense?
6. **Per-provider capability for field-level `additionalProperties: false`.** Pydantic-ai's OpenAI transformer sets it; other transformers don't. What does the framework do when the model-visible schema says `additionalProperties: true` but the validator says `extra='forbid'`?
7. **What is the right path for "model sent `null` for unset field" under strict mode?** OpenAI Agents SDK strips `default: None`; Pydantic-ai preserves it in `description`. CrewAI's strict sanitizer drops `null` from `anyOf`. Which is least surprising?
8. **What does the LLM experience on a 15-tool Anthropic strict cap?** Letta downgrades silently past `ANTHROPIC_MAX_STRICT_TOOLS = 15`. Is there a test that asserts the model gets a coherent tool list under this constraint?
9. **Schema evolution for persisted tools.** Letta persists `tool.json_schema` and relies on the user re-adding the tool to refresh. CrewAI's `BeforeValidator(_deserialize_schema)` round-trips. Neither validates that the persisted schema matches the current source. Is there a test that detects drift?

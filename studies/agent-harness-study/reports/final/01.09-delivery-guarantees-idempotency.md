## Executive Summary

All four systems handle delivery guarantees by **composing primitives** rather than claiming a single end-to-end mode. They all converge on two design choices: a stable identifier per request/signalling unit (correlation ID, RequestID, event UUID), and a deduplication boundary that catches duplicates at the storage layer. They diverge on durability ambition: **temporal** layers RequestID-driven dedup + conditional writes + range-id-protected task IDs + replication clocks into a platform that is *effectively-once for its state machine* but explicitly at-least-once for activity execution; **agent-framework** delegates the entire story to Azure Durable Task and is therefore strong on the durable-agents path but offers no general-purpose idempotency for its in-memory agent loop; **langgraph** builds storage-level idempotency through composite primary keys and ON CONFLICT, exposing three durability modes (`sync`/`async`/`exit`) so users can trade crash-safety for throughput; **openhands** has the most heterogeneous picture — strong edge dedup on provider webhooks via Redis NX EX, last-writer-wins event storage by deterministic path, and at-most-once for everything else (pending messages, callbacks, automation forwards).

The headline distinction is **what the system treats as the unit of work that must be deduplicated**. temporal deduplicates *RPC calls into the state machine*; langgraph deduplicates *writes to a checkpoint*; openhands deduplicates *inbound webhooks* and *event files*; agent-framework deduplicates *signals to a Durable Entity*. None of them deduplicates user-execution side effects — this responsibility is pushed to the user uniformly.

## Core Thesis

A mature delivery-guarantee story is built from four layered primitives, and the four sources differ in how many they implement:

1. **A stable request/unit identifier** that survives retries — RequestID, correlationId, event id, or message id (all four implement this).
2. **A deduplication boundary** at the storage layer — conditional writes (temporal), composite PK + ON CONFLICT (langgraph), deterministic path (openhands event store), per-session entity serialization (agent-framework).
3. **A replay / retry mechanism** that does not double-execute the durable bookkeeping — deterministic replay (agent-framework orchestrator, temporal workflows), per-attempt write clearing + storage idempotency (langgraph), in-memory dedup tables rebuilt from events (temporal Updates).
4. **An explicit honesty about what cannot be guaranteed** — user-side effect idempotency and at-most-once for activity execution (temporal explicitly), at-least-once reliable streaming (agent-framework explicitly), in-process agent loop with no guarantees (agent-framework implicitly), external node side effects (langgraph implicitly).

Choosing a design is therefore a function of which primitive you need and at which layer. The platform-shaped sources (temporal, agent-framework Durable Task) invest in primitives 1–3 deeply but cannot control primitive 4. The application-shaped sources (langgraph, openhands) invert the priority: they implement primitive 2 strongly and lean on the user for primitives 3 and 4.

## Approach Models

The four sources fall into three approach clusters.

**Durable execution platforms (temporal, agent-framework-Durable-Task path).** These use *event-sourced, append-only state machines* with replay and stable RPC identifiers (RequestID / correlationId) attached to every public-facing API. Idempotency lives in conditional persistence writes (`INSERT ... IF NOT EXISTS`, `(create_request_id)` columns, range-id assertions) and in mutable-state dedup tables that are rebuilt from history on recovery. The user is responsible only for activity body idempotency. The framework ships with explicit "delivery semantics" docs and a published honesty about activity at-least-once. Pro: most complete model. Con: complexity, in-memory dedup tables that lose state on errors (temporal Updates, agent-framework entity state).

**Storage-engineered state-write idempotency (langgraph).** This approach pushes dedup *into the database schema* using a composite primary key `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` and asymmetric `ON CONFLICT` semantics: regular channel writes insert-nothing, control channels upsert. The retry layer clears per-attempt writes before invocation, so a partial state cannot persist past a failed attempt. Three durability modes (`sync`, `async`, `exit`) let callers choose the crash-safety vs throughput point. External side effects are entirely the user's problem. Pro: simplest correctness invariant ("one row per key"). Con: limited to state writes — node side effects not enveloped.

**Heterogeneous edge-subsystem mix (openhands).** No single abstraction. Provider webhooks use HMAC + Redis NX EX with 60s TTL; event storage uses deterministic filename `<event_id.hex>.json`; pending messages use delete-after-attempt; export uses a Redis distributed lock with TTL refresh; event callbacks have no dedup at all. Pro: each subsystem gets the cheapest correct primitive. Con: no project-wide contract, several subsystems are at-most-once or best-effort.

## Pattern Catalog

**Stable request identifier on every public-facing API.** All four sources ship this. Temporal applies it uniformly across `StartWorkflow`, `SignalWorkflow`, `UpdateWorkflow`, `PauseActivity`, `RecordWorkflowTaskStarted`, `RecordActivityTaskStarted`, Nexus completions, WorkerDeployment — every public RPC has a `RequestId` and a per-call dedup table. Agent-framework generates a `correlationId` per `RunRequest` (`dotnet/.../RunRequest.cs:37`, `python/.../_models.py:115`). Langgraph uses `(thread_id, task_id, idx)` as the natural identifier. OpenHands uses provider-supplied headers (`x-request_uuid`, `client_msg_id`, `object_attributes.id`) plus a SHA-256 fallback.
*When to copy:* Always. Without a stable identifier you cannot dedup.
*When it is overkill:* Trivial, one-shot, idempotent-by-construction operations (e.g., reading a value, computing a memoized function).
*Risk:* Identifier collision across runs (langgraph notes this for `(task_id)` alone; temporal counters with `run_id` and `RequestID` together).

**Storage-enforced dedup via composite primary key + ON CONFLICT clause.** Langgraph is the clearest example: Postgres `ON CONFLICT (...) DO NOTHING` for regular writes, `DO UPDATE` for control channels; SQLite uses `INSERT OR IGNORE` / `INSERT OR REPLACE`; InMemory does first-write-wins + overwrite per channel class (`libs/checkpoint-postgres/.../base.py:146-159`, `sqlite/__init__.py:463-465`). Temporal applies the same idea via conditional writes on `create_request_id`, `request_id` columns, and Cassandra `INSERT ... IF NOT EXISTS` (`common/persistence/cassandra/queue_v2_store.go:32`). OpenHands applies it as deterministic-path object storage (`event_id.hex` becomes the filename).
*When to copy:* Whenever the storage engine supports conditional semantics and the deduplication key is a natural property of the row.
*When it is overkill:* Ephemeral-only data without a stable key.
*Risk:* Different DB backends need different SQL — langgraph's Postgres vs SQLite vs InMemory asymmetry is a real maintenance hazard.

**Per-attempt write clearing combined with retries.** Langgraph's `task.writes.clear()` runs before each retry attempt in both `_retry.py:600-682` (sync) and `:714-838` (async). This means partial state from a failed attempt cannot persist; combined with storage-level PK dedup, the system has stable semantics even if a single attempt partially records.
*When to copy:* Whenever you have a retry loop that mutates a "result" object before committing — clear it per attempt to avoid leaking partial state.
*When it is overkill:* Pure stateless retries (HTTP with side effect outside the loop).

**Replay-aware side-effect guards.** Agent-framework's orchestrator gates `publish_live_status` (and any orchestrator side effect that should not re-run) on `ctx.is_replaying` (`_workflows/orchestrator.py:755-767`). Temporal achieves the same end via event sourcing: orchestrators replay history events rather than re-executing activities, so anything wrapped in an activity is automatically replay-safe.
*When to copy:* Any time a workflow re-runs its code path but must not re-issue external side effects.
*Risk:* Requires a runtime that exposes `is_replaying` (or equivalent). In agent-framework this is a `WorkflowOrchestrationContext` property — not all host runtimes expose it.

**"Treat-already-X-as-success" idempotency in workers and schedulers.** Temporal codifies this pattern across many code paths: worker deployment operations (`version_workflow.go:513,754`), scheduler migration tasks (`scheduler_migrate_task.go:227-231`), Nexus completion handler (`executors.go:739`). The principle: when a retry would land on an already-terminal state, return success rather than a generic error. Agent-framework uses it for tool registration (`_worker.py:113-115`) and approval replay (`test_approval_replay_is_blocked`). Langgraph uses it for `run.abort()` (`test_pregel_stream_events_v3.py:491`).
*When to copy:* Anywhere the work product is "set state X to terminal" — second arrivals should be no-ops.
*Risk:* A buggy retry loop can mask real failures by always succeeding.

**Stamp-and-clock mismatch rejection.** Temporal gives every history task a `Stamp` (and vector clock for replication). History rejects `RecordActivityTaskStarted` / `RecordWorkflowTaskStarted` calls where the stamp is stale (`service/history/api/recordactivitytaskstarted/api.go:178-184`). This is *not* dedup — it's "different decision, reject". It complements RequestID dedup.
*When to copy:* When a queueing layer may reschedule a task under a new decision while a worker has a stale handle.
*Risk:* Stamp generation must be coordinated across all replays of the same task.

**Edge dedup via Redis SET NX EX with provider-stable key.** OpenHands codifies this for every provider integration: GitLab NX 60s on `object_attributes.id` (`gitlab.py:107`); Bitbucket NX 60s on `(event_key, pr_id, comment_id, x_request_uuid)` (`bitbucket.py:99`); Slack NX 60s on `client_msg_id` (`slack.py:317`); Jira SETEX 300s on signature (`jira.py:316`). HMAC signature is verified *before* dedup is touched.
*When to copy:* Any inbound webhook ingestion surface.
*When it is overkill:* Trivial rest endpoints without provider retry behavior.
*Risk:* TTL must exceed the provider's redelivery window, otherwise dedup silently misses.

**Idempotent object-store writes via deterministic path.** OpenHands writes events to `<conversation_path>/<event_id.hex>.json` so duplicate writes coalesce at the FS/S3/GCS layer (`event_service_base.py:190-197`). This works because the event ID is generated by the producer; if two producers produce two events with the same UUID, one is silently overwritten.
*When to copy:* Append-only object stores where the natural key is stable.
*Risk:* No fencing tokens — concurrent writes from multiple replicas race in last-writer-wins. Torn writes (truncated JSON) appear as missing events on re-read.

**Atomic claim + scheduling via per-session entity.** Agent-framework's `DurableAgentContext` commits the state update and the scheduling in a single transaction (`DurableAgentContext.cs:80-87`). This is the closest the source comes to a transactional outbox. Temporal achieves the same effect differently: per-shard range-id assertion on every state-machine mutation (`service/history/shard/context_impl.go:1147-1199`).

**At-most-once via delete-after-attempt.** OpenHands pending-messages flow stores the row, POSTs to the agent server, and deletes whether or not the POST succeeded (`live_status_app_conversation_service.py:1997-2002`). The docstring states the contract: "deleted after processing, regardless of success or failure". This is acceptable here because messages that fail to POST are recoverable on retry from the producer (UI keeps a queue). It is the cheapest correct design when the upstream is the only consumer.
*When to copy:* When the producer is idempotent by construction and the consumer side has no value of replaying older messages.
*Risk:* A flaky network drops messages silently. Acceptable for some classes of input (chat) and unacceptable for others (commands).

**Redis distributed lock for single-flight long-running work.** OpenHands' conversation export uses `try_acquire_redis_lock` with TTL refresh (`redis_lock.py:25-53`). Falls open if Redis is unavailable when `export_lock_required=False`.
*When to copy:* Long-running operations (multi-minute) where duplicate execution would corrupt the destination.
*Risk:* Requires Redis; if Redis is missing the lock is silently absent.

**Replay-time reapply of previously persisted writes.** Langgraph's `_reapply_writes_to_succeeded_nodes` (`_loop.py:724-737`) attaches previously persisted writes to tasks with the same `task_id` on resume. Errors and interrupt writes are deliberately *not* re-applied so they re-fire or route to error handlers.
*When to copy:* Any system that suspends and resumes a long-running computation.
*Risk:* Filters must be carefully designed — wrong filter leaves stale state visible.

## Key Differences

**Stable identifier generation.** Temporal auto-generates RequestIDs server-side when the caller omits them (`service/frontend/workflow_handler.go:4210-4214`). Agent-framework generates `correlationId` per `RunRequest`, but only if the caller doesn't supply one. Langgraph uses `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` — a composite, not a single ID. OpenHands depends on *provider-supplied* headers and falls back to body-hashing when none are available.

**Where dedup happens.** Temporal: at the history engine via mutable-state maps + conditional persistence. Langgraph: in storage via ON CONFLICT — never in the application code. Agent-framework: in the Durable Entity (per-session) and in the orchestrator (replay). OpenHands: in Redis NX EX (webhooks), in the OS/object store (events), nowhere (callbacks, pending messages).

**Durability modes vs fixed defaults.** Langgraph uniquely exposes three durability modes (`sync`, `async`, `exit`) that change when writes hit the checkpointer (`types.py:87-93`). Temporal fixes the model: state-machine events are always persisted; only user activity is at-least-once. Agent-framework fixes the model: entity writes are atomic; orchestration side effects are gated on replay. OpenHands varies per subsystem.

**At-least-once honesty.** Temporal is the most explicit: `docs/architecture/README.md:35` says activities must be idempotent or non-retryable. Agent-framework's reliable-streaming samples document at-least-once verbatim (`07_ReliableStreaming/README.md:152`). Langgraph never claims exactly-once; the conformance test (`test_put_writes_idempotent`) pins the storage-level invariant. OpenHands never claims exactly-once anywhere — the contract is reconstructed from inline comments and docstrings.

**Failure modes for in-flight state loss.** Temporal: per-call dedup table rebuilt from `UpdateInfo` events on registry clear; lost updates surface as `WorkflowUpdateAbortedErr`. Langgraph: in-process `_delta_write_futs` lost on crash under `durability="async"`; subsequent resumes do not redeliver. Agent-framework: worker crash between agent invocation and state persistence re-runs the agent operation with a fresh `correlationId`. OpenHands: at-most-once everywhere outside the dedup windows.

**Cross-cluster / cross-process semantics.** Temporal has a documented replication story using vector clocks (`service/history/ndc/replication_task.go:118-236`) and `maxRxProcessedTimestamp`. The other sources treat single-process execution as the model; langgraph explicitly notes that distributed execution semantics are not implemented in OSS.

## Tradeoffs

**Storage-enforced vs application-enforced idempotency.**
Langgraph and (for some flows) temporal push idempotency into the storage layer via conditional writes. Application code cannot accidentally break the contract: a duplicate `put_writes` is a no-op at the SQL boundary. The cost is a tight coupling to a particular DB feature set — `ON CONFLICT` semantics differ between Postgres and SQLite (langgraph maintains two implementations and pins behaviour via the conformance suite). Agent-framework and OpenHands keep dedup in application code (entity op hash, Redis client) — easier to evolve, but the responsibility is on every callsite.

**Per-call RequestID table vs single global idempotency store.**
Temporal maintains per-API dedup tables (signal-requested-IDs, Update registry, signal-with-start map, callback table, deployment request set). This is correct for high-throughput, isolated APIs but does not support a generic "idempotency key" contract. A real generic store (à la Stripe) would unify them. Cost: a separate map per feature, each with its own eviction policy.

**In-memory dedup tables rebuilt from events.**
Temporal's Update registry plus `UpdateInfo` event log is the canonical example. Agent-framework keeps `DurableAgentStateRequest`/`Response` in conversation history. This pattern gives O(1) lookup under load and survives process restart by replaying events. Cost: the gap between "admitted in registry" and "persisted to history" is a window of data loss on crash — acknowledged in `docs/architecture/workflow-update.md:70-83`.

**Per-attempt write clearing (langgraph) vs durable history replay (temporal).**
Langgraph clears per-attempt writes to keep state-clean under retry; the cost is no automatic persistence of partial progress — a crashed attempt is fully lost. Temporal replays history events, so activity completion is durable but the activity itself runs again. These are different ways of solving the same problem: "what do I do between retry attempts". Langgraph's approach assumes the user wants simple "node = atomic unit"; Temporal's assumes the user wants durable, retryable activities.

**Conditional writes via IF NOT EXISTS vs UPSERT vs range-id assertion.**
Cassandra `INSERT ... IF NOT EXISTS` returns no row on duplicate (temporal Queue V2). Postgres `INSERT ... ON CONFLICT DO NOTHING` (langgraph) is similar but side-channel `ON CONFLICT DO UPDATE` is needed for control channels. SQLite uses `INSERT OR IGNORE` / `INSERT OR REPLACE`. Range-id assertion (temporal shards) lives at the shard-ownership layer, not the row layer. Each has different failure profiles; langgraph's conformance suite is the only one that pins this across backends.

**Per-attempt determinism vs cached replay.**
Agent-framework's orchestrator uses `context.NewGuid()` for replay-safe IDs and `context.current_utc_datetime` for time (`context.py:39-93`). Temporal uses vector clocks and stamps across the entire history. Langgraph uses pickle-hashing of `(args, kwargs)` for cache keys (`_internal/_cache.py:7-31`). Each imposes different constraints on user code (deterministic idempotency).

**Edge dedup TTL vs unbounded dedup.**
OpenHands uses bounded TTL (60–300s) for webhooks — limits storage growth and matches provider redelivery windows. Temporal keeps dedup rows until the workflow retention period elapses. Langgraph keeps checkpoint writes for the lifetime of the thread. Bounded TTL is correct when the producer is well-behaved; unbounded TTL is correct when retried calls can land days later.

## Decision Guide

**You are building a workflow / orchestration engine.** Copy temporal's pattern: stable RequestID per RPC, conditional persistence writes, range-id or shard-assertion to fence ownership, rebuild dedup tables from history. Expect to maintain per-API dedup tables; budget for in-memory loss windows (Updates, signals-admitted-but-unsent).

**You are building a state-machine-style library with resumable execution.** Copy langgraph's pattern: composite primary key on writes, asymmetric `INSERT` policy per channel class, per-attempt write clearing, three durability modes documented at the API surface. Test with a conformance suite that pins behaviour across backends. External side-effect idempotency stays with the user; document this clearly.

**You are building an SDK on top of a durable-execution platform (e.g., agent-framework on Durable Task).** Delegate idempotency to the platform (Durable Entities, replay-aware orchestrator) and add the SDK-specific glue: correlation-id generation, replay-aware side-effect guards, deterministic new-id helpers. Be explicit about which paths are platform-deduplicated and which are not — agent-framework's biggest weakness is that the in-process agent loop has no guarantees while the Durable Agents path is well-specified.

**You are building a webhook ingestion service.** Copy openhands's pattern: HMAC verification, then Redis NX EX with the most stable provider-supplied identifier, plus a SHA-256 body fallback. Document the TTL relative to provider redelivery windows. Decouple the slow work from the response via background tasks. Be ready to add an idempotency key header per destination — read `X-GitHub-Delivery` even if you do not currently use it.

**You are building any system that stores events in object storage.** Use deterministic, content-independent paths (`<conversation_id>/<event_id>.json`) so duplicate writes coalesce by construction. Use conditional writes (`IfNoneMatch`, `if_generation_match`) where supported. Be aware that last-writer-wins is silent — torn writes disappear from the read path (OpenHands `Event.model_validate_json` swallows parse errors in `_load_event`).

**You are building long-running operations that must not run concurrently.** Use a TTL-refreshed Redis distributed lock (OpenHands `try_acquire_redis_lock`), and document the "fall-open" behaviour under Redis outage explicitly.

## Practical Tips

- Always emit or accept a stable request identifier at every public-facing API. If users omit it, generate one server-side and document that.
- Push dedup into the storage layer whenever the storage engine supports conditional writes or composite keys. Application-level dedup is harder to keep correct.
- Make the retry loop clear per-attempt intermediate state (langgraph `task.writes.clear()`). Combined with storage-level idempotency, it gives a clean retry semantic.
- For orchestrations that re-run their code path, expose an `is_replaying` (or equivalent) API so orchestrator-authored side effects can gate on it.
- When you return "success on already-terminal", say so explicitly — silent success masks real failures.
- For inbound webhooks, verify the signature *before* dedup logic so a malicious actor cannot waste your dedup store.
- Decouple slow work from the HTTP response (Background tasks in FastAPI, durable orchestration in Temporal). The dedicated ack-vs-work pattern is the cheapest correctness primitive for ingestion.
- When storing events in object stores, write to deterministic paths and consider conditional writes to prevent torn-file leaks.
- Treat user-side-effect idempotency as a deliberate API shape, not an implementation detail. Document it.
- For per-call dedup tables, define the rebuild-from-history contract precisely: temporal calls this out for Updates; agent-framework does it for conversation history.
- For multi-cluster / multi-shard systems, vector clocks or transition counts are necessary to detect replays; RequestID alone is insufficient.

## Anti-Patterns / Caution Signs

- **Background tasks that swallow errors and return 200.** OpenHands' `_run_callbacks_in_bg_and_close` and Slack / Bitbucket DC `background_tasks.add_task` paths run callbacks and forget about errors. Failures become invisible. Use an explicit outbox + retry or at minimum a logging surface.
- **Dedup TTL shorter than the producer's redelivery window.** OpenHands chose 60s for most providers; Jira uses 300s. If the provider takes longer than TTL to redeliver, dedup silently misses and the work runs twice. Do not pick TTLs from a default — measure the provider.
- **No fencing tokens on object-store writes.** OpenHands `save_event` does last-writer-wins on `event_id.hex`. Concurrent replicas race silently. Use conditional writes (`IfNoneMatch: *`, `if_generation_match`) where possible.
- **Generate a fresh `correlationId` per retry at the transport layer.** Agent-framework defaults to this. A caller retrying a 5xx will get a new ID and the agent runs again; history grows. Make the caller pass through the original ID.
- **Use of `add_agent` / `register_agent` with inconsistent strictness across hosts.** agent-framework's `AgentFunctionApp.add_agent` silently skips duplicates (`_app.py:572-576`); `DurableAIAgentWorker` raises (`_worker.py:113-115`). Pick one policy.
- **Two-tier idempotency with silent asymmetry.** langgraph's conformance suite exists precisely because Postgres `DO NOTHING` and SQLite `INSERT OR IGNORE` are not identical. Without tests, drift is invisible.
- **Blaming the user for activity idempotency without saying so.** Temporal does say this (`docs/architecture/README.md:35`). Many systems do not. The user must know that retry + side-effect ⇒ potential duplication.
- **In-memory dedup tables without documented loss windows.** Agent-framework and Temporal both have explicit "lost on registry clear" or "lost on worker crash before ack" cases. Both document them. If your system has such windows, document them or make them persistent.
- **Default durability mode in agent-orchestration that loses pending writes on crash.** Langgraph's `durability="async"` does this and the docs acknowledge it. A user who picks it without reading the docs will be surprised. Default to `sync` unless you have a measured reason.
- **Treating `task_id` as a cross-run identity.** Langgraph dedups `(task_id, idx)` within a checkpoint but not across checkpoints. Without a `run_id` in the dedup key, two runs of the same thread will have separate dedup spaces.
- **Composite dedup keys without a conformance test.** Langgraph pins behaviour with `test_put_writes_idempotent`; without similar tests, key semantics drift across DB backends.
- **Pending messages that are deleted before consumer-side ack.** OpenHands's pending-message loop deletes whether or not the POST succeeded. A single network blip drops user input. Acceptable for chat, not for commands.

## Notable Absences

- **A single top-level "Delivery Guarantees" page that consolidates retry + durability + cache + replication.** None of the four sources has one. temporal comes closest with `docs/architecture/README.md:35` for activities and `docs/architecture/workflow-update.md` for updates; agent-framework has `docs/features/durable-agents/README.md:127` for orchestrator replay. The story is distributed.
- **A generic per-API idempotency key store (Stripe-style).** Temporal manages per-API dedup tables; agent-framework uses correlationId only on the durable-agent path. There is no single "if I send you the same `idempotency_key` twice, you do it once and return the same response" abstraction in any source.
- **Idempotency for user-execution side effects in langgraph nodes.** Users have to bring their own. This is documented as a gap for future work.
- **Fencing tokens on object-store writes.** OpenHands uses deterministic paths but no conditionals; torn writes appear as missing events.
- **Distributed execution semantics in OSS.** Langgraph explicitly notes the runner assumes a single in-process event loop; there is no cross-worker coordination in the OSS library.
- **At-most-once delivery for callback fan-out.** OpenHands callbacks re-execute for each redelivery; there's no `(event_id, callback_id)` dedup table.
- **Cross-checkpoint `task_id` uniqueness in langgraph.** Treated as a per-checkpoint constraint.
- **Retry budget / cap on outbound callbacks.** Temporal retries continue until success, DLQ, or workflow retention period — there is no explicit retry cap beyond the circuit-breaker.

## Per-Source Notes

**agent-framework (rating 6, framework-strong-Durable-Entities-8).** Delegates to Durable Task for replay + entity serialization. Has correlation-id pairing, replay-aware side-effect guards, and a documented at-least-once reliable-streaming story with reader-side dedup expectation. Weak: in-process agent loop has no dedup, no general idempotency key on the request path, conversation history grows unbounded across retries, different packages have different duplicate-handling strictness for `add_agent` / worker registration. Test coverage at `python/packages/durabletask/tests/test_durable_entities.py:398-444,517-583` is the strongest durability evidence.

**langgraph (rating 7).** Storage-engineered composite-PK ON CONFLICT, asymmetric per-channel-class insert policies, per-attempt write clearing, three durability modes (`sync`/`async`/`exit`), bounded LRU event-id dedup in SDK. Conformance suite `test_put_writes_idempotent` pins the storage contract. Weak: no general-purpose idempotency-key for user nodes; default `async` durability loses pending writes on crash; cache TTL interacts with retries in a way not tested across crash injection.

**openhands (rating 5).** Most heterogeneous. Strongest: HMAC + Redis NX EX pattern with provider-supplied keys (GitLab `object_attributes.id`, Bitbucket `(event_key, pr_id, comment_id, x_request_uuid)`, Slack `client_msg_id`, Jira SETEX on signature). Event storage idempotent by deterministic path. Export single-flight via Redis distributed lock with TTL refresh. Weakest: pending messages delete-after-attempt (explicit at-most-once), event-callback processors have no per-(callback, event_id) dedup table, `on_event` webhook returns 200 on exception so redeliveries silently repeat, GitHub integration has no dedup layer at all.

## Executive Summary

Tool-calling roundtrip control is the most architecturally defining surface across these seven agent harnesses, and it splits into two camps: **end-to-end in-loop frameworks** (six sources) that own extraction, validation, execution, result formatting, repair, and approval — and **control-plane frameworks** (openhands) that delegate the loop to an external SDK and only configure/observe it. Among the in-loop frameworks, four broad design families emerge: **chat-client-layer wrapping** (agent-framework), **graph-orchestrated pipelines** (langgraph), **typed-parts validate-then-execute** (pydantic-ai), and **dual-path legacy+native with validator ladders** (crewai). Two sources close to the OpenAI Responses API wire format (openai-agents-sdk) or to a sandboxed-services canonical shape (letta) carve out a stricter, more provider-coupled niche.

Across the six in-loop frameworks, the highest-scoring ones (pydantic-ai 9, agent-framework/langgraph/openai-agents-sdk 8) share four traits: **typed content shapes for every tool state**, **Pydantic-driven argument validation**, **explicit per-tool budgets** (retries, concurrency, timeouts), and **first-class approval flows**. The mid-scoring sources (crewai 7, letta 6) trade those guarantees for permissive parsers, hook-based controls, and multi-version flexibility — choices that surface as silent failures, drift between implementations, or missing tests at the parsing seams.

The key insight for practitioners: the framework you choose predetermines whether malformed tool calls **kill the run, get repaired automatically, get fed back to the model as a typed error, or silently become `{}`**. Openhands scores 3/10 not because it is poorly implemented, but because the dimension barely lives in the repo under study — a cautionary tale about studying harnesses without including their agent runtimes.

## Core Thesis

**Tool-calling roundtrip design is fundamentally a choice about who owns tool-call *state*.** The mature frameworks (pydantic-ai, agent-framework, openai-agents-sdk) push every aspect of tool-call state — calls, results, errors, approvals — into typed content objects that the model can react to and the framework can serialize. Less mature ones (letta, crewai) keep that state spread across text-parsing, regex extraction, and unstructured string observations — flexible but harder to debug and easier to silently lose.

Three orthogonal axes separate the implementations:

1. **Where the JSON parse happens**: at the *adapter boundary* (langgraph delegates to `langchain_core`), at *transport-decoded-with-repair* (pydantic-ai's `INVALID_JSON` sentinel, letta's `_safe_load_tool_call_str`), or by the framework itself with multi-strategy fallbacks (crewai's 5-strategy ladder, agent-framework's `Content.parse_arguments`).
2. **How errors are surfaced**: as typed `ToolMessage(status="error")` (langgraph), `Content(type="function_result", exception=...)` (agent-framework), `RetryPromptPart` (pydantic-ai), `ToolExecutionResult(status="error")` (letta), or as bare string observations (crewai's text mode). Typed surfaces are easier to test; string observations are easier to bolt on.
3. **How approval is modeled**: as a content type alongside tool calls (agent-framework, pydantic-ai deferred, openai-agents-sdk `ToolApprovalItem`), as a pre-execution hook callback (crewai), as a graph-level interrupt (langgraph), or as a per-agent policy class (openhands). Content-type approvals are the only ones that survive a checkpointed resume cleanly.

The frameworks that converge on typed content + Pydantic validation + per-tool budgets + content-type approval are observably more robust under malformed input and partial failures. Frameworks that rely on parser ladders and global hooks retain flexibility but trade away the model's ability to self-correct.

## Approach Models

The seven sources cluster into five approach models, ordered by **how much of the roundtrip the framework itself owns**.

### A. Typed-Parts Validate-Then-Execute (pydantic-ai)

The agent loop is a `pydantic_graph` graph with `CallToolsNode` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1046-1382`), and every tool state is a Pydantic typed part — `ToolCallPart`, `ToolReturnPart`, `RetryPromptPart`, `NativeToolCallPart` (`pydantic_ai_slim/pydantic_ai/messages.py:1771-1955`). The framework owns promotion from base `BaseToolCallPart` to typed subclasses via a registry (`_parts_manager.py:86-97`). Validation and execution are separate methods (`ToolManager.validate_tool_call`/`execute_tool_call`, `tool_manager.py:419-501`) sharing a `ValidatedToolCall` carrier (`tool_manager.py:46-74`).

- **Why it wins**: Eight orthogonal safety knobs (`max_retries`, `timeout`, output retries, usage limits, concurrency, sequential flag, defer_loading, requires_approval`); single `RetryPromptPart` retry channel; transport-layer `INVALID_JSON` repair.
- **Cost**: Two parallel resume code paths (`_resolve_single_deferred` vs `_call_tool`) with explicit "keep in sync" comments (`tool_manager.py:861-863`); capability-hook layer adds indirection.

### B. Chat-Client-Layer Wrapping (agent-framework)

The roundtrip sits in `FunctionInvocationLayer` (`python/packages/core/agent_framework/_tools.py:2388-2877`) layered on top of any `BaseChatClient` subclass (`_clients.py:978-996`). Every state is a `Content(type="function_call"|"function_result"|"function_approval_*")` item (`_types.py:338-363`). The loop wrapper catches validation errors and emits a `function_result` carrying `"Error: Argument parsing failed. …"` (`_tools.py:1519-1528`) so the model sees structured failure rather than crash.

- **Why it works**: Approval is a content type (`function_approval_request`/`function_approval_response`, `_types.py:1213-1253`) so it survives serialization; mixed-batch routing hides non-required approval requests behind session state (`_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY`, `_tools.py:94, 1974-2032`); orthogonal budget knobs (`max_iterations`, `max_function_calls`, `max_consecutive_errors_per_request`).
- **Cost**: Dual Pydantic + JSON-Schema validators with different error messages (`_tools.py:668-669` vs `:1109-1141`); `Content.parse_arguments` silently swallows JSON errors (`_types.py:1577-1578`); `terminate_on_unknown_calls=True` raises `KeyError` instead of returning error result.

### C. Graph-Orchestrated Pipeline (langgraph)

The roundtrip is compiled into a `StateGraph` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:278-1002`) alternating between an `agent` node and a `tools` node built from `ToolNode` (`libs/prebuilt/langgraph/prebuilt/tool_node.py:622-1579`). The prebuilt layer does **no JSON parsing** — `langchain_core` model adapters convert the emitted string to a dict before `tool_node` sees it (`tool_node.py:63-70`). Validation happens inside `tool.invoke`; `ValidationError` is caught and converted to `ToolMessage(status="error")` (`tool_node.py:1006-1012`).

- **Why it works**: `wrap_tool_call`/`awrap_tool_call` is an open extension point (`tool_node.py:202-283`); `_filter_validation_errors` strips `InjectedState`/`InjectedStore`/`ToolRuntime` errors so the model only sees fields it controls (`tool_node.py:510-563`); multi-strategy `handle_tool_errors` with type-introspected callables (`tool_node.py:394-507`); `_inject_tool_args` strips LLM-supplied values for injected keys before re-injecting trusted values (`tool_node.py:1421-1430`).
- **Cost**: No retry/repair inside the roundtrip — failures become `ToolMessage(status="error")` and trust the model to fix it; `ValidationNode` deprecated (`tool_validator.py:43-46`); graph-level `interrupt_before=["tools"]` is the only approval.

### D. Dual-Path Legacy + Native (crewai)

The roundtrip exists in two parallel paths. The text/ReAct path goes through `ToolUsage` (`lib/crewai/src/crewai/tools/tool_usage.py:76`) which detects calls via `ACTION_INPUT_REGEX` + `_safe_repair_json` (`agents/parser.py:161-178`), validates via `_validate_tool_input`'s 5-strategy ladder (`tool_usage.py:884-930`), executes via `tool.invoke`/`tool.ainvoke`, and returns the observation as a string (`utilities/agent_utils.py:616`). The native path goes through `_execute_single_native_tool_call` (`agents/crew_agent_executor.py:868-1071` or `experimental/agent_executor.py:1860-2073`) which extracts `(call_id, func_name, func_args)` via `extract_tool_call_info` (`utilities/agent_utils.py:1191-1231`) covering OpenAI, Anthropic, Gemini, and Bedrock shapes.

- **Why it works**: 5-strategy validation ladder (`json.loads` → `ast.literal_eval` → `repair_json` → `json5.loads` → `repair_json(skip_json_loads=True)`) — every step has a dedicated test (`tests/tools/test_tool_usage.py:264-543`); distinct event types (`ToolUsageStartedEvent`/`ToolUsageFinishedEvent`/`ToolUsageErrorEvent`/`ToolValidateInputErrorEvent`/`ToolSelectionErrorEvent`, `events/types/tool_usage_events.py:56-90`); atomic usage enforcement via `BaseTool._claim_usage` lock (`tools/base_tool.py:295-312`); contextvars-safe parallel execution (`crew_agent_executor.py:748-749`).
- **Cost**: Two near-duplicate `_execute_single_native_tool_call` implementations with their own event payload fields; silent argument filtering strips unknown keys from `args_schema` (`tool_usage.py:332-339, 570-579`) without warning; approval only via global `before_tool_call` hooks, not declarative per-tool metadata.

### E. Responses-API Typed-Item Routing (openai-agents-sdk)

A `ModelResponse` arrives with heterogeneous output items (`src/agents/run_internal/turn_resolution.py:1551-1949`); each is classified and routed into a dedicated run container (`ToolRunFunction`, `ToolRunComputerAction`, `ToolRunShellCall`, `ToolRunCustom`, `ToolRunApplyPatchCall`, `ToolRunLocalShellCall`, `ToolRunMCPApprovalRequest`) at `src/agents/run_internal/run_steps.py:62-130`. Function tools are dispatched in parallel inside `_FunctionToolBatchExecutor` (`src/agents/run_internal/tool_execution.py:1358-1970`) with deterministic concurrent-failure arbitration (`_FunctionToolFailure`, `tool_execution.py:173-289`) and bounded post-invoke and cancellation drains.

- **Why it works**: Strict JSON schema by default (`ensure_strict_json_schema`, `src/agents/strict_schema.py:18-149`) with `oneOf` rewritten to `anyOf` and `additionalProperties: false`; two-layer error handling (guarded `_parse_function_tool_json_input` raises `ModelBehaviorError`, then Pydantic validation) routed via `default_tool_error_function` for recoverable strings (`src/agents/tool.py:1575-1618`); per-tool input/output guardrails (`tool_execution.py:2337-2406`); `ToolApprovalItem` interruptions preserve approval state across HITL pause (`items.py:508-580`); `ToolExecutionPlan` is a turn-scoped dispatch table that makes resume logic straightforward (`run_planning.py:177-300`).
- **Cost**: No pluggable repair pipeline — failures either crash the turn via `ModelBehaviorError` or inject a recovery string; approval state lives in-memory on `RunContextWrapper` unless explicitly persisted via `RunState`.

### F. OpenAI-Canonical Sandboxed Execution (letta)

Tool calls follow OpenAI's wire format (`OpenAIToolCall(id, function.name, function.arguments)`, `letta/schemas/openai/chat_completion_response.py:16-22`). The streaming layer accumulates partial state, the loop applies `_safe_load_tool_call_str` (`letta/agents/helpers.py:378-393`) which is lenient — strips concatenated JSON artifacts and demotes failures to `{}`. Execution is mediated by `ToolExecutionManager` (`letta/services/tool_executor/tool_execution_manager.py:68-160`) which dispatches to a `ToolExecutor` subclass per `ToolType` (`LETTA_CORE`/`LETTA_MEMORY_CORE`/`LETTA_SLEEPTIME_CORE`/`LETTA_MULTI_AGENT_CORE`/`LETTA_BUILTIN`/`LETTA_FILES_CORE`/`EXTERNAL_MCP`/sandbox default). Each executor returns `ToolExecutionResult(status, func_return, agent_state, stdout, stderr, sandbox_config_fingerprint)` (`letta/schemas/tool_execution_result.py:8-18`). Approval runs through `RequiresApprovalToolRule` (`letta/schemas/tool_rule.py:348`).

- **Why it works**: `ToolRulesSolver` is a pure-function policy engine that runs on every tool call (`is_requires_approval_tool`, `is_continue_tool`, `get_uncalled_required_tools`; `letta/helpers/tool_rule_solver.py:186-208`); OpenAI strict-mode schema healing pairs `SchemaHealth` classifier with deep property rewriting (`letta/functions/schema_validator.py:12-202`, `letta/helpers/tool_execution_helper.py:89-162`); tool-result truncation has per-tool `return_char_limit` with an opt-out list for search tools (`letta/agents/letta_agent_v3.py:1878`); trace-driven observability via `@trace_method` and `MetricRegistry` histograms.
- **Cost**: `_safe_load_tool_call_str` returns `{}` silently on JSON failure; missing `tool_call_id` at stream end raises `ValueError` swallowed by outer `try/except` (`letta/adapters/letta_llm_stream_adapter.py:184-187`); three concurrent agent loop implementations (`letta/agent.py`, `letta_agent.py`, `letta_agent_v2.py`, `letta_agent_v3.py`); no unit tests for `_safe_load_tool_call_str`, `_extract_tool_calls`, `merge_and_validate_prefilled_args` failure modes.

### G. Control-Plane Configuration Only (openhands)

The OpenHands V1 app-server does **not** implement the in-loop roundtrip; the actual mechanics live in `software-agent-sdk`. This repo participates only at the edges — toolset wiring (`get_default_tools`/`get_planning_tools`, `live_status_app_conversation_service.py:1617-1625`), MCP server hosting (`mcp_router.py:43, 72, 147-487`), confirmation policy selection (`_select_confirmation_policy`, `app_conversation_service_base.py:641-652`), secret-name validation (`validate_secret_name`/`validate_secrets_dict`, `constants.py:44-169`), and observation reconciliation (`webhook_router.py:453-522`).

- **Why it works as a control plane**: Hard split between control plane and agent runtime means the app-server cannot crash or stall a tool call directly; SDK version handshake (`_verify_agent_server_version`, `live_status_app_conversation_service.py:176-181`) catches the failure mode where a custom sandbox image ships an SDK newer/older than what the app-server expects; MCP server uses `FastMCP(..., mask_error_details=True)` so internal exceptions don't leak to the model.
- **Why it scores 3/10 on this dimension**: There is no parser, no argument validator, no executor wrapper, no result formatter, no in-band tool-error handling, and no per-call approval gate inside this source tree. The rating reflects what the dimension asks about, not what the repo is for.

## Pattern Catalog

### Pattern 1: Typed Content for Every Tool State

**Problem**: Without typed content, parsing errors leak into model context as `"<something at 0x...>"`, retries have no structured identity, and serialization round-trips break.

**Sources**: agent-framework, pydantic-ai, openai-agents-sdk, langgraph (typed `ToolCallRequest`/ToolMessage), letta (typed `OpenAIToolCall`/`ToolExecutionResult`).

**Why it works**: Once every state is a typed object, the loop can route by type (`process_tool_calls` in pydantic-ai, `process_model_response` in openai-agents-sdk, `_extract_function_calls` in agent-framework), serialize round-trippable checkpoints (`AgentState.remaining_steps` in langgraph), and emit structured events (`ToolUsageFinishedEvent` in crewai).

**When to copy**: Always. The single most important decision in tool-calling roundtrip design is to define a stable typed shape for *at least* the tool-call request and tool-result message — start with `dataclass` or `Pydantic BaseModel`.

**When overkill**: Greenfield libraries with one tool type and no checkpointing. The cost is more boilerplate per PR; if you have one tool, a `dict` is fine.

**Evidence**: `agent_framework/_types.py:794-874` (function_call/function_result/function_approval_*), `pydantic_ai/messages.py:1771-1955` (BaseToolCallPart/ToolCallPart/NativeToolCallPart/ToolReturnPart/RetryPromptPart), `openai_agents/items.py:347-580` (ToolCallItem/ToolCallOutputItem/ToolApprovalItem), `langgraph tool_node.py:132-199` (ToolCallRequest), `letta/schemas/openai/chat_completion_response.py:16-22` (OpenAIToolCall).

### Pattern 2: Strict Pydantic Validation with Lenient JSON Parse at the Edges

**Problem**: Models emit malformed JSON sometimes; tools want strict schema; users get confused when the parse "succeeds" but the schema rejects.

**Sources**: pydantic-ai uses `args_as_dict(raise_if_invalid=False)` with Anthropic-recommended `INVALID_JSON` sentinel (`messages.py:1826-1850`) and Pydantic `SchemaValidator` (`_function_schema.py:103-281`). Agent-framework uses `Content.parse_arguments` which falls back to `{"raw": original_str}` on `JSONDecodeError` (`_types.py:1577-1578`), then runs Pydantic via `tool.input_model.model_validate(parsed_args)` (`_tools.py:649-655`). OpenAI Agents SDK uses a guarded parser that raises `ModelBehaviorError` (`src/agents/tool.py:1575-1598`) then Pydantic validation.

**Why it works**: The lenient layer survives the model's typos; the strict layer rejects semantically wrong args. The model then sees *one* error message — either a `RetryPromptPart` (pydantic-ai), a `function_result(error=...)` (agent-framework), or a `ToolMessage(status="error")` (openai-agents-sdk) — that can be traced back to a specific field.

**When to copy**: Always. The lenient pass should never raise — it should always produce something the strict layer can reject with a useful message.

**When overkill**: Single-tool systems where the developer owns both ends and can guarantee JSON output.

**Evidence**: `pydantic_ai/messages.py:36-40` (INVALID_JSON sentinel), `agent_framework/_tools.py:649-669, 1510-1528` (dual validation), `openai_agents/tool.py:1575-1618` (guarded parser + `default_tool_error_function`).

### Pattern 3: Per-Tool Budgets (Retries, Timeouts, Concurrency)

**Problem**: A single misbehaving tool can exhaust a global budget, kill the whole run, or block siblings in a parallel batch.

**Sources**: pydantic-ai uses `ctx.retries: dict[str, int]` per tool (`_run_context.py:60-61`) and per-tool `timeout`, `max_retries` (`tools.py:467-580`); agent-framework uses `errors_in_a_row` counter (`_tools.py:1855, 2248-2258`) that increments on exceptions and flips `tool_choice="none"` after `max_consecutive_errors_per_request` (`_tools.py:2632-2644`); openai-agents-sdk uses deterministic failure arbitration (`_FunctionToolFailure`, `tool_execution.py:173-289`) with bounded cancellation drain (0.25s); langgraph uses `AgentState.remaining_steps: int = 25` default (`chat_agent_executor.py:60-74`); letta uses `Tool.return_char_limit` and `MAX_ERROR_MESSAGE_CHAR_LIMIT` per tool.

**Why it works**: Per-tool budgets isolate misbehavior. pydantic-ai's `ctx.retries` means one bad tool can exhaust its own 5 retries without affecting siblings; agent-framework's `errors_in_a_row` flips `tool_choice="none"` so the next iteration must produce text, breaking the broken-call loop deterministically; openai-agents-sdk's arbitration prefers `Exception > asyncio.CancelledError` so the most informative failure wins.

**When to copy**: When you have any tool whose failure mode could cascade — i.e., always for production systems.

**When overkill**: Read-only agent demos where any failure aborts the run anyway.

**Evidence**: `pydantic_ai/tool_manager.py:177-181, 419-465` (`_check_max_retries`, `validate_tool_call`), `agent_framework/_tools.py:2248-2270, 2632-2644` (`errors_in_a_row` + `tool_choice="none"`), `openai_agents/run_internal/tool_execution.py:173-289` (failure arbitration), `openai_agents/tool.py:1806-1847` (per-tool timeout via `asyncio.wait_for`).

### Pattern 4: Approval as a Content Type

**Problem**: When approval is "make a callback, then resume", a crashed run loses the approval state. When approval is a separate HTTP request, the model loses context.

**Sources**: Agent-framework uses `Content(type="function_approval_request")` and `Content(type="function_approval_response", approved)` (`_types.py:1213-1253`) — both flow in the assistant message stream and survive serialization round-trips. Pydantic-ai uses `DeferredToolRequests(approvals=[...])` as the agent output (`tools.py:256-326`) and `DeferredToolResults` as the next-run input (`tools.py:378-421`), with `ToolApproved.override_args` replacing the model's args before re-validation. OpenAI Agents SDK uses `ToolApprovalItem` interruptions (`items.py:508-580`).

**Why it works**: Content-type approvals carry the full call context (name, args, call_id) in the same serialization format as tool calls, so checkpointing "just works". Mixed-batch routing (agent-framework) hides already-approved calls in session state (`_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY`, `_tools.py:94, 1974-2032`) so users only see one approval prompt per batch.

**When to copy**: Whenever tool calls have side effects you don't trust. The cost is one extra content type and one approval-response code path — small relative to the durability benefit.

**When overkill**: Internal-only tools with no destructive side effects; chat-style demos where the developer runs everything manually.

**Evidence**: `agent_framework/_types.py:1213-1253, 1712-1718` (`function_approval_request` content type + loop short-circuit), `pydantic_ai/tools.py:256-421` (`DeferredToolRequests`/`DeferredToolResults`), `pydantic_ai/toolsets/approval_required.py:26-31` (`ApprovalRequiredToolset`), `openai_agents/items.py:508-580` (`ToolApprovalItem`).

### Pattern 5: Multi-Strategy JSON Parser Ladder

**Problem**: Models emit JSON in a spectrum from strict JSON to Python-literal-ish, with occasional truncation during streaming.

**Sources**: CrewAI's `_validate_tool_input` tries `json.loads` → `ast.literal_eval` → `repair_json` → `json5.loads` → `repair_json(skip_json_loads=True)` (`tool_usage.py:884-930`). Letta's `_safe_load_tool_call_str` strips `}{` artifacts, double-decodes non-dict JSON, and returns `{}` on `JSONDecodeError` (`letta/agents/helpers.py:378-393`). Pydantic-ai's `args_as_dict` returns `{INVALID_JSON: '<raw>'}` (`messages.py:1826-1850`).

**Why it works**: The ladder trades strict-JSON purity for robustness on real model output. Each rung has a dedicated test in crewai (`tests/tools/test_tool_usage.py:264-543`) — Python literal form, JSON5 unquoted keys, trailing commas, escaped quotes, large JSON, code content, mixed types, single quotes.

**When to copy**: When you don't trust your model provider's JSON validity — which is "always" for production.

**When overkill**: Strict-mode providers that have already validated JSON server-side (OpenAI Responses API with strict tools, Anthropic). Openai-agents-sdk relies on provider-side strictness and only needs a guarded parser.

**Evidence**: `crewai/tools/tool_usage.py:884-930` (5-strategy ladder), `crewai/tests/tools/test_tool_usage.py:264-411` (test matrix), `letta/agents/helpers.py:378-393` (`_safe_load_tool_call_str`), `pydantic_ai/messages.py:36-40, 1826-1850` (INVALID_JSON).

### Pattern 6: Injected-Argument Filtering

**Problem**: Tools that need `state`, `store`, or `runtime` context can't trust the model to populate those fields correctly, and validation errors on injected fields confuse the model.

**Sources**: Langgraph's `_filter_validation_errors` strips errors whose first `loc[0]` is in the tool's `_InjectedArgs.all_injected_keys` and prunes the same keys from `input_value` (`tool_node.py:510-563`); `_inject_tool_args` then strips caller-supplied values for `InjectedToolArg` keys before re-injecting trusted values (`tool_node.py:1421-1430`). Pydantic-ai handles the same concern by not exposing `RunContext` as a model-callable parameter through the standard `ToolDefinition.parameters_json_schema`.

**Why it works**: Injected args are stripped twice — first from the error message so the model doesn't see "argument `state` is required" (which it can never fix), then from the call payload so the model can't forge them. This is the only model where the model can't accidentally set internal state via tool args.

**When to copy**: Any tool that takes `state`/`store`/`runtime`/`agent` as parameters — i.e., any tool whose correctness depends on caller-supplied values.

**When overkill**: Pure stateless tools.

**Evidence**: `langgraph tool_node.py:510-563, 1315-1430`, `langgraph tests/test_tool_node_validation_error_filtering.py:41-200`.

### Pattern 7: Parallel Execution with Deterministic Arbitration

**Problem**: Parallel tool calls can fail mid-batch; the framework must decide which failure to surface, what to do with siblings, and how long to wait.

**Sources**: OpenAI Agents SDK's `_FunctionToolBatchExecutor` uses arbitration (`_FunctionToolFailure`, `tool_execution.py:173-289`) preferring `Exception > asyncio.CancelledError`, with a 0.25s post-invoke drain (`_FUNCTION_TOOL_CANCELLED_DRAIN_SECONDS`). Pydantic-ai's `_call_tools` cancels sibling tasks on `BaseException` (`_agent_graph.py:1947-1956`) and supports `parallel`/`sequential`/`parallel_ordered_events` (`tool_manager.py:155-168`). Letta splits batches by `enable_parallel_execution` (`letta/agents/letta_agent_v3.py:1843-1862`). Langgraph dispatches in `version="v2"` mode via the `Send` API (`chat_agent_executor.py:846-859`).

**Why it works**: Without arbitration, a slow sibling can deadlock the batch or its failure can be lost. The arbitration policy is documented enough to test (`tests/test_run_step_execution.py:2859-2876`) and the drain gives in-flight work a bounded grace period.

**When to copy**: Whenever tool calls can run in parallel and any of them can fail.

**When overkill**: Single-tool-per-turn agents.

**Evidence**: `openai_agents/run_internal/tool_execution.py:1480-1559, 173-289` (arbitration + drain), `pydantic_ai/_agent_graph.py:1916-1956` (parallel/cancel), `letta/agents/letta_agent_v3.py:1840-1862` (parallel/serial fan-out).

### Pattern 8: Tool-Result Truncation with Opt-Outs

**Problem**: A search tool returning 1MB can blow context; but truncating a search result is often worse than not searching at all.

**Sources**: Letta's `Tool.return_char_limit` (`letta/schemas/tool.py:51-56`) with an opt-out list for `conversation_search`/`conversation_search_date`/`archival_memory_search` (`letta/agents/letta_agent_v3.py:1878`). Pydantic-ai auto-extracts multimodal files into a trailing `UserPromptPart` with `'See file {id}.'` placeholder instead of truncating raw values (`messages.py:1271-1297`).

**Why it works**: Truncation defaults to safe (cap-then-truncate) but search-style tools get a configurable opt-out where partial data is more harmful than large output. The split is documented at the call site, not buried in config.

**When to copy**: When tools can return large values and search tools are part of the surface.

**When overkill**: Stateless tools with bounded output sizes.

**Evidence**: `letta/schemas/tool.py:51-56` (`return_char_limit`), `letta/agents/letta_agent_v3.py:1878-1884` (opt-out list), `pydantic_ai/messages.py:1271-1297` (multimodal file extraction).

## Key Differences

### Where parsing happens

- **Adapter boundary**: langgraph delegates JSON parsing to `langchain_core` model adapters (it never sees raw JSON). This isolates model-format details but means errors are caught upstream and may surface as opaque adapter exceptions.
- **In-pipeline repair**: pydantic-ai uses the `INVALID_JSON` sentinel (`messages.py:36-40`), agent-framework uses `Content.parse_arguments` raw-fallback (`_types.py:1577-1578`), openai-agents-sdk raises `ModelBehaviorError` with redaction (`tool.py:1575-1598`).
- **Multi-strategy ladder**: crewai runs `json.loads` → `ast.literal_eval` → `repair_json` → `json5.loads` → `repair_json(skip_json_loads=True)` (`tool_usage.py:884-930`); letta's `_safe_load_tool_call_str` strips `}{` artifacts and double-decodes (`letta/agents/helpers.py:378-393`).

**Why diverge**: langgraph delegates because it sits atop `langchain_core` and trusts the adapter layer. Pydantic-ai and agent-framework trust the provider's strict schema; the lenient pass is only for malformed JSON shape. CrewAI and letta must accept whatever the model emits, so they get permissive. **Best practice picked up by pydantic-ai**: keep the lenient pass *bounded* (return a sentinel), let the strict layer reject, route through a single retry surface (`RetryPromptPart`).

### How errors are surfaced

- **Typed error messages**:
  - langgraph: `ToolMessage(status="error", content=..., tool_call_id=...)` (`tool_node.py:1006-1012`)
  - agent-framework: `Content(type="function_result", exception=str(exc))` (`_types.py:818-874`)
  - pydantic-ai: `RetryPromptPart(content=list[ErrorDetails] | str, tool_name, tool_call_id)` (`messages.py:1406-1502`)
  - openai-agents-sdk: `function_rejection_item` / `shell_rejection_item` (`items.py:349-409`)
  - letta: `ToolExecutionResult(status="error", ...)` (`schemas/tool_execution_result.py:8-18`) wrapped in `{status, message, time}` JSON envelope via `package_function_response` (`letta/system.py:150-168`)
- **String observations**:
  - crewai text mode: `"Error executing tool: {e}"` or `"Error: Failed to parse tool arguments as JSON: ..."` returned to the agent via the prompt (`crew_agent_executor.py:1009`; `agent_utils.py:1653-1662`)

**Why diverge**: Typed surfaces are easier to test, serialize, and observe; string observations are easier to bolt on but require text-matching for routing. **Best practice**: pick a typed surface from day one — moving from strings to typed content later costs the team weeks.

### How approval is modeled

- **Content type alongside tool calls**: agent-framework (`function_approval_request`/`function_approval_response`), pydantic-ai (`DeferredToolRequests.approvals`), openai-agents-sdk (`ToolApprovalItem` interruption)
- **Graph-level interrupt**: langgraph `interrupt_before=["tools"]` — no in-ToolNode concept of "this specific call needs approval"
- **Hook callback before execution**: crewai `before_tool_call` hook with `request_human_input()` (`hooks/tool_hooks.py:79-121`)
- **Per-agent policy at start**: openhands selects `NeverConfirm`/`ConfirmRisky`/`AlwaysConfirm` once per agent run (`app_conversation_service_base.py:641-652`)
- **Per-tool declarative rule**: letta `RequiresApprovalToolRule` (`letta/schemas/tool_rule.py:348`)

**Why diverge**: Content-type approvals are the only ones that survive checkpoint/resume without external state. Hook callbacks are the most flexible but get lost on resume. Per-agent policies are easy to reason about but can't distinguish tools. **Best practice**: declarative per-tool metadata (letta) that flows into a content-type-shaped output (pydantic-ai's `DeferredToolRequests`). The framework should refuse to run the tool until the caller provides a structured approval response, not by swallowing a `False` return.

### Repair vs. raise

- **Repair in-loop**: agent-framework re-emits a `function_result` with the error and lets the model self-correct; crewai returns the error string and lets the model self-correct; pydantic-ai uses `RetryPromptPart` for the same purpose.
- **Repair at transport layer**: pydantic-ai's `INVALID_JSON` sentinel + JSON-repair scaffolding (`messages.py:36-40`).
- **Raise and let caller decide**: openai-agents-sdk routes through `failure_error_function`: `None` raises, a callable returns a string that loops; letta's manager catches every exception and emits `status="error"`.
- **Trust the model**: langgraph converts every validation error to `ToolMessage(status="error")` with no in-loop retry budget — the user implements retry inside `wrap_tool_call` if needed.

**Why diverge**: Tools have different recovery semantics — some are idempotent (search), some are not (send_email). Frameworks that lean on the model are simplest; frameworks that opt to raise give users control. **Best practice**: a `failure_error_function` per tool (openai-agents-sdk) or per-tool `max_retries` (pydantic-ai) is the sweet spot — the framework can loop the model, and the caller can opt into raising.

### Multi-version drift

Crewai has two near-mirrored `CrewAgentExecutor._execute_single_native_tool_call` implementations (`crew_agent_executor.py:868-1071` and `experimental/agent_executor.py:1860-2073`); letta has three concurrent agent loop implementations (`letta/agent.py`, `letta_agent.py`, `letta_agent_v2.py`, `letta_agent_v3.py`). Pydantic-ai explicitly warns about two parallel resume code paths (`tool_manager.py:861-863` comment "keep the dispatch branches here in sync").

**Why diverge**: Migration safety is a real concern — letting old clients keep working while shipping new loop logic. But unmaintained mirrors are drift magnets. **Best practice**: if you must run two versions, share the *executor* (`execute_single_native_tool_call`-style helper) and only differ in the loop policy.

## Tradeoffs

| Choice | Benefit | Cost | Best-Fit Context | Failure Mode | Alternative |
|--------|---------|------|------------------|--------------|-------------|
| Typed content for all tool states | Easier to test, serialize, observe | Boilerplate per PR; schema must be kept in sync | Production systems with checkpointing | Stale schemas if content types change without updating serialization | String observations (crewai) for prototyping |
| Multi-strategy JSON parser ladder | Survives real model output | Loses strict JSON purity; risks accepting Python literals that should fail | Open-ended multi-provider deployments | Accepting a `dict` that should have failed strict validation | Single `json.loads` + Pydantic validation (openai-agents-sdk) |
| Approval as content type | Survives checkpoint/resume; mixed-batch routing | Requires caller to consume the approval response | Long-running agents with HITL | Lost approvals if caller forgets to thread `DeferredToolResults` | Per-agent policy class (openhands) |
| Per-tool budget (retries/timeout) | Isolates misbehavior | More config; harder to reason about "global" behavior | Multi-tool agents in production | One tool exhausting its budget silently | Global `max_iter` only (crewai) |
| Repair-and-loop on validation error | Maximizes model self-correction | Loops the model into the same broken behavior | Idempotent tools with clear schemas | Infinite retry on a permanently-broken tool | `failure_error_function=None` raises (openai-agents-sdk) |
| Strict schema by default | Provider-native strict tools work | Higher friction for optional fields and `oneOf` | OpenAI Responses API deployments | `UserError` blocking tool definition (`strict_schema.py`) | Hand-rolled JSON Schema validator (agent-framework) |
| Delegating JSON parse to the adapter layer (langgraph) | Framework doesn't care about provider format | Provider bugs surface as opaque adapter exceptions | Above `langchain_core` | Silent schema drift if adapter changes | Own the parser (everyone else) |
| Hook-based approval (crewai) | Maximum flexibility | Caller has to remember to register the hook | Teams wanting their own approval UX | Approval lost if no hook is registered | Declarative per-tool metadata (letta, agent-framework) |
| Per-agent confirmation policy (openhands) | Simple to reason about | Can't distinguish tools within an agent | Homogeneous agent surfaces | User wants to allow one tool but not another | Per-tool rules (letta, agent-framework) |
| Injected-argument filtering (langgraph) | Model can't forge state | Two-layer complexity | Stateful tools that take `state`/`store`/`runtime` | Tool author forgets to mark a field as injected | Bake state into the tool's lifecycle, not its signature |

## Decision Guide

When to use which approach:

- **Greenfield production system with checkpointing** → pydantic-ai (9) or openai-agents-sdk (8). Choose pydantic-ai if you need per-tool retries, three parallel modes, deferred-tool approval, and durable execution. Choose openai-agents-sdk if you target the OpenAI Responses API strictly and want a hook-style guardrail surface.

- **Above `langchain_core` and want graph-level orchestration** → langgraph (8). Choose if you already use LangGraph for state machines and you want approval via interrupts, not content types. Pair with `langchain.agents.create_agent` for first-party validation (`ValidationNode` is deprecated).

- **Multi-provider agent fleet with HITL requirements** → agent-framework (8). The `Content(type="function_approval_*")` shape, mixed-batch approval routing (`_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY`), and hosted-tool passthrough (`mcp_approval_request`) are durable patterns. Trade-off: dual Pydantic + JSON-Schema validation with different messages.

- **Multi-provider agency with permissive parser needs** → crewai (7). If your agents handle 4+ LLM providers and you need lenient JSON repair, the 5-strategy ladder and `extract_tool_call_info` 4-shape adapter are well-tested. Trade-off: two near-duplicate executors and no consecutive-error cap.

- **Persistent agents with sandboxed execution and strict schema healing** → letta (6). If you need per-tool `return_char_limit`, `ToolRulesSolver`-driven policy, and OpenAI strict-mode healing, this fits. Trade-off: 3 concurrent loop implementations and silent `{}` JSON-failure fallback.

- **Control plane over an external SDK** → openhands (3-of-10 on this dimension, but a fine answer for this question). If your repo is just configuration and observation, do not try to score well on this dimension — score on *configuration quality* and *observation fidelity* instead.

## Practical Tips

### Patterns to copy

1. **Start with typed content for tool states**. Define `ToolCall`, `ToolResult`, `ToolError`, `ToolApprovalRequest`, `ToolApprovalResponse` as Pydantic models (or dataclasses). The cost is small and the savings are large. (agent-framework, pydantic-ai, langgraph, openai-agents-sdk all do this.)

2. **Separate validation from execution**. `ValidatedToolCall` (pydantic-ai `tool_manager.py:46-74`) is one way; `wrap_tool_call` + `ToolInvocationError` (langgraph `tool_node.py:202-283, 339-380`) is another. The split lets you emit different events (`FunctionToolCallEvent(args_valid=...)`) and is needed for streaming.

3. **Bound retries per tool, not globally**. `max_retries=5` per tool with `ctx.retries: dict[str, int]` (pydantic-ai) is better than a global `max_iter=25` (crewai). The per-tool budget is what isolates misbehavior.

4. **Use a single retry channel.** All errors funnel into one typed surface — `RetryPromptPart` (pydantic-ai), `function_result(exception=...)` (agent-framework), `ToolMessage(status="error")` (langgraph). The discriminator lives in the message, not in 5 different code paths.

5. **Strip model-injected vs. caller-injected arguments separately.** langgraph's `_filter_validation_errors` (`tool_node.py:510-563`) is a pattern worth copying — the model only sees validation errors on fields it controls.

6. **Pre-compute expensive introspection at construction time.** langgraph's `_InjectedArgs` cache (`tool_node.py:786`), agent-framework's `_input_schema` cached schema (`_tools.py:782-804`), pydantic-ai's `_function_schema.function_schema` (`_function_schema.py:103-281`). One reflection pass at init; cheap at runtime.

### Patterns to avoid or delay

1. **Multi-strategy JSON parsing ladder (crewai-style)** — unless you support >=3 providers with heterogeneous JSON validity. The strict-Pydantic-after-lenient-parse pattern (agent-framework, pydantic-ai, openai-agents-sdk) is simpler and adequate for most cases.

2. **Global `before_tool_call` hooks as the only approval surface (crewai)** — prefer declarative per-tool metadata + content-type approval response. Hooks get lost on resume.

3. **`function_invocation_configuration` per-client, not per-call (agent-framework)** — but accept the trade-off; threading config through each call explodes the surface.

4. **Three concurrent loop implementations (letta)** — one shared executor with options beats three near-mirrors. If you must migrate, share the executor helpers.

5. **Silent `{}` on JSON failure (letta's `_safe_load_tool_call_str`)** — at minimum emit a `WARNING` log so the developer can see the demotion. The downstream "missing required argument" error is misleading.

6. **Best-effort per-tool budgets (agent-framework's `max_function_calls`)** — if the budget matters, enforce it exactly. Document the choice; don't ship it as a surprise.

7. **Hand-rolled JSON-Schema validator as a complete solution (agent-framework's `_validate_arguments_against_schema`)** — narrow coverage (no `oneOf`/`$ref`). Use `jsonschema` package which Pydantic already pulls in.

8. **In-memory approval state without serialization (openai-agents-sdk default)** — `RunState.approve()`/`reject()` is the explicit hook. Add a session-level `persist_approvals=True` default for long-running runs.

### Decision rules

- **Strict-schema provider (OpenAI Responses, Anthropic)**: pass through strict-mode validation; one guard at the parser is enough. *Don't* add a multi-strategy ladder.
- **Multi-provider deployment with no strict-mode**: Pydantic validation with a *single* lenient parse pass returning a sentinel; let Pydantic do the typing work.
- **Long-running / HITL agents**: declarative per-tool approval metadata (letta, agent-framework) flowing into a content-type-shaped response (pydantic-ai's `DeferredToolRequests`). Avoid graph-level interrupts unless you're already in a graph framework.
- **Sandboxed execution with strict tool surface**: `ToolExecutionResult(status, func_return, ...)` (letta) is the canonical structured result. Pair with per-tool `return_char_limit` and trace-driven metrics.

### Caution signs (the dimension is becoming brittle)

- **Two or more implementations of the same executor**: crewai's `_execute_single_native_tool_call` appears in three places (`crew_agent_executor.py:868-1071`, `experimental/agent_executor.py:1860-2073`, `utilities/agent_utils.py:1399-1633`); letta has three concurrent loop implementations (`letta/agent.py`, `letta_agent_v2.py`, `letta_agent_v3.py`). Either consolidate or share the executor helper and only vary the loop policy.
- **No unit tests at the parse seam**: letta has no tests for `_safe_load_tool_call_str`; agent-framework has narrow coverage of `_validate_arguments_against_schema`'s narrow schema subset. Missing tests at the seam are where silent-failure bugs live.
- **String observations where typed messages are easy**: if your `function_calling` path returns `result="Error: Tool X failed: …"` strings rather than typed error messages, you will lose the ability to route by error class later.
- **Global hooks for approval with no declarative alternative**: crewai's pattern works for `request_human_input` but doesn't survive checkpointed resume cleanly. Add a declarative surface as soon as you ship a checkpointed mode.
- **Best-effort budgets**: agent-framework's `max_function_calls` is post-batch (`_tools.py:2636-2644`). If your budget matters, enforce it exactly — any test should pin a single number.
- **Provider-coupling in the parser**: if your parser knows about OpenAI/Anthropic shapes by name, isolate it behind an adapter boundary (langgraph's approach). Otherwise, every new provider is a fork in the parser.
- **Silent tolerance**: returning `{}` on JSON failure (letta), silently dropping unknown keys (crewai), swallowing `try/except` at stream boundaries (letta). Each is "let me not crash" — together they form a system where you can't tell *why* something stopped working.

## Notable Absences

- **JSON parsing inside the prebuilt layer in langgraph** — by design; `langchain_core` adapters own it. Not a gap, but flagged in `tool_node.py:63-70` so a reader doesn't go hunting for the parser.
- **Tool-call recovery / repair in langgraph** — `wrap_tool_call` exists for caller-implemented retry, but the framework ships no default. The `ValidationNode` (`tool_validator.py:43-46`) is the historical "validate, then re-prompt" pattern and is deprecated in favor of `langchain.agents.create_agent`.
- **Per-call approval in openhands** — only `_select_confirmation_policy` exists, applied once at conversation start. Per-tool rules would need to be wired into `create_request()`.
- **Custom repair pipeline in openai-agents-sdk** — `default_tool_error_function` is fixed and per-tool `failure_error_function` is the only customization. A `tool_repair_function`/`arg_repair_callback` would close the gap.
- **`strict_mode=False` ergonomic escape hatch in openai-agents-sdk** — `strict_schema.py` raises `UserError` if a user-generated schema has `additionalProperties: true` without `strict=False`. `function_tool(strict_mode=False)` is supported but the message is unclear.
- **Unit tests for `_safe_load_tool_call_str` in letta** — no tests for the lenient parse fallback despite its centrality to the roundtrip.
- **Tests for `_resolve_single_deferred` vs `_call_tool` symmetry in pydantic-ai** — the inline comment ("keep the dispatch branches here in sync", `tool_manager.py:861-863`) flags this drift risk explicitly.
- **In-app-server hook for "approve this one specific call" in openhands** — the webhook layer captures events but has no API surface to influence them (`webhook_router.on_event` swallows exceptions and returns `Success()` regardless).
- **Per-batch `max_function_calls` pre-check in agent-framework** — currently post-batch (`_tools.py:2636-2644`); a pre-truncation by call-id could enforce exactly.
- **Default tool guardrails in openai-agents-sdk** — every tool only enforces guardrails if the author attaches them. A future-default for shell/computer/apply_patch tools could block obvious dangerous operations.
- **Strict-mode rewriter for Anthropic/Gemini in letta** — `enable_strict_mode` and the `SchemaHealth` classifier exist for OpenAI; other providers get non-strict and may behave inconsistently.

## Per-Source Notes

### microsoft/agent-framework

The most architecturally distinctive choice is that approval lives in the `Content` model — `function_approval_request` and `function_approval_response` are stable content types (`_types.py:1213-1253`). Mixed-batch routing uses session state to hide already-approved calls (`_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY`, `_tools.py:94, 1974-2032`), avoiding per-call approval prompts. The dual validation (Pydantic + hand-rolled JSON Schema) is the main operational risk: the JSON-Schema validator covers `required`/primitive `type`/`enum`/`additionalProperties: false` but skips `oneOf`/`$ref`/nested schemas (`_tools.py:1097-1144`). The `.NET` mirror in `dotnet/src/` uses the MEAI-style `FunctionInvokingChatClient` (`dotnet/AGENTS.md`), so the roundtrip is the same shape across language targets.

### crewai

Two parallel code paths (legacy text/ReAct + native function-calling) share `ToolUsage` primitives but each maintains its own executor. The 5-strategy JSON parser (`tool_usage.py:884-930`) is unusually well-tested — every rung has a dedicated test (`tests/tools/test_tool_usage.py:264-543`). The most operationally risky behavior is the silent argument filtering (`tool_usage.py:332-339, 570-579`): when `args_schema` is set, unknown keys are dropped without warning, and the downstream "missing required argument" error is misleading because it doesn't mention the dropped keys. Approval is hook-only via `ToolCallHookContext.request_human_input` (`hooks/tool_hooks.py:79-121`), which calls `builtins.input()` directly and blocks in async contexts.

### langchain-ai/langgraph

Graph-orchestrated pipeline with explicit `wrap_tool_call` extension point. The prebuilt layer does no JSON parsing — `langchain_core` owns it. The two-layer error handling (`ValidationError → ToolInvocationError → ToolMessage(status="error")`) is well-isolated, and the `handle_tool_errors` knob has six accepted shapes including type-introspected callables. The most distinctive safety mechanism is injected-argument filtering: `_filter_validation_errors` (`tool_node.py:510-563`) strips errors whose first `loc[0]` is in `all_injected_keys`, and `_inject_tool_args` (`tool_node.py:1421-1430`) strips LLM-supplied values for injected keys before re-injecting trusted values. Approval is graph-level only (`interrupt_before=["tools"]`); there is no in-ToolNode concept of "approve this specific call."

### letta

OpenAI-shaped canonical model is the lingua franca across provider adapters, streaming interfaces, and message construction. `ToolExecutionManager` dispatches by `ToolType` to a `ToolExecutor` subclass (`tool_execution_manager.py:35-65`); every executor returns `ToolExecutionResult(status, func_return, agent_state, stdout, stderr, sandbox_config_fingerprint)` (`schemas/tool_execution_result.py:8-18`). `ToolRulesSolver` is a pure-function policy engine that runs on every tool call. The most operationally risky behavior is the lenient JSON parser: `_safe_load_tool_call_str` returns `{}` silently on `JSONDecodeError` (`helpers.py:378-393`), and missing `tool_call_id` at stream end raises `ValueError` swallowed by outer `try/except` (`letta_llm_stream_adapter.py:184-187`). Three concurrent loop implementations (`letta/agent.py`, `letta_agent_v2.py`, `letta_agent_v3.py`) maintain migration safety at the cost of code drift.

### openai/openai-agents-sdk

Responses-API fidelity is the headline architectural choice. The `ToolExecutionPlan` is a turn-scoped dispatch table (`run_planning.py:177-300`) that separates "needs approval" from "execute now" — a resume builds a smaller plan that reuses the same executor. The `_FunctionToolBatchExecutor` (`tool_execution.py:1358-1970`) uses deterministic concurrent-failure arbitration preferring `Exception > asyncio.CancelledError` (`:173-289`) with bounded cancellation drain (0.25s). Strict schema is the default (`ensure_strict_json_schema`, `strict_schema.py:18-149`); tools can opt out via `function_tool(strict_mode=False)`. Two-layer error handling (guarded JSON parser → Pydantic) flows into `default_tool_error_function` for recoverable strings, with `failure_error_function=None` to raise instead. HITL approval via `ToolApprovalItem` interruptions (`items.py:508-580`) — approval state lives in-memory on `RunContextWrapper` by default, with explicit `RunState.approve()`/`reject()` for persistence.

### All-Hands-AI/openhands

V1 app-server (control plane) only; agent runtime lives in `software-agent-sdk`. The roundtrip mechanics — extraction, validation, execution, result formatting, repair, error injection — are entirely downstream of the agent-server. What this repo participates in: toolset wiring (`get_default_tools`/`get_planning_tools`, `live_status_app_conversation_service.py:1617-1625`), MCP server hosting (`mcp_router.py:43, 72, 147-487` with `mask_error_details=True`), confirmation policy selection (`_select_confirmation_policy`, `app_conversation_service_base.py:641-652`), secret-name validation (`validate_secret_name`/`validate_secrets_dict`, `constants.py:44-169`), and observation reconciliation for `SwitchLLMObservation` only (`webhook_router.py:483-496`). The SDK version handshake (`_verify_agent_server_version`, `live_status_app_conversation_service.py:176-181, 908-923`) is the most operationally significant edge — it catches the failure mode where a custom sandbox image ships an SDK version that doesn't match what the app-server expects.

### pydantic-ai

The most mature roundtrip in this study. Eight orthogonal safety knobs (`max_retries`, `timeout`, output retries, usage limits, concurrency, sequential, defer_loading, requires_approval`); single `RetryPromptPart` retry surface (`messages.py:1406-1502`); validate-then-execute split via `ValidatedToolCall` carrier (`tool_manager.py:46-74`); three parallel execution modes (`parallel`/`sequential`/`parallel_ordered_events`, `tool_manager.py:155-168`); transport-layer `INVALID_JSON` repair sentinel (`messages.py:36-40, 1826-1850`). Capability hooks (`before_tool_validate`/`wrap_tool_validate`/`after_tool_validate`/`on_tool_validate_error`/`before_tool_execute`/`wrap_tool_execute`/`after_tool_execute`/`on_tool_execute_error`) compose via `WrapperToolset` subclasses — `ApprovalRequiredToolset` is itself a `WrapperToolset` and doesn't need to know about hooks (`toolsets/approval_required.py:26-31`). Two deferred-tool surfaces share one carrier: `kind='external'` and `kind='unapproved'` both flow through `DeferredToolRequests` (`tools.py:256-326`) and are resolved via `DeferredToolResults` (`tools.py:378-421`) on the next run. The most operationally risky behavior is the manual sync between `_resolve_single_deferred` and `_call_tool`, explicitly flagged in a comment (`tool_manager.py:861-863`).

## Open Questions

1. **How does pydantic-ai's adapter layer handle the `INVALID_JSON` sentinel?** The framework tolerates it via `args_as_dict(raise_if_invalid=False)`, but it's unclear whether providers scrub or rewrite the value before sending it back (`pydantic-ai/messages.py:1826-1850`). Some providers may reject the sentinel key on the next request.

2. **What is the streaming argument reconstruction path?** Openai-agents-sdk has a dedicated test module `tests/test_streaming_tool_call_arguments.py`; pydantic-ai supports partial JSON via `allow_partial='trailing-strings'` (opt-in, not surfaced at agent level). The actual reassembly logic for partial streaming tool-call args across reconnects is not visible in the static path surveyed.

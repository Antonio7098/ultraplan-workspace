## Executive Summary

The two sources land in different places on the event-sourcing spectrum but both treat the question of "what is the source of truth for state?" with rigor. **Temporal** implements the canonical pattern: a pure append-only event log (`history_node`) keyed by `(tree_id, branch_id, node_id, txn_id)` with protobuf-encoded `HistoryEvent` batches; every mutation to `MutableState` piggybacks a `WorkflowEvents` append, and the mutable state in memory is explicitly a derived cache rebuilt by `StateRebuilderImpl` (`common/persistence/cassandra/history_store.go:16-19`, `service/history/ndc/state_rebuilder.go:103-150`). **LangGraph** uses a hybrid: a per-superstep checkpoint snapshot row is authoritative for ordinary channels, and a co-resident append-only `checkpoint_writes` table captures per-task writes that have not yet been folded into channel state; only the opt-in `DeltaChannel` layer (`libs/langgraph/langgraph/channels/delta.py`) elevates the write log to be canonical between snapshots, with periodic `_DeltaSnapshot` blobs bounding replay cost (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:131-159`, `libs/checkpoint/langgraph/ch checkpoint/base/__init__.py:582-690`).

Convergent properties across both: explicit append-only contract at the persistence layer; monotonic event IDs (`EventId` in Temporal, `ChannelVersions` per channel in LangGraph); bounded replay cost via paging or snapshot cadences; tested rebuilder paths; and protobuf-with-extended-types (Temporal) or msgpack-with-extended-types (LangGraph) serialization that allows newer code to read older blobs. Divergent properties: Temporal stores no per-event schema-version integer and relies on protobuf field-tag compatibility plus `DiscardUnknown` (`common/persistence/serialization/serializer.go:227-238`), while LangGraph carries an explicit `Checkpoint["v"]` integer and per-checkpoint migration routine (`libs/langgraph/langgraph/pregel/main.py:1135-1142`). Temporal's `VersionedTransition` fingerprint is a derivative staleness mechanism that lets `MutableState` detect split-brain scenarios without replaying events (`common/persistence/transitionhistory/transition_history.go:51-125`); LangGraph has no equivalent compact fingerprint and detects staleness only via channel-version bump on retry.

If you delete the snapshot row in LangGraph and keep the writes, state reconstruction still works for `DeltaChannel` but fails for ordinary channels. If you delete the snapshot and keep events in Temporal, full reconstruction is possible by design.

## Core Thesis

**For agent systems that need durable, replayable, event-sourced state, Temporal is the textbook reference: a single append-only log keyed by a compound identity, replay is exercised in tests, and mutable state is explicitly a projection.** LangGraph demonstrates the more pragmatic pattern for library-shaped products where the agent framework owns some channels but defers the durability contract to third-party checkpointers: a snapshot-per-step model augmented by an opt-in delta/event-sourced channel for users who need it. The two are not in conflict; they answer different product questions — Temporal asks "how do we make state provably recoverable across failures and clusters?", LangGraph asks "how do we let users attach arbitrary reducers to graph state while keeping a clean serialization boundary?" — and both produce event-sourced replay machinery in the process.

The deepest lesson is that "event sourcing" is a continuum. Treating it as a binary design choice ("we do event sourcing or we don't") produces brittle systems; treating it as a layered design where some state is event-sourced, some is snapshot-derived, and the transition between them is explicit produces systems that scale to real workloads. Both sources encode that layering, just at different layers of the stack.

## Approach Models

Both sources solve the same fundamental problem — recovering deterministic state from durable storage — but with different approach models:

- **Temporal — *Event log as canonical authority, state as derived projection*.** The `history_node` table holds every state change as a `HistoryEvent`; mutable state is loaded from `getWorkflowExecution` and is explicitly a cache (`service/history/workflow/context.go:141-220`, `service/history/workflow/mutable_state_impl.go:128-200`). Replay is a first-class operation invoked by `StateRebuilderImpl.Rebuild` and `MutableStateRebuilderImpl.ApplyEvents` (`service/history/ndc/state_rebuilder.go:103-150`, `service/history/workflow/mutable_state_rebuilder.go:70-101`), tested across `service/history/ndc/state_rebuilder_test.go:124-506` and `service/history/workflow/mutable_state_rebuilder_test.go:164-2211`. The newer `VersionedTransition` fingerprint is a derivative compact staleness signal layered on top, not a competing source of truth.

- **LangGraph — *Hybrid snapshot-plus-write-log with opt-in delta replay*.** Every superstep writes a `checkpoints` row carrying a serialized `Channel` state plus a separate `checkpoint_writes` table capturing per-task writes that have not yet folded into channel state (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:43-91`). For ordinary channels (`LastValue`, `Topic`, `BinaryOperatorAggregate`), the snapshot row is the source of truth. For the opt-in `DeltaChannel` (`libs/langgraph/langgraph/channels/delta.py`), the write log is canonical between snapshots, with `_DeltaSnapshot` blobs materialized to bound replay cost (`libs/langgraph/langgraph/pregel/_checkpoint.py:37-58`). The rebuilder is `channels_from_checkpoint` (`libs/langgraph/langgraph/pregel/_checkpoint.py:136-184`), and the protocol for delta replay is `BaseCheckpointSaver.get_delta_channel_history` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-690`).

The two models reflect their product shapes. Temporal is a workflow server: it owns the event log end-to-end, supports replication and reset across clusters, and is the single source of truth for every workflow. LangGraph is a library: it owns the in-memory graph engine but delegates durability to a third-party `BaseCheckpointSaver`; the hybrid model is the right way to give users choice between "store everything as a snapshot" and "store only the writes when the channel can be reconstructed from them."

## Pattern Catalog

Patterns that appear in both sources, or in one source with unusual clarity:

- **Append-only contract at the persistence layer, enforced by UPSERT on a compound identity key.** Temporal uses `INSERT … ON CONFLICT DO UPDATE` keyed by `(shard_id, tree_id, branch_id, node_id, txn_id)` with the rule "higher `txn_id` wins, no other path mutates the row" (`common/persistence/sql/sqlplugin/postgresql/events.go:12-16`, `common/persistence/cassandra/history_store.go:16-19`). LangGraph uses `INSERT … ON CONFLICT DO UPDATE` keyed by `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` with the rule "collision overwrites payload but never deletes the row" (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-153`). This is the cleanest way to get at-least-once retry safety and append-only immutability in the same operation. *When to copy*: any time you have a write path that may be retried. *When overkill*: single-writer local applications where retry safety is trivial. *Evidence*: Temporal `common/persistence/cassandra/history_store.go:71-83` (Cassandra dedup is "same `txn_id` upserts, no row delete"); LangGraph `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:262-292` (no UPDATE/DELETE on `writes` outside `setup`/`delete_thread`).

- **Compound identity key encodes both logical and physical position.** Temporal: `(tree_id, branch_id, node_id, txn_id)` — branch_id distinguishes forks/reset, txn_id orders the per-node append within a transaction (`schema/cassandra/temporal/versioned/v1.0/schema.cql:53-64`). LangGraph: `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` — `checkpoint_ns` namespaces subgraphs, `task_id` distinguishes per-task writes within a checkpoint (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:43-91`). *Why it works*: the same keying scheme supports ordering, concurrency, and idempotency without needing separate indices. *When to copy*: any system with branching/reset/subgraph semantics. *Evidence*: Temporal `api/history/v1/message.pb.go:73-130`; LangGraph `libs/checkpoint/langgraph/checkpoint/base/__init__.py:795` (`WRITES_IDX_MAP` keeps `ERROR/SCHEDULED/INTERRUPT/RESUME` writes in fixed negative slots to avoid collisions).

- **Monotonic event/sequence IDs as a precondition for append.** Temporal: `serializeAppendHistoryNodesRequest` requires contiguous `EventId` and uniform `Version` within a batch (`common/persistence/history_manager.go:332-360`); `AddOrUpdateVersionHistoryItem` rejects lower version / lower event ID (`common/persistence/versionhistory/version_history.go:66-90`). LangGraph: `ChannelVersions` dict on each checkpoint bumped by `apply_writes` via `checkpointer.get_next_version` (`libs/langgraph/langgraph/pregel/_algo.py:227-345`); `InMemorySaver` uses `<int>.<random>` for globally unique IDs (`libs/checkpoint/langgraph/checkpoint/memory/__init__.py:619-628`). *Why it works*: monotonic IDs let the system detect and reject out-of-order or replayed writes without scanning the log. *When to copy*: any event-log or write-log system. *Evidence*: Temporal `common/persistence/versionhistory/version_history_test.go:166-211` (rejection tests).

- **Paged history fetch during replay.** Temporal: `ReadHistoryBranchByBatch` with `defaultPageSize = 100` (`service/history/ndc/constants.go:4`, `service/history/ndc/state_rebuilder.go:356-400`). LangGraph: paged JSONB key lookup stage-1 plus per-channel UNION ALL stage-2 SQL for delta reconstruction (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:201-288`, executed at `libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:444-550`); streamed stage-1 in SQLite (`libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/_delta.py:34-124`). *Why it works*: a single event log is too large to fetch atomically; paged fetches keep memory and DB transaction size bounded. *When to copy*: any system whose event log can grow unboundedly. *Evidence*: Temporal `service/history/ndc/state_rebuilder_test.go:124-506` (pagination tests); LangGraph `libs/checkpoint-conformance/langgraph/checkpoint/conformance/spec/test_delta_channel_history.py:1-247`.

- **Capability-gated conformance suite.** LangGraph's `libs/checkpoint-conformance/langgraph/checkpoint/conformance/capabilities.py:15-87` defines `DELTA_CHANNEL_HISTORY` as an explicit capability; savers opt in by overriding `aget_delta_channel_history`. *Why it works*: third-party checkpointers can declare support without breaking the abstraction. *When to copy*: any library that defines an interface with optional methods. *Evidence*: `libs/checkpoint-conformance/langgraph/checkpoint/conformance/capabilities.py:53-87`.

- **Selective replay / cherry-pick reset.** Temporal's `WorkflowResetter.ResetWorkflow` calls `stateRebuilder.Rebuild`, then `reapplyEvents`/`reapplyEventsFromBranch` to apply only events the new run cares about (`service/history/ndc/workflow_resetter.go:107-235,767-1100`). LangGraph's `ReplayState` (in `_internal/_replay.py:14-90`) loads the latest-before-replay checkpoint for subgraph replay, gated by `PregelLoop.__enter__` (`libs/langgraph/langgraph/pregel/_loop.py:1606-1686`). *Why it works*: not every workflow re-run needs every event; selective replay is how both systems support reset / fork / time-travel without paying full replay cost. *When to copy*: any system with reset, fork, or replay-from-version semantics.

- **Serializer-aware schema evolution.** Temporal: protobuf wire compat plus `DiscardUnknown: true` on `DeserializeStrippedEvents` (`common/persistence/serialization/serializer.go:227-238`), with `ProtoDiscardUnknownFields` test (`common/persistence/serialization/serializer_test.go:205-243`). LangGraph: `_DeltaSnapshot` blob encoded as dedicated msgpack ext type `EXT_DELTA_SNAPSHOT = 7` (`libs/checkpoint/langgraph/checkpoint/serde/jsonplus.py:295-307, :634-639`); `Checkpoint["v"]` integer for container-format migration (`libs/langgraph/langgraph/pregel/main.py:1135-1142`). *Why it works*: extending the wire format without breaking existing payloads lets the same store hold events from many code generations. *When to copy*: any system whose serialized state outlives code versions. *Evidence*: Temporal `common/persistence/cassandra/history_store.go:78-79` (explicit `data_encoding` per node); LangGraph `libs/checkpoint/langgraph/checkpoint/serde/types.py:19-31`.

- **Compact staleness fingerprint layered on top of the log.** Temporal's `VersionedTransition{NamespaceFailoverVersion, TransitionCount}` lets `MutableState` detect split-brain state cheaply without replaying events (`common/persistence/transitionhistory/transition_history.go:51-125`), with `StalenessCheck` returning `ErrStaleState`/`ErrStaleReference` and per-entity `LastUpdateVersionedTransition` stamping (`service/history/workflow/mutable_state_impl.go:7867-7925`, `api/persistence/v1/executions.pb.go:314-316`). LangGraph has no equivalent — staleness is detected only by channel-version bump on retry. *Why it works*: replaying events is expensive, but many staleness checks don't need full event content. *When to copy*: any system where two writers may race for the same entity and full replay is too costly. *Evidence*: Temporal `common/persistence/transitionhistory/transition_history_test.go:71-116`.

- **Bounded replay cost via multi-axis controls.** Temporal: `defaultPageSize = 100`, `TransactionSizeLimitError`, `HistorySizeLimitError/Warn`, `HistoryCountLimitError/Warn`, `MutableStateSizeLimitError`, `MutableStateTombstoneCountLimit`, plus TTL-bounded mutable-state cache and reference-counted branch GC (`service/history/ndc/constants.go:4`, `common/persistence/history_manager.go:367-373`, `common/dynamicconfig/constants.go:386-430`, `service/history/workflow/cache/cache.go:96-149`, `common/persistence/history_manager.go:130-208`). LangGraph: per-channel `snapshot_frequency`, system-wide `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT` env var (default 5000), and per-channel `counters_since_delta_snapshot` bookkeeping (`libs/langgraph/langgraph/channels/delta.py:60-64`, `libs/langgraph/langgraph/_internal/_config.py:33-35`, `libs/checkpoint/langgraph/checkpoint/base/__init__.py:63-86`, `libs/langgraph/langgraph/pregel/_loop.py:1079-1142`). *Why it works*: replay cost has multiple dimensions (per-write, per-transaction, per-history, per-cache-miss) and each needs its own bound. *When to copy*: any system whose replay cost can blow up under long histories. *Evidence*: Temporal `service/history/workflow/cache/cache.go:96-149`; LangGraph `libs/langgraph/tests/test_delta_channel_supersteps_bound.py:1-195`.

- **History builder immutability at the in-memory layer.** Temporal's `HistoryBuilderStateImmutable`/`HistoryBuilderStateSealed` enforced via `assertMutable`/`assertNotSealed`, with `EventStore.Finish` setting sealed state in deferred call (`service/history/historybuilder/event_store.go:26-30,218-258,290-299`). *Why it works*: once history is committed, the in-memory copy must be sealed to prevent drift between cache and log. *When to copy*: any system with a mutable cache in front of an immutable log. *Evidence*: `service/history/historybuilder/history_builder.go:101-153` (`NewImmutable`/`NewImmutableForUpdateNextEventID` provide read-only iterators).

## Key Differences

The two sources diverge on five substantive axes, each traceable to a difference in product shape:

1. **Authority model.** Temporal's `history_node` log is canonical; mutable state is a projection. LangGraph's checkpoint snapshot row is canonical for ordinary channels; the writes log is canonical only for the opt-in `DeltaChannel`. The difference is "single-log systems own the source of truth end-to-end; library-shaped systems share authority with the persistence plugin." Temporal can be authoritative because it owns the storage; LangGraph must defer to whatever saver the user picks.

2. **Event-id space.** Temporal uses a single global `EventId` per branch, monotonic per version, gated by `serializeAppendHistoryNodesRequest` (`common/persistence/history_manager.go:339-360`). LangGraph uses per-channel `ChannelVersions` bumped by `apply_writes` via `get_next_version`, with `InMemorySaver` falling back to `<int>.<random>` for globally unique IDs (`libs/langgraph/langgraph/pregel/_algo.py:227-345`, `libs/checkpoint/langgraph/checkpoint/memory/__init__.py:619-628`). The difference is "one log = one sequence number; many channels = many independent sequence numbers." LangGraph's design is a natural fit for fan-out graphs where multiple channels update concurrently.

3. **Branching semantics.** Temporal treats branching as a first-class construct used by reset, replication, and continue-as-new, with `VersionHistory`/`VersionHistoryItem` and LCA resolution (`common/persistence/versionhistory/version_history.go:108-135`, `common/persistence/history_manager.go:79-95` for branch UUID generation). LangGraph's "branching" is essentially `fork` and `invoke(checkpoint_id=...)` time-travel (`libs/langgraph/langgraph/pregel/_loop.py:1606-1686`), with `ReplayState` for subgraph replay (`libs/langgraph/langgraph/_internal/_replay.py:14-90`). Temporal's branching is richer because it needs to support cross-cluster replication; LangGraph's is sufficient for in-process time-travel.

4. **Schema evolution mechanism.** Temporal uses protobuf wire compat + `DiscardUnknown` and stores no per-event schema version (`common/persistence/serialization/serializer.go:227-238`). LangGraph uses `Checkpoint["v"]` integer + `_migrate_checkpoint` for container migration, plus `_DeltaSnapshot` as a dedicated msgpack ext type to isolate the delta representation (`libs/langgraph/langgraph/pregel/main.py:1135-1142`, `libs/checkpoint/langgraph/checkpoint/serde/types.py:19-31`). Temporal's design trades per-event version complexity for forward-only protobuf evolution; LangGraph's design keeps an explicit version stamp for migration.

5. **Staleness detection.** Temporal has a dedicated `VersionedTransition` fingerprint layered on top of the log for split-brain detection (`common/persistence/transitionhistory/transition_history.go:51-125`, `service/history/workflow/state_transition_history.go:24-45`). LangGraph has no such fingerprint — staleness is detected only by retry bumping channel versions. The difference is "production-grade event-sourced systems need cheap staleness checks because they handle concurrent writers; agent-framework state usually has a single writer (the agent loop) and retries are exceptional."

## Tradeoffs

- **Single canonical log vs. layered snapshot-plus-log.**
  - *Benefit (Temporal)*: one log = one replay path = simpler mental model; trivial cross-cluster replication.
  - *Cost (Temporal)*: every write must go through the log; no way to opt out of durable storage; per-write serialization cost.
  - *Benefit (LangGraph)*: per-channel choice between snapshot and event-sourced reducer; cheaper writes for channels that don't need replay.
  - *Cost (LangGraph)*: no uniform "events" interface; users have to know which channels are delta and which aren't; `prune`/`copy_thread` failure modes.
  - *Best fit*: Temporal for workflow servers; LangGraph for agent libraries with user-extensible state.
  - *Alternative seen*: hybrid models that snapshot a *derived* projection on top of a pure log (a direction Temporal hints at with `MutableStateSizeLimitError = 8 MiB` and continue-as-new nudges in `common/dynamicconfig/constants.go:386-430`).

- **Per-event schema version vs. protobuf forward-compat.**
  - *Benefit (Temporal / protobuf)*: no schema-version integer to maintain; old events automatically replay under new code as long as field tags are preserved.
  - *Cost (Temporal / protobuf)*: dropping a required field silently breaks old events; no on-disk record of "what schema version was this event written with?".
  - *Benefit (LangGraph / `Checkpoint["v"]`)*: explicit migration routines; clear version boundary for breaking changes.
  - *Cost (LangGraph / `Checkpoint["v"]`)*: every checkpoint writer must respect the version protocol; `_migrate_checkpoint` is a hot path.
  - *Best fit*: protobuf for systems whose event types evolve organically; explicit version for systems with discrete breaking changes.
  - *Alternative*: hybrid — keep protobuf field tags but also record a `schema_version` integer on the blob (a future consideration both sources acknowledge).

- **Compound identity key vs. single primary key.**
  - *Benefit*: handles branching/forking/subgraphs without separate indices.
  - *Cost*: schema migrations are harder (every key component may need to change); query plans are more complex.
  - *Failure mode*: forgetting one component of the key produces silent collisions or fan-out (LangGraph's `WRITES_IDX_MAP` reserves negative slots for `ERROR/SCHEDULED/INTERRUPT/RESUME` to prevent this — `libs/checkpoint/langgraph/checkpoint/base/__init__.py:795`).

- **`VersionedTransition` fingerprint vs. no fingerprint.**
  - *Benefit*: cheap split-brain detection without replaying events.
  - *Cost*: every close of a transaction updates `TransitionHistory` and stamps many child entities (`service/history/workflow/mutable_state_impl.go:7840-7925`), adding write amplification.
  - *When to copy*: any system with concurrent writers or cross-cluster replication.
  - *Failure mode*: `UpdatedTransitionHistory` collapses same-version transitions, losing per-mutation granularity (`service/history/workflow/state_transition_history.go:24-45`) — a tradeoff that makes the log compact but reduces staleness detection resolution.

- **Snapshot cadence per-channel vs. system-wide env-var bound.**
  - *Benefit (LangGraph per-channel `snapshot_frequency`)*: each channel tunes to its own write pattern.
  - *Cost*: developer-tunable, hardcoded defaults, no per-workload auto-tuning (`libs/langgraph/langgraph/_internal/_config.py:33-35`, default 5000).
  - *Failure mode*: too-low `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT` reverts to non-delta-channel size; too-high risks long ancestor walks on read.

## Decision Guide

When to choose each approach:

- **Choose pure append-only event log (Temporal style) if:**
  - You own the storage end-to-end and can guarantee replay semantics.
  - You need cross-cluster replication, fork/reset/continue-as-new, or other branching semantics.
  - Your users need a uniform "give me every event" query interface.
  - You need provable durability across failures and can pay for full replay on cache miss.

- **Choose snapshot-per-step with co-resident write log (LangGraph base) if:**
  - You are a library and must defer durability to a user-provided store.
  - Your state graph has channels with very different write rates (some heavy, some rare).
  - Most channels don't benefit from event-sourced replay and would just add write cost.
  - You want users to opt into event-sourced behavior for specific channels without changing the framework.

- **Choose hybrid with opt-in event-sourced reducer (LangGraph `DeltaChannel`) if:**
  - You have a library-shaped product but some users will want full event sourcing.
  - You can document a deterministic-reducer contract and accept that correctness depends on it.
  - You can ship a conformance suite so users can verify their storage layer.
  - You can give users explicit snapshot knobs (`snapshot_frequency` per channel, system-wide env var).

- **Add a staleness fingerprint if:**
  - You have concurrent writers or cross-cluster replication.
  - Full event replay is too expensive for staleness detection.
  - You can absorb the write amplification of stamping many child entities.
  - If none of those apply, skip it — LangGraph's no-fingerprint design is fine for single-writer agent loops.

## Practical Tips

Patterns to copy:

- **Compound identity key** (`(tree_id, branch_id, node_id, txn_id)` in Temporal, `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)` in LangGraph) is the cleanest way to encode ordering + branching + idempotency in one schema.
- **Append-only contract at the persistence layer**, enforced by `INSERT … ON CONFLICT DO UPDATE` with a strict "only same-key overwrite" rule. This gives you at-least-once retry safety and immutability in the same operation.
- **Paged history fetch with explicit page-size constant.** Temporal's `defaultPageSize = 100` and LangGraph's two-stage SQL are both effective. Make the page size a single named constant so it's tunable.
- **Conformance test suite for third-party storage plugins.** LangGraph's `libs/checkpoint-conformance/` is a model — it lets users write their own savers without breaking the framework's replay contract.
- **Capability-gated optional methods.** `DELTA_CHANNEL_HISTORY` as an explicit capability declaration lets the framework know which savers support which replay paths.
- **History builder immutability at the in-memory layer.** Temporal's `assertMutable`/`assertNotSealed` is a cheap way to prevent cache drift.
- **Serializer-aware schema evolution.** Use protobuf wire compat + `DiscardUnknown` (Temporal) or msgpack ext types + explicit version stamp (LangGraph) to let old events replay under new code.

Patterns to avoid or delay until needed:

- **Per-event `schema_version` integer is only worth it if you anticipate breaking changes.** Temporal doesn't store one because protobuf field tags give it forward compat for free. If you don't need explicit migration routines, skip the integer.
- **`VersionedTransition` fingerprint is overkill for single-writer, no-replication systems.** Add it only when you actually have concurrent writers or cross-cluster scenarios.
- **Pure event sourcing is overkill for library-shaped products.** If you don't own the storage, prefer the hybrid snapshot-plus-log model — users get choice, you avoid ownership of durability semantics.
- **A public "give me every event" query API is overkill unless you have observers.** LangGraph's writes table is internal; if you don't have observer use cases, keep it that way.
- **Auto-tuning snapshot cadence is premature optimization.** LangGraph's hardcoded defaults + env var override is enough until you see real workloads tuning it.

Decision rules:

- **One log or many?** If all state lives in one logical structure (workflow execution, document), use one log with branching. If state is naturally partitioned (graph channels, document sections), use one log per partition with per-partition sequence numbers.
- **Snapshot or no snapshot?** If replay from time zero is bounded (small history, fast reducer), skip snapshots. If replay can be long (large history, complex reducer), add snapshots but make them opt-in per state partition.
- **Versioned transitions or not?** Yes if you have concurrent writers or replication. No if you have a single writer.
- **Per-event version stamp or protobuf forward-compat?** Yes if you expect breaking schema changes. No if you expect additive-only evolution.

## Anti-Patterns / Caution Signs

- **Silent corruption when pruning intermediate writes.** LangGraph documents this explicitly: `BaseCheckpointSaver.prune` with `keep_latest` can sever the chain between a kept checkpoint and the nearest `_DeltaSnapshot` ancestor, silently reconstructing delta channels as empty (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`). If you implement prune, you must preserve the chain.
- **`copy_thread` partial chain.** A naive copy of only the head orphans delta writes from the head back to the nearest snapshot. Must copy the entire parent chain (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:350-372`).
- **Snapshot cadence tuned to one workload.** Hardcoded `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT = 5000` works for typical graphs but not for long-running workflows. If you don't expose a per-workload knob, you'll hit either too-many-snapshots or too-long ancestor walks.
- **Reducer non-determinism caught only by tests.** LangGraph's `DeltaChannel` requires `(state, [writes]) -> new_state` to be deterministic and batching-invariant (`libs/langgraph/langgraph/channels/delta.py:42-49`). The library provides no schema-level versioning for serialized reducer state; migration is on the reducer author. A non-deterministic reducer silently produces different state across replays; only the `test_*` corpus exercises the equivalence property.
- **Permissive default serializer.** LangGraph's `JsonPlusSerializer` runs in "permissive" mode by default and logs unregistered types (`libs/checkpoint/langgraph/checkpoint/serde/jsonplus.py:107-114`). Setting `LANGGRAPH_STRICT_MSGPACK=true` locks deserialization to `SAFE_MSGPACK_TYPES`. If an attacker can write to the checkpoint table, they can trigger code execution on deserialization unless the strict gate is on.
- **DataLoss requiring human recovery.** Temporal's `*serviceerror.DataLoss` from `ReadHistoryBranchByBatch` is re-raised by `stateRebuilder` (`service/history/ndc/state_rebuilder.go:260-262`); there is no automatic recovery. If you can avoid this failure mode (e.g., via cross-cluster replication to a peer), do.
- **Per-batch invariants only enforced at one layer.** Temporal enforces `EventId` continuity + uniform `Version` at the manager layer (`common/persistence/history_manager.go:332-360`); if a different code path bypasses the manager and writes directly to the store, the invariant disappears. Make sure every write path goes through the manager.
- **Two staleness layers that drift.** Temporal has both version-history monotonicity and transition-history compare. They serve different purposes but if their ordering rules drift, debuggability becomes painful. Keep them documented together.
- **Cache miss forces full rebuild.** Temporal's mutable-state cache miss triggers `stateRebuilder.Rebuild` + `taskRefresher.Refresh` (`service/history/ndc/state_rebuilder.go:141-143`). If your cache TTL is too short, you'll rebuild constantly. Make sure your cache TTL is tuned to your replay cost.

## Notable Absences

- **No per-event schema version in Temporal.** Searched `common/persistence/serialization/` and `schema/` — the only versioning concept is `ENCODING_TYPE_*` on `DataBlob` and the DB schema version check at `common/persistence/schema/version.go:10-29`. No `EventBlobVersion` / `EventBlobSchemaVersion` integer stored on the blob itself. This is a deliberate tradeoff but means dropping a once-required proto field cannot be detected at the storage layer — old events would fail to deserialize unless the new code keeps the field tag as reserved.
- **No public append-only "event log" abstraction in LangGraph.** Events are conflated with channel writes; the schema mixes snapshot and write data into the same saver. Treat this dimension accordingly: "yes for `DeltaChannel`, no for the rest."
- **No staleness fingerprint in LangGraph.** No equivalent of `VersionedTransition`. Staleness is detected only by retry bumping channel versions, which means split-brain scenarios are caught only after a retry attempt.
- **No standalone "event log" abstraction in Temporal.** History persistence is folded into the broader `ExecutionManager` interface (`common/persistence/execution_manager.go:1-279`, `common/persistence/data_interfaces.go:1420-1438`); the term "event log" does not appear as a named interface. This is fine for a workflow server but makes it harder to extract a standalone event-log component.
- **No first-party `prune`/`copy_thread` reference implementations in LangGraph.** The Postgres and SQLite concrete savers rely on the base class's `NotImplementedError` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:415, 372, 348`). Implementing them with the documented safeguards would close the documented failure mode for first-party users.
- **No per-workload reducer validation in LangGraph.** A static check that the reducer is associative-across-folds would catch drift before it becomes a silent replay divergence. Only the `test_*` corpus exercises the equivalence property.
- **No replay observability hooks.** Neither source exposes per-rebuild metrics (ancestor-walk length, seed-vs-walk count, snapshot-blob size, history range replayed) that would let operators tune the relevant knobs based on real workloads.
- **No first-class "events" query API in LangGraph.** A unified `events(thread_id, from_version, to_version)` over the writes table would let observers reconstruct a thread without going through `get_state`. Currently the writes table is read only by the saver and the rebuilder.

## Per-Source Notes

**langgraph (7/10).** Snapshot-per-step system with a co-resident per-step write log. The closest thing to event sourcing is the opt-in `DeltaChannel` (`libs/langgraph/langgraph/channels/delta.py`), which uses an immutable write history between periodic `_DeltaSnapshot` blobs. The reconstruction protocol `get_delta_channel_history` → `replay_writes` walks the parent chain by checkpoint id and is the closest the codebase comes to "event sourcing + replay." Strong explicit interfaces (`BaseCheckpointSaver.get`/`get_tuple`/`list`/`put`/`put_writes` at `libs/checkpoint/langgraph/checkpoint/base/__init__.py:227-318`), append-only write path durable by contract (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:146-153`), bounded replay cost via `_DeltaSnapshot` blobs and per-channel counters (`libs/checkpoint/langgraph/checkpoint/pregel/_checkpoint.py:37-58`), checkpoint schema versioning via `Checkpoint["v"]` integer with explicit migration routine (`libs/langgraph/langgraph/pregel/main.py:1135-1142`), and a conformance test suite (`libs/checkpoint-conformance/...`). Held back by: no public append-only event-log abstraction, `prune` can silently corrupt `DeltaChannel` chains (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`), snapshot cadence is developer-tunable rather than observable, and replay correctness depends on user-supplied reducer being deterministic and batching-invariant (`libs/langgraph/langgraph/channels/delta.py:42-49`) — enforced only by documentation.

**temporal (9/10).** Pure append-only event log at the `history_node` table keyed by `(tree_id, branch_id, node_id, txn_id)`, with mutable state as a derived projection rebuilt by `StateRebuilderImpl` (`service/history/ndc/state_rebuilder.go:62-150`). Append-only contract enforced both at the API layer (`common/persistence/history_manager.go:339-360`) and at the SQL/Cassandra persistence layer (`common/persistence/sql/sqlplugin/postgresql/events.go:12-16`, `common/persistence/cassandra/history_store.go:16-19`). Newer `VersionedTransition` mechanism (`api/persistence/v1/hsm.pb.go:457-465`) is layered on top as a compact per-mutation fingerprint to detect stale state without replaying events. Replay explicitly exercised in tests (`service/history/ndc/state_rebuilder_test.go:246-506`, `service/history/workflow/mutable_state_rebuilder_test.go:164-2211`, `service/history/ndc/workflow_resetter_test.go:1037-1746`). Held back by: no per-event schema version (Temporal relies on protobuf field tags + `DiscardUnknown` rather than a per-event schema version), and `*serviceerror.DataLoss` is fatal without automatic recovery (`service/history/ndc/state_rebuilder.go:260-262`).

## Open Questions

- **Capability advertisement in LangGraph.** `get_delta_channel_history` capability is detected by method override (`libs/checkpoint-conformance/langgraph/checkpoint/conformance/capabilities.py:75-87`). A saver that doesn't override but inherits the base walk won't be flagged as having the capability. Is there evidence that capability-based routing in production honors the right path?
- **Encryption boundary in LangGraph.** `EncryptedSerializer` wraps another serde (`libs/checkpoint/langgraph/checkpoint/serde/encrypted.py`); `with_msgpack_allowlist` is propagated through the wrapper. Effect on replay correctness is not separately documented.
- **`JsonPlusSerializer` default permissive mode security policy.** The strict gate is opt-in via env var. Is there a server-enforced default strict mode for production deployments?
- **`BackfillHistoryEvents` failure policy in Temporal.** The `*serviceerror.SyncState` fallback for missing mutable state (`service/history/ndc/history_replicator.go:260-269`) is an in-band retry signal rather than a fully observable event; the exact policy for converting it into a recovery action is in the receiver (out of scope for this dimension).
- **`MutableStateTombstoneCountLimit = 16`** (`common/dynamicconfig/constants.go:427-430`) constrains delete-marker churn but does not directly bound history size. What is the operational relationship?
- **Three rebuilder paths in Temporal.** `Rebuild`, `RebuildWithCurrentMutableState`, and `MutableStateRebuilder.ApplyEvents` (`service/history/ndc/state_rebuilder.go:103-213`, `service/history/workflow/mutable_state_rebuilder.go:70-101`) have overlapping responsibilities. Is consolidation on the roadmap?
- **Replay observability.** Neither source exposes per-rebuild metrics (ancestor-walk length, seed-vs-walk count, snapshot-blob size, history range replayed). Would adding a hook to `StateRebuilderImpl.Rebuild` and `channels_from_checkpoint` let operators tune their respective knobs?
- **Per-event schema version.** If Temporal were to add one (a future consideration they acknowledge), would it replace protobuf `DiscardUnknown` or be additive? Same question for LangGraph if it were to add per-payload versioning.

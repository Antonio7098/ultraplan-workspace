## Executive Summary

No source in this study ships a true pattern-based doom-loop detector. Every agent harness reduces non-progress to a counter — either a turn/step/request cap or a per-tool retry budget — and trusts the LLM or the user to recognise repetition. The strongest detection surface in the study is `agent-framework`'s .NET Magentic orchestrator, which asks the LLM judge `IsInLoop` / `IsProgressBeingMade` each round, decays/increments a `StallCount`, and triggers a `ResetAndReplanAsync` intervention when the threshold (`DefaultMaxStallCount = 3`) is exceeded. Even there, the "detector" is the model itself, not a structural fingerprint of tool calls.

What converges across the seven sources is the *ultimate safety net*: a hard iteration cap that fires after N turns (`DEFAULT_MAX_ITERATIONS = 40` in agent-framework Python, `max_iter = 25` in crewai, `recursion_limit = 10007` in langgraph, `DEFAULT_MAX_STEPS = 50` in letta, `DEFAULT_MAX_TURNS = 10` in openai-agents-sdk, `request_limit = 50` in pydantic-ai). What diverges is everything between "first turn" and "cap reached": the most aggressive sources insert a forced-final-answer LLM call to extract a usable answer from the wasted trajectory; the most passive sources simply raise an exception and hope the caller catches it.

Two patterns emerge from this divergence. First, **the user pays for the full iteration budget whenever a doom loop involves successful calls**, because no source fingerprints `(tool_name, arguments)` over a sliding window — only crewai and agent-framework Python compare tool calls at all, and both compare them only against the immediately preceding call. Second, **compaction systems erase loop evidence by design**: agent-framework collapses older tool-call groups into one-line summaries, crewai's `summarize_messages` clears the entire history except system messages, pydantic-ai's provider-driven `CompactionPart` replaces past history with an opaque token, and openhands' `LLMSummarizingCondenser` summarises everything except the first two events. Because no framework-side detector inspects the raw history anyway, this is not yet a real cost — but it forecloses the cheapest place to add a real detector later.

The headline question — *does the system stop repeated behaviour before the user pays for 20 useless turns?* — is answered **No** by five of seven sources, **Yes only when tuned** by langgraph (recursion_limit), and **Yes only on repeated tool errors or LLM-judged stalls** by agent-framework. The median default cap is 25–50 turns, which is past the 20-turn pain threshold for most users.

## Core Thesis

**Doom-loop detection in the studied agent harnesses is a budget problem, not a recognition problem.** Every framework knows how to count turns; none of them knows how to count repetition. Where a real detector exists — agent-framework .NET Magentic's LLM-judged `IsInLoop` + `IsProgressBeingMade` + reset-and-replan — it works by paying for another LLM call each round and trusting the model to recognise its own stalling. The cheaper structural detector (fingerprinting the last K tool calls and counting consecutive or alternating matches) is absent from every source in the study, including those that explicitly market loop prevention.

This is not a failure of engineering — it is a deliberate architectural choice. The frameworks vary in three axes:

1. **Who is responsible for recognising a loop?** LangGraph, OpenAI Agents SDK, Pydantic AI, and (for its tool loop) agent-framework delegate to the model. Letta delegates to the agent author via declarative tool rules. CrewAI does a one-step byte-identical comparison and otherwise also delegates. Only agent-framework .NET Magentic inserts an explicit judge LLM call per round.
2. **What is the unit of the budget?** A superstep (langgraph), a tool loop iteration (agent-framework Python), a ReAct iteration (crewai, letta), an LLM turn (openai-agents-sdk, pydantic-ai), an MCP "step" (openhands). These are not interchangeable — a single "step" in langgraph can fan out into many parallel tool calls.
3. **What happens at the budget boundary?** The variance is huge. agent-framework Python and crewai force a final-answer LLM call so the user gets a usable response. agent-framework .NET LoopAgent and Magentic log and stop. langgraph raises `GraphRecursionError`. openai-agents-sdk raises `MaxTurnsExceeded` unless a user-supplied handler returns a result. pydantic-ai raises `UsageLimitExceeded`. letta sets `StopReasonType.max_steps` and continues. openhands bubbles up `ConversationExecutionStatus.STUCK` to the SDK caller.

The choice reflects each framework's primary user. Frameworks that target embedded agents in larger applications (agent-framework, crewai) prioritise graceful degradation — the user gets *some* answer even from a doomed trajectory. Frameworks that target library authors (langgraph, pydantic-ai) prioritise correctness — the framework raises a clear error and lets the caller decide. Frameworks that target long-running autonomous work (openhands, letta) prioritise continuation — the framework lets the loop run to its natural end and only labels the trajectory.

## Approach Models

The seven sources cluster into four approach models for this dimension. Use these categories when describing a new framework or comparing against the study.

### Model A — Count-and-Cap (no pattern recognition)

**Sources:** langgraph, letta, openai-agents-sdk, pydantic-ai, openhands (in-source surface).

A single counter — Pregel superstep count, ReAct iteration count, model turn count, or request count — caps the loop. When the cap fires, the framework either raises an exception (`GraphRecursionError`, `MaxTurnsExceeded`, `UsageLimitExceeded`), sets a stop reason (`StopReasonType.max_steps`), or labels the run STUCK and bubbles up. There is no sliding window of recent tool calls, no recent-event ring buffer, no fingerprint of `(tool_name, arguments)`. The only "detection" is "did we exceed N?".

**Unit of budget:** A framework-defined unit — superstep, turn, request — that the framework author chose for reasons unrelated to doom-loop semantics.

**Strength:** Trivial to reason about. Zero false positives. Operationally cheap. The same counter protects against runaway cost regardless of cause.

**Weakness:** Defaults are mis-tuned. langgraph's `10007` is far past the 20-turn pain threshold; openai-agents-sdk's `10` is past the typical ReAct budget. Users *can* tune them but most do not, and most frameworks do not document a recommended value. The detector is blind to *successful* repetition — a tool called 100 times with the same args and the same answer is invisible until the cap.

**Diagnostic shape:** Counter fires → exception or stop reason → user reads it.

### Model B — Cap + Minimal Structural Detector (1-step comparison or error counter)

**Sources:** crewai, agent-framework Python tool loop.

These frameworks add a narrow pattern check to the cap: crewai compares the *current* tool call against `ToolsHandler.last_used_tool` (a single-slot window); agent-framework Python maintains an `errors_in_a_row` counter that increments when any tool result has a non-None `exception` and resets on the first error-free iteration. Both detection surfaces are O(1) and live in the hot path of tool dispatch.

**Unit of budget:** Same as Model A (turn or iteration count), plus a per-step pattern check.

**Strength:** Catches the two most common doom-loop shapes — "same tool called twice in a row with identical args" (crewai) and "consecutive tool exceptions" (agent-framework Python). Cheap. Survives LLM perturbations only when the LLM happens to repeat byte-for-byte.

**Weakness:** Window size is 1. Alternating A → B → A → B patterns slip through. Intermittent failure (error, success, error, success, ...) never trips the consecutive-error counter because it resets on a single clean iteration. crewai's detector can be dodged by reordering dict keys or adding/removing null fields. agent-framework Python's primary non-streaming test for this behaviour is explicitly `@pytest.mark.skip(reason="Error handling and failsafe behavior needs investigation in unified API")` (`python/packages/core/tests/core/test_function_invocation_logic.py:1489-1491`).

**Diagnostic shape:** Detector fires → string injected as next LLM observation (crewai) or `tool_choice="none"` + forced text turn (agent-framework Python). The loop never short-circuits — the LLM is told to do something else and may or may not listen.

### Model C — Cap + LLM-Judged Detector + Reset-and-Replan

**Source:** agent-framework .NET Magentic.

The only source in the study with a true pattern detector. Each round, the Magentic manager model fills in two extra fields on its `MagenticProgressLedger`: `IsInLoop` ("Are we in a loop where we are repeating the same requests and/or getting the same responses as before?") and `IsProgressBeingMade` ("Are we making forward progress? True if just starting, or recent messages are adding value. False if recent messages show evidence of being stuck in a loop..."). The orchestrator increments `StallCount` when either is true and decrements it when progress is being made (`MagenticOrchestrator.cs:289-298`). When `StallCount > DefaultMaxStallCount = 3`, it triggers `ResetAndReplanAsync` — clears `ChatHistory`, increments `ResetCount`, sends `ResetChatSignal`, and calls `UpdatePlanAndDelegateAsync(..., replanAfterStall: true)` to regenerate the plan and pick the next speaker.

**Unit of budget:** Same as Model A (round count), plus a semantic judgement per round, plus a soft stall threshold.

**Strength:** The only intervention in the study that *replans* rather than *aborts*. Tested end-to-end (`MaxStallCount_Triggers_ResetAsync`, `Consecutive_Stalls_Trigger_ResetAsync`). The decay semantics (`StallCount = Math.Max(0, StallCount - 1)` on progress) avoid permanent bad state from one noisy round. The downstream `IsStalled` flag on `MagenticPlanReviewRequest` lets observers distinguish stall-triggered replans from normal ones.

**Weakness:** Cost — an extra manager LLM call per round. Probabilistic — the judge can be wrong. Limited to multi-agent orchestration; the single-agent tool loop in agent-framework .NET uses the same Model B approach as the Python side. Same vulnerability as every other source to *successful* repetition that the LLM judge itself fails to recognise.

**Diagnostic shape:** Detector fires → `ResetAndReplanAsync` → plan regenerated, `ChatHistory` cleared, loop continues with a new direction.

### Model D — Cap + Declarative Author-Time Rules

**Source:** letta.

Letta's `ToolRulesSolver` exposes `TerminalToolRule` (end when tool X is called), `RequiredBeforeExitToolRule` (tool X must be called before exit), `ContinueToolRule`, `InitToolRule`, `ChildToolRule`, `ParentToolRule`, and `ConditionalToolRule`. These rules key off `tool_call_history[-1]` and constrain which tools can be called next; they are author-time configuration, not runtime detection. The only runtime detector is `max_steps = 50`.

**Unit of budget:** Iteration count, plus per-step constraint filtering.

**Strength:** Letta authors can express exit conditions declaratively. The framework's job is to enforce the contract, not to guess at progress. The hard `max_steps` is a true safety net that fires regardless of how clever the agent is.

**Weakness:** Authors must anticipate stuck patterns at config time. There is no "this looks like a loop" runtime signal. The V1 legacy `last_function_failed` filter that prevented the agent from re-invoking a tool that just errored was *dropped* in V2/V3 (`letta/agents/letta_agent_v3.py:2039-2045`), so a flake on tool `T` allows the model to call `T` again on every subsequent iteration. Compaction (`LLMSummarizingCondenser`) is triggered by `context_token_estimate > context_window * 0.9` (`letta/services/summarizer/thresholds.py:27-41`), not by loop state.

**Diagnostic shape:** Rule violation → heartbeat message with `NON_USER_MSG_PREFIX` prefix → continue stepping until `max_steps` caps.

## Pattern Catalog

Each pattern below appears in multiple sources or is unusually well demonstrated in one. Use it as a vocabulary, not a copy-paste.

### Pattern 1 — Hard iteration cap as universal safety net

Every source in the study has a hard iteration cap. The vocabulary differs — `DEFAULT_MAX_ITERATIONS`, `max_iter`, `recursion_limit`, `DEFAULT_MAX_STEPS`, `DEFAULT_MAX_TURNS`, `max_iterations`, `request_limit` — but the pattern is universal. The cap is checked *first* before any softer evaluator fires, which keeps cost predictable but means the cap is the only thing that actually stops a runaway agent.

**Evidence:** agent-framework `_tools.py:90`, `_harness/_loop.py:117`; agent-framework .NET `LoopAgent.cs:68`, `MagenticTaskContext.cs:16-17`; crewai `agents/agent_builder/base_agent.py:286-288`; langgraph `_internal/_config.py:32`; letta `constants.py:74-75`; openai-agents-sdk `run_config.py:33`; pydantic-ai `usage.py:273,379-395`.

**When to copy:** Always. A framework without an iteration cap can run unbounded. Even when you have richer detectors, the cap is the backstop that prevents the LLM-judge detector from misfiring into a longer loop.

**When it is overkill:** Never. The risk of false positives is zero (it is a count, not a judgement). The risk of false negatives is bounded.

**Caveat:** Tune the default. The range in this study is 10 → 10007. 10007 is clearly wrong (langgraph benchmarks use `20000000000` for stress tests); 10 is too tight for any non-trivial agent. A reasonable middle is 25–50.

### Pattern 2 — Forced final-answer turn on cap hit

Three sources convert a doomed trajectory into a usable answer instead of raising an exception.

- **agent-framework Python tool loop** (`_tools.py:2645-2647`): when the consecutive-error counter trips or `max_iterations` is reached, sets `tool_choice = "none"` and runs one more model call that must produce plain text.
- **crewai** (`utilities/agent_utils.py:293-350`): `handle_max_iterations_exceeded` appends the i18n `force_final_answer` prompt and runs one more LLM call to synthesise a final answer.
- **langgraph prebuilt ReAct** (`chat_agent_executor.py:684-692`): when `RemainingSteps < 2 && has_tool_calls`, replaces the AI response with `AIMessage(content="Sorry, need more steps to process this request.")`.

**Evidence:** `_tools.py:2633-2647,2818-2820`; `utilities/agent_utils.py:293-350`; `chat_agent_executor.py:684-692,711-719`.

**Why it works:** The user gets *something* rather than an exception. In an embedded agent context, this is the difference between "the agent did not finish" (recoverable) and "the agent crashed" (typically a red-flag alert).

**When to copy:** When the framework is targeted at embedded agents or end-user-facing applications. When the framework is a library for code that already handles exceptions, raise instead — see openai-agents-sdk's `MaxTurnsExceeded` and pydantic-ai's `UsageLimitExceeded`.

**When it is risky:** When the underlying conversation has overflowed the context window, the forced-answer LLM call will *also* fail. agent-framework's solution is to compact first when possible. crewai's solution is to compact only on context-length error, not preemptively. Neither is robust to a full-context loop.

### Pattern 3 — Non-overlapping verdict markers for ambiguity-tolerant judging

agent-framework Python and .NET both use exactly two verdict markers, deliberately designed so neither is a substring of the other: `JUDGE_VERDICT_DONE = "VERDICT: DONE"` and `JUDGE_VERDICT_MORE = "VERDICT: MORE"` (`_loop.py:70-71`); `.NET` `DoneVerdictMarker` / `MoreVerdictMarker` (`AIJudgeLoopEvaluator.cs:70-76`). When the judge's response is ambiguous, "MORE" wins — keep looping rather than risk stopping on a false positive.

**Evidence:** `_loop.py:70-71,193`; `AIJudgeLoopEvaluator.cs:70-76,163`.

**Why it works:** Pattern-matching on LLM output is brittle. The non-overlapping marker scheme survives substring matching (`"VERDICT: DONE-ISH"` parses as neither). The ambiguity rule biases the system toward running, which combines well with the hard cap to bound cost without surprising the user with premature stops.

**When to copy:** Whenever you build an LLM-as-judge loop-stop mechanism. Pair it with the explicit "MORE wins on ambiguity" rule.

**When it is overkill:** When the judge output is structured (JSON with a known schema). Use Pydantic / Zod validation instead.

### Pattern 4 — Stall counter with decay, not absolute count

agent-framework .NET Magentic's `StallCount` increments on a stalled round and decrements on a progressing round (`MagenticOrchestrator.cs:289-298`). One noisy round cannot permanently mark the run as stuck.

**Evidence:** `MagenticOrchestrator.cs:289-298`; test `Progress_Made_Decrements_StallCountAsync` (`MagenticOrchestrationTests.cs:1065-1132`).

**Why it works:** An LLM judge is probabilistic. Without decay, a single wrong answer locks the run into a reset-replan spiral. With decay, the system needs K consecutive bad answers before acting.

**When to copy:** Whenever the loop-status signal is probabilistic. Pair with a hard absolute threshold (Magentic's `DefaultMaxStallCount = 3`).

**When it is risky:** When the agent's task is genuinely long and a slow start is normal. The decay floor should be `0`, not negative.

### Pattern 5 — Snapshot+restore session for "fresh context per iteration"

agent-framework Python `AgentLoopMiddleware.fresh_context=True` (`_harness/_loop.py:417,507-510,591-596`) and .NET `LoopAgent.FreshContextPerIteration` (`LoopAgent.cs:43-46,500-508`) both serialise the session before the loop and rebuild a fresh session between iterations, threading continuity only through the feedback/progress log.

**Why it works:** A long-running agent that accumulates tool-call history eventually exhausts context. Resetting between iterations keeps the agent's working memory bounded; the feedback log carries forward only the decisions, not the trace.

**When to copy:** For long-running orchestrator loops where individual iterations do not need full conversation continuity.

**When it is risky:** When the agent's reasoning depends on prior tool results. The reset wipes them. Compensate by ensuring the feedback log includes the result of any decision that future iterations need.

### Pattern 6 — Per-tool retry budget keyed on tool name

pydantic-ai's `RunContext.retries: dict[str, int]` (`_run_context.py:60-77`) and crewai's `tool.max_usage_count` enforce a per-tool budget. agent-framework Python's `errors_in_a_row` is a global counter, not per-tool — but it serves the same purpose.

**Why it works:** Different tools have different failure modes. A flaky network tool needs more retries than a deterministic local tool. A key budget on tool name lets the framework give different tools different budgets.

**When to copy:** Always, when you have multiple tools with different reliability profiles. Pair with a global counter (errors_in_a_row or per-tool max_retries * number_of_tools).

**When it is risky:** When the tool name changes between calls (e.g. a dynamic dispatcher that mints new tool names). Key on a stable signature.

### Pattern 7 — `tool_choice` policy reset as "infinite loop prevention"

openai-agents-sdk's `reset_tool_choice` (default `True`, `agent.py:367-369`) and agent-framework Python's `tool_choice = "required"` reset (`_tools.py:2647-2650,2818-2821`) both null the forced-tool-choice flag after the first tool has run, so the model is no longer compelled to call a tool every turn.

**Evidence:** `agent.py:367-369`; `tool_execution.py:541-549`; `_tools.py:2647-2650,2818-2821`.

**Why it works:** A model that is forced to pick a tool (`tool_choice="required"`) will keep calling tools forever once it starts, because each tool result is sent back and the next call still requires a tool pick. Unsetting `tool_choice` lets the model say "I'm done" with a plain text reply.

**When to copy:** When the framework exposes `tool_choice` as a configurable parameter. This is the cheapest possible intervention — one flag flip, no extra LLM call.

**When it is risky:** When the caller genuinely wants every turn to pick a tool (e.g. an explicit "always call a function" contract). Make the reset opt-out.

### Pattern 8 — Managed value exposing loop status to user agents

langgraph exposes `IsLastStep` (`is_last_step.py:9-12`) and `RemainingSteps` (`is_last_step.py:18-24`) as managed values that user agents can read in any node. The prebuilt ReAct agent uses `RemainingSteps` to soft-bail; user code can write a richer predicate.

**Why it works:** Most frameworks hide loop status inside the loop. langgraph exposes it as state, which means user code can implement its own detector without subclassing.

**When to copy:** When the framework is targeted at library authors who may want to write their own detectors. This is the cheapest possible extensibility surface — it adds no runtime cost and creates an obvious place for a community detector pattern.

### Pattern 9 — Declarative tool rules as anti-loop scaffolding

letta's `ToolRulesSolver` (`letta/helpers/tool_rule_solver.py`, `letta/schemas/tool_rule.py`) exposes `TerminalToolRule`, `RequiredBeforeExitToolRule`, `ContinueToolRule`, `ChildToolRule`, `ParentToolRule`, `ConditionalToolRule`. Rules key off `tool_call_history[-1]` and constrain which tools can be called next. Combined with `MaxCountPerStepToolRule`, the author can express "force `finish_task` to be called before exit" or "limit `search_memory` to 2 calls per step".

**Why it works:** Lets the agent author encode their knowledge of the task shape without writing runtime detection logic. For well-understood agent shapes (search-then-summarise, plan-then-execute), this is more reliable than an LLM-judged detector.

**When to copy:** When the framework is targeted at agent authors who can express their intent as a small FSM. Particularly valuable for code-execution agents where the desired state machine is obvious.

**When it is risky:** When the agent's task is open-ended or novel. Declarative rules cannot anticipate every stuck pattern; the framework still needs a hard cap.

### Pattern 10 — Compaction erases loop evidence by design

Every framework that ships compaction collapses older tool-call history before the loop detector (if any) reads it.

- **agent-framework Python** `ToolResultCompactionStrategy(keep_last_tool_call_groups=N)` (`_compaction.py:820-895`): collapses older tool groups into `[Tool results: name: result; ...]` summaries. Default in harness agent: `keep_last_tool_call_groups=1` (`_harness/_agent.py:74-110`).
- **agent-framework .NET** `ToolResultCompactionStrategy.DefaultMinimumPreserved = 16` (`ToolResultCompactionStrategy.cs:54`), `SummarizationCompactionStrategy.DefaultMinimumPreserved = 8`, `SlidingWindowCompactionStrategy.DefaultMinimumPreserved = 1` turn.
- **crewai** `summarize_messages` (`utilities/agent_utils.py:920-1003`): `messages.clear()` and `messages.extend(system_messages)` plus a single summary message. Reached only from `handle_context_length` after `is_context_length_exceeded(e)`.
- **pydantic-ai** provider-driven `CompactionPart` (`messages.py:1682-1730`): replaces past history with an opaque token on OpenAI Responses / Anthropic.
- **openhands** `LLMSummarizingCondenser` with `max_size=240, keep_first=2` (`app_conversation_service_base.py:578-612`): first two events verbatim, everything else summarised.

**Evidence:** as above.

**Why it matters:** None of the framework detectors inspect message history (the only exception is agent-framework .NET Magentic's LLM judge, which *does* see the chat history but is itself an LLM call, not a structural scan). So today, compaction erasing evidence is not a real cost. But it forecloses the cheapest place to add a structural detector later — once history is summarised, you cannot fingerprint raw tool calls anymore. If a future detector wants to do "fingerprint last 8 raw tool calls", it must run on the *pre-compaction* history or maintain a parallel buffer of uncompacted fingerprints.

**When to copy:** Compaction for context-window management is necessary. The contract that matters is "the compaction system does not maintain a parallel fingerprint buffer" — which is what every framework in this study does.

**Caution sign:** If your framework summarises tool calls but does not also keep a rolling `(tool_name, args_hash)` log, a future detector will be blind.

### Pattern 11 — "Reset-and-replan" instead of "abort-and-raise"

Only agent-framework .NET Magentic does this (`MagenticOrchestrator.cs:289-300,337-344`). On stall, it clears `ChatHistory`, increments `ResetCount`, sends `ResetChatSignal`, and regenerates the plan with `replanAfterStall: true`.

**Why it works:** Aborting wastes the work the agent has done. Replanning keeps the trajectory but resets the LLM's context, giving it a chance to see the problem with fresh eyes. The hard round/reset limits (`MaxRoundCount`, `MaxResetCount`) cap the replan budget.

**When to copy:** When the framework is multi-agent or task-decomposition-oriented. When the agent is single-task and the work is sequential, abort is fine.

**When it is risky:** When the agent's plan is a long-running external state mutation (file edits, database writes) — replanning may produce a contradictory plan that conflicts with already-applied changes.

### Pattern 12 — Streaming fast-path re-raise suppression

openai-agents-sdk's streaming runner sets `_max_turns_handled = True` whenever a user-supplied `error_handlers["max_turns"]` handler returns a value (`result.py:793-801`). The post-loop `_check_errors` then does not re-raise.

**Why it works:** Streaming paths cannot raise out of a coroutine mid-stream. The flag distinguishes "handler produced a result" from "no handler configured".

**When to copy:** Whenever the framework supports both streaming and non-streaming entry points with a unified exception path.

## Key Differences

The differences between sources are not quality differences — they are valid responses to different constraints.

### Why sources diverge on detector placement

**Library vs. application.** langgraph, pydantic-ai, openai-agents-sdk are libraries that other developers embed. They raise on overrun so the embedder can decide. agent-framework, crewai, openhands are application runtimes that wrap the loop and prefer to produce a usable output even on overrun.

**Multi-agent vs. single-agent.** agent-framework .NET Magentic's LLM-judged stall detection only makes sense when there is a "manager" LLM that can evaluate progress — that is the multi-agent orchestrator. Single-agent loops (crewai, letta, openai-agents-sdk, pydantic-ai) cannot pay the cost of an extra judge call per round.

**Iteration unit.** langgraph's superstep is not the same as crewai's ReAct iteration or pydantic-ai's model turn. A superstep can fan out into many parallel tool calls; a ReAct iteration is one LLM call + at most one tool. The right detector depends on the unit.

### Why sources diverge on compaction interaction

**Where the evidence lives.** Compaction strategies target the LLM message history. If the detector lives on the LLM message history (langgraph's prebuilt agent reads `messages`), compaction affects detection. If the detector lives on a parallel ring buffer (none of these frameworks have one), compaction does not.

**Who triggers compaction.** crewai, pydantic-ai (provider-driven), and openhands trigger compaction on context-window overflow. agent-framework runs compaction before/after model calls (`CompactionProvider.before_strategy` / `after_strategy`). None of them tie compaction to loop state.

### Why sources diverge on forced-final-answer vs. raise

**Embedding context.** agent-framework Python, crewai, and langgraph's prebuilt ReAct soft-bail are all targeted at embedded agent use — the user wants *some* output, not an exception. openai-agents-sdk, pydantic-ai, langgraph's bare engine are all targeted at library authors — the embedder catches and decides.

**Determinism.** A forced final-answer LLM call is itself a probabilistic step that can fail. A library author who needs determinism would rather get the exception and handle it themselves.

## Tradeoffs

### Cap-then-abort vs. cap-then-force-final-answer

| Aspect | Abort (langgraph, openai-agents-sdk, pydantic-ai) | Force final answer (agent-framework Python, crewai, langgraph prebuilt) |
|--------|--------------------------------------------------|-------------------------------------------------------------------------|
| **Cost on overrun** | Zero further LLM calls | One more LLM call to synthesise answer |
| **Output on overrun** | Exception | Best-effort text response |
| **Failure mode** | User sees error | User sees possibly-wrong answer |
| **Best fit** | Library users with exception handlers | Embedded agents, end-user apps |

### Structural detector vs. LLM-judged detector

| Aspect | Structural fingerprint (not in study) | LLM judge (agent-framework .NET Magentic) |
|--------|--------------------------------------|--------------------------------------------|
| **Cost per iteration** | O(1) — hash + dict lookup | One extra LLM call |
| **Catches semantic repetition** | No — identical `(name, args)` only | Yes — model can recognise paraphrase |
| **Catches alternating A/B** | Yes with K-gram window | Maybe — depends on judge's framing |
| **False positives** | Window-size dependent | Judge-bias dependent |
| **Best fit** | Cost-sensitive loops, well-understood tool set | Open-ended tasks, large action spaces |

### Cap default: tight vs. generous

| Aspect | Tight (openai-agents-sdk = 10, agent-framework Python = 40) | Generous (langgraph = 10007, letta = 50) |
|--------|--------------------------------------------------------------|---------------------------------------------|
| **Cost on overrun** | Low | High |
| **Risk of cutting off legitimate work** | High | Low |
| **Risk of paying for 20+ useless turns** | Low | High |
| **Best fit** | Cost-sensitive production | Research / dev / autonomous tasks |

### Per-tool retry budget vs. global consecutive-error counter

| Aspect | Per-tool (pydantic-ai, crewai max_usage_count) | Global counter (agent-framework Python errors_in_a_row) |
|--------|-----------------------------------------------|---------------------------------------------------------|
| **Granularity** | Different tools have different budgets | One budget for all |
| **Failure-mode coverage** | Catches per-tool flakes | Catches "every tool is failing" |
| **Best fit** | Heterogeneous tool sets | Homogeneous tools |

## Decision Guide

**If you are building a library:** Raise on cap hit. Do not force a final-answer call. Expose iteration count and last-tool-call state as managed values (langgraph's `RemainingSteps` pattern) so library users can write their own detectors. Document the recommended cap default but let users override.

**If you are building an embedded agent runtime:** Force a final-answer call on cap hit (crewai / agent-framework pattern). Keep the structural detector (1-step tool repeat or errors_in_a_row) in the hot path. Pair with an LLM-judged detector if your loop is multi-agent or has a natural "manager" role. Pair with declarative tool rules (letta pattern) if your agents are author-configured FSMs.

**If your agents are long-running autonomous workers (research, dev, etc.):** Higher default cap (50+). External LLM-judge detector optional. Compaction must preserve a parallel tool-call digest buffer if you want post-mortem stuck analysis.

**If your agents are short-running reactive (one tool, one answer):** Tight cap (10–25). Skip the LLM judge. Cap is sufficient.

**Always:** Have a hard cap. Even with a rich detector, the cap is the safety net.

## Practical Tips

### Patterns to copy

1. **A hard iteration cap, checked before any soft evaluator.** Every source has one; the absence is the dimension's failure mode.
2. **A forced final-answer turn on cap hit** if you are building an embedded agent runtime. The user gets *some* answer.
3. **Per-tool retry budgets keyed on tool name.** Different tools have different reliability.
4. **A consecutive-error counter** (`errors_in_a_row`) that resets on the first clean iteration. Catches the cheapest doom-loop shape — repeated tool failures.
5. **A 1-step identical-tool-call detector** that injects a string into the next LLM observation rather than aborting. Cheap, catches the obvious case, lets the LLM self-correct.
6. **Non-overlapping verdict markers + "MORE wins on ambiguity"** for any LLM-as-judge loop-stop.
7. **Stall counter with decay + hard absolute threshold** for probabilistic judges.
8. **Snapshot+restore session for "fresh context per iteration"** in long-running orchestrator loops.
9. **Managed values exposing loop status to user agents** (langgraph's `RemainingSteps`).
10. **Declarative tool rules** (letta's `ToolRulesSolver`) when the framework is author-configured.
11. **Streaming flag `_max_turns_handled`** (openai-agents-sdk) to coordinate handler-based recovery with the streaming post-loop error check.
12. **`tool_choice` policy reset** (openai-agents-sdk's `reset_tool_choice`) as the cheapest possible "infinite loop prevention" for forced-tool-choice callers.

### Patterns to avoid or delay until needed

1. **LLM-judged "IsInLoop" / "IsProgressBeingMade" detectors** unless you are already multi-agent. The cost is significant and the benefit is only on semantic repetition that structural fingerprinting misses.
2. **Reset-and-replan.** Powerful but risky when external state has already been mutated. Only copy if your plan structure is internal.
3. **Heavy per-tool retry budgets.** Default `1` (pydantic-ai) is fine; `> 3` invites masking of real bugs.
4. **Window-of-N structural fingerprinting at the framework layer** — this is cheap and would be valuable, but no source in this study ships one. The risk is "yet another knob the user has to tune".
5. **Default recursion_limit = 10007** (langgraph). Even with a low default, no framework recommends raising it past 50 for a typical ReAct agent.

### Decision rules

- **Default iteration cap:** 25–50 for a typical ReAct agent. 10 if you want the framework to feel "safe" to non-technical users (openai-agents-sdk's choice). 50+ if the agent is autonomous.
- **Default consecutive-error threshold:** 3 (agent-framework Python's choice). 2 is too tight (one transient network blip and you abort). 5 is too loose (you burn budget on real bugs).
- **Default per-tool retry budget:** 1–2. More than 3 means you are masking real issues.
- **Default 1-step identical-tool-call detector:** enabled, with the i18n message as the intervention (crewai pattern). Disable for polling-style agents.
- **When to add an LLM-judge detector:** only when you have a natural "manager" role (multi-agent orchestrator) and the cost of an extra LLM call per round is acceptable.
- **When to add reset-and-replan:** only when (a) the plan is internal state, (b) the agent is multi-step, and (c) the round/reset budget is bounded.

## Anti-Patterns / Caution Signs

- **Default iteration cap that is so high the agent burns real money on a doom loop before stopping.** langgraph's `recursion_limit = 10007` is the canonical example. If your default is > 100, you are optimising for benchmarks, not users.
- **No detector at all, only a cap.** Models that loop on successful calls are invisible until the cap. Five of seven sources in this study are in this state.
- **A "loop prevention" flag that is actually a `tool_choice` policy reset** (openai-agents-sdk's `reset_tool_choice`). The docstring says "infinite loop prevention" but the implementation does not inspect calls. Either rename it or add real detection.
- **Compaction that is silent about loop evidence loss.** Every framework in this study summarises tool calls but none of them warn that a downstream detector would be blind after compaction.
- **A consecutive-error counter that resets on a single clean iteration** (agent-framework Python). An alternating failure/success pattern never trips it. Pair with a "consecutive identical errors" detector that requires *multiple* identical errors in a row, or a "fraction of recent calls that errored" detector.
- **Window = 1 structural detector.** Catches only "same tool twice in a row". Pattern A → B → A → B slips through.
- **A detector that compares arguments structurally.** LLMs can perturb arguments (key order, trailing whitespace, redundant null fields) to dodge byte-identical comparison. Either normalise arguments before comparing or use semantic similarity.
- **An LLM-judged detector that biases toward "MORE" without a hard cap.** A judge that always says "more" combined with a generous cap is worse than no detector.
- **A "STUCK" terminal status that is conflated with "ERROR" downstream** (openhands webhook router). Operators lose the ability to measure doom-loop rate vs. crash rate separately.
- **A declarative rule that authors must remember to configure** (letta's tool rules). Defaults should include at least one anti-loop rule (`TerminalToolRule` with a sensible default, or `RequiredBeforeExitToolRule`).
- **A skipped test on the primary behaviour path.** agent-framework Python's `test_function_invocation_config_max_consecutive_errors` is `@pytest.mark.skip(...)` at `test_function_invocation_logic.py:1489`. The streaming counterpart is tested, but the non-streaming path is the primary user path.
- **A counter that resets on session reset but not on success** — these exist in no source but would be a natural bug to introduce.
- **`max_method_calls = max_iter * 10`** (crewai Flow runtime). Generous, but if a listener legitimately fires 100+ times, the framework aborts with a `RecursionError` rather than a graceful error. Consider raising to `1000` or making it configurable.
- **Tool-result dedupe keyed only on `(type, call_id)`** (openai-agents-sdk's `_dedupe_tool_call_items`). A repeated call with a new `call_id` is invisible. If the goal is "don't run the same call twice", compare `(name, args_hash)`.

## Notable Absences

**Structural `(tool_name, args)` sliding-window fingerprint.** No source in this study fingerprints tool calls over a sliding window of the last K iterations. This is the cheapest possible doom-loop detector and the most general one. Its absence is the most striking finding of the dimension.

**Alternating A/B detector.** No source detects "called tool A, then B, then A, then B" as a pattern.

**No-progress monologue detector.** No source detects "model returned plain text without any tool call for N consecutive turns" as a stuck pattern. agent-framework .NET Magentic's `IsProgressBeingMade` is the closest — it is LLM-judged and one-shot per round, not a rolling window.

**Consecutive identical error response detector.** No source fingerprints the *tool response* (as opposed to the call). A tool that returns the same error message every time (without raising an exception) is invisible to every detector in the study.

**Loop-evidence preservation in compaction.** No source maintains a parallel ring buffer of `(tool_name, args_hash, result_hash)` outside the LLM message history. Compaction therefore forecloses future detectors that wanted raw evidence.

**Dedicated `stuck_loop` span/event in tracing.** No source ships a tracing signal for "loop detector fired". pydantic-ai mentions `stuck_loop` as a future span type; openai-agents-sdk does not; agent-framework logs at WARNING/INFO but does not emit spans.

**Reset-and-replan for single-agent loops.** Only Magentic (multi-agent) does this. A single-agent equivalent would be "clear the recent tool-call history and prompt the model with the same question again" — none of the sources implement that.

**Test coverage for doom-loop patterns.** None of the sources tests "agent that calls the same tool 20 times with the same args stops short of cap". The tests that exist cover the cap and the reset/replan, not the in-between.

**Public tunability of detector windows.** Where detectors exist, their windows are hardcoded (`DEFAULT_MAX_CONSECUTIVE_ERRORS_PER_REQUEST = 3`, `DefaultMaxStallCount = 3`, `max_iter = 25`). Users can override the cap but not the detector sensitivity.

## Per-Source Notes

### agent-framework (score 5)

The framework is split between two implementations that do not share code or contracts. Python's tool loop has a 1-step consecutive-error counter (`errors_in_a_row`, `_tools.py:2249-2259`) plus a hard `max_iterations = 40` cap, but the primary non-streaming test is `@pytest.mark.skip(...)` at `test_function_invocation_logic.py:1489-1491`. .NET's Magentic orchestrator has a tested LLM-judged `IsInLoop` / `IsProgressBeingMade` detector (`MagenticProgressLedger.cs:22-29`), a `StallCount` counter with decay (`MagenticOrchestrator.cs:289-298`), and a `ResetAndReplanAsync` intervention (`MagenticOrchestrator.cs:289-300,337-344`) triggered at `DefaultMaxStallCount = 3` (`MagenticTaskContext.cs:16`). .NET's `LoopAgent` is the orchestrator-level re-invocation loop and uses the same model-B approach as the Python side (`LoopAgent.cs:195-200`). Compaction can erase loop evidence: Python's `ToolResultCompactionStrategy` collapses older groups into `[Tool results: ...]` summaries (`_compaction.py:820-895`), .NET's `DefaultMinimumPreserved = 16` (`ToolResultCompactionStrategy.cs:54`).

### crewai (score 5)

Detection lives in `ToolUsage._check_tool_repeated_usage` (`tools/tool_usage.py:728-738`), which compares the current call against `ToolsHandler.last_used_tool` (a 1-slot window). On a match, an i18n string ("task_repeated_usage") is returned as the tool's output (`translations/en.json:49`). The hard cap is `max_iter = 25` (`agents/agent_builder/base_agent.py:286-288`) and the cap-exceeded handler `handle_max_iterations_exceeded` (`utilities/agent_utils.py:293-350`) forces a final-answer LLM call. Per-tool `max_usage_count`, `_max_parsing_attempts = 3`, agent-level `max_retry_limit = 2`, Plan-and-Execute `max_replans = 3`, and Flow runtime `max_method_calls = 100` are the other narrow guards. There is no test for `_check_tool_repeated_usage` — only mocks. Compaction (`summarize_messages`) is triggered only by context-length error, not by loop detection.

### langgraph (score 3)

Detection is entirely count- and time-based. `DEFAULT_RECURSION_LIMIT = 10007` (`_internal/_config.py:32`) is the superstep cap; `step_timeout` is the per-superstep wall-clock watchdog; `RetryPolicy` and `TimeoutPolicy` are per-task retry/idle mechanisms (`types.py:416-509`). The prebuilt ReAct agent's `RemainingSteps < 2` soft bail (`chat_agent_executor.py:620-634,684-692`) is the closest thing to early stop and is itself count-driven. There is no pattern detection, no recent-tool-call window, no `recent_messages` field on `AgentState` (`chat_agent_executor.py:57-62`). Grep for `doom`, `loop_detection`, `recent_tool_calls`, `no_progress`, `replan` returns 0 hits in the framework code. The `LANGGRAPH_API_URL` references suggest a server component not in this source tree.

### letta (score 2)

Only `DEFAULT_MAX_STEPS = 50` (`constants.py:74-75`) and the declarative `ToolRulesSolver` (`letta/helpers/tool_rule_solver.py`). The V1 legacy `last_function_failed` filter (`letta/agent.py:325-326`) that prevented the agent from re-invoking a tool that just errored was *dropped* in V2/V3. Compaction is context-window driven (`SUMMARIZATION_TRIGGER_MULTIPLIER = 0.9`, `services/summarizer/thresholds.py:27-41`). The 13 stop reasons (`schemas/letta_stop_reason.py:9-22`) include no `stuck`, `no_progress`, or `doom_loop` value. A multi-agent agent calling itself recursively via `send_message_to_agent_async` (`constants.py:155`) has no inner guard; only the outer `max_steps` saves the user.

### openai-agents-sdk (score 2)

`DEFAULT_MAX_TURNS = 10` (`run_config.py:33`) is the only iteration cap. `reset_tool_choice` (default `True`, `agent.py:367-369`) is the only mechanism named "infinite loop prevention" in the docs (`docs/agents.md:423-425`) — it nulls `tool_choice` after the first tool use to break forced-tool-choice chaining. `_dedupe_tool_call_items` (`tool_planning.py:158-174`) catches replays of identical `call_id`s but not new-call-id repetitions. `MaxTurnsExceeded` (`exceptions.py:56-63`) is the universal stuck signal; `error_handlers` allow user-supplied recovery. Compaction (`OpenAIResponsesCompactionSession`, `memory/openai_responses_compaction_session.py:78-340`) is opt-in and erases raw tool outputs.

### openhands (score 2)

The source under study contains no detection logic of its own. The agent loop lives in an external `openhands-sdk==1.29.0` package (`pyproject.toml:64-66`). Within this source: `ConversationSettings.max_iterations` is the only knob (`settings_models.py:109`), forwarded to the SDK via `conv_settings.create_request(...)` (`live_status_app_conversation_service.py:1729,1918`). `ConversationExecutionStatus.STUCK` is recognised as a terminal status (`webhook_router.py:141-144`), and the webhook fires `track_conversation_errored` with `terminal_state='stuck'`. Integration callbacks (Slack/GitHub/etc.) only post on `event.value == 'finished'` (`github_v1_callback_processor.py:54-59`), so stuck conversations leave no integration trace. Compaction is `LLMSummarizingCondenser` (`app_conversation_service_base.py:578-612`) with `max_size=240, keep_first=2`. The SDK's stuck detector is the actual subject and cannot be audited from this source.

### pydantic-ai (score 4)

Five layered caps: per-tool `default_max_retries = 1` (`tool_manager.py:92-93`), per-run `max_output_retries = 1`, `UsageLimits.request_limit = 50` (`usage.py:273`), `UsageLimits.tool_calls_limit = None` (unbounded by default, `usage.py:275`), and three token caps. `ModelRetry` → `RetryPromptPart` is the in-loop retry hint (`tool_manager.py:183-191`) that travels through `_clean_message_history` into the next `ModelRequest`. `pydantic_graph.GraphRun` has no step counter of its own (`graph.py:664-753`); the agent layer adds `request_limit`. Provider-driven `CompactionPart` (`messages.py:1682-1730`) replaces past history with an opaque token on OpenAI/Anthropic. No detector anywhere; no test for non-progress patterns.

## Open Questions

1. **What is the optimal window size for a structural `(tool_name, args_hash)` detector?** No source has one, so there is no empirical guidance. A reasonable default might be 8 turns with a threshold of 6, but this needs measurement against real agent traces.
2. **Does semantic-repetition detection (embedding similarity over the last K tool responses) catch more doom loops than LLM-judge detection?** The frameworks in this study have not tested this. The cost of an embedding call is lower than an LLM-judge call but higher than a structural fingerprint.
3. **Should a forced final-answer turn be triggered at 80% of the iteration cap, not 100%?** Today every framework fires at 100%. A pre-emptive forced answer at 80% would give the user a usable answer while leaving 20% of the budget for retry. None of the studied sources do this.
4. **Is the asymmetry in agent-framework (Python skipped test, .NET Magentic LLM judge) intentional or a side-effect of different teams owning the two implementations?** Without organisational evidence this cannot be answered.
5. **What patterns does openhands' `openhands-sdk` actually detect?** The source under study cannot answer this; the SDK is external and out of scope.
6. **Does the langgraph server (referenced via `LANGGRAPH_API_URL`) implement a doom-loop detector that is not in the framework code?** Out of scope of this study.
7. **How does crewai's 1-step detector interact with multi-step parallel tool calls?** The detector lives in `ToolUsage._use/_ause`; native-tool loops (`_invoke_loop_native_tools`) do not appear to invoke it (`crew_agent_executor.py:484+`). This is a likely coverage gap.
8. **Why was letta's V1 `last_function_failed` filter dropped in V2/V3?** The migration was deliberate but the rationale is not documented in code. The cost is real (a flake on tool T now allows infinite re-invocation up to `max_steps`).
9. **Does the `_clean_message_history` in pydantic-ai preserve the per-tool retry counter?** `_run_context.py:60-77` says yes (counter is per-tool, not per-message), but no test asserts this across message-merge boundaries.
10. **Is `tool_use_behavior="stop_on_first_tool"` (openai-agents-sdk `agent.py:345-365`) a doom-loop defence?** It stops after the first tool returns, which prevents tool-call chains entirely. This is a deliberate early-stop, not a detector — but it overlaps with the doom-loop problem.

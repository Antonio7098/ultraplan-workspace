## Executive Summary

All five sources treat schema evolution as a serious concern, but the solutions cluster into two distinct postures: **envelope-versioned persistence** (agent-framework, langgraph) where the persisted blob itself carries a version field and migrations run on read, and **tool-driven database migrations** (temporal, langfuse, letta) where the database schema is the primary evolution surface and JSON/blob compatibility is layered on top. A hybrid pattern emerges where some sources (langfuse, temporal) build first-class migration registries for database DDL while still relying on Zod `safeParse`, protobuf `DiscardUnknown`, or per-field `JsonExtensionData` for application blobs. Common strengths across the corpus: idempotent DDL with `IF NOT EXISTS` / `IF EXISTS`; literal version discriminators in the persisted envelope; and runtime compat layers (discriminated unions, fallback content types, allow-listed deserializers). Common weaknesses: weak or absent committed old-state fixtures; asymmetric enforcement between write and read paths or between runtimes; and ad-hoc "if nil, populate" patches rather than a single migration registry. **Will today's saved run still load after next month's refactor?** — defensively yes for most sources via additive-only evolution and forward-compat envelopes, but with documented risks in Python runtimes of agent-framework, the SQLite path of langgraph, the informational JSON versioning in letta, and the silent `DiscardUnknown` drop in temporal.

## Core Thesis

**Schema versioning in agent runtimes is best understood as a layered contract problem with three independent layers, and each source chooses a different point to anchor its version-of-record.** The three layers are (a) the database schema, (b) the persisted blob envelope, and (c) the inline/optional fields within those blobs. Most sources anchor at one layer and rely on additive evolution at the others: temporal anchors at the database schema via manifest-driven `MinCompatibleVersion` and lets proto3 wire-format tolerance carry the blob layer; langfuse anchors at the database schema (Prisma + ClickHouse + recoverable background migrations) and lets Zod `.optional()` carry the blob layer; agent-framework and langgraph anchor at the blob envelope itself with first-class `schemaVersion`/`v` fields; letta is the only source where neither anchor is enforced — the JSON file format carries an informational `revision_id` that the importer does not inspect, leaving the blob layer contractually weak.

The strongest patterns are: **manifest-driven, manifest-md5-protected DB migrations with required `MinCompatibleVersion`** (temporal); **recoverable, lock-protected background migrations with persisted `state` Json cursors** (langfuse); **integer/enum version keys on the persisted envelope that gate in-place migration functions called on every read** (langgraph); and **literal semver `schemaVersion` with `$type` polymorphism + an explicit `unknown` content fallback** (agent-framework). The weakest pattern is **"write the version, log the mismatch, proceed anyway"** (letta's legacy marshmallow `check_version` at `letta/serialize_schemas/marshmallow_agent.py:223-230`), which satisfies the contract surface without providing actual safety.

## Approach Models

The five sources implement schema versioning under materially different operational constraints — library vs. multi-tenant SaaS, single-language vs. cross-language, single-database vs. multi-database. The right model depends on whether persisted state is *primarily relational* (langfuse, letta, temporal) or *primarily a structured blob* (agent-framework, langgraph). The following are the dominant archetypes:

### 1. Envelope-versioned structured blob (agent-framework, langgraph)

Both sources declare a top-level version field on the persisted blob and run migrations on read.

- **agent-framework** uses a *string* `schemaVersion` (semver, regex `^\d+\.\d+\.\d+$`) declared as a required first property in a JSON Schema 2020-12 contract (`schemas/durable-agent-entity-state.json:207-216`). Migrations are *not* version-targeted; compatibility is structural via polymorphic `$type` discriminators, an `unknown` content envelope, and `[JsonExtensionData]` round-tripping on every container.
- **langgraph** uses an *integer* `v` on a `Checkpoint` TypedDict (`libs/checkpoint/.../base/__init__.py:92-96`), with `LATEST_VERSION` constants declared per layer (base=2, Pregel=4). Migrations are *explicit and version-gated*: `Pregel._migrate_checkpoint` (`libs/.../pregel/main.py:1135-1142`) and `StateGraph._migrate_checkpoint` (`libs/.../graph/state.py:1612-1690`) rewrite the in-memory checkpoint based on `v` comparisons, wired into the `PregelLoop` constructor and called after every `checkpointer.get_tuple`.

**When to copy:** when the persisted unit is a structured blob whose shape evolves through additive changes and where the *contents* (not just the relational column count) are the migration surface — typical of agent conversation histories, workflow checkpoints, and task state envelopes.

### 2. Manifest-driven DB schema versioning with envelope-level compatibility (temporal)

`temporal` separates the three concerns explicitly:

- **DB schema** — per-store `schema_version` table, `schema_update_history` audit log, `manifest.json` per migration step with mandatory `MinCompatibleVersion`, MD5 of manifest written to audit log. Generic upgrade tool (`tools/common/schema/updatetask.go:77-130`) reads manifests, computes diffs, applies idempotent DDL.
- **Persisted state envelope** — `commonpb.DataBlob{Data, EncodingType}` (proto3 or JSON), decoded via a symmetric codec that returns terminal errors on unknown encodings (`common/persistence/serialization/codec.go:104-121`).
- **Inline fields** — protobuf3 wire-format tolerance, `DiscardUnknown: true` for stripped events and JSON, plus hand-coded `if-nil-init` patches in `serializerImpl`.

The most distinctive feature is that `MinCompatibleVersion` is *required* in every manifest (`tools/common/schema/updatetask.go:288-294`), pushing the compatibility contract into the migration itself rather than into a separate compatibility matrix.

**When to copy:** when persisting across multiple database backends (Cassandra, MySQL, Postgres, SQLite, Elasticsearch), when a deliberate one-way-only migration policy is acceptable, and when the on-wire format is a single canonical type with optional extension fields.

### 3. Tool-driven DB migrations + phase-gated cutovers (langfuse)

`langfuse` runs two separate DB migration systems (Prisma for Postgres, golang-migrate for ClickHouse) and layers a **recoverable background migration** concept on top:

- `BackgroundMigrationManager.run` (`worker/src/.../backgroundMigrationManager.ts:6-185`) holds a serializable-transaction lock with 60-second TTL and 15-second heartbeats; `state` Json column allows resuming mid-batch.
- V4 ingestion uses `LANGFUSE_MIGRATION_V4_WRITE_MODE ∈ {legacy, dual, events_only}` and `LANGFUSE_MIGRATION_V4_NATIVE_OTEL_BEHAVIOUR` env flags to gate dual-write and dual-read; `validateV4Flags` throws on incoherent combinations (`worker/src/env.ts:519-540`).

JSON-blob compat is delegated to Zod discriminated unions plus `.optional()` and `.transform()` reshaping, with one exception (`EvalOutputDefinition`) that uses an explicit `version: z.literal(2)` discriminator with a `LegacyEvalOutputDefinitionSchema` branch (`packages/shared/src/features/evals/outputDefinition.ts:13-134`).

**When to copy:** when long-running, non-atomic data moves are part of the migration surface (backfilling from Postgres to ClickHouse, rewriting millions of rows); when dual-write windows are needed because the read layer cannot be flipped atomically; and when a feature-flag-gated cutover model is acceptable.

### 4. Linear Alembic + informational JSON versioning (letta)

`letta` uses a 167-revision Alembic chain for relational state and a Pydantic `AgentFileSchema` for portable agent files. The JSON file carries `metadata.revision_id` (the *current Alembic revision* at export), but the import path does not consult it (`agent_serialization_manager.py:488`; check absent at `agent_serialization_manager.py:993-1020`). Legacy `AgentSchema` has a `version` field but the marshmallow loader prints mismatch and proceeds anyway (`letta/serialize_schemas/marshmallow_agent.py:223-230`). Legacy format is hard-rejected at the REST API (`agents.py:322-323,580-583`) rather than adapted.

**When to copy:** carefully — this is the weakest pattern in the corpus. Useful elements include Alembic's per-dialect guard (`if not settings.letta_pg_uri_no_default: return` repeated in ~150 migrations) and the `preserve_on_migration` template-migration flag on Block. The informational-versioning pattern should be avoided; the hard-cutover pattern loses existing user data and is the most operationally expensive choice.

## Pattern Catalog

### Pattern A — Schema-as-contract with JSON Schema + `$type` polymorphism (agent-framework)

**Problem solved:** cross-language (.NET + Python) durable state evolution without coordinated major bumps.

**Where seen:** `agent-framework`. Authoritative schema at `schemas/durable-agent-entity-state.json:111-216`; mirror types in `dotnet/.../State/*.cs` and `python/.../_durable_agent_state.py:394-443`; `unknown` content type envelope at `:111-119`; `[JsonExtensionData]` on every container at `DurableAgentStateData.cs:30-31` and equivalents.

**Why it works:** the JSON Schema is the contract; both runtimes consume it. Adding a new content variant requires only a new `$type` branch in the discriminated union, not a schema major bump. Unrecognized content is captured by `unknown` rather than rejected.

**When to copy:** any cross-language persistence boundary where the contract must be stable across runtimes and where new variants are added incrementally. Particularly well-suited for agent message histories with diverse content types.

**When overkill / risky:** if the schema change is *breaking* (rename, remove, split) rather than additive, the entire pattern collapses — .NET `DurableAgentStateJsonConverter` hard-rejects anything beyond `Major == 1` (`DurableAgentStateJsonConverter.cs:38-41`), which is a global data-loss event for any operator that upgrades without a separate data migration. The asymmetric enforcement between .NET (strict) and Python (silent reset on missing `schemaVersion`) is also a known fragility.

### Pattern B — In-place migration on read with integer `v` key (langgraph)

**Problem solved:** resume historical threads written by older code versions without operator action.

**Where seen:** `langgraph`. `Checkpoint` TypedDict declares `v: int` (`libs/checkpoint/.../base/__init__.py:92-96`); `LATEST_VERSION = 4` at `libs/langgraph/langgraph/pregel/_checkpoint.py:21`; `_migrate_checkpoint` called from `PregelLoop` constructor after every `checkpointer.get_tuple` (`libs/.../pregel/_loop.py:1639-1642`); `SAVED_CHECKPOINTS` corpus in `libs/langgraph/tests/test_checkpoint_migration.py:272-1397`.

**Why it works:** the read path normalizes to the current shape before any business logic runs, so the rest of the codebase can assume the latest format. `copy_checkpoint` preserves `v` verbatim so a resumed thread stays at its original `v` until something explicitly writes a new checkpoint.

**When to copy:** when the persisted unit is owned by a single language runtime (not cross-language) and where the migration logic is contained enough to run on every read without measurable cost. Especially useful when the migration is a *relocation* (e.g., `pending_sends` → `channel_values[TASKS]`) rather than a rewrite.

**When overkill / risky:** requires the migration function to be idempotent in case the same tuple is re-read many times; requires the function to be stable across versions (no signature changes); and the SQLite backend relies entirely on the Pregel layer because it has no `MIGRATIONS` runner (`libs/checkpoint-sqlite/.../sqlite/__init__.py:129-166`), so any DDL change must ship as a one-shot script.

### Pattern C — Manifest-driven DB migrations with required `MinCompatibleVersion` (temporal)

**Problem solved:** multi-database schema evolution with a documented compatibility contract per migration.

**Where seen:** `temporal`. Manifests at `schema/cassandra/temporal/versioned/v1.13/manifest.json:1-7` and `schema/mysql/v8/temporal/versioned/v1.19/manifest.json:1-7`; manifest parser at `tools/common/schema/updatetask.go:267-307`; `MinCompatibleVersion` field required at `:288-294`; manifest MD5 written to `schema_update_history` at `:156-167,301-304`.

**Why it works:** the compatibility contract lives *in the migration itself*, so reviewers see it in code review and operators see it in `git log`. The MD5 fingerprint detects post-hoc edits to applied manifests.

**When to copy:** any multi-database system where the schema tool needs to evolve multiple stores in lockstep, and where operator decisions (rollback vs. forward-fix) need to be governed by an explicit version contract rather than tribal knowledge.

**When overkill / risky:** when the migration surface is small enough that a `CURRENT_VERSION` constant in code would suffice, the manifest machinery is over-engineered. Manifests are also typically forward-only; `VerifyCompatibleVersion` accepts code rollback but refuses DB downgrade (`common/persistence/schema/version.go:21-28`), so a botched upgrade cannot be undone by re-running the tool.

### Pattern D — Recoverable, lock-protected background migrations with persisted state cursor (langfuse)

**Problem solved:** long-running data moves (Postgres → ClickHouse backfills, dataset-run backfills) that must survive worker crashes without leaving the system half-migrated.

**Where seen:** `langfuse`. `BackgroundMigrationManager` at `worker/src/backgroundMigrations/backgroundMigrationManager.ts:6-185`; `BackgroundMigration` model at `packages/shared/prisma/schema.prisma:279-292`; `state` Json column persisted between batches in `worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:67-83,129-136`; README at `worker/src/backgroundMigrations/README.md:1-40` codifying idempotency, single-runner, abort/resume.

**Why it works:** the lock + heartbeat pattern prevents two workers from racing on the same migration; the `state` Json column lets the worker resume from the last cursor after a crash. The naming convention (sortable date prefix) guarantees migrations run in order.

**When to copy:** any migration that touches more rows than fit in a single transaction or that needs to be safely interruptible. Particularly important when the migration window is measured in hours/days.

**When overkill / risky:** the hard-coded `backgroundMigrationId` UUIDs (e.g., `"5960f22a-748f-480c-b2f3-bc4f9d5d84bc"` at `migrateTracesFromPostgresToClickhouse.ts:12`) tie a SQL `INSERT` row to a specific script — renaming a script file can silently desynchronize. The README also notes that some migrations have "upgrade stops" (code paths that depend on the migration having finished) but no automatic pause-on-migration-incomplete (`worker/src/backgroundMigrations/README.md:22-23`).

### Pattern E — Phase-gated dual-write/dual-read cutover via feature flags (langfuse)

**Problem solved:** rolling out a new persistence layer (e.g., V4 events) without a maintenance window, by letting the same binary write to both schemas during the transition and then switching the read layer.

**Where seen:** `langfuse`. `LANGFUSE_MIGRATION_V4_WRITE_MODE ∈ {legacy, dual, events_only}` and `LANGFUSE_MIGRATION_V4_NATIVE_OTEL_BEHAVIOUR ∈ {dual_write, direct}` at `worker/src/env.ts:459-468`; `validateV4Flags` throws on incoherent combinations at `:519-540`; read routing at `packages/shared/src/server/repositories/events.ts:1108-1140`.

**Why it works:** the same binary can be rolled out to a fleet, then progressively flipped from `legacy` → `dual` → `events_only` without downtime. Hard-validated flag combinations prevent operator error.

**When to copy:** when the read layer cannot be flipped atomically with the write layer, and when a "write to both for a while" window is acceptable. Particularly useful for backfill scenarios where the new write format needs to coexist with reads from the old format for weeks.

**When overkill / risky:** the read layer's gating is per-environment, not per-record — there is no field on individual records saying "I was written in mode X, route me accordingly". This means a deployment flipped to `events_only` mid-flight loses any records still buffered in the legacy path. The current validation only checks *flag combinations*, not whether the backfill background migrations have actually completed.

### Pattern F — Discriminated unions with literal `version` field on JSON blobs (langfuse, agent-framework)

**Problem solved:** compact versioning of a JSON blob family without an outer envelope or shared schema.

**Where seen:** `langfuse` — `version: z.literal(2)` discriminator with `LegacyEvalOutputDefinitionSchema` union (`packages/shared/src/features/evals/outputDefinition.ts:73,83,94,129-134`); `agent-framework` — `$type` discriminator with `unknown` fallback at `schemas/durable-agent-entity-state.json:121-134` and `DurableAgentStateUnknownContent.cs:25-33`.

**Why it works:** the resolver branches on the literal `version` field at parse time, normalizing both shapes to a single runtime shape. New versions can be added by extending the union without rewriting existing decoders.

**When to copy:** when a single feature owns a self-contained JSON blob family (eval output definitions, agent message content) and where a literal version discriminator is cleaner than an outer envelope.

**When overkill / risky:** doesn't scale across many features; each feature has to invent its own versioning scheme, which is exactly what langfuse does for every other JSON blob. The `z.literal(2)` typing means a feature flag to bump to `version: 3` requires updating the literal in many call sites.

### Pattern G — Discard-unknown + safe-type allowlist for wire-format compat (temporal, langgraph)

**Problem solved:** reading wire-format payloads from older or newer code versions without rejecting the entire message.

**Where seen:** `temporal` — `DiscardUnknown: true` in `DeserializeStrippedEvents` (`common/persistence/serialization/serializer.go:227-233`) and JSON slice decoder (`common/codec/jsonpb.go:128-132`); `langgraph` — `SAFE_MSGPACK_TYPES` allowlist at `libs/checkpoint/.../serde/_msgpack.py:21-85`, `safe types bypass the allowed_json_modules gate` at `libs/checkpoint/.../serde/jsonplus.py:60-70`.

**Why it works:** protobuf's `unknownFields` plus the explicit `DiscardUnknown` flag means unknown fields are preserved across an unmarshal/marshal round-trip rather than rejected. The safe-type allowlist gates code revival (msgpack) to a known set, preventing arbitrary class instantiation from corrupted checkpoints.

**When to copy:** any protobuf/msgpack/JSON pipeline where forward compat across schema additions is more important than strict validation. Particularly important for wire formats shared across many SDK versions.

**When overkill / risky:** silent. A field that disappears from the proto will be silently dropped on the wire, with no telemetry to surface the loss. The TODO at `common/persistence/serialization/serializer.go:565` (forced JSON for `QueueMetadata`) is a known migration debt that this pattern defers rather than solves.

### Pattern H — Two-formula S3 key prefix with regex match for both shapes (langfuse)

**Problem solved:** reading S3 keys produced by both an old code path (raw SDK id) and a new code path (`safeBlobKeySegment`-hashed) without rewriting old keys.

**Where seen:** `langfuse` — `STANDARD_EVENT_KEY_REGEX` and `OTEL_EVENT_KEY_REGEX` at `packages/shared/src/server/ingestion/eventBucketPath.ts:16-23`; `buildEventBucketPrefix` (sanitize fresh) vs `rawEventBucketPrefix` (preserve canonical) at `:71-117`.

**Why it works:** the regex matches both shapes, so a read can locate files written by either generation; the *write* helpers are split so that fresh writes don't double-sanitize.

**When to copy:** any time the on-disk identifier shape has changed but the new shape can't be retroactively applied to old data. Particularly useful for S3, blob storage, file paths, or anywhere the identifier is opaque to consumers.

**When overkill / risky:** a replay that re-sanitizes the canonical segment will point at a different file (`eventBucketPath.ts:84-117`), so callers must know to use the "raw" helper for replay-style reads.

### Pattern I — `LATEST_VERSION` constant baked into the binary (langgraph, temporal)

**Problem solved:** making "what version of schema does this binary speak?" answerable by static analysis.

**Where seen:** `langgraph` — `LATEST_VERSION = 2` at `libs/checkpoint/.../base/__init__.py:811` and `LATEST_VERSION = 4` at `libs/.../pregel/_checkpoint.py:21`; `temporal` — per-store `Version` constants at `schema/cassandra/version.go:6`, `schema/mysql/v8/version.go:6-9`, `schema/postgresql/v12/version.go:6-10`, `schema/sqlite/version.go:4-6`.

**Why it works:** the runtime can compare its expected version against the DB's `schema_version` table at startup, refusing to start if there's a downgrade. The constant is committed in code, so the deploy artifact pins the contract.

**When to copy:** any persistence layer where the on-disk shape has a known good version and where the binary should refuse to operate against a lower-version DB.

**When overkill / risky:** when there are multiple layers with different version cadences, having multiple `LATEST_VERSION` constants (langgraph's base=2 vs Pregel=4) is a known source of confusion; the `Checkpoint` docstring at `libs/checkpoint/.../base/__init__.py:95-96` still says "Currently `1`" and is stale.

### Pattern J — `JsonExtensionData` / unknown-field round-tripping for forward compat (agent-framework)

**Problem solved:** preserving unrecognized fields through serialize/deserialize cycles so new fields added by newer code are not lost when older code reads them.

**Where seen:** `agent-framework` — `[JsonExtensionData]` on every container (`DurableAgentStateData.cs:30-31`, `DurableAgentStateEntry.cs:42-43`, `DurableAgentStateMessage.cs:43-44`, `DurableAgentStateContent.cs:29-30`, `DurableAgentStateUsage.cs:39-40`); Python equivalent in `DurableAgentStateUsage.from_usage/to_usage_details` at `_durable_agent_state.py:1237-1263`.

**Why it works:** a separate `ExtensionData` dict captures keys that don't match any declared property and is re-emitted on serialize. Reads lose type information but preserve the bytes.

**When to copy:** any JSON-based persistence where additive evolution across versions is expected and where losing unknown fields would be operationally worse than losing type information on them.

**When overkill / risky:** `JsonExtensionData` only round-trips — it doesn't *act* on the data. A field round-tripped via extension data still needs a separate mechanism to be promoted to a typed property once both runtimes support it.

## Key Differences

### Anchoring layer

The most consequential difference across sources is *which layer the version-of-record lives on*:

| Source | DB schema | Blob envelope | Inline fields |
|--------|-----------|---------------|---------------|
| agent-framework | (not anchored) | **semver `schemaVersion` (required)** | `$type` discriminator + `unknown` + `[JsonExtensionData]` |
| langgraph | Postgres `MIGRATIONS` list (positional), SQLite none | **integer `v` on `Checkpoint`** | (relies on envelope) |
| temporal | **`schema_version` table + manifest `MinCompatibleVersion`** | `DataBlob.EncodingType` | protobuf3 wire-format + `DiscardUnknown` + hand-rolled nil-init |
| langfuse | **Prisma migrations + ClickHouse `*.up.sql` + `background_migrations` table** | mostly none (Eval output is the exception) | Zod `.optional()` + `.transform()` + `passthrough()` |
| letta | **Alembic linear chain + paired upgrade/downgrade** | `metadata.revision_id` *not inspected* | Pydantic `extra="forbid"` strict mode + ad-hoc legacy-field stripping |

Three observations follow:

- **Only agent-framework and langgraph treat the blob envelope as the source of truth.** They are also the two sources that document a hand-curated schema (JSON Schema 2020-12 / TypedDict) as the contract.
- **Only temporal requires `MinCompatibleVersion` in the migration itself.** That requirement is what makes the manifest tool effective at scale.
- **Only letta's `metadata.revision_id` is unwritten data.** Writing a version field that the importer doesn't check is the worst of both worlds — it appears compliant in audits but provides no runtime safety.

### Migration style — DB vs. blob

- **DB-driven** (langfuse, letta, temporal): schema is the canonical evolution surface; blob compatibility is layered on top.
- **Blob-driven** (agent-framework, langgraph): the persisted blob is the canonical surface; the DB layer is mostly an afterthought (or, in langgraph's SQLite case, nonexistent).

### Migration direction — forward-only vs. reversible

| Source | DB migrations | Blob migrations |
|--------|---------------|-----------------|
| agent-framework | n/a (no DB schema) | additive only (no explicit migration) |
| langgraph | Postgres paired up/down; SQLite none | in-place on read, no down-migration |
| temporal | forward-only (rollback tolerated, downgrade rejected) | additive via proto3 wire format |
| langfuse | forward-only (Prisma); paired `*.up.sql`/`*.down.sql` (ClickHouse, but never invoked by `entrypoint.sh`) | additive via Zod `.optional()` |
| letta | paired upgrade/downgrade | hard-rejection of legacy at HTTP boundary |

Only letta provides fully paired upgrade/downgrade Alembic migrations. Only langfuse ships paired `*.down.sql` files that are never executed in practice. agent-framework and langgraph rely on additive-only evolution; a breaking change in agent-framework bricks every existing durable entity (`DurableAgentStateJsonConverter.cs:38-41`).

### Idempotency

- **langfuse** — Prisma migrations have shipped in non-idempotent form, requiring manual `_prisma_migrations` cleanup (`packages/shared/scripts/cleanup.sql:1-7`); ClickHouse migrations are designed idempotent via "already exists / not found" tolerance in the upgrade tool.
- **temporal** — DDL statements are idempotent (`execStmts` swallows `already exists` / `not found` at `tools/common/schema/updatetask.go:142-147`); manifest MD5 detects post-hoc edits.
- **agent-framework** — JSON reads are not idempotent across major versions because the .NET converter rejects `Major != 1` outright.
- **letta** — MCP encryption migration is idempotent because the `SELECT-then-UPDATE` only touches rows where `_enc IS NULL` (`eff256d296cb_mcp_encrypted_data_migration.py:41`); other migrations are atomic.

### Old-state fixtures

- **langgraph** is the only source that *commits* an in-source corpus of old checkpoints as Python literals (`SAVED_CHECKPOINTS` at `libs/langgraph/tests/test_checkpoint_migration.py:272-1397`), exercised by parametrized `test_saved_checkpoint_state_graph(_async)` tests.
- **agent-framework** uses inline JSON literals in tests (`test_durable_entities.py:190-202`, `test_app.py:511-533`, `DurableAgentStateTests.cs:89-169`) but does not version them in a fixtures directory.
- **temporal** has no committed fixtures; tests construct inputs in-memory.
- **langfuse** has no committed historical Zod-payload fixtures; the closest is the ClickHouse version-band compatibility test (`packages/shared/src/server/clickhouse/compatibility.test.ts:17-118`).
- **letta** has `tests/data/memgpt-0.2.11` and `tests/data/memgpt-0.3.17` fixtures that are *not referenced by any modern test*; agent-file `.af` fixtures are written by the current code, not snapshots.

### Compatibility test coverage

The corpus has three observable patterns for testing compat:

- **Round-trip tests** that lock the on-disk format with explicit assertions (agent-framework's `DurableAgentStateTests.BasicState` at `DurableAgentStateTests.cs:89-169`; temporal's `TestSerializeWorkflowExecutionState` at `serializer_test.go:246-274`).
- **Migration-call tests** that invoke the migration function on a stored old-version snapshot (langgraph's `test_migrate_checkpoints` at `libs/langgraph/tests/test_checkpoint_migration.py:1473-1494`).
- **Compatibility-resolution tests** that pin the runtime behavior of compatibility helpers (temporal's `TestDeserializeStrippedEvents.ProtoDiscardUnknownFields` at `common/persistence/serialization/serializer_test.go:204-243`; langfuse's `parseClickHouseVersion` and `resolveClickHouseCompatibility` at `packages/shared/src/server/clickhouse/compatibility.test.ts:17-118`).

Only langgraph tests the *end-to-end* read+resume from a stored old checkpoint. The others test serialization correctness or helper behavior.

## Tradeoffs

### Forward-only DB migrations (temporal, langfuse)

- **Benefit:** simpler upgrade tool, no down-migration code to maintain, `MinCompatibleVersion` becomes the only compatibility contract.
- **Cost:** a botched upgrade that breaks the application cannot be undone by re-running the schema tool — operators must restore from backup.
- **Best-fit context:** systems where forward progress is more important than reversible upgrades, and where the upgrade tool is run by an operator with backup/restore discipline.
- **Failure mode:** mid-migration failure leaves the DB in an unknown state with no documented recovery path (temporal's `execStmts` handles `already exists` / `not found` but not arbitrary DDL timeouts).
- **Alternative seen:** letta uses paired upgrade/downgrade Alembic migrations, which provides rollback symmetry but doubles the migration maintenance burden and is "data-loss unsafe" for destructive migrations (`d05669b60ebe_migrate_agents_to_orm.py`).

### Semver string vs. integer version key (agent-framework vs. langgraph)

- **Semver string (agent-framework):** allows patch/minor/major distinction; reads can reject breaking majors via a `Major != 1` check; downstream code can branch on minor.
- **Integer (langgraph):** simpler to compare (just `v > N`); fits a linear "every version is a step forward" model.
- **Best-fit context:** semver when the breaking surface is large and the major-version policy matters (agent-framework is cross-language, breaking changes need explicit negotiation); integer when evolution is monotonic (langgraph checkpoints are append-mostly).
- **Failure mode:** semver requires discipline — agent-framework has two `LATEST_VERSION`-equivalent constants (`SCHEMA_VERSION = "1.1.0"` in both runtimes) and the Python implementation doesn't validate the major at all.

### In-place migration on read (langgraph) vs. background migration (langfuse)

- **In-place on read:** invisible to operators, no separate data move required, but the migration runs on every read of an old record.
- **Background migration:** runs once, leaves the DB in a normalized state, but requires a recoverable worker and a separate deployment.
- **Best-fit context:** in-place when the migration is small and contained (renaming a key, moving a field); background when the migration is large (backfilling millions of rows) or when reads need a normalized shape for performance.
- **Failure mode:** in-place migration has no idempotency marker per row, so re-running the checkpointer against the same DB after a downgrade could re-trigger migrations. Background migration has upgrade-stops (code paths that depend on the migration having finished) but no automatic pause-on-migration-incomplete (langfuse README admits this at `worker/src/backgroundMigrations/README.md:22-23`).

### Discriminated union + legacy branch (langfuse Eval output, agent-framework `$type`)

- **Benefit:** compact, type-safe at the parse boundary, the resolver normalizes both shapes to a single runtime shape.
- **Cost:** every new version requires updating the union; legacy branches must be retained indefinitely or the resolver must learn to ignore missing variants.
- **Best-fit context:** single feature owning a self-contained blob family with infrequent major changes.
- **Failure mode:** the resolver becomes a long if/else chain; the literal `z.literal(2)` typing means bumping to `version: 3` requires updating many call sites.

### `[JsonExtensionData]` round-tripping (agent-framework) vs. proto `unknownFields` (temporal)

- **JSON extension data:** preserves unknown fields at the property level; type information is lost.
- **Proto unknown fields:** preserves unknown fields at the wire-format level; type information is also lost, but the byte sequence is preserved for round-trip.
- **Both:** silently preserve data without acting on it; need a separate promotion mechanism.
- **Failure mode:** silent. A field that disappears from the schema is silently dropped on the wire, with no telemetry to surface the loss. Temporal's `DiscardUnknown` has no companion metric.

### Per-dialect migration early-exit (letta)

- **Benefit:** keeps the migration graph readable; SQLite gets a single baseline; developers don't see PG-only changes.
- **Cost:** dialect-specific bugs are masked on the dev DB and only surface in production PG.
- **Best-fit context:** when SQLite is intentionally dev-only and PG is the production target.
- **Failure mode:** a `letta 0.10.0` schema migration that assumes PG semantics will silently no-op on SQLite, then break in production.

## Decision Guide

Choosing between these patterns depends on five operational constraints. The decision tree below assumes the most common cases; if your situation spans multiple branches, prefer the more conservative choice (manifest-driven DB, envelope-versioned blob).

### 1. Is your primary persistence relational or blob-shaped?

- **Relational** (most users, datasets, queue metadata, transactional state) → tool-driven DB migrations (Pattern C, Pattern D) with envelope versioning as a secondary concern. Choose **temporal-style manifests** when you have multiple databases to evolve in lockstep; choose **langfuse-style Prisma + background migrations** when you have long-running data moves.
- **Blob-shaped** (agent conversations, workflow checkpoints, structured events) → envelope-versioned blob (Pattern A, Pattern B). Choose **agent-framework-style JSON Schema + `$type` polymorphism** when the runtime is cross-language and the contract must be externally consumable; choose **langgraph-style integer `v` + in-place migration** when the runtime is single-language and the migration logic is contained.

### 2. Will the schema change frequently, or rarely?

- **Frequently** (every release) → additive-only patterns (Pattern A, Pattern G) plus committed fixtures (langgraph `SAVED_CHECKPOINTS`). Resist the urge to bump major versions.
- **Rarely** (every few releases) → manifest-driven with `MinCompatibleVersion` (Pattern C). Each migration is a discrete event with explicit compatibility contract.

### 3. Is there a long-running data move required?

- **Yes** (e.g., moving from one storage backend to another, encryption-at-rest rollout, partition by new column) → recoverable background migration (Pattern D, langfuse). Persist `state` Json cursor between batches; lock-protect with TTL + heartbeat; add explicit "upgrade stops" documentation.
- **No** → schema-only migrations are sufficient; the tool only needs `MinCompatibleVersion`.

### 4. Do you need a dual-write window during cutover?

- **Yes** (read layer cannot be flipped atomically with the write layer) → phase-gated feature flags (Pattern E, langfuse). Validate flag combinations at boot. Document the cutover order in the manifest.
- **No** → schema migration plus a one-shot data migration is sufficient; no need for `legacy | dual | events_only` flags.

### 5. Do you have a cross-language boundary?

- **Yes** (e.g., .NET + Python, JS + Go) → the schema must be the contract, not the framework types. Use JSON Schema 2020-12 or a TypedDict-equivalent that both runtimes can validate against. Use `$type` polymorphism rather than language-specific class names. Reconcile any asymmetry in how strictly each runtime validates the version field.
- **No** → the schema can be a single-runtime concern; integer `v` key + in-place migration is fine.

## Practical Tips

### Patterns to copy

- **Commit old-state fixtures as executable regression tests.** langgraph's `SAVED_CHECKPOINTS` corpus (`libs/langgraph/tests/test_checkpoint_migration.py:272-1397`) is the gold standard in this study. Even inline JSON literals (agent-framework's `test_durable_entities.py:190-202`) are better than nothing.
- **Manifest `MinCompatibleVersion` is required, not optional.** temporal's `readManifest` rejects empty values at `tools/common/schema/updatetask.go:288-294`; this is the single most effective contract.
- **Recoverable background migrations need a `state` Json cursor.** langfuse's pattern (`worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:67-83`) survives worker crashes without losing progress.
- **`$type` polymorphism with an explicit `unknown` fallback.** agent-framework's `unknown` content type (`schemas/durable-agent-entity-state.json:111-119`; `DurableAgentStateUnknownContent.cs:25-33`) is the cleanest forward-compat envelope in the corpus.
- **Two-formula S3 key handling.** langfuse's `buildEventBucketPrefix` vs `rawEventBucketPrefix` (`packages/shared/src/server/ingestion/eventBucketPath.ts:71-117`) is a good model when identifier shape has changed but old data is in place.
- **`[JsonExtensionData]` / `unknownFields` round-tripping.** agent-framework's `[JsonExtensionData]` on every container, temporal's protobuf3 wire-format tolerance — both ensure new fields added by newer code are not lost when older code reads them.
- **Manifest MD5 in audit log.** temporal writes the MD5 of the applied manifest to `schema_update_history` (`tools/common/schema/updatetask.go:301-304`), detecting post-hoc edits.
- **DDL statement whitelist.** temporal's `validateCQLStmts` restricts migrations to `CREATE`/`ALTER`/`INSERT`/`DROP`/`DO` (`tools/common/schema/updatetask.go:251-265`), preventing accidental data corruption via migration script.
- **Hard-validate feature flag combinations at boot.** langfuse's `validateV4Flags` (`worker/src/env.ts:519-540`) throws on incoherent `legacy` + `direct` combinations. Don't wait for runtime misbehavior.
- **Stale-state detection via range comparison.** temporal's `StalenessCheck` (`common/persistence/transitionhistory/transition_history.go:81-125`) is more expressive than simple equality.

### Patterns to avoid or delay until needed

- **Informational versioning** (letta's `metadata.revision_id` never inspected at `agent_serialization_manager.py:993-1020`). Write the version or don't — an unwritten version is worse than no version.
- **Hard cut-over at the HTTP boundary with no adapter** (letta's HTTP 400 on legacy format at `agents.py:580-583`). If you must hard-cut, ship an adapter layer for at least one release.
- **Print-and-continue version checks** (letta's `marshmallow_agent.py:223-230`). Either raise or migrate — printing is the worst of both.
- **Silent `DiscardUnknown` without telemetry.** temporal drops unknown JSON fields without surfacing the loss (`common/codec/jsonpb.go:128-132`). Add a counter partitioned by message type so operators can see schema drift.
- **Asymmetric enforcement between runtimes** (agent-framework .NET strict vs. Python permissive at `_durable_agent_state.py:419-434`). Pick one strategy and apply it in both.
- **Hand-coded `if-nil-populate` patches scattered through the serializer** (temporal's `serializerImpl` at `common/persistence/serialization/serializer.go:386-470`). Promote to a central migration registry before the third one lands.
- **Multiple `LATEST_VERSION` constants** (langgraph's base=2 vs Pregel=4). Promote to a single source of truth.
- **Migration duplication across layers** (langgraph's `pending_sends` migration in both `PostgresSaver.get_tuple` and `Pregel._migrate_checkpoint`). Centralize in one layer.
- **`dev-tables.sh` bypass of the migration system** (langfuse `packages/shared/clickhouse/scripts/dev-tables.sh:15-26`). Either ship as a real migration or document the bypass prominently.
- **Hard-coded background-migration UUIDs** (langfuse `worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:12`). Use a stable symbolic name so renaming a script can't desync the SQL `INSERT`.

### Decision rules for the main approaches

| Question | Prefer | Avoid |
|----------|--------|-------|
| Cross-language persistence? | agent-framework JSON Schema + `$type` | langgraph single-runtime integer `v` (no cross-runtime contract) |
| Single database, frequent additive changes? | langfuse Prisma + ClickHouse + background migrations | letta linear Alembic with no JSON contract |
| Multiple databases, infrequent breaking changes? | temporal manifest-driven with `MinCompatibleVersion` | langfuse forward-only Prisma (no `MinCompat`) |
| Long-running backfill? | langfuse background migration with `state` Json cursor | on-read migration (re-runs on every access) |
| Need dual-write window? | langfuse phase-gated feature flags | at-the-API hard cut-over |
| Portable file format? | Envelope with `version` field + per-version parser (langgraph) | letta informational `revision_id` not consulted |

## Anti-Patterns / Caution Signs

These are concrete signals that the schema-versioning area is becoming brittle, over-coupled, under-specified, hard to test, hard to operate, or hard to evolve:

- **Hard-rejection of legacy format at the HTTP boundary without an adapter.** letta's `agents.py:580-583` returns HTTP 400 for legacy `AgentSchema` JSON; the only path forward is "wrap your old file in the new shape manually". Any deployment with old user files is a guaranteed support ticket.
- **Print-and-continue version checks.** letta's `marshmallow_agent.py:223-230` prints `Version mismatch: expected {v1}, got {v2}` and proceeds. Looks compliant in code review but provides zero runtime safety.
- **Informational version field that the importer doesn't inspect.** letta's `metadata.revision_id` is set at export (`agent_serialization_manager.py:488`) but never read. Same root cause as above.
- **Hand-rolled `if-nil-init` patches in the serializer.** temporal's `serializerImpl` (`common/persistence/serialization/serializer.go:386-470`) has at least four such patches (`ShardInfoFromBlob`, `WorkflowExecutionInfoFromBlob`, `WorkflowExecutionStateFromBlob`, `ChecksumFromBlob`). Each new optional collection adds a new branch; the next contributor may forget it.
- **Asymmetric enforcement between language runtimes.** agent-framework's .NET converter rejects `Major != 1` (`DurableAgentStateJsonConverter.cs:38-41`) while Python's `from_dict` accepts any value including missing version (`_durable_agent_state.py:419-434`). A bug surfaces asymmetrically depending on which runtime reads the blob.
- **Silent `DiscardUnknown` drops.** temporal's JSON decoder at `common/codec/jsonpb.go:128-132` silently drops unknown fields. Operators upgrading have no way to know they lost data.
- **Multiple `LATEST_VERSION` constants without documentation.** langgraph has `base=2` and `Pregel=4` with no explanation of why they differ. The docstring at `libs/checkpoint/.../base/__init__.py:95-96` still says "Currently `1`" — stale.
- **No migration runner for a persistent backend.** langgraph's SQLite saver has no `MIGRATIONS` list (`libs/checkpoint-sqlite/.../sqlite/__init__.py:129-166`). A future DDL change needs a one-shot script.
- **Migration duplication across layers.** langgraph's `pending_sends` migration is implemented in both `PostgresSaver.list/get_tuple` (`libs/checkpoint-postgres/.../__init__.py:164-188,246-259`) and `Pregel._migrate_checkpoint` (`libs/.../pregel/main.py:1135-1142`). A bug fix in one will be missed by the other.
- **Hard-coded background-migration UUIDs in source code.** langfuse's `migrateTracesFromPostgresToClickhouse.ts:12` references `"5960f22a-748f-480c-b2f3-bc4f9d5d84bc"` — if the SQL `INSERT` and the script's `const` drift, `findUniqueOrThrow` throws at runtime.
- **Per-dialect early-exit repeated 150+ times.** letta's `if not settings.letta_pg_uri_no_default: return` pattern is repeated in nearly every migration, with no central registry of which migrations are PG-only.
- **Migrations that ship with hand-written cleanup scripts.** langfuse's `packages/shared/scripts/cleanup.sql:1-7` deletes rows in `_prisma_migrations` for migrations that were applied in non-idempotent form. Every new such migration adds a new cleanup row.
- **DDL drops in migrations after the schema stabilizes.** letta's early migrations drop columns and tables (`20230518193521_changes/migration.sql:12`, `20230523082455_rename_metrics_to_gradings/migration.sql:14`). Acceptable during bootstrap, destructive thereafter.
- **`dev-tables.sh` bypass of the migration system.** langfuse's `events_full`/`events_core` are managed through `packages/shared/clickhouse/scripts/dev-tables.sh:15-26` because they depend on ClickHouse ≥24.5/25.x features. Operators who forget to run this script get a half-deployed state.
- **JSON validator not invoked.** agent-framework declares the JSON Schema contract (`schemas/durable-agent-entity-state.json`) but no JSON-schema validator import was found in either build script. The contract exists but isn't enforced.
- **Cross-version cross-runtime test missing.** agent-framework has a Python test for `1.0.0` fixtures but no test that round-trips `1.0.0` through both `DurableAgentStateJsonConverter` (NET) and `DurableAgentState.from_dict` (Python). The asymmetry at `_durable_agent_state.py:419-434` is invisible to the test suite.
- **Version not propagated into the JSON blob.** langgraph's `WorkflowCheckpoint.version` is a Python default value (`_checkpoint.py:88`) that is never written to the JSON blob — only the `graph_signature_hash` and the dataclass `version` field carry the version. A consumer reading raw JSON sees no version.
- **No "compatibility matrix" file.** temporal's contract is implicit in `MinCompatibleVersion` and `VerifyCompatibleVersion`; no `compat.md` exists. letta has no documented compatibility matrix either.
- **Wire-format ext codes are positional integers.** langgraph's `_msgpack.py` reserves ext codes 0-7 with no central registry. A new ext type without coordination will collide.

## Notable Absences

Several desirable primitives are *absent* from the corpus. They are worth listing because their absence is a signal that the field has not yet converged.

- **No committed fixtures directory of versioned payloads.** Only langgraph has anything close, and only as in-source Python literals. There is no `tests/fixtures/state-v1.0.0.json` repository in agent-framework, temporal, or letta. langfuse's only ClickHouse compatibility test uses parsed versions, not committed payloads.
- **No machine-readable `schema_version` history catalog.** agent-framework's schema history is implicit in `CHANGELOG.md:37,50`; letta's is implicit in Alembic revisions; langfuse's is implicit in Prisma migration filenames; temporal's is implicit in manifest directories. None of them publish a registry mapping version → change → migration recipe.
- **No automatic downgrade path.** Temporal's `VerifyCompatibleVersion` rejects downgrade (`common/persistence/schema/version.go:21-28`); langfuse's Prisma migrations are forward-only by design; ClickHouse `*.down.sql` files exist but `web/entrypoint.sh` never invokes `down`; langgraph's in-place migration has no downgrade mechanism.
- **No telemetry for `DiscardUnknown` drops.** temporal silently drops unknown JSON fields; agent-framework's `[JsonExtensionData]` silently preserves unknown fields. Neither surfaces a counter or log line.
- **No documented policy on renaming a proto field.** temporal has no `rename`, `deprecate field`, or `reserved` documentation in the persistence proto files.
- **No "migration registry" / upcaster framework.** temporal's `if-nil-init` patches are scattered through `serializerImpl`; there's no central list of (source-shape, target-shape) mappings. Same for agent-framework's compatibility shims.
- **No cross-backend compatibility tests.** temporal stores checkpoints in JSONB on Postgres and BLOB on SQLite; langgraph stores them in `JSONB` (Postgres) and `BLOB` (SQLite). Cross-backend copy is impossible without re-encoding through the serde, and no test covers this.
- **No official "old format → new format" migration shims for portable files.** letta hard-rejects legacy `AgentSchema` JSON at the HTTP boundary; there is no in-process adapter.
- **No CI gating for non-additive migrations.** letta's early migrations drop columns; langfuse's bootstrap migrations drop tables. None of the sources have a CI check that prevents destructive DDL past a certain project age.

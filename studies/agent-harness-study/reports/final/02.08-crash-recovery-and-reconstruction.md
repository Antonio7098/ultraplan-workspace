## Executive Summary

Across the five studied systems, crash recovery converges on three architectural archetypes that map to product shape rather than quality:

1. **Deterministic replay engines** (temporal, langgraph) treat every superstep as a unit of durability. They persist a checkpoint plus a parallel table of pending writes, and resume by reading the last durable point and re-executing only the steps that have no recorded writes. This is the only model that gives strong "resume where you left off" semantics across hard crashes.

2. **Library-layer checkpoint frameworks** (agent-framework) expose a pluggable `CheckpointStorage` protocol and per-executor `on_checkpoint_save` / `on_checkpoint_restore` hooks, letting users pick between in-process checkpoints and a Durable Task host. Recovery is opt-in and operator-initiated for the in-process path; the durable path is host-managed.

3. **Application-level run managers** (letta, openhands) persist a row per run plus streamed chunks, and rely on the operator to call `cancel_run` or `DELETE /conversations/{id}` on stuck rows. There is no periodic sweeper, no DLQ, no automatic requeue of failed runs. Crash recovery is mostly a UI/operator concern, not an engine concern.

Ratings cluster accordingly: temporal (9/10) and the two library/framework sources (8/10 each) have clear models with explicit interfaces; letta (6/10) and openhands (5/10) have partial models that rely on operator action.

The single biggest divergence is **orphaned-run detection**: temporal fences ownership via range-id, langgraph treats orphans as just-non-terminal checkpoints, agent-framework refuses fresh runs unless checkpointing is on, while letta and openhands leave `running` rows in the DB indefinitely. The second biggest is **dead-letter handling**: temporal ships a first-class DLQ with operator workflows; langgraph materializes unrecoverable failures as `(ERROR, exc)` writes; the other three have no DLQ.

## Core Thesis

> A system survives a crash not because it has a backup, but because it has a deterministic point of resumption. The five sources split sharply on whether that point is implicit (temporal's last acked task, langgraph's last durable checkpoint + pending writes), explicit-but-opt-in (agent-framework's `WorkflowCheckpoint`), or simply absent (letta/openhands leave "what was I doing?" to the next caller). Mature recovery is a function of three primitives: durable per-step progress, idempotent re-execution of the step boundary, and a deterministic next-call contract — not of periodic sweepers or alerting.

## Approach Models

Each source falls into one of three archetypes. The vocabulary below reflects the dimension's question "what does the system do after a crash?" rather than general architecture.

### A. Deterministic Replay Engines (temporal, langgraph)

- **Unit of durability**: one task/superstep. Each unit is persisted as a checkpoint plus a per-task write record.
- **Resumption contract**: on the next invocation with the same identity key (`run_id` / `thread_id` / `shard_id`), the engine reads the last durable checkpoint, re-attaches pending writes, and skips tasks that already committed. Un-acked tasks are re-executed.
- **Identity**: temporal uses `shard_id + range_id` for ownership fencing; langgraph uses `(thread_id, checkpoint_ns, checkpoint_id)` for storage scoping.
- **Failure destination**: temporal has a DLQ with operator workflow; langgraph materializes unrecoverable failures as `(ERROR, exc)` writes in the same writes table, surfaced through `get_state`.

### B. Library-Layer Checkpoint Frameworks (agent-framework)

- **Unit of durability**: end-of-superstep. The runner captures `WorkflowCheckpoint` (state + messages + pending request_info events + iteration_count + version) after each superstep.
- **Resumption contract**: explicit. `Workflow.run(checkpoint_id=..., checkpoint_storage=...)` calls `Runner.restore_from_checkpoint` which validates `graph_signature_hash`, clears stale state, replays executor state via `on_checkpoint_restore`, then resumes.
- **Storage pluggability**: `CheckpointStorage` Protocol with in-memory, file, and Cosmos implementations; a separate Durable Task path delegates durability to the Azure host via orchestration replay.
- **Failure destination**: none. `WorkflowCheckpointException` raises to caller; no DLQ, no operator alert.

### C. Application-Level Run Managers (letta, openhands)

- **Unit of durability**: one HTTP request / one run row. Each long-running operation persists a row (`runs`, `app_conversation_start_task`) plus per-step message checkpoints.
- **Resumption contract**: implicit. Next caller's request continues from current state; there is no deterministic "resume from X" call.
- **Stream durability**: Redis SSE buffer (letta) / runtime-as-truth for sandboxes (openhands) decouples client lifetime from processing.
- **Failure destination**: rows transition to `failed` on exception; no DLQ; orphan detection is operator-driven (call `cancel_run` / `DELETE /conversations`).

## Pattern Catalog

The patterns below appear in multiple sources or are demonstrated unusually clearly by one.

### 1. Persist + Enqueue in Single Transaction

**Problem**: how do you guarantee that a unit of work and its trigger both survive a crash, or neither does?

**Sources**: temporal (`service/history/queues/queue_base.go:104-242` — task queue + ack-level checkpoint), langgraph (`libs/langgraph/langgraph/pregel/_loop.py:1182-1189` — checkpoint after each superstep), letta (`letta/services/run_manager.py:48-90` — run row + run_metrics in one transaction).

**Why it works**: a single ACID transaction is the only durable "this step happened" record. The reader can reconstruct exactly where to resume by reading the transaction log.

**When to copy**: any time the engine can drive both the state write and the wakeup trigger (timer, next task, signal).

**When it is overkill**: a stateless HTTP handler or a fire-and-forget notification where the cost of replication outweighs the benefit of durability.

**Evidence**: `common/persistence/sql/shard.go:82-113` (transactional outbox), `libs/langgraph/langgraph/pregel/_loop.py:1507-1524` (order-preserving checkpoint flush), `letta/services/run_manager.py:48-90`.

### 2. Pending Writes as First-Class

**Problem**: mid-superstep, some tasks have committed but the checkpoint hasn't fired. How do you represent partial success?

**Sources**: langgraph (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:795` — `WRITES_IDX_MAP[ERROR] = -1` sentinel, writes table parallel to checkpoints), temporal (ack-level checkpoint + `exclusiveReaderHighWatermark`, `service/history/queues/queue_base.go:118-133`), agent-framework (`WorkflowCheckpoint.pending_request_info_events`, `python/packages/core/agent_framework/_workflows/_checkpoint.py:30-88`).

**Why it works**: on resume, completed tasks are reattached and skipped; failed tasks retry; the checkpoint serves as a stable anchor for the writes table.

**When to copy**: any engine where individual tasks can finish independently of the superstep boundary.

**When it is overkill**: single-threaded runs where each step is atomic.

**Evidence**: `libs/langgraph/langgraph/pregel/_loop.py:724-737` (`_reapply_writes_to_succeeded_nodes`), `service/history/queues/executable.go:656-678` (Ack/Nack gating), `python/packages/core/agent_framework/_workflows/_runner_context.py:414-426` (InProcRunnerContext.apply_checkpoint).

### 3. Range-ID / Version Fencing

**Problem**: how do you know whether a stale host is still allowed to write?

**Sources**: temporal (range-id monotonic counter, `common/persistence/sql/shard.go:115-149`; returns `ShardOwnershipLostError` on mismatch), agent-framework (graph-fingerprint validation, `python/packages/core/agent_framework/_workflows/_runner.py:275-279`; refuses resume on topology change), langgraph (storage-layer `INSERT OR IGNORE` / `ON CONFLICT DO NOTHING` for idempotency, `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:131-159`).

**Why it works**: a fencing token (range-id, graph hash, PK conflict) converts "am I still the owner?" from a soft guess into a hard assertion.

**When to copy**: any system where multiple hosts may attempt writes for the same identity.

**When it is overkill**: single-process systems where there is only one writer.

**Evidence**: `common/persistence/sql/shard.go:115-149`, `python/packages/core/agent_framework/_workflows/_runner.py:275-279`, `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:131-159`.

### 4. Pluggable Storage Protocol

**Problem**: how do you let users pick their durability tier without re-implementing the engine?

**Sources**: agent-framework (`CheckpointStorage` Protocol, `python/packages/core/agent_framework/_workflows/_checkpoint.py:119-189`), langgraph (`BaseCheckpointSaver` with conformance suite, `libs/checkpoint/langgraph/checkpoint/base/__init__.py:176-580`, `libs/checkpoint-conformance/langgraph/checkpoint/conformance/validate.py:45-128`).

**Why it works**: the conformance suite enforces recovery semantics across all implementations; the protocol lets storage evolve independently from engine.

**When to copy**: any library that ships with multiple storage backends.

**When it is overkill**: an application with one well-known backing store.

**Evidence**: `python/packages/core/agent_framework/_workflows/_checkpoint.py:119-189`, `libs/checkpoint-conformance/langgraph/checkpoint/conformance/validate.py:45-128`.

### 5. Runtime as Source of Truth

**Problem**: DB rows can lie about whether the underlying runtime (container, process) actually exists.

**Sources**: openhands (`_get_user_running_sandboxes` intersects DB with `/list` runtime endpoint, `openhands/app_server/sandbox/remote_sandbox_service.py:375-396`; Docker `containers.list(all=True)` enumeration, `openhands/app_server/sandbox/docker_sandbox_service.py:291`), temporal (Cassandra task TTL via column expiry, `common/persistence/cassandra/matching_task_store.go:41-55`).

**Why it works**: when there are two systems of record (DB and runtime), the runtime wins because it is what the user is actually paying for.

**When to copy**: any system where ephemeral compute (containers, processes) is the unit of work.

**When it is risky**: when the runtime is expensive to query, or when the DB is the user's primary audit surface.

**Evidence**: `openhands/app_server/sandbox/remote_sandbox_service.py:375-396`, `common/persistence/cassandra/matching_task_store.go:41-55`.

### 6. Status Synthesis on Abnormal Termination

**Problem**: a stream/handler that ends without a clear terminal signal leaves the operator wondering what happened.

**Sources**: letta (synthesizes `stream_incomplete` + `[DONE]` when async generator exits without completion, `letta/services/streaming_service.py:610-629`, `letta/server/rest_api/redis_stream_manager.py:284-315`), langgraph (`GraphDrained` exception carries `drain_reason`, `libs/langgraph/langgraph/errors.py:54-64`; `RunControl.request_drain()` flips a flag, `libs/langgraph/langgraph/runtime.py:79-104`), temporal (`executable.Execute` defer recovers from panic as a task failure, `service/history/queues/executable.go:302-359`).

**Why it works**: a missing `[DONE]` is just as informative as an explicit one, if the code synthesizes a clear terminal event.

**When to copy**: any streaming or async path where the consumer is waiting for an end-of-stream sentinel.

**When it is overkill**: short request/response with no streaming.

**Evidence**: `letta/services/streaming_service.py:610-629`, `libs/langgraph/langgraph/runtime.py:79-104`, `service/history/queues/executable.go:302-359`.

### 7. Cooperative Drain vs. Forced Kill

**Problem**: how do you tell a long-running task to stop without losing the work it has done?

**Sources**: langgraph (`RunControl.request_drain()` flips a flag, loop checks it at superstep boundary, exits with `GraphDrained`, `libs/langgraph/langgraph/pregel/_loop.py:650-652`, `libs/langgraph/langgraph/runtime.py:79-104`), letta (cancellation as DB-flag polling + `asyncio.Event` propagation, `letta/agents/letta_agent_v2.py:750-757`, `letta/server/rest_api/streaming_response.py:33-194`).

**Why it works**: a flag-based drain lets the task reach its next safe checkpoint; a SIGKILL loses everything since the last flush.

**When to copy**: any task that holds in-flight state and where restart cost is non-trivial.

**When it is risky**: when the task has a long stretch without checkpoints, the drain timeout must bound the worst case (langgraph relies on external watcher SIGKILL).

**Evidence**: `libs/langgraph/langgraph/pregel/_loop.py:650-652`, `letta/agents/letta_agent_v2.py:750-757`.

### 8. Leader-Election for Background Work

**Problem**: how do you avoid N pods all polling the same job queue?

**Sources**: letta (PostgreSQL advisory lock `0x12345678ABCDEF00`, `letta/jobs/scheduler.py:18-99`), temporal (membership-based shard ownership via range-id, `service/history/shard/controller_impl.go:383-521`).

**Why it works**: a single leader is responsible for the work; non-leaders wait or take over shards owned by dead hosts.

**When to copy**: any background scheduler that must run exactly once across a cluster.

**When it is overkill**: stateless workers, or single-pod deployments.

**Evidence**: `letta/jobs/scheduler.py:18-99`, `service/history/shard/controller_impl.go:383-521`.

### 9. Per-Step Message Checkpoint

**Problem**: how do you keep the agent's conversation state durable without waiting for the run to finish?

**Sources**: letta (`_checkpoint_messages` inside a single SQLAlchemy session, `letta/agents/letta_agent_v3.py:758-816`), agent-framework (`AgentExecutor.on_checkpoint_save`/`on_checkpoint_restore` captures cache + session + full_conversation, `python/packages/core/agent_framework/_workflows/_agent_executor.py:320-367`).

**Why it works**: after every successful step the message table is durable; on crash, the next caller sees all messages up to the last checkpointed step.

**When to copy**: any agent loop where the conversation is the primary artifact.

**When it is overkill**: stateless transformations.

**Evidence**: `letta/agents/letta_agent_v3.py:758-816`, `python/packages/core/agent_framework/_workflows/_agent_executor.py:320-367`.

### 10. Schema Auto-Migration on Boot

**Problem**: how do you keep the DB schema in sync with the binary without manual ops?

**Sources**: openhands (`OssAppLifespanService.__aenter__` runs `alembic upgrade head`, `openhands/app_server/app_lifespan/oss_app_lifespan_service.py:15-37`).

**Why it works**: each boot brings a partially-migrated DB to head automatically.

**When to copy**: any app where schema drift is a common deployment failure mode.

**When it is risky**: when migrations are non-additive, boot can break if alembic crashes mid-way (openhands accepts this and crash-loops).

**Evidence**: `openhands/app_server/app_lifespan/oss_app_lifespan_service.py:15-37`.

## Key Differences

### Why Temporal vs. LangGraph Differ

Both use the "checkpoint + pending_writes" pattern, but they differ in what the *identity* of a durable unit is:

- **Temporal** keys durability by `shard_id + range_id`, which is a *host ownership* boundary. Recovery is host-scoped: when a host dies, the shard moves to a new owner with a fresh range-id, and any in-flight writes from the old owner are rejected. This is necessary because Temporal is a multi-host service.

- **LangGraph** keys durability by `(thread_id, checkpoint_ns, checkpoint_id)`, which is a *user conversation* boundary. Recovery is call-scoped: any caller with the same `thread_id` can resume. This fits LangGraph's role as an embedded library where the embedding process is the unit of execution.

### Why agent-framework Differs from Both

Agent-framework exposes both layers but does not pick one. The in-process `WorkflowCheckpoint` is library-style (per-executor hooks, pluggable storage) while the Durable Task path is service-style (host-managed orchestration replay). The dual model means users opt into durability; the framework refuses to silently replay (`python/packages/core/agent_framework/_workflows/_workflow.py:803-810`).

### Why Letta and OpenHands Diverge from the Others

Both are FastAPI applications, not engines. Their "durability" is a database row per run plus streamed chunks. They have:

- No history replay. There is no deterministic "what step am I on?" record to consult on resume.
- No DLQ. Failures are marked `failed`; the next caller starts a fresh run.
- No automatic orphan sweeper. Operator must cancel stuck runs.

This is appropriate for their product shape: a single API call may span minutes, but the unit of work is the call, not a multi-day workflow.

### Why the DLQ Pattern is Rare

Only temporal ships a first-class DLQ (`service/history/queues/dlq_writer.go:64-143`, `service/worker/dlq/workflow.go:203-323`). The others handle unrecoverable failures differently:

- **LangGraph**: materializes as `(ERROR, exc)` writes in the writes table; surfaced via `get_state_history`.
- **Agent-framework**: raises `WorkflowCheckpointException` to the caller; no operator alert.
- **Letta**: writes `stop_reason=error` and a final `[DONE]`; operator inspects via `GET /v1/runs/active`.
- **OpenHands**: writes to `app_conversation_start_task.detail`; user polls `GET /start-tasks/{id}`.

The DLQ is a workflow-engine concern; for application-level run managers, "the run failed" is the terminal state.

### Why the Orphan-Detection Story Diverges

- **Temporal**: range-id fencing makes "ownership lost" detectable at the persistence layer (`common/persistence/sql/shard.go:132-149`); the matching side relies on `liveness` TTL unload (`service/matching/liveness.go:23-49`).
- **LangGraph**: orphan = "the last persisted checkpoint is not terminal." No detector; the next caller resumes.
- **Agent-framework**: orphan = "Workflow instance has in-flight messages." Detected at next `run()` call (`_workflow.py:803-810`).
- **Letta**: orphan = `runs WHERE status='running' AND updated_at < now() - threshold`. No detector; `GET /v1/runs/active` lists them.
- **OpenHands**: orphan = `app_conversation_start_task WHERE status='WORKING' AND app_conversation_id IS NULL`. No detector; `GET /start-tasks/search` lists them.

The presence of an orphan detector correlates with the presence of a deterministic engine.

## Tradeoffs

| Tradeoff | Benefit | Cost | Best-Fit Context | Failure Mode | Alternative |
|----------|---------|------|-------------------|--------------|-------------|
| **Durability at every superstep** (temporal, langgraph) | Resume from exact last state | High write amplification | Multi-day workflows; financial transactions | Write throughput bottleneck | Per-N-steps checkpoint (letta) |
| **Opt-in checkpointing** (agent-framework) | No surprise replay; user controls durability | Users who skip it lose crashes | Library for varied deployment contexts | Silent failure when checkpointing off | Default-on + escape hatch (temporal) |
| **Run-row durability** (letta, openhands) | Simple mental model; DB is the source of truth | No replay of partial work | API-call-shaped work | Stuck `running` rows forever | History replay (langgraph) |
| **Range-id fencing** (temporal) | Strong ownership guarantees | Stale-host writes must be rejected | Multi-host services | `ShardOwnershipLostError` on caller; needs retry | CAS-based optimistic concurrency (langgraph) |
| **Graph-fingerprint check on resume** (agent-framework) | Refuses resume when topology changed | Operator must reconcile manually | Library with evolving graphs | No automatic fix path | Versioned migration (letta alembic) |
| **Pending_writes parallel to checkpoint** (langgraph) | Partial-superstep recovery is explicit | Two-table write on every step | Engines with parallel branches | Storage must enforce idempotent PK | Single-checkpoint model (letta) |
| **DLQ as last-resort** (temporal) | Operator has visibility into terminal failures | DLQ-disabled-by-default silently drops | Engines with retry policies | Operator must opt-in | Inline-error synthesis (letta) |
| **Leader-elected poller** (letta) | Single polling pod across cluster | DB lock dependency | Background batch polling | Falls back to "everyone polls" on SQLite | Each-pod-polls with deduplication |
| **Membership-driven shard ownership** (temporal) | Survives host failure transparently | Slow startup (`initialShardsAcquired` gate) | Multi-host services | Many-shard slow-DB delays serving | Single-host with manual takeover |
| **Runtime-as-truth for sandboxes** (openhands) | Stale DB rows cannot lie about running containers | DB side-effects (like FK) become inconsistent | Ephemeral compute | DB says `RUNNING`, runtime says `MISSING`, no fix | DB-as-truth with periodic reconcile |

## Decision Guide

### Choose deterministic replay (temporal / langgraph style) when:

- Workflows span minutes to days and span process boundaries.
- Failure of one step must not lose the work of prior steps.
- You can define a deterministic "step" boundary (timer, task, signal).
- You have the operational capacity to maintain a checkpoint store (Postgres / MySQL / Cassandra).

### Choose library-layer checkpoints (agent-framework style) when:

- You ship a library and your users have different durability needs.
- Users may deploy to a host with its own durability model (Azure Durable Task).
- You can document the explicit "you must enable checkpointing" contract.
- Your executors can be expressed as `on_checkpoint_save` / `on_checkpoint_restore` hooks.

### Choose application-level run managers (letta / openhands style) when:

- The unit of work is one HTTP call, not a multi-step workflow.
- Resumability is "next call continues from current state", not "resume step 47".
- Operator action (cancel, delete) is an acceptable recovery path.
- Your infra budget does not support a checkpoint store.

### Decision rule for orphan detection:

- **If you have a deterministic engine**: fencing (range-id / graph-hash) makes orphan detection implicit.
- **If you have a run row**: add a scheduled reaper. Without it, you are accumulating debt.
- **If you have neither**: the next caller's behavior is your recovery. Document it.

### Decision rule for dead-letter handling:

- **If retries are bounded by policy**: terminal failures need a destination. Add a DLQ or an inline error.
- **If retries are unbounded**: at minimum log the failure and increment a metric; operators need a signal.
- **If failures are user-actionable**: surface them in the user-visible status; do not bury them in logs.

### Decision rule for checkpoint frequency:

- **Sub-second durability**: per-step checkpoint (temporal default).
- **Multi-second durability**: per-superstep checkpoint (langgraph default).
- **Minute-scale durability**: per-N-steps checkpoint (letta per-step message checkpoint).
- **End-of-run only**: acceptable only when run duration is bounded and the engine can replay from start.

## Practical Tips

### Patterns to Copy

1. **Single-transaction persist + enqueue** (temporal `queue_base.go:104-242`). Use the same transaction for the state mutation and the wakeup trigger. The reader will reconstruct exactly where to resume.

2. **Pending_writes parallel to checkpoint** (langgraph `libs/checkpoint/langgraph/checkpoint/base/__init__.py:795`). Use sentinel negative `idx` values for control writes so they never collide with regular writes.

3. **Cooperative drain with superstep check** (langgraph `libs/langgraph/langgraph/runtime.py:79-104`). A flag-based drain lets tasks reach their next safe checkpoint; a SIGKILL loses everything since the last flush.

4. **Idempotent storage primitives** (langgraph Postgres `ON CONFLICT DO NOTHING`, SQLite `INSERT OR IGNORE`). Replay safety starts at the storage layer.

5. **Graph-fingerprint validation on resume** (agent-framework `python/packages/core/agent_framework/_workflows/_runner.py:275-279`). Refuse to silently replay against a changed topology.

6. **Belt-and-suspenders terminal sentinel** (letta `letta/server/rest_api/redis_stream_manager.py:447-460`). Always write `[DONE]` even if a prior chunk set `complete`; downstream SDKs depend on it.

7. **Per-step message checkpoint** (letta `letta/agents/letta_agent_v3.py:758-816`). After every successful step, persist messages in the same session.

8. **Status synthesis on abnormal termination** (letta `letta/services/streaming_service.py:610-629`). When the upstream generator exits without a terminal signal, synthesize one and mark the run failed.

9. **Idempotent terminal state guard** (letta `letta/services/run_manager.py:342-356`). Refuse to overwrite a terminal run; force `completed_at` on first terminal transition.

10. **Atomic file writes for checkpoint storage** (agent-framework `python/packages/core/agent_framework/_workflows/_checkpoint.py:317-323`). Write to `.tmp`, then `os.replace`. Avoids torn writes on crash.

### Patterns to Avoid or Delay Until Needed

1. **DLQ disabled by default** (temporal `service/history/queues/executable.go:583-585`). Terminal failures silently drop. Either enable by default or document the migration path.
2. **`durability="exit"`** (langgraph `libs/langgraph/langgraph/pregel/_loop.py:1294-1311`). Nothing durable until exit. Operators must understand this trades durability for throughput.
3. **In-memory checkpointer for production** (langgraph `InMemorySaver`). Process restart = total loss. Acceptable only for debugging.
4. **Best-effort pending-message delivery** (openhands `live_status_app_conversation_service.py:1977-2006`). POST then delete on failure silently loses messages on 5xx.
5. **No idempotency check on conversation create** (openhands `start_app_conversation`). Double-click creates two agents.
6. **Cancel writes denials for ALL pending tool calls** (letta `letta/services/run_manager.py:671-781`). Necessary to prevent "stuck" agents, but means cancel is destructive.
7. **Module-level polling task** (openhands `remote_sandbox_service.py:54`). On app-server reload, the next injector recreates it. No `done_callback` to log a failure.

### Caution Signs

The studied design area is becoming brittle when:

1. **No periodic sweeper for `running` rows** — letta and openhands both have this gap. Operators will accumulate stuck runs.
2. **No DLQ or terminal-error destination** — agent-framework raises exceptions but does not record them anywhere persistent. The runner has no `WorkflowEvent` for recovery failures.
3. **No idempotency key on external effects** — agent-framework tool calls, chat client invocations between checkpoints. Application-level idempotency is the developer's responsibility, which usually means it is not implemented.
4. **No graph-fingerprint on resume** — a topology change can cause silent replay against the wrong code path. agent-framework has this guard; letta and openhands do not because they do not have graph topology.
5. **Streaming protocol that depends on `__exit__` running** — `durability="exit"` in langgraph. SIGKILL during the final superstep loses the exit-mode buffer.
6. **Run-row table without TTL** — openhands `app_conversation_start_task` grows monotonically across restarts when users never delete stuck tasks.
7. **DB-side cleanup that fires only on a user action** — openhands `delete_app_conversation_start_tasks` only fires on `DELETE /conversations/{id}`. There is no GC.
8. **Single-process `liveness` TTL without per-task monitoring** — temporal's matching side. A physical task queue idle for `MaxTaskQueueIdleTime` unloads, but a stuck task within a busy queue is invisible.
9. **Versioning field that is set but never compared** — agent-framework `WorkflowCheckpoint.version` is stored but not validated in `from_dict` (`_checkpoint.py:100-117`). Future versions may break round-trips silently.
10. **Per-host restart metric with no aggregate** — temporal `metrics.RestartCount` is emitted per host but not aggregated. Operators must build their own dashboards.

## Anti-Patterns / Caution Signs

| Anti-pattern | Source | Risk |
|--------------|--------|------|
| Silent retry-by-event without max attempts | openhands (`set_title_callback_processor.py:132-138`) | Callback stays `ACTIVE` forever if event of same kind never arrives |
| Per-run asyncio.Event registry never cleaned up | letta (`streaming_response.py:31-40`) | Memory growth across process lifetime |
| `durability="exit"` for production workloads | langgraph (`_loop.py:1294-1311`) | SIGKILL during final superstep loses buffer |
| `Workflow.run(message=...)` silently rejects recovery attempts | agent-framework (`_workflow.py:803-810`) | Operators may not notice their workflow is broken |
| Best-effort "POST then delete" for outbox-style data | openhands (`live_status_app_conversation_service.py:1994-2006`) | Transient 5xx silently drops user messages |
| Module-level singleton polling task | openhands (`remote_sandbox_service.py:54`) | No `done_callback`; failure is invisible |
| In-memory `outstandingTasks` ack map | temporal (`ack_manager.go:14-32`) | Recovery requires persistence reconstruction |
| No idempotency key on conversation create | openhands (`start_app_conversation`) | Double-click creates two agents |
| Cancel that writes denials for ALL pending tool calls | letta (`run_manager.py:671-781`) | Destructive; not always what user wants |
| `_background_tasks` set without cleanup | letta (`utils.py:1131-1162`) | Memory growth; orphan shielded tasks |

## Notable Absences

The following are **absent** in one or more sources, which represents a maturity gap:

1. **No orphan sweeper for `running` rows**: letta, openhands, langgraph all lack a periodic background scan. Operators must call `cancel_run` or `DELETE /conversations/{id}` manually.

2. **No dead-letter queue**: agent-framework, letta, openhands have no DLQ. Failures are either inline errors or marked `failed` with no inspection path.

3. **No built-in alerting on recovery failure**: none of the five sources ships webhook / email / log alert for "graph mismatch during restore" or "stuck `WORKING` task for 2 hours". Operators must wire their own.

4. **No automatic requeue of failed runs**: letta and openhands treat `failed` as terminal; there is no policy that retries with backoff. Only the Anthropic batch poller (`letta/jobs/llm_batch_job_polling.py:170-247`) has a resume flow.

5. **No idempotency key for tool calls**: agent-framework, letta, openhands do not provide a deduplication mechanism for tool invocations between checkpoints. Replay may double-execute side-effecting tools.

6. **No explicit "abandoned run" marker**: langgraph has `delete_thread` but no "mark this thread as abandoned so the next call starts fresh". The only way to recover from a poisoned state is delete + recreate.

7. **No per-iteration checkpointing option**: agent-framework checkpoints at superstep boundary only. A workflow with long supersteps and high-value side effects between checkpoints has no tighter option.

8. **No version-gated checkpoint migration**: agent-framework `WorkflowCheckpoint.version` is set but never compared in `from_dict`. Future schema changes may break round-trips silently.

9. **No automatic orphan detection for matching tasks**: temporal relies on `liveness` TTL unload, not an explicit sweeper for tasks with no poller activity.

10. **No `start_app_conversation` uniqueness check**: openhands does not check for an in-progress task with the same `conversation_id` before the agent-server POST. Double-clicks can create two agents.

11. **No cross-pod cancellation signaling**: letta's cancellation flag is read from the DB on every poll, which is correct but adds DB round-trips. A Redis pub/sub channel could speed it up.

12. **No DLQ-enabled-by-default for non-replication tasks**: temporal `TaskDLQEnabled` defaults to `false`. Terminal failures silently drop unless the operator opts in.

## Open Questions

1. **What is the durability floor for agent-framework's in-process checkpoint model?** With checkpointing on, the worst-case loss is work in the current superstep up to the last committed checkpoint. The framework does not document this SLA. Checkpoint-save failure is swallowed (`_runner.py:236-238`).

2. **What is the canonical behavior when `durability="exit"` and the process is SIGKILLed?** `_put_exit_delta_writes` is gated on `durability == "exit"` but relies on `__exit__` running. SIGKILL during the final superstep loses the exit-mode buffer by definition.

3. **Is `WorkflowCheckpoint.version` used as a compatibility gate?** It is stored on save, copied on load, but never compared (`_checkpoint.py:100-117`).

4. **Does openhands `start_app_conversation` have a uniqueness check on `(conversation_id, user_id)` before the agent-server POST?** No evidence found. A double-click could create two agents.

5. **Is there a documented operational procedure for `running` runs older than X hours in letta?** No code path does this automatically. Operators must wire their own.

6. **What happens to a letta run when Lettuce is enabled but unreachable?** `get_status` falls back to DB status on exception (`run_manager.py:126-129`), so the run stays `running` indefinitely until cancelled.

7. **Does temporal's `RebuildMutableState` verify that the workflow has actually drifted before overwriting?** The `rebuildableCheck` (`workflow_rebuilder.go:110-148`) suggests it gates on certain conditions, but the exact contract is not documented.

8. **What is the maximum safe parent-chain length for DeltaChannel walks in langgraph?** `DELTA_MAX_SUPERSTEPS_SINCE_SNAPSHOT` (default 5000) prevents unbounded walks. Beyond that, the walk is bounded, but no specific test exercises the limit under high fan-out.

9. **Does temporal's `taskReader` / `taskWriter` recover after a matching process restart mid-`getTasks`?** `setReadLevelAfterGap` handles the "no tasks in range" case but does not cover mid-stream crash.

10. **Is the Letta event-loop watchdog test (`tests/test_watchdog_hang.py`) actually run in CI?** No `pytest` test invocation was found referencing it.

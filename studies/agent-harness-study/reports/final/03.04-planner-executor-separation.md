## Executive Summary

The six sources split cleanly along one axis: how much of the planner/executor contract does the framework itself own? At one end, agent-framework and crewai ship typed, LLM-driven planners with explicit plan schemas, bounded replanning, checkpoint persistence, and event-sourced transitions (9/10 and 8/10). At the other end, langgraph and pydantic-ai ship no planner at all and treat planning as a user convention layered on a low-level loop or graph (2/10 each). openai-agents-sdk occupies a narrow middle: it owns a `ToolExecutionPlan` for per-turn tool dispatch but no user-level multi-step plan (4/10). openhands splits the planner and executor across sub-conversations with shared-workspace coupling via a Markdown file (6/10).

The most reusable insights cut across both extremes. (1) Tool-gating by composition is more reliable than prompt-level denial: openhands strips executor tools from the planner's toolset so attempts to execute from plan mode fail at the registry, not the prompt. (2) Bounded adaptation is the universal pattern: every framework that owns a planner surfaces explicit caps on replans, retries, stalls, or round counts. (3) Plan persistence lives wherever the framework's state already lives — workflow checkpoints (agent-framework), executor state (crewai), workspace file (openhands), schema-versioned `RunState` (openai-agents-sdk), not at all (langgraph, pydantic-ai). (4) Plan-vs-actual comparison is universally missing: none of the six ships a first-class diff between what was planned and what actually happened.

The single most consequential divergence is whether the framework treats planning as in-scope or out-of-scope. The choice correlates with library-vs-application posture: langgraph and pydantic-ai are deliberately general-purpose substrates whose authors explicitly defer planning to third-party packages (`pydantic-ai-todo`, `pydantic-deep`); agent-framework, crewai, openai-agents-sdk, and openhands are application-shaped frameworks that own their planning surface. This is not a quality difference — it is a valid response to different audience and constraint profiles.

## Core Thesis

A planner/executor architecture is **not** a model loop with a todo list. It is a contract with five distinct obligations:

1. **Plan as typed artifact.** A plan must be representable as a structured object (schema, slot, or step list) — not as free text in a prompt, not as a model context, not as a stale workspace file.
2. **Plan owns execution control.** The plan must select the next action, dispatch it, gate it by approval or dependency, and decide when the run terminates. The model may contribute content but does not unilaterally steer the loop.
3. **Plan changes are first-class events.** Every plan creation, replan, sign-off, or revision must be observable as a discrete event, not inferred from message history.
4. **Plan persists by the framework's durability mechanism.** A plan must survive interruption, resume, and crash via the same primitive the framework uses for other state — workflow checkpoint, executor state, `RunState`, typed persistence object.
5. **Plan can be compared to actual.** The system must be able to answer "what was planned vs what ran" — whether through a typed diff API, a structured event history, or an explicit ledger.

The six sources cluster into four approach models along these five obligations:

- **Explicit typed planner** (agent-framework Magentic, crewai modern) — meets all five.
- **Advisory planner** (agent-framework harness, openhands sub-conversation) — meets 1, 3, partially 2 and 4; fails 5.
- **Per-turn derived plan** (openai-agents-sdk) — meets 1, 2, 3; fails 4 (per-run only) and 5.
- **No planner / user convention** (langgraph, pydantic-ai) — meets none.

A useful operational rule: **if the plan is enforced by toolset composition or a state machine, it survives model drift; if it is enforced only by prompt, it does not.** openhands demonstrates the strong form (planner cannot execute because the executor tools are not installed); agent-framework harness demonstrates the weak form (the model can call `todos_complete` without doing the work).

## Approach Models

### 1. Typed LLM-driven explicit planner

**Where it appears:** agent-framework Magentic (Python `_magentic.py:859-1185`, .NET `MagenticOrchestrator.cs:80-396`); crewai modern path (`utilities/reasoning_handler.py:30-242`, `experimental/agent_executor.py:126-161, 314-450`).

**How it works:** A separate planner model (or agent) produces a structured artifact — `MagenticTaskLedger(facts, plan)` plus `MagenticProgressLedger(is_request_satisfied, is_in_loop, is_progress_being_made, next_speaker, instruction_or_question)`, or a `ReasoningPlan` compiled into a typed `TodoList` with status, dependencies, and results. The executor consumes the typed artifact: the orchestrator routes by `ProgressLedger.NextSpeaker` and dispatches via `TurnToken`; the crewai `AgentExecutor` schedules by `TodoList.is_ready` checks that honor dependencies.

**Key mechanisms:**
- **Slot-typed progress ledger.** `BooleanProgressLedgerSlot` and `StringProgressLedgerSlot` (agent-framework .NET `MagenticProgressLedger.cs:170-269`) build a JSON schema for the model's response and parse answers out by slot key. Each slot carries a `FormattedQuestion` and parses both `answer` and `reason` sub-fields.
- **Closed planner schema.** crewai's `create_reasoning_plan` function schema declares required fields with `additionalProperties: false` (`reasoning_handler.py:50-104`).
- **Plan ownership by executor.** Plan generation, refinement, replan, and reset are centralized in the executor flow (`agent_executor.py:314-372, 897-1016, 2504-2589`); the model contributes content but cannot mutate plan state arbitrarily.
- **Incremental replanning.** crewai replaces only pending todos on replan, preserving completed/failed/running history (`agent_executor.py:2504-2589`; `planning_types.py:185-195`).

**Best fit:** multi-agent or multi-step applications where plan-vs-actual reconciliation is operationally important, where plans need explicit review or sign-off, where the plan outlives the LLM call that produced it.

**Cost:** an LLM call (or more) per plan and per replan; tolerance for the planner model's mistakes; schema maintenance.

### 2. Advisory planner with mode/todos

**Where it appears:** agent-framework harness (`_harness/_mode.py:118-325`, `_harness/_todo.py:447-615`, .NET `AgentModeProvider.cs:42-281`, `TodoProvider.cs:42-351`); openhands (`live_status_app_conversation_service.py:185-196, 1071-1091, 1330-1373, 1613-1625`).

**How it works:** A mode flag (`plan` vs `execute`) and a structured todo list together steer a single agent. The mode string is injected into the system prompt; the todo list is re-injected as a synthetic user message each turn. The agent itself calls `todos_add`/`todos_complete`/`todos_remove` and `mode_set` tools. Execution is shaped, not controlled: the plan guides but does not gate. openhands takes this further by making the planner a separate `AppConversation` with a different toolset.

**Key mechanisms:**
- **Tool restriction by composition.** openhands's planner has zero executor tools installed (`live_status_app_conversation_service.py:1613-1617`); the planner physically cannot run code. agent-framework takes the weaker form — the harness agent has the same toolset in both modes but is told via prompt which mode to operate in.
- **Synthetic message injection.** When mode changes externally, a `[Mode changed: ...]` user message is injected (agent-framework `_mode.py:285-325`). openhands's `PlanningFileEditorObservation` triggers a file re-read on the frontend (`conversation-websocket-context.tsx:674-709`).
- **Loop as enforcement.** agent-framework's `TodoCompletionLoopEvaluator` (Python `_loop.py:751-767`, .NET `TodoCompletionLoopEvaluator.cs:42-149`) re-invokes the agent until todos are closed — the loop is the hard gate even though the plan itself is advisory.
- **Provider-aware plan path.** openhands's `_compute_plan_path` switches between `.agents_tmp/PLAN.md` (GitHub) and `agents-tmp-config/PLAN.md` (GitLab, Azure DevOps) based on filename constraints of the git provider (`live_status_app_conversation_service.py:1071-1091`).

**Best fit:** single-agent or research-style workflows where the model's ability to deviate is the point; sub-conversation architectures where planning and execution have different sandbox permissions.

**Cost:** the plan can drift silently — the model can mark todos complete without doing the work, and there is no callback to compare tool-call output to todo content (agent-framework harness); the plan is unstructured Markdown with no step schema (openhands).

### 3. Per-turn derived tool execution plan

**Where it appears:** openai-agents-sdk (`src/agents/run_internal/tool_planning.py:177-683`, `src/agents/run_state.py:131-149, 776-823`).

**How it works:** Every turn, a `ToolExecutionPlan` is constructed from the model's `ProcessedResponse`. The plan has six lanes — `function_runs`, `computer_actions`, `custom_tool_calls`, `shell_calls`, `apply_patch_calls`, `local_shell_calls` — dispatched in parallel via `asyncio.gather`. HITL filtering happens at plan construction (`_collect_runs_by_approval`, `_select_function_tool_runs_for_resume`), so rejected calls never reach executors. There is no LLM-callable plan-update tool and no replanning event; replanning across failure is implicit in the model's next turn.

**Key mechanisms:**
- **Schema-versioned persistence.** `CURRENT_SCHEMA_VERSION = "1.11"` plus a `SUPPORTED_SCHEMA_VERSIONS` frozenset gate read support (`run_state.py:131-150`); resume rebuilds the plan from `_last_processed_response` without re-asking the model.
- **Approved-by-sticky-decision.** `_get_approval_status_for_key` reads `_context._approvals` so a previously approved call auto-runs even after serialization/deserialization (`run_context.py:180-200`).
- **Three-lane next-step dispatch.** `NextStepHandoff`, `NextStepFinalOutput`, `NextStepInterruption`, `NextStepRunAgain` mirror each other in streaming and non-streaming loops (`run_loop.py:791-839, 1065-1153`).

**Best fit:** tool-calling agents where each model turn is the planning unit; products that need fast HITL filtering without paying for a separate planner model.

**Cost:** no multi-step user plan means the model must re-issue tool calls every turn to express "do 1-3, branch, then do 4-7"; no plan-trace event means plan lifecycle is invisible in OpenTelemetry traces; HITL filtering at construction time makes post-hoc selection impossible.

### 4. No planner / user convention

**Where it appears:** langgraph (`libs/langgraph/langgraph/pregel/main.py:2979-2999`, `_algo.py:392-513`); pydantic-ai (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2110-2139`).

**How it works:** The framework provides a substrate — BSP/Pregel loop (langgraph) or a 3-node FSM (pydantic-ai) — and leaves planning to user code. langgraph exposes `Command(goto)` and `Send` as routing primitives plus `interrupt_before`/`interrupt_after` pause points; pydantic-ai exposes an `AgentRun` iterator and per-step callable model settings. Users who want planning build it as a node, holding the plan in state and returning routing instructions. The pydantic-ai docs explicitly defer planning to third-party packages (`pydantic-ai-todo`, `pydantic-deep`).

**Key mechanisms:**
- **Substrate exposure.** langgraph's `prepare_next_tasks` (`_algo.py:392-513`) selects PUSH + PULL tasks from trigger channels; pydantic-ai's `build_agent_graph` registers exactly three node types (`_agent_graph.py:2121-2139`).
- **Per-step seams.** langgraph's `RetryPolicy` (`types.py:416-435`) and pydantic-ai's `for_run_step` tool refresh (`toolsets/abstract.py:118-124`) are the closest analogues to per-plan hooks.
- **Error routing.** langgraph's `node_error_handler_map` routes a failing node to a fallback (`pregel/_runner.py:147-174`), which is the only structured "reroute" the framework provides — and it does not touch any plan.

**Best fit:** library authors and power users who want full control; applications where the planner pattern is still being discovered.

**Cost:** the framework itself can never compare planned to actual because there is no plan to compare against; planning is the user's problem; no built-in plan-revision primitive; plan can drift silently with error handlers (langgraph: `Command(goto)` to a non-existent node is silently dropped at `_algo.py:977-979`).

## Pattern Catalog

### Pattern 1: Slot-typed progress ledger with retries

**Problem solved:** routing decisions need structured answers, not free text. The planner must answer "is the request satisfied? is progress being made? who speaks next?" in a way the executor can parse reliably, retry on failure, and audit later.

**Sources:** agent-framework Magentic. .NET `MagenticProgressLedger.cs:17-132` declares slots with `FormattedQuestion`, `SchemaType` (`"boolean"` / `"string"`), and parses both `answer` and `reason` sub-fields. `MagenticManager.UpdateProgressLedgerAsync` retries JSON parsing and throws on exhaustion (`MagenticManager.cs:66-106`). Python mirrors this with inline prompts (`_magentic.py:193-242`) and a `0.25*attempts` sleep between retries (`_magentic.py:687-727`).

**Why it works:** slots give the planner a typed vocabulary and the executor a typed parser. The retry-and-then-fallback path keeps the orchestrator running even when the LLM produces malformed JSON.

**Copy it when:** the planner and executor live in the same process and need to exchange structured judgments every turn.

**Overkill when:** planning is one-shot or done by a single human; a typed function-call schema is sufficient.

### Pattern 2: Plan-to-todo compilation without task mutation

**Problem solved:** a plan must be representable as executable runtime state, but it must not corrupt the original task description. crewai compiles a `ReasoningPlan` into a typed `TodoList` with `pending`, `running`, `completed`, `failed` statuses, dependency edges, and results — the task description itself is left untouched (`agent_executor.py:314-372`). A regression test verifies the isolation (`test_agent_executor.py:2207-2221`).

**Why it works:** the plan is a runtime artifact, not a prompt fragment. The executor can read pending state, dependency readiness, and results without re-parsing the task. Original task text remains a stable reference.

**Copy it when:** plans must outlive the LLM call and survive multiple execution rounds; replanning must preserve historical context.

**Evidence:** `agent_executor.py:347-349` keeps the plan in state rather than mutating task text.

### Pattern 3: Incremental replanning

**Problem solved:** when a plan must change mid-run, the new plan should preserve what already worked. crewai replaces only pending todos on replan; completed, failed, and running records remain as context for the next model call (`agent_executor.py:2504-2589`; `planning_types.py:185-195`).

**Why it works:** the planner gets a compact "what's left" view; replanning stays cheap; observability is straightforward (history is monotonically appended).

**Copy it when:** replanning is a regular occurrence (failure recovery, scope drift) and you want completed work to remain visible.

**Risk:** if the replanner drops required context, downstream steps may execute against assumptions the new plan does not support.

### Pattern 4: Tool-gating by composition

**Problem solved:** the planner must not be able to execute. Prompt-only enforcement fails under pressure; explicit checks at each tool invocation are repetitive. openhands's planner agent has zero executor tools installed (`live_status_app_conversation_service.py:1613-1617`), so attempts to execute from plan mode fail at the registry before any prompt is read. agent-framework does the same for Magentic participants — the manager agent and the worker agents have different tool sets, declared at workflow compile time.

**Why it works:** "you cannot execute" is enforced by absence, not by instruction. Defense in depth comes from also injecting a `PLANNING_AGENT_INSTRUCTION` suffix (`live_status_app_conversation_service.py:185-196`), but the toolset is the structural guarantee.

**Copy it when:** planning and execution must be physically separated (different security contexts, different sandboxes, different audit trails).

**Overkill when:** planning and execution happen in the same agent and the cost of separation exceeds the benefit.

### Pattern 5: Sub-conversation as the planner boundary

**Problem solved:** when the planner and executor have different tool sets, prompts, condensers, and analytics tags, swapping them inside a single agent becomes tangled. openhands spawns the planner as a separate `AppConversation` whose `parent_conversation_id` points at the executor conversation (`app_conversation_models.py:131-132`). Configuration is inherited from the parent (`_inherit_configuration_from_parent` at `live_status_app_conversation_service.py:1014-1042`). The two conversations share the workspace but have separate LLM sessions, separate histories, and separate condenser tags (`'agent'` vs `'planning_condenser'` for SaaS analytics).

**Why it works:** the boundary is a conversation, not a flag. Configuration, history, and toolset are swapped atomically when the user clicks "Plan" or "Build". The Build hand-off is a literal user instruction: `Execute the plan based on the .agents_tmp/PLAN.md file` (`use-handle-build-plan-click.ts:27`).

**Copy it when:** planning and execution need different LLM contexts, different tool registries, or different lifecycle controls; the parent-child relationship can be expressed in your conversation model.

**Risk:** sub-conversations double the LLM call count and complicate replanning loops because there is no shared event bus.

### Pattern 6: Schema-versioned plan persistence

**Problem solved:** plans must survive serialization/deserialization across SDK versions. openai-agents-sdk assigns `CURRENT_SCHEMA_VERSION = "1.11"` plus a `SUPPORTED_SCHEMA_VERSIONS` frozenset (`run_state.py:131-150`). On resume, the plan is rebuilt from `_last_processed_response` without re-asking the model (`run_state.py:776-823`). agent-framework's Magentic checkpoints serialize `MagenticTaskState` (Python `_magentic.py:1266-1321`; .NET `MagenticOrchestrator.cs:355-396`).

**Why it works:** the schema version is a hard migration gate. Forward compatibility is explicit (which versions can be read) rather than accidental.

**Copy it when:** the plan is persisted to durable storage and the SDK may be upgraded between writes and reads.

**Risk:** fail-fast versioning (`run_state.py:126-130`) means a version mismatch halts the run rather than allowing a best-effort migration.

### Pattern 7: Plan sign-off as a first-class public type

**Problem solved:** when a plan must be reviewed by a human before execution, the review process should not be an ad-hoc protocol. agent-framework ships `MagenticPlanReviewRequest(Plan, CurrentProgress, IsStalled)` with `Approve()` and `Revise(message)` factories (Python `_magentic.py:797-855`; .NET `MagenticPlanReviewRequest.cs:18-44`). When sign-off is enabled, the orchestrator explicitly pauses and resumes via the workflow's `request_info` mechanism.

**Why it works:** the review object is part of the framework's typed surface, so tooling, tests, and checkpoint round-trips treat it like any other plan state. A test asserts the round-trip preserves pending plan-review across `checkpoint_id` resume (`test_magentic_checkpoint_resume_round_trip` at `_magentic.py:403-463`).

**Copy it when:** plan approval is a real workflow step rather than an optional guard.

**Caveat:** the dotnet default is `RequirePlanSignoff = true` (`MagenticWorkflowBuilder.cs:35`) but the Python default is `false` (`_magentic.py:883`). Code that ports between the two must flip the flag.

### Pattern 8: Bounded replanning with explicit counters

**Problem solved:** unbounded replanning is the failure mode every planner fears. Both agent-framework (`TaskLimits(MaxStallCount=3, MaxRoundCount=null, MaxResetCount=null, MaxProgressLedgerRetryCount=3)` at .NET `MagenticTaskContext.cs:11-18`) and crewai (`max_refinement_attempts`, `max_replans`, `max_step_iterations`, `step_timeout` at `planning_config.py:98-142`) expose explicit caps on every adaptation axis. Termination on hit produces a final assistant message rather than a crash.

**Why it works:** the orchestrator knows when to stop trying. Limits are configured once, enforced at every replan, and surface in tests (`test_magentic_stall_and_reset_reach_limits` at `_magentic.py:778-820`).

**Copy it when:** replanning can fire multiple times per run and you cannot trust the model to converge.

**Risk:** a too-tight cap aborts recovery; a too-loose cap wastes tokens. Both frameworks expose this as configuration so the caller decides.

### Pattern 9: Event-sourced plan changes

**Problem solved:** consumers must be able to react to plan creation, replan, refinement, observation, and goal completion without coupling to the orchestrator's internals. agent-framework emits `MagenticPlanCreatedEvent`, `MagenticReplannedEvent`, `MagenticProgressLedgerUpdatedEvent` (Python `_magentic.py:781-795`; .NET `MagenticOrchestrator.cs:17-58`). crewai emits reasoning, plan-step, observation, refinement, replan, and goal-achievement events (`events/types/reasoning_events.py:6-48`, `events/types/observation_events.py:29-131`).

**Why it works:** subscribers attach to the event stream without reaching into orchestrator internals. Events are emitted through `ctx.add_event` / `context.AddEventAsync`, which preserves them in the workflow's event log and includes them in checkpoint state.

**Copy it when:** multiple consumers (UI, telemetry, audit) need to observe plan transitions.

**Gap:** neither framework ships a "what changed in this replan" delta event. A user wanting plan-vs-actual must reconstruct it from successive progress-ledger payloads.

### Pattern 10: Per-session locking on shared state

**Problem solved:** concurrent invocations against the same session can race on todo list updates. agent-framework uses `ConditionalWeakTable<AgentSession, SemaphoreSlim>` (.NET `TodoProvider.cs:67-68`) and `weakref.WeakKeyDictionary[AgentSession, asyncio.Lock]` (Python `_todo.py:484-492`) so per-session locks are GC'd with their session.

**Copy it when:** plan state is mutable within a session and the same session may be invoked concurrently.

**Note:** this pattern is invisible until you have two concurrent requests against the same session — at which point the race produces silent data loss without the lock.

## Key Differences

### Plan persistence location

Where the plan lives correlates with the framework's overall state model:

- **Workflow checkpoint state** (agent-framework Magentic: `MagenticTaskState` round-trips via `OnCheckpointingAsync` at `MagenticOrchestrator.cs:355-396`).
- **In-process executor state** (crewai: `AgentExecutorState` at `agent_executor.py:126-161`, persisted only when flow checkpointing is configured at `agent/core.py:1573-1603`).
- **Schema-versioned `RunState`** (openai-agents-sdk: `_last_processed_response` at `run_state.py:776-823`).
- **Workspace file** (openhands: `.agents_tmp/PLAN.md`, lifetime tied to sandbox).
- **None** (langgraph: only state the user puts there; pydantic-ai: only `message_history`).

### Plan scope: multi-step user plan vs per-turn tool plan

- **Multi-step user plan** (agent-framework Magentic, crewai modern, openhands): the plan represents what the agent intends to do across multiple turns or participants. It is the unit of revision.
- **Per-turn tool plan** (openai-agents-sdk): the plan represents how the model turn's tool calls will be dispatched. It is rebuilt every turn.

### Multi-agent vs single-agent

- **Multi-agent** (agent-framework Magentic, crewai): the plan routes between participants. The next speaker is part of the plan.
- **Single-agent with mode flag** (agent-framework harness, openhands): the plan shapes one agent's behavior. Mode changes flip the agent's role.
- **Single-agent with FSM** (langgraph, pydantic-ai, openai-agents-sdk): the plan is incidental to the loop.

### Plan sign-off surface

- **First-class public type** (agent-framework Magentic): `MagenticPlanReviewRequest`/`MagenticPlanReviewResponse` are public types with builder flags and checkpoint-aware state.
- **HITL as pause/resume** (openai-agents-sdk, openhands): approval is expressed as an interruption that resumes the run with a different plan subset.
- **Implicit / none** (crewai modern, langgraph, pydantic-ai): the planner is not gated by an external approval surface.

### Plan trigger for replanning

- **Stall detection** (agent-framework Magentic): progress ledger reports `is_progress_being_made=False`; orchestrator counts stalls and triggers `ResetAndReplanAsync` after a threshold.
- **Observation failure** (crewai modern): a failed observation routes to refinement or replan depending on reasoning effort.
- **Human revise** (agent-framework Magentic, openhands): a reviewer explicitly requests changes.
- **Implicit model re-plan** (openai-agents-sdk, langgraph, pydantic-ai): the next model turn produces a new plan subset implicitly.

### Replan fidelity

- **Structured replan decision** (agent-framework Magentic: progress ledger slots; crewai: typed observation fields driving replan paths).
- **Heuristic replan detection** (crewai also searches the last message for fixed indicators at `agent_executor.py:2452-2502` — this is brittle but practical).
- **No replan** (openai-agents-sdk: implicit through next model turn; langgraph, pydantic-ai: no concept).

## Tradeoffs

### Plan flexibility vs determinism

Magentic's plans are LLM-authored free text; routing depends on the model choosing a name from the participant list and writing valid JSON for the progress ledger. The framework defends with retries (`progress_ledger_retry_count`, default 3) and explicit fallbacks (`.NET MagenticOrchestrator.cs:308-311` chooses first participant when `next_speaker` is empty), but the orchestrator can still misinterpret an ambiguous reply. crewai takes the opposite path: a closed JSON schema (`additionalProperties: false`) makes the plan deterministic at the cost of asking the planner to produce a less expressive artifact.

**When to prefer Magentic-style:** plans must express natural language reasoning and the executor can absorb noise (multi-agent research, brainstorming, open-ended tasks). **When to prefer crewai-style:** plans must drive structured execution with explicit dependencies and the planner model is reliable.

### Centralized mutation vs agent autonomy

crewai routes plan changes through the executor flow (`agent_executor.py:897-1016`), preventing the agent from mutating plan state arbitrarily but also preventing the agent from correcting its own plan via a tool. agent-framework's harness takes the opposite path: the agent itself calls `todos_add`/`todos_complete`/`todos_remove` and `mode_set` tools. The Magentic variant is intermediate: the manager agent can revise the plan only via the orchestrator's replan flow, never directly.

**Benefit of centralized:** auditability, bounded authority, easier testing. **Cost:** no in-flight plan correction; the model cannot refine its plan without a full replan cycle. **Benefit of agent autonomy:** faster plan correction, no need for the executor to model every possible plan change. **Cost:** the agent can mark todos complete without doing the work; no callback compares tool-call output to todo content.

### Plan persistence location vs lifetime

openhands persists the plan as a file in the sandbox workspace, lifetime-bound to the sandbox. When the sandbox stops, the plan is gone — the only durable record is the event stream. agent-framework Magentic persists the plan in workflow checkpoint state, lifetime-bound to the checkpointer. crewai persists the plan in executor state, lifetime-bound to the flow's session. openai-agents-sdk persists the plan in `RunState`, lifetime-bound to the response chain.

**The trade-off:** in-workspace plans are inspectable by humans and other agents; in-checkpoint plans are durable across crashes but invisible to the user; in-executor-state plans are easy to read but lost when the process exits.

### Tool-gating by composition vs prompt-level denial

openhands removes executor tools from the planner's toolset — a structural guarantee that survives prompt failures. agent-framework harness injects the same planning instruction into the system prompt of the same agent that has both plan-mode and execute-mode tools. The structural form is stronger because it fails closed (no executor tools = no execution); the prompt form fails open (the model can ignore the prompt).

### Schema-driven plan vs free-form plan

agent-framework Magentic's progress ledger slots are typed, parseable, and retry-safe. crewai's `ReasoningPlan` is a closed JSON schema with `additionalProperties: false`. openhands's plan is free-form Markdown. openai-agents-sdk's plan is a typed dataclass but for tool dispatch only. langgraph and pydantic-ai have no plan schema at all.

**The trade-off:** typed schemas give the executor a parser and the operator an audit trail. Free-form plans give the planner room to express intent in natural language. The right choice depends on whether plan-vs-actual reconciliation matters operationally.

### Bounded adaptation vs unbounded recovery

Every framework that owns a planner exposes explicit caps: agent-framework (`TaskLimits(MaxStallCount=3, MaxRoundCount=null, MaxResetCount=null, MaxProgressLedgerRetryCount=3)`), crewai (`max_refinement_attempts`, `max_replans`, `max_step_iterations`, `step_timeout`). The trade-off is between "give the model enough rope to recover" and "stop wasting tokens on a doomed plan". Both frameworks expose the caps as configuration rather than hard-coding them.

## Decision Guide

### Choose an explicit typed planner when...

- Plans must outlive the LLM call (cross-turn or cross-session persistence).
- Plans must be reviewed by humans or external systems before execution.
- Plan-vs-actual reconciliation matters (debugging, audit, billing).
- Multi-agent routing needs a typed `next_speaker` decision.
- The plan is the unit of checkpoint and resume.

**Pattern sources:** agent-framework Magentic (most rigorous), crewai modern (typed but executor-owned).

### Choose an advisory planner when...

- Single-agent workflows where model autonomy is the point.
- The "plan" is really a structured todo list, not a multi-step artifact.
- Plan changes are expected to be cheap and frequent.
- A loop wrapper can enforce "run until todos are closed" without a typed plan schema.

**Pattern sources:** agent-framework harness, openhands sub-conversation.

### Choose a per-turn derived tool plan when...

- The model is the planner — every model turn produces the plan.
- Tool parallelism is the main goal (six-lane parallel dispatch).
- HITL filtering at construction time is acceptable.
- Plan scope is "this turn's tool calls", not "this run's overall plan".

**Pattern source:** openai-agents-sdk.

### Choose no framework planner when...

- You are building a library, not an application — the planner pattern is still evolving.
- Power users will bring their own planner and the framework should stay out of the way.
- You want to compose planner and executor as graph nodes without commitment to a schema.

**Pattern sources:** langgraph, pydantic-ai.

### Decision rules for the main approaches

1. **Do you need HITL plan sign-off as a public surface?** Yes → typed explicit planner (agent-framework Magentic). No → any approach.

2. **Do plans need to survive process crashes?** Yes → typed planner with checkpoint persistence (agent-framework Magentic, crewai if flow checkpointing is configured). No → advisory or per-turn may suffice.

3. **Is the planner a different model than the executor?** Yes → typed explicit planner with separate manager agent (agent-framework Magentic). No → advisory planner with mode flag is fine.

4. **Do plans need to drive parallel execution of typed dependencies?** Yes → crewai modern (typed `TodoList` with `is_ready` checks). No → any approach.

5. **Is the planner's toolset different from the executor's?** Yes → sub-conversation model (openhands) or compile-time tool gating (agent-framework Magentic participants). No → single agent with prompt-level mode flag.

6. **Do you need plan-vs-actual diff as an audit artifact?** None of the six ships this out of the box. You must build it from typed event streams (agent-framework Magentic, crewai) or from message history + user convention (langgraph, pydantic-ai).

## Practical Tips

### Patterns to copy

1. **Slot-typed progress ledger with retries.** Build a typed slot for each planner question (`is_request_satisfied`, `next_speaker`, `instruction_or_question`) with a `SchemaType` and a parser that reads both `answer` and `reason`. Add JSON retry with backoff before falling back. (agent-framework Magentic .NET `MagenticProgressLedger.cs:170-269`, Python `_magentic.py:193-242`).

2. **Plan-to-todo compilation that does not mutate the task description.** Keep the plan in executor state; do not splice it into the original task text. Add a regression test asserting the original task is unchanged. (crewai `agent_executor.py:347-349`; test `test_agent_executor.py:2207-2221`).

3. **Incremental replan that preserves history.** Replace only pending todos on replan; keep completed/failed/running records as context. (crewai `agent_executor.py:2504-2589`).

4. **Tool-gating by composition for planner/executor separation.** Strip executor tools from the planner's toolset at compile time. Prompt-level denial is defense in depth, not the primary control. (openhands `live_status_app_conversation_service.py:1613-1617`).

5. **Sub-conversation as the planner boundary.** When planning and execution need different LLM sessions, tool registries, and analytics tags, model the planner as a separate conversation with a `parent_conversation_id` and explicit configuration inheritance. (openhands `app_conversation_models.py:131-132`, `_inherit_configuration_from_parent`).

6. **Schema-versioned plan persistence.** Assign a `CURRENT_SCHEMA_VERSION` and a `SUPPORTED_SCHEMA_VERSIONS` frozenset at write time. Resume rebuilds the plan from the persisted typed object without re-asking the model. (openai-agents-sdk `run_state.py:131-150, 776-823`).

7. **Plan sign-off as a first-class public type.** When review is a real workflow step, make the review object a public type with `Approve()` and `Revise(message)` factories. (agent-framework Magentic `_magentic.py:797-855`; .NET `MagenticPlanReviewRequest.cs:18-44`).

8. **Bounded replanning with explicit counters.** Surface `MaxStallCount`, `MaxRoundCount`, `MaxResetCount`, `MaxProgressLedgerRetryCount` as configuration. Add tests that exercise each limit. (agent-framework Magentic; crewai `planning_config.py:98-142`).

9. **Event-sourced plan changes.** Emit `PlanCreated`, `Replanned`, `ProgressLedgerUpdated` events through the framework's event context. (agent-framework Magentic; crewai reasoning/observation events).

10. **Per-session locking on shared state.** Use weak-keyed maps for per-session locks so locks are GC'd with their session. (agent-framework harness `TodoProvider.cs:67-68`, `_todo.py:484-492`).

### Patterns to avoid or delay until needed

1. **Free-form plans injected into task text.** They cannot be parsed back, replanned against, or audited. Use only as a compatibility fallback. (crewai legacy `planning_handler.py:15-164` is the cautionary example.)

2. **Advisory plans without a loop-level gate.** If the plan is advisory and nothing enforces it, the model can mark todos complete without doing the work. If you go advisory, wrap with a `TodoCompletionLoopEvaluator` or equivalent. (agent-framework harness shows both halves.)

3. **Heuristic replan triggers based on message string matching.** They miss semantically equivalent failures and react to incidental wording. (crewai `agent_executor.py:2452-2502` uses fixed indicators.)

4. **Optimistic observer-failure recovery that treats failure as success.** It allows progress at the cost of correctness. Make this policy configurable rather than the default. (crewai `planner_observer.py:191-213` is the cautionary example.)

5. **Plan persistence co-located with sandbox/workspace lifetime.** If the sandbox stops, the plan is gone. Persist to a durable store keyed by conversation/session, not by sandbox. (openhands's `.agents_tmp/PLAN.md` is the cautionary example.)

6. **Schema versioning that fails fast on mismatch.** A user mid-task who upgrades SDKs and finds the resume halts is a worse experience than a best-effort migration. (openai-agents-sdk `run_state.py:126-130` is the cautionary example.)

7. **Two planner architectures in one repo without clear signposting.** Beginners face real cognitive load. If you ship both (like agent-framework), document the boundary explicitly and provide composition examples. (agent-framework has Magentic + harness + workflows but no documented wiring showing a `HarnessAgent` as a Magentic participant.)

## Anti-Patterns / Caution Signs

1. **The "plan" lives only in the model context window.** If the framework has no plan field, no `Plan` schema, no plan channel, you cannot reconstruct "what was the plan before step N" without parsing message history. langgraph and pydantic-ai exhibit this.

2. **The plan is enforced only by prompt, not by tools or state.** If removing the prompt leaves the agent free to execute, the plan is decorative. openhands's planner is structurally separated by toolset; agent-framework's harness is not.

3. **The planner is allowed to mark todos complete without doing the work.** There is no callback comparing tool-call output to todo content. agent-framework harness and openhands's markdown plan both have this gap.

4. **Replan triggers are heuristic string matches.** "REPLAN" or "TRY_AGAIN" in the last message is brittle. crewai's last-message scanning is a known weakness.

5. **Observer failure is treated as success.** "Mark step successful and continue" is the wrong default for an LLM-based observer that throws. crewai's `planner_observer.py:191-213` does this.

6. **Plan persistence is co-located with ephemeral storage.** A sandbox file, a per-process state object, or an unbounded context window is not a durable plan store. openhands's `.agents_tmp/PLAN.md` and crewai's in-process executor state are cautionary examples.

7. **Plan sign-off has no timeout.** If the human never responds, the workflow stays in `IDLE_WITH_PENDING_REQUESTS` indefinitely. agent-framework Magentic has no sign-off timeout (`test_magentic_checkpoint_resume_round_trip` confirms pending requests survive resume but does not exercise a timeout).

8. **Replan budget and round budget are conflated.** When both counters tick on the same event, total iterations are `outer × inner` and easy to underestimate. agent-framework's loop middleware (`_loop.py:117`) defaults to 10 iterations at the outer level while `FunctionInvocationLayer` (`_tools.py:90`) defaults to 40 inner iterations.

9. **Mode/todo changes are not traceable as events.** agent-framework harness's todo list changes appear only as tool calls in the model's transcript, not as framework events.

10. **Two planner architectures in one repo without documented composition.** agent-framework ships both Magentic and harness but the documentation does not show how to compose them. A user wiring a `HarnessAgent` as a Magentic participant is on their own.

11. **The plan is unstructured and the system has no schema to drive plan-vs-actual comparison.** openhands's Markdown plan is the cautionary example.

12. **Plan changes across turns are not surfaced as replanning events.** openai-agents-sdk's `NextStepRunAgain` is implicit; there is no "plan revised" marker in the run state.

13. **Planner and executor share the same LLM model with no isolation.** crewai's planner resolves to the agent's LLM unless a planning-specific LLM is configured (`reasoning_handler.py:175-185`). A planner that fails in the same way as the executor is a single point of failure.

14. **Graph snapshots cannot be replayed into a different plan.** pydantic-ai's `BaseStatePersistence` (`pydantic_graph/persistence/__init__.py:113-123`) snapshots `(state, next_node)`; restoring gives back the loop at step N. If the user's plan was stored only in `message_history`, restoration works; if in `RunContext.deps`, restoration loses it.

## Notable Absences

These are gaps shared across most or all sources, where evidence in the source reports explicitly notes the absence.

1. **No first-class "plan diff" event or API.** None of agent-framework, crewai, openai-agents-sdk, openhands, langgraph, or pydantic-ai ships a structured "what changed in this replan" delta event or a public function that returns a planned-vs-actual diff. agent-framework source notes: "Neither pattern exposes a structured 'what was planned vs what happened' report." crewai source notes: "no evidence was found of a dedicated report or API that computes a complete planned-versus-actual diff." openai-agents-sdk source notes: "There is no plan-vs-actual reconciliation surface, no `PlanAudit` event, and no test asserts that the plan at execution time matched the plan at construction time." langgraph source notes: "Because planning is purely user-defined, the system cannot compare what it planned to what it actually did unless the user also writes the bookkeeping." pydantic-ai source notes: "The framework cannot compare what it planned to what it actually did, because nothing is ever recorded as the 'plan' in the first place."

2. **No DAG-style plans with dependencies in any "todo" implementation.** agent-framework harness's `TodoProvider` is a flat item list with no dependencies, no priorities, no grouping. openhands's `PLAN.md` is unstructured Markdown. crewai's modern path is the exception with explicit dependency edges (`planning_types.py:112-145`), but only crewai modern has this.

3. **No cross-framework plan schema.** Every framework that owns a plan defines its own schema. There is no shared vocabulary for "plan step", "todo", "next action" across the six sources.

4. **No first-class replanning signal in openai-agents-sdk, langgraph, or pydantic-ai.** The keyword `replan` returns zero hits in their source trees. Replanning is implicit (next model turn) or user convention.

5. **No plan-vs-actual metric in any source.** run-step counters, output-retry counters, and stall counters exist, but none of them measures "the model diverged from any user-supplied plan".

6. **No multi-model isolation between planner and executor by default.** Only agent-framework Magentic uses a separate manager agent by default; every other source reuses the agent's LLM unless explicitly configured otherwise.

7. **No framework exposes a `planning` trace span in OpenTelemetry.** openai-agents-sdk ships turn/agent/function/tool spans but no plan span; the plan lifecycle is invisible in traces.

8. **openhands's actual planner loop is in external packages.** The planner agent's `system_prompt_planning.j2`, the `planning_file_editor` tool, and any replanning triggers live in `openhands-tools==1.29.0` and `openhands-agent-server==1.29.0`, pinned in `pyproject.toml:67-72`. The studied repo only owns the seam.

9. **agent-framework harness mode change made by the agent itself does not produce a notification.** Only external mode changes inject the `[Mode changed: ...]` user message. The asymmetry is intentional but easy to miss.

10. **No first-class "compare planned vs done" helper in agent-framework or crewai.** Both architectures could trivially compute "todos completed vs todos created" or "next-speaker chosen vs plan participant-list," but no public function ships this.

## Per-Source Notes

### agent-framework

**Strengths:** Two parallel first-class architectures (Magentic explicit planner + harness advisory planner) implemented in two languages (Python + .NET) with full parity. Magentic has a slot-typed progress ledger, schema-validated plan, event-sourced plan changes, bounded replanning (stall/reset/round counters), checkpoint persistence, and human-in-loop sign-off. Harness has typed todo state, mode-flag system-prompt injection, per-session locking, loop-evaluator pattern, and file-store path safety. **Weaknesses:** Harness plan is advisory; Magentic plan is LLM-authored free text; no plan-vs-actual diff; two planner architectures in one repo without documented composition; .NET and Python defaults differ on plan sign-off (`true` vs `false`).

**Files of interest:** `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py`, `python/packages/core/agent_framework/_harness/_mode.py`, `python/packages/core/agent_framework/_harness/_todo.py`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs`, `dotnet/src/Microsoft.Agents.AI.Harness/`.

### crewai

**Strengths:** Modern path is a clean planner/executor separation with closed JSON schema, typed todo state, dependency-aware scheduling, parallel-ready execution, observation-driven continuation, event-sourced transitions, bounded adaptation, and incremental replan that preserves history. Legacy path remains supported for compatibility. Tests cover plan isolation, replan triggers, and dependency failures. **Weaknesses:** No plan-specific durable store; no crash-recovery tests for partially completed plans; observer-failure treated as success; heuristic replan triggers (last-message string matching); legacy planner is materially weaker.

**Files of interest:** `lib/crewai/src/crewai/utilities/planning_types.py`, `lib/crewai/src/crewai/utilities/reasoning_handler.py`, `lib/crewai/src/crewai/experimental/agent_executor.py`, `lib/crewai/src/crewai/agents/step_executor.py`, `lib/crewai/src/crewai/agents/planner_observer.py`.

### langgraph

**Strengths:** Powerful substrate (`Command(goto)`, `Send`, `interrupt_before/after`, `node_error_handler_map`) for user-defined planners. **Files of interest:** `libs/langgraph/langgraph/pregel/main.py`, `libs/langgraph/langgraph/pregel/_algo.py`, `libs/langgraph/langgraph/types.py`, `libs/langgraph/langgraph/pregel/_io.py`, `libs/langgraph/langgraph/pregel/_runner.py`.

**Strengths:** Typed `ToolExecutionPlan` for per-turn tool dispatch; schema-versioned persistence; parallel six-lane tool execution; HITL filtering at plan construction; approved-by-sticky-decision. **Weaknesses:** No multi-step user plan; no LLM-callable plan-update tool; no plan-trace event; HITL filtering at construction time means post-hoc selection is impossible; schema versioning is fail-fast; sequential guardrails only on first turn.

**Strengths:** Clean two-mode agent model (`AgentType.DEFAULT` vs `AgentType.PLAN`); tool-gating by composition removes executor tools from the planner; sub-conversation isolation with `parent_conversation_id`; provider-aware plan path; SaaS analytics tagging via distinct `usage_id`s; defense-in-depth via `PLANNING_AGENT_INSTRUCTION` suffix. **Weaknesses:** Plan is unstructured Markdown with no step schema; no in-process step state machine; no plan-vs-actuals comparison; planner loop is in external `openhands-tools` and `openhands-agent-server` packages (not in this repo); replanning on executor failure is implicit (user click); plan persistence co-located with workspace lifetime.

**Strengths:** Minimal 3-node FSM; `AgentRun` iterator exposes every node for user-supplied planning; per-step callable settings as a planning seam; deferred_loading / `for_run_step` for dynamic tool plans; agent delegation as tool call; philosophy of strong primitives over opinionated solutions. **Weaknesses:** No plan schema; no planner agent; no plan-update tool; no replanning event; planning deliberately delegated to third-party packages (`pydantic-ai-todo`, `pydantic-deep`); `run_step` and `output_retries_used` are framework-internal scalars with no plan semantics; graph snapshots cannot be replayed into a different plan.

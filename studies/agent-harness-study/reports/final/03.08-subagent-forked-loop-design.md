## Executive Summary

Across the six sources, **delegation is universally supported**, but the architectural shape of "a parent agent delegates to a child loop" varies dramatically — from a *user-shaped tool that calls another agent's `run()`* (pydantic-ai) to a *structural embedding of a child graph inside the parent's dispatch loop* (langgraph) to *workflow-first orchestration with parallel background tasks and graph executors* (agent-framework). Three convergent truths emerge:

1. **A "child run" is never just a string return.** Every source that exposes delegation either invokes a real child run loop (langgraph, pydantic-ai, openai-agents-sdk, crewai in-process) or wraps an `asyncio.create_task(...)` of one (agent-framework `BackgroundAgentsProvider`, crewai A2A turn loop). The mechanism for returning a value to the parent is the only thing that varies.
2. **Trace and span correlation is everywhere, but explicit `parent_run_id` linkage is rare.** LangGraph relies on langsmith and OTel; openai-agents-sdk relies on trace group_id and the span tree; pydantic-ai relies on OTel baggage; crewai goes furthest with an event-bus contextvars-backed `EventRecord` graph with eight edge types; langgraph holds the most data by carrying a `checkpoint_ns` per task and a `checkpoint_map` for parent → child traversal. openhands ships the weakest runtime story — DB `parent_conversation_id` exists but Laminar/LiteLLM tracing is per-conversation.
3. **State isolation lives on a spectrum, not a binary.** LangGraph lets you choose `checkpointer=True | False | None` per node; agent-framework's `as_tool()` defaults to a fresh `AgentSession` but offers `propagate_session=True`; openai-agents-sdk's `as_tool()` creates a fresh `ToolContext` but inherits usage and approvals; pydantic-ai defaults to total isolation with manual opt-in sharing; crewai inherits the parent's memory by default. The "right" default depends on whether the source optimizes for agent-as-tool ergonomics (agent-framework, openai-agents-sdk) or graph-state machine purity (langgraph) or building-block simplicity (pydantic-ai).

Three architectural patterns are *not yet uniformly available* across the field: **first-class parallel subagent fan-out within one LLM turn** (agent-framework and openai-agents-sdk expose it; crewai declares but does not implement; langgraph requires `Send` or `@task`; pydantic-ai relies on `asyncio.gather`); **async cancellation of background subagents** (agent-framework's `BackgroundAgentsProvider` cannot cancel from outside; langgraph's `_should_stop_others` does cancel siblings; pydantic-ai uses `cancel_and_drain`); and **durable subagent state across replays** (langgraph via checkpoint namespace, agent-framework via Magentic reset + checkpoint, pydantic-ai via DBOS restriction).

The strongest designs (langgraph 9, agent-framework 8, openai-agents-sdk 8) earn their place by exposing a **structured delegation contract** — one mechanism with explicit lifecycle, isolation, error, and trace contracts — rather than several overlapping surfaces that confuse operators.

## Core Thesis

**Delegation in modern agent frameworks converges on "child is a real run, parent sees the result", but diverges on whether child state is structurally isolated (langgraph, pydantic-ai), opt-in isolated (agent-framework, openai-agents-sdk), or shared by default (crewai).** The right model for a given product is dictated by its primary surface: graph-orchestration systems treat child invocation as a node property; SDK/tooling systems treat it as a tool-call facades; ORM-style app-servers (openhands) treat it as a row-level relationship and delegate the run-time semantics to an external SDK.

## Approach Models

The six sources cluster into five overlapping approach archetypes. A given source may combine several — agent-framework is the most combinatorial; openhands and pydantic-ai are the most prescriptive.

**Cluster A — Tool-call delegation (agent-as-tool).** The parent agent's LLM chooses to call a tool whose body invokes a real child agent run and returns its result as a string. This is the dominant pattern in *user-facing SDKs* where the user's mental model is "the LLM can call other LLMs". Members:
- agent-framework `agent.as_tool()` in `python/packages/core/agent_framework/_agents.py:485-579` — synchronous, blocks the parent's tool-call loop; default fresh child session.
- openai-agents-sdk `Agent.as_tool()` in `src/agents/agent.py:508-936` — invokes a nested `Runner.run(...)`; default `failure_error_function` localizes child errors as a string returned to the parent LLM.
- pydantic-ai — *no built-in `as_tool` surface*; users write a `@tool` whose body is `await child.run(...)`. Fully isolated by default.

**Cluster B — Structural embedding (child graph as a node).** The parent runtime discovers a child runnable and runs it as one of its own tasks. The child is structurally a node, not a tool call. Members:
- langgraph — embedding `Pregel` as a node via `find_subgraph_pregel` (`libs/langgraph/langgraph/pregel/_utils.py:47-75`); per-task `checkpoint_ns` (`libs/langgraph/langgraph/pregel/_algo.py:615-624`); runner-level concurrency with sibling cancellation.

**Cluster C — Workflow/routing/structured-orchestration surfaces.** A graph of executors inside a single workflow run; agents are peers, not children of a parent. Members:
- agent-framework `WorkflowExecutor` (`python/packages/core/agent_framework/_workflows/_workflow_executor.py:103-697`); `WorkflowAgent` wrapping a workflow as an agent (`_workflows/_agent.py:52-479`); Handoff orchestrator (`orchestrations/.../_handoff.py:200-561`); Magentic with replan resets (`_magentic.py:1329-1366`); Group chat (`_group_chat.py:96-540`); Sequential and Concurrent builders.
- crewai — sequential `Process.sequential` is the default; `Process.hierarchical` creates a manager agent that delegates via tools (`lib/crewai/src/crewai/crew.py:1494-1518`).
- agent-framework `BackgroundAgentsProvider` (`_harness/_background_agents.py:240-521`) — *out-of-loop* parallel delegation; children are real `asyncio.Task`s with their own sessions.

**Cluster D — Cross-process delegation (A2A).** The framework serializes a delegation hop onto a wire protocol with JSON-RPC/gRPC/HTTP+JSON negotiation and conversation continuity. Members:
- crewai — `wrap_agent_with_a2a_instance` (`lib/crewai/src/crewai/a2a/wrapper.py:97-237`) monkey-patches the agent's `execute_task`/`kickoff` methods via `MethodType` after model construction; drives a multi-turn loop (`wrapper.py:1272-1407`) bounded by `max_turns` (default 10).
- pydantic-ai — `fasta2a.pydantic_ai.agent_to_a2a` (`pydantic_ai_slim/pydantic_ai/_a2a.py:79-319`) for serving; in-process delegation uses `await agent.run(...)`.
- agent-framework — its handoff and workflow surfaces are in-process only; no A2A equivalent observed.

**Cluster E — Persistence-first (parent/child as DB rows, no in-tree run loop).** Members:
- openhands — `parent_conversation_id` column (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:109-111`); sub-conversation search filter, cascade delete, configuration inheritance; the actual run loop is owned by the external `openhands-sdk`.

**Where sources sit:**
- agent-framework covers A + C + background-provider (model-orchestrated async) — most multiplicity by design.
- crewai covers A + D + scope-graph event tracing.
- langgraph covers B + remote (B-via-`RemoteGraph`) + `GraphBubbleUp` family — most uniform.
- openai-agents-sdk covers A + a separate "control transfer" semantic (handoff) that is *not* A.
- openhands is the only E-shaped source.
- pydantic-ai is the most minimal: A as user pattern + B-equivalent (`pydantic_graph` fork/join) + D (A2A serving).

## Pattern Catalog

The following patterns appear in multiple sources or demonstrate a particularly clean shape in one source.

### P1. Agent-as-Tool wrapper

**Problem:** Allow a parent's LLM to choose a subagent like any other tool while still letting the subagent run its own LLM loop, accumulate usage, and return a structured result.

**Where demonstrated:** agent-framework, openai-agents-sdk; pydantic-ai (user pattern); crewai DelegateWorkTool (a coarser flavor — picks a coworker by role rather than generic tools).

**Why it works:** The LLM already understands "call a function with arguments" — wrapping a subagent as `FunctionTool` plugs into existing tool-call dispatch, guardrails, approvals, and tracing without inventing a new dispatcher. Openai-agents-sdk takes this further: a handoff is also a function tool (`src/agents/handoffs/__init__.py:172-176`), so the entire `transfer_to_*` family shares the tool-call surface.

**When to copy:** When (a) the primary surface is "LLM picks what to do next"; (b) usage accounting per child matters; (c) the parent wants a structured string reply; (d) you can tolerate synchronous blocking within a single tool-step.

**When it is overkill or risky:** When you need parallel children inside one LLM turn (only one tool call fires per step in tool-call dispatch — except where the tool planner fans out per-step, which openai-agents-sdk does via `asyncio.gather` in `src/agents/run_internal/tool_planning.py:572-595`).

**Evidence:** `_agents.py:485-579` (agent-framework); `agent.py:508-936` (openai-agents-sdk).

### P2. Structural embedding of a child runnable

**Problem:** Composing a child graph inside a parent graph without inventing a parallel delegation surface.

**Where demonstrated:** langgraph (the only source that elevates this to the *primary* model); agent-framework `WorkflowExecutor` (`_workflows/_workflow_executor.py:103-697`) wraps an entire workflow as a single executor node.

**Why it works:** The runtime already knows how to schedule one task per node. Embedding a child graph as a node reuses that scheduler, the cancellation policy, the checkpoint mechanism, and the trace context — without any new abstraction. Langgraph earns its rating because the entire delegation vocabulary (local, remote, with-state, without-state, interruptable, streamed) bottoms out in `find_subgraph_pregel → PregelNode.subgraphs → get_subgraphs`.

**When to copy:** When the system is *graph-oriented* (state-machine with channels), or when the cost of a separate tool surface outweighs the ergonomics.

**When it is overkill or risky:** For a pure SDK where users do not think in graphs. Tooling libraries (openai-agents-sdk) prefer P1.

**Evidence:** `libs/langgraph/langgraph/pregel/_utils.py:47-75`; `_workflows/_workflow_executor.py:103-697`.

### P3. Persistent child + parent run identity via explicit session/state objects

**Problem:** Distinguishing "child" from "sibling" without external tracing context.

**Where demonstrated:**
- agent-framework — `AgentSession(session_id, service_session_id)` (`_agents.py:411-449`); per-executor session in `AgentExecutor` (`_workflows/_agent_executor.py:169`); per-execution-id `_execution_contexts` in `WorkflowExecutor` (`_workflow_executor.py:103-302`).
- pydantic-ai — `AgentRun.run_id` (UUID7) stamped per response (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:852,912,969`).
- crewai — `reference_task_ids` propagated across A2A turns (`lib/crewai/src/crewai/a2a/wrapper.py:1066-1143`), and event-bus `parent_event_id` (`lib/crewai/src/crewai/events/event_bus.py:540-568`).
- openhands — `parent_conversation_id` column (`sql_app_conversation_info_service.py:109-111`).

**Why it works:** A real identifier on the run object — not just a span or a log line — means downstream consumers can query "what ran as a child of X?" without parsing trace trees. CrewAI's event graph (`EventRecord` with `parent`/`child`/`trigger`/`triggered_by`/`next`/`previous`/`started`/`completed_by` edges at `lib/crewai/src/crewai/state/event_record.py:99-202`) is the most explicit; agent-framework's `AgentSession` is the most typeful.

**When to copy:** When you must support replay, cross-process debugging, durable persistence, or programmatic reconstruction of delegation trees.

**When it is overkill or risky:** When you only need ordered logs; adding identifier plumbing to small runtimes adds friction.

**Evidence:** `_agents.py:411-449`, `event_record.py:99-202`, `sql_app_conversation_info_service.py:109-111`.

### P4. Event-bus scope stack for trace correlation

**Problem:** Correlating nested events from a child back to a parent without threading `parent_id` through every API.

**Where demonstrated:** crewai's `_event_id_stack` `contextvars.ContextVar` with `push_event_scope`, `pop_event_scope`, `get_current_parent_id` (`lib/crewai/src/crewai/events/event_context.py:41-134`) — combined with `VALID_EVENT_PAIRS` to auto-pair `started`/`completed` events (`event_context.py:240-369`). Langgraph uses `var_child_runnable_config` + `_set_tracing_context({"parent": run})` (`libs/langgraph/langgraph/_internal/_runnable.py:67-103`). Pydantic-ai uses OTel baggage (`pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:127-187`).

**Why it works:** Stack-based context is invisible to user code, propagates naturally into asyncio/tasks/threads, and does not require changes to function signatures. CrewAI's stack survives `concurrent.futures.ThreadPoolExecutor` snapshots via `contextvars.copy_context()` (`lib/crewai/src/crewai/a2a/wrapper.py:233-235`).

**When to copy:** When events are the primary observability surface and you need accurate parenting across async/concurrent boundaries without an external tracer.

**When it is overkill or risky:** When observability lives in an existing tracer (LangSmith, Laminar, OpenTelemetry) — duplicating scope stacks adds bookkeeping.

**Evidence:** `events/event_context.py:41-134`, `_internal/_runnable.py:67-103`, `capabilities/instrumentation.py:127-187`.

### P5. Sibling-cancel-on-failure in concurrent runners

**Problem:** When one task in a fan-out fails, draining the others promptly without orphaning them.

**Where demonstrated:** langgraph `_should_stop_others` (`libs/langgraph/langgraph/pregel/_runner.py:616-634`); pydantic-ai `gather()` via AnyIO task group with `ExceptionGroup` unwrap (`pydantic_ai_slim/pydantic_ai/_utils.py:201-229`) and `cancel_and_drain` with `CancelScope.shield=True` (`_utils.py:232-249`); pydantic-ai also has `ReducerContext.cancel_sibling_tasks()` (`pydantic_graph/pydantic_graph/join.py:73-79,141-148`) and `GraphRun.override_next` (`graph_builder.py:134-144`) for hooks-driven recovery; pydantic-ai tool step uses `asyncio.wait(FIRST_COMPLETED)` with cancel on first failure (`_agent_graph.py:1902-1956`).

**Why it works:** Deterministic cancellation makes "first failure wins" idiomatic — the user can write `asyncio.gather(...)` and trust that a failing child does not waste parent time. Pydantic-AI's `CancelScope.shield=True` is the small detail that prevents recursive cancellation while children unwind their `finally` blocks.

**When to copy:** Anywhere multiple concurrent tasks are owned by one request scope.

**When it is overkill or risky:** When you want best-effort fan-out and prefer all branches to run to completion (e.g., diagnostic telemetry). LangGraph excludes `GraphBubbleUp` exceptions from cancellation so root-level interrupts are not killed prematurely (`_runner.py:616-634`).

**Evidence:** `_runner.py:616-634`, `_utils.py:201-249`, `_agent_graph.py:1902-1956`, `graph_builder.py:1049-1059`.

### P6. Background/forked-loop via context-provided tools

**Problem:** Letting the parent's LLM spawn and manage concurrent child loops with structured wait/result semantics.

**Where demonstrated:** agent-framework `BackgroundAgentsProvider` injects six tools: `background_agents_start_task`, `_wait_for_first_completion`, `_get_task_results`, `_get_all_tasks`, `_continue_task`, `_clear_completed_task` (`python/packages/core/agent_framework/_harness/_background_agents.py:316-495`). Children run as `asyncio.Task`s with their own `AgentSession` (`_background_agents.py:338,341`).

**Why it works:** Closing the loop on async delegation inside a synchronous tool-call LLM. The parent's LLM decides *when* to wait, and the framework hands back a `task_id` immediately so the LLM can keep producing other tool calls.

**When to copy:** When the parent agent must fan-out work in parallel and gather results later.

**When it is overkill or risky:** When the parent's tool-call loop is already non-blocking; when you do not want LLM round-trips waiting on async work; when you need cancellation (no `background_agents_cancel_task` exists).

**Evidence:** `_harness/_background_agents.py:240-521`.

### P7. Cross-process delegation via metaclass patching

**Problem:** Adding A2A capability to an existing `Agent` class without forcing a parallel `A2AAgent` hierarchy.

**Where demonstrated:** crewai `AgentMeta` metaclass post-construction rewrites `execute_task`, `aexecute_task`, `kickoff`, `kickoff_async` via `MethodType` (`lib/crewai/src/crewai/a2a/wrapper.py:97-237`, `lib/crewai/src/crewai/agent/internal/meta.py:48-71`).

**Why it works:** Avoids class explosion; the wrapped agent retains its original type. Triggered only when `self.a2a is not None`.

**When to copy:** When the delegation surface is opt-in and orthogonal to the model layer.

**When it is overkill or risky:** When Pydantic v2's `model_config`/`__init_subclass__` semantics matter; metaclass mutation of instances is fragile. Pydantic-AI takes the safer `WrapperAgent` route (`pydantic_ai_slim/pydantic_ai/agent/wrapper.py:35-296`).

**Evidence:** `a2a/wrapper.py:97-237`, `agent/internal/meta.py:48-71`.

### P8. Programmatic parent → child resumption via state keys

**Problem:** Resuming a parent's run after a child hit an interruption, without losing decision context.

**Where demonstrated:**
- openai-agents-sdk `_apply_nested_approvals` mirrors parent approval decisions into nested context (`src/agents/agent.py:665-758`); nested `RunResult` is keyed by parent `tool_call` via `agent_tool_state.py:108` and surfaced via `peek_agent_tool_run_result` / `consume_agent_tool_run_result`.
- agent-framework `ToolApprovalMiddleware`-aware mixed batches hide non-approval requests and re-inject on resume.
- langgraph `Command(graph=Command.PARENT)` for parent-graph routing (`libs/langgraph/langgraph/pregel/_retry.py:618-631,753-769`); `interrupt()` from a child bubbles as `GraphBubbleUp` to the root (`libs/langgraph/langgraph/errors.py:50-129`).

**Why it works:** Persistence keyed to a stable identifier (parent tool_call, child run_id, checkpoint namespace) is the universal substrate for "come back later and finish".

**When to copy:** Anytime a child can interrupt or ask for human input and the parent should resume seamlessly.

**When it is overkill or risky:** Single-shot agent patterns that never need a resume.

**Evidence:** `agent.py:665-758`, `_retry.py:618-631`, `errors.py:50-129`.

### P9. Error-as-string for delegation failures (LM-friendly recovery)

**Problem:** Allowing the parent's LLM to recover from a child failure within the same prompt rather than crashing the run.

**Where demonstrated:**
- openai-agents-sdk `failure_error_function=default_tool_error_function` returns a string (`src/agents/agent.py:524`, `src/agents/tool.py:1609-1618`).
- crewai `BaseAgentTool._execute` swallows exceptions into `agent_tool_execution_error` localized string (`lib/crewai/src/crewai/tools/agent_tools/base_agent_tools.py:121-124`).
- pydantic-ai converts `UnexpectedModelBehavior` to `ModelRetry` so the outer model retries the subagent call (`pydantic_ai_slim/pydantic_ai/common_tools/x_search.py:73-77`, `common_tools/image_generation.py:103-107`).
- agent-framework `BackgroundAgentsProvider` returns `"Task failed: {error}"` strings, never raises (`_harness/_background_agents.py:411-417`).

**Why it works:** LLMs reason over text. Returning a string means the parent can see "what went wrong" and adapt. Cost: stacks and causal context are lost.

**When to copy:** When the LLM is in the loop anyway and can decide whether to retry.

**When it is overkill or risky:** When you need a *programmatic* signal (e.g., alerting, retry policy) — opt out by setting `failure_error_function=None` (openai-agents-sdk) or use typed exceptions (`UserInputRequiredException` in agent-framework `exceptions.py:184-209`).

**Evidence:** `agent.py:524`, `base_agent_tools.py:121-124`, `common_tools/x_search.py:73-77`.

### P10. Graph-level fork/join with deadlock-free parent finding

**Problem:** Expressing parallel branches in a step graph and recovering from early failures without deadlock.

**Where demonstrated:** pydantic-ai's `pydantic_graph` (`pydantic_graph/pydantic_graph/graph_builder.py:992-1035`; `pydantic_graph/pydantic_graph/join.py:101-148`; `pydantic_graph/pydantic_graph/parent_forks.py:1-232`); langgraph `Send`/`@task` (similar surface, different mechanism: PUSH tasks scheduled through the runner).

**Why it works:** A deterministic algorithm (parent-fork dominator) chooses the join target; reducers (including `ReduceFirstValue` for early-stop) express fan-in policies. Langgraph achieves the same surface differently — every node runs as a task and `Send` packages parallel copies.

**When to copy:** When the orchestration is naturally branching.

**When it is overkill or risky:** For simple delegation without fanning out. Use P1 instead.

**Evidence:** `pydantic_graph/graph_builder.py:992-1035`, `pydantic_graph/parent_forks.py:1-232`, `_algo.py:938-999`.

## Tradeoffs

For each major design choice:

### Tool-call delegation vs structural embedding

- **Benefit:** Tool-call shapes are LLM-native (the parent just calls a function); reuses guardrails, approvals, span tracing.
- **Cost:** Sequential per LLM step; one child per tool call unless the tool planner fans out (openai-agents-sdk's `execute_tools_and_side_effects` does — `src/agents/run_internal/tool_planning.py:572-595`).
- **Best-fit:** SDK/tooling libraries (openai-agents-sdk, agent-framework's `as_tool`).
- **Alternative:** langgraph's structural embedding — uniform for graph-shaped workloads.
- **Failure mode:** A tool that returns a long string can blow past context; multi-modal or structured content is silently flattened. Agent-framework's `as_tool` collapses structured content into `text` (`python/packages/core/agent_framework/_agents.py:571`).

### Fresh session vs shared session for child runs

- **Benefit (fresh):** No cross-contamination; explicit isolation boundary; cheaper to reason about (crewai's choice leaves this leaky, agent-framework and openai-agents-sdk opt for fresh).
- **Cost (fresh):** Lose conversation continuity; the child must re-explain if it lacks context.
- **Best-fit:** Agent-framework and openai-agents-sdk default to fresh and let the user opt into sharing via `propagate_session=True` or `message_history=...`.
- **Failure mode:** Pydantic-AI users who forget `message_history=...` get a child that has no clue what was discussed; this is documented (`docs/multi-agent-applications.md:79-148`).

### One canonical delegation mechanism vs several

- **Benefit (one):** Langgraph earns its 9 by having *one* delegation primitive — embed a `Pregel`. Every variant (local, remote, checkpointer-on/off, interruptable, streamed) bottoms out in the same dispatch path.
- **Cost (one):** No first-class tool-call surface; users write a wrapper that calls `subgraph.invoke`.
- **Alternative (many):** agent-framework exposes three surfaces (`as_tool`, `BackgroundAgentsProvider`, `WorkflowExecutor`) — more flexibility, more to learn.
- **Failure mode:** CrewAI's "many" has parallel delegation declared but unimplemented (`A2AParallelDelegationStartedEvent` at `events/types/a2a_events.py:631-661` has no emit site) — confusion at the API surface is a real risk.

### Pinned child session vs checked-out child session

- **Benefit (pinned, agent-framework Magentic reset):** Explicit "discard child state" signal — useful for replanning loops.
- **Cost:** Lose service-side conversation IDs; the next chat-client call allocates a new one (`_magentic.py:1348-1367`).
- **Alternative:** langgraph's checkpoint namespace preserves the *address* but lets the *contents* be reset.

### Tool-as-coworker vs LLM-as-router

- **Benefit:** CrewAI exposes delegation as tools the LLM chooses (`DelegateWorkTool`), keeping it a model-driven loop.
- **Cost:** Role resolution is fragile; `sanitize_agent_name` lowercases and strips quotes (`base_agent_tools.py:20-35`).
- **Alternative:** openai-agents-sdk uses `Handoff` for control transfer and `Agent.as_tool` for sub-call — two distinct surfaces for two distinct semantics.

### Cross-process vs in-process delegation

- **Benefit (cross-process):** Protocol-level isolation; the remote system can be anything (CrewAI pairs with non-CrewAI peers via A2A).
- **Cost:** Latency, transport negotiation (gRPC vs HTTP+JSON), auth wiring (HTTPDigest not supported with gRPC — `delegation.py:782-788`), max-turns semantics, and error mapping (`A2AConnectionErrorEvent` vs `A2AAuthenticationFailedEvent`).
- **Best-fit:** When agents are deployed by different teams or need firewall isolation.

### Memory sharing default

- **Benefit (shared):** Cheap continuity; the agent remembers past work.
- **Cost:** Cross-contamination risks, especially with same-id parents and children.
- **Best-fit (shared):** CrewAI in-process because the team model is "collaborators".
- **Best-fit (isolated):** Langgraph and pydantic-ai because they treat state as a graph/runtime, not as crew memory.

## Decision Guide

Pick your delegation strategy based on which constraints dominate:

### When the parent is graph-shaped

- **Use langgraph's subgraph embedding.** Single mechanism, `Send`/`@task` for parallel, `RemoteGraph` for cross-process, `Command.PARENT` for routing. Earns you sibling cancel, durable per-namespace checkpoints, and uniform trace correlation.
- Agent-framework's `WorkflowExecutor` works as an alternative with more ceremony but better workflow-graph ergonomics (checkpoints, broadcast envelopes via `SubWorkflowRequestMessage`).

### When the parent is an LLM calling tools

- **Use `agent.as_tool()` (agent-framework) or `Agent.as_tool()` (openai-agents-sdk).** They give you a tool-call-shaped child with usage accounting and approval surfaces for free.
- For fan-out, do not rely on the LLM looping through multiple tool calls; instead expose a single *background-agents* style tool (agent-framework `BackgroundAgentsProvider`) or use `asyncio.gather` over multiple tool-call invocations (openai-agents-sdk's tool planner fans out automatically).

### When the parent's *output* is the goal and the LLM is over-routed

- **Use `Handoff` (openai-agents-sdk) or `Magentic` (agent-framework).** Both transfer control to a specialist agent without nesting a sub-loop. Openai-agents-sdk collapses multiple handoffs per turn to the first; agent-framework's handoff middleware routes via the workflow graph.
- Tradeoff: shared state means a poorly-scoped handoff can corrupt the parent's history. Use `input_filter` (openai-agents-sdk) or Magentic replan + reset (agent-framework).

### When you need cross-process delegation

- **Use A2A (crewai, pydantic-ai).** Both implement Google A2A with `context_id`/`task_id` lineage on the wire.
- Pydantic-AI's `fasta2a.agent_to_a2a` serves an Agent as a Starlette app — useful if you control both sides.
- CrewAI's metaclass wrapper is opt-in via `Agent.a2a` field; less code, more magic.

### When you need durable / replayable parallelism

- **Use `pydantic_graph` (pydantic-ai) with `DBOSParallelExecutionMode`** — but be aware it restricts to `sequential` and `parallel_ordered_events`.
- **Use langgraph checkpointer + durability levels `exit | async | sync`** — the parent durability propagates to subgraphs (`tests/test_pregel.py:3286-3325`).

### When you need a child that holds a conversation

- **Use `propagate_session=True` (agent-framework)** to share session; otherwise expect the child to lose context.
- **Use `nest_handoff_history=True` (openai-agents-sdk)** to compress prior transcript into a `<CONVERSATION HISTORY>` block before handoff.

## Practical Tips

### Patterns to copy

1. **Tool wrappers for first-class subagents**, with explicit fresh-context construction (`src/agents/agent.py:630-660`).
2. **`BackgroundAgentsProvider`-style context-injected tool kits** for parallel sub-loop work (`_harness/_background_agents.py:240-521`). The parent's LLM picks when to wait and when to inspect results.
3. **Event-bus scope stacks** for trace correlation across async/concurrent boundaries (`events/event_context.py:41-134`).
4. **`SubWorkflowRequestMessage`/`SubWorkflowResponseMessage` envelopes** for typed parent-child boundary contracts (`_workflow_executor.py:57-100`).
5. **Sibling-cancel-on-failure** in runners (`_runner.py:616-634`, `_utils.py:201-229`).
6. **`failure_error_function` for tool-call children** so the parent LLM can recover with a clear string error (`src/agents/tool.py:1609-1618`).
7. **`SubAgent` typed return values.** LangGraph's `v3 streaming` exposes per-subgraph handles (`SubgraphRunStream` at `stream/run_stream.py:587-649`) — a clean navigable abstraction.
8. **`Magentic`-style child-state reset** when the parent asks for replanning (`_magentic.py:1329-1366`). Useful when state from a prior plan is now wrong.

### Patterns to avoid or delay until needed

1. **Metaclass instance mutation for opt-in capabilities.** CrewAI's `AgentMeta` works but introduces fragility; prefer `WrapperAgent` (pydantic-ai) or composition.
2. **Declared-but-unimplemented abstractions.** CrewAI's parallel delegation events are real symbols (`events/types/a2a_events.py:631-661`) with no producers — readers will assume they exist.
3. **Silent multiple-handoff collapse.** Openai-agents-sdk's `run_handoffs[0]` policy is silent; add structured `ToolCallOutputItem` visibility or fail-loud.
4. **One-line `_parallel_delegation` reserved flags** on config models — dead flags mislead readers about capability parity. CrewAI has one (`a2a/config.py:444,541`).
5. **Sharing memory by default** (crewai). Cheap and easy to live with until the first cross-contamination incident.
6. **`as_tool()` returning only text.** Structured content is collapsed; if the child emits multi-modal or function-call output, the parent never sees it.

### Decision rules

- **Pick tool-call (P1) when the parent's flow is LLM-centric and each delegation is short-lived.**
- **Pick structural embedding (P2) when the system already has a graph runtime.**
- **Pick structured workflow (C) when the orchestration is multi-agent and the parent's job is to coordinate.**
- **Pick A2A (D) only when cross-process is a real constraint.** In-process `await child.run()` is cheaper and more debuggable.
- **Pick persistence-first (E) only when the application is conversation-centric and the run-time is delegated elsewhere** (openhands).
- **Pick user-pattern delegation (pydantic-ai) when the framework commits to minimal surface area.** Add a `SubAgent` capability once you have a real use case.

### Caution signs

1. **Shared mutable state between parent and child.** Any `propagate_session=True` or default memory sharing must be tested under writes from both sides.
2. **Synchronous long-running sub-agent in a tool step.** `as_tool()` blocks the parent's LLM iteration; large work needs `BackgroundAgentsProvider` or external `asyncio.gather`.
3. **No cancellation surface for background tasks.** Agent-framework's `BackgroundAgentsProvider` lacks an external cancel; add one or document the lifecycle explicitly.
4. **Silent error localization.** CrewAI masks subagent errors as localized strings (`base_agent_tools.py:121-124`); provide an opt-out.
5. **Multiple-handoff collapse without visibility.** Track dropped handoffs in `new_items` for diagnostic UI.
6. **Inferred subagent naming.** Langgraph uses `lc_agent_name` (`stream/transformers.py:524-536`) inherited from `langchain.agents.create_agent` — a non-subagent graph inherits it and is surfaced as a subagent; ensure your graphs set a proper name or null it.
7. **`propagate_session=True` for tasks that mutate the session.** Two agents writing to the same session can race; test interleaved writes.
8. **A single trace with no group_id.** Openai-agents-sdk shares the parent's trace by default; without `group_id`, cross-run correlation is impossible.
9. **DB-level parent/child without runtime trace.** Openhands has the first but not the second — bridging requires either Laminar config or an OTel span link that is not yet wired.
10. **Re-using the same agent instance for parent and child.** CrewAI does this in-process; the delegated child mutates `_last_messages` and influences later parent turns (`lib/crewai/src/crewai/agent/core.py:680`).

## Notable Absences

1. **No first-class cross-framework A2A pairing for agent-framework.** Handoff and workflow surfaces are in-process only. There is no `wrap_agent_for_a2a` analog to crewai's wrapper or pydantic-ai's `fasta2a`.
2. **No framework-level `parent_run_id` field on tool responses.** Agent-framework (`AgentResponse`), openai-agents-sdk (`RunResult`), pydantic-ai (`AgentRun`) all rely on trace correlation via OTel/langsmith rather than an explicit parent reference. The only explicit linkage across the sources is crewai's `EventRecord` causal graph and openhands's DB column.
3. **No in-tree subagent dispatch in openhands.** The contracts `enable_sub_agents`, `agent_definitions`, and `get_registered_agent_definitions` are precise, but the run loop that turns them into child runs lives in the SDK and is not vendored here. Auditor cannot determine child-run semantics from this source tree alone.
4. **No documented fan-out tool for in-process delegation in crewai.** `A2AParallelDelegationStartedEvent`/`Completed` events exist with scope pairing in `event_context.py:264,315,366` — but the actual production code path that emits them is absent.
5. **No `SubAgent` capability class in pydantic-ai.** The community `subagents-pydantic-ai` package provides `task`, `check_task`, `wait_tasks`, `list_active_tasks`, `soft_cancel_task`, `hard_cancel_task`, `answer_subagent` (per `docs/capabilities.md:1736-1738`), but the framework itself stops at "user-supplied tool calling `await agent.run()`".
6. **No native cancellation API for agent-framework `BackgroundAgentsProvider`.** Workarounds (`asyncio.Task.cancel()` only via event-loop shutdown) are absent from the public surface.
7. **No centralized "running subagents" registry** in any source. LangGraph's `get_subgraphs` walks the static node graph; agent-framework's `_runtime.background_sessions` is per-session state; crewai's `EventRecord.descendants` reconstructs from history; openai-agents-sdk has none; openhands queries the DB.
8. **No documented cross-task interrupt attribution.** When multiple children each emit an `interrupt()`, which one was approved? LangGraph preserves this in the checkpoint row's `tasks[].interrupts`; openai-agents-sdk mirrors via `nested_approvals`; but no source exposes a clean API for "resume-from-multi-child-interrupt" without consulting internal state.
9. **No clear primary analog of LangGraph's `RemoteGraph` outside langgraph.** Pydantic-AI has it via `fasta2a.agent_to_a2a`; crewai wires A2A at the agent level (different model). LangGraph's "embed as node" model is harder to find a peer for outside graph runtimes.

## Per-Source Notes

**agent-framework** — Most combinatorial. `as_tool()` for sync sub-call, `BackgroundAgentsProvider` (`_harness/_background_agents.py:240-521`) for parallel out-of-loop, `WorkflowExecutor` (`_workflows/_workflow_executor.py:103-697`) for sub-workflow wrapping, Magentic replan + reset for state-mode-aware restarts. Tests cover three delegation levels deep (`tests/workflow/test_workflow_kwargs.py:566-786`). Main gaps: no `parent_run_id` on `AgentResponse`; `BackgroundAgentsProvider` cannot cancel externally; sessions default to fresh but `propagate_session=True` couples writes.

**crewai** — Tool-as-coworker in-process, A2A monkey-patched for cross-process. Strongest explicit trace correlation via event-bus scope stack and `EventRecord` causal graph (`state/event_record.py:99-202`). Most distinctive weakness: declared-but-unimplemented parallel delegation surface. Memory shared by default via `crew._memory` fallback (`agent/core.py:609-620`).

**langgraph** — One mechanism (embed `Pregel`), earns its top-half rating. Sibling-cancel, per-namespace checkpointer, `GraphBubbleUp` exception family, `RemoteGraph` for cross-process, v3 streaming handles per subgraph. Borrowed `lc_agent_name` is the main fragility. No native tool-call surface — wrapper nodes are user-supplied.

**openai-agents-sdk** — Two cleanly separated primitives (`Handoff` vs `Agent.as_tool()`) plus `Runner.run` in `asyncio.gather` for external parallel. `ToolOriginType.AGENT_AS_TOOL` and `AgentToolInvocation` distinguish nested runs in telemetry. Gaps: silent multiple-handoff collapse, default `failure_error_function` localization, no separate child trace.

**openhands** — Strong DB-level parent/child contract (`parent_conversation_id` with `include_sub_conversations` filter, cascade delete, inheritance). Runtime delegation is owned by the SDK and not visible. Laminar tracing captures `trace_user_id` + `session_id=conversation_id` but no `parent_trace_id`. UI consumes `sub_conversation_ids` via WebSocket.

**pydantic-ai** — User-shaped delegation via a tool that calls `await child.run()`. No `SubAgent` primitive; community package is the de-facto answer for task/wait/cancel/answer. `pydantic_graph` provides real fork/join with deadlock-free parent finding and reducers (`reduce_list_append`, `ReduceFirstValue`). Built-in subagent tools (`XSearchSubagentTool`, `ImageGenerationSubagentTool`) convert `UnexpectedModelBehavior` to `ModelRetry`. `WrapperAgent` forwards transparently — durable agents (Temporal/Prefect/DBOS) participate in delegation identically. DBOS restricts parallel modes for determinism.

## Open Questions

1. **What is the *intended* canonical way to share `message_history` across delegated pydantic-ai agents?** Docs and tests cover explicit pass-through; no framework helper copies the parent's history into the child. (pydantic-ai `docs/multi-agent-applications.md:79-148`)
2. **How does a `RemoteGraph`-dispatched child surface to its parent on resume?** `Command.PARENT` rewrites one level per raise (`libs/langgraph/langgraph/pregel/_retry.py:618-631`). Cross-process + multi-deep escalation is not documented.
3. **Does agent-framework's `BackgroundAgentsProvider` survive process restart?** The docstring explicitly says in-flight state is non-serializable (`_background_agents.py:288-292`). If children are critical, they must be checkpointable.
4. **What is the operational story for `"ignore this one"` handoff messages in openai-agents-sdk?** The dropped handoffs are surfaced as `ToolCallOutputItem` strings but there is no structured diagnostic event for "model attempted multiple handoffs in one turn".
5. **Can agent-framework's `as_tool()` stream partial output to the parent?** The streaming variant accepts a callback, but the parent's model only sees the final text (`_agents.py:571`). A `background_agents_get_task_stream`-style tool would close the gap.
6. **Is `lc_agent_name` set explicitly for langgraph subgraph wrappers?** LangGraph recommends nulling it on wrapper nodes to avoid the "non-agent appears as agent" pathology (`stream/transformers.py:529-532`). No first-class API.
7. **What is the relationship between openai-agents-sdk's `group_id` and prompt-cache grouping?** A comment claims order matches prompt-cache grouping (`run_grouping.py:18-21`); no test corroborates.
8. **Does pydantic-ai's `gather()` apply AnyIO task groups transitively to user code?** When a user's tool body does `asyncio.gather(child_a, child_b)` directly without using the framework helper, sibling-cancel-on-failure does not apply. Best practice documentation exists but verification is on the user.
9. **How are multi-child interrupts traced back to the requester in crewai A2A?** `reference_task_ids` carries lineage (`lib/crewai/src/crewai/a2a/wrapper.py:1115-1120`); whether it propagates back to the parent after a `REQUEST` closure is not documented.
10. **Is `enable_sub_agents` honoured in the agent-server, or only the app-server plumbing?** Openhands tests prove the value reaches `get_default_tools(..., enable_sub_agents=...)` (`tests/...test_live_status_app_conversation_service.py:1117,1160`), but SDK-side consumption is out of scope.

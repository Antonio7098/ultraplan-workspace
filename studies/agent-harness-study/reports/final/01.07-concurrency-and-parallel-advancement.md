## Executive Summary

Across seven sources, the agent-harness study finds **two distinct concurrency philosophies** answering the same dimension question — "can read-only work run fast without making stateful work unsafe?" — and one outlier that delegates the agent loop entirely.

**Cluster A — BSP / barrier-based frameworks** (agent-framework, langgraph, pydantic-ai's `pydantic_graph`) solve the problem by making parallel writes invisible to siblings within a step. State is mutated only at step boundaries; intra-step fan-out is safe because parallel branches never see each other's intermediate results. Langgraph (`pregel/main.py:460-476`) and agent-framework (`_workflows/_runner.py:160-210`) implement the Pregel model directly; pydantic-ai uses an `anyio.create_task_group` with a `MemoryObjectStream` (`graph_builder.py:643, 840-863`) for its v2 graph runtime.

**Cluster B — single-loop asyncio + bounded primitives** (openai-agents-sdk, crewai, pydantic-ai core, openhands, agent-framework in the non-workflow path) solve the problem with disciplined single-event-loop design: every parallel surface is `asyncio.gather` / `create_task` / `wait`, every shared resource has an `asyncio.Lock` / `anyio.Lock`, and back-pressure is expressed either through `CapacityLimiter` (pydantic-ai), `Semaphore` (openai-agents-sdk sandbox, agent-framework GAIA), or fixed `ThreadPoolExecutor` caps (crewai).

**Cluster C — single-writer-per-key serialization** (temporal) is the production-grade answer for distributed state machines: every workflow execution holds a `PrioritySemaphore(1)` so persistence cannot see concurrent writers. Per-key `SequentialScheduler` (`common/tasks/sequential_scheduler.go:283-307`) gives "1 queue = 1 worker" ordering for replication tasks.

The outlier is **openhands**, which delegates the agent step loop to an external `agent-server` container and only owns app-server concurrency (`asyncio.gather` for read fan-out, Redis lock for the one operation that needs cross-worker safety).

The dimension's lead question gets a uniform "yes" from every source, but the *cost* differs dramatically — from pydantic-ai's `anyio.CapacityLimiter` with OTel spans and a race-tested `max_queued` to temporal's coarse-grained per-workflow semaphore. The most consistent convergence across sources is on **index-aligned ordered results** as the dominant determinism strategy; the most consistent divergence is on **how to bound fan-out** — explicit `Semaphore` vs. shared `CapacityLimiter` vs. fixed pool size vs. no bound.

## Core Thesis

> A consistent set of primitives — index-aligned ordered results, per-resource locks, deterministic per-edge/per-step ordering — appears across all seven sources, but the *control plane* for parallelism splits into three families: BSP barriers (langgraph, agent-framework, pydantic-ai graph v2), bounded primitives with first-class back-pressure (pydantic-ai core, openai-agents-sdk), and per-key single-writer serialization (temporal, agent-framework sub-workflows, pydantic-ai MCP). Each family answers "what must be serialized" differently — channel writes, memory mutation, or persistence calls — and the choice tracks the source's product shape: workflow framework, agent SDK, application server, or durable execution engine.

## Approach Models

The seven sources map onto five recognizable approach models for concurrency. The same product shape tends to pick the same model, and the differences are mostly validated responses to different constraints.

### A. Pregel / BSP workflow frameworks (agent-framework, langgraph)

Both implement the Bulk Synchronous Parallel model: every step has a Plan, Execution, and Update phase. Within a step, all triggered tasks fan out in parallel; channel/state writes are invisible to siblings until the step boundary. agent-framework's `_runner.py:202-210` and `_runner.py:189-203` gather across source executors and across edge runners per superstep; langgraph's `PregelRunner.tick` / `atick` (`pregel/_runner.py:176-358`, `:360-572`) submit tasks into a `BackgroundExecutor` / `AsyncBackgroundExecutor` and wait on `concurrent.futures.wait(... FIRST_COMPLETED)` or `asyncio.wait(... FIRST_COMPLETED)`. Both serialize per-edge message delivery so "send A then B to target X" semantics are preserved. Both apply writes deterministically: agent-framework sorts by edge group, langgraph sorts by `task_path_str(t.path[:3])` (`pregel/_algo.py:253-256`).

The decisive difference is **where the lock sits**. Langgraph's `FuturesDict` uses a `threading.Lock` to guard its completion counter (`pregel/_runner.py:97, 116-132`); agent-framework uses no lock for `_is_running` and justifies it with a CPython GIL comment (`_workflow.py:368-377`). Langgraph provides an `asyncio.Semaphore` from `max_concurrency` config (`_executor.py:135-140`); agent-framework provides no framework-level cap. Langgraph's `BackgroundExecutor` (sync) and `AsyncBackgroundExecutor` (async) share a typed `Submit` protocol with `__cancel_on_exit__` / `__reraise_on_exit__` / `__next_tick__` flags (`_executor.py:27-37, 54-75, 142-169`); agent-framework uses plain `asyncio.gather` everywhere.

### B. Bounded single-loop asyncio SDKs (openai-agents-sdk, pydantic-ai core)

These are single-loop, cooperative, `asyncio`-native designs where parallelism is bounded and observable. They diverge on which primitives they favor:

- **pydantic-ai** prefers `anyio` (backend-agnostic: works under `asyncio` and `trio`) and treats `anyio.CapacityLimiter` as a first-class, observable, pluggable resource. `ConcurrencyLimiter` wraps it, adds race-free `max_queued` enforcement via `anyio.Lock` (`concurrency.py:186-222`), and emits OTel spans on every wait. Its `_utils.gather` (`_utils.py:201-229`) is a `TaskGroup`-shaped wrapper that cancels siblings on failure and unwraps `ExceptionGroup` for multi-failure — explicitly chosen over `asyncio.gather` because the latter "leaves sibling tasks running on first failure."
- **openai-agents-sdk** prefers bare `asyncio.Lock` / `asyncio.Event` and implements a custom slot-filling executor (`_fill_tool_task_slots`, `tool_execution.py:1438-1448`) where the concurrency cap is a max-active counter inside the executor rather than an `asyncio.Semaphore`. It maintains deterministic output by indexing results into a `results_by_tool_run` dict (`tool_execution.py:1385, 1432-1433, 1929-1955`) and re-emitting in call order.

Both expose user-facing knobs — pydantic-ai's `Agent(max_concurrency=...)` and `ConcurrencyLimitedModel` (`agent/__init__.py:542, 1531-1534`, `models/concurrency.py:26-111`); openai-agents-sdk's `ToolExecutionConfig.max_function_tool_concurrency` (`run_config.py:99-118`). Both serialize the model call itself.

### C. Hybrid asyncio + threadpool application SDKs (crewai)

crewai's concurrency story is more scattered than the BSP or bounded-asyncio clusters but is unified by a dual-execution principle: I/O work runs through awaitables; blocking CPU/IO work runs through `concurrent.futures.ThreadPoolExecutor` with bounded sizes — 10 for the event bus, 8 for parallel tool calls, `min(n, 8)` for memory search, `min(n, 4)` for memory recall. The dominant fan-out idiom is `asyncio.gather(*, return_exceptions=True)` (`flow/runtime/__init__.py:1429`, `events/event_bus.py:450`, `experimental/agent_executor.py:1216-1268`), and the dominant ordering idiom is index-aligned `ordered_results` written from `concurrent.futures.as_completed`. Resource protection is layered by concern: `RWLock` for the event-bus handler dict (`utilities/rw_lock.py:12-81`), `RLock` for singleton init (`events/event_bus.py:117-142`, `crewai_core/telemetry.py:75-84`), `asyncio.Lock` for OAuth2 token fetch (`a2a/auth/client_schemes.py:339-372`), `threading.Lock` for RPM counters (`utilities/rpm_controller.py:60-64`), and a pluggable `crewai_core.lock_store` for cross-process safety (Redis `portalocker.RedisLock` or file `portalocker.Lock`).

### D. Single-writer-per-key distributed engines (temporal)

Temporal's model is fundamentally about **isolation, not throughput**. Every state-mutating operation on a workflow passes through `ContextImpl.Lock` (`workflow/context.go:84-89`) backed by `locks.NewPrioritySemaphore(1)` — only one writer per workflow at any moment. A separate `PrioritySemaphore` caps per-shard persistence concurrency (`shard/context_impl.go:117-126, 246-255`). Within the matching / replication subsystems, `SequentialScheduler` enforces "1 queue = 1 worker" per key (`common/tasks/sequential_scheduler.go:283-307`); the `ReaderGroup` starts/stops all readers in lockstep under one `sync.Mutex` (`service/history/queues/reader_group.go:20-155`); the chasm engine is single-writer per execution by design (`chasm/engine.go:16-59`). The cost — no intra-workflow parallelism — is paid deliberately so the persistence layer never needs optimistic-concurrency checks.

Parallel advancement *across* keys is the default: shard acquisition (`controller_impl.go:443-457`), replication stream tasks bucketed into P slots per workflow (`replication/fx.go:186-217`), and matching-service partition fan-out during shutdown (`workflow_handler.go:3092-3135`).

### E. App-server-only / delegated agent loop (openhands)

openhands is the only source that does not own its own step loop. The agent step loop is delegated to a separately-deployed `agent-server` container (`openhands/app_server/sandbox/sandbox_spec_service.py:16`). What the app server *does* own is read fan-out (`asyncio.gather` in every `batch_get_*` service), semaphore-bounded event loads (`event_service_base.py:58-64`), and one cross-worker operation protected by a Redis lock (conversation export, `live_status_app_conversation_service.py:2354-2421`). It cannot answer the dimension question for the agent step itself.

## Pattern Catalog

### 1. Index-aligned ordered results

**Problem**: parallel branches complete out of order, but callers want a result list in input order.

**Where**: agent-framework (`_edge_runner.py:351` — `FanInEdgeRunner._is_ready_to_send` buffers per source), crewai (`agents/crew_agent_executor.py:743-768`, `experimental/agent_executor.py:1715-1746`, `a2a/wrapper.py:280-298`), openai-agents-sdk (`tool_execution.py:1385, 1432-1433, 1929-1970` — `results_by_tool_run[id(tool_run)]`), pydantic-ai (`_agent_graph.py:1958-1961` — `output_parts.extend([tool_parts_by_index[k] for k in sorted(...)])`), openhands (`async_utils.py:80-89` — `wait_all` re-orders by original task list), langgraph (`_algo.py:253-256` — `apply_writes` sorts by `task_path_str`).

**Why it works**: even when `asyncio.gather` is *guaranteed* to return in argument order, the underlying operations (LLM calls, I/O) finish in arbitrary order; collecting into an index-keyed structure and reassembling in input order is the only way to keep `gather`'s positional guarantee meaningful when results have completion-order side effects.

**When to copy**: any time your fan-out returns N results the caller wants in input order and you cannot guarantee that side effects of N branches are commutative.

**When overkill**: if your fan-out is pure (read-only, no side effects) and you only need any/all results, `asyncio.gather` already does this. The pattern is needed when *individual branch results* need to be paired with *individual input items*.

**Evidence**: see "Where" links; the pattern is a near-universal idiom.

### 2. BSP / per-step barrier for state isolation

**Problem**: parallel branches must not see each other's intermediate writes.

**Where**: agent-framework (`_runner.py:99-153` — drain_messages → fan-out → drain events → commit state → checkpoint per superstep), langgraph (`pregel/main.py:460-476` — Plan / Execution / Update per step, `pregel/_loop.py:676-714` — `after_tick` applies writes then runs `put_checkpoint`), pydantic-ai `pydantic_graph` (`graph_builder.py:634-642, 749-818` — `active_tasks` / `active_reducers` mutated only from the runtime; per-task `CancelScope` keys).

**Why it works**: the cost of one barrier per step is paid up front, eliminating an entire class of mid-step races. Every source using BSP-style serialization cites this as the reason — agent-framework's `_runner.py` docstring says *"Multiple messages from a source to the same target are delivered in the order they were sent"* and langgraph's BSP docstring at `pregel/main.py:460-476` is the canonical reference.

**When to copy**: when your domain has natural step boundaries (workflow iterations, agent turns, graph ticks) and you cannot reason about arbitrary intra-step state interactions.

**When overkill**: when your fan-out is shallow (one or two steps) and you can reason about pairwise state isolation per branch.

**Evidence**: `agent-framework/_workflows/_runner.py:160-203`, `langgraph/pregel/main.py:460-476, 2974-2999`, `pydantic_graph/pydantic_graph/graph_builder.py:634-642`.

### 3. Bounded concurrency with first-class limiter

**Problem**: unbounded fan-out exhausts resources or violates provider rate limits.

**Where**: pydantic-ai's `ConcurrencyLimiter` (`pydantic_ai_slim/pydantic_ai/concurrency.py:79-226`) wraps `anyio.CapacityLimiter`, supports `max_queued` back-pressure with race-free `anyio.Lock` enforcement, emits OTel spans on wait; langgraph's `max_concurrency` config feeds an `asyncio.Semaphore` only on the async path (`pregel/_executor.py:135-140`); openai-agents-sdk's `ToolExecutionConfig.max_function_tool_concurrency` (`run_config.py:99-118`) — a max-active counter inside `_FunctionToolBatchExecutor` rather than a semaphore; agent-framework's `asyncio.Semaphore` is used exactly once, in the experimental GAIA benchmark runner (`packages/lab/gaia/agent_framework_lab_gaia/gaia.py:574`).

**Why it works**: bounded concurrency turns an unbounded resource problem into a tunable knob. pydantic-ai's limiter additionally enforces `max_queued` with race-free accounting (`concurrency.py:186-222`, regression test at `tests/test_concurrency.py:127-181` with an `AsyncBarrier` of 5 parties).

**When to copy**: any time a single fan-out can produce N>16 branches and there is a downstream rate-limited resource (LLM API, DB connection pool, third-party webhooks). pydantic-ai's `ConcurrencyLimiter` is the strongest pattern; openai-agents-sdk's slot-filling executor is the most ergonomic for function-tool fan-out.

**When overkill**: when your fan-out width is naturally small (sequential crew of ≤ 3 agents) and downstream resources absorb the load without explicit accounting.

**Evidence**: `pydantic_ai_slim/pydantic_ai/concurrency.py:79-226`, `langgraph/pregel/_executor.py:135-140`, `openai-agents-sdk/src/agents/run_config.py:99-118`, `agent-framework/packages/lab/gaia/agent_framework_lab_gaia/gaia.py:574`.

### 4. Structured-concurrency gather (TaskGroup-shaped)

**Problem**: `asyncio.gather` leaves sibling tasks running on first failure; a hand-rolled cancel-and-drain loop is error-prone.

**Where**: pydantic-ai's `_utils.gather` (`pydantic_ai_slim/pydantic_ai/_utils.py:201-229`) — wraps `anyio.create_task_group` with `asyncio.gather`-shaped single-exception unwrap; `cancel_and_drain` helper (`_utils.py:232-249`) uses `anyio.CancelScope(shield=True)` + `asyncio.gather(return_exceptions=True)` to avoid both losing exceptions and orphaning tasks.

**Why it works**: structured concurrency guarantees that when control returns from the block, all spawned tasks are done. pydantic-ai specifically calls out that `_utils.gather` "is preferred over `asyncio.gather` because `asyncio.gather` leaves sibling tasks running on first failure, while the `TaskGroup` variant cancels them and re-raises a single exception (or `ExceptionGroup` for multi-failure)."

**When to copy**: in any Python 3.11+ codebase where you have fan-out + per-task error handling. This is what pydantic-ai's `_utils.py:201-229` recommends as the canonical wrapper.

**When overkill**: if your fan-out is fire-and-forget (you do not care about partial completion), or if you have already adopted `asyncio.TaskGroup` directly.

**Evidence**: `pydantic_ai_slim/pydantic_ai/_utils.py:201-249`; the OpenAI Agents SDK's `_FunctionToolBatchExecutor` reimplements the same idea manually at `tool_execution.py:1438-1478` and would benefit from the same wrapper.

### 5. Per-execution / per-key isolation via UUID-keyed context

**Problem**: composing the same workflow / agent / state-machine multiple times concurrently must keep their contexts separate.

**Where**: agent-framework's `WorkflowExecutor` (`_workflow_executor.py:299-301, 376-383, 647-697`) — per-execution `ExecutionContext` keyed by UUID; tested with 5 concurrent sub-workflow executions at `test_sub_workflow.py:361-464`. pydantic-ai's `MCPServer` session task is pinned to one `asyncio.Task` with a reference-counted `async with` under `_enter_lock` (`pydantic_ai_slim/pydantic_ai/mcp.py:1280-1342`) so cancel scopes never cross task boundaries. temporal's per-workflow `PrioritySemaphore(1)` (`workflow/context.go:43, 73-93`) and per-shard `PrioritySemaphore` (`shard/context_impl.go:117-126, 246-255`).

**Why it works**: dict-keyed isolation avoids cross-talk between concurrent invocations; agent-framework's `WorkflowExecutor` documentation explicitly states *"Concurrent executions are fully isolated"* (`_workflow_executor.py:197-264`) and warns that the wrapped workflow and executors must be stateless. pydantic-ai's pinned task pattern handles the case where a transport's cancel scopes are bound to a specific task — entering and exiting from `asyncio.gather` children would otherwise crash.

**When to copy**: any time you wrap a stateful primitive (sub-workflow, MCP session, persistent connection) inside a parallel fan-out. The UUID-keyed dict pattern is the simplest isolation surface; the pinned-task pattern is needed when the underlying transport has its own cancellation protocol.

**When overkill**: if the wrapped primitive is naturally stateless (a pure function, a stateless HTTP client) — UUID-keyed dicts add bookkeeping without isolation benefit.

**Evidence**: `agent-framework/_workflows/_workflow_executor.py:197-264, 299-301, 376-383`, `pydantic_ai_slim/pydantic_ai/mcp.py:1217-1342`, `temporal/service/history/workflow/context.go:43, 73-93`.

### 6. Bounded pool + `index-aligned ordered_results` for threadpool fan-out

**Problem**: ThreadPoolExecutor fan-out completes out of order but caller wants input-order results.

**Where**: crewai throughout — parallel native tool calls (`agents/crew_agent_executor.py:695-784`), experimental tool executor (`experimental/agent_executor.py:1703-1746`), A2A agent-card fetch (`a2a/wrapper.py:262-299`), memory parallel_analyze (`memory/encoding_flow.py:222-345`). All use `ThreadPoolExecutor(max_workers=N)` + `concurrent.futures.as_completed` writing back into `ordered_results[idx]`.

**Why it works**: bounded pools cap concurrent load; `as_completed` yields fastest-first but `ordered_results[idx]` preserves input order. crewai documents the invariant at `experimental/agent_executor.py:1221-1223` ("asyncio.gather preserves input order, so zip gives us the exact todo ↔ result mapping") and applies the same pattern to threadpool work.

**When to copy**: any threadpool fan-out where the order of completion should not be observable to the caller. crewai's pattern is the cleanest — fixed pool size, indexed results, never consume in completion order.

**When overkill**: if completion order is acceptable (e.g., logging an event per task where ordering does not matter), `as_completed` is simpler.

**Evidence**: `crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:695-784`, `lib/crewai/src/crewai/experimental/agent_executor.py:1703-1746`.

### 7. Pluggable lock backend for cross-process safety

**Problem**: a single in-process lock does not protect multi-worker or multi-process deployments.

**Where**: crewai's `crewai_core/lock_store.py:79-121` — `portalocker.RedisLock` (distributed) or `portalocker.Lock` (file) behind `set_lock_backend`; openhands's `redis_lock.py:21-53` — `RedisLockUnavailable` distinguishes "Redis broken" from "lock held", plus a periodic refresh coroutine for long-lived operations.

**Why it works**: deployment topology varies (single process, multi-worker, multi-host). A pluggable backend lets the same code path use file locks locally and Redis in production. openhands's `refresh_lock_periodically` task (`redis_lock.py:36-53`) keeps the lock alive during long exports while still allowing TTL-based recovery if the holder dies.

**When to copy**: any operation that must be exclusive across workers — conversation export, file handler writes, LanceDB commits. crewai wraps every LanceDB mutation with `with store_lock(self._lock_name)` (`memory/storage/lancedb_storage.py:100, 113, 227, 305, 330, 351, 421, 604, 630`).

**When overkill**: if your deployment is single-process or you can use the DB's row-locking primitives directly.

**Evidence**: `crewai/lib/crewai-core/src/crewai_core/lock_store.py:79-121`, `openhands/app_server/utils/redis_lock.py:21-53`.

### 8. Slot-filling executor for bounded parallel work

**Problem**: you want to cap concurrent tasks but the work has natural completion-order side effects.

**Where**: openai-agents-sdk's `_FunctionToolBatchExecutor` (`tool_execution.py:1358-1478`) — `_fill_tool_task_slots` + `_drain_pending_tasks` mimics a worker pool over a `set[asyncio.Task]` with `_wait(FIRST_COMPLETED)` + slot refill; failure triggers sibling drain. Its `gather_in_order` (`materialization.py:23-78`) is a custom worker pool with N workers pulling from a shared `next_index`, used for manifest materialization.

**Why it works**: slots + index-keyed completion give bounded concurrency *and* deterministic output ordering, both of which `asyncio.gather` cannot guarantee together. openai-agents-sdk's `gather_in_order` deliberately reinvents what a bounded `asyncio.Queue` worker would do — gaining order preservation but losing the Queue-native backpressure signaling.

**When to copy**: when you need bounded concurrency with deterministic per-branch result collection and you do not want a full thread pool. The slot pattern is a clean asyncio-native primitive.

**When overkill**: if you can use `asyncio.Semaphore` + `asyncio.gather` (function-tool bodies that do not interact with each other), or `ThreadPoolExecutor` + `as_completed` (CPU-bound work).

**Evidence**: `openai-agents-sdk/src/agents/run_internal/tool_execution.py:1358-1478`, `src/agents/sandbox/materialization.py:23-78`.

### 9. Co-routine hand-off via `asyncio.Event` pair

**Problem**: two coroutines need to cooperate without `await`ing each other directly, so the outer task can be cancelled independently.

**Where**: pydantic-ai's agent run loop ↔ capability middleware (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:634-693`, `pydantic_ai_slim/pydantic_ai/agent/__init__.py:1561-1598`) — `_run_ready` / `_run_done` events with `asyncio.wait(..., return_when=FIRST_COMPLETED)`; either side can short-circuit; `cancel_and_drain` cleans up before re-raising. agent-framework uses the same idiom for the iteration-vs-event-stream race (`_runner.py:105-120`).

**Why it works**: `asyncio.Event` is a one-way signal; a pair lets one coroutine tell another "I am ready" / "I am done" without coupling their lifetimes. When the outer task is cancelled, both events are still observable, and the drain helper ensures neither side leaks.

**When to copy**: any time a long-running task is wrapped by a thin coroutine that needs to be cancellable independently of the underlying work.

**When overkill**: if you can just `await` the inner coroutine directly, the hand-off protocol is unnecessary complexity.

**Evidence**: `pydantic_ai_slim/pydantic_ai/_agent_graph.py:634-693`, `pydantic_ai_slim/pydantic_ai/agent/__init__.py:1561-1598`, `agent-framework/_workflows/_runner.py:105-120`.

### 10. Determinism constraint for durable / replayable engines

**Problem**: non-deterministic Python parallelism cannot be replayed by a durable execution engine.

**Where**: pydantic-ai's durable adapters — DBOS exposes only `'sequential'` and `'parallel_ordered_events'` because `'parallel' cannot guarantee deterministic ordering on replay` (`pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:46-48, 295-302`); Temporal override uses `_utils.disable_threads()` to keep the workflow deterministic (`pydantic_ai_slim/pydantic_ai/durable_exec/temporal/_agent.py:286-289`). agent-framework's `durabletask` host translates the graph topology into Azure Durable Functions primitives — `_prepare_all_tasks` collapses per-agent messages so each agent runs at most once per superstep (`packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:663-679`), then `ctx.task_all(all_tasks)` schedules them in one fan-out (`orchestrator.py:796`).

**Why it works**: durable engines replay workflows from history; non-deterministic completion order breaks replay. Constraining the parallel mode to ordered-events or disabling threads forces determinism at the cost of throughput.

**When to copy**: any time you integrate with a durable workflow engine (Temporal, DBOS, Prefect, Azure Durable Functions). The constraint is non-negotiable.

**When overkill**: pure in-process execution where replay is not a concern.

**Evidence**: `pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:46-48`, `pydantic_ai_slim/pydantic_ai/durable_exec/temporal/_agent.py:286-289`, `agent-framework/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:625-797`.

## Key Differences

### How fan-out is bounded

| Source | Mechanism | Default cap |
|--------|-----------|-------------|
| agent-framework | None at framework level; `asyncio.Semaphore` only in GAIA benchmark | Unbounded |
| crewai | Fixed constants per domain | 8 tool calls, 10 bus, `min(n, 8)` search |
| langgraph | `asyncio.Semaphore` from `max_concurrency` config (async only); thread-pool size from `langchain_core.runnables.config.get_executor_for_config` (sync) | Configurable |
| openai-agents-sdk | Max-active counter inside `_FunctionToolBatchExecutor`; `gather_in_order` worker pool for manifests | `None` (unbounded) for function tools; 4 for manifest entries |
| openhands | `asyncio.Semaphore` from `_event_load_concurrency()` env var | 10 |
| pydantic-ai | `anyio.CapacityLimiter` via `ConcurrencyLimiter` with race-free `max_queued` | No default cap; `Agent(max_concurrency=...)` opt-in |
| temporal | `golang.org/x/sync/semaphore.NewWeighted(concurrency)` per shard | Configurable per deployment |

**Why the divergence**: the choice tracks product shape. Langgraph and agent-framework are workflow frameworks where fan-out width is caller-controlled and back-pressure is the caller's problem. OpenAI Agents SDK defaults to unbounded for function tools (predictable small N from one model response) but bounded for sandbox materialization (user-supplied unbounded N). Pydantic-AI treats the limiter as a first-class, observable resource and exposes it through the public `Agent(max_concurrency=...)` API. Temporal's per-shard semaphore is a deployment knob because persistence is the bottleneck.

### How partial failure is collected

| Source | Default idiom | Where structured concurrency wins |
|--------|---------------|-----------------------------------|
| agent-framework | `asyncio.gather` default (first failure cancels rest) + `ExecutorFailedEvent` emitted before raise | `_functional.py` caching |
| crewai | `asyncio.gather(..., return_exceptions=True)` predominant | Per-future `try/except` + `ordered_results[idx]` |
| langgraph | `_should_stop_others` + cancel siblings; `GraphInterrupt` collected across branches | `_panic_or_proceed` |
| openai-agents-sdk | Priority-ordered failure selection (`BaseException > Exception > CancelledError`); cancel + drain siblings | `_select_function_tool_failure` |
| openhands | Default `asyncio.gather` (first-failure-cancels-rest); `wait_all` aggregates via `AsyncException` | None observed |
| pydantic-ai | `_utils.gather` (TaskGroup-based, structured concurrency) + `cancel_and_drain` | Tool-fan-out per-task error → `RetryPromptPart` |
| temporal | Per-goroutine error capture + `errors.Join` aggregation | `dualWriteWrapper` |

**Why the divergence**: BSP / barrier frameworks can afford to cancel siblings because the barrier is the synchronization point — a partial state would never be observed anyway. CrewAI's `return_exceptions=True` idiom is a "best-effort delivery" model suited to the dependency-tiered event bus. Pydantic-AI uses structured concurrency because the agent loop needs to know which tool index failed to construct a `RetryPromptPart`. Temporal isolates per-goroutine because the writer-per-key model means sibling failures are targeting different keys.

### How deterministic ordering is preserved

| Source | Strategy | Notes |
|--------|----------|-------|
| agent-framework | Per-edge sequential await within gather; per-source gather across edges; `FanInEdgeRunner._is_ready_to_send` buffers per source | `_runner.py:160-203` |
| langgraph | `apply_writes` sorts by `task_path_str(t.path[:3])` before application | `_algo.py:253-256` |
| openai-agents-sdk | `results_by_tool_run: dict[int, Any]` keyed by `id(tool_run)`; final list built in call order | `tool_execution.py:1385, 1929-1950` |
| pydantic-ai | `tool_parts_by_index` + final `output_parts.extend([... for k in sorted(...)])`; preallocated `results: list = [None] * len(inputs)` for Bedrock embeddings | `_agent_graph.py:1958-1961`, `embeddings/bedrock.py:616-630` |
| crewai | `ordered_results[idx]` written from `as_completed`; explicit comment on `asyncio.gather` input-order preservation | `experimental/agent_executor.py:1221-1223` |
| openhands | `asyncio.gather` input order + `wait_all` re-orders by original task list | `async_utils.py:80-89` |
| temporal | Per-key `SequentialScheduler` (1 queue = 1 worker); `chain_future` for commit ordering | `common/tasks/sequential_scheduler.go:283-307` |

**Why the convergence**: every source that needs deterministic ordering across parallel branches ends up using the same trick — assign each branch an index, collect into an index-keyed structure, reassemble in input order. The pattern is universal because the alternatives (sorting after completion, or relying on completion order) either require knowing a global sort key or sacrifice the input-order guarantee that `gather` promises.

### How shared mutable state is protected

| Source | In-process primitives | Cross-process / cross-worker |
|--------|-----------------------|-------------------------------|
| agent-framework | `asyncio.Lock`, `_is_running` flag (lock-free), striped asyncio.Lock pool, `WeakKeyDictionary` of locks | `WorkflowExecutor` UUID-keyed isolation |
| crewai | `RWLock` (event bus), `RLock` (singleton init, memory storage), `asyncio.Lock` (OAuth2), `threading.Lock` (RPM), `LockedListProxy/DictProxy` | `portalocker.RedisLock` or `portalocker.Lock` via `crewai_core.lock_store` |
| langgraph | `FuturesDict.lock = threading.Lock`, `asyncio.Semaphore` (async path), per-step futures dict | `chain_future` for asyncio ↔ `concurrent.futures` cross-loop bridge |
| openai-agents-sdk | Bare `asyncio.Lock` (MCP, sandbox memory, unix sandbox output); module-level `ThreadPoolExecutor` (docker sandbox) | None observed (single-loop SDK) |
| openhands | `asyncio.Semaphore` (event loads), `asyncio.Lock` (per-user profile), module-level `ThreadPoolExecutor` | Redis lock for export; DB row locks assumed elsewhere |
| pydantic-ai | `anyio.Lock` (cached_property on event loop), `anyio.CapacityLimiter` (back-pressure), `asyncio.Event` pairs | `AbstractConcurrencyLimiter` ABC with Redis example in docstring |
| temporal | `locks.PrioritySemaphore`, `locks.PriorityMutex`, `locks.IDMutex` (sharded), `sync.Map` of `*sync.Mutex` | Per-shard IO semaphore + per-workflow PrioritySemaphore(1) |

**Why the divergence**: language idiom and product surface. Go's stdlib provides `sync.Mutex` / `sync.RWMutex` and `errgroup` is intentionally avoided in production (Temporal reserves it for tests). Python's `asyncio.Lock` vs `anyio.Lock` choice depends on whether you care about `trio` backend support. The in-process lock is always a starting point; cross-worker safety requires either DB row locks, Redis, file locks, or workflow-id-keyed serialization.

### How agent / model calls serialize

| Source | Model call concurrency | Tool fan-out |
|--------|------------------------|--------------|
| agent-framework | One model call per turn; `_is_running` flag prevents overlapping workflow runs | Function calls in one response run concurrently in `asyncio.gather` |
| crewai | One model call per agent turn; parallel tool calls bounded to 8 in threadpool | Native tool calls in threadpool with `ordered_results` |
| langgraph | Model call inside actor task; `Submit` protocol re-uses `BackgroundExecutor` per step | ToolNode: `executor.map` (sync) / `asyncio.gather` (async) |
| openai-agents-sdk | One model call per turn; tripwire-cancels-siblings for guardrails (Temporal-replay compat flag) | Slot-filling executor for function tools |
| openhands | Delegated to external agent-server container | Not in scope |
| pydantic-ai | One model call per agent turn; `ConcurrencyLimitedModel` caps concurrent model requests | `_call_tools` fan-out with `ParallelExecutionMode` ContextVar |
| temporal | N/A (no LLM model call in server) | N/A |

**Why the convergence**: every SDK that owns its own step loop serializes the model call to one-at-a-time and parallelizes only the side-effect work. The choice tracks provider rate limits and reasoning-state semantics.

## Tradeoffs

### BSP barrier cost vs. intra-step isolation

**agent-framework and langgraph** pay one synchronization barrier per step. Cost: a workflow that needs inter-task pipelining (e.g., one task reading another's intermediate output) cannot pipeline within a step; users work around by emitting intermediate states via `Send` (`langgraph/types.py:664+`, `pregel/_algo.py:440-466`). Benefit: every workflow is automatically race-free inside a step; the BSP docstring at `langgraph/pregel/main.py:460-476` is the canonical rationale. Best-fit: long-running graph workflows with natural step boundaries. Failure mode: latency-sensitive, intra-step communication patterns. Alternative: hand-rolled locks per channel pair (does not scale to N-branch fan-outs).

### Bounded primitives vs. raw gather

**pydantic-ai's `ConcurrencyLimiter`** and **crewai's fixed-pool ThreadPoolExecutors** pay the cost of accounting for every concurrent task. Benefit: deterministic resource usage, OTel observability on wait (`pydantic_ai_slim/pydantic_ai/concurrency.py:202-219`), back-pressure with `max_queued`. Cost: every fan-out needs to be either capped or consciously unbounded. Best-fit: SDKs that fan out to rate-limited external resources (LLM APIs, third-party APIs). Failure mode: under-sized limits become throughput bottlenecks; over-sized limits lose the back-pressure benefit. Alternative: raw `asyncio.gather` (agent-framework default) — simpler but no rate-limiting.

### Single-writer per key vs. intra-workflow throughput

**temporal's `PrioritySemaphore(1)` per workflow** and **agent-framework's `_is_running` flag** pay the cost of serializing all state mutations on one workflow. Benefit: persistence never sees concurrent writers; no optimistic-concurrency checks above the persistence layer (`service/history/workflow/context.go:84-89`). Cost: a single workflow cannot use multiple workers to update state. Best-fit: state machines where ordering is correctness (workflow execution engines, single-tenant multi-step transactions). Failure mode: throughput-sensitive, in-workflow parallelism patterns. Alternative: per-channel locks (more bookkeeping, but allows intra-workflow parallelism).

### Process-local lock vs. distributed lock

**openhands's per-user `_user_profile_locks: defaultdict[str, asyncio.Lock]`** explicitly admits in a comment that this only protects a single worker and hands multi-worker safety to "DB-level row locks or optimistic concurrency tokens" (`openhands/app_server/settings/settings_router.py:313-319`). **crewai's `crewai_core.lock_store`** uses `portalocker.RedisLock` (distributed) or `portalocker.Lock` (file) by default. Cost: distributed locks add latency and a Redis dependency; process-local locks are insufficient for multi-worker SaaS. Best-fit: deployment-driven — single-process gets the simpler primitive, multi-worker needs the pluggable backend.

### Custom `gather_in_order` vs. `asyncio.Queue`

**openai-agents-sdk's `gather_in_order`** (`materialization.py:23-78`) deliberately reinvents what a bounded `asyncio.Queue` worker would do. Cost: no Queue-native backpressure signaling; long-running queue workloads would not scale. Benefit: order preservation + bounded concurrency + first-error cancel in one primitive. Best-fit: short-lived materialization tasks (manifest entries, file copies) where determinism matters more than long-tail throughput. Failure mode: long-running producer/consumer pipelines. Alternative: bounded `asyncio.Queue` + worker coroutines — better for long-running queue workloads.

### Default thread pool vs. dedicated pools

**agent-framework** offloads sync function executors and sync I/O to `asyncio.to_thread`'s default loop-owned executor (`_function_executor.py:137-151`, `_checkpoint.py:323-450`, `_file_access.py:793-995`); it never creates a custom `ThreadPoolExecutor` in core — the only multi-thread pool is the single-thread executor in the hyperlight sandbox (`packages/hyperlight/agent_framework_hyperlight/_execute_code_tool.py:126`). **openhands** similarly uses one module-level `EXECUTOR` for sync-from-async, async-from-sync, Docker pulls, and the local file store (`openhands/app_server/utils/async_utils.py:7`). **crewai** uses per-domain fixed-size pools. Cost: shared default pool — a misbehaving sync function can starve the pool; per-domain pools — more bookkeeping but isolated failures. Best-fit: shared pool for short sync ops; per-domain for long-running or risky sync work (e.g., crewai's `Memory._save_pool = ThreadPoolExecutor(max_workers=1)` to serialize writes).

### TaskGroup vs. asyncio.gather

**pydantic-ai's `_utils.gather`** (`pydantic_ai_slim/pydantic_ai/_utils.py:201-229`) prefers `anyio.create_task_group` because `asyncio.gather` "leaves sibling tasks running on first failure, while the TaskGroup variant cancels them and re-raises a single exception." Cost: structured concurrency has a learning curve and `ExceptionGroup` semantics differ from `asyncio.gather`'s. Benefit: when control returns from the block, all spawned tasks are done — no orphan tasks, no lost exceptions. Best-fit: any fan-out where partial completion needs structured handling. Failure mode: fire-and-forget workloads where you do not care about cleanup. Alternative: `asyncio.TaskGroup` (Python 3.11+) — same semantics with stdlib instead of `anyio`.

### Determinism constraint for durable engines

**pydantic-ai's DBOS adapter** restricts `ParallelExecutionMode` to `'sequential'` and `'parallel_ordered_events'` because `'parallel' cannot guarantee deterministic ordering on replay` (`pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:46-48`). **Temporal override** uses `_utils.disable_threads()` (`durable_exec/temporal/_agent.py:286-289`). Cost: throughput reduction — full `'parallel'` is unavailable under durable execution. Benefit: deterministic replay. Best-fit: any integration with a durable workflow engine. Failure mode: users who want full throughput under replay will be frustrated. Alternative: none — durable determinism is non-negotiable.

## Decision Guide

### Which concurrency model should I use?

1. **You are building a workflow framework** (graph execution, state machines, branching agents).
   - Pick **BSP barrier** (langgraph, agent-framework style). Implement per-step Plan/Execution/Update; barrier at step boundary; per-edge ordering preserved by sequential await; writes sorted by deterministic key before application.
   - Use `_is_running`-style single-instance lock if your graph is single-tenant.
   - Use per-execution UUID-keyed context (`WorkflowExecutor` pattern) if your graph can be composed.

2. **You are building an agent SDK** (LLM tool calls, single-turn model calls).
   - Pick **bounded single-loop asyncio**.
   - Use pydantic-ai's `ConcurrencyLimiter` (`ConcurrencyLimiter(max_running, max_queued)`) for back-pressure with OTel.
   - Use openai-agents-sdk's slot-filling executor pattern if you need per-tool deterministic ordering and concurrent failure arbitration.
   - Use pydantic-ai's three-mode `ParallelExecutionMode` ContextVar + per-tool `sequential=True` to give users escape hatches.

3. **You are building an application server** (HTTP API, read fan-out, batch processing).
   - Pick **asyncio.gather for read fan-out + bounded sync-pool for blocking I/O**.
   - Use `asyncio.Semaphore` for bounded parallel I/O (openhands `event_service_base.py:58-64`).
   - Use Redis lock with periodic refresh for the one operation that needs cross-worker safety (openhands `redis_lock.py:21-53`).
   - Hand backpressure explicitly; do not rely on default executor limits.

4. **You are building a distributed state-machine engine** (workflow persistence, multi-host coordination).
   - Pick **single-writer-per-key serialization** (temporal style).
   - Implement `PrioritySemaphore(1)` per execution key + `PrioritySemaphore(N)` per shard.
   - Use `SequentialScheduler` for per-key ordered queues with N workers.
   - Use a generic `Future[T]` primitive (atomic state machine, `pending → setting → ready`) for one-shot signaling.
   - Add a deadlock detector with `AdaptivePool` if your lock graph is non-trivial.

5. **You are integrating with a durable workflow engine** (Temporal, DBOS, Prefect, Azure Durable Functions).
   - **Restrict `ParallelExecutionMode` to deterministic subset** (pydantic-ai DBOS pattern).
   - **Disable Python threads inside the workflow** (pydantic-ai Temporal pattern).
   - Translate graph topology into the engine's native fan-out primitive (agent-framework `ctx.task_all` pattern).

### When to use `asyncio.gather` vs. `_utils.gather`-style structured concurrency

- **`asyncio.gather`**: fire-and-forget; partial completion is acceptable; you want the first-exception-wins behavior.
- **`TaskGroup` / `anyio.create_task_group`**: you need guaranteed cleanup; partial completion must be observable; you can handle `ExceptionGroup`.
- **`asyncio.gather(..., return_exceptions=True)`**: you need to inspect each result individually and continue regardless of failure (crewai's dominant idiom).
- **`concurrent.futures.as_completed` + `ordered_results[idx]`**: threadpool fan-out where order of completion is not observable to the caller.

### When to use a Semaphore vs. a CapacityLimiter vs. a fixed thread pool

- **`asyncio.Semaphore`**: simple cap; no back-pressure; no observability. Sufficient for I/O-bound bounded fan-out (openhands event loads).
- **`anyio.CapacityLimiter` / `ConcurrencyLimiter`**: cap with back-pressure (`max_queued`), race-free accounting, OTel spans. Sufficient for SDK-level concurrency that crosses rate-limited boundaries (LLM calls).
- **Fixed `ThreadPoolExecutor(max_workers=N)`**: CPU-bound or blocking-I/O work where the pool size is a deployment knob. Sufficient for memory encoding, parallel tool calls, A2A agent-card fetch.
- **No cap (raw `asyncio.gather`)**: predictable small N (a single model response's tool calls); fan-out is naturally bounded by upstream.

### When to use UUID-keyed isolation vs. pinned-task isolation vs. PrioritySemaphore(1)

- **UUID-keyed isolation**: composing the same workflow / stateful primitive concurrently with logically distinct inputs (`WorkflowExecutor`).
- **Pinned-task isolation**: transports whose cancel scopes are bound to a specific task — MCP session, stdio, streamable HTTP (pydantic-ai's `MCPServer` pattern).
- **`PrioritySemaphore(1)` per key**: distributed state-machine engine where persistence must not see concurrent writers (temporal).

## Practical Tips

### Patterns to copy

1. **`ConcurrencyLimiter` with `max_queued` + OTel** (pydantic-ai). The race-test (`tests/test_concurrency.py:127-181` with an `AsyncBarrier` of 5 parties) proves the atomicity; the OTel span emission on wait is the observability hook. Adopt this for any SDK-level concurrency cap.

2. **Index-aligned ordered results** (universal). Whenever you fan out with `asyncio.gather` or `ThreadPoolExecutor.as_completed`, collect into an index-keyed structure and reassemble in input order. Use `results_by_tool_run[id(call)]` or `tool_parts_by_index[i]` patterns.

3. **BSP barrier for graph execution** (langgraph, agent-framework). If your domain has step boundaries, do not try to reason about intra-step state — barrier at the boundary. Sort writes by a deterministic key before application (`task_path_str(t.path[:3])` or equivalent).

4. **Pluggable lock backend** (crewai). `portalocker.RedisLock` (distributed) or `portalocker.Lock` (file) behind a `set_lock_backend` swap point. Lets the same code path use file locks locally and Redis in production.

5. **Per-WorkflowExecutor UUID-keyed context** (agent-framework). When composing stateful primitives concurrently, key by UUID and document the stateless contract.

6. **Three-mode `ParallelExecutionMode` ContextVar + per-tool `sequential=True`** (pydantic-ai). Gives users both batch-level and per-tool escape hatches; ContextVar ensures task-locality.

7. **`SequentialScheduler` per key** (temporal). One queue = one worker enforced by channel ownership. Eliminates per-queue locks.

8. **`asyncio.shield` for tool bodies with explicit `CancelledError` handling** (openai-agents-sdk `tool_execution.py:1858-1893`). Protects the body from external cancellation while allowing sibling-failure cancellation semantics.

9. **`atomic.CompareAndSwapInt32` lifecycle for daemons** (temporal). Every long-running goroutine uses atomic CAS to guarantee single Start/Stop transitions.

10. **`chain_future` for asyncio ↔ concurrent.futures bridging** (langgraph). Keeps commit ordering and `add_done_callback` semantics working across executor boundaries.

### Patterns to avoid or delay until needed

1. **`_is_running` boolean flag with GIL justification** (agent-framework). Works only because Python's GIL serializes assignment; a multi-threaded caller could race it. Use `asyncio.Lock` if you might ever call from a thread.

2. **Default `asyncio.gather` for partial-success workloads** (agent-framework, openhands default). First-failure-cancels-rest is fine for correctness but poor for availability; use `return_exceptions=True` or structured concurrency for any partial-success path.

3. **One module-level `ThreadPoolExecutor`** (openhands `async_utils.py:7`). Shared default pool means a misbehaving sync function can starve the pool; consider per-concern pools for file I/O vs. Docker SDK vs. zip writing.

4. **Process-local `_user_profile_locks`** (openhands `settings_router.py:320`). Admitted in comment to be insufficient for multi-worker. Use Redis-backed lock if you deploy multi-worker.

5. **Unbounded function-tool fan-out by default** (openai-agents-sdk). The default is unbounded per `ToolExecutionConfig.max_function_tool_concurrency`; if your tools have any external resource cost, set an explicit cap.

6. **Docstring claims that contradict implementation** (openhands `_process_pending_messages`). Concurrency claim at `live_status_app_conversation_service.py:1932-1935` vs. sequential `for` loop at `:1976-1995` — keep doc and code in sync, or add a TODO.

7. **Per-call `threading.Thread` spawns in async path** (crewai `task.py:603-610`). Unbounded thread creation on crews with many `async_execution=True` tasks; should be pooled.

8. **`sync.WaitGroup` over `errgroup` in production** (temporal). Where error collection matters, `errgroup.WithContext` would simplify failure handling.

9. **`asyncio.Lock` for in-process locks without typed mutex abstractions** (openai-agents-sdk). Locks are bare instances stored in dataclass fields; no `RLock` and no named-lock context manager.

10. **`time.sleep(0)` as a scheduling primitive** (langgraph `_executor.py:220-222`). Works because of GIL, but it is a thread-scheduling hack rather than an explicit notification primitive. Replace with an explicit scheduling primitive for portability.

### Decision rules

- **Small N (≤ 8), I/O-bound** → `asyncio.gather` (no cap).
- **Small N (≤ 8), CPU-bound or blocking I/O** → `ThreadPoolExecutor(max_workers=min(N, 8))` + `as_completed` + `ordered_results[idx]`.
- **Large N (≥ 16), I/O-bound, rate-limited** → `ConcurrencyLimiter(max_running, max_queued)` with OTel spans.
- **Large N, one-shot** → `asyncio.Semaphore(N)` + `asyncio.gather`.
- **Graph workflow** → BSP barrier with deterministic write-sort.
- **Distributed state-machine** → per-key `PrioritySemaphore(1)` + per-shard `PrioritySemaphore(N)`.
- **Durable engine integration** → restrict parallelism to deterministic subset; disable threads inside the workflow.

### Caution signs

1. **`asyncio.gather` without `return_exceptions=True` in a partial-success path.** One failing parallel I/O will cancel the rest.
2. **`asyncio.Lock` for state that crosses event-loop boundaries** (multiple processes, multiple threads). Process-local; needs Redis or DB lock for cross-worker.
3. **Fan-out width grows with user input without an explicit cap.** Reads may overwhelm the event loop; writes may overwhelm persistence.
4. **`asyncio.create_task` background work with no error observer.** Unobserved task exceptions are silently dropped.
5. **Bare `_is_running` boolean flag without `asyncio.Lock`.** Safe under GIL but a footgun for multi-thread callers.
6. **`time.sleep(0)` or other GIL-dependent scheduling tricks** for portability-critical code.
7. **Docstrings claiming concurrency but implementation is serial** (openhands `_process_pending_messages`). Either fix the code or fix the docstring.
8. **Shared module-level `ThreadPoolExecutor` with no per-concern isolation.** One misbehaving sync function starves the pool.
9. **`asyncio.gather` used where `TaskGroup` / structured concurrency is needed.** Orphan tasks on first failure.
10. **Two independent `ConcurrencyLimiter` instances per request** (pydantic-ai's per-agent + per-model cap). They compose to `min(agent, model)` but a user setting both should understand the effective cap.
11. **Lock ordering rules enforced only by comments** (temporal `shard/context_impl.go:122`). Move to type-level enforcement to prevent regressions.
12. **Skipped multi-process stress tests** (crewai `test_concurrent_storage.py:11-13`). The lock backend is in place but unverified under fork.

## Anti-Patterns / Caution Signs

- **Default `asyncio.gather` without `return_exceptions=True`** (agent-framework runner, openhands batch services). First failure cancels the rest and propagates a single error. This is fine for correctness but bad for availability — one bad row in a batch degrades the whole page.
- **`asyncio.gather` in a tight loop without a bound** (openai-agents-sdk function-tool fan-out default). Unbounded fan-out under model-emitted tool calls can exhaust provider rate limits.
- **Per-tool/per-task `await` instead of `asyncio.create_task`** in a parallel batch (crewai legacy `execute_async` spawns a fresh `threading.Thread` per task). Unbounded thread creation.
- **Docstring contradicts implementation** (openhands `_process_pending_messages`). Concurrency claim at `live_status_app_conversation_service.py:1932-1935` vs. sequential `for` loop at `:1976-1995`.
- **`asyncio.Lock` used as a process-global** (openhands `_user_profile_locks`). Comment explicitly admits this is process-local and hands multi-worker safety to "DB-level row locks or optimistic concurrency tokens" with no implementation.
- **Singleton polling task without restart-on-failure** (openhands `remote_sandbox_service.py:924-932`). If the task dies, the process silently loses cross-process state updates.
- **Sync `kickoff_async` path that wraps `kickoff` in `asyncio.to_thread`** (crewai `crew.py:1130-1162`). Dual API surface (`kickoff_async` / `akickoff`) means users can pick the wrong entry point.
- **Lock-free `_is_running` flag** (agent-framework `_workflow.py:368-377`). Documented and tested, but relies on GIL — a multi-thread caller could in principle race it.
- **Custom `gather_in_order` reinvents bounded executor** (openai-agents-sdk `materialization.py:23-78`). Loses `asyncio.Queue`-native backpressure signaling.
- **Channel-write sorting by GIL-dependent `time.sleep(0)`** (langgraph `_executor.py:220-222`). Thread-scheduling hack.
- **Skipped multi-process test for cross-worker safety** (crewai `tests/memory/test_concurrent_storage.py:11-13` — "Multiprocessing tests incompatible with xdist"). The lock backend is in place but unverified under fork.
- **`asyncio.gather(..., return_exceptions=True)` without per-element inspection** — exceptions become positional results that consumers must remember to check; silent failures follow.
- **Concurrent `Workflow.run` calls on the same instance raise `RuntimeError`** — this is a feature for agent-framework, but it means a caller who needs parallel composition must use `WorkflowExecutor` explicitly.

## Notable Absences

- **No `asyncio.TaskGroup` adoption** (agent-framework, openai-agents-sdk, openhands). All three use `asyncio.gather` everywhere. Pydantic-AI's `_utils.gather` is the structured-concurrency exception, and langgraph's bespoke `FuturesDict` + `BackgroundExecutor` serves a similar role.
- **No `concurrent.futures.ProcessPoolExecutor` or `multiprocessing` use anywhere in the Python agent SDKs.** CPU-bound parallelism is out of scope for every Python source studied (agent-framework, crewai, langgraph, openai-agents-sdk, openhands, pydantic-ai). Only temporal (Go) uses OS-level goroutines, which are cooperative.
- **No `asyncio.Queue` with bounded `maxsize` used as back-pressure** in any Python source. Pydantic-ai's `MemoryObjectStream` in `ProcessEventStream` (`pydantic_ai_slim/pydantic_ai/capabilities/process_event_stream.py:83`) and the graph runtime (`pydantic_graph/graph_builder.py:643`) are unbounded.
- **No framework-level `asyncio.Semaphore` for tool fan-out** in any Python source except openai-agents-sdk's sandbox materialization (default cap 4) and pydantic-ai's Bedrock embedder. Agent-framework's `asyncio.Semaphore` is used exactly once, in the experimental GAIA benchmark runner.
- **No `asyncio.CancelledError` cancellation tokens in user-facing parallel APIs.** Background-agent tasks in agent-framework use `asyncio.create_task` without explicit cancellation tokens (`_background_agents.py:341, 466`). Pydantic-AI's `cancel_and_drain` is a helper, not a public API.
- **No documented "cancel all parallel work" knob on the public `Agent` API** in pydantic-ai; cancellation is driven by the caller's `asyncio.CancelledError` propagation through the cooperative hand-off.
- **No `errgroup` use in production** (temporal). All production code uses raw `sync.WaitGroup` + `sync.Map`; `errgroup` is reserved for tests.
- **No global concurrency budget that composes agent, model, and toolset caps** (pydantic-ai). Two independent `ConcurrencyLimiter` instances per request.
- **No observability for in-flight tool tasks.** Openai-agents-sdk's `_FunctionToolBatchExecutor` has a `pending_tasks` set but it is not exposed in any span attribute. Pydantic-ai's `ConcurrencyLimiter` does emit OTel spans — this is the contrast.
- **No documented concurrency contract for the agent-server side** (openhands). The app server assumes the agent server is well-behaved.
- **No global shutdown handler that cancels in-flight `asyncio.create_task` background work** (openhands). Relies on FastAPI lifespan.
- **No temp-file name in `LocalFileStore.write` that uses the asyncio Task id** (openhands `file_store/local.py:33-43`). Multiple coroutines on the same OS thread will collide on the temp filename.

## Open Questions

1. **What is the exact behavior when a sub-task scheduled from inside a parallel node (`langgraph/_runner.py:759-781`) completes synchronously before the parent task has started streaming?** Langgraph's `chain_future` ordering comment claims it is preserved, but no direct test was found in the public test suite.
2. **Is `__next_tick__` strictly necessary in the async path where `run_coroutine_threadsafe` already schedules onto the event loop, or is it dead-weight?** Langgraph's `noop in async (always True)` comment (`_executor.py:149`) suggests the latter but does not document why it remains.
3. **What happens to in-flight handlers if the user process is killed mid-`kickoff`?** crewai's `atexit.register(crewai_event_bus.shutdown)` exists, but no test verifies the daemon-thread event loop survives `os.kill`.
4. **Does parallel native tool execution respect `tool.result_as_answer` ordering in the chat message history?** Code path at `experimental/agent_executor.py:1724-1753` writes ordered_results back into the assistant message sequentially; safety under `max_usage_count` exhaustion mid-batch is not explicitly tested.
5. **What is the inter-agent contract on `current_flow_method_name` ContextVar** when a Flow method calls `Agent.kickoff()` (sync) inside another Flow method? Crewai's `asyncio.to_thread(ctx.run, ...)` is supposed to propagate, but no explicit test that survives run_in_executor nesting was found.
6. **Is the per-crew `_save_pool` shared across crews in a multi-crew app?** crewai provides no evidence that it is auto-shared; each `Memory(...)` instance creates its own pool with `max_workers=1`.
7. **What is the exact semantics of `agent_runner_helpers.should_cancel_parallel_model_task_on_input_guardrail_trip` under non-Temporal workflow engines?** Openai-agents-sdk's helper returns `True` if Temporal is not present; users under other workflow engines could get surprising behavior.
8. **Does the `_DOCKER_EXECUTOR` module-level `ThreadPoolExecutor` (openai-agents-sdk) have a documented shutdown hook?** No evidence of `atexit` / `loop.shutdown_default_executor` integration.
9. **What is the worst-case task backlog under a stalled ioSemaphore?** Temporal's shard IO semaphore has metrics (`shard/context_impl.go:1535, 1538, 1540`) but no documented SLO.
10. **Is there a documented invariant about goroutine ownership inside chasm tasks?** Temporal's chasm framework serializes per execution key, but the `invoker_tasks.go` code spawns goroutines inside a task and claims no shared state will be mutated; this is implicit and not enforced by types.
11. **How does `kickoff_async` progress reporting interact with `asyncio.to_thread`?** crewai's `signal_end(ctx.state, is_async=True)` is called from inside the thread; the `state.async_queue.put_nowait` is invoked via `loop.call_soon_threadsafe` — fragile pattern; no clear evidence of tests for cross-thread stream ordering.
12. **What happens to the agent-server when the OpenHands app-server's singleton polling task dies?** Openhands has no restart-on-failure path; the process silently loses cross-process state updates.

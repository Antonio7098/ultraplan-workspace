## Executive Summary

All seven agent frameworks encode "done" as a **multi-layered signal** rather than a single boolean: a per-turn chat finish reason, a per-response/turn lifecycle marker, a per-run terminal status, and (sometimes) a workflow-level state machine. The strongest implementations (pydantic-ai, letta, openai-agents-sdk, langgraph) tie these layers together with **typed enums** and **explicit success terminals** that callers can pattern-match on. The weakest (openhands, crewai) rely on **string equality**, **parser heuristics**, or **pinned-but-unread external runtimes**, which makes "done" brittle to upstream changes.

**Key findings:**

- **Done ≠ correct is universal.** Every framework except pydantic-ai's output-tool path can terminate with empty or malformed output. pydantic-ai is the only one that hard-fails empty outputs under `finish_reason='length'` or `'content_filter'` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1102-1130`).
- **Finish-reason normalization is rare.** Only pydantic-ai maps every provider to a single `Literal['stop','length','content_filter','tool_call','error']` (`pydantic_ai_slim/pydantic_ai/messages.py:114-121`). Letta collapses all non-length reasons to `end_turn` (`letta/agents/letta_agent_v3.py:1989-1998`). agent-framework, openai-agents-sdk, and crewai preserve the raw string but rarely surface it.
- **Pending tool calls are universally handled**, but via three different mechanisms: blocking (agent-framework's `PendingRequests`, letta's `tool_rule`), interruption as a first-class state (openai-agents-sdk's `NextStepInterruption`, pydantic-ai's `state='interrupted'`), or graceful skip (pydantic-ai's `EndStrategy`, agent-framework's `ToolApprovalAgent`).
- **Usage/cost is well-finalized in six sources** (agent-framework, crewai, letta, openai-agents-sdk, pydantic-ai, openhands via persistent storage) but **never at the run level** — most expose per-response or per-step usage and expect callers to aggregate. pydantic-ai is the only one with three distinct guard points (`usage.py:379-419`).
- **No source has a "completed-with-warnings" status.** Warnings live in metadata, callback results, or event streams; the terminal status enum never has a warning axis.
- **Streaming completion is the most fragmented area.** Each framework invents its own terminal marker (sentinel event, `[DONE]`, `is_complete` flag, `state` literal, `run_loop_exception`).

## Core Thesis

> **Completion and finalization in agent frameworks is a layered status problem, not a correctness problem.** The strongest models expose a **typed, narrow success terminal** (a single enum value) and a **typed exception hierarchy** for failures, with usage/cost captured *as a side effect of finalization*, not as a separate concern. The weakest models defer completion to a parser outcome (ReAct), to a string match (`event.value == 'finished'`), or to an external pinned wheel, which leaves "done" answerable only by inference. Across all sources, the question "does done mean correct, merely stopped, or just no more tool calls?" has three different answers — and the right answer depends on which layer you're asking about.

## Approach Models

The seven sources cluster into **five distinct approach models** for completion semantics. Each is a coherent design choice shaped by the framework's primary abstraction (graph, agent loop, workflow, runtime), not a quality ranking.

### Cluster A: Typed state machine with single success terminal

**Sources:** pydantic-ai (8.5), openai-agents-sdk (8), agent-framework (7, in part).

These frameworks encode "done" as a **single, narrow typed value** rather than a boolean. pydantic-ai uses `End(FinalResult[OutputDataT])` as the only success exit (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1359-1379`); openai-agents-sdk uses the `NextStepFinalOutput` variant of a four-value discriminated union (`src/agents/run_internal/run_steps.py:155-174`); agent-framework uses `RunStatus.Idle` as the canonical completion state for workflows (`dotnet/src/Microsoft.Agents.AI.Workflows/RunStatus.cs:8-34`). All three then map the typed value to a typed exception hierarchy for failure paths (`AgentRunError`, `AgentsException`, workflow `WorkflowErrorEvent`). The result is that callers can `match`/`switch`/`isinstance` on a closed set of outcomes.

### Cluster B: Two-level status with bidirectional mapping

**Sources:** letta (8), agent-framework (7, in part for chat-vs-workflow layers).

Letta pairs a granular `StopReasonType` (12 values: `end_turn`, `error`, `max_steps`, `max_tokens_exceeded`, `cancelled`, `tool_rule`, `requires_approval`, `insufficient_credits`, etc.) with a coarse `RunStatus` (5 values: `created`, `running`, `completed`, `failed`, `cancelled`) via the `StopReasonType.run_status` property (`letta/schemas/letta_stop_reason.py:24-49`). Streams emit `StopReasonType`; the database stores `RunStatus`; the Redis stream manager maps back at completion (`letta/server/rest_api/redis_stream_manager.py:430-445`). agent-framework achieves a similar layering with `ChatFinishReason` (per-turn) → `AgentResponse.ContinuationToken` (per-agent) → `RunStatus` (per-workflow), but without a single mapping table — each layer has its own enum.

### Cluster C: Dataflow/BSP stop condition

**Source:** langgraph (8).

LangGraph's `Pregel` runtime treats completion as a **stop condition on the task set**: when `prepare_next_tasks` returns an empty dict, the loop exits (`sources/langgraph/libs/langgraph/langgraph/pregel/_loop.py:646-648`). The status literal (`Literal["input", "pending", "done", "draining", "interrupt_before", "interrupt_after", "out_of_steps"]`, `_loop.py:249-257`) is mapped to user-visible exceptions (`GraphRecursionError` for `out_of_steps`, `GraphDrained` for `draining`) by the driver (`sources/langgraph/libs/langgraph/langgraph/pregel/main.py:3017-3031`). This is the only source where the **graph structure is the completion primitive**: there is no "done" node; `END` is a sentinel channel consumed by `entrypoint.final[Return, Save]` (`sources/langgraph/libs/langgraph/langgraph/func/__init__.py:584-589`).

### Cluster D: Parser-driven action vs finish

**Source:** crewai (6).

CrewAI's ReAct loop treats completion as a **parser outcome**: `parser.parse` recognizes `Action:` vs `Final Answer:` lines, and the agent loop guards on `isinstance(formatted_answer, AgentFinish)` (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:330-468`). Tool calls decorated with `result_as_answer=True` short-circuit the loop in every site (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:1097-1107`, `lite_agent.py:914-929`). The experimental executor adds a `finalize()` with re-entrancy lock (`sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2254-2326`), but the underlying primitive is still "any non-actionable output becomes `AgentFinish`". The `format_answer` permissive fallback catches all parse exceptions and synthesizes `AgentFinish` regardless of whether the model actually said done (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:370-386`).

### Cluster E: External runtime + string match

**Source:** openhands (5).

OpenHands's `openhands-ai` repo does not contain the agent loop — the loop is in pinned external wheels (`openhands-sdk==1.29.0`, `openhands-agent-server==1.29.0`, `openhands-tools==1.29.0` per `pyproject.toml:60-62`). The repo only sees the **receiving side** (webhooks, callbacks, storage). Five integration callbacks (`slack_v1_callback_processor.py:46-57`, `jira_v1_callback_processor.py:52-62`, `github_v1_callback_processor.py:46-59`, plus Jira DC / GitLab / Bitbucket variants) hard-code `event.value == 'finished'` against a lowercase string literal. This is the most brittle approach: a SDK rename to `'completed'` or `'idle'` silently breaks integration summaries without any test catching it.

## Pattern Catalog

The patterns below appear in **multiple sources** or are **unusually clearly demonstrated** in a single source. Each is presented as a learnable pattern with the contexts in which it works.

### Pattern 1: Narrow success terminal + typed exception hierarchy

**Problem:** How do you let callers pattern-match on completion without exhaustive boolean checks?

**Demonstrated by:** pydantic-ai, openai-agents-sdk, agent-framework (workflow layer), letta (via `StopReasonType`).

- pydantic-ai: `End(FinalResult[OutputDataT])` is the single success exit (`_agent_graph.py:1359-1379`); failures are `AgentRunError` subclasses (`UnexpectedModelBehavior`, `ContentFilterError`, `IncompleteToolCall`, `UsageLimitExceeded`, `ModelAPIError`, `ModelHTTPError`, `UserError`, `FallbackExceptionGroup`, `exceptions.py:181-309`).
- openai-agents-sdk: `NextStepFinalOutput`, `NextStepHandoff`, `NextStepRunAgain`, `NextStepInterruption` (`run_steps.py:155-174`); failures are `MaxTurnsExceeded`, `InputGuardrailTripwireTriggered`, `OutputGuardrailTripwireTriggered`, `ModelRefusalError`, `ModelBehaviorError` (`exceptions.py:46-174`).
- letta: `StopReasonType.run_status` property maps 12 reasons to 5 statuses (`letta_stop_reason.py:24-49`).

**Why it works:** Callers can `match`/`isinstance` on a closed set instead of inspecting flags. The exception hierarchy preserves typed context (`ModelBehaviorError` carries the raw message; `IncompleteToolCall` carries the truncated tool call) that would otherwise be lost in a generic `RuntimeError`.

**When to copy:** Any framework where "done" is one of multiple outcomes (handoff, pause, completion, error).

**When it is overkill:** Single-shot, single-outcome tools where the answer is always the same shape.

**Caveat:** All three pay the cost of needing to update the type when a new outcome is added. openai-agents-sdk compensates with `RunErrorHandlers` (`src/agents/run_error_handlers.py:50-55`), letting users extend the failure surface without modifying the SDK.

### Pattern 2: Finish-reason normalization at the adapter boundary

**Problem:** Provider APIs return heterogeneous finish reasons (`"stop"`, `"length"`, `"tool_calls"`, `"content_filter"`, `"safety"`, `"max_tokens"`, `"end_turn"`, `"SAFETY"`); agent loops should reason over a stable vocabulary.

**Demonstrated by:** pydantic-ai (best), letta (partial), agent-framework (provider-native preserved).

- pydantic-ai: every provider maps to `Literal['stop','length','content_filter','tool_call','error']` (`messages.py:114-121`) via per-provider maps (e.g., `models/anthropic.py:3969-4014`, `models/bedrock.py:223`, `models/cohere.py:86, 263-273`, `models/gemini.py:289-299, 878-880`, `models/groq.py:123, 240, 395-407`, `models/xai.py:161-175`). OTel instrumentation reads from the normalized value (`_instrumentation.py:283-284`).
- letta: special-cases only `"length"` → `max_tokens_exceeded` (→ `failed`); everything else collapses to `end_turn` (`letta_agent_v3.py:1989-1998`).
- agent-framework: preserves the raw `ChatFinishReason` and stamps it on `AgentResponse.FinishReason` (`AgentResponse.cs:208`) without domain-level mapping.

**Why it works:** A normalized vocabulary lets the agent loop make a single decision (retry? fail? surface as warning?) regardless of provider.

**When to copy:** Any framework with multi-provider support. The OTel alignment in pydantic-ai is a particularly clean choice because it inherits a stable external standard.

**When it is overkill:** Single-provider systems where the raw value is already canonical.

**Caveat:** Normalization risks hiding provider-specific nuances (e.g., OpenAI's `incomplete_details.reason` is richer than the `finish_reason` string). pydantic-ai addresses this by keeping provider details in `provider_details` (`messages.py:1118-1128`).

### Pattern 3: Pending tool calls as a first-class state, not a crash

**Problem:** A model may produce tool calls alongside a final message, or may require human approval before executing. Treating these as crashes loses the work; treating them as success loses the unfinished tool calls.

**Demonstrated by:** openai-agents-sdk (`NextStepInterruption`), pydantic-ai (`EndStrategy` + `state='interrupted'`), letta (`StopReasonType.requires_approval`, `tool_rule`), agent-framework (`PendingRequests`, `ToolApprovalAgent`).

- openai-agents-sdk: pending tool approvals return `NextStepInterruption`; `RunResult.interruptions: list[ToolApprovalItem]` (`src/agents/result.py:367-368`) carries the approval request; `RunResult.to_state()` packages this into a resumable `RunState` (`src/agents/result.py:393-438`).
- pydantic-ai: `EndStrategy = Literal['early','graceful','exhaustive']` (`_agent_graph.py:73`) decides what to do with parallel function tools after a final result. `'early'` synthesizes "Tool not executed - a final result was already processed." returns (`_agent_graph.py:1666-1680`).
- letta: `RequiredBeforeExitToolRule` blocks end-of-loop unless called (`tool_rule_solver.py:198-223`); `requires_approval` is its own `StopReasonType` mapped to `RunStatus.completed` (resumable on next call).
- agent-framework: `PendingRequests` blocks `RunStatus.Idle` (`StreamingRunEventStream.cs:117-119`); `ToolApprovalAgent` breaks out to caller only when human action is required (`ToolApprovalAgent.cs:107-136`).

**Why it works:** Pending tool calls are not errors — they are legitimate intermediate states. Exposing them as a first-class result lets callers resume rather than restart.

**When to copy:** Any framework with human-in-the-loop, multi-step tool execution, or parallel tool calls.

**When it is overkill:** Read-only or single-tool agents where every turn produces an immediate answer.

**Caveat:** pydantic-ai is explicit about scope (`docs/output.md:956-957, 1010-1011`): the framework records interrupted tool calls verbatim and does not synthesize returns or filter them. This is a deliberate "don't make decisions for the user" choice that pushes cleanup work to consumers.

### Pattern 4: Usage/cost captured as a side effect of finalization

**Problem:** Token counts and cost need to be aggregated across the run, attached to the result, and persist beyond the in-memory session.

**Demonstrated by:** all sources except langgraph (which delegates to LangChain callbacks).

- agent-framework: `AgentResponse.Usage` per response; `MessageMerger.MergeUsage` aggregates across workflow responses (`MessageMerger.cs:271-304`); per-provider translators (`UsageDetails.ToCompletionUsage`, `ToResponseUsage`).
- crewai: `UsageMetrics` with `total_tokens/prompt_tokens/cached_prompt_tokens/completion_tokens/reasoning_tokens/cache_creation_tokens/successful_requests` (`usage_metrics.py:32-77`); `UsageMetrics.from_provider_dict` normalizes OpenAI/Gemini/Anthropic/Bedrock aliases (`usage_metrics.py:79-124`); `calculate_usage_metrics` aggregates across agents (`crew.py:2088-2112`).
- letta: per-step `prompt/completion/cached_input/cache_write/reasoning` (`step_manager.py:416-476`); per-run via `RunManager.get_run_usage` (`run_manager.py:512-540`).
- openai-agents-sdk: `Usage` dataclass with per-request entries preserved (`usage.py:125-136, 198-215`); attached to spans via `attach_usage_to_span`.
- openhands: persistent `accumulated_cost` and token breakdowns in `StoredConversationMetadata` (`sql_app_conversation_info_service.py:89-99`); updates via `process_stats_event` (delta) or `on_conversation_update` (full snapshot).
- pydantic-ai: `RunUsage` aggregated eagerly (`_agent_graph.py:1015-1025`); `cost()` computed lazily with provider-url fallback (`messages.py:2231-2253`); OTel `operation.cost` histogram.

**Why it works:** Usage is intrinsically tied to "the run finished" because (a) the final state is what callers bill on, and (b) cost-pricing depends on knowing whether the run completed or was cancelled.

**When to copy:** Always — but choose between eager (pydantic-ai `RunUsage`) and lazy (pydantic-ai `cost()`) aggregation based on whether downstream callers need pre-computed costs or can compute them on demand.

**When it is overkill:** Test or demo frameworks where cost is irrelevant.

**Caveat:** No source (except openhands' persistent storage) survives a sandbox/process loss for accumulated cost. pydantic-ai explicitly documents that cancelled-stream usage is partial and unsafe for cost-critical accounting (`docs/output.md:1013-1015`).

### Pattern 5: Streaming completion with explicit sentinel

**Problem:** Streaming consumers need to know when the stream is truly done — without conflating "stream ended early due to error" with "run completed normally".

**Demonstrated by:** letta (`[DONE]` marker + synthesized terminal), openai-agents-sdk (`QueueCompleteSentinel` + `is_complete`), agent-framework (`RequestHaltEvent` internal sentinel), pydantic-ai (`FinalResultEvent` + `is_complete`), langgraph (no synthetic marker; relies on exception types).

- letta: `get_finish_chunks_for_stream` emits `[stop_reason, usage_statistics, "[DONE]"]` (`base_agent.py:188-195`); the Redis stream manager **synthesizes** `[DONE]` or a `stream_incomplete` error if the stream ends without one (`redis_stream_manager.py:283-315`).
- openai-agents-sdk: `QueueCompleteSentinel` is the in-process end-of-stream signal (`run_steps.py:52-56`); `RunResultStreaming.is_complete: bool` is set on every terminal branch (`result.py:470`); `run_loop_exception` surfaces silent failures (`result.py:623-646`).
- agent-framework: `RequestHaltEvent` is filtered by `AsyncRunHandle.TakeEventStreamAsync` (`AsyncRunHandle.cs:80-87`) and never exposed to consumers; the consumer iterator uses `break` on `Idle` / `Ended` states.
- pydantic-ai: `FinalResultEvent` is the discriminated `event_kind='final_result'` event (`messages.py:2786-2797`); `StreamedRunResult.is_complete` is set by `_marked_completed` or `cancel()` (`result.py:426-434, 750-772`); `state='interrupted'` on the response indicates cancellation.

**Why it works:** A typed sentinel prevents consumers from guessing "did the stream end because the run finished, or because the connection dropped?"

**When to copy:** Any framework exposing streaming runs.

**When it is overkill:** Batch-only APIs.

**Caveat:** letta's synthesis is the most defensive: if the stream ends without a marker *and* without an error, it writes a `stream_incomplete` error so the consumer never silently loses work (`redis_stream_manager.py:283-315`). Most other frameworks rely on the producer to send a marker.

### Pattern 6: Epoch-tagged completion signals

**Problem:** A consumer iterating a stream may receive stale completion signals from a previous iteration; how do you ensure you only act on the current run's signal?

**Demonstrated by:** agent-framework (`StreamingRunEventStream._completionEpoch`, `StreamingRunEventStream.cs:124-126, 218-235`).

This is a niche but robust pattern: every `InternalHaltSignal` carries a monotonically increasing epoch counter; the consumer compares the signal's epoch to its current iteration and ignores stale signals. Other sources avoid this problem by using typed events (openai-agents-sdk, letta, pydantic-ai) or by re-issuing the loop entirely (langgraph).

### Pattern 7: Idempotent finalization via terminal-trigger gating

**Problem:** A run may transition to terminal state multiple times (callback retries, webhook duplicates, race conditions). How do you ensure each side effect fires exactly once?

**Demonstrated by:** letta (`not_completed_before and is_terminal_update`, `run_manager.py:335-339`), agent-framework (`forceNotify` for continuation tokens, `ChatClientAgent.cs:882-896`), crewai (`_finalize_called` re-entrancy lock, `experimental/agent_executor.py:2261-2269`).

- letta: the callback fires only when `not_completed_before and is_terminal_update`; non-terminal updates never set `completed_at`. Belt-and-suspenders test (`tests/managers/test_run_manager.py:655-723`).
- agent-framework: `forceNotify` skips end-of-run notification when a per-service-call decorator handled persistence, but forces it for continuation-token scenarios where per-service-call persistence is unreliable.
- crewai: `experimental.AgentExecutor.finalize` acquires `_finalize_called`; concurrent branches return `"completed"` immediately. Tests at `test_finalize_called_reset_in_invoke` (`tests/agents/test_agent_executor.py:2181-2195`).

**Why it works:** The terminal transition is the unique moment a callback can fire; gating on it once gives exactly-once semantics for free.

**When to copy:** Any framework with callbacks, webhooks, or downstream notifications.

**Caveat:** letta's gating is enforced by a database constraint; crewai's is a runtime lock. The database approach survives process crashes; the runtime lock does not.

### Pattern 8: Streaming final-yield guarantee

**Problem:** A streaming consumer may receive partial outputs (e.g., partial JSON), and the final partial output may equal the last streamed output. How do you guarantee that the last consumed value is fully validated?

**Demonstrated by:** pydantic-ai (`result.py:76-103`).

`AgentStream.stream_output` always re-yields after `final_result_event` is set, with `allow_partial=False` (`result.py:95-103`). Consumers can rely on "last yielded = fully validated". This is unique among the seven sources.

### Pattern 9: Persistent final-answer contract with content validator

**Problem:** After a run completes, what is "the answer" stored on the conversation record? How do you ensure it's non-empty, sanitized, and bounded?

**Demonstrated by:** letta (`validate_function_response` + `ReturnCharLimit`, `utils.py:898-940`), openhands (`SlackV1CallbackProcessor._get_final_response`, `slack_v1_callback_processor.py:175-178` — though only checks `len(response.strip()) > 0`).

**Why it works:** The final answer is what consumers downstream see; a non-empty check at the boundary prevents empty messages from propagating.

**When to copy:** Any framework with a "get the final answer" endpoint or persisted response.

**Caveat:** letta's `validate_function_response` truncates to `return_char_limit`; openhands has no length cap, allowing a 200KB diff to be pushed to Slack verbatim (`openhands/slack_v1_callback_processor.py:175-178`). Length-bounded final outputs are undertreated across the field.

### Pattern 10: Discriminator-driven streaming schema

**Problem:** A streaming consumer needs to `match` on event type without parsing the payload first.

**Demonstrated by:** letta (`LettaStreamingResponse` RootModel union keyed on `message_type`, `letta_response.py:147-175`), pydantic-ai (`ModelResponseStreamEvent` with `event_kind` discriminator, `messages.py:2800-2802`), openai-agents-sdk (`RunItemStreamEvent` with typed `name`, `stream_events.py:10-61`), langgraph (v3 `LifecycleTransformer` emitting `started/completed/failed/interrupted/drained`, `stream/transformers.py:343`).

## Key Differences

The differences below are **not quality differences** — they reflect valid responses to different constraints (product shape, maturity, user needs, public API needs, operational constraints, compatibility requirements, performance constraints, library-vs-application).

### Difference 1: Where does "done" get decided?

- **At the loop boundary** (pydantic-ai, openai-agents-sdk, agent-framework agent-level, crewai): the agent loop is the source of truth. The model emits output → the loop decides `NextStepFinalOutput` or `AgentFinish` or `End`.
- **At the workflow/dataflow level** (agent-framework workflow layer, langgraph): the workflow decides. In langgraph, "done" = "no more triggerable tasks"; the loop has no opinion on correctness.
- **At the persistence layer** (letta): the `Run` row is the source of truth. The agent loop sets a `StopReasonType`; `RunStatus` is derived from it.
- **At the receiving boundary** (openhands): the webhook handler decides via `is_terminal()` and `event.value == 'finished'`.

**Why it diverges:** Library frameworks (pydantic-ai, openai-agents-sdk, langgraph) need a stable public API for "done" because they have library users. Application frameworks (openhands) have a fixed deployment and can rely on internal contracts. Workflow frameworks (agent-framework, langgraph) have a graph-level concept of completion that the agent-level loop alone doesn't satisfy.

### Difference 2: How is final output validated?

- **Hard schema validation** (pydantic-ai via `_output.run_output_with_hooks`, openai-agents-sdk via `AgentOutputSchema.validate_json`): the framework rejects invalid output and either retries or raises a typed error.
- **Guardrail-driven validation** (openai-agents-sdk `output_guardrails`, crewai `process_guardrail`): the user wires validators; the framework runs them and retries on failure.
- **No validation** (agent-framework XML docs explicitly tell callers to sanitize LLM output themselves, `ChatClientAgent.cs:22-37`; langgraph runtime does not validate `loop.output`; openhands `/agent_final_response` checks only `len(response.strip()) > 0`).

**Why it diverges:** Some frameworks (pydantic-ai) target users who want strict schemas; others (agent-framework) treat validation as out of scope and push it to callers; openhands treats the final answer as a string blob.

### Difference 3: Is `finish_reason` normalized?

- **Fully normalized to OTel literals** (pydantic-ai only): the agent loop only ever sees `Literal['stop','length','content_filter','tool_call','error']`.
- **Single special-case** (letta): only `"length"` is special-cased to `max_tokens_exceeded`.
- **Preserved as-is** (agent-framework, openai-agents-sdk, crewai): the raw provider value is exposed on the response or event but rarely influences control flow.
- **Not surfaced** (openhands): `finish_reason` is not visible at the app-server boundary.

**Why it diverges:** pydantic-ai's OTel alignment is a deliberate choice to interoperate with external observability tools. agent-framework is provider-agnostic and prefers to expose the raw value. crewai preserves the value but never translates it.

### Difference 4: Are pending tool calls a completion blocker?

- **Yes — blocks idle state** (agent-framework `RunStatus.PendingRequests`, `StreamingRunEventStream.cs:117-119`; letta `RequiredBeforeExitToolRule` + `requires_approval`, `tool_rule_solver.py:198-223`).
- **Yes — interrupts run** (openai-agents-sdk `NextStepInterruption`, `run_steps.py:169-174`; pydantic-ai `state='interrupted'`).
- **Yes — synthesized return** (pydantic-ai `EndStrategy='early'` produces `ToolReturnPart('Tool not executed - a final result was already processed.')`, `_agent_graph.py:1666-1680`).
- **No — first tool only** (crewai native path: only the first parsed native call is executed per iteration, `crew_agent_executor.py:667-807`).
- **N/A — graph-driven** (langgraph: tasks drive each other; pending tool calls just become new tasks).

### Difference 5: Is there a "completed-with-warnings" status?

**No source has one.** This is the single most convergent gap. Warnings live in:
- Metadata fields (letta `Run.metadata["error"]`, `routers/v1/agents.py:2153-2157`; openhands `detail: str`, `live_status_app_conversation_service.py:534-538`).
- Callback result enums (openhands `EventCallbackResultStatus.SUCCESS/ERROR`, `event_callback_result_models.py:11-14`).
- Output guardrail results (openai-agents-sdk `result.py:196-197`, but only `tripwire_triggered` blocks).
- Event streams (pydantic-ai's `EndStrategy` status parts are recorded in message history, `_agent_graph.py:1571-1577`).

### Difference 6: Is usage/cost eager or lazy?

- **Eager aggregation** (pydantic-ai `RunUsage`, agent-framework `AgentResponse.Usage`, crewai `UsageMetrics`, openai-agents-sdk `Usage` with per-request entries): usage is updated as responses arrive.
- **Lazy cost** (pydantic-ai `ModelResponse.cost()`): cost is computed on demand via `genai_prices`.
- **Persistent storage** (openhands `accumulated_cost`, letta per-step + per-run): cost is in the database.

### Difference 7: Is there an aggregate `RunOutcome`?

- **Yes (typed)** (openai-agents-sdk `RolloutTerminalMetadata.terminal_state` for sandbox, `pydantic-ai`'s `ModelResponseState` literal).
- **No (fragmented)** (agent-framework: callers must assemble from `AgentResponse` + `Run.OutgoingEvents`; crewai: callers inspect `CrewOutput` + events; langgraph: no per-run status field).

## Tradeoffs

### Tradeoff 1: Narrow success terminal vs. rich status

**Benefit of narrow terminal (pydantic-ai, openai-agents-sdk):** Callers can pattern-match on a closed set; new outcomes require explicit type additions.

**Cost:** Every non-terminal outcome must be a separate variant. pydantic-ai has only one success terminal but loses the ability to say "completed with warnings" without inspecting message history.

**Benefit of rich status (letta):** A 12-value `StopReasonType` lets callers distinguish truncated (`max_tokens_exceeded`), refused (`requires_approval`), and cleanly-stopped (`end_turn`) runs.

**Cost:** More variants to keep in sync; the mapping table (`StopReasonType.run_status`) must be updated for every new reason.

**Best-fit:** Narrow terminal for libraries where the public API matters most; rich status for applications where operational observability matters most.

**Failure mode:** CrewAI demonstrates the risk of a narrow-but-permissive terminal — `format_answer` catches all parse exceptions and synthesizes `AgentFinish` (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:370-386`), making "done" answerable for any garbage output.

### Tradeoff 2: Finish-reason normalization vs. raw preservation

**Benefit of normalization:** A single vocabulary for the agent loop to reason over. pydantic-ai can hard-fail on `finish_reason='length'` with empty output (`_agent_graph.py:1107-1114`).

**Cost:** Loses provider-specific nuance. OpenAI's `incomplete_details.reason` is richer than `finish_reason`; pydantic-ai keeps it in `provider_details` (`messages.py:1118-1128`), but most callers won't look there.

**Benefit of preservation (agent-framework, openai-agents-sdk, crewai):** Provider-native semantics survive for callers who need them.

**Cost:** The agent loop can't reason about finish reason unless it knows the provider vocabulary, which it usually doesn't.

**Best-fit:** Normalize when multi-provider; preserve when single-provider or when callers need access to provider-specific details.

### Tradeoff 3: Framework-side validation vs. caller-side validation

**Benefit of framework-side (pydantic-ai, openai-agents-sdk structured outputs):** Catches invalid output before it propagates. Invalid JSON raises `ModelBehaviorError`; invalid structured output is retried up to `max_output_retries`.

**Cost:** Framework imposes a schema; users who want raw text must opt out (`output_type=None` in openai-agents-sdk).

**Benefit of caller-side (agent-framework, langgraph, openhands):** No framework overhead; user controls validation.

**Cost:** Most users won't validate, and empty/garbage output propagates. agent-framework explicitly tells callers to "treat LLM output as untrusted" (`ChatClientAgent.cs:22-37`).

**Best-fit:** Framework-side validation for typed outputs (Pydantic models, JSON schemas); caller-side for raw text and unstructured responses.

### Tradeoff 4: Pending tool calls as state vs. as interrupt vs. as skip

**Benefit of state (agent-framework `PendingRequests`, letta `requires_approval`):** The run is "not done but blockable"; external input drives completion.

**Cost:** The framework must track unserviced requests and surface them; consumers must implement the resume path.

**Benefit of interrupt (openai-agents-sdk `NextStepInterruption`, pydantic-ai `state='interrupted'`):** The run is "paused"; `RunState.to_state()` packages it for resumption.

**Cost:** The resume mechanism must be round-trippable (pydantic-ai deliberately does not auto-cleanup, leaving work to callers).

**Benefit of skip (pydantic-ai `EndStrategy='early'`):** The run completes with synthetic returns; no resume needed.

**Cost:** Function tools may be silently skipped (synthetic `ToolReturnPart` recorded but user-visible work not done).

**Best-fit:** Interrupt for HITL and approval workflows; state for external requests (HTTP callbacks, queue messages); skip for fire-and-forget parallel tool calls where the answer matters more than the side effects.

### Tradeoff 5: Eager vs. lazy cost

**Benefit of eager (openhands, letta):** Cost is in the database; dashboards can read it directly.

**Cost:** Late stats events with smaller values silently overwrite the persisted peak (openhands `update_conversation_statistics` only writes when not None, `sql_app_conversation_info_service.py:447-465`).

**Benefit of lazy (pydantic-ai `cost()`):** Pricing errors don't crash the run; pricing updates take effect immediately.

**Cost:** Each consumer must recompute; cancelled-stream cost is documented as unsafe (`docs/output.md:1013-1015`).

**Best-fit:** Eager for production multi-tenant systems with billing dashboards; lazy for library users with their own pricing models.

## Decision Guide

### How to choose between approach models

**Choose a typed state machine with narrow success terminal (Cluster A) when:**
- You are building a library with a public API.
- Callers need to `match`/`switch`/`isinstance` on completion outcomes.
- "Done" is one of several legitimate outcomes (handoff, pause, completion, error).

**Example:** pydantic-ai, openai-agents-sdk, agent-framework (workflow layer).

**Choose two-level status with bidirectional mapping (Cluster B) when:**
- You are building a persistent application with a database.
- Operators need granular reasons for observability but consumers need a coarse status.
- Streams and storage speak different vocabularies.

**Example:** letta's `StopReasonType` ↔ `RunStatus` is the canonical instance.

**Choose dataflow/BSP stop condition (Cluster C) when:**
- Your primary abstraction is a graph.
- Completion is naturally expressed as "no more work to do".
- You have channel-based state management.

**Example:** langgraph's `PregelLoop.status` is the only instance.

**Choose parser-driven action vs. finish (Cluster D) when:**
- You are building an LLM-only agent loop with no graph abstraction.
- Completion is recognized by text pattern matching.
- The framework's primary interface is a `kickoff()` / `run()` method.

**Example:** crewai's ReAct parser; **but** be aware this is the lowest-scoring approach and the most fragile.

**Choose external runtime + string match (Cluster E) when:**
- You are building an application server that receives webhooks from a separately-deployed agent runtime.
- The runtime is pinned via external wheels, not vendored.
- You accept that "done" is defined outside your repository.

**Example:** openhands; **but** also be aware this is the lowest-scoring approach and the most brittle.

### How to choose between finish-reason strategies

- **Normalize to OTel literals** (pydantic-ai) when: multi-provider, observability integration, hard-fail on `length` / `content_filter`.
- **Single special-case** (letta) when: limited provider set, only one reason matters operationally.
- **Preserve raw** (agent-framework, crewai) when: callers need provider-native semantics.

### How to choose between usage/cost strategies

- **Eager + persistent** (openhands, letta): production multi-tenant with billing.
- **Eager + per-response** (pydantic-ai, openai-agents-sdk, crewai): library users who aggregate themselves.
- **Lazy** (pydantic-ai `cost()`): when pricing is unstable or provider-specific.
- **None** (langgraph): when downstream LangChain callbacks handle it.

## Practical Tips

### Patterns to copy

1. **Narrow success terminal + typed exception hierarchy.** Every library should adopt this. pydantic-ai's `End(FinalResult)` and `AgentRunError` subtree are exemplary.
2. **Finish-reason normalization at the adapter boundary.** Especially with OTel alignment (pydantic-ai). Even a single special-case (letta's `"length"` → `max_tokens_exceeded`) is better than nothing.
3. **Persistent usage/cost storage with idempotent updates.** letta and openhands demonstrate how to survive process loss. Use a "max-watermark" semantic for cost (`max(stored.cost, new_cost)`) to defend against late-stats retraction (openhands currently lacks this — `sql_app_conversation_info_service.py:447-465` only writes when not None).
4. **Idempotent finalization via terminal-trigger gating.** letta's `not_completed_before and is_terminal_update` and crewai's `_finalize_called` lock both work; prefer the database constraint for crash survival.
5. **Streaming final-yield guarantee.** pydantic-ai's `stream_output` always re-yields with `allow_partial=False` (`result.py:95-103`). Consumers can rely on "last yielded = fully validated".
6. **Discriminator-driven streaming schema.** letta's `message_type` union and pydantic-ai's `event_kind` discriminator both work. Use a typed enum, not string equality.
7. **Synthesized terminal marker for hung streams.** letta's Redis stream manager writes `stream_incomplete` if the stream ends without `[DONE]` (`redis_stream_manager.py:283-315`). This is the most defensive streaming-completion pattern in the field.
8. **Per-step + per-run usage aggregation.** letta's `get_run_usage` + `step_manager.update_step_success_async` is the cleanest example. pydantic-ai's `RunUsage` aggregation is a close second.

### Patterns to avoid or delay until needed

1. **String-equality terminal filtering.** All five openhands integration callbacks hard-code `event.value == 'finished'` (`slack_v1_callback_processor.py:46-57`, etc.). This breaks silently on any SDK rename. **Replace with an enum import or a typed reducer.**
2. **Permissive parser fallback.** crewai's `format_answer` catches all parse exceptions and synthesizes `AgentFinish` (`agent_utils.py:370-386`). This trades correctness for "the loop never panics" and is a non-obvious reason model output can be marked final. **Delay until you have observability on parse failures.**
3. **Two coexisting executors.** crewai's `CrewAgentExecutor` + `experimental.AgentExecutor` both exist (`crew_agent_executor.py:143-151`). Until crews flip the default, every finalization story has to be checked against both. **Avoid by making the new path the only path or by removing the old one.**
4. **Guardrail retries as full task re-execution.** crewai's `_invoke_guardrail_function` calls `agent.execute_task(... context=error ...)` for each retry (`task.py:1320-1324`). This multiplies LLM cost. **Cheaper alternatives exist (re-validate only).**
5. **Hallucination guardrail as a no-op.** crewai's `HallucinationGuardrail` returns `(True, task_output.raw)` and logs a warning (`tasks/hallucination_guardrail.py:20-103`). **This is a UX trap; if a guardrail does nothing, do not expose it.**
6. **No-length-cap final response.** openhands' `/agent_final_response` accepts any size (`slack_v1_callback_processor.py:175`). A 200KB diff pushed to Slack is a real failure mode. **Always bound the final response size.**
7. **Cost on cancel.** pydantic-ai explicitly documents this as unsafe (`docs/output.md:1013-1015`). **If your framework is silent about it, assume it is unsafe and document accordingly.**

### Decision rules

- **"Does done mean correct?"** — Only pydantic-ai's output-tool path hard-fails empty or invalid output. If you need correctness, build an output validator pipeline.
- **"Can the model falsely declare done?"** — Yes, in every framework. Use pydantic-ai's hard-fail on `finish_reason='length'` / `'content_filter'` as a template for catching the most common cases.
- **"Are final outputs validated?"** — Only pydantic-ai and openai-agents-sdk's structured-output path validate by default. Everywhere else, the caller validates.
- **"Is usage/cost finalized?"** — Yes for usage in five sources; yes for cost only in openhands and letta (persistent) and pydantic-ai (lazy). langgraph delegates to LangChain callbacks.
- **"Can a run complete with warnings?"** — No, in every source. Warnings live in metadata, callback results, or event streams. If you need a "completed-with-warnings" status, build it explicitly.

### Caution signs

A completion/finalization design is becoming brittle when:

1. **Callers must inspect multiple fields to determine success** (agent-framework: `AgentResponse` + `Run.OutgoingEvents` + `ContinuationToken`; crewai: `CrewOutput` + events; langgraph: `state.next` + `run_manager.on_chain_end`). This is a sign that no aggregate `RunOutcome` exists.
2. **The framework logs "done" but cannot tell whether the output is correct.** A heuristic for "any non-actionable output is final" is a brittle heuristic (crewai `format_answer`).
3. **Final output is treated as a string blob with no length cap or schema.** openhands' `/agent_final_response` is the canonical example.
4. **Pending tool calls can leave the run silently stuck.** Without an explicit interruption state (openai-agents-sdk `NextStepInterruption`, pydantic-ai `state='interrupted'`), a pending approval is invisible to consumers.
5. **Usage is per-response but cost is not aggregated at run level.** Most sources expose per-response usage but leave run-level aggregation to the caller.
6. **Streaming has no synthetic terminal marker.** Hung streams silently lose work. letta's `stream_incomplete` synthesis is the defensive pattern.
7. **Cancellation collapses into "Ended" without distinguishing user-cancel from natural-completion.** agent-framework's `RunStatus.Ended` covers both (`StreamingRunEventStream.cs:174-176`); letta's `cancelled` vs `completed` does not.
8. **String equality is used to detect terminal state.** openhands' five callbacks are the canonical example. This breaks silently on any upstream rename.
9. **Cost/usage updates are conditional on "not None"** (openhands `sql_app_conversation_info_service.py:447-465`). A late stats event with smaller values silently overwrites the peak.
10. **No test for the value space of terminal-status strings.** If your tests only assert `event.value == 'finished'`, you have no defense against SDK renames.
11. **The terminal success path does not fire `on_agent_end` lifecycle hooks.** openai-agents-sdk fires them via `run_final_output_hooks` (`turn_resolution.py:284-302`); if your framework doesn't, observability will be incomplete.
12. **Finalization side effects are fire-and-forget without idempotency.** openhands' `_run_callbacks_in_bg_and_close` (`webhook_router.py:561-573`) is fire-and-forget; the lack of join means there is no "wait for all finalization side-effects" guarantee.
13. **Two parallel final-output constructors drift apart.** openai-agents-sdk has four call sites that build `RunResult(final_output=..., ...)` (`run.py:971-985, 1107-1121, 1380-1394`, plus streaming) — maintenance burden is duplicated.

## Notable Absences

Things **no source** does well, or that are absent from the entire field:

1. **No source has a `RunOutcome` aggregate type** that unifies success / partial / failed / interrupted / cancelled. `RolloutTerminalMetadata.terminal_state` in openai-agents-sdk (`sandbox/memory/rollouts.py:59-67`) is a candidate vocabulary but only inside sandbox/memory.
2. **No source has a "completed-with-warnings" status** that is a first-class terminal-state modifier. Warnings live in metadata, callback results, or event streams — never as a status enum value.
3. **No source aggregates usage/cost at the framework level for multi-agent runs** unless the user explicitly builds a rollup. letta's `get_run_usage` is the closest; agent-framework's `MessageMerger.MergeUsage` is per-message, not per-run.
4. **No source validates the *final* output schema against an output model** at the framework level (except via user-supplied guardrails). pydantic-ai validates per-response via the output pipeline, but the *final* yield is "the last validated response", not a separate schema check.
5. **No source has a framework-side retry on `finish_reason='content_filter'` with non-empty content.** pydantic-ai treats this as a normal final output (`_agent_graph.py:1116`). The user must wrap in `FallbackModel` to retry (`docs/models/overview.md:353-391`).
6. **No source provides a structured "raw finish reason" field** per provider. Only pydantic-ai has `provider_details` (`messages.py:1118-1128`), and it's not documented as a public contract.
7. **No source has a `cancel()` hook that fires on framework-initiated cancellation** (asyncio shutdown, parent process kill). pydantic-ai's `state='interrupted'` is set by user cancellation (`models/__init__.py:820-844`), not framework-initiated cancellation.
8. **No source has automatic retry on `finish_reason='length'` in non-thinking, non-tool-call responses.** pydantic-ai raises immediately (`_agent_graph.py:1107-1114`).
9. **No source has a run-level warning aggregation field.** Status-style events emitted by pydantic-ai's `EndStrategy` are recorded as message parts but not exposed via a `result.warnings` accessor.
10. **No source has a documented cost-on-cancel guarantee.** pydantic-ai explicitly documents it as unsafe (`docs/output.md:1013-1015`); others are silent.

## Per-Source Notes

### agent-framework

- **Where the studied behavior is implemented:** `dotnet/src/Microsoft.Agents.AI.Abstractions/`, `dotnet/src/Microsoft.Agents.AI/`, `dotnet/src/Microsoft.Agents.AI.Workflows/`, `dotnet/src/Microsoft.Agents.AI.Harness/`, `dotnet/src/Microsoft.Agents.AI.Hosting.OpenAI/`, `dotnet/src/Microsoft.Agents.AI.DurableTask/`.
- **Main mechanism:** Layered — `ChatFinishReason` (chat) → `AgentResponse.ContinuationToken` (agent) → `RunStatus.Idle` (workflow) → orchestrator termination rules (Magentic/LoopAgent).
- **Supporting mechanisms:** `PendingRequests` blocks idle; `_runEnded` flag prevents post-end enqueue; `RequestHaltEvent` is the internal stream-termination sentinel; `InternalHaltSignal` is the run-loop coordination event with epoch tagging; `OutputFilter` allowlists which executors can yield outputs; `WorkflowWarningEvent` is emitted on soft failures but does not roll up into `RunStatus`.
- **Standout patterns:** Epoch-tagged completion signals (`_completionEpoch`); `forceNotify` dual-mode end-of-run notification for continuation tokens; terminator executors per orchestration style (`OutputMessagesExecutor`, `ConcurrentEndExecutor`, `HandoffEndExecutor`); `LoopEvaluator` family with `AIJudgeLoopEvaluator` and `CompletionMarkerLoopEvaluator`.
- **Tradeoffs:** No unified `RunOutcome`; final output not validated by framework (XML docs explicitly warn callers to sanitize LLM output); Magentic trusts LLM's "IsRequestSatisfied" without cross-check; `PendingRequests` blocks idle intentionally.
- **Failure modes:** Cancellation collapses to `Ended` without distinguishing user-cancel from natural-completion; double enumerator on event stream throws; post-end enqueue throws; Magentic progress-ledger failure retries up to 3 times then undefined state.
- **Score 7/10** — clear typed model with operational safety, but no unified `RunOutcome` and no framework-side validation.

### crewai

- **Where the studied behavior is implemented:** `sources/crewai/lib/crewai/src/crewai/agents/`, `sources/crewai/lib/crewai/src/crewai/llms/`, `sources/crewai/lib/crewai/src/crewai/crew.py`, `sources/crewai/lib/crewai/src/crewai/experimental/`, `sources/crewai/lib/crewai/src/crewai/types/usage_metrics.py`.
- **Main mechanism:** ReAct parser resolves `AgentAction` vs `AgentFinish`; tool `result_as_answer=True` short-circuits; budget exhaustion forces one final "force_final_answer" call; experimental `AgentExecutor.finalize()` has re-entrancy lock + last-todo heuristic + fallback stub.
- **Supporting mechanisms:** `_finish_reason_utils.extract_choices_finish_reason_and_id` defensive helper; `UsageMetrics.from_provider_dict` alias normalization; `TaskOutput.json` / `CrewOutput.json` fail-loud properties; `process_guardrail` mutual-exclusion validator; `_create_crew_output` double-validation raises on empty.
- **Standout patterns:** Centralized finish-reason defensive helper; `finalize()` re-entrancy lock with idempotency tests; `AgentFinish`-forcing helper on max-iter; permissive `format_answer` fallback; `UsageMetrics.from_provider_dict` mirroring `_track_token_usage_internal` for alias consistency.
- **Tradeoffs:** `_create_crew_output` is fail-fast, not partial (no `CrewOutput(partial=True)`); two coexisting executors with the legacy one marked deprecated; guardrail retries are full task re-executions; `finish_reason` preserved but not interpreted; `HallucinationGuardrail` is intentionally a no-op in OSS; no `CrewOutput.failed` flag.
- **Failure modes:** Empty-output task aborts the whole kickoff; `max_iter` + empty LLM raises `ValueError`; `OutputParserError` after `log_error_after=3` silently retried up to `max_iter`; native tool call returning non-string silently stringifies; guardrail callables dropped at checkpoint restore with warning; streaming `finish_reason` lost on mid-flight cancellation.
- **Score 6/10** — well-normalized usage tracking, but two coexisting executors and unnormalized finish_reason keep it below 7.

### langgraph

- **Where the studied behavior is implemented:** `sources/langgraph/libs/langgraph/langgraph/pregel/_loop.py`, `sources/langgraph/libs/langgraph/langgraph/pregel/main.py`, `sources/langgraph/libs/langgraph/langgraph/types.py`, `sources/langgraph/libs/checkpoint/langgraph/checkpoint/base/__init__.py`, `sources/langgraph/libs/langgraph/langgraph/stream/transformers.py`.
- **Main mechanism:** BSP/dataflow — loop exits when `prepare_next_tasks` returns empty; status literal (`Literal["input", "pending", "done", "draining", "interrupt_before", "interrupt_after", "out_of_steps"]`) maps to user-visible exceptions.
- **Supporting mechanisms:** Three durability modes (`sync`, `async`, `exit`) decouple when finalization happens; `_suppress_interrupt` is registered via `self.stack.push` for context-manager-style finalization; `_put_exit_delta_writes` stages `DeltaChannel` writes for visibility invariant; per-task `commit()` decides `(NO_WRITES, None)`, `(ERROR, exc)`, `(INTERRUPT, …)`.
- **Standout patterns:** Tasks write to channels, channels drive tasks (no explicit "done" node); `entrypoint.final[Return, Save]` distinguishes returned value from persisted value; lifecycle event fan-out via `LifecycleTransformer` for `started/completed/failed/interrupted/drained`; per-superstep write coalescing via `WRITES_IDX_MAP`; `channel_versions` is monotonic and idempotent.
- **Tradeoffs:** Done ≠ correct — empty writes trigger termination; `durability="exit"` defers visibility (crash safety vs. throughput); recursion limit is global, not per-node; per-run cost not aggregated; no retry / no warning counter.
- **Failure modes:** Infinite loops terminate as `GraphRecursionError`; cooperative drain exits cleanly with `GraphDrained(reason)`; exit-time exception swallowed silently by `_suppress_interrupt`; empty input raises `EmptyInputError`; bad node return value raises `InvalidUpdateError`; cancelled async tasks → `NodeCancelledError` if user-raised; `NO_WRITES` marker re-executes on resume.
- **Score 8/10** — explicit status literal with mapped exceptions, but no usage/cost aggregation and "done" means "stopped".

### letta

- **Where the studied behavior is implemented:** `letta/schemas/letta_stop_reason.py`, `letta/schemas/enums.py`, `letta/schemas/run.py`, `letta/schemas/run_metrics.py`, `letta/schemas/step_metrics.py`, `letta/services/run_manager.py`, `letta/services/step_manager.py`, `letta/agents/letta_agent_v3.py`, `letta/server/rest_api/redis_stream_manager.py`, `letta/utils.py`.
- **Main mechanism:** Two-level — `StopReasonType` (12 values, granular) mapped bi-directionally to `RunStatus` (5 values, coarse) via `StopReasonType.run_status` property; `RunMetrics` and `StepMetrics` carry nanosecond-level timing; `Run.completed_at` and `Run.stop_reason` are persisted.
- **Supporting mechanisms:** `StepProgression` state machine (`START → STREAM_RECEIVED → RESPONSE_RECEIVED → STEP_LOGGED → LOGGED_TRACE → FINISHED`) for atomicity; `Run` is durable, `Step` is best-effort during exception; callback fires only on terminal transition (`not_completed_before and is_terminal_update`); mid-stream vs. pre-chunk error handling; Redis stream manager synthesizes `[DONE]` or `stream_incomplete`; cancellation as a co-routine-level concern via `RunCancelledException` injected via `athrow`.
- **Standout patterns:** Decoupled stop-reason enum and run_status derivation (one source of truth, no parallel mapping tables); discriminator-driven streaming schema (`LettaStreamingResponse` RootModel union keyed on `message_type`); idempotent callbacks via "only-on-terminal" gating plus belt-and-suspenders `[DONE]` appending; parallel tool-call continuation forced (`aggregate_continue=True`); compaction as a first-class step event; token error classification ladder (`"length"` → `max_tokens_exceeded`).
- **Tradeoffs:** No dedicated "completed-with-warnings" status; lifecycle guards are logs only (re-updating an already-terminal run logs but mutates); reliance on the LLM adapter's `finish_reason` mapping (only `"length"` special-cased); callback POST is fire-and-forget at 5-second timeout; streaming finalization is string-sniffed from SSE chunks; `get_run_usage` reads every step's rows per call (no memoization).
- **Failure modes:** Stream ends before any chunk → re-raised; stream ends mid-stream → `LettaErrorMessage` emitted, no `end_turn` finish chunks; cleanup error during finalization → `cleanup_error` emitted; pending approval at cancel time → synthetic `ApprovalReturn(approve=False)` denials injected; task cancellation on infrastructure → `StopReasonType.error` with `stream_task_cancelled`; run cancel after terminal → idempotent no-op; re-cancel `requires_approval` run → allowed (treated as still-active); callback 5xx/timeout → log error, persist `callback_error`, run stays `completed`; auto-mode routing failure mid-flight → `update_step_resolved_model_async` in `finally` ensures partial billing.

## Executive Summary

Across the seven sources, "streaming" actually denotes three different categories of work, and the answer to *"does streaming give responsiveness without corrupting state?"* splits cleanly along that line.

The five LLM-aware sources (agent-framework, crewai, langgraph, openai-agents-sdk, pydantic-ai) all expose some form of token-level streaming, but only **pydantic-ai (9/10)** and **langgraph (8/10)** make the durability/cancellation boundary explicit enough to score "mature." Pydantic-ai persists truncated responses with a first-class `state ∈ {complete, incomplete, interrupted}` on `ModelResponse` (`pydantic_ai_slim/pydantic_ai/messages.py:2159-2161`, `models/__init__.py:884-903`); langgraph scopes streaming to a *projection* layer that never weakens the underlying durability/interrupts (`libs/langgraph/langgraph/pregel/_loop.py:660-664`, `runtime.py:79-104`). The other three LLM frameworks (agent-framework, openai-agents-sdk, crewai) score 6-7 because they expose a clear event protocol but each picks a different point to leave resilience to the caller — agent-framework with append-and-stop partial output (`_types.py:3122-3128`), openai-agents-sdk with retry-unsafe event gating that aborts the entire step after the first non-safe event (`model_retry.py:656-657`), and crewai with fire-and-forget chunks that are never rolled back (`utilities/streaming.py:287-289`).

The two non-LLM sources sit at the extremes. **temporal (8/10)** is a workflow engine, so its "streaming" reduces to one bi-directional replication gRPC stream (`proto/internal/temporal/server/api/historyservice/v1/service.proto:350`) plus eight long-poll RPCs annotated `API_CATEGORY_LONG_POLL` (`matchingservice/v1/service.proto:18-58`); it has no token concept and instead trusts partial output at **watermark granularity** with explicit resume from persisted `QueueReaderState` (`stream_sender.go:362-394`). **openhands (5/10)** sits at the opposite pole — its app_server has *no* token streaming at all, with NDJSON task progress and ZIP export being the only true `StreamingResponse` endpoints (`app_conversation_router.py:981-993, 1559-1604`); the actual token stream lives in a separate `agent_server` package not in this repo, with the app_server receiving `Event` objects in batches via webhook (`event_callback/webhook_router.py:453-466`).

Five convergent patterns emerge: (1) **append-and-stop on failure** is the dominant choice — only langgraph offers checkpoint-anchored mid-stream resume; (2) **provider-issued continuation tokens** enable OpenAI Responses API resume (agent-framework, openai-agents-sdk, pydantic-ai), but are absent from Anthropic/Bedrock/Gemini paths; (3) **per-event deduplication** is the universal mechanism for keeping streaming and non-streaming code paths in sync (openai-agents-sdk's `emitted_tool_call_ids` set, agent-framework's `call_id`-keyed merge, pydantic-ai's typed `narrow_type` promotion); (4) **cancellation is always cooperative** — only pydantic-ai's `iterator_with_cancel_guard` suppresses transport errors so `cancel()` looks clean (`models/__init__.py:806-830`), while every other framework surfaces a `CancelledError` to the consumer; (5) **guardrails run on the final response, not on chunks** — this is the single most consistent cross-source weakness and is acknowledged explicitly in agent-framework, openai-agents-sdk, and crewai.

## Core Thesis

> Streaming execution in agent frameworks converges on a small set of primitives — typed update events, per-item dedupe, idempotent close, async-iterator lifecycle — but diverges on three load-bearing decisions: **when partial output is trusted** (after watermark vs. after coalesced final response vs. never), **how cancellation flows** (provider-level socket close vs. task cancel vs. server-side drain), and **whether resume is in-band** (continuation tokens for OpenAI Responses, checkpoint IDs for workflows, A2A resubscribe only for cross-process). The product shape predicts the answer: a workflow framework with durable state will scope streaming to a projection layer that never weakens durability; an LLM SDK with no durable substrate will choose append-and-stop and delegate resume to caller-driven continuation tokens; an orchestration engine with no LLM at all will reject token streaming entirely and stream only what its state model can guarantee.

## Approach Models

### A. Typed-update iterator (agent-framework, openai-agents-sdk, pydantic-ai)

All three expose `AsyncIterable[UpdateEvent]` as the public surface, but the granularity of "what is an update" differs.

- **agent-framework** ships `ChatResponseUpdate` (provider-level) and `AgentResponseUpdate` (agent-level) wrapped by a generic `ResponseStream[UpdateT, FinalT]` with a finalizer, transform hooks, result hooks, cleanup hooks, and pull-context-manager factories (`python/packages/core/agent_framework/_types.py:2939-3329`). Updates carry `Content` items (text deltas, reasoning deltas, function-call argument fragments) plus optional `continuation_token`, `raw_representation`, and metadata; the finalizer coalesces adjacent text/reasoning/code-interpreter fragments, merges function-call args by `call_id`, and aggregates usage (`_types.py:1869-2072`).

- **openai-agents-sdk** ships a **three-tier** event union: `RawResponsesStreamEvent` (every raw `ResponseStreamEvent` passed through verbatim — `response.output_text.delta`, `response.function_call_arguments.delta`, `response.completed`, etc.), `RunItemStreamEvent` (semantic boundary events: `message_output_created`, `tool_called`, `handoff_occured`, `reasoning_item_created`), and `AgentUpdatedStreamEvent` (handoff signal) (`stream_events.py:10-58`). Two of these are interleaved on a single `asyncio.Queue`; the third (approval items) is *not* streamed at all and surfaces via `result.interruptions` (`run_internal/streaming.py:56-57`).

- **pydantic-ai** ships `ModelResponseStreamEvent = PartStartEvent | PartDeltaEvent | PartEndEvent | FinalResultEvent` (`messages.py:2800-2803`), with the four kinds discriminated by `event_kind` so a UI can group by part index and chain on `previous_part_kind`/`next_part_kind`. This is the only framework where streamed tool-call parts become *typed subclasses* (`NativeToolCallPart`, `ToolSearchCallPart`, ...) via `narrow_type` as soon as the tool name arrives (`messages.py:1877-1895`, `_parts_manager.py:86-97`), giving consumers an `isinstance`-stable API throughout the stream.

The decisive difference: **openai-agents-sdk keeps raw provider events visible** (token-level fidelity for UIs), **agent-framework hides the wire shape behind a typed envelope** (portable across providers), **pydantic-ai synthesizes lifecycle events** (PartEndEvent per part) so the UI never has to infer boundaries.

### B. Event-bus mediated iterator (crewai)

crewai is the outlier on the LLM side. Streaming is **not** an iterator on the LLM call; it is a side-channel on the singleton event bus. Every provider (OpenAI, Anthropic, Gemini, Bedrock, LiteLLM, Azure) emits `LLMStreamChunkEvent` (or `LLMThinkingChunkEvent`) through the same chokepoint (`llms/base_llm.py:595-625`), and the consumer registers a handler when `crew.stream=True` (`crews/utils.py:47`) that pushes each chunk into a `queue.Queue` / `asyncio.Queue` filtered by `stream_id` ContextVar (`utilities/streaming.py:110-144`). A generator (`create_chunk_generator` / `create_async_chunk_generator`) drains the queue into a `CrewStreamingOutput` (`types/streaming.py:240-319`). The synchronous event-bus dispatch (`events/event_bus.py:509-510`) preserves chunk order at the cost of throughput — a slow custom handler on `LLMStreamChunkEvent` blocks the LLM thread.

The trade-off: this design decouples "who produces chunks" from "who consumes them" so a single crew can drive a `Live` panel UI and a log file simultaneously, but it makes per-chunk user callbacks *non-public* (the user-facing API is the iterator, not the bus). It also leaves `LiteAgent` and the old `Agent` without streaming entirely (`lite_agent.py:477-546`, `agent/core.py:1568-1690`).

### C. Multi-mode graph streaming (langgraph)

LangGraph is the only source with **eight named stream modes** (`values`, `updates`, `messages`, `checkpoints`, `tasks`, `custom`, `debug`, `tools` — `pregel/main.py:2697-2710`) and three protocol generations (v1 raw, v2 typed `StreamPart`, v3 `StreamMux`+`StreamTransformer`). The v3 architecture is the most sophisticated in this study:

- A `StreamMux` collects events with monotonic `seq` numbers assigned only at the root mux; child mini-muxes (one per subgraph) inherit the parent's pump binding so cursors on a subgraph handle transparently drive the root run (`libs/langgraph/langgraph/stream/_mux.py:86-89, 193-225`).
- A `StreamTransformer` pipeline (e.g. `ValuesTransformer`, `MessagesTransformer`, `LifecycleTransformer`, `SubgraphTransformer`) processes each event, can suppress it with `return False`, or pass it through (`stream/_mux.py:269-296`).
- The "pump" is **the consumer's iteration** — `GraphRunStream` is documented as "No background thread is used — the caller's `for` loop is the pump" (`stream/run_stream.py:34-37`). This bounds memory to caller pace but means a forgotten consumer stalls the graph.
- Async pump uses an `asyncio.Condition` for take-a-number serialization (`run_stream.py:411-456`); abort cancels in-flight `_anext_task` and closes the mux (`run_stream.py:458-498`).

The non-obvious decision: streaming is a **projection layer** in langgraph, never a mutation path. Durability (`durability` ∈ `{sync, async, exit}`), interrupts (`interrupt_before`, `interrupt_after`, `interrupt()` from inside nodes), and the checkpointer allowlist operate on the channel/checkpoint layer regardless of whether a stream is attached (`pregel/_loop.py:660-664`, `runtime.py:79-104`). This is why langgraph scores 8/10 even though its v3 protocol is `@beta`-tagged — the streaming semantics can be experimental without weakening the durability contract.

### D. Coarse-grained app-server streaming (openhands)

openhands's app_server has the **least** token-streaming responsibility in the study: events arrive at the app_server as fully-formed `Event` objects in batches via webhook (`webhook_router.py:453-466`), persisted as individual JSON files (`event_service_base.py:86-92`). The only true `StreamingResponse` endpoints in this source are (1) NDJSON task progress during `start_app_conversation` (`app_conversation_router.py:981-993`) and (2) ZIP export of conversation trajectory (`app_conversation_router.py:1559-1604`). Token streaming lives in a separate `agent_server` package reachable via WebSocket from the frontend (`frontend/src/utils/websocket-url.ts:78-95`) but not in this repo.

The interesting design pattern is **save-before-yield**: every `AppConversationStartTask` is persisted via `app_conversation_start_task_service.save_app_conversation_start_task` *before* being yielded by the async generator (`live_status_app_conversation_service.py:300-307`), so a polling client can recover by replaying the persisted tasks. The ZIP export couples to a Redis lock with 30 s refresh (`live_status_app_conversation_service.py:2394-2421`) and fails closed with explicit 404/409/503/413 codes (`app_conversation_router.py:1605-1616`).

### E. Long-poll + replication stream engine (temporal)

Temporal is a workflow orchestration engine, not an LLM runtime, so its "streaming" has no token concept. The model is:

- **One bi-directional gRPC stream** (`StreamWorkflowReplicationMessages`, `proto/internal/temporal/server/api/historyservice/v1/service.proto:350`) for cross-cluster replication, wrapped by a generic `BiDirectionStream[Req, Resp]` with a 512-slot recv channel, `CloseSend` on shutdown, and exponential-retry lifecycle via `WrapEventLoop` (`service/history/replication/stream.go:46-92`).
- **Eight long-poll RPCs** annotated `API_CATEGORY_LONG_POLL` (`matchingservice/v1/service.proto:18-58`), each running a `pollLoop` that returns empty just before the caller's deadline via `WithDeadlineBuffer(ctx, longPollInterval, returnEmptyTaskTimeBudget)` (`common/contextutil/deadline.go:16-39`, `service/matching/matching_engine.go:2874`).
- **Pub/sub notifier** for state-change long polls: `events.Notifier` returns a size-1 channel per subscription and `NotifyNewHistoryEvent` non-blocking-enqueues into a 1000-slot central queue (`events/notifier.go:25, 131, 191-211`), so a slow subscriber never back-pressures the publisher.
- **Watermark-trusted partial output**: an empty `WorkflowReplicationMessages{ReplicationTasks: nil, ExclusiveHighWatermark: ...}` is a valid committed payload that advances reader state (`stream_sender.go:478-489`), so partial output is trusted *at watermark granularity* — never per-message.
- **Two-tier cancellation**: `outstandingPollers: SyncMap[pollerID, CancelFunc]` for `CancelOutstandingPoll`, plus a TTL cache and per-worker bulk-cancel map for `CancelOutstandingWorkerPolls` (`matching_engine.go:1200-1221`).
- **Resume by persisted watermark**: receiver-side `QueueReaderState` with `InclusiveMin` task IDs is persisted on every ACK; sender's `sendCatchUp` reads this state on reconnect (`stream_sender.go:338-394`).

The non-obvious choice: `BiDirectionStream` uses `context.Background()` for its internal client context (`bi_direction_stream.go:73`), so a client cancel does *not* stop the server-side sender goroutine; cancellation flows only through `Close()` from explicit callers (`Stop()`, monitor reconciliation). This is a deliberate decoupling — stream lifetime is decoupled from any single RPC call.

## Pattern Catalog

### 1. `AsyncIterable` of typed update events

**Problem**: how to expose a token-by-token stream as a Pythonic surface without leaking the underlying provider's wire format.

**Where**: agent-framework (`ResponseStream` + `ChatResponseUpdate`/`AgentResponseUpdate` at `_types.py:2939-3329`), openai-agents-sdk (`RunResultStreaming.stream_events()` draining `asyncio.Queue[StreamEvent]` at `result.py:696-779`), pydantic-ai (`AgentStream` and `ModelResponseStreamEvent` union at `result.py:55-243`, `messages.py:2800-2803`), langgraph (`AsyncGraphRunStream` at `stream/run_stream.py`), crewai (`StreamingOutputBase.__aiter__` at `types/streaming.py:147-153`).

**Why it works**: `async for` is the lingua franca for streaming in Python; lifecycle hooks (`__aenter__`/`__aexit__`) make cleanup composable; cancel via `asyncio.CancelledError` is well-understood by every consumer framework (FastAPI, Starlette, Jupyter).

**When to copy**: any framework whose consumers are async-Python. The pattern works less well when downstream consumers are sync (FastAPI sync routes, WSGI workers) — agent-framework ships a `.NET` parallel API for this (`AIAgent.RunStreamingAsync` returning `IAsyncEnumerable<AgentResponseUpdate>` at `dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgent.cs:464-506`).

**When overkill**: simple UI bridges where a callback model would do. crewai's event-bus chokepoint would be replaced by a callback here.

**Evidence**: `agent-framework.md:38-103`, `openai-agents-sdk.md:28-100`, `pydantic-ai.md:36-101`, `langgraph.md:29-92`.

### 2. Per-item deduplication

**Problem**: a streamed item and a non-streamed item resolve to the same logical record, but without dedupe the consumer would see the item twice.

**Where**: openai-agents-sdk keeps three sets: `emitted_tool_call_ids`, `emitted_reasoning_item_ids`, `emitted_tool_search_fingerprints` (`run_loop.py:1280-1282, 1529-1625`). agent-framework coalesces adjacent text/reasoning/code-interpreter fragments and merges function-call args by `call_id` (`_types.py:1869-2072`). pydantic-ai promotes `ToolCallPartDelta` to `ToolCallPart` only when `tool_name_delta` is set (`messages.py:2583-2598`), so the same delta chain produces one `PartStartEvent` followed by `PartDeltaEvent`s. langgraph dedupes by `message.id` on v1 (`_messages.py:97-104, 151-178`) and tracks `_streamed_run_ids` on v2/v3 (`_messages.py:402-406`, `transformers.py:300-322`).

**Why it works**: every framework needs to keep streaming and non-streaming code paths behaviorally aligned; the AGENTS.md policy in openai-agents-sdk (`AGENTS.md:21-24`) makes this an explicit invariant. The dedupe sets are the concrete mechanism.

**When to copy**: any time you have both a streaming entry point and a step-result entry point that may converge on the same logical item.

**When overkill**: pure-push systems where there is no second entry point to dedupe against.

**Evidence**: `openai-agents-sdk.md:51-53, 113-115`, `agent-framework.md:46-48`, `pydantic-ai.md:53-59`, `langgraph.md:52-55`.

### 3. Synchronous event-bus dispatch for ordering

**Problem**: chunks emitted concurrently by multiple workers must arrive at the consumer in the order the model produced them.

**Where**: crewai explicitly dispatches sync handlers for `LLMStreamChunkEvent` on the emitting thread, no thread pool, no async hop (`events/event_bus.py:508-510, 627-628`); the same is true for `LLMCallFailedEvent` for error ordering. agent-framework keeps transforms synchronous within `ResponseStream.__anext__` (`_types.py:3134-3140`); pydantic-ai's `iterator_with_final_event` breaks out after the first match then drains remaining events synchronously (`models/__init__.py:751-767`).

**Why it works**: chunk order is a contract the consumer depends on (UI text concatenation, tool-call argument accumulation). An async hop or thread-pool dispatch would allow reordering.

**When to copy**: when chunk order matters and consumers cannot dedupe by sequence number themselves.

**When overkill**: when chunk order is irrelevant (independent events on separate channels) or when the framework already provides sequence numbers (langgraph's monotonic `seq` from root mux).

**Evidence**: `crewai.md:51`, `agent-framework.md:123-125`, `pydantic-ai.md:46-47`.

### 4. `iterator_with_cancel_guard` to suppress transport errors after `cancel()`

**Problem**: closing the underlying HTTP stream raises `httpx.StreamError`/`httpx.TransportError`, but `cancel()` was deliberate — surfacing this as a stream error confuses consumers.

**Where**: pydantic-ai wraps the model stream in `iterator_with_cancel_guard` which `try/except self.get_stream_cancel_errors()` and only flips `_finished` on natural completion (`models/__init__.py:806-830`). Tests confirm `state='interrupted'` is set and the interrupted response appears in `all_messages()` (`tests/test_streaming.py:4678-4714`).

**Why it works**: cancellation is a deliberate user action, not an error. Suppressing the resulting transport exception makes the stream appear to end cleanly.

**When to copy**: any framework whose model stream is backed by an HTTP/WebSocket transport that raises on close.

**When overkill**: frameworks whose providers do not raise on graceful close (e.g. OpenAI's `responses.retrieve(stream=True)` path used in agent-framework — `_chat_client.py:640-665`).

**Evidence**: `pydantic-ai.md:48-50, 95-99`.

### 5. `ContinuationToken` as opaque, JSON-serializable TypedDict

**Problem**: how to resume a half-completed stream from a different process, after restart, or across a fail-over.

**Where**: agent-framework's `ContinuationToken` TypedDict (`_types.py:2078-2102`) is opaque to consumers, JSON-serializable, drives resume on OpenAI Responses via `client.responses.with_raw_response.retrieve(response_id, stream=True)` (`_chat_client.py:640-665`). The .NET parallel ships `ChatClientAgentContinuationToken` for cross-process resume (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgentContinuationToken.cs:1-160`). openai-agents-sdk threads `previous_response_id`/`conversation_id` through `OpenAIServerConversationTracker` and refuses retries unless `replay_safety == "safe"` (`model_retry.py:412-491`). langgraph offers `Command(resume=...)` against a checkpointer as the durable analogue (`pregel/_loop.py:878-921`).

**Why it works**: the model itself owns the resumption cursor. The framework just hands it back. Persisting the token across processes becomes a one-line JSON write.

**When to copy**: any provider whose wire protocol issues a stable id for an in-flight response (OpenAI Responses, A2A's `resubscribe`).

**When overkill**: providers without server-side resume (OpenAI Chat Completions, most non-OpenAI providers). For these, framework resume via checkpoints (langgraph, agent-framework workflow checkpoints at `_workflow.py:914-933`) is the alternative.

**Evidence**: `agent-framework.md:49-50, 64-66`, `openai-agents-sdk.md:70-77`, `langgraph.md:128-131`.

### 6. `state ∈ {complete, incomplete, interrupted}` on `ModelResponse`

**Problem**: after a stream ends, the consumer must distinguish "stream ended cleanly" from "truncated" from "cancelled" without inspecting flags.

**Where**: pydantic-ai alone ships this. `ModelResponse.state` is a `Literal['complete', 'incomplete', 'interrupted']` (`messages.py:123, 2159-2161`), inferred in `StreamedResponse.get()` based on `_cancelled` and `_finished` (`models/__init__.py:884-903`). Truncated responses are persisted to history so the next model request sees the truncated context (`_agent_graph.py:1027-1040`); `state='incomplete'` mid-tool-call raises `IncompleteToolCall` (`_agent_graph.py:146-160`).

**Why it works**: it surfaces a contract directly on the response object. Downstream code (guardrails, history queries, retry policies) can branch on `state` without knowing the streaming internals.

**When to copy**: any framework that retains streamed responses for replay or audit.

**When overkill**: ephemeral streaming where the final response is the only thing kept.

**Evidence**: `pydantic-ai.md:51-52, 72-76`.

### 7. Caller-driven pump instead of background tasks

**Problem**: a background producer can outrun the consumer; an unbounded queue hides the back-pressure problem.

**Where**: langgraph's `GraphRunStream` is documented as "No background thread is used — the caller's `for` loop is the pump" (`stream/run_stream.py:34-37`). Async pump uses `asyncio.Condition` for take-a-number serialization (`run_stream.py:411-456`). pydantic-ai's `agent.run_stream_events(...)` builds an `anyio.create_memory_object_stream` and a producer task via `self.run(..., event_stream_handler=...)` (`agent/abstract.py:1252-1325`) — bounded by the memory stream's capacity and drained by the consumer.

**Why it works**: pump pace equals consumer pace; memory grows only as fast as the consumer reads. Back-pressure is automatic.

**When to copy**: any framework where the producer is cheaper than the consumer (LLM tokens vs. UI render). The langgraph choice to *not* use background tasks is the cleanest version.

**When overkill**: when the producer must keep running regardless of consumer (replication streams — temporal deliberately uses `context.Background()` to decouple stream lifetime from any single RPC call, `bi_direction_stream.go:73`).

**Evidence**: `langgraph.md:64-69, 106-109`, `pydantic-ai.md:66-69`.

### 8. Visitor-pattern transformer pipeline with `return False` to suppress

**Problem**: multiple projections (`values`, `messages`, `lifecycle`, `subgraphs`) must share one event stream without coupling.

**Where**: langgraph's `StreamTransformer.process(event)` returns `False` to suppress the event before it hits the main log (`stream/_mux.py:269-296`). This is how `MessagesTransformer` ignores v1 `AIMessageChunk` tuples (`transformers.py:265-290`) and `LifecycleTransformer` absorbs `tasks` events (`transformers.py:458-473`).

**Why it works**: transformers are pure functions over the event stream; composing them is just registering more; suppression is one boolean.

**When to copy**: any framework with multiple named projections that share an underlying event log.

**When overkill**: frameworks with a single projection type (most SDKs).

**Evidence**: `langgraph.md:60-63, 73-77`.

### 9. Save-before-yield for recovery-by-polling

**Problem**: a streaming endpoint that loses its client mid-flight leaves the server holding state the client may never recover.

**Where**: openhands's `start_app_conversation` async generator calls `app_conversation_start_task_service.save_app_conversation_start_task` *before* yielding each task (`live_status_app_conversation_service.py:300-307`), so a polling client can replay state via `GET /api/v1/app-conversations/start-tasks/{id}`. The ZIP export uses Redis lock with 30 s refresh so the lock survives long exports (`live_status_app_conversation_service.py:2394-2421`).

**Why it works**: the stream becomes a *view* over persisted state; the client's connection is incidental.

**When to copy**: any task-progress or long-running-operation API where the client may reconnect.

**When overkill**: pure UI streams (chat token display) where reconnection logic is the consumer's responsibility.

**Evidence**: `openhands.md:40, 43`.

### 10. Watermark-trusted empty batches

**Problem**: a stream with no actual events must still allow forward progress (acks, liveness detection).

**Where**: temporal's replication sender emits `WorkflowReplicationMessages{ReplicationTasks: nil, ExclusiveHighWatermark: ...}` even when `beginInclusiveWatermark == endExclusiveWatermark` (`stream_sender.go:478-489`). The receiver treats this as a committed payload that advances reader state (`stream_receiver.go:374-377`). A similar `TaskMaxSkipCount = 1000` cap forces a watermark emission when tasks would otherwise be skipped because they belong to a different priority or shard (`stream_sender.go:518-531`).

**Why it works**: it lets the receiver distinguish "no tasks but stream alive" from "stream is dead." Without watermark-only messages, a stuck stream is indistinguishable from an idle one.

**When to copy**: any stream with a notion of "forward progress" that does not require a non-empty payload.

**When overkill**: UI token streams where empty messages are noise.

**Evidence**: `temporal.md:64-65`.

### 11. Retry-unsafe event gating

**Problem**: after the model has produced any non-trivial output, replaying the request would produce a different response; retry must be blocked or the response becomes wrong.

**Where**: openai-agents-sdk defines `_RETRY_SAFE_STREAM_EVENT_TYPES = frozenset({"response.created", "response.in_progress"})` (`model_retry.py:44`) and flips `emitted_retry_unsafe_event = True` on any non-safe event (`model_retry.py:365-367, 656-657`). Subsequent retries are blocked by `_evaluate_retry` (`model_retry.py:370-433`). The runner also refuses retries on stateful requests (`previous_response_id`/`conversation_id`) unless policy explicitly approves replay (`model_retry.py:436-491`).

**Why it works**: it converts a per-stream heuristic ("can we retry?") into a per-event decision ("what has the consumer already seen?"). The framework knows exactly when to stop.

**When to copy**: any LLM SDK that calls a non-idempotent model API. Especially critical when retry would re-charge the user.

**When overkill**: providers whose APIs are truly idempotent (rare).

**Evidence**: `openai-agents-sdk.md:71-77`.

### 12. Per-provider `close_stream()` override for upstream cancellation

**Problem**: a user clicks "Stop" but the model keeps generating until the next network round-trip; billing continues.

**Where**: pydantic-ai ships an abstract `StreamedResponse.close_stream()` (`models/__init__.py:833-868`) with provider-specific overrides: OpenAI (`openai.py:3261`), OpenAI Responses (`openai.py:3461`), Anthropic (`anthropic.py:2382`), Mistral (`mistral.py:698`), Google (`google.py:1311`), Groq (`groq.py:645`). Each closes the SDK-specific connection. `get_stream_cancel_errors()` defaults to `(httpx.StreamError, httpx.TransportError)` and is overridable per transport.

**Why it works**: one abstract method lets each provider cancel the way its SDK requires, without leaking provider-specific code into consumer-facing code.

**When to copy**: any framework whose model stream is backed by an HTTP transport that bills per token and supports abrupt close.

**When overkill**: streams where the producer is server-side state (replication, workflow tick loops).

**Evidence**: `pydantic-ai.md:48-50, 81-87`.

## Tradeoffs

### Append-and-stop vs. transactional rollback

The dominant choice across LLM frameworks is **append-and-stop**: whatever tokens or tool-call deltas were yielded stay yielded; on failure the consumer's buffer is left as-is.

- agent-framework: `_types.py:3122-3128` catches `Exception` in `ResponseStream.__anext__`, sets `_stream_error`, runs cleanup hooks, re-raises. Already-yielded updates are not discarded.
- openai-agents-sdk: `result.py:728-731` triggers `cancel()` on consumer `CancelledError` and re-raises. Already-queued events stay queued until the sentinel arrives.
- crewai: `utilities/streaming.py:287-289` simply raises `item` from the generator. Whatever was already in `state.chunks` stays.

This favors low-latency UIs (no double-buffering) at the cost of consumer-side handling. The only framework that goes the other way is langgraph, where the *checkpointer* sees only durability-controlled writes (`runtime.py:79-104`) — the stream is purely a view, so "rollback" is achieved by *not committing* to the checkpointer rather than by un-doing commits.

**Best-fit context**: append-and-stop is right when the consumer is a UI; transactional rollback is right when the consumer is a state machine that must be all-or-nothing.

**Alternative seen in another source**: langgraph's projection-layer model. A state-machine consumer can treat the stream as ephemeral and act on the next checkpoint instead.

### Cancellation propagation depth

Every framework supports user-initiated cancellation, but only three propagate the cancel *upstream* to the model/server:

- agent-framework: only MCP `tasks/cancel` for long-running tasks (`_mcp.py:2157-2195`); chat-stream cancellation drops the client connection but the server-side stream is not explicitly signaled.
- pydantic-ai: per-provider `close_stream()` override (`openai.py:3261`, `anthropic.py:2382`, `mistral.py:698`, `google.py:1311`, `groq.py:645`) closes the underlying HTTP stream — this stops token billing immediately.
- langgraph: `AsyncGraphRunStream.abort()` cancels in-flight `_anext_task`, which propagates `CancelledError` into the running node and lets the node's `try/finally` run (`run_stream.py:458-498`).

**Best-fit context**: cooperative cancellation (close_stream) is right for HTTP-backed models; task-cancel propagation is right for in-process graph execution.

**Failure mode of the shallow version**: a user clicks "Stop" but the model keeps generating for several more seconds until the next network round-trip — billing continues.

### In-band resume vs. caller-driven vs. none

Three different answers for "the network failed mid-stream; how do I get the rest?"

- agent-framework: caller-driven via `ContinuationToken`. The framework does not auto-retry. Resume requires the consumer to pass `continuation_token` back on the next call (`_chat_client.py:640-665`). Provider-specific: only OpenAI Responses supports it.
- pydantic-ai: HTTP-level retries via `AsyncTenacityTransport` (transport layer, pre-stream); `ModelRetry` for application-level (post-stream, re-builds `ModelRequestNode` reusing the truncated response in history — `_agent_graph.py:1027-1040`); output-tool validation retries explicitly *not* supported in `run_stream()` (`result.py:304-308`).
- langgraph: durability modes (`sync`/`async`/`exit`) flush writes independently from the stream; on next invocation `update_state` re-enters at the last durable checkpoint (`pregel/main.py:2530-2556`).
- openai-agents-sdk: retry-unsafe event gating — once `response.output_text.delta` or any non-safe event has been yielded, retries are blocked (`model_retry.py:656-657`). The step aborts and either the next turn retries or the run fails. Within the same step, no resume.
- temporal: persisted `QueueReaderState` watermark + `sendCatchUp` (`stream_sender.go:362-394`) is the only framework with automatic mid-stream resume from a server-side cursor.

**Best-fit context**: caller-driven continuation tokens are right for SDKs whose provider has resume; durability-anchored resume is right for frameworks with their own state substrate; watermark-based resume is right for server-to-server replication where both sides persist state.

### Unbounded queue vs. pump-paced

- openai-agents-sdk's `asyncio.Queue` is unbounded (`result.py:483-485`) — a fast producer vs. slow consumer grows memory until the run ends.
- langgraph's caller-driven pump bounds memory by consumer pace (`run_stream.py:34-37`).
- temporal's `BiDirectionStream` is bounded at 512 slots with a `ReplicationStreamChannelFull` metric on overflow (`bi_direction_stream.go:22, 186`).
- pydantic-ai uses `anyio.create_memory_object_stream` with capacity set by the consumer; `iterator_with_cancel_guard` does not add back-pressure but suppresses transport errors on cancel.

**Best-fit context**: unbounded queue is right for short-lived streams (single chat turn); pump-paced is right for long-lived graph runs; bounded-with-metric is right for server-to-server streams.

### Sync event-bus dispatch vs. async pipeline

- crewai: sync dispatch on `LLMStreamChunkEvent` to preserve chunk order (`events/event_bus.py:509-510`); cost is throughput — a slow custom handler blocks the LLM thread.
- langgraph: async `aprocess` is explicitly serialised ("slow `aprocess` serializes the pipeline by design", `stream/_mux.py:351-378`) — same trade-off.
- openai-agents-sdk: events go onto a single `asyncio.Queue`; ordering is guaranteed by the single queue, not by sync dispatch.

**Best-fit context**: sync dispatch is right when chunk order matters and producers are single-threaded; queue-mediated is right when producers and consumers may run in different tasks.

### Per-stream failure modes (comparison)

- agent-framework: streaming exceptions captured on span attributes but span not flipped to ERROR (`tests/core/test_observability.py:2023-2050`) because cleanup runs in a different async context.
- crewai: per-chunk guardrails impossible — `before_llm_call`/`after_llm_call` see only the final string (`openai/completion.py:2044-2048`); LiteAgent users get nothing.
- openai-agents-sdk: retry-unsafe event gating is binary — one non-safe event blocks all retries, even if the network had only a 100 ms blip 10 tokens in.
- pydantic-ai: `finish_reason='length'` mid-tool-call raises `IncompleteToolCall` (`_agent_graph.py:146-160`) — safer than guessing partial args but requires clients to bump `max_tokens`.
- temporal: `BiDirectionStream` uses `context.Background()` internally (`bi_direction_stream.go:73`); client cancellation does *not* propagate to the server-side sender goroutine.

## Decision Guide

| If your system is… | Choose… | Because… |
|--------------------|---------|----------|
| An LLM SDK with no durable state | Append-and-stop + caller-driven `ContinuationToken` (agent-framework, openai-agents-sdk pattern) | You cannot resume what you did not persist; let the provider's resume primitive do the work |
| An LLM SDK with checkpoint substrate | Durability-anchored resume with `state ∈ {complete, incomplete, interrupted}` on responses (pydantic-ai pattern) | The state machine needs to know whether the response was truncated, cancelled, or completed |
| A workflow framework with graph execution | Multi-mode `StreamPart` + caller-driven pump + visitor transformers (langgraph pattern) | Multiple projections, subgraph nesting, and abort/drain primitives are first-class; v3 `@beta` is acceptable when the projection layer does not mutate state |
| An event-bus framework with multiple UI consumers | Synchronous event-bus dispatch + queue-mediated iterator façade (crewai pattern) | Decouples producer from consumer; preserves chunk order; at the cost of throughput |
| A server-to-server replication engine | Watermark-trusted empty batches + persisted cursor + receiver-mode lock (temporal pattern) | The only forward-progress signal is a watermark; resume is automatic from `QueueReaderState` |
| A coarse-grained app server that delegates the LLM runtime | File-per-event persistence + NDJSON progress + ZIP export (openhands pattern) | You do not own token streaming; expose state changes via webhook and let the agent runtime handle the token path |

## Practical Tips

1. **Default to `AsyncIterable[TypedUpdateEvent]` with `__aenter__`/`__aexit__` for lifecycle.** Every Python consumer framework (FastAPI, Starlette, Jupyter, asyncio) speaks this dialect; agent-framework's hook triumvirate (`transform_hook` + `result_hook` + `cleanup_hook`) is the cleanest extension surface.

2. **Dedup by stable id, not by sequence number.** OpenAI Responses assigns every output item a stable id; use it. Tool-call args accumulation needs both id (to merge fragments) and index (to handle parallel calls). See `openai-agents-sdk.md:51-53` and `agent-framework.md:46-48`.

3. **Make the partial-vs-final distinction visible on the response object.** Pydantic-ai's `state ∈ {complete, incomplete, interrupted}` is the cleanest version. If you do nothing else, at least set a `finish_reason` and an `_cancelled` flag and let consumers infer.

4. **Treat cancellation as a deliberate user action.** Wrap the model stream in an `iterator_with_cancel_guard` so `httpx.StreamError` after `cancel()` is suppressed. Otherwise UIs see a stream error after every Stop click (`pydantic-ai.md:48-50`).

5. **Override `close_stream()` per provider.** The OpenAI, Anthropic, Mistral, Google, Groq SDKs each have their own cancellation mechanism; an abstract `close_stream()` lets each provider stop billing immediately. Pydantic-ai's pattern (`pydantic-ai.md:81-87`) is the template.

6. **Use a `ContinuationToken` TypedDict for cross-process resume.** Even if you do not implement resume today, ship the typed token and let consumers persist it. agent-framework's `_types.py:2078-2102` is the model.

7. **Save state before yielding, especially for long-running task APIs.** openhands's pattern (`live_status_app_conversation_service.py:300-307`) lets a polling client recover by re-reading state — the stream becomes a view, not the source of truth.

8. **For server-to-server streams, trust empty batches at watermark granularity.** temporal's `WorkflowReplicationMessages{ReplicationTasks: nil, ExclusiveHighWatermark: ...}` is the canonical pattern; without it the receiver cannot distinguish "stream alive, no events" from "stream stuck" (`stream_sender.go:478-489`).

9. **For multi-projection frameworks, prefer a visitor-pattern transformer with `return False` to suppress.** langgraph's `StreamTransformer.process` (`stream/_mux.py:269-296`) lets projections share one event log without coupling.

10. **Be explicit about which events do not stream.** openai-agents-sdk skips approval items and compaction items from the queue entirely (`streaming.py:56-62`); pydantic-ai synthesizes `PartEndEvent` per part; the lesson is the same — surface lifecycle at the right granularity rather than letting consumers infer.

11. **For retry safety, gate on per-event rather than per-stream.** openai-agents-sdk's retry-unsafe event gating (`model_retry.py:43-77`) is the principled version: track which event types have been seen, and freeze the stream once a non-replayable event has been emitted. State requests (`previous_response_id`) need explicit policy approval.

12. **For OpenAI Responses-style server-managed conversations, route retries through a server-conversation tracker.** openai-agents-sdk's `OpenAIServerConversationTracker.rewind_input` (`run_loop.py:1452-1456`) avoids double-recording conversation history that the server already owns.

## Anti-Patterns / Caution Signs

- **Using a single `asyncio.Queue` for streamed and non-streamed events** without separating input-guardrail and output-guardrail queues. openai-agents-sdk's two-queue split (`result.py:483-488`) avoids a race where guardrail results arrive after the consumer thinks the run is done. If you only have one queue, at minimum document the order contract.

- **Per-chunk guardrail enforcement attempted without batching windows.** crewai explicitly notes `before_llm_call`/`after_llm_call` run on the final string only; trying to call them on every chunk will fail because the model has not finished its turn (`crewai.md:108`). If you need token-level guardrails, build a `StreamingGuardrailConfig` that operates on a rolling window.

- **Synchronous event-bus dispatch for non-stream events.** crewai does this only for `LLMStreamChunkEvent` and `LLMCallFailedEvent`; extending it to all events would serialize the entire agent on handler latency.

- **Caller-driven pump without an async-condition or lock for concurrent consumers.** langgraph's `asyncio.Condition` (`run_stream.py:411-456`) gives take-a-number semantics; without it two consumers racing on `_pump_next` can advance the graph twice.

- **Lifecycle state stored as boolean flags (`_cancelled`, `_finished`) without a typed enum on the response.** pydantic-ai's `Literal['complete', 'incomplete', 'interrupted']` on `ModelResponse.state` is the model; consumers should not have to inspect two flags to know what happened.

- **Unbounded queue for server-to-server replication without overflow metric.** temporal's `ReplicationStreamChannelFull` (`bi_direction_stream.go:186`) is the counter-example — back-pressure must be measurable.

- **Persisting chunks by default without an explicit opt-in.** crewai's `CheckpointConfig(events=[...])` is opt-in (`crewai.md:79`); defaulting to "persist everything" will balloon storage for users who only care about final results.

- **Reserving `current_task_info` once and never refreshing.** crewai's bug (`crews/utils.py:381-391`) means `StreamChunk.task_index` is broken in practice for multi-task crews. If you advertise per-task metadata on chunks, refresh it per task.

- **Setting `_cancelled` and then immediately raising on close.** pydantic-ai's pattern of separating `_cancelled` (set in `cancel()`) from `_finished` (set only on natural completion) lets `state` inference distinguish user cancel from truncation. Setting both at once collapses the two states.

- **`context.Background()` for stream lifecycle** when the consumer expects client-cancel propagation. temporal uses this for replication because the stream lifetime is intentionally decoupled from any single RPC; LLM SDKs should not.

- **Streaming span status flipped to ERROR on consumer-side exception.** agent-framework deliberately does not flip because cleanup runs in a different async context (`tests/core/test_observability.py:2023-2050`); documenting the "exception captured on span attributes, span status remains" contract is critical for observability consumers.

- **Tool-call argument parser emitting empty-string `arguments` from `output_item.added`.** openai-agents-sdk's `tests/test_streaming_tool_call_arguments.py` (issue #1629) regression test guards against this — wait for `output_item.done` to read the populated arguments.

- **Bulk-cancel path without signaling the client to reconnect.** temporal's `CancelOutstandingWorkerPolls` cancels outstanding pollers but does not signal the gRPC client; adding a hint response code would help worker shutdown loops react faster (`temporal.md:184`).

## Notable Absences

- **No framework in this study offers an in-band resume of a half-consumed raw LLM stream for non-OpenAI providers.** Only OpenAI Responses has server-side resume (`previous_response_id` / `response.retrieve(stream=True)`); agent-framework uses it, openai-agents-sdk threads it through `OpenAIServerConversationTracker`, pydantic-ai reuses it via `message_history=` re-feed. Anthropic, Bedrock, Gemini, Mistral, Groq have no equivalent in any source.

- **No framework exposes a general per-event authorisation hook.** openai-agents-sdk relies on `ToolApprovalItem` / `MCPAprovalRequestItem` as the unit of human intervention; there is no `should_this_event_be_surfaced` predicate. crewai's `before_llm_call`/`after_llm_call` hooks are batch-level only.

- **No framework offers per-chunk output-guardrail evaluation.** This is acknowledged as a gap in agent-framework (only re-validated at `get_final_response`), openai-agents-sdk (output guardrails fire at `_finalize_streamed_final_output`), and crewai (`before_llm_call`/`after_llm_call` see the final string). Pydantic-ai's `validate_response_output(allow_partial=True)` is partial-output validation of structured output, not content moderation.

- **No framework offers a stream-level retry budget independent of the model retry budget.** openai-agents-sdk has retry-unsafe event gating but it is binary (one non-safe event → no retry); agent-framework has no auto-retry; pydantic-ai's HTTP retries are transport-level only. A "retry the stream up to N times" knob is missing across the board.

- **No framework offers automatic checkpoint-on-stream-error.** Langgraph checkpoints at durability-mode boundaries but not on stream failure; agent-framework's `_workflow.py:803-810` raises `RuntimeError` if the runner has in-flight messages. The combination — checkpoint on every stream error — is absent.

- **No framework bridges SSE/WebSocket reconnect for mid-stream client drops.** openhands's frontend `useWebSocket` only reconnects when explicitly marked (`frontend/src/hooks/use-websocket.ts:28-156`); pydantic-ai's `UIEventStream` provides SSE encoding but no reconnect logic (`ui/_event_stream.py:139-153`).

- **No framework provides a "transactional stream" mode where partial output is buffered until `get_final_response()` succeeds.** agent-framework's `Future Considerations` section notes this as a possible extension (`agent-framework.md:163`); `_finalize_response` is the closest analogue but is not opt-out-able per stream.

- **No framework has a per-event idempotency key for at-least-once delivery on the wire.** Agent-framework tracks conversation_id but not event delivery. Temporal's replication stream relies on ACK watermarks for at-most-once-by-task-id semantics. A "exactly-once event delivery" primitive is absent.

## Open Questions

1. **How would an in-band resume mechanism work for non-OpenAI providers?** No source in this study addresses this. Agent-framework explicitly notes "For non-OpenAI providers, consumers must rely on checkpointing or resend the entire conversation" (`agent-framework.md:170`). Could a provider-agnostic "StreamCursor" abstraction capture client-side state sufficient to resume from any wire-level failure point?

2. **What is the right back-pressure signal from a streaming LLM consumer to the provider?** OpenAI Responses has no provider-side rate-limit signal; the SDK's unbounded queue (`result.py:483-485`) hides this. Pydantic-ai's `iterator_with_cancel_guard` is cancel-only; agent-framework's `ExitStack` per `__anext__` is observability-only. A "slow down" signal would let providers throttle on back-pressure rather than billing for tokens that will be discarded.

3. **Should stream-level retry budget be independent of model-level retry budget?** openai-agents-sdk couples them (retry-unsafe event gating blocks all retries after the first non-safe event, `model_retry.py:656-657`); agent-framework has no retry at all; pydantic-ai retries at transport-level pre-stream. A "retry the stream up to N times, then give up" knob is absent.

4. **Can checkpoint-on-stream-error be implemented as a default?** Langgraph checkpoints at durability-mode boundaries but not on stream failure; agent-framework's `_workflow.py:803-810` raises `RuntimeError` if the runner has in-flight messages. The combination — automatic checkpoint on stream error for any run with a checkpointer — would close a gap that affects every framework.

5. **How should a framework distinguish "stream interrupted" from "stream failed" in observability?** agent-framework's telemetry deliberately captures exceptions on span attributes but does *not* set `span.status = ERROR` (`tests/core/test_observability.py:2023-2050`) because the cleanup runs in a different async context. Pydantic-ai's `state` field on `ModelResponse` is cleaner; the question is whether observability should follow the same pattern.

6. **What is the right granularity for per-chunk output-guardrails?** Acknowledged as a gap in every LLM framework. A rolling-window policy (every N tokens, every M seconds) operating on `get_full_text()` (crewai) or the rolling `stream_text(delta=True)` (pydantic-ai) is one answer; per-token validation is almost certainly too expensive.

7. **Is the `StreamToolCallHandler` ContextVar bridge (langgraph, `_tools.py:25-32`) the right pattern for cross-task tool-call streaming?** The reset is best-effort on async detach (`_tools.py:213-222`); no test covers the failure mode where a worker thread emits a delta after the ContextVar has been reset. An explicit channel-based forwarding primitive might be more robust.

8. **How would a "transactional stream" mode work in practice?** Agent-framework's `Future Considerations` notes this as a possible extension: a `stream(replay=False, transactional=True)` flag that buffers until the final response would let tool-heavy agents avoid partial-state risk. What is the user-facing API for "I want this stream to be transactional"?

9. **Does the `current_task_info` bug in crewai (`crews/utils.py:381-391`) belong in a v1 or v2 fix?** The advertised `chunk.task_index` is broken for multi-task crews; either the docs or the code is wrong. A backward-compatibility-preserving fix would refresh `current_task_info` per task via a mutable container.

10. **Should `tools` be a stream mode in langgraph or a separate channel?** It is both — `stream_mode="tools"` exists alongside the `tools` channel on individual messages. The duplication is documented but the consumer-facing surface area is larger than necessary.

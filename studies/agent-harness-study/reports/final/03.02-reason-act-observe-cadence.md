## Executive Summary

Six of the seven reports show a deliberate action/observation cycle: the model emits a structured or parsed action, tools execute, and a result is injected into the next model input. The dominant modern design is provider-native structured tool calling rather than a textual `Thought`/`Action`/`Observation` protocol. LangGraph and Pydantic AI encode the cycle as a graph; agent-framework, Letta, and the OpenAI Agents SDK use layered typed turn loops; CrewAI retains a text-mode ReAct fallback beside native calls. `[agent-framework]` extracts `function_call` content and appends a `tool` message (`python/packages/core/agent_framework/_tools.py:2347`, `python/packages/core/agent_framework/_tools.py:2261`); `[langgraph]` routes on `AIMessage.tool_calls` through an `agent` to `tools` graph edge (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:831-859`); `[pydantic-ai]` uses `UserPromptNode` → `ModelRequestNode` → `CallToolsNode` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2129-2139`).

The most reusable convergence is not a visible scratchpad. It is a durable, correlated transcript: typed action items, typed tool results, stable call identifiers, explicit error status, bounded retries, and a clear termination policy. Letta, the OpenAI Agents SDK, and Pydantic AI provide the strongest combination of structured reasoning preservation, observation handling, and failure budgets, scoring 8/10 in their reports. Agent-framework, CrewAI, and LangGraph also have clear cadence implementations, but lose points for optional or absent compaction, weaker reasoning semantics, or less aggressive repeated-error handling. `[letta]` persists `step_id` and `run_id` across messages and tool returns (`letta/schemas/message.py:292-293`); `[openai-agents-sdk]` replays typed run items and preserves call/output joins (`src/agents/run_internal/items.py:23-31`, `src/agents/run_internal/items.py:66-83`); `[pydantic-ai]` separates per-tool and global retry budgets (`pydantic_ai_slim/pydantic_ai/_run_context.py:60-77`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:162-180`).

The main weakness across the set is context growth. Most systems return raw tool output by default and make compaction opt-in, provider-specific, hook-driven, or overflow-triggered. Letta is the clearest exception: it combines per-tool truncation, a dynamic context share, proactive post-step compaction, and context-overflow retry. OpenHands presents a useful contrasting architecture: its app repository has typed action/observation events and a configured condenser, but the actual loop is external to the studied source tree. Thus a system can be operationally observable while remaining difficult to audit for the exact cadence contract. `[letta]` implements both dynamic tool-return caps and proactive/reactive compaction (`letta/agents/letta_agent_v3.py:143-153`, `letta/agents/letta_agent_v3.py:1218-1294`, `letta/agents/letta_agent_v3.py:1438-1510`); `[openhands]` delegates the loop to pinned external packages (`pyproject.toml:47-49`) while exposing `ActionEvent`/`ObservationEvent` pairs (`frontend/src/types/v1/core/events/action-event.ts:40`, `frontend/src/types/v1/core/events/observation-event.ts:20`).

## Core Thesis

A robust Reason/Act/Observe cadence is a typed state transition contract, not merely a prompt format:

1. **Reason** produces a model response whose action-bearing items can be identified without ambiguous text matching.
2. **Act** executes each identified call under approval, concurrency, validation, and cancellation policy.
3. **Observe** returns a correlated, bounded, model-readable result, including failures, to the next model turn.
4. **Continue or stop** is decided by explicit budgets and state—not only by whether the model happens to emit another tool call.
5. **History management** preserves the action/result relationship while compacting or truncating old context.

The sources converge on the first three requirements and diverge on ownership of the latter two. Graph-oriented systems make the cadence visible and replaceable but delegate prompt shaping and compaction to callers. Client-loop systems make the common path convenient but hide more policy inside a large executor. Durable agent systems add step identities, checkpoints, and compaction, at the cost of more state-machine surface. A text ReAct fallback remains useful for weak or legacy model interfaces, but it should be isolated behind the same normalized action/observation contract rather than treated as the canonical representation. `[crewai]` demonstrates both the compatibility value and brittleness of this fallback: native tool calls are normalized into assistant/tool messages, while text mode depends on `ACTION_INPUT_REGEX` and stop-word trimming (`lib/crewai/src/crewai/agents/constants.py:18-22`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:843-866`).

## Approach Models

### 1. Provider-native typed turn loops

Agent-framework, Letta, and the OpenAI Agents SDK treat provider-produced structured items as the action boundary. Agent-framework extracts `Content(type="function_call")`, executes calls, and appends `Content(type="function_result")` in a `tool` message. Letta normalizes provider responses into OpenAI-style tool calls and returns `ToolReturn` payloads. The OpenAI Agents SDK classifies `Response.output` into typed run items such as `ToolCallItem`, `ReasoningItem`, and `MessageOutputItem`, then converts tool outputs back to API input items. These designs avoid parsing free text and preserve call identity, but require provider adapters or an API contract that can express tool calls. `[agent-framework]` uses structured extraction and result construction (`python/packages/core/agent_framework/_tools.py:2347`, `python/packages/core/agent_framework/_types.py:818-874`); `[letta]` uses provider-normalized `tool_calls` and `parallel_tool_calls` enforcement (`letta/agents/letta_agent_v3.py:1326-1342`); `[openai-agents-sdk]` performs typed response classification and tool planning (`src/agents/run_internal/turn_resolution.py:1551-2001`, `src/agents/run_internal/tool_planning.py:177-263`).

**Best fit:** a framework that controls provider adapters, needs reliable replay, or exposes tools as a public API. **Risk:** a provider that cannot produce the expected structured shape may terminate, raise, or require a separate compatibility path rather than receiving a textual ReAct fallback. Agent-framework explicitly returns when no structured function calls are available (`python/packages/core/agent_framework/_tools.py:2346-2357`); the OpenAI SDK branches deeply on Responses output types (`src/agents/run_internal/turn_resolution.py:1636-1656`, `src/agents/run_internal/turn_resolution.py:1770-1830`).

### 2. Graph-as-loop

LangGraph and Pydantic AI make the cadence a graph transition. LangGraph routes an `AIMessage` with tool calls from `agent` to `tools`, then back to the model or to `END`. Pydantic AI advances `UserPromptNode`, `ModelRequestNode`, and `CallToolsNode`, with the tool node returning either another model request or an end state. This makes the loop easy to insert into, inspect, and replace with hooks, interceptors, or human-in-the-loop nodes. `[langgraph]` implements conditional routing and v1/v2 dispatch (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:831-859`); `[pydantic-ai]` exposes node iteration through `AgentRun.next` (`pydantic_ai_slim/pydantic_ai/run.py:330-404`).

**Best fit:** applications that need custom routing, pauses, retries, or domain-specific intermediate nodes. **Risk:** the framework may provide a powerful cycle without owning the model-input policy. LangGraph delegates trimming to `pre_model_hook` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-413`), and Pydantic AI delegates history rewriting to `HistoryProcessor` or provider compaction (`pydantic_ai_slim/pydantic_ai/capabilities/process_history.py:31-38`, `pydantic_ai_slim/pydantic_ai/models/__init__.py:288-300`).

### 3. Hybrid native/text ReAct

CrewAI supports native function calling when available and falls back to a textual ReAct protocol when a provider rejects tools. The native path normalizes provider-specific call shapes; the text path prompts for `Thought`, `Action`, and `Action Input`, parses with regex, repairs some malformed JSON, and feeds parser errors back as user messages. This is the strongest compatibility model in the set, but also the most exposed to formatting drift. `[crewai]` documents the fallback route and its test (`lib/crewai/src/crewai/agents/crew_agent_executor.py:577-579`, `lib/crewai/tests/agents/test_native_tool_calling.py:1271`), and the text parser contract (`lib/crewai/src/crewai/agents/parser.py:62`, `lib/crewai/src/crewai/agents/constants.py:18-22`).

**Best fit:** products that must support providers without reliable native tools or legacy prompt contracts. **Risk:** treating the text parser as the main protocol. Stop-word trimming, exact marker order, malformed JSON, and tool output containing answer markers can all interfere with the next action. `[crewai]` uses `stop=["\\nObservation:"]` (`lib/crewai/src/crewai/agent/core.py:1062-1066`) and repairs input with `json_repair` (`lib/crewai/src/crewai/agents/parser.py:161-178`).

### 4. Event-observable externalized loop

OpenHands separates the app server from the agent runtime. The repository stores and routes `ActionEvent`, `ObservationEvent`, error, hook, pause, and condensation events while pinned external packages own model parsing and loop mechanics. This is an architectural model for multi-process agent products: the app layer can render and persist cadence events without owning execution. `[openhands]` routes `send-message` to the agent server (`openhands/app_server/app_conversation/app_conversation_router.py:552-567`) and persists event callbacks (`openhands/app_server/event_callback/webhook_router.py:453-522`).

**Best fit:** a product with a separately deployed sandbox/agent worker and multiple front ends. **Risk:** observability without auditability. The studied repository cannot establish whether errors are re-prompted, how reasoning is reused, or how action output is parsed. The condenser is configured in the app layer, but event replacement remains SDK behavior (`openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`).

## Pattern Catalog

### Pattern 1: Normalize every action and observation around a stable call identifier

**Problem solved:** parallel calls, retries, approval resumes, and replay can otherwise pair a result with the wrong request. The common solution is a call ID carried from model output through tool execution into the observation. LangGraph uses `tool_call_id` on `ToolMessage`; agent-framework requires `function_call.call_id` to match `function_result.call_id`; OpenAI uses a universal call/output index; OpenHands documents the same pairing between `ActionEvent.tool_call_id` and `ObservationEvent.tool_call_id`. `[langgraph]` creates correlated success/error messages (`libs/prebuilt/langgraph/prebuilt/tool_node.py:922-1012`); `[agent-framework]` enforces the call/result relationship (`python/packages/core/agent_framework/_tools.py:1547-1584`); `[openai-agents-sdk]` indexes call/output item types (`src/agents/run_internal/items.py:23-31`); `[openhands]` defines the event pairing contract (`frontend/src/types/v1/core/events/observation-event.ts:20`).

**Why it works:** the ID is independent of message ordering and provider-specific function names. It gives retries, approval handling, event consumers, and replay code a join key. **Copy it when:** the system permits parallel calls, asynchronous tools, or resume after interruption. **Avoid or simplify it only when:** there is a genuinely single synchronous action with no persistence requirement. **Evidence limitation:** OpenHands documents the pair but does not expose the code that mints or consumes the ID inside the agent loop.

### Pattern 2: Make the cadence an explicit state transition, even if the public API is imperative

**Problem solved:** an implicit while-loop tends to mix parsing, execution, error handling, history mutation, and termination. LangGraph's graph edge, Pydantic AI's node classes, Letta's `_step` plus outer `step()` loop, and the OpenAI SDK's turn-resolution/tool-planning split each make the transition boundaries inspectable. `[letta]` isolates one LLM call plus tool execution per `_step` and applies outer limits in `step()` (`letta/agents/letta_agent_v3.py:222-441`, `letta/agents/letta_agent_v3.py:895-1592`); `[openai-agents-sdk]` separates response classification, planning, execution, and replay (`src/agents/run_internal/turn_resolution.py:1551-2001`, `src/agents/run_internal/tool_planning.py:542-682`).

**Why it works:** each boundary can own a contract: model output classification, tool execution, observation construction, and continuation. **Copy it when:** adding approval, streaming, retries, or persistence. **Overkill when:** a one-shot tool call has no continuation. **Failure mode:** the graph or state machine can become a large orchestration surface. CrewAI's new executor is a 3206-line Flow and overlaps with its deprecated executor, illustrating how explicit cadence can still suffer from code duplication (`lib/crewai/src/crewai/experimental/agent_executor.py:164`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:98`).

### Pattern 3: Use text ReAct as an adapter, not as the canonical internal representation

**Problem solved:** some providers cannot perform native tool calls. CrewAI shows a practical compatibility path: prompt for textual `Thought`/`Action` blocks, parse them, repair common JSON errors, and fall back when native tools are rejected. Its native path still converts provider-specific outputs into one assistant/tool wire shape (`lib/crewai/src/crewai/utilities/agent_utils.py:1191-1264`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:843-866`).

**Why it works:** weak or legacy providers can participate without forcing the rest of the harness to understand prose. **Copy it when:** provider coverage is a hard requirement. **Do not copy it as the primary path when:** native structured calls are available. Exact marker order and stop words are brittle, and parser errors can consume iterations (`lib/crewai/src/crewai/agents/constants.py:18-22`, `lib/crewai/src/crewai/utilities/agent_utils.py:660-695`). Keep the parser pure and heavily tested as CrewAI does (`lib/crewai/tests/agents/test_crew_agent_parser.py:12-145`), then immediately normalize into the same typed action model used by native calls.

### Pattern 4: Return failures as model-visible observations with typed status and bounded retry

**Problem solved:** a failed tool must teach the next model step what happened without turning every recoverable error into an opaque process failure. LangGraph returns `ToolMessage(status="error")`; Letta packages `ToolExecutionResult(status="error")` or `ToolReturn(status="error")`; Pydantic AI returns `RetryPromptPart`; agent-framework returns `function_result` with an exception and error payload; CrewAI appends error text in the observation. `[langgraph]` supports configurable error handlers (`libs/prebuilt/langgraph/prebuilt/tool_node.py:383-391`, `libs/prebuilt/langgraph/prebuilt/tool_node.py:749-754`); `[letta]` normalizes rule violations and denials into failed returns (`letta/agents/helpers.py:501-505`, `letta/server/rest_api/utils.py:230-264`); `[pydantic-ai]` formats validation and `ModelRetry` failures as retry parts (`pydantic_ai_slim/pydantic_ai/tool_manager.py:183-191`, `pydantic_ai_slim/pydantic_ai/messages.py:1446-1468`).

**Why it works:** the model receives the failure in the same causal channel as a success and can correct arguments or choose another tool. **Copy it when:** failures are expected, recoverable, or caused by model-generated arguments. Add structured error categories, a safe detail policy, and a retry budget. **Do not assume all failures should be observations:** the OpenAI Agents SDK's function-tool runtime exceptions can abort the turn after sibling draining rather than create a normal tool output (`src/agents/run_internal/tool_execution.py:1480-1554`, `src/agents/run_internal/tool_execution.py:1610-1623`). Use fail-fast behavior for infrastructure or security failures, but make that distinction explicit.

### Pattern 5: Separate per-tool retry budgets from global turn/output budgets

**Problem solved:** one malformed call should not consume all retries for the entire agent, while an agent that produces no actionable output still needs a global recovery limit. Pydantic AI provides the clearest split: `RunContext.retries[name]` and `tool.max_retries` are distinct from `max_output_retries`; agent-framework separately counts model round trips, total function calls, and consecutive errors; Letta separates maximum steps from tool rules and provider retries. `[pydantic-ai]` implements the per-tool/global split (`pydantic_ai_slim/pydantic_ai/_run_context.py:60-77`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:162-180`); `[agent-framework]` exposes independent invocation settings (`python/packages/core/agent_framework/_tools.py:1344-1395`) and forces prose after consecutive errors (`python/packages/core/agent_framework/_tools.py:2248-2258`, `python/packages/core/agent_framework/_tools.py:2632-2644`).

**Why it works:** it prevents both infinite tool-error loops and premature termination caused by a single local failure. **Copy it when:** tools can fail independently or a batch contains multiple calls. **Failure mode:** unclear cap names and scopes cause operators to set one limit while another silently stops the run. Agent-framework has separate chat and harness iteration caps that share `DEFAULT_MAX_ITERATIONS` (`python/packages/core/agent_framework/_tools.py:90`, `python/packages/core/agent_framework/_harness/_loop.py:117-122`); LangGraph similarly combines `remaining_steps` with Pregel `recursion_limit` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:620-634`, `libs/langgraph/langgraph/pregel/main.py:3020-3022`). Name limits by what they count.

### Pattern 6: Preserve provider-native reasoning without making it the action parser

**Problem solved:** modern providers may return reasoning signatures, redacted content, summaries, or encrypted blobs that should survive a follow-up request but should not be confused with a tool decision. Letta, OpenAI Agents SDK, Pydantic AI, and agent-framework all provide first-class or typed reasoning content. Letta distinguishes reasoning, redacted, omitted, and summarized content (`letta/schemas/letta_message_content.py:261-320`); Pydantic AI preserves signatures in `ThinkingPart` (`pydantic_ai_slim/pydantic_ai/messages.py:1629-1678`); OpenAI replays raw `ReasoningItem` objects with an ID policy (`src/agents/items.py:451-459`, `src/agents/run_internal/items.py:308-321`).

**Why it works:** provider-specific reasoning can remain opaque, replayable, and optional while tool extraction stays deterministic. **Copy it when:** supporting reasoning models, replay, or provider cache/signature requirements. Treat it as hidden or opaque by default at the product boundary; preservation is not the same as user-visible rendering. **Alternative:** CrewAI's text `Thought:` is simpler and visible but loses the distinction between rationale and action and has no first-class reasoning type (`lib/crewai/src/crewai/agents/parser.py:131-146`). LangGraph similarly stores any reasoning text as ordinary `AIMessage.content` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:651`, `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:677-707`).

### Pattern 7: Bound observations before they become context debt

**Problem solved:** raw tool output grows faster than the model's useful context. The sources show three useful levels: per-result truncation, proactive history compaction, and provider/server-managed compaction. Letta applies a per-tool limit and a dynamic context-window share before messages enter the next step (`letta/utils.py:898-939`, `letta/agents/letta_agent_v3.py:143-153`). Agent-framework collapses older tool groups into a summary and composes that with token budgeting (`python/packages/core/agent_framework/_compaction.py:820-918`, `python/packages/core/agent_framework/_compaction.py:1307-1400`). OpenAI uses an opt-in `responses.compact` session and can delegate truncation to the Responses API (`src/agents/memory/openai_responses_compaction_session.py:160-234`, `src/agents/model_settings.py:120-122`). OpenHands configures an LLM summarizing condenser and emits condensation events (`openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`, `frontend/src/types/v1/core/events/condensation-event.ts:5-46`).

**Why it works:** it bounds the cost of repeated observations while preserving the newest causal steps. **Copy it when:** tools return logs, search results, files, browser output, or other high-volume data. Keep the latest call/result groups verbatim, mark summaries as summaries, and preserve call IDs and failure status. **Overkill or risky when:** the tool output is already small, exact bytes are required, or the summarizer can erase a safety-critical detail. LangGraph and Pydantic AI intentionally expose hooks rather than a built-in summarizer (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-413`, `pydantic_ai_slim/pydantic_ai/capabilities/process_history.py:31-38`); copy their extension seam if policy varies by application.

### Pattern 8: Validate replay and message ordering as runtime invariants

**Problem solved:** a cadence can appear correct in a live run and fail on resume, after compaction, or when a parallel call is missing. LangGraph checks that every AI tool call has a matching tool message before the next model invocation (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:243-271`, `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:651`). Pydantic AI places tool return/retry parts first in each model request (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2220-2245`). The OpenAI SDK drops orphaned function calls and their preceding reasoning items before replay (`src/agents/run_internal/items.py:98-179`).

**Why it works:** it turns hidden transcript corruption into a clear invariant violation before the model receives invalid context. **Copy it when:** sessions can be resumed, compacted, streamed, or persisted. **Failure mode:** cleanup that removes an orphaned action can also remove useful reasoning or make a partial turn look complete; the cleanup policy must be observable. `[agent-framework]` tests call/result pairing and forced final passes (`python/packages/core/tests/core/test_function_invocation_logic.py:75-88`, `python/packages/core/tests/core/test_function_invocation_logic.py:1047-1160`), showing the value of testing the transcript, not only tool bodies.

### Pattern 9: Make post-tool reflection explicit when the product needs evaluation

**Problem solved:** a tool result is not always enough to tell the model whether it has satisfied the user goal. CrewAI injects a `post_tool_reasoning` instruction after tool results, and agent-framework's optional harness judge returns structured `answered` plus `reasoning` feedback to the next agent iteration. `[crewai]` uses the reflection message (`lib/crewai/src/crewai/translations/en.json:17`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:801-806`); `[agent-framework]` uses `JudgeVerdict` and feeds its reasoning back (`python/packages/core/agent_framework/_harness/_loop.py:99-111`, `python/packages/core/agent_framework/_harness/_loop.py:198-206`).

**Why it works:** it distinguishes “the tool returned” from “the task is complete.” **Copy it when:** completion requires synthesis, multiple criteria, or a reviewer. **Overkill when:** the tool result is itself the final answer or the loop is a simple retrieval. Keep evaluator calls outside the tool loop to avoid recursion, as agent-framework does (`python/packages/core/agent_framework/_harness/_loop.py:154-160`).

## Key Differences

- **Action parsing:** agent-framework, LangGraph, Letta, OpenAI Agents SDK, and Pydantic AI rely on structured provider-native items. CrewAI supports both native and regex text parsing. OpenHands' app repository exposes the resulting event types but not the parser. The difference is primarily provider and repository-boundary strategy, not a simple quality ranking.
- **Where the loop lives:** LangGraph and Pydantic AI put it in an explicit graph; agent-framework puts the tool loop in the chat-client layer and an optional evaluator loop in middleware; Letta and OpenAI use stateful turn orchestrators; CrewAI has overlapping legacy and experimental executors; OpenHands places it in an external agent server. These choices reflect whether the framework optimizes for customizable routing, reusable provider clients, durable sessions, compatibility, or process isolation. `[agent-framework]` identifies `FunctionInvocationLayer` as a chat-client wrapper (`python/packages/core/agent_framework/_tools.py:2388-2478`); `[openhands]` identifies the app server as a proxy (`openhands/app_server/app_conversation/app_conversation_router.py:552-583`).
- **Observation representation:** LangGraph uses `ToolMessage`, agent-framework uses `function_result` content in a tool message, Letta uses JSON-wrapped `ToolReturn` data, Pydantic AI uses `ToolReturnPart`/`RetryPromptPart`, and the OpenAI SDK uses `function_call_output`-family input items. CrewAI often uses raw strings in an observation. Structured wrappers improve status, modality, and correlation; JSON or typed wrappers impose a parsing burden on smaller models. `[letta]` packages status, message, and time (`letta/system.py:150-168`); `[openai-agents-sdk]` emits typed function output items (`src/agents/items.py:782-824`); `[pydantic-ai]` builds typed return parts (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2042-2052`).
- **Reasoning semantics:** Letta, Pydantic AI, OpenAI Agents SDK, and agent-framework preserve provider-native reasoning as structured content. CrewAI makes `Thought:` visible in assistant text. LangGraph has no independent reasoning channel. OpenHands captures `thought`, `reasoning_content`, and `thinking_blocks` in events, but the studied app cannot show whether they return to the model. The important distinction is whether reasoning is a durable provider artifact, a prompt convention, ordinary assistant text, or only an event projection.
- **Failure policy:** Pydantic AI and agent-framework have explicit retry/error budgets. LangGraph converts selected errors into observations but does not automatically re-plan or count consecutive failures. Letta feeds many policy and execution failures back and has stop reasons and routing fallback. CrewAI feeds failures back but can continue until `max_iter`. The OpenAI SDK has configurable model-visible errors for some cases, while ordinary tool runtime exceptions can abort the turn. OpenHands exposes error fields but leaves retry policy outside the repository.
- **Context management:** Letta is proactive and multi-level. Agent-framework has reusable compaction strategies and harness defaults when token budgets are configured. OpenAI delegates to an opt-in OpenAI compaction session or server truncation. CrewAI compacts on context overflow. LangGraph and Pydantic AI expose caller hooks without a built-in implementation. OpenHands configures a separate condenser. The divergence follows library-vs-application constraints and provider capabilities.
- **Streaming and partial observations:** LangGraph provides a dedicated tool event channel and `ToolCallStream`; CrewAI supports multimodal observation bridging; OpenAI and Letta stream reasoning/tool lifecycle events, though Letta's tool result itself is generally emitted after completion; Pydantic AI assembles streamed model parts. No source establishes a universal policy that a model receives partial tool progress before the tool finishes.

## Tradeoffs

- **Structured calls vs textual ReAct.** Structured calls eliminate marker ambiguity, support typed arguments, and preserve correlation. They require provider support or adapter work. Text ReAct broadens compatibility and makes the prompt inspectable, but regex and stop-word errors can consume turns or misparse model output. CrewAI demonstrates the fallback benefit and its parser/stop-word risks (`lib/crewai/src/crewai/agents/parser.py:62`, `lib/crewai/src/crewai/agent/core.py:1062-1066`). Choose structured calls as the primary protocol and isolate text parsing at the boundary.
- **Graph ownership vs encapsulated loop ownership.** A graph makes every transition replaceable and enables custom retry, approval, and routing nodes. It pushes compaction and prompt preparation onto the application. An encapsulated client loop is easier to adopt and standardizes behavior, but its policies are harder to inspect and can be split across layers. LangGraph's hook-based design (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-413`) is the alternative to agent-framework's chat-client wrapper (`python/packages/core/agent_framework/_tools.py:2388-2478`). Use a graph when cadence customization is a product feature; use an encapsulated loop when a stable default is more valuable than per-application routing.
- **Raw observations vs summarized observations.** Raw results preserve exact evidence and are easy to replay, but can overwhelm context and expose irrelevant or sensitive output. Summaries bound tokens, but can omit a failure cause or action/result relationship. Letta's newest-call retention and dynamic cap, and agent-framework's old-group summarization, are safer starting points than summarizing every result immediately (`letta/agents/letta_agent_v3.py:143-153`, `python/packages/core/agent_framework/_compaction.py:820-918`). Preserve raw durable events separately even when the model receives a compact projection.
- **Error-as-observation vs fail-fast.** Feeding a validation, denial, or recoverable tool error back lets the model repair itself. Failing fast is safer for infrastructure, authorization, or corrupted state. Pydantic AI's `RetryPromptPart` and LangGraph's configurable `handle_tool_errors` offer model-visible recovery; the OpenAI SDK's exception propagation shows a fail-fast alternative (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:2010-2024`, `libs/prebuilt/langgraph/prebuilt/tool_node.py:749-754`, `src/agents/run_internal/tool_execution.py:1610-1623`). Classify errors explicitly rather than making one policy cover both.
- **Visible reasoning vs opaque preservation.** In-band `Thought:` text is easy to display and may help legacy models, but conflates explanation with control and may expose content unintentionally. Provider-native reasoning objects preserve signatures or encrypted content and can be replayed without promising user visibility, but downstream UIs must understand raw provider shapes. Letta's four reasoning content types and the OpenAI SDK's raw `ReasoningItem` replay illustrate the durability benefit and rendering cost (`letta/schemas/letta_message_content.py:261-320`, `src/agents/items.py:451-459`).
- **Parallel execution vs deterministic sequencing.** Parallel calls reduce latency and are supported by agent-framework, LangGraph, Letta, OpenAI, Pydantic AI, and CrewAI native paths. Sequencing simplifies dependency ordering, cancellation, and observation presentation. Pydantic AI exposes three modes (`pydantic_ai_slim/pydantic_ai/tool_manager.py:155-168`); LangGraph exposes v1 bulk and v2 fan-out (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:843-859`). If calls are independent, parallelize, but retain per-call IDs, deterministic aggregation rules, and a budget that applies to the entire batch.
- **Externalized loop vs local auditability.** A separate agent server enables reuse across clients and process isolation. It also makes the most important behavior invisible to the app repository and creates webhook delivery dependencies. OpenHands' typed event model is strong for product integration, while the source-level cadence remains unknown (`openhands/pyproject.toml:47-49` [sic: the report evidence uses `pyproject.toml:47-49`], `openhands/app_server/event_callback/webhook_router.py:453-522`). Externalize only with a versioned SDK contract, trace export, and a way to inspect the actual loop implementation.

## Decision Guide

- **Choose a structured provider-native loop** when the supported providers expose reliable tool-call items and the system needs replay, approvals, or parallel calls. Model it after agent-framework, Letta, OpenAI Agents SDK, or Pydantic AI; do not add a textual parser just to make the loop look ReAct-like.
- **Choose a graph-as-loop** when application-specific routing, human pauses, custom retry, or domain nodes are first-class requirements. LangGraph favors flexible graph wiring; Pydantic AI favors typed model/tool parts, validation, and explicit retry budgets.
- **Choose a hybrid native/text adapter** only when provider compatibility requires it. Keep CrewAI-like regex parsing behind a narrow adapter, test malformed output, and normalize immediately to the same action structure as native calls.
- **Choose durable step/checkpoint semantics** for long-running agents, user approvals, cancellation, or resume. Letta's step IDs, progression checkpoints, stop reasons, and compaction are the strongest reference in this set. `[letta]` records step lifecycle progression (`letta/agents/letta_agent_v2.py:941-996`) and explicit stop reasons (`letta/schemas/letta_stop_reason.py:9-22`).
- **Choose the OpenAI Agents SDK** when the product is intentionally built around the Responses API's typed output items, server conversations, and `responses.compact`. Its replay hygiene and item model are strong, but it is not a provider-neutral cadence core.
- **Choose an externalized event architecture** when execution must run in a sandbox or separate service. Copy OpenHands' event taxonomy and action/result IDs, but require access to the SDK loop, versioned event semantics, durable webhook replay, and per-step metrics before treating the cadence as auditable.
- **Set policy based on tool output shape:** exact small results can remain raw; logs, files, browser pages, and search results need per-result truncation and history compaction. Prefer a layered policy: cap one result, preserve the latest causal group, summarize older groups, and retain the full event log outside the model context.
- **Set policy based on failure class:** retry model-generated validation errors, expose denials as typed observations, circuit-break repeated failures, and fail fast on authorization, infrastructure, and transcript-integrity errors. Never rely on `max_iterations` alone as the error policy.

## Practical Tips

- Define one internal action schema and one internal observation schema with `call_id`, tool name, arguments, status, output, error class, truncation metadata, and timestamps; adapt provider messages at the boundary.
- Test the complete transcript for at least: action → success observation → next action; action → failure observation → corrected action; parallel calls; approval/denial; cancellation; and resume after persistence.
- Name budgets by scope: `max_turns`, `max_tool_calls`, `max_consecutive_tool_errors`, `max_retries_per_tool`, `request_limit`, and `recursion_limit`. Document whether a parallel batch counts per call or per model turn.
- Preserve the latest action/result groups verbatim during compaction. Summaries should state which calls they summarize and retain failure status; store the raw observations separately for audit and replay.
- Make reasoning retention configurable independently from user-visible rendering. Preserve provider signatures or encrypted fields when required, but do not automatically expose raw reasoning to users or tools.
- Treat observation formatting as a model-facing API. Keep success/error status explicit, serialize non-string output deterministically, and include safe enough detail for recovery without leaking injected state, secrets, or internal arguments.
- Add a consecutive-failure circuit breaker even if the model can retry. After a small configurable number of identical or related failures, force a final response, request human intervention, or switch to a safer tool policy.
- Expose cadence events separately from chat messages: model request, action parsed, tool started, tool finished, observation injected, retry, compaction, and termination. This supports metrics without requiring consumers to parse assistant text.
- Keep a pure parser test suite if a text fallback exists. Test marker ordering, markdown fences, malformed JSON, tool output containing `Final Answer:`, empty responses, and stop-word boundary cases.
- Verify the history invariant before every model call: every tool call has exactly one matching result unless it is represented as a deliberate pending approval or interruption.
- If a tool can emit large or progressive output, decide explicitly whether the model gets deltas, a final result, or only a summarized projection. Do not let the UI's streaming representation silently become the model's observation contract.

## Anti-Patterns / Caution Signs

- **Text parsing as the default when structured tool calls are available.** This creates avoidable marker, stop-word, and JSON-repair failure modes.
- **Raw observation accumulation with no proactive bound.** Overflow-triggered compaction is too late for a single oversized tool result and makes cost depend on failure timing.
- **A failed tool that disappears from the next model input.** The model cannot correct an unseen failure; this also makes the run look successful to downstream observers.
- **One global retry counter for all failure classes.** A malformed argument, a tool timeout, a provider outage, and a policy denial need different recovery paths.
- **Retrying indefinitely because errors are ordinary observations.** LangGraph and CrewAI demonstrate that model-visible errors alone do not guarantee the model will stop repeating the same call (`libs/prebuilt/langgraph/prebuilt/tool_node.py:383-391`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:343-367`).
- **Multiple loop limits with ambiguous names.** Separate counters are useful only when their scope and termination action are clear. Shared names such as `DEFAULT_MAX_ITERATIONS` make operations and debugging harder (`python/packages/core/agent_framework/_tools.py:90`, `python/packages/core/agent_framework/_harness/_loop.py:117-122`).
- **Compaction that breaks call/result pairs or drops failure status.** A shorter prompt that violates provider history rules is worse than a larger valid prompt. Use LangGraph validation and OpenAI orphan cleanup as models for replay hygiene (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:243-271`, `src/agents/run_internal/items.py:98-179`).
- **Treating a reasoning field as a public explanation by default.** Provider-native reasoning may be redacted, encrypted, omitted, or intended only for replay.
- **Parallel batches with per-iteration-only limits.** A single model response may contain more calls than the next iteration budget anticipates; enforce limits before launching the batch and define partial-failure semantics.
- **Externalizing the loop without an audit path.** Typed events do not substitute for access to the parser, continuation policy, compactor, and retry implementation. OpenHands is the clearest warning: the app repository observes the seam but cannot prove the cadence.
- **Duplicating the cadence in legacy and replacement executors.** Overlapping implementations drift in error handling, final-answer behavior, and message formatting. CrewAI's two executor paths show the maintenance risk (`lib/crewai/src/crewai/agents/crew_agent_executor.py:98`, `lib/crewai/src/crewai/experimental/agent_executor.py:164`).

## Notable Absences

- No source provides a universal, provider-neutral textual `Thought`/`Action`/`Observation` protocol as the preferred internal model. CrewAI alone maintains a substantial text fallback; the other visible implementations use provider-native structured outputs or externalize the parser.
- No source provides a universally enabled, provider-neutral observation compactor. Agent-framework, Letta, CrewAI, and OpenAI have concrete mechanisms, but their defaults and triggers differ. LangGraph and Pydantic AI expose extension points rather than an opinionated implementation.
- No source establishes a universal consecutive-error policy. Agent-framework has an explicit consecutive-error cap; Pydantic AI has per-tool/global retry budgets; other systems generally rely on max turns, recursion, tool rules, or caller hooks.
- No source demonstrates a complete, common model-facing partial-observation contract. LangGraph has a dedicated streaming channel and OpenAI/CrewAI expose rich stream or multimodal events, but the reports do not establish that partial tool output is fed back into reasoning before tool completion.
- No source proves that every preserved reasoning item is rendered consistently, persisted with full provenance, or safely compacted. Letta and Pydantic AI preserve typed reasoning; OpenAI preserves raw items; agent-framework has first-class types; LangGraph and CrewAI have weaker separation. Rendering and compaction behavior remain provider- or caller-dependent.
- The OpenHands report cannot establish the inner parser, scratchpad prompt, retry policy, or observation reinjection because those mechanisms live in external packages. Its 3/10 score reflects missing evidence in the studied repository, not a claim that the external implementation is intrinsically weak.
- Motivations for several design choices are not fully documented in the source reports. The comparative explanations above are fit-to-constraint interpretations grounded in observed mechanisms, not claims of author intent.

## Per-Source Notes

- **agent-framework — 7/10.** A two-layer design combines a provider-native chat-client tool loop with an optional harness judge loop. It has structured function calls/results, explicit error and iteration budgets, reasoning content, and compaction strategies. The main concerns are raw observations unless compaction is wired, provider-dependent reasoning, separate caps with the same name, and an experimental outer harness (`python/packages/core/agent_framework/_tools.py:2551-2698`, `python/packages/core/agent_framework/_harness/_loop.py:212-421`).
- **crewai — 7/10.** The strongest compatibility story: native tool calling is normalized across providers, with a tested text ReAct fallback, parser-error feedback, post-tool reflection, and overflow summarization. The costs are regex and stop-word fragility, free-text reasoning, no consecutive-error circuit breaker, and overlapping deprecated/new executors (`lib/crewai/src/crewai/agents/crew_agent_executor.py:330-595`, `lib/crewai/src/crewai/agents/parser.py:62`).
- **langgraph — 7/10.** The graph is the cadence: `agent` routes to `tools`, typed `ToolMessage` results return to state, and `remaining_steps` plus `recursion_limit` bound the cycle. Error handling and transcript invariants are strong, but compaction, error-driven re-planning, and repeated-failure policy are caller responsibilities (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:243-271`, `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:620-692`).
- **letta — 8/10.** A durable one-step loop with typed reasoning, explicit step/run IDs, uniform tool-return packaging, approvals, tool rules, truncation, proactive/reactive compaction, and LLM fallback routing. The main tradeoffs are JSON-wrapped observations, multiple concurrent loop generations, no dedicated ReAct trace, and provider-specific reasoning/rendering (`letta/agents/letta_agent_v3.py:895-1592`, `letta/agents/letta_agent_v3.py:1966-2036`).
- **openai-agents-sdk — 8/10.** A mature Responses-API item loop with typed dispatch, replayable run items, call/output indexing, streaming event parity, approval resume, sibling draining, and optional server-side compaction. It is tightly coupled to OpenAI item shapes, leaves reasoning opaque to convenience renderers, and can abort on runtime tool failures rather than returning a normal observation (`src/agents/run_internal/turn_resolution.py:1551-2001`, `src/agents/run_internal/items.py:98-179`).
- **openhands — 3/10.** The studied tree is an app-server and frontend boundary, not the agent loop. Its event model is unusually explicit—action, observation, errors, hooks, pauses, and condensation—but the parser, continuation policy, reasoning reuse, failure retry, and compaction replacement live in pinned external packages (`pyproject.toml:47-49`, `frontend/src/types/v1/core/events/index.ts:1-10`).
- **pydantic-ai — 8/10.** A typed graph cycle with validation-before-execution, three execution modes, `ToolReturnPart`/`RetryPromptPart`, per-tool/global retry budgets, comprehensive capability hooks, and preserved provider reasoning. Its main gap is no built-in provider-neutral compactor; history processing and compaction are capabilities or provider hooks (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1180-1298`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1537-1965`).

## Open Questions

- **OpenHands inner contract:** Which exact SDK component parses model output, injects observations, and decides whether `is_error`, timeout, or hook-blocked events cause a retry? The app-server report explicitly cannot determine this (`frontend/src/types/v1/core/events/observation-event.ts:6-21`).
- **Runtime failure semantics:** For the OpenAI Agents SDK, which tool failures are intentionally model-visible output items versus turn-aborting exceptions, and can applications configure that distinction without replacing the executor? The report documents both paths (`src/agents/run_internal/turn_resolution.py:1958-1997`, `src/agents/run_internal/tool_execution.py:1610-1623`).
- **Reasoning persistence and compaction:** Do compaction strategies preserve, summarize, or discard reasoning signatures and redacted content consistently across providers? The source reports identify preserved types and compaction paths but do not establish one cross-provider contract (`python/packages/core/agent_framework/_compaction.py:880-890`, `letta/agents/letta_agent_v3.py:2077-2118`, `src/agents/memory/openai_responses_compaction_session.py:375-401`).
- **Observation-size guarantees:** Which frameworks cap a single result before the model request, and which only compact after a context error? Letta has explicit dynamic caps; several others rely on hooks, overflow, or provider services. A unified single-result budget is not evident in the reports (`letta/agents/letta_agent_v3.py:143-153`, `pydantic_ai_slim/pydantic_ai/messages.py:1244-1297`).
- **Batch budget enforcement:** When a model emits many parallel calls, do iteration, tool-call, and retry budgets apply before launch, per call, or only on the next loop? The reports flag this as a concern in agent-framework and CrewAI, but no shared policy emerges (`python/packages/core/agent_framework/_tools.py:2636-2638`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:697-784`).
- **Repeated identical failures:** Which systems detect that the model is repeating the same invalid action, rather than merely counting total errors? The reports show counters, tool rules, or caller hooks, but no common semantic duplicate-action breaker.
- **Partial observation feedback:** LangGraph streams tool deltas to clients, but the reports do not show a standard path by which the model itself receives those deltas before the final `ToolMessage`. Letta explicitly awaits full tool execution before producing its observation (`letta/agents/letta_agent_v3.py:1822-1862`).
- **Event ordering and delivery:** OpenHands uses webhooks and event files, but ordering guarantees and replay/backfill behavior are not established. A production event-sourced cadence needs an explicit sequence or causal ordering contract (`openhands/app_server/event/event_service_base.py:145-197`).
- **Evaluator placement:** Agent-framework has a standalone judge loop, while CrewAI injects a post-tool reflection message. It remains open when an evaluator should run after every action, after a batch, or only at task boundaries, and how its cost should be included in turn budgets (`python/packages/core/agent_framework/_harness/_loop.py:154-206`, `lib/crewai/src/crewai/translations/en.json:17`).

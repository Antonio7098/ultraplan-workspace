## Executive Summary

Across the seven sources, **tool registration is universally explicit and per-instance**: every framework requires the caller to enumerate which tools are visible to a given agent or run, and none auto-discovers Python functions from imports. What differs is the **shape** of the registration surface and the **depth** of cross-cutting machinery built on top of it. Six of the seven sources converge on a single locally-executed tool class (`FunctionTool`, `BaseTool`, `Tool`, `Tool`, `FunctionTool`, `FunctionTool`, `Tool`) produced either by a decorator or by hand-instantiation. MCP is treated as a first-class tool source in six of seven; only `openhands` makes MCP the *only* local registration surface because the SDK toolset lives in an external pinned package.

A clear pattern emerges: **declarative registration beats registry mutation**, but the way tools compose, namespace, and survive state is where sources diverge most sharply. The two strongest implementations — pydantic-ai (9/10) and agent-framework (8/10) — both treat tools as values rather than members of a singleton, but they reach this in different ways: pydantic-ai via `AbstractToolset` + `WrapperToolset` composition (cross-cutting concerns as decorators on toolsets), and agent-framework via a `FunctionTool` dataclass with `approval_mode`, `kind`, and `max_invocations` baked into the leaf class. The weakest source, openhands (6/10), exposes only a `FastMCP` decorator surface and delegates the entire default toolset to a pinned SDK — informative for what is *not* in scope but limited as an architectural example.

Key cross-source findings:

- **All seven sources enforce name uniqueness somewhere in the call path**, but with different strength: pydantic-ai raises `UserError` at registration and again at `CombinedToolset` merge time; langgraph silently overwrites duplicates in `self._tools_by_name`; openai-agents-sdk raises `UserError` via `validate_function_tool_lookup_configuration` only during response conversion. The strictness of the validation is a stronger signal of durability than its mere presence.
- **MCP integration is the universal extension point.** Stdio / Streamable HTTP / SSE transports exist in every source that touches MCP, but only pydantic-ai (via `MCPToolset`), openai-agents-sdk (via `MCPServer*`), agent-framework (via `MCPTool` transport subclasses), letta (via `MCPServerManager`), and crewai (via `MCPToolResolver` / `MCPServerAdapter`) wrap remote tools in the same first-class surface as local function tools. langgraph ships no MCP adapter in this source tree; openhands uses MCP exclusively.
- **Decorator-based registration is the dominant user-facing API**, but the decorator's backing storage varies wildly. agent-framework's `@tool` returns a `FunctionTool` instance directly; pydantic-ai's `@agent.tool` returns the *original function* so the decorator is transparent; openai-agents-sdk's `@function_tool` returns a `FunctionTool`; crewai's `@tool` returns a `Tool` subclass. The "transparent decorator" pattern is the most ergonomic but the hardest to discover statically.
- **Per-agent tool isolation is universal**, but achieved by different mechanisms: explicit per-agent fields (agent-framework, openai-agents-sdk), keyed DI services (agent-framework .NET, hosted-builder pattern), per-instance state (crewai, pydantic-ai, langgraph), or DB-row scoping (letta). The DB-scoped approach (letta) is the only one where tool identity survives process boundaries by default; others must serialize the tool reference itself.
- **Cross-cutting behavior (approval, filtering, prefixing, defer-loading, retries) is implemented as either leaf-class attributes (agent-framework `approval_mode`, openai-agents-sdk `needs_approval`) or as wrapper composition (pydantic-ai `ApprovalRequiredToolset`, `FilteredToolset`, `PrefixedToolset`).** The wrapper composition pattern wins on extensibility but pays a cost in mental overhead and call-stack depth.
- **No source ships a first-class "global tool catalog" or registration endpoint** that would let an external tool list every tool installed in the runtime. Even openhands's `mcp_server.get_tools()` is FastMCP-internal, not framework-exposed.

## Core Thesis

> **Tool definition and registration in modern agent frameworks has converged on three architectural primitives — a locally-executed tool class, an MCP-driven remote tool source, and a decorator or factory that mints one from a function — but the frameworks diverge sharply on *where* tools live, *how* they compose, and *what* cross-cutting concerns are baked in vs. wrapped on.** The strongest models treat tools as values that participate in a typed composition graph (pydantic-ai's `AbstractToolset`, agent-framework's `FunctionTool`, openai-agents-sdk's `Tool` union), the middling models treat tools as instances attached to an agent via configuration lists (crewai, letta, langgraph), and the weakest model — openhands — delegates tool definition entirely to an external package and uses MCP as the in-tree registration surface. Across all seven, **registration is always explicit, names are always stable, and multi-agent tool isolation is always supported**, but the cost of these guarantees varies from "tested at merge time" to "raised at registration time" — and that distinction determines whether a tool bug surfaces during development or in production.

## Approach Models

The seven sources cluster into **five distinct approach models** for tool definition and registration. Each is a coherent design choice shaped by product shape, user needs, and operational constraints — not a quality ranking.

### Cluster A: Toolsets-as-values with wrapper composition

**Sources:** pydantic-ai (9).

Pydantic-ai anchors registration on `AbstractToolset` (`pydantic_ai_slim/pydantic_ai/toolsets/abstract.py:74-279`), a small port surface (`get_tools`, `call_tool`, `for_run`, `for_run_step`, `apply`, `visit_and_replace`, `id`, `label`). Every concrete toolset — `FunctionToolset`, `MCPToolset`, `ExternalToolset`, `DynamicToolset`, `OutputToolset` — implements this port, and cross-cutting behavior (`FilteredToolset`, `PrefixedToolset`, `RenamedToolset`, `ApprovalRequiredToolset`, `DeferredLoadingToolset`, `SetMetadataToolset`, `IncludeReturnSchemasToolset`, `CapabilityOwnedToolset`) lives as subclasses of `WrapperToolset` (`pydantic_ai_slim/pydantic_ai/toolsets/wrapper.py:14-74`). The agent assembles its per-step tool set via `_build_toolset_list` → `CombinedToolset` → `PreparedToolset` → optional `ToolSearchToolset` (`pydantic_ai_slim/pydantic_ai/agent/__init__.py:2591-2704`). Registration is decorator-based (`@agent.tool`, `@agent.toolset`), list-based (`Agent(tools=[...], toolsets=[...])`), or factory-based (`DynamicToolset(toolset_func)` evaluated per-run or per-step). Name collision raises `UserError` at registration (`FunctionToolset.add_tool`, `function.py:587-588`) and again at merge (`CombinedToolset.get_tools`, `combined.py:73-77`).

### Cluster B: Class-based local tool + decorator + first-class MCP

**Sources:** agent-framework (8), openai-agents-sdk (8), crewai (8).

These three frameworks all share a similar surface: a single locally-executed tool class (`FunctionTool`, `FunctionTool`, `BaseTool`), a decorator (`@tool`, `@function_tool`, `@tool`) that mints one from a typed function, and an MCP adapter that yields tools of the same class. What they differ on is the *shape* of the class and the *strictness* of name collision detection.

- **agent-framework** wraps Pydantic from `inspect.signature` + `Annotated[T, "desc"]` (`python/packages/core/agent_framework/_tools.py:484-514, 1043-1073`), supports `approval_mode` as a literal field on the leaf class, and exposes per-agent overrides via `Agent.run(tools=...)` merged by `_append_unique_tools` (`python/packages/core/agent_framework/_agents.py:111-118, 1258-1281`). MCP is `MCPTool` with transport subclasses and a `tool_name_prefix` knob.
- **openai-agents-sdk** uses a `Tool` union of eleven dataclass variants (`src/agents/tool.py:1355-1368`), with strict JSON schema enforcement by default and a dedicated `_tool_identity.py` module that turns (name, namespace) into a tagged lookup key (`src/agents/_tool_identity.py:36-411`). MCP tools become `FunctionTool`s with origin metadata preserved on `ToolCallOutputItem` (`src/agents/tool.py:270-322`, `src/agents/mcp/util.py:499-568`).
- **crewai** auto-registers subclasses via `__init_subclass__` into a `_TOOL_TYPE_REGISTRY` (`lib/crewai/src/crewai/tools/base_tool.py:109-112`), supports class / decorator / `CrewStructuredTool` factory, and resolves MCP tools lazily via `MCPToolResolver.resolve` (`lib/crewai/src/crewai/mcp/tool_resolver.py:45-640`) with a 5-min schema cache and per-server `tool_filter`.

All three enforce name uniqueness in the call path (pydantic-ai at registration, agent-framework via `_append_unique_tools`, openai-agents-sdk via `validate_function_tool_lookup_configuration`, crewai via `_select_tool` with fallback fuzzy match).

### Cluster C: Node-scoped registry with interceptor extension point

**Source:** langgraph (7).

LangGraph treats the `ToolNode` itself as the registry: `self._tools_by_name: dict[str, BaseTool]` is populated once at `__init__` (`libs/prebuilt/langgraph/prebuilt/tool_node.py:773-786`), and call-time tool resolution defers to interceptor hooks (`wrap_tool_call` / `awrap_tool_call`) that can swap in an unregistered `BaseTool` (`libs/prebuilt/langgraph/prebuilt/tool_node.py:1044-1067, 1354-1361`). Injection metadata (`InjectedState`, `InjectedStore`, `ToolRuntime`, custom `InjectedToolArg` subclasses) is frozen into a per-tool `_InjectedArgs` record at init (`tool_node.py:566-619, 1967-2030`), and the LLM cannot spoof injected values because `_inject_tool_args` strips them and re-adds trusted values (`tool_node.py:1422-1429`). The model is the most explicit-and-static of any source: no `add_tool` after `__init__`, no plugin scanner, no name uniqueness check (duplicates silently overwrite at `tool_node.py:784`). Dynamic tool addition is possible but only via the `wrap_tool_call` escape hatch, which is officially supported but functionally a workaround (`tests/test_tool_node.py:1913-1999`, issue `langchain-ai/langchain#35305`).

### Cluster D: DB-row-per-tool with pluggable executor dispatch

**Source:** letta (8).

Letta stores tools as `ToolModel` rows (`letta/orm/tool.py:17`) managed by a single `ToolManager` service (`letta/services/tool_manager.py:206`). A tool enters via four paths: HTTP `POST/PUT /v1/tools/`, server-startup or `POST /v1/tools/add-base-tools` upsert of `LETTA_TOOL_MODULE_NAMES` (`letta/constants.py:47-53`), `POST /v1/tools/mcp/servers/{name}/resync` (`letta/services/mcp_server_manager.py:284`), or lazy auto-repair on `list_tools_async` (`tool_manager.py:650-674`). Concurrency safety is achieved via `SELECT … FOR UPDATE` (`tool_manager.py:583-611`) and atomic `INSERT … ON CONFLICT (name, organization_id) DO UPDATE` (`tool_manager.py:285-370`). Per-org / per-project scoping is enforced by `OrganizationMixin` + `ProjectMixin` (`letta/orm/tool.py:17-61`); per-agent attachment goes through the `tools_agents` join table plus `tool_rules` and `tool_type` filters (`ToolRuleType` enum at `letta/schemas/enums.py:182-197`). This is the only source where tool identity survives process boundaries by default — tool identity *is* the row.

### Cluster E: MCP-first surface with externally-pinned SDK

**Source:** openhands (6).

OpenHands's in-tree repo contains no Python-level tool abstraction. The local registration surface is a single `FastMCP('mcp', mask_error_details=True)` instance (`openhands/app_server/mcp/mcp_router.py:43`) mounted at `/mcp` (`app.py:33,59`), with five `@mcp_server.tool()`-decorated async functions (`create_pr`, `create_mr`, `create_bitbucket_pr`, `create_bitbucket_data_center_pr`, `create_azure_devops_pr`, `mcp_router.py:147-487`) plus a Tavily proxy mount under the `tavily` namespace (`mcp_router.py:49-75`). The actual default toolset (terminal, file editor, browser, plan tools, sub-agents) is constructed by external `get_default_tools` / `get_planning_tools` factories in `openhands-tools==1.29.0` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:132-139, 1617-1625`). Per-conversation MCP configuration merges the system `default` server (pointing at `/mcp/mcp`) with user-defined `mcp_config.mcpServers` (`live_status_app_conversation_service.py:1219-1315`), with `mcp_config` wholesale-replaced on settings update (`openhands/app_server/utils/jsonpatch_compat.py:7`). This is a thin MCP plumbing layer over an external SDK; the value of the source for this dimension is its explicit boundary.

## Pattern Catalog

The patterns below appear in **multiple sources** or are **unusually clearly demonstrated** in a single source. Each is presented as a learnable pattern with the contexts in which it works and when it is overkill.

### Pattern 1: Decorator + direct-instantiation with shared class

**Problem:** Users want both function-decoration ergonomics and explicit construction for dynamic / data-driven tools, without two parallel APIs that don't compose.

**Demonstrated by:** agent-framework (`FunctionTool` + `@tool` overload set, `python/packages/core/agent_framework/_tools.py:1147-1192, 243-410`), openai-agents-sdk (`FunctionTool` + `@function_tool` overload pair, `src/agents/tool.py:381-470, 1850-2048`), crewai (`BaseTool` + `@tool` factory + `CrewStructuredTool.from_function`, `lib/crewai/src/crewai/tools/base_tool.py:103-198, 676-761`), pydantic-ai (`Tool` dataclass + `FunctionToolset(tools=[...])`, `pydantic_ai_slim/pydantic_ai/tools.py:441-671`, `toolsets/function.py:61-139`).

**Why it works:** Decorator ergonomics for the common case, direct construction for programmatic generation (e.g., tools built from config). The shared class means dispatch, validation, and serialization all see the same shape.

**When to copy:** Any framework where some tools are written as Python functions and others are generated (e.g., from MCP servers, from schema files, from config).

**When it is overkill:** Single-author toolsets where every tool is hand-decorated; direct construction without a decorator is fine.

**Caveat:** pydantic-ai's `@agent.tool` decorator returns the *original function* (`agent/__init__.py:2210-2315`) so decoration is transparent — the cleanest ergonomics but the hardest to discover statically.

### Pattern 2: Introspection-driven JSON schema from function signature

**Problem:** Hand-written JSON schemas drift from the Python signature; users want strict typing without writing a schema twice.

**Demonstrated by:** agent-framework (Pydantic model via `pydantic.create_model(inspect.signature(...) + typing.get_type_hints)`, `python/packages/core/agent_framework/_tools.py:484-514`), openai-agents-sdk (Pydantic model via `function_schema()` + `griffe` docstrings, `src/agents/function_schema.py:224-424`), pydantic-ai (`Tool.function_schema` inspects docstring format `google/numpy/sphinx/auto`, `pydantic_ai_slim/pydantic_ai/tools.py:246-253, 467-557`), letta (AST parsing + Google-style docstring, `letta/functions/schema_generator.py:409-499`), langgraph (delegates to langchain_core's schema generation).

**Why it works:** Eliminates schema drift; enables strict provider-side validation (OpenAI strict mode, Anthropic strict mode); documents each parameter in the source-of-truth place (the docstring).

**When to copy:** Any framework targeting strict-mode providers or where tools have ≥3 parameters.

**When it is overkill:** Single-parameter tools where the schema is trivial; declarative tools where the schema is the input.

**Caveat:** Schemas are frozen at construction (agent-framework `python/packages/core/agent_framework/_tools.py:484-514`); mutating the function later is not reflected. Docstring-style drift is a real failure mode — letta raises on missing type annotations or missing parameter descriptions (`letta/functions/schema_generator.py:457-465`); pydantic-ai's `require_parameter_descriptions=True` is opt-in.

### Pattern 3: First-class MCP via tool-class parity (not a separate subsystem)

**Problem:** MCP servers should be a peer of local function tools so that composition, filtering, and approval work uniformly.

**Demonstrated by:** agent-framework (`MCPTool` is opened at run time and yields `FunctionTool` instances, `python/packages/core/agent_framework/_mcp.py:333-530, 2312-2920`), openai-agents-sdk (`MCPUtil.to_function_tool` mints `FunctionTool`s with `ToolOrigin(type=MCP)`, `src/agents/mcp/util.py:499-568`), pydantic-ai (`MCPToolset(AbstractToolset)` — same port as any other toolset, `pydantic_ai_slim/pydantic_ai/mcp.py:1982-2540`), letta (`ExternalMCPToolExecutor` and `ToolCreate.from_mcp` produce a `ToolModel` row, `letta/services/tool_executor/mcp_tool_executor.py:22-87`, `letta/schemas/tool.py:145-169`), crewai (`MCPNativeTool` / `MCPToolWrapper` from `MCPToolResolver`, `lib/crewai/src/crewai/tools/mcp_native_tool.py:16-132`, `lib/crewai/src/crewai/mcp/tool_resolver.py:45-640`).

**Why it works:** The agent loop and the approval layer see one uniform surface; cross-cutting wrappers compose without special-casing remote vs local.

**When to copy:** Any framework that wants a single approval / observability / filtering model for all tools.

**When it is overkill:** Single-source tool environments (e.g., a CLI tool that wraps one binary); the wrapper machinery has a non-trivial footprint.

**Caveat:** MCP tool schemas may not be strict-mode-compatible; openai-agents-sdk falls back to non-strict on coercion failure (`src/agents/mcp/util.py:540-544`). agent-framework keeps a denylist of framework-named kwargs to strip from forwarded arguments (`python/packages/core/agent_framework/_mcp.py:88-98, 425-432`).

### Pattern 4: Prefix/rename on the wire, identity preserved internally

**Problem:** Two MCP servers can advertise the same tool name; the framework must rename on the wire to avoid collision but keep the original name available to internal dispatchers and audit logs.

**Demonstrated by:** pydantic-ai (`PrefixedToolset` rewrites `name` to `f'{self.prefix}_{name}'` for `get_tools` and restores it in `call_tool`, setting `ctx.tool_name` back to the original, `pydantic_ai_slim/pydantic_ai/toolsets/prefixed.py:11-41`; `RenamedToolset` does the same with an explicit map, `pydantic_ai_slim/pydantic_ai/toolsets/renamed.py:11-42`), agent-framework (`tool_name_prefix` on `MCPTool` with `_normalize_mcp_name` regex, `python/packages/core/agent_framework/_mcp.py:161-177`), openai-agents-sdk (`_safe_tool_name_part` + SHA1 suffix on collisions, `src/agents/mcp/util.py:60-61, 407-496`), crewai (`f"{server_name}_{tool_name}"` on `MCPNativeTool`, `lib/crewai/src/crewai/tools/mcp_native_tool.py:43`).

**Why it works:** Wire-side stability for the model, internal stability for the tool, and observable "which prefix came from where" for debugging.

**When to copy:** Any framework that allows multiple tool sources (MCP servers, plugins, multi-tenant registries) that might collide.

**When it is overkill:** Single-source / single-tenant environments where collision is impossible.

**Caveat:** Hashing on collision (openai-agents-sdk) is lossy and not human-readable; renaming makes the wire name unpredictable. The cleanest tradeoff is "prefix by source, never hash."

### Pattern 5: Per-run / per-call tool override via ContextVar (not agent mutation)

**Problem:** A user wants to test an agent with a different toolset, or layer in an extra tool, without reconstructing the agent object.

**Demonstrated by:** pydantic-ai (`_override_toolsets`, `_override_tools`, `_override_builtin_tools` as `ContextVar[_utils.Option[...]]`, `agent/__init__.py:544-570`; `agent.iter(..., toolsets=...)` and `agent.override(...)`, `agent/wrapper.py:298-352`), agent-framework (`Agent.run(tools=...)` and `_merge_options` special-casing the `tools` key with `_append_unique_tools`, `python/packages/core/agent_framework/_agents.py:111-118, 861-905`).

**Why it works:** No mutation of the agent object means no aliasing bugs in concurrent runs; nested `override` blocks compose correctly via `ContextVar`.

**When to copy:** Any framework with multi-agent orchestration or test harnesses.

**When it is overkill:** Simple single-run agents.

**Caveat:** agent-framework's `_merge_options` raises on duplicate names between default and override (`_tools.py:901-929`); this is correct behavior but can surprise callers expecting "override wins."

### Pattern 6: Cross-cutting concerns as wrapper composition

**Problem:** Approval, filtering, prefixing, and defer-loading are needed across many tool sources. Baking them into the leaf class bloats it; wrapping the entire tool registry adds depth.

**Demonstrated by:** pydantic-ai (`WrapperToolset` base + eight shipped wrappers, `pydantic_ai_slim/pydantic_ai/toolsets/wrapper.py:14-74`; `toolsets/AGENTS.md:5-7` documents the design).

**Why it works:** Adding a new cross-cutting concern is a new `WrapperToolset` subclass that overrides `get_tools` and/or `call_tool`; existing toolsets are untouched.

**When to copy:** Frameworks where users need to compose 2+ of {filter, prefix, rename, approval, defer, metadata, return-schema} per toolset.

**When it is overkill:** Single-axis concerns that can be a leaf-class field (e.g., agent-framework's `approval_mode` is fine as a field; a *stack* of approval policies needs wrappers).

**Caveat:** Wrapper depth adds call-stack layers and complicates stack traces; each wrapper should document its position. pydantic-ai's `apply` / `visit_and_replace` (`abstract.py:182-191`) lets agents inject configuration (e.g., `mcp_sampling_model`) into every leaf toolset without leaking wrapper internals.

### Pattern 7: Tool identity as a typed lookup key, not just a string

**Problem:** String tool names collide between namespaces (e.g., a user tool and a server-injected tool both named `search`). String equality across run boundaries is fragile.

**Demonstrated by:** openai-agents-sdk (`FunctionToolLookupKey = BareFunctionToolLookupKey | NamespacedFunctionToolLookupKey | DeferredTopLevelFunctionToolLookupKey`, `src/agents/_tool_identity.py:10-17`; `serialize_function_tool_lookup_key` for persistence, `_tool_identity.py:115-132`).

**Why it works:** The lookup key carries the kind (bare / namespaced / deferred top-level), so dispatch can `match` on it without parsing the string. Persisted state survives round-trips via the explicit kind.

**When to copy:** Frameworks with persisted run state (sessions, durable execution) where tool identity must survive process boundaries.

**When it is overkill:** Stateless agents where every run rebuilds its tool set.

**Caveat:** Requires a stable canonical name field on every tool; tool renames that change the lookup key break replay.

### Pattern 8: Name uniqueness enforced at registration AND merge

**Problem:** A user can register `f` and `g` independently without collision but combine them in one toolset and collide.

**Demonstrated by:** pydantic-ai (`FunctionToolset.add_tool` raises `UserError`, `toolsets/function.py:587-588`; `CombinedToolset.get_tools` raises again with `tool_name_conflict_hint`, `combined.py:73-77`), agent-framework (`_append_unique_tools` at agent construction AND at per-run merge, `python/packages/core/agent_framework/_tools.py:901-929`, `_agents.py:1258-1281`), openai-agents-sdk (`validate_function_tool_lookup_configuration` raises `UserError` for qualified-name and deferred-top-level collisions, `src/agents/_tool_identity.py:310-349`).

**Why it works:** Errors surface at the earliest possible point — at registration when possible, at merge when not — with actionable hints (`tool_name_conflict_hint` points at `PrefixedToolset`).

**When to copy:** Any framework that lets users combine tool sources dynamically.

**When it is overkill:** Static single-source toolsets where uniqueness is provable by construction.

**Caveat:** The error message must tell users *how* to resolve the collision, not just that it occurred. pydantic-ai's hint to wrap in `PrefixedToolset` is a model for this.

### Pattern 9: Per-agent tool attachment as DB rows with scoped uniqueness

**Problem:** Multi-tenant frameworks need per-org / per-project tool scopes so that two orgs can have a tool with the same name without collision.

**Demonstrated by:** letta (`(name, organization_id)` unique constraint with optional `project_id` filter matching `project_id == X OR project_id IS NULL`, `letta/orm/tool.py:30-32`, `letta/services/tool_manager.py:702-704`; `_atomic_upsert_tool_postgresql` with `ON CONFLICT`, `tool_manager.py:285-370`).

**Why it works:** DB-level enforcement is race-free; per-org scoping is the natural unit of multi-tenancy; per-project adds a second scope without breaking global tools.

**When to copy:** Server products with multi-tenancy.

**When it is overkill:** Single-tenant library frameworks.

**Caveat:** Requires DB migration discipline; adding a new scope dimension means a schema migration, not a config change.

## Key Differences

Where do sources diverge on a question of design philosophy rather than product shape?

- **Tool as data vs. tool as class instance.** Pydantic-ai treats a tool as a *value* (`Tool` dataclass + `ToolDefinition` dataclass + `ToolsetTool` dataclass, three layers, `pydantic_ai_slim/pydantic_ai/tools.py:441-895`, `toolsets/abstract.py:42-71`). Agent-framework, openai-agents-sdk, and crewai treat a tool as a *class instance* (one `FunctionTool` / `FunctionTool` / `BaseTool` per tool). The "value" approach wins on serialization and composition; the "instance" approach wins on extensibility (subclassing).

- **Decorator transparency.** Pydantic-ai's `@agent.tool` returns the original function (`agent/__init__.py:2210-2315`); agent-framework's `@tool` returns a `FunctionTool` instance (`python/packages/core/agent_framework/_tools.py:1147-1192`); crewai's `@tool` returns a `Tool` instance. Transparency is more ergonomic but breaks static introspection; returning the tool is less ergonomic but enables `len(agent.tools)`-style enumeration.

- **MCP as a tool vs. MCP as a server.** Pydantic-ai models MCP servers as *toolsets* (`MCPToolset(AbstractToolset)`); openai-agents-sdk models MCP servers as a *field on Agent* (`Agent.mcp_servers`); agent-framework models MCP servers as *tool instances* (`MCPTool`) that produce `FunctionTool` children. The "toolset" framing composes with all wrappers; the "field on Agent" framing is simple but per-agent; the "tool instance" framing is the most permissive (an MCP server can be passed anywhere `tools=` is accepted).

- **Schema source.** Pydantic-ai builds schema from `inspect.signature` + `Annotated` + docstring (`tools.py:467-557`); letta builds it from `inspect.signature` + Google-style docstring with strict validation (`letta/functions/schema_generator.py:409-499`); agent-framework adds `Annotated[T, "desc"]` → `Field(description=...)` mapping (`_tools.py:1043-1073`); langgraph delegates to langchain_core. Strict validation (raising on missing descriptions) is the strongest — it catches drift at registration time.

- **Approval as attribute vs. wrapper.** Agent-framework uses `FunctionTool.approval_mode` (literal field) + `ToolApprovalMiddleware`; openai-agents-sdk uses `FunctionTool.needs_approval` (callable field); pydantic-ai uses `ApprovalRequiredToolset` (wrapper). The leaf-attribute approach is simpler; the wrapper approach composes with per-tool filtering and per-step state.

- **Discovery surface.** MCP, `is_enabled` (openai-agents-sdk, agent-framework), `wrap_tool_call` (langgraph), `DynamicToolset` (pydantic-ai), `MCPToolResolver` (crewai), `_import_all_tools` (openhands, not a registry). The deepest is pydantic-ai's three-layer (factory per run / factory per step / capability-contributed); the shallowest is openhands's "import for deserialization side effect."

- **Tool versioning.** No source ships a first-class `version` field on a tool. openai-agents-sdk's `_tool_identity.py` is the closest (the lookup key is a typed tuple, so renames are detectable). Letta stores `source_code` separately from `json_schema` and requires `def <new_name> in source_code` to rename (`tool_manager.py:946-966`); this is the only source where rename is gated by source-code introspection.

## Tradeoffs

### T1: Per-agent list vs. global registry

- **Benefit of per-agent list:** explicit, auditable tool scopes; no surprise tools from a shared global; test isolation is trivial.
- **Cost:** cross-agent reuse is by reference equality (or shallow copy); no way to enumerate "all tools installed anywhere" without walking every agent.
- **Best-fit context:** library frameworks where users construct their own agents.
- **Failure mode:** a user wanting to know "what tools can this conversation call" must trace the call path: `Agent.tools` → per-run override → `ContextProvider.before_run` → `MCPTool._functions` (agent-framework, `python/packages/core/agent_framework/_agents.py:1252-1281`). DevUI's `_extract_tools_from_object` (`python/packages/devui/agent_framework_devui/_discovery.py:867-938`) is a best-effort workaround.
- **Alternative:** letta's DB-row model makes "what tools can this conversation call" a single query, but trades that for DB coupling.

### T2: Schema from signature vs. schema as input

- **Benefit of schema from signature:** no drift; strict-mode compatible out of the box.
- **Cost:** mutating the function later doesn't update the schema (agent-framework `python/packages/core/agent_framework/_tools.py:484-514`); forward references are best-effort (`_tools.py:498-501`); missing parameter descriptions surface as errors (letta `letta/functions/schema_generator.py:464-465`).
- **Best-fit context:** strict-mode-targeting frameworks; large tool catalogs.
- **Failure mode:** docstring-style drift; complex Pydantic models with forward references fail silently in agent-framework.
- **Alternative:** letta accepts both signature and explicit schema, with AST parsing as the fallback (`letta/services/tool_schema_generator.py:18-100`); agent-framework accepts `schema=...` Pydantic model or raw JSON-schema dict (`_tools.py:1218-1222`).

### T3: Tool as leaf-class attribute vs. toolset wrapper for cross-cutting concerns

- **Benefit of leaf-class attribute:** simplicity; no extra layer; predictable behavior.
- **Cost:** every cross-cutting concern adds a field; combining 3+ concerns (approval + defer + filter) bloats the dataclass.
- **Best-fit context:** small cross-cutting surface (1-2 axes).
- **Failure mode:** agent-framework's `approval_mode` is Python-only; .NET uses `ApprovalRequiredAIFunction` wrapper — the two systems don't share semantics (`python/packages/core/agent_framework/_harness/_tool_approval.py:87-220` vs `dotnet/src/Microsoft.Agents.AI.Tools.Shell/DockerShellExecutor.cs:266-293`).
- **Alternative:** pydantic-ai's wrapper composition handles 3+ axes cleanly but adds call-stack depth and 8+ small types.

### T4: Decorator returns function vs. tool

- **Benefit of returning function:** transparent; `my_func` is still callable normally; type checkers see the original signature.
- **Cost:** no static introspection; `len(agent.tools)` requires walking all attributes; tool metadata is on a side-table.
- **Best-fit context:** ergonomic-first frameworks.
- **Failure mode:** users can't `for tool in agent.tools` because decoration doesn't expose the tool.
- **Alternative:** agent-framework's `@tool` returns the `FunctionTool` so `agent.tools` is directly enumerable.

### T5: Strict name uniqueness at registration vs. silent overwrite vs. fuzzy match

- **Benefit of strict-at-registration:** bugs surface at construction time, not in production.
- **Cost:** users must resolve collisions manually; `from_two_packages` requires explicit prefixing.
- **Best-fit context:** production-grade frameworks where reliability matters.
- **Failure mode:** langgraph silently overwrites duplicates in `_tools_by_name` (`tool_node.py:784`); crewai's fuzzy match at ratio 0.85 (`lib/crewai/src/crewai/tools/tool_usage.py:759-774`) can mask typos.
- **Alternative:** prefix on the wire, identity preserved internally (Pattern 4).

### T6: Tool persistence as DB row vs. instance field

- **Benefit of DB row:** tool identity survives process boundaries; per-org scoping is natural; cross-version migration is possible (letta's `ToolModel` with `source_code`, `json_schema`, `metadata_`).
- **Cost:** DB coupling; schema migrations for new tool types; latency on every tool read.
- **Best-fit context:** server products with multi-tenancy.
- **Failure mode:** renaming a built-in requires an `update_tool_by_id_async` call gated by source-code introspection (`letta/services/tool_manager.py:946-966`); tag-clearing has a documented bug (`tool_manager.py:334-339`).
- **Alternative:** instance-field model is faster but loses identity across process boundaries.

### T7: MCP as first-class tool vs. MCP as separate subsystem

- **Benefit of MCP-as-tool:** uniform approval, filtering, observability; agents don't need to know which tools are remote.
- **Cost:** schema-validation gaps (openai-agents-sdk falls back to non-strict, `src/agents/mcp/util.py:540-544`); per-call connection cost (crewai's `MCPNativeTool` opens a fresh client per call, `lib/crewai/src/crewai/tools/mcp_native_tool.py:73-99`); MCP allowlist vs. denylist complexity (agent-framework `_MCP_FRAMEWORK_DENYLIST`, `_mcp.py:88-98`).
- **Best-fit context:** any framework with non-trivial tool mix.
- **Failure mode:** langgraph ships no MCP adapter in this source tree; users must build one externally.
- **Alternative:** openhands's "MCP is the only local surface" is viable when the rest of the toolset is delegated to a pinned SDK.

## Decision Guide

### If you are building a library / framework

1. **Default to a single locally-executed tool class.** `FunctionTool` (agent-framework, openai-agents-sdk) or `Tool` (pydantic-ai) is the right primitive. Subclassing and decorator support fall out naturally.
2. **Build the JSON schema from the function signature** with `inspect.signature` + `Annotated` metadata + docstring parsing (Pydantic model generation). Accept an explicit `schema=...` override for dynamic tools.
3. **Make MCP a toolset, not a server.** Tools from MCP servers should be indistinguishable from local tools at the dispatch layer. Use a prefix on the wire, identity preserved internally (Pattern 4).
4. **Enforce name uniqueness at registration AND at merge.** Use two-phase validation: registration-time for single-source toolsets, merge-time for cross-source combinations. Provide an actionable error message that points at `PrefixedToolset` or equivalent.
5. **Pick one of three cross-cutting models:**
   - **Leaf-attribute model** (agent-framework's `approval_mode`): simple, but doesn't compose.
   - **Wrapper model** (pydantic-ai's `WrapperToolset`): composes, but adds depth.
   - **Middleware model** (agent-framework's `ToolApprovalMiddleware`): composes with state, but harder to test in isolation.
6. **Support per-run tool overrides via `ContextVar`** so test harnesses and orchestration layers can layer in extra tools without mutating the agent.
7. **Lazy-import MCP** to keep the import-time footprint small (openai-agents-sdk `src/agents/mcp/__init__.py:32-83`).

### If you are building an application

1. **Start with decorator-based registration.** `@agent.tool` (pydantic-ai) or `@tool` (agent-framework) is the lowest-friction entry point.
2. **If you need cross-cutting policy (approval, filtering), wrap rather than subclass.** `ApprovalRequiredToolset` (pydantic-ai) is more flexible than a leaf-attribute approach.
3. **For MCP, expose a per-conversation `mcp_servers` field** so users can attach / detach remote tools per call (agent-framework's `Agent(mcp_servers=[...])`, openhands's per-conversation merge).
4. **Reserve a namespace per tool source** (e.g., `notion_*`, `github_*`) and use `PrefixedToolset` to enforce it at the framework layer rather than at each call site.
5. **For multi-tenant deployments, model tools as DB rows** (letta's pattern) and scope by `organization_id` + optional `project_id`. Use atomic upsert with `ON CONFLICT` to handle concurrent registration.

### If you are evaluating a framework for adoption

- **Library / SDK use:** prefer pydantic-ai (9/10) for composability, openai-agents-sdk (8/10) for Responses-only hosted tools, or agent-framework (8/10) for dual Python/.NET.
- **Multi-agent / server product:** letta (8/10) for DB-scoped multi-tenancy, crewai (8/10) for class-based + decorator + MCP mix.
- **Graph-based workflows:** langgraph (7/10) — but note the silent overwrite on duplicate tool names and the `wrap_tool_call` workaround for dynamic tools.
- **MCP-heavy / thin app:** openhands (6/10) shows the pattern but the catalog is external; don't use it as your only source of architectural guidance.

## Practical Tips

### Patterns to copy

- **Decorator + class-based tool with schema-from-signature.** Pydantic-ai, agent-framework, openai-agents-sdk, crewai all converge here. It is the lowest-friction surface for users.
- **`CombinedToolset`-style merge with source-aware error messages.** Pydantic-ai's `tool_name_conflict_hint` points at `PrefixedToolset` (`pydantic_ai_slim/pydantic_ai/toolsets/abstract.py:106-108`) — a model for actionable collision errors.
- **Per-org / per-project tool scoping via DB constraints** (letta). The `(name, organization_id)` unique constraint is the cleanest multi-tenancy enforcement.
- **Atomic upsert with `ON CONFLICT`** for race-free tool registration (letta `tool_manager.py:285-370`). Library frameworks should at minimum guard against double-import double-registration.
- **Per-call MCP discovery with caching** (openai-agents-sdk `cache_tools_list=True`). Discovery is expensive; cache with explicit invalidation.
- **Tool origin metadata on output items** (openai-agents-sdk `ToolOrigin` on `ToolCallOutputItem`, `src/agents/tool.py:270-322`). Downstream consumers can distinguish function vs MCP vs agent-as-tool without re-parsing names.
- **Reserved-name guard for system tools** (pydantic-ai reserves `search_tools` for `ToolSearchToolset`, `toolsets/_tool_search.py:298-301`). Prevents collision between framework and user tools.

### Patterns to avoid or delay

- **Global mutable registry.** No source ships one; every source uses per-instance state. Don't add one — it makes tool identity murky and creates import-order dependencies.
- **Fuzzy tool-name matching** (crewai's `_select_tool` ratio 0.85, `lib/crewai/src/crewai/tools/tool_usage.py:759-774`). It feels forgiving but masks real bugs.
- **Silent overwrite of duplicate tool names** (langgraph `tool_node.py:784`). Surface a `UserError` instead.
- **Two parallel runtimes for tools** (crewai's `BaseTool._run` vs `CrewStructuredTool.invoke`). Pick one and migrate.
- **External SDK pin as the only source of tools** (openhands's `openhands-tools==1.29.0` pin, `pyproject.toml:60-62`). Pins change silently; users have no way to inspect what changed.
- **Auto-discovery from package layout.** Langgraph, agent-framework, and pydantic-ai all explicitly avoid this; openhands's `_import_all_tools` is a side-effecting import for deserialization, not registration (`openhands/app_server/event_callback/webhook_router.py:576-586`). Plugin-style discovery is an attack vector.

### Decision rules

- **Local function vs. hosted provider tool:** if the provider has a native wire-format tool (OpenAI `web_search`, Anthropic `web_search_20250305`), prefer the provider variant and add a framework-side `kind=` discriminator (agent-framework's `SHELL_TOOL_KIND_VALUE`, `python/packages/core/agent_framework/_tools.py:92`). Don't re-implement web search as a function tool.
- **MCP server vs. in-tree tool:** if the tool needs server-side credentials (GitHub, Slack), expose it as an MCP server on the app side (openhands's pattern) rather than handing the credential to the sandbox.
- **Decorator vs. direct instantiation:** use a decorator for hand-written functions; use direct construction for tools generated from config, MCP servers, or schema files.
- **Single tool class vs. typed union:** if you have ≥5 tool types with different wire shapes, use a union (openai-agents-sdk's `Tool = FunctionTool | WebSearchTool | ...`); otherwise a single class with a discriminator field is enough.
- **Cross-cutting as attribute vs. wrapper:** if you have 1 cross-cutting axis (approval), use a leaf field; if you have ≥3 axes (approval + defer + filter), use a wrapper (pydantic-ai's `WrapperToolset`).
- **Per-instance toolset vs. DB-row tool:** if your framework has multi-tenancy, use DB-row (letta); if it's a library, use per-instance fields.

### Caution signs that the tool layer is becoming brittle

- **Two or more validation points for the same concern** (e.g., name uniqueness at agent construction *and* at chat-client merge). Both must be kept in sync; one will drift.
- **Multiple ways to register the same tool** (class + decorator + factory + JSON config). Each adds a maintenance burden; collapse to one canonical path.
- **Tool identity as a string only.** Tool renames break replay; persisted state references tools by name with no version field. Adopt a typed lookup key (openai-agents-sdk `_tool_identity.py`).
- **Cross-runtime duplicate semantics** (agent-framework Python `approval_mode` vs .NET `ApprovalRequiredAIFunction`). If you ship two stacks, abstract the concept once.
- **MCP re-connects per call** (crewai's `MCPNativeTool`, `lib/crewai/src/crewai/tools/mcp_native_tool.py:73-99`). Each call pays connect/disconnect; pool aggressively once latency matters.
- **Fuzzy name matching** as a fallback. Replace with strict checks + actionable errors.
- **No audit log of registration events.** When a tool appears in the agent's list, you should be able to answer "where did this come from?" OpenAI's `ToolOrigin` metadata is a clean answer; openhands's lack of introspection is a cautionary tale.

## Anti-Patterns / Caution Signs

- **No name uniqueness check at any layer.** Langgraph's `ToolNode._tools_by_name` (`libs/prebuilt/langgraph/prebuilt/tool_node.py:784`) silently overwrites. Users only discover the bug when the wrong tool executes.
- **Fuzzy tool selection as a primary path.** Crewai's `_select_tool` ratio 0.85 (`lib/crewai/src/crewai/tools/tool_usage.py:768-774`) forgives typos in caller naming but masks production bugs.
- **Two parallel tool runtimes.** Crewai's `BaseTool._run` (legacy) + `CrewStructuredTool.invoke` (native) — usage counters must be mirrored (`lib/crewai/src/crewai/tools/structured_tool.py:373-377`). Drift is a real risk.
- **Hosted tools that are silently incompatible across backends.** Openai-agents-sdk raises `UserError` for hosted tools on Chat Completions (`src/agents/models/chatcmpl_converter.py:881-884`). At least it raises — silently dropping would be worse. Document the constraint at the tool definition.
- **Import-time registration side-effects.** Openhands's `_import_all_tools` (`openhands/app_server/event_callback/webhook_router.py:576-586`) walks `openhands.tools` to trigger Pydantic discriminator resolution. Future SDK relocations break this silently.
- **Captive dependencies in DI-registered tools.** Agent-framework's .NET stack guards against this via `ValidateToolLifetime` (`dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:146-156`). Other DI-based stacks should adopt the same check.
- **MCP discovery failure swallowed silently.** Crewai's `_get_mcp_tool_schemas` returns `[]` on failure (`lib/crewai/src/crewai/mcp/tool_resolver.py:511-519`); agents end up with no tools and no error. Fail loudly or document the silent-degrade contract.
- **MCP init swallows all exceptions.** Openhands's `init_tavily_proxy` catches `Exception` (`openhands/app_server/mcp/mcp_router.py:74-75`); operators only see a log line. Health check on startup is better.
- **Wholesale replacement of user config without per-server toggle.** Openhands's `WHOLESALE_REPLACEMENT_KEYS = frozenset({'mcp_config'})` (`openhands/app_server/utils/jsonpatch_compat.py:7`) — an empty `mcpServers: {}` wipes every previously-configured server.
- **Schema frozen at construction with no rebuild path.** Agent-framework's `_resolve_input_model` runs once at `@tool` time (`python/packages/core/agent_framework/_tools.py:484-514`); forward-reference resolution is best-effort with `try/except` (`_tools.py:498-501`).

## Notable Absences

What the seven sources collectively *do not* ship, and what that absence implies:

- **No source ships a "tool versioning" field.** Openai-agents-sdk's `_tool_identity.py` is the closest (the lookup key carries a kind), but no `version: str` is persisted on the tool itself. Renames are not gated by version. Letta's `metadata_["tool_hash"]` (`tool_manager.py:306-309`) is a hash for cache invalidation, not a public version field.
- **No source ships a "tool deprecation" workflow.** Deprecating a tool means editing every `Agent(tools=[...])` list manually. Agent-framework's future considerations (`agent-framework.md:248`) call this out explicitly.
- **No source ships a public API to enumerate "every tool installed in the runtime."** Per-agent lists are inspectable; cross-agent catalogs are not. Pydantic-ai's `toolsets/AGENTS.md` mentions observability tooling as future work; agent-framework's DevUI `_extract_tools_from_object` (`python/packages/devui/agent_framework_devui/_discovery.py:867-938`) is a best-effort walk.
- **No source ships plugin-style registration via Python entry-points.** Pydantic-ai's `AbstractNativeTool.__init_subclass__` (`native_tools/__init__.py:85-87`) is the closest — but subclasses must be defined at import time.
- **No source ships a generic "deferred-but-not-MCP" toolset.** Pydantic-ai's `ExternalToolset` is the closest (`toolsets/external.py:15-52`): declared `ToolDefinition`s whose execution happens outside the agent run. But there's no first-class "callback when this tool is called" hook.
- **No source ships a "tool approval policy at the agent level."** Approval is per-tool (agent-framework `approval_mode`, openai-agents-sdk `needs_approval`) or per-toolset (pydantic-ai `ApprovalRequiredToolset`); there is no global `Agent.requires_approval_for={"shell", "file_write"}` policy that crosses tool sources.
- **No source ships a "tool quota" field** (calls-per-hour, calls-per-conversation) beyond usage counters. Crewai's `max_usage_count` is per-tool-instance; letta's `LETTA_PARALLEL_SAFE_TOOLS` is a static allowlist; agent-framework's `FunctionInvocationConfiguration["max_function_calls"]` is per-call.
- **Langgraph ships no MCP adapter** (`langgraph.md:152` explicitly notes this is in `langchain-mcp-adapters` or `langchain.agents.create_agent`, not in this source).
- **Openhands ships no in-tree default tools** (`openhands.md:35-40`) — the default toolset lives in pinned `openhands-tools==1.29.0`.

## Per-Source Notes

### agent-framework

A dual-stack tool model (Python + .NET) centered on `FunctionTool` (Python) and `AIFunction` (.NET). The Python `@tool` decorator inspects signatures with `inspect.signature` + `typing.get_type_hints` and builds a Pydantic model via `pydantic.create_model` (`python/packages/core/agent_framework/_tools.py:484-514`); the .NET stack delegates to `Microsoft.Extensions.AI.AIFunctionFactory.Create`. Approval is `FunctionTool.approval_mode` (Python, literal) vs `ApprovalRequiredAIFunction` wrapper (.NET). MCP is `MCPTool` with `tool_name_prefix`, `_MCP_FRAMEWORK_DENYLIST`, sampling safety guardrails, and long-running task support. Per-agent tool lists are merged by `_append_unique_tools` at construction and again at per-run merge. DI-keyed tool registration in .NET with `ValidateToolLifetime` captive-dependency guard. No central registry; tools are discovered dynamically via MCP, in-run `add_tools` / `remove_tools`, and `ContextProvider.before_run`. Rating 8/10 — clear model with explicit interfaces, but Python/.NET approval parity is not yet achieved.

### crewai

Four layered surfaces: `BaseTool` (Pydantic v2 model + ABC) with `__init_subclass__` auto-registration, `@tool` decorator factory, `CrewStructuredTool` shim, and `MCPToolResolver` + `MCPServerAdapter`. Names are sanitized via `sanitize_tool_name` (NFKD, camelCase split, 64-char hash suffix) and may be fuzzy-matched at ratio 0.85. Tools enter via `Agent(tools=[...])`, `Task(tools=[...])`, `Agent(mcps=[...])`, or `Agent(apps=[...])`; the crew merges them in `_prepare_tools` with `_merge_tools` deduplication by sanitized name. Operationally rich: usage lock + `max_usage_count`, `cache_function`, `result_as_answer`, MCP retry/backoff with 5-min schema cache, AMP slug + HTTPS + native MCP resolver branches. Rating 8/10 — clear model with explicit interfaces and tests; the dual-runtime split and fuzzy match hold it back.

### langgraph

No global registry; registration is explicit, instance-scoped, and constructor-bound via `ToolNode(tools=[...])`. `_tools_by_name: dict[str, BaseTool]` is the authoritative set. Injection metadata (`InjectedState`, `InjectedStore`, `ToolRuntime`, custom `InjectedToolArg` subclasses) is frozen at init into `_InjectedArgs` records, and `_inject_tool_args` strips LLM-supplied values for any `all_injected_keys` before re-adding trusted ones. `wrap_tool_call` / `awrap_tool_call` interceptor hooks allow swapping in unregistered tools. `tools_condition` couples routing to the literal node name `"tools"`. Streaming via `StreamToolCallHandler` + `ToolCallTransformer`. Rating 7/10 — explicit and durable for injection security; brittle on dynamic tool addition (no `add_tool` after init) and silent overwrite of duplicates.

### letta

Tools are DB rows (`ToolModel`) managed by a single `ToolManager` service. Four entry paths: HTTP `POST/PUT /v1/tools/`, server-startup or `POST /v1/tools/add-base-tools` upsert of `LETTA_TOOL_MODULE_NAMES`, MCP `POST /v1/tools/mcp/servers/{name}/resync`, and lazy auto-repair on `list_tools_async`. Concurrency-safe via `SELECT … FOR UPDATE` and atomic `INSERT … ON CONFLICT`. Per-org `(name, organization_id)` uniqueness with optional `project_id` scope. `ToolExecutorFactory` dispatches by `ToolType`; sandbox execution opts into Modal / E2B / local via `metadata_["sandbox"]`. MCP servers are first-class (`MCPServerModel`) with health classification (`STRICT_COMPLIANT` / `NON_STRICT_ONLY` / `INVALID`). Rating 8/10 — clear model with transactional concurrency; bootstrap is multi-file edit-heavy and `function_map` dispatch duplicates name knowledge with the schema layer.

### openai-agents-sdk

Eleven-variant `Tool` union with a `@function_tool` decorator and a dedicated `_tool_identity.py` module that turns (name, namespace) into a tagged lookup key (`bare` / `namespaced` / `deferred_top_level`). Strict-mode JSON schema by default. MCP servers are `MCPServer*` classes (stdio / SSE / streamable HTTP) with `cache_tools_list`, `tool_filter`, and async-context-manager `MCPServerManager`. Lazy-import boundary at `agents.mcp.__init__`. `ToolOrigin` metadata on output items (`src/agents/tool.py:270-322`). Chat Completions rejects hosted tools with `UserError`. `is_enabled` callable runs every turn via `asyncio.gather`. `RealtimeAgent` lacks `mcp_servers` and `is_enabled`. Rating 8/10 — durable identity and clean decorator ergonomics; backend parity is the gap.

### openhands

A thin FastMCP surface: one `FastMCP('mcp', mask_error_details=True)` instance with five `@mcp_server.tool()`-decorated async functions for PR/MR creation across GitHub, GitLab, Bitbucket, Bitbucket DC, and Azure DevOps. Tavily proxy mounted under the `tavily` namespace. Default toolset is in pinned external `openhands-tools==1.29.0` and reached via `get_default_tools` / `get_planning_tools` factories. Per-conversation MCP config merges system `default` server with user `mcp_config.mcpServers`, wholesale-replaced on settings update. Rating 6/10 — informative for MCP plumbing but the catalog is external; not a complete architectural example.

### pydantic-ai

`AbstractToolset` is the only port (`get_tools`, `call_tool`, `for_run`, `for_run_step`, `apply`, `visit_and_replace`, `id`, `label`); concrete toolsets include `FunctionToolset`, `MCPToolset`, `ExternalToolset`, `DynamicToolset`, `OutputToolset`. Cross-cutting behavior is `WrapperToolset` subclasses (`FilteredToolset`, `PrefixedToolset`, `RenamedToolset`, `ApprovalRequiredToolset`, `DeferredLoadingToolset`, `SetMetadataToolset`, `IncludeReturnSchemasToolset`, `CapabilityOwnedToolset`). `CombinedToolset` merges children concurrently and raises `UserError` on collisions with `tool_name_conflict_hint`. Native tools register via `__init_subclass__` into `NATIVE_TOOL_TYPES`; the framework builds a `Discriminator`-tagged `Annotated[Union[...]]` from them. Per-run overrides via `ContextVar` (`_override_toolsets`, etc.). `ToolSearchToolset` exposes a `search_tools` function tool with reserved name. Tests cover composition (`test_comprehensive_toolset_composition`), tool conflict (`test_repeat_tool_by_rename`, `test_tool_name_conflict_hint`), retry budget inheritance, and capability defer-loading. Rating 9/10 — the most coherent and composable tool model in the study; the only source where cross-cutting concerns are first-class and tooling is genuinely extensible.

## Open Questions

- **Cross-agent tool catalog (observability).** No source ships a public API to enumerate "every tool installed in the runtime" across all agents / processes. Pydantic-ai's future considerations (`pydantic-ai.md:285`) flag this; agent-framework's DevUI does a best-effort walk. What is the right abstraction for this?
- **Tool identity beyond the name.** Openai-agents-sdk's `_tool_identity.py` is the closest to a versioned identity, but no source ships a true `version: str` field. How do persisted run states survive tool renames?
- **Cross-source uniqueness.** Letta enforces uniqueness only per `(name, organization_id)`, not per `(name, tool_type, organization_id)` (`letta/orm/tool.py:31`). A custom tool and an MCP tool can share a name within an org. Is that intentional?
- **Static vs. dynamic tool search.** Openai-agents-sdk's `ToolSearchTool(execution="client")` is documented as supported (`src/agents/tool.py:1338-1352`) but no client-side executor exists in the standard runner. Pydantic-ai's `ToolSearchToolset` does have a local keyword-overlap fallback. How should this gap be closed?
- **Realtime parity.** Openai-agents-sdk's `RealtimeAgent` lacks `mcp_servers` and `is_enabled` (`src/agents/realtime/agent.py:27`). What is the principled reason for the gap?
- **Hosted tools across backends.** Openai-agents-sdk raises `UserError` for hosted tools on Chat Completions (`src/agents/models/chatcmpl_converter.py:881-884`). Should a future pluggable converter let LiteLLM / AnyLLM backends express their own subsets?
- **Tool approval as agent-level policy.** All sources scope approval to per-tool or per-toolset; none ship an `Agent.requires_approval_for={...}` policy that crosses tool sources. Is there a clean way to express this?
- **Plugin-style registration.** Pydantic-ai's native-tool `__init_subclass__` is import-time only; runtime plugin registration is not supported (`pydantic-ai.md:283`). Is there a use case that justifies it?
- **Schema versioning.** No source stores schema version; `json_schema` is overwritten in place on update (letta `tool_manager.py:881-1086`). Is this a gap or a deliberate simplicity?
- **Tool-call concurrency and registration integrity.** Langgraph's `Send`-distributed execution (`chat_agent_executor.py:849-859`) allows parallel calls but the registration-side guarantee that two `BaseTool`s with the same `name` cannot coexist is unenforced (`langgraph.md:159`). Is this a real risk?

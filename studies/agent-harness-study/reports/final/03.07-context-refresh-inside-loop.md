## Executive Summary

The strongest implementations treat context refresh as a repeated, model-call-adjacent control loop rather than a one-time history cleanup. Agent Framework reassembles context through providers and applies composable, token-aware strategies before model calls; Letta reloads durable in-context IDs every turn and combines proactive and reactive compaction with structured events and post-compaction validation. These are the only two sources with a broadly portable, loop-owned policy rather than merely extension points or provider delegation. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1182-1419`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/letta_agent_v3.py:1218-1294, 1438-1505`)

OpenAI Agents SDK and Pydantic AI occupy a middle ground. Both refresh substantial derived context on each model call and expose strong rewrite seams, but generic long-context safety is opt-in. OpenAI Agents SDK offers session limits, per-call filters, tool-output trimming, provider truncation, and a rollback-capable compaction session; Pydantic AI offers `ProcessHistory`, pre-request usage limits, dynamic tool/capability refresh, and typed provider compaction parts. Neither generic runner supplies a mandatory, provider-neutral token-budget policy or automatic overflow-recompact-retry path. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/turn_preparation.py:51-85`, `src/agents/memory/openai_responses_compaction_session.py:185-306`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/capabilities/process_history.py:27-69`, `pydantic_ai_slim/pydantic_ai/models/openai.py:3995-4194`)

CrewAI has a working but coarse recovery path: an append-only task buffer is summarized wholesale only after a provider reports context overflow, and recovery is disabled by default. LangGraph intentionally delegates model-visible context policy to application code through a `pre_model_hook` and a separate `llm_input_messages` channel. OpenHands delegates the actual loop to an external SDK/agent-server, so its in-scope control plane exposes settings and usage metrics but cannot establish what is compacted or when. Their low scores reflect different causes: CrewAI implements a fragile policy, LangGraph supplies policy-neutral graph primitives, and OpenHands lacks runtime evidence inside the selected source boundary. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:698-749, 920-1003`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-424, 636-658`; [openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/app_conversation/live_status_app_conversation_service.py:415-441, 1635-1641`)

Across all sources, the largest shared weakness is summary faithfulness. Every semantic summarization path ultimately trusts an LLM or provider-generated compacted artifact without checking preserved facts against the discarded transcript. The best systems validate shape, budget reduction, atomic persistence, and fallback behavior, but none demonstrates semantic entailment, required-fact coverage, or drift detection. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:949-1088`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/compact.py:350-411`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/memory/openai_responses_compaction_session.py:404-496`)

## Core Thesis

A loop survives context-window growth when it maintains three distinct things: a durable canonical record, a deliberately rebuilt model-visible projection, and an explicit refresh policy executed immediately before model calls. The policy should reserve output space, protect atomic interactions, reduce low-value bulk before high-value conversational state, disclose omissions, and have both preventive and reactive triggers. Durable storage or token telemetry alone does not solve context pressure; what matters is whether measured pressure changes the next model input through a tested, observable path. This distinction is clearest in Letta's durable-ID rebuild plus dual triggers, Agent Framework's annotated projection and composed strategies, and the contrast between CrewAI's unused post-hoc token metrics and reactive whole-buffer replacement. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/helpers.py:148-317`, `letta/services/summarizer/thresholds.py:27-41`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:569-601, 1307-1419`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/token_counter_callback.py:37-66`)

## Approach Models

### 1. Loop-owned, policy-rich refresh

**Agent Framework and Letta** make context control a first-class part of the loop. Agent Framework uses annotated message groups and strategy composition: older tool-call groups are compacted at a lower pressure threshold, then older non-system groups are truncated at a higher threshold. It projects included messages for the model while retaining exclusion metadata in the canonical objects. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:120-243, 569-601, 1307-1419`)

Letta uses durable IDs as the active-context boundary. Every turn reloads the selected IDs, proactive pressure is checked after successful calls, reactive overflow causes compact-and-retry, and compaction rewrites the active ID set while preserving old rows for later search. Its policy is less strategy-composable than Agent Framework's but more operationally complete because it combines structured events, retries, fallback summarizers, provider fallbacks, and explicit system-prompt-overflow classification. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/helpers.py:148-317`, `letta/services/agent_manager.py:999-1018`, `letta/services/summarizer/compact.py:192-411`)

This model best fits long-lived autonomous agents where the framework, not each application, must guarantee bounded model input.

### 2. Extension-driven and provider-assisted refresh

**OpenAI Agents SDK and Pydantic AI** rebuild or normalize the model-facing request each turn but leave the generic pressure policy to configuration, capabilities, or the provider. OpenAI Agents SDK offers the broader local toolbox: latest-N session reads, a pre-call input filter, explicit tool-output previews, provider truncation, a compaction-aware session, and server-conversation deltas. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/session_persistence.py:54-187`, `src/agents/extensions/tool_output_trimmer.py:105-220`, `src/agents/run_internal/oai_conversation.py:417-510`)

Pydantic AI uses a smaller, more uniform capability model. `ProcessHistory` can rewrite each in-flight request; provider-native Anthropic/OpenAI compaction is represented by typed `CompactionPart`s; tool and capability discovery is recomputed from history. The cost of this generality is that a safe portable policy remains application code, and malformed slicing can separate tool calls from returns. ([pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/capabilities/process_history.py:27-69`, `pydantic_ai_slim/pydantic_ai/messages.py:1681-1727`, `docs/message-history.md:599-600, 661-662`)

This model best fits libraries that must stay provider-neutral and let applications choose whether history is trimmed, summarized, provider-compacted, or retained.

### 3. Reactive monolithic recovery

**CrewAI** builds one mutable message list per task and appends through the ReAct loop. It does not refresh retrieval or preflight a token budget. Only a recognized provider overflow error triggers `summarize_messages`; if enabled, that function preserves system messages and file attachments but replaces the complete non-system transcript with one summary. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/agents/crew_agent_executor.py:362, 432, 783, 806, 1082-1088`, `lib/crewai/src/crewai/utilities/agent_utils.py:920-1003`)

This design minimizes ongoing overhead and is understandable, but it discovers pressure too late and creates a large semantic discontinuity. It is adequate for bounded tasks with occasional overflow, not a strong foundation for indefinite sessions.

### 4. Policy delegation

**LangGraph** delegates context refresh by design. The runtime preserves reducer-managed graph state and exposes `pre_model_hook`, `RemoveMessage`, `REMOVE_ALL_MESSAGES`, and `llm_input_messages`; an application must supply token counting, summarization, retrieval refresh, and overflow behavior. Its `DeltaChannel` bounds checkpoint replay/storage cost, not the LLM prompt. ([langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/langgraph/langgraph/graph/message.py:38, 60-244`, `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-424, 636-658`, `libs/langgraph/langgraph/channels/delta.py:25-204`)

**OpenHands** delegates across a process and source boundary rather than through an in-process application hook. The control plane constructs and posts an SDK `Agent`, loads skills/hooks once, tags condenser usage, and stores returned metrics; the actual runtime policy is not available in the studied source. This is not equivalent to LangGraph's intentional user-programmable policy surface: it is an observability and evidence boundary. ([openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/app_conversation/live_status_app_conversation_service.py:415-441, 1352-1371, 1635-1641`)

## Pattern Catalog

### Pattern 1: Separate canonical history from the model-visible projection

**Problem solved:** shrinking a prompt without destroying auditability, replay, or later retrieval.

Agent Framework stores exclusion reasons and summary links on shared messages, then produces the model view through `project_included_messages`. Letta removes IDs from the active set and flags rows out of context while retaining original messages. LangGraph can leave `messages` intact while a `pre_model_hook` supplies `llm_input_messages`. OpenAI Agents SDK separately retains continuation, session, and result histories. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:569-601, 1067-1083`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/agent_manager.py:999-1018`, `letta/services/conversation_manager.py:752-797`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:636-658`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run.py:1284-1290`)

**Why it works:** destructive model-input decisions no longer erase forensic state. It also enables later recall, alternate readers, and rollback.

**Copy when:** sessions are durable, regulated, resumable, or expected to outlive one context window.

**Risk/overkill:** retaining every excluded record creates unbounded storage unless paired with archival, TTL, or purge. Letta explicitly retains old rows, and Agent Framework's history providers keep excluded messages unless configured to skip them. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/conversation_manager.py:752-797`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_sessions.py:814-890, 894-1134`)

### Pattern 2: Put the refresh seam immediately before every model call

**Problem solved:** tools, instructions, retrieved material, and history pressure can all change during a multi-tool turn.

Agent Framework reapplies compaction in `_prepare_messages_for_model_call`; OpenAI Agents SDK runs input filtering in streaming and non-streaming call preparation; Pydantic AI runs `before_model_request`, message preparation, and dynamic tool/prompt refresh each model request; LangGraph exposes a graph node directly before its prebuilt agent node. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_clients.py:363-391`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/run_loop.py:1370-1405, 1818-1847`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/_agent_graph.py:872-958`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:795-798, 876-879`)

**Why it works:** there is one auditable boundary where the final rendered request can be measured and rewritten.

**Copy when:** a single user turn can contain several model/tool iterations or dynamic tool availability.

**Risk/overkill:** repeated tokenization, normalization, or retrieval adds latency. Incremental annotations, identity checks, and cache-stable prefixes can reduce this cost, as shown by Agent Framework and Pydantic AI. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:392-490`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/_agent_graph.py:945-948`, `pydantic_ai_slim/pydantic_ai/messages.py:1542-1545`)

### Pattern 3: Use a staged pressure ladder

**Problem solved:** avoid spending semantic fidelity before cheaper reductions are exhausted.

Agent Framework first compacts old tool groups and later truncates old non-system groups. Letta clamps tool returns, attempts the configured sliding/all summarizer chain, hard-truncates the summarizer input if needed, and classifies irreducible system-prompt overflow. OpenAI Agents SDK's local tool trimmer similarly targets old oversized outputs while protecting recent turns. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1091-1160, 1307-1419`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/summarizer.py:562-647`, `letta/services/summarizer/compact.py:350-411`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/extensions/tool_output_trimmer.py:105-220`)

**Why it works:** large tool payloads commonly consume far more tokens per unit of future value than system instructions, current goals, unresolved decisions, or the newest interaction.

**Copy when:** agents read files, search corpora, or invoke tools that return bulky payloads.

**Risk/overkill:** a fixed role-based policy can delete the one old result containing an exact number or code fragment. Preserve retrievability and add visible omission markers.

### Pattern 4: Compact interaction units atomically

**Problem solved:** prevent invalid histories containing a tool call without its result, or vice versa.

Agent Framework explicitly groups reasoning, function calls, and tool results into atomic `tool_call` groups. Letta protects pending approval pairs and requires safe cutoff boundaries. OpenAI Agents SDK removes orphan calls and tied reasoning items during normalization. Pydantic AI lacks an enforcing helper and instead warns `ProcessHistory` users to preserve pairs, showing the failure mode when atomicity is not encoded. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:120-243`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/summarizer_sliding_window.py:133-137, 196-202`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/items.py:98-179`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `docs/message-history.md:599-600, 661-662`)

**Why it works:** compaction operates on protocol-valid units rather than arbitrary messages or token slices.

**Copy when:** native function calling, approvals, reasoning blocks, or provider-specific compaction parts are present.

**Risk/overkill:** defining groups is provider-sensitive and complicates cross-provider replay, but relying on documentation alone is more brittle.

### Pattern 5: Combine proactive and reactive triggers

**Problem solved:** estimates can be wrong, but waiting for provider rejection is too late.

Letta is the clearest implementation: it compacts after a successful step when usage exceeds 90% of the context window and also catches `ContextWindowExceededError` for bounded compact-and-retry. Agent Framework is strong on preventive budget strategies but has no reactive retry; CrewAI is the inverse, reacting only after a recognized error. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/thresholds.py:27-41`, `letta/agents/letta_agent_v3.py:1218-1294`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_clients.py:363-391`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/exceptions/context_window_exceeding_exception.py:4-13`)

**Why it works:** proactive control handles normal growth; reactive fallback catches tokenizer drift, provider differences, and hidden schema overhead.

**Copy when:** the loop must continue autonomously through changing providers or tool schemas.

**Risk/overkill:** retries can amplify cost or loop forever. Bound retries and detect irreducible system-prompt overflow, as Letta does. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/settings.py:96`, `letta/services/summarizer/compact.py:391-411`)

### Pattern 6: Make omission and compaction visible

**Problem solved:** the model and operators otherwise cannot distinguish “never happened” from “was removed.”

Letta inserts a summary notice that prior messages were hidden and exposes `CompactionStats` with trigger and before/after counts. OpenAI Agents SDK's tool trimmer emits an explicit preview marker with original length. Agent Framework keeps bidirectional source-summary IDs and exclusion reasons, but does not give the model a canonical omission manifest. CrewAI tells the model that a structured summary replaced prior context but does not enumerate loss. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/system.py:207-224`, `letta/schemas/letta_message.py:406-449`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/extensions/tool_output_trimmer.py:208-220`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:33-35, 582-588, 1067-1083`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/translations/en.json:27`)

**Why it works:** the model can choose to retrieve, re-run a tool, or qualify an answer, while operators can correlate changed behavior with context loss.

**Copy when:** omissions can affect correctness or user trust.

**Risk/overkill:** detailed manifests consume tokens and may expose sensitive metadata. Prefer compact counts, categories, retrieval handles, and stable IDs over replaying removed content.

### Pattern 7: Keep retrieval refresh separate from transcript compaction

**Problem solved:** a summary should not become the only route to old facts.

Agent Framework refreshes RAG/memory providers per run and exposes small indexes plus deeper tools. Letta keeps old messages searchable through conversation and archival tools. Pydantic AI rebuilds discovered tools/capabilities from history, while OpenAI Agents SDK refreshes MCP/tool availability each turn. CrewAI's memory and knowledge retrieval are injected only before the loop, so compaction cannot automatically reground the continuing task. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/azure-ai-search/agent_framework_azure_ai_search/_context_provider.py:580-609`, `python/packages/core/agent_framework/_harness/_memory.py:1161-1325`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/functions/function_sets/base.py:87-130`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1431-1456`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/agent.py:224-266`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/agent/core.py:583-646`)

**Why it works:** active context can stay small while exact old evidence remains recoverable.

**Copy when:** the corpus or transcript is durable and retrieval latency is acceptable.

**Risk/overkill:** model-initiated retrieval may never happen, while automatic retrieval can repeatedly inject stale or irrelevant material. Define when retrieval reruns, how results expire, and whether compaction triggers a mandatory regrounding step.

## Key Differences

### Context assembly: rebuild versus append

- **Durable rebuild:** Letta rereads active message IDs every public turn, making storage the source of truth and allowing compaction to change the next assembly deterministically. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/helpers.py:148-317`)
- **Fresh model projection over evolving local state:** Agent Framework assembles provider contributions and compacts before service calls; OpenAI Agents SDK reconstructs normalized input from original input plus generated items. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_sessions.py:315-348`, `python/packages/core/agent_framework/_clients.py:363-391`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/run_loop.py:280-287`)
- **Append-only raw history with refreshed derived context:** Pydantic AI keeps `message_history` but reruns capabilities, tool resolution, prompt evaluation, and normalization. ([pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/_agent_graph.py:854-958, 1431-1456`)
- **Append-only buffer:** CrewAI passes the same growing list until whole-buffer summarization. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/experimental/agent_executor.py:1486`)
- **Reducer state:** LangGraph accumulates channels across supersteps; rebuilding model context is user code. ([langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/langgraph/langgraph/pregel/_algo.py:232-345`, `libs/langgraph/langgraph/pregel/_io.py:23-53`)
- **Unknown runtime:** OpenHands builds control-plane context once; SDK behavior is outside the source. ([openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1635-1641`)

### Eviction order

There is no industry-wide priority, but the strongest policies prefer targeted, oldest-first reduction before broad loss. Agent Framework compacts old tool groups before old non-system groups; Letta's default sliding mode summarizes the oldest prefix at safe boundaries and separately clamps large tool returns; OpenAI's local trimmer targets old oversized tool outputs. CrewAI replaces all non-system content at once. LangGraph and Pydantic AI drop nothing unless user code says otherwise, while OpenHands cannot establish the SDK policy. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:820-918, 1307-1419`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/summarizer_sliding_window.py:152-198`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/extensions/tool_output_trimmer.py:105-147`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:995-1003`)

### Summarization faithfulness

No semantic summarizer is verified for fact preservation. Agent Framework adds excellent lineage metadata; Letta verifies that compaction achieved the token target; OpenAI and Pydantic AI preserve provider compaction artifacts as typed/replayable items; CrewAI enforces a structured five-section prompt. These are traceability, shape, and budget safeguards—not semantic verification. LangGraph has no built-in summarizer, and OpenHands exposes no faithfulness evidence. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:949-1088`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/compact.py:350-411`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/memory/openai_responses_compaction_session.py:404-496`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/messages.py:1681-1727`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/translations/en.json:26`)

### Tool-result handling

Tool results are the most consistently recognized special class. Agent Framework groups them atomically and can replace old groups with readable labels. Letta preserves them until a cut but clamps oversized returns and exempts recall/search tools. OpenAI preserves recent outputs and previews old oversized ones. CrewAI preserves raw outputs until whole-buffer summarization. Pydantic AI and LangGraph retain them indefinitely by default. This divergence reflects product shape: coding and memory agents need exact recent observations, while generic orchestration libraries avoid imposing semantics. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:820-918`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/letta_agent_v3.py:143-153, 1877-1884`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/extensions/tool_output_trimmer.py:196-220`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:806-810`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/_agent_graph.py:125`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/tool_node.py:1247-1250`)

### Can the model tell what is missing?

Letta provides the clearest in-band disclosure. CrewAI says prior context was summarized but not what was lost. OpenAI's local trimmers disclose omissions, while latest-N sessions and arbitrary filters may be silent. Agent Framework preserves machine-readable exclusion metadata and inserted summaries but lacks one canonical model-facing manifest. Pydantic AI, LangGraph, and the OpenHands harness do not guarantee a model-visible signal. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/system.py:207-224`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/translations/en.json:27`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/session_persistence.py:85-105`, `src/agents/extensions/tool_output_trimmer.py:208-220`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:582-588`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:636-658`)

### Observability

Letta is the operational leader because it emits a pre-compaction event and persists before/after token/message statistics in the summary. Agent Framework has detailed lineage and .NET compaction telemetry but lacks matching Python OpenTelemetry. OpenAI traces per-turn usage and optionally exact model input, yet hides compaction as a public stream event. CrewAI traces retrieval events and usage but has no dedicated compaction event. LangGraph traces state writes and checkpoint deltas rather than semantic prompt changes. Pydantic AI exposes usage but not a standardized context-change event. OpenHands persists agent/condenser metrics but cannot identify what was removed. ## Tradeoffs

### Rebuilding versus incrementally appending

- **Benefit of rebuilding:** fresh durable state, deterministic incorporation of compaction, and easier replacement of stale retrieval. Letta exemplifies this. **Cost:** DB reads, ordering invariants, and concurrency risks around active IDs. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/helpers.py:148-317`, `letta/services/agent_manager.py:999-1018`)
- **Benefit of appending:** low overhead and straightforward continuation. CrewAI and Pydantic AI exemplify it. **Cost:** stale retrieval and unbounded raw history unless a pre-call seam actively rewrites it. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/experimental/agent_executor.py:1486`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/_agent_graph.py:125, 854-1025`)
- **Hybrid alternative:** retain append-only canonical state but rebuild a transient model view, as Agent Framework, LangGraph, and OpenAI Agents SDK permit. This reduces data loss but requires clear ownership of the projection. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1163-1176`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:636-658`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/run_loop.py:280-287`)

### Local policy versus provider-native compaction

- **Local policy** is inspectable, portable, and can respect application semantics such as tool pairs and protected messages. It costs implementation effort and tokenizer maintenance. Agent Framework and Letta lean local. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:42-63`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/context_window_calculator/token_counter.py:21-125`)
- **Provider-native compaction** can exploit server state and hidden provider knowledge. It reduces local payloads but makes removal order, summary contents, and cross-provider replay opaque or locked. OpenAI Agents SDK and Pydantic AI expose this route. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/model_settings.py:187-192`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/models/anthropic.py:2016-2081`, `pydantic_ai_slim/pydantic_ai/models/openai.py:3995-4194`)
- **Best-fit rule:** use provider compaction as an optimization or fallback, not the sole correctness boundary, unless the application accepts provider lock-in and opaque semantics.

### Preventive versus reactive control

- **Preventive control** reduces failed calls and latency but depends on a complete rendered-request estimate. Agent Framework reserves output tokens and uses thresholds; Pydantic AI can pre-count only when explicitly enabled and supported. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1307-1419`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/usage.py:283-296`)
- **Reactive control** catches estimation drift but spends a failed request and can depend on brittle provider error mapping. CrewAI recognizes fixed substrings; Letta bounds retries and separates system-prompt overflow. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/exceptions/context_window_exceeding_exception.py:4-13`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/settings.py:96`, `letta/services/summarizer/compact.py:391-411`)
- **Best-fit rule:** combine both, with one bounded emergency retry after a stricter fallback policy.

### Reversible exclusion versus destructive replacement

- **Reversible exclusion** supports audit and retrieval but grows storage. Agent Framework annotations and Letta's inactive rows demonstrate this. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:569-601`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/conversation_manager.py:752-797`)
- **Destructive replacement** immediately bounds active storage but needs atomic writes and rollback. OpenAI's compaction session snapshots old history and attempts restoration; CrewAI mutates the in-memory list without an equivalent durable rollback contract. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/memory/openai_responses_compaction_session.py:221-306`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:995-1003`)
- **Best-fit rule:** keep canonical originals through at least one retention window, version the active projection, and purge asynchronously under an explicit policy.

### Genericity versus safe defaults

LangGraph and Pydantic AI avoid imposing one model's context semantics, which is appropriate for general libraries. The cost is that a novice application can run an unbounded history with no protection. Letta and Agent Framework make stronger product assumptions and therefore provide safer defaults. OpenHands shows the operational cost of delegating the policy without a rich event contract: the control plane cannot answer what the runtime removed. ([langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-424`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/capabilities/process_history.py:27-69`; [openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/event_callback/webhook_router.py:470-473`)

## Decision Guide

1. **For a durable, long-lived memory agent:** choose the Letta model—durable active IDs, proactive and reactive triggers, explicit summary messages, raw-history retrieval, and structured compaction statistics. Add a retention policy and summary verification because those are its main gaps. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/agents/helpers.py:148-317`, `letta/schemas/letta_message.py:406-449`)

2. **For a portable multi-provider framework that owns loop safety:** choose the Agent Framework model—protocol-based tokenizers and strategies, atomic tool groups, staged budget composition, and separate before/after compaction. Add reactive overflow retry and Python telemetry. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:42-63, 1091-1419`)

3. **For an SDK where applications need maximum policy flexibility:** use the OpenAI Agents SDK or Pydantic AI model. Prefer OpenAI Agents SDK when session compaction, local tool-output trimming, rollback, and server conversation deltas matter. Prefer Pydantic AI when uniform capability composition, typed provider parts, and dynamic tool/capability refresh matter. In either case, ship an application-level default rather than assuming extension points are self-enforcing. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/extensions/tool_output_trimmer.py:90-220`, `src/agents/memory/openai_responses_compaction_session.py:160-306`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/capabilities/process_history.py:27-69`, `pydantic_ai_slim/pydantic_ai/messages.py:1681-1727`)

4. **For graph orchestration with domain-specific state semantics:** LangGraph is appropriate if the application team is prepared to own token counting, summarization, retrieval, and disclosure in a `pre_model_hook` or node. Use `llm_input_messages` when the model view should differ from canonical state; use explicit removal writes only when canonical state should change. ([langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:396-424, 636-658`, `libs/langgraph/langgraph/graph/message.py:209-234`)

5. **For short, task-scoped agents with rare overflow:** CrewAI's reactive summary may be sufficient only if `respect_context_window` is enabled and loss of exact old tool outputs is acceptable. It should not be selected unchanged for indefinite sessions. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/experimental/agent_executor.py:196`, `lib/crewai/src/crewai/utilities/agent_utils.py:732-749`)

6. **For a split control-plane/runtime architecture:** require the runtime to emit a versioned context-change event containing trigger, strategy, before/after tokens and items, removed classes, summary ID, and retry outcome. OpenHands' aggregated metrics and `usage_id` attribution are useful but insufficient to audit context behavior. ([openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/app_conversation/sql_app_conversation_info_service.py:399-466`, `openhands/app_server/utils/llm_metadata.py:40`)

## Practical Tips

- Define separate schemas for **canonical history**, **active context**, **model-visible projection**, and **retrieval corpus**. Do not make one mutable list serve all four roles. Agent Framework's projection and Letta's active IDs are concrete starting points. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:569-601`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/agent_manager.py:999-1018`)
- Run the final context builder immediately before every model invocation, including inner tool-call iterations and streaming paths. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_clients.py:363-391`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/run_loop.py:1370-1405, 1818-1847`)
- Budget the fully rendered input: system instructions, dynamic prompts, tool schemas, retrieval results, message serialization, provider overhead, and reserved output tokens. A count of stored messages is not enough. Agent Framework explicitly subtracts output budget; OpenAI's default candidate-count compaction shows the risk of item-count triggers. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1307-1419`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/memory/openai_responses_compaction_session.py:24-56`)
- Use a pressure ladder: truncate oversized individual tool results with a marker, compact old tool groups, summarize an old conversational prefix, drop old non-system units, and only then fail with an explicit irreducible-overflow reason. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/summarizer.py:562-647`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1091-1160`)
- Encode tool call/result, approval, and provider compaction artifacts as atomic units. Never leave pairing to arbitrary list slicing. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:120-243`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/summarizer_sliding_window.py:133-137`)
- Combine an 80–90% proactive threshold with one bounded reactive retry using a stricter strategy. Calibrate the threshold by model and include a special branch for system-prompt overflow. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/thresholds.py:27-41`, `letta/services/summarizer/compact.py:391-411`)
- Add an in-band omission manifest: “N earlier messages summarized; M tool outputs truncated; raw records retrievable with handles X/Y.” Letta's notice and OpenAI's tool preview marker are good minimums. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/system.py:207-224`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/extensions/tool_output_trimmer.py:208-220`)
- Emit a context-change event with trigger, strategy, estimated and actual tokens, item/group counts, removed classes, summary lineage, duration, and retry outcome. Letta provides most of this; Agent Framework's .NET telemetry supplies a useful metric vocabulary. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/schemas/letta_message.py:406-449`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `dotnet/src/Microsoft.Agents.AI/Compaction/CompactionTelemetry.cs:1-50`)
- Version or transact destructive history replacement. If the store cannot transact, snapshot old state and test rollback failure separately. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/memory/openai_responses_compaction_session.py:242-306`)
- Keep raw evicted content retrievable, but define TTL, archival, legal retention, and purge behavior so “reversible compaction” does not mean indefinite primary-store growth. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/conversation_manager.py:752-797`; [langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/checkpoint/langgraph/store/base/__init__.py:545-567, 1278-1296`)
- Test what the provider actually receives, not only intermediate lists. Pydantic AI's capture fixture and Agent Framework's strategy tests demonstrate useful seams. ([pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `tests/test_history_processor.py:36-100`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/tests/core/test_compaction.py:107-1199`)
- Add adversarial long-run tests: one oversized tool message, non-Latin/code-heavy inputs, summarizer timeout and empty output, malformed provider compaction, repeated overflow, pending approval at the cutoff, concurrent compaction, restart/resume, and a system prompt that cannot fit.
- For semantic summaries, require a structured output carrying current goal, accepted constraints, decisions, unresolved items, critical exact values, active tool state, and source message IDs; then verify required fields or facts before replacing active context. None of the studied semantic summarizers currently completes this verification loop. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:931-1088`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/summarizer/summarizer_config.py:35-45`)

## Anti-Patterns / Caution Signs

- **Post-hoc usage mistaken for control:** token metrics are recorded, but no next-turn policy consumes them. CrewAI and the generic OpenAI runner demonstrate this gap. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/token_counter_callback.py:37-66`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/run_loop.py:1909-1918`)
- **Provider rejection as the only trigger:** this spends a failed call, misses providers that silently truncate, and depends on error-string compatibility. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/exceptions/context_window_exceeding_exception.py:4-13`)
- **Whole-buffer semantic replacement:** replacing all non-system history at once destroys recency, tool IDs, and exact observations in one discontinuity. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:995-1003`)
- **Arbitrary history processors without invariants:** a general list-rewrite hook can break call/return pairing, new-message accounting, or provider-specific parts. ([pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `docs/message-history.md:526-541, 599-600, 661-662`)
- **Silent omission:** latest-N reads or transient filters remove context without telling the model, making absence ambiguous. ([openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/run_internal/session_persistence.py:85-105`, `src/agents/run_internal/turn_preparation.py:66-80`)
- **Storage compression confused with prompt compression:** LangGraph's `DeltaChannel` improves checkpoint storage/replay but does not reduce model input. ([langgraph report](../source/03.07-context-refresh-inside-loop/langgraph.md), `libs/langgraph/langgraph/channels/delta.py:25-204`)
- **Provider/config surface treated as runtime proof:** OpenHands' condenser settings and metrics do not establish trigger behavior, drop order, or faithfulness inside the external SDK. ([openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1641`)
- **Dead or competing configuration paths:** OpenHands' unused condenser factory and migration-driven default changes increase uncertainty about the effective policy. ([openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`, `enterprise/migrations/versions/020_set_condenser_to_false.py:24-34`, `enterprise/migrations/versions/024_update_enable_default_condenser_default.py:24-65`)
- **Provider ordering as a hidden correctness condition:** Agent Framework's compaction provider must run after history loading; the ordering is configured but not enforced. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_harness/_agent.py:123-188`)
- **Heuristic tokenization without a reactive backstop:** four-character/byte approximations can drift badly on code, JSON, or non-Latin text. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:66-70`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:752-761`)
- **Append-only canonical storage with no retention plan:** reversibility becomes a database-growth problem. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/agent_manager.py:999-1018`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:1302-1304`)

## Notable Absences

- **No demonstrated semantic faithfulness verifier** appears in any source. Tests validate triggers, shapes, rollback, lineage, and budget reduction, but not whether required facts survived a summary. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `tests/integration_test_summarizer.py:780-1314`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `tests/memory/test_openai_responses_compaction_session.py:434-544`; [agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/tests/core/test_compaction.py:107-1199`)
- **No common omission-manifest standard** tells the model exactly what categories and time ranges are absent. Letta comes closest; other sources are partial or silent. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/system.py:207-224`)
- **Repo maps are largely absent.** Agent Framework has index/progressive-disclosure analogues through memory and skills providers, but the reports found no AST-aware repo-map refresh in CrewAI, Letta, OpenAI Agents SDK, or OpenHands. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_harness/_file_memory.py:217-397`; [crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/knowledge/knowledge.py:135-152`; [openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/utils/chunk_localizer.py:74-101`)
- **Provider-neutral exact rendered-request counting is uncommon.** Agent Framework and CrewAI default to rough estimates; OpenAI's generic runner has no preflight count; Pydantic AI's opt-in count covers only selected providers and can omit server-side tools on Anthropic. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/agent_framework/_compaction.py:66-70`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `src/agents/usage.py:102-215`; [pydantic-ai report](../source/03.07-context-refresh-inside-loop/pydantic-ai.md), `pydantic_ai_slim/pydantic_ai/usage.py:283-296`, `pydantic_ai_slim/pydantic_ai/models/anthropic.py:850-863`)
- **End-to-end scale/failure proof is uneven.** Agent Framework and OpenAI reports found no real-model test that crosses a context limit and proves recovery; OpenHands cannot inspect the runtime; LangGraph and Pydantic AI deliberately have no default recovery to test. ([agent-framework report](../source/03.07-context-refresh-inside-loop/agent-framework.md), `python/packages/core/tests/core/test_compaction.py:107-1199`; [openai-agents-sdk report](../source/03.07-context-refresh-inside-loop/openai-agents-sdk.md), `tests/memory/test_openai_responses_compaction_session.py:1028-1197`)

- **Cross-provider compaction portability is unresolved.** Provider-owned artifacts can be opaque, and Pydantic AI's provider-name-aware type reduces but does not eliminate misuse. ## Per-Source Notes

### agent-framework — 8/10

A policy-engine approach centered on annotated atomic message groups, tokenizer/strategy protocols, composed token budgets, and before/after provider hooks. ### crewai — 5/10

A task-scoped append-only buffer with one reactive escape hatch. The implementation preserves system messages and files and can chunk/summarize asynchronously, but it compacts only after provider rejection, is disabled by default, replaces all non-system content, and does not refresh memory/knowledge inside the loop. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/utilities/agent_utils.py:819-1003`, `lib/crewai/src/crewai/agent/core.py:583-646`)

### letta — 8/10

A durable memory-agent pipeline with per-turn DB rebuild, 90% proactive pressure checks, reactive compact-and-retry, multiple compaction modes and fallbacks, provider-aware counting, protected approval boundaries, explicit system-overflow failure, and rich client-visible statistics. It is the most operationally complete source, but raw storage grows monotonically and semantic faithfulness is not verified. ### openai-agents-sdk — 7/10

A hybrid model with fresh per-call normalized input, dynamic instructions/tools, local rewrite hooks, session limits, targeted tool trimming, provider context management, and rollback-capable compaction sessions. A control-plane surface around an external runtime. ## Open Questions

- What minimum structured summary schema and verifier can demonstrate preservation of goals, constraints, exact values, unresolved decisions, and tool state without making compaction prohibitively expensive?
- Should the active-context projection be versioned so traces, retries, and user-visible responses can identify the exact context snapshot used for each model call?
- How should proactive thresholds adapt from observed provider usage when local tokenizers omit tool schemas, server tools, or hidden provider overhead?
- What transaction or optimistic-lock contract prevents two concurrent turns from overwriting each other's active-context IDs or compaction results? Letta's active-ID update is the clearest place this question remains open. ([letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/services/agent_manager.py:999-1018`)
- When provider-native compaction is opaque, what portable metadata should be required—source item IDs, trigger tokens, provider response ID, model name, and a human-readable omission notice?
- Should retrieval automatically rerun after compaction, or should the model receive only retrieval handles and choose what to reload? CrewAI's one-shot injection and Letta's tool-driven recall represent opposite ends of this decision. ([crewai report](../source/03.07-context-refresh-inside-loop/crewai.md), `lib/crewai/src/crewai/agent/core.py:583-646`; [letta report](../source/03.07-context-refresh-inside-loop/letta.md), `letta/functions/function_sets/base.py:87-130`)
- What retention policy balances auditability against indefinite growth when excluded messages remain retrievable?
- Can a portable atomic interaction model cover function calls, approvals, reasoning blocks, MCP calls, native search, and provider compaction parts across providers?
- How should control planes verify that a remote agent runtime actually honored configured context policy? OpenHands currently receives usage totals but no typed compaction transition. ([openhands report](../source/03.07-context-refresh-inside-loop/openhands.md), `openhands/app_server/event_callback/webhook_router.py:470-473`)
- Which test should be considered the acceptance criterion for the dimension: a real-provider run that crosses the context limit, survives summarizer failure, compacts, retrieves an old exact fact, and completes without breaking tool-call pairing?

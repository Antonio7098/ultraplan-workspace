> **Note on framing.** OPA is not an LLM agent harness, so it has no native notion of "tools" for a model. The closest analog is the **built-in function registry** that Rego policies (the "model") call: `count`, `sum`, `http.send`, `regex.match`, `json.match_schema`, etc. Each call site supplies typed arguments whose signatures are declared as `*ast.Builtin` literals. This analysis therefore treats `*ast.Builtin` as OPA's "tool schema" and examines (a) how that schema is generated, (b) how arguments are validated, (c) whether descriptions reach the consumer, and (d) what cross-provider portability looks like for CLI/HTTP/SDK surfaces.

## Summary

OPA defines built-in function schemas as **hand-written Go struct literals** in `v1/ast/builtins.go` — there is no annotation-driven or reflection-driven generator. Each `*Builtin` literal builds a `*types.Function` (`v1/types/types.go:690`) whose argument and result slots are `*types.NamedType` (`v1/types/types.go:72`) carrying a `Name` and a `Descr`. The `Builtin` struct itself has a `Description` field (`v1/ast/builtins.go:3596`) that is exported as JSON via struct tags.

Validation happens at **two layers**. Compile-time type checking (`v1/ast/check.go:383`, `checkExprBuiltin`) unifies each operand against the expected type via `unify1` and `unifies` (`v1/ast/check.go:438,445`); errors carry `Have []types.Type` and `Want types.FuncArgs` in `*ArgErrDetail` (`v1/ast/check.go:1052,1184`). At runtime, `builtinErrorWrapper` (`v1/topdown/builtins.go:129`) wraps every registered function and decorates errors with the function name + source location; helpers like `NewOperandTypeErr` (`v1/topdown/builtins/builtins.go:129`) emit `ErrOperand`-typed errors. Errors are accumulated in `builtinErrors` (`v1/topdown/eval.go:49`) and only surfaced as fatal when `StrictBuiltinErrors(true)` is set on `rego.PrepareForEval` (`v1/rego/rego.go:1291`) or `?strict-builtin-errors` is passed to the HTTP API (`v1/server/types/types.go:611`, `v1/server/server.go:1525,1816`).

Descriptions are **mandatory** for all non-deprecated, non-infix, non-`internal.*` builtins — enforced by `TestAllBuiltinsHaveDescribedArguments` (`v1/ast/builtins_test.go:33`) which iterates the registry and asserts every argument and result is a `*types.NamedType` with a non-empty `Descr`. The actual external schema (`builtin_metadata.json`, 29,564 lines) is **generated at release time** by `go generate build/gen-run-go.sh internal/cmd/genbuiltinmetadata/main.go builtin_metadata.json` (`main.go:34`) which walks the `Capabilities.Builtins` registry (`internal/cmd/genbuiltinmetadata/main.go:26-81`) and emits a JSON document with `name`/`type`/`description` for each argument, plus `introduced`, `available`, `wasm`, `deprecated`, `infix`, and a `_categories` index. The docs site (`docs/docusaurus.config.js:418`) reads this file at build time to render the `<BuiltinTable>` component (`docs/src/components/BuiltinTable/index.jsx:86-141`).

JSON Schema is supported **inbound only**: `loadSchema` (`v1/ast/schema.go:41`) parses a JSON Schema document via `internal/gojsonschema` into the Rego `types.Type` representation, and the result is fed into `typeChecker.WithInputType` (`v1/ast/check.go:92`) so that `input.*` accesses are type-checked against the imported schema. There is also a runtime `json.verify_schema` and `json.match_schema` built-in (`v1/topdown/jsonschema.go:65,83`) backed by the same gojsonschema loader, with `interQueryBuiltinValueCache` short-circuiting schema recompilation (`v1/topdown/jsonschema.go:86-92`). OPA does **not** export built-in signatures as JSON Schema; the `Builtin.Decl` MarshalJSON (`v1/types/types.go:787`) emits the Rego-native type tree.

In the dimension's central question — "would the model understand how to call this tool correctly?" — the answer for OPA is **yes for Rego authors** (descriptions are first-class, named, and tested) but **no for LLM agents out of the box**, because the canonical schema is the Rego-native `types.Type` tree, not JSON Schema / OpenAPI / MCP. The external `builtin_metadata.json` is close (name, type, description, deprecation) but uses Rego type strings like `"set[number]"` instead of JSON Schema.

## Rating

**7 / 10** — Clear, hand-curated model with strong compile-time and runtime validation, mandatory named+described arguments enforced by a CI test, and a release-time schema export. Deductions: (a) **schemas are handwritten, not generated** — every builtin requires a `*ast.Builtin` literal in `v1/ast/builtins.go` (`builtins.go:780-3693`), so adding a tool means hand-coding `types.Named(...)` chains and remembering the `.Description(...)` calls; this is error-prone compared to a Pydantic/Zod/Pydantic-style schema decorator approach; (b) **the external schema (`builtin_metadata.json`) is generated but uses Rego-native type strings, not JSON Schema** (`internal/cmd/genbuiltinmetadata/main.go:38,56`), so LLM agents cannot consume it without a translation step; (c) **runtime errors are lenient by default** — `builtinErrors.errs` are accumulated in a slice (`v1/topdown/eval.go:2166`) and only the first is reported by `Eval` (`v1/rego/query.go:519-527`) unless `StrictBuiltinErrors(true)` is passed; (d) **argument coercion is element-wise via ad-hoc helpers** (`v1/topdown/builtins/builtins.go:158-338`) rather than a uniform schema-driven validator, so type-system drift between the declared `Decl` and the runtime check is possible (e.g., `count` accepts a string in `Decl` per its `types.Any(types.S, …)` union, but its runtime is forgiving about any collection shape); (e) **capabilities.json strips descriptions entirely** (`capabilities.json` has only `{name, decl}` per builtin, no `description` field), so the de-facto portable form omits the most useful information for an LLM.

## Answers to Dimension Questions

1. **Are schemas generated or handwritten?**
   Handwritten. Every `*Builtin` is a Go struct literal in `v1/ast/builtins.go` (3,693 lines, ~140 builtins). Each builds its `Decl` via `types.NewFunction(types.Args(types.Named(...).Description(...)), types.Named(...).Description(...))`. The `*types.NamedType` chain is hand-coded per builtin. There is no auto-generation from docstrings, annotations, or runtime reflection. The only **generation** step is the release-time JSON dump from `internal/cmd/genbuiltinmetadata/main.go` (`//go:generate build/gen-run-go.sh` at `main.go:34`) which reads the registry and emits `builtin_metadata.json`; this is a translator, not a generator. The `RegoMetadataChain` / `RegoMetadataRule` builtins (`v1/ast/builtins.go:3169-3191`) are policy-side helpers for accessing annotation objects, not a schema-generation facility. JSON Schema is **imported**, not generated: `loadSchema` (`v1/ast/schema.go:41`) parses user-provided schemas.

2. **Are descriptions useful?**
   Yes for the in-process consumer (Rego compiler, docs renderer). Each argument and result carries both a name (`collection`, `request`, `pattern`) and a description (`"the set/array/object/string to be counted"`) at `v1/ast/builtins.go:790-792`. Descriptions are exported in `builtin_metadata.json` (`builtin_metadata.json:259,411`) and rendered in docs (`docs/src/components/BuiltinTable/index.jsx:118-141`). The CI-style test `TestAllBuiltinsHaveDescribedArguments` (`v1/ast/builtins_test.go:33-44`) fails the build if any non-excluded builtin lacks descriptions. **However**, descriptions live only in the in-memory struct and the generated JSON dump — they are not propagated to runtime error messages. The `ArgErrDetail.Lines()` output is just `"have: (number, string)" / "want: (collection: any, n: number)"` (`v1/ast/check.go:1058-1063`) which includes type names but not the human-readable description. A user seeing `sum: invalid argument(s)` gets no hint that they passed a string where a numeric collection was expected beyond the type string. The runtime `NewOperandTypeErr` (`v1/topdown/builtins/builtins.go:128-135`) likewise only names the type, not the arg position name from `Decl`.

3. **Is validation strict?**
   Strict at compile time, lenient at runtime by default.
   - Compile: `checkExprBuiltin` (`v1/ast/check.go:383-455`) returns errors for wrong arity, undefined function, and incompatible argument types. The error carries both `Have` and `Want` (`v1/ast/check.go:1184-1191`).
   - Runtime: errors are accumulated into `builtinErrors.errs` (`v1/topdown/eval.go:49,2166`) and `Eval` returns only the first error (`v1/topdown/query.go:519-527`). To get all errors the caller must pass `StrictBuiltinErrors(true)` (`v1/rego/rego.go:1290`) or `BuiltinErrorList(&buf)` (`v1/rego/rego.go:1297`). The HTTP API mirrors this with `?strict-builtin-errors` (`v1/server/types/types.go:611`). **Wasm target ignores strict mode**: `rego_wasmtarget_test.go:233` has a comment "ignored for wasm target (always non-strict)". This means a model that submits invalid args to a wasm-compiled policy gets silent success.
   - **Type-checker drift**: the compile-time schema (`types.Function`) and the runtime coercion helpers (`v1/topdown/builtins/builtins.go:158-242`) are independent. `StringOperand` accepts any `ast.String` regardless of what the declared `Decl` said; `NumberOperand` accepts any `ast.Number`. A builtin that declared `types.S` in `Decl` but calls `IntOperand` would reject any string. There is no automatic check that runtime helpers match the declared types.

4. **Are defaults handled clearly?**
   No defaults. OPA builtins are functions; arguments are positional. There is no concept of a default value or optional argument in the `*types.Function` signature. Variadic args exist (`types.NewVariadicFunction` at `v1/types/types.go:732-741`) but require `result == nil` — non-void variadic is a `panic("illegal value")` at `v1/types/types.go:734`. Optional inputs to policies are handled through the `input` document and JSON Schema annotations (`SchemaAnnotation` at `v1/ast/annotations.go:50-55`), not through default values on builtins. The `json.verify_schema` builtin signature at `v1/ast/builtins.go:3078-3094` shows the typical pattern: required arg + structured result, no defaults.

5. **Are schemas portable across providers?**
   Partially. Three portability surfaces exist:
   - **`builtin_metadata.json`** (`builtin_metadata.json:256-262`): closest to JSON Schema, but uses Rego-native type strings (`"number"`, `"set[string]"`, `"array[any]"`) — not interoperable with an LLM tool-call JSON Schema consumer without a translation step. Field names are `name`/`type`/`description` which is recognizably tool-schema-like.
   - **`capabilities.json`** (`capabilities.json:3-30`): portable as JSON but **omits descriptions entirely** — only `{name, decl:{args:[{type}], result:{type}}}` is present, so the most useful field for an LLM is missing. This is the format used for OPA version compatibility and wasm ABI; it is intentionally minimal.
   - **`Builtin` JSON via struct tags** (`v1/ast/builtins.go:3594-3609`): also uses Rego-native type tree (via `types.Function.MarshalJSON` at `v1/types/types.go:787`), preserves `description` and `categories`. Roundtrip is verified by `TestBuiltinDeclRoundtrip` (`v1/ast/builtins_test.go:16-31`).
   - **JSON Schema (inbound)**: `loadSchema` (`v1/ast/schema.go:41`) and the runtime `json.match_schema` builtin (`v1/topdown/jsonschema.go:83`) accept JSON Schema documents. The translator (`schemaParser` at `v1/ast/compile.go:1676`) supports `$ref`, `anyOf`, `AllOf`, recursive schemas. **Pattern enforcement** uses Go RE2 (commented at `v1/topdown/jsonschema.go:50-61`), so ECMA-262-only regex features cause a runtime rejection.
   - **No JSON Schema export**: OPA does not generate JSON Schema for its builtins. `opa inspect` (`cmd/inspect.go:310-321`) prints per-rule schemas from annotations but not the registry. The only `*.schema.json` in the tree is for IR plans (`v1/ir/plan.schema.json`).
   - **LLM agent interop**: OPA has no MCP server mode, no OpenAI function-call transformer, no Anthropic-tool-use adapter. A consumer wanting to expose OPA builtins to an LLM must write their own translator from `builtin_metadata.json` to their model's tool-call schema format.

## Architectural Decisions

- **Schemas-as-code, not schemas-as-data.** Every `*Builtin` is a Go literal, not a JSON/YAML/TOML file. Pro: the schema is type-checked at compile time; the `Builtin` struct is impossible to construct incorrectly. Con: adding a builtin requires a code change, a PR, and a release; contributors cannot add tools without Go knowledge.
- **Two-layer validation.** Compile-time type checking (`v1/ast/check.go`) catches everything before evaluation starts, but the same checks re-run defensively at runtime via `builtinErrorWrapper` (`v1/topdown/builtins.go:129`). Runtime errors are *additionally* structured (per-operand) via `NewOperandTypeErr` / `NewOperandElementErr` / `NewOperandEnumErr` (`v1/topdown/builtins/builtins.go:128-156`) so the error message tells the model *which* arg and *why*. This is the closest OPA gets to a "correction" mechanism — but there is no auto-retry; the error is just surfaced.
- **Lenient-by-default with strict opt-in.** `StrictBuiltinErrors(false)` is the default. The choice between "collect all errors and return the first" vs "fail on the first error" is a policy decision left to the caller. Mirrored at HTTP boundary via `?strict-builtin-errors`. **Caveat**: wasm target silently ignores strict mode (`v1/rego/rego_wasmtarget_test.go:233`).
- **JSON Schema only inbound.** OPA imports JSON Schema to type-check `input.*` access and to power the `json.match_schema` runtime builtin. It does not export JSON Schema for builtins. Rationale (inferred): the Rego type system is richer than JSON Schema (it has Sets, Any, Function, Recursive, NamedType), so a 1:1 export would lose information. The `builtin_metadata.json` export uses Rego-native type strings as a compromise.
- **Schemas are persisted alongside policy metadata.** Policies can declare per-rule input schemas via `# METADATA: schemas:` blocks (`SchemaAnnotation` at `v1/ast/annotations.go:50-55`). These are parsed into `AnnotationSet`, looked up by `getSchemaType` (`v1/ast/check.go:195`), and merged with the global `SchemaSet`. This is OPA's per-policy schema mechanism — analogous to a "tool" that is policy-specific rather than registry-wide.
- **Registry is single global, init-time only.** `RegisterBuiltin` (`v1/ast/builtins.go:22-40`) is called from `init()` and the comment warns "**not** thread-safe" and "Registering built-ins after that point is unsupported". All builtins are registered before the package is usable. Custom builtins can be added via `WithCapabilities` (`v1/ast/compile.go:565`) but only at compile-time, not at evaluation-time. This is unlike Pydantic-decorator-based systems where `MyTool()` is registered dynamically.
- **No model-emitted argument reconciliation loop.** When a Rego policy author calls `sum([1, "foo"])`, the compiler rejects it (`v1/ast/check_test.go:2716-2722`). When a runtime helper detects a wrong type, it returns `ErrOperand`. There is no retry-with-correction, no LLM-driven schema-aware rewrite, no "did you mean X?" prompt. The error is final.

## Notable Patterns

- **Mandatory named+described arguments via CI test** (`v1/ast/builtins_test.go:33-62`). `TestAllBuiltinsHaveDescribedArguments` iterates `Builtins` and asserts every arg/res has `Name != "" && Descr != ""`. Skips deprecated, infix, `print`, and `internal.*`. This is the strongest enforcement of "schema quality" anywhere in the codebase.
- **Distinct error types per failure mode**: `*ArgErrDetail` (compile, `v1/ast/check.go:1052`), `*UnificationErrDetail` (compile, `v1/ast/check.go:1071`), `*RefErrUnsupportedDetail` (`v1/ast/check.go:1090`), `*RefErrInvalidDetail` (`v1/ast/check.go:1109`). Each carries structured detail (`Have`, `Want`, `Ref`, `Pos`, `OneOf`). The CLI (`cmd/test.go`) and HTTP API (`v1/server/types/types.go:131`) propagate these as `*ast.Error` chains.
- **Capability allow-list** as a portability gate. `Capabilities.Builtins` (`v1/ast/capabilities.go:85`) lists every builtin a given OPA version supports; `Compiler.WithCapabilities` (`v1/ast/compile.go:565`) restricts the policy to those builtins. This is the analog of a "tool allow-list" in MCP-style systems. The `allow_net` field (`v1/ast/capabilities.go:100`) gates which remote hosts can serve `$ref` resolutions — a network-side schema portability constraint.
- **`unsafe` builtin classification.** `WithUnsafeBuiltins` (`v1/ast/compile.go:598-604`) marks named builtins (typically `http.send`, `rego.metadata.*`, anything side-effecting) as unsafe. Policies calling them in unsafe positions are rejected (`checkUnsafeBuiltins` at `v1/ast/compile.go:7208`). The `with`-modifier is also restricted (`validateWith` at `v1/ast/compile.go:7017`) so a policy cannot `with http.send as ...` to swap it for a fake. This is the closest OPA gets to a "side-effecting tool" flag.
- **Inter-query schema caching.** `json.match_schema` caches compiled `*gojsonschema.Schema` in `InterQueryBuiltinValueCache` (`v1/topdown/jsonschema.go:86-92`) so repeated evaluations of the same schema skip recompilation. Similar caching for the result-document-loader is not done.
- **`Nondeterministic` flag** (`v1/ast/builtins.go:3608`). Builtins like `http.send`, `rand.intn`, `time.now_ns` are marked non-deterministic; the partial evaluator (`v1/ast/compile.go` partial-eval stages) preserves them rather than caching results. This is the tool-call equivalent of "side-effecting" or "non-idempotent".
- **`CanSkipBctx`** (`v1/ast/builtins.go:3607`). Pure functions skip the heavy `BuiltinContext` (metrics, caches, cancel, tracers) — an internal optimization but also a hint to the type system that the builtin has no observable side effects.
- **IR plan JSON Schema** (`v1/ir/plan.schema.json`) — the one place OPA does publish a JSON Schema. It describes the output of `opa build -t plan`, not the tool-call surface. Generated comment: "Generated from v1/ir/ir.go" (`v1/ir/plan.schema.json:4`).
- **Capabilities JSON vs metadata JSON dichotomy.** `capabilities.json` is the machine-readable portable form (no descriptions, type-only); `builtin_metadata.json` is the human-readable description-rich form. Consumers needing both must load both files.

## Tradeoffs

- **Handwritten schemas** trade flexibility for safety. Adding a new builtin is a multi-step process: write the `*Builtin` literal in `v1/ast/builtins.go`, implement the function in `v1/topdown/`, register it in `DefaultBuiltins`, write tests, regenerate `builtin_metadata.json`. There is no `@tool` decorator that would let a third-party package export a tool without code changes to the registry.
- **Compile-time vs runtime separation.** The compile-time `*types.Function` schema is "what the policy author wrote"; the runtime `builtinErrorWrapper` is "what the evaluator executes". The two are not linked at runtime — there is no validator that walks the `Decl` and ensures each operand is checked. This means a builtin can declare `types.N` but skip numeric coercion; the runtime would still work but reject arguments of the declared type only if a helper raises.
- **`builtin_metadata.json` is generated, not read at runtime.** It is a release artifact. There is no `loadBuiltinMetadata(path)` API; the runtime always reads the in-process `BuiltinMap`. This means a downstream consumer cannot override or extend the schema at evaluation time.
- **Rego-native type strings in `builtin_metadata.json`** like `"set[string]"` (`builtin_metadata.json:6315`) and `"array[any]"` (`builtin_metadata.json:3285`) are not JSON Schema. An LLM agent receiving this schema must either understand Rego types or use a translator. There is no translator in the tree.
- **Strict-mode opt-in is per-call, not per-policy.** A policy that mixes safe and unsafe builtins cannot opt into strict mode for the unsafe ones only. Either all runtime errors are fatal, or none are. (`StrictBuiltinErrors(true)` is global.)
- **Wasm target ignores strict mode.** `v1/rego/rego_wasmtarget_test.go:233` and `:393` confirm this. A model that emits `1/0` against a wasm-compiled policy gets `undefined` back rather than an error — incorrect usage is silently swallowed.
- **JSON Schema import is lossy.** `schemaParser` (`v1/ast/compile.go:1696`) handles `anyOf`/`AllOf`/`$ref`/recursive but does not implement the full draft-07/2020-12 spec. ECMA-262-only regex features are rejected at runtime (`v1/topdown/jsonschema.go:50-61`). A policy author importing a Kubernetes JSON Schema may hit edge cases.

## Failure Modes / Edge Cases

- **Missing description → CI failure**. Adding a builtin without `.Description(...)` on every arg/res fails `TestAllBuiltinsHaveDescribedArguments` (`v1/ast/builtins_test.go:33`). Skipped categories: deprecated, infix, `print`, `internal.*`. A new `internal.*` builtin can silently omit descriptions.
- **Runtime type drift**. The compile-time schema says `types.S` but the implementation calls `IntOperand`; a string arg would still pass `NewOperandTypeErr(pos, x, "string")` from the helper, but the error message would say "must be number but got string" rather than matching the declared type. (`v1/topdown/builtins/builtins.go:163-164`)
- **Variadic + non-nil result is a panic**. `v1/types/types.go:734` panics if `NewVariadicFunction` is called with `result != nil`. Easy footgun for a contributor adding a variadic builtin.
- **`with` cannot replace unsafe builtins** (`v1/ast/compile.go:7017-7126`). Tests at `compile_test.go:7331-7343` confirm `with` of `http.send` fails compile. But `with` of a *safe* builtin in a policy that calls an unsafe builtin still works — the unsafe call must be mocked away via the compiler's `WithUnsafeBuiltins` map, not via `with`.
- **Wasm silent error swallowing**. Division by zero in wasm-compiled policies returns `undefined` instead of an error (`v1/rego/rego_wasmtarget_test.go:233`). A model relying on error feedback from OPA to correct its tool calls would be confused.
- **First-error-wins without strict mode**. `v1/topdown/query.go:519-527`: when `strictBuiltinErrors=false`, only the first error is returned. A policy with multiple type errors gets one fix at a time. The caller cannot know how many errors are pending.
- **`builtin_metadata.json` drift after release**. The file is generated at release time (`main.go:34`). If a contributor adds a builtin and forgets to run `go generate`, the metadata file is stale until release. `CHANGELOG.md:4650` notes the release process was updated to include the metadata regeneration. The CI does not appear to fail on stale metadata.
- **`capabilities.json` lacks descriptions**. A consumer using `capabilities.json` (the more portable format) cannot show users what a builtin does. (`capabilities.json:3-30` only contains `{name, decl:{args:[{type}], result:{type}}}`.)
- **JSON Schema remote `$ref` gated by `allow_net`**. `v1/ast/capabilities.go:94-100` defines `AllowNet` for remote `$ref` resolution. If a policy's input schema references a remote URL and the OPA instance is started without that host in `allow_net`, the schema import fails (`v1/ast/check_test.go:2614-2620` shows the test for "HostNotAllowed").
- **Recursive JSON Schema handling**. `schemaParser.definitionCache` (`v1/ast/compile.go:1676-1732`) tracks `processing` state to detect cycles. A pathological schema could in theory cause `processing` to remain `true` if a parsing error interrupts the deferred finalize, though the test at `compile_test.go:12744` covers the cycle case.
- **Capability version drift across OPA instances**. A bundle compiled against OPA 1.0 with `http.send` may be loaded into OPA 0.59 without it. The capability check (`v1/ast/capabilities.go:296`) rejects the policy at load time, but the model (LLM agent) gets a generic "undefined function" rather than a feature-detection hint.
- **CLI `--schema` directory mode loads all files in tree** (`cmd/eval.go:646`). A misspelled path silently picks up no schemas, and the policy compiles without input validation. There is no warning when the schema directory is empty.

## Future Considerations

- **JSON Schema export of builtin signatures**. A `Builtin.MarshalJSONSchema()` (or `cmd/inspect --format=json-schema`) would let LLM agents consume OPA's tool surface without a custom translator. The information is already there (`*types.Function` + `*types.NamedType`); a generator similar to `genbuiltinmetadata` but emitting true JSON Schema would close the loop.
- **MCP server mode**. Exposing `*Builtin`s as MCP tools (`tools/list` + `tools/call`) would be a 1-page addition given the existing JSON metadata. This would make OPA's tool surface directly callable by any MCP-compatible agent.
- **Runtime schema validation for HTTP API inputs**. Currently the HTTP API accepts arbitrary `input` JSON; the input schema is enforced only at policy compile time. A runtime check against the input schema would catch malformed requests before evaluation.
- **Stricter default mode**. Making `StrictBuiltinErrors` true by default for HTTP API and SDK calls (with an explicit opt-out) would force callers to handle errors, matching the security posture of policy enforcement.
- **Single `Builtin` schema export format**. Maintaining `capabilities.json`, `builtin_metadata.json`, and the runtime JSON form is duplication. Consolidating into one canonical JSON Schema (with optional enrichment) would reduce drift.
- **Generated builtin declarations from a single source**. A YAML/TOML `builtins.yaml` consumed by both `genbuiltinmetadata` and the Go runtime (`go:generate` producing `*Builtin` literals in `v1/ast/builtins.go`) would let contributors add tools without writing Go. The current scheme requires Go fluency for what is essentially data.
- **Per-policy tool allowlist**. `WithUnsafeBuiltins` is coarse. A `SchemaAnnotation`-style per-rule restriction ("this rule may only call `count` and `sum`") would be a finer-grained tool surface for multi-tenant scenarios.
- **Streaming error reporting**. Currently `Eval` returns one error. Streaming each `builtinErrors` entry as it occurs would let an LLM agent batch corrections.
- **Wasm strict-mode parity**. The wasm target ignores strict mode (`v1/rego/rego_wasmtarget_test.go:393`). Closing this gap would unify the behavior across targets.
- **Schema annotations for HTTP API request validation**. `DataRequestV1.Input` (`v1/server/types/types.go:243`) is typed `*any` with no JSON Schema. Adding a `--request-schema` flag analogous to `--schema` for input would let deployments fail malformed requests at the boundary.

## Questions / Gaps

- **Is `builtin_metadata.json` regenerated by CI on every PR?** `main.go:34` declares the generator via `go:generate`, but the PR check policy (`build/policy/pr-check/pr_check.rego:40-56`) only checks the file is present, not that it matches the current `Builtins`. A contributor adding a builtin and forgetting `go generate` could merge a stale file. No evidence of automated regeneration in the tree.
- **How are custom builtins (added via `WithCapabilities`) exposed externally?** `v1/ast/compile.go:593-595` lets callers inject custom `*Builtin` values. These are in-memory only and do not appear in `builtin_metadata.json` (since the file is regenerated only from `CapabilitiesForThisVersion`). A plugin-based system could have undocumented tools.
- **Are runtime errors ever reported with the arg's declared name?** `v1/topdown/builtins/builtins.go:128-135` builds "operand N must be X but got Y" with `pos` as the positional index. The `*Builtin.Decl.FuncArgs().Args[i].Name` (e.g., `"collection"`) is not included in the error. A user-facing fix would name the argument.
- **Does OPA's Rego V1 schema syntax differ from V0?** `v1/ast/parser.go:3282` has `parseSchemaRef` for parsing schema references in annotations. The capabilities JSON at `capabilities.json:3-30` does not encode Rego version, so a consumer must inspect the OPA version separately.
- **What happens to schemas during partial evaluation?** `v1/ast/compile.go` partial-eval stages preserve schema annotations via `AnnotationSet`, but I did not find explicit tests confirming that partial-evaluation output carries the original schema annotations intact.
- **Is there a tool to convert `builtin_metadata.json` to JSON Schema for LLM consumption?** No evidence found in the tree. Consumers must write their own translator.
- **Does the SDK expose builtin metadata to consumers?** `v1/sdk/opa.go` exposes only `DecisionOptions`, `PartialOptions`, etc. — no method like `Builtins() []*Builtin`. A consumer wanting to introspect the tool surface from Go must import `v1/ast` directly, which is an internal-ish path.
- **How are HTTP API parameter schemas validated?** `v1/server/types/types.go:609-612` defines `ParamStrictBuiltinErrors` but there is no equivalent for input validation. A `?strict-input-validation` flag could close the gap noted under "Future Considerations".

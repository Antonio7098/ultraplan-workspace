## Summary

Pydantic AI generates tool argument schemas from a tool function's Python signature + type hints using Pydantic v2's internal schema generator, then renders the resulting core schema into JSON Schema via a small subclass (`GenerateToolJsonSchema`) that strips per-property titles. The same JSON Schema is fed both to the model (after a provider-specific `JsonSchemaTransformer` may rewrite it for strict / structured-output compatibility) and to a `pydantic_core.SchemaValidator` that re-validates every tool call's arguments before execution. Invalid arguments are converted into a `RetryPromptPart` and returned to the model up to `max_retries` times; once exhausted, the run fails with `UnexpectedModelBehavior`. A separate, optional `args_validator` runs after Pydantic schema validation and before execution, and supports custom `ModelRetry` for business-rule failures. Tool and parameter descriptions are sourced from the function's docstring via `griffe`, supporting Google / NumPy / Sphinx styles with `'auto'` inference, and `require_parameter_descriptions=True` enforces them at registration time. Schemas are stored on `ToolDefinition.parameters_json_schema` and travel through the agent run pipeline, where each model provider applies its own `JsonSchemaTransformer` (OpenAI, Anthropic, Bedrock, Gemini, OpenRouter, InlineDefs) before the wire format is built. The design is mature: schemas are generated (not handwritten), per-provider transformation is explicit and tested with snapshot-based golden files, and validation behavior is well covered by unit + VCR tests including invalid-args retry behavior.

## Rating

**9 / 10** — Clear, end-to-end model (function → JSON Schema → validator → provider transformer → retry) with explicit interfaces, snapshot tests for each provider transformer, and operational safeguards (`require_parameter_descriptions`, `max_retries`, `args_validator`, `args_valid` event flag). Deductions: (1) the JSON Schema generator reaches into Pydantic internals (`pydantic._internal._generate_schema.GenerateSchema`, `pydantic.plugin._schema_validator.create_schema_validator`) and the module explicitly acknowledges brittleness in its docstring (`pydantic_ai_slim/pydantic_ai/_function_schema.py:3`); (2) the default tool schema is permissive at the model-facing boundary — `additionalProperties` is only forced to `False` by the OpenAI transformer, not by the framework itself (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:393-419`); (3) tool description defaults propagate from the function docstring but the framework does not warn when neither docstring nor an explicit description is provided (a silent gap that can hurt model guidance).

## Answers to Dimension Questions

1. **Are schemas generated or handwritten?** Generated. Every Python function passed to `Tool`, `@agent.tool`, `@agent.tool_plain`, or `FunctionToolset.tool/tool_plain` is fed through `function_schema()` which uses Pydantic's internal `_generate_schema.GenerateSchema` to build a TypedDict core schema, then renders it to JSON Schema via `GenerateToolJsonSchema` (`pydantic_ai_slim/pydantic_ai/_function_schema.py:103-281`, `pydantic_ai_slim/pydantic_ai/tools.py:427-433`). Hand-written JSON schemas are supported via `Tool.from_schema()` (`pydantic_ai_slim/pydantic_ai/tools.py:582-636`) and used by MCP / external toolsets.

2. **Are descriptions useful?** Yes — they are extracted from the docstring with `griffe` for Google / NumPy / Sphinx (or `'auto'`), merged with `use_attribute_docstrings=True` per-parameter docstrings, attached to the function (as `ToolDefinition.description`) and to every parameter (`FieldInfo.description`) (`pydantic_ai_slim/pydantic_ai/_function_schema.py:125,146,191-192`, `pydantic_ai_slim/pydantic_ai/_griffe.py:18-80`). A `require_parameter_descriptions=True` flag fails registration with a `UserError` listing missing params (`pydantic_ai_slim/pydantic_ai/_function_schema.py:175-176,209-216`; test `tests/test_tools.py:1050-1063`).

3. **Is validation strict?** At the validator level yes: the default TypedDict is built with `extra_behavior='forbid'` so unknown keys and type mismatches raise `pydantic.ValidationError` (`pydantic_ai_slim/pydantic_ai/_function_schema.py:346-352`). At the model-facing schema level, strictness is provider-driven (OpenAI transformer adds `additionalProperties: false` and forces all properties required, `pydantic_ai_slim/pydantic_ai/profiles/openai.py:393-419`; Anthropic / Bedrock only when `strict=True`, `pydantic_ai_slim/pydantic_ai/providers/anthropic.py:158-165`, `pydantic_ai_slim/pydantic_ai/providers/bedrock.py:83-91`). Invalid args are not silently dropped — they are converted to a `RetryPromptPart` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:183-191`).

4. **Are defaults handled clearly?** Yes. Defaults are encoded in JSON Schema (`default: ...`) for the model to see (`tests/test_tools.py:1080-1129`), passed as Python defaults to the underlying function via the TypedDict field (`pydantic_ai_slim/pydantic_ai/_function_schema.py:184-199`), and surfaced in the rendered code-mode signature (`pydantic_ai_slim/pydantic_ai/function_signature.py:194-209`). OpenAI strict mode removes `default` to remain strict-compatible (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:344-351`), so providers without strict support see defaults while strict backends do not.

5. **Are schemas portable across providers?** Yes, with documented caveats. The base schema is a Pydantic-generated JSON Schema; every supported provider applies its own `JsonSchemaTransformer` to make it wire-compatible. `OpenAIJsonSchemaTransformer` (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:309-419`), `AnthropicJsonSchemaTransformer` (`pydantic_ai_slim/pydantic_ai/providers/anthropic.py:121-187`), `BedrockJsonSchemaTransformer` (`pydantic_ai_slim/pydantic_ai/providers/bedrock.py:61-122`), `GoogleJsonSchemaTransformer` (`pydantic_ai_slim/pydantic_ai/profiles/google.py:105-146`), and `InlineDefsJsonSchemaTransformer` (`pydantic_ai_slim/pydantic_ai/_json_schema.py:201-208`) cover the main families. Each one sets `is_strict_compatible` so `Tool(strict=None)` resolves to `True`/`False` per provider (`pydantic_ai_slim/pydantic_ai/models/__init__.py:1288-1299`). Lossy fields (`minLength`, regex lookaround, `dict` fields in strict mode) are preserved in the field's `description` text rather than being silently dropped (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:374-383`, `pydantic_ai_slim/pydantic_ai/providers/bedrock.py:113-121`).

## Architectural Decisions

- **Single source of truth for the schema**: every consumer (model wire, validator, code-mode signature) reads from `ToolDefinition.parameters_json_schema` (`pydantic_ai_slim/pydantic_ai/tools.py:643`, `853-865`). Provider-specific rewrites happen via a transformer at the request-prep step rather than mutating the stored schema (`pydantic_ai_slim/pydantic_ai/models/__init__.py:1288-1309`).
- **Reuse the validator across sources**: the same `pydantic_core.SchemaValidator` produced by `function_schema()` is wired as `args_validator` for function toolsets and MCP toolsets (`pydantic_ai_slim/pydantic_ai/toolsets/function.py:636`, `pydantic_ai_slim/pydantic_ai/mcp.py:1128`). MCP tools skip schema generation entirely and use the auto-validator from MCP's schema (`pydantic_ai_slim/pydantic_ai/mcp.py:2462-2471`), giving one uniform validation path.
- **Separation of schema validation and execution**: `ToolManager.validate_tool_call()` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:419`) runs only validation; execution is a separate `_run_execute_hooks()`. This lets the agent emit accurate `args_valid` flags on `FunctionToolCallEvent` (`tests/test_streaming.py:4332-4390`) and lets approval / deferred flows short-circuit execution.
- **Two-layer validation**: Pydantic schema validation + an optional `args_validator` (`pydantic_ai_slim/pydantic_ai/tools.py:84-94`). The custom validator receives already-converted typed args and can raise `ModelRetry` for business-rule failures. Both layers' errors map to the same `ToolRetryError` / `RetryPromptPart` shape (`pydantic_ai_slim/pydantic_ai/tool_manager.py:183-191`).
- **Custom `GenerateToolJsonSchema`**: only one override (`_named_required_fields_schema` strips property titles, `pydantic_ai_slim/pydantic_ai/tools.py:427-433`); the rest is delegated to Pydantic. This minimizes Pydantic-internal coupling at the schema level, at the cost of depending on `pydantic._internal._generate_schema` at the *core schema* level (`pydantic_ai_slim/pydantic_ai/_function_schema.py:103-235`).
- **Capability-shaped extensibility**: validation and execution are wrapped by `before_/wrap_/after_tool_validate` and `before_/wrap_/after_tool_execute` hooks (`pydantic_ai_slim/pydantic_ai/tool_manager.py:252-292,294-354`). New behaviors (audit, telemetry, retry policies) compose without modifying the base path.
- **Provider-specific schema quirks live in transformers, not in core**: keys that OpenAI rejects, Gemini ignores, or Bedrock rewrites are stripped by the per-provider transformer with their values preserved as text in `description` (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:374-383`, `pydantic_ai_slim/pydantic_ai/providers/bedrock.py:113-121`). The framework never silently drops a model constraint.

## Notable Patterns

- **TypedDict + wrap-validator for single-arg tools**: `function_schema` detects a single `BaseModel` / `TypedDict` / dataclass argument and emits a `no_info_wrap_validator_function` that unwraps `{name: value}` envelopes and lets the model see the inner schema directly (`pydantic_ai_slim/pydantic_ai/_function_schema.py:320-345,360-381`).
- **`extra_behavior='forbid'` by default**: rejects unexpected keys at the validator level without relying on model-side cooperation (`pydantic_ai_slim/pydantic_ai/_function_schema.py:346-352`).
- **Self-resolving placeholder pattern**: `_build_and_register_type` inserts an empty `TypeSignature` placeholder before recursing into a schema's properties to break cycles in recursive models (`pydantic_ai_slim/pydantic_ai/function_signature.py:703-723`).
- **Per-tool conflict renaming in code-mode**: `FunctionSignature.get_conflicting_type_names` finds type-name collisions across tools and emits `<tool>_<Type>` prefixed names so the rendered signatures stay parseable (`pydantic_ai_slim/pydantic_ai/function_signature.py:344-378`).
- **Capability-aware schema inclusion**: `_prepare_return_schemas` clears return schemas that aren't opted-in, warns when `include_return_schema=True` is set but the schema is empty, and injects the schema into the tool description only for models that don't natively support it (`pydantic_ai_slim/pydantic_ai/models/__init__.py:1368-1399`).
- **VCR-backed provider snapshots**: every provider transformer has snapshot tests with realistic models (recursive, regex patterns, ranges, mixed objects) so lossy rewrites are caught in CI (`tests/profiles/test_openai.py:97-173`; `tests/providers/test_bedrock.py:381-876`).

## Tradeoffs

- **Dependency on Pydantic internals** — the module's own docstring states it is brittle to Pydantic changes because it uses `_decorators`, `_generate_schema`, and `plugin._schema_validator.create_schema_validator` (`pydantic_ai_slim/pydantic_ai/_function_schema.py:1-4,125-235`). This buys a richer schema than a handwritten approach but makes upgrades a release-time risk.
- **Strict mode is provider-scoped, not framework-scoped** — schemas leaving the framework are not always `additionalProperties: false` or all-required; that is the OpenAI transformer's job (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:393-419`). Models without a strict transformer (e.g., Gemini, non-OpenAI openai-compatible endpoints) rely on the validator rather than the model to catch extra fields.
- **Lossy strict rewrites preserve info in text** — OpenAI and Bedrock strip fields they can't express and append `key=value` notes to the description (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:374-383`, `pydantic_ai_slim/pydantic_ai/providers/bedrock.py:113-121`). This keeps the model informed but means strict mode is not lossless and `is_strict_compatible = False` triggers for tools that depend on the stripped constraints.
- **`GenerateToolJsonSchema` is shallow** — only the per-property `title` is removed; everything else relies on the per-provider transformer. A user wanting deeper customization must write their own `schema_generator` and pass it through every entry point.
- **`require_parameter_descriptions` is opt-in** — without it, tools with empty docstrings ship to the model with `description: None`, which can hurt tool selection (`pydantic_ai_slim/pydantic_ai/_function_schema.py:175-176`; documented in `pydantic_ai_slim/pydantic_ai/tools.py:478`).
- **Retry budget is per-tool, not per-error** — `_check_max_retries` only counts tool-call retries; once exhausted the entire run fails with `UnexpectedModelBehavior` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:177-181`).

## Failure Modes / Edge Cases

- **Recursive schemas**: handled via the placeholder pattern in `function_signature.py:703-723` and via `_contains_ref` short-circuit in `check_object_json_schema` (`pydantic_ai_slim/pydantic_ai/_utils.py:155-184`); recursive refs are intentionally preserved.
- **Streaming partial JSON**: `allow_partial='trailing-strings'` is set when streaming is active so deltas can be validated mid-stream (`pydantic_ai_slim/pydantic_ai/tool_manager.py:233-243`).
- **Tool with no annotation**: `function_schema` falls back to `Any` for the unannotated param (`pydantic_ai_slim/pydantic_ai/_function_schema.py:153-158`), validated by `tests/test_function_signature.py:594-602`.
- **`Self` return annotation on unbound function**: `_extract_return_schema_type` falls back to `Any` (`pydantic_ai_slim/pydantic_ai/_function_schema.py:409-413`); tested by `tests/test_tools.py:4211-4221`.
- **Incompatible Pydantic return types**: a `UserWarning` is emitted and the schema collapses to `{}` (`pydantic_ai_slim/pydantic_ai/_function_schema.py:256-267`); downstream, `_prepare_return_schemas` warns again when the resolved schema is empty (`pydantic_ai_slim/pydantic_ai/models/__init__.py:1382-1390`).
- **Tool renamed through `prepare`**: when `Tool.prepare` rewrites the tool's name, `FunctionToolset.get_tools` raises `UserError` on conflicts with other (possibly also-renamed) tools (`pydantic_ai_slim/pydantic_ai/toolsets/function.py:625-630`).
- **`Tool.from_schema`** with no schema: function args are not validated; only the optional `args_validator` runs (`pydantic_ai_slim/pydantic_ai/tools.py:582-636`). This is a footgun if a user assumes Pydantic validation applies.
- **Model retries exceeded**: invalid args bubble up through `_check_max_retries` and the agent run raises `UnexpectedModelBehavior`, terminating the agent loop (`pydantic_ai_slim/pydantic_ai/tool_manager.py:177-181`; test `tests/test_tools.py:2358-2371`).
- **Unknown tool name**: `ModelRetry('Unknown tool name: ...')` is raised with a list of available names so the model can self-correct (`pydantic_ai_slim/pydantic_ai/tool_manager.py:356-370`).
- **OpenAI strict toggle absence**: if `openai_supports_strict_tool_definition=False`, the framework still strips fields it considers strict-incompatible in `strict=None` mode (setting `is_strict_compatible=False`) but does not pass `strict=True` on the wire — providers / users must set the flag explicitly (`pydantic_ai_slim/pydantic_ai/profiles/openai.py:91-92`).
- **Bedrock `botocore` too old**: a runtime warning is emitted and the request goes out without `strict` (`pydantic_ai_slim/pydantic_ai/providers/bedrock.py:518-526`).

## Future Considerations

- A future refactor toward "build JSON schema from the type directly" is hinted at in `_json_schema.py:23-24` as an alternative to the current transformer walker.
- The framework should consider warning (rather than silently accepting) when `Tool(description=None)` ships with no docstring — the gap is observable (`pydantic_ai_slim/pydantic_ai/_function_schema.py:245-249` only repromotes the parameter's description when the tool itself has none).
- Schema generation depends on Pydantic's `_internal._generate_schema`; a hardening pass to reduce surface area would lower upgrade risk (`pydantic_ai_slim/pydantic_ai/_function_schema.py:1-4`).
- Strict-mode capability is currently opt-in (`strict=True` or model profile flags). Adding a per-tool capability that auto-decides based on schema lossiness would smooth the Bedrock / Anthropic strict-mode UX, where today a user must set `strict=True` explicitly (`pydantic_ai_slim/pydantic_ai/providers/anthropic.py:121-148`, `pydantic_ai_slim/pydantic_ai/providers/bedrock.py:61-91`).

## Questions / Gaps

- Is there a public, stable contract for "the wire schema"? The per-provider transformers are not part of the public API surface — only `JsonSchemaTransformer` is (`pydantic_ai_slim/pydantic_ai/_json_schema.py:15-24`). Tests treat the post-transformer schema as observable behavior via snapshots (`tests/profiles/test_openai.py:97-173`).
- How does the framework plan to handle newer OpenAI tool features beyond what `OpenAIJsonSchemaTransformer` already encodes (e.g., the Responses-API `defer_loading`, `tool_choice`)? Currently it adds `additionalProperties: false` per-tool when needed (`pydantic_ai_slim/pydantic_ai/models/openai.py:2577-2579`) but does not have a per-feature transformer hook.
- The docstring-extraction step relies on `griffe`; the framework does not pin a major version in any test fixture I inspected. Whether multi-version CI coverage exists would determine whether a `griffe` major bump can regress schema descriptions silently — no evidence found in the selected source.

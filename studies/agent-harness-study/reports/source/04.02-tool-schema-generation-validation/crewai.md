## Summary

CrewAI's tool schema story is built on Pydantic v2 `BaseModel` subclasses: tool authors either subclass `BaseTool` (declare `args_schema: type[BaseModel]`) or use the `@tool` decorator, and CrewAI generates JSON Schema via `BaseModel.model_json_schema()` for both text-rendered prompts and provider-bound native function-calling. Before execution, `_validate_kwargs` (kwargs path) or `_parse_args` (string path) run the schema with `model_validate(...)`, raising a `ValueError` augmented with `build_schema_hint(...)` so the LLM gets the expected properties/required list when it hallucinates. The same schema is sanitized for each provider through `sanitize_tool_params_for_{openai,anthropic,bedrock}_strict` in `crewai/utilities/pydantic_schema_utils.py`, and a parallel `create_model_from_schema` reverse path rebuilds Pydantic classes from inbound JSON Schema for A2A responses, MCP tools, and project configs. Validation is strict by default (`additionalProperties=False`, all properties required for strict mode), invalid kwargs raise `ValueError`, and the framework retries up to `_max_parsing_attempts` times before emitting a `ToolUsageError`. There are two parallel schemas per tool (`args_schema` and `result_schema`), with `_format_tool_output_for_agent` revalidating the output against `result_schema` (warning + fallback on mismatch). Native call paths in `crew_agent_executor._execute_single_native_tool_call` and `execute_single_native_tool_call` route args through `parse_tool_call_args` (JSON parse only) and rely on `_validate_kwargs` inside `tool.run()` to enforce the schema, while provider adapters (`BaseToolAdapter` subclasses) emit provider-specific JSON Schema for OpenAI Agents and LangGraph.

## Rating

**8 / 10**

Rationale:

- Clear, single source of truth (Pydantic models) for argument types; explicit `args_schema` and `result_schema` fields with full descriptions (`tools/base_tool.py:139-158`).
- Generated JSON Schema is sanitized per provider with dedicated pipelines for OpenAI/Anthropic/Bedrock strict mode (`utilities/pydantic_schema_utils.py:605-635`), and there are tests confirming recursive schemas survive sanitization (`tests/utilities/test_pydantic_schema_utils.py:937-952`).
- Validation is strict: required-field failures, type mismatches, and unknown kwargs raise `ValueError` with a schema hint before the tool body runs (`tools/base_tool.py:272-293`, `tools/structured_tool.py:272-294`); tests assert the throw and the incremented-error path (`tests/tools/test_base_tool.py:265-318`, `tests/agents/test_native_tool_calling.py:1340-1365`).
- Schema is portable across providers through separate sanitizers and provider adapters (`agents/agent_adapters/{openai_agents,langgraph}/`).
- However, the tool description presented to the model is itself a hand-formatted string with the JSON Schema inlined (`tools/base_tool.py:482-490`), which can drift from what the provider actually enforces in strict mode. The text-rendering path does not reuse `sanitize_tool_params_for_openai_strict` for the strict-mode schema it sends (`utilities/agent_utils.py:184-189` passes `strip_null_types=False`). Positional args bypass validation entirely (`tools/base_tool.py:319`, `tests/tools/test_base_tool.py:285-296`), and `_validate_kwargs` strips unknown kwargs by default — both are pragmatic but undocumented for the model. Retry/backoff logic for invalid args is asymmetric (tool-side retry, `_max_parsing_attempts=2..3`), and there is no observable signal to the model about what changed between attempts beyond the formatted error string.

## Answers to Dimension Questions

1. **Are schemas generated or handwritten?**
   - Generated whenever the author does not provide one. `BaseTool._default_args_schema` introspects `inspect.signature(cls._run)` (and falls back to `_arun`) and calls `pydantic.create_model(...)` to produce `{ClassName}Schema` (`lib/crewai/src/crewai/tools/base_tool.py:200-247`). The same pattern exists in `Tool.from_langchain` (`lib/crewai/src/crewai/tools/base_tool.py:410-456`) and the `@tool` decorator (`lib/crewai/src/crewai/tools/base_tool.py:707-747`). Handwritten schemas are accepted via `args_schema: type[BaseModel] = MyModel`; both `BaseTool` and `CrewStructuredTool` use a `BeforeValidator(_deserialize_schema)` to round-trip JSON Schema → Pydantic (`lib/crewai/src/crewai/tools/structured_tool.py:126-135`, `lib/crewai/src/crewai/tools/base_tool.py:200-212`).

2. **Are descriptions useful?**
   - Field descriptions are carried through Pydantic `Field(description=...)` and end up in the JSON Schema under `properties.<name>.description`. CrewAI's own agent tools annotate every field (`lib/crewai/src/crewai/tools/agent_tools/ask_question_tool.py:9-11`, `lib/crewai/src/crewai/tools/agent_tools/delegate_work_tool.py:9-12`). However, the text rendering layer (`_generate_description`) embeds the JSON schema inside the tool description string rather than letting the provider use the structured schema; the `strict` provider pipeline at `lib/crewai/src/crewai/utilities/agent_utils.py:184-189` keeps `strip_null_types=False`, so optional fields still appear as `anyOf` rather than being dropped.

3. **Is validation strict?**
   - Yes on the kwarg and string invocation paths. `_validate_kwargs` runs `args_schema.model_validate(kwargs)` and re-raises as `ValueError("Tool '<name>' arguments validation failed: <err><hint>")` (`lib/crewai/src/crewai/tools/base_tool.py:272-293`). `CrewStructuredTool._parse_args` does the same for JSON-string inputs (`lib/crewai/src/crewai/tools/structured_tool.py:272-294`). On failure, `tool_usage._use/_ause` increments `_run_attempts`, retries up to `_max_parsing_attempts` (default 3, 2 for "big" OpenAI models), and emits `ToolUsageError` once exhausted (`lib/crewai/src/crewai/tools/tool_usage.py:662-686`). Strict mode for OpenAI/Anthropic/Bedrock is enforced through `force_additional_properties_false` + `ensure_all_properties_required` (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:181-217`, `lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:421-459`).
   - There are three pragmatic escapes: positional `run(...)` bypasses validation entirely (`lib/crewai/src/crewai/tools/base_tool.py:319`), `tool_usage._use` filters args against `args_schema.model_json_schema()["properties"].keys()` so unknown keys are silently dropped (`lib/crewai/src/crewai/tools/tool_usage.py:571-587`), and the native path's `parse_tool_call_args` only does JSON parsing, deferring to `tool.run(**args_dict)` which then routes back through `_validate_kwargs` (`lib/crewai/src/crewai/utilities/agent_utils.py:1636-1663`).

4. **Are defaults handled clearly?**
   - Defaults are part of the Pydantic model. The auto-inference path distinguishes `param.empty` from `param.default` and feeds the right side into `create_model(...)` (`lib/crewai/src/crewai/tools/base_tool.py:225-245`). The result-schema path treats a missing annotation as `Any` and falls back to `inspect.signature(func).return_annotation` when `get_type_hints` raises (`lib/crewai/src/crewai/tools/structured_tool.py:40-51`). `_format_tool_output_for_agent` revalidates the output against the inferred/declared result schema and emits a `RuntimeWarning` on failure before falling back to `str(raw_result)` (`lib/crewai/src/crewai/tools/structured_tool.py:54-83`). One subtle point: `_generate_description` reads `args_schema` before the model is fully validated, so the rendered text describes the placeholder schema (`lib/crewai/src/crewai/tools/base_tool.py:267-269`).

5. **Are schemas portable across providers?**
   - Yes, through dedicated sanitizers and adapters:
     - OpenAI completion: `sanitize_tool_params_for_openai_strict(params)` (`lib/crewai/src/crewai/llms/providers/openai/completion.py:1622-1624`).
     - Anthropic completion: `sanitize_tool_params_for_anthropic_strict(parameters)` when `strict=True` (`lib/crewai/src/crewai/llms/providers/anthropic/completion.py:517-522`).
     - Bedrock: reuses Anthropic sanitizer (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:627-635`).
     - OpenAI Agents SDK: `OpenAIAgentToolAdapter` calls `tool.args_schema.model_json_schema()` then `force_additional_properties_false` (`lib/crewai/src/crewai/agents/agent_adapters/openai_agents/openai_agent_tool_adapter.py:131-146`).
     - LangGraph: `LangGraphToolAdapter` builds a `langchain_core.tools.StructuredTool` carrying the same `args_schema` (`lib/crewai/src/crewai/agents/agent_adapters/langgraph/langgraph_tool_adapter.py:82-91`).
   - The reverse path (`create_model_from_schema`) keeps MCP, A2A, and `crewai project` JSON configs round-trippable (`lib/crewai/src/crewai/mcp/tool_resolver.py:633-636`, `lib/crewai/src/crewai/a2a/utils/task.py:339`, `lib/crewai/src/crewai/project/json_loader.py:1607-1609`).

## Architectural Decisions

- **Pydantic v2 as the single schema language.** `BaseTool` extends `BaseModel` and uses `Field(description=...)` for every user-facing field (`lib/crewai/src/crewai/tools/base_tool.py:103-191`). This forces every tool to ship with rich metadata, not just names.
- **Two-schema contract per tool.** `args_schema` (input) and `result_schema` (output) are first-class Pydantic fields (`lib/crewai/src/crewai/tools/base_tool.py:149-158`). The result schema is reused to format the tool output back to the model via `_format_tool_output_for_agent` (`lib/crewai/src/crewai/tools/structured_tool.py:54-83`).
- **Auto-inference as a developer-experience shortcut.** `_default_args_schema` and `Tool.from_langchain` read Python type hints to build a Pydantic model so the developer does not have to declare an `args_schema` for trivial tools (`lib/crewai/src/crewai/tools/base_tool.py:200-247`, `lib/crewai/src/crewai/tools/base_tool.py:410-456`).
- **Round-tripping through JSON Schema.** Both `BaseTool` and `CrewStructuredTool` use `BeforeValidator(_deserialize_schema)` + `PlainSerializer(_serialize_schema)` (or `@field_serializer`) so that tool instances serialize as JSON Schema dicts and reload as Pydantic classes (`lib/crewai/src/crewai/tools/structured_tool.py:126-135`, `lib/crewai/src/crewai/tools/base_tool.py:160-170`). This is what makes checkpoint persistence and the `crewai project` JSON loader work.
- **Provider-specific strict sanitizers as a layer on top of Pydantic's JSON Schema.** `_common_strict_pipeline` enforces a shared minimum (resolve `$ref`, close objects, mark all properties required, strip metadata) and OpenAI/Anthropic/Bedrock add their own restrictions on top (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:594-635`).
- **Strict mode default for native tool calling.** `_convert_tools_for_interference` (OpenAI) and the Anthropic completion both pass `strict=True` through to the sanitizer (`lib/crewai/src/crewai/llms/providers/openai/completion.py:1600-1627`, `lib/crewai/src/crewai/llms/providers/anthropic/completion.py:490-535`).
- **Error feedback to the model.** `build_schema_hint` returns a `Expected arguments: {...}\nRequired: [...]` suffix that is concatenated to the validation error so the LLM sees the schema on retry (`lib/crewai/src/crewai/tools/structured_tool.py:90-108`, `lib/crewai/src/crewai/tools/base_tool.py:289-292`).
- **Bounded retry budget.** `_max_parsing_attempts` is explicit (3 default, 2 for `OPENAI_BIGGER_MODELS`) and the `ToolUsage` class emits a `ToolUsageError` once exceeded, so infinite hallucination loops are impossible (`lib/crewai/src/crewai/tools/tool_usage.py:100-119`, `lib/crewai/src/crewai/tools/tool_usage.py:662-686`).
- **Adapters for cross-framework use.** `BaseToolAdapter` is the abstract base for `OpenAIAgentToolAdapter` and `LangGraphToolAdapter`, isolating provider-specific schema emission behind a uniform `configure_tools(tools)` interface (`lib/crewai/src/crewai/agents/agent_adapters/base_tool_adapter.py:13-40`).

## Notable Patterns

- **Field-level descriptions as documentation.** Every Pydantic field in the built-in agent tools has `description=...` (`lib/crewai/src/crewai/tools/agent_tools/add_image_tool.py:10`, `lib/crewai/src/crewai/tools/agent_tools/ask_question_tool.py:9-11`, `lib/crewai/src/crewai/tools/agent_tools/delegate_work_tool.py:9-12`); the framework propagates them straight into the JSON Schema that LLMs see.
- **Cycle-safe JSON Schema → Pydantic.** `_json_schema_to_pydantic_type` and `_build_model_from_schema` use an `in_progress` map keyed by `id(schema_dict)` to short-circuit recursive `$ref` to a `ForwardRef` instead of blowing the stack (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:1138-1143`, `lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:895-953`). Tests confirm it works for self-referential schemas (`lib/crewai/tests/utilities/test_pydantic_schema_utils.py:937-964`).
- **Strict-mode metadata stripping.** `_STRICT_METADATA_KEYS` and `_CLAUDE_STRICT_UNSUPPORTED` enumerate the JSON Schema keys that strict providers reject, then `_strip_keys_recursive` removes them in one pass (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:515-569`).
- **Auto-inference of `result_schema` from return type annotation.** `_infer_result_schema_from_callable` only accepts `BaseModel` subclasses; everything else is `None` and the formatter falls back to `str(...)` (`lib/crewai/src/crewai/tools/structured_tool.py:40-83`).
- **Usage-counter as a side-channel.** `_claim_usage` and `current_usage_count` are tracked alongside validation, so a tool that fails validation does not consume its `max_usage_count` budget (`lib/crewai/src/crewai/tools/base_tool.py:295-312`, `lib/crewai/tests/tools/test_base_tool.py:312-318`).
- **Tool name sanitization for cross-provider compatibility.** `sanitize_tool_name` is applied everywhere tool names flow out to the LLM or to the wire (`lib/crewai/src/crewai/tools/tool_usage.py:155`, `lib/crewai/src/crewai/utilities/agent_utils.py:132`).
- **JSON-repair fallback for malformed LLM output.** `ToolUsage._validate_tool_input` walks `json.loads → ast.literal_eval → json5.loads → json_repair.repair_json` before giving up (`lib/crewai/src/crewai/tools/tool_usage.py:884-930`).

## Tradeoffs

- **Text-rendered description vs. provider JSON Schema drift.** The text prompt gets `Tool Arguments: {<json schema>}` inlined into the description string (`lib/crewai/src/crewai/tools/base_tool.py:482-490`), while the native path sends a sanitized version. The two can diverge because `convert_tools_to_openai_schema` keeps `strip_null_types=False` (`lib/crewai/src/crewai/utilities/agent_utils.py:184-189`), whereas the strict sanitizer strips `null` from `anyOf`/`type` arrays (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:462-512`).
- **Positional args bypass validation.** `_validate_kwargs` is gated on `if not args` (`lib/crewai/src/crewai/tools/base_tool.py:319`, `lib/crewai/src/crewai/tools/base_tool.py:517-518`). This is convenient but means tool author who overload `run(x, y)` is unverified against the JSON Schema the model sees. The behaviour is intentional (`lib/crewai/tests/tools/test_base_tool.py:285-296`) but undocumented for the model.
- **Strict mode drops optional fields by default.** `generate_model_description(..., strip_null_types=True)` (the default) removes `null` from `anyOf`/`type`, so optional fields are surfaced as "required" with no null alternative (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:638-672`). That matches OpenAI's strict requirement but breaks LLMs that send `null` for unset fields.
- **Schema built from raw function signature.** `_default_args_schema` ignores Pydantic-only types like `Field(default_factory=list, ge=...)`; the only metadata forwarded is annotation + default. Authors who want `min_length`, `pattern`, etc. must hand-write a Pydantic model (`lib/crewai/src/crewai/tools/base_tool.py:214-247`).
- **Native-path argument filtering hides errors.** `tool_usage._use` quietly drops unknown kwargs and only retries on `Exception` raised by `tool.ainvoke` (`lib/crewai/src/crewai/tools/tool_usage.py:571-587`). Hallucinated extra keys therefore get silently ignored rather than surfaced as a schema error.
- **`use_stop_words` / parsing_attempts are model-aware but stringly typed.** `OPENAI_BIGGER_MODELS` is a hand-maintained `Literal[...]` list (`lib/crewai/src/crewai/tools/tool_usage.py:44-62`). Adding a new model requires editing this list, otherwise strict-mode retries stay at the default 3 attempts.
- **`create_model_from_schema` does not support all OpenAPI features.** Unsupported `type` values raise `ValueError` (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:1250-1252`) — for example, the code cannot represent `integer`/`number` JSON Schema integers with arbitrary bit width, only Python's `int`/`float`.

## Failure Modes / Edge Cases

- **Required field missing in LLM response → retry then bail.** `tool_usage._use` catches the validation exception, increments `_run_attempts`, retries up to `_max_parsing_attempts` (3 default, 2 for "big" OpenAI models), and on exhaustion emits a `ToolUsageError` formatted for the model to read as text (`lib/crewai/src/crewai/tools/tool_usage.py:662-686`).
- **Wrong type for a field → `ValueError` from Pydantic.** `_validate_kwargs` and `_parse_args` both raise `ValueError("Tool '<name>' arguments validation failed: <err><hint>")` (`lib/crewai/src/crewai/tools/base_tool.py:284-292`, `lib/crewai/src/crewai/tools/structured_tool.py:289-294`).
- **Strict provider rejects schema before invocation.** When the LLM rejects the schema, `is_native_tool_calling_unsupported_error` matches known phrasings (`lib/crewai/src/crewai/utilities/agent_utils.py:1285-1288`) and `build_text_tool_calling_fallback_message` downgrades to text tool calling (`lib/crewai/src/crewai/utilities/agent_utils.py:1291-1304`).
- **Recursive schema → infinite `$ref` resolution.** `_inline_top_level_ref` and the `in_progress` map prevent stack overflow during `create_model_from_schema` (`lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:766-782`, `lib/crewai/src/crewai/utilities/pydantic_schema_utils.py:895-953`); cycle-handling tests assert survival (`lib/crewai/tests/utilities/test_pydantic_schema_utils.py:937-964`).
- **JSON-decode failure from model output.** `ToolUsage._validate_tool_input` cascades through `json.loads` → `ast.literal_eval` → `json5.loads` → `json_repair.repair_json` and finally emits `ToolValidateInputErrorEvent` (`lib/crewai/src/crewai/tools/tool_usage.py:884-948`).
- **Result does not match `result_schema`.** `_format_tool_output_for_agent` warns and falls back to `str(raw_result)`; the model still gets a string, just not a structured one (`lib/crewai/src/crewai/tools/structured_tool.py:54-83`).
- **Tool name typo / fuzzy match.** `_select_tool` uses `SequenceMatcher` with a 0.85 similarity threshold; if no tool clears the bar it raises with a list of available tools (`lib/crewai/src/crewai/tools/tool_usage.py:759-802`).
- **Tool-call id absent.** `extract_tool_call_info` falls back to `f"call_{id(tool_call)}"` so the assistant message can still be built (`lib/crewai/src/crewai/utilities/agent_utils.py:1204-1230`).
- **MCP tool returns no schema.** `MCPNativeTool`/`MCPToolWrapper` fall back to a generic description string and rely on `_validate_kwargs` at runtime to catch bad input (`lib/crewai/src/crewai/tools/mcp_native_tool.py:46-57`, `lib/crewai/src/crewai/tools/mcp_tool_wrapper.py:36-48`).
- **No `args_schema` declared.** `BaseTool._default_args_schema` generates one from `_run` signature; `_validate_kwargs` early-exits if the resulting schema has no fields (`lib/crewai/src/crewai/tools/base_tool.py:209-247`, `lib/crewai/src/crewai/tools/base_tool.py:284-293`).
- **`max_usage_count <= 0`.** `validate_max_usage_count` raises `ValueError` at instantiation (`lib/crewai/src/crewai/tools/base_tool.py:260-265`).

## Future Considerations

- **Description-as-text vs. native strict schema divergence.** Either align the text-rendered description with the strict-mode schema or remove the inlined JSON when `strict=True` is in effect, so the model is not misled by a different schema than what the provider actually enforces. Today the divergence is silent.
- **Validation enforcement on the native path.** `parse_tool_call_args` only parses JSON; the schema check happens inside `tool.run(...)` via `_validate_kwargs`. A top-level validation step inside `_execute_single_native_tool_call` would catch issues before hook/event emission and surface them as `ToolValidateInputErrorEvent` for parity with the text path (`lib/crewai/src/crewai/utilities/agent_utils.py:1636-1663` vs. `lib/crewai/src/crewai/tools/tool_usage.py:884-948`).
- **Carry Pydantic metadata into auto-inferred schemas.** `_default_args_schema` only forwards annotation + default. Supporting `Annotated[..., Field(min_length=...)]` or docstring parsing would let `BaseTool` authors declare constraints without writing a separate `BaseModel` (`lib/crewai/src/crewai/tools/base_tool.py:214-247`).
- **Stop-words / attempt-budget per model, not per `OPENAI_BIGGER_MODELS` list.** The current literal list (`lib/crewai/src/crewai/tools/tool_usage.py:44-62`) ages poorly; routing the retry budget through the LLM provider metadata would remove the manual `Literal[...]`.
- **Telemetry for schema-validation failures.** Today `task.increment_tools_errors()` and `ToolUsageErrorEvent` are emitted, but there is no per-tool counter to detect chronic hallucination. A simple `validation_failures[tool_name]` dict would make operator dashboards actionable.
- **Cleaner openai/agent-style schema divergence.** `OpenAIAgentToolAdapter` and `convert_tools_to_openai_schema` each apply `force_additional_properties_false` independently (`lib/crewai/src/crewai/agents/agent_adapters/openai_agents/openai_agent_tool_adapter.py:134`, `lib/crewai/src/crewai/utilities/agent_utils.py:182-189`); consolidating through `generate_model_description` would keep one canonical place for strictness.
- **Positional-arg validation opt-out.** Positional calls skip `_validate_kwargs` by design (`lib/crewai/src/crewai/tools/base_tool.py:319`), but the framework could surface an opt-in flag (e.g., `BaseTool(..., validate_positional=True)`) for security-sensitive agents without breaking backwards compatibility.

## Questions / Gaps

- **What happens when `args_schema` raises during `model_validate_json` inside `ainvoke` for the native path?** `_execute_single_native_tool_call` swallows the exception via the outer `try/except` in `tool_func(**args_dict)` (`lib/crewai/src/crewai/utilities/agent_utils.py:1530-1534`, `lib/crewai/src/crewai/agents/crew_agent_executor.py:868-891`) but the error path emits `ToolUsageErrorEvent`; the model only sees the resulting string. No `ToolValidateInputErrorEvent` is emitted on the native path.
- **Does `_generate_description` run before or after the auto-inference of `args_schema`?** `_default_args_schema` is a `field_validator(mode="before")` so the placeholder is replaced before `model_post_init` calls `_generate_description` (`lib/crewai/src/crewai/tools/base_tool.py:200-247` vs. `lib/crewai/src/crewai/tools/base_tool.py:267-269`). I did not find a regression test for a tool that *only* declares `_run` and verifies the rendered description.
- **Is `strip_null_types=False` for the native path intentional?** It diverges from the strict-mode sanitizer default; tests in `tests/utilities/test_converter.py` do not assert this behaviour (`lib/crewai/src/crewai/utilities/agent_utils.py:184-189`).
- **No evidence found** for: an "invalid argument correction" prompt that re-sends the failed schema to the model with structured feedback. The hint is concatenated to the error string but no retry-loop scaffolding nudges the LLM with the corrected schema beyond the next parsing attempt.
- **No evidence found** for: tooling that warns authors at decoration time if their `_run` signature cannot be expressed in JSON Schema (e.g., `*args`, `**kwargs`). The auto-inference code skips `VAR_POSITIONAL`/`VAR_KEYWORD` silently (`lib/crewai/src/crewai/tools/base_tool.py:220-221`).
- **No evidence found** for: a centralized place to register per-tool descriptions that flow to providers — descriptions are read directly off `tool.description`, which is mutated by `_generate_description` for text-tool calls but not for the native path (`lib/crewai/src/crewai/tools/base_tool.py:482-490` vs. `lib/crewai/src/crewai/utilities/agent_utils.py:194-196`).

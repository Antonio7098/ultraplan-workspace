## Summary

LangGraph's recovery model is built around *threaded checkpoints*: every superstep durably persists a `Checkpoint` (state snapshot + channel versions + versions_seen) plus a parallel table of `pending_writes` keyed by `(thread_id, checkpoint_ns, checkpoint_id, task_id, idx)`. After a crash, recovery is *implicit and lazy*: on the next invocation with the same `thread_id`, `SyncPregelLoop.__enter__` / `AsyncPregelLoop.__aenter__` calls `checkpointer.get_tuple(config)` to fetch the latest checkpoint, hydrates channels from `channel_values`, then re-attaches the unsaved `pending_writes` via `checkpoint_pending_writes` and the writer re-runs only the tasks that have no recorded writes (`libs/langgraph/langgraph/pregel/_loop.py:1631`, `libs/langgraph/langgraph/pregel/_loop.py:1888`). Nodes that already committed writes before the crash are skipped (`_reapply_writes_to_succeeded_nodes` at `libs/langgraph/langgraph/pregel/_loop.py:724`); nodes that failed with `ERROR_SOURCE_NODE` are re-routed to their error handlers rather than re-executed (`_resume_error_handlers_if_applicable` at `libs/langgraph/langgraph/pregel/_loop.py:739`).

Reconstruction of `DeltaChannel` state on resumption is the only "non-trivial" rebuild path: a `DeltaChannel` is absent from `channel_values` unless a periodic snapshot fired, so restoration walks the parent chain via `BaseCheckpointSaver.get_delta_channel_history` to find the nearest `_DeltaSnapshot` seed and replay the in-between writes (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-690`; `libs/langgraph/langgraph/pregel/_checkpoint.py:136-184`). Postgres and SQLite implement optimized two-stage ancestor walks (Postgres: `libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:444-551`; SQLite: `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:503-583`). There is no separate worker/orphan sweeper; "orphaned runs" are simply runs whose last checkpoint never advanced — the operator invokes the same `graph.invoke(None, config)` and the loop resumes at the latest durable point. Cooperative shutdown is implemented via `RunControl.request_drain()` (`libs/langgraph/langgraph/runtime.py:79-104`), which raises `GraphDrained` at the next superstep boundary (`libs/langgraph/langgraph/errors.py:54-64`) and, with `durability="exit"`, flushes the pending checkpoint before exiting (`libs/langgraph/langgraph/pregel/_loop.py:1294-1311`).

Idempotency is achieved by `INSERT OR IGNORE` / `ON CONFLICT DO NOTHING` for writes and `INSERT OR REPLACE` / `ON CONFLICT DO UPDATE` for checkpoints and blobs in SQLite/Postgres (`libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:462-465`; `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:131-159`), combined with task-ID-based write deduplication in `InMemorySaver.put_writes` (`libs/checkpoint/langgraph/checkpoint/memory/__init__.py:500-509`). There is no explicit dead-letter queue; an unrecoverable failure is materialized as a `(ERROR, exc)` write in `checkpoint_writes` (PG) / `writes` (SQLite) and surfaced to the next caller via `get_tuple`.

## Rating

**8 / 10** — Clear recovery model with explicit, durable, idempotent primitives, tested error-handler resumption and time-travel flows, and observable in-flight progress via checkpoints/pending writes; weaker on long-tail ops visibility (no orphan sweeper, no built-in alerting) and on delta-channel "no parent" silent reconstruction (documented but not auto-repaired).

## Answers to Dimension Questions

1. **What happens to in-progress work after restart?**
   The loop persists a `Checkpoint` (state snapshot + versions) *after* every superstep via `self.submit(self._checkpointer_put_after_previous, ...)` (`libs/langgraph/langgraph/pregel/_loop.py:1182-1189`), and pending channel writes for each task via `checkpointer.put_writes` (`libs/langgraph/langgraph/pregel/_loop.py:474-487`). On restart, `__enter__` fetches the latest `CheckpointTuple`, restores channels via `channels_from_checkpoint`, reattaches `pending_writes`, and the loop re-enters at the same superstep — completed tasks are skipped via `_reapply_writes_to_succeeded_nodes` (`libs/langgraph/langgraph/pregel/_loop.py:724-737`); failed tasks are either retried (if they have a retry policy) or routed to error handlers via `_resume_error_handlers_if_applicable` (`libs/langgraph/langgraph/pregel/_loop.py:739-804`).

2. **Are orphaned runs detected?**
   There is no orphan detector. "Orphaned" simply means the latest persisted checkpoint is not the terminal one; on the next call with the same `thread_id`, the loop continues from that checkpoint with the saved `pending_writes` (`libs/langgraph/langgraph/pregel/_loop.py:1631`, `libs/langgraph/langgraph/pregel/_loop.py:1888`). Operators see orphan runs by inspecting `checkpoint_writes` (a `(ERROR, exc)` row that was never retried) or `get_state_history`. The CLI does not ship a sweeper; `langgraph-cli` only includes `"restart": "on-failure"` for the optional debugger container (`libs/cli/langgraph_cli/docker.py:152`).

3. **Is recovery automatic?**
   Yes — for the same process thread. Recovery happens implicitly on the next `invoke`/`ainvoke`/`stream`/`astream` for the same `thread_id`. There is no "operator-initiated" recovery command distinct from a normal invocation; the runner treats the second call exactly like a continuation, including applying Command(resume=...) for human-in-the-loop (`libs/langgraph/langgraph/pregel/_loop.py:891-919`).

4. **Is recovery idempotent?**
   Idempotent at the *storage layer*: SQLite uses `INSERT OR IGNORE` / `INSERT OR REPLACE` (`libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:462-465`), Postgres uses `ON CONFLICT DO NOTHING` / `ON CONFLICT DO UPDATE` (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:131-159`), `InMemorySaver.put_writes` skips duplicate non-control writes (`libs/checkpoint/langgraph/checkpoint/memory/__init__.py:500-509`). At the *application layer*, re-running the same invoke is also safe because `put_writes` is keyed by `(task_id, idx)` and completed tasks are reattached via `_reapply_writes_to_succeeded_nodes`. Re-entrance under the same `run_id` is recognized and re-applies input instead of treating the call as fresh (`libs/langgraph/langgraph/pregel/_loop.py:847-860`).

5. **Are users/operators notified?**
   Yes, through: (a) Graph lifecycle callback events `GraphResumeEvent` / `GraphInterruptEvent` with `status` and `checkpoint_id` (`libs/langgraph/langgraph/callbacks.py:39-71`; `libs/langgraph/langgraph/pregel/_loop.py:369-406`); (b) `RunControl.drain_reason` exposing the reason a cooperative drain was triggered (`libs/langgraph/langgraph/runtime.py:79-104`); (c) `GraphDrained` exception carrying the drain reason when the run is short-circuited (`libs/langgraph/langgraph/errors.py:54-64`); (d) checkpoint metadata (`source`, `step`, `parents`, `run_id`, `counters_since_delta_snapshot`) inspectable via `get_state_history` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:38-87`; `libs/langgraph/langgraph/pregel/_loop.py:1108-1110`); (e) embedded pending-write error markers (`WRITES_IDX_MAP[ERROR] = -1`) that surface through `get_state` / `get_state_history` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:795`).

## Architectural Decisions

- **Thread-id as the unit of resumability** — every `Checkpoint` row is keyed by `(thread_id, checkpoint_ns, checkpoint_id)`. There is no separate "run" or "job" identifier; recovery granularity is the thread.
- **Pending writes are first-class** — channel writes for a superstep are persisted in a separate `checkpoint_writes`/`writes` table under the same `(thread_id, checkpoint_id)` as the checkpoint. This makes "partial superstep" recovery explicit and atomic-from-the-callers-perspective: if node A succeeds and node B fails, only A's writes are durably attached to that checkpoint, and on resume B is retried while A is skipped.
- **Three durability modes** — `sync`, `async`, `exit` (`libs/langgraph/langgraph/types.py:87-93`). `sync` blocks on `put` before next step starts; `async` returns a future so the next step can overlap with persistence (`libs/langgraph/langgraph/pregel/_loop.py:1182-1189`); `exit` only flushes when the loop exits normally, on a drain, or on an interrupt (`libs/langgraph/langgraph/pregel/_loop.py:1294-1311`).
- **DeltaChannel as opt-in compression** — channels with large/append-only state can opt into delta-encoding so checkpoints only carry a sentinel unless a periodic snapshot fires. Reconstruction on load walks the parent chain via `get_delta_channel_history` (`libs/langgraph/langgraph/pregel/_checkpoint.py:161-184`).
- **Checkpointer conformance suite** — `langgraph-checkpoint-conformance` runs capability-based tests against any `BaseCheckpointSaver` implementation (`libs/checkpoint-conformance/langgraph/checkpoint/conformance/validate.py:45-128`), ensuring all storage backends honor the same recovery contract.
- **Cooperative drain instead of forced kill** — `RunControl.request_drain()` flips a flag rather than terminating the process; the loop checks it at superstep boundaries (`libs/langgraph/langgraph/pregel/_loop.py:650-652`) and exits cleanly with `GraphDrained`. The external watcher (langgraph-api) is responsible for SIGKILL if no `finish` event arrives in time (`libs/langgraph/tests/test_retry.py:1805-1977`).
- **Pending-write durability gating** — checkpoint `put` waits on (a) the previous checkpoint's put future and (b) all in-flight `put_writes` futures for delta channels, preventing a checkpoint from becoming visible without its backing writes (`libs/langgraph/langgraph/pregel/_loop.py:1515-1524`, `libs/langgraph/langgraph/pregel/_loop.py:1769-1778`).

## Notable Patterns

- **Three-table storage (Postgres) / two-table storage (SQLite)** with checkpoints + writes + (PG-only) blobs, all PK-scoped by `(thread_id, checkpoint_ns[, checkpoint_id[, task_id, idx]])`. Channel blobs are stored separately in Postgres (`checkpoint_blobs`) so checkpoint JSONB stays small; SQLite embeds them inline.
- **Sentinel encoding for control writes** — `ERROR`, `SCHEDULED`, `INTERRUPT`, `RESUME` get negative `idx` values via `WRITES_IDX_MAP` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:795`) so they never collide with regular writes and can be deduplicated/upserted independently.
- **Lazy stub on first run with `durability="exit"`** — if the saver returns `None` for the very first call, the loop fabricates an `empty_checkpoint` with `step=-2` and captures it as `_initial_checkpoint_config`; the exit-mode accumulator later creates a real "stub" parent if needed (`libs/langgraph/langgraph/pregel/_loop.py:1633-1652`; `libs/langgraph/langgraph/pregel/_loop.py:1235-1258`).
- **Two-stage DeltaChannel reconstruction (Postgres)** — dynamic SQL builds `K` parallel JSONB key lookups for stage 1 (per-checkpoint version + has-seed probes), then a per-channel UNION ALL in stage 2 to fetch writes + seed blob in one roundtrip (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:478-543`).
- **Streaming DeltaChannel reconstruction (SQLite)** — stage 1 cursor-based parent walk deserializes one checkpoint at a time, then stage 2 issues per-channel `IN (?, ?, ...)` predicates; the `?` placeholder count varies per branch (`libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/_delta.py:42-90`).
- **Walk-once base-class fallback** — `BaseCheckpointSaver.get_delta_channel_history` provides a generic implementation that calls `get_tuple` per ancestor when storage-specific overrides don't exist, so custom savers don't have to implement delta-history logic to be safe (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:582-649`).
- **Resume-as-Command(resume=...)** — same `graph.invoke(Command(resume=...), config)` works for both human-in-the-loop continuation and post-crash continuation; detection reuses `CONFIG_KEY_RESUMING` or the input-shape heuristic (`libs/langgraph/langgraph/pregel/_loop.py:836-919`).

## Tradeoffs

- **Implicit orphan handling vs. explicit sweep** — there's no separate worker that reaps runs left in "halfway done" state for a long time. Operators must call the API; if no one calls, the half-checkpointed state lingers in storage.
- **DeltaChannel silent empty reconstruction** — if the entire ancestor chain is missing (e.g., manually pruned), `get_delta_channel_history` returns no `seed`, the consumer treats it as "start empty" (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:596-603`), and the channel silently rebuilds from nothing. The conformance suite + `test_prune.py` documents this; no automatic alarm.
- **`InMemorySaver` is process-local** — a process restart loses everything. The README explicitly recommends it only for debugging (`libs/checkpoint/langgraph/checkpoint/memory/__init__.py:40-41`); production deployments must use Postgres/SQLite.
- **Async-IO SQLite bridge uses cross-thread `run_coroutine_threadsafe`** — `AsyncSqliteSaver.put_writes` calls the loop from a worker thread and synchronously waits for the result; this serializes writes but blocks the caller thread (`libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/aio.py:246-255`).
- **SQLite single-writer lock** — `SqliteSaver` uses a `threading.Lock` around every cursor (`libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:95`, `libs/checkpoint-sqlite/langgraph/checkpoint/sqlite/__init__.py:181-189`) so concurrent graph runs against the same DB serialize. WAL mode helps readers but not writers.
- **`durability="exit"` puts nothing durable until exit** — if the process dies between superstep N and the final `__exit__`, the last superstep's writes are lost. Operators must understand that "exit" trades durability for throughput (`libs/langgraph/langgraph/pregel/_loop.py:1294-1311`).
- **Checkpoint writes are unbounded** — neither SQLite nor Postgres has a built-in retention/TTL for old checkpoints; `BaseCheckpointSaver.prune` is the API (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`), but it is the operator's responsibility to call it.

## Failure Modes / Edge Cases

- **Process crash mid-superstep before any task completes** — checkpoint exists from previous superstep, no `pending_writes` for the new one. On resume, loop re-runs all tasks of the superstep from scratch (idempotent given deterministic channel versions).
- **Process crash after some tasks commit `put_writes` but before the next checkpoint** — only those task writes are durable. On resume, completed tasks are reapplied via `_reapply_writes_to_succeeded_nodes`, the rest are retried. Pending writes survive SQLite WAL / Postgres ACID.
- **Process crash after `put_writes` for delta-channel writes but before checkpoint `put`** — `_delta_write_futs` draining logic in `_checkpointer_put_after_previous` is what protects against this; if the process dies *between* `put_writes` and `put`, the writes will exist but the checkpoint referencing them won't. The next `_get_tuple` will see the previous checkpoint and replay would not see the orphan writes (they are anchored under the next checkpoint_id). Documented as a known visibility invariant (`libs/langgraph/langgraph/pregel/_loop.py:199-205`).
- **Partial network failure between PostgresSaver writes** — `UPSERT_CHECKPOINTS_SQL` is `DO UPDATE`; if a crash occurs mid-write, the row reflects the last fully-committed state. Postgres WAL guarantees atomicity of the row update.
- **Schema drift** — if the checkpoint table is from an old `MIGRATIONS` version, `setup()` runs the missing migrations in order (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:85-110`).
- **Pending-sends from pre-v4 checkpoints** — `_migrate_pending_sends` rewrites legacy `checkpoint_writes` rows into the `TASKS` channel on read (`libs/checkpoint-postgres/langgraph/checkpoint/postgres/__init__.py:164-188`; `libs/checkpoint-postgres/langgraph/checkpoint/postgres/base.py:308-326`).
- **Cooperative drain in async loop while writes are still inflight** — `_suppress_interrupt` flushes pending delta writes + checkpoint + remaining `pending_writes` before raising `GraphDrained` (`libs/langgraph/langgraph/pregel/_loop.py:1294-1311`).
- **Sync `time.sleep` inside async node defeats `idle_timeout`** — `NodeTimeoutError` is not raised; the node's normal return flows through. The only safety net is the external watcher (langgraph-api) SIGKILLing on no-`finish` deadline. Documented as a "Cooperative cancellation" caveat in `TimeoutPolicy` (`libs/langgraph/tests/test_retry.py:1805-1977`).

## Future Considerations

- **`DELETE FOR RUNS`** — `BaseCheckpointSaver.delete_for_runs` (`libs/checkpoint/langgraph/checkpoint/base/__init.py:331-348`) is part of the contract; tested via conformance, but no built-in scheduled cleanup exists. Operators would need to schedule reaper jobs.
- **Per-thread retention/TTL** — `BaseCheckpointSaver.prune` (`libs/checkpoint/langgraph/checkpoint/base/__init__.py:374-415`) supports `keep_latest` and `delete` strategies; the SQLite store has a sweeper (`libs/checkpoint-sqlite/tests/test_ttl.py:25-92`), but the API isn't yet automatic.
- **DeltaChannel silent-empty on pruned ancestors** — the conformance suite documents this and recommends either retaining ancestors, forcing a snapshot before prune, or skipping threads that use `DeltaChannel`. A future "auto-snapshot before prune" helper could close this gap.
- **Watcher-deadline integration** — the `CONFIG_KEY_TIMED_ATTEMPT_OBSERVER` event stream is the in-process observability hook used by the langgraph-api external watcher. Today this is fully separate from the checkpointer; if a third-party deployment does not implement the watcher, processes can hang indefinitely on cooperative timeouts.
- **Async-IO SQLite serialization** — `AsyncSqliteSaver` serializes writes through a single asyncio loop. For high-throughput agents this is the bottleneck; for production the docs explicitly recommend Postgres.

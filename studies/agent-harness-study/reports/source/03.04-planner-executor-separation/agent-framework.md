## Summary

The framework ships **two distinct, fully implemented planner/executor architectures** that are both first-class in the public API:

1. **Magentic One orchestration** (Microsoft's port of the Magentic-One reference architecture). A dedicated `MagenticOrchestrator` executor (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:859`; .NET: `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:80`) drives a separate `MagenticManager` (Python `StandardMagenticManager` at `_magentic.py:513`; .NET `MagenticManager` at `Specialized/Magentic/MagenticManager.cs:13`) that produces and persists a **Task Ledger** (facts + plan) and a **Progress Ledger** (satisfaction, loop detection, progress, next speaker, instruction). Plans are emitted as first-class workflow events (`MagenticPlanCreatedEvent`, `MagenticReplannedEvent`, `MagenticProgressLedgerUpdatedEvent`) — Python `_magentic.py:781-795`, .NET `MagenticOrchestrator.cs:17-58` — and replanning is triggered automatically on stalls or human revision. Both implementations support human-in-the-loop plan sign-off (Python `MagenticPlanReviewRequest`/`MagenticPlanReviewResponse` at `_magentic.py:797-855`, .NET `MagenticPlanReviewRequest.cs:18` and `MagenticPlanReviewResponse.cs:15`).

2. **Harness "plan/execute mode + todo list"** pattern. `create_harness_agent` (Python `python/packages/core/agent_framework/_harness/_agent.py:239`) and `HarnessAgent` (.NET `dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgent.cs:84`) ship by default an `AgentModeProvider` that tracks a `plan` vs `execute` mode (Python `_mode.py:200`, .NET `AgentModeProvider.cs:42`) plus a `TodoProvider` for todo-list management (Python `_todo.py:447`, .NET `TodoProvider.cs:42`). A `LoopAgent` decorator (.NET `Loop/LoopAgent.cs:65`) and `AgentLoopMiddleware` (Python `_loop.py:212`) re-invoke the wrapped agent until a `LoopEvaluator`/`should_continue` predicate — typically a `TodoCompletionLoopEvaluator` (Python `todos_remaining` helper at `_loop.py:751`, .NET `TodoCompletionLoopEvaluator.cs:42`) — decides the todo list is exhausted.

Plans are **persisted**: the Magentic `TaskLedger` is serialized in `MagenticTaskState` (.NET `Specialized/Magentic/MagenticTaskContext.cs:29`) and `MagenticContext` + `_MagenticTaskLedger` (Python `_magentic.py:336-396` / `_magentic.py:270-285`), and survives checkpoint round-trips (.NET `OnCheckpointingAsync` at `MagenticOrchestrator.cs:355-396`; Python `on_checkpoint_save`/`on_checkpoint_restore` at `_magentic.py:1266-1321`). The harness's todos persist via `AgentSessionStateBag` (.NET `TodoProvider.cs:63`) or `AgentSession.state` (Python `_todo.py:246-285`) and via an optional `TodoFileStore` (Python `_todo.py:287-444`) that atomically writes JSON state.

The orchestrator **controls execution**: in Magentic, the `MagenticOrchestrator` reads the progress ledger, picks the next speaker, dispatches `TurnToken`, and resets participant state on replan (.NET `MagenticOrchestrator.cs:184-345`, Python `_magentic.py:1050-1185`). In the harness, the plan/execute mode string is injected into the system prompt on every invocation (.NET `AgentModeProvider.cs:194-220`, Python `_mode.py:261-325`), and `TodoCompletionLoopEvaluator` will keep invoking the agent until todos are closed.

Plan changes are **traceable**: Magentic emits `MagenticReplannedEvent` whenever the task ledger is regenerated, and the dotnet orchestrator records `MagenticTaskContextKey` and `CurrentSpeakerStateKey` checkpoint keys (`MagenticConstants.cs:5-9`). The harness exposes a `mode_set` notification user message (Python `_mode.py:285-325`, .NET `AgentModeProvider.cs:208-217`) and the `TodoProvider` keeps a structured `{id, title, description, is_complete}` record per item (.NET `Todo/TodoItem.cs:10`, Python `TodoItem` at `_todo.py:46`).

## Rating

**9 / 10** — Two mature, distinct planner/executor architectures with explicit plan/progress ledgers, named events for every transition, persisted state (including a real checkpoint round-trip), bounded re-planning (stall counter + reset counter + round counter + max-iterations safety cap), and human-in-the-loop plan sign-off as a first-class optional. Plans genuinely constrain execution: the orchestrator routes by the plan's next-speaker field and the harness mode string drives the system prompt. Plan changes are observable via emitted events and checkpoint state, so "what was planned vs what was done" is reconstructible. Deductions: the harness pattern (mode + todos) is **advisory** — the plan string lives in the system prompt and the todo list is maintained by the model itself via tools; nothing in `TodoProvider` enforces that the agent actually closes todos before responding. The Magentic `progress_ledger` is also LLM-judged JSON with retry/backoff (`MagenticManager.UpdateProgressLedgerAsync` at `MagenticManager.cs:66-106`), so the orchestrator's view of progress can be wrong. Finally, no `MagenticOrchestrator` unit-test file was located under `dotnet/tests/.../MagenticOrchestratorTests.cs` was confirmed present but coverage of `MaxResetCount` enforcement is partial.

## Answers to Dimension Questions

1. **Is there an explicit plan?**
   Yes, in both architectures.
   - **Magentic**: `TaskLedger(facts, plan)` (Python `_magentic.py:270-285`; .NET `Specialized/Magentic/MagenticTaskContext.cs:20`) is the explicit plan, produced by an LLM call inside `MagenticManager` (Python `_magentic.py:609-685`; .NET `Specialized/Magentic/MagenticManager.cs:36-64`). The plan is rendered into a single combined assistant message via the "Full" prompt template and added to the chat history so participants see it on every turn.
   - **Harness**: Two layered explicit plans — the `mode` string (`plan` vs `execute`) drives the system prompt (Python `_mode.py:14-30`; .NET `AgentModeProvider.cs:44-62`), and the `TodoList` is an itemized structured plan maintained by the agent through tools (Python `_todo.py:24-43`; .NET `TodoProvider.cs:44-61`).

2. **Does the plan control execution?**
   - **Magentic**: Yes, strongly. The orchestrator reads `ProgressLedger.NextSpeaker` and dispatches via `TurnToken` (Python `_magentic.py:1141-1147`; .NET `MagenticOrchestrator.cs:304-335`). It also injects an `InstructionOrQuestion` message aimed at the next speaker (Python `_magentic.py:1133-1139`; .NET `MagenticOrchestrator.cs:324-331`). On replan, all participants are reset via `ResetChatSignal` / `MagenticResetSignal` (Python `_magentic.py:1260-1264`; .NET `MagenticOrchestrator.cs:337-345`).
   - **Harness**: Indirectly. The mode string is injected into the system prompt (Python `_mode.py:252-259`; .NET `AgentModeProvider.cs:194-220`) and the todo list is re-injected as a synthetic user message on each invocation (Python `_todo.py:596-614`; .NET `TodoProvider.cs:154-179`). The agent's *behavior* is expected to differ in `plan` vs `execute` mode, but no code path enforces it — execution is shaped, not controlled. The loop wrapper *does* re-invoke the agent whenever `TodoCompletionLoopEvaluator` reports remaining items (Python `_loop.py:751-767`; .NET `TodoCompletionLoopEvaluator.cs:97-127`), so the harness does control whether the agent runs again.

3. **Can the plan change?**
   - **Magentic**: Yes, explicitly. `MagenticOrchestrator.ResetAndReplanAsync` clears context and calls `UpdatePlanAndDelegateAsync(replanAfterStall: true)` (.NET `MagenticOrchestrator.cs:337-345`). Python mirrors this in `_reset_and_replan` (`_magentic.py:1149-1185`). Replan is also triggered when the human revises a plan during sign-off (Python `_magentic.py:1015-1027`; .NET `MagenticOrchestrator.cs:152-158`).
   - **Harness**: The agent itself can call `todos_add` / `todos_complete` / `todos_remove` (Python `_todo.py:505-589`; .NET `TodoProvider.cs:203-349`) and `mode_set` (Python `_mode.py:289-297`; .NET `AgentModeProvider.cs:255-269`). External callers can also flip modes via `set_agent_mode` (Python `_mode.py:155-196`) and a notification message is injected on the next run (Python `_mode.py:315-325`; .NET `AgentModeProvider.cs:208-217`).

4. **Are plan changes traceable?**
   - **Magentic**: Yes. Plan changes emit `MagenticPlanCreatedEvent` (initial), `MagenticReplannedEvent` (subsequent), `MagenticProgressLedgerUpdatedEvent` (each round) (Python `_magentic.py:781-795`; .NET `MagenticOrchestrator.cs:17-58`). The full task ledger is serialized into checkpoint state under `MagenticTaskContextKey` (Python `_magentic.py:1273-1275`; .NET `MagenticOrchestrator.cs:359-362`) and restored via `MagenticTaskState` (.NET `Specialized/Magentic/MagenticTaskContext.cs:29`). The dotnet version additionally tracks `CurrentSpeakerStateKey` (.NET `MagenticConstants.cs:7-8`). Tests confirm round-trip across `checkpoint_id` resume (Python `test_magentic_checkpoint_resume_round_trip` at `_magentic.py:403-463`; .NET `MagenticOrchestrationTests.cs:112-146` for `PlanCreatedEvent`).
   - **Harness**: Yes for todos (the `TodoList` is structured state on the session). Mode changes inject a `[Mode changed: ...]` user message (Python `_mode.py:31-34`; .NET `AgentModeProvider.cs:215`), and the previous mode is recorded in `_PREVIOUS_MODE_STATE_KEY` (`_mode.py:73`, `_mode.py:194-195`). There is no event for todo list changes — only `todos_add` / `todos_complete` / `todos_remove` tool calls are visible in the model's tool-call transcript.

5. **Is planning useful or decorative?**
   Useful in both. Magentic's plan constrains who speaks next, what they are asked, and when the workflow terminates — without the orchestrator's plan-aware dispatch the system would degenerate into round-robin or chaos. The harness's plan is less binding (the model decides whether to follow the todo list), but `TodoCompletionLoopEvaluator` makes the loop terminate based on plan completion, so without the todo list the loop would either never run or run once. In both cases the planner contributes a *decision* (next speaker / continue-or-stop), not just decoration.

6. **Can the system compare what it planned to what it actually did?**
   - **Magentic**: Yes — the `MagenticProgressLedger` snapshots each round (`progress_ledger: MagenticProgressLedger.to_dict()` in `python/_magentic.py:1276-1278`; `MagenticProgressLedger.State` JsonElement in .NET `MagenticProgressLedger.cs:84`), the chat history is appended verbatim (`_taskContext.ChatHistory.AddRange(messages)` at `MagenticOrchestrator.cs:203`), and the task ledger survives checkpoint/restore. The progress ledger's `is_in_loop` / `is_progress_being_made` fields are themselves a built-in comparison between planned and observed progress.
   - **Harness**: Partially — the structured todo list state can be diffed against the agent's tool-call transcript to reconstruct which todos were completed, removed, or added and when. The mode transitions are similarly reconstructible from the injected notification messages. There is no first-class "plan vs actual" diff API.

## Architectural Decisions

- **Two parallel planner patterns, neither clearly dominant.** `Microsoft.Agents.AI.Harness` (`.NET`) and `agent_framework._harness` (Python) are the "single-agent-with-mode-and-todo" batteries-included path; `Microsoft.Agents.AI.Workflows/Specialized/Magentic` (`.NET`) and `agent_framework_orchestrations._magentic` (Python) are the "multi-agent-manager-and-ledger" orchestration path. The two are composable: a Magentic participant can itself be a `HarnessAgent`.
- **Magentic plan is an LLM artifact, not a hard program.** Plans are produced by calling the manager agent with prompts (`ORCHESTRATOR_TASK_LEDGER_PLAN_PROMPT`, `ToTaskLedgerPlanPrompt`) and stored as opaque `ChatMessage`s. The orchestrator only reads structural fields (`NextSpeaker`, `IsRequestSatisfied`, etc.) out of the progress ledger, which itself is LLM-produced JSON. This keeps plans flexible but means a malformed plan or progress ledger can silently misroute execution.
- **The harness plan is advisory, but the *loop* is a hard gate.** `TodoCompletionLoopEvaluator` enforces "keep running until todos are closed" regardless of whether the model thinks it is done. This is what keeps the harness from being purely decorative.
- **Plans are persistent by construction.** Both patterns use the framework's session-state abstractions (`AgentSessionStateBag` for .NET harness, `AgentSession.state` for Python) and the workflow checkpoint system (workflow `OnCheckpointingAsync` for Magentic, `CheckpointStorage` parameter to `MagenticBuilder`). A workflow interrupted between rounds can resume with the full plan, progress, and chat history intact (test: Python `_magentic.py:403-463`).
- **Human-in-the-loop plan sign-off is the canonical pattern for "useful planning", not a safety hack.** `MagenticPlanReviewRequest.Approve()` / `Revise(message)` are first-class public types; `RequirePlanSignoff(true)` is a builder flag. When sign-off is enabled, the orchestrator explicitly pauses and resumes via the workflow's `request_info` / `IWorkflowContext.RequestInfoAsync` mechanism.
- **Bounded replanning, not unbounded.** `TaskLimits(MaxStallCount=3, MaxRoundCount=null, MaxResetCount=null, MaxProgressLedgerRetryCount=3)` (.NET `MagenticTaskContext.cs:11-18`) and the Python `max_stall_count=3`, `max_reset_count=None`, `max_round_count=None`, `progress_ledger_retry_count=3` defaults (`_magentic.py:540-590`) prevent runaway replanning. Termination on `hit_round_limit || hit_reset_limit` yields a final assistant message (`_magentic.py:1244-1256`; .NET `MagenticOrchestrator.cs:250-258`).
- **Mode change notifications are explicit.** When an external caller flips mode (via `set_agent_mode` or `mode_set`), the next run injects a synthetic user message `[Mode changed: ...]` rather than just changing the system prompt — because system instructions render first and the model tends to anchor on the most recent `mode_set` tool call in history (Python `_mode.py:165-196`, `_mode.py:285-325`).

## Notable Patterns

- **Schema-validated progress ledger slots** (.NET `MagenticProgressLedger.cs:170-269`): `BooleanProgressLedgerSlot` and `StringProgressLedgerSlot` are typed records that build the JSON schema for the model's response and parse answers out of the response by slot key. Each slot carries a `FormattedQuestion`, `SchemaType` (`"boolean"` / `"string"`), and parses both `answer` and `reason` sub-fields. The Python equivalent inlines this into the prompt template (Python `_magentic.py:193-242`).
- **Event-sourced plan changes.** Every transition emits an event (`MagenticOrchestratorEvent` in both languages), so consumers can subscribe without coupling to the orchestrator's internals. Events are emitted through `ctx.add_event` / `context.AddEventAsync` on the workflow context, which preserves them in the workflow's event log and includes them in checkpoint state.
- **Checkpoint-first persistence.** Magentic plans survive checkpoint round-trips via `MagenticTaskState` (Python dataclass + `to_dict`/`from_dict`; .NET record) — not via ad-hoc re-hydration code. The Magentic test suite explicitly round-trips a workflow with a pending plan-review request (`_magentic.py:403-463`) and asserts the second workflow can resume from `checkpoint_id`.
- **Per-session locking on shared state.** Both `.NET` (`ConditionalWeakTable<AgentSession, SemaphoreSlim>` at `TodoProvider.cs:67-68`) and Python (`WeakKeyDictionary[AgentSession, asyncio.Lock]` at `_todo.py:484-492`) use weak-keyed maps so per-session locks are GC'd with their session.
- **Loop pattern as middleware (Python) vs decorator (.NET).** Python exposes looping as `AgentMiddleware` (`AgentLoopMiddleware` at `_loop.py:212`) so it composes with the existing middleware pipeline and is config-only on a regular `Agent`. .NET exposes looping as a `DelegatingAIAgent` subclass (`LoopAgent` at `Loop/LoopAgent.cs:65`) so it composes via the builder chain and is itself an `AIAgent` to be wrapped further.
- **Participant reset on replan.** `MagenticOrchestrator` broadcasts a `ResetChatSignal` / `MagenticResetSignal` after a stall (Python `_magentic.py:1260-1264`; .NET `MagenticOrchestrator.cs:342`), and each participant executor clears its cache, conversation, and rebuilds its session (Python `MagenticAgentExecutor.handle_magentic_reset` at `_magentic.py:1348-1366`). This prevents stale context from leaking into the next plan attempt.

## Tradeoffs

- **Plan flexibility vs determinism.** Magentic's plans are LLM-authored free text; routing depends on the model choosing a name from the participant list and writing valid JSON for the progress ledger. The framework defends against this with retries (`progress_ledger_retry_count`, default 3) and explicit fallbacks (`.NET MagenticOrchestrator.cs:308-311` chooses first participant when `next_speaker` is empty, `_magentic.py:1122-1125` mirrors it) but the orchestrator can still misinterpret an ambiguous reply.
- **Memory consumption during long replanning.** Magentic accumulates the entire `ChatHistory` plus `TaskLedger` plus `ProgressLedger` until reset; on `ResetChatSignal` the chat history is cleared (`_magentic.py:1160`) but the prior progress ledger and task ledger are replaced, not pruned. For workflows with many stalls this can grow without a hard GC.
- **Advisory vs enforced plans.** Harness mode + todos is *advisory*: the agent can call `todos_complete` without doing the work, or stay in `execute` mode while still asking clarifying questions. There is no callback that compares tool-call output to todo content. This is fine for a research-coding agent but the framework does not pretend otherwise.
- **Two planner architectures in one repo.** Beginners face a real cognitive load: should they reach for `create_harness_agent`, `MagenticBuilder`, both, or compose a custom workflow? The README/AGENTS documentation points at "harness = single-agent w/ batteries" vs "workflows/magentic = multi-agent orchestration", but the boundary between them is permeable — a `HarnessAgent` can be a Magentic participant.
- **Loop iteration caps apply to *wraps* of the agent, not to tool-call loops.** Python `_loop.py:117` and .NET `LoopAgent.cs:68` both default to 10 iterations. The `FunctionInvocationLayer` inside one wrap has its own `DEFAULT_MAX_ITERATIONS = 40` (`_tools.py:90`). When both run together, total LLM-call count is `outer × inner`, easy to underestimate.
- **Plan sign-off in dotnet defaults to `true`** (`MagenticWorkflowBuilder._requirePlanSignoff = true` at `MagenticWorkflowBuilder.cs:35`) but the Python builder defaults to `false` (`_magentic.py:883`). Code that ports between the two must flip the flag.

## Failure Modes / Edge Cases

- **Invalid JSON in progress ledger.** `.NET MagenticManager.UpdateProgressLedgerAsync` retries with backoff and throws on exhaustion (`Specialized/Magentic/MagenticManager.cs:66-106`). The orchestrator catches the throw and routes to `ResetAndReplanAsync` (`MagenticOrchestrator.cs:264-279`). The Python equivalent uses the same try/except pattern with `0.25*attempts` sleep (`_magentic.py:1080-1085`).
- **Unknown next speaker.** If the progress ledger names an agent not in the team, the orchestrator logs a warning and falls back to `PrepareFinalAnswerAsync` (`.NET MagenticOrchestrator.cs:312-320`; Python `_magentic.py:1128-1131`). This is silent — there is no event surfacing the misroute.
- **Reset limit exceeded.** `MagenticContext.CheckLimits` returns `(true, …)` for `ResetCount >= MaxResetCount` (Python `_magentic.py:1238-1256`; .NET `MagenticOrchestrator.cs:246-258`). The orchestrator yields a termination message and sets `IsTerminated = true`. Re-running the same workflow raises `RuntimeError("This Magentic workflow has already been completed...")` on the next invocation (Python `_magentic.py:914-918`; .NET `MagenticOrchestrator.cs:186-189`).
- **Plan signoff is required but no human responds.** The workflow stays in `IDLE_WITH_PENDING_REQUESTS` indefinitely — there is no timeout. A test asserts this state is correctly preserved across checkpoint round-trip (`test_magentic_checkpoint_resume_round_trip` at `_magentic.py:403-463`).
- **Todo file-store corruption.** `TodoFileStore._load_state_sync` raises on a malformed JSON file (`_todo.py:404-417`); atomic writes (`_todo.py:432-443`) protect against partial writes. A corrupted file therefore fails the next `todos_get_remaining` call rather than silently returning `[]`.
- **Mode transition race.** Python's `set_agent_mode` records `_PREVIOUS_MODE_STATE_KEY` so the next `before_run` can inject the notification (`_mode.py:190-196`). If two callers race, the last writer wins and the earlier notification is dropped. The .NET equivalent (`AgentModeProvider.cs:177-191`) has the same last-writer-wins semantics.
- **Mode change made by the agent itself does not produce a notification** (Python `_mode.py:289-297`; .NET `AgentModeProvider.cs:255-269`). The model already knows it made the tool call. Only *external* mode changes inject the announcement user message. The asymmetry is intentional but easy to miss.

## Future Considerations

- **Plan-vs-actual diff API.** Neither pattern exposes a structured "what was planned vs what happened" report. Building one for Magentic would be a single function over `MagenticProgressLedger` history + `ChatHistory`; for the harness it would be a diff between the structured `TodoList` state and the model's tool-call transcript. Both could ship as diagnostic helpers without changing the run-time architecture.
- **Replan trigger fidelity.** Today replanning fires on `IsStalled` (`.NET MagenticOrchestrator.cs:289-302`; Python `_magentic.py:1110-1118`) or human revise. A future improvement could let the orchestrator also replan on detected "scope drift" — the model's answers diverge from the task description — surfaced as an additional ledger slot.
- **Stable next-speaker resolution.** The progress ledger asks the model to pick from a comma-separated list (`ToProgressLedgerPrompt`, Python `_magentic.py:212`), but typos fall through to a warning-and-final-answer path. Enumerated-tool-call next-speaker would close the typo loophole without losing the JSON progress ledger for everything else.
- **Magentic per-participant progress.** Today the progress ledger tracks "is anyone making progress"; a per-participant progress slot would let the orchestrator detect one bad actor in an otherwise productive team.
- **Harness enforcement of plan completion.** The harness could attach an evaluator that compares `todos_complete` calls against the actual tool outputs in the transcript, rejecting completions that look vacuous. This would close the advisory-vs-enforced gap.
- **Cross-pattern composition.** A `HarnessAgent` used as a Magentic participant already works in principle, but there is no documentation showing the wiring. A sample combining `create_harness_agent` + `MagenticBuilder` would make this discoverable.

## Questions / Gaps

- **Are there other planner/executor orchestrators in the framework?** Yes — `Sequential`, `Concurrent`, `GroupChat`, `Handoff` are in the orchestrations package (`python/packages/orchestrations/agent_framework_orchestrations/_group_chat.py`, `_handoff.py`, `_sequential.py`, `_concurrent.py`). Only Magentic has a planner; the others are routing patterns without explicit plans. The dotnet equivalents are in `Microsoft.Agents.AI.Workflows`.
- **No evidence of a Magentic "plan diff" event.** Both versions emit `PlanCreated` and `Replanned` events but neither ships a "what changed in this replan" delta event. A user wanting plan-vs-actual must reconstruct it from successive `MagenticProgressLedgerUpdatedEvent` payloads.
- **The .NET harness plan-tracking is not a multi-step plan.** `TodoProvider` is a flat item list with no dependencies, no priorities, no grouping. There is no concept of a "step 2 blocks step 3" DAG. If the user wants DAG-style plans they need to encode dependencies inside the `Description` string.
- **Magentic participant threads vs Magentic session.** In Python, `MagenticAgentExecutor` does not support custom threads because replanning resets them (`_magentic.py:1339-1346`). This is documented but is a real constraint for long-lived participants.
- **The harness `mode` notification only fires on the *next* invocation.** If the user changes mode and then the agent runs multiple iterations within one outer call, only the first iteration sees the notification (`_mode.py:285-325`; .NET `AgentModeProvider.cs:208-217`). For harness agents that don't loop, this is moot, but combined with `AgentLoopMiddleware` it means the model may ignore a mid-loop mode flip.
- **No first-class "compare planned vs done" helper.** Both architectures could trivially compute "todos completed vs todos created" or "next-speaker chosen vs plan participant-list," but no public function ships this. A diagnostic `MagenticTrace.report()` / `HarnessTrace.report()` is a clear follow-up.
- **Tool-level retry policy for `todos_*` is not documented.** If `todos_complete` is called with an ID that does not exist (e.g., the todo was removed concurrently), the tool returns `completed: 0` (`_todo.py:558-561`) without raising. The model sees the silent no-op and may retry; this is the right behavior but is not surfaced to the planner layer.

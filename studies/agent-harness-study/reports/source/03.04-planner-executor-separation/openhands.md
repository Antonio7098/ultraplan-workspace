## Summary

{{approach_summary}}

OpenHands implements planner/executor separation at the **conversation/sub‑conversation level**, not inside the agent loop. A `DEFAULT` executor conversation and a `PLAN` planning conversation are distinct LLM sessions with different system prompts, different tool sets, and a shared workspace file (`PLAN.md`) as the contract between them. The plan is **persisted as plain Markdown in the sandbox filesystem** (`.agents_tmp/PLAN.md` or `agents-tmp-config/PLAN.md` for GitLab/Azure DevOps) — there is no structured plan schema, no step‑state machine, and no automatic comparison of plan versus execution. The executor is launched into the same workspace and receives a literal user instruction `Execute the plan based on the .agents_tmp/PLAN.md file` (`frontend/src/hooks/use-handle-build-plan-click.ts:27`). The actual agent loop, plan tool (`planning_file_editor`), replanning triggers, and execution control live in the external `openhands-agent-server` and `openhands-tools` packages pinned in `pyproject.toml` (lines 67–72); this repo exposes `AgentType`, plumbing, and tests for the seam, but does not implement the planner agent itself.

## Rating

**6/10** — A clear model with explicit types, distinct agent configs, persisted plan file, and unit tests for path computation and condenser behavior. Limited because the plan is an unstructured file (no step schema, no in‑process state machine, no plan‑vs‑actuals comparison); replanning on executor failure is only indirectly supported by editing the same `PLAN.md`; and traceability exists only through the event stream (every `PlanningFileEditorAction` / `PlanningFileEditorObservation` event is persisted, with the file content re‑read by the frontend WebSocket handler on observation).

## Answers to Dimension Questions

1. **Is there an explicit plan?** Yes — the planner agent is required to write `/workspace/project/.agents_tmp/PLAN.md` (or `agents-tmp-config/PLAN.md` on GitLab/Azure) before yielding control. The plan is a Markdown file, not a typed plan record: the planning constraint is enforced through the injected prompt suffix `PLANNING_AGENT_INSTRUCTION` (`live_status_app_conversation_service.py:185-196`) and a dedicated system prompt template `system_prompt_planning.j2` (`live_status_app_conversation_service.py:1332`). The only persistent representation of the plan content the UI uses is the workspace file — re-read from the agent-server on every relevant observation (`conversation-websocket-context.tsx:674-709`).
2. **Does the plan control execution?** Indirectly. There is no in-process state machine binding the plan to the executor. The executor receives the plan as a literal user message: `Execute the plan based on the .agents_tmp/PLAN.md file` (`use-handle-build-plan-click.ts:27`). The frontend Build button is disabled until `planContent` is set (`conversation-tab-title.tsx:36`) and until the agent is no longer running, which gates launch but does not enforce structural conformance. Per-phase rendering of one preview per user turn (`use-plan-preview-events.ts`) is the only mechanical feedback loop.
3. **Can the plan change?** Yes. The planner may `view` and `str_replace` `PLAN.md` via the `planning_file_editor` tool (declared as `tool_name: "planning_file_editor"` in `test-utils.tsx:181-243`); each edit emits a `PlanningFileEditorAction` / `PlanningFileEditorObservation` pair that lands in the event stream. The frontend watches for the **last** such observation per user phase and re-reads the file (`conversation-websocket-context.tsx:674-709` and `use-plan-preview-events.ts:41-104`). The executor can also re-enter plan mode: the Build/Plan toggle and `useHandlePlanClick` allow spawning additional planning sub-conversations against the same parent (`use-handle-plan-click.ts:42-92`).
4. **Are plan changes traceable?** Yes, but only through the append-only event stream stored by the agent server. Each planning action/observation carries an `id`, `timestamp`, `tool_call_id`, and `action_id`, and is persisted alongside regular events via `EventService.save_event` (`event/event_service.py:60-62`). The file itself is a sandbox filesystem path (cleared when the workspace is destroyed). There is no separate durable `Plan` table on the app server — replays derive plan state purely from event history and a re-read of the workspace file.
5. **Is planning useful or decorative?** Useful, with caveats. It is a real second LLM session with its own tool set and prompt: planning tools (`get_planning_tools(plan_path=plan_path)`) replace the default executor toolset (`get_default_tools(...)`), so the planner physically cannot run code or shell commands. Its only writes go through the restricted file editor. The hand-off into execution, however, is the same conversation-store toggle plus a literal plain-text instruction — the system does not parse steps out of the markdown or compare them against subsequent events. So planning is *enforced* (impossible to execute from plan mode) but *not coupled* (no schema linking plan steps to executor steps).

## Architectural Decisions

- **Plan as a file in the sandbox, not a typed record.** The plan is treated as a workspace artifact at `.agents_tmp/PLAN.md` (or `agents-tmp-config/PLAN.md` for providers whose filename constraints reject `.agents_tmp`). This keeps the sandbox‑local concept of "the plan document" portable across frontend, app server, and agent server, and lets the planner edit it with the same `str_replace_editor` machinery as any other file. Trade‑off: no schema to drive plan/execution comparison.
- **Two `AgentType` values, two complete agent configurations.** Rather than toggling a flag inside one agent, `AgentType.PLAN` swaps system prompt file, tool set (`get_planning_tools`), suffix text, and condenser `usage_id`. All four switches happen in one block (`live_status_app_conversation_service.py:1330-1373`, `1613-1625`).
- **Sub‑conversation as the planner boundary.** The plan agent is launched as a separate `AppConversation` whose `parent_conversation_id` points at the executor conversation. The parent/child relationship is a first‑class field on `AppConversationInfo` and survives deletion via cascade (`app_conversation_models.py:131-132`, `live_status_app_conversation_service.py:2203-2236`). Configuration inheritance from parent (`_inherit_configuration_from_parent`, lines 1014‑1042) keeps the workspace and git context aligned.
- **Plan gating is event‑driven and file‑driven on the frontend.** Plan *content* derives from re‑reading the agent‑server's filesystem via `GET /app-conversations/{id}/file` (`app_conversation_router.py:1078-1170`) on every `PlanningFileEditorObservation` whose path ends in `PLAN.md` (`conversation-websocket-context.tsx:674-709`). The Build button is disabled until `planContent` is non‑null and the agent is not running (`conversation-tab-title.tsx:36`).
- **Decoupling of planner loop from executor loop.** Because both sub‑conversations are independent agent‑server sessions with their own `OpenHands` LLM, plans and executions do not share LLM context, history, or step‑state. They are coupled only through the `PLAN.md` file as the contract.
- **SaaS analytics visibility per role.** Distinct `usage_id`s (`'agent'` vs `'planning_condenser'`) feed the Laminar tracer so planner versus executor traffic can be filtered. See the doc note at `AGENTS.md` ("`openhands/workspace/cloud/workspace.py`: `get_llm()` and `get_secrets()` return LookupSecret-backed objects") for the broader SaaS tracing pattern.

## Notable Patterns

- **Mode‑toggle agent selection on sub‑conversation creation.** `useHandlePlanClick` (`frontend/src/hooks/use-handle-plan-click.ts:42-92`) stores the planner child task ID in both `useConversationStore` and `localStorage` so a refresh can resume polling for the planning sub-conversation's start task.
- **Phase grouping of planning observations.** `groupEventsByPhase` (`use-plan-preview-events.ts:9-39`) wraps every user message; `findLastPlanningObservationInPhase` reduces multiple planning edits inside one user turn to a single visible preview — a UI‑side compensation for the lack of a typed plan/step commit boundary.
- **Tool restriction by composition.** `get_planning_tools(plan_path=plan_path)` (imported in `live_status_app_conversation_service.py:137`) returns a different SDK tool list from `get_default_tools(...)` (line 132). Because the planner's tools list is empty of executor tools, runtime tool gating is structural, not just a prompt instruction.
- **Prompt-level execution denial as defense in depth.** Even if the planner could call executor tools, `PLANNING_AGENT_INSTRUCTION` (`live_status_app_conversation_service.py:185-196`) repeats the "you CANNOT execute code" boundary, asks the planner not to ask "Ready to proceed?", and tells it the role ends at plan finalization. The system message is prepended to whatever suffix the user supplies (`live_status_app_conversation_service.py:1594-1600`).
- **History backfill via observed path.** `latestPlanningFileEventRef` (`conversation-websocket-context.tsx:683-687, 290-313`) captures the latest planning observation during history replay and triggers a one-shot file re-read once replay completes, so the UI plan content survives reconnects.
- **Provider-specific config directory.** `_compute_plan_path` switches between `.agents_tmp` and `agents-tmp-config` for GitLab and Azure DevOps because `.agents_tmp` is invalid on those providers (`live_status_app_conversation_service.py:1085-1091`; tests `test_compute_plan_path_gitlab_uses_agents_tmp_config` and `test_compute_plan_path_azure_uses_agents_tmp_config`, `tests/unit/app_server/test_live_status_app_conversation_service.py:857-877`).

## Tradeoffs

- **Strength — tool gating by composition.** The planner does not have executor tools installed, so attempts to execute from plan mode fail at the tool registry layer, not at the prompt or post‑hoc check.
- **Strength — explicit `AgentType` enum and start‑request field.** There is one canonical switch (`AgentType.DEFAULT` vs `AgentType.PLAN`) that flows from the API surface (`AppConversationStartRequest.agent_type`) to the agent server and is testable from this repo (`tests/unit/app_server/test_live_status_app_conversation_service.py`).
- **Weakness — plan is unstructured Markdown.** Without a plan schema, the system cannot enumerate plan steps, mark them done, or fail when an executor step diverges. UI phase grouping is the only progress approximation.
- **Weakness — no plan‑vs‑actuals comparison.** The Build hand‑off is a literal `Execute the plan based on the .agents_tmp/PLAN.md file` message (`use-handle-build-plan-click.ts:27`). There is no machine‑readable mapping from plan sections to executor events.
- **Weakness — replanning on failure is implicit.** The planner can re‑edit `PLAN.md` and be re‑invoked by the user clicking Plan again, but the agent‑server tool/loop and the orchestrator that detects failure is in the external `openhands-agent-server` package — no failure→replan logic is observable in this repo.
- **Weakness — plan persistence co‑located with workspace lifetime.** If the sandbox is stopped or replaced, `.agents_tmp/PLAN.md` is gone; the durable record is only the event stream. There is no app‑server‑side `Plan` table; the conversation store's `planContent` is a UI cache only.
- **Weakness — `_compute_plan_path` decision tied to provider type.** The branching hides from any single observer where the plan actually lives; only the dedicated test for each provider reveals it.

## Failure Modes / Edge Cases

- **Plan path misnamed on a new provider.** Adding a new git provider that rejects either `.agents_tmp` or `agents-tmp-config` would make `_compute_plan_path` fall into the GitHub branch and the planner could be told to write to a directory the provider forbids. The current code has no exception/fallback (`live_status_app_conversation_service.py:1085-1091`).
- **Planner runs in workspace with no git provider.** Returns `/workspace/project/.agents_tmp/PLAN.md`; this is correct only if the workspace itself allows that prefix.
- **Existence of a *pre-existing* `PLAN.md`.** The path resolution is unconditional — the planner always targets the same absolute path; whether the file existed before is left to the planning tool's idempotency. Tests verify path computation only, not collision behaviour (`test_compute_plan_path_*`, lines 844‑877).
- **Plan content deleted after Build.** If `PLAN.md` is removed between Build and Plan mode re-entry, `planContent` becomes `null` in the conversation store and `PlanPreview` quietly returns the "no plan" placeholder (`planner-tab.tsx:38-50`). The next plan attempt starts against a missing file.
- **Replanning mode lacks a step-status diff.** Because there is no plan schema, repeating the plan does not produce a delta; history replay reconstructs plan state from observation events, which the `use-plan-preview-events` hook reduces to "last observation wins per phase" — if the planner regresses, the preview will reflect that only after the file is re-read (`conversation-websocket-context.tsx:688-705`).
- **Sub-conversation start task polling on page refresh.** `useHandlePlanClick` (`use-handle-plan-click.ts:32-40`) restores `subConversationTaskId` from `localStorage` so the loading state survives a refresh; if the value is stale or stale-format, polling silently continues until completion.
- **Plan observation before history load.** A `PlanningFileEditorObservation` arriving during the initial history replay is captured in a ref and the file is re-read once `isLoadingHistoryPlanning` flips to false (`conversation-websocket-context.tsx:288-313`). A race where the ref is unset before the loading flag flips could leave the plan pane empty.
- **Planner condenser analytics tagging.** `_create_condenser` (`app_conversation_service_base.py:578-612`) sets `usage_id='planning_condenser'`; if the LLM isn't using a `tracked` model, the tag is set but the underlying metadata isn't decorated with `litellm_extra_body`. LLM metadata helpers emit only when `should_set_litellm_extra_body(model)` (`live_status_app_conversation_service.py:1340-1349`), so planning traffic can be invisible to telemetry on non‑OpenHands models.

## Future Considerations

- **Add a structured `Plan` schema** on the app server (steps with status, dependencies, owner), independent of the workspace file, to enable plan‑vs‑actuals comparison, replanning triggers on executor failure, and richer UI gating beyond "has plan / no plan".
- **Bridge executor failures back into the planner.** Today the parent→child handoff is one‑shot; a route that ingests executor `AgentErrorEvent` and asks the planner to revise steps would close the loop structurally rather than as a user click.
- **Make plan storage provider‑independent.** Lift `PLAN.md` out of the sandbox workspace lifetime into app‑server storage with its own retention, so plans survive a sandbox change.
- **Reduce reliance on prompt‑level planning boundaries.** The current `PLANNING_AGENT_INSTRUCTION` is a static string; pairing it with structural tool gating is good, but a schema‑driven planner that emits only typed plan operations would remove the role‑by‑prompt fallibility.

## Summary

Temporal is a workflow orchestration engine, not an LLM agent runtime, so it has **no token-level model streaming**. Instead, "streaming execution semantics" in this codebase map to four distinct concerns, each implemented with explicit, well-instrumented primitives:

1. **Cross-cluster replication bi-directional gRPC streams.** `StreamWorkflowReplicationMessages` is the only true server-streaming RPC in the API surface (`proto/internal/temporal/server/api/historyservice/v1/service.proto:350`, `:1027-1039`). Tasks are sent in `WorkflowReplicationMessages` batches carrying `ReplicationTask` + `ExclusiveHighWatermark` + `Priority` (`api/replication/v1/message.pb.go:619-628`), and ACKs flow back as `SyncReplicationState` messages (`api/replication/v1/message.pb.go:419`). Wrapped by `BiDirectionStreamImpl` (`service/history/replication/bi_direction_stream.go:49-174`) which manages lazy connection, a 512-slot recv channel with overflow metric, `CloseSend` on shutdown, and exponential-retry lifecycle via `WrapEventLoop` (`service/history/replication/stream.go:46-92`). Sender and receiver each run 2-3 long-lived goroutines (sendEventLoop, recvEventLoop, livenessMonitor) keyed by `ClusterShardKeyPair` (`service/history/replication/stream_receiver.go:132-153`, `service/history/replication/stream_sender.go:109-142`). A `StreamReceiverMonitor` reconciles inbound and outbound streams every 2 s on cluster metadata change or every 2 s tick, and a status monitor loop every 5 min detects stuck streams (`service/history/replication/stream_receiver_monitor.go:72-89`, `:299-369`).
2. **Long-poll task matching.** `PollWorkflowTaskQueue`, `PollActivityTaskQueue`, `PollNexusTaskQueue`, `QueryWorkflow`, and `ListNexusEndpoints` are unary RPCs categorized as `API_CATEGORY_LONG_POLL` (`proto/internal/temporal/server/api/matchingservice/v1/service.proto:18-58`). Each runs an outer `pollLoop` that re-enters on retry (`service/matching/matching_engine.go:688-689`, `:2560-2561`, `:2722-2742`). Long-poll expiry is bounded via `WithDeadlineBuffer(ctx, longPollInterval, returnEmptyTaskTimeBudget)` (`common/contextutil/deadline.go:16-39`, `service/matching/matching_engine.go:2874`) so the goroutine returns an empty task *just before* the caller's deadline rather than letting the gRPC stack count a context-deadline error against SLO. Forwarded poll jitter (`forwardedPollJitterRatio = 1/6`) caps minimum interval to `common.CriticalLongPollTimeout` (`service/matching/matching_engine.go:78-82`).
3. **Cancellation is two-tiered.** `outstandingPollers` (`collection.SyncMap[string, context.CancelFunc]`) maps a frontend-generated `pollerID` to the inner cancel function so `CancelOutstandingPoll` can release a poll that the gRPC stack hasn't unblocked yet (`service/matching/matching_engine.go:1200-1209`, `:2877-2893`). `shutdownWorkers` (TTL cache) + `workerInstancePollers` (multi-cancel map) handle bulk cancellation keyed by `workerInstanceKey` (`service/matching/matching_engine.go:1211-1221`). Pollers attach their cancel via `e.outstandingPollers.Set(pollerID, cancel)` inside the same `pollTask` call that wraps the deadline context (`service/matching/matching_engine.go:2877-2893`).
4. **Server-side pub/sub for state-change long polls.** `events.Notifier` (`service/history/events/notifier.go:29-67`) implements a per-workflow pub/sub where `WatchHistoryEvent` returns a buffered channel of size 1 (`service/history/events/notifier.go:131`), `NotifyNewHistoryEvent` non-blocking-enqueues into a 1000-slot central queue (`eventsChanSize`, `service/history/events/notifier.go:25`, `:202-211`), and `dispatchHistoryEventNotification` uses non-blocking sends with a `default` branch that drops silently on subscriber backpressure (`service/history/events/notifier.go:191-198`). Consumers include `GetOrPollWorkflowMutableState` (`service/history/api/get_workflow_util.go:128-258`), which loops over `select { case event := <-channel: ... case <-longPollCtx.Done(): ... case <-ctx.Done(): ... }` and explicitly comments that stale notifications can be ignored by re-checking `expectedNextEventID < response.GetNextEventId()` (`:218-247`). A sibling `ChasmNotifier` (`service/history/chasm_notifier.go:35-64`) implements the same pattern for CHASM executions by *closing* a channel on notify and removing the entry atomically.

Partial output **is trusted**, but only at watermark granularity: the replication sender emits `WorkflowReplicationMessages` with `replication_tasks: nil` and an updated `exclusive_high_watermark` to advance ACK even when no tasks are produced (`service/history/replication/stream_sender.go:478-489`, `:651-660`). The receiver treats each `messages` event as a committed batch and tracks `WatermarkInfo` on each trackable task (`service/history/replication/stream_receiver.go:374-377`).

Retry policy is layered: per-task `ReplicationStreamSenderErrorRetryXxx` (`service/history/replication/stream_sender.go:619-625`), whole-stream `ReplicationStreamEventLoopRetryMaxAttempts = 100` (`common/dynamicconfig/constants.go:2794-2798`, `service/history/replication/stream.go:58-60`), and an `isRetryableError` filter that excludes `StreamError`, `ShardOwnershipLost`, and `context.Canceled` (`service/history/replication/stream.go:136-152`). Liveness is monitored via `recvSignalChan` heartbeat and `ReplicationStreamSendEmptyTaskDuration` (1 min default, `common/dynamicconfig/constants.go:2834-2838`); a stalled stream's monitor triggers `Stop()` on the liveness monitor side (`service/history/replication/stream.go:128-129`).

Stream can be resumed after failure because (a) the inclusive low watermark is persisted in the receiver-side `QueueReaderState` (`service/history/replication/stream_sender.go:338-359`), (b) sender `sendCatchUp` queries `getSendCatchupBeginInclusiveWatermark` to resume from the persisted scope (`stream_sender.go:362-394`, `:396-421`), and (c) `reconcileOutboundStreams` restarts any missing stream (`service/history/replication/stream_receiver_monitor.go:267-297`). The stream receiver monitor even detects "stuck" streams where `currentAckLevel == previousAckLevel && currentAckLevel < previousMaxReplicationTaskId` and emits `ReplicationStreamStuck` (`service/history/replication/stream_receiver_monitor.go:346-360`, `common/metrics/metric_defs.go:1064`).

Guardrails are not weakened by streaming: `WithDeadlineBuffer` ensures the long-poll always returns *before* the caller's deadline so RPC stack errors are not counted (`common/contextutil/deadline.go:10-15`, `service/matching/matching_engine.go:2846-2849`), and the notifier channels are size 1 / drop-on-full so the producer never blocks on a slow consumer (`service/history/events/notifier.go:131`, `:191-198`).

## Rating

**8 / 10 — Clear model with explicit interfaces, tests, and operational safeguards.**

- The only true server-streaming RPC in the API is well-defined with typed oneof attributes (`request_response.proto:1027-1039`, `api/replication/v1/message.pb.go:619-628`) and wrapped by a generic `BiDirectionStream[Req,Resp]` abstraction that gives typed `Send` / `Recv()` / `Close` / `IsValid` (`bi_direction_stream.go:30-65`). 
- `stream_batcher.Batcher` is a small, focused primitive (`common/stream_batcher/batcher.go:15-153`) used for matching-engine user-data updates (`service/matching/matching_engine.go:2771-2783`) with batcher tests at `common/stream_batcher/batcher_test.go`.
- Cancellation has two dedicated RPCs (`CancelOutstandingPoll`, `CancelOutstandingWorkerPolls`) wired into a per-poller cancel map and a TTL cache for worker shutdown (`matching_engine.go:1200-1221`).
- The replication-stream stack has focused tests for the retry/stop boundary (`service/history/replication/stream_test.go:16-112`) plus a 554-line `stream_receiver_test.go` that exercises watermark logic, priority validation, and the receiver-mode lock.
- Liveness, channel-full, stuck, panic, and error metrics are defined (`common/metrics/metric_defs.go:1061-1065`) and emitted from the live code paths (`stream_receiver.go:188,219`, `stream_sender.go:177,221`, `bi_direction_stream.go:186`, `stream_receiver_monitor.go:317`).
- Resilience: configurable retry counts, backoff, and a stream-receiver-monitor that re-conciles the stream set on cluster-metadata change and on a 2 s tick (`stream_receiver_monitor.go:133-161`).
- The poll interface exposes 8 long-poll RPCs classified by `API_CATEGORY_LONG_POLL` (`matchingservice/v1/service.proto:18-58`), and `WithDeadlineBuffer` is used consistently to reserve `returnEmptyTaskTimeBudget` (1 s) before the caller's deadline.
- **Deductions**: the only true stream (`StreamWorkflowReplicationMessages`) is internal — there is no public client-side streaming RPC, and the public `GetWorkflowExecutionHistory` uses a non-streaming long-poll (`service/history/api/getworkflowexecutionhistory/api.go:220-279`). The replication BiDirectionStream uses `context.Background()` for its internal lifecycle (`bi_direction_stream.go:73`), so a client cancel does *not* propagate; cancellation flows only through `Close()` from the manager goroutines. There is no partial-tool-call concept (none of the RPCs model one). The chasm `Subscribe/Notify` uses `close(channel)` (`:60-62`) so a subscriber only ever sees *one* notification before it must re-subscribe — fine for a polling API but does not give multi-event streams.

## Answers to Dimension Questions

1. **What is emitted while execution is still running?**
   - **Replication stream (cross-cluster)**: `WorkflowReplicationMessages` carrying `[]*ReplicationTask` plus `ExclusiveHighWatermark` + `Priority`. Empty batches (no tasks, just watermark) are emitted to advance ACK and unstick a skip loop (`service/history/replication/stream_sender.go:478-489`, `:518-531`). ACKs (`SyncReplicationState`) flow back from receiver (`service/history/replication/stream_receiver.go:296-307`).
   - **Task-queue long polls**: each successful long-poll returns at most one `internalTask` (workflow/activity/nexus) — no incremental partial output. When no task arrives before the deadline, the server returns an empty well-typed response (`service/matching/matching_engine.go:241`, `:713`) so the caller's RPC stack doesn't error on context timeout.
   - **Workflow-state long polls**: `GetMutableState` (and `GetWorkflowExecutionHistory` with `WaitNewEvent=true`) returns a single response per call after waking on the next history event; the notifier delivers `*events.Notification` per-event but each long-poll only consumes one of them per inner loop iteration (`service/history/events/notifier.go:37-48`, `service/history/api/get_workflow_util.go:194-247`).
   - **Update polling**: a single `PollWorkflowExecutionUpdate` returns one `*workflowservice.PollWorkflowExecutionUpdateResponse{Outcome, Stage, UpdateRef}` per call after the registered `Update` reaches the requested `LifecycleStage` (`service/history/api/pollupdate/api.go:69-81`).

2. **Are partial outputs trusted?**
   The replication stream **does** trust partial (empty-task, watermark-only) messages — `WorkflowReplicationMessages{ReplicationTasks: nil, ExclusiveHighWatermark: ...}` is a valid committed payload that advances reader state (`service/history/replication/stream_sender.go:478-489`). The receiver-side task tracker uses the watermark to advance `LowWatermark()` even when no tasks were delivered (`service/history/replication/stream_receiver.go:374-377`). For task-queue and history long-polls, there is no "partial" output — either a task/notification arrives, or an empty response is returned; the empty response is treated as authoritative ("nothing available right now") and never as a partial failure.

3. **Can a failed stream resume?**
   Yes — and the resumption protocol is explicit. The receiver persists `QueueReaderState` (with `InclusiveMin` task IDs at scopes `[overall, high, low]`) on every ACK (`service/history/replication/stream_sender.go:338-359`). On reconnect, the sender's `sendCatchUp` reads this state to compute `catchupBeginInclusiveWatermark` and replays missed tasks from the persisted watermark (`service/history/replication/stream_sender.go:362-394`, `:396-421`). The receiver's `validateAndSetReceiverMode` short-circuits a reconnect by returning a `StreamError` when sender-side priority differs, forcing a fresh stream and re-anchor (`service/history/replication/stream_receiver.go:467-483`). Long-polls do not "resume" — a caller whose deadline expires simply re-issues the poll.

4. **Can a user stop the stream safely?**
   - For the replication stream, the only safe-stop is server-initiated: `Stop()` (called by `WrapEventLoop`'s defer, by the liveness monitor when heartbeat fails, or by `StreamReceiverMonitor` reconciliation) flips the status atomically, closes the shutdown channel, calls `r.stream.Close()`, and cancels outstanding task trackers (`service/history/replication/stream_receiver.go:156-171`, `service/history/replication/stream_sender.go:144-155`). Client-side cancel is *not* wired into `BiDirectionStream`'s context — it uses `context.Background()` for the lazy-init ctx (`service/history/replication/bi_direction_stream.go:73`), so a client cancel does not stop the server-side sender goroutine; the sender is only stopped when the gRPC stack surfaces the half-close on `Recv()`.
   - For matching long-polls, `CancelOutstandingPoll` and `CancelOutstandingWorkerPolls` provide immediate cancellation by invoking the stored `context.CancelFunc` keyed by `pollerID` / `workerInstanceKey` (`service/matching/matching_engine.go:1200-1221`). The cancel is captured in `pollTask` (`service/matching/matching_engine.go:2877-2893`) and the map is cleaned up via `defer`.

5. **Does streaming weaken guardrails?**
   No. Specifically: (a) `WithDeadlineBuffer` ensures long-poll goroutines always return *before* the caller's deadline (`common/contextutil/deadline.go:10-39`), so streaming never converts a clean long-poll into a context-deadline SLO error. (b) `events.Notifier` and `ChasmNotifier` use non-blocking sends with `default` drop, so a slow subscriber cannot back-pressure the publisher (`service/history/events/notifier.go:191-198`, `service/history/chasm_notifier.go:35-64`). (c) The replication stream tracks every task under `ExecutableTaskTracker` with explicit size limits (`service/history/replication/executable_task_tracker.go` + `ReplicationReceiverMaxOutstandingTaskCount = 500`) and emits `ReplicationStreamChannelFull` if the recv channel overflows (`service/history/replication/bi_direction_stream.go:186`). (d) Slow-submission flow control pauses send when scheduler latency exceeds `ReplicationReceiverSlowSubmissionLatencyThreshold` (`service/history/replication/stream_receiver.go:401-403` + flow controller). (e) Per-task retry does not bypass the receiver-mode lock — a priority mismatch causes a `StreamError` that forces a stream restart rather than silently crossing modes (`service/history/replication/stream_receiver.go:467-483`).

## Architectural Decisions

- **Polling as the unified streaming primitive.** The vast majority of "streaming" in Temporal is implemented as a long-poll unary RPC, not a true gRPC stream. This keeps the wire protocol simple, makes retries and quotas trivial, and lets the gRPC stack handle flow control. The trade-off is one RTT-per-event semantics for the client; the server keeps state in memory and on disk between polls.
- **One true gRPC stream, internal-only.** The sole true server-streaming RPC (`StreamWorkflowReplicationMessages`) is internal — it never crosses the public client boundary (`proto/internal/temporal/server/api/historyservice/v1/service.proto:350`, `:1027-1039`). This confines the complexity of bi-directional stream state machines to the replication code path.
- **Receiver-mode lock with stream-reconnect-on-change.** The receiver mode (`SingleStack` vs `TieredStack`) is set exactly once on the first message's priority (`service/history/replication/stream_receiver.go:467-495`); changing modes is *only* possible via a fresh stream. This trades flexibility for a strong invariant the watermark logic depends on.
- **Background-context streams, surface-level cancellation.** `BiDirectionStreamImpl` uses `context.Background()` for its internal client ctx (`service/history/replication/bi_direction_stream.go:73`); cancellation flows through `Close()` from explicit callers (`Stop()`, monitor reconciliation). This decouples stream lifecycle from a single RPC call.
- **Per-tier liveness multiplier.** Sender uses `LivenessMultiplier = 10` and receiver `= 3` (`common/dynamicconfig/constants.go:2839-2848`) — asymmetry reflects that the sender-side heartbeats are cheaper (it generates them) while the receiver is the consumer that needs to react to silence sooner.
- **Empty-watermark batches are a first-class protocol.** `sendTasks` always emits at least one watermark-only message even when `beginInclusiveWatermark == endExclusiveWatermark` (`service/history/replication/stream_sender.go:478-489`). This guarantees ACK forward-progress and lets the receiver detect "no tasks but stream alive".
- **Skip-count cap prevents starvation.** `TaskMaxSkipCount = 1000` forces a watermark emission when tasks would otherwise be skipped because they belong to a different priority or shard (`service/history/replication/stream_sender.go:38`, `:518-531`).

## Notable Patterns

- **`BiDirectionStream[Req, Resp]` generic wrapper** with typed `Send`, `Recv() <-chan StreamResp[Resp]`, `Close`, `IsValid` (`service/history/replication/bi_direction_stream.go:30-65`). Hides the gRPC `ClientStream` behind a channel API so producers can range over responses.
- **`WrapEventLoop` exponential-retry-with-stopper** (`service/history/replication/stream.go:46-92`). One function that classifies errors, applies backoff, and guarantees the stopper runs exactly once.
- **`livenessMonitor`** that resets on signal (`service/history/replication/stream.go:94-134`). Classic watchdog pattern using a buffered signal channel of size 1 and a `time.Timer`.
- **Two-tier cancellation registry**: per-poller `SyncMap[pollerID, CancelFunc]` for fast `CancelOutstandingPoll`, plus a per-worker bulk-cancel data structure for `CancelOutstandingWorkerPolls` (`service/matching/matching_engine.go:1200-1221`, `:2877-2893`).
- **`WithDeadlineBuffer`** (`common/contextutil/deadline.go:16-39`). Reusable utility that any long-poll must use to avoid context-timeout SLO errors; appears in matching, history long-poll, and nexus list operations.
- **`stream_batcher.Batcher[T, R]`** with `Add` blocking on `ctx.Done()` and an internal goroutine that exits after `IdleTime` (`common/stream_batcher/batcher.go:15-153`). Eliminates per-namespace goroutines for batched user-data updates.
- **Notifier pattern for long-polls**: `events.Notifier` and `ChasmNotifier` both decouple "produce a notification" from "consume in a long-poll", with the producer never blocking on slow consumers (`service/history/events/notifier.go:180-211`, `service/history/chasm_notifier.go:35-64`).
- **`streamSender.Wait()` server-side gate** (`service/history/handler.go:1916-1920`): the history RPC handler registers the sender then blocks on its `Wait()` shutdown channel. The stream itself is the unit of lifetime, not the RPC call.

## Tradeoffs

- **Internal context.Background() in BiDirectionStream**: stream lifetime is decoupled from the calling RPC, so client cancellation does not propagate. The only safe-stop path is server-initiated via `Close()` from `Stop()` / reconciliation. Pro: clean shutdown ordering; Con: a hung client cannot abort an outbound replication stream.
- **No multi-event notifier**: `ChasmNotifier.Notify` closes and deletes the channel (`service/history/chasm_notifier.go:57-64`), so a chasm subscriber only sees *one* notification before it must re-subscribe via `pollComponent`. This is correct for monotonic predicates but does not model "stream of state changes" semantics.
- **Long-poll-only public history API**: `GetWorkflowExecutionHistory` with `WaitNewEvent=true` returns at most one page per call (`service/history/api/getworkflowexecutionhistory/api.go:220-279`); the client must page. No public token-level model streaming exists because Temporal does not call LLMs itself.
- **Receiver-mode lock sacrifices flexibility for invariant strength**: changing tiered/single stack on the same stream requires reconnect (`service/history/replication/stream_receiver.go:467-483`). Adds a recovery cycle when rolling out the tiered-stack feature.
- **`outstandingPollers` is unbounded**: keyed by frontend-generated `pollerID` strings, only cleaned up by the matching engine's `defer` (`service/matching/matching_engine.go:2887-2892`). A misbehaving frontend that generates many IDs without GC could OOM the matching host. Mitigated by the matching-service rate limit (`service/matching/configs/quotas.go:13-14`).
- **Polling-based visibility for workers**: `WorkerVersionStamp` and `PollerScalingDecision` are returned as fields on each poll response, not streamed. Con: workers can only learn about new scaling decisions when they next poll. Pro: simplicity.
- **Watermark granularity is `int64` task ID** rather than event ID — efficient but means a missing intermediate task ID is unobservable (sender can skip ahead silently up to `TaskMaxSkipCount` then emit a watermark, `service/history/replication/stream_sender.go:518-531`).

## Failure Modes / Edge Cases

- **Stream channel overflow**: 512-slot recv channel in `BiDirectionStream` is blocked on send with metric (`ReplicationStreamChannelFull`); a consumer that doesn't drain in time blocks the sender's `notifyRecvChannel` (`service/history/replication/bi_direction_stream.go:176-188`). Producer-side backpressure.
- **Stuck stream detection**: monitor compares two consecutive 5-min samples; if `AckLevel` does not advance while behind `MaxTaskId`, emits `ReplicationStreamStuck` and a structured error log (`service/history/replication/stream_receiver_monitor.go:339-368`).
- **Cluster-metadata change mid-stream**: `StreamReceiverMonitor` registers a metadata-change callback and re-reconciles both directions (`service/history/replication/stream_receiver_monitor.go:139-160`). Stale inbound streams are explicitly stopped (`stream_receiver_monitor.go:118-131`).
- **Shard ownership lost**: `isRetryableError` returns `false` for `*persistence.ShardOwnershipLostError` and `*serviceerrors.ShardOwnershipLost` (`service/history/replication/stream.go:136-152`), so the stream does not retry but also does not crash — `WrapEventLoop` returns, stopper runs.
- **Priority mismatch on receiver**: `validateAndSetReceiverMode` returns a `serviceerror.NewInvalidArgument`, wrapped in `NewStreamError` (`service/history/replication/stream_receiver.go:467-483`, `:350`), which `isRetryableError` rejects, terminating the loop. The monitor then re-reconciles and starts a fresh stream.
- **Shutdown semantics**: `StreamReceiverMonitor.Stop` honors `EnableCloseInboundReplicationStreamOnShutdown` config — inbound streams are kept open by default to allow graceful drain from the remote sender (`stream_receiver_monitor.go:109-114`).
- **Empty long-poll returning before deadline**: `WithDeadlineBuffer` ensures `returnEmptyTaskTimeBudget = 1s` is reserved; if the parent ctx has no deadline, the child gets the full long-poll interval (`common/contextutil/deadline.go:25-39`).
- **Stale notification in history long-poll**: `GetOrPollWorkflowMutableState` checks `versionhistory.CompareVersionHistoryItem(notifiedEventVersionItem, request.VersionHistoryItem) < 0` and continues the loop on a stale notification (`service/history/api/get_workflow_util.go:218-221`).
- **Race between subscribe and notify**: `ChasmNotifier.Subscribe` is called *under the execution lease* before releasing the lock, so a concurrent notify cannot fire before subscribe completes (`service/history/chasm_engine.go:707-728`).
- **Admin RPC EOF mid-stream**: `admin_handler.StreamWorkflowReplicationMessages` treats EOF as a clean stop (no error log); all other `Recv` errors log at Info and trigger shutdown (`service/frontend/admin_handler.go:2024-2088`).

## Future Considerations

- **BiDirectionStream context propagation**: making the `BiDirectionStreamImpl.ctx` cancellable by the gRPC RPC ctx would let client cancellations stop the stream immediately without waiting for the monitor to notice.
- **Multi-event ChasmNotifier**: replacing close-on-notify with a buffered channel (size 1, drop-old) would let a polling caller consume multiple transitions per subscribe.
- **Stream-consumer back-pressure metrics**: the 512-slot channel in `BiDirectionStream` already has `ReplicationStreamChannelFull`; a per-stream high-water-mark gauge would surface saturation before channel-full events fire.
- **`GetWorkflowExecutionHistory` streaming**: a server-streaming variant that emits one page per notification (rather than polling per page) would reduce latency for UI clients tailing a running workflow.
- **Resume tokens for cross-shard pollers**: a token encoding `(workerInstanceKey, last-seen-event-id)` would let workers resume cleanly across matching-host failovers.
- **Per-poll-type quotas**: the existing `quotas.go` only applies per-RPC, not per-worker — a worker that hammers `PollWorkflowTaskQueue` from many sessions cannot be rate-limited independently.
- **CancelOutstandingWorkerPolls heartbeat**: the bulk-cancel path cancels outstanding pollers but does not signal the gRPC client to reconnect; adding a hint response code would help worker shutdown loops react faster.

## Questions / Gaps

- **What about `start_app_conversation` style streaming for task completion notifications?** No evidence found in this source for SDK-side streaming of task completion back to a workflow waiter other than the long-poll `PollWorkflowExecutionUpdate` API (`service/history/api/pollupdate/api.go`).
- **Partial tool-call parsing**: no concept exists in this code — Temporal does not parse tool calls. Workflow tasks carry `WorkflowExecutionStartedEventAttributes` and command lists; commands are atomic (`api/workflow/v1/message.pb.go`). The "streaming partial command" question is N/A.
- **Public `WorkflowExecutionInfo` change stream**: the public client API exposes `GetWorkflowExecutionHistory` with `WaitNewEvent=true`, which is the closest analog to a state-change stream. It is a long-poll, not a true stream (`service/history/api/getworkflowexecutionhistory/api.go:220-279`). The frontend's `/api/v1/app-conversations/{id}/trajectory`-style streaming does not exist in this source.
- **What is the wire-level back-pressure for replication streams?** `BiDirectionStream` uses `g.Send(...)` for outgoing; gRPC's own flow control applies. The channel size 512 caps *application-level* buffering (`bi_direction_stream.go:22`).
- **No client-side streamed RPCs**: a public client calling the Temporal SDK can stream operations via `RecordActivityTaskHeartbeat` and the notifier-backed long-polls, but the public proto for the frontend service contains no `stream` directives (`proto/internal/temporal/server/api/adminservice/v1/service.proto` and `proto/internal/temporal/server/api/historyservice/v1/service.proto` searched; only the two `StreamWorkflowReplicationMessages` definitions appear).

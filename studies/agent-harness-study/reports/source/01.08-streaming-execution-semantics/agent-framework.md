> Scope note: this study looks at both the Python implementation under `sources/agent-framework/python/` (the bulk of the framework, including `agent-framework-core` plus providers) and the .NET implementation under `sources/agent-framework/dotnet/`. Both implement the same conceptual model (`ChatResponse`/`AgentResponse` updates + `ResponseStream`/streaming enumerable), so the analysis cites evidence from each where relevant.

## Summary

The framework implements streaming as a first-class, **chunked-event** protocol on top of `ChatResponseUpdate` (`python/packages/core/agent_framework/_types.py:2411`) and `AgentResponseUpdate` (`_types.py:2788`), wrapped by a generic `ResponseStream` (`_types.py:2939`). Updates carry `Content` items (text deltas, reasoning deltas, function-call argument fragments, image-generation partials, etc.), optional metadata (response id, conversation id, model, finish reason, usage, continuation token), and are coalesced into a single `ChatResponse`/`AgentResponse` only when the consumer requests `get_final_response()` (or `ChatResponse.from_updates`). Partial tool-call arguments are aggregated during streaming (`_types.py:1908` for `function_call` chunks), and final responses are coalesced in `_finalize_response` (`_types.py:2067`) which preserves metadata, merges raw representations, and collapses adjacent text/reasoning/code-interpreter content.

Streaming output is persisted only when a `HistoryProvider` (e.g., `FileHistoryProvider`/`InMemoryHistoryProvider`) is attached to the agent (`python/packages/core/agent_framework/_sessions.py`); `ResponseStream` itself does not auto-persist. `RawAgent._parse_streaming_response` (`_agents.py:1051`) eagerly propagates `conversation_id` to the session per-update via a transform hook (`_propagate_conversation_id`, `_agents.py:1085`) and runs context-provider `after_run` once when finalization completes (`_post_hook`, `_agents.py:1058`).

User interruption is supported via standard asyncio cancellation, but with **partial output kept** rather than rolled back: stopping mid-stream simply stops iterating; the stream does not undo already-yielded updates. There is **no semantic rollback** for partial text or tool calls — `ResponseStream` only discards state on a stream-level exception (`_types.py:3122-3128`, where the captured `_stream_error` is exposed for hook consumers). Cancellation of MCP long-running tasks is handled via a best-effort `tasks/cancel` (`_mcp.py:2157`), but cancellation of a streaming chat response does not propagate to the upstream provider; the server-side stream simply terminates.

Retry / resume after a stream failure is supported through `ContinuationToken` (`_types.py:2078`, `python/packages/openai/agent_framework_openai/_chat_client.py:136-140`), which exposes a server-issued response id. The OpenAI Responses client honors both background and streaming resumes by calling `client.responses.with_raw_response.retrieve(...)` with `stream=True` (`_chat_client.py:646-650`). No auto-retry is built into `ResponseStream` itself — recovery is a caller-driven pattern.

Streaming observability is wired through OpenTelemetry without weakening guardrails: spans are started non-current (`observability.py:2159-2179`), attached around each iterator pull via `with_pull_context_manager` (`_types.py:3299` + `_activate_span` `observability.py:2117-2134`), and closed by cleanup hooks + a `weakref.finalize` safety net (`observability.py:1583-1592`). Streaming exceptions are captured on the span but do not flip the span to ERROR status because the cleanup runs in a different async context than the throw (`tests/core/test_observability.py:2023-2050`).

**Net effect**: streaming gives sub-second responsiveness through token-level deltas and structured partial updates, and state is not corrupted by mid-stream errors (caught + exposed via `_stream_error`), but the framework explicitly chooses *append-and-stop* over *rollback* for partial output and leaves caller-driven retry/resume via `ContinuationToken` rather than transparent reconnection.

## Rating

**7/10** — Clear model with explicit interfaces, well-tested streaming primitives, and operational safeguards (cleanup hooks, span context attachment, `ContinuationToken` resume, MCP cancel-on-cancellation). Below "mature" because (1) there is no built-in auto-retry of failed streams, (2) partial text/tool-call output is *not* rolled back on stream failure — consumers must handle partial state themselves, (3) the framework does not propagate upstream cancellation for chat streams (only MCP tasks), and (4) finish_reason/error semantics in `ResponseStream.__anext__` are documented but rely on consumer awareness (`_stream_error`).

## Answers to Dimension Questions

1. **What is emitted while execution is still running?**
   - Per-update `ChatResponseUpdate`/`AgentResponseUpdate` objects via `async for` on `ResponseStream`, plus per-update transform hooks (`stream_transform_hooks`/`with_transform_hook`), the chat client telemetry layer's per-pull `_activate_span` context, and (for the OpenAI Responses API) per-update `model` from the served-model header (`_chat_client.py:654,696`), `conversation_id` propagation, and `continuation_token` for background responses (`_chat_client.py:2750-2754`). Workflows additionally emit `WorkflowEvent` discriminated events (`output | intermediate | request_info | status | superstep_* | executor_* | warning | error`, `python/packages/core/agent_framework/_workflows/_events.py:104-143`). Partial function-call arguments stream through as `function_call` `Content` items with `arguments=event.delta` (e.g., OpenAI `_chat_client.py:2997-3011`, Anthropic `_chat_client.py:1344-1366`).

2. **Are partial outputs trusted?**
   - Partially. Updates are passed through *as received*; transform hooks can inspect, mutate, or drop them (a hook returning `None` keeps the previous value, `_types.py:3433-3448`). However, the *final* `ChatResponse`/`AgentResponse` is the trusted object: it is reconstructed from the collected updates via `ChatResponse.from_updates`/`_finalize_response_updates`, which performs coalescing, merges function-call fragments by `call_id`, aggregates usage, and propagates response metadata. Consumers should treat any "live" output as provisional and only treat `get_final_response()` as authoritative. This is reinforced by tests that show partial text fragments with metadata from the first fragment (`test_text_coalescing_preserves_first_properties`, `test_types.py:1529-1540`) and partial function-call fragments that are merged (`test_function_call_merge_in_process_update_and_usage_aggregation`, `test_types.py:1458-1478`).

3. **Can a failed stream resume?**
   - Yes, via the server-issued `ContinuationToken`. The OpenAI Responses client implements both background and streaming resume: when `options["continuation_token"]` is provided, the next call uses `client.responses.with_raw_response.retrieve(response_id, stream=True)` (`_chat_client.py:640-665`). For .NET, `ChatClientAgentContinuationToken` wraps the same primitive and is persisted via JSON, then rehydrated in a later `RunStreamingAsync` (`sources/agent-framework/dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgentContinuationToken.cs:1-160`). The framework does **not** auto-retry — recovery is caller-driven, and the consumer must explicitly forward the latest `ContinuationToken` to the next call. Workflows additionally offer checkpoint-based resume (`workflow.run(checkpoint_id=..., responses=...)`, `_workflow.py:914-933`).

4. **Can a user stop the stream safely?**
   - Yes, via standard asyncio cancellation. Stopping the `async for` loop simply exits iteration; `ResponseStream` does not roll back already-yielded updates and does not propagate cancellation upstream to the chat service. The error path (`ResponseStream.__anext__` catches `Exception`, sets `_stream_error`, runs cleanup hooks, re-raises — `_types.py:3122-3128`) makes the error visible to consumers. For MCP long-running tasks, `cancel_remote_task_on_local_cancellation=True` (default) makes `CancelledError` fire a best-effort `tasks/cancel` to the remote server (`_mcp.py:244-293,2157-2195`); for chat streams there is no equivalent propagation. Tests confirm that streaming telemetry captures exceptions via `_stream_error` and does *not* flip span status to ERROR because the cleanup runs in a different async context (`tests/core/test_observability.py:2023-2050`).

5. **Does streaming weaken guardrails?**
   - No. Approval flows still gate tool calls mid-stream (e.g., `test_streaming_approval_request_generated`, `test_function_invocation_logic.py:2811-2848`), MCP sampling is deny-by-default even during streaming (`python/packages/core/agent_framework/_mcp.py:...`), and tool-call argument allowlists continue to apply because `_process_function_requests` runs after `get_final_response()` and re-validates before dispatch (`_tools.py:2273-2387`). Observability is preserved end-to-end via per-pull span activation (`observability.py:2117-2134,1589`) and a `weakref.finalize` safety net for span closure (`observability.py:1591`). One acknowledged limitation is that partial text/reasoning deltas are emitted *before* any server-side content-filter check completes — but the Responses API's `BadRequestError`/`content_filter` error path raises inside the async generator and is re-raised by `ResponseStream.__anext__` for downstream handling (`_chat_client.py:666-668`, `_completions/_chat_completion_client.py:526-546`).

## Architectural Decisions

- **`ResponseStream` is the single streaming primitive** (`_types.py:2939-3329`). It owns a `_finalizer`, `_transform_hooks`, `_result_hooks`, `_cleanup_hooks`, and `_pull_context_manager_factories`, making it composable across layers (middleware, telemetry, function invocation) without leaking concerns into the consumer code. Decision: keep all stream composition in one class so that observability/function-call loops/middleware can re-wrap it (`_agents.py:1112-1122`, `observability.py:1585-1592`, `_tools.py:2873`).
- **Updates, not events.** `ChatResponseUpdate`/`AgentResponseUpdate` are content carriers, not domain events. They carry `Content` items, metadata, optional `continuation_token`, and `raw_representation`. This is consistent with `Microsoft.Extensions.AI`'s `ChatResponseUpdate` model (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponseUpdate.cs:35-178`), enabling parity with the .NET implementation.
- **Per-pull context activation for observability** (`_types.py:3299-3316` + `observability.py:2117-2179`). The streaming span is *not* attached at creation time (which would cause cross-context detach warnings when cleanup runs in a different async context). Instead, it is attached via a context-manager factory invoked inside `__anext__`'s `ExitStack`. This is a deliberate design choice documented in `_start_streaming_span` (`observability.py:2159-2179`).
- **Single-pull deltas, finalize-once.** `ChatResponse.from_updates` is called once after iteration completes (or on `get_final_response()` if iteration is skipped). Coalescing preserves the first-seen metadata and aggregates usage/reasoning/text/code-interpreter fragments deterministically (`_types.py:1869-2072`). This means consumers that read `response.text` after `get_final_response()` see a canonical view, not a frame-by-frame reconstruction.
- **Function-call streaming runs in iterations, not token-by-token.** `FunctionInvocationLayer._stream` (`_tools.py:2700-2873`) consumes an inner `ResponseStream` to completion, dispatches function calls, then re-issues the next model call. Token-level deltas within a single iteration; function-call boundaries between iterations. This bounds mid-iteration tool-call concurrency and lets the `_post_hook` result hook run at predictable boundaries.
- **Eager conversation-id propagation, lazy history persistence.** `_propagate_conversation_id` runs as a transform hook (`_agents.py:1085-1099`), so the session's `service_session_id` is updated on the *first* update that carries a real conversation id, not deferred to the final response. `after_run` (history persistence) only runs once at the end (`_post_hook`, `_agents.py:1058-1083`).

## Notable Patterns

- **Hook triumvirate** — `transform_hook` (per update), `result_hook` (after finalizer), `cleanup_hook` (after stream consumption, before finalizer). This lets layers express "tweak each update", "annotate the final response", or "close resources deterministically" without subclassing. Examples: telemetry's `_record_duration` as cleanup (`observability.py:1587`), `_finalize_stream` as cleanup that captures errors and skips `get_final_response` on the error path (`observability.py:1539-1548`), the function-invocation loop collecting inner result hooks for later replay (`_tools.py:2156-2164,2701-2703`).
- **`map(chat → agent) + finalize(agent)` composition** (`_agents.py:1112-1122`). Agent streaming is implemented as a transformation of the underlying chat stream plus an agent-specific finalizer. This avoids duplicating update-buffering logic.
- **Idempotent `get_final_response`** (`_types.py:3349-3366`). Calling it twice after auto-finalize returns the cached result without re-invoking the finalizer. Useful for agents whose hooks re-await the final response.
- **Stream context propagation** (`_types.py:3299-3316`). Each `__anext__` runs through an `ExitStack` that activates any registered context-manager factories. This is the pattern that allows telemetry to attach/detach its span safely per pull.
- **Continuation token as an opaque, JSON-serializable TypedDict** (`_types.py:2078-2102`). Consumers treat it as opaque and pass it back, but it can be persisted and resumed across process restarts.

## Tradeoffs

- **Append-and-stop vs. rollback.** The framework chose append-and-stop: `ResponseStream.__anext__` re-raises on error but does not discard already-yielded updates (`_types.py:3122-3128`). Consumers must handle partial state themselves; the framework only exposes `_stream_error` for diagnostics. This favors low-latency UIs over transactional guarantees.
- **Eager aggregation at finalize, lazy during iteration.** During iteration, raw updates flow through; only at finalization does coalescing happen. This keeps the streaming path zero-copy but means consumers cannot reconstruct a `ChatResponse` from mid-stream data alone (they would need to call `from_update_generator` themselves).
- **Auto-retry is absent.** Failed streams surface the error but do not auto-reconnect or resume. Resume requires a `ContinuationToken` and an explicit caller-driven re-invocation. This keeps the streaming layer deterministic but means flaky networks translate into user-visible errors unless the caller wraps it.
- **Cancellation propagates only to MCP tasks.** Asyncio cancellation of `agent.run(..., stream=True)` does not propagate to the chat server; the server-side stream terminates when the connection drops. MCP long-running tasks *do* fire `tasks/cancel` because of the SEP-2663 task lifecycle support (`_mcp.py:2157-2195`), but chat streams lack an equivalent primitive.
- **Streaming exceptions do not flip span status.** Telemetry deliberately captures the exception on the span attributes but does *not* set `span.status = ERROR` because the cleanup runs in a different async context than the throw (`tests/core/test_observability.py:2023-2050`). This avoids cross-context detaches but means downstream observability tooling cannot rely on span status alone to detect streaming failures.
- **Transform-hook order vs. order of side effects.** Transform hooks fire per-update in registration order (`_types.py:3134-3140`, `test_multiple_transform_hooks_chained` asserts `["a", "b", "a", "b"]`). Consumers that mutate updates must be aware of this linear pipeline.

## Failure Modes / Edge Cases

- **OpenAI Responses API stream interrupted after partial deltas.** `_chat_client.py:646-665` re-fetches via `retrieve(..., stream=True)` and continues emitting updates. If `text_format` is used (`responses.stream(text_format=...)`) the served-model header cannot be surfaced, but streaming still completes (`test_streaming_text_format_not_captured_for_streaming_text_format`, `_chat_client.py:844-882`).
- **Function-call fragments with different ids must not merge.** `test_function_call_incompatible_ids_are_not_merged` (line 1481) and `_process_update` (line 1908) explicitly only merge adjacent `function_call` content on the same message. Cross-id fragments stay as separate calls.
- **Reasoning `done` events must not duplicate delta content.** `_chat_client.py:2659-2695` uses `seen_reasoning_delta_item_ids` to emit a `done` event only as a fallback when no delta was seen — preventing downstream accumulators (e.g., ag-ui) from doubling content.
- **`content_part.added` followed by `output_text.delta`.** The OpenAI parser emits both events as separate `TextContent` items; `_finalize_response` then coalesces adjacent text items into one, preserving the first-seen metadata. (`_types.py:1950-1979`.)
- **`response.completed` event does not set `continuation_token`** (`_chat_client.py:2755-2765`, `test_streaming_response_completed_no_continuation_token` line 5674). Consumers can use `continuation_token is None` as a "stream is done" signal, distinct from "still in progress".
- **MCP `partial_json_delta` after a server-tool-use block is silently ignored** (`anthropic/_chat_client.py:1344-1366`, regression test `test_parse_contents_server_tool_use_input_json_delta_ignored`). Server-managed tool calls cannot have client-side partial arguments.
- **`StreamCompletionException` not raised explicitly.** Both OpenAI clients wrap exceptions in `ChatClientException`/`OpenAIContentFilterException` (`_chat_client.py:598-611`, `_chat_completion_client.py:526-546`). These propagate through `ResponseStream.__anext__` via the generic `except Exception` branch (`_types.py:3122-3128`), which then sets `_stream_error`. Consumers that want to distinguish "stream failed mid-flight" from "stream raised before first update" must inspect whether any updates were already received.
- **Cancellation of `ResponseStream` from outside `__anext__`.** The `StopAsyncIteration`/`Exception` branches both call `await self._run_cleanup_hooks()`. There is no explicit cancellation handler at the stream level — cancellation flows through normal asyncio cancellation semantics.
- **Workflows that cannot be resumed mid-stream without checkpoints.** `_workflow.py:803-810` explicitly raises `RuntimeError` if the runner has in-flight executor messages; mid-run recovery requires `checkpoint_id` (`workflow.run(checkpoint_id=...)`). There is no automatic checkpoint-on-error.

## Future Considerations

- **Auto-retry with backoff for transient stream errors.** Currently absent in `ResponseStream`; callers must wrap with their own retry logic. A natural extension would be a `retry` configuration on `BaseChatClient.get_response` (already partially supported via `client_kwargs`) combined with a `max_stream_retries` parameter.
- **Span status = ERROR on streaming failures.** The current implementation captures exceptions but does not flip span status because of cross-context concerns. Resolving this would require either (a) ending the span synchronously from the iteration task, or (b) a `contextvars` copy that preserves the active context across the cleanup boundary.
- **Built-in partial-output rollback for transactional consumers.** A `stream(replay=False, transactional=True)` flag (or a dedicated `TransactionalResponseStream`) that buffers until the final response would let tool-heavy agents avoid partial-state risk without losing the streaming semantic for UI consumers.
- **Per-update `cursor` / `stream_position` for resume.** `ContinuationToken` works for provider-side resumption (OpenAI Responses, A2A), but for clients that do not support server-side resume (most non-OpenAI providers, OpenAI Chat Completions), there is no equivalent. A client-side `StreamCursor` could enable same-process resume.
- **Cancellation propagation to upstream provider streams.** Today only MCP tasks honor local cancellation. For chat streams, a cooperative cancellation token (e.g., `httpx` request abort) would let callers stop billing immediately without waiting for the next natural boundary.
- **Streaming structured-output validation incremental feedback.** `text_format`/`response_format` parsing happens only at `get_final_response()`. Emitting incremental parse warnings (or a "still validating" flag) would help consumers detect malformed structured streams early.

## Questions / Gaps

- **Is there an explicit, documented contract for "the stream may resume from this point" beyond `ContinuationToken`?** The OpenAI Responses path documents it (`_chat_client.py:622-665`); the OpenAI Chat Completions and Anthropic paths do not have a continuation-token equivalent in the codebase. For non-OpenAI providers, consumers must rely on checkpointing or resend the entire conversation.
- **What is the precise async-context relationship between `_run_cleanup_hooks` (in `ResponseStream.__anext__`) and the `weakref.finalize` callback?** The latter is a sync, GC-driven fallback; the former awaits hooks inside the iteration task. The interaction with `_finalize_stream` (cleanup hook) capturing exceptions via `_stream_error` is well-tested, but there is no test for what happens if the iteration task is itself cancelled mid-pull (the `CancelledError` path).
- **How does `stream_response.raw_representation` relate to per-update `raw_representation`?** Each update carries the SDK-level raw event (`ChatResponseUpdate.raw_representation`, `_types.py:2474,2522`). The final `ChatResponse.raw_representation` is a list of these (`_types.py:1927-1933`). There is no test verifying that `raw_representation` ordering matches the order of yielded updates.
- **Is `stream_response.value` (structured output) computed lazily?** `_parse_structured_response_value` is called in `ChatResponse.value` (lazy) and again in `from_update_generator`/`from_updates` (eager via `_finalize_response`→`response_format` propagation). It is unclear whether concurrent access to `value` during iteration is thread-safe (Python `asyncio` is single-threaded per task, so this may not matter, but it is undocumented).
- **Where is the integration test for resume-via-continuation-token across a process boundary?** There are unit tests (`test_openai_chat_client.py:5634-5733`) that verify the token is set/serialized correctly, but no test exercises "stop mid-stream, serialize continuation_token to disk, restart process, resume, get the remaining updates."
- **What is the behavior of `asyncio.shield` around stream iteration?** Not documented in the source. The `with_pull_context_manager`/`ExitStack` design suggests that pull contexts are activated/deactivated within `__anext__`, so `shield` would only protect a single pull, not the entire stream.

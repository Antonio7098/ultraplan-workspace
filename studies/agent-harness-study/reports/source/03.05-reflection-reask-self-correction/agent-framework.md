## Summary

The framework ships a dedicated, first-class reflection / re-invocation primitive in both languages, exposed as an `AgentMiddleware` (Python) and a `DelegatingAIAgent` (C#). Two implementations — `AgentLoopMiddleware` (Python, `python/packages/core/agent_framework/_harness/_loop.py:212`) and `LoopAgent` (C#, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:65`) — wrap an inner agent, run it once, and ask a `should_continue` / `LoopEvaluator` predicate whether to re-invoke, forwarding any returned `feedback` / `LoopEvaluation.Feedback` back into the next turn's input. A chat-client **judge** is a built-in sub-case via `AgentLoopMiddleware.with_judge(...)` (`python/packages/core/agent_framework/_harness/_loop.py:344`) and `AIJudgeLoopEvaluator` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:44`), which asks a structured-output `JudgeVerdict` after each iteration and feeds the gap analysis back as feedback.

Self-correction is bounded by a per-loop `max_iterations` / `MaxIterations` safety cap (Python default 10, judge default 5; .NET default 10 via `LoopAgent.DefaultMaxIterations`) that short-circuits before the predicate runs (`python/packages/core/agent_framework/_harness/_loop.py:661`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:196`). Termination is observable, structured, and optionally evaluated on three signals: predicate says stop (answered), budget exhausted, or pending tool approval (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:190`). The high-level Magentic One / Magentic orchestrator implements a richer self-correction pattern combining plan + fact replanning with a stall counter (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1110-1118`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:289-302`), and the JSON progress ledger it depends on is itself protected by an explicit retry budget (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:709-727`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:72-103`).

A secondary, very different self-correction loop lives in the **function-invocation** layer (`python/packages/core/agent_framework/_tools.py:2540-2696`): consecutive tool errors are counted (`errors_in_a_row`) and force `tool_choice="none"` after `max_consecutive_errors_per_request` (default 3) so the loop never returns orphaned `function_call` items. The whole function-call budget is itself capped by `max_iterations` (default 40) and `max_function_calls`; on exhaustion the framework injects a deterministic failsafe message rather than hallucinating a success.

## Rating

**8 / 10 — Clear model with tests, explicit interfaces, and operational safeguards.**

Rationale:

- Strong: dedicated `LoopEvaluator` / `LoopAgent` abstractions with a real judge pattern (structured-output `JudgeVerdict`/`MagenticProgressLedger`), per-iteration `progress` log, and `fresh_context` reset that snapshots and clones sessions so iteration N+1 does not silently re-read the accumulated history.
- Strong: retry budgets are explicit (`max_iterations`, `max_consecutive_errors_per_request`, `max_function_calls`, `MagenticTaskLimits` `MaxStall/MaxRound/MaxReset/MaxProgressLedgerRetryCount`), validated at construction, and observed via `ILogger` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:489-492`) and OpenTelemetry events (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:268-279`).
- Strong: failed-judge JSON parse in Magentic retries with linear backoff (`Task.Delay(250 * attempts, ...)`) and emits `WorkflowWarningEvent` per attempt before giving up (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:91-103`).
- Strong: correction feedback is concrete evidence, not vibes — judge gap-analysis is substituted via `{gap_analysis}` into the next turn's message (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:173`), so the loop sees why it failed rather than retrying blindly.
- Limits keeping it under 9:
  - `LoopAgent` / `AgentLoopMiddleware` are marked `[Experimental]` in both languages; the abstractions may evolve.
  - "Repeated reflection can loop forever" is only prevented by an explicit cap; an unbounded loop (`max_iterations=None`) silently relies on the predicate eventually returning `False`, and there is no logged counter to make a runaway observable from outside the predicate.
  - Self-correction's evidence visibility is good for the inner loop but **does not bubble up to the outer task** — when a judge or stall triggers a final answer, the last iteration's response is returned without a structured "judge verdict" attached, so a caller cannot programmatically distinguish a corrected answer from a first-try success.
  - No dedicated "critic agent" pattern beyond the judge loop evaluators; richer multi-agent reflection (writer→critic→revise) is documented but only available as a sample workflow (`dotnet/eng/verify-samples/WorkflowSamples.cs:104`), not a framework primitive.

## Answers to Dimension Questions

1. **When does the agent self-correct?**
   - **Loop agent / middleware** triggers a re-invocation whenever a `LoopEvaluator` returns `Continue` / `(true, feedback)` after an iteration (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:329-339`, `python/packages/core/agent_framework/_harness/_loop.py:483-492`).
   - **Judge-driven loop** triggers whenever the judge chat client returns `answered=False` (or emits `VERDICT: MORE`) (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:172-175`, `python/packages/core/agent_framework/_harness/_loop.py:193-196`).
   - **Function-tool loop** triggers whenever a tool returns an error — counted as `errors_in_a_row` against `max_errors` so the model can see the error and revise (`python/packages/core/agent_framework/_tools.py:2248-2270`).
   - **Magentic planner** triggers `_reset_and_replan` whenever `stall_count > max_stall_count` (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1115-1118`), whenever `is_progress_being_made` is false / `is_in_loop` is true (with exponential decrement on actual progress, `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1110-1113`), whenever the progress-ledger JSON parse fails the retry budget (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1082-1085`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:272-279`).

2. **Is correction bounded?**
   - Yes, at four layers:
     - Per-iteration `LoopAgent.MaxIterations` / `AgentLoopMiddleware.max_iterations` (Python defaults to 10, judge variant 5; C# default 10), enforced before the predicate runs (`python/packages/core/agent_framework/_harness/_loop.py:653-664`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:196-200`).
     - Per-tool-call `max_function_calls` and per-run `max_iterations` for function invocation (`python/packages/core/agent_framework/_tools.py:90-96`, `python/packages/core/agent_framework/_tools.py:2558-2565`).
     - Per-progress-ledger `MaxProgressLedgerRetryCount` for the Magentic manager (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticTaskContext.cs:14-17`, `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:588-590`).
     - Per-workflow `WorkflowRunner._max_iterations` (default 100), throws `WorkflowConvergenceException` on overrun (`python/packages/core/agent_framework/_workflows/_runner.py:99-153`, `python/packages/core/agent_framework/_workflows/_const.py:4`).
   - All four are explicitly documented and validated at construction (`python/packages/core/agent_framework/_harness/_loop.py:330-333`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:123`, `python/packages/core/agent_framework/_tools.py:1417-1418`).

3. **What evidence is shown during correction?**
   - Judge verdict carries a structured `gap_analysis`/`reasoning` field that is **substituted into the next iteration's prompt** (not just logged), so the next pass reads the correction reason verbatim (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:173`; Python `_build_judge_condition._next_message` at `python/packages/core/agent_framework/_harness/_loop.py:198-207`).
   - A per-iteration `progress` log (`List[str]`) is exposed via callback kwargs and injected into the next iteration as a `user` message (full log without a session, latest entry only with a session to avoid duplication) (`python/packages/core/agent_framework/_harness/_loop.py:704-748`).
   - Tool errors are surfaced as `function_result` contents on a `tool` message so the next model turn sees them in the conversation, not just in logs (`python/packages/core/agent_framework/_types.py` plus the in-loop processor at `python/packages/core/agent_framework/_tools.py:2261-2270`).
   - Magentic `progress_ledger` records *reasons* (`is_request_satisfied.reason`, `is_progress_being_made.reason`, `instruction_or_question.reason`) in addition to booleans (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:288-303`), so replans inherit a "why stalled" explanation.
   - Events emitted on every retry: `WorkflowWarningEvent` per failed parse (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:96`); logger warning on max consecutive tool errors (`python/packages/core/agent_framework/_tools.py:2252-2256`, `python/packages/core/agent_framework/_tools.py:2318-2321`).

4. **Does correction improve outputs or hide errors?**
   - It improves outputs in the documented cases: tests assert that a `VERDICT: MORE` judge produces a second iteration carrying the judge's reason (`python/packages/core/tests/core/test_harness_loop.py:857-874`), and that a tool error seen by the model is counted against `errors_in_a_row` (`python/packages/core/tests/core/test_function_invocation_logic.py:1000-1044`).
   - Hiding is prevented by three mechanisms:
     - On `max_iterations` exhaustion in the function-call loop, the framework forces a final call with `tool_choice="none"` and emits a deterministic failsafe message rather than returning the in-progress `function_call` chain (`python/packages/core/agent_framework/_tools.py:2664-2696`; test asserts `"I broke out of the function invocation loop..."` at `python/packages/core/tests/core/test_function_invocation_logic.py:1044`).
     - On Magentic stall cap, `_reset_and_replan` either issues another plan-review request (`if require_plan_signoff`) or surfaces a `"Workflow terminated due to reaching maximum {round,reset} count."` assistant message authored by the manager (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1222-1258`).
     - On MaxProgressLedgerRetryCount exhaustion the last exception is rethrown rather than swallowed (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:105`, `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:725-727`).
   - One known gap: when a judge/loop eventually says "stop", the final response is the *iteration's* response — there is no attached `JudgeVerdict` to the returned `AgentResponse`, so the caller cannot programmatically tell whether `Stop` came from a satisfied judge or from the cap. (No evidence found in the loop or workflow code linking the last verdict to the returned object.)

5. **Are repeated failures escalated?**
   - Function-call side: `errors_in_a_row >= max_consecutive_errors_per_request` triggers a forced `tool_choice="none"` so the assistant must respond in prose; the model is then responsible for reasoning about its own failure surface (`python/packages/core/agent_framework/_tools.py:2632-2644`, `python/packages/core/agent_framework/_tools.py:2806-2822`).
   - Magentic side: stalling past `max_stall_count` triggers `_reset_and_replan`, which clears the chat history (`MagenticContext.reset` at `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:385-393`) and broadcasts `MagenticResetSignal` to participants (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1260-1264`); analogous behavior in C# via `ResetChatSignal` (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:342`).
   - If `max_reset_count` is exceeded the workflow terminates with a manager-authored termination message (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1238-1256`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:248-258`).
   - Human-in-the-loop escalation: Magentic supports `require_plan_signoff=True`, which routes stalls back through the plan-review port (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:949-952`, `dotnet/tests/Microsoft.Agents.AI.Workflows.UnitTests/MagenticOrchestrationTests.cs`-adjacent: `MagenticOrchestrator.SubmitPlanReviewRequestAsync` at `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:109-120`).
   - C# only: a pending `ToolApprovalRequestContent` in the inner run is surfaced as a stop signal so the loop cannot resolve approvals autonomously (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:189-193`).

## Architectural Decisions

- **Loop / judge are middleware in Python, agent in C#.** Python treats re-invocation as one of any number of `AgentMiddleware` instances in a pipeline (`python/packages/core/agent_framework/_harness/_loop.py:212`); the runtime is `AgentContext` so it composes naturally with the existing pipeline (`python/packages/core/agent_framework/_middleware.py`). C# models re-invocation as a `DelegatingAIAgent`, so it composes via the agent decorator chain rather than the middleware pipeline (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:65`). The two implementations are deliberate mirrors — each idiom matches the language's preferred extension point.
- **One evaluator class, two entry points.** Both `AgentLoopMiddleware.__init__(should_continue=...)` and `AgentLoopMiddleware.with_judge(judge_client, ...)` map to the same internal state; the judge is a thin builder that hides the async-predicate wiring (`python/packages/core/agent_framework/_harness/_loop.py:344-396`). C# mirrors this with `LoopEvaluator` + `AIJudgeLoopEvaluator` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopEvaluator.cs:28-41`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:43-176`).
- **Two feedback-verdict fallbacks designed to never misclassify.** The judge uses a structured `answered: bool` when the chat client respects `response_format`, and falls back to *non-overlapping* text markers (`VERDICT: DONE` / `VERDICT: MORE`) when it does not; ambiguous replies default to "more" so the loop keeps running instead of stopping on an incomplete answer (`python/packages/core/agent_framework/_harness/_loop.py:65-82`, `python/packages/core/agent_framework/_harness/_loop.py:191-196`; same idea in `dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:60-76,156-164`).
- **`max_iterations` short-circuits before the predicate runs.** This is the only way to make an unbounded-feeling predicate safe; doing the check first prevents a runaway judge from also being a runaway cost (`python/packages/core/agent_framework/_harness/_loop.py:657-664`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:196-200`).
- **`fresh_context` snapshots the session.** Both languages serialize the session once at the start of the run and restore a clone before each re-invocation, so iteration N+1 sees the pristine baseline rather than the accumulated transcript (`python/packages/core/agent_framework/_harness/_loop.py:415-417, 423-437, 506-510`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:151-156, 500-508`). This is a deep framework decision: the agent has to support session serialization, and the round-trip is explicitly noted as a limitation (e.g. conversations-style storage cannot be cleanly cloned, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgentOptions.cs:42-53`).
- **`LoopEvaluation.Stop` is non-vetoing.** When multiple evaluators are configured, the first that requests a re-invocation wins and the rest are skipped (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:329-339`). This makes evaluator order the priority order and means a "stop" never cancels another evaluator's request — explicitly documented at `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:34-37`.
- **Counters exposed via `AdditionalProperties`.** The `LoopContext.AdditionalProperties` bag is the documented escape hatch for evaluator implementations that need run-local state without losing statelessness (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopContext.cs:89-96`). The Python equivalent is `session.state[...]` plus the session snapshot mechanism.
- **Magentic separates "replan" from "stall" from "reset".** Update-plans facts + plan, stall is just a counter threshold, reset clears chat history and broadcasts a reset signal — distinct operations with distinct triggers (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1109-1186`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:288-345`).

## Notable Patterns

- **Predicate + (optional) feedback tuple.** The predicate returns either `bool` or `(bool, str|None)` so the same hook can decide continuation *and* ship a correction note in one call (`python/packages/core/agent_framework/_harness/_loop.py:127-138`). C# uses a richer `LoopEvaluation` with `Continue(string? feedback)`, `ContinueWithMessages(...)`, `Stop()` to allow either feedback-by-string or full message authoring (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopEvaluation.cs:24-86`).
- **Progress log as a first-class side effect.** Separate from feedback, `record_feedback` captures one entry per iteration into `progress`, which is then either injected into the next turn's input (no session / `fresh_context`) or only the latest entry (with a session) so the prompt doesn't bloat (`python/packages/core/agent_framework/_harness/_loop.py:637-651`, `python/packages/core/agent_framework/_harness/_loop.py:730-748`). The C# equivalent is `LoopContext.Feedback` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopContext.cs:77-87`).
- **Streaming parity with non-streaming.** Both languages run the same loop over a `ResponseStream`; the streaming path yields every iteration's updates *and* surfaces the injected "nudge" messages as `user` updates between iterations so consumers see the full back-and-forth (`python/packages/core/agent_framework/_harness/_loop.py:518-610`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:215-321`). Inner-agent cancellations (`MiddlewareTermination`) are explicitly handled so the stream does not leak exceptions (`python/packages/core/agent_framework/_harness/_loop.py:546-552`).
- **`TodoProvider` / `BackgroundAgentsProvider` predicates are factory-returned.** `todos_remaining(provider)` and `background_tasks_running(provider)` are the canonical `should_continue` shapes for those provider patterns (`python/packages/core/agent_framework/_harness/_loop.py:751-796`); C# mirrors with `TodoCompletionLoopEvaluator` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/TodoCompletionLoopEvaluator.cs:42-149`).
- **Multi-criterion judge prompt via `{{criteria}}` placeholder.** Both languages render user criteria into a bullet block and expose them at a `{{criteria}}`/`{criteria}` placeholder in the instructions; absent criteria strip the placeholder (`python/packages/core/agent_framework/_harness/_loop.py:60-95`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/AIJudgeLoopEvaluator.cs:78-95`). The same instruction is also forwarded to the agent via `additional_instructions` so its output is held to the criteria from the start.
- **Magentic progress ledger as a typed slot system.** Both implementations demand a structured JSON object (`is_request_satisfied`, `is_in_loop`, `is_progress_being_made`, `next_speaker`, `instruction_or_question`) with *reasons* attached (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:288-333`, `dotnet/src/Microsoft.Agents.AI.Workflows/MagenticProgressLedger.cs:17-131`). The boilerplate prompt template forces the model into a reviewer role rather than letting it mix roles freely.
- **Explicit retry inside Magentic manager.** `_complete` (Python) and `UpdateProgressLedgerAsync` (C#) wrap the LLM call in `for attempt in range(progress_ledger_retry_count):` with linear-backoff `await asyncio.sleep(0.25 * attempts)` / `Task.Delay(250 * attempts, ...)` between attempts, and emit a warning event at the end of each failed attempt (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:709-727`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:70-105`).
- **Tool-approval queue replays itself into the next iteration.** When `ToolApprovalMiddleware` queues multiple pending approvals, it auto-approves any that match a standing rule and re-injects the rest as the *next* agent invocation's input (`python/packages/core/agent_framework/_harness/_tool_approval.py:560-589`); this is a re-invocation loop on the *function-call* layer, orthogonal to the agent layer.

## Tradeoffs

- **`max_iterations=None` is dangerously quiet.** Opting into an unbounded loop is a single kwarg, and a predicate that always returns `True` will burn tokens indefinitely with only a logger line on completion (`python/packages/core/agent_framework/_harness/_loop.py:117, 264, 661`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:67`). There is no progress observability once the cap is removed.
- **`fresh_context=True` only works for sessions that support serialization.** For hosted conversations services that store messages by reference, "fresh context" still contains the full message log because the snapshot is just an id reference (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgentOptions.cs:46-49`).
- **Order-sensitive evaluator set.** The "first `ShouldReinvoke` wins" rule is documented but means a buggy order (e.g. `CompletionMarkerLoopEvaluator` before a real judge evaluator) silently disables the judge's reflection (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:34-37`).
- **Streaming cannot return only the last iteration.** `LoopAgentOptions.NonStreamingReturnsLastResponseOnly` only applies to non-streaming runs — by design (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgentOptions.cs:91-101`); a streaming caller always sees every iteration's updates and the loop cannot know in advance which is last.
- **Tool-call budget vs LLM-judge budget are not unified.** The agent loop counts iteration attempts; the function-call loop counts `function_call` items. They are independent budgets; nothing stops `max_function_calls=10` from being exhausted in the first iteration of a `max_iterations=10` loop. The framework does not surface a unified retry budget for both layers.
- **`JudgeVerdict` is not attached to the returned response.** When a judge-driven loop terminates, only the final iteration's `AgentResponse` is returned; the verdict that *caused* termination is dropped (No evidence found for attaching it).
- **Magentic plan-review port is opt-in.** `require_plan_signoff=False` is the default, so unattended escalations to humans never happen unless explicitly wired (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:878-895`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:80`).

## Failure Modes / Edge Cases

- **Marker ambiguity in the text fallback.** The non-overlapping `VERDICT: DONE`/`VERDICT: MORE` markers keep the loop safe on ambiguous output; however, a judge that emits *both* on the same turn — or whose reply ends with prose — still defaults to "more" (safer default). Verified by `test_judge_text_fallback_*` (`python/packages/core/tests/core/test_harness_loop.py:776-806`).
- **Substring false-positive guard.** The fallback excludes substring traps like "UNANSWERED"/"NOT ANSWERED" via the explicit marker contract (`dotnet/tests/Microsoft.Agents.AI.UnitTests/Harness/Loop/AIJudgeLoopEvaluatorTests.cs:97-116`).
- **Empty next-speaker / non-string fallback.** Magentic falls back to the *first* registered participant when the ledger returns an unexpected type for `next_speaker` (`python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1122-1126`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:305-311`) and emits a `WorkflowWarningEvent`. A next-speaker that is *invalid* (not in the registry) routes to `PrepareFinalAnswerAsync` (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:313-320`, `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1128-1131`).
- **Progress ledger JSON parse failure.** Above `MaxProgressLedgerRetryCount` the exception is rethrown in C#, and `RuntimeError` is raised in Python — the orchestrator catches it and triggers `ResetAndReplanAsync` once, so a persistent JSON parse failure will look like a stall rather than an outright error (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:272-279`, `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1080-1085`). Tests assert ≥ 3 warnings are emitted for 3 retry attempts (`dotnet/tests/Microsoft.Agents.AI.Workflows.UnitTests/MagenticOrchestrationTests.cs:941-943`).
- **Function-call loop hits `max_iterations`.** Forced `tool_choice="none"` final call; final response is the assistant's reply with `tool_choice=none`, **not** the in-progress `function_call` chain — verified by `test_max_iterations_*` (`python/packages/core/tests/core/test_function_invocation_logic.py:1032-1202`).
- **Pending tool-approval returns.** C# loop stops on the first iteration whose response contains a `ToolApprovalRequestContent`; the response (with the approval request) is returned verbatim — the loop never silently auto-resolves (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:189-193`).
- **Mixed approval-request batches.** `_drain_auto_approvable_queue` + `_process_outbound_messages` may queue non-rule approval requests for the next iteration; "auto_approved" approval in batch hides them but keeps visibility through session-state rebinding (`python/packages/core/agent_framework/_harness/_tool_approval.py:546-589`).
- **Long-running tool abandonment.** For MCP long-running tasks, the function-calling layer treats `max_task_wait` exceedance as abandonment and fires best-effort `tasks/cancel` rather than retrying (only the poll path retries on 408-timeout once) (`python/packages/core/agent_framework/_mcp.py:1775-2110`).
- **Re-invocation without feedback in `additional_instructions`.** With `additional_instructions` set, those instructions are persisted **server-side** across iterations when using a session, so the model sees them on every iteration even when the local transcript is reset (`python/packages/core/agent_framework/_harness/_loop.py:403-411, 326-328`).

## Future Considerations

- **Critical: expose a per-loop “spend” counter to the caller.** Today the only way to observe progress from outside is to wrap the evaluator. Adding a public counter in `LoopContext` (or a `loop_monitor` callback) would let UIs and dashboards detect runaway loops before `max_iterations`.
- **Critical: attach the last `JudgeVerdict` to the returned `AgentResponse`** (e.g. as `additional_properties["judge_verdict"]`) so callers can distinguish "first try right" from "corrected on iteration N".
- **Recommended: unify the function-call and loop budgets** under one counter so callers can declare "spend up to N total LLM/tool calls" instead of having two independent caps.
- **Recommended: log a warning when an unbounded loop is constructed** to reduce accidental footguns; the cap defaults are explicit but the `None` case is silent.
- **Recommended: built-in `Magentic` self-correction sample promoted to a sample with tests**, not just `writer-critic iteration workflow` prose (`dotnet/eng/verify-samples/WorkflowSamples.cs:104-105`). The primitive exists end-to-end in both languages but is exercised primarily via Magentic.
- **Nice-to-have: deduplicate consecutive iterations** detected by the predicate (e.g. detect when `loop_kwargs["last_result"].text == previous_iteration.text` and short-circuit with a warning event).

## Questions / Gaps

1. **Is there a public guarantee about which iteration's verdict is exposed?** No — searched the loop, eval, and middleware modules; the `LoopEvaluation` is consumed only by the loop. The final `AgentResponse` carries only the last agent message. Callers wanting a "judge's final report" must wrap the evaluator themselves.
2. **Does the Python loop emit OpenTelemetry events for each iteration?** The middleware runs inside the agent pipeline (`python/packages/core/agent_framework/_harness/_loop.py:398-421`), but I did not find an explicit `OtelAttr` for the loop in `_telemetry.py`; the evidence for telemetry is only "**`AgentTelemetryLayer`**" mentioned in `_agent.py`. No clear evidence found for fine-grained reflection-loop spans.
3. **What happens when the inner agent's `run` raises an exception?** The streaming path catches `MiddlewareTermination` and stops cleanly (`python/packages/core/agent_framework/_harness/_loop.py:546-552`); an `asyncio.CancelledError` short-circuits the workflow runner (`python/packages/core/agent_framework/_workflows/_runner.py:115-120`). Other exceptions propagate — there is no self-correction on inner-agent *exceptions*, only on bad outputs.
4. **Is there a way to inject a critic-agent that *modifies* the iteration's output rather than just sending a `feedback` string?** `LoopEvaluation.ContinueWithMessages(...)` lets an evaluator send an exact message sequence (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopEvaluation.cs:81-85`), but no built-in helper renders it as a "writer → critic → revise" twin-agent loop.
5. **How do reask loops compose with checkpointing?** Both `LoopAgent` and Magentic write session/ledger state into checkpoints (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:131, 154`; Magentic checkpoints at `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:355-396` and `python/packages/orchestrations/agent_framework_orchestrations/_magentic.py:1267-1286`) — the loop's counters and feedback log are checkpointed, but the next-iteration input is rebuilt fresh on resume rather than replayed.

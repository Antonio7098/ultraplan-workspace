## Summary

OpenHands' open-source repo (`openhands/`) is now a thin **V1 orchestrator** layered over a separately-shipped agent runtime: the `openhands` namespace is shared with `openhands-sdk` and `openhands-agent-server` (`openhands/__init__.py:1-4`; `pyproject.toml:60-62`). The actual agent loop, tool-execution cycle, and any LLM-side reask/critic machinery live behind the agent-server's HTTP API. As a direct consequence, this repo has **no reflection prompt, no critic agent, no per-step retry decorator, and no malformed-output parser** of its own — `grep -r "reflect\|reask\|critic\|tenacity\|@retry" openhands/openhands/app_server --include="*.py"` returns no agents of self-correction. Reflection-style logic is concentrated in exactly one place: a deferred side-effect callback (`SetTitleCallbackProcessor`) that re-polls the agent-server for a conversation title on `MessageEvent`. Everything else in the "self-correction" surface is HTTP retry-on-network-error semantics (or its absence).

The bounded retry pattern that does exist is concrete and well-tested: `SetTitleCallbackProcessor._poll_for_title` runs **at most 4 polls at 3 s intervals** for a total of ~12 s (`openhands/app_server/event_callback/set_title_callback_processor.py:32-34, 52-79`). On all-fail, the callback **stays `ACTIVE`** and the next `MessageEvent` re-triggers it (`openhands/app_server/event_callback/set_title_callback_processor.py:132-138`), so the loop is technically unbounded across messages but bounded per-invocation. The state machine is `ACTIVE → DISABLED` (only on success) or `ACTIVE → ACTIVE` on transient failure — there is no `ERROR` terminal for callbacks, even though `EventCallbackStatus.ERROR` exists in the enum (`openhands/app_server/event_callback/event_callback_models.py:33-37`). Failures are recorded as `EventCallbackResult(status=ERROR, detail=str(exc))` rows (`openhands/app_server/event_callback/sql_event_callback_service.py:241-249`), but those rows are write-only observability, not a re-trigger channel.

Service-level retry/polling is **present but uniformly timeout-bounded and fail-soft**: `wait_for_sandbox_running` polls every 2 s for up to 120 s and re-checks the agent-server `/alive` endpoint to close a startup race (`openhands/app_server/sandbox/sandbox_service.py:93-140`); `poll_agent_servers` runs an infinite `while True` that sleeps on errors and just logs (`openhands/app_server/sandbox/remote_sandbox_service.py:682-769`); `_wait_for_server_ready` polls `/alive` for 30 s (`openhands/app_server/sandbox/process_sandbox_service.py:161-177`). `SetTitleCallbackProcessor` is the only one of these that includes structured backoff state for the caller.

Validation in this layer is overwhelmingly **silent**. The docstring on `send_message_to_conversation` is explicit: "This endpoint is intentionally a thin proxy that forwards messages to the agent server without additional processing logic. Any custom processing (validation, transformation, side effects) should be implemented via webhook callbacks, not in this endpoint" (`openhands/app_server/app_conversation/app_conversation_router.py:467-470`). The `_process_pending_messages` flow forwards queued messages to the agent-server, and on individual `httpx` failures it **deletes the messages anyway** after logging a warning (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1977-2006`) — a "deliver-best-effort" pattern that hides failures. File-backed event deserialization swallows `Exception` and returns `None` (`openhands/app_server/event/filesystem_event_service.py:23-31`); `LLMProfiles._skip_invalid_profiles` skips any profile that fails `LLM.model_validate` so a single bad profile cannot break the whole settings load (`openhands/app_server/settings/llm_profiles.py:130-148`). Both behaviors reduce blast radius but mean the app_server never "fails loud" on a malformed LLM-side payload.

The only output-validation channel that the app_server reads from the agent is **HTTP status**: agent-server 5xx → `502 Bad Gateway` to the user (`openhands/app_server/app_conversation/app_conversation_router.py:570-578, 748-756`; `openhands/app_server/app_conversation/live_status_app_conversation_service.py:443-458`); `httpx.ConnectError` → `502 Failed to reach agent server` (`openhands/app_server/app_conversation/app_conversation_router.py:579-584`); pause/missing/error sandbox states surface as `409`, `410`, `503` respectively (`openhands/app_server/app_conversation/app_conversation_router.py:509-536`). There is no LLM-as-judge, no schema validator over agent output, no re-prompt. The agent's `max_iterations` budget is **forwards-propagated to the SDK** through `ConversationSettings` (`openhands/app_server/settings/settings_models.py:32-40, 105-136`; `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1900-1912`), but enforcement is in the SDK/agent-server, not in this repo.

Reflection-style escalation does exist in one narrow path: the analytics taxonomy `_classify_error_type` (budget, timeout, cancel, model/LLM/rate-limit/auth, runtime) is what the system uses to *retrospectively* tag a terminal `ERROR`/`STUCK` conversation for dashboards (`openhands/app_server/event_callback/webhook_router.py:70-90, 141-173`). It runs **after** the agent-server has already declared the run terminal — it is a post-mortem classifier, not a corrective trigger.

## Rating

**2 / 10 — Absent at the agent loop level; deferred side-effects and HTTP-timeout retries at the orchestrator level only.**

Rationale (evidence-backed):
- − No reflection prompt, reask template, critic agent, or LLM-as-judge exists in `openhands/openhands/`. A targeted search for `reflect | reask | critic | @retry | tenacity | max_iter` over `openhands/app_server/**/*.py` matches only one loop (`SetTitleCallbackProcessor._poll_for_title` at `set_title_callback_processor.py:52-79`), one SDK field name pass-through (`max_iterations` in `settings_models.py:109` and `live_status_app_conversation_service.py:1900-1902`), and runtime-config commentary (`constants.py:67`, `sandbox_spec_service.py:91-100`). The agent loop is in the external `openhands-sdk` package; the open-source repo explicitly does not re-validate or re-ask on its behalf (`app_conversation_router.py:467-470`).
- − No retry decorator or library. `tenacity>=8.5,<10` is a dependency (`pyproject.toml:98, 184`) but the open-source code uses zero `@retry`/`stop_after_attempt` patterns. Every retry is hand-rolled `for _ in range(N): await asyncio.sleep(...)` — fragile because it cannot be retuned per-exception or per-network-condition.
- − Retry budgets are inconsistent and not centralized. `SetTitleCallbackProcessor` uses `_NUM_POLL_ATTEMPTS = 4` × `_POLL_DELAY_S = 3` (`set_title_callback_processor.py:32-34`); `wait_for_sandbox_running` uses `timeout=120, poll_interval=2` (`sandbox_service.py:93-98`); `_wait_for_server_ready` uses `timeout=30, sleep=1` (`process_sandbox_service.py:161-177`); `poll_agent_servers` has no cap and just `await asyncio.sleep(sleep_interval)` on error (`remote_sandbox_service.py:764-768`). No single config knob.
- − Escalation is missing entirely. `EventCallbackStatus` has `ERROR` in the enum (`event_callback_models.py:37`) but no `SetTitleCallbackProcessor` path ever sets it; the processor flips only between `ACTIVE` and `DISABLED` (`set_title_callback_processor.py:151-152`). The `EventCallbackResult(ERROR, detail=...)` row written by `SQLEventCallbackService.execute_callback` (`sql_event_callback_service.py:241-249`) is never re-read to drive anything — it is observability-only.
- − Best-effort delivery hides failures. `_process_pending_messages` deletes every queued message after the per-message `httpx` POST, regardless of success/failure (`live_status_app_conversation_service.py:1994-2006`). A persistent 502 in the agent server silently drops the user's message.
- − `Event.model_validate_json` failures are dropped, not escalated. The filesystem event service returns `None` for any parse error and only logs (`filesystem_event_service.py:23-31`); the same pattern is repeated in `aws_event_service.py:43` and `google_cloud_event_service.py:49`. There is no quarantine or replay path.
- − Reask-style hooks are a side door. The webhook router comment says custom processing "should be implemented via webhook callbacks" (`app_conversation_router.py:467-470`), but the shipped `SetTitleCallbackProcessor` is the only concrete `EventCallbackProcessor` and is not a reask — it is a title fetcher. The architecture is extensible in principle, but only the title fetcher is implemented.
- + There *is* a bounded, tested retry loop. `_poll_for_title` is the rare self-contained retry, with explicit test coverage for "no title yet" (returns None, stays `ACTIVE`) and "all 4 attempts error" (returns None, logs 4 warnings, stays `ACTIVE`) (`tests/unit/app_server/test_set_title_callback_processor.py:138-208, 211-294`). The retry budget is short and predictable.
- + Polling loops are explicit and not silent. `wait_for_sandbox_running` raises `SandboxError` with a clear message and a 120 s cap (`sandbox_service.py:117-140`); `_wait_for_server_ready` returns a bool and lets the caller decide (`process_sandbox_service.py:161-177`). Failures are surfaced, not hidden, at the sandbox layer.
- + The "drop on parse" path is at least centralized. `FilesystemEventService._load_event` is the only reader that needs to know how to handle malformed JSON; new event-store implementations (S3, GCS) just need their own `_load_event` (`filesystem_event_service.py:17-31`, `aws_event_service.py:43`, `google_cloud_event_service.py:49`).

## Answers to Dimension Questions

1. **When does the agent self-correct?**
   In this repo, never. The agent loop is in the external `openhands-sdk` package; the V1 orchestrator explicitly disclaims responsibility for any in-line processing (`openhands/app_server/app_conversation/app_conversation_router.py:467-470`). The only self-correction-shaped loop in the open-source code is `SetTitleCallbackProcessor` polling for a title, which is a deferred side effect rather than an in-loop correction (`openhands/app_server/event_callback/set_title_callback_processor.py:32-79, 132-138`).

2. **Is correction bounded?**
   Per-invocation, yes. `_poll_for_title` caps at 4 attempts × 3 s ≈ 12 s wall time (`openhands/app_server/event_callback/set_title_callback_processor.py:32-34, 52`). Across messages, the cap is *not* bounded: a failing callback stays `ACTIVE` and re-fires on every subsequent `MessageEvent`, so a permanently broken title endpoint will retry indefinitely. The `EventCallbackStatus.ERROR` terminal exists but is never assigned by the title processor (`openhands/app_server/event_callback/event_callback_models.py:33-37`; `set_title_callback_processor.py:92-152`).

3. **What evidence is shown during correction?**
   `_logger.warning('Title poll failed for conversation %s: %s', url, exc)` per failed attempt (`openhands/app_server/event_callback/set_title_callback_processor.py:67-73`); an `EventCallbackResult(status=ERROR, detail=str(exc))` row persisted per thrown exception inside `execute_callback` (`openhands/app_server/event_callback/sql_event_callback_service.py:236-249`). The original agent output is **not** passed back to the LLM as a corrective signal; the callback only sees `Event` and `callback` (`openhands/app_server/event_callback/event_callback_models.py:47-54`). No agent-side reflection prompt is ever built in this repo.

4. **Does correction improve outputs or hide errors?**
   It hides. Best-effort delivery in `_process_pending_messages` drops failed messages after a single attempt (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1994-2006`); the filesystem event loader returns `None` on JSON parse failure and only logs (`openhands/app_server/event/filesystem_event_service.py:23-31`); `LLMProfiles._skip_invalid_profiles` silently drops malformed profiles rather than surfacing them (`openhands/app_server/settings/llm_profiles.py:130-148`); the agent-server version check fails open on any error (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:902-909`). None of these attempt a corrective second pass; they all just continue.

5. **Are repeated failures escalated?**
   No. The only "ERROR" tag is in `EventCallbackStatus` and the `EventCallbackResult` observability row, neither of which feeds back into the conversation lifecycle. `_classify_error_type` (`openhands/app_server/event_callback/webhook_router.py:70-90`) runs *after* the agent-server has already declared the run terminal — it is a post-mortem label for analytics, not a corrective trigger. There is no dead-letter queue, no `Stuck`/red-flag path, no circuit breaker. The system can sit in a permanent retry loop on a broken title endpoint with no human-visible signal beyond per-message log lines.

## Architectural Decisions

- **Split runtime into `openhands-sdk` + `openhands-agent-server` PyPI packages.** The Python `openhands` package is a namespace that extends `__path__` so the orchestrator and the agent share a single import root (`openhands/__init__.py:1-4`). All reflection/correction logic is in the SDK/agent-server; the open-source repo is the V1 control plane.
- **Express any in-loop processing as `EventCallbackProcessor` subclasses** registered against `EventKind` values (`openhands/app_server/event_callback/event_callback_models.py:40-54`; `openhands/app_server/event_callback/webhook_router.py:421-435`). The webhook router comment makes this design intent explicit: "Any custom processing (validation, transformation, side effects) should be implemented via webhook callbacks" (`openhands/app_server/app_conversation/app_conversation_router.py:467-470`).
- **Reuse Pydantic discriminated unions for everything that crosses the agent boundary** (events, actions, callbacks, agent kinds) so the orchestrator can deserialize and dispatch without bespoke parsers (`openhands/app_server/event_callback/event_callback_models.py:19-30`; `openhands/app_server/event_callback/__init__.py:1-9`).
- **Bounded polling for sandbox startup, unbounded polling for remote-sandbox refresh.** `wait_for_sandbox_running` raises on timeout (`openhands/app_server/sandbox/sandbox_service.py:117-140`); `poll_agent_servers` is a long-running background task with `while True` and per-iteration `await asyncio.sleep(sleep_interval)` on error (`openhands/app_server/sandbox/remote_sandbox_service.py:699-768`). The asymmetry is intentional — startup is bounded, steady-state refresh is not.
- **"Best-effort + log" as the default failure policy** for hooks, skills, profiles, events, and pending messages. Every one of these has a `try / except Exception / _logger.warning(...)` path that returns the safe empty value (`filesystem_event_service.py:28-31`; `skill_loader.py:331-337`; `llm_profiles.py:144-147`; `hook_loader.py:127-147`; `live_status_app_conversation_service.py:1994-1995`). The system is designed to keep running on partial failure, not to surface it.

## Notable Patterns

- **Single retry-with-status persistence** for the deferred side-effect callback (`SetTitleCallbackProcessor`). Retry state lives in the `EventCallback.status` column; the next message event re-fires the callback, which is essentially a poor-man's queue (`set_title_callback_processor.py:32-34, 132-138, 151-152`; `sql_event_callback_service.py:228-231`).
- **Sequential per-event callback execution** rather than `asyncio.gather`: "We don't use asynio.gather here because callbacks must be run in sequence" (`openhands/app_server/event_callback/webhook_router.py:571-573`). This prevents the title from racing against itself.
- **Sequential pending-message delivery** with delete-on-failure rather than requeue-on-failure (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1976-2006`). The system prefers losing a message to blocking the start task.
- **Process-sandbox `/alive` polling is fail-soft** — `except Exception: pass` then `await asyncio.sleep(1)` (`openhands/app_server/sandbox/process_sandbox_service.py:174-176`). The caller decides what a 30 s timeout means.
- **Agent-server version verification is fail-open** with an explicit comment: "can't verify, so don't block — the create POST still surfaces a custom-image hint" (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:902-909`).
- **"Drop on parse" event loader** is the inverse of the retry pattern: invalid events are not retried or quarantined, they vanish (`openhands/app_server/event/filesystem_event_service.py:23-31`).
- **HTTP-503 vs HTTP-409 vs HTTP-410 vs HTTP-502 are a tiny failure state machine** for agent-server interactions. Sandbox `MISSING` (archived) → 410, `ERROR` → 503, `PAUSED`/`STARTING` → 409, agent-server 5xx → 502, connection failure → 502 (`openhands/app_server/app_conversation/app_conversation_router.py:509-536, 570-584, 748-756`).
- **Terminal-state error classifier** is a post-hoc label not a trigger — `_classify_error_type` runs on a `ConversationStateUpdateEvent` that is already terminal, mapping the message into a `model_error | budget_exceeded | timeout | user_cancelled | runtime_error | unknown` bucket for analytics (`openhands/app_server/event_callback/webhook_router.py:70-90, 141-173`).

## Tradeoffs

- **Silence over loudness.** Many failure paths in this layer log a warning and continue (`filesystem_event_service.py:28-31`; `skill_loader.py:331-337`; `llm_profiles.py:144-147`; `hook_loader.py:131-147`; `live_status_app_conversation_service.py:1994-1995`). The win is that one bad profile/event/hook/message does not break the user's whole session. The cost is that real failures hide — a broken title endpoint looks identical to "title not generated yet" in user-visible state.
- **Externalization of the agent loop.** The agent-side reflection, reask, and self-correction logic lives in `openhands-sdk==1.29.0` and `openhands-agent-server==1.29.0` (`pyproject.toml:60-62`). This isolates the orchestrator from LLM mechanics, but it also means the open-source repo is structurally unable to add a critic agent, a retry-on-bad-tool-call, or an LLM-as-judge without touching the SDK — and the SDK is a closed external package from this repo's perspective.
- **Hand-rolled retry vs. library.** Every retry loop in this repo is a `for _ in range(N): await asyncio.sleep(...)` pattern (`set_title_callback_processor.py:52-53`; `process_sandbox_service.py:164-176`; `remote_sandbox_service.py:699-768`; `sandbox_service.py:119-138`). `tenacity` is a declared dependency but never imported (`pyproject.toml:98`). The trade is fewer transitive surfaces vs. per-exception-class backoff strategies that the hand-rolled loops cannot express.
- **Permanent ACTIVE on failure.** The `SetTitleCallbackProcessor` only flips to `DISABLED` on success, so a structurally broken title endpoint will retry forever (`set_title_callback_processor.py:132-138, 151-152`). This trades bounded resource use for never silently dropping a missed title.

## Failure Modes / Edge Cases

- **Permanent 502 in the agent server** during pending-message flush silently drops the queued user message (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1994-2006`). There is no DLQ.
- **Malformed agent event** in any of the three event-store implementations is silently dropped rather than quarantined or replayed (`openhands/app_server/event/filesystem_event_service.py:23-31`; `aws_event_service.py:43`; `google_cloud_event_service.py:49`). The conversation timeline can have invisible holes.

## Future Considerations

- The `EventCallbackStatus.ERROR` enum value exists but no shipped processor ever sets it (`openhands/app_server/event_callback/event_callback_models.py:37`). A future `SetTitleCallbackProcessor` could move from "ACTIVE forever on failure" to a bounded counter that flips to `ERROR` and stops retrying after N consecutive failures, with a UI signal.
- A retry decorator via `tenacity` (already a dep) would let each retry loop express `stop_after_attempt`, `wait_exponential`, and `retry_if_exception_type` policies uniformly (`pyproject.toml:98`). - `EventCallbackResult(ERROR)` rows are write-only today (`openhands/app_server/event_callback/sql_event_callback_service.py:236-249`). A future scheduler could re-fire callbacks whose previous result was `ERROR` after a backoff.

A first shipped `EventCallbackProcessor` for a real reask (e.g., "if MessageEvent contains a tool_call whose observation is `None`, send a follow-up `MessageEvent` saying 'please redo'") would be the obvious first in-line correction primitive.

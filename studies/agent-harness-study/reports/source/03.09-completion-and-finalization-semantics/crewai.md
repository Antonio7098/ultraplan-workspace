## Summary

CrewAI has no single terminal "run status" — completion is encoded as a chain of structural contracts plus events. At the lowest level each LLM call carries a raw `finish_reason` (e.g. `"stop"`, `"length"`, `"tool_calls"`) extracted through a shared `_finish_reason_utils.extract_choices_finish_reason_and_id` helper (`sources/crewai/lib/crewai/src/crewai/llms/_finish_reason_utils.py:21-55`); that raw value is propagated into `LLMCallCompletedEvent.finish_reason` (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:547-575`) but **never** normalized into a domain-level stop reason — the agent loop only cares whether the parser produced `AgentFinish` vs `AgentAction`. Termination conditions therefore live in the loop bodies: `has_reached_max_iterations` + `handle_max_iterations_exceeded` (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:280-350`) force one extra LLM call when the budget is exhausted; the ReAct path treats any output containing `FINAL_ANSWER_ACTION` as done (`sources/crewai/lib/crewai/src/crewai/agents/parser.py:91-101`), and the native-tool path treats any tool marked `result_as_answer=True` as a final answer (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:1097-1107`, `sources/crewai/lib/crewai/src/crewai/utilities/tool_utils.py:263`). The `experimental.AgentExecutor` (the default executor used by modern crews, per `CrewAgentExecutor.__init__` deprecation warning at `sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:145-151`) adds a richer Plan-and-Execute `finalize()` with a `_finalize_called` re-entrancy lock, a heuristic for "answer is already strong enough" and a fallback synthesized "no explicit answer" stub (`sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2254-2326`).

Output packaging splits into three classes: `TaskOutput` (`sources/crewai/lib/crewai/src/crewai/tasks/task_output.py:14-104`) per task, `CrewOutput` (`sources/crewai/lib/crewai/src/crewai/crews/crew_output.py:13-60`) aggregating them, and `LiteAgentOutput` (`sources/crewai/lib/crewai/src/crewai/lite_agent_output.py:30-118`) for the standalone agent path. All three carry usage + messages. `_create_crew_output` validates that at least one task produced a non-empty raw output before assembling (`sources/crewai/lib/crewai/src/crewai/crew.py:1858-1864`), flushes the event bus, then emits `CrewKickoffCompletedEvent(output=..., total_tokens=self.token_usage.total_tokens)` (`sources/crewai/lib/crewai/src/crewai/crew.py:1868-1879`); `usage_metrics` and `token_usage` are aggregated in `calculate_usage_metrics` (`sources/crewai/lib/crewai/src/crewai/crew.py:2088-2112`) summing per-agent LLM summaries built by `_track_token_usage_internal` / `UsageMetrics.from_provider_dict` (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:887-911`, `sources/crewai/lib/crewai/src/crewai/types/usage_metrics.py:79-124`).

Output validation is delegated to user-supplied guardrails via `process_guardrail` returning `GuardrailResult(success, result, error)` (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail.py:60-187`); task-level retries execute `guardrail_max_retries + 1` total attempts and raise on exhaustion (`sources/crewai/lib/crewai/src/crewai/task.py:1246-1353`). Failure is observable via paired `TaskStartedEvent`/`TaskCompletedEvent`/`TaskFailedEvent` (`sources/crewai/lib/crewai/src/crewai/events/types/task_events.py:36`), `CrewKickoffFailedEvent` (`sources/crewai/lib/crewai/src/crewai/crew.py:1054-1063`), and `AgentExecutionErrorEvent` (`sources/crewai/lib/crewai/src/crewai/agent/core.py:685-717`), but there is **no** `Crew.status`, no `RunStatus.completed` enum, no `[DONE]`-style terminal marker — finish is a side-effectful event, not a first-class object.

## Rating

**6 / 10 (Present but inconsistent, weakly documented as a unifying concept, and partially fragile.)**

Rationale: there are real machinery pieces — a centralized `_finish_reason_utils` for cross-provider finish reasons (`sources/crewai/lib/crewai/src/crewai/llms/_finish_reason_utils.py:21-55`), an explicit `finalize()` with re-entrancy protection and last-resort fallback in the experimental executor (`sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2254-2326`), an aggregated `UsageMetrics` model with provider normalization (`sources/crewai/lib/crewai/src/crewai/types/usage_metrics.py:32-124`), a `CrewKickoffCompletedEvent` carrying `total_tokens` (`sources/crewai/lib/crewai/src/crewai/events/types/crew_events.py:43-48`), and guardrail retry loops (`sources/crewai/lib/crewai/src/crewai/task.py:1246-1353`). However, three structural issues hold the score below 7: (a) **the `CrewAgentExecutor` (the legacy ReAct executor) is still active in code paths and emits a `DeprecationWarning` rather than being a typed, fail-fast choice** (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:143-151`); (b) **no domain-level stop reason exists** — `finish_reason` reaches `LLMCallCompletedEvent` (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:547-575`) but the agent loop conflates every "not-action" branch into `AgentFinish` regardless of whether the LLM said `length` or `content_filter` (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:370-386`); (c) **`_create_crew_output` raises `ValueError` on empty `task_outputs` but never marks the partial run as failed** — it just aborts the kickoff (`sources/crewai/lib/crewai/src/crewai/crew.py:1862-1864`).

## Answers to Dimension Questions

1. **What counts as done?**
   Three independent "done" signals apply per agent per task:
   - **ReAct text path**: any line containing `FINAL_ANSWER_ACTION` (the literal `Final Answer:`) becomes `AgentFinish` via `parser.parse` (`sources/crewai/lib/crewai/src/crewai/agents/parser.py:91-101`); if neither Action nor Final Answer is present, parsing raises `OutputParserError` which becomes a *retry* signal (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:434-442`).
   - **Native tool path**: a tool decorated with `result_as_answer=True` returns `AgentFinish(thought="Tool result is the final answer", output=result, text=result)` immediately after the tool message is appended (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:1097-1107`); otherwise the LLM responds with a string and the loop converts it to `AgentFinish` (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:543-564`).
   - **Budget exhaustion**: `has_reached_max_iterations(self.iterations, self.max_iter)` triggers `handle_max_iterations_exceeded`, which prompts one final "force_final_answer" call (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:280-350`).
   For a `Crew`, a `CrewOutput` is produced only when at least one task yielded a non-empty `raw` (`sources/crewai/lib/crewai/src/crewai/crew.py:1858-1864`); the boundary is an exception (`ValueError`) rather than a "warning" status. The experimental executor wraps all three paths in a single `finalize()` that, on success, returns `"completed"` and sets `state.is_finished = True` (`sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2254-2326`).

2. **Can the model falsely declare done?**
   Yes, in the sense that **any non-actionable model output becomes a final answer** with no length/refusal classification. The parser only recognizes two outcomes (Action vs Final Answer); anything that does not contain `Action:` and does not contain `Final Answer:` raises `OutputParserError` and is retried. But once the model's text *does* contain `Final Answer:`, `format_answer` returns `AgentFinish` regardless of whether the answer is empty, hallucinated, or truncated by `length`. The `format_answer` fallback even catches all parsing exceptions and synthesizes `AgentFinish(thought="Failed to parse LLM response", output=answer, text=answer)` (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:370-386`). `max_tokens` exhaustion is handled *generously* by the prompt-forcing final-call path — there is no path that classifies a `"length"` finish_reason as `failed`. The provider-level `finish_reason` is captured into `LLMCallCompletedEvent.finish_reason` (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:555-573`) but is **not** translated into a domain-level stop reason, so a `length`-truncated response looks identical to a clean stop from the runner's perspective.

3. **Are final outputs validated?**
   Only when the user opts in. Validation is **not** automatic. The pipeline exposes:
   - `Task.guardrail` (single) and `Task.guardrails` (list) — `GuardrailCallable | str` (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail_types.py:12-18`). `process_guardrail` calls the callable with the `TaskOutput`, wraps the boolean/string result into `GuardrailResult(success, result, error)` and enforces a mutual-exclusion validator (`success + result && error` ⇒ `ValueError`) (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail.py:60-103`).
   - `_invoke_guardrail_function` runs `guardrail_max_retries + 1` attempts and **fully re-executes** the task via `agent.execute_task(...)` on each failure, replacing the `task_output.raw` (and refreshing `pydantic`/`json_dict` via `_export_output`) on success (`sources/crewai/lib/crewai/src/crewai/task.py:1246-1353`). On exhaustion it raises `Exception("Task failed {guardrail_name} validation after {n} retries. Last error: ...")` (`sources/crewai/lib/crewai/src/crewai/task.py:1298-1301`).
   - `LiteAgent.guardrail` runs once after the loop completes; on failure it retries the whole `_execute_core` recursion up to `guardrail_max_retries` then raises `Exception("Agent's guardrail failed validation after {n} retries. Last error: ...")` (`sources/crewai/lib/crewai/src/crewai/lite_agent.py:696-726`).
   - `HallucinationGuardrail` is documented as **no-op** in open-source, returning `(True, task_output.raw)` and emitting a warning each call (`sources/crewai/lib/crewai/src/crewai/tasks/hallucination_guardrail.py:74-103`).
   - Built-in `LLMGuardrail` validates by spawning a *new* `Agent(role="Guardrail Agent")` and asking for `LLMGuardrailResult(valid: bool, feedback: str?)`; it does the kicking off the same `agent.kickoff` as user code, leaking any other crew/agent config into the guardrail call (`sources/crewai/lib/crewai/src/crewai/tasks/llm_guardrail.py:60-117`).
   There is no structural schema validator for *plain raw* outputs; validation only runs when the user wires one of the above. Output format conformance to `OutputFormat.JSON` / `OutputFormat.PYDANTIC` is enforced by `_export_output`, but the `RAW` format is left unexamined.

4. **Is usage/cost finalized?**
   Yes, in two layers and with a well-normalized schema. `UsageMetrics` carries the full set of token primitives (`total_tokens`, `prompt_tokens`, `cached_prompt_tokens`, `completion_tokens`, `reasoning_tokens`, `cache_creation_tokens`, `successful_requests`) and aggregates via `add_usage_metrics` (`sources/crewai/lib/crewai/src/crewai/types/usage_metrics.py:32-77`). Provider normalization in `UsageMetrics.from_provider_dict` resolves the OpenAI/Gemini/Anthropic/Bedrock aliases (`prompt_tokens`/`prompt_token_count`/`input_tokens`, `cached_tokens`/`cache_read_input_tokens`, etc.) and is paired with `BaseLLM._track_token_usage_internal` so per-LLM totals, the `LiteAgentOutput.usage_metrics`, and the `Crew.usage_metrics` all stay in sync (`sources/crewai/lib/crewai/src/crewai/types/usage_metrics.py:79-124`, `sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:887-911`). Per-call aggregation lives in the LLM instance (`self._token_usage`); per-run aggregation is `Crew.calculate_usage_metrics` (line 2088-2112) which sums each BaseLLM agent + each legacy `_token_process`-tracked agent + the manager_agent. The completed event carries `total_tokens` (`sources/crewai/lib/crewai/src/crewai/crew.py:1873-1878`, `sources/crewai/lib/crewai/src/crewai/events/types/crew_events.py:48`). However, **cost is not stored** — there is no `$` field anywhere in the schema; cost must be derived by the caller from `total_tokens` and provider pricing.

5. **Can a run complete with warnings?**
   Partially. The structural answers are split:
   - `Crew` has **no** terminal "with-warnings" state. `_create_crew_output` either succeeds (emits `CrewKickoffCompletedEvent`, returns a `CrewOutput`) or raises `ValueError` (`sources/crewai/lib/crewai/src/crewai/crew.py:1862-1864`). The `CrewOutput` carries no warning field.
   - `CrewOutput.token_usage` and `CrewOutput.tasks_output` (all `TaskOutput`s, each with `messages`) persist everything that happened, but nothing inside the structure is marked as "warning".
   - The experimental executor's `finalize` can return `"skipped"` if `current_answer` is not an `AgentFinish`, but **it returns the string, it does not propagate it into a top-level status field** (`sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2313-2326`).
   - `LiteAgentOutput` retains per-step `TodoExecutionResult.status` in {`completed`, `failed`, `pending`} so a caller can see *partial* completion, but the top-level object carries no `had_partial_failure` flag — `__repr__` shows `todos={completed}/{total} completed` and `replans=N` (`sources/crewai/lib/crewai/src/crewai/lite_agent_output.py:108-118`).
   - `LLMGuardrailCompletedEvent.success: bool` is the closest thing to a "warning" channel — a `success=False, result=feedback, error=feedback` event survives a failed-but-recoverable validation, but failed guardrails that retry are invisible at the top level unless they finally exhaust and raise.
   - `CrewExecutionEventListener`/`EventListener`/`TraceCollectionListener` propagate completion events but the final dispatch is the trace batch manager — there is no aggregate "completed_with_warnings" consumer-side flag.

## Architectural Decisions

- **"Done" is a parser outcome, not a domain-level enum.** CrewAI's ReAct parser resolves two terminal outcomes — `AgentAction` vs `AgentFinish` — and the agent loop guards on `isinstance(formatted_answer, AgentFinish)` (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:330-468`); both branches (text ReAct + native tool) feed the same downstream contract. Provider `finish_reason` is captured per-call (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:547-575`) but no `StopReason` enum maps it to a domain concept; all terminal reasons collapse to "stopping" or "max-iter exceeded".
- **Two coexisting executors by design, with the legacy one marked deprecated.** `CrewAgentExecutor.__init__` raises `DeprecationWarning` and tells callers to migrate to `crewai.experimental.AgentExecutor` (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:143-151`). The new executor carries the richer `finalize()` + `state.is_finished` + `_finalize_called` re-entrancy lock; the legacy executor relies on parsing + max-iter forcing (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:280-350`).
- **`result_as_answer` is the canonical "tool produces final output" hook.** A tool whose `result_as_answer=True` short-circuits the loop in *every* tool-calling site (legacy executor `crew_agent_executor.py:1097-1107`, native executor duplication in `utilities/tool_utils.py:263`, and LiteAgent in `lite_agent.py:914-929`), with the literal `AgentFinish.thought="Tool result is the final answer"` thread shared everywhere.
- **`TaskOutput` carries three parallel representations** — `raw: str`, `pydantic: BaseModel | None`, `json_dict: dict | None` — chosen at execution time based on `OutputFormat` (`sources/crewai/lib/crewai/src/crewai/tasks/task_output.py:14-48`, `sources/crewai/lib/crewai/src/crewai/tasks/output_format.py:6-17`). The choice is preserved in `output_format` and re-derived via the `json` property which raises `ValueError` on mismatch (`sources/crewai/lib/crewai/src/crewai/tasks/task_output.py:62-83`).
- **Finalization events are terminal-only and best-effort.** `CrewKickoffCompletedEvent` is emitted at the end of `_create_crew_output` after `crewai_event_bus.flush()` and carries `total_tokens` for downstream observability (`sources/crewai/lib/crewai/src/crewai/crew.py:1868-1879`). The matching `CrewKickoffFailedEvent` carries only `error: str` and `started_event_id` — no partial output, no metrics (`sources/crewai/lib/crewai/src/crewai/crew.py:1054-1063`, `sources/crewai/lib/crewai/src/crewai/events/types/crew_events.py:51-55`). There is no central store of crew results; consumers must subscribe to events or read `crew.usage_metrics` after kickoff.
- **Guardrail retries drive full re-execution.** When a guardrail rejects, the task code path is `agent.execute_task(... context=error ...)` not a tighter loop (`sources/crewai/lib/crewai/src/crewai/task.py:1320-1324`). This means every retry costs a fresh LLM call and re-enters the agent loop — guardrails become a cheap correctness lever but an expensive one.

## Notable Patterns

- **Centralized `_finish_reason_utils.extract_choices_finish_reason_and_id`** defensive helper (`sources/crewai/lib/crewai/src/crewai/llms/_finish_reason_utils.py:21-55`) is used by both `crewai.llm.LLM` (legacy LiteLLM) and Azure provider, and has its own dedicated test suite (`sources/crewai/lib/crewai/tests/events/test_llm_finish_reason_response_id.py`). It returns `(None, None)` on any error rather than raising, keeping the loop safe against exotic provider types.
- **`finalize()` re-entrancy lock** (`sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2261-2269`) — when concurrent branches both reach a terminal state, the second call returns `"completed"` immediately. Tested explicitly by `test_finalize_called_reset_in_invoke` and the idempotency tests in `tests/test_flow_conversation.py:1450-1483`.
- **`AgentFinish`-forcing helper** — `handle_max_iterations_exceeded` mutates the assistant message and forces one final LLM call (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:280-350`). This is the only sanctioned escape from `max_iter`. When the LLM returns `None` or empty, the helper raises `ValueError("Invalid response from LLM call - None or empty.")` which propagates to the caller as a kickoff failure — not a soft "truncated" completion.
- **`format_answer` permissive fallback** (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:370-386`) — any unhandled `Exception` from `parse()` becomes an `AgentFinish`. This trades correctness for "the loop never panics", and is a non-obvious reason model output can be marked final.
- **`process_guardrail` mutual-exclusion `field_validator`** (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail.py:81-103`) — `success=True` cannot coexist with `error`; `success=False` cannot coexist with `result`. This catches programmer error at the pydantic layer instead of letting ambiguous tuples surface.
- **`TaskOutput.json`/`CrewOutput.json` fail-loud properties** (`sources/crewai/lib/crewai/src/crewai/tasks/task_output.py:62-83`, `sources/crewai/lib/crewai/src/crewai/crews/crew_output.py:30-37`) — accessing `.json` on a non-JSON task raises `ValueError("Invalid output format requested. ...")` rather than returning `None`, which forces callers to acknowledge the format mismatch.
- **`_create_crew_output` double-validation** (`sources/crewai/lib/crewai/src/crewai/crew.py:1858-1864`) — both the empty-list and the all-empty-raw cases raise `ValueError` before any event is emitted. Failure is therefore total or zero, never partial.
- **`UsageMetrics.from_provider_dict` mirrors `BaseLLM._track_token_usage_internal`** — explicit comment in `types/usage_metrics.py:86-87` ensures per-LLM totals, flow-level aggregation, and OTel spans agree on every provider's key aliasing.

## Tradeoffs

- **`_create_crew_output` is fail-fast, not partial.** A single all-empty task outputs raises `ValueError` — there is no `CrewOutput(partial=True)` path; partial work is lost from the top-level return (`sources/crewai/lib/crewai/src/crewai/crew.py:1862-1864`). This is good for *clear* failure semantics, but it ties "the agent partially ran" to "the caller got an exception".
- **Two executors, one deprecating.** `CrewAgentExecutor` is still wired and reachable for legacy crews; the experimental `AgentExecutor` is opt-in via the `experimental` module (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:143-151`). Until crews flip the default, every finalization story has to be checked against *both* executors.
- **Guardrail retries are full task re-executions.** `_invoke_guardrail_function` does not just re-validate — it calls `agent.execute_task(... context=error ...)` for each retry (`sources/crewai/lib/crewai/src/crewai/task.py:1320-1324`). This guards against bad outputs but multiplies LLM cost; users must tune `guardrail_max_retries` carefully.
- **`finish_reason` is preserved but not interpreted.** Every provider completion path captures `finish_reason` into `LLMCallCompletedEvent` (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:547-575`, `sources/crewai/lib/crewai/src/crewai/llms/providers/anthropic/completion.py:924-1016` etc.), but the agent loop treats every `"stop"`/`"length"`/`"tool_calls"` as the same termination category. Operators wanting to track `length`-truncated runs must subscribe to events separately.
- **Hallucination guardrail is intentionally disabled** in the open-source build (`sources/crewai/lib/crewai/src/crewai/tasks/hallucination_guardrail.py:20-103`). The class exists, the docstring exists, but the `__call__` always returns `(True, task_output.raw)` — this is a UX trap.
- **`_finish_execution` is intentionally minimal.** The hook only stops the RPM counter (`sources/crewai/lib/crewai/src/crewai/crew.py:2084-2086`). CrewAI does not persist a `Crew.status`, does not flush metrics to disk, does not emit any extra "post-complete" event. Downstream systems must listen to `CrewKickoffCompletedEvent` themselves.
- **No `CrewOutput.failed` flag.** The class has `raw`, `pydantic`, `json_dict`, `tasks_output`, `token_usage` — none of these encode a "this run had errors" boolean (`sources/crewai/lib/crewai/src/crewai/crews/crew_output.py:13-28`). To distinguish a clean run from one with tool errors, callers must inspect `TaskOutput.messages` or replay events.
- **`max_iter` termination always succeeds** if the LLM returns *something* (even if it is the forced prompt's reflex). `handle_max_iterations_exceeded` only refuses to finish if the LLM call returns `None` or `""` — in which case the exception is `ValueError` (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:334-340`), a higher-severity failure than the original problem.
- **Streaming completion is not a single event.** `LLMStreamChunkEvent` is emitted per chunk and only the *last* chunk carries `finish_reason`; the stream has no synthesized `[DONE]` marker — clients must watch `finish_reason` themselves or wait for `LLMCallCompletedEvent` (`sources/crewai/lib/crewai/src/crewai/llms/providers/anthropic/completion.py:924-1016`, `sources/crewai/lib/crewai/src/crewai/llms/providers/openai/completion.py:1653-1823`).

## Failure Modes / Edge Cases

1. **Empty-output task aborts the whole kickoff.** When all `task_outputs` are empty, `_create_crew_output` raises `ValueError` *after* every task has already executed, so all token-usage work is done but no `CrewOutput` is returned (`sources/crewai/lib/crewai/src/crewai/crew.py:1862-1864`). Subscribers to `CrewKickoffFailedEvent` get `error=str(ValueError)` but no partial output.
2. **`max_iter` + empty LLM response**. If `handle_max_iterations_exceeded`'s forced final call returns `None`/`""`, the helper raises `ValueError("Invalid response from LLM call - None or empty.")` (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:334-340`) — but the choice of `SystemExit` vs exception is split with `handle_context_length` (which raises `SystemExit` if `respect_context_window=False`). Two different unrecoverable terminal paths exist.
3. **`OutputParserError` after `log_error_after=3`** is silently retried up to `max_iter` times by appending the error message back to `messages` as a user turn (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:660-695`). If the LLM is deterministic and stuck on bad output, the run will burn its budget.
4. **Native tool call returning non-string**. `format_native_tool_output_for_agent` returns `str(raw_result)` if the tool has no `format_output_for_agent` formatter (`sources/crewai/lib/crewai/src/crewai/utilities/agent_utils.py:1386-1396`). For complex tool returns this silently stringifies — the model may receive `"{'k': 'v'}"` and lose structured context.
5. **Concurrent finalization races.** `experimental.AgentExecutor.finalize` guards with `_finalize_called`, but the *legacy* `CrewAgentExecutor._invoke_loop` has no such guard — a tool call returning `result_as_answer=True` after the parser produced `AgentFinish` will be re-entered in the loop only by the `while not isinstance(formatted_answer, AgentFinish):` condition, which exits immediately. So the legacy loop is fine, but the absence of explicit finalization means observation tooling can't tell whether a run "really" finished or just exited the loop.
6. **Guardrail callables rejected at checkpoint.** `serialize_guardrails_for_json` drops callable guardrails with a `UserWarning` and replaces them with `None`, so a restored checkpoint loses guardrails silently (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail.py:33-49`). Tests that round-trip a checkpoint through JSON lose that protection.
7. **Task callback + crew callback double-execution**. `_execute_core` calls `self.callback(self.output)` then `crew.task_callback(self.output)` *if* they differ; in sync mode coroutine callbacks are run via `asyncio.run(cb_result)` (`sources/crewai/lib/crewai/src/crewai/task.py:849-863`) — running sync callbacks from inside an already-running event loop will raise `RuntimeError: asyncio.run() cannot be called from a running event loop`.
8. **Tool cache + result mutation**. The tool cache stores `raw_result` before formatting; if the tool mutates state, the cached value is *the raw state at first execution* — but `_append_tool_result_and_check_finality` formats `format_native_tool_output_for_agent(output_tool, cached_result)` from the original `output_tool` reference, leaving a TOCTOU window if the tool's `format_output_for_agent` reads external state (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:925-940`).
9. **`max_usage_count` + retry path**. `_execute_single_native_tool_call` sets `max_usage_reached=True` if the cap is hit and emits `result="Tool '{func_name}' has reached its usage limit..."` (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:899-982`). The model sees the rejection, can react, but no event records the rejection — only `ToolUsageStartedEvent`/`ToolUsageFinishedEvent` (`sources/crewai/lib/crewai/src/crewai/utilities/tool_utils.py:1493-1503` analogous) fire.
10. **Streaming `finish_reason` lost in mid-flight cancellation**. Azure/Gemini streaming accumulation (`sources/crewai/lib/crewai/src/crewai/llms/providers/azure/completion.py:1139-1175`, `sources/crewai/lib/crewai/src/crewai/llms/providers/gemini/completion.py:1199-1237`) reads the last chunk's `finish_reason`; if the consumer drops before the last chunk, the event is silently dropped because there is no synthetic terminal marker.
11. **`LiteAgentExecutionErrorEvent` is emitted only on retry exhaustion**. In `Agent._check_execution_error`, `crewai_event_bus.emit(... AgentExecutionErrorEvent ...)` happens only after `self._times_executed > self.max_retry_limit` (`sources/crewai/lib/crewai/src/crewai/agent/core.py:685-717`), so a transient single-attempt failure does *not* surface as an error event.
12. **`CrewOutput.tasks_output` may include failed-and-retried tasks**. `agent.execute_task` retry is opaque; both the failed result and the successful result may end up inside `messages`, but only the last successful result becomes the `TaskOutput.raw` (`sources/crewai/lib/crewai/src/crewai/task.py:1320-1353`). No "was retried" flag is propagated.

## Future Considerations

- **Promote `experimental.AgentExecutor` and demote `CrewAgentExecutor`.** Today both coexist; the experimental one carries the better finalization model (`finalize()`, re-entrancy lock, last-todo heuristic, fallback stub) but it has not been moved out of `experimental/` (`sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:143-151`, `sources/crewai/lib/crewai/src/crewai/experimental/agent_executor.py:2254-2326`).
- **Introduce a `StopReason` enum at the agent level.** Raw `finish_reason` is captured but never translated; mapping `"length"` → `max_tokens_exceeded` and `"content_filter"` → `refused` would let callers distinguish truncated, refused, and clean-completed runs without subscribing to per-call events (`sources/crewai/lib/crewai/src/crewai/llms/base_llm.py:547-575`, `sources/crewai/lib/crewai/src/crewai/agents/crew_agent_executor.py:330-468`).
- **Add a `Crew.status` field or a `RunStatus`-style enum on `CrewOutput`.** Currently `_create_crew_output` raises `ValueError` instead of returning a partial `CrewOutput`; a `status: Literal["completed", "failed"]` field plus an optional `partial_outputs: list[TaskOutput]` would let callers recover partial work (`sources/crewai/lib/crewai/src/crewai/crew.py:1858-1890`).
- **Synthesize a streaming `[DONE]` marker.** Provider streaming paths lose the terminal chunk if the consumer drops early; a `LLMCallCompletedEvent` is the only guarantee of completion, but it is emitted post-stream — a trailing SSE-style `[DONE]` would let streaming clients match the agent-framework/Letta contract (`sources/crewai/lib/crewai/src/crewai/llms/providers/anthropic/completion.py:924-1016`, `sources/crewai/lib/crewai/src/crewai/llms/providers/azure/completion.py:1139-1175`).
- **Cost accounting.** `UsageMetrics` carries the right primitives but no `$` field; a pricing adapter (per-model cost tables) would close the "is usage/cost finalized?" gap from "tokens" to "dollars" (`sources/crewai/lib/crewai/src/crewai/types/usage_metrics.py:32-77`).
- **Guardrail restore.** `serialize_guardrails_for_json` strips callables silently; surviving checkpoint restore requires a different strategy (capturing the function path, or restoring the callable via a known registry) (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail.py:33-49`).
- **Universal "completed with warnings" outcome.** `LLMGuardrailCompletedEvent.success=False` already encodes recoverable failures, but nothing rolls them up at the `CrewOutput` level; a `warnings: list[str]` (or `partial: bool` flag) on `CrewOutput` would let observability dashboards distinguish clean runs from `tokens=200, todos=3/5` runs (`sources/crewai/lib/crewai/src/crewai/crews/crew_output.py:13-60`).
- **Idempotent outputs from replays.** When `prepare_task_execution` is reached via the replay path (`crew.py:1918-1956`), the stored `TaskOutput` is reconstructed from `_store_execution_log` JSON. If the validation step changes during replay, output determinism is lost; the `messages` list is restored verbatim but the `pydantic` BaseModel instance is reconstructed only by re-running `_export_output`.

## Questions / Gaps

- **What does a `crew.kickoff(...)` that raises `ValueError("No valid task outputs...")` look like to the caller?** The docstring claims "CrewOutput or CrewStreamingOutput" (`sources/crewai/lib/crewai/src/crewai/crew.py:985-997`) but `ValueError` is not a `CrewOutput`; is this acceptable contract or is it a bug?
- **Is `Crew.usage_metrics` mutated to `None` on a `CrewKickoffFailedEvent` path?** Today it is only set after `_create_crew_output` returns (`sources/crewai/lib/crewai/src/crewai/crew.py:1051`), so on failure the field stays at the default `None`. Failure-time observability is partial.
- **Does `_post_kickoff` ever transform `CrewOutput`?** It is a no-op override hook (`sources/crewai/lib/crewai/src/crewai/crew.py:1071-1072`). Production crews may subclass it — what callbacks are observed? The `after_kickoff_callbacks` list is the documented hook (`sources/crewai/lib/crewai/src/crewai/crew.py:285-291`, `:1046-1047`); these run *before* `usage_metrics` is assigned on kickoff path. Is that ordering intentional?
- **What is the `Crew.execute_end_with_at_most_one_async_task` validator's failure mode?** (`sources/crewai/lib/crewai/src/crewai/crew.py:767-785`.) It raises a `PydanticCustomError` at construction, not at kickoff — so a misconfigured crew fails at import/decoration, not at run time, which differs from the documentable promise "kickoff is where validation happens".
- **`max_iter` enforcement on the experimental executor** vs the legacy one — how do they differ? Both call the same `has_reached_max_iterations` helper, but `AgentExecutor.finalize` has additional behavioral paths. Is the difference documented?
- **Guardrail `retry_count` semantics**: `process_guardrail` accepts a `retry_count: int` and emits it on the event (`sources/crewai/lib/crewai/src/crewai/utilities/guardrail.py:123-185`) but the LiteAgent path reads the same field as a singleton counter (`sources/crewai/lib/crewai/src/crewai/lite_agent.py:700-711`), while `Task._invoke_guardrail_function` reads per-index counters (`sources/crewai/lib/crewai/src/crewai/task.py:1246-1353`). The contract is split — is this a deliberate design?
- **`agent.copy()` semantics on completion**: `crew.kickoff_for_each` copies the crew for each input (`sources/crewai/lib/crewai/src/crewai/crew.py:1095-1108`) — does each copy's `usage_metrics` accumulate into the parent? Yes (line 1100-1101), but only when `not self.stream`; the streaming path skips accumulation. Should that be documented?

## Summary

The repository implements two parallel "done" semantics with explicit, typed surfaces.

1. **Per-turn chat completion** is encoded by the underlying `IChatClient` provider via `Microsoft.Extensions.AI.AI.ChatResponse.FinishReason` (`ChatFinishReason` values: `Stop`, `Length`, `ToolCalls`, `ContentFilter`, `Error`, etc.). `AgentResponse` carries it through unmodified (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:208`) and the hosting/streaming layers emit it per chunk (`dotnet/src/Microsoft.Agents.AI.Workflows/MessageMerger.cs:160-191`, `dotnet/src/Microsoft.Agents.AI.Hosting.OpenAI/ChatCompletions/AIAgentChatCompletionsProcessor.cs:75`, `dotnet/src/Microsoft.Agents.AI.Hosting.OpenAI/ChatCompletions/AgentResponseExtensions.cs:37`, `dotnet/src/Microsoft.Agents.AI.Hosting.OpenAI/Responses/AgentResponseUpdateExtensions.cs:170`).
2. **Whole-run completion** for `AIAgent.RunAsync` / `RunStreamingAsync` is implicit: a non-streaming call resolves when the chat client returns; a streaming call resolves when the inner enumerator completes. The wrapper surfaces `AgentResponse.ContinuationToken` only when the run is still in flight (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:179-180`) and `RunCoreStreamingAsync` performs end-of-run notifications in `finally` (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:291-402`).
3. **Workflow completion** uses an explicit state machine. `RunStatus` (`dotnet/src/Microsoft.Agents.AI.Workflows/RunStatus.cs:8-34`) exposes `NotStarted`, `Idle`, `PendingRequests`, `Running`, `Ended`. The `StreamingRunEventStream` transitions `Running → (Idle | PendingRequests | Ended)` and emits an internal `InternalHaltSignal` per superstep (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:117-126, 248-251`). `Run.InProcessRunner` enforces post-end correctness with a `_runEnded` flag and throws `InvalidOperationException` from `CheckEnded()` if a caller tries to enqueue new messages after the run is over (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunnerContext.cs:491-524`).
4. **Orchestration termination** (`LoopAgent`, `MagenticOrchestrator`) is a separate layer that re-invokes the agent or the team and decides done-ness itself: `LoopAgent` enforces `MaxIterations` and `LoopEvaluation.Stop` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:189-211`); `MagenticOrchestrator` checks ledger `IsRequestSatisfied` plus hard `TaskLimits.MaxRoundCount`/`MaxResetCount` (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:248-258, 281-286, 347-353`).
5. **Output validation** is opt-in via `ChatResponseFormat`/`AgentResponse<T>` (`dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgentStructuredOutput.cs:120-139`), but final output is **not validated** by the framework — the XML docs on `ChatClientAgent` explicitly tell developers to sanitize LLM output themselves (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:22-37`).
6. **Usage/cost recording** is propagated through `AgentResponse.Usage` (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:217`) and `UsageContent` inside streaming updates (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:265-310`), aggregated across responses in `MessageMerger.MergeUsage` (`dotnet/src/Microsoft.Agents.AI.Workflows/MessageMerger.cs:271-304`), and translated to per-provider formats for `OpenAI Responses`, `ChatCompletions`, `Foundry`, `DurableTask`, and `GitHub.Copilot`. There is no first-class "finalization event" or aggregated cost summary object — usage lives on the response/update itself.
7. **Run summaries** are not produced. The closest analogues are:
   - `RunStatus` + `Run.OutgoingEvents` (`dotnet/src/Microsoft.Agents.AI.Workflows/Run.cs:46-52`),
   - `WorkflowErrorEvent` (`dotnet/src/Microsoft.Agents.AI.Workflows/WorkflowErrorEvent.cs:13`) / `ExecutorFailedEvent` (`dotnet/src/Microsoft.Agents.AI.Workflows/ExecutorFailedEvent.cs:12-18`) / `WorkflowWarningEvent` (`dotnet/src/Microsoft.Agents.AI.Workflows/WorkflowWarningEvent.cs:9`),
   - and the per-step `SuperStepCompletionInfo` debug struct (`dotnet/src/Microsoft.Agents.AI.Workflows/SuperStepCompletionInfo.cs:11-44`).

Finalization in this codebase is therefore a **layered model**: the chat client decides per-turn completion; the agent wraps it with notification of providers and finalization helpers; the workflow adds a real status state machine and a halt signal; orchestrators add their own termination rules. Done-ness is observable, multi-valued, and explicitly typed — but correctness is not enforced by the framework.

## Rating

**Score: 7 / 10**

Rationale:

- Clear, typed model for "done" in workflows (`RunStatus` enum with 5 values, `StreamingRunEventStream` halt signal, `RequestHaltEvent` as terminal marker).
- Explicit detection of pending external work (`PendingRequests`) that blocks completion while external requests are unserviced (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:117-119`).
- Hard fences in orchestrators prevent runaway loops (`LoopAgentOptions.MaxIterations`, `TaskLimits.MaxRoundCount/MaxResetCount/MaxStallCount`).
- Usage is propagated end-to-end via `UsageDetails` and per-provider translators; no silent loss.
- `RunStatus.Idle` is the only "true done" state, and it is asserted by tests (`tests/Microsoft.Agents.AI.Workflows.UnitTests/InProcessExecutionTests.cs:36`, `tests/Microsoft.Agents.AI.Workflows.UnitTests/InProcessStateTests.cs:106-139`).
- `_runEnded` flag and `CheckEnded()` prevent post-completion enqueue (good operational safety).

Where it falls short of 8+:

- "Done" in `AIAgent.RunAsync` is **implicit** (returns when the chat client returns). There is no typed "RunOutcome" / "RunStatus" with success vs partial vs failure — only `FinishReason` and exceptions.
- The model never asserts that the **model** truthfully declared done. There is no schema-level "is this a final answer?" gate; structural validation only happens if the caller explicitly passes a `ChatResponseFormat` (`AIAgentStructuredOutput.RunAsync<T>`).
- The XML docs repeatedly warn that LLM output must be sanitized by the caller (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:22-37, 25-37`); the framework does not validate final output, so "done" can mean "merely stopped".
- `WorkflowWarningEvent` is fired on intermediate conditions (e.g., Magentic ledger exceptions, invalid next speaker) but no `Warning` aggregate state exists on `Run` (`RunStatus` has no `CompletedWithWarnings`). The closest semantic is `RunStatus.Idle` after warning events were raised.
- `AgentResponse.ContinuationToken` is set even when the response may have terminated; consumers must consult the token presence to detect ongoing vs done (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:170-180`).
- The `MagenticOrchestrator` "completion" yields a final-answer message that the manager chat-client synthesizes (`MagenticManager.PrepareFinalAnswerAsync`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:108-112`) — the framework trusts the LLM to produce the final answer text and does not cross-check it against the task.
- `WorkflowOutputEvent` filters by `OutputFilter` allowlist (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/OutputFilter.cs:8-17`), but the output itself is not type-checked at runtime against a declared schema — only an `InvalidOperationException` if the executor tries to yield something outside its declared yield types (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunnerContext.cs:283`).

## Answers to Dimension Questions

1. **What counts as done?**

   Three different "done" semantics exist depending on the layer:

   - **Per chat turn**: `ChatFinishReason` carries provider-native reasons (`Stop`, `Length`, `ToolCalls`, `ContentFilter`, etc.) on `AgentResponse.FinishReason` (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:208`).
   - **Per agent run**: The non-streaming `AIAgent.RunAsync` resolves as soon as `IChatClient.GetResponseAsync` returns. There is no `RunStatus` field on `AgentResponse`; presence/absence of `ContinuationToken` is the only "still running vs finished" signal (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:170-180`).
   - **Per workflow run**: Explicit `RunStatus` with `Idle` being the canonical completion state; `Ended` indicates the handle was disposed/cancelled; `PendingRequests` indicates completion is blocked on external requests (`dotnet/src/Microsoft.Agents.AI.Workflows/RunStatus.cs:8-34`, `dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:117-126`).

   Done-ness therefore means **"no more actions to perform right now"**, not **"the answer is correct"** — that distinction is left to the caller and to the orchestrator's evaluators.

2. **Can the model falsely declare done?**

   Yes — the framework does not gate completion on truthfulness. The XML docs explicitly recommend that callers treat the final response as untrusted output (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:22-37`). The only structural protection is `AgentResponse<T>.RunAsync<T>` which sets a `ChatResponseFormat` JSON schema (`dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgentStructuredOutput.cs:120-139`), but deserialization of the LLM-produced JSON may still succeed on garbage. `LoopAgent` mitigates by re-invoking the agent and asking evaluators (`AIJudgeLoopEvaluator`, `CompletionMarkerLoopEvaluator`) to confirm completion (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopEvaluator.cs:22-23`, `dotnet/src/Microsoft.Agents.AI/Harness/Loop/CompletionMarkerLoopEvaluator.cs:23-77`).

3. **Are final outputs validated?**

   Only at the type-marshalling level: `OutputFilter` and `IWorkflowContext.YieldOutputAsync` enforce that yielded outputs match the executor's declared `OutputTypes` (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunnerContext.cs:283`). `MessageMerger` rejects merging responses with different `ResponseId` (`dotnet/src/Microsoft.Agents.AI.Workflows/MessageMerger.cs:206`). For chat responses, no schema/format/content validation is performed — `AgentResponse<T>.Result` throws `InvalidOperationException` only on empty text or null deserialized value (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse{T}.cs:51-73`).

4. **Is usage/cost finalized?**

   Yes, per response: `AgentResponse.Usage : UsageDetails?` carries `InputTokenCount` / `OutputTokenCount` / `TotalTokenCount` / `AdditionalCounts` (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse.cs:217`). For multi-response workflows, `MessageMerger.MergeUsage` aggregates them (`dotnet/src/Microsoft.Agents.AI.Workflows/MessageMerger.cs:271-304`). There is **no first-class aggregate cost/summary record** for the whole run — the caller is expected to merge usages themselves or inspect `Run.OutgoingEvents` for `AgentResponseEvent` instances.

5. **Can a run complete with warnings?**

   Yes. `WorkflowWarningEvent` and `SubworkflowWarningEvent` are emitted on non-fatal conditions (e.g., Magentic progress-ledger exceptions, invalid next speaker) and the workflow still proceeds to `RunStatus.Idle` (`dotnet/src/Microsoft.Agents.AI.Workflows/WorkflowWarningEvent.cs:9`, `dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:274, 308, 316`). However, there is **no `RunStatus.CompletedWithWarnings`** value; the only way for callers to discover warnings is to inspect `Run.OutgoingEvents` (`dotnet/src/Microsoft.Agents.AI.Workflows/Run.cs:52`). For chat agents, the framework has no "warning" channel at all — only `FinishReason` and exceptions.

## Architectural Decisions

- **Completion is layered, not unified.** Chat-client finish reasons, agent return values, workflow `RunStatus`, and orchestrator termination rules all coexist without a single "RunOutcome" type. This is intentional — each layer has its own correct notion of done — but it makes cross-layer reasoning harder.
- **`RunStatus.Idle` is the canonical completion state.** Tests assert it directly (`dotnet/tests/Microsoft.Agents.AI.Workflows.UnitTests/InProcessExecutionTests.cs:36`, `dotnet/tests/Microsoft.Agents.AI.Workflows.UnitTests/Sample/01a_Simple_Workflow_Sequential.cs:16`). It is set after the last superstep when neither unprocessed messages nor unserviced external requests remain (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:117-119`).
- **`PendingRequests` blocks Idle intentionally.** A workflow with outstanding `ExternalRequest`s transitions to `PendingRequests`, not `Idle` (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:117-119`). This is the canonical "blocked on external input" state and is the only non-`Running`/`Ended` state that does not yield `break` from the consumer iterator.
- **Termination is one-shot.** Once `RunStatus.Ended` is set, `_runEnded` is latched and any attempt to enqueue throws `InvalidOperationException` (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunnerContext.cs:491-524`). `AsyncRunHandle.TakeEventStreamAsync` enforces single-active-enumerator (throws if a second enumerator is requested) (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/AsyncRunHandle.cs:64-67`).
- **`RequestHaltEvent` is the internal stream-termination sentinel.** It is filtered by `AsyncRunHandle.TakeEventStreamAsync` (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/AsyncRunHandle.cs:80-87`) and never exposed to consumers.
- **InternalHaltSignal is the run-loop coordination event.** It is a private nested class in `StreamingRunEventStream` carrying the captured `RunStatus` and a monotonically increasing epoch (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:332-336`). It is filtered by the same enumerator and never published.
- **Orchestrators reuse the chat client to decide "done."** `MagenticManager.PrepareFinalAnswerAsync` calls the chat client to synthesize the final answer from the ledger (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticManager.cs:108-112`), and `MagenticOrchestrator.PrepareFinalAnswerAsync` yields that answer and sets `IsTerminated = true` (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:347-353`). Termination is therefore tied to the LLM's own "request satisfied" verdict plus `IsRequestSatisfied` flag in `MagenticProgressLedger`.
- **Loop termination has both cooperative and hard limits.** `LoopAgent` stops when all evaluators return `Stop()` OR when `iteration >= MaxIterations` OR when the response contains `ToolApprovalRequestContent` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:189-211`). The hard `MaxIterations` cap is a safety fence against evaluator bugs.
- **Output gating uses both allowlist (`OutputExecutors`) and declared type.** `Workflow.OutputExecutors` controls which executors' `YieldOutputAsync` calls produce events (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/OutputFilter.cs:8-17`); `IWorkflowContext.YieldOutputAsync` enforces that the yielded type matches the executor's declared yield type (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunnerContext.cs:283`).
- **Streaming finalization is `finally`-disciplined.** `ChatClientAgent.RunCoreStreamingAsync` always disposes the inner `responseUpdatesEnumerator` in a `finally` so `PerServiceCallChatHistoryPersistingChatClient` finalizers run on early `yield break` (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:334-401`). The comment at line 335-340 explicitly explains why this matters.

## Notable Patterns

- **Halting-via-internal-event.** `InternalHaltSignal` and `RequestHaltEvent` are both used to break the consumer loop while keeping the producer (run loop) alive — a clean channel-based coordination primitive.
- **Epoch-tagged completion signals.** `StreamingRunEventStream._completionEpoch` ensures stale halt signals from previous consumer iterations are ignored (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:124-126, 218-235`). This is a robust pattern for "I want completion signal N, not N-1."
- **End-of-run notification pattern with `forceNotify`.** `ChatClientAgent.NotifyProvidersOfNewMessagesAtEndOfRunAsync` skips end-of-run notification when a per-service-call decorator already handled persistence, but forces it for continuation-token / background-response scenarios where per-service-call persistence is unreliable (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:882-896`). This dual-mode notification is a thoughtful trade-off between cost and reliability.
- **Two-state loop wrapper** (`ToolApprovalAgent`): keeps looping internally while approval requests can be auto-approved; breaks out to caller only when human action is required. The "yield break after single unapproved request" pattern is reused for both streaming and non-streaming (`dotnet/src/Microsoft.Agents.AI/Harness/ToolApproval/ToolApprovalAgent.cs:107-136, 218-260`).
- **Terminator executors per orchestration style.** `OutputMessagesExecutor` for sequential, `ConcurrentEndExecutor` for fan-in, `HandoffEndExecutor` for handoff, all yield the final messages via `IWorkflowContext.YieldOutputAsync`. They are typed subclasses of `Executor` so they are registered with the `OutputExecutors` allowlist.
- **Optional final-output response aggregation.** `LoopAgent.BuildResult` either returns the last response verbatim or builds an aggregated `AgentResponse(transcript)` copying the final iteration's `AgentId`, `ResponseId`, `CreatedAt`, `FinishReason`, `Usage`, `AdditionalProperties`, and `ContinuationToken` (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:452-469`).

## Tradeoffs

- **No unified "RunOutcome" type.** Cross-layer reasoning about "did this run succeed and what did it cost?" requires the caller to assemble the answer from `AgentResponse`, `Run.OutgoingEvents`, and possibly `AgentRunHandle.ReadAgentResponseAsync`. This is flexible but increases caller burden.
- **Final output is not validated by the framework.** Trust is delegated to the caller. This avoids imposing a validation cost on simple cases but means "done" can mean "merely stopped." (See XML docs at `dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:22-37`.)
- **`RunStatus.Idle` after `WorkflowWarningEvent`.** Warnings do not roll up into a `CompletedWithWarnings` state, so consumers that only check `Idle` can miss operational issues. The mitigation is to inspect `OutgoingEvents`, but this is a known operational burden.
- **Implicit per-turn completion in agents.** Without inspecting `AgentResponse.ContinuationToken`, callers cannot tell whether a response is final or "still in flight" (background mode). The framework provides the signal but does not require consumers to use it.
- **Token-level usage aggregation lives in `MessageMerger`.** Other code paths (single-agent runs, durable tasks) do not aggregate usage — they only expose the per-response `UsageDetails`. Callers building dashboards must aggregate themselves.
- **`MagenticOrchestrator` trusts the LLM's "IsRequestSatisfied" verdict.** The progress ledger's `IsRequestSatisfied` flag is set by the manager chat client; the orchestrator trusts it without cross-validation. This is fast but inherits any hallucination in the manager.
- **Hard limits in Magentic are independent of `LoopAgent.MaxIterations`.** Callers using both layers must reason about two caps; the loop wraps the Magentic workflow so each Magentic step is one iteration of the loop.

## Failure Modes / Edge Cases

- **Cancellation after start, before drain.** `StreamingRunEventStream.StopAsync` cancels the run loop and waits for clean shutdown but does not surface cancellation as `RunStatus.Cancelled` — the status is set to `Ended` (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/StreamingRunEventStream.cs:174-176`). Callers cannot distinguish "user cancelled" from "run completed naturally and was disposed."
- **Double enumerator on event stream.** `AsyncRunHandle.TakeEventStreamAsync` throws `InvalidOperationException("The event stream has already been taken...")` on a second concurrent enumerator (`dotnet/src/Microsoft.Agents.AI.Workflows/Execution/AsyncRunHandle.cs:64-67`). Tests rely on this behavior.
- **Post-end enqueue.** `RunContext.CheckEnded()` throws `InvalidOperationException` if any caller tries to enqueue after `EndRunAsync()` ran (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunnerContext.cs:491-497`). This is checked at 25+ call sites to prevent zombie activity.
- **Mismatched checkpoint restoration.** `InProcessRunner.RestoreCheckpointCoreAsync` throws `InvalidDataException("The specified checkpoint is not compatible with the workflow associated with this runner.")` if the checkpoint and workflow definitions differ (`dotnet/src/Microsoft.Agents.AI.Workflows/InProc/InProcessRunner.cs:411`).
- **Loop with pending tool approval.** `LoopAgent` returns the response unmodified and trusts the caller to resolve approvals. If the caller ignores this, the agent stops dead (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:189-194`).
- **MaxIterations hit while response contains content.** `LoopAgent.LogMaxIterationsReached` logs at `Information` level and returns the current response (`dotnet/src/Microsoft.Agents.AI/Harness/Loop/LoopAgent.cs:487-493`). Callers see a normal-looking response without an explicit "max reached" indicator. (The loop's behavior of building an aggregated transcript is the only side effect.)
- **Magentic progress-ledger failure.** The orchestrator catches exceptions, emits a `WorkflowWarningEvent`, calls `ResetAndReplanAsync`, and retries (`dotnet/src/Microsoft.Agents.AI.Workflows/Specialized/Magentic/MagenticOrchestrator.cs:271-279`). If the retry budget (`MaxProgressLedgerRetryCount = 3`) is exhausted, the orchestrator ends in an undefined state — the final-answer is never produced and the task is terminated via the round/reset limit if reached.
- **`ConversationId` mismatch.** `ChatClientAgent.PrepareSessionAndMessagesAsync` throws `InvalidOperationException` if the caller's session `ConversationId` differs from `chatOptions.ConversationId` (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:740-747`).
- **No `ConversationId` returned despite service-managed session.** `ChatClientAgent.UpdateSessionConversationId` throws `InvalidOperationException("Service did not return a valid conversation id when using an AgentSession with service managed chat history.")` (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:816`).
- **ChatHistoryProvider and conversation-id collision.** `UpdateSessionConversationId` warns and optionally throws (`WarnOnChatHistoryProviderConflict`, `ThrowOnChatHistoryProviderConflict`) when both are present (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:820-840`).
- **Background response without session.** `PrepareSessionAndMessagesAsync` throws if `AllowBackgroundResponses == true` and `session is null` (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:715-718`).
- **Background continuation with new input messages.** Throws if input messages are supplied while continuing a background response (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:732-736`).
- **Multiple top-level JSON objects in structured output.** `AgentResponse<T>.DeserializeFirstTopLevelObject` reads only the first object via `AllowMultipleValues = true` reader, citing the well-known GPT-3.5 multiple-objects bug (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentResponse{T}.cs:78-93`).

## Future Considerations

- **Add a first-class `RunOutcome`/`RunSummary` type.** Could aggregate `AgentResponse`, `RunStatus`, total usage, total cost, warning count, error count, iteration count. Would unify the three layered views (per-turn / per-agent / per-workflow).
- **Distinguish `RunStatus.Idle` from `RunStatus.Cancelled`.** Currently both end up as `Ended` after disposal. Explicit cancellation state would help callers detect intentional vs accidental termination.
- **`RunStatus.CompletedWithWarnings`.** Allow consumers to gate on warnings without inspecting `OutgoingEvents`. Would help observability dashboards.
- **Trust-but-verify final answer in Magentic.** Cross-check the LLM-synthesized final answer against the task definition; emit a warning if the answer is suspiciously short or empty.
- **Per-run usage aggregation in `AIAgent.RunAsync`.** Expose an aggregated `Usage` field across multi-step agent runs, not just per-response.
- **Validation hooks in `AgentResponse`.** Optional `Action<AgentResponse>` validator that runs before returning; today the only structural check is JSON deserialization.
- **Test for harness `LoopAgent` ↔ `MagenticOrchestrator` interaction.** Currently the two termination models operate independently; a single test that confirms one cap does not accidentally bypass the other would harden the multi-layer story.

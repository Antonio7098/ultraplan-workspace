## Summary

Agent Framework partitions state into **eight explicit categories** with named owner modules and clear durability characteristics. The taxonomy is layered, not flat: a top-level `AgentSession` carries `session_id`, `service_session_id`, and a single mutable `state` dict keyed by `source_id` (defined in `python/packages/core/agent_framework/_sessions.py:746-812`); providers slot their own typed state under distinct keys (`source_id`-scoped namespaces) and own serialization to durable stores via the `HistoryProvider` protocol (`python/packages/core/agent_framework/_sessions.py:413-535`). Workflow execution adds a second, parallel state tier: a shared, superstep-buffered `State` object (`python/packages/core/agent_framework/_workflows/_state.py:6-127`) plus per-executor state, both captured by `WorkflowCheckpoint` (`python/packages/core/agent_framework/_workflows/_checkpoint.py:30-117`). Conversation history has three coexisting representations — service-managed (chat client `conversation_id`), in-memory `AgentSession.state[provider.source_id]["messages"]` (default), and JSONL-on-disk via `FileHistoryProvider` (`python/packages/core/agent_framework/_sessions.py:893-1134`). Long-running conversations use the Azure Durable Functions entity model with a versioned `DurableAgentState` schema (1.1.0) serialized via `durable-agent-entity-state.json` (`schemas/durable-agent-entity-state.json:208-217`; `python/packages/durabletask/agent_framework_durabletask/_durable_agent_state.py:365-474`). Harness providers (`TodoProvider`, `ToolApprovalMiddleware`, `MemoryContextProvider`, `BackgroundAgentsProvider`, `AgentModeProvider`, `FileMemoryProvider`, `SkillsProvider`, `CompactionProvider`) store their working state in the same `AgentSession.state` dict under unique `source_id` namespaces, with optional file-backed or Redis-backed stores as separate `Store` classes (`python/packages/core/agent_framework/_harness/_todo.py:227-443`, `python/packages/core/agent_framework/_harness/_memory.py:554-924`). The repo **can** explain what state must survive a crash: workflow checkpoints capture executor state + committed shared state + in-flight messages + pending HITL requests at every superstep boundary, durable agent state captures the full conversation history per schema version, file history providers write JSONL with per-session locks and process-stripe locking, and in-flight workflow state is explicitly marked as ephemeral (lost on `BackgroundAgentsProvider` restart, see `_background_agents.py:288-292`). The framework distinguishes "derived" state (`message_count`, `is_in_loop`, ledger computations in `OrchestrationState`/`MagenticContext`) from "source of truth" (raw persisted message lists) via explicit `to_dict()`/`from_dict()` round-trip classes that all implement `SerializationMixin` or `DictConvertible`.

## Rating

**8 / 10** — Mature, layered state model with explicit owners, durability contracts, and crash-survivable checkpoints for both agent conversations (Durable Agent Entity, JSONL file history) and workflows (`WorkflowCheckpoint` with superstep-buffered shared state + executor snapshots + in-flight messages). Every state category has a named owner module and a documented round-trip serializer. Deducted points: (a) `AgentSession.state` is an untyped `dict[str, Any]` keyed by string `source_id`, with no schema or static enforcement — type safety is entirely the developer's responsibility (`_sessions.py:772`); (b) several harness providers (todo, memory, tool-approval) maintain **non-serializable per-session locks and runtime maps** (`_todo.py:482-492`, `_memory.py:1019-1038`, `_tool_approval.py:489-499`) that are intentionally lost across process restarts; the framework documents this but does not offer a recovery protocol; (c) the `_state_cache` on `AgentEntityStateProviderMixin` (`_entities.py:44`) is a plain instance attribute that re-reads from `get_state()` lazily but is also `set` to a fresh `DurableAgentState()` on `reset()` — the on-entity state is owned by the durable-task framework, not the agent framework itself; (d) checkpoint deserialization uses Python `pickle` with a restricted allowlist (`_workflows/_checkpoint_encoding.py:115-145`) — the docs explicitly disclaim this as defense-in-depth rather than a security boundary (`_checkpoint_encoding.py:18-45`), so checkpoint integrity depends on access-controlled storage; (e) `FileHistoryProvider` stores conversation history as plaintext JSONL with explicit security guidance (`_sessions.py:902-911`).

## Answers to Dimension Questions

1. **What kinds of state exist?**
   The framework recognizes, with distinct owner modules and storage strategies: (a) **session identity state** (`session_id`, `service_session_id` on `AgentSession`); (b) **session state** — an untyped `dict[str, Any]` on `AgentSession` partitioned by `source_id` (`_sessions.py:772`); (c) **per-invocation state** — `SessionContext.context_messages`, `instructions`, `tools`, `middleware`, `metadata`, `response` (`_sessions.py:154-348`); (d) **conversation history** — three coexisting representations: service-managed (`service_session_id` + chat-client `conversation_id`), session-state `["messages"]` list (default `InMemoryHistoryProvider`), and JSONL-on-disk (`FileHistoryProvider`); (e) **workflow shared state** — `State._committed`/`_pending` superstep-buffered dict (`_workflows/_state.py:6-127`); (f) **executor state** — per-executor dict under reserved `EXECUTOR_STATE_KEY = "_executor_state"` (`_workflows/_const.py:7`); (g) **runtime state** — live `asyncio.Task` handles, child `AgentSession` references, per-session `asyncio.Lock`, per-loop lock stripes (`_background_agents.py:110-115`, `_memory.py:1019-1038`, `_todo.py:482-492`); (h) **durable conversation state** — versioned `DurableAgentState` with schema 1.1.0 (`_durable_agent_state.py:395`); (i) **eval state** — `EvalItem`/`EvalScoreResult`/`EvalResults`/`RubricScore` records (`_evaluation.py:182-373`); (j) **checkpoint metadata** — `iteration_count`, `previous_checkpoint_id`, `metadata`, `version` (`_checkpoint.py:74-88`).

2. **Which state is source-of-truth?**
   For long-running durable conversations, the **Durable Entity storage** (Azure Durable Functions storage via `durabletask` SDK) is the source of truth; `AgentEntityStateProviderMixin._get_state_dict`/`_set_state_dict` delegate to the entity's `get_state`/`set_state` (`_entities.py:46-75`). The in-memory `_state_cache` is a mirror that lazily rehydrates from storage. For workflow execution, the **`CheckpointStorage` backend** (file, in-memory, or external — protocol at `_checkpoint.py:119-189`) is the source of truth for resumable runs; shared `State` is committed at superstep boundaries before checkpoint capture (`_runner.py:140-144, 218-225`). For short-lived local conversations with no durable backend, `AgentSession.state[provider.source_id]["messages"]` (mutated via `InMemoryHistoryProvider.save_messages` at `_sessions.py:878-890`) is the source of truth as long as the process lives. When `service_session_id` is set and the chat client returns its own `conversation_id`, that **server-side conversation log** is the source of truth (`_agents.py:1406-1408`).

3. **Which state is derived?**
   `AgentResponse.usage_details` (assembled from `UsageContent`), `DurableAgentState.message_count` (`_durable_agent_state.py:445-448`), `WorkflowCheckpoint.timestamp` (auto-generated), `MagenticProgressLedger` items (`_magentic.py:288-333`), `AgentLoopMiddleware`'s `progress` log and `next_message` feedback (`_loop.py:135-200`), `CompactionProvider`'s post-strategy projection (`_compaction.py:1270-1305`), and `MemoryContextProvider._topic_score` (`_memory.py:1072-1075`) are all computed from a primary source and re-derivable on demand. None of these are authoritative inputs.

4. **Which state is ephemeral?**
   `SessionContext` (`_sessions.py:154-348`), per-call `_finalize_response`'s local sentinel `conversation_id` (`_sessions.py:537-699`), `BackgroundAgentsProvider._runtime` (in-flight `asyncio.Task`s — `_background_agents.py:288-292`), `MemoryContextProvider` per-loop `_STORE_LOCKS_BY_LOOP` (`_memory.py:930-935`), `TodoProvider._mutation_locks` (`_todo.py:482-492`), `AgentLoopMiddleware`'s `fresh_context=True` ephemeral re-snapshot of the session (`_loop.py:420-580`), `MemoryContextProvider`'s `state.clear()` reset each invocation (`_memory.py:1170-1171`), and `State._pending` (cleared on `commit()`/`discard()`, never checkpointed — `_workflows/_state.py:30-104`, `_checkpoint.py:56-58`) are all explicitly ephemeral and lost on process or workflow restart. `BackgroundAgentsProvider._refresh_task_state` (`_background_agents.py:207-232`) intentionally marks such orphans as `LOST`.

5. **Which state is safe to lose?**
   Ephemeral items above (in-flight `asyncio.Task`s, pending `FunctionInvocationContext`, per-loop locks, `SessionContext`, `State._pending`). Tool-approval `_FUNCTION_INVOCATION_BUDGET_STATE_KEY` (`_tool_approval.py:27`) is session-scoped but reconstructable from message history. The framework explicitly does not lose conversation history or workflow state via two mechanisms: (i) **`State.commit()` at every superstep boundary** before checkpoint capture (`_runner.py:141, 224`), so only committed-but-not-yet-applied writes are discarded; (ii) **`DurableAgentState.to_dict()`** is called on every turn's completion via `AgentEntity.persist_state()` (`_entities.py:71-75, 173-191`) — the durability is delegated to the underlying Durable Entity storage.

## Architectural Decisions

- **Session-state-as-namespaced-dict**: `AgentSession.state: dict[str, Any]` keyed by `source_id` (`_sessions.py:772`) — explicit choice to avoid a rigid base-class hierarchy. Every provider picks its own `source_id` constant (`DEFAULT_TODO_SOURCE_ID = "todo"` at `_todo.py:23`, `DEFAULT_MEMORY_SOURCE_ID = "memory"` at `_memory.py:29`, etc.) and stores a typed payload under that key. Type safety is restored by `SerializationMixin.to_dict()`/`from_dict()` round-trip and a process-global `_STATE_TYPE_REGISTRY` (`_sessions.py:42-147`).
- **`HistoryProvider` is the conversation persistence abstraction** (`_sessions.py:413-535`). The protocol is dual-purpose — load + store, store-only (audit/eval), or load-only — controlled by `load_messages`, `store_inputs`, `store_context_messages`, `store_context_from`, `store_outputs`. This is the seam between framework and storage backends (in-memory, file, Redis, Cosmos, custom).
- **Superstep-buffered `State` (`_workflows/_state.py:6-127`)**: writes go to `_pending`, `get()` reads pending-then-committed, `commit()` flushes at superstep boundaries, `discard()` rolls back. This avoids mid-superstep write conflicts across concurrent executors (mirroring .NET behavior per `_state.py:39-41`).
- **`WorkflowCheckpoint` captures committed-only state**, deliberately excluding `_pending` (`_checkpoint.py:56-58`, `_runner.py:218-225`). Resumed runs therefore start from a consistent snapshot, not a torn one.
- **`DurableAgentState` uses a versioned JSON schema** (`SCHEMA_VERSION = "1.1.0"`, `_durable_agent_state.py:395`) matching `schemas/durable-agent-entity-state.json`. Future schema changes get explicit detection: `from_dict()` warns and resets when `schemaVersion` is missing (`_durable_agent_state.py:426-430`).
- **`AgentEntity.run` rebuilds the full chat message list from `conversation_history` on every invocation** (`_entities.py:150-175`) rather than relying on incremental state — this makes the durable model trivially replayable and resilient to partial writes.
- **`FileHistoryProvider` uses per-file striped locks** (`_sessions.py:916-920, 1103-1115`) — `_FILE_WRITE_LOCKS` (process-local `threading.Lock` × 64 stripes) and a `weakref.WeakKeyDictionary` of event-loop-local `asyncio.Lock` × 64 stripes. Lock contention is reduced to per-file, and locks are GC'd when the loop is.
- **`FileCheckpointStorage` uses atomic write**: writes to `*.json.tmp` then `os.replace()` (`_checkpoint.py:317-322`), same atomic pattern used by `TodoFileStore` (`_todo.py:432-444`) and the memory file store (`_memory.py:119-132`). A crash mid-write leaves the previous file intact.
- **Hybrid JSON+pickle for workflow checkpoints** (`_checkpoint_encoding.py:18-145`): JSON for primitives and structure, base64-pickle for complex Python objects. `_RestrictedUnpickler` enforces an allowlist (built-in safe types + `agent_framework.*` + `openai.types.*`), but the docs are explicit: this is **defense in depth, not a security boundary** (`_checkpoint_encoding.py:18-45`). Treat the storage backend as access-controlled.
- **Per-service-call history middleware**: `PerServiceCallHistoryPersistingMiddleware` (`_sessions.py:570-743`) drives `HistoryProvider` around every model call rather than once per agent run, using a local sentinel `LOCAL_HISTORY_CONVERSATION_ID = "agent_framework_local_history_persistence"` (`_sessions.py:537-542`) that is stripped before the leaf client is invoked. This keeps the function-invocation loop agnostic of which conversation-id strategy is in play.

## Notable Patterns

- **`SerializationMixin` / `DictConvertible` round-trip contracts**: every persisted state class exposes `to_dict()` / `from_dict()` and either registers in `_STATE_TYPE_REGISTRY` (`_sessions.py:69-147`) or implements the `DictConvertible` mixin (`_magentic.py:270-394`). This gives a uniform "wire-format → in-memory → wire-format" cycle across `AgentSession`, `ToolApprovalState`, `TodoItem`, `Message`, `OrchestrationState`, `MagenticContext`, `DurableAgentState*`.
- **`source_id` as a namespace boundary**: every `ContextProvider`, `HistoryProvider`, and harness provider declares a `DEFAULT_SOURCE_ID` class constant and stores its working state under `session.state[source_id]`. This decouples providers (no cross-coupling, no shared schema) and lets multiple providers of the same type coexist if their source IDs differ.
- **Atomic-write-as-default**: `TodoFileStore._save_state_sync` (`_todo.py:432-444`), `FileCheckpointStorage.save` (`_checkpoint.py:317-322`), and `_memory._atomic_write_text` (`_memory.py:119-132`) all use the `temp + os.replace` pattern. The pattern is documented as "a crash, OOM, or disk-full mid-write leaves the previous file (if any) intact" (`_memory.py:120-124`).
- **Per-session locks stored in `WeakKeyDictionary`**: `_todo.py:482-492`, `_memory.py:1019-1038`, `_background_agents.py:294-299` keep per-session `asyncio.Lock` instances in `weakref.WeakKeyDictionary` keyed by `AgentSession`. The comment at `_todo.py:483-484` is explicit: "so per-session locks are evicted automatically when the session is GC'd rather than accumulating in long-running services".
- **`State` pending-buffer with `_DeleteSentinel`**: `_workflows/_state.py:30-127` treats deletes as values, not operations. This lets concurrent executors stage writes (including deletes) into the same pending buffer without races; `commit()` reconciles.
- **`register_state_type()` for cross-process round-trips**: `_sessions.py:69-93` lets callers register a type with its `__name__.lower()` (or `_get_type_identifier`) so that `to_dict()`/`from_dict()` round-trips work even on a cold-start process that has not yet serialized the type. Pydantic models are auto-registered on first serialization (`_sessions.py:107-110`).
- **Schema-version-aware deserialization**: `DurableAgentState.from_dict` (`_durable_agent_state.py:419-434`) reads `schemaVersion` first and resets cleanly on mismatch. This is the durable storage equivalent of a "migration forward, never read backwards" guarantee.

## Tradeoffs

- **`AgentSession.state` is `dict[str, Any]`** (`_sessions.py:772`) — easy to extend, hard to validate. A bug in one provider can corrupt another's namespace; there's no schema enforcement at session-creation time. The mitigation is the `_STATE_TYPE_REGISTRY` + `SerializationMixin` round-trip, but type errors surface at read time, not at write time.
- **Ephemeral runtime state is lost on process restart**, deliberately. `BackgroundAgentsProvider._runtime` (`_background_agents.py:288-292`) holds live `asyncio.Task` handles — there is no protocol to re-establish them after a restart. The framework marks orphans `LOST` via `_refresh_task_state` (`_background_agents.py:207-232`) rather than recovering them. Documented as "inherently non-serializable" (`_background_agents.py:288-291`).
- **Hybrid JSON+pickle checkpoints require trusted storage**: `_checkpoint_encoding.py:18-45` is explicit that the `_RestrictedUnpickler` is "not a security boundary". The `getattr` builtin must be allowlisted for legitimate enum/named-tuple reconstruction; this means an attacker who can write to a checkpoint file can likely achieve code execution. The mitigation is "trust your storage backend", which is documented but easy to overlook.
- **`FileHistoryProvider` stores plaintext JSONL**: explicitly documented as "treat storage_path as trusted application storage, not as a secret store" (`_sessions.py:902-911`). Encoded fallback filenames and resolved-path validation help prevent path traversal but do not encrypt file contents.
- **Service-managed conversations (`service_session_id`)** out-source the source-of-truth to the chat provider (`_agents.py:1406-1408`). If the provider's `conversation_id` round-trip breaks, local state becomes stale silently; `_sessions.py:682-688` logs a WARNING in `PerServiceCallHistoryPersistingMiddleware` when the chat client returns no `conversation_id` despite storing history server-side.
- **`CompactionProvider.after_strategy` mutates the history provider's stored messages directly** (`_compaction.py:1287-1304`). Two providers sharing the same `history_source_id` would interfere; the design assumes a single owner.
- **`State.commit()` happens before checkpoint creation, but per-executor state is staged via `_set_executor_state` -> `state.set` and committed at superstep end** (`_runner.py:218-225, 302-379`). A crash between executor completion and `commit()` discards the executor's last writes but keeps the message-log checkpoint — recoverable but lossy at the boundary.

## Failure Modes / Edge Cases

- **`DurableAgentState.from_dict` on missing `schemaVersion`** resets state silently with a WARNING (`_durable_agent_state.py:426-430`). All history is lost.
- **Cross-process locking is best-effort**: `FileHistoryProvider` uses process-local `threading.Lock` stripes (`_sessions.py:916-920, 1113-1115`); two processes writing to the same `storage_path` race at the OS level — the framework documents this as "no cross-process / cross-host locking" (`_sessions.py:909`).
- **`State.delete()` on an absent key raises `KeyError`** (`_workflows/_state.py:68-77`). Executor code calling `delete()` after a previous superstep committed-then-discarded the key will crash the superstep rather than silently no-op.
- **`RedisHistoryProvider.max_messages`** triggers `LTRIM` after `RPUSH` (`_history_provider.py:162-165`). Pipeline is `transaction=True`, but the count check (`llen`) is a separate round-trip after the pipeline — there's a window where the count exceeds the cap by exactly the batch size until `LTRIM` lands. Acceptable for a max-messages soft-cap.
- **`MemoryContextProvider.state.clear()` each `before_run`** (`_memory.py:1170-1171`) wipes the provider-scoped state and re-exports the routing owner — any caller-supplied non-routing data on the provider's state is lost on every turn. Documented in the comment but easy to overlook.
- **`ToolApprovalState.queued_approval_requests` survives between turns** but is drained by `_drain_auto_approvable_queue` (`_tool_approval.py:379-381`); a crash mid-drain leaves the queue populated, which is the correct behavior but means callers must guard against replaying already-approved requests.
- **`FileCheckpointStorage._validate_file_path`** uses `Path.is_relative_to` for path-traversal defense (`_checkpoint.py:282-300`); this requires Python 3.9+. Older Python versions would be vulnerable to `../` escapes in `checkpoint_id`.
- **Schema mismatch in Redis context**: `RedisContextProvider._validate_schema_compatibility` (`_context_provider.py:249-298`) compares only "significant" parts of the schema, so trivial differences (e.g., tag separator) do not force a rebuild.
- **`AgentLoopMiddleware.fresh_context=True`** snapshots and restores `session.state` between iterations (`_loop.py:420-580`); restoring a session from dict loses any non-serializable runtime that was attached (in-process locks, asyncio handles). This is intentional but can surprise callers.
- **`BackgroundAgentsProvider` orphans**: when an in-flight `asyncio.Task` disappears (e.g., process restart), `_refresh_task_state` marks it `LOST` (`_background_agents.py:217-222`). The agent cannot recover the work — it must restart the task.

## Future Considerations

- **Schema evolution**: `DurableAgentState.SCHEMA_VERSION = "1.1.0"` (`_durable_agent_state.py:395`). When 1.2.x lands, `from_dict` should follow a migration ladder rather than the current "reset on mismatch" (`_durable_agent_state.py:426-430`).
- **Compaction-vs-history interaction**: `CompactionProvider.after_strategy` rewrites stored history in place (`_compaction.py:1287-1304`). If compaction is enabled together with `FileHistoryProvider`'s JSONL append-only model, the two can race. A future implementation could move compaction into a separate `after_strategy` over a copy-then-swap.
- **Cross-process session recovery**: today, `AgentSession.state` is owned by the holding process. A future `SessionStore` protocol mirroring `HistoryProvider`'s shape would let `AgentSession.state` itself cross processes (already partially addressed by `MemoryFileStore.export_provider_state` / `import_provider_state` at `_memory.py:712-729`).
- **Pickle-free checkpoints**: `_checkpoint_encoding.py:148-209` relies on pickle for non-JSON-native values. A `dataclasses-json` / Pydantic-native path would let checkpoint storage be safely exposed to less-trusted environments.
- **GroupChat / Magentic checkpointing**: `GroupChatState` (`_group_chat.py:71-93`) is a frozen dataclass passed per-round but not yet wired into the durable activity bridge. `MagenticContext` (`_magentic.py:336-394`) is `DictConvertible` but persistence is not exposed through a `WorkflowCheckpoint` payload.

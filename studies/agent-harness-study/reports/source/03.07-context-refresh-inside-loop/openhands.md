## Summary

OpenHands in the selected source is a **control-plane harness**, not the agent loop itself. Per the in-tree comments (`openhands/app_server/status/system_stats.py:1-7`, `openhands/app_server/user_auth/default_user_auth.py:4`, `openhands/app_server/user_auth/user_auth.py:4`), the V1 agentic core lives in the external `openhands-sdk` package (`software-agent-sdk#…`) and runs in a remote **agent-server** hosted inside a sandbox. This codebase compiles an `Agent` (with `AgentContext`, `Condenser`, `LLM`, hooks, skills, plugins) through `AgentSettings.create_agent()` and posts it to the agent-server (`open code in live_status_app_conversation_service.py:1641`, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1729` and `openhands/app_server/app_conversation/live_status_app_conversation_service.py:436` for the POST). The actual per-turn context assembly, token counting, condenser invocation, and event compaction happen **inside the agent-server / SDK** and are not observable here except through outbound HTTP and inbound webhook events.

What is observable in the harness for this dimension:

- **Context assembly**: `AgentContext(system_message_suffix=..., secrets=...)` is constructed once at conversation start (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1635`) and skills are merged via `model_copy` into `agent.agent_context.skills` (`openhands/app_server/app_conversation/app_conversation_service_base.py:161-187`, `openhands/app_server/app_conversation/app_conversation_service_base.py:243`). There is no per-turn rebuild path in this harness — the SDK owns that.
- **Token counts / metrics**: stored from SDK `ConversationStats.usage_to_metrics` (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:391-470`) on `ConversationStateUpdateEvent(key='stats')` (webhook at `openhands/app_server/event_callback/webhook_router.py:470`), persisted as `context_window`, `per_turn_token`, `prompt_tokens`, `completion_tokens`, cache and reasoning token columns (`openhands/app_server/app_lifespan/alembic/versions/001.py:172-180`).
- **Summarization / compaction**: the harness defines a `_create_condenser` factory that builds an `LLMSummarizingCondenser` from the SDK (`openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`), but the live code path actually relies on whatever `CondenserSettings` is already embedded in `user.agent_settings` — i.e. the SDK constructs the condenser via `AgentSettings.create_agent()` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1641`). The harness only attaches the LLM-tracing `usage_id` (`'condenser'` / `'planning_condenser'`) and `litellm_extra_body.metadata` to that condenser LLM after the fact (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1352-1371`).
- **Retrieval / repo maps**: this source has **no harness-side retrieval, semantic / vector indexing, or repo-map implementation**. Skills, hooks, and plugins are the only "context loading" mechanisms, all loaded once per `StartConversationRequest` (`openhands/app_server/app_conversation/skill_loader.py:265`, `openhands/app_server/app_conversation/hook_loader.py:109`). The chunk-localization helper at `openhands/app_server/utils/chunk_localizer.py:1-101` is a fuzzy LCS function used outside the loop (typically for diff/edit verification — see docstring at `openhands/app_server/utils/chunk_localizer.py:1-5`).
- **Overflow handlers**: no harness-level overflow/truncation logic; the system relies on the SDK condenser to keep the conversation under the model context window, with no per-turn guards or fallback refusal.
- **Trace of context changes**: token usage + condenser metrics arrive in the harness as separate `MetricsSnapshot` keyed under `usage_id` (`'agent'`, `'condenser'`, `'planning_condenser'`, see `openhands/app_server/utils/llm_metadata.py:40` and `tests/unit/app_server/test_webhook_router_stats.py:115`, `tests/unit/app_server/test_webhook_router_stats.py:237-241`), so per-role LLM spend is observable downstream but **which events were dropped or summarized is not surfaced** by the harness.

The central answer to the dimension's driving question — "Can the loop keep working after the conversation exceeds the context window?" — cannot be confirmed from this source. The SDK is the only place that owns compaction, and it is not in scope.

## Rating

**3 / 10** — Absent in this source.

Rationale: the harness has a partial, two-tier surface for context management: a settings-side representation (`CondenserSettings`, `openhands.sdk.settings.model.CondenserSettings`, see `tests/unit/storage/data_models/test_settings.py:19`), a well-tested factory (`_create_condenser`, `tests/unit/app_server/test_app_conversation_service_base.py:297-439` for DEFAULT/PLAN × None/custom max_size), persisted downstream counters (`context_window`, `per_turn_token`, `openhands/app_server/app_lifespan/alembic/versions/001.py:179-180`, `openhands/app_server/app_conversation/sql_app_conversation_info_service.py:436-465`), and an LLM `usage_id`/metadata patch for the condenser (with explicit tests at `tests/unit/app_server/test_live_status_app_conversation_service.py:981-1009`). However:

- The factory is dead code — it is defined in `AppConversationServiceBase` (line `openhands/app_server/app_conversation/app_conversation_service_base.py:578`) but never invoked (verified by absence of callers in production code; the actual flag-bearing path is `user.agent_settings.condenser.*` → SDK).
- The control plane does not implement retrieval, repo maps, or any progressive disclosure of code context. The only relevant helper (`chunk_localizer`, `openhands/app_server/utils/chunk_localizer.py:1-101`) is a fuzzy-LCS match for verification rather than a context assembler.
- The harness has no overflow / truncation logic, no per-turn metrics inspection, no ability to "tell what context is missing" (the model has no signal that condensation occurred beyond whatever the SDK emits as events).
- Migration history shows the default for `enable_default_condenser` has been **flipped twice** in production (`enterprise/migrations/versions/020_set_condenser_to_false.py:24-34` and `enterprise/migrations/versions/024_update_enable_default_condenser_default.py:24-65`, also a bump in `enterprise/migrations/versions/087_bump_condenser_defaults.py:25-60` from 120 → 240). This suggests the system was unstable around condenser rollouts.

Where evidence exists (settings/test surface, persistence, usage_id tagging) the implementation is consistent and tested; but **none of the per-turn runtime decisions** (when to compact, what to drop, how to summarize, what retrieval to re-run) live in this source. A rating above 3 would require evidence of any in-harness per-turn rebuild, retrieval refresh, or overflow handler, which is not present. Rating 4-6 ("present but inconsistent") is not warranted because the *runtime* mechanism is wholly outside the source.

## Answers to Dimension Questions

1. **Is context static or rebuilt each turn?** Static for the harness's lifetime of a conversation. The harness builds one `AgentContext(system_message_suffix, secrets)` and merges `Skill` objects once at `StartConversationRequest` time. The actual per-turn rebuild happens in the SDK agent-server (out of source).
   - Evidence: `AgentContext(system_message_suffix=effective_suffix, secrets=secrets)` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1635-1638`); skills merged via `model_copy` once (`openhands/app_server/app_conversation/app_conversation_service_base.py:161-187`, called from `app_conversation_service_base.py:243`); the agent is constructed once via `configured_agent_settings.create_agent()` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1641`), then posted to the agent-server in one `model_dump` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:415-441`).
2. **What gets dropped first?** No clear evidence found in this source. The harness neither decides nor observes the policy. The SDK's `LLMSummarizingCondenser` defaults are referenced as `max_size=240, keep_first=2` (`openhands/app_server/app_conversation/app_conversation_service_base.py:594`), implying a "keep first N + summarize the rest" model, but the actual dropping order is implemented elsewhere.
3. **Is summarization faithful?** No evidence of faithfulness checks in this source. The harness creates the condenser with the user LLM and a different `usage_id` for tracing (`openhands/app_server/app_conversation/app_conversation_service_base.py:599-604`, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1352-1371`) and forwards `litellm_extra_body.metadata` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1358-1365`). No summary quality metric, hash comparison, or faithfulness test exists in source.
4. **Are tool results preserved or summarized?** No clear evidence found. The harness has no concept of "tool result preservation policy" and the SDK does not surface a specific event type to the app-server that distinguishes preserved-vs-summarized tool output.
5. **Can the model tell what context is missing?** No clear evidence found. The harness stores raw prompt/completion/cache/reasoning counts (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:431-445`) but does not expose to the model whether its last turn was condensed, how many events were summarized, or which tool results were dropped. Condenser events do not appear to be inspected beyond stats aggregation in `update_conversation_statistics` (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:390-470`).

## Architectural Decisions

- **Decouple the agentic core from the control plane.** The loop, condenser, prompt assembly, and event compaction live in the external `openhands-sdk` package (`openhands/app_server/status/system_stats.py:1-7`); the harness only sets parameters and records results. Implication: most "context refresh" evidence lives outside this study's source boundary.
- **Per-conversation `usage_id` taxonomy.** Every LLM call gets a `usage_id` (`'agent'`, `'condenser'`, `'planning_condenser'`; see `openhands/app_server/utils/llm_metadata.py:40`) so downstream metrics attribution can separate agent / condenser spend. Visible to the harness via `ConversationStats.usage_to_metrics` (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:400-405`).
- **Let the SDK own condenser configuration; harness only tags.** `_create_condenser` (`openhands/app_server/app_conversation/app_conversation_service_base.py:578`) is provided as a convenience but the production path goes through `user.agent_settings.condenser` → `AgentSettings.create_agent()` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1641`). The harness then post-applies `usage_id` + tracing metadata (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1352-1371`).
- **Persist tokens + costs *only when non-null*.** `update_conversation_statistics` silently retains prior values if new fields are missing (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:447-465`). This means a missing condenser report after a turn looks identical to "no condensation yet" — observability of *whether* condensation happened is implicit, not explicit.
- **Conversation settings live independently from agent settings.** `Settings.agent_settings.condenser.*` belongs to agent settings (`openhands/app_server/settings/settings_models.py:108`); max-iteration and security analyzer belong to `conversation_settings` (line `:109`). Switching LLM profiles deliberately preserves the condenser sub-tree (test at `tests/unit/storage/data_models/test_settings.py:314-346`).
- **Skills, hooks, and plugins are loaded once per `StartConversationRequest`** via `load_skills_onto_request` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1746-1773`) and `_load_hooks_from_workspace` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1671-1689`). There is **no per-turn reload**.

## Notable Patterns

- **Settings deep-merge via SDK diff applier.** `apply_agent_settings_diff` from `openhands.sdk.settings` is the canonical way to update agent settings inside the harness; condenser + verification + mcp_config all survive partial updates (`openhands/app_server/settings/settings_models.py:218`, test at `tests/unit/storage/data_models/test_settings.py:73-87`).
- **Webhooks as the only event mirror.** The agent-server streams events back via the app-server webhook router; `ConversationStateUpdateEvent(key='stats')` triggers `process_stats_event` which writes token usage into the ORM (`openhands/app_server/event_callback/webhook_router.py:470-473`).
- **Saas vs OSS settings divergence via migrations.** `enterprise/migrations/versions/108_add_agent_settings_to_enterprise_settings.py:140-202` translates `agent_settings.condenser.{enabled,max_size}` back into the legacy `enable_default_condenser` + `condenser_max_size` columns so OSS-locked users with a `CondenserSettings` round-trip into the SaaS schema.
- **Skill dedup by name, last-wins** (`openhands/app_server/app_conversation/app_conversation_service_base.py:189-206`).
- **Permissive model_copy pattern when overriding fields.** Agent/condenser/LLM overrides are applied via `agent.model_copy(update={...})` instead of mutating in place (`openhands/app_server/app_conversation/app_conversation_service_base.py:175-181`, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1352-1371`, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:1657-1664`).

## Tradeoffs

- **Outsourcing loop control to a closed-core SDK makes the harness simple but unobservable.** Operators cannot audit which events were summarized without tearing open the SDK. They can only react to aggregated `usage_to_metrics` totals and per-`usage_id` cost (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:399-466`).
- **Dead `_create_condenser` factory.** Preserved but uncalled, increasing the surface that reviewers must read without adding production behavior (`openhands/app_server/app_conversation/app_conversation_service_base.py:578-612`); the test suite at `tests/unit/app_server/test_app_conversation_service_base.py:297-439` is therefore testing implementation that is not on the critical path.
- **`update only if non-null` persistence policy** (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:447-465`). Cheap to write, but hides partial SDK errors as zeros from the previous turn; no diff signal is emitted to clients.
- **No per-turn refresh of skills/hooks/plugins.** Long-running tasks cannot pick up new microagents that appear later in the workspace. The cost-saving tradeoff is that context size is bounded by the initial snapshot.
- **No detection of "context overflow" at the harness level.** The harness never inspects the LLM's last `prompt_tokens / context_window` ratio; if the SDK's condenser fails to keep up, the failure surfaces only at the agent-server, after potentially multiple failed model calls.
- **Flat storage of LLM metrics by `usage_id`** (`tests/unit/app_server/test_webhook_router_stats.py:227-241`). Enables dashboards but lacks event-level attribution; impossible to trace a specific tool-call observation to a specific compaction decision.

## Failure Modes / Edge Cases

- **Condenser denylist storm from SaaS migrations.** Three migrations flip `enable_default_condenser` default in production: `020_set_condenser_to_false.py` (False), `024_update_enable_default_condenser_default.py` (True), `087_bump_condenser_defaults.py` (max_size 120→240). Each migration writes `UPDATE ... SET condenser_max_size = 240 WHERE condenser_max_size IN (...)` for every user — DB-side noise plus breakage risk for users whose billing/UX depends on a known condenser state.
- **Mismatch between local SDK pin and remote agent-server.** Detected lazily on `agent-server 500` with an actionable error suggesting users rebuild the sandbox image (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:448-457`). A version drift where the condenser protocol changes is a real failure mode.
- **Custom sandbox image with old SDK pinning condenser defaults** — there is no proactive gate; only the first 500 reaction.
- **Lossy conversation export.** Export capped at `export_max_events=10000` by default (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:223`); events beyond that are not exported, so external reconstructions of past context windows will be incomplete. This doesn't affect the loop, but downstream "what was condensed" analytics cannot rely on the export stream.
- **Secrets leak in the agent-server through ACP secret serialization** (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1815-1821`). Documents a redaction-induced auth failure for ACP runs, not a context-refresh issue, but it confirms the harness has non-trivial failure modes from serializer/secret patterns.
- **`SendLLMObservation` mid-run** updates `app_conversation_info.llm_model` (`openhands/app_server/event_callback/webhook_router.py:482-496`). The condenser LLM, however, is *not* recomputed — so a runtime LLM swap can leave the condenser running on an old model with the new agent's tracing metadata attached.

## Future Considerations

- The harness side would benefit from surfacing a per-turn context-budget signal to the agent (e.g., a system message delta or a tool observation) so the model can reason about its own context. Today there is no such hook (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1635-1638` constructs a single `system_message_suffix` for the entire run).
- The `_create_condenser` factory is a hint that the SDK was once considered "build a condenser here"; replacing it with a producer/consumer pattern would let the harness swap policies without shipping SDK changes.
- `chunk_localizer` (`openhands/app_server/utils/chunk_localizer.py:1-101`) is a verification helper, but the same LCS logic could back a "show what is in the current context" UI panel.
- Token-budget sanity checks (prompt_tokens > 0.8 * context_window → emit a `ConversationStateUpdateEvent(key='context_pressure')`) would close the observability loop and let the frontend warn the user.

## Questions / Gaps

- **What is the threshold and policy of `LLMSummarizingCondenser`?** No evidence in source beyond the documented SDK defaults `max_size=240, keep_first=2` (`openhands/app_server/app_conversation/app_conversation_service_base.py:594`); the actual trigger (per-token, per-event-count, sliding, rolling) is SDK-internal.
- **What does the agent-server emit when condensation runs?** No event kind for `condensation` was found in this source; only `ConversationStateUpdateEvent(key='stats')` is handled (`openhands/app_server/event_callback/webhook_router.py:470`). If the SDK emits a typed "Condenser" or "Summary" event, the harness has no handler in source.
- **Is `context_window` measured by the SDK's tokenizer or by the provider?** `TokenUsage.context_window` is opaque in source; testing shows it defaults to 0 if missing (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:535`, `tests/unit/app_server/test_webhook_router_stats.py:87-89`).
- **Does `user.agent_settings.condenser` accept anything besides LLMSummarizingCondenser?** The harness imports only `LLMSummarizingCondenser` from `openhands.sdk` (`openhands/app_server/app_conversation/app_conversation_service_base.py:34`). The discriminated union in SDK is not observable; this source assumes a single strategy.
- **Do ACP agents get the same condenser handling?** `agent_kind='acp'` branch builds the agent through `ACPAgentSettings.create_agent()` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1891`), but `_apply_server_agent_overrides` (which attaches `usage_id='condenser'` / `'planning_condenser'` to the condenser LLM) is applied only to the non-ACP path — gap.

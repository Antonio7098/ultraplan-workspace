## Summary

The Microsoft Agent Framework treats context as an **explicit, incrementally-annotated message log**, not an opaque conversation buffer. The `python/packages/core/agent_framework/_compaction.py:1-1456` module is the centerpiece: it defines a `TokenizerProtocol` (`_compaction.py:42`), a `CompactionStrategy` protocol (`_compaction.py:51`), a stable message-grouping system (system / user / assistant_text / tool_call groups, `_compaction.py:24-243`), and a rich taxonomy of strategies (`TruncationStrategy`, `SlidingWindowStrategy`, `SelectiveToolCallCompactionStrategy`, `ToolResultCompactionStrategy`, `SummarizationStrategy`, `TokenBudgetComposedStrategy`, `ContextWindowCompactionStrategy`) plus a default `CharacterEstimatorTokenizer` (`_compaction.py:66`).

Context assembly is per-turn through `SessionContext.get_messages()` (`_sessions.py:315`) and a chain of `ContextProvider.before_run` hooks (`_sessions.py:370`). The `CompactionProvider` (`_compaction.py:1182`) sits between the history provider and the model: it runs a `before_strategy` on loaded context and an `after_strategy` on the just-persisted history. Token-aware strategies compose into a two-phase pipeline (`ContextWindowCompactionStrategy`, `_compaction.py:1307`) that first collapses older tool-call groups into summaries at 50% of the input budget and then truncates oldest non-system groups at 80% of the input budget. Summarization is LLM-driven (`SummarizationStrategy.__call__`, `_compaction.py:995`) and writes **bidirectional trace links** (`_summary_of_message_ids` / `_summarized_by_summary_id`, `_compaction.py:33-35`) so an excluded message carries the id of the summary that replaced it and vice versa. Retrieval (RAG) and "repo-map"-style progressive disclosure are handled by sibling `ContextProvider`s (e.g. `AzureAISearchContextProvider.before_run` in `azure-ai-search/_context_provider.py:580`, `MemoryContextProvider.before_run` in `_harness/_memory.py:1161`, `SkillsProvider.before_run` in `_skills.py:2058`, `FileMemoryProvider.before_run` in `_harness/_file_memory.py:237`).

The function-calling loop also does incremental context work: `_prepare_messages_for_model_call` (`_clients.py:363`) re-runs compaction before **every** chat-client call inside `FunctionInvocationLayer.get_response` (`_tools.py:2388`), so each tool-call iteration sees a freshly compacted message list. Per-call history persistence (`PerServiceCallHistoryPersistingMiddleware`, `_sessions.py:570`) makes the loop re-load + re-save history around each model call, which in turn means the `after_run` `CompactionProvider` strategy can run between iterations, not only at the end of an `Agent.run`.

The main weaknesses: **no OpenTelemetry instrumentation for compaction** on the Python side (the .NET sibling `CompactionTelemetry.cs:1-50` does emit `compaction.before.tokens`, `compaction.after.tokens`, `compaction.strategy`, etc., but no parallel exists in `observability.py`), no automatic reactive handler when the model rejects the prompt with `context_length_exceeded` (overflow is purely preventive via the pre-call compaction), and a default tokenizer that is a 4-chars-per-token heuristic (`_compaction.py:66-70`). The framework is otherwise mature, well-tested (50+ dedicated tests in `tests/core/test_compaction.py`), and ships pre-wired in the harness factory (`create_harness_agent`, `_harness/_agent.py:239`).

## Rating

**8 / 10** — Clear model with tests, explicit interfaces, and operational safeguards.

The framework has an unusually complete and well-decomposed design for context refresh: a token-aware protocol, six first-class strategies with composable two-phase pipelines, atomic message groups, traceable exclusions, and per-turn re-compaction across both the agent loop and the function-calling loop. Tests cover group annotation, incremental annotation, every strategy, idempotency, summarization-failure handling, and the `CompactionProvider` hooks. Deductions: (1) the Python side lacks the explicit `compaction.*` telemetry tags that the .NET side has — operators cannot directly answer "what context is the model seeing?" from OTel; (2) the default tokenizer is a 4-char heuristic, and only the model-call surface knows about token counts (the model itself is never queried for actual usage); (3) there is no reactive overflow handler (no automatic re-compaction after a `context_length_exceeded` exception); (4) summarization faithfulness relies on a single LLM call with no verification step and no metadata-preserving guarantees beyond bidirectionally linked ids.

## Answers to Dimension Questions

1. **Is context static or rebuilt each turn?**
   Rebuilt per turn and per *service call*. `SessionContext` (`_sessions.py:154-348`) is constructed fresh for every `Agent.run`; `SessionContext.get_messages()` (`_sessions.py:315`) flattens `context_messages` (keyed by `source_id` from each `ContextProvider`), then optionally appends `input_messages` and `response.messages`. With `require_per_service_call_history_persistence=True` (the harness default), the function-calling loop builds **another** per-call `SessionContext` inside `PerServiceCallHistoryPersistingMiddleware._prepare_service_call_context` (`_sessions.py:605-627`) so each chat-client call sees fresh history. Compaction then re-projects on every call via `_prepare_messages_for_model_call` (`_clients.py:363-391`).

2. **What gets dropped first?**
   Oldest non-system groups, by default. `TruncationStrategy` (`_compaction.py:691-718`) walks ordered group ids and excludes whole groups until included token/count ≤ `compact_to`. `SlidingWindowStrategy` (`_compaction.py:746-764`) drops everything but the last `keep_last_groups` non-system groups. `SelectiveToolCallCompactionStrategy` (`_compaction.py:793-817`) drops the oldest tool-call groups but leaves user/assistant text untouched. `TokenBudgetComposedStrategy` (`_compaction.py:1122-1160`) chains the configured strategies then falls back to oldest-first non-system exclusion, and finally to system-only ("strict") exclusion as a last resort (`_compaction.py:1152-1160`). `ContextWindowCompactionStrategy` (`_compaction.py:1307-1419`) escalates from collapsing older tool groups (50% threshold) to full truncation of older non-system groups (80% threshold). System groups are treated as anchors and only excluded under the strict fallback. Tool-call groups are kept atomic — they cannot be partially trimmed (`_compaction.py:172-216`).

3. **Is summarization faithful?**
   No verification step, but traceability is bidirectional. `SummarizationStrategy` (`_compaction.py:949-1088`) makes a single LLM call to produce a ≤5-sentence summary (prompt at `_compaction.py:931-946`), stores `_summary_of_message_ids` and `_summary_of_group_ids` on the summary message (`_compaction.py:1067-1079`), and writes `_summarized_by_summary_id` on every original message (`_compaction.py:1081-1083`). On summarizer failure the strategy returns `False` and logs a warning instead of mutating the list (`_compaction.py:1053-1058`); on empty summary the same (`_compaction.py:1061-1063`). Tests assert both the bidirectional links (`tests/core/test_compaction.py:393-421`) and the failure paths (`tests/core/test_compaction.py:424-465`). What it does **not** guarantee: factual fidelity (no fact extraction / cross-check) and consistency across re-summarizations. There is no metric for summarization drift, and the summary is always treated as authoritative for the next `get_messages` projection.

4. **Are tool results preserved or summarized?**
   Both, depending on the strategy. `SelectiveToolCallCompactionStrategy` (`_compaction.py:767-817`) drops older tool-call groups entirely — only the function call/result payloads vanish. `ToolResultCompactionStrategy` (`_compaction.py:820-918`) preserves a readable trace: for each collapsed group it builds a `summary_label = "func_name: result_text; func_name2: result_text2"` and emits an assistant message `[Tool results: <summary_label>]` (`_compaction.py:880-912`). The newest `keep_last_tool_call_groups` (default 1 in `SelectiveToolCallCompactionStrategy`, default 1 in `ToolResultCompactionStrategy`, default 4 in `ContextWindowCompactionStrategy`) remain verbatim so the model still has the structured `function_call` / `function_result` content for the most recent turn. Tool messages without matching function calls (orphan tool results) are absorbed into the same `tool_call` group in `group_messages` (`_compaction.py:218-231`).

5. **Can the model tell what context is missing?**
   Implicitly, yes. Excluded messages carry `_excluded=True` plus an `_exclude_reason` such as `"truncation"`, `"sliding_window"`, `"tool_call_compaction"`, `"tool_result_compaction"`, `"summarized"`, or `"token_budget_fallback[/_strict]"` (`_compaction.py:582-588, 717, 763, 816, 897, 1083, 1146, 1157`). The model itself is not directly told "you are seeing a compacted transcript", but it sees the summary message that replaced the originals (role `assistant`, text starts with `[Tool results: ...]` for tool compaction or free-form for summarization). Excluded originals remain in storage and re-appear whenever `InMemoryHistoryProvider`/`FileHistoryProvider` is loaded with `skip_excluded=False` (the default), so a second consumer without compaction sees the full log. The `FileMemoryProvider` also rebuilds and exposes a `memories.md` index capped at 50 entries before every run (`_harness/_file_memory.py:217-235`), and the `MemoryContextProvider` exposes `list_memory_topics` / `read_memory_topic` / `search_memory_transcripts` tools (`_harness/_memory.py:1196-1268`) so the model can pull deeper on demand when it suspects a missing memory. The skills system similarly advertises names only and provides `load_skill` / `read_skill_resource` (`_skills.py:2058-2094`). There is, however, no canonical "context is missing" surface message — the model has to infer it from the inserted summary text.

## Architectural Decisions

- **In-place annotation on shared `Message` objects.** `_set_group_*`/`_write_*` helpers mutate `Message.additional_properties` directly (`_compaction.py:288-332`). `_prepare_messages_for_model_call` deliberately operates on the caller's list when it's already a `list` to keep inserted summary messages alive across the function-invocation loop (`_clients.py:380-391`, with an explicit comment about issue #4991 — a previous version silently dropped summaries because it compacted a throwaway copy while keeping exclusions on the shared objects).
- **Atomic message groups.** A `tool_call` group is the smallest evictable unit; reasoning + assistant function_call + tool messages co-located in one assistant message are absorbed into the same group (`_compaction.py:172-216`). The framework never splits a partial tool-call group, so the model never sees a function_call without its result or vice versa.
- **Two-phase compaction at the context-window layer.** `ContextWindowCompactionStrategy` (`_compaction.py:1307-1419`) runs `ToolResultCompactionStrategy` first (cheap, reversible in the sense that summary text stays readable) and `TruncationStrategy` second (destructive). Each phase has its own `TokenBudgetComposedStrategy` with its own threshold so they fire independently. The defaults (50% / 80% of the input budget) give the model a "soft" signal that summarization has happened before any message is outright dropped.
- **Separation of trigger and strategy.** Strategies expose a single `async __call__(messages) -> bool` (`_compaction.py:51-63`); the trigger is whatever condition the strategy checks internally (count threshold, token threshold). The .NET side makes this even more explicit with a `CompactionTrigger` abstraction (`dotnet/src/Microsoft.Agents.AI/Compaction/CompactionTrigger.cs`), which Python doesn't yet have.
- **`before_strategy` vs `after_strategy` split.** `CompactionProvider` (`_compaction.py:1182-1305`) explicitly separates the strategy that compacts **loaded** context (so the next model call sees a smaller input) from the strategy that compacts **persisted** history (so the next `before_run` starts from a smaller base). Each phase may independently be `None`. This is the closest the framework has to a re-entrant refresh: between iterations, `after_run` shrinks storage, then the next iteration's `before_run` shrinks the input.
- **Skip-excluded opt-in.** `InMemoryHistoryProvider` and `FileHistoryProvider` accept `skip_excluded: bool = False` (`_sessions.py:839, 956`); `CompactionProvider.after_run` deliberately **does not** delete excluded messages from storage (`_compaction.py:1302-1304`) — it relies on `skip_excluded=True` to filter on the next load. This keeps compaction reversible from a different reader's perspective.
- **Per-service-call history persistence.** `PerServiceCallHistoryPersistingMiddleware` (`_sessions.py:570-743`) is what makes the function-calling loop aware of history. When `service_stores_history=False` (local), it loads + persists around every chat call and uses `LOCAL_HISTORY_CONVERSATION_ID = "agent_framework_local_history_persistence"` (`_sessions.py:537-542`) as a sentinel so the function loop drives continuation without forwarding a fake id to the leaf client.
- **Progressive disclosure for retrieval.** The framework does not pull "the entire repo" into context. Instead each provider exposes a small always-on signal plus tools for deeper pulls: `SkillsProvider` advertises ~100 tokens per skill and provides `load_skill` / `read_skill_resource` / `run_skill_script` (`_skills.py:2058-2094`); `MemoryContextProvider` injects an index and auto-loaded topic files and exposes `read_memory_topic` / `search_memory_transcripts` (`_harness/_memory.py:1161-1325`); `FileMemoryProvider` injects a 50-entry `memories.md` index and exposes `file_memory_search_files` (`_harness/_file_memory.py:217-397`); `AzureAISearchContextProvider` runs semantic/agentic search and injects only `top_k` hits per turn (`azure-ai-search/_context_provider.py:580-725`). This is the "repo map" analogue — a small always-on pointer set, not the full corpus.
- **Default factory wires it all together.** `create_harness_agent` (`_harness/_agent.py:239-525`) is the canonical configuration: history provider first, then compaction provider with `ContextWindowCompactionStrategy` defaults when token budgets are supplied (`_harness/_agent.py:72-120`), then todo/mode/memory/access/skills/background, then `require_per_service_call_history_persistence=True` (`_harness/_agent.py:519`). Off-the-shelf context refresh.

## Notable Patterns

- **Protocols over base classes for the contracts.** `TokenizerProtocol` and `CompactionStrategy` are `@runtime_checkable` `Protocol`s (`_compaction.py:41-63`), so user code can plug in any callable/duck-typed object that satisfies the interface — no inheritance required.
- **Token annotation is opt-in and lazy.** `_first_annotation_gaps` (`_compaction.py:392-410`) and `_first_untokenized_index` (`_compaction.py:385-389`) compute the smallest index from which annotation must restart, and `annotate_message_groups` reuses prefix annotations (`_compaction.py:428-490`). Token counts are only computed when a tokenizer is supplied (`_compaction.py:488-489`), so message-count strategies cost nothing.
- **Trace metadata in `_attribution` and `_group`.** Every context message gets `additional_properties["_attribution"] = {"source_id": ..., "source_type": ...}` from `SessionContext.extend_messages` (`_sessions.py:220-251`); compacted groups get `_group` annotations including `id`, `kind`, `index`, `has_reasoning`, and optional `token_count` (`_compaction.py:296-332`). The result is a fully attributable message log.
- **In-place mutation + projection, never reordering-without-trace.** Compaction mutates the shared list (adds summary messages, flips `_excluded` flags) but does not silently drop content. `project_included_messages` (`_compaction.py:600-601`) is the single point that produces the model-facing view, so a different consumer can always see the full annotated log.
- **Bidirectional exclusion-summary links.** `_summary_of_message_ids` ↔ `_summarized_by_summary_id` (`_compaction.py:33-35, 288-293, 1067-1083`) let downstream tooling reconstruct what was dropped and why.
- **Pre-call tokenized hint, post-call untracked.** `_inner_get_response` and the providers don't write token-usage telemetry into the message log itself; the only token counts that flow into the next compaction cycle are the model's own `response.usage_details` which are not used by `_prepare_messages_for_model_call` (it relies solely on the local tokenizer).

## Tradeoffs

- **Heuristic tokenizer by default.** `CharacterEstimatorTokenizer` (`_compaction.py:66-70`) is 4-chars/token. For non-Latin text, code-heavy prompts, or anything with structured payloads (function args, JSON), it can drift significantly. The protocol allows a real `TokenizerProtocol` implementation, but nothing in `BaseChatClient` consumes `response.usage_details` to refine the annotation — once a strategy decides based on the heuristic, the model's actual token usage does not feed back.
- **No reactive overflow handler.** Compaction is purely preventive. If the model's actual input exceeds the model's stated window, the framework does not catch a `context_length_exceeded` error and retry with a smaller input — that's left to the leaf provider client. `_prepare_messages_for_model_call` is only invoked *before* the call (`_clients.py:363-391`).
- **Two-tier annotation cost.** Token annotation requires serializing each message to JSON via `_serialize_message` (`_compaction.py:502-509`); for large histories this is non-trivial. The implementation avoids re-tokenizing fully-annotated prefixes, but on every compaction that crosses a token threshold the working set is re-tokenized (`_compaction.py:1123-1132`).
- **Compaction provider order matters.** `_assemble_context_providers` (`_harness/_agent.py:143-188`) places history before compaction so the compaction strategy sees loaded messages. A user that puts `CompactionProvider` first would get an empty target list. This is a correctness footgun that isn't enforced.
- **Per-call compaction mutates the caller's list.** This is intentional (issue #4991) but means `_prepare_messages_for_model_call` has a documented side effect on the input list (`_clients.py:380-391`). Code that holds onto a `messages` reference across `client.get_response` calls will see it modified.
- **`after_run` does not free storage by default.** `CompactionProvider.after_run` mutates `_excluded` flags but keeps the original messages in `state["messages"]` (`_compaction.py:1302-1304`). On very long sessions the session dict grows monotonically unless `skip_excluded=True` is set on the history provider.
- **Summarization prompt is hard-coded.** `DEFAULT_SUMMARIZATION_PROMPT` (`_compaction.py:931-946`) is shipped as-is unless the user replaces it; `summarizationPrompt` in `.NET SummarizationCompactionStrategy.cs:88` is identical in spirit. No domain-specific templates or guardrails.
- **Python ↔ .NET feature skew.** The .NET side has `CompactionTriggers`, `ChatReducerCompactionStrategy`, and full `CompactionTelemetry.cs` instrumentation; the Python side lacks both. This is a parity gap that surfaces as "the Python build cannot answer 'was compaction triggered this turn and by how much?' from OTel".

## Failure Modes / Edge Cases

- **Summarizer returns empty text** → strategy logs `"Skipping summarization compaction: summarizer returned no text."` and returns `False`; no exclusion flags are flipped (`_compaction.py:1060-1063`). Test at `tests/core/test_compaction.py:446-465`.
- **Summarizer raises** → same no-op path, warning `"summary generation failed"` (`_compaction.py:1053-1058`). Test at `tests/core/test_compaction.py:424-443`.
- **Token budget unreachable even with anchors dropped** → strict fallback excludes system groups with `reason="token_budget_fallback_strict"` (`_compaction.py:1152-1159`). The model is told the context was force-shrunk; recovery requires a smaller system prompt or larger budget.
- **Service-side storage returns no conversation_id** → `PerServiceCallHistoryPersistingMiddleware._finalize_response` logs `"cross-turn history may not resume"` (`_sessions.py:682-688`). Cross-turn compaction/loading silently becomes ineffective.
- **`store=False` mismatch with `service_stores_history=True`** → user warning at agent construction (`_agents.py:1228-1239`); provider is loaded into a service-managed conversation, mismatch is logged but not raised.
- **Tool approval request inside a tool-call batch** → handled by `ToolApprovalMiddleware` with a .NET-style bypass that hides approval requests for known non-approval-required tools and reinjects them only when the visible flow resumes (`packages/core/AGENTS.md` "Tool Approval Harness" section).
- **Compaction with no `before_strategy` and no `after_strategy`** → `CompactionProvider.before_run` and `after_run` return immediately (`_compaction.py:1258-1259, 1284-1285`); `create_harness_agent` skips building the provider entirely (`_harness/_agent.py:111-113`).
- **Compaction in the per-service-call loop** → enabled by `require_per_service_call_history_persistence=True` + per-call provider loading inside the middleware; otherwise compaction only runs once per `Agent.run` (`_sessions.py:570-743, 1221-1239`).
- **Re-annotation with stale prefix annotations** → `_reannotation_start` (`_compaction.py:413-425`) walks back to the start of the prior group so the suffix annotation continues seamlessly. `_first_annotation_gaps` (`_compaction.py:392-410`) catches any inconsistency (e.g. annotation present but tokens missing).

## Future Considerations

- **Add `CompactionTrigger` to Python**, mirroring the .NET abstraction. Decouples "when to compact" from "how to compact" and makes policies like "compact when usage > 0.8 × budget" first-class.
- **Promote `CompactionTelemetry` to Python.** Emit `compaction.before.tokens`, `compaction.after.tokens`, `compaction.strategy`, `compaction.triggered`, `compaction.before.messages`, `compaction.after.messages` on `CompactionProvider.before_run` / `after_run`. This closes the loop with operators who need to answer "why is the model seeing a compacted transcript this turn?".
- **Wire `response.usage_details` back into token annotation.** After a model call, refine `_token_count` for the user/assistant messages that were just sent, so the heuristic self-corrects over time. This requires per-model tokenizers (tiktoken for OpenAI, etc.) and is the only way to make `CharacterEstimatorTokenizer` stop drifting.
- **Reactive overflow handler.** Catch provider-specific `context_length_exceeded` exceptions inside `_inner_get_response`, run an emergency pass of `TruncationStrategy(max_n=budget, compact_to=budget/2)`, and retry once. The strategies already support this; only the wiring is missing.
- **Optional hard-delete in `after_run`.** Add a `purge_excluded: bool = False` to `CompactionProvider.after_run` for users who want storage to actually shrink on compaction, not just on `skip_excluded=True` filtering.
- **Stable summary IDs.** Today `summary_id = f"summary_{len(messages)}"` (`_compaction.py:1064`). A re-compaction that produces a fresh summary gets a new id and loses the link to the prior summary. A content-hash id would let downstream consumers detect "summary drift".
- **Summarization verification.** Pair `SummarizationStrategy` with a cheap verifier (second LLM call or rule-based check) that asserts the summary contains at least the ids of `_summary_of_message_ids`. No such check exists today.
- **Provider source attribution in compaction.** The `CompactionProvider.before_run` projects messages from all `context_messages` source_ids (`_compaction.py:1270-1273`). A user-defined "do not compact messages from `source_id='external_rag'`" filter would be useful when retrieval content must be retained for the duration of the run.
- **Streaming-friendly compaction.** Today `_inner_get_response` is the single integration point (`_clients.py:363-391`); for `stream=True` the compaction still runs once before the stream (`_clients.py:521-541`). Compaction that progressively shrinks during streaming is not supported.

## Questions / Gaps

- **Is there an integration test that exercises a `Agent.run` past the model context window?** Tests cover each strategy in isolation (`tests/core/test_compaction.py`) and per-call client compaction (`tests/core/test_clients.py`), but no clear end-to-end test in the selected source asserts "after N turns of a 128k-token budget, the loop still produces a coherent response". No evidence found in `tests/` or `samples/` of such a scenario.
- **What happens when the summarizer's chat client is the same as the agent's chat client?** `SummarizationStrategy.__call__` calls `self.client.get_response(...)` with no separate conversation id (`_compaction.py:1042-1052`); on OpenAI Assistants/Responses APIs this could pollute the agent's history. No evidence found of a dedicated `conversation_id=None` enforcement for the summarizer call.
- **Does `_resolve_compaction_overrides` (`_clients.py:393-408`) prefer agent-level or per-call overrides?** Per-call wins (`_clients.py:403-407`), but documentation about this precedence is thin.
- **Are there benchmarks for the cost of `apply_compaction` on real workloads?** No benchmark files found in `tests/`, `samples/`, or `python/packages/core/tests/`. Token annotation + summarization calls are both LLM-billable; the heuristic path is the only cheap one.
- **Is there a per-strategy retry/back-off?** No evidence found. `SummarizationStrategy` simply returns `False` on failure (`_compaction.py:1053-1058`) and lets the next `before_run` retry.
- **How does the framework surface to the user that compaction actually happened this turn?** Today the only signals are: (a) `_excluded` flags on stored messages, (b) inserted `[Tool results: ...]` / free-form summary messages visible in the model-facing list, (c) optional log warnings. There is no `AgentRunResult.compaction_summary` or similar field; no evidence found in `_types.py` of such a field.
- **Per-service-call compaction is gated on `require_per_service_call_history_persistence=True` + a non-service-stores chat client.** What happens when `require_per_service_call_history_persistence=False` and the user only wants compaction? The answer is "compaction runs once per `Agent.run` only". This is documented in `_harness/_agent.py:519` but not in the docstring of `Agent.__init__` or `CompactionProvider`.

## Summary

Letta's LLM turn loop is implemented as **three coexisting generations** (`LettaAgent` V1, `LettaAgentV2`, `LettaAgentV3`) that all share the same template: a `for i in range(max_steps)` outer driver, an inner `_step()` coroutine that performs exactly one LLM call plus tool execution, and a `StepProgression` state machine that brackets each turn (`START → STREAM_RECEIVED → RESPONSE_RECEIVED → STEP_LOGGED → LOGGED_TRACE → FINISHED`). The factory `AgentLoop.load()` (`letta/agents/agent_loop.py:15-63`) picks `LettaAgentV3` for the canonical `letta_v1_agent` agent type and falls back to `LettaAgentV2` for legacy types — so V3 is the reference implementation, but V1/V2 still ship and run.

Each turn is durably persisted as a `Step` ORM row (`letta/schemas/step.py:16-66`) plus a `StepMetrics` record (`letta/schemas/step_metrics.py:13-26`), the LLM call is wrapped in a `LettaLLMAdapter` strategy (`letta/adapters/letta_llm_adapter.py:16-131`) that selects blocking vs streaming vs SGLang-native transports, and per-step tokens are aggregated into a `LettaUsageStatistics` carried in `LettaResponse` (`letta/schemas/letta_response.py:52-88`) along with an optional multi-turn `TurnTokenData` list for RL training (`letta/schemas/letta_response.py:33-49`).

The loop is **generic over the runner pattern** but **specialized per agent type**: `BaseAgentV2` (`letta/agents/base_agent_v2.py:18-105`) abstracts `step / stream / build_request`; subclasses add voice (`letta/agents/voice_agent.py:115`), batch (`letta/agents/letta_agent_batch.py:622`), sleeptime groups (`letta/groups/sleeptime_multi_agent_v3.py:42`), and ephemeral summarizers (`letta/agents/ephemeral_summary_agent.py:47`). Multiple tool calls per turn are supported: V3 executes parallel tool calls via `asyncio.gather` (`letta/agents/letta_agent_v3.py:1857`), V2 only handles a single tool call (`letta/agents/letta_agent_v2.py:582-584` raises if none). Compaction (a separate sub-loop that may itself trigger LLM calls) is invoked both inline when context overflows (`letta/agents/letta_agent_v3.py:1218-1299`) and post-step (`letta/agents/letta_agent_v3.py:1438-1509`).

You can draw the loop from reading three files: `letta/agents/base_agent_v2.py` (interface), `letta/agents/letta_agent_v3.py:328-441` (blocking driver), and `letta/agents/letta_agent_v3.py:895-1592` (single-turn body). The cost of that clarity is that `_step()` is ~700 lines because compaction, system-prompt rebuilds, model routing, and fallback handling are all inlined rather than factored out.

## Rating

**8 / 10** — Clear model with explicit interfaces, durable persistence, observable telemetry, and operational safeguards (retries on context overflow, fallback model routing, parallel tool calls, credit checks). Deductions: (a) three loop generations co-exist and one of them (`LettaAgentV2`) still raises `LLMError` when the LLM returns no tool call — a restriction removed in V3; (b) `_step()` is a ~700-line god coroutine whose per-phase code is interleaved with compaction, system-prompt rebuilds, OTel events, and the `StepProgression` switch; (c) some loop invariants are implicit (e.g. "step_id is generated once per `_step` invocation but is also reused on approval-resume paths" — `letta/agents/letta_agent_v3.py:1019-1027`); (d) the `agent_step` OTel span is created at step start (`letta/agents/letta_agent_v2.py:944`) but the `tool_execution_started` / `tool_execution_completed` events are scattered between V2 and V3 helpers.

## Answers to Dimension Questions

**1. What happens during one turn?**
A turn = one `_step()` invocation. The phases are: (a) `step_id = generate_step_id()` (`letta/agents/helpers.py:373-375`); (b) `_step_checkpoint_start` creates a `Step` row with `status=PENDING` and an `agent_step` OTel span (`letta/agents/letta_agent_v2.py:941-966`); (c) `_refresh_messages` scrubs inner thoughts but skips system-prompt rebuild to preserve prompt-cache (`letta/agents/letta_agent_v2.py:760-792`); (d) build the request via `llm_client.build_request_data(messages, llm_config, tools, force_tool_call, ...)` and possibly patch `parallel_tool_calls` per provider rules (`letta/agents/letta_agent_v3.py:1099-1149`); (e) call `llm_adapter.invoke_llm(...)` and accumulate `tool_calls`, `reasoning_content`, `usage` on the adapter (`letta/adapters/letta_llm_request_adapter.py:48-98`); (f) handle the response with `_handle_ai_response`, which executes tools (serial or `asyncio.gather` for parallel), truncates client-side tool returns, and returns `(messages_to_persist, should_continue, stop_reason)` (`letta/agents/letta_agent_v3.py:1595-1964`); (g) checkpoint: `_step_checkpoint_finish` updates the Step row with usage + stop_reason, then `_checkpoint_messages` writes messages + updates `agent_state.message_ids` / `conversation_messages` (`letta/agents/letta_agent_v3.py:758-816`, `letta/agents/letta_agent_v2.py:988-1034`); (h) optionally post-step compaction (`letta/agents/letta_agent_v3.py:1438-1509`); (i) yield `LettaMessage` chunks to the streaming response, or accumulate them for the blocking response (`letta/agents/letta_agent_v3.py:1412-1436`).

**2. Does every turn call the model?**
Effectively yes, with two exceptions. The normal path always invokes the LLM at step (e) above. Exceptions: (a) when the input contains an `approval` response, `_step` short-circuits past the LLM call and reuses the existing `step_id` (`letta/agents/letta_agent_v3.py:973-1027`); (b) when a context-window overflow is raised by the LLM client, the LLM call IS made but its result is replaced by a compaction LLM call (`letta/agents/letta_agent_v3.py:1218-1299`). So the per-iteration outer-loop counter still increments on those turns, but the user-visible "model spoke" event may be absent.

**3. Can a turn include multiple tool calls?**
Yes in V3, no in V2. V3: `tool_calls = llm_adapter.tool_calls or [llm_adapter.tool_call]` then for `len(tool_calls) > 1` it either runs them in parallel via `asyncio.gather` (for tools with `enable_parallel_execution=True`) or sequentially (`letta/agents/letta_agent_v3.py:1328-1342`, `:1840-1862`). V2 raises `LLMError` if there is no tool call (`letta/agents/letta_agent_v2.py:582-584`) and uses only the first `tool_call` (`letta/agents/letta_agent_v2.py:594`). The decision is also gated by `active_llm_config.parallel_tool_calls` — when false, V3 truncates to the first tool call (`letta/agents/letta_agent_v3.py:1337-1342`).

**4. Is turn state persisted?**
Yes, with two layers of durability. Per-step: a `Step` ORM row is created at `_step_checkpoint_start` with `status=PENDING` (`letta/agents/letta_agent_v2.py:947-962`) and updated to `SUCCESS` / `FAILED` at `_step_checkpoint_finish` (`letta/agents/letta_agent_v2.py:1022-1033`, `letta/services/step_manager.py:419-476`); `StepMetrics` captures `step_start_ns`, `llm_request_start_ns`, `llm_request_ns`, `tool_execution_ns`, `step_ns` (`letta/schemas/step_metrics.py:13-26`); a webhook fires on success (`letta/services/step_manager.py:475`, `letta/services/webhook_service.py:16-55`). Per-message: all new messages go through `_checkpoint_messages` (`letta/agents/letta_agent_v3.py:758-816`), which (i) stamps `step_id`, `run_id`, `conversation_id`, (ii) calls `message_manager.create_many_messages_async(...)`, (iii) updates `agent_state.message_ids` or `conversation_messages`. On exception the persistence is intentionally rolled back — "message persistence does not happen in the case of an exception (rollback to previous state)" (`letta/agents/letta_agent_v3.py:1513`); only the Step row remains with an error payload via `update_step_error_async`.

**5. Is the loop generic or agent-specific?**
Both. The shape is generic: `BaseAgentV2` (`letta/agents/base_agent_v2.py:18-105`) declares three abstract methods — `step`, `stream`, `build_request` — and `AgentLoop.load()` (`letta/agents/agent_loop.py:15-63`) chooses the concrete class by agent type. Specializations: `LettaAgentV3` (canonical for `letta_v1_agent`, supports parallel tool calls, client-side tools, conversation mode), `LettaAgentV2` (legacy, single tool call, no client tools), `LettaAgent` V1 (in `letta/agents/letta_agent.py`), `LettaAgentBatch` (batch SDK), `VoiceAgent` (`letta/agents/voice_agent.py:115`), `VoiceSleeptimeAgent` (`letta/agents/voice_sleeptime_agent.py:70`), `EphemeralSummaryAgent` (`letta/agents/ephemeral_summary_agent.py:47`), `EphemeralAgent` (`letta/agents/ephemeral_agent.py:39`), and the `SleeptimeMultiAgent` family (`letta/groups/sleeptime_multi_agent_v3.py:42`, `:v4.py:44`). The body of `_step()` is duplicated between V2 and V3 with V3 as the canonical path; V1 has its own inline implementation.

## Architectural Decisions

- **Factory selection, not polymorphism on agent_state**: `AgentLoop.load()` (`letta/agents/agent_loop.py:15-63`) inspects `agent_state.agent_type` + `enable_sleeptime` + `multi_agent_group` to return the concrete class. This makes the loop variant a static dispatch decision per request rather than a runtime strategy.
- **Step ID as the durable turn key**: `generate_step_id()` (`letta/agents/helpers.py:373-375`) produces `step-<uuid4>` once per `_step` and stamps every message, Step row, StepMetrics row, OTel span, ProviderTrace, and webhook payload with it (`letta/agents/letta_agent_v2.py:947-962`, `letta/agents/letta_agent_v3.py:1037`, `letta/services/step_manager.py:475`). Step IDs are reused when resuming from an `approval` message (`letta/agents/letta_agent_v3.py:1019-1027`).
- **Adapter pattern for LLM transport**: `LettaLLMAdapter` (`letta/adapters/letta_llm_adapter.py:16-131`) abstracts blocking / streaming / SGLang-native behind the same `invoke_llm(request_data, messages, tools, ...)` generator. The agent loop never directly calls the provider SDK — it always goes through the adapter, which also extracts reasoning/tool_calls/usage into instance attributes the agent reads (`letta/adapters/letta_llm_request_adapter.py:57-98`).
- **StepProgression as a failure-path sentinel**: `StepProgression` (`letta/schemas/step.py:68-74`) is an int enum used in the `finally` block to decide whether to call `update_step_error_async`, `update_step_stop_reason`, or nothing (`letta/agents/letta_agent_v3.py:1541-1592`, `letta/agents/letta_agent_v2.py:666-726`). It survives exceptions and gives the loop a deterministic cleanup contract.
- **Two-phase checkpoint (start vs finish)**: `_step_checkpoint_start` creates a PENDING step row + OTel span immediately; `_step_checkpoint_finish` updates the row + closes the span only on the success path (`letta/agents/letta_agent_v2.py:941-1034`). This is the durability primitive that makes "the loop crashed at iteration N" visible in the DB rather than silent.
- **LLM request retry inside the step, not around the loop**: When a `ContextWindowExceededError` is raised by the LLM client, the retry happens inside `_step` (not in the outer `for i in range(max_steps)`) and triggers a compaction sub-loop (`letta/agents/letta_agent_v3.py:1093-1299`). This decouples "user-visible turn count" from "LLM call count".
- **Compaction is part of the step body, not the outer driver**: V3 invokes compaction both inline on context overflow (`letta/agents/letta_agent_v3.py:1218-1299`) and after a successful step when `context_token_estimate > compaction_trigger_threshold` (`letta/agents/letta_agent_v3.py:1438-1509`). The threshold comes from `get_compaction_trigger_threshold(llm_config)`.
- **Credit check runs in parallel with the next step**: `safe_create_task_with_return(self._check_credits())` is fired at the end of each iteration and awaited at the top of the next, so the credit-verification latency hides behind LLM/tool execution (`letta/agents/letta_agent_v3.py:389-390`).
- **Cancellation via Redis run_id**: The HTTP handler stores `run.id` under `{REDIS_RUN_ID_PREFIX}:{agent_id}` in Redis, and `_check_run_cancellation(run_id)` polls the DB Run row each iteration for `RunStatus.cancelled` (`letta/server/rest_api/routers/v1/agents.py:1747-1765`, `letta/agents/letta_agent_v2.py:750-757`).
- **Multi-turn RL training opt-in**: V3 keeps a `self.turns: list[TurnTokenData]` (`letta/agents/letta_agent_v3.py:140`) only populated when `llm_config.return_token_ids=True` and the handle/provider match `sglang/*`. Output_ids and logprobs come from `SGLangNativeAdapter` (`letta/adapters/sglang_native_adapter.py:289`), and tool-return content is also captured so the trainer can mask it (`letta/agents/letta_agent_v3.py:1314-1401`).

## Notable Patterns

- **Idempotent resume on approval**: When the last in-context message is an `approval`, V3 reuses the prior `step_id` so the resumed execution produces a single Step rather than two (`letta/agents/letta_agent_v3.py:1019-1027`). The StepMetrics row is fetched instead of re-created.
- **Heartbeat-driven continuation (V2 only)**: V2's `_decide_continuation` treats `request_heartbeat` as a "keep going" flag and injects a `NON_USER_MSG_PREFIX` system message explaining why the loop continued (`letta/agents/letta_agent_v2.py:1200-1285`). V3 removes heartbeats in favor of a "every tool call continues the loop" rule (`letta/agents/letta_agent_v3.py:1977-1986`).
- **Async generator as the turn contract**: `_step()` returns `AsyncGenerator[LettaMessage | dict, None]`, yielding chunks for streaming mode and `request_data` only when `dry_run=True` (`letta/agents/letta_agent_v3.py:909-929`, `:1150-1152`). The same method powers blocking `step()`, streaming `stream()`, and `build_request()` dry-run mode.
- **Auto-mode content-based rerouting**: When `llm_config.handle` is in `AUTO_MODE_HANDLES`, `_step` resolves a concrete handle via `get_llm_routing_client().resolve_auto_mode_config` and re-routes via `apply_reroute_rules(messages, ...)` — e.g. images → vision-capable model (`letta/agents/letta_agent_v3.py:1042-1087`).
- **Prefix-cache friendly system prompt**: `_refresh_messages` deliberately skips system-prompt rebuild on every step; it only rebuilds when memory/tool-rules/directories have changed, and only forces a rebuild after compaction (`letta/agents/letta_agent_v2.py:760-792`, `letta/agents/letta_agent_v3.py:965-972`).
- **Tracked-but-not-asserted parallelism**: `len(tool_calls) > 1 and not active_llm_config.parallel_tool_calls` truncates to the first call with a warning log because "Some providers (e.g. Gemini) don't respect this setting via API, so we enforce it client-side" (`letta/agents/letta_agent_v3.py:1337-1342`).
- **Cleanup on every path**: Streaming mode has explicit `try / except / finally` with an `error_aware_stream` wrapper that emits a terminal `[DONE]` and error event even on early failure (`letta/services/streaming_service.py:561-737`); the V3 stream also has an outer `try / except / finally` that yields an error event if cleanup itself fails (`letta/agents/letta_agent_v3.py:649-739`).

## Tradeoffs

- **3 loop generations side-by-side**: V1/V2/V3 all compile and ship, with V1 still receiving `@trace_method` updates. Maintaining the same loop shape across three classes plus the `BaseAgent` / `BaseAgentV2` abstraction doubles the surface area of changes (any fix to message persistence must be applied to both `letta/agents/letta_agent_v2.py:643-646` and `letta/agents/letta_agent_v3.py:758-816`).
- **`_step()` is too long (~700 lines)**: It mixes (a) message refresh, (b) LLM request building, (c) LLM call + retry + fallback, (d) response handling, (e) parallel/serial tool execution, (f) continuation decision, (g) persistence, (h) post-step compaction, (i) span/trace emission. Extracting sub-methods like `_execute_tools_for_step()` or `_emit_step_telemetry()` would make the loop shape clearer but would also add indirection.
- **V2 forbids no-tool-call responses**: V2 raises `LLMError("No tool calls found in response")` when the LLM returns text only (`letta/agents/letta_agent_v2.py:582-584`). V3 removed this — both `continue_stepping` and `end_turn` are valid terminal states for content-only responses (`letta/agents/letta_agent_v3.py:1621-1679`). API clients that depend on V2 therefore cannot use non-tool-call agent types.
- **Magic-index follow-up messages**: `if i == 1 and follow_up_messages: input_messages_to_persist = follow_up_messages; follow_up_messages = []` (`letta/agents/letta_agent_v3.py:329-331`) depends on the approval being at index 0 and the follow-up at index 1. This is tightly coupled to `_prepare_in_context_messages_no_persist_async`'s handling of approval flows (`letta/agents/helpers.py:222-369`).
- **Compaction sub-loop shares step state**: When `ContextWindowExceededError` triggers compaction inside `_step` (`letta/agents/letta_agent_v3.py:1218-1299`), the compaction call uses the same `step_id`, so a single Step row records both the failed LLM call and the summarizing LLM call. This is operationally convenient but means one Step can hold multiple `provider_name` / `model` values.
- **Cancellation is best-effort**: `_check_run_cancellation` polls the DB at step boundaries (`letta/agents/letta_agent_v2.py:750-757`), so a long-running LLM call will not be preempted mid-request — it will only stop on the next iteration. Mid-stream cancellation exists for streaming responses but still has to wait for the current `_step` to yield.
- **Inconsistent parallelism policy per provider**: Anthropic/Bedrock/MiniMax toggle `disable_parallel_tool_use`, OpenAI toggles `parallel_tool_calls`, Gemini is a no-op (`letta/agents/letta_agent_v3.py:1118-1146`). Provider-specific hand-tuning is required because not every provider exposes the same flag.

## Failure Modes / Edge Cases

- **LLM returns no tool call (V2)**: Raises `LLMError`, sets `stop_reason=no_tool_call`, terminates the loop with `run_status=RunStatus.failed` (`letta/agents/letta_agent_v2.py:582-584`, `letta/schemas/letta_stop_reason.py:32-43`).
- **Context window exceeded**: Caught in the inner LLM retry loop (`letta/agents/letta_agent_v3.py:1218-1299`), triggers compaction, retries up to `summarizer_settings.max_summarizer_retries = 3` times (`letta/settings.py:96`). If compaction itself fails with `SystemPromptTokenExceededError`, the agent exits with `context_window_overflow_in_system_prompt`.
- **LLM rate limit / server error**: Falls back to a configured fallback model handle via `LLMRouter.get_fallback_handle` and `record_failure` / `record_success` (`letta/agents/letta_agent_v3.py:1184-1212`). After `max_summarizer_retries` the loop raises with `stop_reason=llm_api_error`.
- **Invalid tool call (rule violation)**: `_decide_continuation` returns `continue_stepping=True` and injects a `Continuing: tool rule violation.` heartbeat message so the LLM can self-correct (`letta/agents/letta_agent_v2.py:1255-1257`, `letta/agents/letta_agent_v3.py:2004-2006`).
- **Pre-filled args from tool rules are invalid**: V3 catches the `ValueError` from `merge_and_validate_prefilled_args` and emits an `error` field on the execution spec; the tool is then short-circuited with `continue_stepping=False` and `stop_reason=invalid_tool_call` (`letta/agents/letta_agent_v3.py:1788-1912`).
- **Insufficient credits**: `credit_task` resolves to `False`, `stop_reason=insufficient_credits`, loop exits before next iteration (`letta/agents/letta_agent_v3.py:334-339`).
- **Cancellation**: At top of every step (`letta/agents/letta_agent_v3.py:1031-1035`), `_check_run_cancellation` reads the Run row; if `RunStatus.cancelled`, sets `stop_reason=cancelled` and breaks the loop. Streaming variant also handles this post-step (`letta/agents/letta_agent_v3.py:358-360`).
- **Approval flow dropped messages**: If an approval request message lacks `step_id`, V3 logs a warning and generates a new `step_id` (`letta/agents/letta_agent_v3.py:1021-1024`). If all three of `tool_calls / tool_call_denials / tool_returns` come back empty, V3 logs `"Invalid approval response: ... This likely indicates a corrupted or malformed approval payload."` and terminates with `stop_reason=invalid_tool_call` (`letta/agents/letta_agent_v3.py:1009-1017`).
- **Post-loop cleanup fails**: V3 stream's outer `except Exception as cleanup_error` emits an `event: error` with `error_type=cleanup_error` even when streaming has already produced terminal events (`letta/agents/letta_agent_v3.py:718-736`).
- **Stream ends without terminal event**: `error_aware_stream` in the streaming service emits `error_type=stream_incomplete` + `[DONE]` so clients never hang (`letta/services/streaming_service.py:610-627`).
- **Truncated client-side tool returns**: V3 caps tool-return strings at `_compute_tool_return_truncation_chars() = max(5000, context_window * 0.2 * 4)` (`letta/agents/letta_agent_v3.py:143-153`), with JSON-aware truncation that only chops the `message` field when present (`letta/agents/letta_agent_v3.py:1715-1746`).
- **Provider doesn't honor `parallel_tool_calls=false`**: Client-side enforcement truncates to the first tool call with a warning (`letta/agents/letta_agent_v3.py:1337-1342`).
- **LLM response with `finish_reason="length"`**: V3 maps this to `stop_reason=max_tokens_exceeded` (`letta/agents/letta_agent_v3.py:2000-2002`).

## Future Considerations

- **Convergence on V3**: V1 (`letta/agents/letta_agent.py`) and V2 (`letta/agents/letta_agent_v2.py`) are still actively maintained but the canonical path is V3. A future cleanup would delete V1/V2 and migrate batch/voice/sleeptime agents to inherit from V3.
- **Extract `_step()` sub-phases**: Methods like `_execute_tools_in_step`, `_emit_step_finish_telemetry`, `_maybe_compact`, `_apply_continuation_decision` could make `_step` linear and drawable in ~150 lines instead of ~700.
- **Mid-request cancellation for long LLM calls**: Today cancellation is checked at step boundaries; adding a cancellation-token-aware client SDK or a `cancel_stream` RPC could shorten cancellation latency on long-running steps.
- **Unified tool-call truncation contract**: Tool-result truncation is currently policy-per-tool (`return_char_limit` per tool) plus a global cap. Centralizing this would reduce surprise when a tool returns 1 MB of JSON.
- **Approval flow index dependency**: `if i == 1 and follow_up_messages` (`letta/agents/letta_agent_v3.py:329-331`) is brittle. A state machine that tracks "currently in approval-resume mode" would be clearer.
- **Observability: stream-level metrics**: TTFT and step durations are tracked (`letta/schemas/step_metrics.py:13-26`), but there is no per-stream metric for "time to first step" or "total steps". Adding a stream-level metrics record would help diagnose slow first responses.
- **Step-level provider_trace consistency**: ProviderTrace records per `invoke_llm` invocation, but a single Step may contain two LLM calls when compaction fires inside `_step` (`letta/agents/letta_agent_v3.py:1218-1299`). Adding a `trace_type` ("primary" / "compaction") to ProviderTrace would clarify which call is the user-visible one.
- **Multi-agent parent-child step attribution**: `SleeptimeMultiAgent` reuses `step_id` across sub-agents via the approval flow (`letta/groups/sleeptime_multi_agent_v3.py:42-85`). A `parent_step_id` field on `Step` would make parent-child causality explicit.

## Summary

Temporal Server implements a **highly layered scheduling model** with two parallel implementations of the user-facing Schedule feature (legacy V1 workflow-based scheduler and CHASM component-based scheduler), plus several lower-level scheduling primitives. Triggers can be grouped into five surfaces:

1. **Interactive API triggers** — `StartWorkflowExecution` (`service/frontend/workflow_handler.go:538`) routes to `historyEngineImpl.StartWorkflowExecution` (`service/history/history_engine.go:410`), which builds an in-memory `Starter` (`service/history/api/startworkflow/api.go:86`) that owns a workflow lease and creates a transfer + workflow-task pair. Idempotency is enforced via `RequestId` on the `StartWorkflowExecutionRequest` and `WorkflowIdReusePolicy`.
2. **Scheduled / delayed workflow start** — persisted as a `WorkflowBackoffTimerTask` (`service/history/tasks/workflow_delay_timer.go:14`) with `VisibilityTimestamp = startTime + FirstWorkflowTaskBackoff`. Generated by `GenerateDelayedWorkflowTasks` (`service/history/workflow/task_generator.go:378-409`), with the `WorkflowBackoffType` distinguishing `RETRY`, `CRON`, and `DELAY_START`. Cron-style schedules additionally re-enqueue a fresh `WorkflowBackoffTimerTask` via `transferQueueActiveTaskExecutor` (`service/history/transfer_queue_active_task_executor.go:1689` — `CronSchedule` propagated to the next ContinueAsNew).
3. **Schedules (cron-like, declarative)** — implemented twice:
   - **V1 (legacy)** — a workflow named `temporal-sys-scheduler-workflow` (`service/worker/scheduler/fx.go:26`) with id `temporal-sys-scheduler:<schedule-id>` (`service/worker/scheduler/workflow.go:72`). Driven by `schedulerWorkflowWithSpecBuilder` (`service/worker/scheduler/workflow.go:230`) which loops over `processTimeRange` → `addStart` → `processBuffer` and selects on signals (update/patch/refresh/force-CAN/migrate-to-chasm) plus a `workflow.NewTimer` (`service/worker/scheduler/workflow.go:755-824`).
   - **V2 (CHASM)** — pure timer tasks (`GeneratorTask`, `InvokerProcessBufferTask`, `InvokerExecuteTask`, `SchedulerIdleTask`, `BackfillerTask`, `SchedulerCallbacksTask`) registered in `chasm/lib/scheduler/library.go:83-114`. The generator is the only thing that drives the cron-style loop (`chasm/lib/scheduler/generator_tasks.go:50` re-schedules itself at `result.NextWakeupTime`). Documentation: `docs/architecture/schedules.md:8-15`.
4. **Timer queue + transfer queue + outbound queue** — internal task queues (`service/history/timer_queue_factory.go:33-79`, `service/history/transfer_queue_factory.go`, `service/history/outbound_queue_factory.go`) backed by `queues/scheduler.go`. Tasks are persisted in `tasks` table with `VisibilityTimestamp` as the schedule key. `scheduledQueue` (`service/history/queues/queue_scheduled.go:43-123`) gates firing through a `timer.LocalGate` and a `lookAhead` cache.
5. **Background scanners / per-namespace workers** — `service/worker/scanner/scanner.go:117` spins up SDK workflows at boot, most of which are **cron-triggered** via `CronSchedule: "0 */12 * * *"` (`service/worker/scanner/workflow.go:62`, `service/worker/scanner/build_ids/scavenger.go:39`, plus the schedule-invariant scanners at `service/worker/scanner/scheduleinvariants/workflows.go:49-66`). `PerNamespaceWorkerManager` (`service/worker/worker/pernamespaceworker.go:103`) maintains an SDK worker per namespace on a stable set of hosts, refreshed on namespace / membership / dynamic-config changes.

The trigger and scheduling semantics are **mature, durable, observable, and tested**: triggers are persisted (`GetHistoryTasks` reads by `VisibilityTimestamp`), idempotent at the request-id level (workflow creation uses `RequestId`, scheduler workflow uses `WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE` for scheduled actions at `service/worker/scheduler/workflow.go:1480`), the queue framework retries via `QueuedExecutable` (`service/history/queues/executable.go`) with backoff, and scheduled work survives shard ownership changes because it is indexed by time, not host. The CHASM scheduler (`chasm/lib/scheduler/`) explicitly moves the long-running state from a workflow to a typed component so that triggers and idempotency are owned by the framework rather than the workflow code itself.

## Rating

**9/10 — Mature, durable, observable, extensible, and proven under failure or scale.**

The rating is anchored on:

- **Persisted triggers.** Every scheduled/cron/callback retry is a row in the `tasks` table keyed by `(category, VisibilityTimestamp, TaskID)` (`service/history/tasks/category.go:18-28`, `service/history/tasks/workflow_delay_timer.go:25`, `service/history/queues/queue_scheduled.go:54-87`). A host / shard restart resumes from `taskReader` (no in-memory-only queues for timer/transfer).
- **Idempotency at multiple layers.** (a) Workflow start via `RequestId`; (b) `WorkflowIdReusePolicy = WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE` for scheduled actions (`service/worker/scheduler/workflow.go:1480`); (c) `ConflictToken` (`service/worker/scheduler/workflow.go:89`, used in `checkConflict` at `:1283`) for Schedule Update conflicts; (d) `WORKFLOW_ID_REUSE_POLICY_ALLOW_DUPLICATE` plus `StartWorkflowOptions.ID` for cron-triggered scanner workflows (`service/worker/scanner/workflow.go:61`) — start-with-idempotency per `service/worker/scanner/scanner.go:330-339` swallows `WorkflowExecutionAlreadyStarted`.
- **Idempotency-tested** explicitly: `service/worker/scanner/scanner.go:334` returns nil on `WorkflowExecutionAlreadyStarted`; CHASM scheduler's `InvokerExecuteTaskHandler` records duplicate-execute drops (`chasm/lib/scheduler/invoker_tasks.go:129`).
- **Idempotency of schedule triggers.** The V1 scheduler workflow's catch-up window (`service/worker/scheduler/workflow.go:706-716`) and missed-catchup metric (`ScheduleMissedCatchupWindow`) prevent runaway duplicate actions when the worker recovers late.
- **Observable.** The CHASM scheduler exposes `ExecutionStatus`, `ScheduleNextActionTime`, `ScheduleIdleCloseTime`, `ScheduleRunningWorkflowCount`, `ScheduleBufferedStartsCount` as search attributes (`chasm/lib/scheduler/scheduler.go:78-83`); the legacy scheduler exposes `TemporalSchedulePaused` (`service/worker/scheduler/workflow.go:1275`). Every executor emits per-task metrics (`service/history/queues/metrics.go:54-244`).
- **Foreground / background parity.** The same `Starter` path (`service/history/api/startworkflow/api.go:180`) serves user, signalWithStart, retry, child-workflow, and schedule-started workflows. The only differentiator is whether the start was initiated from a workflow event (recorded in `ParentExecutionInfo`) or from an external RPC.

Reasons this is **not** a 10:

- The V1 scheduler workflow uses `workflow.NewTimer` (`service/worker/scheduler/workflow.go:803`) which still relies on ContinueAsNew + signal selectors for the wakeup loop; this is robust but not the cleanest possible model (CHASM is the cleaner successor and the code explicitly migrates via `SignalNameMigrateToChasm` at `service/worker/scheduler/workflow.go:82`).
- Two parallel scheduler implementations co-exist (`service/worker/scheduler/workflow.go:1` vs `chasm/lib/scheduler/library.go:1`) and the rollout is gated by dynamic config (`service/worker/scheduler/fx.go:92-95`).
- `MigrateScheduleToChasm` activity (`service/worker/scheduler/workflow.go:1033`) is best-effort: errors only log and bump a metric (`service/worker/scheduler/workflow.go:347-354`).

## Answers to Dimension Questions

### 1. What starts execution?

Five trigger paths:

- **External RPC** — `StartWorkflowExecution` via `service/frontend/workflow_handler.go:538` → `service/history/history_engine.go:410` → `service/history/api/startworkflow/api.go:180` (`Starter.Invoke`). Each request carries a `RequestId` (idempotency key).
- **Schedule timer fire** — `WorkflowBackoffTimerTask` (`service/history/tasks/workflow_delay_timer.go:14`) when `VisibilityTimestamp` ≤ now, dispatched by `scheduledQueue` (`service/history/queues/queue_scheduled.go:175-201`) to `timerQueueActiveTaskExecutor.executeWorkflowBackoffTimerTask` (`service/history/timer_queue_active_task_executor.go:465-523`) which schedules the first WorkflowTask.
- **Cron-style schedule trigger** — either V1 (`service/worker/scheduler/workflow.go:230`) or CHASM (`chasm/lib/scheduler/generator_tasks.go:50-179`). Both keep a "high-water mark" of processed time and reschedule themselves. V1 uses `workflow.NewTimer`; CHASM schedules a pure `GeneratorTask` at `result.NextWakeupTime`.
- **User / activity / WFT / run / execution timeout** — generated by `CreateNextUserTimer`/`CreateNextActivityTimer` (`service/history/workflow/timer_sequence.go:69-156`) and run by the timer queue (`service/history/timer_queue_active_task_executor.go:104-131`).
- **Background scanners** — three bootstrap paths: (a) `Scanner.Start` (`service/worker/scanner/scanner.go:156-321`) launches SDK workflows, (b) `PerNamespaceWorkerManager` (`service/worker/pernamespaceworker.go:103-242`) hosts scheduler/scheduler-style workflows per namespace, (c) cron-scheduled workflows with `CronSchedule: "0 */12 * * *"` (`service/worker/scanner/workflow.go:62-74`) which internally use the same `WorkflowBackoffTimerTask` mechanism for each tick.

### 2. Are triggers durable?

Yes. Every scheduled trigger is a row in the `tasks` table indexed by `(CategoryID, VisibilityTimestamp, TaskID)`:

- `service/history/tasks/category.go:20-28` — `CategoryIDTransfer=1`, `CategoryIDTimer=2`, `CategoryIDReplication=3`, `CategoryIDVisibility=4`, `CategoryIDArchival=5`, `CategoryIDMemoryTimer=6`, `CategoryIDOutbound=7`. Warning comment at `:18`: "These IDS are persisted in the database. Do not change them."
- `service/history/queues/queue_scheduled.go:54-87` — pagination function fetches by `NewKey(r.InclusiveMin.FireTime, 0)`, upper-bounded by `common.ScheduledTaskMinPrecision`.
- `service/worker/scheduler/workflow.go:283-388` — V1 scheduler's `continueAsNew` uses `suggestContinueAsNew` from `info.GetContinueAsNewSuggested()` *or* `iters <= 0` *or* `forceCAN`; on the new run, `State.LastProcessedTime` is restored from history, so no scheduled work is lost.
- `chasm/lib/scheduler/scheduler_tasks.go:50-62` — idle task closes the scheduler only after a retention window so a subsequent re-create can resume from the same schedule ID.
- `service/worker/pernamespaceworker.go:325-363` — `handleError` retries with exponential backoff (`retrier.NextBackOff`) so a transient connection failure during worker registration does not lose scheduled workers.

### 3. Are duplicate triggers safe?

Yes, at multiple layers:

- **Workflow start**: `Starter.Invoke` uses `RequestId` and `WorkflowIdReusePolicy` (`service/history/api/startworkflow/api.go:86`, default `WORKFLOW_ID_REUSE_POLICY_ALLOW_DUPLICATE` but `WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE` for V1 scheduler at `service/worker/scheduler/workflow.go:1480`). Replay of the same `StartWorkflowExecutionRequest` is deduped.
- **Scheduled workflows (cron)**: each tick re-creates a `WorkflowBackoffTimerTask` whose `VisibilityTimestamp` is computed from `FirstWorkflowTaskBackoff` (`service/history/workflow/task_generator.go:387-388`). On the timer's fire, if `mutableState.HadOrHasWorkflowTask()` is already true, the executor no-ops (`service/history/timer_queue_active_task_executor.go:515-519`).
- **V1 scheduler actions**: `WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE` for scheduled actions and `WORKFLOW_ID_REUSE_POLICY_ALLOW_DUPLICATE` for manual triggers (`service/worker/scheduler/workflow.go:1480-1483`). The workflow also has a hard `MaxBufferSize = 1000` to drop duplicates if it can't keep up (`service/worker/scheduler/workflow.go:212, 1328-1332`).
- **Catch-up window**: actions whose nominal time exceeds `getCatchupWindow()` are explicitly dropped and counted (`ScheduleMissedCatchupWindow`) rather than re-enqueued indefinitely (`service/worker/scheduler/workflow.go:706-716`). This is the safe-degrade path on long host downtime.
- **CHASM scheduler**: `InvokerExecuteTaskHandler.recordDuplicateExecuteDrops` (`chasm/lib/scheduler/invoker_tasks.go:129`) counts duplicates and `InvokerExecuteTaskHandler` validates via `Validate` (`chasm/lib/scheduler/invoker_tasks.go:138`) before execution.
- **Conflict tokens**: V1 scheduler rejects conflicting updates (`checkConflict` at `service/worker/scheduler/workflow.go:1283-1288`) and the CHASM scheduler exposes `ConflictToken` (`chasm/lib/scheduler/scheduler.go:128-133`).
- **Scanner workflows**: `startWorkflow` swallows `WorkflowExecutionAlreadyStarted` so a re-spawn at server boot is safe (`service/worker/scanner/scanner.go:334`).

### 4. Can scheduled work be observed like interactive work?

Yes. Three channels:

- **Search attributes**: legacy `TemporalSchedulePaused` (`service/worker/scheduler/workflow.go:1275`) plus CHASM's 5 attributes (`chasm/lib/scheduler/scheduler.go:78-83`) are written via `workflow.UpsertSearchAttributes`. They are queryable from any list/visibility operation.
- **Queries**: V1 exposes `QueryNameDescribe` and `QueryNameListMatchingTimes` (`service/worker/scheduler/workflow.go:84-85`); both are registered in `scheduler.run()` (`:257-262`).
- **Event log**: CHASM scheduler maintains a per-component event log (`chasm/lib/scheduler/eventlog.go`) included via `getOrCreateEventLog(ctx).LogEvent(ctx, ...)` calls in every task handler (`chasm/lib/scheduler/generator_tasks.go:63`, `:109`, `:150`).
- **Metrics**: every executor / queue / activity has its own metric. Example: `metrics.ScheduleMissedCatchupWindow` (`service/worker/scheduler/workflow.go:714`), `metrics.ScheduleBufferOverruns` (`:1330`), `metrics.ScheduleActionDelay` (`:1525`), `metrics.ScheduleActionE2EDelay` (`:1527`).
- **DLQ on poison tasks**: `service/history/queues/dlq_writer.go` sends unprocessable tasks to a DLQ workflow (`service/worker/dlq/workflow.go`) for human review.

### 5. Do background and foreground execution share semantics?

Foreground and background share **the same primitive task layer** (timer queue, transfer queue, outbound queue), but **diverge in their drivers**:

- **Foreground (interactive)**: an `AddWorkflowTask` arrives at `matchingEngineImpl.AddWorkflowTask` (`service/matching/matching_engine.go:580-626`) where it is either sync-matched to a poller or persisted. The poller returns a `PollWorkflowTaskQueueResponse`. There is no scheduler state; this is a synchronous add/poll.
- **Scheduled (background)**: history-side generated tasks (`CreateNextUserTimer`, `CreateNextActivityTimer`, `GenerateDelayedWorkflowTasks`) are added via `mutableState.AddTasks(...)` (`service/history/workflow/timer_sequence.go:100-105, 146-154`). These are persisted into the `tasks` table and re-read by `scheduledQueue` based on `VisibilityTimestamp` (`service/history/queues/queue_scheduled.go:54-87`).
- **Background scans / batch operations**: SDK workflows running on the system worker (`service/worker/worker.go:46-101`) or per-namespace workers (`service/worker/pernamespaceworker.go:103`). These execute activities that touch persistence/visibility and are observable through their own metrics.

The **scheduler for end-user Cron-style schedules** is its own case: in V1, it is itself a workflow (`SchedulerWorkflow` running in the system worker); in V2, it is a CHASM component with typed tasks. Both end up calling `FrontendClient.StartWorkflowExecution` to materialize the scheduled workflow — meaning scheduled work eventually **does** pass through the same matching service path that interactive work uses.

The **error handling** does differ: scheduled tasks have explicit retry / DLQ policies in the queue framework (`service/history/queues/scheduler.go`, `service/history/queues/dlq_writer.go`), while an interactive API call returns the error directly to the caller.

## Architectural Decisions

1. **Two scheduler implementations** with a migration path. V1 (`service/worker/scheduler/workflow.go:1`) is a workflow-driven scheduler; V2 (`chasm/lib/scheduler/library.go:1`) is a CHASM component-driven scheduler. The V1 workflow has `SignalNameMigrateToChasm` (`service/worker/scheduler/workflow.go:82`) and `executeMigration` (`:1033`) which delegates to `MigrateScheduleToChasm` activity. This is a deliberate staged rollout (`service/worker/scheduler/fx.go:92-95` — `EnableCHASMSchedulerMigration` and `CHASMSchedulerMigrationRolloutPercent` dynamic config).
2. **Tasks are first-class persisted entities**, not in-memory jobs. `CategoryID*` constants at `service/history/tasks/category.go:18-28` are explicitly warned as persisted.
3. **High water marks and conflict tokens** are used for stateful trigger deduplication. `ConflictToken` (V1 scheduler, `:1283`) and the `LastProcessedTime` HWM (V1 and CHASM) prevent the same time-range from being processed twice.
4. **Time-skipping is integrated** with timer regeneration: `RegenerateTimerTasksForTimeSkipping` (`service/history/workflow/task_generator.go:1039`) re-derives user/exec/run/fast-forward/backoff timers with virtual-time tracking.
5. **Forever-running system workflows use `ContinueAsNew`** (`service/worker/scheduler/workflow.go:283-388`) and `infiniteDuration = 20 * 365 * 24h` (`service/worker/batcher/workflow.go:22`, `service/worker/scanner/workflow.go:19`) to avoid history bloat.
6. **Per-namespace workers** rather than per-cluster workers (`service/worker/pernamespaceworker.go:103-242`) so that namespace-level configuration and rate limits can be applied independently. Host selection is via consistent hashing on namespace ID (`:268-281`).
7. **Rate limiting at the source**: V1 scheduler rate-limits StartWorkflow calls via `quotas.RateLimiter` (`service/worker/scheduler/activities.go:38, 89-106`) with a non-retryable `RateLimited` error type that converts to a `workflow.Sleep(delay)` (`service/worker/scheduler/workflow.go:1511-1517`).
8. **Dedicated DLQ workflow** (`service/worker/dlq/workflow.go`) for unprocessable tasks, with `MaxMessageID` pagination so deletes and re-enqueues can be retried atomically per shard.
9. **`CronSchedule` is set on background workflows**, not on user-facing Schedule executions — the latter is handled by the dedicated Scheduler workflow/component. This separation keeps the two semantics independent.

## Notable Patterns

- **`workflow.NewTimer` + signal selector** for V1 scheduler wakeups (`service/worker/scheduler/workflow.go:755-824`). The selector listens on signal channels (update/patch/refresh/force-CAN/migrate) plus a timer future plus a watcher future (`s.watchingFuture`).
- **`SideEffect` + `MutableSideEffect`** in V1 scheduler (`service/worker/scheduler/workflow.go:524, 596, 1300`) so non-deterministic spec parsing can be patched without breaking history.
- **Reusable timer**: `currentTimer` + `currentTimerDeadline` (`:124-125`) so re-arms don't always create a new timer (`tweakables.ReuseTimer` at `:214`).
- **Watchdog via long-poll activity**: `startLongPollWatcher` with 365-day `StartToCloseTimeout` and 65-second `HeartbeatTimeout` (`service/worker/scheduler/workflow.go:1597-1604`) so a paused scheduler still wakes up when a watched workflow closes.
- **`nexusoperations` and `callbacks` components** (`components/callbacks/`, `components/nexusoperations/`) treat external completion callbacks as a state machine task, decoupling HTTP-triggered completions from the workflow timer queue.
- **`scanner/scheduleinvariants`**: three cron-style workflows (`OverdueNextActionTime`, `StuckOpen`, `UnknownState`) that exist specifically to detect when the scheduler itself has drifted from its invariants. This is a self-healing safety net.
- **Outbound queue as a state-machine category** (`service/history/tasks/category.go:83-87`) lets outbound HTTP calls (Nexus) be a peer to timer/transfer tasks with the same retry / DLQ semantics.

## Tradeoffs

- **V1 scheduler as a workflow** trades simplicity for resource cost: every schedule is a running workflow with full history, must `ContinueAsNew` periodically, and depends on the SDK timer + signal mechanisms for wakeups. CHASM removes these costs at the price of a much larger code surface (`chasm/lib/scheduler/` ~7k LOC vs `service/worker/scheduler/workflow.go` ~1.7k LOC).
- **Catch-up window dropping** (`service/worker/scheduler/workflow.go:706-716`) is an explicit design choice: better to skip missed actions than replay them all after a long downtime. This means user-visible `MissedCatchupWindow` metric but no auto-retry.
- **MaxBufferSize = 1000** for V1 scheduler (`:211`): prevents runaway growth but also means in extreme backpressure the scheduler drops actions, surfaces them as `ScheduleBufferOverruns`, and requires manual intervention via Update/Patch.
- **Per-namespace workers** can leave a node with many workers if the cluster has many namespaces. This is mitigated by `perNamespaceWorkerStartRate` (`service/worker/pernamespaceworker.go:123`) and the `startLimiter`.
- **Two scheduler implementations** in coexistence (V1 + CHASM) doubles the maintenance burden and testing surface. The migration path uses sentinel workflows (`service/frontend/workflow_handler.go:3664-3692`) to atomically claim the schedule ID across both implementations.
- **Time-skipping** (`RegenerateTimerTasksForTimeSkipping`) is opt-in; default servers run on wall-clock time. Time-skipping complicates cron semantics because virtual time differs from real time.

## Failure Modes / Edge Cases

- **Cluster / shard restart**: timer tasks persist across restarts because they're in the `tasks` table; `scheduledQueue.notify(time.Time{})` re-arms on boot (`service/history/queues/queue_scheduled.go:138`).
- **Host loses membership**: `membershipChangedListener` (`service/worker/pernamespaceworker.go:206-219`) refreshes per-namespace workers on the new set of hosts.
- **Long downtime / catch-up skipped**: explicit `ScheduleMissedCatchupWindow` metric and `Info.MissedCatchupWindow` counter; UI surfaces this via Describe.
- **Time going backwards** in scheduler workflow: handled at `service/worker/scheduler/workflow.go:301-305` (`t2 = t1`).
- **Clock skew across hosts**: `IsTimeExpired` truncates to `common.ScheduledTaskMinPrecision` and uses the task's `FireTime` as a floor (`service/history/queues/queue_scheduled.go:286-301`). CHASM treats sub-precision drift as "fire anyway" (`chasm/lib/scheduler/scheduler_tasks.go:88-90`).
- **WorkflowBackoffTimerTask retry stamp mismatch**: executor checks `task.Version` against the workflow start version (`service/history/timer_queue_active_task_executor.go:413-416`).
- **CHASM Idle task invalidation**: `Validate` checks `scheduler.Closed`, `isHeldOpen()`, and `idleDeadline` (`chasm/lib/scheduler/scheduler_tasks.go:71-103`).
- **V1 scheduler workflow task overflow**: `IterationsBeforeContinueAsNew` plus `impossibleHistorySize = 1e6` hard cap (`service/worker/scheduler/workflow.go:100, 285`).
- **Catch-up skip for ALLOW_ALL**: ALLOW_ALL overlap runs are not tracked in `RunningWorkflows` (`:1472-1476`), preventing replay duplication.
- **Batcher activity restart**: `batchActivityOptions.StartToCloseTimeout = infiniteDuration` with `HeartbeatTimeout` (`:107-111`) plus `ContinueAsNew` ensures the batch keeps going.
- **Scanner deadlock**: TODO comment at `service/worker/scanner/scanner.go:212, 239, 256, 268, 286` flags that workers don't listen for fatal errors. Not a correctness issue but operational visibility gap.

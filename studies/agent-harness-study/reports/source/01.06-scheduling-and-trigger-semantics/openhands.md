## Summary

OpenHands treats "what triggers execution" as primarily a **user-driven, request/response story** with a thin **task-persistence layer** and a **callback processor loop** on top. Every conversation start is recorded as a durable `AppConversationStartTask` (`openhands/app_server/app_conversation/app_conversation_models.py:281-309`), so the long-running "start" flow is observable and resumable per user. Messages arriving before the agent is ready are buffered in a dedicated `pending_messages` table (`openhands/app_server/pending_messages/pending_message_service.py:27-38`, alembic `openhands/app_server/app_lifespan/alembic/versions/007.py:21-34`) and flushed once the conversation transitions to READY (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:525-533`, `:1925-2006`). Trigger *origin* is captured as a first-class `ConversationTrigger` enum (`openhands/app_server/app_conversation/app_conversation_models.py:37-48`) with 11 named values covering GUI, resolvers, suggested tasks, Jira/Slack/Linear/Bitbucket integrations, MICROAGENT_MANAGEMENT, REMOTE_API_KEY and a generic AUTOMATION bucket; the enum also feeds analytics and limits how integrations can mutate state.

There is **no in-process scheduler, no cron expression surface, and no internal retry/replay loop**. Background work is composed of a few well-defined primitives: (a) `asyncio.create_task(...)` continuations in routers (`openhands/app_server/app_conversation/app_conversation_router.py:405`, `:968`), (b) Redis-locked long-running exports refreshed by `refresh_lock_periodically` (`openhands/app_server/utils/redis_lock.py:36-53`), (c) a **dev-only** `poll_agent_servers` coroutine that runs only when no public `web_url` is configured (`openhands/app_server/sandbox/remote_sandbox_service.py:682-925`), and (d) ad-hoc polling loops for sandbox readiness (`openhands/app_server/sandbox/sandbox_service.py:93-140`) and SetTitle callback title retries (`openhands/app_server/event_callback/set_title_callback_processor.py:37-79`). The architecture therefore depends on **external triggers** (HTTP requests from GUI, integration webhooks, REST API from SDK) and on **durable middleware-level state** (DB rows + Redis) rather than on a process-internal scheduler.

If the app-server process restarts, **no background scheduler reaps in-flight tasks**. Tasks are durable in DB but require client polling (`/start-tasks/search`, `/start-tasks/{id}`) — there is no built-in reaper or timed retry. This makes restart safe for *new* requests (they re-create task rows) but leaves already-running-but-not-yet-READY start tasks in `WORKING`/`WAITING_FOR_SANDBOX`/`STARTING_CONVERSATION` states with no automatic resumption; the only "recovery" is the next HTTP read on the task.

## Rating

**6/10 — Clear trigger taxonomy with durable persistence, but no scheduler, no retry, and restart recovery is implicit.**

The trigger model is well-named and persisted in DB columns (e.g., `trigger: Mapped[str | None]` at `openhands/app_server/app_conversation/sql_app_conversation_info_service.py:84`), the start-task state machine is explicit (`AppConversationStartTaskStatus` enum at `openhands/app_server/app_conversation/app_conversation_models.py:262-271`), and webhook + event-callback ingestion has idempotency guards (`merge_conversation_tags` at `openhands/app_server/event_callback/webhook_router.py:93-108`, the `detect_automation_trigger` "don't override existing" guard at the same file `:200-201`). The pending-message queue has a hard cap of 10 to bound blast radius (`openhands/app_server/pending_messages/pending_message_router.py:86-91`) and Redis-lock refresh during long exports (`openhands/app_server/utils/redis_lock.py:36-53`, used at `openhands/app_server/app_conversation/live_status_app_conversation_service.py:2397-2403`).

But several rubric gaps prevent a higher score: (1) no scheduler/cron layer exists in this source at all — the only background coroutine is a dev-mode polling fallback (`openhands/app_server/sandbox/remote_sandbox_service.py:919-932`); (2) there is no scheduled reaper that resumes in-flight `AppConversationStartTask`s after restart, even though they are stored durably (`openhands/app_server/app_conversation/app_conversation_start_task_service.py:15-70`); (3) webhook idempotency is best-effort (no provider-supplied event id de-dup; same conversation webhook with same body causes a duplicate `merge_conversation_tags`/callback run); (4) trigger enum coverage is integration-driven — additions require code changes, not config; (5) the only event-loop background task is fire-and-forget (`asyncio.create_task`) with no observable supervisor (`openhands/app_server/app_conversation/app_conversation_router.py:405`, `:968`).

## Answers to Dimension Questions

1. **What starts execution?**
   - HTTP request via the V1 router: `start_app_conversation` (`openhands/app_server/app_conversation/app_conversation_router.py:363`), `stream-start` (`:981`), `send-message` (`:429`), `switch_profile` (`:619`), `start_sandbox` (`openhands/app_server/sandbox/sandbox_router.py:75`), `pause_sandbox` (`:84`), `resume_sandbox` (`:95`), `delete_sandbox` (`:108`).
   - Webhook callbacks from agent servers (or remote runtimes): `/webhooks/conversations` (`openhands/app_server/event_callback/webhook_router.py:333`), `/webhooks/events/{conversation_id}` (`:453`), `/webhooks/secrets` (`:531`); see also `WEBHOOK_CALLBACK_VARIABLE = 'OH_WEBHOOKS_0_BASE_URL'` (`openhands/app_server/sandbox/sandbox_service.py:26`) and `f'{self.web_url}/api/v1/webhooks'` (`:273`).
   - Pending-message REST submitter: `POST /api/v1/conversations/{id}/pending-messages` (`openhands/app_server/pending_messages/pending_message_router.py:37`).
   - Event-driven triggers running through the `EventCallbackService` loop, fired by webhook event arrivals (`openhands/app_server/event_callback/sql_event_callback_service.py:208-251`).
   - **No scheduler, cron, timer, queue worker, or internal retry loop is present.** The only periodic loop is the dev-only polling fallback at `openhands/app_server/sandbox/remote_sandbox_service.py:919-932`.

2. **Are triggers durable?**
   - **Conversation-start triggers** are persisted: every state transition in `start_app_conversation` writes through `save_app_conversation_start_task` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`), and tasks are queryable through `start-tasks/search|/{id}|/count` (`openhands/app_server/app_conversation/app_conversation_router.py:996-1075`).
   - **Conversation info** rows include a `trigger` column that survives DB round-trips (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:84`, `:361`, `:560`).
   - **Pending messages** are persisted in `pending_messages` (`openhands/app_server/pending_messages/pending_message_service.py:27-38`) with the alembic migration at `openhands/app_server/app_lifespan/alembic/versions/007.py:21-34`.
   - **Event callbacks** are persisted in `event_callback` table with composite index on `(conversation_id, status, event_kind)` (`openhands/app_server/event_callback/sql_event_callback_service.py:48-73`).
   - **What is *not* durable**: background `asyncio.create_task(...)` continuations — if the app process restarts while a start task is mid-`WAITING_FOR_SANDBOX` or `PREPARING_REPOSITORY`, the DB row stays but no background coroutine resumes it; the next read will observe a stale `WORKING` state forever (or until the user retries).

3. **Are duplicate triggers safe?**
   - **Webhooks**: Webhook replay safety is partial. `merge_conversation_tags` is a shallow dict merge so calling the same webhook twice is idempotent for `tags` (`openhands/app_server/event_callback/webhook_router.py:93-108`). `on_conversation_update` overwrites `app_conversation_info` by id (no upsert-by-event-id), so a replay inside the same minute overwrites the previous run’s merged tags. `detect_automation_trigger` does not override existing triggers (`openhands/app_server/event_callback/webhook_router.py:212-213`), but that is the only idempotency guard.
   - **EventCallback execution**: Each callback runs once per event kind match in `execute_callbacks` (`openhands/app_server/event_callback/sql_event_callback_service.py:208-231`). Webhooks call this through `asyncio.create_task(_run_callbacks_in_bg_and_close, ...)` (`openhands/app_server/event_callback/webhook_router.py:513-517`), which means a duplicate webhook fires callbacks twice — `_run_callbacks_in_bg_and_close` does not deduplicate by `event_id`.
   - **Conversation start**: The router returns the first yielded task synchronously and detaches the rest with `asyncio.create_task` (`openhands/app_server/app_conversation/app_conversation_router.py:380-410`). Submitting the same start request twice creates two distinct `AppConversationStartTask` rows (no idempotency key on the request body). The DB-level `app_conversation_id` is the join key, but a second submission may run in parallel and create a second conversation.
   - **Sandbox delete**: guarded by reference count (`if conversation_count == 0`) at `openhands/app_server/app_conversation/app_conversation_router.py:880-883`.
   - **Conversation export**: Redis lock prevents concurrent runs; missing Redis in non-SaaS is silently tolerated (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:2365-2376`).

4. **Can scheduled work be observed like interactive work?**
   - **Yes, via the start-task API.** Interactive POST `/api/v1/app-conversations` and the stream-start variant both return `AppConversationStartTask` objects (`openhands/app_server/app_conversation/app_conversation_router.py:373`, `:985`), and the same task model is queryable afterwards via `GET /start-tasks/search` and `GET /start-tasks` (`:996`, `:1059`). Internally every transition goes through `save_app_conversation_start_task` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`), so polling clients see progress through `WORKING → WAITING_FOR_SANDBOX → PREPARING_REPOSITORY → RUNNING_SETUP_SCRIPT → SETTING_UP_GIT_HOOKS → SETTING_UP_SKILLS → STARTING_CONVERSATION → READY (or ERROR)` (`openhands/app_server/app_conversation/app_conversation_models.py:262-271`).
   - **Pending messages are observable as a queue with positions** (`openhands/app_server/pending_messages/pending_message_models.py:28-33`).

5. **Do background and foreground execution share semantics?**
   - **Shared status surface**: both produce and update an `AppConversationStartTask` row (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`, the `_start_app_conversation` async generator yields after each step).
   - **Shared DB session / httpx client**: keep-open flags are set on the request state so background continuations share the same session (`openhands/app_server/app_conversation/app_conversation_router.py:374-376`, `:962-964`).
   - **Different error handling**: foreground HTTP path returns the task synchronously and runs `_consume_remaining` in background; if `await anext(async_iter)` raises, db_session+httpx are explicitly closed (`openhands/app_server/app_conversation/app_conversation_router.py:407-410`). Background path swallows errors by emitting `task.status = ERROR; task.detail = ...` then yielding (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:534-538`). So errors in the foreground result in HTTP 500, errors in the background appear as `AppConversationStartTaskStatus.ERROR`.
   - **Callback background errors**: `execute_callback` catches every exception and writes `EventCallbackResultStatus.ERROR` to the result table (`openhands/app_server/event_callback/sql_event_callback_service.py:241-249`). The callback itself stays `ACTIVE` so a future event could retry.
   - **No automatic retry for start-task failures**: a start task in `ERROR` state stays ERROR; clients must re-POST.

## Architectural Decisions

- **Trigger as a first-class enum, not a free-form string.** Captures the *origin* of the conversation for analytics and conditional logic (`openhands/app_server/app_conversation/app_conversation_models.py:37-48`). Integrations like GitHub/Jira/Slack/Bitbucket each set their own enum value at conversation-create time (e.g. `ConversationTrigger.JIRA` at `enterprise/integrations/jira/jira_view.py:241`, `ConversationTrigger.RESOLVER` at `enterprise/integrations/github/github_view.py:222`); the AUTOMATION bucket is the catch-all for tag-driven runs (`openhands/app_server/event_callback/webhook_router.py:191-232`).
- **Long-running starts are persisted state machines, not async coroutines.** `_start_app_conversation` is an async generator that yields after each meaningful state change and persists via `save_app_conversation_start_task` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`). Clients poll the resulting task; the app does not push or wake them.
- **Webhooks carry the message and the agent runs out-of-process.** The /webhooks endpoints do not run any agent logic themselves; they persist conversation info or events and *may* invoke registered `EventCallbackProcessor`s for derived state (titles, etc.) (`openhands/app_server/event_callback/webhook_router.py:333-522`). This isolates trigger ingestion from agent execution.
- **Pending messages are a separate durability boundary from the conversation itself.** They survive sandbox startup races (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:525-533`), and a hard cap of 10 bounds memory (`openhands/app_server/pending_messages/pending_message_router.py:85-91`).
- **Lock-based coordination instead of a queue.** Conversation export uses Redis with periodic TTL refresh (`openhands/app_server/utils/redis_lock.py:36-53`), which is the closest thing to a "scheduled background job" in this codebase.
- **No internal scheduler.** Search of the source for `cron`, `schedule`, `add_job`, `apscheduler`, `interval` reveals only: (a) `refresh_lock_periodically` (a custom TTL refresh loop, not a scheduler), (b) `sandbox_startup_poll_frequency` (a poll interval inside an existing coroutine), (c) the dev-only `poll_agent_servers` fallback when there is no public URL (`openhands/app_server/sandbox/remote_sandbox_service.py:919-932`). Cron-style expression parsing is absent.

## Notable Patterns

- **`async_generator` start-task state machine.** `LiveStatusAppConversationService._start_app_conversation` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:309-538`) is a single async generator yielding `AppConversationStartTask` instances. Each yield is followed by a DB write. The router consumes one yield synchronously and the rest in a background task (`openhands/app_server/app_conversation/app_conversation_router.py:380-405`). This is the canonical pattern for *long-running but observable* operations in this codebase.
- **Sticky `keep_open` flags on db_session/httpx_client for background continuations.** Pattern at `openhands/app_server/app_conversation/app_conversation_router.py:374-376`, `:962-964`, and `set_*_keep_open` defined at `openhands/app_server/services/db_session_injector.py` / `openhands/app_server/services/httpx_client_injector.py`. Without this, async continuations would see closed resources.
- **Polling + event webhooks.** When the app server is reachable, agent servers call `/api/v1/webhooks/*` (`openhands/app_server/sandbox/remote_sandbox_service.py:273`). When it is not (local development), a single `asyncio.create_task` polls the runtime `/list` endpoint every `polling_interval` seconds (default 15) and reconciles (`openhands/app_server/sandbox/remote_sandbox_service.py:919-932`, `:682-925`).
- **Detect-vs-set trigger semantics on webhook.** `detect_automation_trigger` is careful not to override an existing trigger (`openhands/app_server/event_callback/webhook_router.py:212-213`), which means a conversation that was created GUI cannot accidentally be relabeled AUTOMATION just because a later webhook delivered automation tags.
- **Idempotent tag merge.** `merge_conversation_tags(existing, incoming)` is a deterministic shallow merge with `incoming` winning on collisions (`openhands/app_server/event_callback/webhook_router.py:93-108`). This lets the same webhook replay without losing previously merged keys.
- **Rate-limit exemption on sandbox resume.** `_RESUME_RE` skips the in-process rate limiter (`openhands/app_server/middleware.py:140-141`) because operators frequently re-resume after sandbox restarts.
- **Pending-message queue with explicit lifecycle.** `add_message → update_conversation_id (task-id → real id) → get_pending_messages → sequential replay → delete` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1951-2002`). The terminal `delete` regardless of send result (`live_status_app_conversation_service.py:1997-2002`) means the queue is at-least-once, not exactly-once.

## Tradeoffs

- **No scheduler = simple model, but limited operational behaviors.** Adding a periodic "expire stale tasks after 24h" cleanup, or "retry on transient DB errors", or "prune orphaned sandboxes", requires either a cronjob hitting REST endpoints or external infrastructure — the app itself does not run any periodic logic in production. The dev-only poller demonstrates that the architecture *can* support a periodic loop but only uses it for reconciliation when webhooks are absent.
- **Foreground returns immediately, background does the work.** This gives low-latency UX, but observability burden shifts to the client (poll `/start-tasks/{id}`). Operators rely on the same client surface to monitor production behavior.
- **No event-id dedup on webhook ingestion.** Two identical webhook deliveries fire the callback fan-out twice. The webhook *body* could be deduplicated by event-id, but no such logic is implemented; the only guard is "don't override existing trigger" (`openhands/app_server/event_callback/webhook_router.py:212-213`).
- **Pending-message queue is per-conversation and capped at 10.** Simple and predictable, but loses cross-conversation ordering. Multiple concurrent start-task → same conversation id races are theoretically possible if two clients race.
- **Redis is optional in non-SaaS.** `LiveStatusAppConversationService._conversation_export_lock_required()` defaults to SaaS only (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:2323-2326`), so concurrent exports in OSS are *not* guarded and may produce duplicate zips.
- **Async tasks are fire-and-forget.** `asyncio.create_task(...)` patterns (`openhands/app_server/app_conversation/app_conversation_router.py:405`, `:968`, `openhands/app_server/event_callback/webhook_router.py:513`) are not supervised — if they raise, the exception is logged but no automatic retry or alerting exists. The exception path inside `_start_app_conversation` does update `task.status = ERROR` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:534-538`), but tasks created by `_finalize_sandbox_delete` or `_run_callbacks_in_bg_and_close` have no such refinement.
- **Triggers are coded into the enum.** Adding a new integration requires code (enum + integration code path), configuration is not exposed for end users.

## Failure Modes / Edge Cases

- **App-server restart mid-start-task.** The DB task row persists (`openhands/app_server/app_conversation/sql_app_conversation_start_task_service.py:54-74`), but the background asyncio continuation started via `asyncio.create_task(...)` (`openhands/app_server/app_conversation/app_conversation_router.py:405`) is gone — its in-memory state (sandbox connect attempt, profile seeding, agents-kill list) is lost. The next client poll on `/start-tasks/{id}` will see a stale `WORKING`/`WAITING_FOR_SANDBOX` row that will not progress until a fresh request re-runs the full pipeline. This is a real durability gap.
- **Polling fallback runs in single instance only.** The dev-only `polling_task` is a module-global `asyncio.Task | None = None` (`openhands/app_server/sandbox/remote_sandbox_service.py:54`) and is created inside `RemoteSandboxServiceInjector.inject`. If two app-server instances were both configured with `web_url=None`, both would create competing tasks that race on `/list`. The code does not guard against this.
- **Webhook flood → callback storm.** Each webhook schedules `asyncio.create_task(_run_callbacks_in_bg_and_close)` (`openhands/app_server/event_callback/webhook_router.py:513-517`). A burst of webhooks fires N parallel callback fan-outs against the agent server with no rate limit, no dedup, no shared resource cap. The DB session is scoped tightly per webhook but the agent-server gets hammered.
- **Sandbox readiness polling can outlive timeout.** `wait_for_sandbox_running` raises `SandboxError` after `sandbox_startup_timeout` seconds (default 120) (`openhands/app_server/sandbox/sandbox_service.py:119-140`), but anything the caller did *before* that (e.g., `_seed_sandbox_profiles` at `openhands/app_server/app_conversation/live_status_app_conversation_service.py:925-1000`) is best-effort and silently logged on failure rather than surfaced back to the start task.
- **Retry-on-failure not modeled.** A failed start-task remains `ERROR` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:534-538`); there is no automatic retry, and the start-task service exposes no `restart_task` endpoint — the client must re-POST.
- **Background callbacks fire after error path.** Even if `event_service.save_event` fails, the webhook still triggers `_run_callbacks_in_bg_and_close` because the `try/except` only wraps the save path (`openhands/app_server/event_callback/webhook_router.py:519-522`). State divergence between stored events and triggered callbacks is possible.
- **StopIteration / fast path bug.** `start_app_conversation` calls `await anext(async_iter)` once before saving the first yielded task (`openhands/app_server/app_conversation/app_conversation_router.py:381`). If the generator returns before the first yield (no path does so today, but the structure is fragile), the await raises `StopAsyncIteration` and the call surfaces as a 500 instead of a graceful task payload.
- **Pending-message conversion race.** `_process_pending_messages` does `update_conversation_id(task_id, conversation_id)` then sequential replay (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1951-2002`). Two starts sharing the same task-id hex *and* both racing to mark themselves `READY` will both attempt the update — the SQL `update` uses `select then write`, not atomic single SQL, so the last writer wins.

## Future Considerations

- **Add a minimal scheduler.** The codebase would benefit from a tiny in-process loop that reaps stuck `AppConversationStartTask`s (e.g., tasks older than 30 minutes in non-terminal states) and periodically prunes orphaned sandboxes. A primitive version could live inside `OssAppLifespanService.__aenter__` (`openhands/app_server/app_lifespan/oss_app_lifespan_service.py:15-18`) without disturbing the rest of the architecture.
- **Webhook idempotency keys.** Accept an optional `X-Webhook-Id` and store `(provider, webhook_id) → conversation_id` in a small dedup table, then short-circuit repeat deliveries inside `on_conversation_update` and `on_event`. The current trigger-merge logic covers state, not events.
- **Task resumption on startup.** When `OssAppLifespanService` starts, scan for `AppConversationStartTask` rows in `WORKING`/`WAITING_FOR_SANDBOX`/`STARTING_CONVERSATION` and re-drive them via `start_app_conversation` for the same `request`. This closes the durability gap.
- **Promote `polling_task` to a proper supervisor.** Replace the global single-shot `asyncio.create_task` at `openhands/app_server/sandbox/remote_sandbox_service.py:925-932` with an explicit supervisor service that owns lifecycle, cancellation, and re-acquire behavior.
- **Make `ConversationTrigger` extensible from config.** Today the enum is closed (`openhands/app_server/app_conversation/app_conversation_models.py:37-48`); a follow-up extension could read an `OH_EXTRA_TRIGGERS` env var (or admin config) to allow SaaS operators to register custom trigger values without changing core code.
- **Observability hooks for background tasks.** Wrap `asyncio.create_task` calls (`openhands/app_server/app_conversation/app_conversation_router.py:405`, `:968`, `openhands/app_server/event_callback/webhook_router.py:513`) in a helper that records task names, trace ids, and exceptions into the analytics service (which is already imported in `webhook_router.py:17`). This makes fire-and-forget truly observable.
- **Sandbox resume path could be a generic timer.** `_RESUME_RE` exemption exists because resuming a sandbox is a legitimate operation that exceeds the default 10 req/sec cap (`openhands/app_server/middleware.py:84-112`). A future timer/scheduler component would also need to be exempted in the same way; folding this exception into a "trusted internal" path rather than a regex would scale better.

## Questions / Gaps

- **No evidence of any cron, APScheduler, Celery, RQ, Dramatiq, or any external scheduler integration in the selected source.** The closest pattern is `refresh_lock_periodically` (`openhands/app_server/utils/redis_lock.py:36-53`), which is a TTL refresh, not a scheduler. The `web_url`-less polling path (`openhands/app_server/sandbox/remote_sandbox_service.py:919-932`) is documented as "primarily used for local development rather than production" (`:920`), not as a production scheduler.
- **No evidence of an internal retry/replay loop** for failed start tasks or failed webhook deliveries. Errors propagate to `ERROR` state or to logs; there is no automatic retry. **No clear evidence found** of operator-facing retry endpoints.
- **No evidence of a process-restart reaper for stuck tasks.** Search for `WORKING`, `pre-existing` or `recovery` produced no recovery code path. The system assumes either external HTTP polling or front-end polling will eventually trigger a new request.
- **No evidence that webhook event_ids are de-duplicated.** The webhook signature is verified against session api key (`openhands/app_server/event_callback/webhook_router.py:235-278`), but payloads are not deduplicated by an event-id field.
- **Sandbox start has no idempotency key.** A POST `start_sandbox` with no parameters always returns a new sandbox; calling it twice creates two sandboxes (with limit-check via `max_num_sandboxes` in `pause_old_sandboxes` at `openhands/app_server/sandbox/sandbox_service.py:203-244`).
- **Skills trigger is a separate concept.** `SkillResponse.triggers` (`:318`) and `SkillLoader` keyword/task matching (`openhands/app_server/app_conversation/skill_loader.py:370-382`) handle *content triggers* (which skills to inject for a given message), not conversation triggers. They are mentioned for completeness but are not execution triggers in the dimension's sense.

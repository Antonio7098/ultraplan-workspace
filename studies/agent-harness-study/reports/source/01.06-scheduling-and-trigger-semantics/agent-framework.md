## Summary

Microsoft Agent Framework (MAF) treats "what triggers execution" and "how scheduled work runs" as a deliberately **durable-first, host-agnostic** problem. The base agent in `agent_framework` is an in-process, prompt-driven object that has no built-in trigger surface (`agent-framework:46`), so the trigger story lives entirely in two add-on packages — `agent-framework-durabletask` (`python/packages/durabletask/agent_framework_durabletask/`) and `agent-framework-azurefunctions` (`python/packages/azurefunctions/agent_framework_azurefunctions/`) (Python), and `Microsoft.Agents.AI.DurableTask` + `Microsoft.Agents.AI.Hosting.AzureFunctions` (`dotnet/src/Microsoft.Agents.AI.DurableTask`, `dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions`) (.NET).

Every recognized trigger is a Durable Task primitive: HTTP triggers dispatch through a Durable Task client into a Durable Entity that owns the conversation state (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:687`), orchestrations drive multi-agent and HITL flows (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:327`), entity triggers queue operations one at a time per session (`python/packages/durabletask/agent_framework_durabletask/_entities.py:335`), and MCP tool triggers fan into the same entity (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:875`). There is no in-process scheduler, no cron expression surface, and no built-in timer in the framework itself — the only "timed" trigger is the durable orchestrator's `create_timer` (`python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:134`).

Durability, idempotency, and replay semantics are strong: signals are routed through entities that serialize per-session access (`python/packages/durabletask/agent_framework_durabletask/_entities.py:335`), state is persisted after every conversation turn (`python/packages/durabletask/agent_framework_durabletask/_entities.py:173`), and TTL background deletion is implemented as a self-signal (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211`). Work is therefore safe across worker restarts, but it relies entirely on the underlying Durable Task scheduler and on a single-user, single-process persistent event loop in Python (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:33`) to keep async resources stable.

## Rating

**9/10 — Mature, durable, observable, extensible, and proven under failure or scale.**

The trigger model is one of the clearest parts of the codebase: every trigger reaches an entity keyed by a stable `AgentSessionId` (`python/packages/durabletask/agent_framework_durabletask/_models.py:219`), every dispatch has a `correlation_id` (`python/packages/durabletask/agent_framework_durabletask/_models.py:97`), custom status is published per superstep (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:754`), and there are extensive integration tests against the Durable Task Scheduler (`python/packages/durabletask/tests/integration_tests/`). The model is also explicitly designed for failure: an orchestrator can pause for human input indefinitely and resume days later (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:864`), TTL self-signals expire sessions safely (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:166`), and poison payloads are gated by the same pickle-marker sanitization in both HTTP and orchestrator paths (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:469`, `python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:885`).

It is not a 10 only because (a) the framework does not provide a built-in cron/timer trigger — orchestration timers must be combined with `Task.WhenAny`-style wrappers (`dotnet/src/Microsoft.Agents.AI.DurableTask/Workflows/DurableExecutorDispatcher.cs:102`), (b) duplicate HTTP triggers are not formally idempotent — duplicate requests both append a request entry to the durable state and rely on the caller to filter by `correlation_id` (`python/packages/durabletask/agent_framework_durabletask/_entities.py:151`), and (c) the persistent-event-loop bridge needed to keep async resources alive (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:33`) is an acknowledged workaround for a host limitation rather than a clean trigger abstraction.

## Answers to Dimension Questions

1. **What starts execution?**
   Four trigger families in production hosting (Azure Functions), all of which funnel into a single entity or orchestration: HTTP triggers (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:672-798`, `dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:256-349`), orchestration triggers (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:326`, `dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:224-233`), activity triggers (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:199-218`), entity triggers (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:830`), MCP tool triggers (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:832-889`, `dotnet/src/Microsoft.Agents.AI.DurableTask/BuiltInFunctions.cs:351-382`), and an optional health-check route (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:990-1010`). On standalone Durable Task, the same entity/activity/orchestration model is exposed via `DurableAIAgentWorker.add_agent` / `configure_workflow` (`python/packages/durabletask/agent_framework_durabletask/_worker.py:87-253`) and started by `DurableAIAgentClient.get_agent` / `DurableWorkflowClient.start_workflow` (`python/packages/durabletask/agent_framework_durabletask/_client.py:73`, `python/packages/durabletask/agent_framework_durabletask/_workflows/client.py:67`). There is no framework-level cron/timer trigger; the framework relies on the host's Durable Task primitives (an orchestrator can yield a timer via `ctx.create_timer(...)` at `python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:134`).

2. **Are triggers durable?**
   Yes, by construction. HTTP and MCP requests become signals (`client.SignalEntity(entity_instance_id, "run", run_request)` at `python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:739`) that Durable Task persists and re-delivers on worker restart. Workflow runs are `schedule_new_orchestration` instances (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:361`, `python/packages/durabletask/agent_framework_durabletask/_workflows/client.py:81`). Custom status (`publish_live_status`) is persisted after every superstep (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:754-768`). Even TTL expiration is a self-signaled deletion (`Context.SignalEntity(this.Context.Id, nameof(CheckAndDeleteIfExpired), options: new SignalEntityOptions { SignalTime = scheduledTime })` at `dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211-215`).

3. **Are duplicate triggers safe?**
   Yes at the delivery layer (Durable Task guarantees at-least-once with exactly-once entity semantics) but the framework does not deduplicate semantically: every `signal_entity("run", ...)` call appends a `DurableAgentStateRequest` to `conversation_history` (`python/packages/durabletask/agent_framework_durabletask/_entities.py:151`). Two duplicate HTTP calls therefore yield two state entries and two agent invocations; the HTTP client distinguishes them only by `correlation_id` (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:721`, `python/packages/durabletask/agent_framework_durabletask/_models.py:198-200`). Tests explicitly verify that `DurableAgentTask` ignores duplicate completion (`python/packages/durabletask/tests/test_executors.py:443-446`). Worker registration *rejects* duplicate agent names (`python/packages/durabletask/agent_framework_durabletask/_worker.py:114`; test at `python/packages/durabletask/tests/test_worker.py:69-78`), and `DurableWorkflowsFunctionMetadataTransformer` dedupes function names via a `HashSet<string>` (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/Workflows/DurableWorkflowsFunctionMetadataTransformer.cs:55-57`). Workflow-level status/respond endpoints are gated by `IsOrchestrationOwnedByWorkflow` so a leaked `instanceId` cannot read or perturb another workflow (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:680-691`).

4. **Can scheduled work be observed like interactive work?**
   Yes. The orchestrator publishes a per-superstep custom status (`{state, events?, pending_requests?}` at `python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:754-768`); HITL requests are surfaced in `pending_requests` so an external client can drive them via the respond endpoint (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:843-855`, `python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:441-487`). `DurableWorkflowClient.stream_workflow` polls custom status and yields typed `WorkflowEvent`s (`python/packages/durabletask/agent_framework_durabletask/_workflows/client.py:179-310`). Azure Functions hosts cap custom status at 16 KB so they omit `events` (`python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:55-69`). The .NET Functions host exposes a status JSON with `runtimeStatus`, `output`, `failureDetails`, and pending `RequestPort`s (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:83-116`). Entity calls are also observable via the normal Durable Task dashboard (`docs/features/durable-agents/README.md:46`, `dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:231`).

5. **Do background and foreground execution share semantics?**
   They share the **execution** model (the same `AgentEntity.run`, the same agent state, the same `correlation_id` tracking) but differ in the **caller contract**:
   - Foreground (default `wait_for_response=True`) blocks on `signal_entity` then polls until either a response with the matching `correlation_id` appears in state (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:1047-1086`) or the timeout (`DEFAULT_MAX_POLL_RETRIES=30`, `DEFAULT_POLL_INTERVAL_SECONDS=1.0` at `python/packages/durabletask/agent_framework_durabletask/_constants.py:28-29`).
   - Background (fire-and-forget, `wait_for_response=False`) immediately returns an acceptance envelope (`"Request accepted for processing (correlation_id: ...)"` at `python/packages/durabletask/agent_framework_durabletask/_executors.py:193-206`) and never polls. HTTP-layer this maps to `x-ms-wait-for-response: false` -> HTTP 202 (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:316-322`, `dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:256-349`). Orchestrator-side fire-and-forget yields a `PreCompletedTask` so the workflow itself does not block (`python/packages/azurefunctions/agent_framework_azurefunctions/_orchestration.py:201-217`).
   - Reliable streaming is a third mode: the `IAgentResponseHandler`/callback protocol runs alongside the entity return, allowing token-level delivery without blocking the entity call (`docs/features/durable-agents/README.md:197-205`, `python/packages/durabletask/agent_framework_durabletask/_callbacks.py:25-40`).
   - HITL is a fourth "block until event" mode: `wait_for_external_event(request_id)` paused the orchestrator indefinitely, with no built-in timeout unless the user wraps it (`python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:123-132`; .NET equivalent at `dotnet/src/Microsoft.Agents.AI.DurableTask/Workflows/DurableExecutorDispatcher.cs:104-130`).

   If the process restarts, durable execution resumes correctly: the durable task backend persists the orchestration history and entity state, on restart the orchestrator replays from the last checkpoint (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:757-758`), and in-flight HTTP callers resume polling on the same `correlation_id` once the entity catches up. The known cross-loop hang (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:1-89`) is contained: it is explicitly not part of the public trigger contract but a defensive workaround for the entity handler being invoked on arbitrary worker threads.

## Architectural Decisions

- **One entity, one session, one process state.** Every durable-agent session is a Durable Entity keyed by `dafx-{name}` with the session key embedded in the entity key (`python/packages/durabletask/agent_framework_durabletask/_models.py:227`, `dotnet/src/Microsoft.Agents.AI.DurableTask/AgentSessionId.cs`). Entity serialization guarantees single-threaded execution per session, eliminating concurrency races (`python/packages/durabletask/agent_framework_durabletask/_entities.py:335-353`).

- **Triggers are generators of signals, not direct callers.** Every external entry point (HTTP, MCP tool, queue, workflow node) eventually emits a `signal_entity` or `call_entity` against the entity. The same `AgentEntity.run` runs no matter the trigger (`python/packages/durabletask/agent_framework_durabletask/_entities.py:126`), so background and foreground triggers share the execution body but differ in their caller contract.

- **Replay-safe orchestrator with replay-safe logger and custom-status diffing.** Side effects that should only run once (publishing live status, raising external events) are gated by `ctx.is_replaying` (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:757`). The orchestrator publishes compact status every superstep and a richer `{state, pending_requests, events}` payload only when the host can carry it (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:733-768`).

- **Async-bridge as a cross-host workaround.** Because Durable Functions hosts invoke entity handlers on arbitrary worker threads (each one with its own thread-local event loop context), the Python implementation spins up a single persistent event loop on a daemon thread and runs every agent coroutine through `run_coroutine_threadsafe` so shared async resources (locks, connection pools) remain valid across invocations (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:33-89`).

- **TTL via self-signaled expiration, not polling.** Each entity's first interaction schedules a future `SignalEntity` call to itself with `SignalTime = scheduledTime` (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:211-215`). Subsequent interactions only bump `ExpirationTimeUtc`; the running check decides to delete or reschedule (`dotnet/src/Microsoft.Agents.AI.DurableTask/AgentEntity.cs:120-195`).

- **Workflow registration is plan-then-apply.** Both hosts (standalone durabletask via `DurableAIAgentWorker.configure_workflow`, Azure Functions via `AgentFunctionApp`) consume the same `WorkflowRegistrationPlan` from `plan_workflow_registration`, so the only thing that differs is the *mechanism* (`worker.add_entity` vs. `@app.entity_trigger`); the *decision* (agent -> entity, non-agent -> activity, single orchestrator) is identical (`python/packages/durabletask/agent_framework_durabletask/_workflows/registration.py:51-74`).

- **Function metadata transformers, not attribute-based registration.** Azure Functions isolated workers cannot use C#-attribute trigger decorators for dynamically-generated agents, so the .NET hosting layer implements `IFunctionMetadataTransformer` to inject entity, HTTP, MCP, activity, and orchestration triggers after the Functions runtime queries the worker for its metadata (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/DurableAgentFunctionMetadataTransformer.cs:18-87`, `dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/Workflows/DurableWorkflowsFunctionMetadataTransformer.cs:25-165`).

- **HITL pause as the canonical long-running primitive.** Orchestrators yield `wait_for_external_event(request_id)` and stay paused indefinitely; an external caller raises the event via `RaiseEventAsync` (.NET Functions) or `DurableWorkflowClient.send_hitl_response` (standalone) (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:122-193`, `python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:441-487`, `python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:863-900`). The same pause semantics is what enables "human-in-the-loop can wait days/weeks with zero compute" (`docs/features/durable-agents/README.md:13`).

## Notable Patterns

- **Correlation-id matched pickup.** Every HTTP/MCP request generates `self._generate_unique_id()` (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:721`); the HTTP poll loop reads entity state and matches responses by that exact id (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:1103`), so a stale completion cannot be misattributed to a fresh request.
- **Response callback + reliable streaming.** Streaming updates flow through `AgentResponseCallbackProtocol` (`python/packages/durabletask/agent_framework_durabletask/_callbacks.py:25-40`); entity callbacks fail-soft so callback exceptions do not derail the entity result (`python/packages/durabletask/agent_framework_durabletask/_entities.py:279-317`).
- **Host adapter pattern (`WorkflowOrchestrationContext`).** Shared generator-based orchestrator programs against an abstract protocol (instant id, replaying, agent/activity dispatch, when_all/when_any, external events, create_timer) implemented by `DurableTaskWorkflowContext` and `AzureFunctionsWorkflowContext` (`python/packages/durabletask/agent_framework_durabletask/_workflows/dt_context.py:30-103`, `python/packages/azurefunctions/agent_framework_azurefunctions/_workflow_af_context.py`).
- **Tool-property schema baked into MCP triggers.** Both Python (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:844-860`) and .NET (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/DurableAgentFunctionMetadataTransformer.cs:78-82`) emit identical `query` + `threadId` tool property JSON for the MCP tool trigger, keeping client integrations portable across hosts.
- **Replay-aware publishing.** Side effects (custom-status writes, pending-request publication) skip during replay so events are not re-emitted on resume (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:757-758, 868-892`).
- **Trigger-orchestrator cross-cutting security.** Status/respond endpoints gate by `WORKFLOW_ORCHESTRATOR_NAME` to prevent leaked `runId`s from accessing foreign orchestrations; untrusted inputs run through `strip_pickle_markers` (HTTP respond path and orchestrator HITL path) so pickle RCE attempts cannot reach `deserialize_value` (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:469`, `python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:885`).
- **Status-aware health check.** The Azure Functions health route returns a JSON describing every registered agent, including `http_endpoint_enabled` and `mcp_tool_enabled` flags — useful for ops dashboards that want to see which triggers are live (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:990-1010`).
- **Function-name sanitization.** Agent names with spaces or non-alphanumeric characters are sanitized to function-name-safe slugs (`http-Weather_Agent`) and prefixed with leading digits are renamed (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:1014-1028`).

## Tradeoffs

- **Cross-loop hangs warrant a workaround, not a fix.** The persistent event loop (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:33-71`) is a known workaround for entity handlers running on arbitrary worker threads. It works but couples the framework to a single in-process loop and complicates scaling.
- **No idempotency key on triggers.** Two duplicate HTTP `/run` requests each produce a new entry in `conversation_history` with no dedup; only `correlation_id` distinguishes them post-hoc. The framework expects the caller to enforce idempotency, or to use a single conversation thread (`python/packages/durabletask/agent_framework_durabletask/_entities.py:151`).
- **Custom-status size cap on Azure Functions.** Because Azure Functions caps Durable Functions custom status at 16 KB, the streaming event timeline is opt-in per host (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:731-733`, `python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:55-69`). Standalone Durable Task host publishes the full timeline; AF does not.
- **Streaming is "reliable", not true end-to-end.** The entity returns a single `AgentResponse`; streaming updates are pushed via callback during the run. A streaming client must consume those updates through a side channel (e.g., Redis Streams) and reconcile with the final response (`docs/features/durable-agents/README.md:197-205`).
- **Fire-and-forget returns before effect.** The acceptance response the client gets does not guarantee the entity will ever run successfully. Errors during processing surface only in the persisted state and via the optional callback (`python/packages/durabletask/agent_framework_durabletask/_executors.py:184-206`).
- **Custom status and dynamic shared-state diffs.** The orchestrator updates `shared_state` from each activity result inside the durable loop (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:301-313`); users can mutate it from inside an activity, which means a buggy activity can poison subsequent supersteps.
- **Function metadata transformers add an additional moving part.** Because Azure Functions isolated workers cannot use C# attribute decorators at runtime, every entity/HTTP/MCP trigger must be discovered through `IFunctionMetadataTransformer.Transform` (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/DurableAgentFunctionMetadataTransformer.cs:18-87`). Skipping it leaves the agent unreachable.
- **Registering the same workflow name under different apps requires namespace discipline.** Multiple workflows sharing a task hub must keep their `instanceId` and orchestrator names separate; the `IsOrchestrationOwnedByWorkflow` check is the only guard (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:680-691`, `python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:506-523`).
- **No explicit framework-level dead-letter / retry API.** Failures inside the entity are caught and converted to a `DurableAgentStateErrorContent`; orchestrations rely on the underlying Durable Task retry policy (default varies by host). No framework-side retry decorator is exposed.
- **TTL requires at least one worker available to delete on time.** If every worker is down when a TTL `SignalEntity` fires, deletion is delayed until one reattaches (`docs/features/durable-agents/durable-agents-ttl.md:147`).

## Failure Modes / Edge Cases

- **Worker eviction during agent run.** A worker that crashes mid-call drops in-flight signals; Durable Task re-delivers them on the next worker, and the entity resumes with the latest persisted state (`python/packages/durabletask/agent_framework_durabletask/_entities.py:172-173`).
- **Orphan TTL signals in an immutable trace.** TTL is wall-clock-based; if a signal that was scheduled lands on a worker that has been recycled, deletion is delayed rather than dropped (`docs/features/durable-agents/durable-agents-ttl.md:144-147`).
- **Multi-message race.** Two HTTP requests routed to the same session ID but with different `correlation_id`s will queue at the entity; the second one will run only after the first completes (`python/packages/durabletask/agent_framework_durabletask/_entities.py:335`).
- **HTTP caller timeout smaller than agent run.** `max_poll_retries * poll_interval_seconds` (defaults: `30 * 1s = 30s`) may be shorter than the agent run; the HTTP response will say `status=timeout` and `response="Agent is still processing or timed out..."` while the entity continues (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:1141-1149`). The framework accepts this and does not abort the durable run.
- **Fire-and-forget discard.** A `wait_for_response=False` HTTP caller gets HTTP 202 with the `correlation_id`; if the agent later fails, the failure stays in durable state and there is no built-in callback to surface it to the original caller (`python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:762-771`, `python/packages/durabletask/agent_framework_durabletask/_executors.py:255-260`).
- **Pickle-marker injection via HITL payloads.** Hit by-design but mitigated: rejected responses do not consume the request, so the same call can be retried (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:881-893`).
- **Cross-orchestration ID probing.** Mitigated by `IsOrchestrationOwnedByWorkflow` checks for status/respond endpoints (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:680-691`, `python/packages/azurefunctions/agent_framework_azurefunctions/_app.py:506-523`).
- **Async cross-loop hangs on entity coroutines.** Mitigated via `run_agent_coroutine` pinning all entity coroutines to the shared loop (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:73-89`).
- **Async edges in orchestrations.** Edge conditions that return awaitables are deliberately dropped rather than blocking, because the durable host cannot evaluate async conditions (`python/packages/durabletask/agent_framework_durabletask/_workflows/orchestrator.py:140-154`).
- **Direct raise_event without validation status.** `RespondToWorkflowAsync` accepts an event during early execution (when no live status is yet set) because the orchestrator safely queues it until `WaitForExternalEvent` is reached (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:165-180`).
- **`activity_trigger` validation gaps.** Non-agent workflow executors are activities; if a developer registers an executor with no input schema, the activity runs with `input` as a JSON-encoded string and the dispatch is via `WorkflowOrchestrationContext.prepare_activity_task` (`python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:89-99`).
- **`run(stream=True)` not supported by every agent.** Fallback is automatic: `AgentEntity._invoke_agent` falls back to `run()` when streaming raises a `TypeError` mentioning `__aiter__` or `stream` (`python/packages/durabletask/agent_framework_durabletask/_entities.py:238-251`).
- **Validation gaps on content rejection during state replay.** Conversation history entries that contain only `reasoning` content are skipped during replay (`python/packages/durabletask/agent_framework_durabletask/_entities.py:195-208`), which means a model that only emits reasoning produces no entry on replay — observable as an empty history for that turn.
- **Activity context cannot resolve HITL sends.** The `CapturingRunnerContext` (used by Azure Functions activities) explicitly does not support `send_request_info_response` — HITL must be resolved at the orchestrator level (`python/packages/azurefunctions/agent_framework_azurefunctions/_context.py:166-176`).
- **Scheduling with empty input.** An HTTP `/run` with no `message` and no `thread_id` falls back to `context.InvocationId` as the session key (.NET) or generates a random one (Python); both yield a fresh session rather than a shared one (`dotnet/src/Microsoft.Agents.AI.Hosting.AzureFunctions/BuiltInFunctions.cs:301-304`).

## Future Considerations

- **Cron-style triggers.** No cron-style or scheduled-time trigger is documented. Build a `TimerTrigger -> schedule_new_orchestration` recipe at the host level (.NET Functions does provide a `TimerTrigger`, but the framework does not auto-wire it). Orchestrations can already yield `ctx.create_timer(fire_at)` (`python/packages/durabletask/agent_framework_durabletask/_workflows/context.py:134`).
- **Idempotency keys for HTTP triggers.** Add a request-level idempotency key (e.g., `Idempotency-Key` header) that the entity checks before appending, to make duplicate triggers a no-op rather than two history entries.
- **Streaming-event backlog overflow.** Azure Functions custom status is 16 KB capped; long-running fan-outs will overflow. A side-channel event sink (Service Bus / Event Grid) could replace custom status for long histories.
- **Backpressure across worker processes.** Today the Python bridge assumes one process owns one event loop (`python/packages/durabletask/agent_framework_durabletask/_async_bridge.py:33`). Multi-worker deployments need a per-process bridge; consider a `run_agent_coroutine_in_process` API.
- **TTL granularity.** The minimum signal delay is bounded at 5 minutes today (`docs/features/durable-agents/durable-agents-ttl.md:33`). For shorter-lived agents, support per-agent minimum delays.
- **Trigger observability.** The framework relies on the host's Durable Task dashboard for entity/orchestration traces (`docs/features/durable-agents/README.md:212-218`). A first-class OTel/Application Insights trigger span would shorten mean-time-to-diagnose.
- **Stateless fire-and-forget callbacks.** A `wait_for_response=False` HTTP caller currently has no built-in way to receive a completion notification beyond polling; first-class `notify_url` or Service Bus push sinks would close the loop.
- **Trigger authorization scope.** `enable_http_endpoints`/`enable_mcp_tool_trigger` are booleans; splitting these into per-agent scopes (with auth level / RBAC) would let one app safely expose a subset of agents.
- **Test coverage for restart.** Integration tests do not kill the worker mid-run (verified by `python/packages/durabletask/tests/integration_tests/`); adding a "kill worker, expect replay-and-resume" scenario would prove the durability claim end-to-end.

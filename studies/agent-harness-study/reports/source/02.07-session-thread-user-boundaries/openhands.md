## Summary

OpenHands is in the middle of a V0 → V1 migration, and the answer to "how is state scoped across users, threads, sessions, runs?" differs sharply between the two deployment modes that share this repo:

1. **OSS / single-tenant mode (`AppMode.OPENHANDS`).** The app_server ships with `DefaultUserAuth` whose `get_user_id()` always returns `None` (`openhands/app_server/user_auth/default_user_auth.py:32-34`), and both `FileSettingsStore.get_instance` and `FileSecretsStore.get_instance` *ignore* their `user_id` argument and return one shared store per process (`openhands/app_server/settings/file_settings_store.py:48-56`, `openhands/app_server/secrets/file_secrets_store.py:37-44`). The base SQL conversation store (`SQLAppConversationInfoService._secure_select`) only filters by `conversation_version == 'V1'` and does not include a `user_id` predicate (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:514-518` — the docstring even comments "filters for V1 and user if available", but the user filter is never actually applied). The base remote sandbox store does filter by `created_by_user_id` when a user is present (`openhands/app_server/sandbox/remote_sandbox_service.py:208-213`), but in OSS mode the column is always `None` and the filter is a no-op. **In OSS mode, a process-local multi-user deployment would let any caller see any other caller's conversations, settings, and secrets** — that is by design, not a bug, because OSS is single-tenant.

2. **SaaS / multi-tenant mode (`AppMode.SAAS`, the `enterprise/` extension).** Identity is a `SaasUserAuth` carrying a Keycloak `user_id`, an optional `api_key_org_id`, an optional `X-Org-Id` header, and a `user.current_org_id`; the resolved *effective* org follows a documented precedence (`enterprise/server/auth/saas_user_auth.py:171-274`, `enterprise/server/auth/org_context.py:1-22`). The store hierarchy enforces user + org filters at SQL level: `SaasSQLAppConversationInfoService._apply_user_and_org_filter` joins through `StoredConversationMetadataSaas` (an explicit `(user_id, org_id)` mapping) and raises `AuthError` if `user_id` is missing — secure default deny (`enterprise/server/utils/saas_app_conversation_info_injector.py:51-91`). `SaasSecretsStore` and `SaasSettingsStore` likewise key off `(user_id, org_id)` (`enterprise/storage/saas_secrets_store.py:28-72`, `enterprise/storage/saas_settings_store.py:68-87`). Pending messages validate ownership per request (`enterprise/server/utils/saas_pending_message_injector.py:66-115`). Sandboxes carry a `created_by_user_id` and are looked up via a hashed `X-Session-API-Key`, but session keys are *only* honored for `SandboxStatus.RUNNING` sandboxes (`openhands/app_server/sandbox/session_auth.py:73-87`) and an additional `validate_session_key_ownership` cross-checks the key's owning user against the authenticated caller (`openhands/app_server/sandbox/session_auth.py:103-139`).

3. **Thread / sub-conversation boundary.** Conversations have UUIDs and an optional `parent_conversation_id` + `sub_conversation_ids: list[OpenHandsUUID]` (`openhands/app_server/app_conversation/app_conversation_models.py:131-132`). Sub-conversations inherit the parent's `sandbox_id`, git repo/branch, and LLM model via `_inherit_configuration_from_parent` (`openhands/app_server/app_conversation/live_status_app_conversation_service.py:1014-1042`), and are excluded from default listings via a `parent_conversation_id IS NULL` predicate (`openhands/app_server/app_conversation/sql_app_conversation_info_service.py:148-151`). There is **no general-purpose cross-conversation memory / recall service** in this repo: a grep for `MEMORY`, `recall`, or `cross.thread` finds no production code path (only test fixtures). Per-user memory is per-conversation; the only cross-conversation state that exists is `Settings`, `Secrets`, and the persisted event stream.

4. **Sandbox-as-run, session-as-auth.** A sandbox is the runtime container, not the conversation thread; one sandbox can host multiple conversations (controlled by `max_num_conversations_per_sandbox` and the `SandboxGroupingStrategy` enum, `openhands/app_server/app_conversation/live_status_app_conversation_service.py:753-774`). The `session_api_key` is a sandbox-scoped credential, not a per-conversation or per-user credential; it identifies a sandbox and (in SAAS) is tied to a `created_by_user_id`. The `valid_sandbox` dependency in the webhook router uses the session key as proof of user identity by setting a `SpecifyUserContext(sandbox_record.created_by_user_id)` on the request (`openhands/app_server/event_callback/webhook_router.py:235-279`).

5. **Tenant boundaries at storage level.** Yes — every SAAS-protected store has a `user_id` predicate *and* an `org_id` predicate in its `WHERE` clause, both backed by a `(user_id, org_id)` index added in migration 118 (`enterprise/migrations/versions/118_add_conversation_metadata_saas_indexes.py:1-56`). The OSS store has no such filter.

## Rating

**7 / 10** — Clear model with tests, explicit interfaces, and operational safeguards for the multi-tenant case; weaker on the single-tenant seam and on cross-conversation memory.

Rationale (see "Notable Patterns" / "Tradeoffs" below for evidence):

- **Multi-tenant SAAS path is genuinely well-engineered (9/10 in isolation):** a documented precedence chain for "effective org id" (`saas_user_auth.py:171-274`), an "AuthError on missing user" secure default (`saas_app_conversation_info_injector.py:76-78`), `created_by_user_id` filters on every store, a `(user_id, org_id)` composite index in migration 118, and the session-key re-validation against the authenticated user on every sandbox-scoped call (`session_auth.py:103-139`).
- **V0 → V1 migration leaks the user_id across two tables** (`conversation_metadata.created_by_user_id` is no longer the source of truth; SAAS overrides it from `conversation_metadata_saas.user_id` at `sql_app_conversation_info_service.py:552-554` and `_to_info_with_user_id` at `saas_app_conversation_info_injector.py:415-430`). The wiring works but the comment at `sql_app_conversation_info_service.py:552` is itself evidence that this is a known inconsistency.
- **Single-tenant path is silent on cross-user leakage.** `SQLAppConversationInfoService._secure_select` does not filter by user; the docstring even claims it does (`sql_app_conversation_info_service.py:514-518`). The "secure default" guarantee exists *only* in the SAAS subclass. If someone deploys OSS over a shared DB and bolts on a non-default `UserAuth` that returns a real `user_id`, the conversations table will still leak unless they also swap the conversation info injector.
- **No cross-conversation memory service exists in this repo.** "Memory" is per-conversation (events list). The question "is cross-thread memory explicit?" is answered by absence: there is no shared memory store, so no implicit leakage is possible — but there is also no explicit cross-conversation feature.
- **Sub-conversations are safe by construction** (parent_conversation_id index at `sql_app_conversation_info_service.py:109-111`, excluded from default listings at line 150), but the inherit-from-parent logic never re-validates that the *caller* owns the parent (`live_status_app_conversation_service.py:320-331`). The owning check happens later via the SAAS filter on `save_app_conversation_info`, which raises only via assertion (`saas_app_conversation_info_injector.py:397-400`) — not via a 401.

## Answers to Dimension Questions

### 1. What scopes state?

State is scoped by a layered set of identifiers, each with a distinct lifetime:

- **`user_id`** — the Keycloak `sub` claim in SAAS (`enterprise/server/auth/saas_user_auth.py:59`, `:276-277`); `None` in OSS (`openhands/app_server/user_auth/default_user_auth.py:32-34`). Owns: settings (`SaasSettingsStore`), custom secrets (`SaasSecretsStore`), conversations (via `conversation_metadata_saas.user_id`), sandboxes (`v1_remote_sandbox.created_by_user_id`), API keys (`api_keys.user_id`).
- **`org_id`** (tenant) — `Org.id` (`enterprise/storage/org.py:29-77`). Each user belongs to one or more orgs through `OrgMember(org_id, user_id)` (`enterprise/storage/org_member.py:20-27`); `User.current_org_id` is the default. The "effective" org is `api_key_org_id > X-Org-Id header > user.current_org_id` (`enterprise/server/auth/saas_user_auth.py:171-274`).
- **`conversation_id`** — `AppConversationInfo.id: OpenHandsUUID` (`openhands/app_server/app_conversation/app_conversation_models.py:115`). Owns: events (`{user_id}/v1_conversations/{conversation_id.hex}/*.json` at `openhands/app_server/event/event_service_base.py:66-84`), title, repo/branch, parent_conversation_id, sub_conversation_ids.
- **`sandbox_id`** — a base62 string, generated by `base62.encodebytes(os.urandom(16))` at `openhands/app_server/sandbox/remote_sandbox_service.py:441`, persisted in `v1_remote_sandbox.id` (`remote_sandbox_service.py:81`). A sandbox is the runtime container; it can host multiple conversations via `SandboxGroupingStrategy` (`live_status_app_conversation_service.py:753-774`). The sandbox's `session_api_key` is a hashed (SHA-256) bearer secret that authenticates only that sandbox's calls (`remote_sandbox_service.py:62-70`).
- **`session_api_key`** — a sandbox-scoped credential. Lives only as a SHA-256 hash in `v1_remote_sandbox.session_api_key_hash` (`remote_sandbox_service.py:90-92`) and as the plaintext `SandboxInfo.session_api_key` (only returned for `SandboxStatus.RUNNING` per `sandbox_models.py:55-62`). It is **not** a user or conversation identity — it is a sandbox identity that *implies* the owner's identity via `sandbox.created_by_user_id` (`openhands/app_server/sandbox/session_auth.py:89-100`).
- **`run_id` / conversation turn id** — not separately modeled. The `AppConversationStartTask.id` is a transient UUID that becomes the `AppConversation.id` once the agent_server reports back via webhook (`app_conversation_router.py` and `webhook_router.py:333+`).
- **No `workspace_id`, no `tenant_id`, no `project_id`.** The closest analog to a "workspace" is the `org`. Jira integrations have `jira_dc_workspace_id` (e.g. `enterprise/integrations/jira_dc/jira_dc_conversation_secret_enricher.py:71,81`) but those are external ids, not internal scope identifiers.

### 2. Can state leak across users or sessions?

- **SAAS mode, when wired correctly: no.** Every multi-tenant store raises `AuthError` on missing `user_id` (`saas_app_conversation_info_injector.py:76-78`) and applies both `user_id` and `org_id` predicates (`saas_app_conversation_info_injector.py:81-89`). Pending messages cross-check the requested `conversation_id` against `conversation_metadata_saas.user_id` and `org_id` (`saas_pending_message_injector.py:66-115`). The session-key path re-validates ownership against the authenticated caller on every request (`session_auth.py:103-139`).
- **SAAS mode has one escape hatch: `ADMIN`.** `SpecifyUserContext(user_id=None)` (`specifiy_user_context.py:52`) is installed by the webhook router (`webhook_router.py:251-252`) and by all SAAS-side V1 callback processors (`github_v1_callback_processor.py:252`, `jira_v1_callback_processor.py:113`, `slack_v1_callback_processor.py:231`, etc.). The SAAS conversation service deliberately *skips* `_apply_user_and_org_filter` when `user_context == ADMIN` (`saas_app_conversation_info_injector.py:70-73`) so that webhook callbacks can look up conversations by session key without impersonating a user. This is the standard admin bypass pattern; it is gated by possession of the session key (which the running sandbox holds) and by the implicit trust that a sandbox only calls webhooks for its own data.
- **OSS mode (default `DefaultUserAuth`): trivially yes — and that is by design.** `DefaultUserAuth.get_user_id()` always returns `None` (`default_user_auth.py:32-34`), so any user-context-aware filter is a no-op; the file stores ignore `user_id` (`file_settings_store.py:48-56`, `file_secrets_store.py:37-44`); the base `SQLAppConversationInfoService._secure_select` does not include a `user_id` predicate (`sql_app_conversation_info_service.py:514-518`). The `SetAuthCookieMiddleware` in `enterprise/server/middleware.py` reads a Keycloak cookie, but in OSS mode that middleware is not loaded. OSS is documented as a single-host single-user product (`README.md:69-79`, `AGENTS.md` setup notes); it is not safe to multi-tenant.
- **Sandbox↔conversation bind.** A conversation is bound to its `sandbox_id` at creation (`live_status_app_conversation_service.py:479-497`), and `_inherit_configuration_from_parent` reuses the parent's sandbox for sub-conversations (`live_status_app_conversation_service.py:1028-1030`). In `_find_running_sandbox_for_user`, a new conversation is allowed to join an existing running sandbox only if `sandbox.created_by_user_id == user_id` (`live_status_app_conversation_service.py:696-701`). So a user can be put onto a sandbox they own but never onto one they don't.

### 3. Is cross-thread memory explicit?

There is no cross-conversation memory service in this repo. The grep for `MEMORY|recall|cross.thread|cross_session` over `sources/openhands/` returns no production code (only `enterprise/dev_config/python/mypy.ini` `[mypy-openhands.memory.condenser.impl.*]` referring to a memory condenser in the SDK, which is per-conversation). State that does span conversations is limited to:

1. **Settings / secrets** (`SaasSettingsStore`, `SaasSecretsStore`) — keyed by `(user_id, org_id)`, scoped to a user, not a thread.
2. **LLM profiles** — live on `org.llm_profiles` in SAAS (`enterprise/storage/saas_settings_store.py:226-256`), with explicit "this org's profiles are available to all members" semantics.
3. **Sub-conversations** — `parent_conversation_id` is the only thread-of-threads relation, and it is used for cascade delete (`live_status_app_conversation_service.py:2203-2236`) and config inheritance (`live_status_app_conversation_service.py:1014-1042`), not for sharing memory.
4. **Shared conversations** — `AppConversationInfo.public: bool \| None` (`app_conversation_models.py:134`) plus the unauthenticated `/api/shared-conversations` batch endpoint (`enterprise/server/sharing/shared_conversation_router.py:31-45`). The shared router only exposes conversations whose `public == True` and joins through `conversation_metadata_saas` to read the owning user_id for path resolution (`sql_shared_conversation_info_service.py:46-100`). Sharing is opt-in and explicit.

So the answer is: **no implicit cross-thread memory**; **no explicit cross-thread memory either**, only sharing of conversation artifacts (read-only) and inheriting of config (write-new-thread).

### 4. Are tenant boundaries enforced at storage level?

Yes, in SAAS, at the SQL `WHERE` clause level:

- `SaasSQLAppConversationInfoService._secure_select` joins through `StoredConversationMetadataSaas` and applies `user_id = :uid AND org_id = :oid` (`saas_app_conversation_info_injector.py:106-116`).
- `SaasSecretsStore` filters `StoredCustomSecrets` by both `keycloak_user_id == :uid AND org_id == :oid` and **deletes-and-replaces** on `store()` so secrets cannot leak across orgs via the per-(user,org) row (`saas_secrets_store.py:36-72`).
- `SaasSettingsStore.load()` reads `Org`, then `OrgMember` for the calling `(user_id, org_id)`, and returns `None` if the user is not a member (`saas_settings_store.py:142-156`).
- `StoredConversationMetadataSaas` has a composite `(user_id, org_id)` index added in migration 118 specifically to keep user_id/org_id lookups off a full table scan (`enterprise/migrations/versions/118_add_conversation_metadata_saas_indexes.py:1-56`).
- `SaasPendingMessageService` validates ownership per-call (`saas_pending_message_injector.py:66-115`).

In OSS, the boundary is **the process itself** — there is no multi-tenant storage enforcement.

### 5. Can users fork sessions safely?

Partially. `AppConversationStartRequest.parent_conversation_id` (`app_conversation_models.py:222`) is the fork hook. Sub-conversations inherit `sandbox_id`, repo/branch, and LLM model (`live_status_app_conversation_service.py:1014-1042`), and the resulting row's `parent_conversation_id` is set at `live_status_app_conversation_service.py:493`.

Safety considerations:
- **Fork reads the parent through the standard SQL filter.** If you know a conversation_id and start a sub-conversation pointing at it, the call goes through `_start_app_conversation` → `get_app_conversation_info(parent_conversation_id)` (line 323), which in SAAS is filtered by `(user_id, org_id)`. So a user *cannot* fork someone else's conversation, because the lookup returns `None` and the start raises `ValueError(f'Parent conversation not found: ...')` (`live_status_app_conversation_service.py:327-331`).
- **Cascade delete is per-conversation, not per-parent.** `_delete_app_conversation` only deletes the requested conversation; deleting a parent does **not** automatically cascade to its sub-conversations in the standard `delete_app_conversation` path — the helper `_delete_sub_conversations` (`live_status_app_conversation_service.py:2203-2236`) exists but I did not see it called from the public `delete_app_conversation` route, which means orphaned sub-conversations are possible if a parent is deleted out of band.
- **Sub-conversation listing default hides them.** `search_app_conversations` excludes rows where `parent_conversation_id IS NOT NULL` (`sql_app_conversation_info_service.py:148-151`), so forks are hidden from the default list but still reachable by direct id.
- **The fork does not copy conversation history** — it starts a new conversation in the same sandbox with the same config. There is no "clone state" feature; the new conversation begins empty (events will accumulate from the agent server's point of view).

## Architectural Decisions

- **Two-tier user model:** `UserContext` is an abstract base (`openhands/app_server/user/user_context.py:13`) with three concrete implementations: `AuthUserContext` (production auth, `auth_user_context.py:27-166`), `SpecifyUserContext` (explicit user_id override for background/admin tasks, `specifiy_user_context.py:13-49`), and the `ADMIN` sentinel (`specifiy_user_context.py:52`). The injector pattern (`AuthUserContextInjector.inject`, `auth_user_context.py:152-166`) caches one `AuthUserContext` per request via `request.state`.
- **Two-tier conversation store:** `SQLAppConversationInfoService` is a pure DB CRUD layer with a `_secure_select` that "filters for V1 and user if available" (`sql_app_conversation_info_service.py:1-16, 514-518`). The SAAS variant `SaasSQLAppConversationInfoService` overrides `_secure_select` to join through `conversation_metadata_saas` and apply user+org predicates (`saas_app_conversation_info_injector.py:106-116`). The class comment in the base file is explicit: "Security and permission checks are handled by wrapper services" (`sql_app_conversation_info_service.py:5-12`) — but the wrapper is the SAAS subclass, not a general-purpose check.
- **Sandbox as the unit of authorization for runtime callbacks.** Webhooks from the agent_server carry only an `X-Session-API-Key`. The app_server trusts the key only for RUNNING sandboxes (`session_auth.py:73-87`) and then synthesizes a `SpecifyUserContext(sandbox_owner)` for the rest of the request (`webhook_router.py:264-269`). This avoids having to send Keycloak tokens inside a sandbox.
- **Effective org resolution is a single precedence chain.** Documented in `enterprise/server/auth/org_context.py:1-22` and `enterprise/server/auth/saas_user_auth.py:171-202`: API-key org binding > `X-Org-Id` header > `user.current_org_id`. The resolved value is cached on `SaasUserAuth._effective_org_id` and invalidated by `set_effective_org_id_override` (`saas_user_auth.py:101-119`) so background resolver contexts can pin to a specific org without re-validating.
- **Member-private keys are split out of org-level settings.** `MEMBER_PRIVATE_AGENT_KEYS` (`saas_settings_store.py:43`) — currently `mcp_config` and `acp_env` — are routed through `_split_member_private_keys` (`saas_settings_store.py:46-65`) so they are written only to the acting member's `agent_settings_diff` and broadcast copies are stripped from the org dump before merge (`saas_settings_store.py:176-184`, `:388-396`). This is an explicit "private-by-default" decision.
- **Session key is not a long-lived credential.** The comment block at `session_auth.py:1-16` states that session keys are "only valid while the sandbox is RUNNING" and `validate_session_key` enforces that with a 401 on non-RUNNING sandboxes (`session_auth.py:73-87`). Combined with the storage of only the SHA-256 hash (`remote_sandbox_service.py:62-70`), the design treats session keys as a sandbox-lifetime proof, not a bearer token.
- **Sharing is opt-in via `public: bool`.** `AppConversationInfo.public` (`app_conversation_models.py:134`) gates the unauthenticated `/api/shared-conversations` endpoint (`shared_conversation_router.py:31-45`). The shared service still joins through `conversation_metadata_saas` to read the owner for event storage paths (`sql_shared_conversation_info_service.py:46-100`), so ownership metadata is preserved even on public conversations.

## Notable Patterns

- **The `ADMIN` sentinel + `user_context == ADMIN` check** is the central pattern for bypassing per-user filters in background/integration code paths. Used in webhook entry (`webhook_router.py:251-252`), every SAAS-side V1 callback processor (`github_v1_callback_processor.py:252`, `jira_v1_callback_processor.py:113`, `jira_dc_v1_callback_processor.py:123`, `bitbucket_v1_callback_processor.py:155`, `bitbucket_dc_v1_callback_processor.py:157`, `gitlab_v1_callback_processor.py:239`, `slack_v1_callback_processor.py:231`), and in `session_auth.py:60-66` for session-key lookups. The SAAS filter explicitly recognises the sentinel: `if self.user_context == ADMIN: return query` (`saas_app_conversation_info_injector.py:70-73`).
- **Secure-default-deny on missing user.** `_apply_user_and_org_filter` raises `AuthError('User authentication required')` when `get_user_id()` returns `None` (`saas_app_conversation_info_injector.py:76-78`); `_validate_conversation_ownership` does the same (`saas_pending_message_injector.py:84-86`). Same pattern in `session_auth.py:124-128`.
- **Composite indexes for the hot path.** Migration 118 explicitly adds `(user_id, org_id)` and `(org_id)` indexes to `conversation_metadata_saas` because the existing PK-only index caused ~680K seq scans and ~42B rows read in production (`enterprise/migrations/versions/118_add_conversation_metadata_saas_indexes.py:5-12`).
- **Two-stored-row pattern for the V0 → V1 split.** `conversation_metadata` (owned by the V1 app_server) and `conversation_metadata_saas` (owned by SAAS) carry overlapping `conversation_id` rows. SAAS-reads always go through `SaasSQLAppConversationInfoService._secure_select_with_saas_metadata` which returns both rows in a tuple; SAAS-writes do a base save + a SAAS save in `save_app_conversation_info` (`saas_app_conversation_info_injector.py:353-413`). The `assert existing is None or existing.user_id == user_id_uuid` at line 397 is a defense-in-depth check that the row's owner has not changed.
- **Effective-org-id caching per request.** Once `SaasUserAuth.get_effective_org_id()` resolves, `_effective_org_id_resolved = True` prevents repeated membership lookups (`saas_user_auth.py:192-202`); caches for org-scoped settings/secrets/MCP keys are cleared in one place via `_clear_org_scoped_caches` (`saas_user_auth.py:106-119`).
- **Hybrid keycloak-cookie / bearer-api-key auth.** `SaasUserAuth.get_instance` first tries an API key (`saas_user_auth_from_bearer`, `saas_user_auth.py:676-701`), then a Keycloak cookie (`saas_user_auth_from_cookie`, `saas_user_auth.py:704-711`). The two paths populate the same `SaasUserAuth` dataclass but the BEARER path deliberately *does not* load an offline token and degrades gracefully in `get_access_token` (`saas_user_auth.py:404-424`) — a 401 from a missing offline session is converted to `None` for API-key callers.
- **Authorization as a typed dependency.** `require_permission(Permission)` is a FastAPI dependency that reads the path's `org_id`, fetches `user_role` for that org, and either returns the `user_id` or raises 403 (`enterprise/server/auth/authorization.py:246-341`). Permission enum covers MANAGE_SECRETS, EDIT_LLM_SETTINGS, INVITE_USER_TO_ORGANIZATION, etc. (`authorization.py:47-91`).
- **Sandbox grouping as user-isolated** by `_find_running_sandbox_for_user` which only joins sandboxes whose `created_by_user_id == user_id` (`live_status_app_conversation_service.py:696-701`).

## Tradeoffs

- **User_id is duplicated in two tables.** `conversation_metadata.created_by_user_id` exists but is no longer authoritative; SAAS treats `conversation_metadata_saas.user_id` as truth (`saas_app_conversation_info_injector.py:425-428`). The base class comment "User ID is now stored in ConversationMetadataSaas" at `sql_app_conversation_info_service.py:552-554` is a literal admission of this duplication. The cost is one extra join on every conversation read; the benefit is that OSS can run without the SAAS table.
- **OSS-default is "no auth, single tenant."** That is the right product decision for self-hosted, but the seam is silent: nothing in the OSS code path raises if a second `user_id` starts appearing. A future "multi-user OSS" feature would have to either replace `SQLAppConversationInfoService` with a user-aware version or reintroduce the filter in `_secure_select`.
- **`session_api_key` is the runtime's identity proof, not the user's.** A compromised sandbox process has access to its own session key and can therefore authenticate as its owner (`webhook_router.py:264-269`). Mitigation: the key is only valid for RUNNING sandboxes (`session_auth.py:73-87`) and only hashes are persisted (`remote_sandbox_service.py:62-70`), so a key disclosed after sandbox shutdown is useless.
- **Pending-message validation is per-call, not pre-validated.** `add_message` and friends re-look up `conversation_metadata_saas` on every call (`saas_pending_message_injector.py:91-95`), which is correct but adds a round trip per message.
- **Forking inherits sandbox but not history.** `_inherit_configuration_from_parent` copies config but not events (`live_status_app_conversation_service.py:1014-1042`); a fork is a fresh conversation in the same workspace. This is the right default for an agent harness — copying events would be a memory leak — but it means users cannot "clone and continue" the agent's reasoning, only the environment.
- **Sharing bypasses per-user filtering but still records ownership.** Public conversations are readable without auth (`shared_conversation_router.py:31-45`), but the SAAS join preserves the original `created_by_user_id` and `org_id` for downstream path resolution (`sql_shared_conversation_info_service.py:60-100`). There is no separate `shared_with: list[user_id]` model; sharing is binary public/private.
- **Sub-conversation lifecycle is half-managed.** `_delete_sub_conversations` exists (`live_status_app_conversation_service.py:2203-2236`) but it is not invoked from the public `delete_app_conversation` route; the `_secure_select` excludes them from default listings (`sql_app_conversation_info_service.py:148-151`); and there is no API endpoint that surfaces a parent's `sub_conversation_ids`. A parent can therefore be deleted leaving orphan sub-conversations visible only via direct ID.

## Failure Modes / Edge Cases

- **OSS `SQLAppConversationInfoService._secure_select` silently returns all users' conversations** when `user_id` is `None` (always in OSS). Symptom: `GET /api/v1/app-conversations/search` returns every conversation in the database to every caller. Evidence: `sql_app_conversation_info_service.py:514-518`.
- **Stale `current_org_id` after a user switches orgs.** Without an `X-Org-Id` override, requests resolve to the user's last-set `current_org_id`. SAAS mitigation: API-key-bound keys and the `X-Org-Id` header both take precedence over `current_org_id` (`saas_user_auth.py:200-274`, `org_context.py:1-22`); secrets and settings stores also accept an explicit `effective_org_id` (`saas_secrets_store.py:25-32`, `saas_settings_store.py:71-76`).
- **API key org binding vs. path org mismatch.** `require_permission` and `get_effective_org_id` both raise 403 if `api_key_org_id != path_org` (`authorization.py:284-298`, `saas_user_auth.py:220-236`).
- **`conversation_metadata_saas` may not exist for a brand-new task-id.** `_validate_conversation_ownership` allows the lookup to return `None` for the early-task window before the conversation has been persisted (`saas_pending_message_injector.py:97-101`); the message is validated later when the conversation row is created. This is a deliberate "fail-open for not-yet-existent ids" trade.
- **`ADMIN` bypass is enabled by `user_context == ADMIN` identity check.** If a request path sets `state.USER_CONTEXT_ATTR = ADMIN` (e.g. a misconfigured middleware), all SAAS filters are skipped. Mitigation: `as_admin` in `specifiy_user_context.py:55-65` raises if a non-admin context is already on the request, forcing ordering.
- **`save_app_conversation_info` uses `assert`** to verify ownership invariant (`saas_app_conversation_info_injector.py:397-400`). Assertions are stripped under `python -O`; this is a defense-in-depth check, not the primary guard.
- **`_inherit_configuration_from_parent` does not re-validate parent ownership** at the call site (`live_status_app_conversation_service.py:320-331`); it relies on the SAAS filter applied inside `get_app_conversation_info`. If a future caller passes a parent conversation id they do not own, they will get a `ValueError('Parent conversation not found')` rather than a 403, which leaks existence but not contents.
- **Session key disclosed after sandbox pause/stop.** `validate_session_key` rejects non-RUNNING sandboxes (`session_auth.py:73-87`) so a paused/stopped sandbox's leaked key cannot be replayed.
- **Sandbox sharing across users.** `_find_running_sandbox_for_user` enforces `created_by_user_id == user_id` (`live_status_app_conversation_service.py:696-701`); the only way to share a sandbox is for the owner to expose it explicitly (e.g. by handing over session_api_key, which would then fail `validate_session_key_ownership` unless the new user is the same as the owner).
- **No default cap on conversations per sandbox.** Bounded by `max_num_conversations_per_sandbox` (`live_status_app_conversation_service.py:746-747`) but a misconfiguration could allow many users' conversations to share one sandbox. Today the user filter prevents cross-user sharing, so this is a resource cap not a security one.

## Future Considerations

- **Add a user_id predicate to the base `SQLAppConversationInfoService._secure_select`** behind an OSS-mode flag, or split the OSS path explicitly. The current docstring promises a filter that does not exist; either ship the filter or update the comment (`sql_app_conversation_info_service.py:514-518`).
- **Wire `_delete_sub_conversations` into the public delete path** so deleting a parent cleans up its children (`live_status_app_conversation_service.py:2203-2236`).
- **Add a per-conversation owner check** at the fork site (line 320-331), so that an attempt to fork someone else's conversation returns a 403 (or, at least, a non-existence-indicating 404) rather than a `ValueError` whose string reveals "not found" vs. "not yours."
- **Replace the assertion-based ownership invariant** at `saas_app_conversation_info_injector.py:397-400` with an explicit HTTPException; assertions are stripped under `python -O`.
- **Document the `ADMIN` escape hatch centrally** in `saas_user_auth.py` or a top-level security doc — today it is only documented inline in `specifiy_user_context.py:55-65` and re-asserted in the SAAS filter (`saas_app_conversation_info_injector.py:70-73`). The list of callers (every V1 callback processor + webhook router + `session_auth.py`) is hard to enumerate.
- **Consider a real cross-conversation memory service.** Today there is none, but the `(user_id, org_id)` plumbing is already in place; an explicit, opt-in memory service scoped to a user (and optionally an org) would slot in without disturbing existing boundaries.
- **Add an end-to-end test that two SAAS users with similar runs do not see each other's rows** in `tests/unit/app_server/` — the existing `test_user_isolation` family (`tests/unit/app_server/test_sql_app_conversation_start_task_service.py:230-641`) only covers the start-task service, not the conversation info service.

## Questions / Gaps

- **Is the `ADMIN` bypass auditable?** I could not find a structured log of every place `state.USER_CONTEXT_ATTR = ADMIN` is installed. A grep enumerates them (`github_v1_callback_processor.py:252`, `jira_v1_callback_processor.py:113`, etc.), but a single ADR or runtime audit log would be more discoverable.
- **How are shared conversations revoked?** The `public` flag exists (`app_conversation_models.py:134`) but I did not find a route that flips it back to private. The `update_app_conversation` PATCH route accepts `public: bool | None` (`app_conversation_models.py:256`) — verifying that it persists correctly is open.
- **What happens to a sub-conversation when its parent sandbox is deleted?** If the parent conversation row goes away (e.g. via `delete_app_conversation`), sub-conversations still reference `parent_conversation_id = <gone>` and become orphans reachable only by direct id.
- **Is `conversation_metadata_saas` ever written before `conversation_metadata`?** `save_app_conversation_info` calls `super().save_app_conversation_info(info)` first (`saas_app_conversation_info_injector.py:358`) and then inserts the SAAS row. If the base save succeeds and the SAAS insert fails (e.g. DB error), the conversation exists without SAAS metadata, so all SAAS reads return `None` for it. The rollback story is implicit (the base call commits before the SAAS call).
- **Are there tests for `require_permission`?** I did not find them in `tests/unit/app_server/`; the user-isolation tests only cover start tasks. Permission tests would harden the SAAS authorization story.
- **What is the lifespan of `conversation_metadata.created_by_user_id`?** It is still written (`live_status_app_conversation_service.py:483`) and is used by the OSS path. A migration to deprecate it (now that SAAS owns user_id) is unclear.

## Summary

The agent-framework project ships two cooperating implementations of the same boundary model: a Python SDK around `AgentSession` / `SessionContext` / `ContextProvider` / `HistoryProvider`, and a .NET SDK around `AIAgent` / `AgentSession` / `AgentSessionStateBag` / `ChatHistoryProvider` / `AIContextProvider`. Both treat the *session* as the primary state-bearer (UUID-generated local ID plus an optional *service*-managed ID for hosted services like OpenAI Assistants / Azure AI Foundry / Responses API), and treat *user* and *tenant* boundaries as integration-layer concerns: they are not first-class fields on `AgentSession`, but each history / context provider in the monorepo accepts an explicit scope / key (Mem0, Redis, Foundry Memory, Azure Cosmos, .NET `IsolationKeyScopedAgentSessionStore`, `ClaimsIdentitySessionIsolationKeyProvider`). Cross-thread memory is explicit through `scope` (Python: `FileMemoryProvider` defaults to `session_id`, opt-in `scope="shared"`; Foundry / Mem0 / Redis require a non-null scope) and explicit through `IsolationKey` providers in .NET, with a hard runtime check that rejects resuming a session under a different user/chat identity (HTTP 403 `hosted_session_identity_mismatch`). The model is mature and tested, but the safety guarantees depend on which provider is configured — core defaults (`InMemoryHistoryProvider`, `InMemoryChatHistoryProvider`) have no built-in cross-user guard and rely entirely on the caller for isolation. The story is therefore *clear and well-typed* but *not automatically safe*; tooling for the failure mode is present (sentinel `LOCAL_HISTORY_CONVERSATION_ID`, `service_session_id` propagation rules, .NET 403 on resume-mismatch), so the rating lands in the upper-mid band.

## Rating

**8/10 — Clear model with tests, explicit interfaces, and operational safeguards; missing first-class user/tenant fields in the Python core means the framework can't prevent misuse at the API surface, but every integration implements its own tenant boundary.**

Rationale:
- **+ Clear typed session model** — `AgentSession` / `SessionContext` / `HistoryProvider` / `ContextProvider` is a well-defined four-piece model (`python/packages/core/agent_framework/_sessions.py:746-772`, `:154-213`).
- **+ Explicit per-session state namespace** — `session.state[provider.source_id]` is the canonical place; tests assert it (`tests/core/test_sessions.py`, `tests/core/test_harness_todo.py:67-83`).
- **+ Cross-thread memory is explicit** — `scope` parameter is a first-class argument (`python/packages/core/agent_framework/_harness/_file_memory.py:176-201`, `python/packages/foundry/agent_framework_foundry/_memory_provider.py:76-99`).
- **+ Hard tenant boundary in hosted mode** — .NET refuses to resume a session whose persisted identity disagrees with the live request (`dotnet/src/Microsoft.Agents.AI.Foundry.Hosting/AgentFrameworkResponseHandler.cs:99-109`).
- **+ Authorization filters are validated in providers** — Redis/Mem0 raise `ValueError` if no `agent_id`/`user_id`/`application_id` is configured (`python/packages/redis/agent_framework_redis/_context_provider.py:404-407`, `python/packages/mem0/agent_framework_mem0/_context_provider.py:238-241`); Foundry Memory raises if `scope` is missing (`python/packages/foundry/agent_framework_foundry/_memory_provider.py:130-131`).
- **+ Cosmos hierarchical partition** — `TenantId`/`UserId`/`ConversationId` are part of the partition key (`dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs:197-215`, `:557-580`).
- **− No `user_id`/`tenant_id` field on `AgentSession`** — those values exist only as forwarded function-invocation kwargs; the framework cannot reject calls that should have carried them (`python/packages/core/agent_framework/_sessions.py:758-772`).
- **− In-memory providers rely on caller-supplied `session_id`** — `InMemoryHistoryProvider` (`python/packages/core/agent_framework/_sessions.py:867-890`) and `InMemoryChatHistoryProvider` (`dotnet/src/Microsoft.Agents.AI.Abstractions/InMemoryChatHistoryProvider.cs:71-86`) accept whatever `session_id` they're given; cross-session leakage is possible if the caller mis-routes.
- **− Local "service call" sentinel** (`LOCAL_HISTORY_CONVERSATION_ID`) is treated as a controller-only value but is still mutated on the response (`python/packages/core/agent_framework/_sessions.py:537-542`, `:697-698`) — easy to mistake for a real ID if not guarded.

## Answers to Dimension Questions

**1. What scopes state?**
- `AgentSession` (Python, `_sessions.py:746-772`) and `AgentSession` (.NET, `dotnet/src/Microsoft.Agents.AI.Abstractions/AgentSession.cs:59-118`) are the primary state carriers. Each holds: (a) a **local session ID** (UUID4 by default), (b) an optional **service-managed session ID** for hosted services (Python: `service_session_id`; .NET: opaque per-history-provider state like `ConversationId`), and (c) a **mutable state dict / StateBag** that providers use for their per-session working state.
- Provider state is namespaced by `provider.source_id` (Python `session.state[source_id]`, .NET `session.StateBag[provider.StateKey]`). Default StateKey is the concrete type name (`Provider.StateKeys = [GetType().Name]`).
- Per-invocation context (`SessionContext` / `AIContext`) is constructed fresh each `agent.run()` call so request-scoped additions don't leak across calls. `SessionContext` mirrors `session_id` and `service_session_id` from `AgentSession` (`_sessions.py:154-213`).
- Workflows add: `Workflow.id` (UUID), `CheckpointID` (UUID) per checkpoint, `WorkflowMessage.source_id/target_id` for executor routing, `request_id` for in-flight request_info events (`_workflows/_workflow.py:329`, `_checkpoint.py:74`, `_workflow_context.py:407-419`).
- Background agents add: numeric `task_id` minted per `start_task` call (`_harness/_background_agents.py:325-326`), with a per-session `dict[task_id → AgentSession]` of sub-sessions held in non-serializable `_RuntimeState` (`_harness/_background_agents.py:110-298`).

**2. Can state leak across users or sessions?**
- **Default core providers (in-memory): no auth boundary.** `InMemoryHistoryProvider` reads from `session.state["in_memory"]["messages"]` and is keyed only by the `session_id` it is given (`_sessions.py:867-890`). Two callers passing each other the same `session_id` would share history. The same applies to `InMemoryChatHistoryProvider` on .NET (`dotnet/src/Microsoft.Agents.AI.Abstractions/InMemoryChatHistoryProvider.cs:71-86`).
- **File / Cosmos / Redis providers: isolation comes from the storage layer.** Cosmos filters by `session_id` AND `source_id` (`_history_provider.py:131-152`) and uses session_id as the partition key (`_history_provider.py:264-274`). Redis keys by `{prefix}:{session_id}` (`redis/_history_provider.py:106-108`) and tags include `conversation_id`, `thread_id`, `application_id`, `agent_id`, `user_id` (`redis/_context_provider.py:210-237`). File history provider writes one file per session and rejects path escape (`_sessions.py:1087-1101`).
- **Foundry hosted (.NET): actively enforced.** `AgentFrameworkResponseHandler` compares the persisted `HostedSessionContext(UserId, ChatId)` against the live request's headers. A mismatch raises 403 `hosted_session_identity_mismatch` (`dotnet/src/Microsoft.Agents.AI.Foundry.Hosting/AgentFrameworkResponseHandler.cs:79-110`).
- **Foundry agent (Python):** the hosted-session path requires an explicit `isolation_key` and refuses to run without one (`python/packages/foundry/agent_framework_foundry/_agent.py:721-794`), so sessions are isolated on the platform side.
- **Python core helpers (`user_id`, `tenant_id`):** there is **no** dedicated field on `AgentSession`. The framework only forwards these names through `function_invocation_kwargs` so tools can read them (`tests/core/test_as_tool_kwargs_propagation.py:60-100`, `tests/core/test_middleware_with_agent.py:849-910`). Anything stronger requires the user-supplied `MemoryContextProvider` (Mem0 / Redis / Foundry / File) or the .NET `IsolationKeyScopedAgentSessionStore`.

**3. Is cross-thread memory explicit?**
- Yes, by configuration:
  - Python `FileMemoryProvider(scope="shared")` writes under that scope; omitting `scope` falls back to `session_id` (`_harness/_file_memory.py:176-215`). Tests `test_scope_isolates_memories_across_sessions` and `test_explicit_scope_shares_memories_across_sessions` make this contract explicit (`tests/core/test_harness_file_memory.py:226-254`).
  - Foundry memory `FoundryMemoryProvider(scope=...)` writes under `scope` or `session_id`; the constructor raises if `scope is None` (`_memory_provider.py:128-138`).
  - Mem0 partitions by `user_id`/`agent_id`/`app_id`; multiple users sharing one provider cannot see each other's records (`mem0/_context_provider.py:117-241`).
  - Redis uses `user_id`/`agent_id`/`application_id` as filter tags; `_redis_search` AND's them all into a single query (`redis/_context_provider.py:357-372`).
  - On the file-access side, the `FileAccessProvider` is documented as **deliberately shared** across sessions and agents (`_harness/_file_access.py:6-50`, AGENTS.md), which is the explicit acknowledgement that cross-session sharing is a feature, not an accident.
  - .NET: cross-thread memory is explicit through `ApplicationId`/`UserId`/`ConversationId` partition keys (Cosmos) and through `AgentId`/`UserIsolationKey`/`ChatIsolationKey` headers on the hosting surface.

**4. Are tenant boundaries enforced at storage level?**
- **Cosmos (.NET):** yes, hierarchical partition `TenantId + UserId + ConversationId` is the storage-level key when both are supplied (`dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs:197-215`). Tenant boundary breaks if both are not supplied — falls back to `ConversationId` only.
- **Cosmos (Python):** partition-key path is `/session_id`; no built-in tenant field (`azure-cosmos/agent_framework_azure_cosmos/_history_provider.py:264-285`). Tenant boundaries in Python are provided by integrating `FoundryMemoryProvider`/`Mem0ContextProvider`/`RedisContextProvider`, which are parameter-driven.
- **Redis / Mem0:** filters are validated at the client level (`_validate_filters` raises if no `agent_id`/`user_id`/`application_id` is set) so a misconfigured provider can't accidentally return everyone else's data — but actual tenant isolation is the storage server's job (`redis/_context_provider.py:404-407`, `mem0/_context_provider.py:238-241`).
- **.NET Hosting stack:** `IsolationKeyScopedAgentSessionStore` (re-)writes the session key with the prefix `{key}::` (`dotnet/src/Microsoft.Agents.AI.Hosting/IsolationKeyScopedAgentSessionStore.cs:50-93`). The `ClaimsIdentitySessionIsolationKeyProvider` sources the key from `ClaimTypes.NameIdentifier` by default, and the docs explicitly warn that display names / usernames / emails are unsafe (`dotnet/src/Microsoft.Agents.AI.Hosting.AspNetCore/ClaimsIdentitySessionIsolationKeyProvider.cs:24-34`).
- **In-memory / file:** no storage-level tenant boundary; you get whatever your caller routes.

**5. Can users fork sessions safely?**
- **Python:** `AgentSession.to_dict()` / `from_dict()` round-trips state via a type-tagged registry of any class registered via `register_state_type` (`_sessions.py:69-89`). Copying the dict to a new `AgentSession(new_id, new_service_session_id)` produces a fork with isolated identity but copied messages; provider state is duplicated because it's in the dict.
- **.NET:** `AIAgent.SerializeSessionAsync(session, …)` and `DeserializeSessionAsync(JsonElement, …)` are abstract methods (`dotnet/src/Microsoft.Agents.AI.Abstractions/AIAgent.cs:187-235`). `AgentSession.StateBag.Serialize()` / `Deserialize()` are production-tested utilities (`dotnet/src/Microsoft.Agents.AI.Abstractions/AgentSessionStateBag.cs:124-144`).
- The `AgentSession` design does not include a built-in "fork" semantic; callers compose it from `create_session()` + `state` copy. Forking is safe **as long as the new session gets a new `session_id`** and any `service_session_id` is either `None` (creating a new service thread) or a deliberate continuation point. The code actively avoids replaying the source session's `service_session_id` (`_agents.py:1075, 1095`) so a forked session won't accidentally re-enter the original hosted conversation.

## Architectural Decisions

1. **One object, two IDs.** `AgentSession.session_id` (local UUID) and `AgentSession.service_session_id` (optional, hosted) deliberately separate the caller's bookkeeping ID from the platform-issued one. Service-thread propagation is centralized in `_propagate_conversation_id` / `_finalize_response` and the `LOCAL_HISTORY_CONVERSATION_ID` sentinel (`_sessions.py:537-542`, `_agents.py:1068-1099, 1399-1414`). Trade-off: caller's local UUID is opaque to the platform, which is why it isn't used as a thread identifier downstream.
2. **Provider-scoped state inside session.state.** Both `AgentSession.state[provider.source_id]` (Python) and `AgentSession.StateBag[state_key]` (.NET) guarantee that two providers of the same type don't collide when wired into the same session. The .NET side documents it as a contract on `ChatHistoryProvider` (`dotnet/src/Microsoft.Agents.AI.Abstractions/ChatHistoryProvider.cs:34-49`).
3. **Context providers receive a one-shot `SessionContext`.** Provider-contributed state stays in the invocation context for the duration of a single `run()` and is not persisted anywhere; persisted changes go via `provider.after_run → session.state`. (`_sessions.py:154-348`, `_agents.py:1419-1501`). This is how the framework prevents the *invocation* scope from leaking.
4. **Cross-thread sharing requires an explicit `scope` (Python) / hierarchical partition key (.NET).** Every "memory" provider in the monorepo accepts a scope or partition tuple. There is no implicit cross-thread memory.
5. **Tenant keys are not first-class.** The framework treats user/tenant/app IDs as *content of each provider's API*, not as fields on `AgentSession`. This keeps the core domain model lean but shifts the safety burden to whichever provider the host wires in.
6. **Defensive reads on resume.** .NET rejects resuming a hosted session whose persisted identity disagrees with the live request (`AgentFrameworkResponseHandler.cs:79-110`), preventing the most common cross-user leak. Python's hosted path requires `isolation_key` up front, with no equivalent resume check in core.
7. **Per-service-call history persistence option** (`require_per_service_call_history_persistence`) is a deliberate knob that lets callers force history to be saved between every model call rather than once at the end of the run; it also refuses to coexist with an existing service-managed conversation (`_agents.py:810-823`). The decision matrix is fully tested (`tests/core/test_agents.py:2691-3122`).

## Notable Patterns

- **Attribution tags.** Every message placed into `SessionContext.context_messages` carries `_attribution = {source_id, source_type}` in `additional_properties` (`_sessions.py:236-249`). Downstream providers can filter by source without needing a registry. This is the durable audit trail for "which provider put this message here."
- **Type-tagged session serialization.** `_serialize_state` / `_deserialize_state` round-trip arbitrary objects by their class name (`_sessions.py:69-147`). This is how forks/backups avoid losing custom state.
- **Hierarchical partition keys for Cosmos.** .NET supports `TenantId + UserId + ConversationId` as a Cosmos `PartitionKeyBuilder` triple (`dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs:203-215`). Tenant boundary at storage level is built in.
- **Strict-mode isolation store decorator.** `IsolationKeyScopedAgentSessionStore.Strict` makes "no isolation key ⇒ fail closed" the default in every `.AddAgent(...)`-style registration (`dotnet/src/Microsoft.Agents.AI.Hosting/IsolationKeyScopedAgentSessionStore.cs:50-63`).
- **Forbidden-name reservation for internal files.** `FileMemoryProvider` and `FileHistoryProvider` reject filenames matching internal markers (`_description.md` sidecars, `memories.jsonl`) so the model can't overwrite the provider's bookkeeping (`_harness/_file_memory.py:84-90`, `_sessions.py:1122-1134`). This is a defensive design choice about how patient a malicious model could be with file-system shenanigans.
- **Per-invocation runtime vs. per-session serializable state split.** `BackgroundAgentsProvider` keeps a per-session `_runtime: dict[session_id → _RuntimeState]` holding in-flight `asyncio.Task` objects separately from the serializable `provider_state["tasks"]`, with explicit `LOST` semantics for refs that disappear across restarts (`_harness/_background_agents.py:288-302`). This is a careful separation of "what can survive a checkpoint" from "what cannot."
- **Server-label boundary on tool approval.** Same-named tools on different hosted servers get independent approval rules via `ToolApprovalRule.server_label` (`_harness/_tool_approval.py:61-72`, `:99-156`). The principle generalizes: every cross-cutting identity should be part of an approval key, not the agent name alone.

## Tradeoffs

- **Safety vs. simplicity.** Not having `user_id`/`tenant_id` on `AgentSession` keeps the core simple but pushes the cross-user-isolation problem to every integrator. Memory providers raise on missing scope, but a Python harness that wires only `InMemoryHistoryProvider` has no built-in protection.
- **In-memory default is permissive.** `InMemoryHistoryProvider` auto-injects when nothing else is configured (`_agents.py:1200-1213`). That's a friendly default for prototyping but a footgun in production.
- **Python hosted isolation key is a caller-supplied string.** `FoundryAgent._resolve_service_session_isolation_key` requires the caller to provide a stable string per logical partition (`_agent.py:721-728`). If the caller reuses one, isolation collapses; the framework cannot verify uniqueness.
- **Local-history sentinel mix.** `LOCAL_HISTORY_CONVERSATION_ID` is a string sentinel that flows through middleware as if it were a real conversation id and is later marked internal via `mark_internal_conversation_id` (`_sessions.py:696-698`, `_types.py:2245-2254`). This is convenient for the function-invocation loop but a developer who logs conversation_ids may still see the literal sentinel.
- **State on `AgentSession.state` is per-process.** `_serialize_state`/`_deserialize_state` work fine for snapshots but `state` is a plain dict — concurrent mutations from multiple asyncio tasks will race. Only some providers (e.g., `BackgroundAgentsProvider` `_runtime`, `FileMemoryProvider` `_write_lock`) put their own lock around state mutation.
- **Cross-thread sharing is intentional on the file-access path.** `FileAccessProvider` is documented as shared and the agent harness uses it as a shared working directory (`_harness/_file_access.py:1-50`). Callers must not assume file access is per-session the way file memory is.

## Failure Modes / Edge Cases

1. **Caller-supplied `session_id` collision.** Two calls passing the same `session_id` to `InMemoryHistoryProvider` share state. The framework logs nothing on collision because there's no per-instance identity; detection is the caller's responsibility.
2. **`service_session_id` rotation on Responses APIs.** The framework refreshes `session.service_session_id` whenever the response carries a newer value (`_agents.py:1068-1099, 1399-1414`). If a caller reads `service_session_id` mid-run, it can see a stale value across the function-invocation loop unless the chat client opts into `require_per_service_call_history_persistence`.
3. **`require_per_service_call_history_persistence` + service-stored history.** Guarded: the framework raises `AgentInvalidRequestException` when both an existing service-managed conversation and a HistoryProvider are present (`_agents.py:810-823`).
4. **Path-traversal.** `FileHistoryProvider` encodes risky session IDs and rejects literal path separators and Windows reserved names (`_sessions.py:1122-1134`); `FileMemoryProvider` does its own `_normalize_relative_path` validation (`_harness/_file_access.py:1010-1080` referenced). Callers that bypass these helpers (e.g., write their own store) lose this protection.
5. **Resume under different identity.** On .NET hosted mode, caught at the boundary with a 403 (`AgentFrameworkResponseHandler.cs:99-110`). On Python hosted mode, the same attack is mitigated by `isolation_key` being mandatory but not by a runtime identity check against the resumed session.
6. **`.NET` `IsolationKeyScopedAgentSessionStore` when key provider returns null.** In `Strict` mode the next operation throws `InvalidOperationException`; non-strict silently falls back to bare `conversationId`, removing the tenant boundary. The hosting extensions register with `Strict = isolationKeyProvider != null` (`dotnet/src/Microsoft.Agents.AI.Hosting.AGUI.AspNetCore/AGUIEndpointRouteBuilderExtensions.cs:110-115`), so the boundary is enforced only when a provider is configured.
7. **`RedisHistoryProvider.max_messages` overflow.** When set, the provider calls `LTRIM` to keep only the last `max_messages` (`redis/_history_provider.py:162-165`) — slow cross-session growth is prevented, but a misconfigured value silently drops context.
8. **`FoundryMemoryProvider.update_delay=0`** immediately triggers memory updates (`_memory_provider.py:78-82`) — fine in tests, but a runaway loop in production can stampede the memory store.

## Future Considerations

- **First-class `user_id` / `tenant_id` field on `AgentSession`.** Would let core reject runs that lack the required partition key, instead of relying on each provider's `_validate_filters`. The headers exist on the .NET hosting side; Python's analogue would be a small, explicit `isolation` argument on `AgentSession.__init__`.
- **Authorization at the `ContextProvider.before_run` boundary.** Today any provider can read context, but core doesn't expose an "is this message authorized for this user" filter. The .NET `AgentRequestMessageSourceAttribution` enum is a step in that direction.
- **Lock around `session.state` writes.** Today individual providers (BackgroundAgents, FileMemory, FileHistory) re-implement locking; a core-level `session.state_lock` would centralize it and document the contract.
- **`_STATE_TYPE_REGISTRY` is process-local.** Forks restored from a different process must re-register their types ahead of time (`_sessions.py:69-89`). A persistent registry (`register_state_type` could be wrapped at module import) would catch mis-registrations at startup.
- **Resume-identity guard on Python hosted sessions.** The .NET 403 check (`AgentFrameworkResponseHandler.cs:79-110`) has no Python equivalent. Copying that pattern would close the Python side's only obvious gap.
- **Default `InMemoryHistoryProvider` opt-out.** A misconfigured harness that forgets to provide a real provider gets "convenient" auto-injected `InMemoryHistoryProvider` (`_agents.py:1200-1213`). Surfacing a warning when the provider is auto-injected and the harness talks to a hosted client would make the unsafe-default case harder to miss.

## Questions / Gaps

- **Where is `user_id` first-class in core Python?** No clear evidence found. It appears only as a forwardable kwarg via `function_invocation_kwargs`/`ChatOptions`, not as a typed field on `AgentSession` or `SessionContext`. Anyone relying on it for authorization must add a middleware.
- **Cross-process `state` registry?** `register_state_type` mutates a module-global dict (`_sessions.py:43-89`). For long-lived stateful services that deserialize sessions in worker processes, a mis-deployed module would raise on first decode.
- **Are there existing tests for the .NET 403 cross-user guard?** No clear evidence found in the dotnet tests directory; the implementation is present and the docstring names the test scenario indirectly (`AgentFrameworkResponseHandler.cs:99-110`) but the corresponding test file was not inspected.
- **What happens if a Python hosted agent runs without `isolation_key`?** `_resolve_service_session_isolation_key` raises `ValueError("isolation_key is required")` at hosted-session creation time (`_agent.py:721-728`); a latent failure mode is "developer runs a hosted agent that *appears* stateless (no `service_session_id` yet) and never trips the guard."
- **Two run calls into the same in-memory session with different `store=False`/`store=True`?** `require_per_service_call_history_persistence` covers the per-call load/store mechanics but the test matrix in `tests/core/test_agents.py:2691-3122` only exercises sequential runs, not interleaved concurrent runs on the same session.

## Summary

OpenHands has explicit state vocabularies and service-layer mutator interfaces, but the implementation is mostly direct object mutation plus last-writer-wins persistence rather than a central state-machine model. Conversation metadata is persisted through `AppConversationInfoService.save_app_conversation_info` (`sources/openhands/openhands/app_server/app_conversation/app_conversation_info_service.py:95-117`) and live runtime status is assembled separately in `LiveStatusAppConversationService._build_conversation` (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:630-663`). Mutation observability is stronger for agent/runtime state because webhooks persist events and process callbacks (`sources/openhands/openhands/app_server/event_callback/webhook_router.py:453-518`), while app-server metadata edits such as `LiveStatusAppConversationService.update_app_conversation` simply mutate fields and save (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2114-2147`).

## Rating

5/10 — clear enums, typed services, event persistence, and some operational safeguards exist (`sources/openhands/openhands/app_server/app_conversation/app_conversation_models.py:262-271`, `sources/openhands/openhands/app_server/sandbox/sandbox_models.py:9-15`, `sources/openhands/openhands/app_server/event_callback/webhook_router.py:453-518`), but legal transitions are not centrally enforced, generic saves use merge/commit without version checks (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:347-388`), and direct assignment is routine (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2141-2145`).

## Answers to Dimension Questions

1. **Can any module mutate state?** No single global state-machine boundary was found. Mutation entry points are explicit at service interfaces such as `AppConversationInfoService.save_app_conversation_info` and `AppConversationInfoService.process_stats_event` (`sources/openhands/openhands/app_server/app_conversation/app_conversation_info_service.py:95-117`), but call sites mutate model objects directly before saving, as in `LiveStatusAppConversationService.update_app_conversation` (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2141-2145`) and `SetTitleCallbackProcessor` (`sources/openhands/openhands/app_server/event_callback/set_title_callback_processor.py:140-152`).

2. **Are illegal transitions prevented?** Partially. Intended start-task progression is documented in `AppConversationService.start_app_conversation` (`sources/openhands/openhands/app_server/app_conversation/app_conversation_service.py:92-105`) and implemented by ordered generator steps (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:300-307`, `sources/openhands/openhands/app_server/app_conversation/app_conversation_service_base.py:247-280`), but persistence accepts arbitrary enum updates; `test_update_task_status` mutates a task to `READY` directly and saves it (`sources/openhands/tests/unit/app_server/test_sql_app_conversation_start_task_service.py:200-227`). Runtime guards exist at API boundaries, especially message sends requiring `SandboxStatus.RUNNING` (`sources/openhands/openhands/app_server/app_conversation/app_conversation_router.py:508-529`).

3. **Are concurrent writes safe?** Mixed. Conversation export is protected by a per-conversation Redis lock (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2354-2401`) and profile writes use per-user in-process locks with an explicit multi-worker limitation (`sources/openhands/openhands/app_server/settings/settings_router.py:313-320`). Conversation metadata saves use SQLAlchemy `merge` plus `commit` with no visible compare-and-swap token (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:347-388`); `conversation_version` is stored as a V1 discriminator rather than an update counter (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:105-107`, `sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:375-376`). No evidence found for `with_for_update`, `version_id`, `If-Match`, or `ETag` guards after searching `sources/openhands/openhands/app_server` for those terms.

4. **Are mutations observable?** Agent/runtime mutations are observable through stored events and callbacks: `on_event` persists received events, applies stats, updates model metadata for switch observations, detects terminal execution status, and schedules callbacks (`sources/openhands/openhands/app_server/event_callback/webhook_router.py:453-518`). The frontend consumes state-update events over WebSocket (`sources/openhands/frontend/src/contexts/conversation-websocket-context.tsx:449-460`). App-server metadata changes such as PATCH updates are less observable because `update_app_conversation` saves and returns the object without emitting an event (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2114-2147`).

5. **Can state become internally inconsistent?** Yes. `AppConversation.execution_status` is documented as `None` when the sandbox is not running, but this is a field description rather than an enforced validator (`sources/openhands/openhands/app_server/app_conversation/app_conversation_models.py:173-181`). The frontend works around one impossible combination by forcing archived `sandbox_status === "MISSING"` to stopped even if live status says running (`sources/openhands/frontend/src/hooks/use-agent-state.test.tsx:80-99`), and cache utilities explicitly clear `execution_status` when sandbox status is not `RUNNING` (`sources/openhands/frontend/src/hooks/mutation/conversation-mutation-utils.ts:95-111`). Post-switch model persistence is best-effort, so the agent-server can switch while stored metadata remains stale (`sources/openhands/openhands/app_server/app_conversation/app_conversation_router.py:593-617`, `sources/openhands/tests/unit/app_server/test_app_conversation_router.py:781-814`).

6. **Can impossible states be represented?** Yes. The local model can represent a non-running sandbox with a non-null execution status because `AppConversation` stores those fields independently (`sources/openhands/openhands/app_server/app_conversation/app_conversation_models.py:173-181`). The frontend test for `sandbox_status === "MISSING"` plus live `V1ExecutionStatus.RUNNING` confirms such a combination is possible enough to require precedence logic (`sources/openhands/frontend/src/hooks/use-agent-state.test.tsx:80-99`).

## Architectural Decisions

- Conversation state is split between durable metadata (`AppConversationInfo`) and runtime-derived status (`AppConversation`) rather than persisted as one aggregate (`sources/openhands/openhands/app_server/app_conversation/app_conversation_models.py:112-181`). `LiveStatusAppConversationService._build_app_conversations` fetches live agent status only for running sandboxes and then composes response objects (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:540-587`).
- Runtime status is delegated to sandbox adapters. Docker, remote runtime, and process implementations each derive `SandboxStatus` from their backing system instead of sharing a transition table (`sources/openhands/openhands/app_server/sandbox/docker_sandbox_service.py:115-127`, `sources/openhands/openhands/app_server/sandbox/remote_sandbox_service.py:188-206`, `sources/openhands/openhands/app_server/sandbox/process_sandbox_service.py:179-194`).
- Agent state changes are treated as event-stream updates. `ConversationStateUpdateEvent` is typed in the frontend as full state, execution status, or stats (`sources/openhands/frontend/src/types/v1/core/events/conversation-state-event.ts:64-109`), and the backend persists/processes those events through `on_event` (`sources/openhands/openhands/app_server/event_callback/webhook_router.py:453-518`).
- Some high-risk write paths have targeted safeguards rather than universal locking: export has Redis locking (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2354-2401`), and settings profile writes have process-local locks (`sources/openhands/openhands/app_server/settings/settings_router.py:313-320`).

## Notable Patterns

- Direct mutation plus save is the dominant pattern: `update_app_conversation` uses `setattr` over submitted fields (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2138-2145`), `save_app_conversation_info` merges the resulting object (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:353-388`), and callback processors can mutate callback status (`sources/openhands/openhands/app_server/event_callback/set_title_callback_processor.py:150-152`).
- Lifecycle status is explicit but enforcement is procedural. The intended start-task sequence is documented (`sources/openhands/openhands/app_server/app_conversation/app_conversation_service.py:92-105`), and each implementation step assigns an enum value (`sources/openhands/openhands/app_server/app_conversation/app_conversation_service_base.py:254-275`, `sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:409-423`).
- Event observation is append-style and storage-backed: `EventServiceBase.save_event` writes by event id (`sources/openhands/openhands/app_server/event/event_service_base.py:190-197`), filesystem writes are direct JSON writes (`sources/openhands/openhands/app_server/event/filesystem_event_service.py:33-36`), and cloud storage variants use object writes (`sources/openhands/openhands/app_server/event/aws_event_service.py:54-62`, `sources/openhands/openhands/app_server/event/google_cloud_event_service.py:57-62`).
- Client state uses precedence rules instead of a normalized state machine: `useAgentState` prioritizes live status, falls back to cached status, and treats archived sandboxes specially (`sources/openhands/frontend/src/hooks/use-agent-state.ts:57-74`).

## Tradeoffs

- Splitting durable metadata from runtime-derived status keeps the app server decoupled from sandbox internals, but it allows temporary or representable mismatches between stored metadata and live agent state (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:630-663`, `sources/openhands/frontend/src/hooks/use-agent-state.test.tsx:80-99`).
- Last-writer-wins persistence is simple and broadly reusable, but it does not prevent concurrent metadata overwrites (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:347-388`).
- Webhook-driven observability preserves agent trajectory and supports callbacks (`sources/openhands/openhands/app_server/event_callback/webhook_router.py:453-518`), but `on_event` catches exceptions and still returns success, so persistence or callback failures may rely on logs rather than retry semantics (`sources/openhands/openhands/app_server/event_callback/webhook_router.py:519-522`).
- Best-effort post-switch persistence avoids rolling back agent-server success, but the UI can show stale model metadata until a later refresh (`sources/openhands/openhands/app_server/app_conversation/app_conversation_router.py:593-617`).

## Failure Modes / Edge Cases

- Concurrent metadata updates can lose fields because `save_app_conversation_info` rebuilds a full `StoredConversationMetadata` and commits without a version predicate (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:353-388`).
- Start-task records can skip intended phases because the save layer accepts direct enum changes and tests demonstrate a direct `WORKING` to `READY` update (`sources/openhands/tests/unit/app_server/test_sql_app_conversation_start_task_service.py:200-227`).
- Conversation deletes are not fully atomic across agent server, sub-conversations, local metadata, and sandbox cleanup; sub-conversation deletion logs and continues on child failures (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2203-2236`), and the router schedules sandbox deletion after committing conversation deletion (`sources/openhands/openhands/app_server/app_conversation/app_conversation_router.py:959-978`).
- Runtime status can be stale or unavailable; remote polling explicitly fetches conversations, releases DB sessions, then writes refreshed data and events later (`sources/openhands/openhands/app_server/sandbox/remote_sandbox_service.py:771-869`).
- Event callback execution runs applicable callbacks concurrently and stores results afterward, but callback status mutations are saved after callback execution and committed once (`sources/openhands/openhands/app_server/event_callback/sql_event_callback_service.py:208-231`).
- Stats update errors are swallowed after logging, so usage metrics can remain stale (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:472-512`), and tests lock in that behavior (`sources/openhands/tests/unit/app_server/test_webhook_router_stats.py:399-421`).

## Future Considerations

- Add explicit transition methods for start tasks and sandbox lifecycle states, with a transition matrix covering `AppConversationStartTaskStatus` (`sources/openhands/openhands/app_server/app_conversation/app_conversation_models.py:262-271`) and `SandboxStatus` (`sources/openhands/openhands/app_server/sandbox/sandbox_models.py:9-15`).
- Add optimistic concurrency to conversation metadata and start-task persistence, for example by comparing `updated_at` or a real version counter before `merge`/`commit` (`sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_info_service.py:347-388`, `sources/openhands/openhands/app_server/app_conversation/sql_app_conversation_start_task_service.py:235-248`).
- Emit or persist app-server metadata mutation events for PATCH updates and model switches so those changes have the same observability as agent events (`sources/openhands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:2114-2147`, `sources/openhands/openhands/app_server/app_conversation/app_conversation_router.py:593-617`).
- Add illegal-transition tests beyond persistence round trips, especially for phase skipping and terminal-state mutation (`sources/openhands/tests/unit/app_server/test_sql_app_conversation_start_task_service.py:200-227`).
- Add retry/idempotency policy for webhook event save and callback execution failures, since `on_event` logs and returns success on exceptions (`sources/openhands/openhands/app_server/event_callback/webhook_router.py:519-522`).

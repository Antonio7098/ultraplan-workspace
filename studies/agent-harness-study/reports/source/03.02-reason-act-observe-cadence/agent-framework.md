## Summary

The framework does not implement an explicit "Thought → Action → Observation" ReAct parser. Instead it relies on (a) the provider's native structured `function_call` outputs and (b) a chat‑client‑layer iterative loop that alternates **model call → tool execution → tool‑role observation → next model call** until a budget cap or a no‑more‑`function_call`/`function_approval_request` signal terminates the turn.

The model output parsing happens at `python/packages/core/agent_framework/_tools.py:2347` (`_extract_function_calls`) and uses **structured `Content` items** (`type="function_call"` at `_types.py:344`; constructor at `_types.py:794-815`); the framework never inspects free‑text "Action:" / "Observation:" markers. Tool execution is `_try_execute_function_calls` (`_tools.py:1658-1830`) which dispatches the requested calls in parallel via `_auto_invoke_function` (`_tools.py:1424-1645`). The observation is then injected as a single `Message(role="tool", contents=function_call_results)` appended to `response.messages` (`_tools.py:2261-2262`); for structured‑output approvals the assistant message itself becomes the carriage and the loop returns `action="return"` (`_tools.py:2228-2246`). Models thus see **the raw structured tool payloads** (`result` text + a uniform `items` list — see `Content.from_function_result` at `_types.py:818-874`), not a summarized value.

A second, fully ReAct‑shaped loop lives one layer up as `AgentLoopMiddleware` (`python/packages/core/agent_framework/_harness/_loop.py:212`). It is opt‑in via `Agent(middleware=[…])` and drives whole‑agent re‑invocations with a user `should_continue` predicate or a chat‑client **judge** that returns a structured `JudgeVerdict(BaseModel) { answered: bool, reasoning: str }` (`_harness/_loop.py:99-111`). The judge's `reasoning` is fed back to the agent as the next iteration's "feedback" via the `feedback` keyword (`_harness/_loop.py:198-206`), so the model is explicitly told **why** its previous answer was judged incomplete. When the structured output is unavailable, the fallback path looks for non‑overlapping sentinel markers `VERDICT: DONE` / `VERDICT: MORE` in the text (`_harness/_loop.py:184-194`).

Reasoning is **first‑class but optional**. `ContentType` enumerates `text_reasoning` (`_types.py:340`); `Content.from_text_reasoning` exists (`_types.py:611-631`) and reasoning items can be merged via `_add_text_reasoning_content` (`_types.py:1441-1465`) with an optional `protected_data` field for opaque/redacted reasoning. Provider adapters that emit reasoning (e.g., OpenAI Responses) put reasoning next to `function_call` in the same assistant message; the loop drives off the `function_call` items and treats the reasoning as ordinary message content.

Observations can be **compacted** by `ToolResultCompactionStrategy` (`python/packages/core/agent_framework/_compaction.py:820-918`) which replaces older `function_call`/`function_result` groups with a single `[Tool results: name: result; …]` assistant message; `ContextWindowCompactionStrategy` (`_compaction.py:1307-1456`) composes this with token‑budget truncation. Failed observations are surfaced back to the model as `function_result` items with `exception` populated and a synthetic `"Error: Function failed."` payload (`_tools.py:1556-1568`), or with the full exception when `include_detailed_errors=True` (`_tools.py:1632-1645`). The loop tracks `errors_in_a_row` (`_tools.py:2556`, `_tools.py:2248-2258`) with a default cap `DEFAULT_MAX_CONSECUTIVE_ERRORS_PER_REQUEST=3` (`_tools.py:91`); consecutive errors flip `tool_choice="none"` so the model is forced to speak in prose instead of looping (`_tools.py:2632-2644`).

There are **two unconnected iteration caps** that share the name `DEFAULT_MAX_ITERATIONS`: 40 for the chat‑layer tool loop (`_tools.py:90`) and 10 / 5 (judge) for the harness outer loop (`_harness/_loop.py:117, 122`).

## Rating

**7 / 10** — The cadence is implemented through two clearly separated layers (chat‑client tool loop + harness outer loop) with structured inputs (`function_call`, `function_result`), explicit budgets (`max_iterations`, `max_function_calls`, `errors_in_a_row`), a generous compaction subsystem that can summarize old observations, and a fully ReAct‑shaped judge pattern in the harness (`JudgeVerdict` structured output + `reasoning`‑as‑feedback). Reasoning items are first‑class `Content` and `text_reasoning` is enumerated. However, the framework does not (a) actually parse a "Thought/Action/Observation" textual protocol — providers must return structured `function_call`, (b) guarantee that every tool result is **summarized** before it returns to the model — raw `items` lists are passed back unless the user wires `ToolResultCompactionStrategy`, and (c) the two loops use the same `DEFAULT_MAX_ITERATIONS` constant name despite measuring different things, which makes the cadence harder to reason about for callers. The harness loop is `ExperimentalFeature.HARNESS` (decorated `@experimental(feature_id=...)` at `_harness/_loop.py:211`).

## Answers to Dimension Questions

1. **How does the model decide actions?**
   By emitting structured `function_call` items in an assistant message (`Content(type="function_call", call_id, name, arguments)` — `_types.py:794-815`). The framework **does not parse free text** for "Action: …" / "Thought: …". For providers that don't return structured calls, the framework will fail in `_extract_function_calls` (`_tools.py:2347`) and the loop returns `action="return"` (`_tools.py:2348-2357`).

2. **How are observations returned?**
   As an appended `Message(role="tool", contents=function_call_results)` (`_tools.py:2261-2262`) on the response that the next iteration will pass back to the model. The observation is the **raw structured payload**: each `Content(type="function_result", call_id, result, items, exception)` built by `Content.from_function_result` (`_types.py:818-874`). Pydantic/JSON objects are dumped to JSON text via `json.dumps(..., default=str)` (`_tools.py:859-867`). Approval‑gate responses instead extend the assistant message with `function_approval_request` items and return `action="return"` so the caller sees the request verbatim (`_tools.py:2228-2246`).

3. **Is reasoning visible or hidden?**
   **Visible, structured, optional, provider‑dependent.** Reasoning items are first‑class `Content(type="text_reasoning", text, id, protected_data, annotations)` (`_types.py:340, 611-631, 1441-1465`); the `protected_data` field exists for opaque/redacted reasoning. Provider adapters that emit reasoning alongside a tool call put both items in the same assistant message; the chat‑client loop drives off the `function_call` items and treats the reasoning as ordinary content. There is **no fallback "scratchpad prompt"** that forces the model to write `Thought:` / `Action:` / `Observation:` in prose; reasoning visibility is whatever the provider supports. The harness layer's `DEFAULT_NEXT_MESSAGE` (`_harness/_loop.py:59`) is a *prompt* nudge ("Continue working on the task. If it is complete, say so."), not a parser. `DEFAULT_HARNESS_INSTRUCTIONS` in the harness agent explicitly asks the model to "Explain your reasoning and thought process as you work through tasks" (`_harness/_agent.py:44-59`), which is the closest thing the framework ships to an explicit ReAct‑style prompt template.

4. **Are failed observations fed back?**
   **Yes, explicitly.** Exceptions in tool execution produce a `function_result` with `exception=str(exc)` and a `"Error: Function failed."` body (or `"Error: Function failed. Exception: …"` when `include_detailed_errors=True`) returned to the model (`_tools.py:1556-1568, 1632-1645`). The loop tracks `errors_in_a_row` and forces `tool_choice="none"` after `max_consecutive_errors_per_request` (default 3) so the model is forced to speak prose instead of looping (`_tools.py:2248-2258, 2632-2644`). Argument‑parsing failures get a `"Error: Argument parsing failed."` payload (`_tools.py:1519-1528`). Approval‑required / unknown / declaration‑only calls take a separate path (`_tools.py:1700-1765`).

5. **Can observations be compacted?**
   **Yes, via a separate strategy layer.** `ToolResultCompactionStrategy` (`_compaction.py:820-918`) collapses older `function_call`/`function_result` groups into a single summary message `[Tool results: name: result; …]`, retaining `keep_last_tool_call_groups` of the newest group verbatim (default 1; the harness defaults `after_strategy` to 2 — `_harness/_agent.py:109`). `ContextWindowCompactionStrategy` composes it with token‑budget truncation thresholds (`_compaction.py:1307-1456`). The harness wires both by default when token budgets are supplied (`_harness/_agent.py:72-120`). Without explicit strategy wiring, observations are **not** summarized; raw payloads flow back to the model on every iteration.

## Architectural Decisions

- **Reuse the provider's structured `function_call` output instead of parsing text.** Anchored by `ContentType` enumerating `function_call` / `function_result` / `function_approval_request` / `function_approval_response` (`_types.py:338-363`). This makes the cadence provider‑portable — but assumes every provider can emit structured tool calls; there is no narrative‑text fallback path.
- **Place the action/observation loop at the chat client, not the agent.** `FunctionInvocationLayer` is a `BaseChatClient` mixin (`_tools.py:2388-2478`); every agent that uses a decorated client gets the cadence for free. The trade‑off is that the loop's `attempt_idx` is not surfaced to `HistoryProvider` or `ToolApprovalMiddleware`.
- **Distinguish three termination triggers with explicit action codes.** `_process_function_requests` / `_handle_function_call_results` use `{ "action": "return" | "stop" | "continue" }` so the model call layer can decide whether to (a) return final answer, (b) flip `tool_choice="none"` and re‑call, or (c) loop again (`_tools.py:2217-2270, 2346-2357, 2580-2644`).
- **Tri‑state tool choice control.** `tool_choice="auto"` by default; flipped to `"none"` on cap hit or error threshold (`_tools.py:2586-2592, 2644, 2673, 2816, 2844`); reset when `tool_choice="required"` so it doesn't loop forever (`_tools.py:2648-2652, 2819-2823`).
- **Iteration budget is per model roundtrip, plus a separate per‑tool budget.** `max_iterations` (default 40, `_tools.py:90`) counts model roundtrips; `max_function_calls` (default `None`, `_tools.py:1394`) counts cumulative tool invocations; `max_consecutive_errors_per_request` (default 3, `_tools.py:91`) caps consecutive failures. The docstring at `_tools.py:1373-1378` explains the split.
- **Conversation‑managed vs stateless payload handling inside the loop.** For service‑managed APIs the loop drops `prepped_messages = []` and keeps only the latest assistant tool result; otherwise it accumulates (`_tools.py:2613-2661, 2780-2832`).
- **Pre‑pass for approval responses before each model call.** First half of each turn in `_process_function_requests(..., response=None, ...)` (`_tools.py:2287-2334`) executes approved calls and feeds results back so the model sees those results in its next generated message.
- **Reasoning items survive the loop untouched.** `text_reasoning` is a distinct `Content` type (`_types.py:340`); `_add_text_reasoning_content` lets providers merge consecutive reasoning items by `id` (`_types.py:1441-1465`); `protected_data` carries opaque/redacted reasoning across serialization.
- **Harness layer adds an explicit ReAct‑style outer loop with user feedback.** `AgentLoopMiddleware` (`_harness/_loop.py:212`) re‑invokes the whole agent in a `while True`, calls a `should_continue` predicate (sync or async), exposes `iteration` / `last_result` / `progress` / `feedback` kwargs, and injects a "Progress so far: …" `user` message back to the agent (`_harness/_loop.py:704-708`).
- **Judge pattern implemented as structured output, with a text fallback for clients that ignore `response_format`.** `JudgeVerdict(BaseModel) { answered, reasoning }` (`_harness/_loop.py:99-111`); the loop builds the judge messages from the original request verbatim (so multi‑modal inputs survive — `_harness/_loop.py:170-181`) and the `reasoning` flows back as the next iteration's "feedback" (`_harness/_loop.py:198-206`).
- **Two unrelated caps share the name `DEFAULT_MAX_ITERATIONS`.** Tool‑loop cap = 40 (`_tools.py:90`); harness outer‑loop cap = 10 / 5 for judge (`_harness/_loop.py:117, 122`).
- **Compaction is provider‑level rather than loop‑level.** `CompactionProvider.before_run` and `after_run` mutate `context_messages` / session‑state lists (`_compaction.py:1249-1304`). The function loop never compacts in place; each iteration passes `prepped_messages` to the chat client, which projects through any registered `compaction_strategy` (`_tools.py:374-389`).

## Notable Patterns

- **Two unconnected iteration counters, deliberately scoped.** A model roundtrip (`attempt_idx` in `_tools.py:2567`) and an agent re‑invocation (`iteration` in `_harness/_loop.py:454`) measure different things and are budgeted separately. Code search returns both `DEFAULT_MAX_ITERATIONS` constants.
- **Per‑iteration metrics without per‑iteration telemetry hooks.** `aggregated_usage` is built per pass via `add_usage_details` (`_tools.py:2605, 2685`) but is surfaced only at the end; callers must wrap `ChatMiddleware` to capture each pass individually.
- **Standalone structured judge.** `_build_judge_condition` calls the judge *directly* (no agent tools, session, or middleware) with fresh messages — explicitly avoiding recursion (`_harness/_loop.py:154-160`).
- **Progress log + session snapshot/restore for `fresh_context`.** With `fresh_context=True`, the loop snapshots `session.to_dict()` once and restores `service_session_id` and `state` between iterations so continuity is carried only by the progress log (`_harness/_loop.py:417, 423-437, 507-510, 592-596`).
- **Non‑overlapping verdict markers.** `JUDGE_VERDICT_DONE` ("VERDICT: DONE") and `JUDGE_VERDICT_MORE` ("VERDICT: MORE") are deliberately non‑substring of each other nor of the `{"answered": …}` JSON field (`_harness/_loop.py:66-72, 191-193`); **MORE wins ambiguous cases** to avoid premature stop (`_harness/_loop.py:188-193`).
- **`add_tools(...)` / `remove_tools(...)` from inside a tool.** `FunctionInvocationContext.tools` is the live run‑local tools list (`_middleware.py:264-285`); mutations take effect on the **next** model iteration only (`_tools.py:2544-2550`).
- **Per‑call persisted history opt‑in.** `PerServiceCallHistoryPersistingMiddleware` (`_sessions.py:570-744` per 03.01 evidence) wraps the chat client so the local transcript survives a mid‑loop crash; the harness enables it by default (`_harness/_agent.py:519`).

## Tradeoffs

- **No textual ReAct parser.** The cadence relies entirely on the provider's structured `function_call`; if a provider stops emitting structured items the loop simply ends at `action="return"` (`_tools.py:2346-2357`) without retrying with a `Thought/Action` prompt. A user wanting a "scratchpad must be in text" guarantee must wire their own prompt and parser middleware.
- **Raw observations by default.** `FunctionTool.parse_result` and `Content.from_function_result` pass through JSON dumps and Pydantic objects verbatim (`_tools.py:818-867`); there is no built‑in token cap on `Content.items[].text`. Reduction is opt‑in via `ToolResultCompactionStrategy` (`_compaction.py:820`).
- **Loop split across files.** Iterative loop in `_tools.py`, reasoning/storage/observation types in `_types.py`, persistence in `_sessions.py`, harness re‑invocation in `_harness/_loop.py`, compaction in `_compaction.py`. Tracing "one step" requires four file reads.
- **Two `DEFAULT_MAX_ITERATIONS` constants share the name.** Tool‑loop cap is 40 (`_tools.py:90`); harness outer‑loop cap is 10 / 5 (`_harness/_loop.py:117, 122`). Search hits collide.
- **Conversation‑managed APIs force a unique `prepped_messages` strategy.** With `response.conversation_id is not None` the loop only retains the latest assistant message and tool result (`_tools.py:2613, 2654-2661`); reasoning text emitted earlier in the assistant message can be lost from `prepped_messages` (it is *not* re‑added unless the response includes them in the trailing tool result message).
- **Harness loop is opt‑in (`ExperimentalFeature.HARNESS`) and uses a separate parse‑free callback contract.** `should_continue` may be sync or async and may return either `bool` or `(bool, feedback)`; building a textual ReAct parser on top is left to the caller (`_harness/_loop.py:131-138, 666-669`).
- **Reasoning content is provider‑dependently populated.** `text_reasoning` and `protected_data` are types, not mandates; OpenAI Responses and Anthropic populate them differently, and a chat client that emits no reasoning yields a `assistant` message with only `text`/`function_call`.
- **Successful textual fallback for the judge exists but is fragile to client quirks.** `MORE` wins ambiguous text (`_harness/_loop.py:188-193`), so a judge prompt that occasionally omits the marker will keep looping rather than halting on a near‑complete answer.

## Failure Modes / Edge Cases

- **`max_function_calls` is best‑effort.** A single batch of 20 parallel calls plus a cap of 10 still runs all 20 (`_tools.py:2636-2638`).
- **`tool_choice="required"` infinite loop** mitigated by resetting to `None` after one iteration (`_tools.py:2648-2652, 2819-2823`).
- **Orphan `function_call` after cap.** Mitigated by a forced text‑only final pass (`_tools.py:2664-2696` and stream mirror `2835-2866`); covered by `test_max_iterations_no_orphaned_function_calls` (`test_function_invocation_logic.py:1047-1100`).
- **Tool error in a row reaching cap.** `errors_in_a_row >= max_errors` returns `action="stop"` and flips `tool_choice="none"` (`_tools.py:2248-2258`); model is forced to speak prose instead of looping forever.
- **`UserInputRequiredException` routing.** Caught in `_try_execute_function_calls`, returned as a normal `function_result` ("Tool requires user input but no request details were provided.") when there is no content, otherwise propagated (`_tools.py:1800-1819`).
- **`fresh_context=True` + attached session.** Snapshot/restore must include service‑side conversation id; the loop does `_restore_session(...)` on every iteration and the snapshot is captured once at the start (`_harness/_loop.py:417, 423-437, 510-511`).
- **`prepare_messages` for service APIs loses messages outside the last `function_result`.** For services that return `response.conversation_id`, `prepped_messages.clear(); prepped_messages.append(response.messages[-1])` strips earlier reasoning/user items from the next round; the service retains them server‑side, but a stateless transport reading `prepped_messages` would lose them.
- **`include_detailed_errors=False`** can suppress the real error cause from the model, leaving it guessing — but it keeps the observation a single short line that plays well with compaction.
- **Reasoning items are NOT merged into `function_result`.** A `text_reasoning` item appears as its own `Content` in the assistant message; tools that read `assistant_message.text` (via `Message.text` at `_types.py:1762-1768`) get only the `text` items, missing reasoning unless the caller reads `contents` directly.
- **Harness loop service‑managed conversation id in `fresh_context`.** The snapshot is restored between iterations, but the harness loop's outer re‑invocation clears any service‑side conversation id set during the iteration (`_harness/_loop.py:507-510, 592-596`); the next model call starts with a clean conversation.
- **Judge marker ambiguity.** When `JudgeVerdict` is ignored, `MORE` wins (`_harness/_loop.py:188-193`); a near‑complete answer without a marker is treated as incomplete.

## Future Considerations

- **Add an opt‑in "scratchpad prompt" that asks the model to emit `Thought:` / `Action:` / `Observation:` blocks.** The framework already has the pieces — `DEFAULT_HARNESS_INSTRUCTIONS` (`_harness/_agent.py:44-59`) and `DEFAULT_NEXT_MESSAGE` (`_harness/_loop.py:59`) — but neither is the structured ReAct prompt the dimension describes; a small parser middleware could close the gap for providers that don't emit reasoning.
- **Default‑on observation summarization for the tool loop.** `ToolResultCompactionStrategy` exists but is wired only when the harness agent is used with token budgets (`_harness/_agent.py:72-120`). Promoting it to a default post‑iteration compact step would bound `prepped_messages` growth even for plain `Agent` users.
- **Unify or rename `DEFAULT_MAX_ITERATIONS`.** Tool loop (`_tools.py:90`) and harness loop (`_harness/_loop.py:117`) share the name but mean different things; a `TOOL_LOOP_MAX_ITERATIONS` / `HARNESS_LOOP_MAX_ITERATIONS` rename would help.
- **Surface per‑iteration usage and per‑iteration finish reasons.** `add_usage_details` is currently accumulated and dropped at the end (`_tools.py:2605, 2685`); `attempt_idx` is internal but unused outside the budget state.
- **Treat `MiddlewareTermination` inside the tool loop as a first‑class cadence signal**, not an exception. Today it bubbles from `_auto_invoke_function` (`_tools.py:1612-1629`) and forces a tool‑result‑less `function_approval_request` to be carried forward; a typed signal would be clearer for harness middleware that wants to intervene between iterations.

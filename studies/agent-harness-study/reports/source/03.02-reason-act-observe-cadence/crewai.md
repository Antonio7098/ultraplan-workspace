## Summary

CrewAI ships a textbook **ReAct** parser for text-mode agents and a **native function-calling** path for providers that support it; both paths run inside an explicit thought → tool → observation loop with stop-word trimming of `Observation:`, tool-result "remember-format" reinjection every N steps, structured planning/observation for Plan-and-Act, and a separate hookable cache + summarization layer that compacts observations when context overflows. Reasoning is rendered in-band (`Thought:` lines in the same assistant message) and hidden behind the agent's chosen `Observation:` stop sequence; failed observations are returned as text (`Error executing tool: …` or `Error: Failed to parse tool arguments as JSON: …`) so the model sees them on the next iteration. Two coexisting executors implement the cadence: the legacy `CrewAgentExecutor` (`lib/crewai/src/crewai/agents/crew_agent_executor.py:98`) which is explicitly deprecated, and the new `crewai.experimental.AgentExecutor` (`lib/crewai/src/crewai/experimental/agent_executor.py:164`) — a Flow-based orchestrator that adds per-step `StepObservation` (Pydantic) and a low/medium/high `reasoning_effort` policy. Tool-call content is normalized to OpenAI-style `{role: "assistant", content: None, tool_calls: [...]}` + `{role: "tool", tool_call_id, name, content}` messages (`lib/crewai/src/crewai/agents/crew_agent_executor.py:843-866, 1082-1088`) so providers that return native `function_call` items speak the same wire format as the text-parsed path. The trade-off: every "observation" is rendered as a string appended to the assistant/user message history, so large tool outputs dominate the context unless `respect_context_window=True` triggers `summarize_messages` (`lib/crewai/src/crewai/utilities/agent_utils.py:920`).

## Rating

**7 / 10** — The cadence is implemented end-to-end across two executors with explicit regex-based ReAct parsing (`lib/crewai/src/crewai/agents/parser.py:62`), structured tool-call extraction that covers OpenAI, Anthropic, Gemini and Bedrock shapes (`lib/crewai/src/crewai/utilities/agent_utils.py:1191-1231`), per-step `post_tool_reasoning` reflection prompts (`lib/crewai/src/crewai/agents/crew_agent_executor.py:801-806, 1082-1087`), a Plan-and-Act `StepObservation` Pydantic model with three reasoning-effort levels (`lib/crewai/src/crewai/utilities/planning_types.py:212-278`), a hookable before/after tool lifecycle (`lib/crewai/src/crewai/utilities/tool_utils.py:96-141`), and an LLM-driven summarizer that compacts observations on context overflow (`lib/crewai/src/crewai/utilities/agent_utils.py:712-749, 920-1003`). The repository includes 13+ dedicated parser tests (`lib/crewai/tests/agents/test_crew_agent_parser.py`) and explicit loop-fallback tests (`lib/crewai/tests/agents/test_native_tool_calling.py:1271`). However: (a) reasoning is **not first-class** — there is no `text_reasoning` content type; the LLM must emit `Thought:` in-band with the action; (b) the legacy executor is `DeprecationWarning`-emitted at construction (`lib/crewai/src/crewai/agents/crew_agent_executor.py:145-151`) and the new executor's iteration loop is a 3206-line Flow with no per-iteration unit tests asserting the cadence; (c) the system still leans on `stop=["Observation"]` and a regex parser (`lib/crewai/src/crewai/agent/core.py:1062`) — providers without strong structured-tool support fall back to brittle text parsing, and `Format: I missed the 'Action:' after 'Thought:'` is fed back into the prompt (`lib/crewai/src/crewai/utilities/agent_utils.py:680`) which sometimes just produces another parse error.

## Answers to Dimension Questions

1. **How does the model decide actions?**
   Two complementary paths. (a) **Native function calling** when the LLM exposes `supports_function_calling()`: the executor builds OpenAI-format schemas (`lib/crewai/src/crewai/utilities/agent_utils.py:154-209` via `convert_tools_to_openai_schema`), passes them via `tools=openai_tools`, and parses the response through `extract_tool_call_info` (`utilities/agent_utils.py:1191-1231`) or `_parse_native_tool_call` (`agents/crew_agent_executor.py:809-841`). (b) **ReAct text mode** otherwise: the executor prompts the model with the canonical `Thought / Action / Action Input / Observation` template (`lib/crewai/src/crewai/translations/en.json:12`) and parses the LLM's reply with the regex `ACTION_INPUT_REGEX` (`agents/constants.py:18-20`), falling through to `json_repair.repair_json` if the `Action Input` JSON is malformed (`agents/parser.py:161-178`). The `Plan-and-Act` variant adds an explicit `create_reasoning_plan` function-call LLM call (`utilities/reasoning_handler.py:368-385`) and a separate `StepExecutor` that asks the LLM to pick a single tool per step (`agents/step_executor.py:317-578`).

2. **How are observations returned?**
   Three wire formats, one per path. **Native single-call**: a `{role: "tool", tool_call_id, name, content}` message (`agents/crew_agent_executor.py:1082-1088`). **Native batched**: a single assistant message with `tool_calls=[...]` followed by N tool messages (`agents/crew_agent_executor.py:843-866` + `1701-1801`). **Text-mode**: the assistant text is mutated in place to append `\nObservation: {result}` (`utilities/agent_utils.py:616`). After every observation the executor injects a `post_tool_reasoning` user message — `"Analyze the tool result. If requirements are met, provide the Final Answer. Otherwise, call the next tool."` (`translations/en.json:17`) — so the model is explicitly told what to do with the new information (`agents/crew_agent_executor.py:801-806, 1082-1087`; `experimental/agent_executor.py:1623-1628, 1644-1649`). Tool outputs are formatted through `format_output_for_agent` when the tool exposes that hook (`utilities/agent_utils.py:1386-1396`), and large text observations may be wrapped as multimodal content blocks (e.g. `VISION_IMAGE:` sentinel becomes an `image_url` block — `agents/step_executor.py:462-502`). The model therefore sees **the raw tool string** plus, when present, an optional structured image payload.

3. **Is reasoning visible or hidden?**
   **Visible, but only as free text.** There is **no first-class reasoning content type** in `crewai.types.LLMMessage` (only `system`/`user`/`assistant`/`tool` roles with `content` / `tool_calls`). The model's reasoning is whatever it wrote in the `Thought:` line of its previous turn, which `_extract_thought` (`agents/parser.py:131-146`) and `formatted_answer.thought` capture — but the executor only persists the **assistant message text** to history (`agents/crew_agent_executor.py:432` via `_append_message(formatted_answer.text)`), so the LLM sees its own `Thought:` on the next iteration. The system prevents the LLM from leaking the next iteration's `Observation:` by setting `stop=["\nObservation:"]` via `stop_words` (`agent/core.py:1062`) and threading it through `call_stop_override` (`utilities/agent_utils.py:261-277`). Reasoning **outside** the ReAct template (e.g. `o1`-style hidden reasoning) is **not represented** in the message model — the framework relies on text-mode `Thought:` blocks or, when the provider supports it, structured `function_call` outputs. The Plan-and-Act layer adds structured per-step reflection via `StepObservation` (`utilities/planning_types.py:212-278`) — but that is the planner's view of step outcomes, not the LLM's own chain-of-thought.

4. **Are failed observations fed back?**
   **Yes, every failure mode returns a string observation to the model.** Tool execution errors become `\nObservation: Error executing tool: {e}` (`experimental/agent_executor.py:1618-1620`) and the action text is updated with the error so the next LLM call sees it. JSON parse failures of native tool args produce `Error: Failed to parse tool arguments as JSON: {e}. Please provide valid JSON arguments for the '{name}' tool.` (`utilities/agent_utils.py:1653-1662`). Tool-selection misses raise `Exception` in `_select_tool` with `f"Action '{tool_name}' don't exist, these are the only available Actions:\n{self.tools_description}"` (`tools/tool_usage.py:784-793`). Max-usage breaches inject `Tool '{name}' has reached its usage limit of {N} times and cannot be used anymore.` (`agents/crew_agent_executor.py:981`). Hook-blocked calls inject `Tool execution blocked by hook. Tool: {name}` (`agents/crew_agent_executor.py:978`). `OutputParserError` is converted into a user message `I did it wrong. Invalid Format: I missed the 'Action:' after 'Thought:'…` and appended to history (`utilities/agent_utils.py:660-695`) so the next LLM call sees the format hint. **However**, there is **no automatic consecutive-error cap that flips `tool_choice` to `none`** the way `agent-framework` does; the model can keep emitting invalid tool calls until `max_iter` is hit, at which point `handle_max_iterations_exceeded` appends `force_final_answer` and asks for a final answer in one shot (`utilities/agent_utils.py:293-350`).

5. **Can observations be compacted?**
   **Yes, in two layers.** (a) **Per-tool cache**: `ToolsHandler.cache` deduplicates identical `(tool_name, input)` pairs and returns the cached string on subsequent calls (`agents/tools_handler.py:26-52`; cache check at `agents/crew_agent_executor.py:929-937`). (b) **Context-window summarization**: `summarize_messages` (`utilities/agent_utils.py:920-1003`) splits `messages` into chunks sized by `llm.get_context_window_size()`, asks the LLM to produce structured `<summary>` blocks, and replaces the entire non-system history with a single user message that contains the merged summary — preserving any attached `files` (`utilities/agent_utils.py:941-1003`). The trigger is the `is_context_length_exceeded` exception during the LLM call (`utilities/agent_utils.py:698-709`) and only fires when `respect_context_window=True` on the executor; otherwise the system `SystemExit`s with `"Context length exceeded and user opted not to summarize"` (`utilities/agent_utils.py:732-749`). **Not built in**: an always-on per-iteration compact step that bounds `messages` growth regardless of context length; without `respect_context_window=True` the history only grows. The `LiteAgent` does not implement compaction itself (`lite_agent.py:860-972`) — it relies on the same `handle_context_length` utility.

## Architectural Decisions

- **Two coexisting executors**: the legacy `CrewAgentExecutor` (`agents/crew_agent_executor.py:98`) raises `DeprecationWarning` at construction (`agents/crew_agent_executor.py:145-151`) and the canonical implementation is now `crewai.experimental.AgentExecutor`, a `Flow[AgentExecutorState]` (`experimental/agent_executor.py:164`) with explicit `@router(...)` nodes (`call_llm_and_parse`, `execute_tool_action`, `execute_native_tool`, `check_max_iterations`, `observe_step_result`).
- **State is a single Pydantic model**: `AgentExecutorState` holds `messages`, `iterations`, `current_answer`, `pending_tool_calls`, `observations: dict[int, StepObservation]`, `todos: TodoList`, `plan`, `replan_count`, `last_replan_reason`, `execution_log` (`experimental/agent_executor.py:126-160`) — one mutable object threaded through the Flow so listeners and routers see the same snapshot.
- **Parser as a module-level function, not a class**: `parse(text)` (`agents/parser.py:62`) is pure (no I/O, no side effects) — easy to test in isolation (13+ dedicated tests in `tests/agents/test_crew_agent_parser.py`). The text-mode loop calls `process_llm_response` (`utilities/agent_utils.py:567-586`) which wraps `parse` with `OutputParserError` recovery.
- **One observation wire format regardless of provider**: `extract_tool_call_info` (`utilities/agent_utils.py:1191-1231`) and `_parse_native_tool_call` (`agents/crew_agent_executor.py:809-841`) collapse OpenAI, Anthropic (`ToolUseBlock`), Gemini (`function_call`), and Bedrock (`toolUseId`) shapes into `(call_id, func_name, func_args)`. The assistant message is always `{role: "assistant", content: None, tool_calls: [...]}` (`agents/crew_agent_executor.py:843-866`) and the tool message is always `{role: "tool", tool_call_id, name, content}` (`agents/crew_agent_executor.py:1082-1088`), so downstream LLM adapters only see OpenAI-compatible content.
- **Provider-fallback with prompt downgrade, not silent retry**: when the native path raises a "does not support tools" error (`is_native_tool_calling_unsupported_error` at `utilities/agent_utils.py:1285-1288`), the executor appends `build_text_tool_calling_fallback_message` to `messages` and routes to `_invoke_loop_react` / `_ainvoke_loop_react` (`agents/crew_agent_executor.py:577-579, 1399-1401`). This is covered by `test_native_tool_loop_falls_back_when_provider_rejects_tools` (`tests/agents/test_native_tool_calling.py:1271`).
- **Stop-word-driven truncation**: `_llm_stop_words_applied` (`utilities/agent_utils.py:261-277`) pushes `stop=["\nObservation:"]` into `call_stop_override` for the duration of one call so the LLM stops at the observation boundary, then `process_llm_response` strips a stray `Observation:` if the LLM emits it anyway (`utilities/agent_utils.py:583-584`). This avoids the model ever writing its own observation.
- **Plan-and-Act is a separate code path, not a flag**: `AgentReasoning` (`utilities/reasoning_handler.py:107-598`) produces a `ReasoningPlan(plan, steps, ready)` Pydantic object before any tool call; each `PlanStep` becomes a `TodoItem` (`utilities/planning_types.py`), and `StepExecutor` runs each in isolation with its own `messages` list (`agents/step_executor.py:233-247`). Observations between steps flow through `PlannerObserver.observe` → `StepObservation` Pydantic model (`utilities/planning_types.py:212-278`) and influence replan decisions via `decide_next_action` (`experimental/agent_executor.py:814-895`).
- **Reasoning effort as a single knob**: `planning_config.reasoning_effort` ∈ `{low, medium, high}` controls whether the planner LLM is called per step (`experimental/agent_executor.py:518-535, 609-652`). Low: heuristic observation only. Medium: replan on failure. High: full `decide_next_action` → `refine_and_continue` chain.
- **Iteration cap is per agent roundtrip, not per LLM call**: `max_iter` defaults to 25 (`agents/agent_builder/base_agent_executor.py:27`) and is checked at the top of each loop iteration (`agents/crew_agent_executor.py:343, 503`; `experimental/agent_executor.py:2119`). When exceeded, `handle_max_iterations_exceeded` issues **one final LLM call** with a `force_final_answer` nudge (`utilities/agent_utils.py:293-350`).
- **Tool-call lifecycle is hookable**: `get_before_tool_call_hooks()` / `get_after_tool_call_hooks()` (`utilities/tool_utils.py:105-141`) iterate global hook registries; a hook returning `False` short-circuits the call with a "Tool execution blocked by hook" observation (`agents/crew_agent_executor.py:967-969, 977-978`).

## Notable Patterns

- **Co-located parser tests**: 14+ tests in `tests/agents/test_crew_agent_parser.py` cover every JSON edge case (`_with_incomplete_json`, `_with_quotes`, `_with_curly_braces`, `_safe_repair_json_missing_quotes`, `_safe_repair_json_unclosed_brackets`, `_safe_repair_json_extra_commas`, etc.), so the regex+repair_json pipeline is regression-safe.
- **Tool-cache-as-observation**: a tool's output that is cacheable by `(name, input)` becomes effectively a memoized observation (`agents/tools_handler.py:26-52`), so the same `SearchTool("CrewAI")` call returns identical bytes and the model can't drift on rerun.
- **"Remember-format" reinjection**: every N tool calls (`_remember_format_after_usages`, default 3; lower for big-model OpenAI families — `tools/tool_usage.py:114-119`), the tool result string is suffixed with the full tool-schema block so the model can't drift away from the canonical tool format (`tools/tool_usage.py:716-726`). This is an in-observation reminder pattern unique to text-mode agents.
- **`StepExecutor` as a black-box sub-agent**: `StepExecutionContext(task_description, task_goal, dependency_results)` (`utilities/step_execution_context.py:13-41`) is **frozen** and contains **only final dependency results**, never LLM message histories. `StepResult(success, result, error, tool_calls_made, execution_time)` (`utilities/step_execution_context.py:44-64`) is the only thing returned. The sub-agent's internal `messages` list is garbage-collected after each step.
- **`vision sentinel` as an observation-bridge format**: tools that produce images return `VISION_IMAGE:<media_type>:<base64>` instead of a base64 blob; the executor parses the sentinel (`agents/step_executor.py:462-472`) and replaces the plain-text observation with a multimodal `image_url` content block (`agents/step_executor.py:488-501`). This is the framework's only first-class multimodal-observation bridge.
- **`mark_cache_breakpoint` for provider-level prompt caching**: stable prefix (system + user prompt) is flagged with `cache_breakpoint: True` (`agents/crew_agent_executor.py:192-204`; `experimental/agent_executor.py:2775-2789`) so providers like Anthropic can cache the prefix; per-step the prefix doesn't change so cost drops as iteration count rises. Provider adapters strip the marker if the cache directive is unsupported (`llms/cache.py:35-37`).
- **Three-tier observability**: `AgentReasoning*Event` (planning), `ToolUsageStartedEvent` / `ToolUsageFinishedEvent` / `ToolUsageErrorEvent` (cadence), `AgentLogsStartedEvent` / `AgentLogsExecutionEvent` (per-step start/end) — all emitted through `crewai_event_bus` so the cadence can be traced without monkey-patching the loop.
- **Parser-error resilience in the prompt**: `MISSING_ACTION_AFTER_THOUGHT_ERROR_MESSAGE` and `MISSING_ACTION_INPUT_AFTER_ACTION_ERROR_MESSAGE` (`agents/constants.py:8-13`) are written in first-person ("I did it wrong…") so the LLM treats them as its own self-correction rather than an external rejection.

## Tradeoffs

- **No first-class reasoning content type.** `crewai.types.LLMMessage` has only `system / user / assistant / tool` roles and string-or-list `content`. The model's `Thought:` is embedded in the assistant message text; providers that emit native `text_reasoning` (Anthropic, OpenAI o1) lose that structure when fed through CrewAI's content adapters.
- **Regex-driven action parsing is brittle.** `ACTION_INPUT_REGEX` (`agents/constants.py:18-20`) requires `Action: name\nAction Input: {...}` in that exact order. Models that emit `Action Input` first, or wrap the action in markdown fences, will fall through to `OutputParserError` and the format-error prompt fires. The 14+ parser tests cover only the **success path**; there is no test that verifies `Action: ... Action Input: ...` with intervening prose or trailing fences.
- **Stop-word fragility.** `\nObservation:` only stops at a newline boundary. A model that writes `Action: ... Observation: my_observation_thought\nAction Input: ...` (one-line form) will have the LLM cut off after `Observation:` — then the executor's parser fails to find `Action Input:` and triggers the format-error recovery.
- **Compaction only on overflow.** `summarize_messages` fires only when the LLM raises a context-length exception (`utilities/agent_utils.py:698-709`). A long-running session that stays under the model's nominal window keeps every observation verbatim and never compacts — context can grow unbounded until the exception.
- **Two executors with overlapping semantics.** `CrewAgentExecutor` is deprecated (`agents/crew_agent_executor.py:145-151`) but its `_invoke_loop_react`, `_invoke_loop_native_tools`, `_handle_native_tool_calls`, `_execute_single_native_tool_call`, `_append_assistant_tool_calls_message`, `_append_tool_result_and_check_finality`, `_handle_agent_action`, `_setup_messages`, `_show_logs` are mirrored with slightly different signatures in the `experimental.AgentExecutor`. There is no shared helper extracted yet — code drift between the two is possible.
- **No automatic consecutive-error cap.** A model that emits three bad tool calls in a row gets three observations back and continues; only `max_iter` stops the loop. `agent-framework` flips `tool_choice="none"` after a configurable `max_consecutive_errors_per_request`; CrewAI has no equivalent.
- **`memory._save_to_memory` runs only on `AgentFinish`** (`agents/agent_builder/base_agent_executor.py:31-65`). Mid-loop observations are never persisted to long-term memory — only the final task result is. If the agent is interrupted (raise + recovery) the in-flight tool results stay in `messages` only.
- **`LiteAgent` has no `execution_lock` or `is_executing` guard** (`lite_agent.py:860-972`), so a `LiteAgent` instance can be invoked twice concurrently and both invocations will mutate `self._messages` and `self._iterations` — a divergence from the legacy `CrewAgentExecutor` / `AgentExecutor` (which both use `with self._execution_lock:`).

## Failure Modes / Edge Cases

- **Format-error loop.** `OutputParserError` → `handle_output_parser_exception` appends a self-correction prompt, but if the model keeps emitting bad output the loop runs until `max_iter` and then `handle_max_iterations_exceeded` forces one Final-Answer call. There is no guarantee the Final-Answer call succeeds — it just goes through `format_answer` again (`utilities/agent_utils.py:342-350`).
- **`Action Input` JSON decode failure in text mode.** `_safe_repair_json` (`agents/parser.py:161-178`) uses `json_repair.repair_json` to fix minor issues, but the fallback returns the original string when repair yields `'""'` or `'{}'` (`agents/constants.py:17`). A model that emits a tool name with no payload will then hit `_tool_calling` with empty args, and depending on the tool's signature may raise a `ValidationError` that surfaces as `"Error: ..."` from the tool itself.
- **Parallel native tool calls with `result_as_answer=True` are short-circuited.** When a batch contains any tool with `result_as_answer=True`, the legacy executor skips parallel execution and falls back to sequential so the answer-short-circuit can stop immediately (`agents/crew_agent_executor.py:698-724`). The `experimental.AgentExecutor` does the same check before parallelizing (`experimental/agent_executor.py:1827-1858`).
- **`max_iteration` forced final answer may itself raise.** `handle_max_iterations_exceeded` raises `ValueError("Invalid response from LLM call - None or empty.")` if the final LLM call returns nothing (`utilities/agent_utils.py:334-340`). Callers that don't catch this get an unhandled exception instead of a clean "agent failed" answer.
- **Context-overflow SystemExit when `respect_context_window=False`.** `handle_context_length` calls `raise SystemExit("Context length exceeded and user opted not to summarize.")` (`utilities/agent_utils.py:747-749`), which is not catchable as `Exception` by the loop's `except Exception` blocks. Crews that opt out of compaction will be terminated abruptly.
- **Vision sentinel parse failure.** `StepExecutor._parse_vision_sentinel` returns `None` if the sentinel prefix is missing or the `:` separator is at position 0 (`agents/step_executor.py:462-472`); the executor falls back to plain text `Observation: {raw}` which floods the context with base64 noise.
- **`step_timeout` in `StepExecutor._execute_text_parsed`** is checked at the top of every iteration but not inside the tool call itself (`agents/step_executor.py:336-340`); a single tool that hangs for 60s will blow past `step_timeout=30s`.
- **`LiteAgent` reentrancy.** Two concurrent `kickoff()` calls on the same `LiteAgent` race on `self._messages` and `self._iterations`; the loop has no guard, so the second call resets `_iterations` while the first is still running.
- **Delegate tool short-circuits memory.** `if f"Action: {sanitize_tool_name('Delegate work to coworker')}" in output.text: return` (`agents/agent_builder/base_agent_executor.py:40-41`) — when the final action is delegation, nothing is saved to memory, even if the delegation succeeded.

## Future Considerations

- **Promote `StepExecutor`'s message list to a public API.** Today it lives entirely inside one method (`agents/step_executor.py:126-231`). Exposing it would let callers implement custom post-step reflection without monkey-patching.
- **Default-on observation summarization.** Today `summarize_messages` only fires on context overflow (`utilities/agent_utils.py:920`). A budget-based periodic compaction (e.g. every 8 tool calls) would bound growth proactively.
- **First-class reasoning content type.** Replace the in-band `Thought:` line with a structured `reasoning` field on the assistant message. This would let native reasoning providers (Anthropic, o1) flow their hidden reasoning into memory without losing it on serialization.
- **Replace regex parsing with a structured-output parser.** `parse()` (`agents/parser.py:62`) could be re-implemented as a Pydantic `Action | Finish` discriminated union via `response_model`. The current text-mode path is only needed because some providers don't support native tools; a Pydantic response model gives structured parsing without depending on the provider's `function_call` support.
- **Share helpers between `CrewAgentExecutor` and `AgentExecutor`.** The `_invoke_loop_*`, `_execute_single_native_tool_call`, `_append_assistant_tool_calls_message`, `_append_tool_result_and_check_finality` blocks are duplicated (`agents/crew_agent_executor.py:667-1071` mirrors `experimental/agent_executor.py:1654-1858` plus `utilities/agent_utils.py:1399-1663`). A shared module would reduce drift.
- **Add a consecutive-error cap** that flips the executor into a "Final Answer only" mode after N tool errors, mirroring the `max_consecutive_errors_per_request` pattern from agent-framework.
- **Persist mid-loop observations to memory** when a session is paused via `invoke(..., _resuming=True)` so an interrupted agent can be resumed with full prior context rather than only the final `AgentFinish`.

## Questions / Gaps

- **Is the `Thought:` block ever preserved verbatim across iterations?** `_append_message(formatted_answer.text)` (`agents/crew_agent_executor.py:432, 1270`) stores the full assistant text including the `Thought:` line, so the model should see it on the next call — but no test asserts this round-trip. **No clear evidence found** in `tests/agents/`.
- **Does `LiteAgent` expose a public way to inspect the current observation history?** `_messages` is private; `kickoff()` returns `LiteAgentOutput` only. **No clear evidence found** for a logging or history accessor that includes raw tool outputs.
- **How does `cache_breakpoint: True` interact with `assistant` and `tool` messages that are appended per-iteration?** The markers are set on `system` + `user` only (`agents/crew_agent_executor.py:192-204`), so the cacheable prefix ends at the user prompt. New `tool`/`assistant` messages become the "volatile" suffix — but is this consistent across all LLM adapters that honor cache directives? **No clear evidence found** in a per-adapter test that asserts which segment is cache-hit.
- **What happens if `max_iter` is reached mid-batch in native mode?** `_handle_native_tool_calls` short-circuits on `result_as_answer` (`agents/crew_agent_executor.py:697-784`) but the only `max_iter` check is at the top of `_invoke_loop_native_tools` (`agents/crew_agent_executor.py:503-513`). If the LLM returns a 20-tool batch and `max_iter=1`, the whole batch is processed before the next iteration check. **No clear evidence found** for a per-call budget enforcement.
- **Is the `pre_tool_reasoning` reflection prompt ever skipped?** It is unconditionally appended after every tool result (`agents/crew_agent_executor.py:801-806, 1082-1087`; `experimental/agent_executor.py:1623-1628, 1644-1649`). If the LLM is a provider that supports structured reasoning, the extra `user` message may be redundant or noisy. **No clear evidence found** for a config flag that disables it.
- **How does the executor distinguish "Final Answer:" in a tool's output from the model's own Final Answer marker?** `FINAL_ANSWER_ACTION = "Final Answer:"` (`agents/constants.py:7`) is matched against the LLM's reply (`agents/parser.py:92`). If a tool returns a string containing `Final Answer: foo`, the model's next observation will include it verbatim — but a future model call would then see its own `Final Answer:` substring in the observation and might prematurely terminate. **No clear evidence found** for an escape mechanism (e.g. `&#10;Final Answer:`) that distinguishes tool-output text from agent-output text.

## Summary

The tool-calling roundtrip is implemented as a **chat-client-layer iterative loop** (`FunctionInvocationLayer`, `python/packages/core/agent_framework/_tools.py:2388-2877`) layered on top of any `BaseChatClient` subclass (see layering at `_clients.py:978-996`). It relies entirely on **structured `Content` items** the provider returns — there is no `<tool_call>`/`<tool_result>` XML or text parsing. Tool calls appear as `Content(type="function_call", call_id, name, arguments)` (`_types.py:794-815`) and results as `Content(type="function_result", call_id, result, items, exception)` (`_types.py:818-874`). The execution pipeline is `_extract_function_calls` → `_try_execute_function_calls` → `_auto_invoke_function` (per call) → `FunctionTool.invoke` → `_handle_function_call_results` (`_tools.py:2167-2187, 1658-1830, 1424-1645, 2217-2270`).

Argument validation runs **twice** in the pipeline: first inside `FunctionTool.invoke` via Pydantic (`_tools.py:649-669`) and the lightweight JSON-Schema `_validate_arguments_against_schema` (`_tools.py:1097-1144`), and again in the loop wrapper (`_tools.py:1510-1518`) — the second validation catches arguments the loop sees before the tool even runs. `Content.parse_arguments()` (`_types.py:1539-1579`) is the front door for converting the model-emitted JSON string into a `dict`, and falls back to `{"raw": original_str}` on JSON failure rather than raising.

Invalid tool calls are **never repaired silently** — instead, the failure is converted into a `function_result` carrying `"Error: Argument parsing failed. …"` plus `exception=str(exc)` and routed back to the model so it can correct itself on the next roundtrip (`_tools.py:1519-1528`). Runtime exceptions use the same channel: by default `"Error: Function failed."` plus `exception=str(exc)`; with `include_detailed_errors=True`, the full `repr` is appended (`_tools.py:1556-1568, 1632-1645`). Unknown tools take one of two paths depending on `terminate_on_unknown_calls`: by default they return a synthetic `function_result` so the loop continues (`_tools.py:1469-1479`), but `True` raises `KeyError` and aborts (`_tools.py:1719-1720`).

Errors **continue the loop** under two separate budgets. `errors_in_a_row` is incremented whenever a tool returns an `exception` (`_tools.py:1855, 2248-2258, 2315-2322`); when it reaches `max_consecutive_errors_per_request` (default `3`, `_tools.py:91`), the loop flips `tool_choice="none"` and forces a final plain-text pass (`_tools.py:2632-2644` for non-stream, `2802-2816` for stream). A separate `max_function_calls` budget (default `None`) counts total tool invocations across the whole request and is checked *after* each parallel batch — best-effort, not strict (`_tools.py:2636-2644, 2808-2816`).

Approval is a first-class concept. `Content(type="function_approval_request")` (`_types.py:1213-1231`) and `Content(type="function_approval_response", approved)` (`_types.py:1234-1253`) are stable content types. A tool declared with `approval_mode="always_require"` (`_tools.py:398, 1147-1190`) triggers the loop to short-circuit: no execution, just `Content.from_function_approval_request(...)` plus the original `function_call` are placed back into the assistant message and the loop returns `action="return"` so the caller can show the user (`_tools.py:1712-1718, 1721-1755, 2228-2246`). Mixed batches where only some tools require approval use a session-state trick (`_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY` at `_tools.py:94, 1974-2032`) to hide approval requests for non-approval-required tools, which are replayed only when the visible batch resumes. A separate opt-in `ToolApprovalMiddleware` (`python/packages/core/agent_framework/_harness/_tool_approval.py:345-617`) adds standing per-tool or per-tool+arguments rules, async auto-approval callbacks, and streamed approval prompts — it is experimental (`@experimental(feature_id=ExperimentalFeature.HARNESS)` at `_harness/_tool_approval.py:345`) and requires an `AgentSession`.

## Rating

**8 / 10** — The roundtrip is implemented as a layered chat-client pipeline with a typed, Pydantic-validated `FunctionTool`, dual-mode argument validation (Pydantic for function-annotated tools, lightweight JSON Schema for declaration-only tools), a structured `Content` representation for every state, dual streaming/non-streaming loops, and four orthogonal safety knobs (`max_iterations`, `max_function_calls`, `max_consecutive_errors_per_request`, `terminate_on_unknown_calls`). All errors are converted to `function_result` items and surfaced to the model, with configurable detail levels. Approval is a first-class content type with mixed-batch routing. The code is heavily tested (`test_function_invocation_logic.py:1-4440`, `test_tools.py:1-1508`, `test_harness_tool_approval.py:1`).

Gaps that hold it below 9–10:
- **`Content.parse_arguments()` silently swallows JSON errors** (`_types.py:1577-1578`); a malformed `arguments` string becomes `{"raw": original}` and is only caught later as a Pydantic/Schema validation error — no diagnostic warning at parse time.
- **`terminate_on_unknown_calls=True` raises `KeyError`** (`_tools.py:1719-1720`) instead of returning an error result, so the loop cannot recover; a model that hallucinates a tool name kills the run.
- **`max_function_calls` is best-effort**: a parallel batch that exceeds the cap still runs to completion (`_tools.py:2636-2644`); the framework documents this but offers no per-batch pre-check.
- **The custom JSON-Schema validator** (`_validate_arguments_against_schema`, `_tools.py:1097-1144`) is a small hand-rolled subset: it covers `required`, primitive `type`, `enum`, and `additionalProperties: false`, but not `oneOf` / `$ref` / nested `anyOf` — declaration-only tools with complex schemas are effectively unchecked.
- **Result-parser failures fall back silently to `str(result)`** (`_tools.py:702-706, 765-767`) with a `WARNING` log; the model sees a `"<MyObject instance at 0x…>"` instead of a structured error, and there is no retry/raise path.
- **`function_invocation_configuration` is per-client, not per-call**, so adjusting `max_iterations` mid-run requires client mutation (`_tools.py:2540-2541` shows mutable options but not mutable config).

## Answers to Dimension Questions

1. **How are tool calls represented?**
   As structured `Content` items of `type="function_call"` (`_types.py:338-363, 794-815`). The model emits `call_id` + `name` + `arguments` (a JSON string or mapping); the framework never inspects textual `<tool_call>` / `<tool_result>` markers. Extraction is `_extract_function_calls(response)` (`_tools.py:2167-2187`) which dedupes by `call_id` and skips calls that already have matching `function_result`s. Provider-specific adapters map the wire-format items (e.g. OpenAI Responses `ResponseOutputFunctionCall`, `mcp_approval_request`, `code_interpreter_tool_call`) into the unified `Content` shape (see `_chat_client.py:2374-2399`).

2. **What happens if arguments are invalid?**
   The loop wrapper catches `ValidationError` and `TypeError` and converts the failure into a `Content.from_function_result(call_id, result="Error: Argument parsing failed. …", exception=str(exc), …)` (`_tools.py:1519-1528`); the same path handles missing required arguments, wrong types, and enum violations. Runtime tool exceptions become `Content.from_function_result(..., result="Error: Function failed.", exception=str(exc))` (`_tools.py:1556-1568, 1632-1645`). The detail level toggled by `FunctionInvocationConfiguration["include_detailed_errors"]` (`_tools.py:1398`). Unknown tool names: `terminate_on_unknown_calls=False` (default) → synthetic `function_result` so the loop continues; `True` → `KeyError` raised from `_try_execute_function_calls` (`_tools.py:1469-1479, 1719-1720`).

3. **Are tool results structured?**
   Yes — every tool result is `Content(type="function_result", call_id, result, items, exception, …)` (`_types.py:818-874`). The `result` field carries the joined text of `items`; `items` is `list[Content]` so multi-modal results (text + images + files) are preserved. The result is always appended to the conversation as a `Message(role="tool", contents=function_call_results)` (`_tools.py:2261-2262`) unless the run was an approval round, in which case the assistant message itself is extended with the `function_approval_request` items (`_tools.py:2228-2246`). `FunctionTool.parse_result` (`_tools.py:827-867`) is the single normalization step that converts Python return values (`str`, `Content`, `list[Content | Any]`, arbitrary objects) into `list[Content]`.

4. **Are tool errors visible to the model?**
   Yes, by construction — exceptions are caught at `_auto_invoke_function` (`_tools.py:1556-1569, 1632-1645`) and `invoke_with_termination_handling` (`_tools.py:1772-1819`), then re-emitted as `function_result` items with `exception` populated so the model sees `"Error: Function failed."` (or with `"Exception: …"` when `include_detailed_errors=True`). The loop tracks `errors_in_a_row` (`_tools.py:1855, 2248-2258`) and after `max_consecutive_errors_per_request` consecutive errors flips `tool_choice="none"` and forces a plain-text final pass (`_tools.py:2632-2644, 2802-2816`). Successful calls reset the counter (`_tools.py:2257-2259, 2752-2805` test confirms).

5. **Can tool calls be approved before execution?**
   Yes, in three layers:
   - **Per-tool opt-in**: `FunctionTool(approval_mode="always_require", …)` (`_tools.py:96, 398, 1147-1190`). Any batch containing an approval-required call returns the `function_approval_request` items; the loop returns `action="return"` without executing (`_tools.py:1712-1718, 2228-2246`).
   - **Persistent per-session rules**: `ToolApprovalMiddleware` (`_harness/_tool_approval.py:345-617`, experimental) maintains standing `ToolApprovalRule`s keyed by tool name and (optionally) exact serialized arguments and `server_label`. Rules can be added implicitly by responding with `create_always_approve_tool_response` / `create_always_approve_tool_with_arguments_response` (`_harness/_tool_approval.py:220-247`).
   - **Mixed-batch routing**: when one batch contains both approval-required and non-required tools, the non-required approvals are hidden in session state (`_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY`, `_tools.py:94, 1974-2032`) and replayed via `_pop_already_approved_approval_responses` (`_tools.py:1998-2037`) when the visible approvals are answered.
   Hosted tools (MCP) take a parallel path: `function_approval_request` items carrying `server_label` in `function_call.additional_properties` (`_tools.py:1927-1937`) are forwarded to the API as-is via `_is_hosted_tool_approval`.

## Architectural Decisions

- **Loop placement at the chat client, not the agent.** `FunctionInvocationLayer` (`_tools.py:2388-2877`) sits between the agent and the underlying `BaseChatClient` (see layering at `_clients.py:978-996`); every agent that uses a wrapped client gets the roundtrip for free. Trade-off: per-iteration context (`attempt_idx`) is local to `_get_response()` / `_stream()` and is not surfaced to `HistoryProvider` or context providers.
- **Structured `Content` everywhere, no text parsing.** Anchored by `ContentType` enumerating `function_call`/`function_result`/`function_approval_request`/`function_approval_response` (`_types.py:338-363`); arguments are a JSON string until `Content.parse_arguments()` (`_types.py:1539-1579`) lazily materializes them. This makes the loop provider-portable but assumes every provider can emit structured items.
- **Dual validation — Pydantic for typed tools, hand-rolled JSON Schema for declaration-only tools.** `FunctionTool.invoke` runs both `tool.input_model.model_validate(...)` (`_tools.py:649-655`) and `_validate_arguments_against_schema(...)` (`_tools.py:671-675, 1510-1518`). The JSON-Schema path is intentionally narrow (no `oneOf`/`$ref`); the cost of staying narrow is documented in `_tools.py:1097-1103`.
- **Errors as `function_result`, never as exceptions out of the loop.** All exceptions during tool execution (`_tools.py:1556-1569, 1632-1645`) and argument parsing (`_tools.py:1519-1528`) become normal `function_result` items. The only exceptions that bubble out are `MiddlewareTermination` (which carries a typed `term_exc.result`), `UserInputRequiredException` (re-raised for the agent layer), and `KeyError` from `terminate_on_unknown_calls=True`.
- **Approval is a content type, not a callback.** `Content.from_function_approval_request`/`from_function_approval_response` (`_types.py:1213-1253`) keep approvals in the same message stream as tool calls. The `Content.to_function_approval_response(approved)` helper (`_types.py:1286-1302`) bridges request→response without leaving the Content model. The optional `ToolApprovalMiddleware` (`_harness/_tool_approval.py:345`) adds standing rules and async callbacks on top.
- **Mixed-batch approval optimization.** When a single model response contains multiple function calls with mixed approval requirements, non-required calls are hidden in `_ALREADY_APPROVED_APPROVAL_REQUEST_GROUPS_KEY` (`_tools.py:94, 1974-2032`) and replayed via `_pop_already_approved_approval_responses` (`_tools.py:1998-2037`) when the visible batch's approvals are answered — avoiding per-call approval prompts.
- **Per-iteration budget is split into three knobs.** `max_iterations` (default 40, model roundtrips), `max_function_calls` (default None, total tool invocations across the request, best-effort post-batch), `max_consecutive_errors_per_request` (default 3, triggers `tool_choice="none"`). All three live in `FunctionInvocationConfiguration` (`_tools.py:1344-1398`) and are normalized by `normalize_function_invocation_configuration` (`_tools.py:1401-1421`).
- **Hosted-tool passthrough.** `_is_hosted_tool_approval(content)` (`_tools.py:1927-1937`) checks `function_call.additional_properties.server_label` and forwards hosted approval flows to the provider API; `_auto_invoke_function` returns the original `function_call` content for unknown hosted tools (`_tools.py:1489-1493`) without local execution.
- **Streaming and non-streaming share the same dispatcher.** `_get_response()` (`_tools.py:2553-2698`) and `_stream()` (`_tools.py:2703-2866`) both call `_execute_function_calls`/`_try_execute_function_calls`; only the outer call to `super_get_response` differs (`stream=False` vs `stream=True`).
- **Result hooks bridge streaming → middleware.** `_get_result_hooks_from_stream(stream)` (`_tools.py:2156-2164`) plucks `_result_hooks` off the inner stream so telemetry/middleware hooks fire when the inner stream finalizes (`_tools.py:2757, 2860-2866`).
- **Conversation-id aware message reduction.** When the provider returns a `conversation_id`, the loop drops `prepped_messages` to `[]` and re-sends only the latest assistant tool-result message (`_tools.py:2613-2661, 2780-2832`) — preventing service-managed APIs from seeing duplicate history.
- **Per-tool lifetime caps alongside per-request caps.** `FunctionTool.max_invocations` and `max_invocation_exceptions` (`_tools.py:308-309, 325-330, 516-543`) are checked inside `__call__`; they are *instance-lifetime* counters, not per-request. Documented at `_tools.py:333-340` as "this counter lives on the tool instance and is never automatically reset."

## Notable Patterns

- **Tri-state loop control.** `FunctionRequestResult.action` is `"return"` / `"stop"` / `"continue"` (`_tools.py:2209, 2240-2265`); `"stop"` is "error threshold reached, force a text answer", `"return"` is "nothing left to do", `"continue"` is "more model calls needed."
- **Pre-pass for approval responses.** First half of each turn in `_process_function_requests(response=None, prepped_messages=…)` (`_tools.py:2287-2334`) executes approved calls and feeds results back via `_replace_approval_contents_with_results(prepped_messages, fcc_todo, …)` (`_tools.py:2059-2153`) so the model sees results in its *next* generation, not the current one.
- **Progressive tool exposure from inside a tool.** `FunctionInvocationContext.tools` (`_middleware.py:264-285`) holds the live, run-local tools list; `add_tools(...)` / `remove_tools(...)` take effect on the *next* iteration (`_tools.py:2544-2550`).
- **Pydantic schemas generated lazily with `model_rebuild`.** `_input_schema` (`_tools.py:782-793`) calls `model_rebuild(force=True, raise_errors=False)` to handle forward references; cached after first access.
- **Result schema is uniform.** `Content.from_function_result` always builds `items: list[Content]` and a joined `result: str`; this is the canonical shape regardless of whether the function returned a string, a `Content`, a list, or a Pydantic model.
- **Two normalization paths for arguments.** The inner path (`_tools.py:649-655`) runs in `FunctionTool.invoke` for direct calls (e.g., from tests or `await tool(...)`), the outer path (`_tools.py:1510-1518`) runs in `_auto_invoke_function` for loop-driven calls. They share `_validate_arguments_against_schema` (`_tools.py:1097-1144`).
- **Mixed pre-batch already-approved routing.** `_store_already_approved_approval_requests` (`_tools.py:1974-1996`) and `_pop_already_approved_approval_responses` (`_tools.py:1998-2037`) keep the roundtrip cheap when batches mix approval-required and non-required tools.

## Tradeoffs

- **Two validation paths, two error messages.** The Pydantic path produces `"Invalid arguments for '<name>': <exc>"` (`_tools.py:668-669`); the JSON-Schema path produces `"Missing required argument(s) for '<name>': …"` / `"Invalid type for '…'"` (`_tools.py:1109, 1131-1141`). The two layers wrap into a single `"Error: Argument parsing failed. …"` at the call site (`_tools.py:1519-1528`), so the model sees a uniform error string but the underlying messages differ.
- **`Content.parse_arguments()` swallows JSON errors silently.** Returns `{"raw": original}` on `json.JSONDecodeError`/`TypeError` (`_types.py:1577-1578`); the error is only surfaced later when Pydantic/`_validate_arguments_against_schema` rejects it. A user wanting strict JSON-only parsing must wrap or pre-validate.
- **Custom JSON Schema validator is narrow.** Only primitive `type`, `enum`, `required`, and `additionalProperties: false` are checked (`_tools.py:1097-1144`); `oneOf`, `anyOf`, `allOf`, `$ref`, and array `items` are ignored. Declaration-only tools with complex schemas rely on the provider to do the heavy lifting.
- **`terminate_on_unknown_calls=True` aborts the run.** Unlike the default path, it raises `KeyError` (`_tools.py:1719-1720`) instead of returning an error result — the model never gets a chance to retry with a corrected tool name.
- **`max_function_calls` is checked after a parallel batch, not before.** A batch of 20 parallel calls with a cap of 10 still runs all 20 (`_tools.py:2636-2644, 2808-2816`); the docstring at `_tools.py:1360-1363` explicitly calls this out as best-effort.
- **Result-parser failures fall back to `str(result)`.** A custom `result_parser` that raises is caught and the original value is stringified into `[Content.from_text(str(result))]` (`_tools.py:702-706, 765-767`); the model sees `<MyObject instance at 0x…>` if `str()` doesn't return JSON-serializable text.
- **Approval-rules middleware is opt-in and experimental.** `ToolApprovalMiddleware` (`_harness/_tool_approval.py:345`) is gated by `@experimental(feature_id=ExperimentalFeature.HARNESS)` and *requires* an `AgentSession` (`_harness/_tool_approval.py:373-374`); without a session the middleware refuses to run.
- **Per-tool invocation caps are instance-lifetime.** `max_invocations`/`max_invocation_exceptions` (`_tools.py:308-309`) live on the tool instance and are not reset per request; module-level singletons accumulate across all calls.
- **`function_invocation_configuration` is per-client, not per-call.** Adjusting `max_iterations` mid-run requires mutating the client (`_tools.py:2403-2405`); `mutable_options` carries `tool_choice` and `tools` but not the configuration dict.
- **Argument validation runs in two places.** Inside `FunctionTool.invoke` (`_tools.py:649-669, 671-675`) and again inside `_auto_invoke_function` (`_tools.py:1510-1518`). For loop-driven calls the inner path is redundant work; for direct calls (e.g., `await tool.invoke(...)`) it is the only path.
- **Streaming hooks need inner stream access.** `_get_result_hooks_from_stream` (`_tools.py:2156-2164`) walks `_inner_stream._result_hooks`, which is brittle if a chat client wraps the stream without preserving the attribute.

## Failure Modes / Edge Cases

- **Malformed JSON in `arguments`.** `Content.parse_arguments` returns `{"raw": original_str}` (`_types.py:1577-1578`); the subsequent Pydantic/JSON-Schema validation then rejects it, producing `"Error: Argument parsing failed. …"` (`_tools.py:1519-1528`). Covered by `test_argument_validation_error_*` (`test_function_invocation_logic.py:1845-1914`).
- **Tool raises `UserInputRequiredException`.** Caught at `_auto_invoke_function` (`_tools.py:1554-1555, 1630-1631`) and at the gather layer (`_tools.py:1800-1819`); if the exception carries `contents`, the first content is returned and the rest are appended to `extra_user_input_contents`; otherwise a synthetic `"Tool requires user input but no request details were provided."` is returned.
- **Tool call_id missing.** Hard-required: `if call_id is None: raise KeyError(f'Function "{name}" is missing call_id.')` (`_tools.py:1580-1581`).
- **Tool raises during `asyncio.to_thread`.** Caught at `_auto_invoke_function` (`_tools.py:1556-1569, 1632-1645`); becomes `function_result` with `exception=str(exc)`; `errors_in_a_row` increments.
- **Loop exceeds `max_consecutive_errors_per_request`.** `errors_in_a_row >= max_errors` → `action="stop"` (`_tools.py:2248-2258`); the next iteration runs with `tool_choice="none"` so the model is forced to speak prose (`_tools.py:2632-2644, 2802-2816`).
- **Loop exhausts `max_iterations`.** Final pass with `tool_choice="none"` to drain orphan `function_call`s (`_tools.py:2664-2696, 2835-2866`); covered by `test_max_iterations_no_orphaned_function_calls` (`test_function_invocation_logic.py:1047-1107`).
- **Unknown tool name in strict mode.** Raises `KeyError("Error: Requested function … not found.")` (`_tools.py:1719-1720`); covered by `test_function_invocation_config_terminate_on_unknown_calls_true` (`test_function_invocation_logic.py:1626-1656`).
- **Middleware raises `MiddlewareTermination`.** Caught at `_auto_invoke_function` (`_tools.py:1612-1629`) and at the gather layer (`_tools.py:1790-1799`); `term_exc.result` is preserved and used to short-circuit the loop with `action="return"`.
- **Two concurrent function calls collide on the same tool's `max_invocations`.** The counter is checked at `FunctionTool.__call__` (`_tools.py:520-523`); if two parallel calls hit the limit at once, both increment and the second may pass before the first decrements (the cap is "max total invocations", not "concurrent invocations").
- **`function_approval_response.approved=False`.** `_replace_approval_contents_with_results` (`_tools.py:2122-2129`) replaces the response with `Content.from_function_result(call_id, result="Error: Tool call invocation was rejected by user.")` and forces `msg.role = "tool"`; the model sees a clean rejection message.
- **Hosted tool approval mixed with local approval.** `_is_hosted_tool_approval` (`_tools.py:1927-1937`) routes hosted approvals through `_collect_approval_responses`/`_replace_approval_contents_with_results` so they are passed to the API untouched; covered by `test_hosted_tool_approval_response` and `test_hosted_mcp_approval_response_passthrough` (`test_function_invocation_logic.py:1917-2000`).
- **Streaming path with `terminate_on_unknown_calls=True`.** A `KeyError` raised inside `_try_execute_function_calls` is *not* caught by the streaming `_stream()` outer wrapper (`_tools.py:2783-2793`); it propagates up and aborts the stream.
- **Result-parser raises.** Fallback to `[Content.from_text(str(result))]` with `WARNING` log (`_tools.py:702-706, 765-767`); the model sees whatever `str(...)` produces, which can be `<... object at 0x...>` for objects without `__str__`.
- **Provider returns `function_call` with `arguments: null`.** `Content.parse_arguments` returns `{}` (`_types.py:1565-1568`); the loop wrapper then either passes `{}` to the tool (if no required args) or rejects with `"Missing required argument(s)"`.

## Future Considerations

- **Repair malformed `arguments` instead of returning an error result.** The current path produces `"Error: Argument parsing failed. …"` and lets the model retry; an opt-in auto-repair middleware could attempt a single JSON-fixup pass via a separate chat-client call before failing.
- **Strict mode for `terminate_on_unknown_calls` that still recovers.** Today `True` raises `KeyError` and aborts; a mode that returns a synthetic error result and increments `errors_in_a_row` would let the model self-correct without losing the run.
- **Pre-batch `max_function_calls` check.** Currently post-batch (`_tools.py:2636-2644`); a sliding counter that pre-truncates a batch by call-id could enforce the cap exactly.
- **Unify Pydantic and JSON-Schema validation messages.** Both layers wrap into `"Error: Argument parsing failed. …"` (`_tools.py:1519-1528`) but the underlying messages differ (`_tools.py:668-669` vs `_tools.py:1109, 1131-1141`); a single helper that emits one canonical message would help observability.
- **Expand the JSON-Schema validator.** `_validate_arguments_against_schema` (`_tools.py:1097-1144`) intentionally skips `oneOf`/`$ref`/nested schemas; using `jsonschema` (already a transitive dependency via Pydantic) would close the gap without hand-rolling.
- **Promote `ToolApprovalMiddleware` out of experimental.** Currently `@experimental(feature_id=ExperimentalFeature.HARNESS)` (`_harness/_tool_approval.py:345`); the roundtrip protocol itself is stable, but the harness is not.
- **Reset `max_invocations` per request, not per instance.** The current lifetime counter (`_tools.py:333-340`) is a footgun for module-level tool singletons in long-running apps; an opt-in per-request reset (similar to `FunctionInvocationConfiguration["max_function_calls"]`) would be safer.
- **Per-tool cap that survives retries.** `max_invocations` is checked before `__call__` (`_tools.py:520-523`); if the model retries the same call after an error, the retry consumes a slot. A separate `max_retry_invocations` could distinguish first attempts from retries.

## Questions / Gaps

- **Where does the loop teach the model about expected schema details for malformed arguments?** The current model-visible error is a single line (`_tools.py:1519-1528`); the framework never returns the full Pydantic `errors()` list or the JSON-Schema path. **No clear evidence found** for a "schema hint" mode.
- **Is `parse_arguments()` ever patched by a middleware to repair JSON?** The middleware pipeline runs *after* argument parsing (`_tools.py:1530-1629` show middleware sees `arguments` already parsed); `FunctionMiddleware.process` cannot intercept the raw JSON. **No clear evidence found** for a JSON-repair middleware; would need to add a wrapper layer between `_auto_invoke_function` argument parsing and middleware.
- **How does the loop handle `function_call.arguments` being a dict vs a string?** `Content.from_function_call` allows both (`_types.py:799`); `_auto_invoke_function` does `parsed_args = dict(function_call_content.parse_arguments() or {})` (`_tools.py:1498`) which normalizes both. **No clear evidence found** for a provider that emits `arguments` as a non-JSON-string non-dict (e.g., a list of fragments).
- **Are MCP server tool results (rather than function-call results) run through the same `function_result` Content path?** The `_mcp.py` module builds `Content.from_mcp_server_tool_call`/`from_mcp_server_tool_result` (`_types.py:1168-1210`); the function-calling loop only knows about `function_call`/`function_result`/`function_approval_*`. **No clear evidence found** that MCP-specific results flow through `_handle_function_call_results` — likely they are passed back to the provider API directly.
- **Does `ToolApprovalMiddleware` short-circuit when no session is attached?** It raises `RuntimeError("ToolApprovalMiddleware requires an AgentSession.")` (`_harness/_tool_approval.py:373-374`) — explicit fail-fast, but means sessions are mandatory. **No clear evidence found** for a fallback that returns an "unapproved" `function_result` when no session exists.
- **What happens if `max_function_calls` is reached mid-batch and a tool needs `UserInputRequiredException`?** The error path bypasses the budget (`_tools.py:1554-1555, 1630-1631`); the tool still emits its content. **No clear evidence found** for a test that combines `max_function_calls` with `UserInputRequiredException`.
- **Is there a per-tool override for `include_detailed_errors`?** The flag is global to `FunctionInvocationConfiguration` (`_tools.py:1398`); `_auto_invoke_function` reads it once per call (`_tools.py:1521, 1562, 1638`). **No clear evidence found** for a per-tool override; tools with sensitive internals cannot opt out individually.

## Summary

CrewAI implements the tool-calling roundtrip in **two parallel code paths** that share utilities but each maintain their own execution wrapper. (a) The legacy text/ReAct path is anchored on `ToolUsage` (`lib/crewai/src/crewai/tools/tool_usage.py:76`) which detects tool calls via a regex→`Converter`→Pydantic pipeline, validates arguments through a five-strategy `_validate_tool_input` (`tool_usage.py:884-930`), executes via `tool.invoke`/`tool.ainvoke` with an in-memory `CacheHandler` (`agents/cache/cache_handler.py:10-50`), and returns a string observation through `format_output_for_agent` (`tools/structured_tool.py:54-83`). (b) The native-function-calling path goes through `CrewAgentExecutor._execute_single_native_tool_call` (`agents/crew_agent_executor.py:868-1071`) or `experimental.AgentExecutor._execute_single_native_tool_call` (`experimental/agent_executor.py:1860-2073`) which extracts `(call_id, func_name, func_args)` via `extract_tool_call_info` (`utilities/agent_utils.py:1191-1231`), parses args with `parse_tool_call_args` (`agent_utils.py:1636-1663`), then executes `available_functions[func_name](**args_dict)` and emits a `{role: "tool", tool_call_id, name, content}` message (`crew_agent_executor.py:1082-1088`). Both paths wrap every call with global `before_tool_call` / `after_tool_call` hooks (`hooks/tool_hooks.py:24-122`), enforce `max_usage_count` (`tools/base_tool.py:295-312`), honor a per-tool `cache_function` (`tools/base_tool.py:81-83, 176-179`), and emit `ToolUsageStartedEvent` / `ToolUsageFinishedEvent` / `ToolUsageErrorEvent` / `ToolValidateInputErrorEvent` / `ToolSelectionErrorEvent` for downstream observability (`events/types/tool_usage_events.py:56-90`). Failed tool calls return a string observation (`Error executing tool: ...` or `Error: Failed to parse tool arguments as JSON: ...`) so the loop can continue, but **there is no automatic consecutive-error cap** that flips `tool_choice` to `none` — invalid tool calls are fed back to the LLM until `max_iter` is reached (`tools/tool_usage.py:426-446, 666-686`). Human approval is supported via `ToolCallHookContext.request_human_input` (`hooks/tool_hooks.py:79-121`) which a `before_tool_call` hook can call and return `False` to block execution; this is the canonical "approve before execution" pathway.

## Rating

**7 / 10** — The roundtrip is implemented end-to-end across two executors with explicit argument validation (regex → `json` → `ast.literal_eval` → `json5` → `json_repair` ladder — `tool_usage.py:884-930`), provider-portable native-tool extraction covering OpenAI, Anthropic, Bedrock, and Gemini (`agent_utils.py:1191-1264`), Pydantic-based structured output rendering (`tools/structured_tool.py:54-83, 212-214`), and a hookable lifecycle with human approval (`hooks/tool_hooks.py:79-121, 196-215`). The repository includes 28 dedicated `test_tool_usage.py` tests (`lib/crewai/tests/tools/test_tool_usage.py`), 38+ `test_native_tool_calling.py` tests including malformed-JSON recovery (`tests/agents/test_native_tool_calling.py:1146-1338`), and explicit event-emission assertions for every failure path (`tests/tools/test_tool_usage.py:632-836`). However: (a) the validation/repair ladder silently **strips unknown argument keys** when `args_schema` is set (`tool_usage.py:332-339, 570-579`), which is fragile if the LLM produces a near-miss schema; (b) `ToolUsage._tool_calling` recursion (`tool_usage.py:861-882`) has no bound beyond `_max_parsing_attempts` (2-3) — a permanently malformed string consumes the budget and surfaces `"tool_usage_error"` with no automatic downgrade; (c) the legacy `CrewAgentExecutor` is `DeprecationWarning`-emitted at construction (`agents/crew_agent_executor.py:145-151`) so two near-identical implementations exist; (d) there is no first-class "consecutive tool-error cap" analogous to `agent-framework`'s `max_consecutive_errors_per_request` — the LLM can keep emitting the same broken call until `max_iter` runs out; (e) human approval is **only** available through global hooks (you cannot configure per-tool approval declaratively in `BaseTool`); the hook itself must call `context.request_human_input(...)` and act on the response, with the tool selection baked into hook logic rather than schema metadata.

## Answers to Dimension Questions

1. **How are tool calls represented?**
   Two distinct shapes that share an internal canonical form. **Native** tool calls arrive as provider objects: OpenAI `{id, function: {name, arguments: str}}` (`utilities/agent_utils.py:1202-1209`), Anthropic `ToolUseBlock` with `.name` + `.input` (`agent_utils.py:1218-1221`), Gemini `function_call` with `.args` dict (`agent_utils.py:1210-1217`), or Bedrock dict `{toolUseId, name, input}` (`agent_utils.py:1222-1230`). `extract_tool_call_info` normalizes them to `(call_id, func_name, func_args)` tuples. The assistant message is always `{role: "assistant", content: None, tool_calls: [{id, type: "function", function: {name, arguments}}]}` and each tool result is `{role: "tool", tool_call_id: call_id, name: func_name, content: result}` (`agents/crew_agent_executor.py:843-866, 1082-1088`). **Text-mode** tool calls come back as free-form strings with `Thought:`, `Action:`, `Action Input:` sections; `ACTION_INPUT_REGEX` (`agents/constants.py:18-20`) extracts `(tool_name, tool_input)` and `_safe_repair_json` (`agents/parser.py:161-178`) normalizes the `Action Input` to valid JSON via `json_repair.repair_json`. The result is an `AgentAction(thought, tool, tool_input, text)` dataclass (`agents/parser.py:25-33`). Both paths converge on `ToolCalling` / `InstructorToolCalling` Pydantic models with `{tool_name: str, arguments: dict[str, Any] | None}` (`tools/tool_calling.py:11-24`).

2. **What happens if arguments are invalid?**
   **Five-strategy validation ladder** for text mode (`tools/tool_usage.py:884-930`): (i) `json.loads` → success path; (ii) `ast.literal_eval` → Python literal form; (iii) `repair_json` from `json_repair`; (iv) `json5.loads` → JSON5 with comments/unquoted keys; (v) `repair_json(skip_json_loads=True)` as final attempt. If all fail, `_emit_validate_input_error` emits a `ToolValidateInputErrorEvent` (`tools/tool_usage.py:932-948`) and `Exception` is raised, which the caller's retry handler converts to `ToolUsageError("tool_usage_error")` and feeds back as a string observation to the LLM (`tools/tool_usage.py:879-881`). For **native** mode, `parse_tool_call_args` only attempts `json.loads` (`utilities/agent_utils.py:1636-1663`) — on failure it returns a ready-to-render error dict with `Error: Failed to parse tool arguments as JSON: {e}. Please provide valid JSON arguments for the '{name}' tool.` (covered by `test_malformed_json_returns_parse_error`, `tests/agents/test_native_tool_calling.py:1146-1172`). **Schema-level filtering** also silently strips unknown keys: when `tool.args_schema.model_json_schema()` has keys and `calling.arguments` contains extras, only matching keys are passed (`tools/tool_usage.py:332-339, 570-579`) — there is no warning, just a quiet key drop (the second-attempt path that re-sends the original arguments only fires if the filtered invoke raises).

3. **Are tool results structured?**
   **Yes, in two layers.** (a) **Per-tool structured output**: `BaseTool.result_schema: type[BaseModel] | None` (`tools/base_tool.py:154-158`) and `format_output_for_agent` (`tools/structured_tool.py:54-83, 212-214`) validate the raw tool output against the schema and emit `model.model_dump_json()`; on failure, fall back to `str(raw_result)` with a `RuntimeWarning`. This is exercised by `test_tool_usage_returns_json_agent_text_for_typed_output` (`tests/tools/test_tool_usage.py:144-164`). (b) **Native wire format**: tool messages use the OpenAI-compatible `{role: "tool", tool_call_id, name, content}` (`agents/crew_agent_executor.py:1082-1088`) and assistant messages use `{role: "assistant", content: None, tool_calls}` (`agents/crew_agent_executor.py:843-866`). Text-mode observations are inlined into `formatted_answer.text` as `f"\nObservation: {tool_result.result}"` (`utilities/agent_utils.py:616`). After every native tool execution, a `{role: "user", content: post_tool_reasoning}` message is appended (`agents/crew_agent_executor.py:801-806`) so the model gets an explicit reflection prompt.

4. **Are tool errors visible to the model?**
   **Yes — every error mode becomes a string observation.** Tool execution errors: `Error executing tool: {e}` (`agents/crew_agent_executor.py:1009`; `experimental/agent_executor.py:2002`). JSON parse errors: `Error: Failed to parse tool arguments as JSON: {e}. Please provide valid JSON arguments for the '{name}' tool.` (`utilities/agent_utils.py:1656-1662`). Tool selection miss: `Action '{name}' don't exist, these are the only available Actions:\n{self.tools_description}` (`tools/tool_usage.py:785-793`). Max-usage breach: `Tool '{name}' has reached its usage limit of {N} times and cannot be used anymore.` (`agents/crew_agent_executor.py:981`; `tools/tool_usage.py:756`). Hook block: `Tool execution blocked by hook. Tool: {name}` (`agents/crew_agent_executor.py:978`). OutputParserError: `I did it wrong. Invalid Format: I missed the 'Action:' after 'Thought:'` (`utilities/agent_utils.py:680-695`). All emit `ToolUsageErrorEvent` / `ToolValidateInputErrorEvent` / `ToolSelectionErrorEvent` for observability. **However**, there is no automatic consecutive-error cap. A model that emits broken tool calls N times in a row gets N observations back; only `max_iter` stops the loop (`base_agent_executor.py:27`, default 25). After `max_iter`, `handle_max_iterations_exceeded` issues one final LLM call with a `force_final_answer` nudge (`utilities/agent_utils.py:293-350`).

5. **Can tool calls be approved before execution?**
   **Yes, via global `before_tool_call` hooks.** `ToolCallHookContext.request_human_input(prompt, default_message)` (`hooks/tool_hooks.py:79-121`) calls `builtins.input()`, pauses/resumes the event listener's live updates, and returns the user's response. A hook can call this and return `False` to block execution — the executor replaces the observation with `Tool execution blocked by hook. Tool: {name}` (`agents/crew_agent_executor.py:977-978, 1527`). The pattern is documented as the canonical approval gate (`hooks/tool_hooks.py:95-107`). The hook can also rewrite inputs in place (`hooks/tool_hooks.py:33-36`). **Limitations**: approval is **only** reachable through the hook registry, not declarative per-tool metadata; no built-in tool-level `requires_approval: bool = True` field exists on `BaseTool`. Hooks must be wired globally via `register_before_tool_call_hook` or via the `@before_tool_call` decorator on a crew function (`hooks/decorators.py`). Both `before_tool_call` and `after_tool_call` hooks are invoked from three execution sites — the legacy `_use`/`_ause` text path (`tools/tool_usage.py`), `_execute_single_native_tool_call` in `CrewAgentExecutor` (`agents/crew_agent_executor.py:868-1071`), and the experimental `AgentExecutor._execute_single_native_tool_call` (`experimental/agent_executor.py:1860-2073`) — plus `execute_tool_and_check_finality` for text mode (`utilities/tool_utils.py:96-141`). Hook parity across sequential and parallel native paths is asserted by `test_openai_parallel_native_tool_calling_tool_hook_parity_crew` and `_agent_kickoff` (`tests/agents/test_native_tool_calling.py:326-448`).

## Architectural Decisions

- **Two parallel executors with shared primitives.** The deprecated `CrewAgentExecutor` (`agents/crew_agent_executor.py:98`) and the canonical `crewai.experimental.AgentExecutor` (`experimental/agent_executor.py:164`) each maintain their own `_execute_single_native_tool_call`, `_append_assistant_tool_calls_message`, and `_handle_native_tool_calls` despite sharing `ToolUsage` for the text path. The native execution wrapper is duplicated across three locations (`agents/crew_agent_executor.py:868-1071`, `experimental/agent_executor.py:1860-2073`, `utilities/agent_utils.py:1399-1633`) — the latter is the canonical `execute_single_native_tool_call` helper used by `StepExecutor`.
- **Separate `ToolCalling` and `InstructorToolCalling` models.** Two Pydantic models with identical fields but different semantics (`tools/tool_calling.py:11-24`): `ToolCalling` for the text-mode path, `InstructorToolCalling` when `function_calling_llm.supports_function_calling()` is true and an Instructor-based schema is requested (`tools/tool_usage.py:812-816`).
- **Five-strategy argument validation ladder.** `json.loads` → `ast.literal_eval` → `repair_json` → `json5.loads` → `repair_json(skip_json_loads=True)` (`tools/tool_usage.py:884-930`). Each strategy is exercised by a dedicated test in `tests/tools/test_tool_usage.py` (`test_validate_tool_input_*`). This is more permissive than strict JSON, accepting Python literal syntax and JSON5 forms, but trades strict validation for robustness on real-model output.
- **Atomic usage enforcement.** `BaseTool._claim_usage()` uses a `threading.Lock` (`tools/base_tool.py:295-312`) so a `ThreadPoolExecutor(max_workers=8)` parallel batch cannot exceed `max_usage_count` even under contention. The structured tool also has `has_reached_max_usage_count` and `_increment_usage_count` (`tools/structured_tool.py:366-377`).
- **Hooks are the cross-cutting interception point.** `ToolCallHookContext.request_human_input` (`hooks/tool_hooks.py:79-121`) is the only built-in approval primitive. Hooks can mutate `context.tool_input` in place (must not rebind), block execution by returning `False`, or rewrite results by returning a string. Hook ordering: before_hooks → cache read → execution → after_hooks (the legacy and new paths both follow this ordering, see `crew_agent_executor.py:954-1063` and `agent_executor.py:1962-2049`).
- **Provider-fallback with prompt downgrade.** When a native tool call error message matches one of 7 known substrings (`is_native_tool_calling_unsupported_error`, `utilities/agent_utils.py:1285-1288`), the executor appends `build_text_tool_calling_fallback_message` to `messages` and routes to `_invoke_loop_react` (`agents/crew_agent_executor.py:577-579`). The fallback is covered by `test_native_tool_loop_falls_back_when_provider_rejects_tools` (`tests/agents/test_native_tool_calling.py:1271-1313`).
- **Cache as observation shortcut.** `ToolsHandler.on_tool_use` (`agents/tools_handler.py:26-52`) keys on `{tool}-{json(args)}`; cached outputs skip execution entirely and become the observation verbatim. The `cache_function` callback (`tools/base_tool.py:81-83`) lets tools decide cacheability per call (e.g. time-sensitive tools can return `False`). `CacheTools` (the in-tool cache-bypass tool) is exempt from caching (`agents/tools_handler.py:40`).
- **Argument schema filtering is silent.** When the executor knows the tool's `args_schema` it strips keys not in `tool.args_schema.model_json_schema()["properties"]` (`tools/tool_usage.py:332-339, 570-579`) without warning. The fallback path only fires when the filtered invocation raises; the original `calling.arguments` are then sent unchanged. There is no test that asserts the warning behavior or that the user is informed when keys are dropped.
- **Delegation is a special-case tool.** `Delegate work to coworker` and `Ask question to coworker` are name-matched at `_use`/`_ause` (`tools/tool_usage.py:316-326, 556-566`) to increment `task.increment_delegations(coworker)` (`task.py:1067-1071`). This is the only domain-specific routing baked into `ToolUsage` itself; other special behaviors live in `track_delegation_if_needed` (`utilities/agent_utils.py:1174-1189`).

## Notable Patterns

- **Validation ladder with explicit test matrix.** Every step of `_validate_tool_input` has a corresponding test in `tests/tools/test_tool_usage.py` (lines 264-543) — Python literal form, JSON5 unquoted keys, trailing commas, escaped quotes, large JSON, code content, mixed types, single quotes, special characters, etc. The ladder is regression-safe.
- **Hook-before-cache ordering.** In the native path, `before_tool_call` hooks run **before** the cache read (`agents/crew_agent_executor.py:963-969, 1491-1524`) so a hook can mutate `tool_input` and the mutated input is what gets cached. This means the cache is sensitive to input mutation order — a hook that reformats timestamps will cache under the reformatted key. The post-hook ordering is reverse: `after_tool_call` runs after the tool but before `ToolUsageFinishedEvent` (`agents/crew_agent_executor.py:1026-1063`).
- **`extract_tool_call_info` as a 4-shape adapter.** Recognizes OpenAI (`hasattr(tool_call, "function")`), Gemini (`hasattr(tool_call, "function_call")`), Anthropic (`hasattr(tool_call, "name") and hasattr(tool_call, "input")`), and generic dict (`isinstance(tool_call, dict)` with `function` or `name`+`input` keys). Returns `None` on unrecognized shapes; callers handle `None` as an error and emit `"Invalid native tool call format"` (`experimental/agent_executor.py:1863-1875`).
- **Contextvars-safe parallel execution.** Parallel native batches wrap each `_execute_single_native_tool_call` in `contextvars.copy_context().run(...)` (`agents/crew_agent_executor.py:748-749, 1717-1719`) so each thread inherits the executor's context (including `_is_executing`, telemetry vars, stop-word overrides).
- **Distinct events for distinct failure modes.** `ToolUsageErrorEvent` (executor raised), `ToolValidateInputErrorEvent` (parse failed), `ToolSelectionErrorEvent` (tool not found), and `ToolExecutionErrorEvent` (event_listener-internal) are separately typed (`events/types/tool_usage_events.py:72-97`) so observability consumers can route by failure type. `ToolUsageFinishedEvent` is suppressed after an error (`test_tool_error_does_not_emit_finished_event`, `tests/tools/test_tool_usage.py:836-905`) — the executor relies on `error_event_emitted=True` flag instead of try/finally (lines 950-1063).
- **`max_usage_count` with original/structured copy sync.** `BaseTool.to_structured_tool()` propagates `current_usage_count` into the structured copy (`tools/base_tool.py:393-408`), and `CrewStructuredTool._increment_usage_count` writes back to `_original_tool` (`tools/structured_tool.py:373-377`), so the two counters stay in sync even though the executor mostly operates on the structured copy.
- **Cache-bypass tool.** `CacheTools` (`tools/cache_tools/cache_tools.py`) is excluded from the cache (`agents/tools_handler.py:40-48`), preventing recursive caching of cache-management operations.

## Tradeoffs

- **Two near-duplicate executors.** `CrewAgentExecutor._execute_single_native_tool_call` (`agents/crew_agent_executor.py:868-1071`) and `experimental.AgentExecutor._execute_single_native_tool_call` (`experimental/agent_executor.py:1860-2073`) are 200+ line near-mirrors with subtle differences (event payload fields, hook signature, `_should_parallelize_native_tool_calls`). The canonical `execute_single_native_tool_call` in `utilities/agent_utils.py:1399-1633` is used by `StepExecutor` but not by either top-level executor. Code drift is a known risk.
- **Silent argument schema filtering.** When `tool.args_schema` is set, unknown keys are dropped without warning (`tools/tool_usage.py:332-339, 570-579`). A model that emits a near-miss schema (e.g. `search_query` instead of `query`) gets a silently truncated call that may then fail validation, surface as `Error executing tool:`, and produce a misleading error message that doesn't tell the model the field was dropped. There is no telemetry event for "filtered args".
- **No automatic consecutive-error cap.** A model that emits the same broken tool call N times gets N identical observations back; only `max_iter` stops the loop. `agent-framework`'s `max_consecutive_errors_per_request` pattern (flip to `tool_choice="none"`) is absent — CrewAI has `increment_tools_errors` on `Task` (`task.py:1063-1065`) but no policy that reads it.
- **`_validate_tool_input` raises `Exception` (not a custom error).** The catch in `_tool_calling` (`tools/tool_usage.py:871`) catches `Exception` and converts to `ToolUsageError`. Callers cannot easily distinguish "JSON parse failed" from "Pydantic validation failed" from "tool raised" — the difference is in the message string only.
- **Cache is process-local and unbounded.** `CacheHandler._cache` (`agents/cache/cache_handler.py:20-50`) is an in-memory dict with no size cap. A long-running crew can grow without bound. There is no eviction policy.
- **Parallel skip on `result_as_answer` or `max_usage_count`.** A native batch containing any such tool runs **sequentially** (`agents/crew_agent_executor.py:698-723, agent_executor.py:1827-1858`). This is correctness-first — `result_as_answer` must short-circuit and `max_usage_count` must be enforced atomically — but means a single such tool in a 5-tool batch forces all 5 to run serially.
- **Hook approval is global, not declarative.** `BaseTool` has no `requires_approval: bool = True` field; approval must be coded into a global `before_tool_call` hook. This means every crew on the process pays the cost of an `input()` call unless the hook short-circuits on `tool_name`. There's no per-tool or per-crew declarative gate.
- **Default `ToolUsage._max_parsing_attempts = 3`** (2 for big-model OpenAI families — `tools/tool_usage.py:101-119`) is the only retry budget for `_tool_calling`. Once exhausted, the executor returns `ToolUsageError("Moving on then. …")` and continues the loop with that string as the observation. There's no per-error-class backoff or retry budget.
- **`LiteAgent` does not implement tool approval hooks.** It uses `execute_tool_and_check_finality` (`lite_agent.py:104`) and inherits the global hooks, but does not emit its own `before_tool_call` registration call. Hooks must be registered separately by the host process.
- **`request_human_input` calls `builtins.input()` directly** (`hooks/tool_hooks.py:114`). This blocks in async contexts and will deadlock if invoked from inside an event loop without an async-aware input mechanism.

## Failure Modes / Edge Cases

- **Malformed JSON in native args returns an error observation** (`Error: Failed to parse tool arguments as JSON: {e}. Please provide valid JSON arguments for the '{name}' tool.` — `utilities/agent_utils.py:1653-1662`) and `tool.current_usage_count` is **not** incremented (`test_malformed_json_returns_parse_error`, `tests/agents/test_native_tool_calling.py:1146-1172`). The loop continues with the error observation.
- **Tool selection miss returns `Exception` in text mode** with the full available tool list (`tools/tool_usage.py:784-793`). This bubbles up to `_tool_calling` which catches it, increments `_run_attempts`, and either retries (3 attempts) or returns `ToolUsageError`. The model sees `I forgot the Action name, these are the only available Actions: …` (`tools/tool_usage.py:794-801`) when the tool name is empty.
- **Schema validation miss in native path runs Pydantic validation through `CrewStructuredTool._parse_args`** (`tools/structured_tool.py:272-294`), which raises `ValueError("Arguments validation failed: {e}{hint}")` with a schema hint. The hint is `f"\nExpected arguments: {json.dumps(properties)}\nRequired: {json.dumps(required)}"` (`tools/structured_tool.py:90-108`). The model sees the full JSON schema in the error message.
- **Tool execution exceptions are caught at three nested levels**: (1) `_execute_single_native_tool_call` catches and emits `ToolUsageErrorEvent` (`agents/crew_agent_executor.py:1008-1024`); (2) `_use`/`_ause` re-throw into a wrapper `try/except` that returns `ToolUsageError.message` as the observation (`tools/tool_usage.py:422-446, 662-705`); (3) `execute_tool_action` in the experimental executor appends `"\nObservation: Error executing tool: {e}"` and continues (`experimental/agent_executor.py:1612-1631`).
- **Hook block observation**: `Tool execution blocked by hook. Tool: {name}` (`agents/crew_agent_executor.py:978, 1527`). The block is a soft block — the loop continues with this observation; no error event is emitted (the executor sets `hook_blocked=True` but doesn't emit `ToolUsageErrorEvent`).
- **Max-usage breach in parallel batch**: Each `_execute_single_native_tool_call` checks `current_usage_count >= max_usage_count` independently (`agents/crew_agent_executor.py:902-909`). If 8 parallel calls all hit a max-usage-3 tool, the first 3 succeed, the next 5 each get `Tool '{name}' has reached its usage limit of 3 times and cannot be used anymore.` The atomic `_claim_usage` lock prevents over-shooting, but the order of execution is `as_completed`, not deterministic.
- **Cache hit → silent observation**: A cached result is formatted via `format_native_tool_output_for_agent` (`utilities/agent_utils.py:1489-1490, 1547`) and emitted as the tool observation with `from_cache=True` on the finished event. The model cannot distinguish cache-hit from fresh execution.
- **Empty `func_args` (native)**: `parse_tool_call_args({"code": ""})` is treated as a dict (no JSON parsing needed) and passed to the tool. The tool's `args_schema` then validates — if `code` is required, Pydantic raises `ValidationError`, surfaced as `Error executing tool: 1 validation error…`.
- **`_validate_tool_input` returns empty dict on `None` input** (`tools/tool_usage.py:885-887`), but on non-string input raises `Exception` (`tools/tool_usage.py:888-891`). The two paths are not symmetric.
- **Concurrent `_execute_single_native_tool_call` calls share `ToolsHandler.last_used_tool` updates** through `on_tool_use` (`agents/tools_handler.py:39`). The repeated-usage short-circuit (`_check_tool_repeated_usage`, `tools/tool_usage.py:728-738`) reads `last_used_tool` without a lock — under parallel native execution, a tool may be reported as "repeated" before its result is observed by a sibling call.
- **`request_human_input` swallows `KeyboardInterrupt` only via the `finally` clause that resumes live updates** (`hooks/tool_hooks.py:118-121`). The exception itself propagates (`test_request_human_input_resumes_on_keyboard_interrupt`, `tests/hooks/test_human_approval.py:202-220`); in `BaseTool.run` (`tools/base_tool.py:314-331`), an uncaught exception from a sync tool bubbles all the way up.
- **Sequential native single-call loop processes only the FIRST tool call per LLM turn** (`agents/crew_agent_executor.py:786-807`). If the LLM returns 3 tool calls and the first succeeds without `result_as_answer`, the second and third are queued in `parsed_calls` but not executed in this iteration — only one tool runs, then a `post_tool_reasoning` prompt is appended and the LLM is re-asked. This is intentional ("sequential tool execution with reflection") but means a model that batches tool calls must rely on the LLM re-emitting the remaining calls after seeing the first result.

## Future Considerations

- **Unify the two `_execute_single_native_tool_call` implementations.** The legacy `agents/crew_agent_executor.py:868-1071`, the experimental `experimental/agent_executor.py:1860-2073`, and the canonical `utilities/agent_utils.py:1399-1633` differ in event payload fields, hook invocation patterns, and parallelism gating. A single function with options would reduce drift.
- **First-class `requires_approval` field on `BaseTool`.** A declarative `requires_approval: bool = False` would let `BaseTool.run` invoke `request_human_input` automatically before invoking `_run`. Combined with a default approval policy at the agent level (`agent.approval_required_tools: list[str]`), this would replace the global-hook pattern for most users.
- **Schema-filter warnings.** When `args_schema` filters out unknown keys, emit a `ToolValidateInputErrorEvent` (or a new `ToolArgumentFilterWarning` event) so observability consumers can detect the pattern. The current silent drop is the highest-risk behavior in the roundtrip.
- **Consecutive-error cap.** Mirror `agent-framework`'s `max_consecutive_errors_per_request`: after N consecutive `ToolUsageErrorEvent` emissions, force `tool_choice="none"` for the next iteration and surface a final answer. `Task.tools_errors` (`task.py:1063-1065`) is incremented but never read for policy decisions.
- **Per-tool structured `tool_choice`.** Today the executor sends the full tool list to the LLM. A `BaseTool.tool_choice: Literal["auto", "required", "none"]` would let callers force specific tools (e.g. a final-answer tool for exit), matching OpenAI's `tool_choice` parameter.
- **Cache eviction policy.** `CacheHandler` (`agents/cache/cache_handler.py:10-50`) is unbounded. Adding a TTL or LRU cap would bound memory for long-running crews.
- **Async `request_human_input`.** Replace `builtins.input()` (`hooks/tool_hooks.py:114`) with an injectable human-input provider that returns a coroutine in async contexts. Today calling `context.request_human_input` from inside an event loop blocks the loop until `input()` returns.
- **Observability for `ToolUsageError` retries.** The retry budget (`_max_parsing_attempts`, `tools/tool_usage.py:101-119`) is invisible to operators. Emitting `ToolUsageRetryEvent` on each retry attempt would let dashboards track argument-repair failure patterns.
- **Per-tool `cache_function` default policy.** Currently the default `_default_cache_function` (`tools/base_tool.py:81-83`) returns `True` for every call — every tool output is cached by default. Tools like `get_current_time` or `get_weather` would benefit from a default-deny policy with opt-in caching.

## Questions / Gaps

- **Is the `Result-as-answer` flag honored on a tool after `after_tool_call` hook rewrites the result?** The check `original_tool.result_as_answer` runs before the post-hook in `crew_agent_executor.py:1097-1107` but after the post-hook in `utilities/agent_utils.py:1618-1624`. The two paths disagree; **no clear evidence found** for an end-to-end test asserting which path wins when both fire.
- **Does `ToolsHandler.last_used_tool` update race-safe under parallel native execution?** `on_tool_use` writes without a lock (`agents/tools_handler.py:39`). The repeated-usage short-circuit reads it without a lock (`tools/tool_usage.py:732-738`). **No clear evidence found** for a test asserting `_check_tool_repeated_usage` returns the expected value when 8 parallel calls all complete near-simultaneously.
- **What happens to the cache when a tool's `args_schema` changes at runtime?** The cache key is `{tool}-{json(args)}`, so the JSON form is the only cache discriminator. If a tool is updated and `args_schema` changes, stale cache entries may match new invocations and return wrong results. **No clear evidence found** for a cache-invalidation hook tied to schema changes.
- **Are Pydantic validation errors visible to the model with full context?** `_parse_args` raises `ValueError(f"Arguments validation failed: {e}{hint}")` (`tools/structured_tool.py:289-294`) with a schema hint. The error surfaces as `Error executing tool: Arguments validation failed: 1 validation error for CodeInput\ncode\n  Field required [type=missing, input_value={'lang': 'python'}, input_type=dict]\nExpected arguments: {...}\nRequired: ["code"]`. The model sees both the error and the schema hint. **No clear evidence found** for whether the schema hint is consistently included across all error paths (e.g. when the Pydantic error is wrapped inside another exception).
- **Does the `cache_breakpoint` marker on `system` and `user` prompts work across providers that don't recognize it?** `mark_cache_breakpoint` (`agents/crew_agent_executor.py:192-204`) flags messages for prompt caching. The flag is documented as stripped by adapters that don't support it (`llms/cache.py:24-37`), but **no clear evidence found** in tests that the marker survives intact when passed to a provider that supports it (Anthropic, OpenAI cache-control). Cache-hit/miss assertions are absent.
- **What happens when `request_human_input` is called from inside an event loop?** `builtins.input()` blocks the loop (`hooks/tool_hooks.py:114`). **No clear evidence found** for an async path or for tests asserting non-blocking behavior under `asyncio.run`.
- **Is there a documented contract for `cache_function` returning `False` mid-batch?** If `cache_function` returns `False` for a parallel tool call, the result is **not** added to cache, but sibling parallel calls may have already added their result. The cache state is therefore non-atomic with respect to `cache_function`. **No clear evidence found** for a test asserting per-call atomicity.
- **Are tool-call IDs stable across retries?** `_parse_native_tool_call` falls back to `f"call_{id(tool_call)}"` when no `id` is present (`agents/crew_agent_executor.py:813, 817, 826, 832`). Python `id()` may be reused after garbage collection, so a retry could reuse a previously-issued `call_id`. **No clear evidence found** for ID-stability guarantees.

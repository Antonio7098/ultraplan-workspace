## Summary

Pydantic AI's tool-calling roundtrip is structured around typed message parts (`ToolCallPart`, `ToolReturnPart`, `RetryPromptPart`, `NativeToolCallPart`, `NativeToolReturnPart`) with a discriminated `ModelRequestPart` union at `pydantic_ai_slim/pydantic_ai/messages.py:2036-2044`. The agent loop (a `pydantic_graph` graph with `CallToolsNode`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1046-1382`) is **provider-agnostic**: model adapters emit base `ToolCallPart`s, the framework owns typed-subclass promotion (`_agent_graph.py:2142-2161`, `_parts_manager.py:86-97`).

The pipeline is **explicitly split into validation and execution**, mediated by `ToolManager` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:78-936`) and a `ValidatedToolCall` carrier (`tool_manager.py:46-74`). Validation goes through Pydantic's `SchemaValidator` (`_function_schema.py:103-281`), then an optional `args_validator` user callback (`tools.py:84-94`, `tool_manager.py:215-250`). Execution runs in parallel via `asyncio.create_task` (`_agent_graph.py:1916-1946`) with three modes: `parallel` (default), `sequential`, `parallel_ordered_events` (`tool_manager.py:155-168`). Every tool failure path — schema validation, unknown tool, `ModelRetry` from the body, timeout, custom validator — is normalized into a `RetryPromptPart` (`messages.py:1406-1502`) and sent back to the model, gated by per-tool `max_retries` (`tool_manager.py:177-181`, `_check_max_retries`).

Approval is a first-class flow: `requires_approval=True` on `Tool` or `FunctionToolset` causes `ApprovalRequiredToolset` (`toolsets/approval_required.py:15-32`) to raise `ApprovalRequired` (`exceptions.py:98-113`), which surfaces as a `DeferredToolRequests` output (`tools.py:256-326`) and is resolved on the next run via `DeferredToolResults` (`tools.py:378-421`). External tools (`kind='external'`) follow the same deferred path with `CallDeferred` (`exceptions.py:80-95`). Malformed arguments are repaired at the transport layer using the Anthropic-recommended `'INVALID_JSON'` sentinel key (`messages.py:36-40, 1826-1850`).

## Rating

**9 / 10** — The roundtrip is built on a typed part model with strict discriminated unions, a clean validation/execution split with `ValidatedToolCall`, parallel execution with three modes and proper cancellation, four orthogonal safety knobs (`max_retries`, `timeout`, usage limits, output retries), first-class approval and deferred execution, capability-hook composition (`before_tool_validate`, `wrap_tool_validate`, `after_tool_validate`, `on_tool_validate_error`, `before_tool_execute`, `wrap_tool_execute`, `after_tool_execute`, `on_tool_execute_error`), and structured `RetryPromptPart` errors that always return to the model. Heavy test coverage (timeout tests at `tests/test_tools.py:2749-2940`, args_validator tests at `3246-3439`, unknown-tool recovery at `1920-2018`, retry-budget exhaustion at `1465-1502`).

Gaps that hold it below 10:
- **`CallDeferred` after approval-and-resume still surfaces through `tool_manager._resolve_single_deferred`** (`tool_manager.py:847-935`) with multiple branches that have to be manually kept in sync with `_agent_graph._call_tool`; the two paths are explicitly cross-referenced as a comment ("keep the dispatch branches here in sync", `tool_manager.py:861-863`).
- **Validation error mapping for non-tool-call model output goes through `_make_retry_prompt`** (`_output.py:121-129`) which serializes `ValidationError.errors()` into a `RetryPromptPart` — the JSON-serialized form uses `exclude={'ctx'}` (`messages.py:1457-1462`), but the `input` is still included for tool retries, meaning the model sees its own (possibly large) malformed input echoed back.
- **`wrap_validation_errors=False` propagation** (`tool_manager.py:419-465, 467-502`) is opt-in but easy to forget — raw `ValidationError`/`ModelRetry` propagates out, and the caller must catch.
- **Schema validator is built from the function signature, but `from_schema()` skips schema validation** (`tools.py:583-636`), relying solely on `args_validator`. There's no warning to the user that schema validation is disabled.
- **`allow_partial` validation** (`tool_manager.py:234-243`) uses `'trailing-strings'` for streaming partial JSON, but this is opt-in per call path; full strict mode is the default and the `AllowPartial` capability is not surfaced at the agent level.

## Answers to Dimension Questions

1. **How are tool calls represented?**
   As typed dataclass parts. The base is `BaseToolCallPart` (`messages.py:1771-1867`) with `tool_name`, `args: str | dict[str, Any] | None`, `tool_call_id`, `tool_kind`. Concrete subclasses are `ToolCallPart` (local function tool, `messages.py:1870-1894`), `NativeToolCallPart` (provider-native tool, `messages.py:1897-1955`), and typed subclasses registered via `_TOOL_CALL_NARROWERS` for tools with `tool_kind` (e.g. `ToolSearchCallPart`, `LoadCapabilityCallPart`). Schema for the wire is `ToolDefinition` (`tools.py:686-826`), which carries `parameters_json_schema`, `description`, `kind` (`'function' | 'output' | 'external' | 'unapproved'`), `strict`, `sequential`, `timeout`, `return_schema`, `defer_loading`. Streaming parts are `ToolCallPartDelta` (built up by `ModelResponsePartsManager.handle_tool_call_delta`, `_parts_manager.py:292-388`), upgraded to `ToolCallPart` once the `tool_name` is known.

2. **What happens if arguments are invalid?**
   The Pydantic `SchemaValidator` built in `_function_schema.function_schema` (`_function_schema.py:103-281`) raises `ValidationError`. The agent catches this in `ToolManager.validate_tool_call` (`tool_manager.py:419-465`) and wraps it via `_wrap_error_as_retry` (`tool_manager.py:183-191`) into a `RetryPromptPart` with `content=list[ErrorDetails]`, `tool_name`, `tool_call_id`. After the per-tool retry budget is exhausted (`_check_max_retries`, `tool_manager.py:177-181`), `UnexpectedModelBehavior` is raised. Unknown tool names take a separate path: `_resolve_tool` (`tool_manager.py:356-370`) raises `ModelRetry(f'Unknown tool name: {name!r}. Available tools: ...')`, which is wrapped into a retry prompt the same way. A second user-level `args_validator` callback runs after schema validation passes (`tool_manager.py:215-250`); the same retry path applies if it raises `ModelRetry`. JSON parse failures at the transport layer (e.g. truncated streaming JSON) are wrapped into the Anthropic-recommended `INVALID_JSON` sentinel key (`messages.py:36-40`) so the model receives the raw args on the next roundtrip without crashing.

3. **Are tool results structured?**
   Yes, doubly. The wire-level `ToolReturnPart` (`messages.py:1332-1357`) carries `tool_name`, `tool_call_id`, `content: ToolReturnContent`, `metadata`, `timestamp`, `outcome: 'success' | 'failed' | 'denied'`. Provider- and content-type-aware serialization is provided by `model_response_str()` (JSON string), `model_response_object()` (dict with `RETURN_VALUE_KEY` wrapping for Gemini), and `model_response_str_and_user_content()` (split for multimodal file extraction) at `messages.py:1244-1297`. The user-level `ToolReturn` wrapper (`messages.py:891-918`) lets a tool body return `(return_value, content=separate_user_prompt, metadata=not_sent_to_llm)` so the model-visible content can be split from the result used downstream. Multimodal files (`BinaryContent`, `ImageUrl`, etc.) inside the content list are auto-extracted into a trailing `UserPromptPart` with `'See file {id}.'` placeholders (`messages.py:1271-1297`).

4. **Are tool errors visible to the model?**
   Yes — always. Every error path produces a structured `RetryPromptPart` (`messages.py:1406-1502`) that is appended to the next `ModelRequest`. Formatting distinguishes the two cases: tool-call retries keep the `input` field so the model sees its own malformed args alongside the error (`messages.py:1462`); native-output retries drop the `input` to avoid duplicating JSON already in context (`messages.py:1457-1460`). Timeouts, schema failures, custom validator failures, `ModelRetry` from tool bodies, and unknown tool names all flow through the same channel. The retry budget is checked per tool (`_check_max_retries`, `tool_manager.py:177-181`) and per output (`max_output_retries`, `_output.py:121-129`), and reaching it raises `UnexpectedModelBehavior` to the caller rather than silently dropping the error.

5. **Can tool calls be approved before execution?**
   Yes, via two mechanisms that share `DeferredToolRequests` (`tools.py:256-326`) as the output surface. (1) `Tool(requires_approval=True)` or `FunctionToolset(requires_approval=True)` causes the tool to be advertised with `kind='unapproved'`; on call, `ApprovalRequiredToolset.call_tool` (`toolsets/approval_required.py:26-31`) raises `ApprovalRequired` unless `ctx.tool_call_approved=True`. The agent run terminates with `DeferredToolRequests(approvals=[...])`. (2) `kind='external'` tools raise `CallDeferred` for execution outside the agent run. Both are resolved on the next run by passing `DeferredToolResults` (`tools.py:378-421`): `ToolApproved()` (with optional `override_args` to replace the model's args), `ToolDenied(message)`, raw `ModelRetry`, raw `RetryPromptPart`, or a plain `ToolReturn` value. `_call_tool` (`_agent_graph.py:1984-2052`) dispatches each variant and re-validates approved calls. A capability-level `HandleDeferredToolCalls` handler (`tool_manager.py:830-845`) can resolve deferred calls inline within a single run.

## Architectural Decisions

- **Validate-then-execute split via `ValidatedToolCall`.** `tool_manager.py:46-74` defines a carrier that holds `call`, `tool`, `ctx`, `args_valid`, `validated_args`, `validation_error`. `validate_tool_call` (`tool_manager.py:419-465`) and `execute_tool_call` (`tool_manager.py:467-501`) are separate methods. This lets the loop emit accurate `FunctionToolCallEvent(args_valid=...)` events (`_agent_graph.py:1734`), pre-budget usage limits (`_agent_graph.py:1705-1708`), and skip approved calls' validation. Streaming uses `wrap_validation_errors=False` (`tool_manager.py:419-465`) so raw `ValidationError`/`ModelRetry` can be caught by the streaming handler instead of being wrapped.
- **Parallel execution with three modes.** Default `parallel` emits events as tasks complete (`asyncio.FIRST_COMPLETED`, `_agent_graph.py:1940-1945`); `parallel_ordered_events` waits for all tasks (`ALL_COMPLETED`) and yields in call order to keep message history stable for testing; `sequential` loops one at a time. Any tool with `sequential=True` (`tools.py:481, 720`) forces sequential mode for the entire call batch (`tool_manager.py:162-164`). Sibling task cancellation on error is at `_agent_graph.py:1947-1956`.
- **Capability hook layer around every stage.** Validation and execution are each wrapped in a before/wrap/after + on_error pattern (`tool_manager.py:252-354`), and there are parallel hook layers for output validation, output processing, and model requests. Hooks can short-circuit cleanly via `SkipToolValidation`/`SkipToolExecution`/`SkipModelRequest` exceptions (`exceptions.py:116-151`).
- **Typed-subclass registry, not hard-coded if/elif.** New native tools (e.g. `tool-search`) register `(part_kind, tool_kind) → Tag` entries in `_TYPED_PART_TAGS` at import time (`messages.py:1971-2006`); the same lookup drives streaming promotion (`_parts_manager.py:86-97`) and non-streaming promotion (`_agent_graph.py:2142-2161`). The comments at `messages.py:801-825, 1880-1894` lay out the four-step procedure to add a new typed subclass.
- **Per-call `Tool`/`FunctionToolset` config.** `Tool(timeout=..., max_retries=..., requires_approval=..., args_validator=..., defer_loading=...)` (`tools.py:467-580`) plus the matching `FunctionToolset.__init__` parameters (`toolsets/function.py:61-78`) allow per-tool and per-toolset overrides. Defaults flow from agent → toolset → tool, with `None` meaning "inherit."
- **Retry budgets are per-tool, not global.** `ctx.retries: dict[str, int]` tracks per-tool retry counts (`_run_context.py:60-61`), so a single bad tool doesn't burn the budget for others. `last_attempt` (`_run_context.py:154-156`) is exposed to validators. Output retries use a separate global `max_output_retries` budget (`ctx.state.output_retries_used`, `_agent_graph.py:1146-1171`).
- **Two deferred-tool surfaces sharing one carrier.** `kind='external'` and `kind='unapproved'` both flow through `DeferredToolRequests` (`tools.py:256-326`); the only difference is which field they're collected into (`calls` vs `approvals`) and the exception raised (`CallDeferred` vs `ApprovalRequired`, `exceptions.py:80-113`). Resolution via `DeferredToolResults` (`tools.py:378-421`) supports five `DeferredToolResult` variants: `ToolApproved`, `ToolDenied`, `ModelRetry`, `RetryPromptPart`, plain value (wrapped to `ToolReturn`).
- **Output tools use a different validation hook stack.** Output-tool calls (the model's "submit final answer" call) go through `validate_output_tool_call` and `execute_output_tool_call` (`tool_manager.py:505-699`), which use the **output** validate/process hooks rather than the tool validate/execute hooks. This keeps output validators in one pipeline and tool hooks in another, matching the text-vs-tool output paths.

## Notable Patterns

- **`RetryPromptPart` is the single retry surface.** Schema failures, `ModelRetry`, unknown tools, timeouts, custom validator failures, and external handler retries all funnel into one discriminated-union member with a uniform formatting rule (`messages.py:1406-1502`).
- **`ToolReturn` wrapper for two-channel returns.** A tool can return `(return_value, content=separate_user_prompt, metadata=not_sent)` (`messages.py:891-918`); `content` becomes a separate `UserPromptPart` after the `ToolReturnPart`, useful for "Here's the data + here's a separate explanation for the model."
- **`ValidatedToolCall.validation_error` carries the pre-wrapped `ToolRetryError`.** This lets the caller (e.g. streaming output handler) raise the retry without re-wrapping, while still counting the call against the retry budget during `validate_tool_call` (`tool_manager.py:389-417`).
- **`args_as_dict(raise_if_invalid=False)` with `INVALID_JSON` sentinel** lets downstream code re-serialize a malformed `ToolCallPart` for a retry roundtrip without losing the raw model-emitted text (`messages.py:1826-1850`).
- **`ctx.tool_call_approved: bool`** is the simple bit that flips when a deferred tool is approved (`_run_context.py:80-81`); `ApprovalRequiredToolset` reads it (`toolsets/approval_required.py:29`) and falls through if set.
- **`_populate_deferred_calls` and `_narrow_tool_call_parts`** are tiny pure functions that operate on already-collected data structures (`_agent_graph.py:1968-1981, 2142-2161`). Both reflect the broader pattern of using dataclass-replace flows rather than mutation.

## Tradeoffs

- **Single retry channel simplifies the agent but compresses error semantics.** Every tool problem becomes "fix and try again." If a caller wants to differentiate schema errors from `ModelRetry` from timeouts, they must inspect `RetryPromptPart.content` (str vs `list[ErrorDetails]`) — there is no separate field for error class. This is mitigated by `tool_call_id` and `tool_name` always being set, but a class discriminator would be cleaner.
- **`ValidationError.errors(include_url=False, include_context=False)`** is the canonical serialization (`tool_manager.py:186-189`), losing the URL hint and ctx dict. The error message itself includes the type and input, but in-band context is dropped — fine for model retry, less so for log inspection.
- **Output-tool retries use `max_output_retries`, tool retries use per-tool `max_retries`.** Two separate budgets make sense but require understanding both (`_agent_graph.py:1171`, `tool_manager.py:177-181`). The `ctx.retry` field shows the global output retry, `ctx.retries[name]` shows the per-tool retry (`_run_context.py:60-77`).
- **Per-tool `max_retries` defaults to the agent's tool retry count.** The agent-level retry count defaults to `1`, so most tools get one retry before `UnexpectedModelBehavior`. Callers wanting zero retries must opt in.
- **Capability hooks layer adds indirection.** Tracing is one capability, instrumentation is another; understanding tool execution often requires reading both `tool_manager.py` and `capabilities/`. The benefit is composability (e.g. `ApprovalRequiredToolset` is itself a `WrapperToolset` that doesn't need to know about hooks).
- **Parallel cancellation does not roll back completed siblings.** If 4 calls run in parallel and 2 succeed, 1 fails, 1 is cancelled — the agent continues with the 2 succeeded results and a `RetryPromptPart` for the failed call. There is no "atomic batch" semantic; each tool result is its own message part.

## Failure Modes / Edge Cases

- **Unknown tool name.** Recoverable. `ModelRetry(f'Unknown tool name: {name!r}. Available tools: ...')` (`tool_manager.py:369`) sends the available names to the model. Repeated unknown names exhaust the retry budget and raise `UnexpectedModelBehavior`.
- **Malformed arguments JSON.** Two layers. Transport-layer `args_as_dict` returns `{INVALID_JSON: '<raw>'}` (`messages.py:1847-1850`) so the value can be round-tripped to the model. Schema validation then raises `ValidationError`, which becomes a retry prompt with the raw input echoed back.
- **Tool body raises `ModelRetry`.** Recoverable via `_run_execute_hooks` (`tool_manager.py:294-354`) catching and wrapping with `_wrap_error_as_retry` (`tool_manager.py:183-191`). If `wrap_validation_errors=False`, the raw exception propagates (streaming path).
- **Tool body raises `CallDeferred` or `ApprovalRequired`.** Surfaces as a deferred call (`_agent_graph.py:1889-1894`), collected into `DeferredToolRequests` and returned as agent output. If the agent's output schema doesn't include `DeferredToolRequests`, a `UserError` is raised (`_agent_graph.py:1850-1855`).
- **Timeout exceeded.** `FunctionToolset.call_tool` (`toolsets/function.py:644-658`) catches `TimeoutError` and raises `ModelRetry(f'Timed out after {timeout} seconds.')`. Counts against the tool's retry budget; exhausted budget raises `UnexpectedModelBehavior`.
- **Async sibling cancellation.** On any non-`CancelledError` exception in `handle_call_or_result`, all sibling tasks are cancelled (`_agent_graph.py:1950-1956`); on `CancelledError`, all are cancelled with the original message (`_agent_graph.py:1947-1949`).
- **`approved=True` but validation now fails.** `_resolve_single_deferred` (`tool_manager.py:847-935`) re-validates with `approved=True`; if it fails, the retry path runs the same way.
- **`override_args` not provided.** Falls back to the original args (`_agent_graph.py:1811-1813`).
- **Mixed batch of function + output + deferred calls.** `process_tool_calls` (`_agent_graph.py:1537-1860`) classifies calls by `kind` and routes each bucket separately. Output calls may finish first, set `final_result`, and stub the remaining function calls with `'Tool not executed - a final result was already processed.'` if `end_strategy='early'` (`_agent_graph.py:1666-1680`).
- **Streaming partial JSON.** `allow_partial='trailing-strings'` is used by `ToolManager._validate_tool_args` (`tool_manager.py:234-243`) but only on `validate_output_tool_call`; regular tool validation defaults to strict. Streaming partial input is not enabled for tool calls.
- **Durable execution (Temporal, Prefect, DBOS).** `ValidatedToolCall.validated_args` is the cross-actor payload; tools wrap `_call_tool` in activities (`durable_exec/temporal/_toolset.py:122-168`), with `_wrap_call_tool_result`/`_unwrap_call_tool_result` to serialize roundtrippable values.

## Future Considerations

- **Unify `_resolve_single_deferred` with `_call_tool`.** Both dispatch `DeferredToolResult` variants (`tool_manager.py:847-935`, `_agent_graph.py:1984-2052`); the inline comment ("keep the dispatch branches here in sync", `tool_manager.py:861-863`) flags this as tech debt. A shared `_resolve_deferred_result(call, result, validated_or_call)` helper would reduce drift.
- **Add a typed error discriminator to `RetryPromptPart`.** Currently `content: list[ErrorDetails] | str` and the caller inspects the type. A small enum (`reason: Literal['validation' | 'model_retry' | 'timeout' | 'unknown_tool' | 'output_validation']`) would let tooling route retries differently.
- **Per-call budget pre-check.** `_agent_graph.py:1705-1708` pre-checks `usage_limits.tool_calls_limit` before running tools but does so after validation. Validating fewer calls (or short-circuiting validation when the cap is exhausted) would save Pydantic validator cost on already-over-budget runs.
- **Surface `AllowPartial` at the agent level.** `allow_partial=True` on `validate_output_tool_call` (`tool_manager.py:505-574`) is set for streaming output validation; the same opt-in for tool-call validation would help with long-argument streaming.
- **Drop `from_schema()` schema-validation skip.** `Tool.from_schema()` (`tools.py:583-636`) bypasses schema validation entirely, relying on `args_validator` or the tool body. A schema is required to construct the `Tool`, but it isn't enforced. Either enforce it or document the trade-off in the public API.
- **Improve `output_retries_used` counter visibility.** It's tracked in `GraphAgentState.output_retries_used` (`_agent_graph.py:1146, 1171`) and incremented in several places, but `RunContext.last_attempt` (`_run_context.py:154-156`) reflects the per-tool retry, not the output retry. Users writing validators often confuse the two.

## Questions / Gaps

- **What does the model adapter do with `INVALID_JSON`?** The framework tolerates it via `args_as_dict(raise_if_invalid=False)`, but it's unclear whether adapters scrub or rewrite the value before sending back. Some providers may reject the sentinel key on the next request.
- **How are native tool calls (e.g. OpenAI's built-in `WebSearchTool`) routed through the same roundtrip?** `_parts_manager.py:292-388` shows the streaming path; the non-streaming path (`_narrow_tool_call_parts`, `_agent_graph.py:2142-2161`) only promotes after the model response arrives. There must be a separate execution path for native tool calls that bypasses `_call_tools`. No clear evidence found in the files inspected — would need to inspect `models/openai.py` and `native_tools/`.
- **What happens if `args_validator` itself raises a non-`ModelRetry` exception?** `_run_validate_hooks` (`tool_manager.py:282-285`) catches `(ValidationError, ModelRetry)` and routes to `on_tool_validate_error`; other exceptions propagate. Unclear whether `on_tool_validate_error` is allowed to raise and what the resulting message looks like.
- **Are streaming partial-JSON tool calls supported?** `tool_manager.py:234-243` hard-codes `allow_partial='off'` for tool calls, but `'trailing-strings'` is available. The streaming parser (`_parts_manager.py:292-388`) does emit partial `ToolCallPart`s; unclear whether anyone re-validates them progressively or only after the call is complete.
- **How does `ModelRetry` interact with `wrap_validation_errors=False`?** `tool_manager.py:419-465` propagates the raw `ValidationError`/`ModelRetry`; the outer code (`_agent_graph._run_stream`, `result.py:validate_response_output`) must catch. Inspection needed at `result.py` to confirm.

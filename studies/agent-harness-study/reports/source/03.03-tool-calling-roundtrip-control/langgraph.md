## Summary

The tool-calling roundtrip in LangGraph's `prebuilt` library is **not a single iteration loop** — it is a graph-orchestrated pipeline where `create_react_agent` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:278-1002`) builds a `StateGraph` that alternates between an LLM-emitting `agent` node and a `tools` node built from `ToolNode` (`libs/prebuilt/langgraph/prebuilt/tool_node.py:622-1579`). Tool calls arrive as plain Python dicts attached to the last `AIMessage.tool_calls` (`tool_node.py:1224-1266`); they are extracted by `_parse_input`, dispatched to each tool through `_run_one` / `_arun_one` (`tool_node.py:1014-1067`, `1161-1222`), and returned as `ToolMessage` objects or LangGraph `Command` objects. The prebuilt layer does **no JSON parsing**: tool arguments must already be a `dict` because they come from `langchain_core`'s model adapters (which have already turned the model's emitted JSON string into a `dict` before it lands in `tool_calls`). This is the opposite of frameworks that have to scrape `{"name":...,"arguments":...}` strings from model output.

Argument validation is delegated to Pydantic through LangChain's `BaseTool.invoke` (`tool_node.py:957-958`); when validation fails, the Pydantic `ValidationError` is caught at `tool_node.py:959`, reformatted into `ToolInvocationError` (`tool_node.py:339-380`), then passed through `_filter_validation_errors` (`tool_node.py:510-563`) to strip out errors for `InjectedState`, `InjectedStore`, and `ToolRuntime` arguments so the model only sees feedback on arguments it controls. The current control flow does **not** retry or repair: invalid args produce a `ToolMessage(status="error")` (`tool_node.py:1006-1012`) which the LLM sees on the next round-trip and uses to emit corrected args. The framework has no in-pipeline "repair" step; it relies entirely on the LLM fixing itself.

Tool errors are first-class model-visible signals: every error is converted into `ToolMessage(content=..., status="error", tool_call_id=...)` (`tool_node.py:1006-1012`, `1152-1159`, `1061-1067`, `1214-1222`) and rejoined to the message stream so the next `agent` invocation sees the failure. `GraphBubbleUp` (used by interrupts) is the only exception type that always re-raises (`tool_node.py:982-983, 1129-1130`). Errors that are **not** in the configured `handle_tool_errors` types re-raise and abort the run (`tool_node.py:1002-1003, 1149-1150`). The `handle_tool_errors` flag is a six-way union (`tool_node.py:749-753`): `bool`, `str`, `type[Exception]`, `tuple[type[Exception], ...]`, or `Callable[..., str]`. The callable form's accepted exception types are inferred via reflection (`_infer_handled_types`, `tool_node.py:444-507`), giving typed error routing. There is no global "consecutive-error" budget inside the prebuilt layer; the agent loop uses `remaining_steps` (`chat_agent_executor.py:620-634`) to terminate after `recursion_limit - total_steps_taken` reaches zero, with a friendly final `AIMessage("Sorry, need more steps…")` instead of an exception.

Approval is achieved **indirectly** through the `interrupt_before=["agent"]` / `["tools"]` mechanism (`chat_agent_executor.py:447-454`), leveraging LangGraph's durable interrupts at the graph level — there is no built-in "approve this specific tool call" object that flows back into the LLM. For durable inspection, callers can put a `Send` payload containing `ToolCallWithContext` (`tool_node.py:286-306`) so each tool call becomes its own interrupted subgraph step.

Tool result formatting is a deliberate boundary pass: `msg_content_output` (`tool_node.py:309-336`) accepts `str` or `list[dict]` content blocks and JSON-serializes anything else with a `str()` fallback. After raw tool invocation, `_normalize_tool_response` (`tool_node.py:1432-1453`) enforces that responses are `ToolMessage`, `Command`, or a list of `Command | ToolMessage` — tools returning a dict or tuple raise `TypeError` unless the caller wraps it. `Command` returns are validated for a terminating `ToolMessage` matching the outer `tool_call_id` (`tool_node.py:1455-1501`, `1503-1579`), which is essential to keep the chat history consistent with LLM provider requirements.

Streaming tool calls get their own first-class infrastructure: `StreamToolCallHandler` (`libs/langgraph/langgraph/pregel/_tools.py:35-268`) emits `tool-started` / `tool-output-delta` / `tool-finished` / `tool-error` events on the `tools` stream channel, and `ToolRuntime.emit_output_delta` (`tool_node.py:1732-1750`) reads a `ContextVar` (`_tool_call_writer`, `libs/langgraph/langgraph/pregel/_tools.py:25-32`) so tools can stream partial output without threading a writer through their signature. `ToolCallTransformer` (`libs/prebuilt/langgraph/prebuilt/_tool_call_transformer.py:44-165`) projects those events into `ToolCallStream` handles (`_tool_call_stream.py:17-117`).

## Rating

**8 / 10** — The roundtrip is implemented as a graph-orchestrated pipeline with typed `ToolNode` and `ValidationNode` classes, a per-call `ToolCallRequest` abstraction that supports decorators (`wrap_tool_call` / `awrap_tool_call`), strict `Command`-shape validation with terminating-`ToolMessage` enforcement, a multi-strategy `handle_tool_errors` knob with type-introspected callables, and a separate `ValidationNode` that explicitly produces re-prompting flow. The implementation has comprehensive test coverage (`tests/test_tool_node.py:2430 lines`, `tests/test_tool_node_validation_error_filtering.py:470 lines`, `tests/test_tool_node_interceptor_unregistered.py:804 lines`, `tests/test_validation_node.py:88 lines`).

Gaps that hold it below 9–10:
- **No retry/repair inside the roundtrip.** Malformed or invalid-argument failures are converted into a `ToolMessage(status="error")` and handed back to the model; there is no scheduler or repair hook between detection and replay. The `wrap_tool_call` interface supports the user implementing their own retry (see `ToolCallWrapper` examples at `tool_node.py:226-276`), but the framework ships no built-in repair strategy and no default retry budget.
- **No parallel-bailout when one call fails.** When the loop is in `version="v2"` mode (`chat_agent_executor.py:846-859`), each `tool_call` is dispatched in parallel via the `Send` API; the executor cancels or fast-fails nothing — all `Send` branches complete (success or error) before the `agent` node reruns, so a slow tool blocks the model. `version="v1"` runs them inside one `ToolNode` invocation, which is also non-cancelable.
- **`ValidationNode` is deprecated** (`tool_validator.py:43-46`) in favor of `langchain.agents.create_agent`, so first-party "validate before run" lives downstream of this codebase. The re-prompt graph pattern (`tests/test_validation_node.py:88`) is no longer maintained here.
- **Graph-level interrupts are the only approval mechanism.** `ToolNode` has no `approval_required`, no per-call `approve(decision)`, and no concept of "model proposed but user rejected." A user wanting approval must compose the existing `interrupt_before=["tools"]` plus a custom edge that emits an `HumanInterrupt`-shaped message — none of which is encoded in `ToolNode` itself.
- **`Command.update` typing requires matching the ToolNode input shape** (`tool_node.py:1511-1534`); tools that return `Command(update=[...])` against a `dict`-shaped input raise `ValueError`. The mismatch surface is undocumented until runtime.
- **Tool-result formatting silently coerces** (`msg_content_output`, `tool_node.py:309-336`): any non-`str`/non-list result is `json.dumps`'d, with `str()` as last resort. A tool returning a non-JSON-serializable object produces `"<MyObject …>"` in the model's view with no diagnostic.

## Answers to Dimension Questions

**1. How are tool calls represented?**
Tool calls are plain dicts with `name`, `args` (dict), `id`, and `type="tool_call"`, attached to an `AIMessage.tool_calls` field (`langchain_core.messages`). `ToolNode` accepts three input shapes: `[AIMessage(...)]`, `{messages: [AIMessage(...)]}`, or `[ToolCall, ...]` (`libs/prebuilt/langgraph/prebuilt/tool_node.py:1224-1266`). They are also wrapped in `ToolCallRequest` dataclass instances (`tool_node.py:132-199`) inside the wrapper layer, and `ToolCallWithContext` TypedDicts (`tool_node.py:286-306`) when dispatched via `Send`. There is no in-house JSON parser — `langchain_core` model adapters convert the model's emitted string to a dict before the value reaches the prebuilt layer.

**2. What happens if arguments are invalid?**
Pydantic raises `ValidationError` when `BaseTool.invoke` runs (`tool_node.py:957-958`). `ToolNode` catches it, builds a `ToolInvocationError` (`tool_node.py:339-380, 959-966`), and the filter `_filter_validation_errors` (`tool_node.py:510-563`) removes errors whose first `loc` element is in the tool's `_InjectedArgs.all_injected_keys`, so the model never sees "argument `state` is required" for parameters it doesn't control. The error message is then routed through `_handle_tool_error` with the default template `TOOL_INVOCATION_ERROR_TEMPLATE` (`tool_node.py:117-121, 1006-1012`), producing `ToolMessage(status="error", content="Error invoking tool … Please fix the error and try again.")`. The tool is **not** re-invoked automatically — repair is delegated to the model's next `AIMessage`. Verified by `tests/test_tool_node.py:269-294` ("default invocation" produces a `status="error"` message) and the filtering tests `tests/test_tool_node_validation_error_filtering.py:41-200`.

**3. Are tool results structured?**
Yes, but the surface is `ToolMessage | Command | list[Command|ToolMessage]`. `msg_content_output` (`tool_node.py:309-336`) accepts `str` and `list[ContentBlock]` already in tool-message-block format; everything else is JSON-serialized with a `str()` fallback. The `Command` path carries `goto`, `graph=Command.PARENT|None`, and `update`, which LangGraph's reducer treats as a parent-edge send. The framework validates that **exactly one terminating `ToolMessage`** with the outer `tool_call_id` is present in `Command.update` when the command targets the current graph (`tool_node.py:1455-1579`), preventing the agent loop from getting stuck in broken chat histories.

**4. Are tool errors visible to the model?**
Always when `handle_tool_errors` resolves truthy for the raised type. Each error becomes `ToolMessage(content=..., name=call["name"], tool_call_id=call["id"], status="error")` (`tool_node.py:1006-1012, 1061-1067, 1152-1159, 1214-1222`) and rejoins the message stream. The default handler only catches `ToolInvocationError` (i.e. arg-validation failures); runtime exceptions re-raise (`tool_node.py:383-391`). Callers can widen to `True`, `str`, `tuple[Exception, ...]`, `type[Exception]`, or a `Callable[..., str]` whose first parameter's annotation specifies accepted exception types (`tool_node.py:444-507`). Per-tool overrides via `tool.handle_tool_error` (e.g. `tool5.handle_tool_error = "foo"`) still go through the same routing (`tests/test_tool_node.py:536-562`). `GraphBubbleUp` is the only escape hatch — it always re-raises so interrupts can pause the run (`tool_node.py:982-983`, `tests/test_tool_node.py:594-623`).

**5. Can tool calls be approved before execution?**
Not as a first-class construct inside `ToolNode`. Approval is achieved via the graph-level `interrupt_before=["tools"]` (or `"agent"`) parameter on `create_react_agent` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:302-303, 447-454, 823-825, 998-1000`). When the run is resumed via a checkpointered `invoke(None)`, LangGraph's interrupt machinery re-enters the `tools` node and the tool runs. The deprecated `HumanInterrupt` / `ActionRequest` / `HumanResponse` TypedDicts (`libs/prebuilt/langgraph/prebuilt/interrupt.py:1-105`) are kept only for back-compat; current human-in-the-loop typing has moved to `langchain.agents.interrupt`. There is no built-in per-call "approve / edit / reject" object that gets attached to a `ToolMessage` after the fact.

## Architectural Decisions

- **Graph-native execution** (`chat_agent_executor.py:278-1002`): the roundtrip is compiled into a `StateGraph` rather than a Python `while` loop. This makes the loop checkpointable, restartable, and observable through the standard `Pregel` machinery, but it means "reordering" or "early-exit" tool calls requires graph-level edges (`Send`, `Command.PARENT`), not function calls.
- **`ToolNode` is a `RunnableCallable`, not a graph** (`tool_node.py:622, 772`): the node holds the routing logic and is added to the graph as a leaf, so it benefits from `Pregel.stream` callbacks and parallel dispatch (`get_executor_for_config`, `tool_node.py:800-826`) but is not itself a sub-graph.
- **Tool-call detection is delegated to `langchain_core`** (`tool_node.py:63-70`): the prebuilt layer never sees raw `{"name": ..., "arguments": ...}` JSON. This isolates model-format details (OpenAI vs Anthropic vs Bedrock tool-call shapes) to the provider packages and lets `ToolNode` focus on dispatch, validation, and error routing.
- **`wrap_tool_call` is the open extension point** (`tool_node.py:202-283, 755-778`): retry, caching, request modification, and approval are all expressed as middleware around a single `execute(request)` callable. The interface deliberately passes `ToolCallRequest` rather than a flat dict so the wrapper can `request.override(...)` for retry.
- **Parallel dispatch via `Send` (`version="v2"`) vs in-node `executor.map` (`"v1"`)** (`chat_agent_executor.py:843-859, 800-826`): the v2 path sends each tool call as a separate task so each can independently checkpoint / interrupt — useful for human-in-the-loop pauses per call — at the cost of losing parallel-bailout. v1 keeps tool calls in one task, simpler but no per-call interrupt.
- **Default error handler is narrow (`ToolInvocationError` only)** (`tool_node.py:383-391`): explicit-runtime-error failures bubble up rather than feeding the loop. The rationale lives implicitly in the docstring example (`tool_node.py:690-694`) — most runtime errors in user tools are bugs, not user-fixable prompts.
- **Errors are surfaced as `status="error"` `ToolMessage`s, not retries** (`tool_node.py:1006-1012`): repair is the model's job. This fits the LangGraph ethos of being a low-level orchestration framework, not an opinionated agent.
- **Injected-argument filtering** (`tool_node.py:510-563, 1421-1430`): two layers of safety — first, injected args are filtered out of validation error messages so the model only sees relevant feedback; second, the framework strips LLM-supplied values for `InjectedToolArg` keys before re-injecting trusted values, preventing the model from forging runtime state.

## Notable Patterns

- **Immutable `ToolCallRequest` with `override(...)`** (`tool_node.py:151-199`): discourages in-place mutation; encourages constructing the next request from the previous one. Mirrors LangGraph's broader preference for immutable state transitions.
- **`_InjectedArgs` pre-computed per tool at `ToolNode.__init__`** (`tool_node.py:786`): one reflection pass at construction, then `_inject_tool_args` becomes a dict lookup + state dot-access — cheap at run time.
- **Type-driven error routing** (`tool_node.py:444-507, 982-1012`): a single `handle_tool_errors` knob has six accepted shapes, and the `Callable` shape uses `inspect.signature` + `get_type_hints` to figure out which exceptions the handler accepts. Surfaces intentional behavior without forcing runtime type checks.
- **`Send` + `ToolCallWithContext`** (`tool_node.py:286-306, 1281-1313; chat_agent_executor.py:846-859`): tool calls become separate graph tasks, each carrying the state needed for its injection. Decouples parallelism from parallel-bailout.
- **Stream channel + `ContextVar` writer** (`libs/langgraph/langgraph/pregel/_tools.py:25-32, 121-166`; `tool_node.py:1732-1750`): tools stream partial output without taking an explicit `StreamWriter` parameter. The handler sets a per-call context, and `ToolRuntime.emit_output_delta` reads it. Mirrors langchain-core's `get_stream_writer` (`tests/test_tool_node.py:26`).
- **`_combine_tool_outputs` coalesces parent `Command(PARENT, goto=[Send, Send, …])`** (`tool_node.py:891-919`): parallel tool calls issuing parent navigation collapse their `goto` lists into one combined `Command`, so the supervisor sees a single navigation decision per round.

## Tradeoffs

- **Repair vs raise.** The prebuilt layer converts every argument-validation error to a `ToolMessage(status="error")` and trusts the model to fix it. This is permissive — runs don't die on bad arguments — but it leaves the "did the model actually fix it?" question to the LLM with no in-house retry budget. A consumer needing guaranteed convergence must implement it inside `wrap_tool_call` (`tool_node.py:226-276`).
- **Parallelism vs duration control.** v2 parallelizes tool calls via `Send` (`chat_agent_executor.py:846-859`) — great throughput, but a single slow tool blocks the model because the agent waits for all branches to finish. v1 keeps parallel calls in one task via `executor.map` (`tool_node.py:800-826`) for the same effect. Neither offers timeout / cancel.
- **Cancellation vs preserving audit.** `GraphBubbleUp` is the only exception type that always re-raises (`tool_node.py:982-983`). Everything else becomes a `ToolMessage`. This keeps the run producing chat history even on unexpected failures, but means "uncaught exception that crashes the graph" semantics are intentionally narrow.
- **Per-tool customization vs global defaults.** The `handle_tool_errors` knob is six-way and supports per-tool override via `tool.handle_tool_error`, but the messaging format is fixed (`TOOL_CALL_ERROR_TEMPLATE`, `INVALID_TOOL_NAME_ERROR_TEMPLATE`, `TOOL_INVOCATION_ERROR_TEMPLATE` — `tool_node.py:108-121`). To customize wording across the board a caller wires a `Callable` that returns its own string, but the templates themselves aren't pluggable.
- **Validation done at execution time, not gate time.** `ToolNode` doesn't pre-validate args before invocation — pydantic runs when `tool.invoke` is called (`tool_node.py:957-958`). The deprecated `ValidationNode` (`tool_validator.py:43-221`) is the only "validate then continue" gate, and it's deprecated.
- **Streaming is opt-in.** `ToolCallTransformer` is registered explicitly by users (`_tool_call_transformer.py:61-63`); the framework does not auto-attach. This is consistent with LangGraph's stream-mode opt-in model but means out-of-the-box tool-call streaming is invisible unless the caller asks.

## Failure Modes / Edge Cases

- **Unknown tool name** → `ToolMessage(status="error", content="Error: <name> is not a valid tool, try one of [<available>].")` (`tool_node.py:1268-1279`). Verified by `tests/test_tool_node.py:565-591`. Loop continues; the LLM sees which tools exist.
- **Missing required argument** → pydantic `ValidationError` → `ToolInvocationError` → `ToolMessage(status="error", content="Error invoking tool … field required …")`. Injected-arg errors are filtered out (`tests/test_tool_node_validation_error_filtering.py:95-151`).
- **Type mismatch** → same path as above; filtered errors retain only LLM-controlled fields (`tests/test_tool_node_validation_error_filtering.py:154-200`).
- **Runtime exception in tool body** → by default re-raises; `handle_tool_errors=True` converts to `ToolMessage(status="error", content=TOOL_CALL_ERROR_TEMPLATE.format(error=repr(e)))` (`tool_node.py:982-1012`). Verified in `tests/test_tool_node.py:319-377`.
- **Unregistered tool but `wrap_tool_call` returns a result** → supported; interceptor can short-circuit without calling `execute(request)` (`tests/test_tool_node_interceptor_unregistered.py:48-169`). If it calls `execute`, the run falls through to `_validate_tool_call` and produces the standard "not a valid tool" error.
- **`Command(update={"messages": ...})` against a list-shaped input** → `ValueError` (`tool_node.py:1511-1520`). Documented only in the docstring + tests; symmetry mismatch surfaces at runtime.
- **List of `Command`/`ToolMessage` from a tool with zero or multiple terminators** → `ValueError` with the count of bound terminators (`tool_node.py:1479-1485`). The current contract is **exactly one** matching terminator.
- **Recursion exhaustion** → agent returns a final `AIMessage("Sorry, need more steps to process this request.")` instead of raising (`chat_agent_executor.py:620-634, 684-694`). No `GraphRecursionError` for the v2 path.
- **Chat history with dangling `tool_calls`** → `ValueError` with `ErrorCode.INVALID_CHAT_HISTORY` at the start of every model invocation (`chat_agent_executor.py:243-271, 651`). Prevents sending broken histories to providers.
- **Tool returns non-`str` / non-`list[ContentBlock]` output** → coerced via `json.dumps` with `str()` fallback (`tool_node.py:309-336`). Possible silent data-loss for non-JSON-serializable objects (`repr` in the model's view).
- **Wrapper exception raised outside `tool.invoke`** → caught and converted via the same `handle_tool_errors` pipeline (`tool_node.py:1054-1067`). Bad wrapper code still produces a structured error message rather than crashing.
- **Streaming tool emits deltas after error** → `on_tool_error` resets the `ContextVar` and emits `tool-error`; leftover in-flight writers from the start hook are dropped (`libs/langgraph/langgraph/pregel/_tools.py:167-201`).

## Future Considerations

- **Move to `langchain.agents.create_agent`.** The prebuilt module's `create_react_agent` and `AgentState` are wrapped in `@deprecated(category=LangGraphDeprecatedSinceV10)` (`chat_agent_executor.py:53-56, 65-68, 84-87, 100-103, 274-277`). The deprecation messages point at `langchain.agents.create_agent`, which uses `ToolNode` internally and adds middleware. Anyone building new agents should consume the upstream package; `ToolNode` itself remains the dispatch primitive.
- **`ValidationNode` is deprecated** (`tool_validator.py:43-46`); the "validate, then re-prompt the model on errors" pattern needs to be implemented via `wrap_tool_call` plus the upstream `create_agent`'s tool-error middleware.
- **No native approval content type.** Adding `function_approval_request` / `function_approval_response` shapes (à la `microsoft/agent-framework` `_types.py:1213-1253`) would close the gap between graph-level interrupts and per-call approval. Currently there is no idiomatic way for the LLM to ask "should I do this?" mid-loop.
- **Repair scheduling.** A first-party "retry up to N times with the model's own corrections" policy would reduce the burden on callers — the wrapper examples at `tool_node.py:226-276` already show the shape, but the framework doesn't ship a default.
- **Timeout / cancel for parallel tool calls.** v2's `Send` dispatch has no deadline; a runaway tool holds the whole agent step. Adding a per-call timeout or a `asyncio.wait_for`-style first-success would let supervisors avoid wall-clock hazard.
- **Schema-based runtime gating.** Pydantic validation happens inside the tool; pairing it with an early "looks like a known tool with a known schema" check (analogous to `ValidationNode` but for the live path) could turn some argument failures into early-exits.
- **Per-call observability.** `ToolCallTransformer` (`_tool_call_transformer.py:44-165`) is opt-in; making the `tool-started` / `tool-output-delta` / `tool-finished` / `tool-error` stream part of default observability would surface more diagnostics without ceremony.

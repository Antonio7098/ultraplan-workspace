## Summary

The repository implements a deliberate, versioned durable-agent state schema shared between its .NET and Python implementations. The schema is documented as a JSON Schema 2020-12 contract (`schemas/durable-agent-entity-state.json`), which mandates a semver `schemaVersion` string as the first root property and wraps a `data` object that holds an ordered `conversationHistory` (`schemas/durable-agent-entity-state.json:207-216`). Both runtimes reproduce this contract with parallel type hierarchies (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/*.cs`, `python/packages/durabletask/agent_framework_durabletask/_durable_agent_state.py`) and a dedicated JSON converter that performs validation before constructing the in-memory object (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateJsonConverter.cs:17-56`; `python/packages/durabletask/agent_framework_durabletask/_durable_agent_state.py:419-443`). Polymorphism is implemented through `$type` discriminators (`schemas/durable-agent-entity-state.json:121-134`; `dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateContent.cs:12-23`; `dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateEntry.cs:12-14`) so that new AI content types can be added without breaking older payloads, and a catch-all `unknown` content type preserves unrecognized data (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateUnknownContent.cs:12-43`; `python/packages/durabletask/agent_framework_durabletask/_durable_agent_state.py:1298-1334`). All persistence-bearing models in .NET also use `[JsonExtensionData]` to round-trip foreign keys (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentStateData.cs:30-31`; `DurableAgentStateEntry.cs:42-43`; `DurableAgentStateMessage.cs:43-44`; `DurableAgentStateContent.cs:29-30`; `DurableAgentStateUsage.cs:39-40`). The workflow checkpoint pipeline uses a different, simpler versioning story: a `version: str = "1.0"` field on `WorkflowCheckpoint` (`python/packages/core/agent_framework/_workflows/_checkpoint.py:88`) plus a `graph_signature_hash` topology hash that blocks restore across incompatible graphs (`_functional.py:921-926`). The primary durability story does not include explicit migration scripts or stored old-state fixtures; compatibility relies on additive, optional evolution, graceful handling of missing/unknown versions, and runtime `JsonExtensionData`/`unknown` envelopes.

## Rating

**7 / 10** — Clear, documented, contract-first model with explicit version validation, additive extensibility points (`$type` discriminators, `unknown` content, `JsonExtensionData`), and round-trip tests. Missing: stored old-state fixtures, automated downgrade/upgrade tests across version bumps, dedicated migration scripts, and an explicit `schemaVersion` history beyond the README. The Python and .NET implementations also diverge on how strictly they reject missing or malformed `schemaVersion`.

## Answers to Dimension Questions

1. **Does persisted state carry a version?**
   - Yes — but only on one of two pipelines. The durable-agent entity state carries `schemaVersion` (semver) as a required root property (`schemas/durable-agent-entity-state.json:207-216`), mirrored by both .NET (`DurableAgentState.cs:25-26`) and Python (`_durable_agent_state.py:394-398`). The in-process workflow checkpoint pipeline carries its own `version: str = "1.0"` field on `WorkflowCheckpoint` (`_checkpoint.py:88`) and additionally a `graph_signature_hash` for topology mismatch detection (`_checkpoint.py:48-49`; `_functional.py:921-926`). Other persisted shapes (Durable Functions orchestration history payloads, framework-level `AgentSession`/`RunRequest`) do not embed a separate `schemaVersion`.

2. **What happens when old state is loaded?**
   - .NET `DurableAgentStateJsonConverter` rejects anything that is not `Major == 1`: a missing property throws `"missing the 'schemaVersion' property"` (`DurableAgentStateJsonConverter.cs:30`); a non-semver value throws `"invalid 'schemaVersion' property"` (`DurableAgentStateJsonConverter.cs:35`); any non-`1.x.x` throws `"schema version '{x}' is not supported"` (`DurableAgentStateJsonConverter.cs:40`). A correct `1.x.x` payload is parsed, with the parsed `schemaVersion` retained on the in-memory object (`DurableAgentStateJsonConverter.cs:51-55`). The Python `DurableAgentState.from_dict` is much more permissive: a missing `schemaVersion` produces a warning log and silently substitutes a fresh state, losing all history (`_durable_agent_state.py:426-429`); any present value (including `"2.0.0"`) is accepted without validation (`_durable_agent_state.py:431`). For .NET workflow orchestrator state, `WorkflowCheckpointException` is raised on bad payloads (`_checkpoint.py:115-116`), and a mismatched `graph_signature_hash` raises a clear "different version of workflow ... not compatible" error (`_functional.py:922-926`).

3. **Are migrations automatic or manual?**
   - There are no explicit migration scripts or version-bump handlers. Compatibility is enforced structurally: (a) new properties are declared optional in the schema (`schemas/durable-agent-entity-state.json:17-21` and per-content definitions); (b) `[JsonExtensionData]` (`DurableAgentStateData.cs:30-31`, etc.) preserves unrecognized keys; (c) the `unknown` content type and the `case _:` wildcard (`_durable_agent_state.py:223-227`, `DurableAgentStateUnknownContent.cs:25-33`) absorb new AI content types so future payloads decode without code changes; (d) polymorphic `$type` discriminators (`schemas/durable-agent-entity-state.json:168`, `189`; `DurableAgentStateContent.cs:12-23`) let new variants be added without touching old ones. There is no migration tool or version-targeted upcast/downcast routine.

4. **Are JSON blobs versioned?**
   - Yes for the durable-agent blob (`schemaVersion` is required in the schema and required by the .NET converter). The workflow checkpoint JSON uses a separate `version` field at the dataclass level (`_checkpoint.py:88`) but it is not propagated into the JSON itself in the same explicit way; version drift there is mostly caught via `graph_signature_hash`. Inline content blobs are versioned by their `$type` discriminator, which doubles as the addition marker. Provider-specific JSON RPC payloads (A2A, AG-UI, Foundry Responses) use protocol-defined `version` fields outside this schema (e.g., `agent_definitions.py:44,92,114,136`), but those are wire protocols, not persisted state.

5. **Can old checkpoints/events still be read?**
   - Old major-version `1.x.x` durable-agent state can be read by the .NET implementation today (no minor-version comparison is enforced — `DurableAgentStateJsonConverter.cs:38-41` checks only `Major == 1`). Tests demonstrate `1.0.0` JSON being parsed and round-tripped (`DurableAgentStateTests.cs:89-169`). Python tests also exercise `1.0.0` fixtures (`test_durable_entities.py:187-210`; `azurefunctions/tests/test_entities.py:63-201`; `azurefunctions/tests/test_app.py:503-1118`), but only for `reset`/`restore` semantics, not for full content fidelity across versions. For workflow checkpoints, the `graph_signature_hash` mismatch is the dominant guard; older checkpoints with the same topology hash still load, but a refactor that changes the topology causes restore to fail (`_functional.py:921-926`). Unknown `$type` discriminators degrade to base/unknown content rather than failing (`_durable_agent_state.py:124`, `223-227`), so unrecognized entries and contents survive.

## Architectural Decisions

- **Schema-as-contract, not framework mirror.** Durable agent state uses its own parallel type hierarchy rather than directly persisting `ChatMessage`/`AIContent` (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/README.md:7`). The justification recorded in the README is explicit: cross-language stability (.NET vs Python), versioning control, and external-tool consistency. The schema file is the single source of truth consumed by both runtimes.
- **Required, semver, first-property `schemaVersion`.** The schema enforces `schemaVersion` as the first property and requires it (`schemas/durable-agent-entity-state.json:207-216`), and the README records the policy that durable agents "should not attempt to deserialize semver-incompatible versions" (`dotnet/src/Microsoft.Agents.AI.DurableTask/State/README.md:18`).
- **Polymorphic `$type` discriminator.** Every entry and content item carries a `$type` discriminator (`schemas/durable-agent-entity-state.json:121-134`, `162-194`; `DurableAgentStateContent.cs:12-23`; `DurableAgentStateEntry.cs:12-14`). This decouples addition of new content types from version bumps and is what enables the `unknown` fallback.
- **`unknown` content + `[JsonExtensionData]` as forward-compat envelopes.** The schema explicitly defines an `unknownContent` type (`schemas/durable-agent-entity-state.json:111-119`); both runtimes route anything they cannot classify into it (`DurableAgentStateUnknownContent.cs:25-33`; `_durable_agent_state.py:223-227`). Combined with `[JsonExtensionData]` on every container, this preserves unknown fields across round-trips.
- **Source-generated JSON context for AOT/trimming.** .NET uses `JsonSerializerContext` for durable agent types (`DurableAgentStateJsonContext.cs:10-39`) plus a chained singleton `JsonSerializerOptions` (`DurableAgentJsonUtilities.cs:59-79`), which keeps the schema-driven contract stable under Native AOT.
- **Workflow checkpoints use a simpler versioning story.** Rather than a per-payload `schemaVersion`, the workflow pipeline attaches a topology hash (`graph_signature_hash`) and a `version: str = "1.0"` field (`_checkpoint.py:48-49,88`). Compatibility is by structural hash; runtime errors are raised when the hash mismatches (`_functional.py:921-926`). This is appropriate for short-lived, in-process state but is a much weaker contract than the durable-agent schema.
- **Defense-in-depth for pickle-based workflow encoding.** Checkpoint values are encoded as JSON with embedded pickle+base64 (`_checkpoint_encoding.py:1-100`). The pickle path is sandboxed via `RestrictedUnpickler` allowlists and the workflow client strips pickle markers from server responses (`_workflows/client.py:303-306`). This is a security control, not a versioning control, but it materially affects how unknown future types are reconstructed.

## Notable Patterns

- **First-class forward compatibility.** Every persisted object has an `ExtensionData` (`JsonExtensionData` on .NET containers, `extension_data`/`extensionData` on Python equivalents). The Python `DurableAgentStateUsage` even merges non-standard fields into a typed `extensionData` and rehydrates them on `to_usage_details` (`_durable_agent_state.py:1237-1263`), providing a path for provider-specific fields.
- **Discriminator-driven polymorphism with explicit fallback.** Both runtimes enumerate known `$type` values and always provide a base or `unknown` branch (e.g., `case _:` at `_durable_agent_state.py:223`, the `default:`-style `_ =>` at `DurableAgentStateContent.cs:57`). New content types can be added by adding a single line without coordinating a schema major bump.
- **Two-tier error reporting.** Validation failures in .NET throw `InvalidOperationException` with caller-actionable text (`DurableAgentStateJsonConverter.cs:30,35,40`); Python downgrades missing `schemaVersion` to a `logger.warning` plus silent state reset (`_durable_agent_state.py:426-429`). Workflow checkpoint failures raise `WorkflowCheckpointException` (`_checkpoint.py:115-116`).
- **Round-trip tests as the contract.** Both `DurableAgentStateTests.BasicState` (.NET) and `test_durable_agent_state.test_round_trip_serialization` (Python) lock in the on-disk format with explicit JSON literal assertions (`DurableAgentStateTests.cs:89-169`; `test_durable_agent_state.py:198-219`). These are the most concrete artifacts describing the current schema.
- **Schema fixtures in tests, not in `fixtures/`.** Tests inline small JSON literals carrying `1.0.0` (`test_durable_entities.py:190-202`; `test_app.py:511-533`; `test_entities.py:72-89`). There is no `tests/fixtures/v1.0.0/state.json` repository of golden payloads, which means upgrades must construct fixtures by hand.

## Tradeoffs

- **Additive-only evolution.** Choosing extension data + `unknown` discriminators avoids migration code but means there is no formal mechanism for renaming, splitting, or removing fields. Any breaking shape change must bump to `2.0.0`, after which the .NET converter will hard-reject every existing durable entity (`DurableAgentStateJsonConverter.cs:38-41`) — a global data-loss event for any operator that upgrades without a separate data migration.
- **Asymmetric validation rigor between runtimes.** .NET enforces `schemaVersion` presence and major-version compatibility (`DurableAgentStateJsonConverter.cs:28-41`); Python silently resets state on missing `schemaVersion` and does not validate the major (`_durable_agent_state.py:426-431`). A payload with `schemaVersion: "2.0.0"` written by a hypothetical future release will load on Python but not on .NET. There is no test for cross-runtime version-skew.
- **Strict default vs. silent reset are both problematic.** .NET's hard rejection will surface bugs early but can brick long-lived agent entities. Python's silent reset will mask schema divergence but destroy user history without warning. Neither path provides a "downgrade or migrate" hook.
- **Version history is implicit.** The only documentation of past schema versions is `dotnet/src/Microsoft.Agents.AI.DurableTask/CHANGELOG.md:37,50` (orchestration ID added; empty content filtered) and the README's "should strive to be compatible with older schema versions" guidance. There is no machine-readable changelog of `schemaVersion` history or migration recipes.
- **Workflow checkpoint version is fixed at `"1.0"`.** The constant is a default on the dataclass (`_checkpoint.py:88`) but there is no logic to write or read a different value, and no test that validates this default propagates. Forward evolution of the checkpoint format is gated entirely by the `graph_signature_hash` check, which conflates refactor with incompatibility.
- **JSON pickle encoding accepts arbitrary framework classes.** `RestrictedUnpickler` allowlists `agent_framework.*` and `openai.types.*` by module prefix (`_checkpoint_encoding.py:67-71`). This permits reconstruction of any current or future framework class from a checkpoint — convenient but means a single corrupted checkpoint can pull in arbitrary code if the storage backend is ever compromised.

## Failure Modes / Edge Cases

- **Old `1.0.0` payload after orchestrator-id introduction (PR #2137).** The schema added `orchestrationId` to requests (`schemas/durable-agent-entity-state.json:169-172`; `DurableAgentStateRequest.cs:17-18`; `_durable_agent_state.py:564,589-590,609`). There is no documented migration for `1.0.0` state lacking this field; both runtimes treat it as optional, so reads work, but the Python regression test `test_durable_agent_state.py:43-114` covers only the new path.
- **Unknown AI content type from a newer framework version.** Covered gracefully: routed to `DurableAgentStateUnknownContent` on both runtimes (`DurableAgentStateUnknownContent.cs:25-33`; `_durable_agent_state.py:1298-1334`); Python's `to_ai_content` falls back to `Content(type="unknown", additional_properties={"content": self.content})` when `Content.from_dict` cannot parse the inner object (`_durable_agent_state.py:1325-1334`). Round-trip is preserved at the state layer even if the AI content layer cannot fully hydrate it.
- **Forward compatibility for legacy function-call arguments.** Older `functionCall` content stored `arguments` as a dict; current code accepts both dict and string (`_durable_agent_state.py:966-978`; covered by `test_function_call_content_supports_legacy_mapping_arguments` at `test_durable_agent_state.py:240-251`). The Python implementation now serializes `arguments` as a JSON string (`to_ai_content` at `_durable_agent_state.py:980-981`), but legacy dict payloads still load.
- **MCP-tool content with non-JSON-serializable nested objects.** Issue #4719 produced the regression test `test_state_with_mcp_content_is_json_serializable` (`test_durable_agent_state.py:495-521`), which now relies on `DurableAgentStateUnknownContent.from_unknown_content` to coerce `Content` instances to dicts (`_durable_agent_state.py:1320-1323`). This is the only migration-shaped test added in response to a real-world breakage.
- **Missing `data` envelope in .NET.** `MissingData` test confirms the .NET converter throws (`DurableAgentStateTests.cs:40-53`). Python silently substitutes an empty `DurableAgentStateData()` (`_durable_agent_state.py:432`), again divergent.
- **Breaking major version.** .NET rejects `2.0.0` payloads (`DurableAgentStateTests.BreakingVersion` at `:25-38`). There is no Python test for a `2.0.0` payload because Python does not validate major versions.
- **Workflow topology drift.** A non-matching `graph_signature_hash` raises and aborts restore (`_functional.py:921-926`). This is a hard failure, not a migration.
- **`expirationTimeUtc` and other runtime-only fields.** Defined on `DurableAgentStateData` (`DurableAgentStateData.cs:24-25`) and surfaced as `[JsonExtensionData]`; not part of the published schema, so older code reading newer state ignores it. There is no explicit schema-level definition for this field, which is a small consistency gap with the schema's role as single source of truth.

## Future Considerations

- **Publish an authoritative schema-version registry.** A machine-readable map from `schemaVersion` → introduction PR → migration recipe would let operators reason about durability. Today, the only hints are scattered across `dotnet/src/Microsoft.Agents.AI.DurableTask/CHANGELOG.md` and `State/README.md`.

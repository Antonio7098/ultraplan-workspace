## Summary

Temporal implements a layered schema-versioning strategy that separates three concerns: (1) **database schema versions** are tracked in per-store `schema_version` tables and managed through a generic, manifest-driven upgrade tool (`tools/common/schema/updatetask.go:77-106`) that runs only forward migrations with explicit `CurrVersion` / `MinCompatibleVersion` manifests per step (`schema/cassandra/temporal/versioned/v1.13/manifest.json:1-7`, `schema/mysql/v8/temporal/versioned/v1.19/manifest.json:1-7`); (2) **persisted state envelopes** are `commonpb.DataBlob` records that carry an explicit `EncodingType` enum (proto3 or JSON) and are decoded with strict mapping in `common/persistence/serialization/codec.go:104-121`, with backward-compatibility achieved through protobuf's `unknownFields` plus targeted `DiscardUnknown: true` unmarshalers (`common/persistence/serialization/serializer.go:227-233`, `common/codec/jsonpb.go:128-132`); and (3) **runtime versioning of workflow executions** uses two parallel mechanisms — a legacy `VersionHistory` event-id list (`common/persistence/versionhistory/version_history.go:66-90`) and the newer `VersionedTransition` + `TransitionHistory` HSM vector (`common/persistence/transitionhistory/transition_history.go:51-125`, `api/persistence/v1/hsm.pb.go:457-510`) — that are designed to be safely initialized from missing or partial fields. Migration is **deliberately one-way** for the database schema (`common/persistence/schema/version.go:21-28`) — the `VerifyCompatibleVersion` function explicitly accepts a code rollback (DB > code) but refuses a DB downgrade (code > DB). Old checkpoints and events are decodable today primarily through protobuf wire-format tolerance; there is no dedicated "old state fixtures" directory and no automated cross-version state-load tests. Several hand-rolled "if nil, populate" patches inside `serializerImpl` (e.g. `WorkflowExecutionStateFromBlob` at `common/persistence/serialization/serializer.go:461-470`, `ShardInfoFromBlob` at `common/persistence/serialization/serializer.go:386-404`) document the ad-hoc nature of field additions. The Elasticsearch visibility schema additionally relies on `sadefs.ErrInvalidName` silently swallowing unknown field names (`common/persistence/visibility/store/elasticsearch/visibility_store.go:1063-1065`) as a forward-compat escape hatch.

## Rating

**7 / 10** — Multi-database schema versioning is fully engineered, with declarative manifests, a stable upgrade tool, `MinCompatibleVersion` enforced at startup (`common/persistence/schema/version.go:21-28`), and protobuf's wire-format tolerance providing forward/backward compatibility for stored blobs. Versioned transitions are an explicit design with deterministic staleness checks (`common/persistence/transitionhistory/transition_history.go:81-125`). Migrations are forward-only by policy, with documented rationale and rollback semantics. Missing: dedicated old-state fixtures, no automated tests asserting that yesterday's `DataBlob` decodes today, ad-hoc "if nil, populate" patches rather than a single migration registry, and a sharp gap between the production-grade DB schema toolchain and the more ad-hoc in-protobuf handling of evolving message types.

## Answers to Dimension Questions

1. **Does persisted state carry a version?**
   - The answer is **layered**. The database schema does: every store (Cassandra / MySQL / Postgres / SQLite) writes its current `Version` to a `schema_version` table and a matching `schema_update_history` audit row (`tools/cassandra/cqlclient.go:53-72`; `common/persistence/sql/sqlplugin/mysql/admin.go:9-34`; `common/persistence/sql/sqlplugin/postgresql/admin.go:9-34`; `common/persistence/sql/sqlplugin/sqlite/admin.go:9-25`). The on-disk `DataBlob` envelope carries an `EncodingType` enum (`common/persistence/data_blob.go:10-21`), which acts as the serialization-format version. The mutable-state protobuf carries `VersionHistory` (legacy event-id vector, `common/persistence/versionhistory/version_history.go`) and the newer `TransitionHistory []*VersionedTransition` plus `InitialVersionedTransition` / `LastUpdateVersionedTransition` on each HSM node (`api/persistence/v1/executions.pb.go:260-335`, `api/persistence/v1/hsm.pb.go:27-58`). Task-queue user data is explicitly wrapped in `VersionedTaskQueueUserData{Data, Version}` (`api/persistence/v1/task_queues.pb.go:715-722`) for optimistic concurrency. The Cassandra-style `serialized_event_batch` UDT has a `version int` field on the row itself (`schema/cassandra/temporal/versioned/v1.0/schema.cql:1-5`).

2. **What happens when old state is loaded?**
   - **Database**: `VerifyCompatibleVersion` enforces DB-version ≥ code-version at startup (`common/persistence/schema/version.go:21-28`) — old databases (DB ≥ code) are explicitly accepted to support a code rollback; new databases that the code can't read (DB > code) are rejected.
   - **DataBlob**: decode dispatches on `EncodingType` and only accepts `PROTO3` or `JSON` (`common/persistence/serialization/codec.go:104-121`). Unknown encodings yield a terminal `UnknownEncodingTypeError` and the task is DLQ'd (`common/persistence/serialization/serializer.go:330`). Within an accepted encoding, protobuf's `unknownFields` plus the explicit `DiscardUnknown: true` options in `DeserializeStrippedEvents` and `JSONPBEncoder.DecodeSlice` (`common/persistence/serialization/serializer.go:227-233`; `common/codec/jsonpb.go:128-132`) preserve unrecognized fields without failing. Then a handful of fields are explicitly back-filled with default values if missing (`common/persistence/serialization/serializer.go:386-470`).
   - **Mutable state**: `LastVersionedTransition` returns nil when transition history is disabled (`common/persistence/transitionhistory/transition_history.go:34-42`), and `StalenessCheck` raises typed `ErrStaleState` / `ErrStaleReference` errors with the version delta (`common/persistence/transitionhistory/transition_history.go:81-125`). The legacy `VersionHistory` API refuses to add items with a lower version or event id (`common/persistence/versionhistory/version_history.go:66-90`).
   - **Elasticsearch**: unknown search-attribute names are silently dropped (`common/persistence/visibility/store/elasticsearch/visibility_store.go:1061-1066`).

3. **Are migrations automatic or manual?**
   - **Database migrations are manual, scripted, and one-way**. Operators invoke `temporal-sql-tool setup-schema` / `update-schema` (`tools/sql/handler.go:19-57`) or the equivalent Cassandra tool (`cmd/tools/cassandra/main.go`). Each migration is a directory under `schema/<db>/v<n>/<table>/versioned/v<x.y>/` containing a `manifest.json` (`CurrVersion`, `MinCompatibleVersion`, `Description`, `SchemaUpdateCqlFiles`) and idempotent DDL files (`schema/cassandra/temporal/versioned/v1.13/manifest.json`; `schema/mysql/v8/temporal/versioned/v1.19/manifest.json`). `UpdateTask.Run` reads the current DB version, computes the change set, applies it, and writes the new version (`tools/common/schema/updatetask.go:77-167`). The tool also supports a dry-run on a separate database (`tools/common/schema/updatetask.go:420-427`). There is no auto-migration at server startup.
   - **In-protobuf field evolution is automatic** by virtue of proto3's wire-format tolerance, with manual, per-call-site `if-nil-populate` patches in the serializer (`common/persistence/serialization/serializer.go:386-470`).

4. **Are JSON blobs versioned?**
   - Indirectly: every `commonpb.DataBlob` carries `EncodingType` (`common/persistence/data_blob.go:10-21`), which records whether the body is proto3 or JSON. Inside the body, the protobuf message itself carries no per-blob `schemaVersion` string — schema evolution relies on proto3's wire format and the explicit `DiscardUnknown: true` flags in `DeserializeStrippedEvents` (`common/persistence/serialization/serializer.go:227-233`) and `DecodeSlice` (`common/codec/jsonpb.go:128-132`). Queue metadata is hard-coded to JSON (`common/persistence/serialization/serializer.go:566-567`) with a `TODO` comment to migrate to proto3, which itself is a documented migration debt. The `TESTSPEC_TEST_DATA_ENCODING` env var (`common/persistence/serialization/codec.go:24-39`) lets tests round-trip via JSON, and `ReencodeEventBlobsAsProto3` (`common/persistence/serialization/serializer.go:682-700`) re-encodes non-proto3 event blobs on read paths (`service/history/api/getworkflowexecutionrawhistoryv2/api.go:142`).

5. **Can old checkpoints/events still be read?**
   - For `proto3` blobs in a `DataBlob`: yes, provided the schema change was additive and the `EncodingType` is still recognized. The `StrippedHistoryEvent` decode test demonstrates that an unknown payload decodes lossily without error (`common/persistence/serialization/serializer_test.go:204-243`). The risk surface is **structural breaks** (renamed/removed fields, changed oneof members) which are not handled — they will surface as a proto unmarshal error or a missing-required-field error from the ad-hoc nil-checks.
   - For the database: the `MinCompatibleVersion` field of the manifest and the `VerifyCompatibleVersion` start-up check (`common/persistence/schema/version.go:21-28`) make forward-reads safe across at least one major version bump, but there is no automated test that loads yesterday's schema and exercises today's code against it.
   - For the HSM transition history: `TransitionHistory` is opt-in (the proto field comment notes "If transition history is disabled, the transition_count field will be 0", `api/persistence/v1/hsm.pb.go:36-38`) and `LastVersionedTransition` returns nil in that case (`common/persistence/transitionhistory/transition_history.go:34-42`), so old mutable state without transition history continues to function.
   - For `VersionHistory` event-id vectors: `FindLCAVersionHistoryItem` raises `serviceerror.NewInternal("version history is malformed. No joint point found.")` if two histories share no common ancestor (`common/persistence/versionhistory/version_history.go:134`), so a rebalance across a broken-history boundary will surface as a hard error.

## Architectural Decisions

- **One-way forward migrations with rollback tolerance.** The DB schema tool is intentionally designed to apply changes in one direction only; rolling back code is supported because the server explicitly accepts a DB-version ≥ code-version relationship (`common/persistence/schema/version.go:21-28`). This is a deliberate simplification: down-migrations are an operator problem, not a runtime problem.
- **Manifests with explicit `MinCompatibleVersion`.** Every migration manifest must declare the oldest code version that can read what the migration produces (`schema/cassandra/temporal/versioned/v1.13/manifest.json:3`, `tools/common/schema/updatetask.go:282-298`). This puts the compatibility contract inside the migration itself rather than in a separate compatibility matrix.
- **MD5 of the manifest as a content fingerprint.** Each manifest is hashed and written to `schema_update_history` so that operators can detect when a manifest has been edited after a migration was applied (`tools/common/schema/updatetask.go:301-304`). This guards against silent drift between intended and applied schema changes.
- **Idempotent DDL with allowed-error swallowing.** Both `execStmts` (Cassandra / generic) and the SQL plugin tolerate `already exists` and `not found` errors so that a partially-completed previous migration can be re-run safely (`tools/common/schema/updatetask.go:142-147`). This is a pragmatic answer to migration durability.
- **Statement whitelist.** Only `CREATE`, `ALTER`, `INSERT`, `DROP`, `DO` are accepted in migration files (`tools/common/schema/updatetask.go:62-65, 251-265`), preventing migrations from smuggling in `DELETE`/`UPDATE` that would mutate rows.
- **`DataBlob` envelope with explicit encoding type.** All persisted records are wrapped in a `commonpb.DataBlob{Data, EncodingType}` (`common/persistence/data_blob.go:10-21`), which gives the codec a single place to dispatch on format and a natural extension point for new encodings.
- **Discard-unknown as the primary forward-compat mechanism.** `StrippedHistoryEvent` decode and the JSON slice decoder both use `DiscardUnknown: true` (`common/persistence/serialization/serializer.go:227-233`; `common/codec/jsonpb.go:128-132`), trading type-safety for forward compatibility. There is no equivalent field-level mechanism in the SQL side.
- **Two parallel versions of versioning for mutable state.** The legacy `VersionHistory` (event-id vector per branch) and the newer `VersionedTransition` + `TransitionHistory` (HSM-style vector) coexist (`api/persistence/v1/hsm.pb.go:457-510`; `common/persistence/versionhistory/version_history.go`). The new design adds explicit `Initial` / `LastUpdate` per HSM node and deterministic staleness ranges (`common/persistence/transitionhistory/transition_history.go:81-125`). The previous-transition-history fields document that history may be disabled and re-enabled (`api/persistence/v1/executions.pb.go:332-335`).
- **`VersionedTaskQueueUserData` envelope for optimistic concurrency.** Task-queue user data is wrapped with a `Version int64` field that the update path requires callers to pass in (`api/persistence/v1/task_queues.pb.go:715-722`; `common/persistence/data_interfaces.go:1197-1203`). This is application-level optimistic concurrency, not a schema migration tool.
- **Per-plugin admin DB interface.** Each SQL plugin exposes the same `ReadSchemaVersion` / `UpdateSchemaVersion` / `WriteSchemaUpdateLog` / `VerifyVersion` contract (`common/persistence/sql/sqlplugin/interfaces.go:79-114`), making the upgrade tool database-agnostic.
- **Dry-run sandbox.** `setupDryRunDatabase` materializes a fresh DB to apply migrations against before touching the real one (`tools/common/schema/updatetask.go:420-427`), reducing operator risk.
- **Hand-rolled nil-map backfills in `serializerImpl`.** Rather than introducing a migration framework for added map fields, the deserializer patches missing collections inline (`common/persistence/serialization/serializer.go:386-470`). This is a deliberate trade-off: the upgrade surface is the proto schema (additive), and the deserializer is the compatibility shim.
- **Elasticsearch forward-compat via silent-drop.** `sadefs.ErrInvalidName` is explicitly swallowed so that older indexes with extra fields don't break newer clients (`common/persistence/visibility/store/elasticsearch/visibility_store.go:1061-1066`). The Elasticsearch upgrade path itself is a shell script per version, not a Go-driven tool (`schema/elasticsearch/visibility/versioned/v14/upgrade.sh:1-57`).

## Notable Patterns

- **Single-version-source-of-truth baked into the binary.** `Version` and `VisibilityVersion` are `const` strings in `version.go` files under each `schema/<db>/v<n>/` directory (`schema/cassandra/version.go:6`; `schema/mysql/v8/version.go:6-9`; `schema/sqlite/version.go:4-6`). The server reads them via `db.ExpectedVersion()` (`common/persistence/sql/sqlplugin/mysql/db.go:124-134`) and the Cassandra equivalent. This makes "what version of schema does this binary speak?" answerable by static analysis.
- **Embedded schemas via `go:embed`.** The whole `schema/` tree is shipped inside the binary via `//go:embed *` (`schema/embed.go:12-13`), so the upgrade tool can run without an external filesystem path.
- **`IsTerminalTaskError` on serialization errors.** A deserialization failure routes the task to the DLQ rather than retrying forever (`common/persistence/serialization/serializer.go:372`). This is a "fail loud, don't loop" pattern that protects the system from a stuck blob.
- **Manifest whitelist of statements.** Restricting migrations to `CREATE` / `ALTER` / `INSERT` / `DROP` / `DO` (`tools/common/schema/updatetask.go:62-65`) makes accidental data corruption through a schema migration script impossible.
- **Per-plugin `VersionedBlob` abstraction in SQL.** The `sqlplugin.VersionedBlob{Version, Data, DataEncoding}` struct (`common/persistence/sql/sqlplugin/interfaces.go:24-28`) is the in-process equivalent of the on-disk `DataBlob`. The pattern is repeated from the protocol boundary to the storage adapter.
- **DiscardUnknownProto utility.** A standalone helper at `common/util.go:682-684` performs unknown-field stripping for arbitrary proto messages, used outside the persistence layer (e.g., `common/util_test.go:278-339`).
- **Range-based staleness checks.** The newer `StalenessCheck` function operates on min/max transition-count ranges per version (`common/persistence/transitionhistory/transition_history.go:81-125`), which is more expressive than the legacy LCA search.
- **Two separate task-table layouts.** Older task categories (transfer / timer / visibility / replication) still have their own tables; newer categories use the generic `history_immediate_tasks` and `history_scheduled_tasks` tables (`common/persistence/sql/execution_util.go:697-775`). The dual paths are kept for backward compatibility.
- **Manual DDL with explicit semantics.** Even schema changes that *could* be done as defaults (e.g., `add_current_executions_start_time.sql`, `fix_data_encoding_column.sql`) ship as discrete versions so each migration has its own audit row.
- **Server-driven `VerifyCompatibleVersion` at startup.** The check runs both on the main store and on the visibility store (`common/persistence/sql/version_checker.go:13-26`), enforcing schema compatibility before any workload is accepted.

## Tradeoffs

- **No automatic rollback / down-migration.** `VerifyCompatibleVersion` rejects code > DB but accepts code < DB (`common/persistence/schema/version.go:21-28`). This means a botched upgrade that breaks the application cannot be undone by re-running the schema tool; the operator must restore a backup. There is no versioned "downgrade" manifest format.
- **Proto-only forward compatibility.** Renaming, splitting, or removing a field has no built-in migration path; the deserializer's `if-nil-populate` patches are per-call-site, and there is no central registry that ties a "before" shape to an "after" shape. A breaking field change relies on protobuf's wire-format breakage as the safety net.
- **Hand-coded nil initialization in `serializerImpl` is fragile.** Each new optional collection requires a new branch in `serializerImpl` (`common/persistence/serialization/serializer.go:386-470`), which is easy to forget and hard to audit. A migration registry or a code-generated "if nil, init" would be more robust.
- **Two parallel versioning schemes for mutable state.** `VersionHistory` and `VersionedTransition` both exist (`common/persistence/versionhistory/version_history.go`; `api/persistence/v1/hsm.pb.go:457-510`); this is a known transition cost paid for backwards compatibility. `PreviousTransitionHistory` / `LastTransitionHistoryBreakPoint` are kept around as scaffolding for the eventual cut-over (`api/persistence/v1/executions.pb.go:332-335`).
- **Elasticsearch migration is a shell script per version.** Unlike Cassandra/SQL, there is no Go-driven migration loop for ES (`schema/elasticsearch/visibility/versioned/v14/upgrade.sh:1-57`). ES index mappings are re-applied via curl, which means compatibility is enforced by the operator running the script, not by the server.
- **`Manifest` MD5 is weak.** md5 is acknowledged in the source as "weak cryptographic primitive" and used only as a content fingerprint (`tools/common/schema/updatetask.go:5-7`). A determined attacker could engineer a collision, but the actual threat is operator confusion, which md5 is fine for.
- **`DiscardUnknown` is silent.** When the JSON decoder drops fields, there is no telemetry to surface it; an operator upgrading a server will not see "this old JSON had fields X, Y, Z that the new schema doesn't know about". Visibility into lost fields is effectively zero.
- **No stored old-state fixtures.** Searches for `*fixture*` and `oldVersion*` produced no committed artifacts; tests construct fixtures in-memory (`common/persistence/serialization/serializer_test.go:204-243`). This means "will today's server load yesterday's blob?" is answered only by the protobuf wire-format tolerance and the few hand-rolled nil-checks.
- **Per-version sqlite bundle version (v3)** The whole sqlite schema lives under `schema/sqlite/v3/`, separate from MySQL/Postgres's v8/v12 versioning. This is the result of a historical fork and is not actively migrated; both trees are maintained in parallel (`schema/sqlite/v3/temporal/versioned/` vs. `schema/mysql/v8/temporal/versioned/`).

## Failure Modes / Edge Cases

- **Code rollback after schema upgrade.** `VerifyCompatibleVersion` accepts a DB > code relationship (`common/persistence/schema/version.go:21-28`). If the new server reads records written by an older version of itself, it relies on proto3 wire-format tolerance and the `serializerImpl` nil-checks (`common/persistence/serialization/serializer.go:386-470`). There is no test that exercises this path explicitly.
- **Old proto record with a renamed oneof.** Proto3 oneof members are not backward-compatible in the usual sense: a member renamed in the proto produces a different wire tag, and the old tag becomes unknown. The decoder will silently drop it (`common/persistence/serialization/serializer.go:227-233`) unless the deserializer explicitly handles it.
- **Unknown encoding on a `DataBlob`.** A blob with `EncodingType == ENCODING_TYPE_UNSPECIFIED` (e.g., from an old writer that used `NewDataBlob` with an unrecognized string, `common/persistence/data_blob.go:11-15`) hits `NewUnknownEncodingTypeError` and the task is DLQ'd (`common/persistence/serialization/serializer.go:330`). Recovery requires manual operator action.
- **Dry-run migration against a real DB.** If an operator misconfigures `--schema-name` vs `--schema-dir`, the `dbschemas.Assets()` path can be bypassed in favor of a filesystem path (`tools/common/schema/updatetask.go:175-181`). A typo could result in `task.setupDryRunDatabase` running against the production DB only if `IsDryRun` is misconfigured; the code is careful to require the flag explicitly.
- **`QueueMetadata` forced to JSON forever.** The TODO at `common/persistence/serialization/serializer.go:565` notes that `QueueMetadata` is hard-coded to `JSON`, with a deferred plan to migrate. This is a known lock-in that requires a separate upgrade for legacy queues.
- **Reading old `ShardInfo` without `ReplicationDlqAckLevel` or `QueueStates`.** Both nil-map fallbacks exist in `ShardInfoFromBlob` (`common/persistence/serialization/serializer.go:386-404`), but a field added later without an explicit nil-check here will surface as a nil-map panic downstream.
- **Mismatch on `MinCompatibleVersion` in a manifest.** `readManifest` rejects an empty `MinCompatibleVersion` (`tools/common/schema/updatetask.go:288-294`), but there is no semantic check that the declared `MinCompatibleVersion` is actually achievable — a manifest with `MinCompatibleVersion = "0.0"` would silently disable the check.
- **`DiscardUnknown` is silent and could mask schema corruption.** A version that accidentally removes a field and writes `unknownFields` to the wire will be silently dropped on read by clients that haven't updated, hiding the bug from telemetry.
- **`VersionHistory` LCA failure.** If two histories share no ancestor (e.g., after a broken replication cutover), `FindLCAVersionHistoryItem` raises `serviceerror.NewInternal("version history is malformed. No joint point found.")` (`common/persistence/versionhistory/version_history.go:134`). This is a hard failure rather than a recoverable state.
- **`TransitionHistory` with `transition_count = 0`.** When the history is disabled, the `transition_count` is `0` and cannot be used to uniquely identify a node (`api/persistence/v1/hsm.pb.go:36-38`). Code that implicitly assumes `transition_count > 0` will misclassify state. `LastVersionedTransition` returning nil is the safe path (`common/persistence/transitionhistory/transition_history.go:34-42`), but consumers must remember to check.

## Future Considerations

- **Promote nil-init patches to a migration registry.** Replace the scattered `if result.X == nil { result.X = ... }` in `serializerImpl` (`common/persistence/serialization/serializer.go:386-470`) with a declarative upcaster layer keyed on proto field tags. This would make field evolution auditable and prevent the next contributor from forgetting the patch.
- **Add stored old-state fixtures.** Check in a small set of representative old-version `DataBlob`s (one per major schema milestone) under `tests/fixtures/` and write a fixture-driven decoder test, similar to what the `agent-framework` source documents in its `1.0.0` JSON tests. Today the closest analogue is the inline `1.0` blob in `TestDeserializeStrippedEvents.ProtoDiscardUnknownFields` (`common/persistence/serialization/serializer_test.go:204-243`), but there is no committed file.
- **Emit telemetry for `DiscardUnknown` drops.** Today the JSON and stripped-event decoders silently drop unknown fields (`common/persistence/serialization/serializer.go:227-233`; `common/codec/jsonpb.go:128-132`). A counter for dropped fields, partitioned by message type, would let operators notice accidental schema drift.
- **Defer schema_version table creation to the upgrade tool, not the per-DB code.** Each SQL plugin already defines `schema_version` and `schema_update_history` DDL inline (`common/persistence/sql/sqlplugin/mysql/admin.go:18-34`), and the Cassandra tool does the same (`tools/cassandra/cqlclient.go:59-72`). Hoisting this into the upgrade tool's manifest would centralize the meta-schema.
- **Finish the `QueueMetadata` JSON → proto3 migration.** The TODO at `common/persistence/serialization/serializer.go:565` is a long-standing debt. Once removed, the `ENCODING_TYPE_JSON` branch in `Encode`/`Decode` (`common/persistence/serialization/codec.go:81-89, 110-111`) becomes exclusively test-environment.
- **Unify the two mutable-state versioning schemes.** The transition from `VersionHistory` to `TransitionHistory` should have a clear sunset date; today both live in the same `WorkflowExecutionInfo` proto (`api/persistence/v1/executions.pb.go:260-335`), which keeps the wire format large and the code path duplicated.
- **Drive Elasticsearch upgrades from Go.** Replace the per-version `upgrade.sh` files (`schema/elasticsearch/visibility/versioned/v14/upgrade.sh:1-57`) with a Go tool similar to the SQL upgrade tool, so compatibility is enforced uniformly across stores.
- **Use a stronger hash than MD5 for manifest content fingerprinting.** Although the source correctly notes md5 is acceptable for content fingerprinting (`tools/common/schema/updatetask.go:5-7`), SHA-256 is the modern default and costs nothing here.
- **Cross-plugin compatibility tests.** Write a test that loads the same `DataBlob` through MySQL, Postgres, Cassandra, and SQLite deserializers and asserts they all produce identical proto values. Today each plugin has its own test suite (`common/persistence/tests/mysql_test.go`, etc.) with no shared fixture.
- **Document `MinCompatibleVersion` semantics in the manifest schema.** The field is required by `readManifest` (`tools/common/schema/updatetask.go:288-294`), but there is no machine-checkable guidance for what value to put. A `docs/migrations.md` (or equivalent) would help contributors pick the right value.

## Questions / Gaps

- **Where are stored old-state fixtures?** Searches for `*fixture*` and `oldVersion*` inside `studies/agent-harness-study/sources/temporal` returned no committed fixtures. Tests construct inputs in-memory (`common/persistence/serialization/serializer_test.go:204-243`). The closest analogue is the embedded `manifest.json` files under `schema/<db>/v<n>/<table>/versioned/v<x.y>/`, but those describe the DDL, not the runtime blob shape.
- **What is the policy for renaming a proto field?** No search hit on `rename`, `deprecate field`, or `reserved` in the persistence proto files. There is no documented "what to do when a field has to be removed" runbook.
- **How is the Elasticsearch `versioned/v0..v14` chain intended to migrate?** The per-version `upgrade.sh` scripts (`schema/elasticsearch/visibility/versioned/v14/upgrade.sh:1-57`) reference env vars (`ES_VERSION`, `ES_VIS_INDEX_V1`) but there is no top-level orchestration. An operator running `temporal-elasticsearch-tool update-schema` would need to know the version history by hand.
- **Is the `SchemaVersion` constant enforced against the DB at compile time?** The constant lives in `schema/cassandra/version.go` and `schema/mysql/v8/version.go` etc., but there is no test that cross-checks the binary's expected version against the version declared in the highest manifest. A mismatch would surface only at runtime as a `VerifyCompatibleVersion` failure.
- **What happens to a `ShardInfo` blob that has `QueueStates` for a category that has since been deleted?** `ShardInfoFromBlob` initializes missing collections but does not prune stale categories (`common/persistence/serialization/serializer.go:386-404`). A category that was removed in a later release will continue to live in the shard blob until it is overwritten.
- **Is there a documented "compatibility matrix" of code × schema?** No file matching `compat*.md`, `matrix*.md`, or similar was found in the repo. The contract is implicit in `MinCompatibleVersion` of each manifest and the rollback tolerance of `VerifyCompatibleVersion`.
- **What is the operational procedure to recover from a partial migration?** `execStmts` handles `already exists` / `not found` (`tools/common/schema/updatetask.go:142-147`), but a DDL failure that is neither of those (e.g., a timeout mid-statement) will leave the DB in an unknown state with no documented recovery path.
- **How does `QueueMetadata` (locked to JSON, `common/persistence/serialization/serializer.go:565-567`) interact with a future code path that reads it as proto3?** The current code path is JSON-only; a future migration would require either re-decoding and re-encoding all queues or a one-time on-read fallback.

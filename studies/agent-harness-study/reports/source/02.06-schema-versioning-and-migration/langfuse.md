## Summary

Langfuse uses a layered schema-versioning strategy that combines (1) timestamped Prisma migrations for Postgres with forward-only `migration.sql` files (`packages/shared/prisma/migrations/`, 409 entries from `20230518191501_init` through `20260619120000_add_blob_storage_export_tuning`), (2) numbered ClickHouse migrations under `clustered/` and `unclustered/` paired with `.up.sql`/`.down.sql` files (34 pairs as of `0034_add_long_string_value_to_scores`) executed by `golang-migrate` (`packages/shared/clickhouse/scripts/up.sh:54-71`), (3) a recoverable `BackgroundMigrationManager` (`worker/src/backgroundMigrations/backgroundMigrationManager.ts:6-185`) that backs long-running data moves with a `background_migrations` Prisma table (`packages/shared/prisma/schema.prisma:279-292`) plus a persisted `state` Json column, (4) Zod-based discriminated-union schemas that branch on a `version: z.literal(2)` discriminator for legacy JSON blobs (`packages/shared/src/features/evals/outputDefinition.ts:13-134`), (5) optional fields with `.optional()` plus inline `// backward compatibility` comments on every queue payload (`packages/shared/src/server/queues.ts:28-33,73,118-121`), and (6) explicit "legacy / dual / events_only" write-mode environment flags that gate dual-write and dual-read paths through the V4 ingestion migration (`worker/src/env.ts:459-540`, `packages/shared/src/server/repositories/events.ts:1115-1140`).

There is no central `schema_version` envelope on persisted blobs — versioning is expressed through (a) the migration history tables themselves, (b) the migration-script ordering by sortable prefix (the README explicitly requires sortable, date-prefixed names: `worker/src/backgroundMigrations/README.md:25`), (c) per-feature literal-version discriminators on the JSON side, and (d) feature-flag-gated "phased" cutovers. The strongest versioning artifacts are the Eval output-definition `v2` schema (`packages/shared/src/features/evals/outputDefinition.ts:72-134`), the dashboard-widget `min_version` column (`packages/shared/prisma/schema.prisma:1524`) paired with migration `20260224000000_add_widget_version/migration.sql:1`, and the `eventTypes.OBSERVATION_CREATE`/`OBSERVATION_UPDATE` legacy ingestion branches retained alongside `eventCreateEvent`/`spanCreateEvent` (`packages/shared/src/server/ingestion/types.ts:259-279,282-356,689-718`). JSON blobs are not versioned in a unified way; instead each consumer (eval output, score config, prompts, blob storage export tuning, automations, dashboard widgets) owns its own backward-compatibility rule and resolver. Old-state fixtures are not committed as inputs to round-trip tests; compatibility is exercised by Zod `safeParse` regression tests (`packages/shared/src/server/queues.test.ts:25-47`, `web/src/__tests__/server/dashboard-widget-version.servertest.ts:213-331`) and a single ClickHouse version-band compatibility test (`packages/shared/src/server/clickhouse/compatibility.test.ts:17-118`).

## Rating

**6 / 10** — Mature infrastructure for *database schema* evolution (Prisma + ClickHouse migrations, recoverable background migrations, lock-protected runs, dual-write / phased cutover flags, V4 read-routing wrappers) and a clear versioning pattern for at least one JSON blob family (Eval output `version: z.literal(2)` with a discriminator resolver). Each Prisma migration is forward-only and dated, migrations are applied automatically on container start (`web/entrypoint.sh:90-115`), background migrations are recoverable, lock-protected, and resumable from `state` Json (`worker/src/backgroundMigrations/backgroundMigrationManager.ts:38-162`). However: (a) there is no project-wide `schema_version` envelope on JSON blobs — each feature reimplements backward-compat via `optional()` and `.transform()`; (b) no committed fixtures of "old" state are used in round-trip tests, so the test suite does not pin specific versions; (c) at least one reversible migration must be repaired manually via `_prisma_migrations` cleanup (`packages/shared/scripts/cleanup.sql:1-7`), suggesting non-idempotent migrations have shipped; (d) several migrations use raw destructive DDL (column DROP, table DROP) without preserving prior data; (e) the `dev-tables.sh` events-core/events-full definitions explicitly bypass the migration system because of ClickHouse feature gates (`packages/shared/clickhouse/scripts/dev-tables.sh:15-26`), and (f) failure modes include: missing background-migration "upgrade stops" documentation beyond the README template, no automatic downgrade path, and JSON compatibility relies on Zod `passthrough`/`.optional()` discipline rather than tested fixtures. Compared with the `agent-framework` study's documented `schemaVersion` JSON envelope + `$type` discriminator + round-trip tests, Langfuse's model is more *operationally* robust for DB changes but less *explicitly* versioned for application JSON payloads.

## Answers to Dimension Questions

1. **Does persisted state carry a version?**
   - Yes for the dashboard-widget table (`min_version INTEGER NOT NULL DEFAULT 1`, `packages/shared/prisma/schema.prisma:1524`, migration `packages/shared/prisma/migrations/20260224000000_add_widget_version/migration.sql:1`) and for Eval output-definition JSON blobs (`version: z.literal(2)`, `packages/shared/src/features/evals/outputDefinition.ts:73,83,94`). No for the rest of the JSON column space: `score_configs`, `automations`, prompt content, blob-storage export field groups, dataset items, etc. — each consumes Zod `optional()` and `.transform()` instead.
   - Migration history itself is versioned by sortable timestamp prefix (`20230518191501_init`, `…20260619120000_add_blob_storage_export_tuning`); Prisma's `_prisma_migrations` table records applied versions.
   - Background migrations are versioned by hard-coded UUID `backgroundMigrationId` referenced from `INSERT INTO background_migrations` (e.g. `worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:12`, `migrateEventLogToBlobStorageRefTable.ts:19`).
   - View-declaration API versions are explicit (`viewVersions = z.enum(["v1","v2"])`, `packages/shared/src/features/query/types.ts:101-102`) and per-widget `min_version` is the discriminator (`packages/shared/src/features/query/dataModel.ts:1393-1410`).

2. **What happens when old state is loaded?**
   - DB schemas: Prisma `migrate deploy` is forward-only (`web/entrypoint.sh:94`); migrations are additive (mostly `ALTER TABLE … ADD COLUMN … DEFAULT`, e.g. `packages/shared/prisma/migrations/20260224000000_add_widget_version/migration.sql:1`). Some migrations explicitly drop columns and tables (`packages/shared/prisma/migrations/20230523082455_rename_metrics_to_gradings/migration.sql:14` drops `metrics`, `…20230518193521_changes/migration.sql:12` drops `parentObservationId`), so old data is *not* preserved through DDL drops. At least one migration is known to be non-idempotent and is patched by hand-editing `_prisma_migrations` (`packages/shared/scripts/cleanup.sql:1-7`).
   - Ingestion events: legacy `OBSERVATION_CREATE`/`OBSERVATION_UPDATE` types remain in the discriminated union (`packages/shared/src/server/ingestion/types.ts:689-718`) and are routed to the same processing path as the new types. Legacy `LegacySpanPostSchema`/`LegacyGenerationsCreateSchema`/`LegacyObservationBody` (`packages/shared/src/server/ingestion/types.ts:282-383`) keep accepting the old field names alongside the new ones.
   - Usage payloads: `usage = MixedUsage.nullish().transform(...).pipe(Usage.nullish())` (`packages/shared/src/server/ingestion/types.ts:45-69`) rewrites the old `{ promptTokens, completionTokens, totalTokens }` shape into the new `{ input, output, total, unit }` shape, and `UsageDetails = z.union([OpenAICompletion, OpenAIResponse, Raw])` (`packages/shared/src/server/ingestion/types.ts:217-223`) accepts three generations of usage payloads.
   - Eval output-definition JSON: `PersistedEvalOutputDefinitionSchema = z.union([LegacyEvalOutputDefinitionSchema, NumericEvalOutputDefinitionV2Schema, BooleanEvalOutputDefinitionV2Schema, CategoricalEvalOutputDefinitionV2Schema])` (`packages/shared/src/features/evals/outputDefinition.ts:129-134`); `resolvePersistedEvalOutputDefinition` (`packages/shared/src/features/evals/outputDefinition.ts:182-212`) branches on `if (!("version" in outputDefinition))` to normalize legacy rows to the v2 runtime shape.
   - Queue payloads: in-flight jobs that were enqueued before a payload field was added still deserialize because the new fields are `.optional()` (`packages/shared/src/server/queues.ts:28-33,73`). There is no automatic migration of in-flight messages — the consumer must tolerate both shapes.
   - Dashboard widgets: `minVersion` defaults to 1 for legacy rows and the `requiresV2` helper picks the right view declaration per row (`web/src/__tests__/server/dashboard-widget-version.servertest.ts:101-166`).
   - S3 event files: `parseEventKey` accepts both legacy raw-id and newer `safeBlobKeySegment`-hashed shapes (`packages/shared/src/server/ingestion/eventBucketPath.ts:16-23,52-69`).

3. **Are migrations automatic or manual?**
   - Automatic on container start: `web/entrypoint.sh:90-95` runs `prisma migrate deploy`; `web/entrypoint.sh:108-115` runs ClickHouse `up.sh`. Both are guarded by `LANGFUSE_AUTO_POSTGRES_MIGRATION_DISABLED` / `LANGFUSE_AUTO_CLICKHOUSE_MIGRATION_DISABLED` env flags.
   - Background migrations are also automatic: `BackgroundMigrationManager.run` (`worker/src/backgroundMigrations/backgroundMigrationManager.ts:38-165`) loops over pending rows in `background_migrations` ordered by `name`, acquires a serializable-transaction lock, heartbeats, runs, marks complete, repeats. They can be invoked manually with `npx ts-node src/backgroundMigrations/<script>.ts` (`worker/src/backgroundMigrations/README.md:9-16`).
   - Local dev migrations use `prisma migrate dev` (`packages/shared/package.json:75`) which generates a new migration after auto-applying the shadow DB diff.
   - The README explicitly calls out GitLab-style "upgrade stops" as a documentation requirement but does not auto-pause: `worker/src/backgroundMigrations/README.md:22-23` ("We must highlight in the changelog and potentially another page if the code relies on some background migration having finished.").

4. **Are JSON blobs versioned?**
   - Eval output-definition: yes, `version: z.literal(2)` (`packages/shared/src/features/evals/outputDefinition.ts:73,83,94`).
   - Dashboard widgets: stored `minVersion` column tracks the schema generation; the column is *not* inside the JSON blob but on the row.
   - Most other JSON blobs (`automations.headers`, prompt versions, score configs, blob-storage export field groups, dataset items) rely on Zod `.optional()` + `.transform()` for forward compat. No envelope `schema_version` key is present.
   - `bucketPrefix` on the IngestionEvent was added later than the original payload and was made optional specifically to consume legacy queue messages (`packages/shared/src/server/queues.ts:28-33`).

5. **Can old checkpoints/events still be read?**
   - Old Postgres rows: yes, because migrations are additive and defaulted (`min_version` default 1, added columns default to null/0). Rows dropped by `DROP COLUMN`/`DROP TABLE` migrations are gone and not recoverable through the application.
   - Old queue messages in flight: yes, by design (`.optional()` fields). There is no replay pipeline for the queue itself beyond BullMQ's default dead-letter retry (`packages/shared/src/server/redis/dlqRetryQueue.ts`).
   - Old ClickHouse rows: yes; the `events_proto` virtual table plus `events_core`/`events_full` physical tables co-exist (`packages/shared/src/server/clickhouse/schema.ts:1-33`), and the read layer branches on `LANGFUSE_MIGRATION_V4_WRITE_MODE` (`packages/shared/src/server/repositories/events.ts:1115-1140`).
   - Old S3 event files: yes; both raw-id and sanitized-prefix variants are read (`packages/shared/src/server/ingestion/eventBucketPath.ts:16-23,52-69,88-117`).
   - Old background-migration progress: yes, via the `state` Json column checkpoint (`worker/src/backgroundMigrations/backgroundMigrationManager.ts:17-36`).
   - Old Langfuse deployments upgrading to V4 events: must run a backfill background migration (`migrateTracesFromPostgresToClickhouse`, `migrateObservationsFromPostgresToClickhouse`, `migrateScoresFromPostgresToClickhouse`, `migrateDatasetRunItemsFromPostgresToClickhouse*`) — the legacy tables are read for as long as `LANGFUSE_MIGRATION_V4_WRITE_MODE !== "events_only"`.

## Architectural Decisions

- **Two-database migration systems**, each with its own tooling:
  - Postgres: Prisma Migrate (`prisma migrate dev` / `prisma migrate deploy`) — schema is the source of truth in `packages/shared/prisma/schema.prisma` (1841 lines), migration files are timestamped forward-only `migration.sql` (Prisma `migration_lock.toml` pins `provider = "postgresql"`).

## Notable Patterns

- **Co-existence of `legacy`/`dual`/`events_only` write modes** with hard validation of legal combinations (`worker/src/env.ts:459-540`) is the single most distinctive pattern. It avoids the classic big-bang migration failure by letting the same binary write to both schemas during cutover and toggle the read layer independently.
- **Migration history embedded in code**: `_prisma_migrations` is the canonical record; `migration_lock.toml` pins the dialect; `migration.sql` files are immutable. Background migrations are added via a *separate* Prisma migration that runs `INSERT INTO background_migrations (id, name, script, args)` (`packages/shared/prisma/migrations/20241024121500_add_generations_cost_backfill_background_migration/migration.sql:1`), which is the only way to enroll a background migration.
- **Lock-protected single-runner background migration**: serializable transaction, `workerId = randomUUID()`, `lockedAt` heartbeat, 60-second lock TTL (`worker/src/backgroundMigrations/backgroundMigrationManager.ts:7-97`). A worker crash releases the lock after 60s, the next worker resumes from `state` Json.
- **Dual Zod schema parsing for ingestion events**: every event type has a `LegacyXxxSchema` (kept for compat) and a new typed variant; the discriminated union `ingestionEvent = z.discriminatedUnion("type", […])` includes both (`packages/shared/src/server/ingestion/types.ts:699-719`).
- **Two-formula S3 key prefix**: `buildEventBucketPrefix` (sanitize fresh) vs `rawEventBucketPrefix` (preserve canonical). The comment at `packages/shared/src/server/ingestion/eventBucketPath.ts:102-106` explicitly says "The point isn't legacy compatibility — it's that the caller has already observed the literal S3-side string and we must reproduce it byte-for-byte."
- **Migration-state Json column for resumable batch loops**: `state: { maxDate }` or `state: { offset }` is rewritten after each batch so a worker crash mid-migration resumes at the right cursor (`worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:67-83,129-136`, `worker/src/backgroundMigrations/migrateEventLogToBlobStorageRefTable.ts:79-146`).
- **Cleanup script for known-bad migrations**: `packages/shared/scripts/cleanup.sql:1-7` is run before `prisma migrate deploy` in `web/entrypoint.sh:91` to delete rows in `_prisma_migrations` for migrations that have shipped in non-idempotent form.
- **Discriminated union over webhook envelopes**: `WebhookOutboundEnvelopeSchema = z.discriminatedUnion("type", [promptVersionWebhookEnvelopeSchema, MonitorWebhookQueueEventSchema])` (`packages/shared/src/server/queues.ts:247-264`) — each new envelope type is a new union member, no payload-shape mutation.
- **ClickHouse compatibility is data, not code**: a single array `CLICKHOUSE_COMPATIBILITY_RULES` controls which ClickHouse-version bands get which setting overrides (`packages/shared/src/server/clickhouse/compatibility.ts:59-69`).

## Tradeoffs

- **Two migration systems** (Prisma for Postgres, golang-migrate for ClickHouse) double the operational surface area (two command runners, two histories, two rollback paths) but each tool matches its DB's semantics best (declarative schema diff for relational, raw SQL for ClickHouse DDL).
- **Forward-only Prisma migrations + manual cleanup scripts** trade idempotency for simplicity; when a non-idempotent migration ships, the workaround is a hand-written `DELETE FROM _prisma_migrations` row rather than rewriting the migration (`packages/shared/scripts/cleanup.sql:1-7`).
- **No global `schema_version` envelope on JSON blobs** keeps the wire format lean but pushes backward-compat into every consumer's Zod schema, which makes it easy to forget `.optional()` and accidentally tighten a contract.
- **Background migrations gated by a separate Prisma table** decouples "shape" from "data" migrations but creates a coupling: adding a new background migration requires a Prisma migration to `INSERT INTO background_migrations`, and the worker must be deployed before any data-touching code that depends on the migration running. The README acknowledges this as "upgrade stops" (`worker/src/backgroundMigrations/README.md:22-23`).
- **`dev-tables.sh` bypasses the migration system** for `events_full`/`events_core` (`packages/shared/clickhouse/scripts/dev-tables.sh:15-26`) because they depend on ClickHouse ≥24.5/25.x features; this is a deliberate exception that keeps older self-hosters unblocked but creates a deployment procedure that is not pure migration-driven.
- **Destructive DDL in early migrations** (`DROP COLUMN`, `DROP TABLE` in `20230518193521_changes/migration.sql:12`, `20230523082455_rename_metrics_to_gradings/migration.sql:14`, `20230530204241_auth_api_ui/migration.sql:46-58`) is acceptable in the bootstrap era but illustrates that pre-1.0 schemas were not held to a preservation contract. Newer migrations are additive and defaulted (`packages/shared/prisma/migrations/20260224000000_add_widget_version/migration.sql:1`).
- **Hard-coded background-migration IDs** (`worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:12`, `…migrateEventLogToBlobStorageRefTable.ts:19`) tie the script to a specific row in `background_migrations` and make renaming the migration script file risky — the SQL must be updated atomically.

## Failure Modes / Edge Cases

- **In-flight BullMQ jobs with old payload shapes** are accepted because every added field is `.optional()` (`packages/shared/src/server/queues.ts:28-33`). If a future change ever removes the `.optional()`, in-flight jobs will be moved to the dead-letter queue and need manual replay (`packages/shared/src/server/redis/dlqRetryQueue.ts`).
- **ClickHouse version mismatch**: if a self-hoster runs on ClickHouse < 25.x, `enable_full_text_index = 1` (`packages/shared/shared/clickhouse/scripts/dev-tables.sh:281`) will fail. `dev-tables.sh` is a documented escape hatch but is not part of `migrate deploy`; running the production migration stack without the dev tables script leaves `events_core`/`events_full` undefined.
- **Pre-V4 deployments upgrading without backfill**: switching `LANGFUSE_MIGRATION_V4_WRITE_MODE=events_only` before the backfill background migrations complete (`migrateTracesFromPostgresToClickhouse`, `migrateObservationsFromPostgresToClickhouse`, etc.) breaks historical-trace queries. `validateV4Flags` enforces `LANGFUSE_MIGRATION_V4_ALLOW_PREVIEW_OPT_IN=true` for `events_only` (`worker/src/env.ts:538-540`), but there is no automatic check that the backfill migrations have actually finished.
- **Worker crash mid-background-migration**: the lock expires after 60s and the next worker resumes from `state` Json (`worker/src/backgroundMigrations/backgroundMigrationManager.ts:38-97`). If `state` itself is corrupted, the migration must be reset manually (the manager does not auto-repair).
- **Hard-coded background-migration ID drift**: if the SQL `INSERT` uses one UUID and the script `const backgroundMigrationId = "…"` uses another, `findUniqueOrThrow` will throw at runtime (`worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:69`).
- **Migrations that DROP tables/columns** in early bootstrap (`packages/shared/prisma/migrations/20230523082455_rename_metrics_to_gradings/migration.sql:14`, `…20230518193521_changes/migration.sql:12`) mean any tenant upgrading from those early revisions lost the dropped data permanently.
- **Incompatible OTel behavior + write mode** is caught by `validateV4Flags` (`worker/src/env.ts:519-540`), but only at boot — late changes to env vars do not re-validate.
- **Two-generation bucket key collision**: raw SDK-id segments may contain `/`, `safeBlobKeySegment` segments are hashed; both are valid for `STANDARD_EVENT_KEY_REGEX` (`packages/shared/src/server/ingestion/eventBucketPath.ts:9-17`), but a replay that re-sanitizes the canonical segment will point at a different file (`packages/shared/src/server/ingestion/eventBucketPath.ts:84-117`).
- **`safeParse` rejection on queue payloads**: queue handlers may receive `{ type: "bogus" }` (`packages/shared/src/server/queues.test.ts:35-39`); the DLQ catches those, but the test only covers the webhook envelope, not all queue payload schemas.
- **Backward-compat for score ingestion**: `validateAndInflateScore` (`packages/shared/src/server/ingestion/validateAndInflateScore.ts:23-81`) infers a `dataType` when absent and applies config-based overrides. If a config's `isArchived` is `true`, ingestion throws (`packages/shared/src/server/ingestion/validateAndInflateScore.ts:218-222`), but archived configs are silently ignored at read time, which can lead to "ghost" score rows pointing at archived configs in legacy exports.
- **Missing `_prisma_migrations` rows**: if a previous deploy applied a migration outside of `prisma migrate deploy` (e.g. via `prisma db execute`), the next `migrate deploy` will try to re-apply it. The cleanup script (`packages/shared/scripts/cleanup.sql:1-7`) handles only the named failures, not new ones.

## Future Considerations

- **Promote `events_full` / `events_core` from `dev-tables.sh` to a real migration** once the ClickHouse ≥25.x minimum can be assumed (`packages/shared/clickhouse/scripts/dev-tables.sh:15-26`). This eliminates the second deployment procedure and removes a class of "I forgot to run dev-tables.sh" outages.
- **Add a project-wide `schema_version` envelope on user-controlled JSON blobs** (prompts, automations, blob-storage export field groups, score configs) so that deserializers can branch on a single field instead of probing shape with Zod `.optional()` chains.
- **Commit a fixtures directory of historical payloads** (v1 trace events, v2 score configs, pre-`bucketPrefix` queue messages) and assert round-trip through the current deserializers; this would catch silent contract tightenings that current tests miss.
- **Move `web/entrypoint.sh` migration gating to a pre-deploy health check** so `events_only` cannot be enabled until the background migrations have actually completed (`worker/src/env.ts:519-540` only validates flag combinations, not migration completion).
- **Replace hard-coded `backgroundMigrationId` strings** with stable symbolic names (e.g. a `name` lookup) so renaming a script cannot desynchronize the SQL `INSERT` and the script's `const backgroundMigrationId = "…"` (`worker/src/backgroundMigrations/migrateTracesFromPostgresToClickhouse.ts:12`).
- **Add `prisma migrate diff` CI gating** to detect non-additive migrations (column drops, table drops) past a certain project age — the early bootstrap migrations set a precedent but should not be repeated on a mature schema.
- **Test the union path of `PersistedEvalOutputDefinitionSchema`** end-to-end against a fixture of legacy rows to lock the `resolvePersistedEvalOutputDefinition` (`packages/shared/src/features/evals/outputDefinition.ts:182-212`) behavior.
- **Document upgrade-stops**: surface in `CHANGELOG.md` which code paths require which background migration to have completed (the README mentions this obligation at `worker/src/backgroundMigrations/README.md:22-23` but there is no auto-pause or warning in code).

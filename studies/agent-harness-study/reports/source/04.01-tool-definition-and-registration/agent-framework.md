## Summary

Agent Framework ships a **dual-stack tool model** with a Python core (`agent_framework._tools`) and a .NET core built on top of the `Microsoft.Extensions.AI` `AIFunction`/`AITool` abstractions. Both stacks center on a single locally-executed tool class — `FunctionTool` (Python, `python/packages/core/agent_framework/_tools.py:243`) and `AIFunction` (.NET, used via `AIFunctionFactory.Create` and the typed `AIFunction` shape consumed by `FunctionInvokingChatClient`).

The Python `@tool` decorator (`python/packages/core/agent_framework/_tools.py:1148-1338`) is the canonical user-facing entry point. It inspects the wrapped callable with `inspect.signature` + `typing.get_type_hints`, builds a Pydantic model from the signature via `pydantic.create_model` (`_tools.py:484-514`), and stores a JSON-schema cache on the instance. It also accepts an explicit `schema=...` Pydantic model or raw JSON-schema dict, a `kind=` provider-agnostic classification (e.g., `SHELL_TOOL_KIND_VALUE = "shell"`, `_tools.py:92`), and an `approval_mode` literal (`"always_require"` / `"never_require"`).

The .NET stack delegates tool definition to `Microsoft.Extensions.AI.AIFunctionFactory.Create(...)` (.NET `dotnet/src/Microsoft.Agents.AI/Skills/Programmatic/AgentInlineSkillScript.cs:46`, `dotnet/src/Microsoft.Agents.AI/Skills/Programmatic/AgentInlineSkillResource.cs:55`, `dotnet/src/Microsoft.Agents.AI.LocalCodeAct/LocalExecuteCodeFunction.cs:69`, `dotnet/src/Microsoft.Agents.AI.Hyperlight/HyperlightExecuteCodeFunction.cs`). The framework's own role is to **host** and **catalog** these tools: keyed DI registration in `HostedAgentBuilder.WithAITool` / `WithAITools` / `WithAITool(factory:)` (`dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:87-140`), `function_invocation_configuration` knobs on the chat client (`python/packages/core/agent_framework/_tools.py:1344-1421`), and per-agent `tools=[...]` lists that are normalized by `normalize_tools(...)` (`python/packages/core/agent_framework/_tools.py:941-1004`).

Discovery is **explicit and per-invocation** — there is no global registry. Tools are listed on the agent at construction (`Agent(client=..., tools=[...])`, `python/packages/core/agent_framework/_agents.py:674, 861-905`), declared on `ChatOptions.Tools` (Python `ChatOptions` TypedDict field "tools"; .NET `ChatOptions.Tools`, `dotnet/src/Microsoft.Agents.AI.Abstractions/AIContext.cs:108`), or pushed by an `AIContextProvider` / `ContextProvider` (`.NET: AIContext.Tools`, `dotnet/src/Microsoft.Agents.AI.Abstractions/AIContext.cs:85-108`; Python: `context.extend_tools(self.source_id, self.get_tools())`, `python/packages/core/agent_framework/security.py:2050`). MCP server tools are discovered at run time and converted into `FunctionTool` instances (`python/packages/core/agent_framework/_mcp.py:333`, `dotnet/src/Microsoft.Agents.AI.Mcp/McpClientTaskExtensions.cs:34-60`).

Names are stable because they live on immutable instance attributes (`FunctionTool.name`, `FunctionTool.kind`), and duplicate names are caught with a dedicated helper (`_append_unique_tools`, `python/packages/core/agent_framework/_tools.py:901-929`) used both at agent construction and at per-run tool-merge time (`python/packages/core/agent_framework/_agents.py:1258-1281`). Multiple agents see different tool sets because the merged tool list is built fresh inside `_prepare_run_context` for every `run()` call (`_agents.py:1160-1339`), and the live list is also exposed to `FunctionInvocationContext` so tools can `add_tools` / `remove_tools` mid-run (`_middleware.py:287-360`).

## Rating

**8/10**

Rationale:

- **Strengths**
  - Class-based `FunctionTool` / `AIFunction` with introspection-driven JSON schema (Pydantic model from signature, `python/packages/core/agent_framework/_tools.py:484-514`).
  - Decorator + direct-instantiation both supported (`@tool` overload set, `python/packages/core/agent_framework/_tools.py:1147-1192`; .NET `AIFunctionFactory.Create` everywhere).
  - Strong duplicate-name handling (`_append_unique_tools`, `_tools.py:901-929`; `_raise_duplicate_tool_name`, `_tools.py:896-898`).
  - Normalization that flattens tool-collection wrappers (bundles, mapping-like toolboxes, base models), with tests (`_tools.py:941-1004`; `python/packages/core/tests/core/test_tools.py:1167-1295`).
  - MCP is a first-class, transport-pluggable, tool source: `MCPStdioTool` / `MCPStreamableHTTPTool` / `MCPWebsocketTool` (`_mcp.py:2312, 2494, 2734`), all yielding `FunctionTool` objects, with per-tool `allowed_tools`, `tool_name_prefix`, `additional_tool_argument_names`, and an `MCP_FRAMEWORK_DENYLIST` safety net (`_mcp.py:88-98, 425-432`).
  - Layered docstring flow / chat-client decorator stack (`FunctionInvocationLayer`, `_tools.py:2388-2873`; `_apply_get_response_docstrings`, `python/packages/core/agent_framework/_clients.py:978-999`).
  - DI-aware tool lifetime validation in .NET (`HostedAgentBuilderExtensions.ValidateToolLifetime`, `dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:146-156`) — refuses captive-dependency configurations up front.
  - Multiple agents see different tool sets: per-agent list (`Agent(tools=...)`), per-call override (`Agent.run(tools=...)`, `python/packages/core/agent_framework/_agents.py:861-905`), and mid-run progressive tool exposure (`FunctionInvocationContext.add_tools/remove_tools`, `_middleware.py:287-360`).
  - Tool-approval is a first-class cross-cutting concern (`approval_mode` on `FunctionTool`, `ToolApprovalMiddleware` in `_harness/_tool_approval.py`, `ToolApprovalAgent` decorator in .NET, and the auto-bypass layer `NonApprovalRequiredFunctionBypassingChatClient` at `dotnet/src/Microsoft.Agents.AI/ChatClient/NonApprovalRequiredFunctionBypassingChatClient.cs:20-50`).
- **Limitations**
  - No central registry. The "registry" is the union of `Agent.tools` and `ChatOptions.Tools` lists, with each chat client constructing its own tool map at run time (`_get_tool_map`, `_tools.py:1648-1655`). Hot-reload of a tool from outside the agent requires mutating one of these.
  - The Python side relies on the **Pydantic model built at `@tool` time**; later mutations to the wrapped function are not reflected without recreating the tool (`_tools.py:484-514`). Forward-reference resolution is best-effort (`_resolve_input_model` `try/except`, `_tools.py:498-501`).
  - Hosted tools (e.g., `WebSearchTool`, `FileSearchTool`, `CodeInterpreterTool`, `HostedMcpServerTool`) are defined by **provider SDKs** (azure-ai-projects, openai SDK), not by Agent Framework itself; the framework only re-exports providers and providers handle them. Names are SDK-defined. No uniform `kind=` for hosted tools in the core.
  - `MCPTool.name` is reused as the tool-group identifier while the *exposed* function name is normalized + optionally prefixed (`_normalize_mcp_name`, `_build_prefixed_mcp_name`, `_mcp.py:161-177`). This split is documented but easy to miss.
  - In .NET, the `HostedMcpToolboxAITool` marker requires the hosted server to recognize `ServerAddress` (`dotnet/src/Microsoft.Agents.AI.Foundry/HostedMcpToolboxAITool.cs:10-25, 65-75`); the marker is opaque to the local model and depends on platform support.
  - Tool approval is split between Python (`FunctionTool.approval_mode`, `ToolApprovalMiddleware`) and .NET (`ApprovalRequiredAIFunction` wrapper at `dotnet/src/Microsoft.Agents.AI.Tools.Shell/DockerShellExecutor.cs:266-293`); semantics differ (per-call vs wrapped) and are not abstracted behind a single concept.
  - Naming for hosted tools is provider-coupled; Chat Completions vs Responses vs Anthropic transports name the same concept differently (e.g., web search `"type": "web_search"` vs `"type": "web_search_20250305"`).
  - `BaseChatClient` only logs a warning if the client does not subclass `FunctionInvocationLayer` (`_agents.py:722-725`), so a misconfigured client silently degrades tool-calling.

## Answers to Dimension Questions

1. **How does a new tool enter the system?**
   - **Python — local function tool**: decorate a Python function with `@tool` (with or without parens) or hand-instantiate `FunctionTool(name=..., description=..., func=..., schema=..., approval_mode=..., kind=...)` (`python/packages/core/agent_framework/_tools.py:1147-1338`, `:243-410`). The function's signature is inspected for schema, with `Annotated[T, "desc"]` becoming `Field(description="desc")` and `FunctionInvocationContext` parameters being injected by the runtime.
   - **Python — declaration-only tool**: `FunctionTool(name=..., description=..., func=None, input_model=...)` lets the agent reason about a tool whose implementation is handled elsewhere (e.g., client-side rendering). `declaration_only` is computed as `func is None` (`_tools.py:444-451, 518-543`).
   - **Python — hosted tool from a provider SDK**: `FoundryChatClient.get_web_search_tool(...)` / `get_code_interpreter_tool(...)` / `get_mcp_tool(...)` / `OpenAIChatClient.get_image_generation_tool(...)` / `AnthropicClient.get_shell_tool(func=...)` return SDK-typed objects (e.g., `WebSearchTool`, `CodeInterpreterTool`, `HostedMcpServerTool`) or plain dicts (Anthropic) that are then passed via `tools=[...]`. Discovery is gated by `isinstance(client, SupportsXxxTool)` (`python/packages/core/agent_framework/_clients.py:665-842`).
   - **Python — MCP server**: instantiate `MCPStdioTool(name=..., command=...)` / `MCPStreamableHTTPTool(name=..., url=...)` / `MCPWebsocketTool(name=..., url=...)` and pass it as part of `tools=...`. The MCP tool is opened lazily inside `_prepare_run_context` via `AsyncExitStack.enter_async_context` if `is_connected` is `False` and its `functions` (a list of `FunctionTool` produced from the server's advertised `tools/list`) are merged into the per-run tool list (`python/packages/core/agent_framework/_agents.py:1262-1281`).
   - **Python — context provider tools**: a `ContextProvider.before_run(...)` may call `context.extend_tools(source_id, get_tools())` to push tools (e.g., `FileAccessProvider`, `FileMemoryProvider`, `AgentSkillsProvider`, `TodoProvider`, security tools) into the run (`python/packages/core/agent_framework/security.py:2050`; `python/packages/core/agent_framework/_harness/_file_access.py:1056+`; `python/packages/core/agent_framework/_harness/_todo.py:447+`).
   - **.NET — local function tool**: `AIFunctionFactory.Create(method, new AIFunctionFactoryOptions { Name, Description })` (e.g., `dotnet/src/Microsoft.Agents.AI/Skills/Programmatic/AgentInlineSkillScript.cs:46`, `dotnet/src/Microsoft.Agents.AI.LocalCodeAct/LocalExecuteCodeFunction.cs:69`).
   - **.NET — DI-registered tool**: `hostBuilder.AddAIAgent(...).WithAITool(tool)`, `.WithAITools(tool1, tool2, ...)`, or `.WithAITool(sp => tool, lifetime)` — keyed services are resolved by agent name at construction time (`dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:87-140`).
   - **.NET — chat options tool**: pass `tools` to `ChatClientAgent(...)` or via `ChatClientAgentOptions.ChatOptions.Tools`; on construction these are set on `FunctionInvokingChatClient.AdditionalTools` (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientExtensions.cs:99-105`).
   - **.NET — hosted tool from a provider**: `FoundryAITool.CreateWebSearchTool(...)`, `CreateFileSearchTool(...)`, `CreateMcpTool(...)`, `CreateComputerTool(...)`, `CreateCodeInterpreterTool(...)`, `CreateImageGenerationTool(...)`, `CreateHostedMcpToolbox(...)` — these return `AITool` (`dotnet/src/Microsoft.Agents.AI.Foundry/FoundryAITool.cs:29-220`).
   - **.NET — MCP**: `McpClient.ListAgentToolsWithTaskSupportAsync(options, ct)` returns `IReadOnlyList<AIFunction>`, each one either the upstream `McpClientTool` (cast to `AIFunction`) or a `TaskAwareMcpClientAIFunction` wrapper when the server declares `execution.taskSupport == "required"` (`dotnet/src/Microsoft.Agents.AI.Mcp/McpClientTaskExtensions.cs:34-60`).
   - **.NET — attribute-discovered scripts**: methods marked with `[AgentSkillScript("...")]` on an `AgentClassSkill<T>` subclass are wrapped into `AIFunction` via `AIFunctionFactory` (`dotnet/src/Microsoft.Agents.AI/Skills/Programmatic/AgentSkillScriptAttribute.cs:46-72`).

2. **Is tool registration explicit?**
   Yes, on both stacks. There is no implicit scan or convention-based pickup.
   - **Python**: tools are passed via the `tools=` kwarg on `Agent(...)` / `RawAgent(...)` / `BaseAgent` and/or embedded in `ChatOptions["tools"]`. `BaseChatClient` warns (does not raise) if the client is not a `FunctionInvocationLayer` (`python/packages/core/agent_framework/_agents.py:722-725`). `FunctionTool` instances are duplicated-name-checked at every tool list merge via `_append_unique_tools` (`_tools.py:901-929`); duplicate names raise `ValueError("Tool names must be unique.")`. Local MCPTools that clash produce a guidance message ("Consider setting `tool_name_prefix` on the MCPTool.", `_agents.py:1255, 1266-1270`).
   - **.NET**: tools are passed via `ChatOptions.Tools` (set by `ChatClientAgentOptions` constructor, persisted to `FunctionInvokingChatClient.AdditionalTools`) or registered through `IHostedAgentBuilder.WithAITool(...)` as a keyed service keyed by the agent's name (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientExtensions.cs:99-105`; `dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:87-140`). Lifetime validation enforces no captive dependencies (`dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:146-156`).

3. **Can tools be discovered dynamically?**
   Yes, in three places:
   - **MCP**: each `MCPTool` runs `session.list_tools()` on connect (lazily, with a `function_load_lock` for reload) and exposes the result as `self._functions: list[FunctionTool]` (`python/packages/core/agent_framework/_mcp.py:333-530`). The function names are normalized with `_normalize_mcp_name` and optionally prefixed via `tool_name_prefix`. `MCPLONG_RUNNING_TASKS` adds a second discovery layer for tools that advertise `execution.taskSupport == "required"` (`_mcp.py:244-332`).
   - **In-run progressive tool exposure**: a tool invoked during a run can call `ctx.add_tools([...])` or `ctx.remove_tools([...])`, mutating the live `tools` list for subsequent iterations (`python/packages/core/agent_framework/_middleware.py:287-360`). The framework's normalization rejects atomic duplicate batches (all-or-nothing semantics).
   - **Context providers**: every `ContextProvider.before_run(...)` may push new tools via `context.extend_tools(source_id, get_tools())` (e.g., `FileAccessProvider.get_tools` → six `file_access_*` `FunctionTool`s; `python/packages/core/agent_framework/_harness/_file_access.py:1056-1190`). `SecureAgentConfig.get_tools()` adds `quarantined_llm` and `inspect_variable` (`python/packages/core/agent_framework/security.py:2054-2060`).
   - **.NET**: `AIContextProvider` may populate `AIContext.Tools`, which is merged into the per-invocation tool set (`dotnet/src/Microsoft.Agents.AI.Abstractions/AIContext.cs:85-108`). `AgentSkillsProvider` discovers skills from the file system / programmatic sources and exposes them on demand (`dotnet/src/Microsoft.Agents.AI/Skills/AgentSkillsProvider.cs`).

4. **Are names stable?**
   - `FunctionTool.name` is the immutable string set at construction (`@tool` defaults to `func.__name__`, override via `name=...` in the decorator). MCP tool exposed names are the normalized upstream name (with optional `tool_name_prefix`); collisions surface via `_append_unique_tools` rather than auto-suffixing. Foundry `_convert_tools` requires that mapping-shaped tools include a `type` discriminator for SDK rehydration (`python/packages/foundry/agent_framework_foundry/_to_prompt_agent.py:306-323`).
   - The schema is cached on the instance (`_input_schema_cached`, `_cached_parameters`, `_tools.py:386-397, 802-804`); the same tool object reused across runs returns the same `parameters()` and `to_json_schema_spec()`.
   - .NET tools are name-keyed in DI: keyed service resolution `sp.GetKeyedServices<AITool>(name)` uses the agent's name as the key (`dotnet/tests/Microsoft.Agents.AI.Hosting.UnitTests/HostedAgentBuilderToolsExtensionsTests.cs:54-251`).
   - The `[AgentSkillScript("name")]` attribute lets you pin a stable name independent of the method name (`dotnet/src/Microsoft.Agents.AI/Skills/Programmatic/AgentSkillScriptAttribute.cs:63-71`).

5. **Can multiple agents see different tool sets?**
   Yes.
   - **Python**: each `Agent` stores `default_options["tools"]` separately (`_agents.py:770`). Per-run overrides are passed via `Agent.run(..., tools=[...])` and merged by `_merge_options` (which special-cases `tools` to merge with `_append_unique_tools` rather than last-wins, `_agents.py:111-118`). Each run also gets its own normalized list (a fresh `list[ToolTypes]`) inside `FunctionInvocationLayer.get_response` so the original list isn't mutated (`_tools.py:2549-2550`). `mcp_tools` are kept on the agent instance and only opened when actually needed (`_agents.py:748-797`).
   - **.NET**: keyed services ensure per-agent isolation (`RegisteredTools_IsolatedPerAgent` test passes — `agent1`'s tools do not leak into `agent2`, `dotnet/tests/Microsoft.Agents.AI.Hosting.UnitTests/HostedAgentBuilderToolsExtensionsTests.cs:79-114`). `ChatClientAgent.ChatOptions.Tools` is per-agent; `FunctionInvokingChatClient.AdditionalTools` is set from those options for the lifetime of the chat client (`dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientExtensions.cs:99-105`).
   - **In-run progressive tool exposure** is the same on both: each tool invocation receives a `FunctionInvocationContext` whose `tools` attribute is the live, mutable list — `add_tools` / `remove_tools` change what the model sees on the next iteration.

## Architectural Decisions

- **One local tool class, one decorator.** The Python stack centers on `FunctionTool` and the `@tool` decorator (`python/packages/core/agent_framework/_tools.py:243, 1148`). The .NET stack centers on `AIFunction` and `AIFunctionFactory.Create` (used in `AgentInlineSkillScript.cs:46`, `LocalExecuteCodeFunction.cs:69`, `HyperlightExecuteCodeFunction.cs`). Both expose one canonical locally-executed tool with the same `Name` / `Description` / `JsonSchema` / `InvokeAsync` shape, plus an `additional_properties` bag.
- **Pydantic model is built at construction time.** The signature-driven schema in `_resolve_input_model` is evaluated once; subsequent invocations read `_input_schema_cached` and `_cached_parameters` (`_tools.py:386-397, 802-804`). The schema is also exposed via `to_json_schema_spec()` in the OpenAI function-spec shape (`_tools.py:869-882`).
- **`Annotated[T, "desc"]` → `Field(description=...)` mapping** keeps decorator ergonomics simple while still producing strict Pydantic JSON schemas (`_tools.py:1043-1073`).
- **Provider-agnostic `kind` and provider-specific wire shape.** `FunctionTool.kind` is a string (e.g., `SHELL_TOOL_KIND_VALUE = "shell"`) that transport layers key off; the wire shape lives in `additional_properties` (e.g., Anthropic shell `type="bash_20250124"`, `python/packages/anthropic/agent_framework_anthropic/_chat_client.py:421-468`).
- **`FunctionInvocationLayer` as a `Generic[OptionsCoT]` decorator on `BaseChatClient`.** The tool loop is reusable across providers; the framework's `BaseChatClient` warns at agent construction time if the client is not a `FunctionInvocationLayer` (`_agents.py:722-725`). Layered docstring alignment in `_apply_get_response_docstrings` (`_clients.py:978-999`) keeps the public docstrings on the lowest layer.
- **`FunctionInvocationConfiguration` as a `TypedDict`.** Per-client knobs (`enabled`, `max_iterations`, `max_function_calls`, `terminate_on_unknown_calls`, `additional_tools`, `include_detailed_errors`) are validated centrally in `normalize_function_invocation_configuration` (`_tools.py:1344-1421`).
- **`ToolTypes` is intentionally permissive.** `ToolTypes: TypeAlias = FunctionTool | MCPTool | Mapping[str, Any] | object` (`_tools.py:893`) plus callable inputs means the framework accepts:
  - decorated functions (auto-converted via `tool(func)` in `normalize_tools`),
  - explicit `FunctionTool` instances,
  - `MCPTool` instances (auto-flipped into their `.functions` at run time),
  - dict-shaped tools (provider-native or mapping form),
  - arbitrary objects implementing a `tools` collection (Toolbox-style),
  - and iterable tool-bundle wrappers (`_tools.py:941-1004`).
- **MCP has both a server-class surface and a tool-result surface.** `MCPTool` is the unit of registration; the unit of execution is `FunctionTool` produced from `session.list_tools()`. The `tool_name_prefix` lets multiple MCP servers coexist without name collisions; `additional_tool_argument_names` is configuration-only (no per-call override) and the framework strips an `_MCP_FRAMEWORK_DENYLIST` of reserved kwargs before forwarding (`_mcp.py:88-98, 425-432`).
- **Approval is a first-class `approval_mode` on the tool, not a wrapper layer.** Python: `FunctionTool.approval_mode = "always_require" | "never_require"` plus `FunctionInvocationConfiguration["additional_tools"]` for non-advertised tools (`_tools.py:303-322, 1344-1421`). .NET: `ApprovalRequiredAIFunction` wraps an `AIFunction` to gate execution (`dotnet/src/Microsoft.Agents.AI.Tools.Shell/DockerShellExecutor.cs:266-293`); `NonApprovalRequiredFunctionBypassingChatClient` sits **above** `FunctionInvokingChatClient` to bypass human approval for non-approval-required tools in mixed batches (`dotnet/src/Microsoft.Agents.AI/ChatClient/NonApprovalRequiredFunctionBypassingChatClient.cs:20-50`).
- **DI-keyed tool registration with lifetime validation** in .NET (`HostedAgentBuilderExtensions.ValidateToolLifetime`, `dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:146-156`). Refuses `toolLifetime > agentLifetime` to prevent captive-dependency bugs.
- **Hosted tools are SDK-shaped, not framework-shaped.** The framework does not define `HostedWebSearchTool` etc.; it relies on the provider SDK types (`Azure.AI.Projects.Agents.ProjectsAgentTool`, `OpenAI.Responses.ResponseTool`, `Microsoft.Extensions.AI.HostedMcpServerTool`). The Foundry package wraps these into `AITool` via `FoundryAITool.CreateXxxTool(...)` (`dotnet/src/Microsoft.Agents.AI.Foundry/FoundryAITool.cs:29-220`). The Python Foundry client returns SDK-typed objects directly (`get_web_search_tool` → `WebSearchTool`; `get_code_interpreter_tool` → `CodeInterpreterTool`).
- **In-run progressive tool exposure via `FunctionInvocationContext`**. A live `tools` list is plumbed through to every tool call; `add_tools` / `remove_tools` are experimental (`_middleware.py:287-360`) and atomic on a per-batch basis.

## Notable Patterns

- **Decorator + direct-instantiation** on the Python side (`@tool` overloads `_tools.py:1147-1192`; `FunctionTool(...)` direct constructor with `declaration_only=True` when `func=None`).
- **Provider-aware factory methods** on chat clients (`get_code_interpreter_tool`, `get_web_search_tool`, `get_image_generation_tool`, `get_mcp_tool`, `get_file_search_tool`, `get_shell_tool`). Discovery is gated by `isinstance(client, SupportsXxxTool)` (`_clients.py:665-842`), so users get static duck-typing on provider capabilities.
- **Tool-collection flattening** that recognizes bundle classes (iterable), mapping-like toolboxes (`.tools` attribute), and explicit `BaseModel` passthrough. Recurses on nested bundles (`_tools.py:965-1004`).
- **Schema-from-source-of-truth** via `inspect.signature` + `typing.get_type_hints(include_extras=True)` + `Annotated` metadata, with a `pydantic.create_model` step that builds a strict JSON schema (`_tools.py:484-514, 869-882`). Forward references are best-effort with a `try/except` on `model_rebuild(force=True, raise_errors=False)` (`_tools.py:498-501, 786-789`).
- **Pydantic strict-mode generation** is implicit (Pydantic's `model_json_schema` is strict by default in v2).
- **Concurrency-aware MCP tool load** uses a `function_load_lock` and a `lifecycle_lock` plus a queued lifecycle owner task to prevent reentrancy during connect/load/reload (`_mcp.py:445-472`).
- **MCP sampling safety guardrails**: deny-by-default `sampling_callback`, with per-session rate limit + approval gate + token cap. Documented in `AGENTS.md` (`python/packages/core/AGENTS.md`).
- **Declarative vs imperative tool sources**: tools can be declared in code (`@tool`), in YAML/JSON via the declarative package (`python/packages/declarative/agent_framework_declarative/`), or as attributes on a skill class (`[AgentSkillScript]` / `[AgentSkillResource]` in .NET).
- **Layered docstring alignment**: `_apply_get_response_docstrings` (`_clients.py:978-999`) walks the chat-client decorator chain and aligns docstrings from the lowest layer upward.
- **Captive-dependency guard** for tool lifetimes in .NET (`HostedAgentBuilderExtensions.ValidateToolLifetime`).
- **Auto-bypass layer for non-approval-required tools** in mixed batches (`NonApprovalRequiredFunctionBypassingChatClient`) mirrors the Python `.NET-style bypass` documented in `python/packages/core/AGENTS.md` for `ToolApprovalMiddleware`.
- **Tool name prefixing for MCP** (`tool_name_prefix`) gives users a way to disambiguate multiple MCP servers without depending on upstream names staying stable.
- **Per-agent isolation by keyed services** in .NET DI (`AddKeyedSingleton(builder.Name, tool)` / `AddKeyedService(...)`, `dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:92, 137`).
- **`ToolApprovalMiddleware` is middleware, not a wrapper.** Tools don't have to be wrapped to be approval-gated; the middleware handles the round trip via the agent session's state bag (`python/packages/core/agent_framework/_harness/_tool_approval.py:87-220`).

## Tradeoffs

- **Per-agent/per-client tool lists vs. global registry.** No global registry means explicit, auditable tool scopes. The cost: there is no single "list of all tools installed anywhere"; to find every tool an agent could possibly call, you have to walk `Agent.tools`, the per-run override in `_prepare_run_context`, every `ContextProvider.before_run` that calls `context.extend_tools`, and every MCP server's `.functions`. Cross-agent sharing requires passing the same `FunctionTool` instance by reference.
- **Schema from signature** freezes the schema at `@tool` time; later signature changes are not reflected without recreating the tool (`_tools.py:484-514`). The trade-off is no schema drift between Python signature and tool spec.
- **`kind=` and `additional_properties` for provider adaptation** vs. fully-typed tool classes. Anthropic shell tools, for example, are regular `FunctionTool` instances with `kind="shell"` and `additional_properties={"type": "bash_20250124"}`; this keeps the framework's tool core small at the cost of more transport-specific metadata to track (`python/packages/anthropic/agent_framework_anthropic/_chat_client.py:421-468`).
- **Hosted-tool provider coupling.** `FoundryAITool.CreateXxxTool` exists specifically because the Foundry SDK's `ResponseTool` types don't implement `AITool` directly; the framework re-wraps them (`dotnet/src/Microsoft.Agents.AI.Foundry/FoundryAITool.cs:21-28, 36-46`). Similar in Python: `FoundryChatClient.get_mcp_tool` returns an `azure.ai.projects.models.Tool` subclass.
- **`ToolTypes` permissive alias.** Accepts `FunctionTool | MCPTool | Mapping[str, Any] | object | Callable`; the price is that `normalize_tools` has a long `isinstance` ladder and the `else` branch passes through whatever it is given (`_tools.py:962-1003`). New tool categories need a new branch.
- **MCP allowlist vs. denylist model.** Tools declare their own `inputSchema.properties`; the framework allowlists those + user-extras, with a denylist of framework-named kwargs (`_MCP_FRAMEWORK_DENYLIST`) as a safety net. The `_meta` key is extracted as request metadata, never forwarded as an argument. Trade-off: tool-side schema is the source of truth, but the framework must read the schema to build the allowlist, which is fragile for `additionalProperties: true` schemas.
- **DI-keyed tool resolution** in .NET depends on the agent name being a stable string; renaming the agent changes the keyed-service key. No warning today.
- **Approval semantics differ between stacks.** Python's `FunctionTool.approval_mode="always_require"` is interpreted inside the framework's `_try_execute_function_calls`. .NET's `ApprovalRequiredAIFunction` wraps the tool and gates via the framework's `NonApprovalRequiredFunctionBypassingChatClient`. The two systems are not abstracted behind a single concept.
- **Progressive tool exposure is `experimental`.** Marked with `ExperimentalFeature.PROGRESSIVE_TOOLS` and gated by the experimental warning system (`_middleware.py:287, 324`).
- **`include_detailed_errors` default is `False`.** Tool errors are returned to the model as `Error: Function failed.` by default (`_tools.py:1556-1569, 1632-1645`); opting in exposes exception text.

## Failure Modes / Edge Cases

- **`max_invocations < 1` or `max_invocation_exceptions < 1`** raises `ValueError` at construction (`_tools.py:399-402`).
- **Duplicate tool name** (different object, same name) raises `ValueError("Duplicate tool name '<n>'. Tool names must be unique.")` via `_append_unique_tools` (`_tools.py:901-929`); MCP local-server duplicates get a guidance message ("Consider setting `tool_name_prefix` on the MCPTool.", `_agents.py:1255, 1266-1270`).
- **Adding a tool with the same name to a running agent's live list** raises `ValueError` unless it's the same object (`_middleware.py:300-308, 320-322`); the batch is validated atomically against a throwaway copy before mutation.
- **Declaration-only tool invoked** raises `ToolException("Function '<n>' is declaration only and cannot be invoked.")` from `__call__` and `invoke` (`_tools.py:518-519, 618-619`).
- **Sync callable + `tool_choice="required"` with `allowed_tools`** is rejected at the OpenAI Chat Completions layer (`python/packages/openai/agent_framework_openai/_chat_completion_client.py:670-672`).
- **MCP server list before connect** raises `ToolExecutionException` (in the Python MCP code path; not opened in this analysis but consistent with the `is_connected` / `enter_async_context` flow at `_agents.py:1264-1265, 1275-1276`).
- **MCP `sampling/createMessage` by default is denied.** Deny-by-default callback + per-session cap + per-request `maxTokens` cap. To opt in, callers must pass `lambda params: True` (`python/packages/core/agent_framework/_mcp.py:107-117, 411-414`).
- **Local MCPTool in `PromptAgent` tool list** raises `ValueError` ("Local MCP tool ... cannot be published as a prompt-agent tool. Use FoundryChatClient.get_mcp_tool(...) to register a hosted MCP server instead.", `python/packages/foundry/agent_framework_foundry/_to_prompt_agent.py:275-279`).
- **Mapping tool without `type` discriminator** raises `ValueError("Dict-shaped tools must include a 'type' field matching a Foundry tool discriminator.")` (`python/packages/foundry/agent_framework_foundry/_to_prompt_agent.py:318-319`).
- **Tool lifetime longer than agent lifetime in .NET DI** raises `InvalidOperationException` ("A tool with lifetime '<X>' cannot be registered for an agent with lifetime '<Y>'...", `dotnet/src/Microsoft.Agents.AI.Hosting/HostedAgentBuilderExtensions.cs:152-155`).
- **Empty `function_call_content.call_id` while processing a function call** raises `KeyError` (`_tools.py:1580-1581`).
- **`result_parser` raising** falls back to `Content.from_text(str(result))` with a `WARNING` log (`_tools.py:702-706, 763-767`).
- **Schema supplied as a `Mapping` containing complex JSON Schema features (`oneOf`, `$ref`)** is not supported — only flat object schemas (`python/packages/core/agent_framework/_tools.py:1218-1222`).
- **Forward reference resolution failure** is silently swallowed (`try/except` around `model_rebuild(force=True, raise_errors=False)`, `_tools.py:498-501, 786-789`).
- **`max_invocations` counter is per-instance and never reset.** A long-lived `FunctionTool` accumulates invocations across requests; the docstring explicitly recommends using `FunctionInvocationConfiguration["max_function_calls"]` for per-request limits (`_tools.py:325-340`).
- **Unknown tool in model output** with `terminate_on_unknown_calls=True` raises `KeyError`; otherwise returns an error result to the model (`_tools.py:1719-1720, 1470-1479`).
- **Two `MCP_FUNCTION_CALL`s with the same `call_id`** are deduplicated by `_extract_function_calls` (`_tools.py:2167-2187`).
- **`FunctionInvocationLayer` not applied to `BaseChatClient`** logs a warning but does not fail the agent (`_agents.py:722-725`).

## Future Considerations

- **Tool registry / cross-agent catalog.** A first-class registry would make cross-agent reuse safer and let DevUI / observability tools enumerate "every tool installed anywhere" (a known gap in DevUI's `_extract_tools_from_object` at `python/packages/devui/agent_framework_devui/_discovery.py:867-938`, which walks `agent.default_options["tools"]`, `agent.tools`, and per-provider `get_*_tool` factories but has no global view).
- **Tool version field.** No version field on `FunctionTool`; deprecating a tool today means manual edits to every `Agent(tools=[...])` list. Would also help with persisted session state that needs to deserialize a tool reference.
- **MCP tool list hot-reload.** `MCPTool` supports a `pending_reload_tasks` set (`_mcp.py:472`) but no agent-level subscription to change notifications.
- **Stable identity for tool references in sessions.** Tool references in session state today are not first-class; the `ToolApprovalMiddleware` and `_harness/_tool_approval.py:160-220` use `ToolApprovalState` models for standing rules, but tool identity across serialization is name-based.
- **Unified hosted-tool abstraction.** Today, `FoundryAITool.CreateXxxTool` and the Python provider's `get_*_tool` factories are SDK-coupled. A framework-level `HostedTool` base with provider-agnostic configuration (kind, options dict) could collapse the surface.
- **Tool approval parity between Python and .NET.** Python uses `FunctionTool.approval_mode`; .NET uses `ApprovalRequiredAIFunction` wrapper. The two systems should ideally have one shared model.
- **DI-keyed tool resolution by name is fragile to agent renames.** A capability-keyed or `nameof`-keyed alternative would be more robust.
- **Tool-collection flattening is a hidden layer.** Calling code can pass `tools=[toolbox, my_func]` and have it just work, but the flattening in `normalize_tools` (`_tools.py:941-1004`) has surprising branches (`BaseModel` is excluded; mapping-like tool collections are detected by `.tools` attribute). This is well tested but not broadly documented.
- **Tools inside `ChatOptions` vs. tools inside `AIContext` (.NET) overlap.** `ChatOptions.Tools` is per-invocation base; `AIContext.Tools` is per-invocation override. The merge semantics for the two are not explicitly documented in code.
- **Progressive tool exposure is `experimental`.** Promote to stable once the semantics (atomic batch add/remove, live list ownership) have been stable long enough.

## Questions / Gaps

- **Visibility of "tools available to this agent" at runtime.** The merge happens inside `_prepare_run_context` (`_agents.py:1252-1339`) but the final list is not stored back on the agent; it lives on `chat_options["tools"]` and on the live `tools` list inside `FunctionInvocationContext`. There is no public API to dump the resolved tool set after the run starts. DevUI's `_extract_tools_from_object` does its best (`python/packages/devui/agent_framework_devui/_discovery.py:867-938`) but it walks `default_options["tools"]` and `obj.tools` only, missing `ContextProvider`-injected tools.
- **Exact wire shape for hosted tools in the Python core.** The framework itself does not define `HostedCodeInterpreterTool` / `HostedWebSearchTool` etc.; it relies on `azure.ai.projects.models`, the openai SDK, or plain `dict`. A search of `python/packages/core/agent_framework` for `HostedCodeInterpreterTool` returned no hits. Provider-specific tool classes live in the provider packages.
- **`@tool` vs `@ai_function`.** `ai_function = tool` is an alias (`_tools.py:2876-2877`) so there is no behavioral difference; the alias is provided "for security tools and samples" but the underlying type is `FunctionTool`. No clear evidence of two semantically distinct tool types in Python.
- **`FunctionTool.approval_mode` and `MCPSpecificApproval`.** `MCPTool` accepts an `MCPSpecificApproval` TypedDict with `always_require_approval` / `never_require_approval` collections (`_mcp.py:58-69`). `FunctionTool.approval_mode` is a flat literal. The relationship is not symmetric — MCP-specific lists are evaluated per-MCP tool name; `FunctionTool.approval_mode` is per-tool instance. No global "this agent requires approval for everything" knob was found in the core; the closest is `ToolApprovalMiddleware` in `_harness/_tool_approval.py`.
- **`MCPWebsocketTool` lazy import boundary.** Unlike `MCPStdioTool` / `MCPStreamableHTTPTool`, the WebSocket transport may require an optional dependency. The lazy-import pattern is consistent with the rest of the framework, but no test or docstring in this analysis directly confirmed whether `MCPWebsocketTool` works out-of-the-box on Python 3.13 / non-ws environments. (Searched `_mcp.py:2734-2920`; the import is guarded.)
- **Default tool set in `BaseChatClient.as_agent` and `Agent`.** There is no framework default tool set on `Agent(...)`. The default tool set only materializes in `HarnessAgent` (`dotnet/src/Microsoft.Agents.AI.Harness/HarnessAgent.cs:267-279`, `HostedWebSearchTool` + optional `ShellExecutor.AsAIFunction()`) and in `create_harness_agent` (`python/packages/core/agent_framework/_harness/_agent.py:239-...`). Plain `Agent(tools=...)` users get exactly the tools they pass.
- **Per-call override precedence vs. `default_options["tools"]`.** `_merge_options` in `_agents.py:111-118` deliberately treats `tools` as a list-merge (via `_append_unique_tools`) rather than last-wins. The exact conflict-resolution rule when the same name appears in both is "raise ValueError" rather than "override" or "defer to runtime" — verified in tests at `python/packages/core/tests/core/test_function_invocation_logic.py:4351-4440`.

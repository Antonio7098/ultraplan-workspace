## Summary

Temporal is a workflow orchestration server, not an agent harness, so the "who picks the next step?" question resolves into three layered control-flow regimes, all of which are runtime- and framework-owned rather than model-owned. The LLM (or user worker code) only enters through well-defined task delivery channels; the framework always reserves the authority to refuse, redirect, terminate, or replay next-step decisions.

The three regimes are:

1. **V1 workflow task state machine** — `service/history/workflow/workflow_task_state_machine.go:39-44` runs an event-sourced `Scheduled → Started → Completed/Failed/TimedOut` machine that the history service drives (`service/history/transfer_queue_active_task_executor.go:126-184`). User code executes inside an SDK worker that returned commands, but the next state and the next task are decided by `TaskGeneratorImpl` (`service/history/workflow/task_generator.go:34-105`) and committed by the framework. The framework can also **cancel or reroute** at multiple checkpoints: `ScheduleWorkflowTask` checks guard conditions (`workflow_task_state_machine.go:574`, `workflow_task_state_machine.go:721-736`), and the matching service drops superseded tasks (`service/matching/matching_engine.go:802-833`).

2. **HSM (Hierarchical State Machines)** — `service/history/hsm/sm.go:18-66` provides a generic `StateMachine[S]` with a `Transition[S, SM, E]` value type whose `Sources []S` and `Destination S` fields make every transition explicit. `Tree.MachineTransition` (`service/history/hsm/tree.go:579-629`) is the only legal mutation path, performs rollback on error, and increments a `TransitionCount` that `validateStateMachineRef` (`service/history/statemachine_environment.go:223-289`) uses to reject stale or dangerous moves. Tasks emitted by transitions are typed `Immediate`/`Timer`/`Outbound` executors (`service/history/hsm/registry.go:151-260`, `service/history/tasks/state_machine_task.go:11-103`).

3. **ChASM (Coordinated Heterogeneous Application State Machines)** — `chasm/statemachine.go:14-79` is the newer uniform runtime that replaces bespoke HSMs. `Transition.Apply` (`chasm/statemachine.go:52-78`) verifies `Sources` membership, runs the user `apply` function, then sets `Destination`. All `AddTask` calls in the transition commit atomically with the state change (transactional outbox, `docs/architecture/chasm.md:233-238`). The engine exposes a read-only `PollComponent` that long-polls on a monotonic predicate and is woken by `NotifyExecution` (`chasm/chasmtest/test_engine.go:259-323`); runtime overrides exist via `WithSpeculative`/`WithBusinessIDPolicy` options (`chasm/engine.go:84-173`), conflict/reuse policies in `StartExecution` (`chasm/chasmtest/test_engine.go:344-401`), `DeleteExecution` and `Terminate` requests, and the staleness check at `service/history/statemachine_environment.go:223-325`.

Across all three regimes, the LLM is not in the loop at all: the LLM equivalent is the SDK worker process, which receives `ActivityTask`/`WorkflowTask` from the matching service, returns a `RespondWorkflowTaskCompletedRequest`, and the framework re-derives the next state from the request via `mutableStateImpl` (`service/history/workflow/mutable_state_impl.go`) — the worker cannot unilaterally invent a "next step" that the framework has not already prepared a task for, and several explicit rejection paths exist (`ErrStaleReference`, `ErrWorkflowCompleted`, `ErrStaleState`).

Transitions are explicit and centralized: every state change passes through one of `MachineTransition` (HSM, `service/history/hsm/tree.go:579`), `Transition.Apply` (ChASM, `chasm/statemachine.go:52`), or the V1 `workflowTaskStateMachine.Apply*` family (`service/history/workflow/workflow_task_state_machine.go:62, 169, 247, 256`). There is no string-based "next action" command; the next action is a typed transition with a known source-state set, a known destination, and a registered executor.

Control flow is testable without any "LLM": `chasmtest.Engine` (`chasm/chasmtest/test_engine.go:82-323`) is a full in-memory implementation of `chasm.Engine` that exercises every state machine, transition, and task without an SDK worker. `TransferQueueActiveTaskExecutor` tests (`service/history/transfer_queue_active_task_executor_test.go`) drive state machines end-to-end with `gomock`-driven history clients. `spec_processor.go:62-208` and `generator_tasks.go:40-180` exercise scheduler transitions without invoking any external service.

## Rating

**9 / 10** — Control flow is mature, durable, observable, and has the runtime override authority required by the rubric. State machine transitions are typed, validated against an explicit source set, atomically committed with their emitted tasks, and the framework can always reject a stale or invalid move via `validateStateMachineRef` (`service/history/statemachine_environment.go:223-325`), the `Apply` `ErrInvalidTransition` check (`chasm/statemachine.go:70-72`, `service/history/hsm/sm.go:60-62`), the `validateNotZombieWorkflow`/`ZOMBIE` check (`service/history/statemachine_environment.go:349-361`), and the `ExecutionAwareScheduler.HandleBusyWorkflow` busy-workflow gate (`service/history/queues/scheduler.go:218-223, 29-36`). There is a public test surface (`chasmtest.Engine`, `TransferQueue*TaskExecutor` tests) that does not require an SDK worker or "LLM". It does not reach 10 because (i) the V1 workflow state machine still coexists with HSM/ChASM and the V1 path encodes control flow across multiple large files (`service/history/transfer_queue_active_task_executor.go`, `timer_queue_active_task_executor.go`, `mutable_state_impl.go`) where transitions are not always encoded as a single typed `Transition` value, and (ii) some V1 components rely on side-effect tasks dispatched to `tasks.Task` queues whose category/dispatch is a string-keyed registry, weakening the test-without-LLM guarantee for those code paths.

## Answers to Dimension Questions

1. **Who decides what happens next?**
   The framework. In V1, the history service drives the next step via `TaskGeneratorImpl` (`service/history/workflow/task_generator.go:34-105`); the SDK worker reports what it did, not what should happen next, and the framework re-derives the next state via `mutable_state_impl`. In ChASM/HSM, the next step is a typed `Transition` whose `Sources`/`Destination` are fixed at construction time (`chasm/statemachine.go:23-31`, `service/history/hsm/sm.go:31-39`) and is executed by registered executors (`chasm/registry.go:60-278`, `service/history/hsm/registry.go:60-278`). The "user" picks which transition to take by emitting an event (e.g., `RespondActivityTaskCompletedRequest` triggers `TransitionCompleted`), but the framework decides whether that event is *legal* via `Apply`'s `Sources` check.

2. **Can the LLM (or worker) bypass runtime control?**
   No. The worker can return commands inside `RespondWorkflowTaskCompletedRequest`, but those commands are processed by `mutable_state_impl` which validates each command against the workflow state. Outside of V1's `mutable_state_impl`, the only way to drive a state machine is through a registered `ImmediateExecutor`, `TimerExecutor`, or `PureTaskHandler`/`SideEffectTaskHandler`, and each is dispatched by the framework. The LLM equivalent (the SDK worker) is a side-effect task that must re-enter via the engine API to mutate state. The `validateNotZombieWorkflow` and `validateStateMachineRef` checks explicitly prevent a worker from driving a state machine that the framework has marked closed or stale.

3. **Can the runtime reject or rewrite the next action?**
   Yes. Explicit rejection: `Apply` returns `ErrInvalidTransition` (`chasm/statemachine.go:70-72`, `service/history/hsm/sm.go:60-62`); `validateStateMachineRef` returns `ErrStaleReference`/`ErrStaleState` (`service/history/statemachine_environment.go:223-325`); `validateNotZombieWorkflow` returns `ErrWorkflowZombie` (`service/history/statemachine_environment.go:349-361`); `CheckRunning` returns `ErrWorkflowCompleted` (`service/history/hsm/tree.go:483-507`). Explicit rewrite: the speculative `WorkflowTask` mechanism lets the framework pre-schedule a WFT and discard it on conflict (`workflow_task_state_machine.go:120-167, 574, 721-736`); the matching service drops superseded tasks (`service/matching/matching_engine.go:802-833`); the `BusinessIDConflictPolicyTerminateExisting` policy lets the framework terminate an in-flight execution on a new request (`chasm/engine.go:78-82`); `DeleteExecution` first terminates the execution then queues a delete task (`service/history/chasm_engine.go:548-625`).

4. **Are transitions explicit or scattered?**
   Explicit. In ChASM/HSM, every transition is a single typed value (`Transition[S, SM, E]`) and the only mutation path is `MachineTransition`/`Apply`. The `Sources` slice is a closed set; `Possible` is `slices.Contains` over that set (`chasm/statemachine.go:45-47`, `service/history/hsm/sm.go:52-54`). The V1 path is more distributed: `workflow_task_state_machine.go` is the closest thing to a single file for WFT transitions, but `mutable_state_impl.go` and the executor pair `transfer_queue_active_task_executor.go` / `timer_queue_active_task_executor.go` split the responsibilities. The framework is migrating to ChASM to consolidate.

5. **Is control flow testable without calling an LLM/SDK?**
   Yes. `chasmtest.Engine` (`chasm/chasmtest/test_engine.go:82-323`) is a complete in-memory `chasm.Engine` implementation that supports every operation, conflict/reuse policy, long-poll, and notifier. It is used in `chasm/lib/scheduler/helper_test.go:161-220`, `chasm/lib/scheduler/scheduler_test.go`, `chasm/lib/scheduler/invoker_execute_task_test.go:32-535`, and the entire `chasm/lib/*/statemachine_test.go` corpus. `service/history/hsm/sm_test.go` and `service/history/hsm/tree_test.go` test the HSM primitives without any engine. The only thing that is *not* fully tested without a real backend is the long-poll / `BUSY_WORKFLOW` busy-workflow gate (`service/history/queues/scheduler.go:29-36`).

## Architectural Decisions

- **ChASM as the consolidation target.** The V1 path uses bespoke `workflow_task_state_machine.go` + per-component `MutableState`; the migration target is ChASM, which exposes a uniform `Engine` interface (`chasm/engine.go:16-59`) and a uniform `Transition` value type (`chasm/statemachine.go:23-31`). The chasm engine is wired in as the `chasmEngine chasm.Engine` parameter on `transferQueueActiveTaskExecutor` (`service/history/transfer_queue_active_task_executor.go:73, 86`) and the V1 path forwards CHASM tasks via `executeChasmSideEffectTask` (`service/history/transfer_queue_active_task_executor.go:186-223`).
- **Atomicity is the load-bearing invariant.** Every transition emits tasks via `ctx.AddTask`; the engine commits state and tasks in one transaction (transactional outbox, `docs/architecture/chasm.md:233-238`). This is what lets the framework assert "no work is silently lost on crash" (`docs/architecture/chasm.md:244`). It also means a malicious or buggy transition cannot emit a task and then fail to commit state — the task is in the same write.
- **Transitions are *declarative*, not *imperative*.** `Sources []S` and `Destination S` are fixed at construction (`chasm/statemachine.go:25-28`); the `apply` function mutates fields and schedules tasks but cannot move the state. The state is set by `Apply` after the apply function returns successfully (`chasm/statemachine.go:77`).
- **Staleness is detected, not relied on.** Every state change increments a `TransitionCount` and stamps a `VersionedTransition` (`service/history/hsm/tree.go:589-598`, `chasm/tree.go:1215-1221`). Tasks carry the ref's transition count, and `validateStateMachineRef` checks both the initial and last-update transitions (`service/history/statemachine_environment.go:222-289`). A task whose embedded transition count does not match the current node is rejected.
- **Conflicts are resolved at the engine, not in user code.** `BusinessIDConflictPolicy{TerminateExisting, UseExisting, Fail}` is an `Engine` option (`chasm/engine.go:78-82`) and the engine applies it on `StartExecution`/`UpdateWithStartExecution` (`chasm/chasmtest/test_engine.go:344-401`). User code never sees the conflict.
- **Lifecycle is exposed.** `LifecycleState(C) -> LifecycleState` (`chasm/component.go`) lets a component report `Running`/`Completed`/`Failed`; the engine uses this for retention and auto-close (`docs/architecture/chasm.md:151-160`). The `chasm` workflow's `LifecycleState` is hardcoded to `Running` because detach mode is not implemented (`chasm/lib/workflow/workflow.go:53-61`).
- **Two task types — pure and side effect.** Pure tasks run in the same transaction (or a new one if a deadline is set); side-effect tasks run asynchronously and must re-enter via the engine API to mutate state (`docs/architecture/chasm.md:255-261`). This split enforces the boundary between "I can mutate state" and "I can call external services".
- **Speculative execution is supported and discarded on conflict.** `WithSpeculative()` (`chasm/engine.go:146-150`) and V1 `WORKFLOW_TASK_TYPE_SPECULATIVE` (`workflow_task_state_machine.go:120-167, 472-520`) pre-schedule WFTs; the runtime discards them on conflict (`workflow_task_state_machine.go:574, 721-736`). This is how speculative-but-correct execution is achieved without leaking half-finished state.

## Notable Patterns

- **Pure-value `Transition` type.** `Transition[S comparable, SM StateMachine[S], E any]` is a value, not an interface; instances are constructed once and reused (`chasm/lib/callback/statemachine.go:19-30`). The apply function is captured in a closure. This makes transitions trivially serializable, diffable, and reflectable.
- **Operation log for replication.** `MachineTransition` appends a `TransitionOperation` to the root's `opLog` (`chasm/tree.go:619-626`); the log is compacted via a tree-walk that filters out operations on deleted subtrees (`chasm/tree.go:709-805`). This pattern keeps the change set small across replication while preserving auditability.
- **Dual-side registration.** `chasm.Registration` and `service/history/hsm` registry both separate `RegisterMachine`, `RegisterTaskSerializer`, `RegisterImmediateExecutor`, `RegisterTimerExecutor`, and `RegisterRemoteMethod` (`chasm/registry.go:60-278`, `service/history/hsm/registry.go:60-278`). This split makes the runtime's authority clear: a machine can be registered for state without the right to be executed.
- **Two-tier ownership (V1 vs ChASM).** V1 still owns the WFT state machine (`workflow_task_state_machine.go`), `MutableState`, and the queue dispatch; ChASM owns per-component state machines. The bridge is `executeChasmSideEffectTask` (`service/history/transfer_queue_active_task_executor.go:186-223`), which releases the workflow lock before delegating to the ChASM engine so the ChASM task can take its own locks.
- **Speculative workflow task.** The V1 path supports `WORKFLOW_TASK_TYPE_SPECULATIVE` with `ApplyTransientWorkflowTaskScheduled` and the matching discard path (`workflow_task_state_machine.go:120-167, 574, 721-736`). The speculative WFT is created without an event in the history; if it succeeds, the event is backfilled; if it fails, it is discarded.
- **Generator / Invoker / Backfiller decomposition (ChASM scheduler).** Three sibling components, each with its own state machine, drive the scheduler: `Generator` runs a self-rescheduling `GeneratorTask` to buffer actions (`chasm/lib/scheduler/generator.go:47-58`); `Invoker` runs a `ProcessBufferTask` then an `ExecuteTask` (`docs/architecture/schedules.md:65-82`); `Backfiller` handles manual backfills (`chasm/lib/scheduler/backfiller_tasks.go:70-196`). Each is testable in isolation.
- **V1 SDK workflow scheduler (legacy).** Uses Temporal SDK primitives (`workflow.NewSelector`, `workflow.NewTimer`, `workflow.GetSignalChannel`, `workflow.Saga`) inside `SchedulerWorkflow` (`service/worker/scheduler/workflow.go:230-303, 755-824, 1067-1079`). This is the only place where control flow is "user code that runs on a worker", and it exists for backwards compatibility — the migration target is the ChASM scheduler.
- **`ScheduleNextActionTime` etc. as search attributes.** `chasm/lib/scheduler/scheduler.go:73-84` exposes schedule metadata as search attributes; the engine's visibility task keeps them in sync (`chasm/visibility.go`).
- **Engine interceptor for observability.** `chasm/interceptors.go` provides `EngineInterceptor` to instrument every engine call; `interceptor_test.go` validates it. This is how telemetry and tracing are uniformly applied without modifying the engine.
- **Pure task self-scheduling pattern.** `Generator.Generate` and `Invoker.addTasks` always re-arm themselves by calling `ctx.AddTask` with the next `GeneratorTask`/`ProcessBufferTask` (`chasm/lib/scheduler/generator.go:47-58`, `chasm/lib/scheduler/invoker.go:51-57`). The "next step" is not a runtime decision; it is *baked into* the current task handler's return.

## Tradeoffs

- **Two engines, one product.** V1 (HSM + `MutableState`) and ChASM coexist. V1's path has more moving parts and more places where "next step" is implicit (e.g., `WorkflowTask` handling in `mutable_state_impl.go` rather than a typed `Transition`). The migration to ChASM is in progress but not complete. The tradeoff: a uniform framework costs more up-front, but the V1 path is already battle-tested.
- **Pure tasks are coupled to the engine.** A pure task's deadline runs in a new transaction, but the task must be registered against an `Engine`. A misconfigured executor cannot silently drop tasks; the registration is checked at startup (`chasm/registry.go:62-147`).
- **Speculative workflow tasks add complexity.** The speculative path requires dual-write logic — the speculative WFT lives only in mutable state until validated, and the matching service can drop it on conflict (`matching_engine.go:802-833`). This adds code paths but reduces WFT latency for the common case.
- **`BusinessIDConflictPolicyTerminateExisting` is destructive.** When a user requests start with `TerminateExisting`, the engine terminates the running execution (`chasm/chasmtest/test_engine.go:344-401`). The runtime has authority to override the user's prior decision; the user can opt out by choosing `UseExisting`/`Fail`. The default is `Fail` (`chasm/chasmtest/test_engine.go:75-78`).
- **`ExecutionAwareScheduler` busy-workflow rerouting adds latency.** Tasks that hit a busy workflow are rerouted to a per-execution sequential queue (`service/history/queues/scheduler.go:29-36, 218-223`). This protects against head-of-line blocking but can lengthen p99 latency for hot workflows.
- **Hierarchical state machines complicate replication.** Each node carries its own `TransitionCount` and `VersionedTransition`; the tree's `opLog` must be compacted on subtree delete (`chasm/tree.go:709-805`). The pattern is correct but more complex than flat state.
- **The SDK workflow scheduler (V1) is hard to statically analyze.** `service/worker/scheduler/workflow.go` is a single 1724-line file that interleaves state, control flow, signals, and timers in a `Selector` loop. It is correct but harder to read than the ChASM decomposition.
- **Callback state machines exist twice.** `components/callbacks/statemachine.go` (HSM) and `chasm/lib/callback/statemachine.go` (ChASM) implement the same state machine twice. The HSM one is marked `Deprecated: HSM Callback is no longer supported.` (`components/callbacks/statemachine.go:36-39`).

## Failure Modes / Edge Cases

- **Invalid source state.** `Apply` returns `ErrInvalidTransition` and the state is not changed (`chasm/statemachine.go:70-78`). The transition function is not called.
- **Stale reference.** `validateStateMachineRef` returns `ErrStaleReference`/`ErrStaleState`. The framework reloads mutable state and retries once (`service/history/statemachine_environment.go:200-220`); on second failure, the task is dropped.
- **Workflow closed mid-transition.** `CheckRunning` returns `ErrWorkflowCompleted` for any state machine attached to a closed workflow (`service/history/hsm/tree.go:483-507`). Side-effect tasks that try to mutate after close are rejected by the engine.
- **Zombie workflow.** `validateNotZombieWorkflow` rejects any write against `WORKFLOW_EXECUTION_STATE_ZOMBIE` and treats task refs as writes even on read (`service/history/statemachine_environment.go:349-411`). This is the runtime's hard veto over replay attempts.
- **Conflict on StartExecution.** `BusinessIDConflictPolicyFail` returns an error; `TerminateExisting` terminates and replaces; `UseExisting` returns the existing ref (`chasm/chasmtest/test_engine.go:344-401`). The runtime decides.
- **Speculative WFT conflict.** V1 discards the speculative WFT and reschedules a real one (`workflow_task_state_machine.go:120-167, 574, 721-736`).
- **Task already started.** Matching service drops the task with `TaskAlreadyStarted` (`matching_engine.go:799-813`).
- **Build ID / deployment mismatch.** Matching service drops the task with `ObsoleteDispatchBuildId`/`ObsoleteMatchingTask` (`matching_engine.go:802-833`). The runtime has authority to discard rather than deliver to a worker.
- **Namespace handover.** `transferQueueActiveTaskExecutor.execute` returns `ErrNamespaceHandover` without executing during handover (`service/history/transfer_queue_active_task_executor.go:143-153`). The runtime pauses task execution.
- **Resource exhausted.** `ResourceExhausted` from history returns to the client (or triggers `BUSY_WORKFLOW` rerouting, `matching_engine.go:833-840`).
- **Pure task immediate execution failure.** `executeImmediatePureTasks` runs pure tasks at the end of the transaction; failures roll back the transaction (`chasm/tree.go:1683-1700+`).
- **Node deleted mid-transition.** `DeleteChild` marks the entire subtree as deleted; subsequent transitions on a deleted node return `ErrStateMachineInvalidState` (`chasm/tree.go:404-439, 579-583`).
- **Duplicate registration.** `chasm/registry.go:62-147` returns `ErrDuplicateRegistration`; the same `ImmediateExecutor` cannot be registered twice.
- **ChASM task validation.** `Validate(ref, node)` on a `Task` returns `ErrStaleReference`/`ErrWorkflowCompleted` to discard a task whose target is no longer relevant (`service/history/hsm/tasks.go:55-81`).

## Future Considerations

- **Complete the V1 → ChASM migration.** The dual-engine setup is a known cost; the HSM callback is already deprecated (`components/callbacks/statemachine.go:36-39`). Future work should keep migrating V1 components (e.g., the SDK workflow scheduler, which is scheduled to be retired after `EnableCHASMMigration` / `MigrateWithRunningWorkflows` ship widely, see `service/worker/scheduler/workflow.go:117-122, 165-167`).
- **Unify execution authority.** Right now the V1 `MutableState` is the source of truth for WFT state, and ChASM components have their own state. Once ChASM owns everything, the framework can simplify the `execute` switch in `transfer_queue_active_task_executor.go:156-177` to a single ChASM-side handler.
- **Tighter "no LLM in the loop" guarantee for tests.** A `chasmtest.Engine` exists, but several task executors in `service/history/queues` and `service/matching` are not directly unit-tested with an `Engine` mock. Future test infra should mock the engine and assert task emission, not just call the handler.
- **Improved observability hooks.** The `chasm.transition` span event is gated on `telemetry.DebugMode()` (`chasm/statemachine.go:56`); a default-on trace event would help production debugging.
- **Transition history (mentioned in TODOs).** `CompareState` exists for backward compatibility (`service/history/hsm/registry.go:151-180`); a TODO mentions removing it once transition history is fully implemented (`service/history/hsm/tree.go:55-58, 519-520`).
- **Detached mode for ChASM workflow.** `chasm/lib/workflow/workflow.go:53-61` notes detached mode is not implemented; tasks for callback components can't run after the workflow is closed until then.
- **Speculative ChASM transitions.** `WithSpeculative()` exists but speculative semantics for ChASM components (not just V1 WFT) are not fully realized. The framework could speculatively pre-schedule a ChASM transition and discard on conflict.
- **Stale-state retry budget.** `service/history/statemachine_environment.go:200-220` retries once; a budget-based retry with circuit-breaking could be more robust under load.

## Questions / Gaps

- **How is "next step" chosen when a user submits an arbitrary `UpdateComponent` event?** The engine looks up the registered `Transition[S, SM, E]` for that machine type and event type (via `Registry.Machine` and the `apply` function pointer in `Transition`). If no transition matches, the call returns `ErrInvalidTransition`. The selection of *which* transition to run is therefore: the registered `Transition` for the (component type, event type) pair — there is no runtime "smart" choice. Evidence: `chasm/registry.go:60-147` and `chasm/statemachine.go:70-78`.
- **Can the SDK worker refuse to take a task?** Yes — long-poll will time out, and the matching service will keep the task in the backlog (`service/matching/matcher.go`). The runtime is patient, but the framework can reroute via `ExecutionAwareScheduler.HandleBusyWorkflow` (`service/history/queues/scheduler.go:218-223`).
- **What happens if a state machine is registered but the executor is missing?** `ExecuteImmediateTask` / `ExecuteTimerTask` / `ExecuteRemoteMethod` return `ErrNotRegistered` (`service/history/hsm/registry.go:151-221, 193-221, 225-260`). `notRegisteredError.IsTerminalTaskError() bool { return true }` (`service/history/hsm/registry.go:29-31`) marks it as a terminal task error, so the framework does not retry indefinitely.
- **What is the difference between V1 WFT state machine and ChASM components?** V1 is owned by `mutable_state_impl` and persists to history; ChASM is owned by the ChASM engine and persists to `ChasmNode`s. Both are atomic; the difference is in the *shape* of the state and the *style* of mutation. See `service/history/chasm_engine.go:516-625` for the bridge.
- **Can the LLM directly drive a ChASM transition?** Only by submitting a `historyservice`/`workflowservice` request that the history service maps to an `Engine.UpdateComponent` call. The worker cannot directly call `Engine.UpdateComponent` — that is a server-side API. The SDK exposes `UpdateWorkflowExecutionOptions`/`UpdateHandler`/etc., all of which are routed through the history service. So no, the LLM cannot directly mutate ChASM state without the framework's intermediary.

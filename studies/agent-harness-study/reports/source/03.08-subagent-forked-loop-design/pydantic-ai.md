## Summary

Pydantic AI has **no first-class "child run" / `SubAgent` class** in its public API. Delegation is a deliberate user-level pattern: an outer agent defines a tool that calls `await inner_agent.run(prompt, usage=ctx.usage, deps=...)` and returns the inner agent's output (`docs/multi-agent-applications.md:25-57`, `examples/pydantic_ai_examples/medical_agent_delegation.py:180-238`). The framework treats that inner `agent.run()` call as a real, full-fledged child run with its own `_graph_run`, its own `run_id`, its own message history, its own `AgentRunResult`, and its own OTel span — but **shares no conversation state** unless the user explicitly passes `message_history=...` or copies fields. Usage accounting is opt-in via `usage=ctx.usage` (`docs/multi-agent-applications.md:44`, `tests/test_usage_limits.py:188-310`).

Concurrency inside an agent loop comes from two orthogonal mechanisms: (1) `ParallelExecutionMode` inside a single agent's tool-call step (`pydantic_ai_slim/pydantic_ai/tool_manager.py:38-110`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1902-1956`) — this fans out independent **tool calls** requested by the same model response, not sub-agents; and (2) the `pydantic_graph` graph executor, which has a first-class `Fork` / `Join` / reducer abstraction with `asyncio` task groups, deadlock-free parent-fork detection, broadcast and map over iterables / `AsyncIterable`s, and reducers like `reduce_list_append`, `reduce_list_extend`, `reduce_dict_update`, `reduce_sum`, `reduce_null`, `ReduceFirstValue` (`pydantic_graph/pydantic_graph/graph_builder.py:992-1035`, `pydantic_graph/pydantic_graph/join.py:101-148`, `pydantic_graph/pydantic_graph/node.py:60-95`, `pydantic_graph/pydantic_graph/parent_forks.py:1-232`, `docs/graph/builder/parallel.md`). Forks are step-level (broadcast a value or map an iterable to N step tasks), not agent-level, but a step body is free to call any agent — so the framework deliberately leaves "subagent within a forked branch" to user code.

There is no built-in `SubAgent` capability class; the closest primitives that *delegate to a subagent internally* are the common-tools wrappers for image generation (`pydantic_ai_slim/pydantic_ai/common_tools/image_generation.py:60-159`) and X search (`pydantic_ai_slim/pydantic_ai/common_tools/x_search.py:35-128`), which construct a fresh `Agent` per call, run it with the supplied model + native tool, and translate `UnexpectedModelBehavior` into `ModelRetry` so the outer model can self-correct. The same idea is exposed as a higher-level `ImageGeneration` / `XSearch` capability with a `fallback_model=...` keyword that selects the subagent branch automatically (`pydantic_ai_slim/pydantic_ai/capabilities/image_generation.py:26-31`, `pydantic_ai_slim/pydantic_ai/capabilities/x_search.py:28-31,144-156`).

Across-process delegation is exposed through `fasta2a.pydantic_ai.agent_to_a2a` (`pydantic_ai_slim/pydantic_ai/_a2a.py:79-319`, `docs/a2a.md`), which packages an `Agent` as a Starlette/FastA2A server speaking the Google A2A protocol — this is an inter-agent *handoff* boundary, not an in-process subagent. Durable agents (`TemporalAgent`, `PrefectAgent`, `DBOSAgent`) are `WrapperAgent` subclasses (`pydantic_ai_slim/pydantic_ai/agent/wrapper.py:35`, `pydantic_ai_slim/pydantic_ai/durable_exec/temporal/_agent.py:57`, `pydantic_ai_slim/pydantic_ai/durable_exec/prefect/_agent.py:45`, `pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:52`); they participate in delegation exactly as the wrapped agent would, since `WrapperAgent.iter()` forwards every argument to `self.wrapped` (`pydantic_ai_slim/pydantic_ai/agent/wrapper.py:277-296`).

Traces are correlated through OTel baggage rather than a pydantic-ai-managed parent/child tree: each `Instrumentation` capability opens its own agent-run span with `gen_ai.agent.call.id = ctx.run_id` and sets the same `run_id` as OTel baggage so nested `current_otel_traceparent()` calls can resolve back to the active span (`pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:127-187`, `pydantic_ai_slim/pydantic_ai/_instrumentation.py:31-368`). `AgentRun._traceparent` and `AgentRunResult._traceparent` expose this to user code, and `Message.run_id` is stamped per response (`pydantic_ai_slim/pydantic_ai/run.py:106-117,484-498`, `pydantic_ai_slim/pydantic_ai/_utils.py:399-407`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:129,852,912,969`).

## Rating

**7 / 10 — Clear delegation model with tests, explicit interfaces, and operational safeguards, but a real first-class subagent / child-loop primitive is absent; what exists is a set of user-pattern primitives plus graph-level parallelism.**

Rationale (anchored to evidence):

- **Delegation is fully supported and tested.** `tests/test_usage_limits.py:188-310` snapshot-tests usage propagation across two `Agent`s; `examples/pydantic_ai_examples/medical_agent_delegation.py:1-238` is a reference orchestration with routing, escalation, and try/except. The docs distinguish "agent delegation" (return-from-tool) from "programmatic hand-off" (`docs/multi-agent-applications.md:13-177`).
- **Trace correlation works end-to-end via OTel baggage and `current_otel_traceparent()`.** `pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:156-159` attaches baggage on the agent-run span; `pydantic_ai_slim/pydantic_ai/run.py:106-117` exposes the traceparent to consumers like `OnlineEvaluation`. Subagent spans therefore nest under the parent's span tree automatically when the inner agent is started inside the active OTel context.
- **Parallel forked loops are a real first-class abstraction in `pydantic_graph`**, with deadlock detection via `ParentForkFinder` (`pydantic_graph/pydantic_graph/parent_forks.py:72-197`), join reducers (`pydantic_graph/pydantic_graph/join.py:101-148`), fork stacks for tracking parallel lineage (`pydantic_graph/pydantic_graph/id_types.py:31-55`), AnyIO task groups (`pydantic_graph/pydantic_graph/graph_builder.py:457,838`), and `cancel_sibling_tasks` for early-stop joins (`pydantic_graph/pydantic_graph/join.py:73-79,141-148`, `pydantic_graph/pydantic_graph/graph_builder.py:1049-1059`).
- **Tool-call parallel execution is explicit and tunable.** `pydantic_ai_slim/pydantic_ai/tool_manager.py:38-110` defines `ParallelExecutionMode = Literal['parallel', 'sequential', 'parallel_ordered_events']` and exposes `parallel_execution_mode()` as both a contextmanager and (deprecated) `sequential_tool_calls` alias; `_agent_graph.py:1902-1956` switches between sequential loop, `asyncio.create_task` + `asyncio.wait(FIRST_COMPLETED)`, and `ALL_COMPLETED` based on the mode; sibling tasks are explicitly cancelled on cancellation / error via `cancel_and_drain` (`pydantic_ai_slim/pydantic_ai/_utils.py:232-249`).
- **Error propagation from subagent to parent is explicit.** The built-in subagent tools convert `UnexpectedModelBehavior` to `ModelRetry` so the parent model is asked to fix the issue (`pydantic_ai_slim/pydantic_ai/common_tools/x_search.py:73-77`, `pydantic_ai_slim/pydantic_ai/common_tools/image_generation.py:103-107`). The graph layer surfaces errors as `ErrorMarker` so callers can recover via `GraphRun.override_next(...)` or re-raise (`pydantic_graph/pydantic_graph/graph_builder.py:134-144,661-668`). `pydantic_ai_slim/pydantic_ai/_utils.py:201-229` defines `gather` so a failing sibling task cancels the rest instead of leaving them orphaned (unlike plain `asyncio.gather`).
- **Gaps that keep this below 8.** (a) There is no `SubAgent` / `child_agent_run` first-class type — the `Subagents-pydantic-ai` third-party integration is explicitly mentioned as the answer for "task / check_task / wait_tasks / list_active_tasks / soft_cancel_task / hard_cancel_task / answer_subagent" (`docs/capabilities.md:1736-1738`). (b) No parent/child *run-id* linkage: each agent run gets a fresh UUID7 (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:107-129`); correlation across a delegation hop relies on OTel baggage / spans, not on a framework field. (c) The "deep agents" guidance lists `subagents-pydantic-ai` as a community package rather than an in-tree primitive (`docs/multi-agent-applications.md:338-340`). (d) In-process parallel *agents* (not tools) are not provided — only graph steps, which are user-defined, so users must wire `await agent.run()` inside step bodies themselves.

## Answers to Dimension Questions

1. **Can agents delegate?** Yes. Delegation is an explicit, documented pattern: define a tool on the parent that does `await child_agent.run(prompt, usage=ctx.usage, deps=ctx.deps)` and return the child's output. The child run is a real `AgentRun` with its own graph, message history, result, run_id, and OTel span. Documented in `docs/multi-agent-applications.md:13-77`; canonical example in `examples/pydantic_ai_examples/medical_agent_delegation.py:99-238`; tested in `tests/test_usage_limits.py:188-310`.
2. **Is delegation just a tool call or a real child run?** It is a real child run. The parent's `await child_agent.run(...)` is awaited inside a tool and yields an `AgentRunResult`. The framework exposes `AgentRun._traceparent`, `AgentRun.run_id`, `AgentRun.all_messages`, and `AgentRun.new_messages` so the inner run is a first-class async context — `pydantic_ai_slim/pydantic_ai/run.py:33-200`. The child agent graph is fully built and iterated (`pydantic_ai_slim/pydantic_ai/agent/abstract.py:255-449`, `pydantic_ai_slim/pydantic_ai/agent/__init__.py:1054-1349`), not a sub-tool that returns a string.
3. **Is child state isolated?** Yes by default; the user must opt in to sharing. Child runs have their own `GraphAgentState` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:127-129`), their own `message_history` (built from the child's `user_prompt` and any explicit `message_history=` the user passes), and their own `usage` (started from `RunUsage()` or the `usage=` argument). To share, the user explicitly passes `usage=ctx.usage` and `deps=ctx.deps` to the child; conversation continuity requires `message_history=...` (see `docs/multi-agent-applications.md:79-148`). There is no automatic message propagation.
4. **Are child traces linked to parent traces?** Yes via OpenTelemetry context propagation. The `Instrumentation` capability opens the agent-run span with `gen_ai.agent.call.id = ctx.run_id` and attaches `gen_ai.agent.call.id` as OTel baggage (`pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:127-176`). When a subagent's `agent.run()` is invoked inside that span, its own `Instrumentation.wrap_run` opens a *child* span (any OTel tracer does this automatically because the parent's span is the active span via `start_as_current_span`). `AgentRun._traceparent` resolves to the child's span; `AgentRunResult._traceparent_value` captures it for later inspection (`pydantic_ai_slim/pydantic_ai/run.py:106-117,484-498`). Per-message `run_id` is stamped via `fill_run_metadata` (`pydantic_ai_slim/pydantic_ai/_utils.py:399-407`). Note: pydantic-ai does **not** maintain its own parent/child relationship between `run_id`s — correlation is entirely an OTel concern, and would break if the user clears the OTel context between the parent tool and the child call.
5. **Can child loops run in parallel?** Two answers depending on what "child" means:
   - *Sub-agents invoked from a parent's tool* — sequential by default (a single tool returns one value), but the framework places **no restriction** on `asyncio.gather([agent_a.run(...), agent_b.run(...)])` inside a tool body. The framework's `gather()` helper cancels siblings on failure (`pydantic_ai_slim/pydantic_ai/_utils.py:201-229`).
   - *Tool calls inside a single agent step* — yes, with explicit modes: `parallel`, `sequential`, `parallel_ordered_events` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:38-110`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:1902-1956`). Sibling tasks are cancelled via `cancel_and_drain` on `CancelledError` or `BaseException`.
   - *Forked branches in `pydantic_graph`* — yes, first-class: broadcast, map over `Iterable` / `AsyncIterable`, nested broadcasts and maps, deadlock-safe join via `ParentForkFinder`, reducers including `ReduceFirstValue` for early cancellation (`pydantic_graph/pydantic_graph/node.py:60-95`, `pydantic_graph/pydantic_graph/graph_builder.py:992-1085`, `pydantic_graph/pydantic_graph/join.py:101-148`, `pydantic_graph/pydantic_graph/parent_forks.py:1-232`, `docs/graph/builder/parallel.md`).

## Architectural Decisions

- **Delegation as a tool, not a primitive.** The framework deliberately keeps `Agent` simple and pushes delegation to a user-defined tool that calls `await other_agent.run(...)`. This avoids baking a parent/child model into `Agent.run`'s signature and keeps a delegated agent identical to a top-level one (`docs/multi-agent-applications.md:13-23`). The cost is that there is no framework-level "subagent list" / lifecycle; the benefit is that any agent can be reused either as a parent, a child, or both without configuration.
- **`WrapperAgent` as the cross-cutting integration point.** Durable execution (Temporal/Prefect/DBOS) and similar are implemented as `WrapperAgent` subclasses that override `iter()`/`run()`/`run_stream()`/`run_stream_events()` and forward to `self.wrapped`. Because `WrapperAgent.iter()` delegates transparently (`pydantic_ai_slim/pydantic_ai/agent/wrapper.py:277-296`), a durable agent participates in delegation identically to a vanilla one — `pydantic_ai_slim/pydantic_ai/durable_exec/temporal/_agent.py:57-426`, `pydantic_ai_slim/pydantic_ai/durable_exec/prefect/_agent.py:45-300`, `pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:52-179`.
- **Parallel tool-call dispatch is decoupled from model-side parallelism.** Three explicit modes exist (`parallel`, `parallel_ordered_events`, `sequential`) so callers can trade event-ordering guarantees for throughput, and per-tool `sequential=True` overrides the contextvar default (`pydantic_ai_slim/pydantic_ai/tool_manager.py:155-168`). DBOS durably limits itself to a subset (`pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:46-83`) — model provider `parallel_tool_calls` is a separate, provider-controlled concern (`pydantic_ai_slim/pydantic_ai/models/openai.py:490,1013`, `pydantic_ai_slim/pydantic_ai/models/anthropic.py:1200-1201`, `pydantic_ai_slim/pydantic_ai/models/groq.py:356`).
- **Fork/Join in `pydantic_graph` is step-level, not agent-level.** The fork node broadcasts a value or maps over an iterable; the joined node reduces with a user-supplied reducer. The framework does **not** start an agent per branch automatically — branch bodies are step functions the user supplies, and they may choose to call `await agent.run(...)`. This is a deliberate separation of "graph state machine" from "agent loop" (`pydantic_graph/pydantic_graph/node.py:60-95`, `pydantic_graph/pydantic_graph/join.py:151-216`, `docs/graph/builder/parallel.md:1-10`).
- **Error-as-retry for subagent failures inside the built-in common tools.** `ImageGenerationSubagentTool` and `XSearchSubagentTool` convert `UnexpectedModelBehavior` from the inner agent into `ModelRetry`, which the outer `ToolManager` translates into a `RetryPromptPart` and feeds back to the outer model. The subagent's textual failure becomes a model-readable retry hint, not an exception (`pydantic_ai_slim/pydantic_ai/common_tools/x_search.py:73-77`, `pydantic_ai_slim/pydantic_ai/common_tools/image_generation.py:103-107`). This is consistent with Pydantic AI's general philosophy of staying on the agentic loop until output is valid.
- **Trace correlation via OTel, not framework state.** No `parent_run_id` / `child_run_id` fields exist on `AgentRun`. Each run gets a fresh UUID7; cross-run correlation flows through OTel baggage (`gen_ai.agent.call.id`, `gen_ai.conversation.id`) set by `Instrumentation.wrap_run` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:129`, `pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:127-176`, `pydantic_ai_slim/pydantic_ai/_instrumentation.py:31,357-368`). Aggregated usage across parent + children is reported via `gen_ai.aggregated_usage.*` when configured (`pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:221-228`).

## Notable Patterns

- **Two parallel dispatch APIs.** `gather()` (`pydantic_ai_slim/pydantic_ai/_utils.py:201-229`) and `ParallelExecutionMode` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:38-110`) coexist: the first is a generic utility that cancels siblings on first failure (via AnyIO task group + `ExceptionGroup` unwrap); the second is an event-ordering-aware scheduler inside the agent tool step. Both are safe by default (no orphan tasks).
- **`GraphRun.override_next(...)` recovery loop.** Graph-level errors are surfaced as `ErrorMarker`, not exceptions, so a hook can recover by calling `GraphRun.override_next([GraphTaskRequest(...)])` or `EndMarker(value)` (`pydantic_graph/pydantic_graph/graph_builder.py:134-144,559-571,659-680`). This is the recovery primitive that makes capability hooks like `on_node_run_error` viable — and a future `SubAgent` capability would inherit this for free.
- **`CancelScope.shield=True` during cancel-and-drain.** `pydantic_ai_slim/pydantic_ai/_utils.py:232-249` shields the drain so that anyio's level-cancelling behaviour can't recurse into the children while they unwind their own `finally` blocks. This is a small but important detail that prevents orphan tasks on outer cancellation.
- **`ReducerContext.cancel_sibling_tasks()`.** The only branch-cancel mechanism in the join abstraction; implemented by `_cancel_sibling_tasks` walking `active_tasks` and matching fork ids (`pydantic_graph/pydantic_graph/graph_builder.py:1049-1059`, `pydantic_graph/pydantic_graph/join.py:73-79`). This is the "race the first finisher and drop the rest" pattern, exposed to reducers.
- **`current_otel_traceparent()` as a graph-aware view.** The fallback for getting a traceparent when `GraphRun._traceparent` is `None` — reads the active OTel span via `opentelemetry.trace.get_current_span()`, so a user-side `OnlineEvaluation` consumer can pick up the right span even if it didn't receive the `GraphRun` (`pydantic_ai_slim/pydantic_ai/_instrumentation.py:357-368`, `pydantic_ai_slim/pydantic_ai/run.py:106-117`).
- **`A2A` for cross-process.** `fasta2a.pydantic_ai.agent_to_a2a` exposes a Pydantic AI agent as a Google-A2A-protocol Starlette/FastA2A app. This is the standardized cross-process sibling of in-process delegation, with `Storage` (task state) / `Broker` (task scheduling) / `Worker` (execution) slots (`pydantic_ai_slim/pydantic_ai/_a2a.py:79-319`, `docs/a2a.md:39-100`).

## Tradeoffs

- **Delegation is user-shaped, not framework-shaped.** Pro: zero ceremony, no schema migration to introduce a `SubAgent`. Con: no shared convention for "given a parent prompt, which subagent should run?" — users must hand-write the routing tool (`docs/multi-agent-applications.md:79-148`, `examples/pydantic_ai_examples/medical_agent_delegation.py:180-238`).
- **No framework-level parent/child `run_id` linkage.** Pro: avoids storing extra metadata on every run; OTel already provides hierarchical trace context. Con: correlation requires an OTel exporter; in a tool-only-no-instrumentation setup, there is no on-the-`AgentRun` parent reference.
- **Built-in subagent tools are domain-specific.** `ImageGenerationSubagentTool` and `XSearchSubagentTool` exist, but there is no `GenericSubagentTool` that delegates to an arbitrary `Agent` from inside a tool. The user has to write that wrapper themselves (`pydantic_ai_slim/pydantic_ai/common_tools/x_search.py:35-77`, `pydantic_ai_slim/pydantic_ai/common_tools/image_generation.py:60-107`).
- **Parallel mode in durable execution is restricted.** DBOS excludes bare `'parallel'` mode because deterministic replay cannot guarantee order (`pydantic_ai_slim/pydantic_ai/durable_exec/dbos/_agent.py:46-83`). Users on durable runners give up true `FIRST_COMPLETED` ordering. This is documented as `use_aggregated_usage_attribute_names`-style explicit constraint, not silent.
- **Fork stack only tracks fork lineage, not agent lineage.** The graph executor can answer "which fork run produced this join input?", but not "which subagent produced this message?" — those are different scopes and the framework does not unify them (`pydantic_graph/pydantic_graph/id_types.py:31-55`, `pydantic_ai_slim/pydantic_ai/_agent_graph.py:129`).
- **Graph's "parent fork" logic is purely structural.** `ParentForkFinder` chooses a dominating fork via a dominator-tree fixed-point over the static graph; if no dominating fork exists, the user must supply `parent_fork_id` manually (`pydantic_graph/pydantic_graph/parent_forks.py:96-134`). This is correct but adds friction for non-trivial graphs.

## Failure Modes / Edge Cases

- **Sibling-task orphaning on plain `asyncio.gather`.** If user code does `asyncio.gather(*[tool_a(), tool_b()])` directly and `tool_a` raises, `tool_b` keeps running. Pydantic AI ships `gather()` (`pydantic_ai_slim/pydantic_ai/_utils.py:201-229`) precisely to avoid this; agents should use that helper or `ParallelExecutionMode` (`pydantic_ai_slim/pydantic_ai/tool_manager.py:38-110`).
- **`MessageHistory` not auto-propagated.** A delegated agent has no knowledge of the parent's conversation unless the user explicitly passes `message_history=...` (`docs/multi-agent-applications.md:79-148`). A subagent without that argument will respond to only its own prompt.
- **Token-cost attribution.** Without `usage=ctx.usage`, the parent's `RunUsage` will not reflect subagent requests; the user must opt in (`tests/test_usage_limits.py:188-310`, `docs/multi-agent-applications.md:42-57`). If the child uses a *different model* than the parent, `result.usage` of the parent still reflects cumulative cost, but per-model cost breakdown is not computed (`docs/multi-agent-applications.md:22-23`).
- **`AgentRun.next()` requires capability-aware iteration.** Bare `async for node in agent_run` skips `wrap_node_run` hooks and fails to drain `when_idle` queued messages, raising `UndrainedPendingMessagesError` at end of run (`pydantic_ai_slim/pydantic_ai/run.py:189-233`). A subagent that relies on `enqueue(..., priority='when_idle')` may deadlock or silently drop messages if the user iterates with `async for` instead of `agent.run()` or explicit `next()`.
- **`GraphRun.override_next` must be called between iterations.** Calling it while an iteration is in flight is unsupported; error semantics on misuse are not enumerated (`pydantic_graph/pydantic_graph/graph_builder.py:559-572`).
- **Image-only models as fallback_agent.** `_IMAGE_ONLY_MODELS` (`pydantic_ai_slim/pydantic_ai/common_tools/image_generation.py:37-46`) and the `_check_image_only_model` guard raise `UserError` to prevent `gpt-image-1` etc. from being used as a conversational subagent.
- **Deterministic fork map ordering.** Map forks iterate the iterable in input order; async-iterable maps yield a new task per item asynchronously (`pydantic_graph/pydantic_graph/graph_builder.py:1011-1028`). Order-sensitive reducers (e.g. `reduce_list_append`) emit in completion order under `'parallel'` mode — users wanting input order must use `parallel_ordered_events` (`pydantic_ai_slim/pydantic_ai/_agent_graph.py:1928-1933`) or a deterministic reducer.

## Future Considerations

- **First-class subagent API.** The natural next step is a `SubAgent(agent=..., name=...)` capability that exposes `task`, `check_task`, `wait_tasks`, `list_active_tasks`, `soft_cancel_task`, `hard_cancel_task`, `answer_subagent` — the surface area already prototyped in `subagents-pydantic-ai` (`docs/capabilities.md:1736-1738`). Because `Instrumentation` is already a capability, instrumentation of subagents (parent/child span wiring, usage roll-up) would compose cleanly with `gen_ai.aggregated_usage` (`pydantic_ai_slim/pydantic_ai/capabilities/instrumentation.py:221-228`).
- **Framework-level parent/child run linkage.** Today `run_id` is opaque; an explicit `parent_run_id` field on `AgentRun` (with a contextvar or explicit param) would make non-OTel setups still correlate correctly. The hook system already supports this: `before_node_run`/`after_node_run` could stamp parent ids on the child's run (`pydantic_ai_slim/pydantic_ai/run.py:267-329`).
- **Map-broadcast over agents, not just steps.** `GraphBuilder` already supports arbitrary step bodies in branches (`docs/graph/builder/parallel.md:60-66`). A first-class `agent_branch(agent, num_workers=N)` primitive that fanned out N copies of an agent over a list, joined via a reducer, would close the loop on "parallel subagents" at the agent level rather than the step level.
- **Reduce API surface on `join.py`.** `ReduceFirstValue` exists but `reduce_min` / `reduce_max` do not. Adding numeric reducers alongside `reduce_sum` would round out the common parallel-collection aggregation cases (`pydantic_graph/pydantic_graph/join.py:101-148`).
- **`Runner`-level cancellation.** `cancel_sibling_tasks` cancels tasks in the same parent fork (`pydantic_graph/pydantic_graph/graph_builder.py:1049-1059`); a "cancel all branches after N failures" mode would help for fan-out / fan-in search patterns.

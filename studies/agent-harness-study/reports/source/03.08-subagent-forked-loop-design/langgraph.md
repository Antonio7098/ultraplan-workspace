## Summary

LangGraph treats delegation as **graph embedding**: a child `Pregel` is composed into a parent `Pregel` by being passed as a node (`graph.add_node("child", compiled_subgraph)`). There is **no `as_tool`, no `handoff`, no agent-as-tool wrapper** in the framework itself — the construction is structural, not a tool-call facade. The `PregelNode` constructor resolves the embedded subgraph via `find_subgraph_pregel(self.bound)` (`libs/langgraph/langgraph/pregel/_read.py:182-194`, `libs/langgraph/langgraph/pregel/_utils.py:47-75`) and stores it in `PregelNode.subgraphs: Sequence[PregelProtocol]` (`libs/langgraph/langgraph/pregel/_read.py:147`). Pregel then schedules a regular PULL task for that node; the task body is the embedded graph's `Runnable.invoke`, with the parent's channels and checkpointer surfaced as a child namespace.

There are **three distinct surfaces that look like "subagent"** but operate at different layers:

1. **Embedded subgraphs (real child runs).** When a `CompiledStateGraph` (or `RemoteGraph`) is added as a node, `find_subgraph_pregel` walks the bound runnable's `RunnableSequence`/`RunnableLambda`/`RunnableCallable` graph and collects the first encountered `PregelProtocol` (`libs/langgraph/langgraph/pregel/_utils.py:47-75`). The discovery is shallow but recursive across composition wrappers. Disabled checkpointer detection — `Pregel` instances with `checkpointer=False` are excluded (`libs/langgraph/langgraph/pregel/_utils.py:54-56`) — but other patterns (returning `Runnable` from a callable, bare async functions) are still wrapped via `coerce_to_runnable` (`libs/langgraph/langgraph/_internal/_runnable.py:529-560`) and only produce a subgraph if the wrapped result is itself a `Pregel`.

2. **Functional API tasks (`@task` + `@entrypoint`).** Tasks (`libs/langgraph/langgraph/func/__init__.py:59-251`) return a `SyncAsyncFuture`, which lets the parent fan them out concurrently (`asyncio.gather(*futures)`, `libs/langgraph/langgraph/func/__init__.py:209`). The entrypoint compiles to a single-node `Pregel` whose body uses `ChannelWriteEntry(PREVIOUS, ...)` to thread state forward (`libs/langgraph/langgraph/func/__init__.py:576-609`); tasks are scheduled as `PUSH` tasks via the topic channel (see `prepare_push_task_functional`, `libs/langgraph/langgraph/pregel/_algo.py:813-935`), not as sub-Pregels.

3. **v3 streaming subgraph handles.** The new `version="v3"` `stream_events` path produces in-process `SubgraphRunStream` / `AsyncSubgraphRunStream` handles (`libs/langgraph/langgraph/stream/run_stream.py:587-649`) backed by per-subgraph mini-muxes (`libs/langgraph/langgraph/stream/_mux.py:193-225`, `libs/langgraph/langgraph/stream/transformers.py:670-925`). This is an **observability projection**, not a delegation mechanism; it discovers subgraph scopes that are *already* running as part of the parent Pregel execution. The transformer learns of subagents via `metadata["lc_agent_name"]` carried on `tasks` events — the same convention used by `langchain.agents.create_agent` to mark named agents (`libs/langgraph/langgraph/stream/transformers.py:524-536`).

Delegation semantics are tight but limited:
- **No "agent-as-tool" or handoff tool surface in langgraph.** Both live in the upstream `langchain.agents.create_agent` package; langgraph is downstream and treats any `Pregel` as a delegatable unit.
- **No per-call child session** — the embedded subgraph inherits the parent's `Runtime` and `BaseStore`; `thread_id` is reused; `checkpoint_ns` is extended with a `node:<task_id>` segment (`libs/langgraph/langgraph/pregel/_algo.py:615-624`, `libs/langgraph/langgraph/pregel/_algo.py:833-843`).
- **Per-namespace checkpoint isolation is configurable**, not automatic: `checkpointer=True` opts the subgraph into its own persisted state under the parent's saver (`libs/langgraph/langgraph/types.py:98-104`, `libs/langgraph/langgraph/pregel/main.py:2824-2844`), `checkpointer=False` disables it, `checkpointer=None` inherits the parent's saver.
- **Parallelism is real but happens inside one superstep**: Send-style fan-out dispatches PUSH tasks (`libs/langgraph/langgraph/pregel/_algo.py:986-1000`); the runner executes them concurrently via `concurrent.futures` or `asyncio.wait(FIRST_COMPLETED)` (`libs/langgraph/langgraph/pregel/_runner.py:282-291`, `:481-491`), with sibling-cancel-on-failure via `_should_stop_others` (`libs/langgraph/langgraph/pregel/_runner.py:616-634`).
- **Error propagation is sharp**: a child failure re-raises through the task path; `GraphBubbleUp` exceptions (`GraphInterrupt`, `ParentCommand`, `GraphDrained`) are not re-raised as errors but bubbled to the root (`libs/langgraph/langgraph/errors.py:50-129`, `libs/langgraph/langgraph/pregel/_runner.py:585-594`, `libs/langgraph/langgraph/pregel/_retry.py:618-634`).
- **Trace nesting is wired through `var_child_runnable_config` and LangSmith's `_set_tracing_context`** (`libs/langgraph/langgraph/_internal/_runnable.py:67-103`). Each task is registered as a child run via `run_manager.get_child(...)` (`libs/langgraph/langgraph/_internal/_runnable.py:409`, `:482`, `:668-670`, `:716-718`), producing nested spans by construction.
- **Cross-process delegation is supported** through `RemoteGraph` (`libs/langgraph/langgraph/pregel/remote.py:118-1308`), a `PregelProtocol` that translates the embedded-subgraph contract into HTTP calls; the parent cannot distinguish `RemoteGraph` from a local `Pregel` from the dispatcher's perspective.

## Rating

**9 / 10 — Mature, durable, observable, extensible, and proven under failure or scale.**

LangGraph's delegation model is intentionally uniform: one mechanism (embed a `Pregel` as a node), with all the variation happening in the contract surface — `checkpointer=True/False/None`, `interrupt_before/after`, `subgraphs=True` for streaming, `Command.PARENT` for parent-graph routing, `RemoteGraph` for cross-process, and the v3 streaming projection for navigation. This earns the top-half rating because: (1) every claim about delegation maps to a single, well-tested Pregel-internal contract — `find_subgraph_pregel` (`libs/langgraph/langgraph/pregel/_utils.py:47-75`), `PregelNode.subgraphs` (`libs/langgraph/langgraph/pregel/_read.py:147-194`), `checkpoint_ns` propagation (`libs/langgraph/langgraph/pregel/_algo.py:615-624`), `GraphInterrupt`/`ParentCommand`/`GraphBubbleUp` bubbling (`libs/langgraph/langgraph/errors.py:50-129`); (2) parallel child execution is a first-class concern of the runner, not an afterthought, with sibling cancellation on failure (`libs/langgraph/langgraph/pregel/_runner.py:616-634`), explicit error-handler routing (`libs/langgraph/langgraph/pregel/_runner.py:171-176`, `:297-323`), per-task `RetryPolicy` and `TimeoutPolicy`, and per-task cache; (3) v3 streaming exposes `run.subgraphs` as `SubgraphRunStream` handles (`libs/langgraph/langgraph/stream/run_stream.py:587-649`), each backed by a per-scope mini-mux (`libs/langgraph/langgraph/stream/_mux.py:193-225`); (4) cross-process delegation via `RemoteGraph` re-uses the same protocol contract (`libs/langgraph/langgraph/pregel/remote.py:118-1308`); (5) trace context is propagated through `var_child_runnable_config` + langsmith `_set_tracing_context` (`libs/langgraph/langgraph/_internal/_runnable.py:67-103`), producing nested runs naturally. Deductions for: (a) **no "subagent" naming convention in core** — `lc_agent_name` discrimination is a streaming-time inference rule (`libs/langgraph/langgraph/stream/transformers.py:524-536`), and the test suite acknowledges the failure mode where a non-agent subgraph inherits the parent's `lc_agent_name` and surfaces as a subagent (`libs/langgraph/langgraph/tests/test_stream_lifecycle_transformer.py:549-596`); (b) **the synchronous, single-run delegation surface is the `invoke` body itself** — there is no tool-call wrapper, so calling out to a child "as a tool" requires the user to write a node that calls `child.invoke(input, config)` and merge the output back; (c) **the lifecycle transformer does not surface subagent errors as data unless the child emits an explicit `error` field on its `TaskResultPayload`** — failures during a streamed `messages` mode show up as raw exceptions that abort the run, not as `lifecycle.error` events; (d) **`Agent`-style semantics** (session, memory, tool registry) live in the upstream `langchain.agents` package; the langgraph core treats each `Pregel` as a stateless from-the-outside POV (state lives only in the channels + checkpointer).

## Answers to Dimension Questions

1. **Can agents delegate?**
   Yes, structurally. A node body can be a `Pregel` (compiled `StateGraph`, `RemoteGraph`, or anything satisfying `PregelProtocol`) and Pregel discovers this via `find_subgraph_pregel` (`libs/langgraph/langgraph/pregel/_utils.py:47-75`). There is no separate agent-as-tool API; the user either embeds a `CompiledStateGraph` as a node or calls `child.invoke(input, config)` from inside a node body (e.g. `libs/langgraph/langgraph/tests/test_pregel.py:3358`).

2. **Is delegation just a tool call or a real child run?**
   A real child run. The embedded subgraph runs its own `PregelLoop` inside the parent's `tick` (`libs/langgraph/langgraph/pregel/_loop.py:307` `is_nested=True`); it has its own channels, its own scratchpad, and its own `checkpoint_ns`. It executes concurrently with sibling tasks in the same superstep via the PregelRunner's `concurrent.futures.wait(FIRST_COMPLETED)` loop (`libs/langgraph/langgraph/pregel/_runner.py:282-291`). Cross-process delegation is a real child run too: `RemoteGraph` translates the protocol events into HTTP calls and re-uses the same `stream_subgraphs=True` mechanism (`libs/langgraph/langgraph/pregel/remote.py:780-1041`).

3. **Is child state isolated?**
   Partially — by namespace, not by session. The child shares the parent's `BaseStore`, `Runtime`, `checkpointer` (unless `checkpointer=True` opts in to its own persisted state), and `thread_id`. The child's *channels* are isolated inside the parent `PregelLoop` because they only exist when the child is running; the child's *checkpoints* are isolated by `checkpoint_ns` (`libs/langgraph/langgraph/pregel/_algo.py:615-624`). There is no per-call "session" or fresh memory scope in langgraph itself — that semantics lives in the upstream `langchain.agents.create_agent` / `langchain` packages.

4. **Are child traces linked to parent traces?**
   Yes, by construction. Each task is registered as a child callback via `run_manager.get_child(...)` (`libs/langgraph/langgraph/pregel/_algo.py:716-720`, `:901-903`), and `RunnableCallable.invoke/ainvoke/stream/astream` wrap every user step in `set_config_context(child_config, run)` which sets `var_child_runnable_config` *and* langsmith's `_set_tracing_context({"parent": run})` (`libs/langgraph/langgraph/_internal/_runnable.py:67-119`, `:417-419`, `:491-493`). The v3 streaming layer adds a separate navigation graph: `SubgraphRunStream.handle.path` is the namespace tuple, `graph_name` is the `lc_agent_name` from metadata, and the v3 `lifecycle` channel emits `started`/`completed`/`interrupted`/`failed`/`drained` events with the namespace embedded (`libs/langgraph/langgraph/stream/transformers.py:608-668`). Trace nesting is correlated via the namespace, not a separate span tree.

5. **Can child loops run in parallel?**
   Yes — at two granularities:
   - **Siblings in the same superstep.** `Send`-style fan-out and `@task` futures both translate to PUSH tasks that the runner executes concurrently (`libs/langgraph/langgraph/pregel/_runner.py:282-291`, `:481-491`). Sibling tasks share the parent's channels; their writes commit at the end of the superstep via `apply_writes` (`libs/langgraph/langgraph/pregel/_algo.py:294-345`).
   - **Cross-tick via `Send`-pushed subgraph invocations.** A node can return `[Send("child", payload) for payload in payloads]` to dispatch parallel copies of a child graph; each becomes its own `PregelExecutableTask` with a separate `task_id` (`libs/langgraph/langgraph/pregel/_algo.py:986-1000`). The `create_react_agent(v2)` path uses exactly this to run tool calls in parallel: `Send("tools", ToolCallWithContext(...))` for each `AIMessage.tool_call` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:849-859`, `libs/langgraph/langgraph/tests/test_pregel.py:8835` `test_subgraph_streaming_sync`).
   - **Function-API `@task` futures.** `asyncio.gather(*futures)` is documented as the canonical parallelism pattern (`libs/langgraph/langgraph/func/__init__.py:156-209`).

## Architectural Decisions

- **Single delegation primitive: embed `Pregel` as a node.** All delegation — local subgraph, cross-process `RemoteGraph`, even the "subagent" naming convention — bottoms out in the same discovery path: `find_subgraph_pregel` (`libs/langgraph/langgraph/pregel/_utils.py:47-75`) → `PregelNode.subgraphs` (`libs/langgraph/langgraph/pregel/_read.py:147-194`) → `get_subgraphs` (`libs/langgraph/langgraph/pregel/main.py:1075-1112`). No separate agent-as-tool surface; the user composes structurally.
- **Checkpoint namespace as the unit of isolation.** Each task gets `f"{parent_ns}{NS_SEP}{name}{NS_END}{task_id}"` and the parent → child relationship is recoverable from the namespace (`libs/langgraph/langgraph/pregel/_algo.py:615-624`, `:833-843`). `recast_checkpoint_ns` strips `task_id` segments to recover the subgraph-level namespace (`libs/langgraph/langgraph/pregel/main.py:1407`, `:1451`, `:1501`, `:1554`, `:1629`, `:2102`).
- **`GraphBubbleUp` family for non-error bubbling.** `GraphInterrupt`, `ParentCommand`, `GraphDrained` are exceptions that propagate *up* the call stack without being treated as failures; the runner's `_should_stop_others` excludes them (`libs/langgraph/langgraph/pregel/_runner.py:616-634`). `GraphInterrupt` is the documented channel for surfacing `interrupt()` calls in a child up to the root (`libs/langgraph/langgraph/errors.py:102-104`).
- **`checkpointer=True | False | None` per node.** Lets each embedded subgraph opt into its own persisted state, opt out entirely, or inherit the parent's checkpointer (`libs/langgraph/langgraph/types.py:98-104`). This is the canonical way to control "is this child persistent?".
- **`Command(graph=Command.PARENT)` for parent-graph routing.** Lets a child node route control up to the enclosing graph by raising `ParentCommand`; the retry layer handles `cmd.graph == Command.PARENT` by mutating `exc.args = (replace(cmd, graph=_checkpoint_ns_for_parent_command(ns)),)` and re-raising (`libs/langgraph/langgraph/pregel/_retry.py:618-631`, `:753-769`).
- **Per-task `RetryPolicy` and `TimeoutPolicy`.** Configurable at the node level and respected by the runner's `_Runner.tick` and `_Runner.atick` (`libs/langgraph/langgraph/pregel/_runner.py:176-358`, `:360-572`, `libs/langgraph/langgraph/pregel/_retry.py:613-683`, `:696-784`). Each task gets a separate `concurrent.futures.Future` or `asyncio.Future`, and `_should_stop_others` cancels siblings on failure.
- **Streaming projection, not delegation.** The v3 `SubgraphTransformer` and `LifecycleTransformer` do not run subgraphs; they *observe* scopes from the raw `tasks` event stream and surface handles/events. The transformer mux (`libs/langgraph/langgraph/stream/_mux.py:193-225`) creates a per-scope mini-mux whose pump delegates to the parent's pump, so the child's events flow through transparently.
- **`is_nested` flag drives interrupt-deduplication.** `PregelLoop.is_nested` is computed from the presence of `CONFIG_KEY_TASK_ID` in the config (`libs/langgraph/langgraph/pregel/_loop.py:307`). `_suppress_interrupt` only converts a `GraphInterrupt` into a payload when `not self.is_nested` (`libs/langgraph/langgraph/pregel/_loop.py:1313-1352`), preventing duplicates.
- **Top-level `CONFIG_KEY_CHECKPOINT_NS` reset.** When a graph is invoked at the top level (not via another graph's `invoke`), the loop resets its own `checkpoint_ns` to `""` to avoid leaking from the parent's view (`libs/langgraph/langgraph/pregel/_loop.py:334-338`).

## Notable Patterns

- **Subgraph discovery by closure walking.** `find_subgraph_pregel` walks a node's runnable and closure non-locals looking for a `PregelProtocol` — this is what makes `builder.add_node("child", compiled_subgraph)` work without an explicit "this is a subgraph" flag (`libs/langgraph/langgraph/pregel/_utils.py:47-75`).
- **`@task` future pattern.** Functional API tasks return a `SyncAsyncFuture` that can be awaited or `asyncio.gather`-ed, producing parallel execution with deterministic ordering (`libs/langgraph/langgraph/func/__init__.py:59-251`).
- **`@entrypoint` as a one-node `Pregel`.** The functional `@entrypoint` decorator compiles to a single-node Pregel that uses `ChannelWriteEntry(END, mapper=_pluck_return_value)` and `ChannelWriteEntry(PREVIOUS, mapper=_pluck_save_value)` to split "return value" from "saved value" (`libs/langgraph/langgraph/func/__init__.py:516-609`). This lets a function be checkpointed like a graph without explicit state machinery.
- **Per-scope mini-mux for child streaming.** `SubgraphTransformer._on_started` creates a new `StreamMux` via `mux._make_child(ns)` and wraps it in a `SubgraphRunStream` (`libs/langgraph/langgraph/stream/transformers.py:713-736`). The child mini-mux forwards events from the parent pump, supports the same transformers, and produces nested grandchildren via its own `SubgraphTransformer`.
- **`_on_started` backward-compatible `cause` delivery.** `cause` is delivered via `self._pending_cause` (instance state), not via a new positional arg, so old subclass overrides (e.g. `deepagents.SubagentTransformer`) keep working (`libs/langgraph/langgraph/stream/transformers.py:421-444`, `libs/langgraph/langgraph/tests/test_stream_lifecycle_transformer.py:684-712`).
- **`var_child_runnable_config` + langsmith `_set_tracing_context`.** A single `set_config_context` context manager propagates both the callback tree and the langsmith parent for downstream observers (`libs/langgraph/langgraph/_internal/_runnable.py:67-119`).
- **`send_command_to_parent` via `Command.PARENT`.** A child node can return `Command(graph=Command.PARENT, goto=..., update=...)` to route control to the enclosing graph. The retry layer detects `ParentCommand` and rewrites the command's `graph` to the parent's checkpoint namespace before re-raising (`libs/langgraph/langgraph/pregel/_retry.py:618-631`, `:753-769`).
- **Error-handler fan-in via `node.error_handler_node`.** `StateGraph.add_node(..., error_handler=...)` registers a sibling node that runs when the original fails; the runner routes the failed task to the handler via `schedule_error_handler` (`libs/langgraph/langgraph/pregel/_runner.py:297-323`, `:496-532`).

## Tradeoffs

- **No built-in "agent-as-tool" surface.** Users wanting to expose a `Pregel` as an LLM-callable tool have to write a thin node wrapper that calls `subgraph.invoke(input)` and merges the result into a channel update. This is straightforward but means tool-call semantics (failure localization, multi-call concurrency, output schemas) are user-supplied.
- **Shared memory by default, not opt-in.** The embedded subgraph inherits the parent's `BaseStore` and `Runtime`; only `checkpointer=True` gives it its own checkpoint space. A child that should not see the parent's `store` entries has to be wrapped in a node that calls a fresh `Runtime` — not native.
- **Trace correlation via langsmith, not core.** Nested tracing only nests because `set_config_context` calls `langsmith.run_helpers._set_tracing_context`. If langsmith is not installed or tracing is disabled, the callback tree still nests, but observability tools that read langsmith will see flat runs.
- **Subagent naming convention is borrowed.** `lc_agent_name` is set by `langchain.agents.create_agent`, not langgraph itself. A user writing their own graph with no `name=` metadata inherits the parent's `lc_agent_name` and gets surfaced as a subagent (the framework documents and tests this trade-off at `libs/langgraph/langgraph/tests/test_stream_lifecycle_transformer.py:549-596`).
- **Parallel child fan-out requires Send or `@task`.** Default node execution is sequential within a superstep. The framework exposes `Send` and `@task` for explicit parallelism but does not auto-fan-out children even if they have independent inputs.
- **`interrupt()` from a child must surface as `GraphInterrupt`.** A child using `interrupt(...)` bubbles through `GraphBubbleUp` to the root, which converts it into an `interrupts` payload only at the top level (`libs/langgraph/langgraph/pregel/_loop.py:1313-1352`). Multiple interrupts from different children are merged into a single tuple, but the parent cannot tell which child emitted each one without consulting the checkpoint's `tasks[].interrupts`.
- **`v3 streaming` is experimental.** Marked with `@beta(...)` (`libs/langgraph/langgraph/stream/run_stream.py:30`). The streaming interface is still evolving.

## Failure Modes / Edge Cases

- **Subgraph that didn't opt in to its own checkpointer.** When `checkpointer=False`, the subgraph shares the parent's checkpoint namespace; its state lives inside the parent's checkpoint rows. `find_subgraph_pregel` excludes these (`libs/langgraph/langgraph/pregel/_utils.py:53-57`), so `get_subgraphs` returns them only via the runnable-walk path. The parent cannot `update_state` directly on a disabled-checkpointer child because there's no per-subgraph checkpoint to write to.
- **Unhandled child `interrupt()` in a v2 streaming run.** With `stream_mode="messages"`, an interrupt inside a child surfaces as a `GraphInterrupt` bubble; the v2 stream path converts it into an `interrupts` field on the message chunk, but only at the top level. A nested-only `interrupt()` does not stop sibling tasks if its parent task continues.
- **Sibling task cancellation order.** `_should_stop_others` cancels remaining tasks on first sibling failure (`libs/langgraph/langgraph/pregel/_runner.py:330-333`, `:539-542`). Children that are mid-write at cancellation may have partial state if durability is async.
- **`ParentCommand` rewrap loop.** `_retry.run_with_retry` only rewrites `cmd.graph` if it equals `Command.PARENT`; a deeper-graph `Command(graph="grandparent")` is re-raised unchanged. This means `ParentCommand` only crosses one level per raise; deeper escalation requires the intermediate levels to handle `ParentCommand` themselves.
- **Non-agent subgraph inheriting `lc_agent_name`.** Documented in `libs/langgraph/langgraph/stream/transformers.py:529-532`: a plain `StateGraph` invoked inside a `create_agent(name=...)` tool body inherits the parent's `lc_agent_name` and is surfaced as a subagent even though it isn't. The mitigation is to call with a config that nulls `lc_agent_name`.
- **`RemoteGraph` with `stream_subgraphs=False`.** Some events from a remote sub-run are filtered out; tests assert that messages from inside a remote child are dropped unless `subgraphs=True` (`libs/langgraph/langgraph/pregel/_messages.py:140-147`).
- **Multi-handler node retry.** When a node has both `retry_policy` and an `error_handler_node`, the retry layer runs the retry first; only after the policy gives up does `schedule_error_handler` get the task. This is correct but not obvious from the API surface.
- **Edge case: `interrupt` inside a `PUSH` task path with `True` flag.** A `@task` invocation appends `True` to its task path to mean "interrupt came from me, parent should emit"; the loop suppresses its own `INTERRUPT` emit in this case (`libs/langgraph/langgraph/pregel/_loop.py:1401-1406`).

## Future Considerations

- **v3 streaming is still `@beta(...)`.** The transformer mux / `SubgraphRunStream` / `LifecycleTransformer` API is "experimental and may change" (`libs/langgraph/langgraph/stream/run_stream.py:30`). Expect minor breaking changes to the projection keys and event shapes.
- **`create_agent` has moved to `langchain.agents`.** The prebuilt `create_react_agent` is deprecated in favor of `langchain.agents.create_agent` (`libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:53-57`). The langgraph side retains only the lower-level infrastructure; the agent-name convention and tool-call streaming semantics are owned upstream.
- **First-class tool surface for subgraphs.** A `Pregel.as_tool(...)` (mirroring `langchain.agents.create_agent`'s `tool` decorator or openai-agents' `as_tool`) would close the "no agent-as-tool" gap; current workaround is a wrapper node that calls `subgraph.invoke(input, config)`.
- **Per-child memory isolation.** `BaseStore` is shared by default; an opt-in `fresh_store=True` or per-invocation `store_namespace` parameter would let users give a child a clean memory scope without rewriting the node body.
- **Cross-process parent/child tracing.** Today, `RemoteGraph` translates the streaming protocol; a future "subagent ID" wire field would let consumers correlate a remote child run to a local parent invocation without parsing namespaces.
- **Parallel subagent fan-out as a first-class Send helper.** `Send("child", ...)` already works, but a helper like `Send.fan_out([...], node="child")` would make explicit batching the same ergonomic level as `asyncio.gather`.

## Questions / Gaps

- **What does a child return to the parent when invoked via `Send`?** The child's writes are channel-updates applied at the end of the superstep (`libs/langgraph/langgraph/pregel/_algo.py:294-345`). There is no explicit "return value" from a Send-dispatched subgraph — the parent's node body merges via the channels that were its inputs. No evidence of a richer result-handshake contract.
- **How are multiple interrupts from a child propagated to the parent at resume time?** The framework's `interrupt()` API collects a tuple, but the mechanics of resume-after-multi-child-interrupt (which child emitted which interrupt, how to route the resume) are not surfaced as a clear contract in the code; tests cover the case but the API does not label this prominently.
- **No evidence of a "child sandbox" or "isolation level" flag.** Checkpointer inheritance is the only isolation knob (`Checkpointer = None | True | False`, `libs/langgraph/langgraph/types.py:98-104`). There is no analogue of CrewAI's `propagate_session` or Agent Framework's "fresh vs shared session".
- **State isolation for `RemoteGraph` children.** A `RemoteGraph` call crosses a process boundary, but the framework does not document or enforce what state (memory, threads, run IDs) the remote peer retains. The remote endpoint is treated as opaque.
- **No centralized subagent registry or discovery API.** `get_subgraphs` walks the static node graph; there is no registry that tracks "subagents currently running" or "subagents that ran in this thread". v3 streaming's `run.subgraphs` is the closest analog, and it is event-driven, not registry-driven.

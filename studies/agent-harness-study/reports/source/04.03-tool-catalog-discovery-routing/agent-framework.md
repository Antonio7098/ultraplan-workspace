## Summary

The source implements an explicit, layered tool model rather than one universal catalog. Python agents combine agent defaults, per-run tools, context-provider tools, and connected MCP functions before sending the resulting list to the chat client (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_agents.py:1253-1316`). The .NET agent follows the same broad shape: constructor-level tools are stored as lifecycle-wide additional tools, while request-level `ChatOptions.Tools` are unioned with agent-level tools (`studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientExtensions.cs:97-105`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:625-649`).

Selection is primarily explicit configuration, model `tool_choice`, MCP allow-lists, provider-added tools, and progressive exposure. Python supports a run-local mutable tool list whose additions and removals affect the next function-calling iteration (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:2544-2550`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_middleware.py:287-370`). MCP tools are discovered from the server and filtered by `allowed_tools` before being exposed through the `functions` property (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:745-762`).

Permission is enforced mainly at execution time: Python tool approval, function middleware, MCP argument allow-lists, and security-label policy enforcement can stop or gate a call (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1691-1755`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/security.py:1726-1836`). .NET documents the same default risk explicitly: provided tools are invoked without approval unless the application adds approval behavior (`studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:21-37`).

Discovery exists for skills, MCP servers, and search context, but no general semantic or capability-retrieval service was found that selects arbitrary tools from a large registry. Skills use advertise/load/read/run progressive disclosure (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_skills.py:1719-1739`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/Skills/AgentSkillsProvider.cs:20-33`), while .NET text search either injects retrieved context before invocation or exposes a single on-demand search tool (`studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/TextSearchProvider.cs:21-27`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/TextSearchProvider.cs:109-127`).

Availability is observable but only partly explainable. Python emits tool definition attributes on agent spans and per-tool execution spans with tool name, type, call ID, arguments/results when sensitive capture is enabled (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:2020-2058`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:2228-2238`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:717-779`). The traces show what was configured and executed, but not a durable reason for each visibility decision or a catalog diff across iterations.

## Rating

**8/10 — Clear model with explicit interfaces, tests, and meaningful operational safeguards.**

The rating is supported by separate agent/run/provider catalogs, duplicate-name validation, MCP exposure filtering, next-iteration progressive tool mutation, approval workflows, security-policy enforcement, retrieval-aware providers, and OpenTelemetry coverage. The rating is held below 9 because default catalog behavior is mostly additive rather than task-minimal, authorization is generally checked after the model has seen or requested a tool, arbitrary-tool capability retrieval was not found, and traces do not explain why a tool was included, hidden, or removed.

## Answers to Dimension Questions

1. **Does every agent see every tool?** No. Tools are scoped to an agent's defaults, a run, context providers, or connected MCP servers. However, all tools in those configured sources are normally unioned into the request rather than automatically reduced by task (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_agents.py:1260-1316`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/ChatClient/ChatClientAgent.cs:625-649`).

2. **Are tools filtered by task?** Partly. Callers can pass a per-run list, use `tool_choice`, load skills or search on demand, and mutate the list during the function loop (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_agents.py:1160-1316`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_types.py:3335-3347`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/TextSearchProvider.cs:109-127`). No evidence found of a general task classifier, capability graph, or embedding-based retriever that chooses arbitrary tools from a catalog.

3. **Are tools filtered by permission?** Yes, at several boundaries. MCP has a named exposure allow-list (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:351-388`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:745-762`); local and skill tools can require explicit approval (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:301-409`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_skills.py:2156-2161`); security middleware evaluates trust and confidentiality labels before execution (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/security.py:1726-1836`).

4. **Can tools be hidden from the model?** Yes, but with different semantics. Python `additional_tools` are explicitly not advertised, while still available to the invocation map as declaration-only tools (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1366-1371`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/tests/core/test_function_invocation_logic.py:1659-1707`). MCP `allowed_tools` removes tools from the exposed `functions` view (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:745-762`). A normal approval-required or policy-protected local tool is generally still presented to the model and gated when requested, not hidden by the permission mechanism (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1691-1755`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/security.py:1645-1679`).

5. **Is tool availability explainable?** Partly. Tool definitions, tool-call metadata, MCP discovery/call spans, approval request metadata, and security audit records provide evidence of what was visible, requested, executed, or blocked (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:207-214`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:2228-2238`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/security.py:1904-1921`). No clear evidence found of a first-class availability decision record explaining why a tool was filtered, added, removed, or selected for a specific task.

## Architectural Decisions

- **Additive catalog composition:** Agent defaults, per-run tools, provider tools, and MCP functions are combined, with duplicate names rejected rather than silently shadowed (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_agents.py:1254-1281`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:901-929`). This favors predictable routing and makes collisions visible, but does not minimize the model's catalog automatically.
- **Separate exposure from execution:** The model-facing tool list and execution tool map can differ through `additional_tools`, declaration-only tools, MCP allow-lists, and runtime mutation (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1344-1371`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1648-1655`).
- **Progressive disclosure over frontloading:** Skills advertise compact metadata and load full content/resources/scripts on demand; the sample explicitly motivates this for large tool counts (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_skills.py:1728-1739`; `studies/agent-harness-study/sources/agent-framework/python/samples/02-agents/tools/dynamic_tool_exposure.py:17-24`).
- **Execution-time policy gates:** Approval and trust/confidentiality policies are implemented as middleware around actual function invocation, allowing rejection, pause-for-approval, audit, or continuation (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_middleware.py:524-585`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/security.py:1681-1836`).
- **Remote-server boundary hardening:** MCP forwards only declared or explicitly configured argument names and defaults server sampling to denial without an approval callback (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:1745-1791`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:1088-1159`).
- **Layered observability:** Agent, chat, function, and MCP operations are instrumented separately, with sensitive payload capture gated by settings (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:683-812`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:1725-1917`).

## Notable Patterns

- **Progressive tool loader:** A small visible loader can add capability only after the model asks for it; tests confirm that the newly added tool is advertised on the next iteration (`studies/agent-harness-study/sources/agent-framework/python/samples/02-agents/tools/dynamic_tool_exposure.py:28-59`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/tests/core/test_function_invocation_logic.py:4068-4131`).
- **Provider-contributed catalogs:** Context providers return tools and instructions through a provider pipeline, and the agent merges those contributions into chat options before invoking the client (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_agents.py:1425-1501`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/AIContextProviderDecorators/AIContextProviderChatClient.cs:144-181`).
- **Skill catalog as a retrieval interface:** Skills use a compact advertised index and explicit load/resource/script functions instead of injecting every skill body into the initial context (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_skills.py:1942-2008`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_skills.py:2127-2179`).
- **Capability-specific MCP filtering:** Remote tools are discovered from `tools/list`, normalized into local wrappers, and filtered by configured names including prefixed and remote aliases (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:1489-1519`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:745-762`).
- **Approval queueing:** .NET approval middleware retains standing rules in session state and serializes multiple unresolved requests into one-at-a-time caller interactions (`studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/Harness/ToolApproval/ToolApprovalAgent.cs:22-47`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/Harness/ToolApproval/ToolApprovalAgent.cs:415-500`).
- **Tool-definition telemetry:** Availability at the model boundary is represented as serialized tool definitions, while individual calls carry separate tool execution spans (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:2203-2238`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/observability.py:2020-2058`).

## Tradeoffs

- **Context size versus ease of composition:** Additive agent/run/provider merging is easy to reason about and tested, but it can present a large catalog. The progressive-exposure sample states that frontloading hundreds of tools hurts selection accuracy, increases context, and raises cost (`studies/agent-harness-study/sources/agent-framework/python/samples/02-agents/tools/dynamic_tool_exposure.py:17-24`).
- **Permission versus model visibility:** Approval and security middleware protect execution, but an ordinary local tool can remain visible to the model until it is requested (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1691-1755`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/security.py:1726-1783`). This allows the model to understand that approval is needed, but does not provide least-privilege model context by itself.
- **MCP filtering versus discovery cost:** `allowed_tools` filters the exposed function property, while the server's tool list is still loaded into the internal `_functions` collection (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_mcp.py:1424-1521`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/tests/core/test_mcp.py:1591-1600`). This improves model exposure control but may not reduce remote discovery, metadata handling, or startup cost.
- **Hidden tools versus predictable execution:** `additional_tools` support server-side or declaration-only workflows without advertising tools, but a model-requested call can still be recognized and returned as a declaration-only request rather than producing an executed result (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:1658-1765`; `studies/agent-harness-study/sources/agent-framework/python/packages/core/tests/core/test_function_invocation_logic.py:1691-1707`).
- **Retrieval freshness versus prompt-injection exposure:** Search can be on demand and can retain recent message state for relevance, but retrieved external text is injected into model context and is explicitly treated as a possible indirect prompt-injection source (`studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/TextSearchProvider.cs:30-44`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/TextSearchProvider.cs:143-195`).
- **Observability versus data exposure:** Tool arguments and results are available for traces only when sensitive telemetry is enabled in the Python path, while .NET likewise defaults to omitting raw messages and function payloads (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:733-775`; `studies/agent-harness-study/sources/agent-framework/dotnet/src/Microsoft.Agents.AI/OpenTelemetryAgent.cs:99-124`). This is safer by default but leaves availability explanations less detailed unless operators opt in.

## Failure Modes / Edge Cases

- **Duplicate names stop composition:** Merging a different tool object with an existing name raises instead of selecting a winner (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_tools.py:918-929`). MCP-specific merges provide a clearer prefix recommendation when the conflict is remote (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_agents.py:1254-1281`).
- **Unknown model calls are configurable:** The default path returns a function-result error to the model, while `terminate_on_unknown_calls=True` raises a `KeyError` (`studies/agent-harness-study/sources/agent-framework/python/packages/core/tests/core/test_function_invocation_logic.py:1589-1656`).
- **In-flight batch is not retroactively filtered:** Tool removals affect the next iteration, while calls already requested in the current batch still execute (`studies/agent-harness-study/sources/agent-framework/python/packages/core/agent_framework/_middleware.py:324-334`).

## Summary

agent-framework ships two parallel, full-fidelity checkpoint systems (Python and .NET), both keyed off a **superstep / iteration boundary** in their respective Pregel-style execution loops. Checkpoints capture the **committed** slice of execution state — messages in flight, executor state, shared state, pending external request events, iteration counter, and graph signature hash — and are persisted through a pluggable storage protocol (`CheckpointStorage` in Python at `python/packages/core/agent_framework/_workflows/_checkpoint.py:119-189`; `ICheckpointStore<TStoreObject>` in .NET at `dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/ICheckpointStore.cs:12-46`). Each implementation exposes a chained parent link (`previous_checkpoint_id` in Python `_checkpoint.py:75`; `Parent: CheckpointInfo?` in .NET `dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/Checkpoint.cs:39`) so writes form a single linear chain per workflow definition. The Python wire format is a hybrid JSON + base64-pickle document with a `RestrictedUnpickler` allowlist (`python/packages/core/agent_framework/_workflows/_checkpoint_encoding.py:115-145`); the .NET wire format is `System.Text.Json` over a portable value graph that defers deserialization via `IDelayedDeserialization` (`dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/JsonWireSerializedValue.cs:16`). Both implementations support **resume** (load by id, restore state, continue from the next superstep), but neither supports **time travel or branching**: there is no "fork from arbitrary ancestor" API, and the only pruning hook is in the foundry-hosting layer (`python/packages/foundry_hosting/agent_framework_foundry_hosting/_responses.py:702-712`). Durable execution is delegated to Azure Durable Entities (Python: `python/packages/durabletask/agent_framework_durabletask/_durable_agent_state.py:365-474`; .NET: `dotnet/src/Microsoft.Agents.AI.DurableTask/State/DurableAgentState.cs:11-27`), which carry a versioned `DurableAgentState` schema (`SCHEMA_VERSION = "1.1.0"`); the `CapturingRunnerContext` inside durable activities explicitly disables checkpointing by raising `NotImplementedError` (`python/packages/durabletask/agent_framework_durabletask/_workflows/runner_context.py:81-105`) because the orchestrator owns the state machine.

## Rating

**8 / 10** — Clear superstep-aligned checkpoint model with explicit `WorkflowCheckpoint`/`Checkpoint` schemas, a typed storage protocol, executor hooks (`on_checkpoint_save`/`on_checkpoint_restore` in Python `_workflows/_executor.py:493-517`; `OnCheckpointingAsync`/`OnCheckpointRestoredAsync` in .NET `dotnet/src/Microsoft.Agents.AI.Workflows/Executor.cs:338-347`), atomic file writes (`os.replace` at `python/packages/core/agent_framework/_workflows/_checkpoint.py:317-322`), graph-signature validation to refuse incompatible checkpoints (Python `_workflows/_runner.py:275-279`, .NET `WorkflowInfo.IsMatch` at `dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/WorkflowInfo.cs:46-103`), parent-link chain for linear history, and three production storage backends (in-memory, file, Cosmos). Deducted points: (a) **no branching / time-travel API** — the `previous_checkpoint_id` is captured but only consumed to thread the chain; there is no `fork(checkpoint_id)`; (b) **no built-in pruning / retention policy** in the core — `FileCheckpointStorage.list_checkpoints` / `delete` are exposed (`python/packages/core/agent_framework/_workflows/_checkpoint.py:158-167, 392-410`) but no consumer in core calls them; the only auto-pruning lives in the foundry-hosting shim (`_responses.py:702-712`); (c) **`InMemoryCheckpointStorage.get_latest` iterates all entries every call** (`_checkpoint.py:225-232`) — `O(n)` scan with no per-workflow index; (d) **`CosmosCheckpointStorage.load` queries by `checkpoint_id` only, then cross-partition scans for a single hit** (`python/packages/azure-cosmos/agent_framework_azure_cosmos/_checkpoint_storage.py:247-272`) — and raises if multiple workflows share an id, which becomes a footgun for operators who copy/replay checkpoint ids across workflows; (e) **pickle-based payload** despite the restricted unpickler is explicitly disclaimed as "defense in depth, not a security boundary" (`_checkpoint_encoding.py:18-45`); (f) **the .NET `ICheckpointManager` is internal** (`dotnet/src/Microsoft.Agents.AI.Workflows/Checkpointing/ICheckpointManager.cs:11`) and the public `CheckpointManager` (`dotnet/src/Microsoft.Agents.AI.Workflows/CheckpointManager.cs:13`) hides its actual impl behind a static factory, leaving only `InMemory` and `FileSystemJson` as shipped backends; (g) **durable-task activities do not checkpoint at all** — state survives only because the orchestrator's replay log includes it.

## Answers to Dimension Questions

1. **When is a checkpoint written?**
   - **Graph workflow (Python)**: at the end of every superstep, including an initial checkpoint at "superstep 0" that captures state after the start executor's first dispatch (`_runner.py:92-97`). Triggered by `Runner._create_checkpoint_if_enabled` (`_runner.py:212-238`), which is a no-op when `self._ctx.has_checkpointing()` is false.
   - **Functional @workflow (Python)**: after every **live** step execution (cache hits do not save), and once at the end of the run or on `WorkflowInterrupted` (`_functional.py:961-1009, 1042-1047`).
   - **.NET graph workflow**: at the end of every superstep via `InProcessRunner.CheckpointAsync` (`InProcessRunner.cs:332`), invoked unconditionally when a `CheckpointManager` is configured. There is no Python-style "first-checkpoint-at-superstep-0" — the .NET path creates a checkpoint at the start of the run when `StepNumber == -1` (`Checkpoint.IsInitial` at `Checkpoint.cs:30`) but only after the first superstep completes.
   - **Durable task / Foundry hosting (Python)**: durable tasks do **not** create explicit checkpoints — `CapturingRunnerContext` raises `NotImplementedError` (`runner_context.py:81-105`), so state survives because the durable task framework replays the orchestrator with the same state delta. Foundry hosting implicitly checkpoints at the end of each request turn via the `FileCheckpointStorage` it constructs (`_responses.py:593-712`).

2. **What is inside it?**
   - **Python `WorkflowCheckpoint`**: `workflow_name`, `graph_signature_hash`, `checkpoint_id` (UUID), `previous_checkpoint_id` (chain link), `timestamp` (ISO 8601 UTC), `messages` (per-source in-flight `WorkflowMessage` list), `state` (committed shared state, with `_executor_state` reserved key), `pending_request_info_events` (HITL requests still open), `iteration_count`, `metadata` dict, `version="1.0"` (`_checkpoint.py:30-117`). Per-executor state is captured separately via `executor.on_checkpoint_save()` and merged under `state["_executor_state"]` (`_runner.py:302-312, 367-379`). **Excluded**: pending state (`State._pending`), pending `asyncio.Queue` events, runtime locks.
   - **.NET `Checkpoint`**: `StepNumber` (-1 = initial), `Workflow: WorkflowInfo` (executor bindings, edges, ports, outputs), `RunnerData: RunnerStateData` (`instantiatedExecutors`, `queuedMessages`, `outstandingRequests`), `StateData: Dictionary<ScopeKey, PortableValue>`, `EdgeStateData: Dictionary<EdgeId, PortableValue>`, `Parent: CheckpointInfo?` (`Checkpoint.cs:10-39`).
   - **Durable `DurableAgentState`**: `schemaVersion="1.1.0"` + `data.conversationHistory` with request/response entries and typed content (`durable_agent_state.py:365-474`). Versioning is the only schema-evolution knob.

3. **Can it restore execution or only data?**
   Both implementations restore **full execution state**: shared workflow state, per-executor state (Python: via `_set_executor_state`/`_restore_executor_states`, `_runner.py:302-340`; .NET: via `OnCheckpointRestoredAsync` hooks called on every executor, `InProcessRunnerContext.cs:403-414`), pending messages, pending external requests, iteration counter. After restore, `runner.run_until_convergence()` (Python) or `RunSuperstepAsync` (.NET) resumes from the next superstep. HITL pending requests are **re-emitted as live events** after restore (.NET `RestoreCheckpointAsync` calls `RepublishUnservicedRequestsAsync` at `InProcessRunner.cs:387`; Python `_runner_context.py:414-426` re-adds pending request_info events to the event queue during `apply_checkpoint`). The "agent loop" inside the foundry-hosting workflow restore also feeds prior state back into the start executor as a continuation rather than a fresh run (`_responses.py:622-637`).

4. **Can checkpoints branch?**
   **No explicit branch API** in either implementation. The chain is **linear**: each checkpoint records exactly one `previous_checkpoint_id` (Python) or `Parent` (`.NET`). The .NET side does, however, expose a `withParent` filter in `ICheckpointManager.RetrieveIndexAsync(sessionId, withParent?)` (`ICheckpointManager.cs:41`) and `ICheckpointStore.RetrieveIndexAsync` (`ICheckpointStore.cs:24`), so the **storage backends can answer "what are the children of X?"** — but no code path in the repo ever **writes** a second checkpoint with a non-null `Parent` other than the previous one, so the data structure supports branching-shaped queries without anyone using them. The `FileSystemJsonCheckpointStore` even tracks `HasParentMetadata` separately from `ParentCheckpointId` to distinguish checkpoints written with explicit parent linkage from inherited linkage (`FileSystemJsonCheckpointStore.cs:84-89`). This is dead infrastructure for branching today.

5. **Are checkpoints pruned safely?**
   - **Core**: no built-in pruning. `FileCheckpointStorage.delete` exists (`_checkpoint.py:392-410`) but is never called from core. Operators must implement retention themselves.
   - **Foundry hosting**: prunes to latest per turn via `_delete_not_latest_checkpoints` (`_responses.py:702-712`). Safe because each turn is restored from the previous response_id directory before writing the new one (`_responses.py:593-712`), so deleting non-latest checkpoints within a single turn loses no resumable state — but it does destroy the per-turn chain.
   - **Cosmos**: no pruning; storage grows unbounded.
   - **Atomicity**: `FileCheckpointStorage.save` writes via `tmp + os.replace` (`_checkpoint.py:317-322`), so a crash mid-write leaves the previous file intact. `.NET FileSystemJsonCheckpointStore.CreateCheckpointAsync` rolls back both the on-disk file and the in-memory index entry on failure (`FileSystemJsonCheckpointStore.cs:164-178`), and uses `FileShare.None` on `index.jsonl` for process exclusivity (`FileSystemJsonCheckpointStore.cs:60-65`).
   - **Path safety**: Python's `FileCheckpointStorage._validate_file_path` rejects checkpoint IDs that resolve outside the configured root (`_checkpoint.py:282-300`); the foundry-hosting shim adds an extra layer for caller-controlled `context_id`s (`_responses.py:217-260`) — both with explicit regression tests (`tests/test_responses.py:2930-3316`).

## Architectural Decisions

- **Superstep-aligned, not event-aligned**: Both implementations checkpoint at superstep boundaries (Pregel model), so a checkpoint is always a consistent state slice that can be replayed without partial-write hazards. Python explicitly commits pending state before capture (`_runner.py:218-225`) and excludes `_pending` from the checkpoint payload (`_checkpoint.py:56-58`). .NET's `StateManager.ExportStateAsync` asserts "no queued updates" before export (`StateManager.cs:245-248`).
- **Committed-only state captures**: The same constraint is enforced in both languages — pending writes are discarded on checkpoint capture so the snapshot represents a deterministic, replayable point.
- **Pluggable storage via protocol/interface**: Python uses `typing.Protocol` with 6 methods (`_checkpoint.py:119-189`), .NET uses `ICheckpointStore<TStoreObject>` (open generic) + `IWireMarshaller<TStoreObject>` for serialization (`Checkpointing/ICheckpointStore.cs:12-46`, `JsonMarshaller.cs:9-?`). New backends plug in without touching the runner.
- **Hybrid JSON + pickle (Python)**: chosen for human-readable structure + Python-object fidelity. The docs explicitly disclaim pickle safety and treat the restricted unpickler as "defense in depth" (`_checkpoint_encoding.py:18-45`).
- **Portable value graph (.NET)**: every non-primitive value is wrapped in `PortableValue` and serialized via `JsonMarshaller`, with deferred deserialization via `IDelayedDeserialization`/`JsonWireSerializedValue` (`JsonWireSerializedValue.cs:16`) so the JSON shape stays stable even when types change between runs.
- **Graph signature / topology hash**: Both implementations validate structural compatibility on restore — Python's `_compute_signature_hash` mixes name, step names, `co_code`, `co_names` (`_functional.py:1144-1164`); .NET's `WorkflowInfo.IsMatch` checks start executor, executor bindings, edge list, request ports, and output tag sets (`WorkflowInfo.cs:46-103`). A checkpoint from a structurally different workflow is refused.
- **Per-executor hooks**: Both languages expose `on_checkpoint_save` / `on_checkpoint_restore` (Python) or `OnCheckpointingAsync` / `OnCheckpointRestoredAsync` (.NET) so domain-specific executors can persist their own state. In .NET, the runner context iterates every instantiated executor at superstep end (`InProcessRunnerContext.cs:394-400`).
- **Linear chain (single parent)**: Both implementations record exactly one parent link per checkpoint. .NET's storage backends are structured to support a multi-child branching tree (`withParent` filter), but nothing writes a branching checkpoint.
- **Durable-task separation**: Durable-task activities disable checkpointing entirely; the durable task framework's replay log carries state instead (`runner_context.py:81-105`). This avoids double-bookkeeping but couples the system to Azure Durable Functions storage.
- **Atomic file writes**: `os.replace` after writing to `*.json.tmp` (`_checkpoint.py:317-322`) is the same pattern used elsewhere in the framework (`_sessions.py:1134`, `_todo.py:432-444`).
- **Process-exclusivity on the .NET file store**: `FileSystemJsonCheckpointStore` opens `index.jsonl` with `FileShare.None` and fails fast if another process already holds it (`FileSystemJsonCheckpointStore.cs:60-65`), making cross-process corruption impossible but blocking cooperative hosting.
- **Cosmos `workflow_name` as partition key**: enables efficient per-workflow list queries, but means `load(checkpoint_id)` requires a cross-partition query and then a uniqueness assertion (`python/packages/azure-cosmos/agent_framework_azure_cosmos/_checkpoint_storage.py:247-272`).

## Notable Patterns

- **HitL-aware restore (Python)**: `apply_checkpoint` re-queues pending `request_info` events onto the runner's event stream so the next iteration sees them (`_runner_context.py:414-426`). The functional workflow mirrors this by restoring `_pending_requests` into the `RunContext` (`_functional.py:929-934`).
- **Step cache bypass on restore (functional workflow)**: replayed steps don't re-execute because `_step_cache` is pre-populated from the checkpoint; the framework emits `executor_bypassed` events instead of `executor_invoked`/`executor_completed` (`_functional.py:419-433`). This makes resume skip already-completed work.
- **Checkpoint chain as crash recovery**: the functional workflow's per-step checkpointing is explicitly designed for "crash recovery: every step is replayable" (`_functional.py:419-433`). Combined with the step cache, this turns an arbitrary in-flight step back into a no-op after restore.
- **Per-turn checkpoint deletion (foundry hosting)**: `_delete_not_latest_checkpoints` (`_responses.py:702-712`) is the only place in the codebase that actively manages retention — and it lives in the hosting layer, not the core.
- **Initial-checkpoint-as-flag (.NET)**: `StepNumber == -1` is the canonical "initial checkpoint" sentinel (`Checkpoint.cs:21, 30`), instead of a nullable separate field.
- **Two-tier restore on .NET runtime**: `RestoreCheckpointAsync` (public, republishes pending requests) vs `RestoreCheckpointCoreAsync` (private, used by `ResumeStreamAsync` which defers event republish to the subscriber) (`InProcessRunner.cs:380-388` ; `ICheckpointingHandle.cs:24-36`). This is an explicit API split for the streaming-vs-runtime distinction.
- **Deferred deserialization on .NET**: `JsonWireSerializedValue : IDelayedDeserialization` lets the wire format round-trip without materializing every object up front (`JsonWireSerializedValue.cs:16`).
- **Sub-workflow rehydration (Python)**: `WorkflowExecutor.on_checkpoint_restore` decodes execution contexts and re-registers pending request_info events on the sub-workflow's runner context (`_workflow_executor.py:483-535`) — a hard-coded workaround marked `TODO(@taochen): Issue #1614`.
- **Checkpoint failure tolerance (Python)**: `Runner._create_checkpoint_if_enabled` swallows exceptions (`_runner.py:236-238`) so a transient storage failure cannot abort the workflow. Verified by `test_runner_checkpoint_creation_failure` (`test_runner.py:542-586`).
- **Cross-run replay state on functional workflow**: `_last_message`, `_last_step_cache`, `_last_pending_request_ids` survive between runs of the same `FunctionalWorkflow` instance to enable response-only replay without an explicit checkpoint (`_functional.py:684-689, 941-947`). They are cleared on clean completion (`_functional.py:1017-1021`).

## Tradeoffs

- **Pickle over a typed schema (Python)**: gains Python-object fidelity, loses portability, and forces a pickle-deserialization surface that the framework itself disclaims as not-a-security-boundary (`_checkpoint_encoding.py:18-45`). The `RestrictedUnpickler` narrows the attack surface but cannot eliminate it (e.g., it allows `getattr` to support enum reconstruction, `_checkpoint_encoding.py:95`).
- **Linear chain, no branching API**: simpler invariants, but no way to fork "what would happen if this step had gone differently?" — operators who want experimentation must clone the workflow manually.
- **No built-in retention**: storage grows unbounded unless the caller wires up pruning (only the foundry-hosting shim does, and only for single-turn responses).
- **Cosmos cross-partition lookups**: `load(checkpoint_id)` is a cross-partition query with a uniqueness assertion (`python/packages/azure-cosmos/agent_framework_azure_cosmos/_checkpoint_storage.py:247-272`) — slow and raises if ids ever collide across workflows.
- **In-memory `get_latest` is O(n)**: with many checkpoints under one workflow, retrieval becomes a full scan plus `max` over timestamps (`_checkpoint.py:225-232`). No per-workflow index.
- **.NET `ICheckpointManager` is internal**: only the public `CheckpointManager` facade is exposed, and it has only two factories (`InMemory`, `Json`). Custom backends must implement the internal interface.
- **.NET file store is single-process**: `FileShare.None` on the index makes cross-process hosting impossible by design (`FileSystemJsonCheckpointStore.cs:60-65`).
- **Durable-task disables checkpointing**: simpler model but couples durable execution to Azure Durable Functions storage; an alternative durable runtime would need to implement its own `RunnerContext`.
- **Graph signature is a digest, not a canonical contract**: a checkpoint from a structurally identical workflow rebuilt from different source still passes the hash check, but a checkpoint from a workflow whose step names changed but whose shape is identical is rejected — there is no semantic versioning.

## Failure Modes / Edge Cases

- **Crash mid-`os.replace`**: impossible to leave a half-written checkpoint on disk; the previous file survives (`_checkpoint.py:317-322`).
- **Crash mid-`InMemoryCheckpointManager.Add` (`.NET`)**: the cache insertion is in-memory and atomic at the dictionary level; no corruption risk, but the entire checkpoint state is lost on process exit. The file store has its own rollback path (`FileSystemJsonCheckpointStore.cs:164-178`).
- **Workflow topology drift after restore**: caught by `WorkflowInfo.IsMatch` (`.NET`) and `graph_signature_hash` check (Python), but the error messages do not enumerate what changed — operators must re-rebuild the original workflow manually (`_runner.py:275-279`, `_functional.py:921-925`, `WorkflowInfo.cs:46-103`).
- **Pending requests after restore**: `.NET RestoreCheckpointAsync` republishes them via `RepublishUnservicedRequestsAsync` (`InProcessRunner.cs:387`); Python `apply_checkpoint` re-adds them to the event queue (`_runner_context.py:414-426`). Both treat unhandled pending requests as still-pending after restore, which is correct for HITL but can produce double-fire if the caller doesn't de-duplicate downstream.
- **Checkpoint with service-managed conversation state**: `AgentExecutor.on_checkpoint_save` documents that "if the session uses service-side storage, the full session state may not be serialized locally" (`_agent_executor.py:323-325`) — restoring from a checkpoint in this mode may lose server-side context.
- **Cosmos id collisions across workflows**: `CosmosCheckpointStorage.load` raises on duplicate `checkpoint_id` even when they're in different workflows (`python/packages/azure-cosmos/agent_framework_azure_cosmos/_checkpoint_storage.py:264-270`). UUID collisions are statistically negligible but cross-workflow id reuse (e.g., copy/replay) trips the check.
- **Cosmos DB system field leakage**: `_document_to_checkpoint` strips Cosmos internals before decode, but `_document_to_checkpoint` does not check for collisions with checkpoint field names — a Cosmos system field named `state` or `messages` would shadow the checkpoint payload (`python/packages/azure-cosmos/agent_framework_azure_cosmos/_checkpoint_storage.py:435-447`).
- **Path traversal in checkpoint id**: guarded by `_validate_file_path` (Python core, `_checkpoint.py:282-300`) and an additional layer in foundry hosting (`_responses.py:217-260`). Both are covered by explicit regression tests (`test_responses.py:2930-3316`).
- **Disallowed pickle types**: `_RestrictedUnpickler.find_class` raises `pickle.UnpicklingError` wrapped in `WorkflowCheckpointException` for non-allowlisted classes (`_checkpoint_encoding.py:139-144`). The error message tells the caller which `allowed_checkpoint_types` entry to add.
- **Checkpoint version mismatch**: `DurableAgentState.from_dict` warns and silently resets state to empty when `schemaVersion` is missing (`python/packages/durabletask/agent_framework_durabletask/_durable_agent_state.py:426-430`) — destructive default rather than a hard failure.
- **Step cache key corruption**: `RunContext._import_step_cache` raises `ValueError` with the offending key when a checkpoint's encoded cache key cannot be parsed (`_functional.py:382-389`) — caught but not auto-recovered.

## Future Considerations

- **Branching / fork API**: the .NET storage layer already supports parent-child queries (`withParent` filter); the runner would only need to allow callers to set `Parent` to a non-previous checkpoint. Python would need an equivalent for `previous_checkpoint_id`.
- **Typed schema for Python payloads**: a Pydantic/dataclass schema (instead of pickle blobs) would eliminate the pickle attack surface entirely, at the cost of forbidding arbitrary Python objects in `state`/`messages`. The foundry-hosting checkpoint already accommodates Azure-specific message role enums via `allowed_checkpoint_types` (`_responses.py:255-259`) — the same extension could grow into a typed schema.
- **Built-in retention policy**: add a `RetentionPolicy(max_count | max_age)` parameter to `CheckpointStorage` so consumers don't have to implement `_delete_not_latest_checkpoints` themselves.
- **`.NET` public `ICheckpointManager`**: making the interface public would let third-party stores (Redis, Cosmos for .NET) plug in without re-deriving the abstraction.
- **Cross-process locking**: replace `FileShare.None` with an external lock (e.g., `fcntl.flock`) so the .NET file store can be hosted across multiple processes safely.
- **Time-travel debugging**: with a branching API, "rewind to checkpoint X and replay with different inputs" becomes a single `restore(checkpoint_id)` call — currently only re-implementable by manually cloning the workflow.
- **Schema evolution for Python `WorkflowCheckpoint`**: `version = "1.0"` is in the dataclass (`_checkpoint.py:88`) but no consumer reads it. Loading a v2 checkpoint into a v1 runtime would silently mis-interpret fields.
- **Streaming checkpoint creation**: large executors (e.g., an LLM agent with full conversation) currently snapshot the entire `AgentSession.to_dict()` on every checkpoint (`_agent_executor.py:329`). A delta-based approach would cut storage cost on long conversations.
- **Multi-host checkpoint portability**: Cosmos storage is shared across hosts, but the file store uses an `index.jsonl` per directory with `FileShare.None` — there's no story for relocating checkpoints between storage backends, no export/import API.

## Questions / Gaps

- No evidence found for: an explicit **branching** or **fork** API. The chain is linear in both implementations. The .NET storage layer has the query primitives for parent-child filtering, but no caller writes a non-linear parent link.
- No evidence found for: a built-in **pruning / retention policy** in core. Only the foundry-hosting shim actively prunes.
- No evidence found for: **time-travel debugging** (rewind to a past checkpoint, replay with different inputs while preserving earlier checkpoints). The chain is destroyed as soon as `previous_checkpoint_id` advances.
- No evidence found for: a public `.NET` `ICheckpointManager` contract. The facade exposes only two factories; custom backends must implement the internal interface.
- No evidence found for: any non-file, non-Cosmos (Python) storage backend. The README mentions `Redis` for history providers (`python/packages/redis/agent_framework_redis/_history_provider.py:20-194`), but no Redis-backed `CheckpointStorage` exists.
- No evidence found for: a documented SLO / durability guarantee for the file and Cosmos backends beyond atomic write semantics. No metrics, no health checks.
- No evidence found for: how the framework handles **schema migration** of `WorkflowCheckpoint` payloads across versions. The `version="1.0"` field exists (`_checkpoint.py:88`) but no consumer branches on it.
- No evidence found for: a public API to **export** a checkpoint out of one storage backend and **import** it into another. Relocating between file/Cosmos requires manual JSON editing.
- No evidence found for: a documented limit on how many `WorkflowCheckpoint`s a single `FileCheckpointStorage` directory can hold before `list_checkpoints` / `list_checkpoint_ids` (both linear scans over `*.json` glob, `_checkpoint.py:362-449`) becomes a performance issue.

## Summary

Letta's "checkpoint" model is **not a Pregel-style channel-state snapshot** in the classical sense — it is a **layered, append-only event log** over relational tables (`runs`, `steps`, `messages`, `messages.conversation_messages`, plus optional Redis SSE streams). Each agent step performs **four discrete, transactional writes**: a `Step` row (`letta/orm/step.py:20`), a `StepMetrics` row (`letta/orm/step_metrics.py:20`), a `messages` insert via `MessageManager.create_many_messages_async` (`letta/services/message_manager.py:477`), and an update of `agent.message_ids` (`letta/services/agent_manager.py:999`) or `conversation_messages.in_context` (`letta/services/conversation_manager.py:752`). Checkpoint capture is bracketed by `_step_checkpoint_start` / `_step_checkpoint_finish` in the agent loop (`letta/agents/letta_agent_v2.py:941-1034`), which create the step row with `status=StepStatus.PENDING` (`letta/services/step_manager.py:96-173`) before LLM traffic starts and finalize it with token usage and stop reason once execution succeeds or throws.

Granularity is **one Step row per LLM call** (capped by `max_steps` per `Run`), with **all persisted messages attached via `step_id`/`run_id` foreign keys** (`letta/orm/message.py:48-53`). A `Run` (`letta/orm/run.py:22`) is the top-level envelope: it carries `agent_id`, optional `conversation_id`, `status`, `stop_reason`, `ttft_ns`, `total_duration_ns`, `request_config`, callback fields, and a paired `RunMetrics` row (`letta/orm/run_metrics.py:19`) for `num_steps` and `tools_used`. `RunStatus` enum (`letta/schemas/enums.py`) drives lifecycle transitions (`created` → `running` → `completed`/`failed`/`cancelled`). The `step_id` ↔ `run_id` ↔ `agent_id` chain is the implicit parent–child linkage; there is no `parent_step_id` column.

There is **no native time-travel, branching, or `fork_from_step` API**. The closest analog is **two resume surfaces**: (1) **stream reconnection** via Redis-stored SSE chunks (`letta/server/rest_api/redis_stream_manager.py:199-526`) with cursor-based `starting_after` replay (`letta/server/rest_api/redis_stream_manager.py:463-526`), gated by `RedisSSEStreamWriter` (`letta/server/rest_api/redis_stream_manager.py:26-198`) with `stream_ttl=10800s` default and `max_stream_length=10000`; and (2) **conversation-mode resumption** via `ConversationManager.fork_conversation` (`letta/services/conversation_manager.py:104-172`), which copies message references into a new `Conversation` row with shared `messages` rows (via the `conversation_messages` junction) — a one-way soft branch only. The conventional "agent resume from checkpoint" semantics are implemented implicitly by **re-reading `agent.message_ids` (or `conversation_messages` ordered list) on agent load** and replaying the in-context messages at the next `step()` call (`letta/schemas/agent.py:78`, `letta/orm/agent.py:71`). The `LettaAgentV3._checkpoint_messages` method (`letta/agents/letta_agent_v3.py:758-816`) is the only place messages are written during a step.

**Failure semantics are explicit**: an exception inside `_step` skips message persistence ("rollback to previous state") (`letta/agents/letta_agent_v3.py:1513`), updates the step row with `StepStatus.FAILED` + `error_type` + `error_data` + traceback (`letta/services/step_manager.py:368-414`), and fires `WebhookService.notify_step_complete` (`letta/services/step_manager.py:412`). Cancellation takes a similar but distinct path: `RunManager.cancel_run` (`letta/services/run_manager.py:619-782`) marks the run `cancelled`, then calls `AgentLoop._checkpoint_messages` to write a synthetic approval-denial + tool-error message pair so the next agent invocation resumes with a consistent `message_ids` list rather than an orphaned tool-call reference.

Storage is **production-ready Postgres/SQLite via SQLAlchemy** with full migration history (`letta/alembic/versions/`), composite indexes on `(agent_id, conversation_id, sequence_id)`, `(run_id, sequence_id)`, and `(step_id)` (`letta/orm/message.py:27-37`). Optional Redis stream caching is TTL-bounded but not pruned by default; the system has **no built-in retention policy** for `messages`, `steps`, or `runs`. Pruning exists only for Redis SSE buffers (`stream_ttl` + `max_stream_length`) and isolated OAuth sessions (`letta/services/mcp_manager.py:1071-1087`).

## Rating

**5 / 10 — Present and durable, but missing core checkpoint semantics.**

The system has explicit, well-named checkpoint primitives (`Run`, `Step`, `StepMetrics`, `_step_checkpoint_start/finish`, `agent.message_ids`), transactional boundaries, failure-rollback semantics, lifecycle states, a fork path for conversations, and an SSE-replay resume path. The relational schema is mature (alembic migrations, composite indexes, organization/project scoping). What keeps the rating out of the 7-8 band: (a) **no `parent_step_id` / `parent_run_id` column** anywhere — parent–child relationships are encoded only by FK chain `agent_id → run_id → step_id → messages(step_id, run_id)` with no branching semantics; (b) **no time-travel API** — `ConversationManager.fork_conversation` (`letta/services/conversation_manager.py:104-172`) is the only branch primitive and operates at the conversation, not step, level; (c) **message_ids is a JSON column on `Agent`** (`letta/orm/agent.py:71`) — flagged in-source as "dangerously flexible with the JSON type" and as a known TODO ("This should be a separate mapping table"), which makes in-context state mutation a JSON-rewrite rather than a relational update; the relational `conversation_messages` table exists but only applies when a `conversation_id` is supplied; (d) **no pruning/retention for `runs`, `steps`, `messages`** — `delete_run` (`letta/services/run_manager.py:307-315`) hard-deletes with cascade but is not exposed via the REST API and no scheduled job prunes old data; (e) **no replay/replay-from-step API** — clients can only continue from the head of `agent.message_ids`; (f) **two parallel in-context mechanisms coexist** (`agent.message_ids` JSON vs `conversation_messages` junction, `letta/agents/letta_agent_v3.py:788-816`) and the code is full of branch-by-flag (`if self.conversation_id:`) without a clear migration story. The model is sufficient for crash-recovery-style continuity and human-in-the-loop pause/resume via approval messages, but not for snapshot-rewind or branching.

## Answers to Dimension Questions

1. **When is a checkpoint written?**
   - **Run row**: created up-front via `RunManager.create_run` (`letta/services/run_manager.py:47-90`) before any LLM traffic, with paired `RunMetrics` row. Updated to `running` once the agent loop has started (`letta/services/streaming_service.py:444-449`) and to terminal `completed`/`failed`/`cancelled` in `RunManager.update_run_by_id_async` (`letta/services/run_manager.py:317-481`).
   - **Step row**: created at the start of each agent step via `_step_checkpoint_start` (`letta/agents/letta_agent_v2.py:940-966`, mirrored in `letta/agents/letta_agent_v3.py:1024`, `:1038`) with `status=StepStatus.PENDING` (`letta/services/step_manager.py:96-173`). Finalized to `SUCCESS`/`FAILED`/`CANCELLED` in `_step_checkpoint_finish` (`letta/agents/letta_agent_v2.py:987-1034`) and post-step exception handlers (`letta/agents/letta_agent_v3.py:1551-1592`).
   - **Messages**: persisted only at the end of a successful step via `_checkpoint_messages` (`letta/agents/letta_agent_v3.py:758-816`). On any exception inside `_step`, messages are NOT persisted ("rollback to previous state", `letta/agents/letta_agent_v3.py:1511-1515`); the step is marked `FAILED` with traceback (`letta/services/step_manager.py:368-414`).
   - **In-context pointer**: rewritten after each successful step via `AgentManager.update_message_ids_async` (`letta/services/agent_manager.py:997-1020`) or `ConversationManager.update_in_context_messages` (`letta/services/conversation_manager.py:750-797`), depending on whether `conversation_id` is provided.
   - **Compaction**: triggers a mid-step `compact()` call (`letta/agents/letta_agent_v3.py:1241-1272`, `:1464-1505`) which persists a single summary `Message` and rewrites the in-context list — a checkpoint-adjacent event that replaces earlier messages with one synthesized one.
   - **Cancellation**: `RunManager.cancel_run` (`letta/services/run_manager.py:619-782`) marks the run `cancelled` and, if an approval request was pending, persists synthetic approval-denial + tool-return messages so the next agent invocation finds a consistent `message_ids` tail.
   - **Stream chunks (Redis)**: per-chunk via `RedisSSEStreamWriter.write_chunk` (`letta/server/rest_api/redis_stream_manager.py:92-130`) in background mode; flushed every 50 chunks / 0.5s / on terminal (`letta/server/rest_api/redis_stream_manager.py:123-128`).
   - **OTEL trace + StepMetrics**: written via `record_step_metrics_async` (`letta/services/step_manager.py:562-652`) inside `_step_checkpoint_finish` and exception handlers (`letta/agents/letta_agent_v3.py:1581-1590`).

2. **What is inside it?**

No parent-run FK.
   - **Step** (`letta/orm/step.py:20-98`, `letta/schemas/step.py:12-66`): `id` (step- prefix), `origin`, `organization_id`, `provider_id`, `run_id` (FK SET NULL on delete), `agent_id`, `provider_name`, `provider_category`, `model`, `model_handle`, `model_endpoint`, `context_window_limit`, `completion_tokens`, `prompt_tokens`, `total_tokens`, `cached_input_tokens`, `cache_write_tokens`, `reasoning_tokens`, `completion_tokens_details`, `prompt_tokens_details`, `stop_reason`, `tags` (JSON list), `tid`, `trace_id`, `request_id`, `feedback`, `error_type`, `error_data`, `status`. No parent-step FK.
   - **Message** (`letta/orm/message.py:23-96`, `letta/schemas/message.py`): `id`, `role`, `text`, `content` (list of MessageContent), `model`, `name`, `tool_calls`, `tool_call_id`, `step_id` (FK SET NULL on delete), `run_id` (FK SET NULL on delete), `otid` (offline threading id for idempotency), `tool_returns`, `group_id`, `sender_id`, `batch_item_id`, `conversation_id` (FK SET NULL on delete), `is_err`, `approval_request_id`, `approve`, `denial_reason`, `approvals`, `sequence_id` (bigint unique monotonic).
   - **RunMetrics** (`letta/orm/run_metrics.py:19-50`): `id` (== `run_id`, PK + FK CASCADE), `run_start_ns`, `run_ns`, `num_steps`, `tools_used` (JSON list of tool IDs).
   - **StepMetrics** (`letta/orm/step_metrics.py:20-72`): `id` (== `step_id`, PK + FK CASCADE), `organization_id`, `provider_id`, `run_id` (FK SET NULL on delete), `step_start_ns`, `llm_request_start_ns`, `llm_request_ns`, `tool_execution_ns`, `step_ns`, `base_template_id`, `template_id`.

3. **Can it restore execution or only data?**
   - **Data restoration (yes, fully)**: reloading an agent by `id` rebuilds its in-context state from `Agent.message_ids` JSON list (`letta/orm/agent.py:71`, `letta/schemas/agent.py:78`), hydrating each `Message` row by id. Conversation mode reads ordered message IDs from `conversation_messages` ordered by `position` (`letta/services/conversation_manager.py:617-630`).
   - **Execution restoration (partial)**: there is **no `restore_from_run(run_id)` or `restore_from_step(step_id)` API**. Execution always resumes from the **head** of `agent.message_ids` (or the conversation's tail). The next `step()` call starts a new `Step` row (PENDING) and runs the agent loop from the current in-context messages — already-executed steps are not re-run.
   - **Stream reconnection (yes, but only for in-flight runs)**: `redis_sse_stream_generator` (`letta/server/rest_api/redis_stream_manager.py:463-526`) replays Redis-stored SSE chunks from a `starting_after` cursor. This is **transport-level replay** of an active background run, not agent-state replay. The window is bounded by `stream_ttl=10800s` (`letta/server/rest_api/redis_stream_manager.py:42-43`).
   - **HITL pause/resume (yes, by design)**: when an `approval_request` Message is the last in-context item (`is_approval_request()`), the agent loop terminates with `StopReasonType.requires_approval` (`letta/agents/letta_agent_v3.py:1706-1709`, `letta/agents/letta_agent_v2.py:1138-1153`). The next client invocation provides `ApprovalCreate` (`letta/server/rest_api/utils.py:175-227`), which converts to an `approval_response` Message; `_maybe_get_approval_messages` (`letta/agents/helpers.py:524-528`) picks up the persisted pair and the loop continues without re-running completed tool calls.
   - **No replay-with-alternate-paths**: there is no way to rewind a `Run` to a previous `Step` and retry with different inputs.

4. **Can checkpoints branch?**
   - **Conversation fork (one level)**: `ConversationManager.fork_conversation` (`letta/services/conversation_manager.py:103-172`) and `fork_default_conversation` (`:174-219`) create a new `Conversation` row that **shares the same `Message` rows** via the `conversation_messages` junction table (`letta/orm/conversation_messages.py:30-31` unique constraint makes the junction 1:1 per message). Subsequent writes to either conversation fork the timeline: new messages get appended to whichever conversation produced them. This is a **soft branch at the conversation level** — both forks retain read access to the shared ancestor messages but accumulate divergent tails.
   - **No step-level branching**: there is no `parent_step_id` column on `Step`, no `parent_run_id` on `Run`, no `get_state_history(run_id)` API, no `fork_from_step(step_id, new_input)` operation.
   - **No message-tree versioning**: every persisted `Message` has a unique `id` (uuid4) and a monotonic `sequence_id`. A message is not versioned; updating it via `MessageManager.update_message_by_letta_message_async` (`letta/services/message_manager.py:664-...`) replaces in place.

5. **Are checkpoints pruned safely?**
   - **Manual delete only**: `RunManager.delete_run` (`letta/services/run_manager.py:306-315`) hard-deletes a run; cascade rules on the ORM (`letta/orm/run.py:76-77`: `steps` and `messages` cascade `all, delete-orphan`) remove associated rows. No REST endpoint exposes this — the only caller is internal.
   - **No scheduled retention job**: no background task sweeps old `runs`/`steps`/`messages`. `_log_step` / `_verify_run_access` (`letta/services/step_manager.py:654-682`) are ad-hoc, not periodic. `ConversationManager.delete_conversation` (`letta/services/conversation_manager.py:554-609`) soft-deletes a conversation and only hard-deletes `Message` rows that are not referenced by any other active conversation — safe against shared-reference corruption.
   - **Redis SSE auto-prune**: `RedisSSEStreamWriter` caps each stream at `max_stream_length=10000` entries with `XADD MAXLEN ~` (`letta/server/rest_api/redis_stream_manager.py:146`) and sets `EXPIRE stream_ttl` (`:148`), so the SSE buffer is bounded but the persistent tables are not.
   - **OAuth session pruning**: `cleanup_expired_oauth_sessions` (`letta/services/mcp_manager.py:1071-1087`, mirrored in `letta/services/mcp_server_manager.py:1267-1283`) — unrelated to agent checkpoints, mentioned only because it is the closest analog in the codebase.
   - **Compaction as a soft-prune**: `compact()` (`letta/agents/letta_agent_v3.py:1241`, `:1464`) replaces a range of in-context messages with one summary message. The originals are **NOT deleted**; they remain queryable by id (`MessageManager.get_message_by_id_async`). This is "logical pruning of context" not "physical pruning of storage".
   - **Test guard for orphaned step rows on cancellation**: there is no automated test verifying that a failed-then-cancelled run leaves a clean `runs`/`steps` table state.

## Architectural Decisions

- **Append-only event log over snapshots**: each agent step writes one row to `steps`, one row to `step_metrics`, and one or more rows to `messages`. There is no "snapshot blob" — current state is the head of `agent.message_ids` (or `conversation_messages` ordered by `position`). This keeps the schema normalized and makes it cheap to inspect history, at the cost of recomputing context from N rows on every agent load.
- **Layered checkpoint scope**: a `Run` is the top-level envelope; `Step` is the unit of execution; `Message` is the unit of conversation state; `RunMetrics` and `StepMetrics` are denormalized 1:1 rows for fast aggregation; `conversation_messages` is the junction that lets the same `Message` participate in multiple conversations (forks).
- **Explicit "rollback on exception" semantics** (`letta/agents/letta_agent_v3.py:1511-1540`): the agent loop deliberately skips message persistence on uncaught exception so `agent.message_ids` reverts to the pre-step state. The `Step` row, however, is marked `FAILED` with `error_type` + `error_data` + traceback (`letta/services/step_manager.py:368-414`) — execution metadata persists even when conversation state does not.
- **JSON-column "in-context" list on Agent** (`letta/orm/agent.py:71`): the agent row carries an ordered JSON array of message IDs as a denormalized cache, replacing a would-be `agent_in_context_messages` junction table. The inline comment at `letta/orm/agent.py:69-70` flags this as "TODO: This should be a separate mapping table. This is dangerously flexible with the JSON type." The relational alternative (`conversation_messages`) is used only when a `conversation_id` is supplied, leaving a permanent split-brain.
- **Dual in-context mechanisms (Agent vs Conversation)**: when `conversation_id` is supplied, in-context tracking goes through `conversation_messages`; otherwise through `agent.message_ids`. The code branches by flag in `_checkpoint_messages` (`letta/agents/letta_agent_v3.py:788-816`) and `ConversationManager.update_in_context_messages` (`:750-797`). The two paths must be kept in sync manually.
- **OTID-based idempotency for background streaming**: every inbound `MessageCreate` carries a `Message.otid` (`letta/schemas/message.py:137`) used as a dedupe key. `set_otid_run_mapping` (`letta/data_sources/redis_client.py:252-...`, used at `letta/services/streaming_service.py:341-347`) stores `otid → run_id` so a retried request recovers the existing run stream via `try_recover_duplicate_request` (`:83-118`).
- **Pause/resume via approval messages**: instead of a "pause token" or explicit `pause_run` API, pause is implemented by emitting an `approval_request` Message that becomes the tail of `message_ids`; the agent loop terminates with `requires_approval`; the next invocation appends an `approval_response` Message and continues. The `ApprovalCreate`/`ToolCallDenial` schema (`letta/server/rest_api/utils.py:175-227`) carries tool-call approval, denial reason, and inline tool returns.
- **Run-level metadata JSON** (`letta/orm/run.py:46`, `letta/services/run_manager.py:60-62`): `RunUpdate` merges into `metadata_` rather than overwriting it ("This is important for streaming/background flows where different components update different parts of metadata"), supporting `run_type`, `error` payload, and `result` payload as separate sub-keys.
- **Sequence-id-based message ordering**: `Message.sequence_id` (`letta/orm/message.py:86-91`) is a server-default bigint unique column providing global insertion order. Indexes `(agent_id, conversation_id, sequence_id)`, `(run_id, sequence_id)`, `(agent_id, sequence_id)` (`letta/orm/message.py:29-35`) make cursor-paginated reads fast.
- **Webhooks for terminal step events**: `StepManager.update_step_success_async`, `update_step_error_async`, and `update_step_cancelled_async` each fire `WebhookService.notify_step_complete` (`letta/services/step_manager.py:412`, `:474`, `:553`) so external systems can react to per-step terminal state without polling.
- **Run callback URL**: a Run can carry `callback_url` (`letta/orm/run.py:60`, `letta/services/run_manager.py:336-339`, `:484-510`) which is POSTed once with the full `LettaResponse` on terminal state. Callback status is recorded back into `callback_sent_at` / `callback_status_code` / `callback_error`.
- **Two parallel LLM adapters**: `SimpleLLMRequestAdapter` (`letta/adapters/simple_llm_request_adapter.py`) for synchronous requests, `SimpleLLMStreamAdapter` (`letta/adapters/simple_llm_stream_adapter.py`) for streaming. Both call `_step_checkpoint_*` at the same boundaries; checkpoint behavior is independent of adapter choice.
- **Compaction is mid-step, not post-step**: `compact()` is called inside `_step` either when context-window is exceeded mid-LLM (`letta/agents/letta_agent_v3.py:1217-1294`) or post-step if usage crosses the trigger threshold (`:1438-1509`). A persisted summary message is appended and the in-context list is rebuilt; the original messages are not deleted.

## Notable Patterns

- **StepProgression as a single-int state machine** (`letta/schemas/step.py:68-74`): the agent loop threads a `StepProgression` enum value through checkpoint calls, allowing the `finally` block (`letta/agents/letta_agent_v3.py:1541-1592`, `letta/agents/letta_agent_v2.py:666-720`) to know exactly where the step was when an exception fired, and react with the appropriate `update_step_error_async` / `update_step_stop_reason` call. This avoids scattering try/except inside the step body.
- **Pre-creation of Step row with PENDING status** (`letta/services/step_manager.py:96-173`): `_step_checkpoint_start` inserts a `Step` row with `status=PENDING` and a captured `trace_id` (from current OTEL span) **before** any LLM call. This means if the agent dies mid-step, there is still a row to update (via the orphan-recovery / timeout sweeper if one exists — none observed in code; comments at `letta/services/step_manager.py:248-249` warn that orphaned steps are detected only by application-level reconciliation).
- **PENDING-then-finalized lifecycle**: `PENDING` → `SUCCESS` (normal), `FAILED` (exception), or `CANCELLED` (explicit cancellation) — never overwritten. The `StepMetrics` row has the same id as the Step and is written in parallel via `record_step_metrics_async`.
- **`Run` + `RunMetrics` 1:1 with shared PK** (`letta/orm/run_metrics.py:25-29`): RunMetrics uses `id = run.id` as PK and FK to `runs.id` with `ondelete=CASCADE`, so deleting a run drops its metrics. Same pattern for `Step` + `StepMetrics`.
- **ConversationMessage junction with `position` integer** (`letta/orm/conversation_messages.py:52-56`): allows in-conversation messages to be reordered (e.g., summary inserted before approval request — see `ConversationManager.update_in_context_messages` at `letta/services/conversation_manager.py:750-797`). Position is rewritten on every in-context update to reflect the canonical order.
- **Async iteration over `_step` with pre-checkpoint early-exit**: `letta/agents/letta_agent_v3.py:1029-1035` checks `_check_run_cancellation(run_id)` before starting each step, allowing cancellation to short-circuit before any DB write. The same hook exists in `letta/agents/letta_agent_v2.py:505-509`.
- **OTEL-traced sub-checkpoints**: `_request_checkpoint_start/ttft/finish` (`letta/agents/letta_agent_v2.py:915-938`) emit OTEL spans for "time_to_first_token" with LLM config attributes, so per-request timing data is available even without persisted steps. This is the "checkpoint as observability signal" pattern.
- **Webhook on every step terminal** (`letta/services/step_manager.py:412`, `:474`, `:553`): `WebhookService.notify_step_complete(step_id)` is called outside the DB session (after commit) for success, failure, and cancellation paths. Avoids webhook failures aborting the checkpoint write.
- **`fork_conversation` shares `Message` rows** (`letta/services/conversation_manager.py:103-172`): the fork does not duplicate message data — only the junction rows in `conversation_messages`. This means both forks see the same historical messages, but new messages appended after the fork are visible only to the conversation that produced them. The "safe delete" path in `delete_conversation` (`letta/services/conversation_manager.py:584-597`) explicitly checks for cross-conversation references before soft-deleting messages.
- **Streaming-resume deduplication via OTID → run_id mapping** (`letta/data_sources/redis_client.py:252-...`, `letta/services/streaming_service.py:341-347`): a client retrying a streaming request with the same OTID gets back a reader attached to the existing run instead of starting a new one. This is checked both pre-lock (`:286-295`) and as a second-chance post-lock fallback (`:305-318`).
- **Cancellation-aware stream wrapper** (`letta/server/rest_api/streaming_response.py:162-...`): a stream wrapper periodically checks the cancellation flag for the run and injects `RunCancelledException` into the stream generator, allowing graceful mid-stream cancellation without dropping the [DONE] marker.

## Tradeoffs

- **No step-level rewind / replay**: state is the head of `agent.message_ids`; there is no API to fork from `step_id=N`. To "branch from step 5" today, an operator would need to export messages up to step 5, create a new agent or conversation, and inject them as the starting in-context — manual and brittle.
- **JSON-column in-context pointer**: `Agent.message_ids` JSON column (`letta/orm/agent.py:71`) is convenient but bypasses relational integrity (no FK to `messages.id`), cannot enforce ordering at the DB level, and makes transactional in-context updates require a JSON rewrite. The relational alternative (`conversation_messages`) exists but is conditional on `conversation_id`. The in-source TODO at `letta/orm/agent.py:69-70` acknowledges this.
- **Permanent dual-write path for in-context state**: agent-mode and conversation-mode code paths must be kept in sync manually (compare `_checkpoint_messages` at `letta/agents/letta_agent_v3.py:788-816` to `ConversationManager.update_in_context_messages` at `:750-797`). A bug in one will not be caught by the other.
- **No retention/TTL on `runs`/`steps`/`messages`**: storage grows monotonically. Only the SSE Redis buffer and OAuth sessions have TTLs (`letta/server/rest_api/redis_stream_manager.py:42-43`, `letta/services/mcp_manager.py:1071-1087`). Operators running long-lived deployments need external retention (manual SQL or scheduled job).
- **`delete_run` not exposed via REST API**: the only way to delete a run is via `RunManager.delete_run` (`letta/services/run_manager.py:306-315`), used only internally. There is no `DELETE /v1/runs/{run_id}` route — only `delete_run_by_id` is registered in the route module (`letta/server/rest_api/routers/v1/runs.py:310-321`).
- **No rollback of partial compaction**: `compact()` (`letta/agents/letta_agent_v3.py:1241`, `:1464`) appends a summary Message and rewrites in-context. If a downstream tool call after compaction fails, the summary Message remains — there is no automatic undo. The original messages are still in the table but not in `message_ids`.
- **Webhook fires outside the DB transaction** (`letta/services/step_manager.py:412-414`, `:474-476`, `:553-555`): each terminal webhook is fired after the DB commit. A webhook crash will not roll back the Step, but a webhook delivered before the DB commit (impossible here because of the order) would orphan state.
- **Cancellation adds denial messages to the message log** (`letta/services/run_manager.py:748-776`): a cancelled run with a pending approval writes a synthetic approval denial + tool-error message pair. This is intentional — it keeps the conversation coherent — but it pollutes the message log with synthetic content that callers must filter out (`Message.is_approval_request()` etc.).
- **`approved_partial` on `create_many_messages_async`** (`letta/services/message_manager.py:485-507`): callers can opt into a "skip duplicates" mode for re-running agents. Useful for idempotent retries, but means an interrupted agent can be safely restarted without producing duplicate rows. This pattern is rare and not all callers use it.
- **`StepProgression` integer auto values** (`letta/schemas/step.py:68-74`): the enum uses `auto()` so values are implementation-defined. Downstream code must compare by name (`StepProgression.FINISHED`), not by integer. Most code follows this, but it is a footgun.
- **Message persistence skipped on exception means the `Step` row can be `FAILED` while no messages were added**: there is no automatic rollback of the `Step` row's `completion_tokens` etc. The `finally` block at `letta/agents/letta_agent_v3.py:1551-1592` records partial step metrics, but the Step row's status flips to `FAILED` with the error_type/error_data fields populated.
- **No "resume from step X with new input" API**: clients can only resume a `Run` via the stream reconnect endpoint (`letta/server/rest_api/routers/v1/runs.py:324-411`), which replays SSE chunks — not the agent loop. To inject new user input mid-stream, clients send a new `step()` call with the next message appended to in-context, which starts a **new Run** (or continues the current one if `conversation_id` is set).
- **Stream chunk TTL of 3 hours**: clients who disconnect for longer than `stream_ttl` (`letta/server/rest_api/redis_stream_manager.py:42-43`) cannot recover the stream; the run remains in `runs` table but the SSE buffer is gone. The retrieval endpoint at `letta/server/rest_api/routers/v1/runs.py:357-411` rejects runs older than 3 hours with `LettaExpiredError`.

## Failure Modes / Edge Cases

- **Orphaned `Step` row if process dies before `_step_checkpoint_finish`**: a PENDING step row stays in the table. No automated sweeper. Reconciliation requires a custom job querying `steps WHERE status = 'pending' AND created_at < NOW() - INTERVAL ...` and either marking them `FAILED` or retrying.
- **Mid-compaction crash leaves a summary message with no in-context update**: if `compact()` (`letta/agents/letta_agent_v3.py:1241`) returns successfully and the summary message is persisted via `_checkpoint_messages` (`:1267-1272`) but then the agent process dies before the in-context list is finalized, the next agent load will see the summary message as the tail of `message_ids` without the rebuilt context. The next step's `_refresh_messages` (`letta/agents/letta_agent_v2.py:760-782`) will rebuild the system prompt from the current state, which may be incomplete.
- **Stream chunk race on duplicate OTID**: `try_recover_duplicate_request` (`letta/services/streaming_service.py:83-118`) is called both pre-lock and post-lock. Between the two, a new run_id mapping may be stored (`letta/services/streaming_service.py:341-347`); a race can produce a recovery response for a run that is no longer current. The post-lock fallback at `:305-318` polls up to 3 times with backoff before giving up.
- **Cancellation denial message references tool calls from a different step**: `RunManager.cancel_run` (`letta/services/run_manager.py:683-768`) collects all pending tool calls (including from prior steps) into one denial batch. If the agent has been running for many steps, this single denial could be very large.
- **StepMetrics partial writes during exception**: the `finally` block at `letta/agents/letta_agent_v3.py:1581-1590` computes `step_ns = get_utc_timestamp_ns() - step_metrics.step_start_ns` and calls `_record_step_metrics`. If `_record_step_metrics` itself raises, the exception is caught at `:1591-1592` and logged but not re-raised, so partial step metrics may be missing.
- **JSON message_ids update loses ordering if the same message appears twice**: the update at `letta/services/agent_manager.py:1016` writes `message_ids = message_ids` directly, so dedup is the caller's responsibility. `_checkpoint_messages` (`letta/agents/letta_agent_v3.py:758-816`) builds the list from `input_messages_to_persist + new_messages` and dedups implicitly because each message has a unique id; but a caller that passes the same message twice will store both ids.
- **Sequence-id atomicity for SQLite**: `set_sequence_id_for_sqlite` (`letta/orm/message.py:130-264`) uses a custom `message_sequence` table with atomic increments because SQLite lacks PostgreSQL's sequence generator. If the trigger fails (e.g., the auxiliary table is missing), the `Message.sequence_id` ends up `None` and downstream reads that rely on ordering (`get_run_messages`, etc.) will fail or return non-deterministic results.
- **OTID replay race across two clients**: if two clients send messages with the same OTID at exactly the same time, both may create separate runs before either's OTID is mapped. The OTID-to-run_id mapping stored at `letta/services/streaming_service.py:341-347` is a `SET` (last write wins), so the second client may incorrectly recover the first client's run.
- **Webhook retry on transient failure**: `WebhookService.notify_step_complete` is fire-and-forget at the call sites; no retry/backoff is observable. A transient webhook receiver outage will lose the notification.
- **`delete_run` cascades but `cancel_run` does not**: cancelling a run leaves all rows intact (just updates `status` and `stop_reason`); deleting a run hard-deletes everything. There is no "soft delete with cascade" or "archive" path.
- **`Conversation.fork_conversation` skips system message** (`letta/services/conversation_manager.py:148-150`): "Skip the system message (always position 0); the fork gets its own." This means the fork's system prompt may diverge from the original. The `compile_and_save_system_message_for_conversation` call at `:165-170` recomputes from current memory blocks, which may differ from the original's.
- **Redis stream key collision**: `RedisSSEStreamWriter` keys are `sse:run:{run_id}` (`letta/server/rest_api/redis_stream_manager.py:139`). If `run_id` ever collides (uuid4 collision is statistically negligible but possible), two runs would share the same stream. Run IDs are uuid4 with `run-` prefix (`letta/orm/run.py:37`).
- **Compaction summary message competes with approval request** (`letta/services/conversation_manager.py:765-766`): "This is critical for correctness: when summarization inserts a summary message that needs to appear before an approval request, the positions must reflect the intended order, not the insertion order." The fix relies on rewriting `position` on every `update_in_context_messages` call, which is correct but easy to break.

## Future Considerations

- **Relational replacement for `Agent.message_ids` JSON**: drop the JSON column and use a `agent_in_context_messages(agent_id, conversation_id, message_id, position, in_context)` mapping table unconditionally. The `conversation_messages` table is the prototype; extending it to a default-conversation row per agent would unify the two paths in `_checkpoint_messages` (`letta/agents/letta_agent_v3.py:788-816`).
- **REST `DELETE /v1/runs/{run_id}`**: the endpoint at `letta/server/rest_api/routers/v1/runs.py:310-321` is registered but uses `delete_run_by_id` (likely a typo for `delete_run`). Plumb it through to `RunManager.delete_run` so operators can prune from the UI.
- **Retention job for `runs`/`steps`/`messages`**: add a configurable TTL or row-count cap, with a background sweeper. Could leverage `created_at` + a new `RetentionPolicy` setting. The Redis SSE writer already has the right TTL/maxlen pattern (`letta/server/rest_api/redis_stream_manager.py:42-43`); apply the same to the relational store.
- **Step-level branch API**: add `parent_step_id` column on `Step` + `parent_run_id` on `Run`. Implement `RunManager.fork_run(run_id, from_step_id) → new_run_id` that copies all `Message` rows up to `from_step_id` into a new `Run` with `parent_run_id` set. The junction table pattern at `conversation_messages` shows the storage side is straightforward.
- **`get_run_history(run_id)` API**: traverse `(run_id, sequence_id)` index (`letta/orm/message.py:35`) to return messages in step order. Combine with `RunMetrics.num_steps` (`letta/orm/run_metrics.py:40-44`) for a "what happened in this run" view.
- **PENDING-step sweeper**: a scheduled job that marks `Step.status='pending'` rows older than N minutes as `FAILED` with a synthetic timeout error. Prevents indefinite "PENDING" state from accumulating.
- **Webhook retry/backoff**: `WebhookService.notify_step_complete` should retry with exponential backoff on 5xx responses, with a max-attempts cap. Mirror the pattern in `_dispatch_callback_async` (`letta/services/run_manager.py:484-510`), which already wraps the POST in a try/except.
- **Time-travel / replay UI**: with a `parent_step_id` schema, expose `GET /v1/runs/{run_id}/history` and `POST /v1/runs/{run_id}/replay?from_step_id=X` endpoints. The junction-table pattern at `conversation_messages` shows the storage side is straightforward.
- **Branch via fork_default_conversation + clone agent**: a single API call that creates a new agent with the same in-context messages as an existing agent at a given step. `AgentSerializationManager` already does the import side (`letta/services/agent_serialization_manager.py:720-790`); wiring this to a fork endpoint is incremental.
- **Schema migration for OTID**: `Message.otid` is currently a free-form string (`letta/schemas/message.py:137`). Adding a unique index `(agent_id, otid)` would let OTID serve as a primary idempotency key in addition to `id`.
- **Compaction retry/undo**: track compaction events as separate rows (similar to `BlockHistory` for memory blocks) so a failed compaction can be undone by reading the original messages and re-running `_checkpoint_messages` with the pre-compaction list.

## Questions / Gaps

- **Why is there no `parent_step_id` or `parent_run_id` column?** The schema has every other relation (FK from `Step` to `Run`, from `Message` to `Step`/`Run`), but no self-referential FKs. No evidence found for an explicit decision to omit branching; the code comments suggest it was simply not in scope.
- **What is the operational story for very long-running agents?** No evidence found for: a documented retention policy, a soft-delete + archive path, or a scheduled cleanup job for `runs`/`steps`/`messages`. The Redis SSE buffer has TTL (`letta/server/rest_api/redis_stream_manager.py:42-43`); the relational tables do not.
- **How does `ConversationManager.fork_conversation` interact with `_step_checkpoint_finish`?** The fork code (`letta/services/conversation_manager.py:103-172`) creates a new Conversation + system message + junction rows, but does not call `_checkpoint_messages`. There is no evidence that subsequent `step()` invocations in the forked conversation correctly apply `_refresh_messages` (`letta/agents/letta_agent_v2.py:760-782`).
- **What happens to `agent.last_run_completion` / `last_run_duration_ms` / `last_stop_reason` on `RunManager.cancel_run`?** The update at `letta/services/run_manager.py:400-410` writes these after the run is marked terminal. But if the run is cancelled mid-step, `RunMetrics.run_ns` is computed from `RunMetrics.run_start_ns` (`letta/services/run_manager.py:439-443`), which is set at run creation. There is no evidence that the cancelled run's "actual duration" matches what the user perceived.
- **Is the `WebhookService` retry-safe?** No evidence found for: retry on 5xx, deduplication of webhook deliveries, or signed callbacks. The current pattern (`letta/services/step_manager.py:412`) fires once and relies on the receiver's reliability.
- **What is the exact atomicity guarantee of `RunManager.create_run`?** It creates `Run` + `RunMetrics` rows inside one `db_registry.async_session()` block (`letta/services/run_manager.py:50-86`). The `commit` is at the context manager exit. If the process crashes after the `RunMetrics.create_async` call but before the context manager exits, the `Run` row may exist without a paired `RunMetrics` row, violating the 1:1 invariant.
- **Are there race conditions on `agent.message_ids`?** Two concurrent `step()` calls on the same agent would each read `message_ids`, append their new messages, and write back. The second write would clobber the first, losing one tail of the conversation. No evidence found for an explicit lock or optimistic-concurrency check. The conversation lock (`letta/data_sources/redis_client.py:196-247`) protects only the conversation-scoped path.
- **How are SSE chunks batched and what happens if a chunk write fails?** `_flush_run` (`letta/server/rest_api/redis_stream_manager.py:132-163`) buffers chunks and flushes via a pipeline; on failure it re-buffers and re-raises. A persistent Redis outage will cause the background stream processor to spin on retry. No evidence found for a circuit breaker.
- **Does `RedisSSEStreamWriter._flush_task` (`letta/server/rest_api/redis_stream_manager.py:76`) survive process restart?** It is an asyncio task created in `start()` (`letta/server/rest_api/redis_stream_manager.py:72-76`).

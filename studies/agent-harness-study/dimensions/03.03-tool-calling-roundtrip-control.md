# Dimension 03.03: Tool-Calling Roundtrip Control

## Purpose

Analyze how tool calls are detected, executed, validated, retried, and returned to the model.

## Steps

1. Locate tool-call extraction.
2. Identify argument validation.
3. Inspect execution wrapper.
4. Inspect result formatting.
5. Check whether invalid tool calls trigger repair.
6. Check whether tool errors continue the loop.

## Evidence to Capture

- Tool call parsers
- Function call schemas
- Argument validators
- Tool result messages
- Invalid-tool recovery logic

## Questions

1. How are tool calls represented?
2. What happens if arguments are invalid?
3. Are tool results structured?
4. Are tool errors visible to the model?
5. Can tool calls be approved before execution?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can malformed tool calls be repaired without crashing the run?

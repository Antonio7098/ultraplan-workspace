# Dimension 14.01: Human-in-the-Loop Trigger Policy

## Purpose

Study what triggers human review: tool risk, uncertainty, failed validation, policy violation, budget, or user request.

## Steps

1. Identify conditions that trigger human review.
2. Check whether triggers are configurable.
3. Inspect whether multiple trigger conditions can combine.
4. Check whether users can request human review explicitly.
5. Check whether trigger events are recorded.

## Evidence to Capture

- Trigger condition code
- HITL config
- User request handlers
- Trigger audit logs

## Questions

1. What triggers human review?
2. Are triggers configurable?
3. Can users request human review?
4. Are trigger decisions auditable?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Does the system know when to ask for help?

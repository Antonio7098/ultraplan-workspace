# Dimension 13.03: Failure Visibility

## Purpose

Study whether failures are visible to the model, user, developer, and operator at the right level of detail.

## Steps

1. Check what the model sees on failure.
2. Check what the user sees on failure.
3. Check what the developer sees in logs/traces.
4. Check what the operator sees in monitoring.
5. Check whether failure detail levels are configurable.

## Evidence to Capture

- Model error messages
- User-facing error pages
- Log formats
- Trace error events
- Monitoring dashboards

## Questions

1. Is the model informed of failures?
2. Is the user informed appropriately?
3. Can developers debug failures?
4. Can operators detect failure patterns?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Does each stakeholder see the right amount of failure information?

# Dimension 24.03: API Versioning and Compatibility

## Purpose

Study how public APIs, schemas, protocols, persisted contracts, and extension points change without breaking users.

## Steps

1. Identify versioned APIs, schema versions, protocol versions, migration paths, and deprecation policy.
2. Inspect how breaking changes are documented, detected, staged, and released.
3. Check whether backwards compatibility is tested for clients, plugins, saved state, traces, prompts, and tool schemas.
4. Look for feature flags, capability negotiation, or version negotiation.
5. Assess whether compatibility expectations differ across SDK, CLI, server, and extension surfaces.

## Evidence to Capture

- Version fields and compatibility policy
- Changelogs, migration guides, deprecation notices
- Backwards-compatibility tests
- Capability or version negotiation code
- Schema migration and contract evolution logic

## Questions

1. Which APIs are stable, experimental, deprecated, or internal?
2. How are users warned before breaking changes?
3. Are old clients, plugins, traces, or persisted artifacts still usable?
4. Does compatibility rely on policy alone or executable tests?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can a production integration upgrade safely without auditing every internal change?

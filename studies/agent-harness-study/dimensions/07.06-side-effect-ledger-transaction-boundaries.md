# Dimension 07.06: Side-Effect Ledger and Transaction Boundaries

## Purpose

Study whether side effects are recorded, confirmed, rolled back, or compensated.

## Steps

1. Identify side-effecting tools.
2. Check whether side effects are logged before/after execution.
3. Inspect transaction boundaries.
4. Check rollback/compensation logic.
5. Check user-facing side-effect summaries.
6. Inspect audit trail.

## Evidence to Capture

- Side-effect logs
- Transaction wrappers
- Compensation handlers
- Audit events
- Confirmation records
- External IDs

## Questions

1. What external changes did the agent make?
2. Are side effects auditable?
3. Can failed side effects be compensated?
4. Are external IDs stored?
5. Are users shown what changed?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can the system answer 'what did the agent change?'

# Dimension 01.09: Delivery Guarantees and Idempotency

## Purpose

Study whether execution is at-most-once, at-least-once, effectively-once, or best-effort.

## Steps

1. Identify whether jobs/events/messages can be redelivered.
2. Look for idempotency keys.
3. Inspect deduplication logic.
4. Identify side-effecting operations.
5. Check whether retries can repeat side effects.
6. Document delivery semantics per subsystem.

## Evidence to Capture

- Job IDs
- Idempotency stores
- Deduplication tables
- Queue acknowledgement logic
- Event sequence numbers
- Transaction boundaries

## Questions

1. Can the same work run twice?
2. Is duplicate execution safe?
3. Which side effects are idempotent?
4. Does the system claim exactly-once semantics?
5. Are guarantees documented?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> If a worker crashes after doing the work but before acknowledging it, what happens?

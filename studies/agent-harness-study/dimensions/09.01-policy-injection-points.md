# Dimension 09.01: Policy Injection Points

## Purpose

Study where governance rules enter the system.

## Steps

1. Identify where policies are defined: prompt, runtime code, tool metadata, eval configuration, deployment config, or external policy engine.
2. Check whether policies can be updated without code changes.
3. Inspect policy precedence when multiple sources conflict.
4. Check whether policies are versioned and audited.

## Evidence to Capture

- Policy config files
- Prompt-based rules
- Tool metadata annotations
- External policy engine integration
- Policy versioning

## Questions

1. Where do governance rules live?
2. Can policies be updated at runtime?
3. What happens when policies conflict?
4. Are policy changes audited?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can a governance rule be added without modifying agent code?

# Dimension 04.06: Tool Result Contract and Error Envelope

## Purpose

Analyze how tool outputs and errors are represented.

## Steps

1. Inspect tool return types.
2. Identify success/error envelopes.
3. Check whether errors are model-readable.
4. Check whether internal errors are redacted.
5. Inspect whether large results are summarized.
6. Check whether artifacts are linked.

## Evidence to Capture

- Tool result classes
- Error envelopes
- Output serializers
- Redaction logic
- Artifact references
- Tool result messages

## Questions

1. Is every tool result structured?
2. Can the model distinguish success from failure?
3. Are stack traces hidden from users/models?
4. Are large results handled safely?
5. Can tool results be replayed?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can a failed tool call become useful feedback rather than a crash?

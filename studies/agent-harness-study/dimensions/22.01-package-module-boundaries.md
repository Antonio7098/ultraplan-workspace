# Dimension 22.01: Package and Module Boundaries

## Purpose

Study whether runtime, tools, memory, evals, tracing, UI, and provider adapters are cleanly separated.

## Steps

1. Identify top-level package structure.
2. Check dependency direction between modules.
3. Inspect whether modules can be used independently.
4. Check for circular dependencies.
5. Check whether internal APIs are distinguished from public APIs.

## Evidence to Capture

- Package structure
- Module dependency graphs
- Module boundaries
- API visibility annotations
- Separation tests

## Questions

1. Are modules cleanly separated?
2. Do dependencies flow in one direction?
3. Can modules be used independently?
4. Are public APIs distinguished from internal ones?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can you use the tool system without pulling in the entire runtime?

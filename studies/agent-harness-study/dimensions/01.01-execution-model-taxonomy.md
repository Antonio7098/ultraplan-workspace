# Dimension 01.01: Execution Model Taxonomy

## Purpose

Identify the fundamental execution model: loop, graph, event stream, queue, recursive evaluator, workflow engine, or hybrid.

## Steps

1. Locate the main runtime entry point.
2. Identify what causes execution to advance.
3. Classify the model: step loop, ReAct loop, graph/superstep model, event-driven router, queue worker, workflow task state machine, recursive evaluator, streaming turn loop.
4. Note whether the repo uses one model or several layered models.

## Evidence to Capture

- Runtime entrypoint
- Main loop implementation
- Graph executor
- Queue worker
- Event handler
- Scheduler
- Workflow/task processor

## Questions

1. What is the primary execution model?
2. Is it explicit or emergent?
3. Does the model match the product shape?
4. Is the model easy to explain to a new contributor?
5. Does the system mix models cleanly or accidentally?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can you explain what advances execution in one sentence?

## Output

Write findings to `reports/repo/01.01-execution-model-taxonomy/{repo-name}.md`

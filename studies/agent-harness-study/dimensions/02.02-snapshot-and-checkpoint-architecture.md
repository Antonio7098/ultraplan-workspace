# Dimension 02.02: Snapshot and Checkpoint Architecture

## Purpose

Study how the system captures point-in-time execution state.

## Steps

1. Identify checkpoint creation points.
2. Determine checkpoint granularity.
3. Determine what is included/excluded.
4. Identify parent/child checkpoint relationships.
5. Check whether checkpoints support resume, time travel, branching, or debugging.
6. Inspect checkpoint storage backends.

## Evidence to Capture

- Checkpointer interfaces
- Snapshot schemas
- Parent checkpoint IDs
- Resume APIs
- Thread/session IDs
- Stored state blobs

## Questions

1. When is a checkpoint written?
2. What is inside it?
3. Can it restore execution or only data?
4. Can checkpoints branch?
5. Are checkpoints pruned safely?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can execution resume from a checkpoint without redoing completed work?

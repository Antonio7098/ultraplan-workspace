# Dimension 10.04: Export, Interoperability, and Observability Backends

## Purpose

Study whether traces can flow to OpenTelemetry, Langfuse, LangSmith, Honeycomb, custom sinks, or local files.

## Steps

1. Identify trace export mechanisms.
2. Check support for OpenTelemetry (OTLP) export.
3. Check support for commercial platforms (Langfuse, LangSmith, etc.).
4. Inspect whether export is configurable at runtime.
5. Check whether trace format is standard or custom.

## Evidence to Capture

- OTLP exporters
- Langfuse/LangSmith SDK integration
- Custom sink implementations
- Export configuration
- Trace format schemas

## Questions

1. Can traces be exported to external backends?
2. Are standard protocols supported?
3. Is export configurable without code changes?
4. Can multiple backends receive traces simultaneously?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can you send traces to your existing observability stack without writing an adapter?

# Dimension 13.02: Retry, Fallback, and Degraded Mode

## Purpose

Study retries, backoff, fallback models, backup providers, degraded behavior, and circuit breakers.

## Steps

1. Identify retry policies and backoff strategies.
2. Check whether fallback models/providers are configured.
3. Inspect circuit breaker or degradation behavior.
4. Check whether degraded mode reduces functionality gracefully.
5. Check whether retry state is persisted.

## Evidence to Capture

- Retry configs
- Backoff implementations
- Fallback model config
- Circuit breaker code
- Degraded mode handlers

## Questions

1. Are retries configurable?
2. Are fallback providers available?
3. Does the system degrade gracefully?
4. Are circuit breakers used to prevent cascading failure?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can the system survive a provider outage without failing all requests?

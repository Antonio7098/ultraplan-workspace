# Dimension 04.04: Tool Context and Dependency Injection

## Purpose

Analyze how tools receive runtime context without depending directly on global state.

## Steps

1. Identify tool context objects.
2. Inspect injected dependencies: user/session IDs, stores, logger, secrets, workspace, permissions, cancellation token.
3. Check whether context is typed.
4. Check whether context differs across tools.
5. Inspect testability.

## Evidence to Capture

- Tool context classes
- Dependency providers
- Runtime context
- Store injection
- Test fixtures

## Questions

1. What context does a tool receive?
2. Is context explicit or global?
3. Are secrets passed safely?
4. Can tools be unit tested?
5. Can context enforce permissions?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

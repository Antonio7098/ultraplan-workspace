# Dimension 02.05: Persistence Durability Tiers

## Purpose

Determine which state survives process death, machine death, deployment, and version change.

## Steps

1. Classify stores: in-memory, file-backed, SQLite, Postgres, object storage, queue, external service.
2. Map state categories to durability tiers.
3. Identify silent no-op persistence.
4. Check production vs test storage.
5. Identify recovery behavior for each tier.

## Evidence to Capture

- Store interfaces
- Concrete store implementations
- Persistence config
- Startup recovery logic
- No-op stores
- Storage docs

## Questions

1. What survives a restart?
2. What survives redeploy?
3. What survives multi-instance operation?
4. What is silently dropped?
5. Is durability configurable?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can operators tell which state is production-durable?

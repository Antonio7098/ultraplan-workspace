# Dimension 02.03: Event Sourcing and Replay State

## Purpose

Analyze whether state is derived from an append-only event log.

## Steps

1. Identify event schemas.
2. Determine whether events are source-of-truth or telemetry.
3. Inspect replay code.
4. Check whether events are immutable.
5. Look for event versioning.
6. Check whether replay is tested.

## Evidence to Capture

- Event log implementation
- History builder
- Rebuilder/replayer
- Sequence numbers
- Event version fields
- Append-only write paths

## Questions

1. Are events authoritative?
2. Can state be rebuilt from events?
3. Are events immutable?
4. Can old events still replay after schema changes?
5. How is replay cost managed?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

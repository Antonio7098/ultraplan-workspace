# Dimension 11.01: Context Selection Policy

## Purpose

Study how the system decides what information enters the model context.

## Steps

1. Identify context assembly logic.
2. Determine which signals are included: conversation history, tool results, retrieved documents, system prompt, user profile, workspace state.
3. Check whether context selection is static or dynamic.
4. Inspect whether the model can request additional context.
5. Check whether sensitive data is filtered before inclusion.

## Evidence to Capture

- Context builders
- Message selectors
- Filter rules
- Retrieval calls
- Policy-based inclusion/exclusion

## Questions

1. What decides what goes into context?
2. Is selection policy explicit or implicit?
3. Can the model influence what it sees?
4. Are sensitive fields redacted?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can the system explain why a particular document was included in context?

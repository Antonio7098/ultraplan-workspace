# Dimension 02.06: Schema Versioning and Migration

## Purpose

Analyze how persisted state evolves over time.

## Steps

1. Search for schema version fields.
2. Inspect migration files.
3. Check JSON blob versioning.
4. Identify deserialization validation.
5. Look for backwards/forwards compatibility strategy.
6. Check whether old state fixtures are tested.

## Evidence to Capture

- schema_version
- Migration scripts
- Pydantic/Zod/protobuf models
- Deserializers
- Compatibility tests
- Old-state fixtures

## Questions

1. Does persisted state carry a version?
2. What happens when old state is loaded?
3. Are migrations automatic or manual?
4. Are JSON blobs versioned?
5. Can old checkpoints/events still be read?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

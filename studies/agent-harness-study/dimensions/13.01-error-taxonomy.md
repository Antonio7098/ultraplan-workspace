# Dimension 13.01: Error Taxonomy

## Purpose

Study whether errors are classified by source: model, provider, tool, validation, policy, context, user, infrastructure, timeout.

## Steps

1. Identify error type definitions.
2. Check whether errors are classified by source category.
3. Inspect whether error types are used for routing and handling.
4. Check whether error taxonomy is documented.

## Evidence to Capture

- Error type enums
- Error classification code
- Error handling dispatch
- Error documentation

## Questions

1. Are errors classified by source?
2. Is the taxonomy used for handling?
3. Are error categories documented?
4. Can new error types be added without breaking existing handling?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can you tell from the error type whether to retry, escalate, or stop?

# Dimension 07.03: Idempotency and Retry Semantics

## Purpose

Study how tool retries avoid repeating dangerous side effects.

## Steps

1. Identify retry wrappers.
2. Classify retryable vs non-retryable failures.
3. Search for idempotency keys.
4. Inspect duplicate detection.
5. Check external side-effect handling.
6. Check whether retries are visible to the model/user.

## Evidence to Capture

- Retry policies
- Idempotency stores
- Tool attempt records
- Error classification
- External API wrappers
- Retry traces

## Questions

1. Which tool failures are retried?
2. Are repeated attempts safe?
3. Is retry state persisted?
4. Are non-idempotent tools protected?
5. Can retries create duplicate side effects?

# Dimension 21.01: Plugin and Extension Points

## Purpose

Study where third parties can extend tools, agents, memory, evals, providers, prompts, policies, and UI.

## Steps

1. Identify extension point interfaces.
2. Check whether plugins can be loaded dynamically.
3. Inspect plugin lifecycle management.
4. Check whether plugin isolation exists.
5. Check whether extension points are documented.

## Evidence to Capture

- Extension interfaces
- Plugin loaders
- Plugin lifecycle hooks
- Plugin isolation mechanisms
- Extension documentation

## Questions

1. What can be extended via plugins?
2. Can plugins be loaded at runtime?
3. Are plugins isolated from each other?
4. Are extension points documented and stable?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can a third party add a new tool type without modifying core code?

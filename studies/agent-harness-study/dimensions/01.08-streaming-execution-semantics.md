# Dimension 01.08: Streaming Execution Semantics

## Purpose

Analyze how streaming affects state, safety, cancellation, retries, and user experience.

## Steps

1. Identify whether model responses stream token-by-token, event-by-event, or batch-only.
2. Identify how partial tool calls are represented.
3. Determine whether streaming output is persisted.
4. Check whether users can interrupt mid-stream.
5. Check whether partial output can be rolled back.
6. Inspect retry behavior after stream failure.

## Evidence to Capture

- Stream event types
- Partial message buffers
- Streaming callbacks
- Cancellation handlers
- Partial tool call parsers
- UI event emitters

## Questions

1. What is emitted while execution is still running?
2. Are partial outputs trusted?
3. Can a failed stream resume?
4. Can a user stop the stream safely?
5. Does streaming weaken guardrails?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

# Dimension 07.08: Tool Failure Escalation

## Purpose

Study how tool failures are escalated to the model, user, operator, or retry system.

## Steps

1. Classify tool failure types.
2. Inspect model-facing error messages.
3. Inspect user-facing error messages.
4. Inspect operator alerts/logs.
5. Check retry exhaustion behavior.
6. Check escalation to human approval/supervision.

## Evidence to Capture

- Error envelopes
- Failure statuses
- Escalation handlers
- Alerting hooks
- User messages
- Retry exhaustion records

## Questions

1. Who sees tool failure?
2. Is the error actionable?
3. Can the model recover?
4. When is failure escalated to a human?
5. Are failures grouped by cause?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Does a failed tool become a recovery path or a dead end?

## Output

Write findings to `reports/repo/07.08-tool-failure-escalation/{repo-name}.md`

# Dimension 19.03: Adapter and Interop Boundary Design

## Purpose

Study whether standards are core abstractions, optional adapters, or afterthought integrations.

## Steps

1. Identify whether protocol support is in core or in adapters.
2. Check whether adapters are swappable at runtime.
3. Inspect adapter conformance tests.
4. Check whether adding a new protocol requires core changes.
5. Check whether interop boundaries are documented.

## Evidence to Capture

- Core protocol abstractions
- Adapter implementations
- Plugin/extension points
- Conformance tests
- Protocol documentation

## Questions

1. Are protocols core or adapter-layer?
2. Can adapters be added without core changes?
3. Are adapters tested for conformance?
4. Are interop boundaries documented?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can a new protocol be added through an adapter without modifying core code?

# Dimension 24.04: Embedding and Host Integration Ergonomics

## Purpose

Study how easily the agent harness can be embedded into another application, service, CLI, workflow engine, or developer platform.

## Steps

1. Identify supported embedding modes: library, SDK, CLI subprocess, server API, worker, plugin, or hosted service.
2. Inspect initialization, configuration injection, lifecycle management, logging, cancellation, and shutdown behavior.
3. Check whether host applications can provide dependencies such as storage, tools, identity, policy, telemetry, and secrets.
4. Assess how errors, progress, streaming output, and approvals are surfaced to the host.
5. Evaluate whether embedding requires global state, singleton configuration, process-wide side effects, or hidden background work.

## Evidence to Capture

- Embedding examples and SDK entry points
- Initialization and dependency injection APIs
- Host-provided storage, telemetry, policy, and tool hooks
- Progress, streaming, approval, and error callback contracts
- Shutdown, cleanup, and resource ownership behavior

## Questions

1. Can the harness run inside another application without owning the whole process?
2. Can the host supply policy, tools, identity, storage, telemetry, and secrets?
3. Are lifecycle, cancellation, shutdown, and error propagation explicit?
4. Does the integration model work for both local-first and service deployments?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can a host product embed the harness while retaining ownership of policy, state, telemetry, and UX?

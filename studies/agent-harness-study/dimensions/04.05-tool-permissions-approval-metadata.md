# Dimension 04.05: Tool Permissions and Approval Metadata

## Purpose

Study how tools declare risk and whether dangerous tools require approval.

## Steps

1. Identify permission fields.
2. Classify tools by risk: read-only, write, delete, execute code, network, secret access, money/external side effect.
3. Check approval rules.
4. Inspect whether permission metadata is visible to runtime.
5. Check whether permissions are enforced or decorative.

## Evidence to Capture

- Permission enums
- Approval policies
- Tool metadata
- Risk classifications
- Confirmation states
- Tests for denied tools

## Questions

1. Are tools risk-classified?
2. Are permissions enforced?
3. Can users approve selectively?
4. Are approvals persisted?
5. Can policy block a model-requested tool?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can the runtime stop a high-risk tool even if the model asks for it?

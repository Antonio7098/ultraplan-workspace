# Dimension 06.02: Task Decomposition Representation

## Purpose

Analyze how tasks are broken into subtasks.

## Steps

1. Identify plan/task schemas.
2. Inspect whether subtasks have status.
3. Check dependencies between subtasks.
4. Check whether subtasks map to tool calls.
5. Inspect whether tasks can be reordered.

## Evidence to Capture

- Todo schemas
- Task objects
- Dependency graphs
- Step records
- Plan update tools

## Questions

1. How is a task decomposed?
2. Are subtasks typed?
3. Can dependencies be represented?
4. Are statuses tracked?
5. Can decomposition be evaluated?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Can the system tell which subtask is currently blocking progress?

# Dimension 03.09: Completion and Finalization Semantics

## Purpose

Study how the system decides an agent has finished and how it packages the final answer.

## Steps

1. Identify final output types.
2. Identify model finish reasons.
3. Inspect whether pending tool calls block completion.
4. Check whether finalization records usage/cost/status.
5. Inspect whether final output is validated.
6. Check whether incomplete runs produce clear status.

## Evidence to Capture

- Final output schemas
- Completion statuses
- Usage records
- Output validators
- Finalization events
- Run summaries

## Questions

1. What counts as done?
2. Can the model falsely declare done?
3. Are final outputs validated?
4. Is usage/cost finalized?
5. Can a run complete with warnings?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Does done mean correct, merely stopped, or just no more tool calls?

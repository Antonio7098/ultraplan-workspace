# Dimension 05.01: Short-Term Conversation Memory

## Purpose

Analyze how recent conversation history is stored, selected, and passed back to the model.

## Steps

1. Locate conversation history storage.
2. Identify how messages are selected for each model call.
3. Check whether history is full, windowed, summarized, or retrieved.
4. Inspect role handling.
5. Check whether tool messages are included.

## Evidence to Capture

- Message stores
- Session history builders
- Context windows
- Message selectors
- Role conversion logic

## Questions

1. What conversation history does the model see?
2. What gets dropped?
3. Are tool messages retained?
4. Is memory per user/thread/session?
5. Can history be edited or forked?

## Rating

| Score | Meaning |
|-------|---------|
| 1-3 | Absent, implicit, ad-hoc, or unsafe |
| 4-6 | Present but inconsistent, weakly documented, or fragile |
| 7-8 | Clear model with tests, explicit interfaces, and operational safeguards |
| 9-10 | Mature, durable, observable, extensible, and proven under failure or scale |

> Does the model see enough recent context without overflowing?

## Relevant Patterns

- **Thin command wiring, study-owned behavior:** The sprint's command additions fit the dominant pattern where CLI code handles flags, calls domain services, renders output, and maps errors, while business rules remain inward. This is supported by `01-project-structure` evidence for thin entrypoints and unidirectional flow (`main.go:26-34`, `cmd/gh/main.go:6`, `cmd/restic/main.go:37-114`) and `02-command-architecture` evidence for command factories delegating to action/domain logic (`pkg/cmd/install.go:132-145`, `pkg/cmdutil/factory.go:16-43`). For this sprint, it pressures `internal/app` to remain a command/rendering layer and `internal/study` to own validation semantics.
- **Stable machine output as a separate rendering path:** Evidence from command architecture and IO reports favors explicit output abstractions rather than direct `fmt` or terminal-coupled output. Gh-cli's `IOStreams.Test()` returns in-memory stdout/stderr buffers (`pkg/iostreams/iostreams.go:551-568`), go-task injects stdout/stderr via functional options (`executor.go:553-564`), and terminal UX reports caution against assuming TTY or mixing progress/color with output (`lib/terminal/terminal.go:82-86`). This pattern pressures JSON output to be centralized, ANSI-free, and test-captured independently from text rendering.
- **Typed diagnostics with preserved error chains:** Error evidence favors wrapping with `%w`, sentinel or typed errors, and explicit exit-code mapping. Gh-cli maps distinct exit categories at `internal/ghcmd/cmd.go:44-49`, go-task exposes a `TaskError` with `Code()` at `errors/errors.go:47-50`, and restic's `fatalError` is classified through `errors.IsFatal()` from `internal/errors/fatal.go:10`. For this sprint, validation failures need structured check data while lower-level causes remain inspectable for safe diagnostics.
- **Redaction as a type/policy boundary before logs or JSON:** Security evidence supports secret-aware data types and scrubbers rather than relying on callers to remember not to print secrets. Restic's `SecretString` redacts through `String()` (`internal/options/secret_string.go:15-20`), helm scrubs auth headers before logging (`pkg/registry/transport.go:37-41`), and gh-cli masks tokens in status output (`status.go:332-338`). This pattern pressures validate/status/health output to redact before both text and JSON rendering, not only in error printing.
- **Fake-first inspection commands:** Testing and DI evidence supports constructing commands with injectable dependencies and command-level tests that parse output. Chezmoi runs txtar command tests (`internal/cmd/main_test.go:64-174`), gh-cli combines acceptance scripts with HTTP mocks (`acceptance/acceptance_test.go:26-29`, `pkg/httpmock/stub.go:35-199`), and restic uses mock backends (`internal/backend/mock/backend.go:14-26`). This pattern applies directly to runtime-free validation/status and fake-backed health checks.
- **Bounded state/artifact inspection:** Performance evidence warns against eager initialization and unbounded scans. Gh-cli's factory defers expensive dependencies through `func()` fields (`pkg/cmdutil/factory.go:27-42`), yq provides streaming document evaluation for bounded memory (`pkg/yqlib/stream_evaluator.go:78-113`), and gdu bounds parallel traversal (`pkg/analyze/parallel.go:13`). For this sprint, validation should inspect known study artifacts and durable state, not recursively scan source repos or initialize runtime paths.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Thin app rendering plus study-owned validation | Keeps command code testable and preserves package ownership; aligns with helm/gh-cli command-to-action separation (`pkg/cmd/install.go:132-145`, `pkg/cmdutil/factory.go:16-43`) | Requires clear result models so `internal/app` can render without re-deriving validation meaning | When deciding whether validation summaries/check models live in `internal/study` or are shaped ad hoc in command handlers |
| Central JSON envelope helper vs per-command JSON structs | Enforces schema version, timestamp, command, workspace, status, and ANSI-free rendering consistently; follows output abstraction pressure from gh-cli IOStreams (`pkg/iostreams/iostreams.go:551-568`) | Risks becoming a generic diagnostics framework if it absorbs command-specific logic | When adding `validate --json`, `status --json`, and `health --json` without over-centralizing product semantics |
| Typed validation findings vs simple error strings | Enables stable JSON fields, exit-code mapping, tests, and actionable guidance; supported by structured error examples like `TaskError.Code()` (`errors/errors.go:47-50`) and `FileTooSmallError` (`fs/fs.go:61-73`) | More model surface to maintain; over-typing one-off failures can slow implementation | When deciding how much structure each validation check, severity, path, observed detail, and guidance should carry |
| Redacted safe diagnostics vs raw runtime/config details | Prevents credential and payload leakage, following `SecretString` (`internal/options/secret_string.go:15-20`) and auth scrubber evidence (`pkg/registry/transport.go:37-41`) | Can remove context that would help debugging unless safe summaries and cause chains are preserved | When rendering health runtime failures, run-state diagnostics, config summaries, and report parse errors |
| Lazy/runtime-free status and validation vs eager health checks | Keeps validation/status deterministic, offline, and fast; avoids gh-cli-style eager config startup penalties noted in performance evidence (`pkg/cmdutil/factory.go:27-42`) | Some failures are only visible when runtime health is explicitly queried | When separating `study status --json` and `study validate` from `health --json` runtime-boundary checks |
| Golden/schema output assertions vs flexible substring tests | Stable JSON schemas and command output regressions become visible; supported by helm golden testing (`internal/test/test.go:43`) and go-task normalized golden tests (`task_test.go:166-169`) | Golden/schema maintenance adds review overhead and can over-constrain intentionally flexible text output | When choosing tests for JSON field names, envelope versions, redaction, exit codes, and deterministic text summaries |

## Anti-Patterns And Warnings

- **Business logic in `RunE`:** `02-command-architecture` flags long `RunE` bodies as logic leakage, with opencode `cmd/root.go:49-183` and yq `cmd/evaluate_sequence_command.go:152` as cautionary examples. Validation should not become a long command handler that duplicates study validators.
- **Direct `os.Stdout`/ANSI/progress in machine output:** `06-io-abstraction` warns about hardcoded output paths such as rclone `cmd/ls/ls.go:42`, and `09-terminal-ux` warns that progress/color should respect TTY detection (`lib/terminal/terminal.go:82-86`). JSON output must not share paths that emit color, progress, spinners, or mixed human diagnostics.
- **Config/runtime globals hidden behind validation:** `03-dependency-injection` identifies globals as the key testability risk, including rclone `fs/config.go:793` fallback and yq `pkg/yqlib/lib.go:13`. Validation/status should not pull runtime/config singletons implicitly or become impossible to test with isolated fixtures.
- **Panic or string-only parse failures for user artifacts:** `05-error-handling` flags panic for parse errors (`dive/image/docker/manifest.go:18`) and untyped string-only errors as weak diagnostics. Malformed report frontmatter, summaries, and durable state should be reported as structured validation findings, not crashes.
- **Secret leakage through logs, JSON, or observed details:** Security evidence shows why secret types and scrubbers matter (`internal/options/secret_string.go:15-20`, `pkg/registry/transport.go:37-41`). Observed validation details, runtime stderr snippets, prompt bodies, config values, and environment-derived diagnostics need redaction before rendering.
- **Unbounded recursive scans or eager expensive setup:** `14-performance` warns against eager initialization and unbounded accumulation (`gh version` style eager config penalties, `pkg/cmdutil/factory.go:27-42`; goroutine explosion without bounding, `pkg/analyze/parallel.go:13`). Validation should stay bounded to known workspace/study artifacts.
- **Implementation-detail tests instead of behavior/schema tests:** `11-testing-strategy` calls out k9s hint count assertions (`internal/view/pod_test.go:23`) as brittle. Sprint tests should assert public output shape, exit codes, redaction, and runtime-free behavior rather than private list lengths or incidental formatting.

## Design Pressures

- `validate`, `status --json`, and `health --json` all need stable machine-readable shapes, but each surface has different ownership: study validation/result semantics, durable status summaries, and runtime/config/workspace health summaries.
- Command-level JSON needs a shared envelope without turning into a cross-project diagnostics framework; report-specific or run-state-specific semantics should stay in their owning packages.
- Validation and status must be offline and runtime-free, while health is allowed to touch the existing runtime health boundary; the handbook evidence supports lazy construction and explicit dependency seams to keep those paths separate.
- User-facing diagnostics must be actionable and causal, but redacted; preserving cause chains and exposing safe observed details pull in opposite directions.
- JSON must be stable and automation-friendly, while text output must remain concise and deterministic for humans.
- Durable `run-state.json` handling must diagnose malformed or unsupported schema versions without migration, and status must preserve unknown usage/cost as unknown rather than zero.
- Validation needs to distinguish missing from inapplicable artifacts because Markdown applicability already exists in prior sprint scope; this raises pressure for typed statuses beyond pass/fail.
- Tests must cover command behavior, output shape, redaction, and no-runtime invocation while remaining deterministic and offline.
- Performance pressure is less about raw speed and more about bounded known-artifact inspection and avoiding recursive source scans or expensive runtime setup.
- Package boundary pressure is high: no new global `internal/validation`, `internal/reports`, `internal/diagnostics`, or `internal/status` package is supported by the evidence unless reasoning finds a concrete exception.

## Open Questions For Reasoning

- What is the minimal validation result model that supports stable JSON, deterministic text, severity, path, observed safe detail, expected condition, and guidance without overfitting to this sprint?
- Should the JSON envelope encode command status as `ok`/`failed`/`degraded`, `success`/`failure`, or another stable vocabulary, and how should it compose with result-level validation statuses?
- How should validation failures map to exit code `5` while preserving existing non-validation command errors and wrapped causes?
- Which validation checks should be modeled as `inapplicable` rather than `skipped`, and how should those statuses appear in text and JSON?
- What redaction function shape is sufficient for nested validation/status/health structures without creating a broad diagnostics framework?
- How much run-state validation belongs in `internal/study` validation versus existing run-state/status code, and where should unsupported schema diagnostics be produced?
- What exact fields should `status --json` include for unknown usage/cost so callers can distinguish unknown from zero?
- How should health JSON expose runtime/config/workspace diagnostics through the existing platform boundary without importing study or product semantics into runtime packages?
- Should text output have golden tests, or should tests assert selected deterministic lines while JSON gets strict schema/field assertions?
- How can command tests prove validation/status do not invoke agentwrap, OpenCode, network, subprocesses, or source cloning?

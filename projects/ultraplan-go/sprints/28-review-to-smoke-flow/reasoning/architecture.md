# Architecture: Integrated Review-to-Smoke Verification Flow

Scope: integrate the existing execute, review, and smoke capabilities into one resumable, read-only verification flow with shared CLI/JSON/TUI semantics. This document decides module ownership, state boundaries, dependency direction, orchestration shape, and the minimum abstractions needed for Sprint 28. It does not introduce issue management, automatic fixes, Git mutation, a third assessment artifact, a general workflow engine, or a browser surface.

## Area Decisions

### Architecture fit and implementation shape

- **Decision: retain the module-driven architecture and perform a small refactor before integration.** The existing shape is partially sufficient: Sprint 26 review and Sprint 27 smoke remain product capabilities inside `internal/sprint`, but their status, gating, freshness, and recovery projections must be composed into a shared verification result before adding `verify` and `flow --to smoke`. Wholesale replacement of either stage is rejected.
- **Decision: `internal/sprint` is the sole owner of verification policy.** It owns stage ordering, review gating, diagnostic override policy, freshness manifests and comparisons, focused-rerun rules, verdict synthesis, overall assessment, artifact validation, next-action selection, and mutation exclusion. Focused files such as `flow.go`, `state.go`, `domain.go`, `service.go`, `review.go`, `smoke.go`, and `validation.go` remain one package unless concrete package-size or dependency pressure later earns a split.
- **Decision: `internal/app` exposes shared typed use cases, not a second workflow layer.** Status, flow, review, smoke, verify, rerun, cancellation, and recovery requests call `internal/sprint` services and return one typed semantic result. CLI text and JSON render that result; TUI actions consume it directly. CLI handlers and TUI models must not calculate verdicts, staleness, gates, or next actions.
- **Decision: preserve unidirectional dependencies.** The allowed path is `cmd/ultraplan` and `internal/tui` -> `internal/app` -> `internal/sprint` -> `internal/project`, `internal/workspace`, `internal/platform/runtime`, and `internal/platform/process`. Platform packages must not import product modules or receive sprint-specific types.

### Canonical state and derived truth

- **Decision: use one versioned `flow-state.json` as durable verification state and no third assessment artifact.** Review and smoke stage records persist execution status, completed verdict, artifact path, artifact digest, governed-input fingerprint, timestamps, attempt metadata, safe diagnostics, smoke run/evidence identifiers, explicit diagnostic override metadata, and recovery information needed after process exit.
- **Decision: separate attempts from last complete evidence.** Starting a review or smoke records a new running attempt atomically without deleting the last complete stage record or replacing `review.md`/`smoke.md`. Cancellation, timeout, malformed output, or failed validation closes the attempt with a classified outcome while preserving the prior complete artifact and its fingerprint. Only a fully validated result atomically replaces the Markdown artifact and promotes the attempt to the current complete record.
- **Decision: compute freshness and overall assessment from canonical inputs at read time.** Persisted fingerprints and artifact digests are facts; current staleness, readiness, overall assessment, and next action are deterministic projections produced by `internal/sprint`. Status may serialize those projections, but no CLI, JSON renderer, or TUI cache becomes authoritative.
- **Decision: avoid self-referential fingerprints.** A stage fingerprint covers normalized inputs consumed to produce that stage. The resulting artifact has a separate content digest. Review freshness compares its governed-input fingerprint and `review.md` digest; smoke freshness includes the current review identity plus smoke inputs and separately compares the `smoke.md` digest. A stale review always forces smoke stale regardless of a matching smoke artifact digest.
- **Decision: fingerprint deterministic manifests, not timestamps.** Manifests use workspace-relative normalized paths, stable lexical ordering, explicit missing-file markers, content digests, selected catalog identities, execute evidence identity, and normalized changed-path scope. Target identity is the configured target root plus Git commit and changed-file content digests when trustworthy Git metadata exists; without it, identity is a full contained regular-file manifest excluding only VCS metadata. Timestamps are metadata only. External harness evidence uses stable run/evidence identity and validated references rather than copying raw evidence into the workspace.
- **Decision: support one explicit forward migration from the flow-state schema current before Sprint 28.** The migration atomically adds verification fields while preserving existing stage facts and last-complete review/smoke references; older, unknown, or malformed schemas are blocked with recovery guidance and are never silently treated as fresh. The new schema has one strict representation and no parallel legacy verification truth.

### Workflow and rerun semantics

- **Decision: keep top-level orchestration sequential: `execute -> review -> smoke`.** `verify --to smoke` and `flow --to smoke` call the same `internal/sprint` transition logic. Smoke requires a current, non-blocking review by default. An explicit diagnostic override may permit smoke execution, but the request and rationale are recorded and cannot improve or conceal the review verdict in the overall assessment.
- **Decision: use bounded concurrency only inside independent review work.** Reviewer fan-out remains owned and joined by the review service with one propagated context, a configured bound, complete result collection where independent findings remain useful, and no fire-and-forget goroutines. Smoke gating and stage transitions are not parallelized.
- **Decision: focused reruns update canonical evidence only through containing-scope validation.** A focused review rerun must merge into and validate one complete current review result under Sprint 26 verdict rules. A focused smoke test or suite can add diagnostic evidence, but a passing narrow run cannot erase a current failure or replace required containing-suite evidence. The smoke result remains non-clean until the required containing scope is current and acceptable.
- **Decision: serialize verification mutations per sprint.** Review, smoke, verify, and flow operations that can update verification state or canonical summaries acquire a sprint-owned mutation guard. Read-only status and freshness inspection may run concurrently from immutable snapshots. Conflicts return a typed, actionable result rather than racing atomic writes.

### Boundaries and abstractions

- **Decision: add only consumer-shaped volatile-boundary interfaces.** Review uses the existing generic agentwrap-backed runtime seam. Smoke uses a narrow process-runner seam accepting executable, argv, contained cwd, allowlisted environment, timeout, and context, and returning generic exit/signal/timing/bounded-output/cleanup facts. Neither boundary knows review findings, smoke levels, harness issues, sprint paths, or verdicts.
- **Decision: create dependencies at the app composition root and pass them inward.** Runtime, process runner, filesystem/store, clock, and progress sink are explicit collaborators where nondeterminism or external effects require substitution. Context carries cancellation, deadlines, and request metadata only; it is not a service locator. No package-global current sprint, runtime, runner, or verification registry is introduced.
- **Decision: use a typed semantic projection shared by all surfaces.** The projection represents stage execution status separately from verdict, freshness and stale reasons, artifact/evidence references, relevant issue IDs, override facts, overall assessment, and required next action. It is a product result, not a rendering DTO and not another persisted aggregate.
- **Decision: keep progress ephemeral and final output deterministic.** CLI text and TUI may subscribe to bounded progress events. JSON mode is non-streaming unless an existing explicit event mode is selected and receives only the final typed result on stdout. Diagnostics and logs use stderr or configured sinks and never contaminate stable JSON.

### Failure, cancellation, and safety

- **Decision: distinguish blocked, failed, stale, malformed, cancelled, timed out, and conflicting outcomes.** `internal/sprint` returns typed categories with safe fields and a next action; `internal/app` maps them to stable JSON and exit behavior and renders human guidance. Process exit zero cannot produce success without valid current artifacts and referenced evidence.
- **Decision: propagate the caller context through review requests, process execution, progress delivery, and waits.** Owned work is joined before return. Process-tree or artifact cleanup that must continue after cancellation uses a separate bounded cleanup context and reports cleanup failure without replacing the primary outcome.
- **Decision: verification remains read-only outside its owned state and summaries.** Allowed writes are atomic updates to verification state and the canonical `review.md`/`smoke.md` summaries, plus external harness-owned evidence under its declared roots. Product source, product tests, governed planning inputs, and Git state are never mutated. Harness invocation uses executable plus argv, contained paths, bounded environment forwarding, timeout, redaction, and no shell evaluation of Markdown or README prose.

### Test and rollout structure

- **Decision: test policy in `internal/sprint`, composition/parity in `internal/app`, and interaction/rendering in `internal/tui`.** Domain tests cover transition tables, fingerprint normalization, stale propagation, deterministic assessment precedence, override behavior, rerun containment, issue-aware smoke verdicts, and interrupted-attempt preservation. App and command tests cover typed mapping, exit behavior, and text/JSON parity. TUI tests assert the same semantic fields and actions without terminal or CLI subprocess dependencies.
- **Decision: normal tests use fake runtime and process/harness boundaries.** Cancellation, timeout, malformed evidence, missing evidence, path escape, cleanup failure, and atomic-preservation cases must be deterministic. Real OpenCode and external harness tests remain gated. Race tests specifically cover concurrent status reads, progress delivery, cancellation, and sprint mutation exclusion.
- **Decision: roll back by disabling the new integrated command routing and reading the previous complete supported state, not by reverting artifacts during a failed operation.** Schema changes must be introduced atomically with strict readers and fixtures. If a supported migration is required, it must preserve last complete review/smoke identities and reject ambiguous legacy evidence as current.

## Trade-Offs

| Decision | Benefit | Accepted Cost |
| --- | --- | --- |
| Small refactor around shared sprint projections instead of direct command additions | Prevents three surfaces from becoming independent freshness and verdict engines while preserving Sprint 26/27 implementation | Adds typed request/result structures and requires existing call sites to converge on them |
| Persist stage facts but derive freshness and overall assessment | Avoids a third artifact and contradictory cached verdicts | Status reads must inspect and hash the current manifest; careful normalization and fixtures are required |
| Preserve last complete artifacts separately from active attempts | Keeps evidence trustworthy through cancellation, timeout, or malformed replacement | State transitions and recovery output are more explicit than simply overwriting files |
| Sequential stage flow with bounded review fan-out | Makes review-before-smoke gating and cancellation comprehensible | Does not optimize smoke latency through speculative execution |
| Focused reruns constrained by containing-scope evidence | Saves diagnostic time without converting a narrow pass into false confidence | Some reruns still require broader suite work before the canonical verdict can improve |
| Narrow runtime/process interfaces at volatile boundaries | Enables deterministic fakes and keeps platform packages product-neutral | Introduces limited indirection and requires disciplined result translation |
| One product projection with surface-specific rendering | Gives CLI text, JSON, and TUI semantic parity | Complete rendering parity still requires separate adapter tests and selected golden fixtures |
| Per-sprint mutation exclusion | Prevents semantically conflicting operations despite atomic individual writes | A second operator may receive a conflict and need to retry after the active operation |

Rejected alternatives:

- **Surface-owned integration:** rejected because implementing gates, freshness, or assessment separately in CLI and TUI would duplicate business knowledge and violate the shared-use-case architecture.
- **A new generic workflow engine or verification plugin registry:** rejected because only one ordered sprint flow and two established external boundaries are in scope; the handbook explicitly warns against abstractions without a named pressure.
- **A durable `assessment.md` or separate verification database:** rejected because requirements define assessment as a deterministic projection from current review, smoke, flow state, and harness issues.
- **Recompute all truth from Markdown alone:** rejected because interruption, attempts, run IDs, overrides, and input fingerprints need strict machine-readable state; process success and prose are insufficient evidence.
- **Treat persisted staleness or assessment as authoritative:** rejected because editable governed inputs and externally edited summaries can change outside the process and invalidate cached conclusions.
- **Replace Sprint 26 review or Sprint 27 smoke wholesale:** rejected because the sprint is an integration and hardening step; replacement increases migration and regression risk without evidence that existing module boundaries are wrong.
- **Expose shell command strings to the smoke adapter:** rejected because catalog and Markdown content cross a trust boundary and explicit argv is required.
- **Fail fast across independent review checks:** rejected as the default because one failed reviewer should not hide other actionable independent findings; bounded collection is more useful, while prerequisite or cancellation failures still stop dependent work.

## Evidence

- The technical handbook's **Thin surfaces over a shared operation** pattern cites Helm delegation and gh-cli factories as high-confidence evidence that command code should parse, call a reusable operation, and render (`technical-handbook.md`, lines 30-32). This supports typed app use cases and rejects CLI/TUI-owned verification policy.
- **Explicit, inspectable lifecycle state** and **Cancellation from entrypoint through every blocking boundary** support durable attempts, caller-context propagation, and a separate bounded cleanup policy (`technical-handbook.md`, lines 34-38). The cited Go task, Helm, Restic, and OpenCode examples also support localized, joined concurrency rather than detached work.
- The handbook's typed-failure evidence shows that machine-readable categories and human recovery rendering can remain separate (`technical-handbook.md`, lines 40-42). This is the basis for distinct blocked, stale, malformed, cancelled, timeout, conflict, and failed outcomes without a large class hierarchy.
- Injected IO and process seams in gh-cli, Restic, and lazygit support fake runtime/harness tests and explicit boundary effects (`technical-handbook.md`, lines 42-46). The security evidence for executable plus argv directly supports a non-shell smoke process contract.
- Structured status separated from diagnostics supports one stable typed result for text, JSON, and TUI while keeping logs, spinners, ANSI, and raw harness output away from JSON stdout (`technical-handbook.md`, lines 48-50 and 80-86).
- The handbook's state trade-off explicitly warns that centralized state can drift from editable artifacts (`technical-handbook.md`, lines 56-60). Separating persisted facts and digests from read-time freshness/assessment resolves that pressure while preserving resumability and last-complete evidence.
- The focused-rerun trade-off states that a narrow pass may not establish the broader invariant (`technical-handbook.md`, lines 60-62). This is the evidence basis for retaining containing-suite requirements in the canonical smoke verdict.
- The bounded concurrency and philosophy reports support accepting complexity only at volatile boundaries and where current sprint pressure earns it (`technical-handbook.md`, lines 21-26 and 92). Therefore this design adds a shared projection, external seams, and mutation exclusion, but no general scheduler, plugin system, or package split.
- Project architecture constraints independently align with the evidence: `ARCHITECTURE.md` assigns sprint semantics to `internal/sprint`, composition and typed use cases to `internal/app`, interface concerns to `internal/tui`, and generic execution to platform packages. The PRD and TRD require review-before-smoke, validated evidence rather than exit-code success, atomic canonical summaries, external raw smoke evidence, and CLI/TUI parity.

Key conclusion: the current module architecture remains correct. The simplest honest design is to consolidate integrated verification semantics and durable transitions inside `internal/sprint`, expose one typed app projection to every surface, and keep runtime/process concerns generic and injected. Evidence does not justify a new workflow framework or replacement of the existing review and smoke stages.

## Risks

- **Fingerprint cost and scope:** the non-Git fallback hashes every regular file under the contained target root except VCS metadata. This is deliberately conservative and may mark evidence stale because of generated files; changed-path optimization is permitted only when its provenance is trustworthy and complete.
- **State/artifact reconciliation:** external edits can create a valid-looking artifact whose digest no longer matches state. Such evidence is stale or malformed, never silently adopted as current; recovery may explicitly rerun the owning stage.
- **Schema migration:** the one-version migration must distinguish absent historical fields from known values and must not invent freshness. Fixtures must prove preservation of prior stage facts and rejection of older or ambiguous state.
- **Focused review merge semantics:** preserving complete coverage while rerunning only selected findings can accidentally retain stale reviewer conclusions. A focused review may improve the canonical result only after every non-rerun result is proven to share the current review fingerprint.
- **Harness issue selection:** issue relevance must be resolved by the sprint smoke policy from validated harness metadata, not free-form text. Missing or ambiguous relevance blocks a clean pass without expanding UltraPlan into issue management.
- **Concurrent operators:** atomic files alone do not prevent two valid operations from overwriting each other's logical state. The sprint mutation guard and conflict result are required, and stale-lock recovery must be conservative and observable.
- **Cleanup portability:** descendant-process termination differs by operating system. The generic process result must expose cleanup status, and cleanup failure must leave a recoverable non-pass state rather than claim successful cancellation.
- **Performance:** content hashing and external evidence validation increase status cost. Optimize only after measurement; deterministic manifests may reuse safely keyed digests, but cached values cannot bypass current file identity checks.
- **Projection growth:** a shared verification result could become a god DTO. Keep it use-case-oriented, omit raw runtime/harness payloads, and split request-specific result types only when consumers demonstrably need different concepts.

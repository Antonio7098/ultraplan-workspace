## Relevant Patterns

- **Thin CLI, Study-Owned Orchestration:** The project-structure and command reports consistently show CLI layers parsing flags and delegating into business logic rather than owning workflows (`main.go:16`, `cmd/root.go:112-127`, `pkg/cmd/install.go:132-145`). For this sprint, the pressure is to keep `run-loop` state transitions, scheduling, resume validation, synthesis gating, and persistence in study-owned code while the command layer handles args, config overrides, cancellation setup, and rendering.
- **Explicit Composition Root And Test Seams:** DI evidence favors explicit factories and constructor injection over global state (`pkg/cmdutil/factory.go:16-43`, `pkg/cmd/factory/default.go:26-46`, `executor.go:22-24`). Run-loop reasoning should identify the smallest useful seams for runtime, state store, lock store, clock, and output, so fake-runtime and failure-injection tests can exercise durable behavior without real OpenCode or subprocesses.
- **Context-Propagated Cancellation:** Long-running CLI evidence repeatedly wires root signal handling into `context.Context` and passes it through task execution (`pkg/cmd/install.go:333-347`, `cmd/restic/cleanup.go:24-38`, `task.go:89`). Run-loop needs the same discipline for stop-new-work, cancel-active-runs, drain events, call `Wait`, persist cancellation/retryable status, and return the existing cancellation exit convention.
- **Bounded Scheduling With Structured Completion:** Concurrency evidence favors `errgroup`, worker pools, semaphore channels, and timeout-bounded waits over fire-and-forget goroutines (`go-task/task.go:87`, `gdu/pkg/analyze/parallel.go:13`, `k9s/internal/pool.go:21,30,37`, `opencode/cmd/root.go:130,252`). This sprint's scheduler must reason about bounded runtime tasks, bounded queueing, no unbounded goroutine-per-task launch, and safe cleanup if workers block or cancellation arrives.
- **Durable State As Product Truth, Not Runtime Truth:** The state/context, observability, and testing evidence show mature CLIs track app-level state explicitly through `Config`/`App`/`Options` or lock/session models (`internal/global/global.go:46-89`, `internal/restic/lock.go:105`, `internal/session/session.go:12-23`). For UltraPlan, agentwrap run metadata is evidence to persist, but task completion remains gated by product validation and output existence.
- **Atomic Persistence And Conservative Locking:** Restic evidence around lock modeling and cleanup (`internal/restic/lock.go:47`, `internal/restic/lock.go:105`, `internal/restic/lock.go:290-305`) plus filesystem abstraction evidence (`internal/fs/interface.go:10-31`) points to explicit lock records, scoped lock ownership, and cleanup contexts. Reasoning must settle stale-lock policy without auto-deleting active-looking locks.
- **Classified Errors And Retry Metadata:** Error evidence supports behavioral or typed classifications for retry/fatal/cancel decisions (`fs/fserrors/error.go:22-29`, `go-task/errors/errors.go:47-50`, `internal/errors/fatal.go:10`, `internal/ghcmd/cmd.go:44-49`). Runtime errors should be inspected structurally, not string-parsed, and task state should preserve safe category, operation, provider/model, attempt, retry/fallback, validation, usage, and cleanup facts.
- **Runtime-Free Status From Persisted Facts:** Observability and terminal UX evidence favors operator-readable status from logs/state rather than live runtime probing (`internal/ui/progress/printer.go:15-34`, `pkg/iostreams/iostreams.go:52-54`, `internal/slogs/keys.go:6-231`). Status should answer active, pending, failed, retrying, cancelled, recovered, and artifact path questions by loading durable state and lock data only.
- **Fake Runtime, Fakes, And Golden/Command Tests:** Testing evidence strongly favors deterministic fake dependencies and command-level tests (`pkg/httpmock/stub.go:35-199`, `internal/backend/mock/backend.go:14-26`, `lazygit/pkg/commands/oscommands/fake_cmd_obj_runner.go:17-26`, `helm/internal/test/test.go:43`). This sprint should verify task-state transitions, retries/fallback, cancellation, locks, atomic write failure preservation, and status rendering with local fixtures.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| `errgroup`-style structured fan-out vs explicit worker pool/semaphore | Automatic error collection and sibling cancellation, shown in `go-task/task.go:87` and `restic/internal/repository/repository.go:567` | `errgroup` needs explicit limits or it can still over-launch; worker pools make task ordering and state persistence more explicit | Run-loop must decide how to bound runtime starts, preserve task-level state transitions, and cancel active work without starving synthesis |
| Atomic write after every meaningful transition vs batched state writes | Durable recovery after interruption and clear status; aligns with explicit state evidence and filesystem abstraction patterns (`internal/fs/interface.go:10-31`) | More filesystem writes, more chances for transient write errors, and more complex tests for partial write preservation | Required when a runtime run starts, validates, retries, fails, cancels, or unblocks synthesis |
| Conservative stale-lock handling vs automatic stale cleanup | Avoids corrupting state from a still-active process; restic-style lock modeling treats lock conflicts as structured facts (`internal/restic/lock.go:47`, `internal/restic/lock.go:105`) | Operators may need explicit force-unlock even when a process is truly gone | Per-study `run-loop` lock acquisition, stale lock detection, and force-unlock UX |
| Persisting rich runtime metadata vs minimal task status | Helps status, debugging, retry/fallback reasoning, and cost/usage accounting; observability evidence supports structured metadata (`internal/slogs/keys.go:6-231`, `fs/accounting/prometheus.go:78-108`) | Increases schema surface, redaction burden, and migration/rejection decisions | Agentwrap metadata includes retry, fallback, validation, cleanup, permission, usage, cost, and raw-payload safety facts |
| Completed-output revalidation on resume vs trusting persisted completion | Prevents invalid success when files were deleted or corrupted; product evidence says runtime success is not product success | Resume can be slower and may move previously completed tasks back to schedulable or failed states | Any existing run-state loads before scheduling and synthesis gating |
| Single study package with focused files vs new scheduler/workflow packages | Preserves module ownership and avoids global technical layers, matching evidence for business logic inside protected app packages (`internal/chezmoi/chezmoi.go:1-2`, `pkg/action/install.go:73-140`) | One package can grow dense if files are not well-focused | Run-loop adds scheduling, state, locks, validation, and status behavior to `internal/study` |

## Anti-Patterns And Warnings

- **Do Not Let `RunE` Become The Run Loop:** Command architecture flags long `RunE` handlers as a maintainability smell (`opencode/cmd/root.go:49-183`, `yq/cmd/evaluate_sequence_command.go:152`). CLI code should not own task state machines, retry loops, or synthesis unblocking.
- **Avoid Fire-And-Forget Runtime Goroutines:** Concurrency evidence warns about untracked goroutines and missing waits (`dive/cmd/dive/cli/internal/command/adapter/resolver.go:70`, `gh-cli/pkg/cmd/extension/manager.go:196-206`). Every started runtime run needs event draining or safe consumption, cancellation handling, and `Wait`.
- **Do Not Use `context.Background()` In Work Paths:** State/context evidence calls out isolated backgrounds as cancellation leaks (`internal/cmd/templatefuncs.go:215`, `task.go:341`, `context.TODO()` in `internal/restic/lock.go` refresh paths). Long-running run-loop work should inherit the command context unless reasoning deliberately defines a short cleanup context.
- **Do Not String-Parse Runtime Errors:** Error evidence favors `errors.Is`/`errors.As`, sentinels, and typed errors over strings (`fs/fserrors/error.go:22-29`, `internal/errors/fatal.go:10`, `api/client.go:42-49`). Runtime classifications should come from agentwrap `SDKError` fields and policy metadata.
- **Do Not Convert Unknown Usage To Zero:** Observability/cost metadata needs to preserve unknown as unknown; otherwise status and accounting become false. This is consistent with evidence against untyped fields and with the sprint scope's cost/usage preservation pressure.
- **Do Not Leak Secrets, Prompts, Markdown Bodies, Or Unsafe Native Payloads:** Security evidence supports redaction types and scrubbers (`internal/options/secret_string.go:15-20`, `pkg/registry/transport.go:37-41`, `status.go:332-338`). Run-state and status diagnostics should store safe summaries and omission facts, not raw sensitive material.
- **Do Not Auto-Delete Active-Looking Locks:** Lock evidence from restic shows lock conflicts are meaningful structured errors (`internal/restic/lock.go:47`). Stale detection should be conservative and explicit force-unlock should be scoped to the selected study.
- **Avoid Global Mutable Config Or Service Singletons:** DI and configuration reports flag globals as testability and initialization-order hazards (`pkg/yqlib/lib.go:13-21`, `fs/config.go:793`, `fs/cache/cache.go:16-21`). Run-loop dependencies should be explicit enough for isolated tests.
- **Avoid A Generic Workflow Engine:** Philosophy and project-structure evidence show complexity should be accepted only for real product needs. This sprint needs a durable study run loop, not DAG authoring, plugin workflows, target/sprint execution, or global `workflow` packages.

## Design Pressures

- The run loop is long-running, cancellable, stateful, and retrying; evidence suggests context propagation, bounded concurrency, and explicit state transitions must be treated together rather than as independent features.
- The runtime boundary must stay generic. Evidence supports adapter/decorator boundaries, but sprint scope requires product semantics such as source applicability, report validation, task matrix construction, and synthesis gating to remain study-owned.
- State is both recovery substrate and status source. The handbook evidence points toward runtime-free status, which means each meaningful transition and safe metadata summary must be persisted clearly enough for operators to inspect later.
- Retry/fallback policy creates pressure for typed task outcomes. The sprint must preserve enough classification to distinguish validation failure, cancellation, rate limit, permission failure, timeout, cleanup failure, write failure, and lock conflict without relying on user-facing strings.
- Locking and atomic writes are reliability features but also UX features. Operators need deterministic paths, lock metadata, and safe force-unlock guidance, not hidden background cleanup.
- Validation must remain a product gate. Evidence and project scope both reject treating runtime exit success as task success when required reports are missing or invalid.
- Markdown document applicability adds matrix pressure. Inapplicable pairs must not be created, retried, synthesized, warned as missing, or counted as failures.
- Observability must balance detail with safety. Runtime metadata, retry decisions, usage/cost, and cleanup facts are useful; prompts, embedded document bodies, API keys, and unsafe native payload bytes are not safe by default.
- Normal verification must be deterministic and offline. Evidence strongly supports fakes and fixtures, and this sprint explicitly excludes default real OpenCode smoke tests.
- The architectural boundary pressure is high because durable orchestration could tempt global `scheduler`, `workflow`, `prompts`, `reports`, or `validation` packages. Evidence favors keeping behavior near owned state.

## Open Questions For Reasoning

- What exact task-state statuses and transition rules should represent `pending`, `running`, `retrying/waiting`, `validating`, `completed`, `failed`, `cancelled`, `skipped`, and `synthesis-unblocked` without overfitting a generic workflow model?
- When a prior task is found `running` on resume, should it reset to pending, become failed, or become retryable, and what recovery metadata must be preserved for status?
- How should completed-output revalidation classify missing or invalid artifacts: failed, pending, retryable, or validation-failed, and how should attempts be preserved?
- Should synthesis tasks be materialized in initial state, created when unblocked, or inferred from analysis completion, and what state shape gives truthful runtime-free status?
- What lock path and payload fields are sufficient for PID, command, timestamp, stale diagnosis, and scoped force-unlock without exposing sensitive command args?
- How should write failures during required state persistence affect active worker scheduling and final exit status?
- Which agentwrap metadata fields should become durable task fields, which should be summarized, and which should be omitted with safety facts?
- How should cancellation distinguish user interrupt from timeout/runtime cancellation and map those cases to exit code, task status, retryability, and cleanup metadata?
- How should retry and fallback decisions be persisted when agentwrap policy retries the same logical task across multiple targets or models?
- What is the minimum schema/version compatibility behavior: migrate compatible Sprint 8 state, reject old state, or accept only exact current version?
- How should status present active and recent tasks without becoming nondeterministic or making runtime calls?
- Which tests need race-sensitive coverage beyond normal fake-runtime unit tests?

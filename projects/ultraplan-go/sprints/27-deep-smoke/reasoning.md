## Sprint Technical Handbook Summary

- **Relevant Patterns:** Thin interface adapters over shared use cases; constructor-injected process and IO seams; explicit executable/argv; versioned machine-readable subprocess contracts; post-merge config validation; end-to-end context propagation; one owned and bounded lifecycle; typed progress/errors; separate user, JSON, and diagnostic output; fake runners plus child-process fixtures and selective goldens.
- **Important Trade-Offs:** Generic process mechanics require sprint-owned protocol validation; strict protocol checks can temporarily block newer harnesses; containing-suite evidence costs more than a focused test; bounded excerpts move deep diagnostics to external evidence; one-process supervision delegates test concurrency to the harness; artifact and flow state cannot be committed as one filesystem transaction.
- **Warnings / Anti-Patterns:** No shell interpolation or prose-derived commands, no pass from exit code alone, no detached cleanup, no unbounded output/goroutines, no mutable global smoke dependencies, no context service locator, no warning-only safety/capability validation, no diagnostic leakage, no TUI alternate workflow, and no speculative plugin/workflow abstraction.
- **Evidence Confidence:** High for architecture, command/use-case separation, process safety, cancellation, strict contracts, diagnostics, and testing because the selected reports cite recurring patterns across mature Go CLIs. Medium for numeric timeout/capture defaults and long-term protocol evolution because the studies establish boundedness and versioning but not UltraPlan-specific values; conservative Sprint 27 defaults and explicit revisit triggers are therefore recorded below.

## Trade-Off And Debt Analysis

### Accepted Trade-Offs

| Trade-Off | Benefit | Cost / Constraint Accepted | Why Acceptable Now | Revisit Trigger |
| --- | --- | --- | --- | --- |
| Keep all smoke semantics in `internal/sprint` | One traceable owner for gate, scope, evidence, verdict, validation, and state. | The package gains several focused files and collaborators. | This follows established module ownership and avoids a speculative verification layer. | `sprint` becomes difficult to understand despite focused files, or a second product module needs the same smoke semantics. |
| Add only a generic process runner interface | Isolates the volatile OS boundary and permits realistic fakes. | Sprint code must own all protocol decoding and validation. | Process mechanics are genuinely reusable; smoke meaning is not. | A second external process use case proves a missing generic capability. |
| Support only protocol major version 1 | Fail-closed compatibility is simple and prevents ignored safety/coverage requirements. | A newer harness blocks until explicit support is added. | There is one current harness and no migration promise requiring multi-version conversion. | A cataloged harness ships v2 or a compatibility requirement is approved. |
| Prefer containing-suite evidence | Produces defensible sprint-level proof. | Default runs can cost more than one mapped test. | The sprint goal is acceptance evidence, not merely diagnosis. | Discovery gains a signed/validated coverage-equivalence model for narrower tests. |
| Supervise one harness process | Gives one cancellation, output, identity, and cleanup owner. | UltraPlan cannot tune test-level parallelism. | The harness already owns suites/tests and their raw evidence. | The manifest protocol explicitly delegates schedulable independent units to the host. |
| Cap and redact product-side process output | Bounds memory and disclosure risk. | Complete diagnostics may require following external evidence links. | Raw output explicitly belongs to the harness. | A required diagnostic cannot be represented safely by structured response/evidence. |
| Commit `smoke.md` before flow state | Preserves a validated canonical summary and gives it a clear commit point. | A later flow-state failure creates a detectable split state. | Cross-file/external evidence cannot be one atomic transaction; reconciliation is safer than rollback. | Storage gains a transactional primitive spanning both product artifacts. |

## Decisions

### Decision 1: Preserve Module Ownership And One Shared Use Case

- **Decision:** Implement smoke domain types, orchestration, review/freshness gate, protocol decoding, selection, evidence validation, verdicts, `smoke.md`, validation, and flow/status integration as focused files in `internal/sprint`. Add a consumer-shaped generic runner in `internal/platform/process`. Construct dependencies and expose typed readiness/preview/run/validate/status/flow operations in `internal/app`; CLI and TUI adapt those types only. `internal/project` supplies the catalog entry and `internal/workspace` supplies path mechanics. `platform/runtime` and `agentwrap` review behavior remain unchanged. Add no harness registry, plugin system, generic workflow package, or smoke subpackage tree.
- **Rationale:** Smoke changes sprint state and interpretation, while process launch is the sole genuinely generic volatile boundary. One use case prevents CLI/TUI drift and leaves durable files as the source of truth.
- **Study / Source Grounding:** `technical-handbook.md` Relevant Patterns lines 31-32 and reports `01-project-structure`, `02-command-architecture`, `03-dependency-injection`, and `06-io-abstraction`; concrete references include `helm/pkg/action/install.go:73-140`, `gh-cli/pkg/cmdutil/factory.go:16-43`, `opencode/internal/app/app.go:42-81`, and `lazygit/pkg/commands/oscommands/cmd_obj_runner.go:18-23`. `reasoning/architecture.md` lines 9-29 and `ARCHITECTURE.md` lines 215-299, 343-364 are adopted.
- **Trade-Offs Accepted:** `internal/sprint` grows, and renderers independently adapt typed events/results; in return ownership and dependency direction remain explicit.
- **Technical Debt / Future Impact:** A future second harness may require deliberate adaptation, but current concrete sprint types preserve a clean seam. Sprint 28 can compose the typed use cases without relocating them.
- **Alternatives Rejected:** A harness-aware platform runner is rejected because it leaks product semantics into platform code. A global verification/workflow package is rejected as premature and state-fragmenting. TUI-to-CLI subprocess calls and stdout parsing are rejected because they create an alternate workflow. A generic harness registry is rejected because one protocol does not justify plugin infrastructure.
- **Contracts Satisfied:** Architecture, LLM Runtime, CLI Surface, Workflows; `AC-09`-`AC-11`; `C-01`-`C-04`, `C-10`.
- **Evidence Required:** Architecture Review confirms imports and ownership; tests prove CLI and TUI call the same app operation; process-runner fakes are injected at composition; no product import exists under `internal/platform/process`; no direct OpenCode or self-CLI invocation is introduced.

### Decision 2: Use A Strict Cataloged Protocol-v1 Preflight

- **Decision:** Resolve the harness only from the project catalog root and manifest reference. Canonicalize existing catalog root, manifest, executable, working directory, and declared evidence roots with symlink-aware containment before use; recheck returned evidence paths after execution. Support manifest protocol major version `1` only. Reject malformed JSON, unsupported major versions, missing required fields/capabilities, duplicate identities, and escaping paths. Permit unknown additive fields only when required v1 semantics remain unambiguous. The v1 manifest must provide protocol/schema version, executable plus fixed argument prefix, discovery/run argument forms, cwd, run/issue evidence roots, required capabilities, and declared environment variable names. Discovery must return protocol identity, deterministic levels/suites/tests, sprint mappings and containment relationships, prerequisites, expected duration/cost class, evidence schema, and safe run-command inputs. README or Markdown prose is never executable input.
- **Rationale:** Catalog and machine-readable contract validation are the first trust boundary. Failing closed prevents a partially loaded harness from producing a false pass.
- **Study / Source Grounding:** Handbook lines 33, 47, 57, 63, 73-74 and reports `12-extensibility` and `13-security`; Helm's `runtime_subprocess.go:65-79` and `metadata_v1.go:24-48` support isolated execution and explicit versions, while K9s `plugin.go:158-164` demonstrates the risk of warning-only schema failures. Architecture reasoning lines 33-40, 62-67 supplies the adopted strictness and containment policy.
- **Trade-Offs Accepted:** Strict v1 checks can block a newer otherwise compatible harness. This is preferable to silently ignoring coverage, environment, evidence, or safety semantics.
- **Technical Debt / Future Impact:** No v2 conversion exists. Diagnostics must report manifest path, observed/supported version, failed field/capability, and corrective action so later compatibility work is bounded.
- **Alternatives Rejected:** Hardcoded developer commands are rejected because they are non-portable and bypass the catalog. README parsing is rejected as unsafe and ambiguous. Warning-only compatibility is rejected because missing semantics could look successful. Lexical path-prefix checks alone are rejected because symlinks can escape.
- **Contracts Satisfied:** Architecture, Configuration, Errors, Persistence And Migrations, Security, Workflows; `AC-02`, `AC-03`, `AC-05`, `AC-06`; `C-01`, `C-05`, `C-08`, `C-10`.
- **Evidence Required:** Manifest/discovery fixtures for valid v1, unknown additive fields, malformed JSON, unsupported version, missing capability, duplicate IDs, missing manifest, lexical escape, symlink escape, executable/cwd escape, and returned evidence escape; project-index validation and actionable text/JSON error assertions.

### Decision 3: Resolve Options Once, Gate Review, And Select Sufficient Scope Deterministically

- **Decision:** Build one source-aware effective request in this precedence order: manifest defaults, workspace smoke configuration, bounded environment-derived values, flow/TUI request values, then explicit CLI overrides; validate only after merge. A normal run requires a valid, current `review.md` whose fingerprint matches governed inputs and whose verdict is `pass` or `pass_with_findings`. A current `fail` or `blocked` review may run only with an explicit diagnostic override confirmed by an interactive surface or explicitly flagged by a non-interactive caller; the request, result, `smoke.md`, JSON, and flow state retain the original review verdict/fingerprint and override fact. Missing, malformed, or stale review cannot be overridden in Sprint 27.
- **Decision:** Default selection first chooses a complete sprint-specific suite; otherwise it chooses the deterministic smallest set of directly mapped suites that discovery proves covers the sprint, ordered by declared specificity and stable ID. If equally narrow mappings cannot prove complete coverage, return `blocked` with an explicit level/suite action rather than guess. Explicit level/suite/test values must exist in discovery. A selected suite satisfies evidence only when it contains all required mapped coverage. A test or narrower diagnostic suite cannot replace its required containing suite unless discovery explicitly declares equivalent complete coverage. Missing required prerequisites or coverage yields `blocked`; discovery-proven irrelevance yields `not_applicable`; neither launches the run process. Before launch, all interfaces expose scope/rationale, prerequisites, runtime/model if supplied by the harness, duration/cost class, safe command, override status, and allowed mutation/evidence roots.
- **Rationale:** Review prevents spending on known conformance failures, while deterministic containing-suite selection balances cost against credible sprint evidence. One merge and selection implementation guarantees interface parity.
- **Study / Source Grounding:** Handbook lines 37, 48, 53, 87-92 and report `04-configuration-management` (`go-task/internal/flags/flags.go:314-327`, `k9s/internal/config/k9s.go:423-451`) ground merged validation. Bounded-work evidence in `14-performance` supports narrow defaults but does not define UltraPlan coverage; the exact hierarchy is requirement- and architecture-reasoning-driven (`requirements.md` `AC-01`, `AC-03`, `AC-04`; architecture reasoning lines 35-43). No study source determines review verdict policy because it is a product workflow contract; the project requirements and TRD are authoritative.
- **Trade-Offs Accepted:** Containing suites may cost more than focused tests, and failed-review diagnostic runs require conspicuous metadata. Missing/stale review remains a hard block rather than an emergency bypass.
- **Technical Debt / Future Impact:** Focused rerun recovery and full stale workflow remain Sprint 28 work. Preserved fingerprints, override facts, scope rationale, and next actions are the integration seam.
- **Alternatives Rejected:** Broadest-level default is rejected as unnecessarily expensive. Smallest test default is rejected as insufficient sprint proof. Arbitrary pass-through names are rejected because discovery must validate scope. Treating forced smoke as review approval is rejected because it erases governance. Allowing stale/missing review override is rejected because no current reviewed baseline exists.
- **Contracts Satisfied:** Configuration, CLI Surface, LLM Evaluation / Cost / Safety, Observability, Workflows; `AC-01`, `AC-03`, `AC-04`, `AC-08`, `AC-10`, `AC-11`; `C-06`, `C-07`.
- **Evidence Required:** Table tests for precedence/source metadata; fresh `pass`, fresh `pass_with_findings`, current `fail`/`blocked` with and without override, stale/missing/malformed review; deterministic mapping ties; sprint suite, suite-set, level, suite, test, unknown override, insufficient containing-suite coverage, blocked prerequisite, and not-applicable scenarios; confirmation/preview assertions across CLI and TUI.

### Decision 4: Supervise One Direct Process With Bounded Lifecycle And Environment

- **Decision:** `internal/platform/process` accepts `context.Context` plus executable, argv, cwd, complete environment, timeout, stdout/stderr limits, and an optional non-blocking progress sink; it returns start/finish/duration, exit/signal, bounded streams, truncation/drop counts, and cleanup facts. It calls the executable directly, never a shell. `internal/sprint` resolves the executable and builds argv from validated manifest constants plus validated scope values. It forwards no ambient environment wholesale: only a minimal platform set needed for direct execution and manifest-declared names allowed by workspace policy are copied, with values absent from every display and diagnostic. Discovery defaults to a 30-second timeout with 4 MiB stdout and 1 MiB stderr caps. Run defaults to the validated manifest duration/timeout or the existing 30-minute execution default, with the same product-side caps; positive bounded config/request overrides may change these values and must be reported by source.
- **Decision:** UltraPlan starts one harness process; harness-owned suite/test parallelism is not reimplemented. On Linux/macOS the runner starts an owned process group, propagates context cancellation/timeout, sends graceful termination, waits up to 5 seconds, escalates remaining owned descendants, and always waits/reaps. Cleanup failure or uncertainty is returned separately and cannot yield a passing verdict. Progress delivery is bounded/coalesced; optional intermediate events may be dropped with a count, but process draining, cleanup, and the terminal result cannot be blocked or dropped.
- **Rationale:** Direct argv, explicit environment, one lifecycle owner, bounded capture, and process-group cleanup provide the minimum safe external execution boundary while preserving responsive CLI/TUI progress.
- **Study / Source Grounding:** Handbook lines 32, 34-35, 49, 57, 59-60, 77-80 and reports `06`, `07`, `08`, `09`, `13`, and `14`; key references are `lazygit/cmd_obj_builder.go:38`, `helm/pkg/cmd/install.go:333-347`, `opencode/cmd/root.go:261-279`, and `lazygit/pkg/tasks/tasks.go:189-217`. The studies require bounds but do not establish UltraPlan-specific numeric defaults; 30 seconds, 30 minutes, 4 MiB/1 MiB, and 5 seconds are conservative Sprint 27 operational choices, configurable where stated and reviewable from recorded effective settings.
- **Trade-Offs Accepted:** Capped streams can omit decisive diagnostics, one-process supervision cannot optimize test scheduling, and cleanup behavior is currently Unix-targeted. Linked harness evidence retains detail.
- **Technical Debt / Future Impact:** Windows cleanup is not claimed. Defaults may be tuned from gated-run measurements without changing the request/result architecture.
- **Alternatives Rejected:** Shell command strings are rejected for injection risk. Implicit full environment inheritance is rejected for secret leakage and non-reproducibility. Per-test product workers are rejected as duplicate scheduling. Unbounded capture and blocking progress channels are rejected for memory/deadlock risk. Fire-and-forget kill is rejected because cleanup must be observable.
- **Contracts Satisfied:** Configuration, Errors, Observability, Performance, Security, Testing, Workflows; `AC-03`, `AC-05`, `AC-06`, `AC-11`-`AC-13`; `C-02`, `C-05`, `C-08`, `C-09`.
- **Evidence Required:** Recording-runner assertions for exact executable/argv/cwd/env; child-process fixtures for success, nonzero exit, timeout, cancellation, ignored graceful signal, descendants, malformed/large output, stdout/stderr truncation, slow/disconnected progress, and cleanup failure; race/leak checks; secret values in env, URLs, argv, and excerpts remain absent from all product output.

### Decision 5: Validate Evidence, Synthesize Verdicts, Then Commit The Summary

- **Decision:** Strictly decode the run response and cross-check manifest/discovery/run protocol identity, run ID, requested and observed scope, containing-suite coverage, exit/result counts, duration, runtime/model/usage metadata when supplied, external summary/detail identity, hashes when supplied, fallback identity metadata when hashes are absent, and relevant open/resolved issue IDs. Every referenced path must exist under the canonical manifest-declared run/issue roots and identify the reported run/issue. Unknown usage/runtime/model values remain absent or unknown, never zero/invented. Raw JSON, streams, per-test artifacts, and issue files remain external.
- **Decision:** Compute exactly one verdict in product code: valid selected smoke failures produce `fail`; unavailable required prerequisites or required coverage produce `blocked`; discovery-proven irrelevance produces `not_applicable`; valid passing evidence with relevant open issues produces `pass_with_open_issues`; valid passing evidence with no relevant open issues produces `pass`. A nonzero process exit may still produce `fail` only when a complete valid run response/evidence set supports that test verdict. Malformed output, timeout, cancellation, unexplained process failure, missing/escaping/mismatched evidence, or failed/uncertain cleanup is an execution failure and never a pass.
- **Decision:** For a completed evidence-backed run, including a valid test failure, and for truthful preflight `blocked`/`not_applicable`, render a candidate `smoke.md` containing review verdict/fingerprint/override, harness/protocol, selected scope/rationale, date, environment/runtime/model facts, run ID when one exists, safe argv, counts, duration, evidence links/identity, issue references, verdict, and required next action. Validate the candidate, write a same-directory temporary file, flush/close, and atomically rename over `smoke.md`. Malformed run/process/evidence/cancellation/timeout/cleanup failures do not replace the last valid summary. After summary commit, atomically update versioned flow state with execution status separate from verdict, fingerprint, artifact, run/evidence ID, and safe diagnostics. If flow-state update fails, keep `smoke.md`, report reconciliation required, and let status/validation detect artifact/state mismatch.
- **Rationale:** Process completion is not product success. Validate-before-trust and artifact-first commit preserve the strongest truthful user-facing evidence without pretending multiple filesystems are transactional.
- **Study / Source Grounding:** Handbook lines 33, 36, 47, 49, 58, 63-64, 89-95 and reports `05-error-handling`, `10-logging-observability`, `12-extensibility`, and `13-security` support typed failures, strict external contracts, safe diagnostics, and evidence beyond exit status. Exact verdict rules and artifact ownership come from `requirements.md` `AC-06`-`AC-09`, `TRD.md` lines 1704-1739 and 1846-1854, and architecture reasoning lines 42-46, 68-74. No repository study defines UltraPlan's verdict vocabulary; it is necessarily product-contract-driven.
- **Trade-Offs Accepted:** Artifact-first commit can leave stale flow state after a crash, and linked external evidence can later drift. Strong identity metadata, validation, and explicit reconciliation mitigate both.
- **Technical Debt / Future Impact:** Sprint 28 may automate reconciliation/recovery. Protocol evolution may make hashes mandatory later; v1 accepts run ID plus contained stable path, schema, size, and modification identity when hashes are unavailable.
- **Alternatives Rejected:** Exit-code verdicts are rejected as false confidence. Copying raw evidence into the sprint is rejected as duplication and disclosure risk. Incremental or pre-validation replacement is rejected because it can corrupt the current summary. Rolling back a valid summary after flow-state failure is rejected because it discards stronger committed evidence. Treating malformed/timeout/cancel as a smoke `fail` artifact is rejected because no valid smoke result exists.
- **Contracts Satisfied:** Documentation, Errors, LLM Evaluation / Cost / Safety, Observability, Persistence And Migrations, Security, Workflows; `AC-06`-`AC-10`; `C-07`, `C-08`.
- **Evidence Required:** Verdict table tests for all five verdicts and all non-verdict execution failures; mismatched run ID/count/scope/hash/path/issue fixtures; absent optional metadata; atomic write and injected flow-write failure tests; validator rejects placeholders/raw streams/secrets/invalid verdicts/missing next action; mutation snapshots prove only allowed sprint state and external evidence roots change.

### Decision 6: Expose One Typed State And Progress Model Across Every Surface

- **Decision:** Define typed app requests/results/errors and smoke progress phases: `preflight`, `discovery`, `selection`, `running`, `validating_evidence`, `writing_artifact`, `completed`, `cancelled`, and `failed`, with optional suite/test identity, counts, dropped-event count, and safe diagnostic fields. Harness-native event names stay behind the sprint decoder. CLI text, stable JSON, status, `validate smoke`, `flow --to smoke`, and TUI render the same readiness, effective scope/rationale, review gate/override, prerequisites, execution state, verdict, artifact/evidence links, staleness, diagnostics, and required next action. Execution status and verdict remain separate. Status/validation perform catalog, manifest, artifact, fingerprint, and path checks without launching a run; machine-readable discovery is performed only by preview/run when dynamic suite or prerequisite facts are required.
- **Decision:** The TUI reuses existing guarded confirmation, progress, cancellation, result, issue-summary, and preview patterns. It calls app use cases with its context and persists no alternate smoke state. JSON contains no ANSI or raw process streams; human progress and operational diagnostics remain separate output channels. Stable machine error codes cover review gate/stale review, manifest/protocol/capability, scope/coverage/prerequisite, path containment, process/timeout/cancellation/cleanup, evidence, validation, and persistence/reconciliation categories.
- **Rationale:** A shared typed model is the only reliable way to meet parity while preserving non-TTY automation and TUI responsiveness.
- **Study / Source Grounding:** Handbook lines 31, 38, 40, 65, 87, 92, 97-98 and reports `02-command-architecture`, `05-error-handling`, `06-io-abstraction`, `09-terminal-ux`, and `10-logging-observability`; concrete support includes K9s's bifurcation warning, gh-cli IOStreams, and lazygit/restic progressive display. The exact UltraPlan phase vocabulary comes from architecture reasoning line 74 and has no repository-source analogue requiring a different choice.
- **Trade-Offs Accepted:** Interface renderers require separate presentation tests, and static status cannot know all dynamic discovery facts. Status must label unknown dynamic readiness rather than launch the harness implicitly.
- **Technical Debt / Future Impact:** Sprint 28 can add orchestration/recovery states by composing results rather than parsing output. Stable JSON additions must remain additive within its schema version.
- **Alternatives Rejected:** Parsing terminal output, TUI-owned verdict/state, hidden eager harness launch during status, and raw native event exposure are rejected for coupling, latency, and safety. A TUI-only capability is rejected because CLI/JSON remain the automation contract.
- **Contracts Satisfied:** CLI Surface, Documentation, Errors, Observability, Performance, Testing, Workflows; `AC-01`, `AC-03`, `AC-10`, `AC-11`; `C-03`, `C-06`, `C-08`, `C-09`.
- **Evidence Required:** Semantic parity tests over one fake use case for CLI text/JSON/status/validate/flow/TUI; stable JSON/help goldens; narrow-terminal TUI model/update tests; cancellation and slow-progress tests; static status test proving no run launch; stale/reconciliation and next-action scenarios.

### Decision 7: Verify In Layers And Gate Real Harness Use

- **Decision:** Build a layered suite: pure/table tests for gate, precedence, selection, verdict, fingerprints, validators, and paths; fixture tests for manifest/discovery/run/evidence schemas; an in-memory recording runner for orchestration; a real fake-harness child executable for OS process, timeout, cancellation, output, malformed response, and descendant cleanup; command and shared-use-case scenarios for CLI/JSON/status/validate/flow; deterministic TUI model/update tests; semantic assertions for safety/verdict logic; and explicit goldens for `smoke.md`, stable JSON envelopes, help, and selected TUI presentation. Cover success, selected-test failure, blocked prerequisite, not applicable, timeout, cancellation, malformed discovery/run JSON, missing/mismatched evidence, issue references, path/symlink escape, insufficient coverage, review gate/override, output truncation, cleanup failure, state-write failure, and redaction.
- **Decision:** Keep real `ultraplan-go-smoke` integration opt-in behind explicit environment/config gating. Missing harness, credentials, network, or runtime is a truthful skip/blocked result, never pass evidence. Required release evidence is `go test ./...`, `go test -race ./...`, and `go build ./cmd/ultraplan`, plus fake-harness suite results and either a valid gated real-harness run or its explicit blocked reason. Update embedded `smoke.md` defaults, command/help and user workflow/recovery documentation; workspace override files remain optional.
- **Rationale:** Pure and fake layers provide deterministic breadth; child processes exercise the actual safety boundary; the real harness provides environment confidence without making normal development nondeterministic.
- **Study / Source Grounding:** Handbook lines 39, 52, 75, 81-82, 96 and report `11-testing-strategy`; gh-cli acceptance scripts, lazygit's fake command runner, and Helm's golden helper support complementary test levels. `requirements.md` `AC-12`-`AC-13` supplies the mandatory failure matrix and commands.
- **Trade-Offs Accepted:** The suite has more fixture types and explicit golden maintenance. This is justified by the security-sensitive process boundary and multiple stable surfaces.
- **Technical Debt / Future Impact:** Real-harness coverage remains environment-dependent; CI may later provide a controlled gated lane. Golden updates must remain explicit to avoid blessing regressions.
- **Alternatives Rejected:** Live-harness-only testing is rejected as slow and nondeterministic. In-memory-only testing is rejected because it cannot prove process-tree cleanup or output bounds. Snapshotting every view is rejected as brittle. Silently skipping a required real run as pass is rejected as false evidence.
- **Contracts Satisfied:** Documentation, Security, Testing, Performance, Observability; `AC-02`, `AC-05`-`AC-07`, `AC-09`, `AC-11`-`AC-13`; `C-05`, `C-08`-`C-10`.
- **Evidence Required:** Named tests/fixtures for the complete matrix, explicit golden review, race output, build output, fake-harness command logs, architecture/security review checks, and opt-in real-harness result or blocked diagnostic.

## Implementation Constraints

- Keep smoke semantics, protocol types, selection, evidence, verdicts, `smoke.md`, validation, status, and flow in `internal/sprint`.
- Keep `internal/platform/process` generic and free of project, sprint, review, harness, suite, issue, evidence, and verdict types.
- Introduce no competing LLM runtime abstraction; review remains through `agentwrap`, smoke through the generic process boundary.
- Use one typed `internal/app` operation family for CLI and TUI; neither interface may invoke or parse the other.
- Support protocol major v1 only for this sprint; reject unsupported major versions and missing required capabilities before launch.
- Resolve catalog and all executable/cwd/evidence paths canonically with symlink-aware containment; revalidate returned evidence.
- Execute direct executable/argv only. Never execute joined display strings, Markdown, README, generated artifacts, or shell syntax.
- Forward an explicit bounded environment only; never inherit the host environment wholesale and never display secret values.
- Resolve effective options once with the precedence fixed in Decision 3 and retain source metadata.
- Require a valid current review. Only a current `fail`/`blocked` review has the explicit diagnostic override path; missing, malformed, or stale review remains blocked.
- Use the deterministic scope hierarchy and never let diagnostic test evidence replace required containing-suite evidence without explicit discovery-proven equivalence.
- Supervise one harness process; do not add product-owned per-test concurrency.
- Propagate context through discovery, run, progress, cleanup, and completion; all goroutines/processes must have a bounded wait and observable terminal state.
- Keep raw JSON, unrestricted streams, per-test artifacts, and issue files in cataloged harness roots.
- Keep execution status separate from verdict and compute verdict only from validated product-owned rules.
- Validate the complete candidate `smoke.md` before same-directory atomic replacement; preserve the last valid summary on execution/evidence failures.
- Update flow state only after summary commit and expose reconciliation rather than rollback if the state write fails.
- Normal smoke may modify only `smoke.md`, approved flow/status state, and manifest-declared external run/issue evidence roots; it must not modify product source/tests, governed planning inputs, harness maintenance inputs, or Git state.
- Keep embedded defaults sufficient; do not require workspace `prompts/` or `templates/` overrides.
- Use deterministic fake runtimes/harnesses in normal tests; keep live harness execution opt-in and truthful when unavailable.
- Do not implement Sprint 28 `verify`, focused-rerun recovery, full stale recovery, issue management, automatic fixes, Git operations, hosted/remote smoke, or plugin infrastructure.

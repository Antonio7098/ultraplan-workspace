## Relevant Patterns

- **Thin interfaces over shared product use cases:** The evidence repeatedly separates command parsing/rendering from reusable behavior. Helm's command delegates to an action object (`helm/pkg/cmd/install.go:132-145` and `helm/pkg/action/install.go:73-140`), while gh-cli constructs commands from a dependency factory (`gh-cli/pkg/cmdutil/factory.go:16-43`). Gdu demonstrates a shared UI boundary with separate TUI, stdout, and report implementations (`gdu/cmd/gdu/app/app.go:30-49`). For this sprint, the pressure is to make smoke requests, progress, results, and errors usable by CLI text, JSON, status/flow, and TUI without making terminal output an integration protocol.
- **A narrow, injectable volatile-process boundary:** External subprocesses are commonly isolated behind runners or plugin runtimes. Lazygit's `ICmdObjRunner` records command behavior for tests (`lazygit/pkg/commands/oscommands/cmd_obj_runner.go:18-23`), and Helm isolates subprocess plugins behind a runtime (`helm/internal/plugin/runtime_subprocess.go:65-79`). The security report establishes explicit executable plus argument arrays as the baseline (`lazygit/cmd_obj_builder.go:38`). For this sprint, the pressure is to keep generic process mechanics substitutable while leaving harness discovery, scope, evidence, and verdict meaning outside that boundary.
- **Versioned, machine-readable external contracts:** Helm's plugin metadata uses an explicit version and conversion path (`helm/internal/plugin/metadata_v1.go:24-48`, `helm/internal/plugin/metadata.go:114-130`), while age versions its plugin protocol (`age/plugin/plugin.go:159-167`). The extensibility report warns that schema failures treated only as warnings leave users with partially loaded behavior (`k9s/internal/config/plugin.go:158-164`). A smoke harness contract therefore creates pressure for strict compatibility and capability validation before execution, without implying which compatibility range or migration policy sprint reasoning should choose.
- **Context propagated through every long-running layer:** Helm wires signals to cancellation and passes the resulting context into its action (`helm/pkg/cmd/install.go:333-347`, `helm/pkg/action/install.go:284`); go-task uses `errgroup.WithContext` so sibling work observes failure (`go-task/task.go:87-89`). The state report contrasts this with detached `context.Background()` work (`chezmoi/internal/cmd/templatefuncs.go:215`). Smoke discovery, execution, output collection, progress delivery, and cleanup all face the same cancellation pressure.
- **Bounded lifecycle and output ownership:** K9s bounds workers with a semaphore (`k9s/internal/pool.go:21-37`), and opencode pairs cancellation with an explicit bounded wait (`opencode/cmd/root.go:261-279`). Lazygit streams process output line by line (`lazygit/pkg/tasks/tasks.go:189-217`) rather than accumulating an unbounded result. For this sprint, external process output, progress events, descendants, and timeout cleanup need explicit owners and bounds; the studies do not settle the exact limits.
- **Success requires validated evidence, not only process completion:** The testing report's CLI integration patterns assert observable behavior (`chezmoi/internal/cmd/main_test.go:64-174`, `gh-cli/acceptance/acceptance_test.go:26-29`), while the broader IO evidence shows how fake runners and captured streams make failure modes reproducible. Error evidence also favors typed or sentinel classifications over strings (`go-task/errors/errors.go:47-50`, `helm/pkg/storage/driver/driver.go:27-48`). Applied to smoke, a zero exit status is only one input; run identity, counts, paths, protocol shape, and coverage remain separate validation pressures.
- **Explicit configuration precedence followed by merged validation:** Go-task centralizes source precedence (`go-task/internal/flags/flags.go:314-327`), restic tracks whether flags were explicitly changed (`restic/internal/global/global.go:139,147`), and k9s validates the effective result after merging (`k9s/internal/config/k9s.go:423-451`). The sprint must reconcile catalog values, manifest defaults, environment, and explicit smoke overrides without allowing default flag values or direct environment reads to silently win.
- **Separate user output, machine output, and diagnostics:** Gh-cli's IOStreams makes output channels injectable (`gh-cli/pkg/iostreams/iostreams.go:49-54,551-568`); modern logging examples use structured fields and stderr (`helm/internal/logging/logging.go:31-71`). K9s routes a failure to both operational logging and UI notification (`k9s/internal/model/flash.go:100-103`). Smoke progress and diagnostics therefore need enough structure to support CLI, JSON, and TUI while keeping raw or sensitive process output out of summaries.
- **Layered deterministic testing around a fake executable:** Lazygit's fake command runner records calls without side effects (`lazygit/pkg/commands/oscommands/fake_cmd_obj_runner.go:17-26`), testscript exercises complete CLI workflows (`gh-cli/acceptance/acceptance_test.go:26-29`), and golden helpers protect output contracts (`helm/internal/test/test.go:43`). The evidence supports combining narrow unit cases, a deterministic fake harness process, command/use-case scenarios, and explicit snapshot or golden coverage rather than relying on a live harness for normal tests.
- **Progress as domain events rather than terminal scraping:** Lazygit's channel-driven view buffer and restic's status worker demonstrate thread-safe progressive display (`lazygit/pkg/tasks/tasks.go:31-435`, `restic/internal/ui/termstatus/status.go:197-205`). Chezmoi and gh-cli also provide non-TTY paths (`chezmoi/internal/cmd/prompt.go:20-256`, `gh-cli/internal/prompter/`). This creates pressure for typed progress that can be rendered differently by text, JSON, and TUI consumers and can stop cleanly on cancellation.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Thin generic process runner vs harness-aware runner | A generic runner is reusable, easy to fake, and keeps external mechanics independently testable, as in Lazygit's runner interface and Helm's subprocess runtime (`lazygit/pkg/commands/oscommands/cmd_obj_runner.go:18-23`, `helm/internal/plugin/runtime_subprocess.go:65-79`). | A thin boundary pushes protocol decoding, evidence checks, and lifecycle orchestration into its caller; an overly broad one couples platform code to harness semantics. | When assigning executable/argv/cwd/env/timeout/output limits versus manifest, suite, run, issue, and verdict responsibilities. |
| Strict manifest/protocol rejection vs tolerant compatibility | Versioned metadata and conversion can produce predictable evolution (`helm/internal/plugin/metadata_v1.go:24-48`, `helm/internal/plugin/metadata.go:114-130`). | Strict rejection can block otherwise usable harnesses; tolerant parsing increases ambiguity and risks silently ignoring required capabilities, as warning-only validation illustrates (`k9s/internal/config/plugin.go:158-164`). | When defining supported protocol versions, unknown fields, required capabilities, and migration diagnostics. |
| Narrowest default scope vs broader confidence | Narrow selection reduces duration, cost, process output, and resource pressure; bounded and incremental work is favored by the performance evidence (`gdu/pkg/analyze/parallel.go:13`, `yq/pkg/yqlib/stream_evaluator.go:78-113`). | Too narrow a run may omit containing-suite behavior and create misleading confidence; broad runs increase latency and cancellation exposure. | When mapping a sprint to levels, suites, and tests and when deciding what an explicit diagnostic rerun can prove. |
| Streaming/capped output vs complete captured output | Streaming keeps memory bounded and gives prompt feedback (`lazygit/pkg/tasks/tasks.go:189-217`, `age/internal/stream/stream.go:195-219`). | Truncation can hide the decisive diagnostic, and streaming adds synchronization and ordering complexity. | When choosing capture limits, excerpts, progress events, external raw evidence retention, and safe user diagnostics. |
| Fail-fast sibling cancellation vs collecting all outcomes | `errgroup.WithContext` simplifies failure propagation (`go-task/task.go:87-89`, `restic/internal/repository/repository.go:567`). | Early cancellation can reduce the amount of suite evidence and make aggregate result counts incomplete; independent waits require more manual lifecycle/error handling. | If harness discovery or execution exposes parallel sub-operations that UltraPlan itself owns rather than one harness process. |
| Central composition object vs focused constructor dependencies | A composition root makes wiring traceable (`opencode/internal/app/app.go:42-81`, `gh-cli/pkg/cmdutil/factory.go:16-43`). | Large shared config/factory structs become god objects and make unrelated commands depend on smoke collaborators (`chezmoi/internal/cmd/config.go:193-291`). | When adding process, clock, filesystem, catalog, and renderer seams to existing app and sprint construction. |
| Golden output coverage vs semantic assertions | Golden files expose intentional text/JSON/Markdown changes clearly (`helm/internal/test/test.go:43`, `go-task/task_test.go:166-169`). | They are maintenance-heavy and can obscure behavior beneath formatting diffs; unprotected updates can bless regressions. | For `smoke.md`, JSON envelopes, help text, and status/TUI rendering where both schema and human readability matter. |
| Eager preflight vs lazy dependency setup | Eager checks surface missing configuration before costly work; lazy factories avoid charging simple commands for unused dependencies (`gh-cli/pkg/cmdutil/factory.go:27-42`, `helm/pkg/action/lazyclient.go:35-53`). | Eager setup slows unrelated commands; lazy setup defers errors until an operation has begun. | When deciding what status/help/validation needs to discover versus what only an actual smoke run should initialize. |

## Anti-Patterns And Warnings

- **Shell-interpolated or prose-derived execution:** The security evidence treats `exec.Command(executable, args...)` as the baseline (`lazygit/cmd_obj_builder.go:38`). Do not turn README text, Markdown, user-edited artifacts, or a joined display command into executable shell input.
- **Exit code as the verdict:** Structured error and test evidence separates process completion from product meaning (`go-task/errors/errors.go:47-50`, `gh-cli/acceptance/acceptance_test.go:26-29`). A zero exit code without valid identity, evidence, counts, and required coverage must not imply a smoke pass.
- **Detached cancellation or fire-and-forget cleanup:** Detached contexts and unowned goroutines leave work running after cancellation (`chezmoi/internal/cmd/templatefuncs.go:215`, `dive/cmd/dive/cli/internal/command/adapter/resolver.go:70`). Every discovery/run process, progress producer, and descendant-cleanup path needs an observable completion path.
- **Unbounded capture or goroutine creation:** The concurrency report calls out unbounded per-item goroutines and waits without timeouts (`gh-cli/pkg/cmd/extension/manager.go:196-206`); the performance report favors bounded workers and streaming (`k9s/internal/pool.go:26-48`, `lazygit/pkg/tasks/tasks.go:189-217`). Raw stdout/stderr should not accumulate without limits in the product process.
- **Global mutable smoke dependencies:** Global config, caches, and registries cause test contamination and implicit ordering (`rclone/fs/config.go:793`, `rclone/fs/cache/cache.go:16-21`). Avoid package-level harness paths, process runners, mutable verdict state, or UI channels when they can be composed per application/run.
- **Context as a service locator:** K9s's context keys trade compile-time guarantees for convenience (`k9s/internal/keys.go:10-38`). Context should carry lifecycle and request-scoped values; process, catalog, filesystem, and clock collaborators should remain explicit unless reasoning identifies a concrete exception.
- **Warning-only contract validation:** K9s demonstrates how schema failures that merely warn can leave extensions mysteriously unavailable (`k9s/internal/config/plugin.go:158-164`). Unsupported protocol versions, malformed discovery, escaping evidence paths, and absent required capabilities need explicit classification rather than partial continuation that could look successful.
- **Diagnostic leakage into output or artifacts:** Debug output on stdout breaks scripting (`fzf/src/core.go:325`), and direct environment access bypasses centralized redaction/precedence (`opencode/internal/config/config.go:163`). Safe argv must not become a route for secrets, and unrestricted process streams should remain external evidence rather than sprint Markdown.
- **TUI as an alternate workflow engine:** The command study identifies bifurcated TUI-only commands as invisible to automation (`k9s/internal/view/command.go:176`), while IO evidence shows direct terminal coupling harms tests. TUI code should not shell out to the product, parse CLI text, or synthesize a separate verdict/state model.
- **Over-abstraction before a proven boundary:** The studies support interfaces for volatile dependencies, but warn that broad UI, filesystem, and config interfaces can become costly (`restic/internal/fs/interface.go:10-31`, `chezmoi/internal/cmd/config.go:193-291`). The harness protocol does not by itself justify a plugin marketplace, generic workflow engine, or interface for every helper.

## Design Pressures

- The same smoke operation must remain coherent across CLI text, stable JSON, status/flow, and TUI, favoring typed requests, progress, results, and errors over output parsing.
- Product-owned smoke meaning and generic process mechanics need a boundary narrow enough to preserve dependency direction but rich enough to enforce executable, argv, cwd, environment, timeout, capture, and cleanup controls.
- The harness is independently persisted and may evolve independently, so discovery and run responses need protocol/version/capability validation before evidence is trusted.
- Review gating and smoke execution are separate outcomes: a process can complete while the gate, coverage, evidence, or synthesized verdict is non-passing.
- Narrow scope reduces runtime cost, but selected coverage must remain sufficient and explainable; focused investigation evidence cannot silently replace containing-suite evidence.
- Cancellation must remain responsive from CLI/TUI through discovery, process execution, progress delivery, descendant cleanup, and final state reporting.
- Raw external evidence may be large or sensitive, while product surfaces still need useful excerpts, links, counts, identity, and safe diagnostics.
- The prior valid `smoke.md` and durable state must survive malformed output, timeout, cancellation, and partial persistence attempts, creating pressure for validate-before-replace behavior.
- Environment forwarding must be sufficient for real credentials and runtime prerequisites but bounded and redacted in displays, logs, JSON, and Markdown.
- Fake-harness tests need to exercise actual process-boundary behavior without making normal tests depend on the real harness, runtime credentials, network, or an interactive terminal.
- TUI progress can be high frequency, while JSON and status consumers need stable, deterministic summaries; event delivery should not permit slow consumers to block process cleanup.
- Startup and status inspection should avoid unnecessary harness execution or expensive initialization while still reporting truthful readiness.

## Open Questions For Reasoning

- What is the smallest process-runner request/result contract that supports executable, argv, cwd, bounded environment, timeout, output limits, cancellation, exit status, and cleanup facts without importing smoke semantics?
- Which protocol versions and manifest/discovery/run fields are required, optional, forward-compatible, or fatal when unknown?
- How will catalog root, manifest path, executable path, working directory, and returned evidence paths be canonicalized and proven contained, including symlink cases?
- What exact precedence should apply among manifest defaults, workspace configuration, environment-derived values, CLI flags, flow settings, and TUI selections, and how will the effective source be displayed safely?
- What algorithm defines “narrowest sufficient” scope, and what evidence proves an explicit test or suite override does or does not satisfy required containing-suite coverage?
- Which review verdicts are non-blocking, what freshness evidence is mandatory, and what fields must record a diagnostic override without normalizing the review failure away?
- What timeout and output-capture defaults are appropriate for discovery versus execution, and which limits must be configurable?
- What descendant-process cleanup guarantee is feasible on each supported platform, and how will cleanup failure affect execution status, verdict synthesis, and diagnostics?
- What structured progress vocabulary is stable enough for CLI, JSON, flow/status, and TUI while remaining independent of harness-specific event names?
- How will slow or disconnected progress consumers be handled without blocking the process runner or losing terminal completion facts?
- Which errors require sentinels, typed fields, or stable machine codes for review gate, manifest, discovery, timeout, cancellation, evidence, containment, and blocked prerequisites?
- What is the exact commit point between validated external evidence, atomic `smoke.md` replacement, and flow-state replacement, and how are partial failures represented without corrupting the last valid artifact?
- Which evidence identity fields are mandatory when hashes are unavailable, and how are run ID, summary path, detail path, issue references, and reported counts cross-checked?
- What safe argv and environment representation remains useful while preventing secrets embedded in argument values, URLs, paths, or environment variables from leaking?
- Which fake-harness fixtures need to be real child processes to validate timeout, signal, output limits, malformed streams, and descendant cleanup, and which cases can use an in-memory runner?
- What output should be golden-tested versus semantically asserted, especially for editable `smoke.md`, stable JSON, progress text, and narrow-terminal TUI views?
- Is any concurrency inside UltraPlan needed for one smoke run, or should the harness exclusively own suite/test parallelism while UltraPlan supervises one process?
- Which readiness checks should status and validation perform without launching the harness, and which facts can only be known through machine-readable discovery?

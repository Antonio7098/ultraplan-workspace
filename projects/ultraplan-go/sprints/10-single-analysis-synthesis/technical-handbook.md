## Relevant Patterns

- **Thin CLI With Study-Owned Behavior:** Mature CLIs keep command handlers limited to parsing, dependency construction, delegation, output rendering, and error mapping. Evidence from project structure and command studies includes chezmoi `main.go:26-34`, gh-cli `cmd/gh/main.go:6`, helm `pkg/cmd/install.go:132-145`, and gh-cli `pkg/cmdutil/factory.go:16-43`. For this sprint, this pattern pressures `study run` and `study synthesize` reasoning to keep workflow semantics in the study module and command wiring thin.
- **Explicit Composition Root And Runtime Seam:** Manual DI is the norm in high-scoring Go CLIs; app or factory roots assemble dependencies, and volatile boundaries get interfaces. Evidence includes opencode `internal/app/app.go:42-81`, gh-cli `pkg/cmd/factory/default.go:26-46`, go-task `executor.go:22-24`, and restic `internal/backend/backend.go:19-90`. For this sprint, the runtime dependency should be inspectable and fakeable without forcing command code to construct runtime internals.
- **Output Validation As Product Gate:** Evidence reports stress structured error handling and validation-driven success, not exit-code-only success. The error report cites go-task `errors/errors.go:47-50`, restic `internal/errors/fatal.go:10`, and helm `pkg/storage/driver/driver.go:27-36`; the security and philosophy reports reinforce explicit trust boundaries through opencode `internal/permission/permission.go:44-108` and restic `internal/options/secret_string.go:15-20`. For this sprint, runtime completion should remain distinct from report existence and report validator acceptance.
- **Injectable IO And Fake External Systems:** Testable CLI layers use writer injection and fake dependencies rather than hardcoded `os.Stdout` or real services. Evidence includes gh-cli `pkg/iostreams/iostreams.go:551-568`, go-task `executor.go:553-564`, restic `internal/ui/mock.go:10-53`, and lazygit `pkg/commands/oscommands/fake_cmd_obj_runner.go:17-26`. For this sprint, command tests and study tests can exercise fake runtime paths and deterministic stdout/stderr behavior.
- **Context-Aware Single Execution Without Batch Machinery:** Context propagation and cancellation are visible in high-scoring projects, but concurrency machinery is applied when there is fan-out. Evidence includes gh-cli `internal/ghcmd/cmd.go:142`, helm `pkg/cmd/install.go:333-347`, go-task `task.go:89`, and restic `internal/repository/repository.go:567`. For this sprint, single analysis and single synthesis need context-aware runtime calls and event draining, while worker pools and full task scheduling remain outside the selected scope.
- **Safe, Structured Diagnostics:** Observability and security evidence converge on safe diagnostics: structured fields, stderr separation, secret redaction, and no parsing of human runtime text for control flow. Evidence includes helm `internal/logging/logging.go:31-66`, k9s `internal/slogs/keys.go:6-231`, gh-cli `pkg/iostreams/iostreams.go:52-54`, helm `pkg/registry/transport.go:37-41`, and restic `internal/options/secret_string.go:15-20`.
- **Configuration Precedence Before Work Launch:** Mature CLIs resolve and validate effective config before expensive work. Evidence includes chezmoi `internal/cmd/config.go:2253-2287`, go-task `internal/flags/flags.go:314-327`, restic `internal/global/global.go:139,147`, and opencode `internal/config/config.go:609-641`. For this sprint, runtime/provider/model/timeout settings should be resolved before launching agent-backed execution.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Thin command handlers vs. command-contained logic | Better testability and clearer ownership; supported by helm `pkg/cmd/install.go:132-145` and gh-cli `pkg/cmdutil/factory.go:16-43` | Requires service/use-case methods and explicit result types instead of quick inline `RunE` code | `study run` and `study synthesize` command wiring could accidentally absorb prompt, validation, and runtime mapping logic |
| Generic runtime boundary vs. product-aware runtime helpers | Keeps platform runtime reusable and avoids reverse imports, matching `internal/` dependency direction evidence like yq `cmd/root.go:9` and restic `internal/restic/repository.go:18` | Study code must translate study metadata into generic runtime requests and validators | The sprint must pass task metadata and expected artifact validation without making platform runtime know studies, dimensions, or reports |
| Agentwrap validation hook vs. study post-run validation | Runtime-level validation can fail early and attach validation events; post-run validation keeps report semantics fully product-owned | Duplicating checks in both places can produce inconsistent diagnostics; placing all checks in runtime can leak study semantics | Reasoning must decide where generic expected-output validation ends and study-owned report validation begins |
| Minimal single-task orchestration vs. reusable batch scheduler | Smaller surface and easier review for one analysis/synthesis task | Future batch/run-loop may need additional state transitions or scheduling hooks later | Sprint scope excludes `run-all`, worker pools, retries across process restarts, locks, and automatic synthesis |
| Structured error taxonomy vs. simple returned errors | Enables exit-code mapping, user/actionable diagnostics, and `errors.Is`/`errors.As` inspection, as in go-task `errors/errors.go:47-50`, rclone `fs/fserrors/error.go:22-29`, and restic `internal/errors/fatal.go:10` | More types and mapping code than plain `fmt.Errorf` | Runtime failure, validation failure, inapplicable skip, usage errors, and missing output need distinct command outcomes |
| Deterministic text output vs. rich progress UX | Stable command tests and script-friendly behavior; aligns with IO and testing evidence from gh-cli `pkg/iostreams/iostreams.go:551-568` and testscript examples | Less interactive feedback during long runtime operations | Single execution commands need clear summaries and diagnostics without introducing complex TUI/progress machinery |

## Anti-Patterns And Warnings

- **Do not let CLI handlers become orchestration bodies:** Reports flag long `RunE` functions and command-contained business logic as weak architecture, including opencode `cmd/root.go:49-183`, yq `cmd/evaluate_sequence_command.go:152`, and restic `cmd/restic/cmd_backup.go` large run functions. Sprint reasoning should guard against moving prompt building, report gating, or runtime semantics into `internal/app`.
- **Do not put study semantics in platform runtime:** The structure report favors unidirectional dependency flow such as CLI to domain and no reverse imports (`cmd/root.go:9`, `internal/restic/repository.go:18`). Platform runtime should not import study types or enforce synthesis gating/report semantics.
- **Do not treat runtime exit success as product success:** Error, security, and philosophy evidence favors explicit trust and validation gates, such as restic `internal/errors/fatal.go:10`, opencode `internal/permission/permission.go:44-108`, and restic `internal/options/secret_string.go:15-20`. This sprint's reasoning must preserve artifact existence and validator success as separate checks.
- **Do not hardcode stdout/stderr or real runtime in tests:** IO and testing reports warn that direct `os.Stdout`/`os.Stderr` and sparse integration-only tests reduce confidence; examples include rclone `cmd/ls/ls.go:42`, age `cmd/age/tui.go:31`, and opencode sparse coverage `internal/tui/theme/theme_test.go:1-89`. Use fake runtime and injectable output seams for default tests.
- **Do not introduce speculative plugin/scheduler frameworks:** Extensibility and philosophy reports show registries/plugins are expensive and justified only by scope, with examples like helm `internal/plugin/runtime_subprocess.go:65-79` and rclone `fs/registry.go:407`. Single execution does not need new plugin systems, broad registries, or worker-pool abstractions.
- **Do not parse human runtime text for control flow:** Error handling evidence prefers wrapped and typed errors over string inspection, including rclone `fs/fserrors/error.go:22-29`, go-task `errors/errors.go:47-50`, and gh-cli `api/client.go:42-49`. Runtime classification should come from the platform/agentwrap abstraction, not message text.
- **Do not let event channels block the final result path:** Context and concurrency reports emphasize cleanup and waiting discipline, including opencode `cmd/root.go:130,252`, lazygit `pkg/tasks/tasks.go:338-357`, and restic `internal/repository/repository.go:567`. Single execution still needs to consume or drain runtime events and obtain the final result.
- **Do not leak secrets or unsafe native payloads in diagnostics:** Security and observability evidence includes restic `internal/options/secret_string.go:15-20`, helm `pkg/registry/transport.go:37-41`, and gh-cli `status.go:332-338`. Runtime/provider/model diagnostics should stay safe and redacted.

## Design Pressures

- The sprint needs one-shot execution semantics while preserving future compatibility with batch/run-loop state vocabulary.
- CLI output must be deterministic and script-friendly, but runtime execution may produce streaming events and diagnostics.
- Study code needs enough runtime metadata for observability without moving study/domain concepts into `internal/platform/runtime`.
- Runtime request construction must combine prompt text, work directory, model/provider/timeout, permissions, health/capability requirements, metadata, and output validation while staying reviewable.
- Markdown source applicability creates a product-level skip path distinct from runtime failure, validation failure, and success.
- Synthesis has a preflight gate: applicable per-source reports must already be present and valid before runtime invocation.
- Agentwrap owns OpenCode supervision, event projection, policy, permission translation, health checks, and validation wrappers; UltraPlan should not create a parallel supervisor.
- Tests must cover runtime-facing behavior with fakes and should not require OpenCode, provider credentials, network access, wall-clock sleeps, or real subprocesses.
- Diagnostics need to be actionable enough to identify report paths and failed checks while omitting unsafe prompt/native/runtime payloads unless explicitly safe.
- Implementation should remain minimal because the sprint excludes `run-all`, worker pools, durable restart recovery, summary generation, code extraction, repair workflows, and target/sprint execution.

## Open Questions For Reasoning

- Where should generic expected-output validation be attached to the runtime request, and where should study-owned per-source/final report validation run after completion?
- What exact status/result fields are needed for single analysis and synthesis without pre-building the full batch/run-loop result model?
- How should inapplicable Markdown analysis be represented in service results and CLI output so it is clearly successful but skipped without invoking runtime?
- Which runtime event consumption pattern is sufficient for a single run so event channels cannot block while avoiding a general event processing subsystem?
- What metadata keys should be included in runtime requests to preserve observability without leaking study types into `internal/platform/runtime`?
- How should runtime failure diagnostics and validation failure diagnostics differ in stdout, stderr, result fields, and exit-code mapping?
- Should study-owned validators be exposed as agentwrap custom validators, used only as post-run checks, or split between both mechanisms?
- What is the minimum fake-runtime interface needed for study tests and CLI tests while keeping app composition explicit?
- How should directory-source and Markdown-source work directories differ in runtime requests while preserving source/document isolation?
- If runtime validation reports missing output, how should reasoning avoid double-reporting when study post-run validation also checks missing output?

## Relevant Patterns

- **Thin CLI, sprint-owned behavior:** Sprint reasoning should inspect whether `internal/app` only parses arguments, wires dependencies, renders output, and maps errors while selection parsing, subset validation, prompt rendering, runtime flow rules, and state transitions stay inside `internal/sprint`. The selected reports repeatedly show this boundary: `01-project-structure` cites thin entrypoints at chezmoi `main.go:26-34`, gh-cli `cmd/gh/main.go:6`, and k9s `cmd/root.go:112-127`; `02-command-architecture` cites helm command wrappers delegating to action logic at `pkg/cmd/install.go:132-145` and gh-cli factory wiring at `pkg/cmdutil/factory.go:16-43`.
- **Command factories with explicit dependencies:** Select-stage commands need enough dependencies for project catalog loading, sprint storage, prompt rendering, validation, clock/runtime fakes, and output streams. Evidence favors factory/manual DI over global state: gh-cli uses factory dependency functions at `pkg/cmd/factory/default.go:26-46`, go-task uses functional options at `executor.go:22-24`, and helm constructs action configuration at `pkg/action/action.go:118-146`. The sprint question is how much should be passed into `internal/sprint.Service` vs kept in app wiring.
- **Validated artifact completion gate:** Runtime success must not imply sprint-stage success. Evidence from error, testing, and runtime-oriented reports supports explicit validation after execution: `05-error-handling` cites typed/task errors and exit-code mapping in go-task `errors/errors.go:47-50` and gh-cli `internal/ghcmd/cmd.go:44-49`; `11-testing-strategy` cites command-level behavior tests in chezmoi `internal/cmd/main_test.go:64-174`; scope context from TRD requires planning completion only after runtime success, expected artifact existence, validator pass, and atomic `flow-state.json` update.
- **Injectable IO, filesystem, and runtime seams:** Prompt preview, validation diagnostics, and fake-runtime flow tests require command paths that can capture output and avoid real runtime calls. `06-io-abstraction` cites gh-cli `IOStreams.Test()` at `pkg/iostreams/iostreams.go:551-568`, go-task stream options at `executor.go:541-577`, restic filesystem interfaces at `internal/fs/interface.go:10-31`, and chezmoi `System` at `internal/chezmoi/system.go:25`. The sprint can use this evidence to reason about minimal seams without inventing broad framework packages.
- **Context propagation for runtime-backed flow:** Non-dry-run `flow --to sprint-index` may call the generic runtime and should carry cancellation/timeout through the boundary. `07-state-context` cites root context execution in gh-cli `internal/ghcmd/cmd.go:142`, `errgroup.WithContext` in go-task `task.go:89`, signal-context wiring in helm `pkg/cmd/install.go:333-347`, and cleanup signal handling in restic `cmd/restic/cleanup.go:24-38`.
- **Explicit config precedence and redaction:** Select-stage prompt and flow commands may accept config/model/runtime overrides, but diagnostics must not leak provider or secret values. `04-configuration-management` cites explicit precedence logic in chezmoi `internal/cmd/config.go:2253-2287`, generic precedence in go-task `internal/flags/flags.go:314-327`, and validation in opencode `internal/config/config.go:609-641`; `13-security` cites redaction-oriented types and scrubbing in restic `internal/options/secret_string.go:15-20` and helm `pkg/registry/transport.go:37-41`.
- **Fixture-first command and validation tests:** Sprint 18 has many invalid-reference and unsupported-stage cases, which suit table-driven validation tests plus command-level output tests. `11-testing-strategy` cites testscript integration in chezmoi `internal/cmd/main_test.go:64-174`, gh-cli acceptance scripts at `acceptance/acceptance_test.go:26-29`, golden helpers in helm `internal/test/test.go:43`, and mock backend patterns in restic `internal/backend/mock/backend.go:14-26`.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Keep select-stage logic in `internal/sprint` instead of extracting global planning/validation/prompt packages | Preserves product ownership and mirrors thin CLI/deep domain evidence from chezmoi `main.go:26-34`, yq `cmd/root.go:9`, and architecture evidence that global layers fracture context | Some repeated helper code may appear in `internal/project` and `internal/sprint` until stable shared mechanics are proven | When implementing sprint-index parsing, validation, prompts, and flow without creating forbidden global packages |
| Central service struct/factory vs narrow function parameters | A service/factory makes command wiring consistent and test setup simpler, as seen in gh-cli `pkg/cmdutil/factory.go:16-43`, go-task `executor.go:27-84`, and opencode `internal/app/app.go:25-40` | Large config/service structs can become god objects; reports call out chezmoi `internal/cmd/config.go:193-291` and the risk of huge central objects | When choosing the surface of `internal/sprint.Service`, store/runtime collaborators, and app command constructors |
| Strict validation with actionable diagnostics vs permissive Markdown acceptance | Strict checks catch missing sections, catalog mismatches, placeholders, and unsupported stages before later planning stages depend on invalid context; this aligns with typed/error-chain evidence from go-task `errors/errors.go:47-50` and restic `internal/errors/fatal.go:10` | Over-strict Markdown parsing can reject useful human edits or create brittle tests if it depends on exact prose | When defining required `sprint-index.md` sections, subset checks, excluded context requirements, and repair guidance |
| Runtime-backed flow through a generic platform boundary vs direct OpenCode/agentwrap adapter calls in sprint logic | Keeps `internal/sprint` product-owned and platform runtime generic; selected reports support adapter/decorator boundaries such as dive `cmd/dive/cli/internal/command/adapter/resolver.go:17-21` and opencode pub/sub separation `internal/pubsub/broker.go:10-19` | More indirection and fake setup in tests; failures must be mapped back into sprint diagnostics carefully | When implementing non-dry-run `flow --to sprint-index` and fake-runtime tests |
| Golden/fixture command tests vs looser substring assertions | Golden/fixture tests protect scriptable CLI output and prompt previews; evidence includes go-task `task_test.go:166-169`, helm `internal/test/test.go:43`, and rclone `cmd/bisync/bisync_test.go:1435-1479` | Golden files require explicit update workflow and can become noisy if prose is overasserted | When testing prompt previews and help/output for `validate`, `prompt`, and `flow --dry-run` |
| Structured logs/diagnostics vs minimal stderr messages | Structured facts make validation, runtime failures, and flow-state updates inspectable; evidence includes k9s structured keys `internal/slogs/keys.go:6-231` and helm slog handler `internal/logging/logging.go:31-66` | More plumbing and redaction discipline; too much log output can break calm CLI UX | When surfacing validation failures, runtime metadata summaries, and failed flow-state writes |

## Anti-Patterns And Warnings

- **Do not let CLI handlers become business logic:** Reports warn about long `RunE` or `main` functions, including opencode `cmd/root.go:49-183`, yq `cmd/evaluate_sequence_command.go:152`, and age `cmd/age/age.go:105-321`. For this sprint, argument parsing and output rendering belong in `internal/app`; subset rules, stage rules, prompt content, and flow-state transitions should not.
- **Do not introduce global mutable config, logger, or planning registries:** `03-dependency-injection` flags rclone `fs/config.go:14-51`, yq `pkg/yqlib/lib.go:13`, and cache singletons such as rclone `fs/cache/cache.go:16-21` as testability risks. Sprint reasoning should avoid global selected-context registries or stage registries that bypass explicit project-index input.
- **Do not treat runtime exit success as stage success:** Product scope and evidence both reject this. The selected reports emphasize validation and typed error mapping, with examples including go-task `errors/errors_task.go:13-32`, restic `internal/errors/fatal.go:10`, and gh-cli `internal/ghcmd/cmd.go:44-49`. A generated `sprint-index.md` still needs existence, non-empty/placeholder checks, catalog subset validation, and atomic state write.
- **Do not bypass injectable output and filesystem seams:** Direct output or filesystem access undermines command tests. `06-io-abstraction` warns about hardcoded `os.Stdout` in rclone `cmd/ls/ls.go:42`, `os.Stderr` in age `cmd/age/tui.go:31`, and template stderr bypass in chezmoi `internal/cmd/templatefuncs.go:296`.
- **Do not use shell/OpenCode/provider calls directly from sprint logic:** Security evidence favors explicit trust boundaries and controlled execution, including opencode permissions at `internal/permission/permission.go:44-108`, banned/allowlist filtering at `internal/llm/tools/bash.go:41-55`, and go-task's controlled interpreter at `internal/execext/exec.go:59-66`. Sprint logic should only use the generic runtime boundary selected by project architecture.
- **Do not silently accept unknown or out-of-scope stages:** The sprint scope explicitly excludes implementation, smoke, review, issues, and Git mutation. Security and philosophy reports warn against implicit trust and accumulated complexity; see yq default-allow security flags `pkg/yqlib/security_prefs.go:3-7` as a default-allow caution and lazygit `VISION.md:97` for rejecting complexity that does not serve the core purpose.
- **Do not make prompt-preview tests brittle by asserting full generated prose:** The testing report warns about fragile regex/output assertions, including gh-cli `pkg/cmd/issue/list/list_test.go:88-91` and inline fixture brittleness in mitchellh-cli `cli_test.go:1586-1614`. For this sprint, assertions should focus on required variables, catalog entries, output path, non-goals, and no-mutation instructions.

## Design Pressures

- Sprint reasoning must preserve `internal/sprint` ownership while still using `internal/project` as the catalog source for subset validation.
- `sprint-index.md` is editable Markdown and authoritative context; parsing must tolerate human formatting enough to be useful while rejecting missing sections, placeholders, and catalog mismatches.
- Prompt rendering must be deterministic and previewable without runtime invocation or artifact mutation.
- Non-dry-run flow must treat runtime execution as one step in a larger product gate, not as completion by itself.
- Flow-state updates must remain strict, versioned, and atomic, preserving prior valid state on write failure.
- Diagnostics must include enough section/name/path context for repair while preserving redaction and stdout/stderr separation.
- Unsupported future stages must be rejected without accidentally modeling execution, smoke, review, issues, or Git mutation as current Planning Phase 2 behavior.
- Tests need fake runtime coverage for success, missing artifact, invalid artifact, runtime failure, validation failure, dry-run, and state write failure without OpenCode, network, credentials, or Git.
- Reuse should favor mechanical infrastructure such as workspace path safety and atomic writes, not study-owned report, rating, prompt, scheduler, or run-loop semantics.
- Catalog subset validation creates a dependency on `project-index.md` shape; errors should distinguish missing catalog entries from path mismatches and section omissions.

## Open Questions For Reasoning

- What is the minimal `internal/sprint` data model for selected contracts, evidence reports, reasoning templates, review protocols, and excluded context without creating a generic catalog abstraction outside `internal/project`?
- How strict should Markdown table parsing be for selected context, especially around table column names, whitespace, duplicate rows, and extra explanatory prose?
- Should validation collect all section/catalog failures before returning, or fail fast for missing required sections that make later checks unreliable?
- What exact diagnostic structure should represent section, selected entry, observed problem, referenced path/name, and suggested repair while preserving existing exit-code conventions?
- Where should the runtime seam live so `internal/sprint` can invoke generic platform runtime but never know about OpenCode, agentwrap adapters, shell execution, or provider APIs?
- How should `flow --to sprint-index --dry-run` describe planned stages and inputs without writing `sprint-index.md` or marking state complete?
- What should a successful select-stage flow mark as ready, complete, missing, failed, or skipped, especially when reasoning templates are empty by explicit selection?
- Should prompt preview output be treated as stable golden text, or should tests assert only required variables and invariants to avoid brittle prose fixtures?
- How should unsupported stages like `technical-handbook`, `area-reasoning`, `reasoning`, and `plan` be represented before later sprints implement them, if at all, without violating Sprint 18 scope?
- What is the smallest reusable file/atomic-write helper that avoids importing study semantics while preserving Sprint 17 flow-state guarantees?

## Sprint Technical Handbook Summary

- **Relevant Patterns:** Thin CLI adapters with domain-owned behavior; command factories with explicit dependencies; validated artifact completion gates; injectable IO, filesystem, and runtime seams; context propagation for runtime-backed commands; explicit config precedence and redaction; fixture-first command and validation tests.
- **Important Trade-Offs:** Keep select-stage logic in `internal/sprint` rather than global planning packages; use a central sprint service carefully without creating a god object; enforce strict subset validation while tolerating normal Markdown table whitespace; use generic runtime indirection instead of direct OpenCode calls; protect prompt and CLI output with focused fixture or invariant tests rather than brittle full-prose assertions.
- **Warnings / Anti-Patterns:** Do not let `internal/app` contain business rules; do not introduce global mutable registries; do not mark a stage complete on runtime success alone; do not bypass IO/filesystem/runtime seams; do not invoke shell/OpenCode/provider APIs directly from sprint logic; do not silently accept unsupported stages.
- **Evidence Confidence:** High for package boundaries, command architecture, DI, error handling, IO seams, state/context, observability, testing, and security because the handbook cites multiple mature Go CLIs and concrete source references. Medium for terminal UX and philosophy because those findings are more qualitative, but they align strongly with the project requirements.

## Final Decisions

### Decision 1: Keep Select-Stage Ownership In `internal/sprint`

- **Decision:** Implement `sprint-index` domain structures, parsing, validation, prompt rendering, flow execution rules, stage-state updates, and sprint service use cases inside `internal/sprint`. `internal/sprint` may depend on `internal/project`, `internal/workspace`, and generic platform/runtime/config/filesystem helpers, but it must not depend on `internal/study` or introduce global `internal/catalog`, `internal/planning`, `internal/workflow`, `internal/stages`, `internal/validation`, `internal/reports`, or `internal/prompts` packages.
- **Rationale:** Sprint 18 is product behavior for sprint planning state. The project architecture requires module-owned behavior and the requirements explicitly assign selection, validation, prompts, and flow rules to `internal/sprint`.
- **Study / Source Grounding:** `technical-handbook.md` cites thin entrypoints in chezmoi `main.go:26-34`, gh-cli `cmd/gh/main.go:6`, k9s `cmd/root.go:112-127`, and yq `cmd/root.go:9`; command/action separation in helm `pkg/cmd/install.go:132-145`; DI patterns in gh-cli `pkg/cmd/factory/default.go:26-46` and opencode `internal/app/app.go:42-81`; and philosophy evidence from lazygit `VISION.md:97` and gh-cli `docs/project-layout.md:68-69`.
- **Trade-Offs Accepted:** This accepts some sprint-local helper code instead of global reuse, in exchange for preserving boundaries and avoiding premature abstractions.
- **Technical Debt / Future Impact:** Later planning stages may reveal reusable non-semantic helpers, but extraction must wait until concrete repetition exists and must remain product-neutral.
- **Alternatives Rejected:** A global planning/workflow package is rejected because it centralizes product rules away from owned state. Reusing `internal/study` services or report validators is rejected because study semantics are explicitly non-reusable for Planning Phase 2. Putting selection logic in `internal/app` is rejected because CLI adapters must remain thin.
- **Contracts Satisfied:** Architecture, Testing, Security, Workflows, REQ-AC-23, REQ-AC-24, REQ-AC-25, REQ-AC-26, REQ-C.
- **Evidence Required:** Import review proving allowed dependency direction; unit tests under `internal/sprint`; command tests proving `internal/app` delegates without owning business rules; Architecture Review protocol checks.

### Decision 2: Validate `sprint-index.md` As Structured Editable Markdown With Catalog Subset Checks

- **Decision:** Implement an artifact parser in `internal/sprint/index.go` that reads required sections and Markdown tables for sprint scope, selected contracts, selected evidence reports, selected reasoning templates, required review protocols, prior carry-forward notes, and excluded context. Validation in `internal/sprint/validation.go` must reject missing, empty, placeholder-containing, section-incomplete, duplicate, malformed, or uncataloged selections. Selected contracts, evidence reports, reasoning templates, and review protocols must match the corresponding `project-index.md` catalog entries by selected name and, where a path is present, by path. Diagnostics must be deterministic and include section, selected entry, referenced name/path, observed problem, and suggested repair.
- **Rationale:** `sprint-index.md` is the authoritative selected context for later stages. Later planning artifacts cannot trust it unless it is both human-editable and semantically checked against the project catalog.
- **Study / Source Grounding:** `technical-handbook.md` cites error and diagnostics evidence from gh-cli `internal/ghcmd/cmd.go:44-49`, go-task `errors/errors.go:47-50`, and restic `internal/errors/fatal.go:10`; testing evidence from chezmoi `internal/cmd/main_test.go:64-174`; and anti-pattern warnings against treating runtime output or loose text as success.
- **Trade-Offs Accepted:** Strict validation can reject loosely edited Markdown, but deterministic repair guidance makes failures actionable and prevents invalid context from flowing downstream.
- **Technical Debt / Future Impact:** Parser strictness may need tuning after real user edits. Keep tests focused on section/table semantics, whitespace tolerance, duplicate handling, and catalog matching rather than exact prose.
- **Alternatives Rejected:** Existence-only validation is rejected because it would pass invalid selected context. Free-form prose parsing is rejected because diagnostics would be unreliable. Converting `sprint-index.md` to JSON/YAML is rejected because the product requires editable Markdown artifacts and the template is Markdown.
- **Contracts Satisfied:** Errors, Security, Testing, Documentation, Persistence And Migrations, REQ-AC-07, REQ-AC-08, REQ-AC-09, REQ-AC-10, REQ-AC-11, REQ-AC-12, REQ-AC-13, REQ-AC-14, REQ-AC-30, REQ-AC-32.
- **Evidence Required:** `sprint_index_test.go` covering valid content, missing required sections, empty file, placeholders, duplicate rows, invalid contract/evidence/template/protocol selections, path mismatches, explicit no-template selection, excluded context, and deterministic diagnostic ordering.

### Decision 3: Render Deterministic Runtime-Free Prompt Previews For `sprint-index`

- **Decision:** Implement `internal/sprint/prompts.go` so `ultraplan sprint <project> <sprint> prompt sprint-index` renders a deterministic prompt preview without invoking runtime and without writing `sprint-index.md`. The prompt must include project slug, sprint slug, sprint path, requirements path, docs list, roadmap path, project-index path, available catalog entries, prior sprint carry-forward constraints from the selected context, output path, workspace-relative paths by default, instructions to select only cataloged context, and explicit no-mutation instructions for project docs, project index, roadmap, prior reviews, source repositories, Git state, and implementation files.
- **Rationale:** Prompt preview is a safety boundary before runtime execution. It must be inspectable, deterministic, and non-mutating so users can verify inputs before generating an artifact.
- **Study / Source Grounding:** `technical-handbook.md` cites IO seam evidence from gh-cli `pkg/iostreams/iostreams.go:551-568`, calm output evidence from chezmoi `internal/cmd/prompt.go:124-137` and restic `internal/ui/terminal.go:10-34`, configuration precedence evidence from go-task `internal/flags/flags.go:314-327`, and security/redaction evidence from restic `internal/options/secret_string.go:15-20` and helm `pkg/registry/transport.go:37-41`.
- **Trade-Offs Accepted:** Prompt tests should assert required variables and invariants rather than full prose; this protects contract content without making harmless wording changes expensive.
- **Technical Debt / Future Impact:** Later prompt stages should reuse only mechanical rendering patterns, not copy Sprint 18 wording blindly.
- **Alternatives Rejected:** Building prompt text in CLI handlers is rejected because prompt logic belongs to `internal/sprint`. Invoking runtime from `prompt sprint-index` is rejected because prompt preview must be runtime-free. Embedding full environment/config dumps or absolute paths by default is rejected because it risks secrets and non-relocatable prompts.
- **Contracts Satisfied:** Configuration, Security, Documentation, CLI Surface, Testing, REQ-AC-02, REQ-AC-15, REQ-AC-16, REQ-AC-17, REQ-AC-27, REQ-AC-28, REQ-AC-33.
- **Evidence Required:** Prompt unit tests and command tests asserting required variables, selected catalog entries, workspace-relative paths, output path, non-goals, no-mutation instructions, no runtime calls, no artifact writes, stdout/stderr separation, and no ANSI output.

### Decision 4: Gate Non-Dry-Run Flow On Requirements Validation, Generic Runtime, Artifact Validation, And Atomic State

- **Decision:** Implement `flow --to sprint-index` so dry-run reports planned stages and inputs without invoking runtime, writing `sprint-index.md`, or marking the stage complete. Non-dry-run flow must validate `requirements.md` first, invoke only the generic platform runtime boundary, require the generated `sprint-index.md` to exist, run sprint-index validation, and then atomically update `flow-state.json`. A successful flow marks `requirements` complete and `sprint-index` complete, marks `technical-handbook` ready, leaves later required stages missing, and marks `area-reasoning` skipped only when no reasoning templates are explicitly selected. A failed flow records `sprint-index` failed with a safe error and does not mark later stages ready or complete.
- **Rationale:** Product success requires validated artifacts and durable state, not only runtime success. This preserves Sprint 17 atomic-state guarantees while adding select-stage execution.
- **Study / Source Grounding:** `technical-handbook.md` cites runtime-success-is-not-completion evidence from error and testing reports, context propagation from gh-cli `internal/ghcmd/cmd.go:142`, go-task `task.go:89`, helm `pkg/cmd/install.go:333-347`, and restic `cmd/restic/cleanup.go:24-38`, runtime boundary warnings from opencode `internal/permission/permission.go:44-108` and `internal/llm/tools/bash.go:41-55`, and atomic/durable state requirements from the project TRD and Sprint 17 carry-forward constraints.
- **Trade-Offs Accepted:** Runtime-backed generation has more steps and failure modes, but each failure is visible and recoverable. Generic runtime indirection costs some setup but enables fake-first tests and preserves architecture.
- **Technical Debt / Future Impact:** Flow-stage transition helpers must remain compatible with Sprint 19 through Sprint 21 without modeling future execution/smoke/review/issues behavior.
- **Alternatives Rejected:** Marking `sprint-index` complete on runtime success alone is rejected because it violates the product principle and requirements. Direct OpenCode, agentwrap adapter, shell, Git, or provider calls from `internal/sprint` are rejected because runtime execution belongs behind the generic platform boundary. Supporting `flow --to technical-handbook`, `reasoning`, or `plan` in Sprint 18 is rejected beyond unsupported/not-implemented handling because those belong to later sprints.
- **Contracts Satisfied:** LLM Runtime, Workflows, Persistence And Migrations, Errors, Security, Testing, REQ-AC-03, REQ-AC-04, REQ-AC-05, REQ-AC-06, REQ-AC-18, REQ-AC-19, REQ-AC-20, REQ-AC-21, REQ-AC-22, REQ-AC-31, REQ-AC-34, REQ-AC-35.
- **Evidence Required:** `flow_test.go` covering dry-run, fake-runtime success, runtime success without artifact, runtime success with invalid artifact, runtime failure, validation failure after generation, flow-state write failure, unsupported stages, success transitions, failure transitions, area-reasoning skip semantics, and no unsupported stage persistence.

### Decision 5: Use Thin CLI Wiring And A Sprint Service With Explicit Store, Runtime, Clock, And Output Boundaries

- **Decision:** Implement command wiring in `internal/app/sprint_commands.go` and registration/help in `internal/app/app.go` as thin adapters. They parse arguments, construct request structs, call `internal/sprint.Service`, render returned results or diagnostics, and map exit codes. `internal/sprint/store_fs.go` owns workspace-safe reads and writes under the sprint root. The sprint service owns use cases for validate, prompt, and flow and receives explicit collaborators for store, runtime, clock/time, project catalog access, and output-safe rendering data.

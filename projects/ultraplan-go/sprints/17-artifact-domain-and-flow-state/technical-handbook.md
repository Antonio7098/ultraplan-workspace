## Relevant Patterns

- **Thin CLI adapter over sprint-owned behavior:** The command layer should remain command-shaped: parse arguments, resolve dependencies, call a use-case surface, render output, and map exit codes. Evidence reports repeatedly favor thin wrappers and factory command construction: `01-project-structure` cites thin CLI entrypoints such as chezmoi `main.go:26-34`, fzf `main.go:51-103`, and gh-cli `cmd/gh/main.go:6`; `02-command-architecture` cites gh-cli `pkg/cmdutil/factory.go:16-43`, helm `pkg/cmd/install.go:132-145`, and restic `cmd/restic/main.go:37-114`. For this sprint, the pressure is to keep `ultraplan sprint <project> <sprint> status` as wiring around `internal/sprint`, not a home for artifact, status, or flow-state rules.
- **Module-owned domain and persistence:** Mature projects put behavior near the state it transforms rather than splitting by generic technical labels. `01-project-structure` describes application repos using `internal/` for protected business behavior and one-way import flow, with restic `cmd/restic/` to `internal/global/` to `internal/restic/` to `internal/repository/` as an acyclic chain. `15-philosophy` highlights interface-driven boundaries such as restic `internal/backend/backend.go:19-90`, rclone `fs/types.go:16-59`, and chezmoi `internal/chezmoi/system.go:25-45`. For this sprint, that pattern supports reasoning about `internal/sprint` owning sprint discovery, artifact paths, `flow-state.json`, status derivation, and validation-shaped findings without creating a generic workflow package.
- **Explicit composition and injectable seams:** The evidence favors manual construction over hidden global state. `03-dependency-injection` cites centralized roots such as chezmoi `internal/cmd/config.go:362`, gh-cli `pkg/cmd/factory/default.go:26-46`, opencode `internal/app/app.go:42-81`, and restic `cmd/restic/main.go:181-183`; it also warns that globals-heavy architectures create test and coupling problems. For this sprint, reasoning should decide the narrow seams needed for filesystem, clock/time, and command IO while avoiding interfaces for every internal helper.
- **Strict, inspectable state with cause-preserving errors:** State and errors are most useful when they preserve classification and context. `05-error-handling` shows typed and classified errors via go-task `errors/errors.go:47-50`, restic `internal/errors/fatal.go:10`, and gh-cli exit constants `internal/ghcmd/cmd.go:44-49`; `07-state-context` shows central state roots such as restic `internal/global/global.go:46-89` and explicit session structures such as opencode `internal/session/session.go:12-23`. For this sprint, flow-state reasoning should treat invalid schema, unsupported stages/statuses, unsafe paths, and write failures as inspectable validation or filesystem failures with the state path and cause preserved.
- **Filesystem and IO abstractions where they buy tests:** The selected reports support injectable IO for CLI output and filesystem seams for stateful artifact inspection. `06-io-abstraction` cites gh-cli `pkg/iostreams/iostreams.go:551-568`, go-task `executor.go:553-564`, restic `internal/fs/interface.go:10-31`, and chezmoi `internal/chezmoi/system.go:25`. The pressure is not to over-abstract all helpers, but to make status output, artifact inspection, state loading, and atomic-write error paths testable without real terminal or unsafe filesystem mutation.
- **Deterministic command output and no accidental terminal UX:** `09-terminal-ux` shows rich TUI investments are justified for interaction-heavy tools, while CLI-first tools stay plain and scriptable. The status command's sprint scope is runtime-free and non-interactive, so evidence such as gh-cli prompter/TTY handling `internal/prompter/prompter.go:16-81`, chezmoi non-TTY fallback `internal/cmd/prompt.go:20-256`, and yq's filter-style simplicity suggests deterministic text without ANSI dependencies unless sprint reasoning proves otherwise.
- **Security boundaries as code, not comments:** `13-security` emphasizes explicit path, command, and secret trust boundaries. Relevant examples include gh-cli `safeexec` usage at `pkg/surveyext/editor_manual.go:23`, restic `SecretString` redaction at `internal/options/secret_string.go:15-20`, k9s schema validation at `internal/config/json/validator.go:146`, and path/shell caution examples such as gdu raw path ignore matching `internal/common/ignore.go:16-37`. For this sprint, the security pressure is safe sprint slug resolution, artifact paths that cannot escape the sprint root, no runtime or shell invocation during status, and diagnostics that do not reveal unsafe details.
- **Behavior-focused and golden-compatible tests:** `11-testing-strategy` favors table-driven subtests, integration-style command tests, output regression checks, and centralized mocks where reused. Evidence includes chezmoi `internal/cmd/main_test.go:64-174`, helm golden helper `internal/test/test.go:43`, go-task normalized golden checks `task_test.go:166-169`, and restic functional field mocks `internal/backend/mock/backend.go:14-26`. For this sprint, tests should cover observable status, exit codes, strict state loading, artifact status derivation, ordering, and write failures rather than private implementation details.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Sprint-owned model vs generic workflow/stage abstraction | Keeps planning-stage semantics close to sprint artifacts and avoids study/workflow leakage; aligns with evidence favoring `internal/` protected application behavior and module-owned interfaces such as restic `internal/backend/backend.go:19-90`. | Some repeated mechanics may exist between study and sprint until a stable shared helper is proven. | When deciding whether to introduce shared `planning`, `workflow`, `stages`, or validation packages for only this sprint. |
| Concrete filesystem helpers vs broad filesystem interface | Concrete helpers are smaller and easier to understand; `06-io-abstraction` notes direct use can be acceptable for narrow paths. | Harder to inject write/rename failures and path-edge cases unless seams are placed deliberately. | When designing artifact inspection, strict `flow-state.json` load/write tests, and workspace-safe path validation. |
| Strict flow-state rejection vs lenient compatibility | Rejecting unknown schema versions, stages, statuses, and path escapes keeps state trustworthy and matches evidence favoring structured errors and validation gates, such as go-task `TaskError` `errors/errors.go:47-50` and helm sentinels `pkg/storage/driver/driver.go:27-36`. | Older or manually edited files may fail loudly instead of being partially recovered. | When reasoning about malformed JSON, unsupported `implementation` or `review` stages, unknown statuses, duplicate stages, and project/sprint mismatch. |
| Refreshing state from artifact inspection vs only reading persisted state | Status can recover from missing or stale `flow-state.json` and reflect human-edited files; fits product artifact-as-state pressure. | Refresh behavior can accidentally mask invalid state if not carefully separated from strict load failures. | When deciding the order of load, validate, inspect artifacts, retain failures, and atomically write refreshed state. |
| Plain deterministic text vs richer terminal formatting | Easier command tests, no ANSI in CI, and cleaner stdout/stderr separation; supported by CLI-first evidence and `10-logging-observability` stdout/stderr separation. | Less visually expressive for humans. | When rendering `sprint status` output and command help. |
| Explicit command enumeration vs registration/registry mechanisms | Simple, discoverable, and type-checkable for a small sprint command family; `02-command-architecture` calls static `AddCommand` common for most tools. | Grows linearly as command count grows. | When adding only the `sprint` command family and avoiding plugin-like extensibility for deferred stages. |

## Anti-Patterns And Warnings

- **Putting sprint business rules in command handlers:** `02-command-architecture` warns about long `RunE` functions and logic leakage, citing opencode `cmd/root.go:49-183`, yq `cmd/evaluate_sequence_command.go:152`, and restic `runBackup()` as caution cases. Sprint status rendering can live in app wiring, but stage ordering, state validation, artifact rules, and path safety should not.
- **Creating generic planning/workflow packages before there are multiple proven uses:** `15-philosophy` warns that complexity should be accepted deliberately, while `01-project-structure` favors clear domain ownership. A global `internal/workflow`, `internal/stages`, or `internal/validation` package would risk recreating the fractured technical layers cautioned by the architecture context and would exceed the evidence needed for a sprint-scoped status/state model.
- **Globals for state, config, or registries:** `03-dependency-injection` identifies global config and singleton state as primary testability hazards, citing rclone `fs/config.go:14-51`, rclone `fs/cache/cache.go:16-21`, yq `pkg/yqlib/lib.go:13`, and opencode `internal/config/config.go:123`. Flow-state status should avoid package-level mutable state that would pollute tests or concurrent commands.
- **Direct `os.Stdout`, `os.Stderr`, or terminal formatting in status logic:** `06-io-abstraction` warns about hardcoded output paths such as rclone `cmd/ls/ls.go:42`, urfave-cli `cli.go:47`, and yq `pkg/yqlib/logger.go:18`. Status output needs buffer-based tests and no ANSI surprises.
- **Breaking error chains or flattening validation causes:** `05-error-handling` flags `fmt.Errorf` without `%w` at mitchellh-cli `cli.go:205-206` and inconsistent wrapping in fzf `src/server.go:68`. Strict `flow-state.json` load/write failures need cause chains that internal handling can inspect while rendering safe user diagnostics.
- **Path trust by string convention:** `13-security` warns against no path canonicalization before comparison in gdu `internal/common/ignore.go:16-37` and highlights schema/path validation as a trust boundary. Sprint slugs, artifact paths, and persisted workspace-relative paths need normalization and containment checks before filesystem access or persistence.
- **Using runtime, shell, or study-service behavior for a runtime-free status command:** `12-extensibility` and `13-security` show subprocess/plugin models add trust and timeout risks, such as helm plugin subprocesses `internal/plugin/runtime_subprocess.go:65-79` and no timeout warning. Sprint 17 status should not invoke agentwrap, OpenCode, shell commands, study services, or plugin-like mechanisms.
- **Testing private implementation detail instead of behavior:** `11-testing-strategy` calls out fragile hint-count assertions at k9s `internal/view/pod_test.go:23` and inline string fixtures at mitchellh-cli `cli_test.go:1586-1614`. Sprint tests should assert resolved behavior: output, stage status, paths, exit codes, validation diagnostics, and persistence outcomes.

## Design Pressures

- Keep `internal/sprint` cohesive enough to own planning stages, artifact paths, flow-state schema, discovery, status derivation, and state persistence without depending on study-owned models.
- Preserve the boundary where `internal/app` wires commands and renders status, while `internal/sprint` owns rules and state transitions.
- Keep status runtime-free: no agentwrap, OpenCode, network, prompt generation, study run-loop, source cloning, shell execution, project-index mutation, or planning artifact generation.
- Treat `flow-state.json` as durable product state, not a cache that can be accepted leniently after corruption.
- Derive status from artifacts deterministically while preserving recorded failed states and safe error details.
- Normalize and validate sprint references, slugs, artifact paths, and persisted paths before filesystem access.
- Avoid global workflow abstractions even though planning stages look like a small workflow; the selected evidence favors scoped ownership until reuse is proven.
- Keep output plain, stable, and script-friendly with strict stdout/stderr separation.
- Make tests offline, deterministic, fake-first, and able to inject or simulate filesystem write failures.
- Keep generated/user-editable Markdown artifacts as filesystem state and avoid opaque database-only status.

## Open Questions For Reasoning

- What is the smallest `internal/sprint` public surface that lets `internal/app` render status and map exit codes without exposing internal helpers?
- Should flow-state loading fail before artifact refresh when persisted state is invalid, or may refresh repair only missing/stale valid state? Reasoning must preserve strict rejection requirements for malformed or unsafe persisted content.
- What exact path representation should be stored in each `StageState.Path`: workspace-relative, sprint-root-relative, or another normalized form? The sprint must keep paths reviewable and must reject path escapes.
- How should recorded `failed` stage errors interact with artifact-derived completion? Should a present artifact clear a prior safe error only through explicit refresh, or should failure persist until a later stage action changes it?
- What evidence is sufficient to mark `area-reasoning` as `skipped` when `sprint-index.md` selects no reasoning templates, without implementing broader sprint-index validation that is out of scope?
- Which filesystem seam is necessary to test atomic write failures and path safety without over-abstracting the entire `internal/sprint` package?
- How should command diagnostics mention unsafe paths or invalid JSON while preserving cause chains internally and redacting unsafe detail externally?
- Should command-level status output use golden tests or structured table assertions, and where should help text stability be verified?
- How should ambiguous sprint prefix resolution report candidates deterministically while avoiding leaking absolute host paths unnecessarily?
- How can `internal/sprint` reuse generic atomic-write or workspace path helpers without importing study-owned persistence or run-loop state concepts?

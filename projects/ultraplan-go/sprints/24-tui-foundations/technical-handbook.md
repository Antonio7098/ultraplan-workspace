## Relevant Patterns

- **Thin local interface over shared use cases:** Mature CLIs keep command entrypoints and command handlers thin, with import flow from command/UI layers into product logic and not the reverse. This appears in thin main/command examples such as `cmd/gh/main.go:6`, `pkg/cmd/install.go:132-145`, and `cmd/root.go:9`, and it matters because `ultraplan tui` must call typed application use cases rather than CLI command handlers or rendered text.
- **Composition root plus explicit dependency seams:** High-scoring projects assemble dependencies in a visible root or factory and pass them into commands or apps, as in `internal/ghcmd/cmd.go:52-132`, `pkg/cmdutil/factory.go:16-43`, `pkg/action/action.go:118-146`, and `internal/app/app.go:42-81`. For this sprint, that pattern supports one `internal/app` boundary consumed by CLI and TUI while preserving fake-use-case tests.
- **Injectable IO and terminal abstractions:** Repos with robust tests provide production and test IO constructors or terminal mocks, including `pkg/iostreams/iostreams.go:551-568`, `ui.go:19-43`, `internal/ui/terminal.go:10-36`, and `internal/ui/mock.go:10-53`. This directly applies to deterministic TUI view/model tests and command startup tests that cannot require a real terminal.
- **Message/model separation for TUI state:** Bubble Tea-style and TUI projects separate update state from rendering through models and messages, with examples in `internal/chezmoibubbles/passwordinputmodel.go:8-53`, `internal/tui/tui.go:717-723`, and the fzf renderer split at `src/tui/tcell.go:1-19` plus `src/tui/light.go:110-143`. The pressure for this sprint is deterministic navigation, refresh, preview, error, and narrow-terminal behavior without product-state ownership.
- **Context propagation for refresh and later cancellation:** Mature long-running CLIs create root contexts and pass them through I/O or workflow layers, as shown by `internal/ghcmd/cmd.go:142`, `task.go:89`, and `pkg/cmd/install.go:333-347`. Even if Sprint 24 is read-only, TUI startup and refresh should remain context-aware so Sprint 25 operation/cancellation does not require redesign.
- **User-facing error panes separated from operational details:** Error studies show explicit user/operational separation through `cmd/age/tui.go:37-54`, `internal/model/flash.go:100-103`, `cmd/restic/main.go:199-209`, and `internal/ghcmd/cmd.go:281-301`. The TUI needs actionable panes, but displayed diagnostics must preserve classification and redaction rather than dumping raw runtime/config internals.
- **Bounded preview and streaming discipline:** Performance evidence favors chunked and streaming processing for large input, including `age/internal/stream/stream.go:20`, `age/internal/stream/stream.go:195-219`, and `yq/pkg/yqlib/stream_evaluator.go:78-113`. Artifact previews should therefore cap reads and avoid recursive repository scans or full artifact loading when only a preview is needed.
- **Structured and behavior-focused testing:** Strong CLI projects use testscript, golden fixtures, fakes, and behavior assertions, including `internal/cmd/main_test.go:64-174`, `task_test.go:166-169`, `internal/backend/mock/backend.go:14-26`, and `pkg/httpmock/stub.go:35-199`. This sprint should use fakes for app use cases and assertions against model/view behavior rather than requiring an interactive terminal.
- **Explicit read-only trust boundary:** Security evidence shows that tools score higher when dangerous operations are visibly gated or disabled, as with `internal/permission/permission.go:44-108`, `internal/config/plugin.go:64`, and `pkg/yqlib/security_prefs.go:3-7`. For Sprint 24 the read-only boundary should be visible in command labels, model actions, tests, and documentation.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Bubble Tea-style model/update vs tcell/tview widget model | Bubble Tea evidence shows more testable message/model separation for prompts and overlays (`internal/chezmoibubbles/passwordinputmodel.go:8-53`, `internal/tui/tui.go:717-723`). | tcell/tview offers richer widgets for multi-pane dashboards, but tighter library coupling and harder unit testing are noted in the terminal UX report. | Choose during reasoning for `internal/tui` dependency containment, deterministic model tests, and narrow-terminal rendering. |
| Shared typed app use cases vs direct product service calls from TUI | A shared boundary keeps CLI and TUI reusable and avoids TUI-owned workflow logic; evidence includes factory/use-case patterns at `pkg/cmdutil/factory.go:16-43` and app composition at `internal/app/app.go:42-81`. | Adds a layer and result types that must be maintained. | Matters when deciding what belongs in `internal/app/usecases.go` versus product modules and `internal/tui`. |
| Explicit constructor injection vs context-based service propagation | Constructor/factory injection is traceable and testable (`pkg/cmdutil/factory.go:16-43`, `executor.go:93-114`). | Deep TUI trees can become verbose; context propagation like `internal/view/xray.go:412` reduces passing but loses compile-time safety and can panic on missing keys (`internal/dao/ds.go:72`). | Matters for TUI program setup, child components, and fake use-case plumbing. |
| Immutable startup config vs hot-reload/session overrides | Immutable merged config is simpler and aligns with one-shot CLI precedence patterns (`internal/cmd/config.go:2253-2287`, `opencode/internal/config/config.go:609-641`). | Long-running TUIs may benefit from session-specific updates; k9s uses manual override fields at `internal/config/k9s.go:317-337`. | Matters if refresh should reload workspace/config or only product status artifacts. |
| Golden/rendering tests vs flexible substring assertions | Golden or snapshot tests catch rendering regressions (`internal/test/test.go:43`, `task_test.go:166-169`). | Golden diffs can be brittle for layouts and ANSI styling; overly broad regexes can also be fragile. | Matters for `views_test.go`, especially narrow-terminal output and error panes. |
| Bounded preview reads vs complete artifact display | Bounded reads protect memory and responsiveness, consistent with streaming evidence (`age/internal/stream/stream.go:20`, `yq/pkg/yqlib/stream_evaluator.go:78-113`). | Users may not see complete artifacts in the TUI and need clear truncation indicators. | Matters for Markdown/JSON previews of `execute.md`, `.run-state.json`, reports, and indexes. |
| Read-only dashboard only vs adding guarded actions early | Strict read-only scope reduces security and mutation risk, aligning with explicit trust-boundary evidence (`internal/permission/permission.go:44-108`, `pkg/yqlib/security_prefs.go:3-7`). | Users may expect validation/run controls in a TUI; deferring actions can feel limited. | Matters for key bindings, action labels, command docs, and Sprint 25 handoff. |

## Anti-Patterns And Warnings

- **Do not let the TUI call CLI handlers or scrape CLI output:** The command architecture report warns against bifurcated CLI/TUI behavior hiding commands inside interactive interpreters (`internal/view/command.go:176`) and emphasizes command-to-library delegation (`pkg/cmd/install.go:132-145`). The TUI should consume typed results, not stdout text.
- **Do not let `internal/tui` own product state machines or persistence:** Project-structure evidence favors unidirectional boundaries and protected internals (`internal/chezmoi/chezmoi.go:1-2`, `cmd/root.go:9`), while philosophical evidence warns against accidental complexity. Product modules should remain state owners.
- **Avoid global mutable app/config state in TUI tests:** DI evidence identifies globals and singleton config as testability risks (`pkg/yqlib/lib.go:13`, `fs/config.go:793`, `fs/cache/cache.go:16-21`). TUI tests should use fresh fake use cases and not shared mutable registries.
- **Do not use direct `os.Stdout`, `os.Stderr`, or real terminal dependencies in normal tests:** IO evidence flags hardcoded streams as untestable (`cmd/age/tui.go:31`, `cmd/ls/ls.go:42`, `cli.go:47`). The TUI should isolate terminal effects behind package-level setup and test rendering as strings/models.
- **Avoid `context.Background()` inside refresh/status operations:** State/context evidence warns that isolated backgrounds break cancellation (`internal/cmd/templatefuncs.go:215`, `task.go:341`, `dive/image/analysis.go:20`). Startup and refresh use cases should accept caller context.
- **Do not expose raw secrets, models, provider env, or unredacted diagnostics:** Security and logging evidence supports redaction types and header scrubbing (`internal/options/secret_string.go:15-20`, `pkg/registry/transport.go:37-41`, `status.go:332-338`). TUI error and execute panes should display redacted summaries only.
- **Avoid unbounded artifact reads, recursive source scans, or eager runtime setup on TUI startup:** Performance evidence warns against eager initialization and unbounded accumulation (`pkg/cmdutil/factory.go:27-42`, `cmd/restic/main.go:128-133`, `gdu/pkg/analyze/parallel.go:13`). The dashboard should discover known artifacts through product use cases rather than scanning repositories broadly.
- **Do not silently downgrade invalid config or preview path errors:** Config and security reports flag silent misconfiguration and weak validation as risks (`cmd/gdu/main.go:242-244`, `internal/config/json/validator.go:146`). Missing files, invalid workspace, and path escapes should become actionable error states.
- **Avoid implementing plugins, subprocess extensions, or external command launchers in the read-only dashboard:** Extensibility evidence shows plugin/subprocess models carry timeouts, trust, and collision risks (`helm/internal/plugin/runtime_subprocess.go:65-79`, `rclone/fs/rc/registry.go:41-48`). Sprint 24 does not need that surface.

## Design Pressures

- The sprint needs an interactive dashboard, but the evidence favors keeping terminal UI as a local interface over typed use cases rather than a new product owner.
- The TUI must start without runtime credentials, provider availability, network access, or subprocess self-invocation, so dependency construction has to separate read-only status dependencies from runtime-backed execution dependencies.
- Existing CLI text and JSON output remain automation surfaces; TUI work must not alter command rendering or error/exit mapping except to add `ultraplan tui` startup wiring.
- Artifact previews have high UX value but create security and performance pressure: paths must stay workspace-contained, secrets must be redacted, and reads must be bounded.
- The dashboard must display execute status and diagnostics safely, but selected evidence warns against direct JSON parsing in UI layers when product status logic should own durable state interpretation.
- Terminal rendering must work in narrow terminals and not rely on ANSI-only meaning; terminal UX evidence supports fallback and plain-text visibility as core behavior.
- Tests need to prove read-only guarantees, which requires fake app use cases and behavior assertions against actions, not only import-boundary review.
- The selected evidence supports explicit non-goals and deliberate complexity control; Sprint 24 should resist turning the dashboard into plugin, workflow, scheduler, or runtime operation infrastructure.

## Open Questions For Reasoning

- Which concrete terminal library best fits the sprint's deterministic model/update/rendering test needs while staying contained inside `internal/tui`?
- What exact result types belong in `internal/app/usecases.go`, and which product-specific status derivations must stay in `internal/project`, `internal/study`, or `internal/sprint`?
- Should TUI refresh reload workspace/config, or should it only refresh product state from durable artifacts using the startup config snapshot?
- How should artifact previews communicate truncation, missing files, invalid JSON/Markdown, redaction, and path rejection without implying mutation or repair?
- Which execute diagnostics and model/runtime fields need redaction before the TUI can display them, and should redaction happen in product use cases, app use cases, or TUI rendering?
- What is the narrow-terminal minimum that tests should enforce for critical status and error visibility?
- Should the first read-only dashboard include background periodic refresh, or only explicit user-triggered refresh to keep lifecycle and test behavior simpler?
- How should read-only guarantees be tested: import scans, fake mutation counters, absence of runtime/use-case methods, fixture file snapshots, or a combination?

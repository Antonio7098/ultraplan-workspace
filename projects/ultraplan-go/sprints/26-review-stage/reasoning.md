## Sprint Technical Handbook Summary

- **Relevant Patterns:** Thin interface adapters over shared typed use cases; visible manual composition with lazy volatile dependencies; central config merge and post-merge validation; semantic error classes; injected and separated output channels; root-context cancellation; bounded structured fan-out; isolated concurrent collection followed by deterministic aggregation; TTY-aware progress; behavior-level fake-runtime and command tests; mechanically enforced path, redaction, and permission boundaries.

## Decisions

The eight final decisions below are authoritative for Sprint 26. They fix review ownership, manifest and fingerprint construction, reviewer orchestration, deterministic verdict rules, persistence and freshness behavior, shared CLI/TUI surfaces, security boundaries, embedded assets, and verification evidence so `plan.md` can proceed without reopening architecture.

## Final Decisions

### Decision 1: Keep Review Product-Owned And Share Typed Use Cases

- **Decision:** Implement review as focused behavior in `internal/sprint`: scope resolution, manifest creation, prompts, orchestration, structured validation, deterministic checks, verdict synthesis, Markdown validation, and persistence. Extend existing domain/service/store files plus `review.go` and `review_validation.go`. `internal/app` will construct dependencies and expose typed readiness/status, dry-run, prompt-preview, execute, progress, cancellation, and result operations used by both CLI and TUI. Platform runtime remains generic; CLI and TUI only adapt requests and render results.
- **Rationale:** Review transforms sprint-owned artifacts and state. Keeping policy in one module prevents duplicated CLI/TUI behavior and avoids leaking contract/verdict semantics into generic runtime infrastructure.
- **Study / Source Grounding:** `technical-handbook.md:31-32,59-60,102`; project-structure evidence (`cmd/restic/main.go:37-114`), command factory evidence (`pkg/cmdutil/factory.go:16-43`), and manual composition evidence (`internal/cmd/config.go:193-291`). `reasoning/architecture.md:9-14,40-53` adopts the same boundary.
- **Trade-Offs Accepted:** `internal/sprint` grows and each interface needs mapping code, but focused files and one typed semantic result are simpler than a new subsystem.
- **Technical Debt / Future Impact:** Package size is monitored through Sprint 27/28. No generic workflow/reviewer framework, plugin registry, or context service locator is introduced.
- **Alternatives Rejected:** A global `internal/review` or workflow engine is rejected because no other module owns these semantics; putting orchestration in `internal/app` is rejected because app is composition, not product policy; TUI-to-CLI calls/stdout parsing are rejected because they violate parity and testability; extending platform runtime with sprint concepts is rejected because it reverses dependencies.
- **Contracts Satisfied:** Architecture, CLI Surface, Testing; AC-02, AC-05, AC-17, AC-18; RP-01, RP-02.
- **Evidence Required:** Architecture Review confirms `sprint -> project/workspace/platform/runtime`, no `platform/* -> product`, no `sprint -> study`, no TUI-to-CLI dependency, and no direct OpenCode import; shared-use-case tests feed identical scope/result fixtures to CLI and TUI.

### Decision 2: Resolve, Validate, Freeze, And Fingerprint One Review Manifest

- **Decision:** Before runtime work, resolve every selected contract and review protocol dynamically through `project-index.md`; reject missing, duplicate, unknown, unreadable, or canonical paths escaping allowed roots. Require valid prerequisites through execute. Build an immutable manifest containing workspace-relative governed inputs, selected scope, selected protocols, target implementation identity, explicit changed-path scope, `plan.md`, `execute.md`, `.run-state.json` when present, effective review config/model source, permission policy, reviewer count, permitted writes, and a deterministic content/identity fingerprint. Workers consume this frozen manifest. Recheck governed and target identity before persistence; drift fails the attempt as stale and preserves prior evidence.
- **Rationale:** A review is meaningful only for one stable evidence identity. Dynamic catalog resolution honors project governance, while freezing prevents reviewers in one invocation from silently assessing different inputs.
- **Study / Source Grounding:** `technical-handbook.md:33,40,96-100`; configuration evidence (`chezmoi/internal/cmd/config.go:2253-2287`, `go-task/internal/flags/flags.go:314-327`) and security evidence (`internal/backend/shell_split.go:45-76`, `internal/permission/permission.go:44-108`). TRD `18.9.1` and architecture reasoning `31-38` require this consistency boundary.
- **Trade-Offs Accepted:** Canonicalization and hashing add preflight I/O, and strict identity requirements may block review where target identity is inadequate. Correct freshness is preferred over a false current review.
- **Technical Debt / Future Impact:** Git-independent identity may initially use execute target identity plus explicit changed paths and contained content fingerprints. If that cannot be trustworthy, review blocks rather than weakens the fingerprint.
- **Alternatives Rejected:** Hardcoded contract/protocol paths are rejected because project index is authoritative; reading live files independently in each worker is rejected because inputs can drift; using timestamps alone is rejected because they are not content identity; silently omitting absent `.run-state.json` when execute claims it should exist is rejected because missing evidence must remain visible.
- **Contracts Satisfied:** Architecture, Configuration, Persistence And Migrations, Security, Workflows; AC-01, AC-02, AC-03, AC-04, AC-13, AC-16.
- **Evidence Required:** Fixture tests for missing/duplicate/unknown/unreadable/escaping and symlinked paths; deterministic fingerprint tests including each governed input and changed scope; mutation-during-run test proving stale failure and preservation; dry-run/prompt/TUI confirmation assertions for model source, reviewer count, concurrency, scope, selected contracts/protocols, and permitted writes.

### Decision 3: Run Independent Structured Reviewers With Bounded Collect-All Fan-Out

- **Decision:** UltraPlan will create exactly one independent structured agentwrap request per selected contract and one technical-handbook request. Use a configurable positive per-invocation concurrency bound, immutable per-reviewer input, context propagation, bounded agentwrap retries, required capabilities, read-only policy, and one owner that drains events and awaits every started run. Ordinary reviewer failures, malformed output, or missing output become terminal failed coverage units while healthy siblings continue. User cancellation, deadline expiry, or invocation-wide infrastructure failure stops new scheduling, cancels active runs, and performs bounded awaited cleanup.
- **Rationale:** Independent coverage units can run concurrently, but complete diagnostics require collecting unaffected results. A failed or missing reviewer must remain visible and make passing impossible without hiding other conformance evidence.
- **Study / Source Grounding:** `technical-handbook.md:20-21,36-37,48-50,61-62`; bounded pool examples (`restic/internal/repository/repository.go:567`, `k9s/internal/pool.go:21,30,37`) and timed shutdown (`lazygit/pkg/tasks/tasks.go:338-357`). Architecture reasoning `24-27,64-70` resolves collect-all versus fail-fast.
- **Trade-Offs Accepted:** Collect-all consumes more runtime after one unit fails, and concurrency adds lifecycle/race complexity. Bounds, cancellation, isolated results, and sequential aggregation contain those costs.
- **Technical Debt / Future Impact:** No cross-process/global throttle or partial result cache is added. Both require measured need and separate versioned designs.
- **Alternatives Rejected:** Fail-fast cancellation for an ordinary reviewer failure is rejected because it hides unaffected coverage; sequential execution is rejected because independent units and AC-05 call for bounded fan-out; unbounded goroutines are rejected because resource use and cleanup become unauditable; runtime-spawned subagents are rejected because UltraPlan owns fan-out; bespoke OpenCode process/retry handling is rejected because agentwrap owns it.
- **Contracts Satisfied:** Errors, LLM Evaluation / Cost / Safety, LLM Runtime, Observability, Performance, Security, Testing, Workflows; AC-05, AC-06, AC-07, AC-20.
- **Evidence Required:** Fake-runtime assertions for request count and unique coverage keys; maximum-active count never exceeds the configured bound; sibling completion after one unit fails; missing/malformed output; retries/timeouts; cancellation stops scheduling, calls cancellation, drains events, waits for started work, and yields no goroutine leaks under `go test -race ./...`.

### Decision 4: Validate Evidence And Compute The Verdict Deterministically

- **Decision:** Accept only the versioned structured reviewer schema. Validate coverage identity, applicability classification (`direct`, `partial`, `not_triggered`, or `deferred`), finding severity, contained workspace/target-relative citations, and valid line references before findings enter the aggregate. Independently run product checks for final-decision conformance, plan-task execution evidence, approved verification results, complete reviewer coverage, deviations, missing evidence, and citation validity. Normalize/sort coverage and findings by stable product keys. Product code computes only `pass`, `pass_with_findings`, `fail`, or `blocked`: applicable blocker/high findings force `fail`; medium/low/info may produce `pass_with_findings`; missing/failed/malformed coverage, required runtime/environment absence, cancellation, or insufficient required evidence cannot produce pass.
- **Rationale:** Model prose can identify potential findings but cannot establish completeness, validate its own citations, or select the product verdict. Deterministic checks make identical validated evidence produce identical output.
- **Study / Source Grounding:** `technical-handbook.md:34,37,93-100`; error aggregation (`helm/pkg/action/uninstall.go:232-254`), deterministic-concurrency warnings (`gdu/pkg/analyze/file.go:159,185-200`), and behavior test evidence (`chezmoi/internal/cmd/main_test.go:64-174`). TRD `18.6` and `18.9.1` define required sections, coverage, citations, and verdict rules.
- **Trade-Offs Accepted:** Structured schemas and strict validation reject some usable free-form model output and add product code. This is necessary to prevent false passes and nondeterministic canonical artifacts.
- **Technical Debt / Future Impact:** Schema evolution must be explicit and versioned. New finding kinds or severities must update validators and deterministic rules before use.
- **Alternatives Rejected:** Letting each model author a review section or verdict is rejected because ordering, safety, completeness, and severity become nondeterministic; treating runtime exit zero as success is rejected because output/evidence can be absent; downgrading failed coverage to a warning is rejected because 100% coverage is required; trusting citation text without path/line validation is rejected because model output is untrusted.
- **Contracts Satisfied:** Errors, LLM Evaluation / Cost / Safety, LLM Runtime, Security, Testing, Workflows; AC-07, AC-08, AC-09, AC-10, AC-20.
- **Evidence Required:** Pure table tests for all verdict combinations and stable ordering; fake outputs for success, medium findings, blocker/high failure, blocked preflight, missing result, malformed schema, citation escape, invalid line, deviations, missing plan evidence, and failed verification; repeated shuffled inputs produce byte-stable normalized sections apart from normalized timestamps.

### Decision 5: Persist Only Complete Current Evidence And Reconcile Freshness

- **Decision:** Render the complete review from validated aggregate data into staging, validate all required sections/no placeholders/coverage/citations/verdict, flush and close, then atomically replace sprint-root `review.md`. Only after successful artifact replacement atomically update the versioned review portion of `flow-state.json` with execution status, verdict, artifact path, timestamp, input fingerprint, staleness, and safe diagnostics. Preserve execution status separately from verdict. Preserve the last complete `review.md` on any failed/incomplete attempt. Status reconciles file validity and matching fingerprints and treats mismatch as stale/non-current; existence alone is never proof. Rerun may replace only `review.md` and review flow-state fields.
- **Rationale:** Runtime success is not product completion, and a partially written review is worse than a known prior complete review. Per-file atomicity plus fingerprint reconciliation is the smallest reliable local-filesystem design.
- **Study / Source Grounding:** `technical-handbook.md:51,96,101`; narrow I/O seam evidence (`internal/backend/mem/mem_backend.go:52-255`) and project architecture requirements. Architecture reasoning `31-38,84-90` explicitly rejects pretending to have a cross-file transaction.
- **Trade-Offs Accepted:** Process interruption can occur between two atomic replacements, leaving a complete artifact and stale state. Reconciliation exposes rather than hides this condition.
- **Technical Debt / Future Impact:** Flow-state schema gains review fields and migration/compatibility obligations. Partial reviewer outputs are not durable and a recovery reruns all selected units.
- **Alternatives Rejected:** Directly streaming reviewer prose into `review.md` is rejected because failure corrupts canonical evidence; deleting the previous artifact before write is rejected because it loses last-known-good evidence; a database or transaction journal is rejected as disproportionate; inferring completion from `review.md` existence is rejected because legacy/manual/stale files may exist; adding reviewer cache files is rejected because it creates another artifact hierarchy and invalidation problem.
- **Contracts Satisfied:** Documentation, Observability, Persistence And Migrations, Security, Workflows; AC-03, AC-04, AC-09, AC-11, AC-12, AC-13, AC-14, AC-15.
- **Evidence Required:** Failure injection at stage/write/flush/close/rename/flow-state update; previous review remains byte-identical on failed replacement; strict malformed/unsupported state load tests; legacy/manual review is not current; artifact/state mismatch reports stale and next action; successful rerun atomically replaces only allowed files.

### Decision 6: Expose One Truthful CLI, JSON, Status, Flow, And TUI Surface

- **Decision:** Extend `status`, `validate review`, `prompt review`, `flow --to review`, and `review` with text and stable JSON results from the same app DTOs. DTOs carry readiness, scope, selected model source, reviewer count/concurrency, execution status, progress lifecycle, verdict, staleness, findings, diagnostics, artifact path, and next action. CLI stdout in JSON mode contains only the final machine envelope; progress/diagnostics use injected non-data channels. Prompt preview and dry-run do not initialize runtime, invoke reviewers, update state, or write `review.md`. TUI provides confirmation, live progress, cancellation, finding navigation, verdict, and artifact preview without recomputing product rules.
- **Rationale:** CLI automation and TUI supervision need different presentation but identical product facts. Separating progress from final data preserves scriptability and testability.
- **Study / Source Grounding:** `technical-handbook.md:19,22-24,35,38-39,53-55`; testable streams (`pkg/iostreams/iostreams.go:478-523,551-568`), non-interactive prompt paths (`internal/cmd/prompt.go:124-137`), and structured logging (`helm/internal/logging/logging.go:31-71`). Lazy factories (`gh-cli/pkg/cmdutil/factory.go:27-42`) support runtime-free inspection paths.
- **Trade-Offs Accepted:** Multiple renderers and TUI update states add code and parity testing. The shared typed operation prevents semantic duplication.
- **Technical Debt / Future Impact:** JSON schema compatibility must be documented before public release. Progress ordering is lifecycle-oriented and may vary; final result ordering is deterministic.
- **Alternatives Rejected:** TUI shelling out or calling CLI handlers is rejected; parsing native provider payloads in TUI is rejected; writing progress to JSON stdout is rejected; requiring credentials/runtime for status, validation, preview, or dry-run is rejected; streaming unvalidated findings as final findings is rejected.
- **Contracts Satisfied:** Architecture, CLI Surface, Configuration, Documentation, Errors, Observability, Performance, Testing, Workflows; AC-14, AC-15, AC-16, AC-17, AC-18.
- **Evidence Required:** Command tests for every required command and exit class; structural JSON tests with no ANSI/progress contamination; dry-run/prompt tests proving zero runtime calls and zero writes; shared DTO parity tests; deterministic TUI model/update tests for readiness, confirmation, progress, cancellation, findings, result, and preview; narrow-terminal rendering checks where existing TUI patterns require them.

### Decision 7: Enforce Read-Only Review And Safe Diagnostics Mechanically

- **Decision:** Build reviewer requests with explicit read-only agentwrap permission policy, required capabilities, contained working directories, context deadlines, and only the read/search/list operations needed. Canonicalize all workspace, target, selected artifact, citation, and output paths. Deny source/test/governed-input/workspace-config/Git mutation and reject unsupported required enforcement before launch. Permit product writes only to the staged/canonical `review.md` and review fields in flow state. Persist/display safe classified diagnostics and redacted metadata only; omit raw native payloads, unrestricted stdout/stderr, secrets, and unnecessary absolute paths.
- **Rationale:** Prompts and conventions cannot enforce a trust boundary. Review reads untrusted repositories and model output, so permissions, paths, output policy, and redaction must be executable checks.
- **Study / Source Grounding:** `technical-handbook.md:25,40,55,67,97`; permission enforcement (`internal/permission/permission.go:44-108`), redacting values (`internal/options/secret_string.go:15-20`), and explicit argv/path handling (`internal/backend/shell_split.go:45-76`). Requirements `47,53-54,79-89` and TRD `22` make these controls mandatory.
- **Trade-Offs Accepted:** Strict capability checks can block a runtime that cannot prove read-only behavior. This is safer and more truthful than silently treating policy as best effort.
- **Technical Debt / Future Impact:** Agentwrap capability evolution may require policy translation updates, but sprint semantics remain unchanged behind the generic runtime boundary.
- **Alternatives Rejected:** Prompt-only "do not edit" instructions are rejected; banned shell-command lists are rejected because they do not provide filesystem isolation; direct OpenCode invocation is rejected; storing raw payloads for convenience is rejected due to secret/source leakage; absolute generated paths are rejected except safe local diagnostics where unavoidable.
- **Contracts Satisfied:** Configuration, Errors, LLM Evaluation / Cost / Safety, LLM Runtime, Observability, Security; AC-01, AC-02, AC-06, AC-07, AC-11, AC-20.
- **Evidence Required:** Request-policy inspection; unsupported-capability blocked preflight; path escape, symlink/canonical containment, citation line, and output-path tests; filesystem snapshots proving no source/test/governed-input/config/Git mutation; secret and absolute-path redaction fixtures; raw payload omission assertions.

### Decision 8: Ship Embedded Assets And Verify Behavior At Product Boundaries

- **Decision:** Add embedded defaults for the structured review prompt and `review.md` template through the existing workspace asset mechanism. Readable workspace files remain optional intentional overrides and materialization occurs only through `ultraplan defaults install`. Test deterministic pure rules, fake runtime orchestration, failure-injected persistence, app/command wiring, output modes, and TUI updates. Real OpenCode review remains gated and optional. The implementation gate is `go test ./...`, `go test -race ./...`, and `go build ./cmd/ultraplan` from `../ultraplan-go`, followed by the selected Architecture Review and Sprint Review protocols.
- **Rationale:** Embedded defaults make a normal workspace complete without copied assets, while boundary-level tests catch wiring, concurrency, persistence, and presentation failures that unit-only testing misses.
- **Study / Source Grounding:** `technical-handbook.md:39,68,86,101`; command behavior tests (`chezmoi/internal/cmd/main_test.go:64-174`), fake runner (`lazygit/pkg/commands/oscommands/fake_cmd_obj_runner.go:17-26`), synchronized I/O (`executor_test.go:146-151`), and explicit golden fixtures (`go-task/task_test.go:166-169`). No separate repository study was needed for embedded asset precedence because that policy is project-specific and fixed by PRD `2.2`, TRD `18.7`, and Architecture `workspace` ownership.
- **Trade-Offs Accepted:** Goldens/fixtures require deliberate updates and fake runtimes cannot prove provider behavior. Gated real-runtime evidence remains optional so normal tests are deterministic and portable.
- **Technical Debt / Future Impact:** Prompt/template schema changes must update embedded assets, validators, and fixtures together. Do not create mandatory workspace copies or duplicate runtime fixtures outside existing test conventions.
- **Alternatives Rejected:** Requiring `prompts/` or `templates/` files is rejected because embedded defaults are the product baseline; exporting defaults during workspace initialization is rejected because materialization is explicit; normal tests requiring OpenCode/credentials are rejected; library-only tests are rejected because they miss CLI/TUI/output/persistence wiring; brittle tests tied to private helper counts are rejected.
- **Contracts Satisfied:** Documentation, LLM Runtime, Persistence And Migrations, Security, Testing; AC-09, AC-19, AC-20, AC-21; RP-01, RP-02.
- **Evidence Required:** Embedded fallback, valid override, unreadable/invalid override, and explicit materialization tests; prompt and review Markdown fixtures/goldens; the complete fake-runtime scenario matrix; command/app/TUI tests; successful required Go commands; completed Architecture Review and Sprint Review evidence.

## Implementation Constraints

- Preserve dependency direction: `cmd/ultraplan -> internal/app`, `internal/tui -> internal/app`, `internal/app -> internal/sprint` and platform composition, `internal/sprint -> project/workspace/platform/runtime`, and `platform/* -> no product modules`.
- Do not import `study` from `sprint`, direct OpenCode packages from product code, CLI handlers from TUI, or terminal/TUI types into app/sprint DTOs.
- Keep review semantics, structured result schema, deterministic checks, verdict rules, Markdown validation, and persistence policy in `internal/sprint`.
- Resolve selected contracts and protocols through the project catalog; do not hardcode active contract names or paths.
- Freeze one immutable manifest and fingerprint before scheduling; do not let workers reread mutable governance inputs as their authority.
- Issue one independent request per selected contract plus one handbook request. Selected protocols constrain checks but are not extra reviewer units.
- Use bounded per-invocation concurrency, context propagation, bounded retries owned by agentwrap, event draining, and awaited terminal results for every started run.
- Continue healthy siblings after an individual reviewer failure; stop scheduling for user cancellation, deadline, or invocation-wide invalidation; no failed/missing unit may yield pass.
- Validate and normalize all model output before aggregation. Only product code computes the verdict and writes canonical Markdown.
- Keep execution status separate from conformance verdict in domain, flow state, DTOs, JSON, CLI, and TUI.
- Permit review writes only to staged/canonical `review.md` and review fields in `flow-state.json`; never mutate product source/tests, governed planning inputs, workspace config, unrelated sprint artifacts, or Git state.
- Stage, flush, close, validate, and atomically replace each file. Preserve the last complete review on any failed/incomplete replacement and reconcile cross-file fingerprints.
- Use workspace/target-relative contained paths in generated artifacts and citations. Redact secrets and omit unsafe raw runtime payloads and unrestricted output.
- Keep runtime creation lazy for help, status, validation, prompt preview, and dry-run; those paths must not invoke runtime or write review state.
- JSON stdout must contain only machine output. Progress and diagnostics use injected separate channels; TUI consumes typed progress, not terminal text or native provider payloads.
- Use embedded review prompt/template defaults with optional explicit workspace overrides and `defaults install` materialization; do not make workspace copies prerequisites.
- Normal tests must use deterministic fixtures, fake runtimes, fake clocks/stores where needed, and non-interactive TUI updates. Real OpenCode tests remain gated.
- Do not implement smoke, `verify`, issue tracking, automatic fixes, remote execution, Git operations, a reviewer cache, a new runtime API, or a generic workflow/plugin framework in Sprint 26.

## Relevant Patterns

- **Thin interface adapters over shared use cases:** The project-structure and command reports show entrypoints and commands that construct options, resolve dependencies, delegate, and render results rather than own business rules (`studies/go-cli-study/reports/final/01-project-structure.md`, `cmd/restic/main.go:37-114`; `studies/go-cli-study/reports/final/02-command-architecture.md`, `pkg/cmd/install.go:132-145`). This is relevant to a review stage exposed through both CLI and TUI because it puts pressure on the shared operation and typed result to remain independent of either terminal surface.
- **Visible composition with lazy volatile dependencies:** Manual composition roots and lazy factories make dependency ownership inspectable while avoiding unnecessary startup work (`studies/go-cli-study/reports/final/03-dependency-injection.md`, `internal/cmd/config.go:193-291`, `pkg/cmd/factory/default.go:26-46`; `studies/go-cli-study/reports/final/14-performance.md`, `helm/pkg/action/lazyclient.go:35-53`). Review reasoning should inspect where runtime, stores, clocks, streams, and configuration are constructed and whether simple status/help paths avoid expensive setup.
- **Central configuration merge followed by validation:** Explicit changed-flag tracking and generic precedence accessors keep defaults from overriding file or environment values (`studies/go-cli-study/reports/final/04-configuration-management.md`, `chezmoi/internal/cmd/config.go:2253-2287`, `go-task/internal/flags/flags.go:314-327`). This matters when review has stage-specific model, timeout, parallelism, permission, output, and dry-run settings whose effective source must be visible.
- **Semantic errors with boundary-specific presentation:** Wrapped causes, typed or sentinel classifications, multi-error aggregation, and application-boundary exit mapping preserve both machine decisions and user guidance (`studies/go-cli-study/reports/final/05-error-handling.md`, `go-task/errors/errors_task.go:13-32`, `helm/pkg/action/uninstall.go:232-254`, `restic/cmd/restic/main.go:199-209`). Review reasoning should distinguish preflight, blocked prerequisites, reviewer failure, timeout, cancellation, validation failure, persistence failure, and a completed review with a failing verdict.
- **Inject I/O at command and external boundaries:** Production/test stream constructors and separate normal/error writers make text, JSON, progress, and diagnostics independently assertable (`studies/go-cli-study/reports/final/06-io-abstraction.md`, `pkg/iostreams/iostreams.go:551-568`, `internal/ui/mock.go:10-53`). Concurrent capture also needs synchronization (`executor_test.go:146-151`). This pattern supports CLI/TUI parity without treating rendered terminal output as an integration contract.
- **Root cancellation plus structured, bounded fan-out:** Signal-derived contexts and `errgroup.WithContext` demonstrate cancellation propagation, while semaphores and worker pools impose resource limits (`studies/go-cli-study/reports/final/07-state-context.md`, `helm/pkg/cmd/install.go:333-347`; `studies/go-cli-study/reports/final/08-concurrency.md`, `restic/internal/repository/repository.go:567`, `k9s/internal/pool.go:21,30,37`). Sprint reasoning must determine whether one reviewer failure cancels peers or whether independent results should be collected, but every started unit still needs an owned and awaited lifecycle.
- **Deterministic aggregation after concurrent work:** The concurrency and I/O reports expose risks around shared result mutation, interleaved output, blocked channels, and non-thread-safe buffers (`studies/go-cli-study/reports/final/08-concurrency.md`, `gdu/pkg/analyze/file.go:159,185-200`, `client.go:981`; `studies/go-cli-study/reports/final/06-io-abstraction.md`, `executor_test.go:146-151`). For a review artifact and verdict, evidence suggests separating concurrent collection from stable ordering and final rendering.
- **TTY-aware progress with a first-class non-interactive path:** Mature tools detect terminal capabilities, suppress or redirect progress for pipes, and avoid prompting in automation (`studies/go-cli-study/reports/final/09-terminal-ux.md`, `internal/cmd/prompt.go:124-137`, `iostreams/iostreams.go:514-516`; `studies/go-cli-study/reports/final/10-logging-observability.md`, `helm/internal/logging/logging.go:31-71`). The review stage must serve human terminals, JSON consumers, and TUI event rendering without allowing one mode to contaminate another.
- **Behavior-level tests around volatile boundaries:** Command tests, fake runners, reusable mocks, real fixtures where needed, and explicit goldens test observable contracts without requiring live services (`studies/go-cli-study/reports/final/11-testing-strategy.md`, `chezmoi/internal/cmd/main_test.go:64-174`, `lazygit/pkg/commands/oscommands/fake_cmd_obj_runner.go:17-26`, `helm/internal/test/test.go:43`; `studies/go-cli-study/reports/final/06-io-abstraction.md`, `internal/backend/mem/mem_backend.go:52-255`). This directly supports deterministic reviewer, persistence-failure, cancellation, output, and TUI model tests.
- **Enforce trust boundaries mechanically:** Explicit argv, canonicalized paths, redacting types, and permission enforcement make safety properties testable (`studies/go-cli-study/reports/final/13-security.md`, `internal/backend/shell_split.go:45-76`, `internal/options/secret_string.go:15-20`, `internal/permission/permission.go:44-108`). For this sprint, the pattern applies to selected-path containment, read-only runtime capabilities, diagnostics, and the prohibition on source or Git mutation.

## Trade-Offs

| Trade-Off | Benefit | Cost | When It Matters |
| --- | --- | --- | --- |
| Central composition versus distributed construction | Dependencies and lifecycle are visible and replaceable (`studies/go-cli-study/reports/final/03-dependency-injection.md`, `internal/cmd/config.go:193-291`). | The composition object can become a god object with broad, hard-to-test responsibilities. | When CLI and TUI must share runtime, config, service, stream, and persistence setup. |
| Constructor injection versus context-held services | Constructors retain compile-time dependency visibility and per-test isolation. | Explicit plumbing grows with each dependency; context lookup reduces signatures but hides requirements (`internal/view/xray.go:412`, `internal/dao/ds.go:72`). | When deciding how reviewer dependencies and progress metadata enter the shared review operation. |
| Fail-fast `errgroup` versus collect-all reviewers | Fail-fast cancellation reduces wasted work and propagates the first terminal error (`studies/go-cli-study/reports/final/08-concurrency.md`, `restic/internal/repository/repository.go:567`). | Early cancellation can prevent complete coverage evidence; collect-all requires separate error aggregation and more resource use. | When one reviewer fails but the product still needs truthful per-reviewer outcomes. |
| Parallel reviewers versus sequential execution | Bounded fan-out lowers wall-clock time and suits independent work (`k9s/internal/pool.go:21,30,37`). | It introduces ordering, race, cancellation, quota, and shutdown complexity; sequential work is simpler but slower. | When selecting the review concurrency bound and deterministic aggregation policy. |
| Streaming progress/findings versus final deterministic rendering | Streaming improves perceived latency and can bound retained output (`studies/go-cli-study/reports/final/09-terminal-ux.md`, `pkg/tasks/tasks.go:31-435`). | Concurrent events can interleave, destabilize tests, and contaminate JSON/stdout unless channels are separated. | When defining CLI progress, TUI updates, and final `review.md` ordering. |
| Broad filesystem interfaces versus narrow boundary seams | Broad fakes offer strong isolation (`studies/go-cli-study/reports/final/06-io-abstraction.md`, `internal/fs/interface.go:10-31`). | Large interfaces impose implementation and maintenance burden; narrow seams may require more focused fakes. | When testing atomic replacement, path containment, stale inputs, and failure injection. |
| Typed error taxonomy versus minimal wrapping | Typed errors carry stable structured facts and support `errors.As` (`studies/go-cli-study/reports/final/05-error-handling.md`, `helm/pkg/storage/driver/driver.go:27-48`). | A large taxonomy creates maintenance and classification ambiguity; sentinels are simpler but data-poor. | When mapping preflight, blocked, runtime, cancellation, validation, persistence, and verdict outcomes to JSON and exit codes. |
| Rich interactive UX versus CLI-first output | Progress and navigation improve long review supervision. | Terminal state, synchronization, accessibility, and cleanup add complexity; plain output remains easier to script and test (`studies/go-cli-study/reports/final/09-terminal-ux.md`, `internal/ui/termstatus/status.go:197-205`). | When choosing what belongs in shared progress DTOs versus TUI-only rendering. |
| Lazy initialization versus eager preflight | Lazy factories keep help, status, and version paths fast (`studies/go-cli-study/reports/final/14-performance.md`, `gh-cli/pkg/cmdutil/factory.go:27-42`). | First-use latency and cached-error behavior move failures later. | When deciding which review runtime/config checks occur before reviewer scheduling. |
| Strict permission prompts versus non-interactive execution | Interactive approval makes dangerous operations visible (`studies/go-cli-study/reports/final/13-security.md`, `internal/permission/permission.go:44-108`). | Prompts block CI and can cause approval fatigue; non-interactive defaults need a separate enforceable policy. | When defining read-only review permissions for CLI, JSON automation, and TUI confirmation. |

## Anti-Patterns And Warnings

- **Business behavior in command or TUI handlers:** Oversized handlers and hidden TUI command systems obscure the product boundary (`studies/go-cli-study/reports/final/02-command-architecture.md`, `cmd/root.go:49-183`, `internal/view/cmd/interpreter.go:62-86`). Avoid duplicating review orchestration, validation, verdict, or persistence in either interface.
- **Hidden mutable process-wide state:** Global configuration, singleton caches, and mutable concurrency controls undermine parallel review runs and test isolation (`studies/go-cli-study/reports/final/03-dependency-injection.md`, `internal/config/config.go:123`, `fs/cache/cache.go:16-21`; `studies/go-cli-study/reports/final/15-philosophy.md`, `pkg/analyze/parallel.go:13`).
- **Presence-only context use:** `context.Background()` below the root, stored contexts, context-based service location, and detached work can make cancellation cosmetic (`studies/go-cli-study/reports/final/07-state-context.md`, `cmd/dive/cli/internal/ui/v1/app/controller.go:18`, `k9s/internal/keys.go:10-38`, `task.go:341`).
- **Unbounded or fire-and-forget reviewer work:** Goroutine-per-item fan-out, indefinite waits, incompletely drained channels, or returning before workers stop can leak work and falsely report completion (`studies/go-cli-study/reports/final/08-concurrency.md`, `gh-cli/pkg/cmd/extension/manager.go:196-206`, `dive/cmd/dive/cli/internal/command/adapter/resolver.go:70`, `client.go:981`).
- **Mixed output channels:** A single direct `os.Stdout`, `os.Stderr`, or debug print can bypass capture and corrupt JSON or piped output (`studies/go-cli-study/reports/final/06-io-abstraction.md`, `cmd/ls/ls.go:42`, `cli.go:47`; `studies/go-cli-study/reports/final/10-logging-observability.md`, `fzf/src/core.go:325`).
- **Interactive assumptions in automation:** Unconditional prompts, ANSI output without TTY checks, or cancellation reported as success can hang CI or misstate review status (`studies/go-cli-study/reports/final/09-terminal-ux.md`, `pkg/action/package.go:200-208`, `internal/cmd/prompt.go:351-360`).
- **Lossy or string-based error handling:** Formatting without `%w`, comparing error strings, panicking on external input, and magic exit values destroy classification evidence (`studies/go-cli-study/reports/final/05-error-handling.md`, `mitchellh-cli/cli.go:205-206`, `dive/image/docker/manifest.go:18`).
- **Configuration bypass and silent correction:** Direct `os.Getenv`, ignored config errors, and unchanged flag defaults overriding loaded values make effective review policy hard to audit (`studies/go-cli-study/reports/final/04-configuration-management.md`, `internal/config/config.go:163`, `cmd/gdu/main.go:46-112`).
- **Documented rather than enforced safety:** Shell strings built from input, unvalidated paths, plaintext credential fallback, or plugin/agent permissions enforced only by convention leave the trust boundary open (`studies/go-cli-study/reports/final/13-security.md`, `internal/common/ignore.go:16-37`, `config.go:368-372`, `internal/llm/tools/bash.go:41-55`). A banned-command list is not filesystem or network isolation.
- **Tests coupled to implementation shape:** Fixed UI element counts, fragile regexes, and large inline output constants create noisy tests without proving stable behavior (`studies/go-cli-study/reports/final/11-testing-strategy.md`, `k9s/internal/view/pod_test.go:23`, `gh-cli/pkg/cmd/issue/list/list_test.go:88-91`, `mitchellh-cli/cli_test.go:1586-1614`).
- **Speculative optimization:** `sync.Pool`, GC tuning, global hot-path locks, and caches without eviction add lifecycle complexity without measured need (`studies/go-cli-study/reports/final/14-performance.md`, `rclone/lib/pool/pool.go:17-24,52-53`).

## Design Pressures

- The review operation has two different outcomes to represent truthfully: execution completion and the deterministic conformance verdict. Error, status, and observability shapes must not collapse them.
- Independent reviewers favor concurrency, while complete coverage, deterministic ordering, bounded resource use, and preservation of every failed/missing result constrain how that concurrency can be used.
- CLI text, JSON, and TUI need the same typed scope, progress, cancellation, result, and recovery facts, while terminal rendering and machine output need separate channels.
- A frozen input manifest and atomic artifact replacement create a consistency boundary: work may run concurrently, but the final artifact must describe one stable input identity and failed replacement must preserve the prior complete artifact.
- Review reads implementation and governed evidence while being prohibited from mutating them. Path containment, runtime permissions, output paths, and diagnostics therefore need enforceable boundaries rather than prompt-only instructions.
- Runtime/model configuration must be flexible enough for stage overrides but deterministic and inspectable enough that dry-run, status, and persisted metadata explain the effective source without exposing secrets.
- Cancellation must stop scheduling, reach active runtime work, await or bound cleanup, preserve truthful state, and produce an outcome distinguishable from pass or ordinary failure.
- Reviewer findings may arrive nondeterministically, but citation validation, severity rules, verdict synthesis, Markdown rendering, JSON output, and tests require stable ordering and reproducible results.
- The sprint needs failure-injection and fake-runtime coverage across runtime, output validation, path safety, persistence, race behavior, and TUI updates without broad interfaces or global test hooks becoming a second architecture.
- Product-owned review semantics must remain separate from generic runtime execution, while composition and adapters still need enough metadata for permissions, cancellation, observability, and safe diagnostics.

## Open Questions For Reasoning

- Should a failed reviewer cancel sibling reviewers immediately, or should the stage collect all possible results while ensuring a failed/missing reviewer prevents a passing verdict?
- What per-invocation concurrency limit balances review latency, runtime quotas, memory, filesystem pressure, and readable progress? Is any cross-run throttle needed?
- Should findings be streamed to interactive surfaces as they arrive and sorted only for final output, or should all externally visible result ordering wait for aggregation?
- Which dependencies belong in explicit review constructors, which are generic app composition concerns, and which metadata belongs in context without turning context into a service locator?
- Which review failures need stable typed identity for retry, blocked classification, JSON, or exit mapping, and which only need wrapped causes?
- What is the exact stdout/stderr/event-channel policy for text mode, JSON mode, prompt preview, dry-run, and TUI progress?
- Which configuration values may be overridden at review invocation, how are empty values interpreted, and when must effective configuration be frozen and validated?
- Which preflight checks should happen eagerly, and which expensive runtime dependencies should remain lazy so status, help, validation, and preview stay fast?
- Does deferred cleanup inherit the cancelled review context, or receive a fresh bounded context so runtime and temporary-resource cleanup can complete?
- What is the minimum filesystem abstraction needed for deterministic atomic-write failure, containment, staleness, and prior-artifact-preservation tests?
- Which generated text and JSON structures are stable contracts suitable for golden tests, and which timestamps, progress events, or diagnostics should be normalized rather than frozen?
- How should TUI confirmation and interactive permission feedback coexist with a default non-interactive, read-only review policy that cannot prompt in CI?
- What panic policy applies inside reviewer workers: process termination, conversion to a failed reviewer result, or another explicitly bounded outcome?
- Which implementation identity and changed-path evidence are sufficient when Git metadata is unavailable, and how should that choice affect fingerprinting and citation containment?
- Which observed abstractions serve this product's stated local, governed workflow, and which would add plugin, public API, or compatibility obligations outside sprint scope?
